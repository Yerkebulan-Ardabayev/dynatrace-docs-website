    Лучшие практики масштабирования тегирования и правил зон управления¶

    Лучшие практики масштабирования тегирования и правил зон управления¶

    • Explanation
    • 8-min read
    • Published May 19, 2021

    Автоматическое назначение тегов и зон управления помогает поддерживать порядок в окружении мониторинга и направляет внимание отдельных команд на инфраструктуру, за которую они отвечают.

    Dynatrace предоставляет мощные фильтры и условную обработку для выбора частей отслеживаемой топологии, которым должен присваиваться определённый тег или которые должны включаться в зону управления. Эти фильтры — условия, задаваемые при создании правила автоматического тегирования или зоны управления. Несмотря на наличие у Dynatrace ограничений стабильности, администратор каждого окружения мониторинга несёт ответственность за производительность тегирования и правил зон управления.

    Излишне сложные фильтры правил работают медленнее простых, а общее количество фильтров определяет скорость автоматического тегирования и назначения зон управления в окружении мониторинга. Для ориентира: цикл тегирования с небольшим числом фильтров (порядка 100) занимает примерно одну минуту; при тысячах сложных фильтров цикл может занимать несколько часов.

    Поскольку многие фильтры включают скрытый обход топологии и другие «дорогостоящие», времязатратные операции, данный раздел призван помочь оптимизировать скорость тегирования в вашем окружении. Приведённые ниже рекомендации помогут ускорить назначение тегов и зон управления.

    Минимизируйте количество правил¶

    Минимизация числа правил автоматического тегирования и правил зон управления — наиболее эффективный способ обеспечить быстрое назначение. В одной конфигурации тегирования или зоны управления можно использовать несколько правил, однако именно общее их число во всех конфигурациях влияет на назначение тегов и зон управления.

    Зоны управления могут строиться на основе тегированных объектов, поэтому назначение зон управления может зависеть от правил автоматического тегирования. Это также означает, что рабочий процесс тегирования должен последовательно оценивать теги и зоны управления, что замедляет назначение зон в зависимости от числа настроенных правил автоматического тегирования.

    Задержка цикла тегирования определяется прежде всего числом правил в сочетании с числом отслеживаемых объектов в окружении. Тысячи сложных правил могут быть приемлемы в небольших окружениях с сотнями хостов, однако в окружениях с тысячами хостов возможны значительные задержки тегирования.

    На одно окружение мониторинга можно настроить:

    • Любое количество автоматических тегов.
    • 300 правил автоматического тегирования объектов через интерфейс (150 для версий Dynatrace 1.323 и ниже).
    • 300 текстовых правил на основе entity selector для автоматического тегирования (150 для версий 1.323 и ниже).
    • 100 000 условий для всех правил автоматического тегирования вместе взятых (не применяется к правилам на основе entity selector).
    • Не более 1000 ручных тегов и 1000 автоматических тегов для каждого отдельного объекта.

    Дополнительно для одного окружения мониторинга можно настроить:

    • 5 000 зон управления по умолчанию. По дополнительным вопросам обратитесь к эксперту Dynatrace через онлайн-чат.
    • 300 правил зон управления для объектов через интерфейс (150 для версий 1.323 и ниже).
    • 300 правил зон управления для размерных данных через интерфейс (150 для версий 1.323 и ниже).
    • 300 текстовых правил на основе entity selector для зон управления (150 для версий 1.323 и ниже).
    • 100 000 условий для всех правил зон управления вместе взятых (не применяется к правилам на основе entity selector).
    • Назначение каждого отдельного объекта не более чем в 200 зон управления (выполните API-запрос GET an entity для просмотра зон управления объекта).

    Запросы топологии работают медленно¶

    Фильтры, выполняющие запросы топологии, используют свойство родительского объекта для фильтрации списка дочерних объектов. Например, можно использовать свойство хоста для фильтрации всех экземпляров групп процессов, работающих на этом хосте, или всех сервисов на этом хосте.

    Как бы полезны они ни были, такие запросы топологии дорогостоящи и медленны: каждый фильтр вынужден обходить топологию, в ряде случаев через несколько шагов, для назначения тега или извлечения значения для данного объекта. Обход топологии особенно дорог в крупных окружениях, где каждый такой фильтр добавляет значительную задержку к общему назначению тегов и зон управления.

    Если действительно необходимо передавать значения от одного типа объекта к другому (например, от хоста к сервисам), сохраняйте очень небольшое число таких сложных правил.

    По возможности избегайте извлечения значений¶

    Назначение одного тега выполняется быстро. Извлечение значений из свойства данного объекта выполняется медленно. Если правило извлечения значений можно заменить назначением одного тега с использованием статического правила, это ускорит общую производительность тегирования.

    В случаях, когда число значений является ограниченным статическим набором, несколько простых статических правил могут заменить правило автоматического извлечения значений.

    Например, если у вас работает несколько кластеров Kubernetes, вы можете использовать фильтр извлечения для назначения идентификатора кластера K8S в качестве значения в правиле. Это можно легко заменить одним правилом для каждого из ваших кластеров Kubernetes. Такой подход менее гибок, чем правило извлечения, но значительно быстрее.

    Используйте многоуровневые фильтры для сужения результатов поиска¶

    Если «дорогостоящее» условие фильтрации, например регулярное выражение, неизбежно, можно также настроить «дешёвый» предварительный фильтр, который сокращает общее число проверяемых объектов.

    Пример дешёвого фильтра — фильтр «имя начинается с», который сначала сужает набор объектов до значительно меньшего подмножества, после чего можно безопасно применить дорогостоящий фильтр на основе регулярных выражений.

    Порядок фильтров в конфигурации одного правила не имеет значения: Dynatrace всегда выбирает сначала самый дешёвый фильтр для сокращения результирующего набора перед применением дорогостоящих фильтров.

    Избегайте регулярных выражений или запросов «содержит»¶

    Несмотря на огромную мощь регулярных выражений, их главный недостаток — значительно более медленное вычисление по сравнению с простыми фильтрами «начинается с» или «равно». То же самое относится к запросам «содержит», при которых вся строка проверяется на наличие подстроки.

    Запросы «начинается с» или «равно» значительно эффективнее запросов «содержит» или запросов на основе регулярных выражений.

    Один из наиболее эффективных способов организации данных и топологии — использование общей схемы именования с иерархической структурой, которая обеспечивает максимальную эффективность при построении запросов с фильтрами «начинается с».

    Пример эффективного иерархического шаблона именования, где иерархия выстроена от наиболее общего к наиболее конкретному слева направо:

    Схема именования объектов: <stage>.<app>.<cluster>.<entityname>

    Используйте ручные, массовые или теги окружения¶

    Помимо правил автоматического тегирования, существуют альтернативные способы тегирования объектов, обладающие определёнными преимуществами в части производительности.

    Ручное тегирование¶

    Одна из альтернатив автоматическому тегированию — ручное тегирование объектов. Это может быть неудобно для большого набора объектов, но работает для небольшого набора стабильных объектов.

    Поскольку ручные теги могут использоваться автоматическим правилом зоны управления, такой подход позволяет улучшить общую производительность назначения тегов и зон управления.

    Custom tags API¶

    Когда объектов слишком много для эффективного ручного тегирования, Dynatrace предлагает Custom tags API для удобного назначения тега большой группе объектов в рамках одного API-вызова. Мощь этого подхода определяется стандартизированным entity selector — гибким и мощным инструментом выбора объектов.

    Entity selector также может содержать запрос топологии, например запрос всех дисков заданного набора хостов, фильтруемых по любому атрибуту объекта. Ниже приведены примеры использования entity selector.

    Следующий entity selector выбирает все сервисы, имя которых начинается с book:

    type(SERVICE),entityName.startsWith("Book")

    Следующий entity selector выбирает все экземпляры контейнерных групп заданного кластера Kubernetes:

    type(CONTAINER_GROUP_INSTANCE),fromRelationship.isCgiOfCluster(type(KUBERNETES_CLUSTER) AND entityName.equals(my-prod-cluster-name))

    API-вызов выполняется немедленно, что является существенным преимуществом перед правилами автотегирования, которые планируются через процесс тегирования Dynatrace. Это помогает ускорить выполнение там, где необходимы сложные правила тегирования.

    Недостаток этого подхода в том, что это разовое действие тегирования: вновь обнаруженные объекты, соответствующие правилу, не будут тегированы автоматически. Это легко решается регулярным API-запросом (например, запускаемым внешним cron-заданием).

    Теги окружения¶

    OneAgent обеспечивает удобный способ чтения и назначения тегов из локальных переменных окружения. Использование тегов окружения вместо правил автоматического тегирования значительно улучшает производительность тегирования, поскольку каждый экземпляр OneAgent самостоятельно назначает такие теги в высокораспределённом режиме. Более того, этот подход быстр и никак не замедляет выполнение серверных правил тегирования.

    Поскольку большинство современных автоматизированных систем развёртывания всё равно используют скрипты и рецепты развёртывания, такие рецепты также можно применять для определения тегов Dynatrace через системные переменные окружения.

    Используя переменные окружения для создания и назначения тегов, вы передаёте управление назначением тегов собственному рецепту развёртывания и не зависите от дорогостоящего правила тегирования.

    Используйте дешёвые атрибуты вместо дорогостоящих¶

    Значительная часть модели топологии Dynatrace Smartscape индексируется в памяти для обеспечения быстрого выполнения запросов. Dynatrace хранит индекс большинства стабильных атрибутов объектов, избегая индексации высоковолатильной информации.

    Это означает, что правила тегирования и зон управления, определённые на основе индексированных атрибутов объектов, значительно дешевле, чем правила, запрашивающие атрибуты, отсутствующие в индексе топологии.

    Следующие атрибуты объектов следует избегать, так как они недоступны через индекс топологии:

    • Имя контейнера Docker
    • Полное имя образа Docker
    • Топология сервиса
    • Программная технология
    • Версия программной технологии
    • Версия образа Docker
    • Сокращённое имя образа Docker
    • Метки Kubernetes
    • Метки облачного приложения
    • Метки пространства имён облачного приложения
    • Тип экземпляра виртуальной машины OpenStack
    • Публичные IP-адреса GCE
    • IP-адреса хоста
    • IP-адреса пользовательского устройства
    • Лимит памяти PaaS
    • Тип синтетического движка
    • Имя синтетического движка
    • Группы безопасности EC2
    • Внешние порты ELB
