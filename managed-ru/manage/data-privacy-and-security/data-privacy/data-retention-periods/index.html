    Сроки хранения данных¶

    Сроки хранения данных¶

    • Чтение 8 мин
    • Updated on Jan 28, 2026

    Dynatrace хранит и сохраняет различные типы данных мониторинга из ваших окружений. Данные мониторинга хранятся в кластере Dynatrace. В следующей таблице приведены общие сроки хранения для данных сервисов (распределённые трассы), Real User Monitoring (действия пользователей и пользовательские сессии), синтетических мониторов и метрик временных рядов.

    Пробные аккаунты¶

    После истечения срока действия 15-дневного пробного аккаунта Dynatrace продолжает хранить данные мониторинга этого аккаунта ещё 30 дней, чтобы гарантировать отсутствие потери данных.

    Платные аккаунты¶

    Для активных аккаунтов Dynatrace по умолчанию установлены следующие сроки хранения:

    Тип данных Срок хранения
    Распределённые трассы Настраиваемый, с максимальным сроком хранения 365 дней
    Сервисы: запросы и атрибуты запросов Настраиваемый, с максимальным сроком хранения 365 дней
    RUM: данные действий пользователей Настраиваемый, с максимальным сроком хранения 35 дней
    RUM: пользовательские сессии 35 дней
    RUM: сбои мобильных приложений 35 дней
    RUM: Session Replay Настраиваемый, с максимальным сроком хранения 35 дней
    Synthetic Настраиваемый, с максимальным сроком хранения 35 дней
    Log Monitoring Classic 35 дней
    Metrics Classic 5 лет
    Диагностика OneAgent (архивы поддержки и результаты анализа) Настраиваемый, по умолчанию 30 дней
    Проблемы и события Davis 14 месяцев
    Данные безопасности Classic Открытые уязвимости: хранятся до устранения; Устранённые уязвимости: 365 дней; События: 365 дней; Атаки: 550 дней

    Distributed traces Classic¶

    Dynatrace хранит полные сведения о каждой транзакции в течение максимум 365 дней в зависимости от конфигурации. Это позволяет анализировать отдельные транзакции и получать все доступные сведения с помощью инструментирования.

    Для пробных пользователей применяется дополнительное ограничение по размеру хранилища, что может приводить к сокращению сроков хранения.

    Данные хранятся в хранилище транзакций по пути DATASTORE_PATH/server/tenantData. Данные не реплицируются между узлами кластера.

    Детализация на уровне кода¶

    Детализация на уровне кода доступна при инструментировании OneAgent в течение 10 дней.

    По истечении 10 дней данные сессий оптимизируются для агрегированных представлений. Неагрегированные и агрегированные данные на уровне кода дают сопоставимые результаты для более длительных временных интервалов, тогда как для более коротких интервалов могут наблюдаться различия.

    Services Classic: запросы и атрибуты запросов¶

    Краткосрочное хранение данных, связанных с метриками сервисов, используемых в многомерном анализе и построении графиков запросов. Эти данные доступны в течение максимум 365 дней в зависимости от конфигурации со следующими уровнями детализации интервалов:

    Временной интервал Детализация интервала
    Менее 20 минут 10 секунд
    20–40 минут 20 секунд
    40–60 минут 30 секунд
    Более 1 часа 1 минута

    Краткосрочный анализ обращается к данным на уровне кода, доступным в течение 10 дней.

    По истечении 10 дней данные сессий оптимизируются для агрегированных представлений. Неагрегированные и агрегированные данные на уровне кода дают сопоставимые результаты для более длительных временных интервалов, тогда как для более коротких интервалов могут наблюдаться различия.

    Данные хранятся в хранилище транзакций по пути DATASTORE_PATH/server/tenantData. Данные не реплицируются между узлами кластера.

    RUM: данные действий пользователей¶

    Агрегированные метрики действий пользователей, используемые в таблицах Top user actions и Top JavaScript errors, доступны в течение максимум 365 дней в зависимости от конфигурации. По истечении 10 дней данные действий пользователей оптимизируются для агрегированных представлений, и некоторые отдельные действия пользователей становятся недоступны для индивидуального анализа. Однако выборка достаточно велика для статистически корректных агрегаций.

    Для ключевых действий пользователей необработанные данные действий также хранятся в течение максимум 365 дней в зависимости от конфигурации. Срок хранения данных временных рядов ключевых действий пользователей совпадает со сроком хранения метрик временных рядов.

    Данные хранятся в хранилище транзакций по пути DATASTORE_PATH/tenantData. Данные не реплицируются между узлами кластера.

    RUM: пользовательские сессии¶

    Включает данные Session Replay. Все данные пользовательских сессий хранятся в течение максимум 365 дней в зависимости от конфигурации. Обратите внимание, что данные анализа водопадных диаграмм и ошибок JavaScript хранятся вместе с детализацией на уровне кода распределённых трасс и ошибками.

    Данные хранятся в хранилище Elasticsearch по пути DATASTORE_PATH/elasticsearch. Данные реплицируются между узлами кластера. Коэффициент репликации равен трём.

    RUM: сбои мобильных приложений¶

    Включает все данные о сбоях и трассировки стека мобильных и пользовательских приложений. Данные хранятся в течение максимум 365 дней в зависимости от конфигурации.

    Примечание

    Данные об аварийных завершениях работы, отображаемые на подробной странице приложения, могут отличаться от данных на странице статистики. Для этих страниц данные извлекаются из разных хранилищ и имеют разные сроки хранения.

    Количество аварийных завершений на подробной странице приложения

    Общее количество аварийных завершений на странице статистики

    Данные хранятся в хранилище Elasticsearch по пути DATASTORE_PATH/elasticsearch. Данные реплицируются между узлами кластера. Коэффициент репликации равен трём.

    RUM: Session Replay¶

    Минимальный размер требуемого тома хранилища Session Replay полностью зависит от нагрузки. Максимальный размер не ограничен.

    Каталог хранилища данных Session Replay — это специализированное файловое хранилище по пути DATASTORE_PATH/server/replayData, используемое исключительно для данных Session Replay. Подробнее о рекомендациях по размеру хранилища см. в разделе Настройка вторичного диска.

    Log Monitoring Classic¶

    Log Monitoring Classic позволяет хранить все логи централизованно во внешнем хранилище. Это делает данные логов доступными независимо от самих файлов логов.

    Для централизованного хранения файлов логов уже используется хранилище Elasticsearch по пути DATASTORE_PATH/elasticsearch для хранения файлов логов в кластере Dynatrace Managed. Коэффициент репликации равен двум. Для настройки хранилища логов см. раздел Правила приёма логов (Logs Classic).

    Дампы памяти¶

    Дампы памяти немедленно удаляются с диска после загрузки на ActiveGate. Если загрузка невозможна, дампы памяти объёмом до 20 ГБ хранятся на диске до 2 часов.

    Metrics Classic¶

    Для доступа через панели мониторинга и API доступны следующие уровни детализации интервалов:

    Временной интервал Детализация интервала
    0–14 дней 1 минута
    14–28 дней 5 минут
    28–400 дней 1 час
    400 дней — 5 лет 1 день

    Данные хранятся в репозитории метрик по пути DATASTORE_PATH/cassandra. Данные реплицируются между узлами кластера. Коэффициент репликации равен трём.

    Для обеспечения точных вычислений метрик временных рядов Dynatrace использует алгоритм P2 для динамического вычисления квантилей. Этот алгоритм известен хорошими результатами и хорошо работает со значениями в длинных хвостах распределений значений. Однако алгоритм агрегации не является ни ассоциативным ((a + b) + c == a + (b + c)), ни коммутативным (a + b + c == c + b + a). Для некоторых метрик, например времени отклика, это может приводить к различным значениям квантилей при каждом запуске алгоритма или при агрегации данных разными способами, например когда одна метрика разбита по URL, а другая — по браузеру.

    Диагностика OneAgent и ActiveGate¶

    Диагностика OneAgent и ActiveGate — это дополнительные функции, позволяющие собирать и анализировать архивы поддержки на наличие аномалий.

    Архивы поддержки создаются Dynatrace OneAgent или Dynatrace ActiveGate и хранятся в Cassandra, где автоматически удаляются через 30 дней. Когда вы разрешаете Dynatrace анализировать проблему, дополнительная копия архива поддержки сохраняется в настроенном бакете AWS S3. Результаты анализа и архив поддержки также автоматически удаляются из бакета AWS S3 через 30 дней. Dynatrace OneAgent и Dynatrace ActiveGate не хранят копий созданных архивов поддержки.

    Вы можете удалить проблемы диагностики OneAgent или ActiveGate в любое время. При удалении проблемы связанный архив поддержки и отчёт об анализе немедленно удаляются из Cassandra и бакета AWS S3. Результат анализа в Dynatrace Health Control удаляется через 30 дней.

    Данные безопасности Classic¶

    Уязвимости¶

    • Открытые уязвимости сторонних компонентов и уязвимости на уровне кода хранятся, пока они остаются открытыми, независимо от временного интервала.

    • Срок хранения устранённых уязвимостей сторонних компонентов и уязвимостей на уровне кода зависит от времени их устранения:

    • Если уязвимость устранена до истечения 365 дней с момента первого обнаружения, она удаляется через 365 дней.

    • Если уязвимость устранена после истечения 365 дней с момента первого обнаружения, она удаляется в ближайшую годовщину даты первого обнаружения.

    Примеры:

    Дата первого обнаружения Дата первого устранения Повторное открытие Повторное устранение Дата удаления
    2022-05-12 2023-05-06 2023-05-13
    2022-05-12 2023-08-06 2024-05-13
    2022-05-12 2023-08-06 2024-01-01 2024-02-08 2024-05-13

    События¶

    События эволюции уязвимостей сторонних компонентов и события эволюции уязвимостей на уровне кода хранятся в течение 365 дней и могут запрашиваться только до временной метки первого обнаружения уязвимости.

    Временная метка первого обнаружения

    Атаки¶

    Атаки хранятся в течение 550 дней.

    Поскольку управление сроками хранения осуществляется ежемесячными интервалами, точная продолжительность может незначительно варьироваться в зависимости от времени возникновения атаки и времени выполнения очистки.
