    Хранилище учётных данных¶

    Хранилище учётных данных¶

    • Руководство
    • Чтение 21 мин
    • Обновлено 27 сен 2024 г.

    Хранилище учётных данных — это централизованный репозиторий для безопасного хранения и управления учётными данными (пары имя пользователя — пароль, сертификаты или токены), используемыми синтетическими мониторами (browser и HTTP), расширениями Extensions, а также приложениями AppEngine.

    Чтобы перейти в хранилище учётных данных, откройте Credential Vault.

    Безопасность учётных данных¶

    Учётные данные хранятся в зашифрованном виде по стандарту Advanced Encryption Standard (AES-256). Доступ к данным шифруется с использованием TLS 1.3. Содержимое учётных данных в хранилище не видно ни одному пользователю, включая создателя; оно доступно только синтетическим мониторам, которые на них ссылаются. Пользователи с доступом к хранилищу учётных данных могут перезаписать некоторые или все учётные данные — подробнее об этом в разделе Кто может редактировать или перезаписывать учётные данные.

    Архитектура безопасности и взаимодействия хранилища учётных данных

    Хранилище учётных данных разработано полностью Dynatrace и расположено в Dynatrace Cluster. Ниже описаны ключевые компоненты синтетического мониторинга, участвующие в безопасности и взаимодействии хранилища учётных данных. Все коммуникации между ключевыми компонентами зашифрованы.

    Взаимодействие хранилища учётных данных

    • Dynatrace Cluster

    Функция: Dynatrace Cluster хранит хранилище учётных данных в базе данных кластера как часть вашей среды мониторинга Dynatrace.

    Доступ к учётным данным и безопасность: Все учётные данные создаются, изменяются и удаляются на Dynatrace Cluster. Хранилище учётных данных привязано к конкретному окружению и не используется совместно между окружениями. Только кластер имеет доступ к хранилищу учётных данных и разрешает идентификаторы учётных данных в фактические значения при подготовке мониторов к выполнению.

    Взаимодействие: Все коммуникации между ключевыми компонентами зашифрованы. Если монитор нужно выполнить из частного Synthetic-расположения, кластер разрешает и отправляет учётные данные в составе конфигурации монитора в Synthetic engine при создании или обновлении (когда мониторы или связанные учётные данные создаются или обновляются), а не перед каждым выполнением. Аналогично, если монитор нужно выполнить из публичного Synthetic-расположения, кластер разрешает и отправляет учётные данные в публичную Synthetic-инфраструктуру при создании или обновлении.

    • Synthetic engine

    Функция: Synthetic engine выполняет синтетические мониторы как в частных, так и в публичных Synthetic-расположениях и имеет доступ к конфигурациям мониторов и разрешённым учётным данным для этой цели.

    Взаимодействие: Synthetic engine не взаимодействует напрямую с хранилищем учётных данных и не видит его содержимого. На частных Synthetic-расположениях Synthetic engine запрашивает и получает расписание мониторов для выполнения и их конфигурации (с разрешёнными учётными данными) от Dynatrace Cluster при создании или обновлении, то есть когда мониторы или связанные учётные данные создаются или обновляются. На публичных Synthetic-расположениях это же взаимодействие происходит с публичной Synthetic-инфраструктурой. Все коммуникации между ключевыми компонентами зашифрованы.

    Доступ к учётным данным и безопасность: Synthetic engine имеет доступ только к тем разрешённым учётным данным, которые являются частью мониторов, которые ему нужно выполнить; он не может получить или просмотреть другие учётные данные. Все конфигурации мониторов хранятся в памяти Synthetic engine (без уровней постоянного хранения) до тех пор, пока мониторы должны выполняться. Если монитор выключен или удалён, его конфигурация удаляется из памяти Synthetic engine.

    Все данные учётных данных передаются через внутренний API, защищённый API-токеном, и доступ строго ограничен на уровне сетевого протокола. Каждое выполнение монитора изолировано; память или скрипты между выполнениями не используются совместно.

    • Публичная Synthetic-инфраструктура

    Функция: Включает всю инфраструктуру, необходимую для выполнения мониторов из публичных расположений. Публичная Synthetic-инфраструктура хранит и передаёт информацию в публичные Synthetic-расположения для выполнения синтетических мониторов.

    Взаимодействие: Если монитор нужно выполнить из публичного Synthetic-расположения, Dynatrace Cluster отправляет конфигурацию монитора (с разрешёнными учётными данными) в публичную Synthetic-инфраструктуру при создании или обновлении (когда мониторы или связанные учётные данные создаются или обновляются), а не перед каждым выполнением. Публичная Synthetic-инфраструктура затем передаёт ту же информацию в публичные Synthetic-расположения, как кластер делает для частных. Все коммуникации между ключевыми компонентами зашифрованы.

    Доступ к учётным данным и безопасность: Конфигурации мониторов с разрешёнными учётными данными шифруются и сохраняются в базе данных Synthetic. Учётные данные не сохраняются напрямую как учётные данные, а только как часть конфигураций мониторов.

    Dynatrace управляет и обеспечивает безопасность всей публичной Synthetic-инфраструктуры и публичных Synthetic-расположений. Доступ возможен только из корпоративной сети Dynatrace и защищён аутентификацией по SSH-ключу. Доступ к закрытым ключам ограничен небольшой группой сотрудников Dynatrace.

    Доступ к хранилищу учётных данных¶

    Для просмотра и записи в хранилище учётных данных пользователю необходимо разрешение View environment (просмотр окружения) на уровне окружения (управление разрешениями).

    Если у вас есть только разрешение View environment (без других более высоких разрешений, таких как Change monitoring settings), вы можете:

    • Просматривать хранилище учётных данных.
    • Просматривать все «публичные» учётные данные и «только для владельца», которыми вы владеете.
    • Создавать учётные данные в хранилище. Однако использовать синхронизацию с внешним хранилищем не разрешается.
    • Перезаписывать любые учётные данные (публичные или только для владельца), которыми вы владеете; перезаписывать учётные данные другого пользователя не разрешается.
    • Удалять любые учётные данные (публичные или только для владельца), которыми вы владеете; удалять учётные данные другого пользователя не разрешается.

    Для перезаписи и удаления учётных данных, принадлежащих другим пользователям, необходимо разрешение Change monitoring settings на уровне окружения.

    Пользователи с разрешением View environment или Change monitoring settings на уровне зоны управления могут только просматривать публичные учётные данные в хранилище; создавать учётные данные в хранилище они не могут.

    Для создания или редактирования синтетических мониторов необходимо разрешение Change monitoring settings (управление разрешениями) на уровне окружения или зоны управления.

    Подробнее о требуемой области действия токена для доступа к хранилищу учётных данных через API — в разделе API хранилища учётных данных ниже.

    Учётные данные только для владельца, общие и публичные¶

    Каждые учётные данные имеют уровни доступа. При первоначальном создании (в хранилище или при создании и редактировании синтетического монитора) учётным данным присваивается уровень Owner only (только для владельца).

    Владелец учётных данных может отключить переключатель Owner access only в хранилище, чтобы изменить уровень доступа учётных данных на All (все). Владелец также может ограничить доступ к учётным данным для указанных пользователей, предоставив разделённый запятыми список имён пользователей (который может включать адреса электронной почты, используемые как имена пользователей) в поле Users with access, например username1, username2, userx@company.com. Уровень доступа учётных данных в этом случае устанавливается как Shared (общие).

    Чтобы предоставить другим пользователям доступ к учётным данным, отключите Owner access only. Если вы отключите Owner access only и не ограничите доступ к именам пользователей в поле Users with access, учётные данные будут доступны всем пользователям.

    Другие пользователи могут изменить уровень доступа учётных данных, став новым владельцем и перезаписав их с новыми данными аутентификации.

    Пользователи с разрешением View environment на уровне окружения могут перезаписывать только те учётные данные, которыми они владеют.

    Уровень доступа учётных данных (Owner only, Shared или All) определяет, кто может использовать их в синтетическом мониторе. Учётные данные только для владельца может использовать только сам владелец, например, для создания или редактирования синтетического монитора. Общие учётные данные доступны для использования указанному списку пользователей, например, в синтетических мониторах и приложениях. Публичные учётные данные доступны всем пользователям.

    Подробнее в разделе Работа с учётными данными.

    Области действия учётных данных¶

    Область действия учётных данных представляет сценарий использования; можно выбрать несколько областей одновременно. В таблице ниже описана каждая область и применимые типы учётных данных.

    Область Типы учётных данных Описание
    Synthetic * Имя пользователя и пароль * Сертификат * Токен Учётные данные для использования в синтетических мониторах
    Extension authentication * Имя пользователя и пароль Учётные данные, используемые расширениями (источниками данных) для аутентификации в ресурсе и получения данных
    Extension validation * Публичный сертификат Используется для подписи расширений
    AppEngine * Имя пользователя и пароль * Сертификат * Токен * Публичный сертификат Для предоставления доступа к учётным данным из приложений Dynatrace (последняя версия Dynatrace) и ad hoc функций (без контекста приложения)

    Область AppEngine¶

    Выберите область AppEngine, чтобы разрешить встроенным приложениям из последней версии Dynatrace доступ к учётным данным. Эта область доступна для всех типов учётных данных.

    При первом выборе области для учётных данных доступ по умолчанию устанавливается на All applications (все приложения) в поле Dynatrace apps with access.

    Значение по умолчанию для области AppEngine

    Для ограничения доступа к учётным данным конкретными приложениями установите курсор в поле и выберите из списка доступных встроенных приложений в вашем окружении. Можно выбрать несколько приложений по одному. Доступ к учётным данным теперь ограничен выбранными приложениями.

    Выбранные приложения для области AppEngine

    При желании включите Allow access without app context (разрешить доступ без контекста приложения). Это позволяет ad hoc функциям, например из Workflows и Notebooks, получать доступ к учётным данным вне контекста приложения.

    • Нельзя разрешить доступ без контекста приложения для учётных данных с публичным доступом. Это означает, что Dynatrace не позволит создавать новые учётные данные, если они одновременно удовлетворяют всем трём условиям:

    • Allow access without app context включён.

    • Owner access only отключён.
    • Список Users with access пуст.

    Учётные данные, созданные до версии Dynatrace 1.337+ с такой конфигурацией, будут работать как есть, если вы не начнёте их редактировать. При редактировании таких учётных данных сохранить изменения не удастся, пока хотя бы одно из указанных выше условий не будет нарушено. Например, можно сохранить такие учётные данные после добавления пользователя в список Users with access или включения Owner access only.

    • Чтобы пользователь мог обращаться к учётным данным в контексте приложения, и пользователь, и приложение должны иметь доступ к учётным данным по отдельности. Если пользователь X имеет доступ к учётным данным, а приложение A — нет, пользователь X не сможет использовать учётные данные из приложения A.

    Просмотр хранилища учётных данных¶

    Перейдите в Credential Vault, чтобы увидеть все учётные данные в вашем окружении.

    Пользователи с разрешением View environment на уровне окружения могут просматривать только публичные учётные данные и те, которыми они владеют.

    Доступные типы учётных данных для синтетического мониторинга: пары имя пользователя — пароль, сертификаты и токены. Пары имя пользователя — пароль можно синхронизировать с внешними хранилищами — подробнее в Интеграция с внешним хранилищем.

    Каждые учётные данные отображаются с иконкой, идентифицирующей Type (тип), Name (имя), мониторы, в которых они используются (Used by), Owner (владелец), уровень Access (доступа), Scope (область) и элементы управления для редактирования и Delete (удаления).

    Список можно фильтровать по Type, Name, Access, Owner или Scope (выберите Synthetic).

    Хранилище учётных данных

    Вы можете видеть идентификатор и свойства учётных данных, но содержимое учётных данных не отображается; его можно только перезаписать. Содержимое учётных данных видно только синтетическим мониторам, приложениям или ad hoc функциям, которые на них ссылаются.

    Если у вас нет доступа к учётным данным, вы увидите сообщение об использовании или перезаписи учётных данных. Смотрите раздел Кто может редактировать или перезаписывать учётные данные ниже.

    Выберите HTTP или Browser рядом с учётными данными, чтобы увидеть связанные мониторы на странице Synthetic. Список автоматически фильтруется по имени и владельцу учётных данных.

    Мониторы, связанные с учётными данными

    При необходимости включите Turn on local playback of Synthetic browser monitors without entering credentials, чтобы пользователи могли воспроизводить browser clickpaths локально без ввода связанных токенов или пар имя пользователя — пароль, к которым у них есть доступ. Если отключено, пользователю нужно вводить связанные учётные данные для локального воспроизведения browser clickpaths. Обратите внимание, что воспроизводить browser clickpath могут только пользователи, имеющие доступ к соответствующим учётным данным.

    Создание учётных данных в хранилище¶

    Создавать учётные данные можно непосредственно в хранилище или в процессе создания и редактирования синтетического монитора.

    Для синтетического мониторинга можно создать учётные данные следующих типов:

    • Пары имя пользователя — пароль
    • Сертификатные учётные данные
    • Токен-учётные данные

    Пары имя пользователя — пароль¶

    Пары имя пользователя — пароль можно использовать для базовой аутентификации, а также аутентификации через веб-форму в browser-мониторах с одним URL, browser clickpath и HTTP-мониторах.

    Пары имя пользователя — пароль в хранилище учётных данных Dynatrace можно синхронизировать с внешним хранилищем (Azure Key Vault, HashiCorp Vault или CyberArk Vault). Синтетические мониторы, содержащие эти учётные данные, используют синхронизированные значения, полученные из внешних хранилищ — подробнее в Интеграция с внешним хранилищем.

    Создание учётных данных для входа в хранилище

    1. Нажмите Add new credential в правом верхнем углу.
    2. Выберите User and password в качестве Credential type.
    3. Выберите Synthetic для Credential scope. Подробнее о других областях учётных данных, включая область AppEngine.
    4. Измените стандартное Credential name для правильной идентификации новых учётных данных.
    5. Если синхронизируете пару имя пользователя — пароль с внешним хранилищем (Synchronization with external vault), следуйте инструкциям в Интеграция с внешним хранилищем.
    6. Введите Username и Password. Пароль автоматически маскируется при вводе. Обратите внимание, что эти поля не отображаются для синхронизированных учётных данных.

    Поддерживаемые форматы имён пользователей

    • Browser-мониторы: <username> и <domain>\<username>
    • HTTP-мониторы: <username>
    • NTLM-аутентификация в browser- и HTTP-мониторах: <username>
    • При желании укажите Description (описание).
    • По умолчанию учётным данным устанавливается уровень Owner access only. Отключите этот переключатель, чтобы сделать учётные данные доступными другим пользователям. При желании ограничьте доступ для указанных пользователей, предоставив разделённый запятыми список имён пользователей в Users with access. Подробнее в разделах Учётные данные только для владельца, общие и публичные и Работа с учётными данными.
    • Сохраните учётные данные. Обратите внимание, что после создания содержимое учётных данных можно только перезаписать.

    Создание пары имя пользователя — пароль в хранилище

    Сертификатные учётные данные¶

    Сертификатные учётные данные используются в HTTP-мониторах — вы можете добавить клиентский сертификат к HTTP-запросу.

    Создание сертификатных учётных данных в хранилище

    1. Нажмите Add new credential в правом верхнем углу.
    2. Выберите Certificate в качестве Credential type.
    3. Выберите Synthetic для Credential scope. Подробнее о других областях учётных данных, включая область AppEngine.
    4. Загрузите Certificate file. Допустимые форматы файлов: PFX, P12 и PEM.

    Если возникают проблемы с использованием PEM-сертификата, смотрите раздел Конвертация PEM-сертификатов ниже. 5. Введите Certificate password (пароль сертификата). 6. Укажите Credential name и при желании Description. 7. По умолчанию учётным данным устанавливается уровень Owner access only. Отключите этот переключатель, чтобы сделать учётные данные доступными другим пользователям. При желании ограничьте доступ для указанных пользователей, предоставив разделённый запятыми список имён пользователей в Users with access. Подробнее в разделах Учётные данные только для владельца, общие и публичные и Работа с учётными данными. 8. Сохраните учётные данные. Обратите внимание, что после создания содержимое учётных данных можно только перезаписать.

    Конвертация PEM-сертификатов¶

    Если при создании учётных данных с использованием PEM-сертификата возникают проблемы, рассмотрите возможность конвертации файла сертификата в форматы P12 или PFX, которые поддерживаются стандартами Java.

    Используйте утилиту командной строки openssl для конвертации файла сертификата. Например, следующая команда конвертирует PEM-сертификат в формат P12:

    openssl pkcs12 -export -in /path/to/certificate.pem -out /path/to/certificate.p12

    Токен-учётные данные¶

    Токен — это универсальный тип учётных данных с одним значением. Вы можете создавать токены в хранилище учётных данных и вставлять ссылки на них из HTTP-мониторов — в URL запросов, значениях HTTP-заголовков и в теле запроса. В clickpaths вы можете вставить идентификатор токена в событие Keystroke.

    Токен-учётные данные в хранилище учётных данных Dynatrace можно синхронизировать с внешним хранилищем (Azure Key Vault или HashiCorp Vault). Синтетические мониторы, содержащие эти учётные данные, используют синхронизированные значения, полученные из внешних хранилищ — подробнее в Интеграция с внешним хранилищем.

    Создание токен-учётных данных в хранилище

    1. Нажмите Add new credential в правом верхнем углу.
    2. Выберите Token в качестве Credential type.
    3. Выберите Synthetic для Credential scope. Подробнее о других областях учётных данных, включая область AppEngine.
    4. Если синхронизируете токен с внешним хранилищем (Synchronization with external vault), следуйте инструкциям в Интеграция с внешним хранилищем.
    5. Введите значение Token.
    6. Укажите Credential name и при желании Description.
    7. По умолчанию учётным данным устанавливается уровень Owner access only. Отключите этот переключатель, чтобы сделать учётные данные доступными другим пользователям. При желании ограничьте доступ для указанных пользователей, предоставив разделённый запятыми список имён пользователей в Users with access. Подробнее в разделах Учётные данные только для владельца, общие и публичные и Работа с учётными данными.
    8. Сохраните учётные данные. Обратите внимание, что после создания содержимое учётных данных можно только перезаписать.

    Работа с учётными данными¶

    Уровень доступа учётных данных (только для владельца или публичный) определяет, кто может использовать учётные данные для:

    • Создания нового синтетического монитора или связывания существующего монитора с учётными данными.
    • Редактирования монитора, связанного с учётными данными.

    Пользователи могут удалять/перезаписывать учётные данные и изменять их уровни доступа.

    Кто может использовать учётные данные в мониторе¶

    Эти операции требуют разрешения Change monitoring settings (управление разрешениями).

    При создании монитора или редактировании существующего монитора без связанных учётных данных вы можете:

    • Использовать существующие учётные данные из хранилища в мониторе — вы можете выбирать публичные учётные данные или те, которыми владеете, или которые предоставлены вам совместно.

    • Создать новые учётные данные в процессе создания/редактирования монитора. Учётные данные автоматически назначаются как только для владельца и сохраняются в хранилище. После создания учётных данных другой пользователь может изменить уровень доступа, став новым владельцем и перезаписав их с новыми данными аутентификации.

    • Вы можете сохранять пароли, записанные в clickpaths, в хранилище (с именем пользователя или как токен с одним значением). Они сохраняются как только для владельца. Смотрите, как использовать событие Keystroke.

    Либо вы можете отредактировать записанное событие, чтобы использовать существующие учётные данные из хранилища или создать собственные непосредственно в clickpath.

    Кто может редактировать монитор, связанный с учётными данными¶

    Эти операции требуют разрешения Change monitoring settings (управление разрешениями).

    • Если монитор связан с публичными учётными данными, любой член вашей команды может включать/выключать, удалять или редактировать монитор.

    • Вы можете включать/выключать или удалять синтетический монитор, защищённый учётными данными только для владельца или общими учётными данными другого пользователя.

    • Если browser-монитор (clickpath или с одним URL) связан с ограниченными учётными данными (только для владельца или общими для нескольких пользователей), любой пользователь может вносить изменения в определённые поля, даже если у него нет доступа к используемым учётным данным. Вы можете редактировать имя монитора, настройки эмуляции устройства, условия ожидания, частоту, расположения, оповещение о сбоях, пороги производительности, метрики, связанные приложения, валидацию и игнорируемые коды HTTP-статуса. И, конечно, вы можете изменить токен или учётные данные имя пользователя/пароль. Вы можете создать учётные данные в настройках монитора в режиме редактирования. Вам потребуется заменить все учётные данные в мониторе теми, к которым у вас есть доступ. Обратите внимание, что замена учётных данных другого пользователя на доступные вам необратима.

    Элементы управления, которые нельзя редактировать — например, URL, включение/выключение HTTP-аутентификации, добавление или удаление событий clickpath, ввод данных в Keystroke, а также Advanced setup в настройках монитора — выделены серым или отображают сообщение об ошибке при попытке сохранить изменения, как в скриптовом, так и в режиме UI.

    • Если HTTP-монитор связан с ограниченными учётными данными (только для владельца или общими для нескольких пользователей), любой пользователь может вносить изменения в определённые поля, даже если у него нет доступа к используемым учётным данным. Вы можете редактировать имя монитора, расположения, валидацию, пороги и, конечно, изменить сертификат или пару имя пользователя/пароль. Вы можете редактировать и изменять учётные данные в URL, значении заголовка или теле запроса. Вы можете создать учётные данные в настройках монитора в режиме редактирования. Вам потребуется заменить все учётные данные в мониторе теми, к которым у вас есть доступ. Обратите внимание, что замена учётных данных другого пользователя на доступные вам необратима.

    Элементы управления, которые нельзя редактировать — например, URL запроса, метод HTTP, скрипт перед выполнением, скрипт после выполнения, HTTP-заголовки, тело запроса, опция следования редиректам и добавление/удаление HTTP-запросов — выделены серым или отображают сообщение об ошибке при попытке сохранить изменения, как в скриптовом, так и в режиме UI.

    Если вам не удаётся редактировать монитор с связанными учётными данными, вы можете найти владельца учётных данных, чтобы обсудить изменения или запросить доступ. Вы также можете перезаписать учётные данные другого пользователя.

    Кто может редактировать или перезаписывать учётные данные¶

    Содержимое учётных данных видно только синтетическим мониторам, приложениям или ad hoc функциям, которые на них ссылаются. Вы можете видеть идентификатор и свойства учётных данных в хранилище, но содержимое учётных данных можно только перезаписать.

    Пользователи с разрешением Change monitoring settings (управление разрешениями) на уровне окружения могут перезаписать учётные данные другого пользователя и тем самым стать новым владельцем. В процессе перезаписи (или в любое время после) они могут изменить уровень доступа учётных данных (общие, публичные или только для владельца).

    Пользователи с разрешением View environment на уровне окружения могут перезаписывать или удалять только те учётные данные, которыми они владеют.

    Выберите учётные данные и нажмите Overwrite credential или Overwrite certificate.

    Для перезаписи учётных данных необходимо предоставить полный набор данных аутентификации. Это означает, что перезаписать учётные данные (и стать владельцем) можно только указав новую пару имя пользователя — пароль, новый сертификат или новый токен. В случае пар имя пользователя — пароль, если оставить поле пароля пустым, старое значение пароля не сохраняется.

    Перезапись учётных данных

    • Владельцы учётных данных могут изменять данные аутентификации и/или уровень доступа учётных данных.

    • Если вы не являетесь владельцем учётных данных:

    • Вы увидите предупреждение об использовании или перезаписи учётных данных.

    • Вы станете новым владельцем учётных данных при их перезаписи. В этом случае рекомендуется уведомить предыдущего владельца.
    • Изменить уровень доступа учётных данных можно только путём полной перезаписи с новыми данными аутентификации и становясь новым владельцем.

    Пользователи с разрешением Change environment settings на уровне окружения могут удалять учётные данные другого пользователя — при удалении учётных данных, используемых в мониторе, этот монитор становится неактивным.

    Как найти владельца учётных данных¶

    Хранилище учётных данных отображает все учётные данные в вашем окружении с именем владельца и уровнем доступа. Вы можете сортировать и фильтровать учётные данные по Owner в хранилище учётных данных.

    Вы можете искать владельцев учётных данных в Synthetic Classic. Вы можете фильтровать мониторы по конкретным учётным данным (Associated credential) и/или их владельцам (Associated credential owner). Обратите внимание, что эти фильтры доступны только при наличии хотя бы одних учётных данных из хранилища, используемых в мониторе. Фильтры показывают все учётные данные (и их владельцев), используемые в мониторах.

    При открытии монитора с учётными данными типа «только для владельца» или «общими» имя владельца выделяется.

    API хранилища учётных данных¶

    Вы можете обращаться к хранилищу учётных данных через API, что открывает широкий спектр сценариев автоматизации.

    • Чтение хранилища учётных данных требует области токена Read credential vault entries. Также можно использовать более широкую область Read configuration.
    • Запись в хранилище учётных данных требует области токена Write credential vault entries. Также можно использовать более широкую область Write configuration. Обратите внимание, что области записи не включают области чтения, которые должны предоставляться отдельно (см. выше).
    • Обновление синтетических мониторов через API требует области токена Create and read synthetic monitors, locations, and nodes.
    • При использовании API для редактирования или обновления монитора с учётными данными, API-токен должен принадлежать пользователю, имеющему доступ к учётным данным, назначенным монитору.

    Подробнее об областях токенов в документации по аутентификации API.

    Рекомендации по работе с учётными данными¶

    • Рекомендуется использовать выделенные тестовые учётные данные для синтетических мониторов.
    • Если многим людям нужно редактировать синтетический монитор с учётными данными, лучше сделать связанные учётные данные публичными.
    • Если аккаунт владельца учётных данных больше не существует в вашей системе, выполнение связанных синтетических мониторов не нарушается; мониторы продолжают выполняться как прежде. Однако рекомендуется, чтобы учётные данные были перезаписаны другим пользователем, который станет новым владельцем. Без этого вы не сможете использовать учётные данные только для владельца в новых мониторах или редактировать существующие мониторы, связанные с такими учётными данными.
    • При перезаписи учётных данных уведомьте предыдущего владельца. При удалении учётных данных уведомьте владельца.
    • При использовании API для редактирования или обновления монитора с учётными данными, API-токен должен принадлежать пользователю, имеющему доступ к учётным данным, назначенным монитору.
    • По возможности используйте узкие области токена API, ограничивая доступ на чтение и запись только хранилищем учётных данных.
