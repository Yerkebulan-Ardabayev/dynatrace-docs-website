    Использование обогащения логов для трассировок при решении проблем¶

    • Tutorial
    • 8-min read
    • Published Jan 10, 2022

    Логи — важнейший компонент для понимания поведения вашего окружения. Объединяя логи с распределёнными трассировками, можно просматривать записи логов в полном контексте транзакции. Автоматическая контекстуализация данных логов работает «из коробки» для популярных языков, таких как Java, .NET, Node.js, Go и PHP, а также для веб-серверов NGINX и Apache.

    Перед началом¶

    Подключите данные логов к трассировкам для Logs Classic:

    • Автоматически через функции OneAgent — для поддерживаемых фреймворков логирования.
    • Вручную через открытые стандарты с соответствующей контекстной информацией — для технологий, не отслеживаемых OneAgent или пока не поддерживаемых «из коробки».

    Сценарий¶

    Шаги¶

    Начинаем анализ с затронутого сервиса Go и проверяем его динамические запросы.

    Анализ логов в распределённой трассировке — 7

    Для изучения частоты отказов выбираем плитку Failure rate. Это переводит нас на вкладку Failure rate страницы Details.

    Анализ логов в распределённой трассировке — 8

    На диаграмме выделен временной период, в течение которого произошёл рост частоты отказов. Для получения дополнительной информации выбираем Analyze failure rate degradation.

    Анализ логов в распределённой трассировке — 9

    Сразу видно, что затронуто большое количество запросов, и Dynatrace предлагает несколько возможных первопричин. Выбираем Details для первой из них, чтобы изучить эту возможную первопричину.

    Первое расширение в списке — проблема с оплатой кредитной картой, которая серьёзно влияет на пользователей, и её необходимо расследовать. Связанные логи можно найти внизу страницы. А пока выберем View all logs in the log viewer, чтобы проверить все возможные логи.

    Анализ логов в распределённой трассировке — 2

    Сразу видно, что есть проблема с загрузкой праздничных дней для доставки. Развернём запись лога, чтобы увидеть подробности. Среди дополнительных атрибутов можно найти свойство trace_id, которое связывает запись лога с распределённой трассировкой.

    Анализ логов в распределённой трассировке — 3

    Выберем значение свойства для перехода к связанной распределённой трассировке. Она содержит детальный обзор поведения приложения и пользовательского опыта для данной конкретной транзакции.

    Анализ логов в распределённой трассировке — 4

    С первого взгляда видно, что две трассировки находятся в состоянии ошибки. Если мы их просмотрим, обнаружим лог ошибки для трассировки /cart.

    Анализ логов в распределённой трассировке — 5

    В логе отображается ошибка при попытке загрузить праздничные дни для доставки, поэтому можно проверить эту трассировку для получения дополнительной информации, так как она тоже содержит ошибку, указывающую на то, что может быть причиной ухудшения времени отклика.

    Изучая распределённую трассировку, приходим к выводу, что что-то не так со службой GetCart, которая существенно влияет на общее время отклика. Если проверить её логи, обнаружим запись slow request.

    Определив компоненты, способствующие возникновению проблемы, можно связаться с ответственными командами и попросить их провести расследование.

    Вернёмся и проверим логи на наличие других ошибок. Поскольку мы уже занялись проблемой с праздничными днями для доставки, можно отфильтровать эти логи с помощью режима advanced query.

    Анализ логов в распределённой трассировке — 10

    Оставшиеся логи указывают на проблему с неподдерживаемым типом карты. Развернём лог и перейдём к распределённой трассировке.

    Анализ логов в распределённой трассировке — 11

    Просматривая распределённую трассировку, можно увидеть, что приложение функционирует нормально и проблемы вызваны неподдерживаемым типом карты.

    Поскольку это не то, что можно исправить в нашем приложении, обращаемся к нашему провайдеру обработки платежей, чтобы выяснить, как эту проблему можно решить.

    Как побочный эффект этого анализа, замечаем, что номер карты отображается в логе, поэтому также стоит обратиться к ответственной команде с просьбой изменить правила логирования, чтобы предотвратить запись конфиденциальной информации.

    Анализ автоматически обнаруженных проблем при первопричине в виде сбоя сервиса

    Сценарий¶

    В этом примере анализируется онлайн-магазин HipsterShop. Это демонстрационное облачное приложение на основе микросервисов, позволяющее пользователям просматривать товары, добавлять их в корзину и совершать покупки.

    Дашборд — логи в анализе первопричин

    На Dashboard видно, что за 24 часа было зафиксировано две проблемы.

    Шаги¶

    Выбрав плитку Problems для их изучения, обнаруживаем, что HipsterShop пострадал от роста частоты ошибок JavaScript, что повлекло за собой последствия для конечных пользователей. Для получения подробностей открываем проблему P-2205042.

    Обзор проблем — логи в анализе первопричин

    На странице проблемы контекстуализированные данные позволяют понять влияние на конечных пользователей, сколько пользователей затронуто и подробности о возникшей ошибке JavaScript. В разделе Root cause узнаём, что проблема берёт начало от изменения конфигурации рабочей нагрузки Kubernetes AdService, активировавшего функциональность Expanded Ads.

    Детали проблемы — логи в анализе первопричин

    Выбрав Analyze Failure rate degradation, получаем обзор всех неудачных запросов с выделением:

    • Каких запросов не удалось выполнить.
      В данном случае проблемы затронули запрос GetAds.
    • Деталей исключений, связанных с неудачными запросами.
      В данном случае в AdService было выброшено исключение NullPointerException, где ожидался объект.
    • Related errors logs и предупреждений.

    Проактивное устранение неполадок с помощью логов (Kubernetes)

    Сценарий¶

    С дашборда необходимо исследовать сервис frontend, развёрнутый в качестве рабочей нагрузки Kubernetes. Чтобы получить информацию о рабочей нагрузке, выбираем плитку рабочей нагрузки Kubernetes frontend.

    Дашборд — логи для Kubernetes

    На странице сведений о рабочей нагрузке перечислены все критические метрики работоспособности для рабочей нагрузки, такие как использование CPU и памяти и их ограничения для базовых подов, а также время отклика, частота отказов и пропускная способность для развёрнутых сервисов. Когда один из сервисов (в нашем случае frontend) затронут ошибкой, можно исследовать его через страницу сервиса или, как в нашем случае, проверить раздел Logs.

    Анализ рабочей нагрузки — логи для Kubernetes

    Таблица логов показывает информацию Status. Красные полосы указывают на наличие ошибок в логах. Можно исследовать их, отфильтровав доступные логи по status error. Видно, что все они связаны с неудачным процессом оформления заказа из-за неверных данных кредитной карты. Поскольку каждая из этих записей лога происходит в контексте распределённой трассировки, также можно просмотреть связанную трассировку для конкретных записей лога, выбрав View trace.

    Распределённые трассировки — логи для Kubernetes

    Это показывает, что выбранная запись лога для микросервиса frontend на Go является лишь частью более сложного паттерна ошибок. Видно, что он вызывает метод PlaceOrder нижестоящего hipstershop.CheckoutService. Выбрав этот сервис, можно увидеть, что OneAgent автоматически собирает дополнительную полезную информацию, например тот факт, что это вызов gRPC. На вкладке Logs видно, что во время этого вызова было создано две записи лога, включая сообщение об ошибке. Выбрав Show related entry 'Code level' tab, также можно понять, в какой части микросервиса Go была сгенерирована конкретная строка лога.

    Выбрав строку NGINX frontend reverse proxy, видно, что вызов вернул клиенту 500 — Internal Server Error, что, похоже, не является корректным кодом ошибки для неверных данных кредитной карты.

    Какие тарифный план и пакет необходимы для начала работы?

    Требуются тарифный план и пакет как для мониторинга логов, так и для трассировок.

    Ваша технология отслеживается через OneAgent Дополнительная информация о тарифных планах и пакетах
    Применимо * Для логов см. Log Monitoring Classic. * Для трассировок (Full-Stack Monitoring) см. Application and Infrastructure Monitoring (Host Units).
    Не применимо * Для логов см. Log Monitoring Classic. * Для трассировок (приём открытых трассировок с OpenTelemetry) см. DDU для бессерверных функций и DDU для пользовательских трассировок (Trace API).
