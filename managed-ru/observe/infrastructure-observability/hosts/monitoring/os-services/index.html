    OS services monitoring¶

    Мониторинг служб ОС¶

    • How-to guide
    • 14-min read
    • Updated on Jan 14, 2026

    Dynatrace предоставляет готовый мониторинг доступности служб ОС.

    Можно отслеживать хосты в режиме мониторинга полного стека или использовать облегчённые режимы мониторинга. Подробнее см. в разделе OneAgent monitoring modes.

    Требования¶

    • На отслеживаемом хосте для мониторинга связей между процессами и службами ОС требуется Linux systemd version 230+.
    • На отслеживаемом хосте для улучшенного обнаружения служб ОС с менее частым обновлением требуется Linux systemd version 250+.

    Варианты оповещений для служб ОС¶

    В зависимости от требований к мониторингу можно выбрать базовое или расширенное оповещение о службах ОС. Режим Discovery допускает только базовое оповещение, тогда как режимы Full-Stack и Infrastructure также поддерживают расширенное оповещение.

    Базовое оповещение (Service status)¶

    Базовое оповещение предоставляет сведения только о состоянии службы. Система отслеживает текущее состояние службы ОС и оповещает вас при его изменении с «running» на «failed».

    Состояние службы можно проверить на странице OS services monitoring выбранного хоста или запросить с помощью Settings API — таблица схемы OS services monitoring.

    Расширенное оповещение (Service availability metric)¶

    Расширенное оповещение предоставляет доступ к состоянию службы и метрике доступности службы. В дополнение к изменениям состояния службы ОС вы получаете более детальную информацию о доступности службы с точностью до одной минуты. Это позволяет реализовывать более сложную бизнес-логику, например получать уведомления, если служба остановлена более 10 минут. Можно создавать правила оповещений для мониторинга доступности и инициирования оповещения при необходимости.

    Пример¶

    Расширенное оповещение можно настроить в Data Explorer, выбрав нужные компоненты. Подробнее см. в разделе Data Explorer.

    Set an advanced alerting in Data explorer for Managed.

    Мониторинг службы¶

    Чтобы настроить мониторинг службы ОС, выполните следующие шаги.

    1. Доступ к мониторингу служб ОС

    В Dynatrace перейдите в OS services monitoring для нужного уровня настройки.

    • Настройки определяются на уровне среды.
    • Уровень группы хостов наследует все настройки среды. Дополнительно можно добавить конфигурации для отдельных групп хостов.
    • Уровень хоста наследует все настройки группы хостов. Дополнительно можно добавить конфигурации для отдельных хостов.

    Уровень среды¶

    Перейдите в Settings > Collect and capture > Infrastructure > OS > OS services monitoring.

    Уровень группы хостов¶

    1. Перейдите в Deployment Status > OneAgents.
    2. На странице OneAgent deployment отключите Show new OneAgent deployments.
    3. В поле Filter by введите Host group и выберите нужную группу хостов из раскрывающегося списка.

    Список хостов теперь отфильтрован по выбранной группе хостов. Каждый перечисленный хост имеет ссылку Host group: <имя группы>, где <имя группы> — имя группы хостов, которую требуется настроить.

    Свойство Host group не отображается, если выбранный хост не принадлежит ни одной группе хостов. 4. Выберите имя группы хостов в любой строке.

    Так как фильтр применён по группе хостов, все отображаемые хосты относятся к одной группе.

    1. В настройках группы хостов выберите OS services monitoring.

    Уровень хоста¶

    1. Перейдите в Hosts.
    2. Найдите и выберите хост для отображения страницы его обзора.

    3. В правом верхнем углу страницы обзора хоста выберите More (…) > Settings.

    4. В настройках хоста выберите OS services monitoring.

    5. Добавление политики мониторинга служб

    На основе состояния службы и правил политика мониторинга служб определяет способ мониторинга службы в Dynatrace. По умолчанию Dynatrace поставляется с политиками Auto-start Windows OS Services и Auto-start Linux OS Services для автоматически запускаемых служб Windows и Linux со статусом «failed».

    Ограничения

    Обратите внимание, что стандартный лимит сущностей служб ОС составляет 100 000 на кластер. В крупных средах с большим количеством хостов рекомендуется создавать точные правила, охватывающие только важные для вашей инфраструктуры службы. Создание слишком широких правил (например, охватывающих все службы на тысячах хостов) может привести к достижению лимита (взрыв сущностей) и исчезновению служб ОС из Dynatrace. Политики Auto-start Windows OS Services и Auto-start Linux OS Services можно использовать в качестве отправной точки для дальнейшей настройки.

    Порядок политик мониторинга служб важен. Политики, расположенные выше в списке, обрабатываются раньше остальных. Это позволяет создавать избирательные оповещения или мониторинг с минимальным числом политик. Например, если вы хотите отслеживать все автоматически запускаемые службы, а не только созданные Microsoft, добавьте политику с отключёнными оповещениями и/или мониторингом для проверки, является ли производитель Microsoft.

    1. На странице OS services monitoring для нужного уровня выберите Add policy и определите политику — набор правил.
    2. System: выберите операционную систему.
    3. Rule name: введите имя, которое будет отображаться в поле Summary.
    4. Monitor: определите, хотите ли вы отслеживать доступность службы с помощью метрики OS service availability (builtin:osservice.availability). При наличии метрика отправляет состояние службы каждые 10 секунд. Состояние передаётся через измерение Service status (dt.osservice.status).
      Обратите внимание, что метрика потребляет единицы данных (data points). Подробнее см. в разделе Metrics powered by Grail.
    5. Alert: определите, нужны ли оповещения для данной политики.
    6. OneAgent version 1.257+ Alert if service is not installed: следует ли получать оповещения о службах ОС, не установленных на хосте.
    7. Service status: задайте состояние службы, при котором должно создаваться оповещение.

    Для мониторинга состояния службы можно использовать логические операции. Например, $eq(running) отслеживает состояние running.

    Доступные логические операции:

    • $not($eq(paused)) — совпадает со службами, не находящимися в состоянии paused.
    • $or($eq(paused),$eq(running)) — совпадает со службами в состоянии paused или running.

    Доступные состояния службы (используйте в качестве параметра условия):

    • running
    • stopped
    • start_pending
    • stop_pending
    • continue_pending
    • pause_pending
    • paused

    Для мониторинга состояния службы можно использовать логические операции. Например, $eq(active) отслеживает состояние active.

    Доступные логические операции:

    • $not($eq(active)) — совпадает со службами, не находящимися в состоянии active.
    • $or($eq(inactive),$eq(failed)) — совпадает со службами в состоянии inactive или failed.

    Доступные состояния службы (используйте в качестве параметра условия):

    • reloading
    • activating
    • deactivating
    • failed
    • inactive
    • active
    • Необязательно OneAgent version 1.257+ Alerting delay: количество 10-секундных циклов измерения, в течение которых служба должна находиться в заданном состоянии до генерации события. Не применяется к оповещениям о неустановленных службах.

    Далее необходимо выбрать службы для мониторинга на основе их свойств.

    1. Выбор служб для мониторинга

    2. Выберите Add rule.

    3. Необязательно Rule scope: выберите OS Service или Host. По умолчанию выбран вариант OS Service.

    Если выбрано Host:

    • OneAgent version 1.277+ Custom metadata для сопоставления:

    • Key — ключ метаданных для сопоставления.

    • Condition — строка, которая:

      • Dynatrace version 1.310+ $match(ver*_1.2.?) — сопоставляет строку с подстановочными знаками. * — любое число символов (включая ноль), ? — ровно один символ.
      • $contains(production) — совпадает, если production встречается в значении метаданных хоста.
      • $eq(production) — совпадает, если production точно совпадает со значением метаданных хоста.
      • $prefix(production) — совпадает, если production является префиксом значения метаданных хоста.
      • $suffix(production) — совпадает, если production является суффиксом значения метаданных хоста.

      Доступные логические операции:

      • $not($eq(production)) — совпадает, если значение метаданных хоста не равно production.
      • $and($prefix(production),$suffix(main)) — совпадает, если значение начинается с production и заканчивается на main.
      • $or($prefix(production),$suffix(main)) — совпадает, если значение начинается с production или заканчивается на main.

      При включении специальных символов, например скобок ( и ), в выражения сопоставления, экранируйте их тильдой ~. Например, для сопоставления со значением my(amazing)property используйте $eq(my~(amazing~)property).

      Условия нечувствительны к регистру.

    Если выбрано OS Service, продолжайте в соответствии с вашей операционной системой.

    • Display name — отображаемое имя для системного пользователя.

    • Path — путь к исполняемому файлу, используемому службой ОС.

    Правило мониторинга может включать несколько правил обнаружения. Все правила обнаружения должны выполняться для соответствия службе ОС, так как к ним применяется логическая операция AND.

    Дополнительные сведения о Display name, Path, Manufacturer и Service Name

    С помощью этих свойств определяются службы для мониторинга на основе:

    • Display name — отображаемое для системного пользователя имя
    • Path — путь к исполняемому файлу службы
    • Manufacturer — производитель службы
    • Service name — имя или идентификатор, под которым служба ОС распознаётся системой

    • Condition — строка, которая:

    • Начинается с — используйте квалификатор $prefix, например $prefix(ss).

    • Заканчивается на — используйте квалификатор $suffix, например $suffix(hd).
    • Равна — используйте квалификатор $eq, например $eq(sshd).
    • Содержит — используйте квалификатор $contains, например $contains(ssh).
    • Dynatrace version 1.310+ Совпадает — используйте квалификатор $match, например $match(ip?tables*), где * — любое число символов, ? — ровно один символ.

    Доступные логические операции:

    • $not($eq(sshd)) — совпадает, если значение свойства службы не равно sshd.
    • $and($prefix(ss),$suffix(hd)) — совпадает, если значение начинается с ss и заканчивается на hd.
    • $or($prefix(ss),$suffix(hd)) — совпадает, если значение начинается с ss или заканчивается на hd.

    При включении специальных символов, например скобок ( и ), в выражения сопоставления, экранируйте их тильдой ~. Например, для сопоставления со значением my(amazing)property используйте $eq(my~(amazing~)property).

    Условия нечувствительны к регистру.

    Дополнительные сведения о Startup Type

    С помощью этого свойства определяются службы для мониторинга на основе типа их запуска.

    • Condition — строка, которая:

    • Равна — используйте квалификатор $eq, например $eq(manual).

    Доступные логические операции:

    • $not($eq(auto)) — совпадает со службами, у которых тип запуска не Automatic.
    • $or($eq(auto),$eq(manual)) — совпадает со службами с типом запуска Automatic или Manual.

    Используйте одно из следующих значений в качестве параметра условия:

    • manual — Manual: служба запускается только при необходимости или по запросу.
    • manual_trigger — Manual (Trigger Start): служба запускается вместе с запуском другой службы.
    • auto — Automatic: служба запускается автоматически.
    • auto_delay — Automatic (Delayed Start): запуск службы откладывается до завершения загрузки системы.
    • auto_trigger — Automatic (Trigger Start): служба запускается автоматически при старте системы и может быть запущена или остановлена при определённых событиях ОС.
    • auto_delay_trigger — Automatic (Delayed Start, Trigger Start)
    • disabled — Disabled

    Service property для сопоставления:

    • Service Name
    • Startup Type

    Дополнительные сведения о Service Name

    С помощью этого свойства определяются службы для мониторинга на основе имени службы.

    Condition — строка, которая:

    • Dynatrace version 1.310+ $match(ip?tables*) — сопоставляет строку с подстановочными знаками. * — любое число символов, ? — ровно один символ.
    • $contains(ssh) — совпадает, если ssh встречается в значении свойства службы.
    • $eq(sshd) — совпадает, если sshd точно совпадает со значением свойства службы.
    • $prefix(ss) — совпадает, если ss является префиксом значения свойства службы.
    • $suffix(hd) — совпадает, если hd является суффиксом значения свойства службы.

    Доступные логические операции:

    • $not($eq(sshd)) — совпадает, если значение свойства службы не равно sshd.
    • $and($prefix(ss),$suffix(hd)) — совпадает, если значение начинается с ss и заканчивается на hd.
    • $or($prefix(ss),$suffix(hd)) — совпадает, если значение начинается с ss или заканчивается на hd.

    Дополнительные сведения о Startup Type

    С помощью этого свойства определяются службы для мониторинга на основе типа их запуска.

    Condition — строка, которая:

    • $eq(enabled) — совпадает со службами с типом запуска enabled.

    Доступные логические операции:

    • $not($eq(enabled)) — совпадает со службами с типом запуска, отличным от enabled.
    • $or($eq(enabled),$eq(disabled)) — совпадает со службами с типом запуска enabled или disabled.

    Используйте одно из следующих значений в качестве параметра условия:
    enabled, enabled-runtime Служба помечена как готовая к запуску.
    static Файл юнита не включён и не содержит настроек для включения в разделе install. Статические юниты устанавливаются как зависимости, их можно только маскировать, но они выполняются только при зависимости других юнитов или при ручном запуске.
    disabled Файл юнита не включён, но содержит раздел install с инструкциями по установке.

    Управление отслеживаемыми службами ОС¶

    Для управления службами ОС:

    1. В Dynatrace перейдите в OS services monitoring для нужного уровня.

    Уровень хоста

    1. Перейдите в Hosts.
    2. Найдите и выберите хост для отображения страницы его обзора.

    3. В правом верхнем углу страницы обзора хоста выберите More (…) > Settings.

    4. В настройках хоста выберите OS services monitoring.

    Уровень группы хостов

    1. Перейдите в Deployment Status > OneAgents.
    2. На странице OneAgent deployment отключите Show new OneAgent deployments.

    3. В поле Filter by введите Host group и выберите нужную группу хостов из раскрывающегося списка.

      Список хостов теперь отфильтрован по выбранной группе хостов. Каждый перечисленный хост имеет ссылку Host group: <имя группы>.

      Свойство Host group не отображается, если выбранный хост не принадлежит ни одной группе хостов. 4. Выберите имя группы хостов в любой строке.

      Так как фильтр применён по группе хостов, все отображаемые хосты относятся к одной группе.

    4. В настройках группы хостов выберите OS services monitoring.

    Уровень среды

    Перейдите в Settings > Monitoring > OS services monitoring. 2. Отслеживаемые службы ОС отображаются в таблице под кнопкой Add policy.

    • Чтобы прекратить мониторинг указанной службы, отключите параметр Enabled.
    • Чтобы удалить службу из таблицы, нажмите кнопку удаления в столбце Delete.
    • Для просмотра и редактирования деталей выберите элемент управления раскрытием в столбце Details.

    Мониторинг доступности службы¶

    2. Выберите любой хост для перехода на страницу его обзора.
    3. В разделе OS services analysis выберите имя службы, чтобы открыть страницу Service overview.

    Подробнее см. в разделе OS services analysis.

    Конфигурация в масштабе через Settings API¶

    Для масштабной настройки мониторинга доступности служб можно использовать Settings API.

    1. Для изучения схемы используйте GET a schema со значением builtin:os-services-monitoring в качестве schemaId.
    2. На основе схемы builtin:os-services-monitoring создайте объект конфигурации.
    3. Для создания конфигурации используйте POST an object.

    Ограничения¶

    В Linux поддерживаются службы systemd ОС со следующими типами запуска:

    • enabled
    • enabled-runtime
    • static
    • disabled
    • indirect
