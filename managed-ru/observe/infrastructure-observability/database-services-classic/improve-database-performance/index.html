    Улучшение производительности базы данных¶

    Улучшение производительности базы данных¶

    • Практическое руководство
    • Чтение 6 мин
    • Опубликовано 20 марта 2019 г.

    Базы данных, это сложные приложения, а доступ к базе данных, ключевая функция многих приложений. Чтобы избежать сбоев или низкой производительности, важно, чтобы базы данных размещались надёжно и получали достаточно ресурсов для наилучшей работы.

    Оптимизировать базы данных можно с помощью:

    • данных сервера, поддерживающих мониторинг состояния хоста
    • метрик гипервизора и виртуальных машин, поддерживающих мониторинг уровня виртуализации
    • данных приложений, оптимизирующих доступ к базе данных
    • сетевых данных, дающих представление о влиянии сетевого взаимодействия на работу базы данных

    Следующие несколько шагов для простой настройки производительности базы данных позволяют значительно ускорить работу большинства приложений.

    Шаг 1. Проверка состояния базы данных¶

    Первый шаг, убедиться, что хост, обслуживающий процесс базы данных, обладает достаточными ресурсами: CPU, памятью и дисковым пространством.

    CPU¶

    • Оснастить каждый хост минимум двумя ядрами CPU. Соответствие количества CPU хоста требованиям помогает:

    • обеспечить отзывчивость хоста, поскольку серверы баз данных создают постоянную базовую нагрузку на машины;

    • избежать перерасхода средств или аппаратных ограничений, поскольку лицензирование серверов баз данных зависит от количества CPU.
    • При мониторинге виртуальных машин нужно также отслеживать хост, на котором они работают. Это даёт более полную картину, чем метрики CPU отдельных виртуальных машин, которые показывают только доступность процессорного времени для конкретной машины.

    Память¶

    • Помимо мониторинга метрики Memory usage, нужно отслеживать Page faults в секунду, чтобы понять, сколько дополнительной памяти требуется. Тысячи отказов страниц (page faults) в секунду указывают на нехватку памяти на хосте.

    Дисковое пространство¶

    • Нужно убедиться, что доступное хранилище для сервера базы данных превышает объём дискового пространства, требуемый для самих данных.

    Из-за индексов и других механизмов повышения производительности базы данных могут использовать больше дискового пространства, чем требуется самим данным. Например, NoSQL базы данных (такие как Cassandra и MongoDB) потребляют значительно больше дискового пространства, чем ожидается. По сравнению с обычными SQL базами данных, базы данных MongoDB могут потреблять меньше оперативной памяти, но больше дискового пространства. * Нужно убедиться, что база данных работает на выделенных жёстких дисках, чтобы снизить фрагментацию диска, вызванную другими процессами. * Проверить Disk latency (задержку диска).

    В зависимости от нагрузки на жёсткий диск задержка диска может увеличиваться, что приводит к снижению производительности базы данных. Высокую задержку диска можно предотвратить, максимально задействуя механизмы кэширования приложения и базы данных.

    Задержка диска * Если результаты описанных выше мер неудовлетворительны, стоит рассмотреть следующее.

    1. Добавить дополнительные жёсткие диски.

      Производительность чтения можно кратно увеличить простым зеркалированием жёстких дисков. Производительность записи выигрывает от использования RAID 1 или RAID 10 вместо RAID 6. 2. Попробовать твердотельные накопители.

      Нужно выбирать модель, предназначенную для использования в базах данных, поскольку базы данных выполняют больше циклов чтения/записи в хранилище, чем большинство обычных приложений. Твердотельные накопители дороже традиционных жёстких дисков, но обеспечивают существенный прирост производительности.

    Шаг 2. Понимание того, как выполняется доступ к базе данных¶

    Как только база данных размещена на исправном оборудовании, стоит взглянуть на приложения, которые к ней обращаются. Если известно приложение или сервис с плохой производительностью базы данных, не стоит предполагать, что именно это приложение влияет на производительность базы данных: возможно, дело совсем в другом приложении или сервисе.

    Обзор базы данных | Unified analysis

    Снижение производительности базы данных может затрагивать всю базу данных целиком или отдельного клиента.

    • Если плохую производительность испытывают все клиенты, нужно проверить исправность хоста. В большинстве случаев причина в том, что оборудование не справляется с нагрузкой.
    • Если от низкого времени отклика страдает только один сервис, стоит глубже изучить метрики этого сервиса, чтобы найти первопричину проблемы.

    Шаг 3. Понимание нагрузки и индивидуального времени отклика каждого экземпляра сервиса¶

    Когда у сервиса плохая производительность базы данных, можно проанализировать его взаимодействие с базой данных через database statements (операторы базы данных). Это даёт представление о количестве выполненных запросов, частоте выполнения запроса на один request, количестве строк, возвращаемых каждым запросом, и так далее.

    Страница деталей базы данных

    • При наличии нескольких экземпляров сервиса нужно проверить, затронуты ли все экземпляры, а не только один.

    • Нужно проверить, как часто запросы вызываются на один request. Возможно, получится сократить количество запросов к базе данных, улучшив кэш базы данных сервиса. Если один и тот же запрос выполняется более одного раза на request, можно раскрыть потенциал производительности, применив продуманные стратегии кэширования.

    Шаг 4. Проверка количества подключений к базе данных¶

    Плохая производительность базы данных может сохраняться даже при корректно настроенных запросах к базе данных. В таких случаях нужно проверить, правильно ли настроен размер пула подключений к базе данных у приложения.

    При настройке пула подключений стоит учитывать следующее:

    • максимальное количество подключений, которое может обработать база данных;
    • корректный размер пула подключений, требуемый для приложения.

    Поскольку приложение может быть не единственным клиентом, подключённым к базе данных, нужно убедиться, что размер пула подключений не установлен на максимум. Если приложение занимает все подключения, сервер базы данных не будет работать так, как ожидается.

    Как определить максимальное количество подключений

    Максимальное количество подключений к базе данных зависит от ресурсов базы данных. Чтобы найти максимальное количество подключений, нужно постепенно увеличивать нагрузку и количество разрешённых подключений к базе данных.

    При этом нужно следить за метриками сервера базы данных: CPU, памятью и производительностью диска. Как только любая из них достигает предела, значит, лимит найден. Если доступного количества подключений недостаточно для приложения, стоит рассмотреть обновление оборудования.

    Подробнее о размере пула подключений к базе данных см. About Pool Sizing﻿.

    Как определить корректный размер пула подключений для приложения

    Количество разрешённых одновременных подключений к базе данных соответствует объёму параллельной нагрузки, которую приложение создаёт на сервер базы данных. Есть определённые инструменты, которые можно использовать для определения правильного числа.

    Увеличение нагрузки приводит к росту времени отклика транзакций, даже если сервер базы данных исправен. Нужно измерить время отклика транзакции от начала до конца, чтобы увидеть, увеличивается ли время Connection acquisition (получения подключения) при высокой нагрузке. Если это так, пул подключений может быть исчерпан. Если нет, стоит проверить метрики сервера базы данных, чтобы определить максимальное количество подключений, которое способна обработать база данных.

    Размер пула подключений должен быть постоянным. Поэтому минимальный и максимальный размер пула нужно устанавливать равными.

    Шаг 5. Проверка сети¶

    Физические ограничения виртуализированной инфраструктуры могут влиять на производительность базы данных: кабели могут выходить из строя, а маршрутизаторы ломаться. Сетевые метрики дают представление о невиртуальных проблемах. Например, если проблемы появляются спустя месяцы или даже годы безупречной работы, инфраструктура может страдать от физических неполадок. Стоит проверить маршрутизаторы, кабели и сетевые интерфейсы.

    Анализ процессов хоста

    Чаще всего перегруженные процессы начинают терять пакеты при исчерпании ресурсов. Если сетевая проблема не связана с оборудованием, видимость на уровне процессов помогает выявить неисправный компонент.
