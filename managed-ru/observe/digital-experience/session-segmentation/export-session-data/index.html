    Экспорт пользовательских сессий¶

    Экспорт пользовательских сессий¶

    • Практическое руководство
    • Чтение 12 минут
    • Обновлено 06 сентября 2022 г.

    Dynatrace может непрерывно отправлять данные пользовательских сессий на заданную конечную точку webhook.

    Чтобы начать получать данные пользовательских сессий, нужно настроить конечную точку, а затем настроить экспорт сессий через веб-интерфейс Dynatrace или через Settings API.

    Dynatrace отправляет данные по завершённым пользовательским сессиям пакетами на все заданные конечные точки, со сбросом каждые несколько секунд, чтобы экспортировать данные сразу после того, как пользовательская сессия помечена как завершённая. Передача данных происходит, как только выполняется одно из следующих условий:

    • завершилось 1000 пользовательских сессий;
    • размер пакета превышает 896 000 байт;
    • ни одна пользовательская сессия не завершилась за последние 30 секунд.

    Чтобы предотвратить перегрузку системы, Dynatrace отменяет запрос, если конечная точка не отвечает в течение 30 секунд. Dynatrace повторяет попытку ещё три раза, прежде чем окончательно отбросить запрос и отправить уведомление об оповещении с сообщением об ошибке Request timeout.

    Настройка конечной точки¶

    Сервер, предоставляющий webhook, должен прослушивать запросы PUT или POST по URL, указанному в настройках экспорта сессий. Кроме того, по соображениям безопасности Dynatrace допускает только конечные точки HTTPS.

    • Метод HTTP: PUT или POST
    • Путь: как настроено
    • Тип контента: application/json или application/x-ndjson. При отправке в Elasticsearch установить application/x-ndjson.
    • Код статуса ответа: 200

    Настройка экспорта сессий через веб-интерфейс Dynatrace¶

    Чтобы настроить экспорт пользовательских сессий через веб-интерфейс Dynatrace

    1. Перейти в Settings > Integration > User session exports.
    2. Выбрать Add item.
    3. Указать Endpoint URL и включить Enable user session export, если готово начать получение данных пользовательских сессий.

    По соображениям безопасности Dynatrace допускает только конечные точки HTTPS. 4. Установить Content type на application/json или application/x-ndjson. При отправке данных сессий в Elasticsearch установить application/x-ndjson. 5. Включить Use POST method (instead of PUT) при отправке данных сессий в Elasticsearch или если конечная точка настроена на приём запросов POST. 6. Опционально: настроить аутентификацию, включить передачу данных в Elasticsearch или настроить область экспорта, оповещения и расширенную конфигурацию.

    Также можно протестировать экспорт пользовательских сессий, а также скачать пример набора данных или пример сопоставления.

    Можно настроить до трёх конечных точек HTTPS.

    Настройка аутентификации¶

    Dynatrace может отправлять данные пользовательских сессий с использованием базовой аутентификации или OAuth 2.0. Эти типы аутентификации позволяют защитить конечные точки.

    При включении аутентификации для конфигурации экспорта сессий нужно протестировать конфигурацию перед сохранением.

    По соображениям безопасности при тестировании конфигурации экспорта сессий нужно повторно ввести пароль базовой аутентификации или client secret OAuth 2.0. Также нужно повторно ввести пароль или secret при редактировании существующей конфигурации экспорта сессий с включённой аутентификацией.

    Настройка базовой аутентификации¶

    Чтобы настроить экспорт пользовательских сессий с базовой аутентификацией

    1. На странице User session exports развернуть нужный Endpoint URL.
    2. В разделе Authentication включить Activate.
    3. Установить Basic authentication как тип аутентификации.
    4. Ввести имя пользователя и пароль.

    Пароль шифруется и маскируется в среде.

    Настройка аутентификации OAuth 2.0¶

    Чтобы настроить экспорт пользовательских сессий с аутентификацией по учётным данным клиента OAuth 2.0

    1. На странице User session exports развернуть нужный Endpoint URL.
    2. В разделе Authentication включить Activate.
    3. Установить OAuth 2.0 как тип аутентификации.
    4. Ввести URL access token, client ID, client secret и scope (опционально).

    Client secret шифруется и маскируется в среде.

    Дополнительная информация об аутентификации OAuth 2.0

    OAuth 2.0ï»¿ предлагает различные grant types, но Dynatrace поддерживает только grant type client credentials. Информация об этом grant type доступна в следующих источниках:

    • Client Credentials Flowï»¿
    • Call Your API Using the Client Credentials Flowï»¿
    • The OAuth 2.0 Authorization Framework | Client Credentials Grantï»¿

    Чтобы отправлять данные пользовательских сессий с аутентификацией OAuth 2.0, нужно настроить сервер авторизации OAuth2, имеющий конечную точку access token. Также нужно убедиться, что Endpoint URL работает с access token.

    Ниже описаны шаги того, как работает аутентификация OAuth 2.0 для экспорта пользовательских сессий.

    1. Dynatrace отправляет запрос на сервер авторизации для получения access token. В запрос включаются Access token URL, Client ID, Client secret и Scope (опционально).

    curl --insecure -w "\nHTTP Status: %{http_code}\n" \



--location --request POST 'https://[your-access-token-url]/oauth2/token' \



--header 'Content-Type: application/x-www-form-urlencoded' \



--header 'Accept: application/json' \



--data-urlencode 'client_id=[your-client-id]' \



--data-urlencode 'client_secret=[your-client-secret]' \



--data-urlencode 'grant_type=client_credentials' \



--data-urlencode 'scope=[your-scope]'
    2. Сервер авторизации отвечает access token, например:

    {



"scope":"session-export-api",



"token_type":"Bearer",



"expires_in":300,



"access_token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJodHRwczovL2V4YW1wbGUuYXV0aDAuY29tLyIsImF1ZCI6Imh0dHBzOi8vYXBpLmV4YW1wbGUuY29tL2NhbGFuZGFyL3YxLyIsInN1YiI6InVzcl8xMjMiLCJpYXQiOjE0NTg3ODU3OTYsImV4cCI6MTQ1ODg3MjE5Nn0.CA7eaHjIHz5NxeIJoFK9krqaeZrPLwmMmgI_XiQiIkQ"

    3. Dynatrace отправляет запрос POST на настроенный Endpoint URL. Запрос включает данные пользовательской сессии, а также access token, полученный на предыдущем шаге, который передаётся как Bearer token в заголовке Authorization.

    curl --request POST \



--url https://[your-endpoint-url] \



--header 'Authorization: Bearer ACCESS_TOKEN' \



--header 'Content-type: application/json'

    Отправка данных напрямую в Elasticsearch¶

    Чтобы отправить данные напрямую в Elasticsearch

    1. Убедиться, что сервер кластера имеет доступ к развёртыванию Elasticsearch.
    2. Убедиться, что URL экспорта соответствует формату: https://<your_host>:9200/_bulk. Заменить <your_host> на актуальное значение.
    3. На странице User session exports развернуть нужный endpoint URL.
    4. В разделе Define your endpoint убедиться, что Content type установлен в application/x-ndjson и включена опция Use POST method (instead of PUT).
    5. В разделе Send data directly to Elasticsearch включить Activate.
    6. Указать имя индекса, в который отправляются данные и тип документов в индексе Elasticsearch.

    Индекс Elasticsearch¶

    Создать индекс, в который нужно отправлять данные пользовательских сессий, и определить маппинг для этого индекса. Подробнее о том, как скачать образец маппинга для конфигурации экспорта пользовательских сессий, см. в разделе Download a sample mapping.

    Если индекс не создан до включения экспорта пользовательских сессий, развёртывание Elasticsearch автоматически создаёт маппинги для полей. Такой автоматический маппинг не всегда создаёт корректные маппинги полей. Например, поля даты маппятся как long.

    Тип индекса Elasticsearch¶

    Elasticsearch в настоящее время убирает поддержку типов маппинга. Способ создания индексов и настройки типов зависит от версии Elasticsearch. Подробности ниже.

    Рекомендуется указывать _doc в качестве типа документа независимо от используемой версии Elasticsearch.

    • Elasticsearch версии 6

    • Указывать один единственный тип на индекс.

    • При создании индекса указывать параметр строки запроса include_type_name, чтобы обозначить, что запросы и ответы должны включать имя типа. Значение этого параметра по умолчанию true. Если его не задать, появится предупреждение об устаревании. Если при создании индекса тип не указан, используется _doc.

    • Elasticsearch версии 7

    • Указание типов в этой версии Elasticsearch устарело. Параметр include_type_name по умолчанию имеет значение false.

    • Можно принудительно заставить Elasticsearch использовать имя типа, установив параметр в true, что приведёт к предупреждению об устаревании.

    • Elasticsearch версии 8

    • Указание типов больше не поддерживается. Начиная с этой версии, тип документа нужно опускать. Подробнее см. Removal of mapping typesï»¿.

    Type относится к «типу индекса», используемому в Elasticsearch, и не ограничивает, какие пользовательские сессии экспортируются. Независимо от настроенного типа, экспортируются все данные пользовательских сессий, включая действия пользователей, события и ошибки.

    Чтобы ограничить экспортируемые пользовательские сессии, можно определить management zone. См. раздел Set up export scope, alerting, and advanced configuration ниже.

    Настройка области экспорта, оповещений и расширенной конфигурации¶

    В разделе Export scope, alerting, and advanced configuration конечной точки можно сузить область экспорта сессий, отключить уведомления и настроить некоторые другие параметры экспорта пользовательских сессий.

    • Область экспорта (Export scope): чтобы определить область экспорта пользовательских сессий, выбрать нужную Management zone. После настройки management zone Dynatrace отправляет только те пользовательские сессии, в которых есть хотя бы одно действие пользователя с соответствующим приложением.

    Ограничение экспорта сессий только синтетическими пользовательскими сессиями в настоящее время невозможно. * Оповещения (Alerting): включить Disable notification, если уведомления о сбоях экспорта пользовательских сессий не нужны. * Пользовательская конфигурация (Custom configuration): указать Custom configuration properties для дополнительной настройки конфигурации экспорта пользовательских сессий. Перед изменением этого поля нужно связаться с экспертом по продуктам Dynatrace через онлайн-чат.

    При возникновении проблем с экспортом пользовательских сессий нужно связаться с экспертом по продуктам Dynatrace через онлайн-чат, прежде чем настраивать дополнительные свойства.

    Тестовый экспорт¶

    Чтобы протестировать конфигурацию экспорта пользовательских сессий, нужно развернуть нужный endpoint URL на странице User session exports и выбрать Test export.

    Если кнопка Test export неактивна, вероятная причина в том, что Endpoint URL или Access token URL указывает протокол HTTP. По соображениям безопасности Dynatrace допускает только протокол HTTPS.

    Dynatrace использует текущую конфигурацию экспорта пользовательских сессий для экспорта до 50 пользовательских сессий за последние семь дней. Если за этот период данных нет, тестовый экспорт недоступен. Как только тестовый экспорт завершится, придёт уведомление о результатах.

    Сохранять конфигурацию для тестирования конечной точки не обязательно.

    При настройке конечной точки, защищённой с помощью аутентификации:

    • Протестировать конфигурацию перед сохранением.
    • Повторно ввести пароль базовой аутентификации или client secret OAuth 2.0 при тестировании конфигурации.

    Скачивание образца набора данных¶

    Чтобы посмотреть, как выглядят данные пользовательских сессий при экспорте на конечные точки, на странице User session exports нужно развернуть требуемый URL конечной точки и выбрать Download sample export data.

    Образец набора данных состоит максимум из 50 пользовательских сессий, отслеженных в среде за последние семь дней. Если за этот период не было пользовательских сессий, скачивание образца данных недоступно.

    Скачивание примера отображения¶

    Чтобы экспортировать данные пользовательских сессий напрямую в собственный экземпляр Elasticsearch, можно скачать пример отображения (mapping) для индексов. Разверни нужный URL конечной точки на странице User session exports и выбери Download mapping. Скачанный файл шаблона отображения содержит маппинг для каждого экспортируемого поля.

    Созданный пример отображения отражает текущие настройки, поэтому его можно использовать при создании индекса, в который будут экспортироваться данные пользовательских сессий.

    PUT /my-usersession-index



{



"settings": {



"index": {



"number_of_shards": 3,



"number_of_replicas": 1



}



},



"mappings": {



"properties": {



// add the mappings from the downloaded file here ...



}



}



}

    Пример

    {



"mappings" : {



"dynamic_templates" : [ {



"string_fields" : {



"match" : "*",



"match_mapping_type" : "string",



"mapping" : {



"norms" : "false",



"type" : "keyword"



}



}



} ],



"properties" : {



"applicationType" : {



"type" : "keyword"



},



"appVersion" : {



"type" : "keyword"



},



// ...



}



}



}

    Настройка экспорта сессий через API¶

    Конфигурация экспорта пользовательских сессий хранится с использованием фреймворка Settings 2.0. Он предоставляет REST API, который можно использовать для создания, чтения, обновления и удаления конфигураций экспорта сессий. Подробности см. в разделе Settings API.

    • Конечные точки для доступа к Settings API:

    • Для развёртываний Dynatrace Managed: https://{your-domain}/e/{your-environment-id}/api/v2/settings/objects/{objectId}

    • Для Environment ActiveGate: https://{your-activegate-domain}/e/{your-environment-id}/api/v2/settings/objects/{objectId}
    • Адрес документации OpenAPI: https://{your-environment-id}.live.dynatrace.com/rest-api-doc/index.jsp?urls.primaryName=Environment%20API%20v2#/Settings%20-%20Objects.
    • ID схемы для конфигурации экспорта пользовательских сессий: builtin:elasticsearch.user-session-export-settings-v2.
      С этим ID схемы можно, например, прочитать текущую конфигурацию экспорта пользовательских сессий с помощью любого REST-клиента.
    GET https://{your-domain}/api/v2/settings/objects?schemaIds=builtin:elasticsearch.user-session-export-settings-v2&scopes=tenant&fields=objectId,value

    Чтобы добавить конфигурацию экспорта сессий через REST API

    POST https://{your-domain}/api/v2/settings/objects?schemaIds=builtin:elasticsearch.user-session-export-settings-v2&scopes=tenant

{

"schemaId": "builtin:elasticsearch.user-session-export-settings-v2",



"scope": "tenant",



"value": {

"enableUserSessionExport": true,

"usePost": false



},



"authentication": {



"active": false



},



"sendDirect": {



"active": false



},



"exportBehavior": {



"managementZone": null,



"disableNotification": false,



"customConfiguration": null



}



}



}



]

    За один запрос можно добавить только одну конфигурацию экспорта пользовательских сессий. Если нужно добавить вторую конфигурацию экспорта сессий, отправь новый запрос POST.

    Подробности об обновлении или удалении объекта настроек см. в разделах Settings API - PUT an object и Settings API - DELETE an object.

    Проверка конфигурации экспорта сессий¶

    Чтобы проверить конфигурацию экспорта сессий, выполни команду ниже.

    curl -v -H "Content-Type: application/json" -X PUT -d '{"visitorId":"14804637803609BCTKP776NMJBOIF3R8OD6R0E4NQALJO","visitId":"16229530","startTime":1480463779085,"endTime":1480463784889,"visitType":"SYNTHETIC"}' http://localhost:3000/export/events

    При необходимости можно задать следующие дополнительные флаги.

    • --insecure для отключения проверки SSL
    • --http1.1, если команда возвращает ошибку REFUSED_STREAM
