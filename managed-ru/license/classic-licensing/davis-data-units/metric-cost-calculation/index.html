    DDU для метрик¶

    DDU для метрик¶

    • время чтения 17 мин
    • Опубликовано 30 марта 2021 г.

    На этой странице описано, как рассчитываются DDU для метрик, что такое пользовательские метрики, а также как оценить и отслеживать потребление DDU для метрик.

    Какие типы метрик потребляют DDU?¶

    • Пользовательские метрики
    • Некоторые встроенные метрики (см. ниже)

    Что такое пользовательская метрика?¶

    Пользовательские метрики, это метрики, которые определяются или устанавливаются самостоятельно. Они расширяют возможности мониторинга Dynatrace, позволяя интегрировать сторонние источники данных, рассчитывать пользовательские метрики, импортировать метрики через API, использовать extensions и многое другое.

    Ниже приведён неполный список типов пользовательских метрик:

    • Все метрики, полученные через API

    • Расчётные метрики служб, пользовательские DEM-метрики и метрики логов

    • Например, расчётные метрики служб, созданные на основе атрибутов запроса

    • Все метрики, предоставляемые вручную установленными Dynatrace Monitoring Extensions

    Полную информацию о настройке и приёме пользовательских метрик в Dynatrace см. в разделе Приём метрик.

    Какие встроенные метрики потребляют DDU?¶

    К метрикам, потребляющим DDU, относятся:

    • Метрики, предоставляемые некоторыми встроенными extensions, которые не включены по умолчанию

    • Переключатели включения/отключения этих метрик в веб-интерфейсе Dynatrace помечены так, чтобы было понятно, влияют ли они на потребление DDU.

    • Все метрики, принимаемые Extensions Public Cloud (AWS, Azure и Google Cloud).

    • Например, метрики, полученные из облачных сервисов.

    Как рассчитывается потребление DDU для метрик?¶

    Потребление происходит при приёме точек данных метрик. Точка данных метрики, это одно измеренное значение, сохранённое в Dynatrace. Значение принадлежит метрике, которая идентифицируется ключом метрики и имеет временную метку. Метрика временного ряда, это последовательность таких точек данных, например использование CPU для данного хоста за анализируемый период времени.

    Каждая принятая точка данных списывает 0,001 DDU с доступной квоты.

    Метрики, предоставляемые Dynatrace, обычно записываются раз в минуту. Это даёт следующее типичное годовое потребление DDU для одной метрики:

    1 точка данных метрики × 60 мин × 24 ч × 365 дней × 0,001 = 525,6 DDU на метрику в год

    Как измерения метрики влияют на потребление DDU?¶

    Точки данных метрик могут иметь связанные измерения, которые идентифицируются ключами и значениями измерений. Например, с использованием синтаксиса протокола приёма метрик два значения могут быть приняты следующим образом:

    my_cpu_utilization,hostname=hostA 55 1609459200000



my_cpu_utilization,hostname=hostB 45 1609459200000

    Для одной и той же временной метки (1609459200000) Dynatrace хранит два разных измеренных значения для одной и той же метрики на двух разных хостах. Это достигается с помощью ключа измерения hostname и значений измерения hostA и hostB. 55 и 45, это измеренные проценты использования CPU для данной метрики на двух хостах.

    Обратите внимание, что это можно рассматривать как две метрики: один график можно создать для использования CPU hostA, а другой, для использования CPU hostB. Если точки данных периодически принимаются для обоих хостов, конечный эффект, это удвоение числа принятых точек данных, потому что здесь каждое измерение по хосту приводит к отдельному значению метрики, которое нужно сохранить.

    Это удвоение принятых точек данных также отражается на соответствующем потреблении DDU: теперь потребление DDU фактически составляет 0,002 в минуту, потому что принимаются данные с двух хостов. Поэтому, чтобы рассчитать потребление DDU на основе ключей метрик, нужно учитывать, какие данные принимаются, и принимать во внимание измерения этих данных.

    Если добавляется больше измерений и число точек данных увеличивается, то потребление DDU также увеличится. Вот пример:

    my_cpu_utilization,hostname=hostA,cpu=1 55 1609459200000



my_cpu_utilization,hostname=hostA,cpu=2 11 1609459200000



my_cpu_utilization,hostname=hostB,cpu=1 45 1609459200000



my_cpu_utilization,hostname=hostB,cpu=2 45 1609459200000

    Этот пример потребит 0,004 DDU в минуту из-за двух хостов и двух CPU.

    Однако дополнительное потребление вызвано не увеличением числа измерений, а увеличением числа принятых точек данных, которое обычно сопровождает рост числа измерений.

    Если измерения добавляются, но число точек данных остаётся прежним, потребление DDU не меняется. Это происходит, когда добавляются чисто описательные измерения, значения которых определяются другими уже существующими измерениями:

    my_cpu_utilization,hostname=hostA,host_ip="127.0.0.23" 55 1609459200000



my_cpu_utilization,hostname=hostB,host_ip="127.0.0.42" 45 1609459200000

    Эти принятые точки данных дают такое же потребление DDU, как если бы они были отправлены без измерения host_ip. Формально говоря, если добавляются измерения, значения которых функционально определяются﻿ другими уже существующими измерениями, то потребление DDU не изменится.

    Включённые метрики на единицу хоста¶

    Если хосты отслеживаются с помощью OneAgent и потребляются host units, то предоставляется определённое число «включённых в хост метрик» на основе потребления host units отслеживаемых хостов. Точнее, для каждой заданной минуты хост, потребляющий host units, имеет бюджет включённых в хост точек данных метрик. Когда принимаются точки данных метрик «о хосте» (например, с формальным измерением хоста), сначала расходуется включённый в хост бюджет точек данных, прежде чем это повлияет на квоту DDU. Одна включённая в хост метрика определяется как одна точка данных метрики в минуту, с учётом предыдущего раздела об измерениях принятых точек данных.

    1000 метрик на host unit включены на каждом отслеживаемом хосте (в режиме Full-Stack Monitoring). Эти метрики включены в стоимость host units Full-Stack и поэтому не потребляют DDU.

    • Каждый отслеживаемый OneAgent хост с включённым Full-Stack Monitoring включает 1000 метрик на Host Unit (подробности см. в таблице ниже).
    • Отслеживаемые OneAgent хосты с включённым Infrastructure Monitoring всегда включают 200 метрик, которые не потребляют DDU.
    • Все метрики, превышающие включённые для хоста метрики, будут потреблять DDU.
    • Если все DDU в пуле Metrics израсходованы, то включённые метрики на единицу хоста также становятся недоступны.
    • Расчётные метрики служб, пользовательские DEM-метрики и метрики логов не учитываются в этих включённых метриках на единицу хоста.

    Обратите внимание, что включённые метрики привязаны к конкретным хостам, отслеживаемым через OneAgent. Это означает, что учитываются только метрики, зафиксированные либо на отслеживаемом хосте, либо на процессе отслеживаемого хоста. Сюда также относятся метрики, зафиксированные на сущностях Amazon EC2, Azure VM и виртуальных машинах VMware (если эти виртуальные машины отслеживаются OneAgent). Обратите внимание, что для метрик, принимаемых через API, это применимо только к метрикам, зафиксированным на отслеживаемом хосте.

    Оценка того, превышает ли хост включённые для хоста метрики, основана на общем числе точек данных метрик, поступающих с частотой раз в минуту (для этих метрик нет скользящего временного окна). Когда принятые точки данных превышают включённые для хоста метрики, за разницу списываются DDU.

    Если хост лицензируется на основе host-unit hours, а не host units, расчёт работает так же: можно потребить до числа включённых для хоста метрик на host-unit hour. Каждая точка данных сверх этого числа потребляет DDU.

    Пример влияния включённых метрик на потребление DDU¶

    Рассмотрим следующий пример.

    • Хост имеет 16 ГиБ RAM, другими словами, это 1 host unit
    • Хост мониторится через OneAgent в режиме Full-Stack

    (1 host unit = 1 000 включённых в тариф метрик хоста)

    • На этом хосте, мониторимом через OneAgent, учитывается 1 500 метрик из различных расширений и локального приёма API (например, JMX и RabbitMQ)
    • 300 метрик отправляются через публичный API без связи с хостом, мониторимым через OneAgent

    В этом сценарии 1 000 метрик/мин включены за счёт host unit, а хост отправляет избыток в 500 метрик, которые будут расходовать DDU. 300 метрик, отправленных через API и не связанных с хостом, мониторимым через OneAgent, также будут расходовать DDU. В сумме получается 800 метрик/мин, расходующих DDU.

    Вот ещё несколько простых сценариев (все значения ниже основаны на метриках, отправляемых с интервалом 1 мин).

    Full-Stack 0.5 500 300 0 0
    Full-Stack 1 1 000 1 500 500 0.5
    Full-Stack 1 1 000 500 0 0
    Full-Stack 4 4 000 5 000 1 000 1
    Infrastructure 0.6 200 150 0 0

    Отражённые DDU и потреблённые DDU¶

    Отражённые DDU определяются как все DDU до применения учёта включённых в хост метрик. Потреблённые DDU отражают общий объём «оплачиваемых» DDU, которые списываются с баланса DDU.

    Можно сравнить отражённые DDU для конкретного хоста с числом включённых в host unit метрик, создав диаграмму в Data Explorer с использованием следующих метрик. Это позволяет получить представление о количестве доступных и потреблённых DDU для конкретного хоста.

    • builtin:billing.ddu.includedMetricDduPerHost Эта метрика разбита по ID хоста. Она показывает включённые в хост DDU. Например, если у хоста 1 000 включённых метрик, значение этой метрики будет равно 1 (другими словами, 1 000 × 0.001)
    • builtin:billing.ddu.metrics.byEntityRaw Метрика byEntityRaw разбита по мониторимой сущности и отражает общее число отражённых DDU до применения учёта включённых в хост метрик.
    • builtin:billing.ddu.metrics.byEntity Как и метрика byEntityRaw, метрика byEntity разбита по мониторимой сущности. Она отражает общее число потреблённых DDU (другими словами, всё потребление сверх количества включённых метрик на host unit). Это соответствует информации, показанной на странице обзора потребления DDU (см. пример ниже).

    Рекомендации по работе с диаграммой¶

    Отражённые и потреблённые DDU

    • Так как биллинг DDU рассчитывается на основе минутных интервалов (подробнее см. пример потребления DDU для метрик), рекомендуется работать в диаграмме также с разрешением в 1 минуту.
    • Использование Split by и Filter by позволяет выбрать конкретный интересующий хост.

    Приведённый выше пример запроса Data Explorer показывает следующую информацию:

    • Фиолетовая линия отражает включённые в хост DDU. В приведённом выше примере это 0.25 DDU (другими словами, 250 метрик) в пересчёте на минуту.
    • Жёлтая линия отражает метрику byEntityRaw. Она показывает общее число отражённых DDU для мониторимой сущности, отфильтрованной по хосту. Любое потребление DDU, превышающее бюджет, включённый в хост, превышает порог фиолетовой линии.
    • Бирюзовая линия отражает информацию из метрики byEntity. Она показывает общее число потреблённых метрик. В примере выше лимит, включённый в хост, превышается примерно каждые 15 минут в период с 09:30 до 10:45, а начиная с 10:45 превышается непрерывно. Это приводит к потреблённым DDU, которые представлены графиком бирюзовой линии.

    Обратите внимание, что точки данных из метрики byEntity могут задерживаться на несколько минут.

    Как оценить необходимое количество DDU для метрик¶

    Ограниченный приём и анализ метрик предоставляется в рамках стандартной поддержки технологии Dynatrace в виде включённых метрик на host unit. Чтобы организовать дополнительный приём и анализ метрик, свяжитесь с отделом продаж Dynatrace﻿.

    Как отслеживать потребление DDU для метрик¶

    В этом разделе описаны два варианта отслеживания потребления DDU для метрик.

    Страница обзора Davis data units¶

    Чтобы узнать, сколько DDU потребила ваша среда

    1. Перейдите в Account Management﻿ > License / Subscription > Overview.

    Для доступа к этой странице нужно право Manage monitoring settings или учётная запись администратора. 2. На странице Davis data units (DDU):

    • В таблице Consumption by DDU pool проверьте строку Metrics.
    • В разделе DDU consumption details проверьте вкладку Metrics.

    Там можно посмотреть потребление DDU и определить основных «виновников» потребления DDU.

    Потребление «Не связанное с мониторимой сущностью»¶

    Источником этого потребления является одно из следующего:

    • Приём метрик для данных без измерения dt.entity.host
    • Вычисляемые метрики из пользовательских сессий для веб, мобильных и пользовательских приложений
    • Вычисляемые лог-метрики v2 - в разбивке по ключам метрик они имеют префикс log.

    FAQ¶

    Какие метрики потребляют DDU?

    Все метрики, отправляемые через стандартный OneAgent (например, метрики хоста), включены бесплатно и не потребляют DDU. Метрики, поступающие через расширение и привязанные к конкретным хостам (или процессам), сначала расходуют все доступные включённые метрики, прежде чем начать потреблять дополнительные DDU. Метрики, поступающие через API, расходуют все доступные включённые метрики, привязанные к конкретным хостам (не процессам), прежде чем начать потреблять дополнительные DDU.

    Следующие типы метрик потребляют DDU:

    Тип метрики Потребляет DDU Включено в host unit
    Встроенные расширения Да Да
    Rest API Да Да1
    JMX Да Да
    Python Да Нет
    Расчётные метрики сервисов Да Нет
    Расчётные метрики логов Да Нет
    Расчётные метрики DEM Да Нет
    Метрики AWS Да Да2
    Метрики Azure Да Да2
    Метрики Google Cloud Да Да2
    Cloud Foundry Нет Нет
    Метрики кластера Kubernetes Нет Нет
    Метрики VMware Нет Нет

    1

    Если для отправки метрик на хост, отслеживаемый OneAgent, используется REST API, эти метрики будут засчитываться в счёт выделенного объёма включённых метрик host unit. Если метрики отправляются на другие сущности, они будут потреблять DDU.

    2

    Метрики облачных сервисов, привязанные к хостам, отслеживаемым OneAgent, вычитаются из выделенного объёма включённых метрик host unit. Например, метрика инстанса EC2 засчитывается в счёт включённых метрик host unit этого инстанса EC2. Все остальные метрики облачных сервисов подлежат потреблению DDU.

    Какие метрики считаются включёнными метриками host unit?

    • Любая метрика, зарегистрированная на отслеживаемом хосте или экземпляре группы процессов, поскольку их можно сопоставить с конкретным хостом. Обратите внимание, что для метрик, поступающих через API, это применимо только к метрикам, зарегистрированным на отслеживаемом хосте.
    • Начиная с версии 1.203 Dynatrace, приём метрик можно регистрировать напрямую на отслеживаемых хостах, что позволяет лучше использовать включённые метрики хоста.
    • В одном из будущих релизов расширения ActiveGate также смогут регистрировать метрики на отслеживаемых хостах и таким образом использовать включённые метрики хоста, так же как это делают расширения OneAgent.

    Используются ли DDU только для расширений?

    Нет, для расширений любая метрика, зарегистрированная на хосте или процессе, потребляет DDU. Для приёма через API сначала расходуются включённые метрики на host unit (иными словами, 1000 включённых метрик для Full-Stack Monitoring), прежде чем начинают потребляться DDU. Подробнее см. Включённые метрики на host unit.

    Какой вес (стоимость) у одной точки данных?

    Каждая точка данных потребляет 0.001 DDU.

    Сколько DDU потребляет одна метрика с частотой 1 минута за год?

    1 точка данных × 60 мин × 24 часа × 365 дней × вес 0.001 DDU = 525.6 DDU

    Почему разбивка потребления по отслеживаемой сущности отличается от разбивки по метрике?

    Разбивка потребления DDU, показанная на страницах Monitored entities и Metrics, а также метрики byEntity и byMetric в API, могут немного отличаться, потому что:

    • Разбивка по отслеживаемой сущности учитывает включённые DDU на host unit, а разбивка по метрике этого не учитывает.
    • Потребление по отслеживаемым сущностям может отставать от метрик byEntity и byMetric в API на срок до 10 минут.

    Поэтому для целей лицензирования и биллинга учитывается только разбивка по отслеживаемой сущности (разбивка по метрике предоставляется исключительно в информационных целях).

    Для метрик, не связанных с отслеживаемым хостом, потребление по отслеживаемой сущности и потребление по метрике идентичны.

    У нас запущено много расширений OneAgent. Покрываются ли эти метрики включёнными метриками? Только в режиме Full-Stack Monitoring?

    Да, метрики через расширение OneAgent покрываются включёнными метриками. Для режима Full-Stack Monitoring это означает 1000 включённых метрик. Для режима Infrastructure это означает 200 включённых метрик.

    Верно ли, что метрики, связанные с хостом или процессом, засчитываются в счёт включённых метрик на host unit?

    Да. Обратите внимание, что для метрик, поступающих через API, это применимо только к метрикам, зарегистрированным на отслеживаемом хосте.

    У нас есть расширения, использующие API для custom device для приёма метрик. Засчитывается ли это в счёт включённых метрик?

    Нет, у custom device нет включённых метрик. Включённые метрики доступны только для хостов и процессов, отслеживаемых OneAgent. Следующие расширения используют custom device для приёма метрик: F5, Windows Server, Apigee Edge, SAP, Juniper Network, IBM I, IBM DataPower, Citrix Netscaler и другие.

    Что произойдёт, если отслеживать удалённые источники с помощью расширений ActiveGate? Будут ли эти метрики засчитываться в счёт включённых метрик host unit?

    Расширения ActiveGate не потребляют включённые метрики host unit. Включённые метрики доступны только для метрик хостов и процессов, отслеживаемых OneAgent.

    Что произойдёт, если изменить конфигурацию с режима Full-Stack на режим Infrastructure Monitoring? Адаптируются ли включённые метрики немедленно?

    Изменение конфигурации должно занять всего несколько секунд, без негативного влияния на потребление метрик.

    Что произойдёт, если отправлять пользовательские метрики с более высокой частотой, например, каждые 10 секунд?

    Рекомендуется передавать пользовательские метрики с частотой 1 раз в минуту. Чем больше точек данных метрик передаётся в минуту, тем больше DDU потребляется. Например, пользовательская метрика, передаваемая с частотой 10 секунд, эквивалентна 6 точек данных × 60 мин × 24 часа × 365 дней × вес 0.001 DDU = 3153.6 DDU

    Потребляют ли облачные сервисы для AWS и Azure DDU, или они входят в включённые метрики на хост?

    Облачные сервисы для AWS и Azure потребляют DDU. Все метрики, принадлежащие хосту, отслеживаемому OneAgent, либо входят в базовый пакет (это не метрики), либо засчитываются в счёт включённых метрик, предоставляемых с каждым host unit. Все остальные метрики расширений или интеграций потребляют DDU.

    Можно ли выбирать, какие сервисы из AWS или Azure отслеживаются, чтобы контролировать потребление DDU?

    Да, можно выбирать, какие отдельные сервисы AWS и Azure отслеживаются. Все «cloud services» доступны для отслеживания по принципу opt-in. Кроме того, с помощью фильтрации по тегам можно отключить отслеживание отдельных «Built-in services» (сервисов, которые отслеживаются автоматически).

    Потребляют ли метрики Kubernetes (использование/лимит/ресурсы CPU) DDU, даже если OneAgent не установлен?

    Узлы Kubernetes отслеживаются автоматически при установке OneAgent; эти метрики входят в базовое предложение OneAgent и включены на host unit.
