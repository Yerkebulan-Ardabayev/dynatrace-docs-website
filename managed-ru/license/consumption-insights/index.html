    Детальная аналитика потребления¶

    Детальная аналитика потребления¶

    • How-to guide
    • 11-min read
    • Published Mar 31, 2025

    В этом разделе описано, как определять первопричины роста использования по каждой возможности DPS и как устранять проблемы с нежелательным или чрезмерным потреблением. Большинство возможностей DPS предоставляют набор метрик потребления DPS, которые можно анализировать в Data Explorer для выявления роста соответствующих затрат.

    Full-Stack Monitoring¶

    Возможные первопричины¶

    • Могло увеличиться количество хостов, отслеживаемых с Full-Stack Monitoring.
    • Мог увеличиться объём памяти хостов, отслеживаемых с Full-Stack Monitoring.

    Как проверить¶

    Используйте Data Explorer для построения графика потребления (DPS) Full-Stack Monitoring по хостам с разбивкой по хостам, чтобы определить, увеличилось ли количество хостов с Full-Stack Monitoring или вырос объём памяти мониторируемых хостов, и если да — выявить затронутые хосты.

    Варианты решения¶

    Отключите мониторинг хостов на непреднамеренно отслеживаемых Full-Stack-хостах или переключитесь на Infrastructure Monitoring для некритичных хостов.

    Infrastructure Monitoring¶

    Возможная первопричина¶

    Увеличилось количество хостов, отслеживаемых с Infrastructure Monitoring.

    Как проверить¶

    Используйте Data Explorer для построения графика потребления (DPS) Infrastructure Monitoring, чтобы определить, увеличилось ли количество мониторируемых хостов.

    Варианты решения¶

    Отключите мониторинг хостов на непреднамеренно отслеживаемых хостах или переключитесь на Foundation & Discovery для некритичных хостов.

    Code Monitoring¶

    Возможные первопричины¶

    • Под существующие правила попали новые хосты/контейнеры.
    • Мониторируемые хосты/контейнеры работают дольше, чем необходимо.

    Как проверить¶

    • Проверьте настройки Observability for Developers и правила мониторинга, чтобы убедиться, что хосты/контейнеры инструментированы с Code Monitoring.
    • Используйте Account Management для просмотра использования на уровне окружения.
    • Используйте Notebooks для детального анализа конкретных хостов/контейнеров, ответственных за рост.

    Варианты решения¶

    • Используйте правила Observability for Developers, чтобы включать только бизнес-критичные сервисы и приложения.
    • Если нет необходимости в данных на уровне кода, отключите Code Monitoring на уровне окружения.

    Runtime Vulnerability Analytics (RVA)¶

    Возможные первопричины¶

    • Был включён мониторинг Application Security с Third-party или Code-Level Vulnerability Analytics.
    • Правила мониторинга Application Security изменились, что привело к увеличению числа мониторируемых хостов.
    • Обнаружены новые хосты, соответствующие существующим критериям мониторинга Application Security.
    • Мониторируемые хосты работают дольше.
    • Мониторируемым хостам выделено больше памяти.

    Как проверить¶

    • Проверьте настройки Application Security и правила мониторинга.
    • Используйте приложение Security Overview для просмотра текущей статистики охвата хостов.
    • Используйте Account Management для просмотра использования на уровне окружения и Data Explorer для детального анализа хостов, ответственных за рост (например, (DPS) Runtime Vulnerability Analytics billing usage per host с разбивкой по хостам).

    Варианты решения¶

    • Скорректируйте правила мониторинга Application Security, чтобы включать только бизнес-критичные сервисы и приложения.
    • Отключите мониторинг неиспользуемых или некритичных технологий (например, .NET и Go).
    • Отключите Code-level Vulnerability Analytics, если мониторинг пользовательского кода не нужен.
    • Отключите Third-party Vulnerability Analytics для полного отключения мониторинга Application Security.

    Runtime Application Protection (RAP)¶

    Возможные первопричины¶

    • Был включён мониторинг Application Security с Runtime Application Protection.
    • Правила мониторинга Application Security изменились, что привело к увеличению числа мониторируемых хостов.
    • Обнаружены новые хосты, соответствующие существующим критериям мониторинга Application Security.
    • Мониторируемые хосты работают дольше.
    • Мониторируемым хостам выделено больше памяти.

    Как проверить¶

    • Проверьте настройки Application Security и правила мониторинга.
    • Используйте Account Management для просмотра использования на уровне окружения и Data Explorer для детального анализа хостов, ответственных за рост (например, (DPS) Runtime Vulnerability Analytics billing usage per host с разбивкой по хостам).

    Варианты решения¶

    • Скорректируйте правила мониторинга Application Security, чтобы включать только бизнес-критичные сервисы и приложения.
    • Отключите мониторинг неиспользуемых или некритичных технологий (например, .NET и Go).
    • Отключите Runtime Application Protection для полного отключения защиты Application Security.

    Real User Monitoring (RUM)¶

    Возможная первопричина¶

    Увеличение числа сессий по следующим причинам:

    • Включение RUM для большего числа frontend-приложений.
    • Увеличение коэффициента захвата до 100% (если ранее он был ниже).
    • Рост трафика на клиентских frontend-приложениях.

    Как проверить¶

    В Data Explorer постройте график метрики (DPS) Real-user monitoring (web and/or mobile) billing usage by frontend application, чтобы увидеть:

    • Количество frontend-приложений с включённым RUM.
    • Коэффициент захвата для каждого frontend-приложения.

    Варианты решения¶

    • Снизьте коэффициент захвата.
    • Отключите RUM для выбранного frontend-приложения.

    RUM с Session Replay¶

    Возможная первопричина¶

    Увеличение числа сессий с Session Replay по следующим причинам:

    • Session Replay включён для большего числа приложений.
    • Коэффициент захвата увеличен до 100%.
    • Рост трафика в клиентских приложениях.

    Как проверить¶

    В Data Explorer постройте график метрики (DPS) Real-user monitoring (web) with Session Replay billing usage by application, чтобы увидеть:

    • Количество приложений с включённым Session Replay.
    • Коэффициент захвата для каждого приложения.

    Варианты решения¶

    • Отключите Session Replay для приложений.
    • Снизьте коэффициент захвата.

    Browser monitor / clickpath¶

    Возможная первопричина¶

    Увеличение числа выполнений по следующим причинам:

    • Добавлены новые мониторы, увеличена частота или клиент начал запускать их из большего числа локаций.
    • Изменилось определение clickpath: добавлены шаги с веб-запросами или XHR.

    Как проверить¶

    • Уточните у команды, какие изменения в конфигурации synthetic-мониторов были недавно выполнены.
    • Проверьте конфигурацию.

    Варианты решения¶

    Для ограничения потребления рассмотрите ограничение числа выполнений за счёт уменьшения числа мониторов, частоты, количества локаций или шагов. Либо отключите часть мониторов.

    Third-party Synthetic API¶

    Возможная первопричина¶

    Больше результатов тестов поступает в Dynatrace через Synthetic Third-party API.

    Как проверить¶

    • Проверьте, появились ли новые сторонние результаты тестов.
    • Проверьте, увеличились ли частота или число локаций для существующих сторонних тестов.

    Варианты решения¶

    Рассмотрите способы ограничения потребления за счёт уменьшения объёма данных, поступающих через Synthetic Third-Party API.

    HTTP monitor¶

    Возможная первопричина¶

    Увеличение числа выполнений по следующим причинам:

    • Добавлены новые мониторы, увеличена частота или клиент начал запускать их из большего числа локаций.
    • Изменилась конфигурация мультишагового HTTP-монитора: добавлены шаги.

    Как проверить¶

    • Уточните у команды, какие изменения в конфигурации synthetic-мониторов были недавно выполнены.
    • Проверьте конфигурацию.

    Варианты решения¶

    Рассмотрите способы ограничения числа выполнений за счёт уменьшения числа мониторов, частоты, количества локаций или шагов. Либо отключите часть мониторов.

    Custom Metrics Classic¶

    Возможная первопричина¶

    • Проблема: Неожиданный рост числа пользовательских метрик.
    • Первопричина: Определённая пользовательская метрика генерирует высокое потребление точек данных метрик.

    Как проверить¶

    Используйте Data Explorer, чтобы:

    • Построить график (DPS) Recorded metric data points per metric key — для выявления роста потребления точек данных метрик.
    • Определить, какая пользовательская метрика начала формировать повышенное число точек данных.

    Варианты решения¶

    • Если пользовательская метрика поступает из облачного сервиса, перейдите в Settings и добавьте/удалите пользовательскую метрику из облачного сервиса.
    • Если пользовательская метрика поступает из расширения, перейдите в конфигурации мониторинга расширения и отключите набор функций для данной метрики.

    Log Monitoring Classic¶

    Возможная первопричина¶

    • Проблема: Неожиданный рост приёма логов.
    • Первопричина: Кто-то включил приём всех логов в правилах приёма.

    Как проверить¶

    В Data Explorer постройте график метрики (DPS) Total Log Monitoring Classic billing usage, чтобы проверить, увеличилось ли общее число записей логов.

    Варианты решения¶

    • Используйте фильтрацию OneAgent для приёма только релевантных логов.
    • Добавьте детализированное управление доступом к конфигурации логов.

    Custom Traces Classic¶

    Возможная первопричина¶

    • Увеличение нагрузки на существующие сущности (например, сценарии «Чёрной пятницы»).
    • Увеличение числа мониторируемых сущностей.

    Как проверить¶

    • Определите сущности, генерирующие трассы, построив график метрики builtin:billing.custom_traces_classic.usage_by_entity:splitBy("dt.entity.monitored_entity"):auto:splitBy():count в Data Explorer.
    • Определите объём трасс по сущностям, построив график метрики builtin:billing.custom_traces_classic.usage_by_entity:splitBy("dt.entity.monitored_entity") в Data Explorer.

    Варианты решения¶

    Оцените, для каких сущностей необходима трассировка, и рассмотрите возможность уменьшения числа мониторируемых сущностей.

    Custom Events Classic¶

    Возможная первопричина¶

    Кто-то отправляет больше данных на конечную точку API.

    Serverless Functions Classic¶

    Возможные первопричины¶

    • Увеличение нагрузки на существующие serverless-функции (например, сценарии «Чёрной пятницы»).
    • Увеличение числа мониторируемых serverless-функций.

    Как проверить¶

    • Проанализируйте среднее число вызовов по функциям, используя среднее значение метрики builtin:billing.serverless_functions_classic.usage_by_function в Data Explorer.
    • Проверьте, увеличилось ли общее число мониторируемых функций, используя подсчёт для метрики builtin:billing.serverless_functions_classic.usage_by_function в Data Explorer.

    Варианты решения¶

    Оцените, для каких serverless-функций необходима трассировка, и рассмотрите использование только метрик там, где это уместно. Либо уменьшите число мониторируемых функций.

    Дополнительное замечание о Workflows¶

    Workflow может вызывать рост затрат для любой лицензионной возможности в зависимости от того, что он делает и как настроен. Например, рост затрат в Query и AppEngine Functions может быть вызван workflow, запускающимся каждую минуту. Это может быть ошибкой конфигурации или намеренным решением. В любом случае необходимо учитывать Workflows в дополнение к отдельной возможности (Query или AppEngine Function в данном примере).
