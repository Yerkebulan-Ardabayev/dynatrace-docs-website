    Обновление конфигурации в SaaS Upgrade Assistant¶

    Обновление конфигурации в SaaS Upgrade Assistant¶

    • Updated on Mar 11, 2025

    Latest Dynatrace

    Импортированную конфигурацию из Dynatrace Managed можно обновить непосредственно в SaaS Upgrade Assistant и развернуть в вашей среде следующими способами:

    • Обновление через Enter edit mode — позволяет изменять конфигурации без необходимости соблюдать форматы вручную, без повторной загрузки пакета и без потери данных развёртывания.
    • Обновление через редактируемые свойства — можно выбрать режим редактирования одиночной записи или массового редактирования.

    Структура конфигурации¶

    Рассмотрим структуру конфигурации, чтобы полностью понять, как работает режим редактирования.

    При экспорте или развёртывании конфигураций каждый элемент представлен в виде JSON-нагрузки. Однако одних JSON-нагрузок недостаточно для эффективного выполнения таких операций, как массовое редактирование или составление списка зависимостей. Поэтому часть информации дополнительно извлекается в виде метаданных. Извлечённая информация может быть использована внутри JSON через специальный формат.

    Таким образом, каждая конфигурация состоит из следующих данных:

    • Raw JSON — JSON после извлечения метаданных (извлечённые свойства указываются по их ID в формате {{.propertyId}})

    • Метаданные, включая:

    • Имя

    • Область применения (только для «встроенных» конфигураций)
    • Обычные свойства — свойства, которые можно редактировать с помощью функции Bulk edit или на вкладке Edit properties.
    • Ссылки на конфигурации — также называемые «ссылочными свойствами». Это свойства, создающие зависимости между конфигурациями, которые можно просмотреть на вкладке Dependencies.

    При просмотре конфигураций в SaaS Upgrade Assistant вкладка JSON отображает не исходный JSON, а JSON с уже заполненными метаданными. Это позволяет просмотреть финальную нагрузку перед развёртыванием в целевой среде. Однако при использовании режима редактирования для корректировки конфигураций структура становится ближе к описанной выше — это даёт полный контроль над исходным JSON и извлечёнными метаданными.

    Редактирование конфигурации в SaaS Upgrade Assistant¶

    Переход в режим редактирования¶

    Перейти в режим редактирования можно двумя способами:

    • Выбрать значок контекстного меню нужной конфигурации, затем выбрать Enter edit mode.
    • Выбрать значок карандаша в правом нижнем углу области предварительного просмотра JSON.

    Как перейти в режим Enter edit mode

    Мастер режима редактирования¶

    В верхней части страницы отображается имя редактируемой конфигурации и доступны следующие действия:

    • Для изменения макета страницы используйте значки столбцов и строк.
    • Для отображения встроенных инструкций выберите Learn more about edit mode.

    Встроенное мини-руководство содержит подробные инструкции по выполнению следующих операций.

    Редактирование JSON-нагрузки:

    • Редактирование значений
    • Добавление нового JSON-узла
    • Удаление JSON-узла

    Управление свойствами конфигурации (метаданными):

    • Редактирование значений
    • Удаление свойства
    • Добавление существующего свойства в JSON-нагрузку
    • Добавление нового свойства
    • Для предварительного просмотра изменений выберите Validate and preview changes. Эта опция активна только при наличии изменений. В окне предварительного просмотра можно сохранить изменения.
    • Для отмены изменений выберите Cancel.

    Редактор JSON¶

    Редактор позволяет изменять исходный JSON: добавлять, редактировать или удалять узлы. Наведите указатель мыши на JSON-узел, чтобы увидеть набор значков с доступными действиями. Редактор следит за корректностью формата JSON.

    Таблица свойств извлечённых метаданных¶

    В верхней части таблицы находится строка поиска и возможность создать свойство. Для каждого свойства в таблице можно:

    • Просмотреть и скопировать ID свойства
    • Просмотреть категорию свойства
    • Редактировать значение свойства
    • Удалить свойство
    • Найти свойство в JSON

    Доступные действия различаются для каждого типа свойства. Например, свойство name нельзя удалить, а свойство scope может изменить свою категорию. Подробнее об этих различиях — в разделе Типы свойств и управление ими.

    Поддерживайте согласованность между JSON и таблицей

    Не все свойства в таблице обязательно должны присутствовать в JSON. Например, свойство scope редко включается (вместо этого оно намеренно добавляется в предварительный просмотр на верхнем уровне).

    Однако для всех ID свойств, присутствующих в JSON, в таблице должно быть соответствующее свойство. Например, если JSON-узел содержит значение {{.extractedIDs.id_APPLICATION_2BAAB3940FB8931A}}, то в таблице должно быть свойство с ID .extractedIDs.id_APPLICATION_2BAAB3940FB8931A. В противном случае валидация завершится с ошибкой, и перед сохранением изменений потребуется исправить эти ошибки.

    Обратите внимание: удалённые свойства, несмотря на то что они только визуально изменяются и остаются в таблице (чтобы можно было отменить удаление), пропускаются при валидации.

    Типы свойств и управление ими¶

    Имя¶

    ID: .name

    Категория: Metadata

    Это простейший тип свойства. Каждая конфигурация имеет имя. Оно используется в SaaS Upgrade Assistant для отображения и не может быть удалено. Для некоторых конфигураций имена извлекаются из JSON на этапе обработки пакета при загрузке и могут представлять различные JSON-узлы. Если подходящий узел не найден, будет присвоено имя по умолчанию, например unnamed disk options.

    Изменения таких значений unnamed не будут развёрнуты в целевой среде, однако они могут упростить поиск конфигураций при работе с SaaS Upgrade Assistant.

    Обычное свойство¶

    ID: начинается с .extractedIDs или .customIDs, например .extractedIDs.id_APPLICATION_2BAAB3940FB8931A.

    Категория: варьируется

    Это тип свойства, который можно редактировать с помощью функции Bulk edit или на вкладке Edit properties. Помимо редактирования значения, режим редактирования позволяет создавать или удалять такие свойства. Созданные свойства впоследствии можно редактировать через Bulk edit. Также появляется возможность добавлять в JSON дополнительные вхождения.

    Ссылочное свойство конфигурации (зависимость)¶

    Пример ID: .syntheticlocation__SYNTHETIC_LOCATION22AAF80F0E2CAF4B__id

    Категория: Reference

    Иногда конфигурации ссылаются друг на друга, что влияет на порядок развёртывания и требования к нему. Зависимые конфигурации всегда должны развёртываться вместе со своими зависимостями. Это не всегда желательно, особенно при наличии циклических зависимостей. Такие случаи можно обработать в режиме редактирования, удалив или изменив ссылочные свойства. При необходимости можно добавить новые зависимости.

    Примечание: любые изменения ссылочных свойств требуют перестройки графа зависимостей. Это происходит автоматически, однако при работе с такими свойствами может наблюдаться увеличенное время сохранения.

    Область применения (scope)¶

    ID: .scope

    Категории: String, Config reference или Property reference.

    Это свойство доступно только для некоторых типов конфигурации, однако при наличии оно обязательно и не может быть удалено. Это единственный тип свойства, способный менять свою категорию. Возможны три варианта:

    • String — обычно environment, означает глобальную область применения; можно свободно изменять в соответствии с потребностями.
    • Config reference — работает так же, как ссылочные свойства конфигурации, описанные выше. Изменение влияет на зависимости.
    • Property reference — обычно используется, когда некая сущность является областью применения вашей конфигурации. Например, disk options определяются на уровне хоста. Каждая конфигурация disk options будет иметь обычное свойство категории host, например HOST-51FEDB44E1BB4DF4 с ID свойства .extractedIDs.id_HOST_51FEDB44E1BB4DF4. Область применения ссылается на свойство хоста, используя его ID в качестве значения.

    Создание свойства¶

    Чтобы создать новое свойство — обычное или ссылочное — выберите Create property и заполните всплывающее окно.

    Для обычных свойств сначала выберите категорию, затем укажите значение свойства. ID свойства будет заполнен автоматически на основе значения.

    Если требуется указать собственный ID свойства, отключите Use the value as property ID.

    Откат изменений¶

    Режим редактирования позволяет свободно изменять конфигурацию, однако допускает и внесение нежелательных изменений. Если внесённые изменения не дали ожидаемого результата или были сохранены по ошибке, можно вернуться к исходному состоянию.

    При откате конфигураций необходимо учитывать следующее:

    • Откат к исходному состоянию означает возврат к состоянию сразу после загрузки пакета конфигурации. Все изменения, внесённые через редактирование свойств, обновление владельцев дашбордов или режим редактирования, будут потеряны — в том числе изменения, внесённые другими пользователями.
    • Эту операцию нельзя отменить.
    • Конфигурации, уже развёрнутые в целевой среде, затронуты не будут. Если часть конфигураций уже была развёрнута, необходимо учитывать возможные несоответствия.
    • Эта операция всегда включает перезагрузку конфигураций и перестройку графа зависимостей, поэтому может занять некоторое время.

    Откат конфигураций возможен на нескольких уровнях:

    • Отдельная конфигурация — как и в режиме редактирования, выберите значок контекстного меню нужной конфигурации, затем выберите Revert changes.

    • Отфильтрованные конфигурации

    • Перейдите на вкладку Upgrade details: configuration и установите фильтры.

    • Выберите значок контекстного меню под графиком Configuration objects.
    • Выберите Revert changes to all (number).

    • Все конфигурации

    • Перейдите на вкладку Upgrade details: configuration.

    • Выберите значок контекстного меню в нижней части страницы.
    • Выберите Revert changes to all (number).

    При откате только части конфигураций остальные останутся без изменений.

    После операции отката будет отображён список конфигураций с восстановленным содержимым.

    Также возможна ситуация, когда все конфигурации, участвующие в откате, уже находятся в исходном состоянии. В этом случае откат не будет выполнен.

    Использование режима редактирования для решения известных проблем¶

    Не удалось сохранить конфигурацию (недопустимый пробел в имени конфигурации)¶

    Некоторые конфигурации не допускают пробелов в начале или конце имени, например конфигурации типа application-web. После развёртывания может появиться следующее сообщение об ошибке:

    Given property 'applicationName' with value: 'Title ' violates the following constraint: Must not have leading or trailing spaces.

    Эту проблему можно решить, отредактировав свойство name в режиме редактирования. В данном случае Title (обратите внимание на пробел) необходимо изменить на Title (без пробела).

    Недопустимый пробел

    Исключение при разборе DQL-запроса¶

    Правила сопоставления логов необходимо конвертировать из LQL в DQL перед развёртыванием. SaaS Upgrade Assistant не выполняет конвертацию самостоятельно. Подробнее см. в разделе Conversion to DQL for Logs.

    Однако с помощью режима редактирования можно вручную исправить простые запросы.

    Рассмотрим в качестве примера конфигурацию типа builtinlogmonitoring.schemaless-log-metric:

    {



enabled: true



key: "log.test1"



query: "host.name="HOST1""



measure: "OCCURRENCE"



dimensions: []



}

    При попытке развернуть эту конфигурацию возникнет ошибка развёртывания со следующим сообщением:

    Caught exception when parsing DQL query 'host.name="HOST1"'.

    С помощью режима редактирования можно исправить эту конфигурацию, заменив запрос на matchesValue(host.name, "HOST1"), который является корректным DQL.

    Нарушение ограничений¶

    Эта проблема может возникать в различных случаях. Рассмотрим пример, связанный с calculated-metrics-service. После развёртывания может появиться следующее сообщение об ошибке:

    At least one condition of the following types must be used: [SERVICE_DISPLAY_NAME, SERVICE_PUBLIC_DOMAIN_NAME, SERVICE_WEB_APPLICATION_ID, SERVICE_WEB_CONTEXT_ROOT, SERVICE_WEB_SERVER_NAME, SERVICE_WEB_SERVICE_NAME, SERVICE_WEB_SERVICE_NAMESPACE, REMOTE_SERVICE_NAME, REMOTE_ENDPOINT, AZURE_FUNCTIONS_SITE_NAME, AZURE_FUNCTIONS_FUNCTION_NAME, CTG_GATEWAY_URL, CTG_SERVER_NAME, ACTOR_SYSTEM, ESB_APPLICATION_NAME, SERVICE_TAG, SERVICE_TYPE, PROCESS_GROUP_TAG, PROCESS_GROUP_NAME]

    Проблему можно решить, добавив соответствующий JSON-узел в режиме редактирования — в данном случае условие внутри массива conditions, например:

    conditions: [



0: {



attribute: "SERVICE_WEB_SERVICE_NAME"



comparisonInfo: {



caseSensitive: false



comparison: "EXISTS"



negate: false



type: "STRING"



value: null



values: null



}



}



]

    Идентификатор сущности не найден¶

    Не все сущности нужно мигрировать при переходе с Managed на SaaS. Иногда достаточно просто удалить ID сущности — например, убрать шаги тестирования из Synthetic-мониторов. Это можно сделать, удалив JSON-узлы в режиме редактирования.

    FAQ¶

    Что означает «исходный» (raw) JSON?

    Исходный JSON — это содержимое JSON-файла нагрузки, хранящегося в SaaS Upgrade Assistant без какой-либо обработки. Именно это отображается в редакторе JSON при использовании режима редактирования. Это означает, что ID извлечённых свойств становятся видимыми, и ими необходимо управлять вручную.

    Что происходит при экспорте конфигурации?

    При экспорте конфигураций Monaco создаёт отдельную директорию для каждого типа конфигурации. Внутри директории находятся множество исходных JSON-файлов и один файл config.yaml. Этот YAML-файл содержит метаданные для всех JSON-файлов в директории.

    Как редактировать файл config.yaml в режиме редактирования?

    Файл config.yaml содержит несколько записей — по одной для каждой конфигурации. Изменяется только элемент, связанный с редактируемой конфигурацией. Кроме того, не все YAML-свойства доступны для редактирования в режиме редактирования. Рассмотрим следующий фрагмент файла config.yaml в качестве примера:

    configs:



- id: 955423ac-a4e5-3dc6-952a-adc35b736f01



config:



name: [Built-in] cloud:aws:cloudtrail



parameters:



extractedIDs:



type: value



value:



id_69de2278_ef7a_4b9a_bdc5_a78c9210ea4e: 69de2278-ef7a-4b9a-bdc5-a78c9210ea4e



id_249491df_4970_421d_8575_5bf555357c14: 249491df-4970-421d-8575-5bf555357c14



networkzone__useast1__id:



configId: us-east-1



configType: network-zone



property: id



type: reference



template: 955423ac-a4e5-3dc6-952a-adc35b736f01.json



skip: false



originObjectId: vu9U3hXa3q0AAAABACNidWlsdGluOmxvZ21vbml0b3JpbmcubG9nLWRwcC1ydWxlcwAGdGVuYW50AAZ0ZW5hbnQAJGM1ODQxMTJmLTk1MWUtNDRkNC1hNjg3LTg4MmMyYmEyZDhlMr7vVN4V2t6t



type:



settings:



schema: builtin:logmonitoring.log-dpp-rules



schemaVersion: 1.0.20



scope: environment

    В режиме редактирования доступны для изменения:

    • config.name
    • type.settings.scope
    • config.parameters — поддерживаются все операции; можно управлять свойствами типа value и reference, изменять их значения, удалять или добавлять новые свойства.

    Режим редактирования не поддерживает изменение следующих YAML-свойств:

    • id — необходим для идентификации конфигурации.
    • config.template — связывает JSON с его метаданными.
    • config.skip — автоматически изменяется в процессе развёртывания в зависимости от включённых конфигураций.
    • config.originObjectId — ссылается на исходную конфигурацию и используется в процессе развёртывания как дополнительный идентификатор.
    • type.settings.schema и type.settings.schemaVersion (для «встроенных» конфигураций) — необходимы для развёртывания конфигураций в правильные API.
    • type.api (если не «встроенная») — необходим для развёртывания конфигураций в правильные API.
