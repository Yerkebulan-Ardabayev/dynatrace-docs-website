    Планирование модернизации развёртывания¶

    Планирование модернизации развёртывания¶

    • Опубликовано 24 апреля 2023 г.

    На этапе планирования собираются требования, учитываются различия, определяется область охвата и выбирается подход к переходу на SaaS. Результатом этапа планирования должен стать подробный план перехода на развёртывание SaaS и хорошо проработанная архитектура целевого решения в рамках платформы Dynatrace.

    Планирование перехода с Managed на SaaS

    Шаг 1. Анализ¶

    Основные задачи этого шага:

    • оценить текущую ситуацию;
    • выявить проблемы и собрать требования;
    • учесть требования к безопасности данных, конфиденциальности и соответствию нормативным требованиям;
    • оценить целесообразность перехода на SaaS.

    Для достижения перечисленных задач нужно рассмотреть следующие темы:

    Оценка текущих потребностей¶

    Как указано в разделе Информация перед модернизацией, перед переходом на развёртывание SaaS нужно знать о различиях между SaaS и Managed. По каждому различию, требующему дополнительных действий, нужно создать пункт действия в документе плана модернизации.

    Проверка требований к безопасности данных, конфиденциальности и соответствию нормативным требованиям¶

    Чтобы приносить пользу клиентам, Dynatrace собирает большие объёмы данных, часто включающих конфиденциальную и персональную информацию пользователей. Dynatrace осведомлён о рисках, связанных с безопасностью и конфиденциальностью данных, и подход можно кратко описать так:

    Безопасность данных¶

    Подход, ориентированный на конкретные сценарии использования, позволяет настроить Dynatrace под конкретные потребности и требования соответствия, включая GDPR и CCPA. Можно легко исключать конфиденциальные данные, включая персональные данные, ещё на этапе захвата: настройка простая, а строгие настройки по умолчанию автоматически исключают данные, в том числе номера IBAN и номера платёжных карт, для соответствия PCI-DSS.

    Вместо этого можно обрабатывать конфиденциальные данные в кластере, получая пользу от первичного анализа данных без их хранения в кластере: данные исключаются на этапе хранения, настройка при этом простая, с полезными значениями по умолчанию.

    Кроме того, доступен детальный контроль доступа к данным и возможность автоматизировать прозрачность событий доступа через журнал аудита API, который позволяет интегрировать события журнала аудита с внутренними решениями SIEM (security information and event management), а также управление правами доступа API для автоматизации ограничений доступа в ответ на события SIEM.

    Подробнее об этом можно узнать на следующих страницах:

    • Средства контроля безопасности данных
    • Средства контроля безопасной разработки

    Конфиденциальность данных¶

    Строгие принципы privacy-by-design и privacy-by-default позволяют настроить Dynatrace так, чтобы максимизировать пользу при соблюдении юридических требований. Нужно изучить и составить список действий, которые требуется предпринять по следующим аспектам:

    • конфиденциальность данных;
    • API конфиденциальности данных;
    • настройки на уровне Environment;
    • Session Replay;
    • Real User Monitoring.

    Информацию по каждой из этих тем можно найти в Trust Center Dynatrace﻿. В частности, рекомендуется ознакомиться со следующими темами:

    • Персональные данные, захватываемые Dynatrace
    • Уровни защиты данных

    Сроки хранения данных¶

    Развёртывание Dynatrace Managed даёт гибкость в настройке сроков хранения ценой соответствующей инфраструктуры. В Dynatrace SaaS значения защитных ограничений хранения по умолчанию оптимизированы и применяются, однако каждое из них можно скорректировать с учётом индивидуальных особенностей.

    Защитные ограничения платформы¶

    Если в среде Dynatrace Managed ранее были увеличены лимиты, подробный анализ сроков хранения и лимитов поможет выявить защитные ограничения, которые могут превышать значения по умолчанию в среде Dynatrace SaaS.

    Для каждого защитного ограничения Dynatrace определяет значение по умолчанию и жёсткий предел. В SaaS настройки можно увеличить сверх значения по умолчанию, вплоть до жёсткого предела, однако такое увеличение может повлечь дополнительные затраты на лицензирование из-за более высокого использования хранилища или вычислительных ресурсов.

    Если лимиты хранения данных нужно увеличить сверх значения по умолчанию для SaaS, рекомендуется заранее согласовать требования по увеличению лимитов с командой Dynatrace и уточнить возможные последствия таких запросов.

    Как правило, лимиты нельзя увеличить сверх жёсткого предела, и если есть необходимость превысить жёсткий предел, нужно обратиться в службу поддержки Dynatrace﻿.

    Переменные привязки SQL (bind variables)¶

    Переменные привязки создают высокую нагрузку на сеть и хранилище, поэтому захват переменных привязки SQL по умолчанию не включён. Чтобы узнать об условиях доступности этой функции в развёртывании Dynatrace SaaS и о том, как начать работу, см. раздел Поддержка переменных привязки SQL.

    Лимиты API¶

    API Dynatrace можно использовать для автоматизации задач мониторинга и экспорта различных типов данных в сторонние инструменты отчётности и анализа. Между Dynatrace Managed и SaaS нет различий в обработке лимитов запросов API. См. API Dynatrace - ограничение доступа. Если ограничение частоты запросов API признано критичным или если кластер Dynatrace Managed был явно рассчитан или настроен для обработки большого количества вызовов API, нужно обратиться в команду Dynatrace.

    Обновления программного обеспечения Dynatrace¶

    В Dynatrace SaaS все обновления программного обеспечения кластера управляются Dynatrace, что автоматически обеспечивает наиболее безопасную и оптимизированную версию Dynatrace, а также доступ ко всем последним разработкам, включая функции Early Access. Обновления SaaS выполняются автоматически, без прерываний и простоев мониторинга, и спроектированы так, чтобы проходить незаметно. Кластеры Dynatrace SaaS обновляются примерно раз в две недели. Хотя кластер обновляется автоматически, полный контроль сохраняется над обновлением OneAgent и настройками обновления ActiveGate.

    Adaptive Traffic Management¶

    Одна из ключевых возможностей Dynatrace, это автоматический захват распределённых трасс. Dynatrace защищает производительность и доступность платформы, проактивно управляя интенсивностью захвата трафика. Метод Adaptive Traffic Management в Dynatrace SaaS отличается от Dynatrace Managed: он автоматически адаптируется и настраивается с меньшей зависимостью от архитектуры конкретного отслеживаемого приложения. Защитные ограничения применяются к общему объёму данных, отправляемых в среду Dynatrace, что позволяет лимитам масштабироваться вместе с лицензированием и даёт точкам входа с высоким объёмом трафика возможность отправлять больше данных. Подробную информацию можно найти в разделе Переход на Dynatrace SaaS﻿, где указаны конкретные лимиты.

    Предотвращение перегрузки RUM¶

    Можно контролировать количество захватываемых пользовательских действий в веб-, мобильных и OpenKit-приложениях, задав максимальное количество пользовательских действий в минуту. По достижении этого лимита Dynatrace автоматически снижает количество захватываемых пользовательских сессий, пока число пользовательских действий не вернётся в рамки лицензионных ограничений. Настройку предотвращения перегрузки RUM нужно спланировать в соответствии с требованиями в среде Dynatrace SaaS. Подробности см. в статье RUM: что означает сообщение «Превышено максимальное количество пользовательских действий в минуту»?﻿ в сообществе Dynatrace.

    Если требуется дополнительная информация для решения вопросов безопасности данных, конфиденциальности, соответствия нормативным требованиям и управления, нужно обратиться в отдел продаж Dynatrace.

    Шаг 2. Разработка стратегии¶

    Основные задачи этого шага:

    • оценить область перехода на Dynatrace SaaS;
    • понять объём усилий, необходимых для перехода на Dynatrace SaaS;
    • оценить свои возможности и ресурсы;
    • выбрать между самостоятельным переходом или переходом с помощью Dynatrace ACE Services;
    • составить план перехода на Dynatrace SaaS.

    Следующие темы нужно рассмотреть на этом шаге, они помогут в достижении перечисленных задач:

    Миграция данных¶

    Dynatrace хранит три основных типа данных:

    • Конфигурационные данные
    • Учётные записи пользователей и права доступа
    • Историческая мониторинговая информация

    Миграция конфигурационных данных, таких как дашборды, приложения и теги, важна для подготовки новой SaaS-среды к приёму мониторинговых данных от всех источников, ранее передававших их в кластер Dynatrace Managed. Независимо от того, переносится ли конфигурация или создаётся с нуля, нужно настроить новые учётные записи пользователей и права доступа в SaaS-среде. Объём и сложность этой работы зависят от количества пользователей, групп, management zones и IAM-политик, которые планируется перенести.

    Из-за технических ограничений перенести историческую мониторинговую информацию (такую как метрики, трейсы, проблемы, события и логи) из одной среды в другую невозможно. Однако доступ к этим данным сохраняется в кластере Dynatrace Managed. Кластеры Dynatrace Managed можно эксплуатировать параллельно, если требуется доступ к историческим данным. Это может потребовать одновременного использования лицензий Dynatrace Managed и SaaS.

    Также может быть полезно настроить Set up cross-environment tracing и Create remote/multi-environment Dynatrace dashboards, чтобы связать среды Dynatrace Managed и SaaS для доступа к данным.

    Одновременное использование лицензий Dynatrace Managed и SaaS¶

    Dynatrace может по запросу предоставить дополнительные лицензии на Dynatrace SaaS, которые можно использовать бесплатно в процессе апгрейда. Это даёт достаточно времени для сбора более длительных трендовых данных в SaaS-среде, а также гарантирует, что целевая SaaS-среда работоспособна до переключения. SaaS-лицензии можно использовать только для рабочих нагрузок и настроек, которые переносятся, и они предоставляются только на период апгрейда, до шести месяцев. По завершении процесса апгрейда лицензии Dynatrace Managed будут деактивированы, и оплата пойдёт за SaaS-лицензии. За подробностями нужно обратиться к команде продаж Dynatrace.

    Оценка объёма работ и выбор подходящего подхода¶

    Нужно оценить объём работ, необходимые навыки и ресурсы для выполнения апгрейда. Начать стоит с каталогизации, оценки среды и определения конфигурационных элементов, которые нужно перенести. Далее следует составить каталог и провести обзор экземпляров OneAgent и ActiveGate, развёрнутых в среде. Это поможет определить предпочтительный подход к апгрейду до SaaS.

    Какие варианты подхода к апгрейду существуют?¶

    У каждой организации свои потребности. Dynatrace предлагает два различных подхода к апгрейду:

    • Подход Big Bang
      Рекомендуется клиентам с сильно взаимосвязанной и взаимозависимой инфраструктурной средой.
    • Поэтапный подход (Phased)
      Рекомендуется организациям с изолированными командами и инфраструктурными сущностями.

    У каждого метода есть свои преимущества и особенности, которые нужно учитывать. Стоит выбрать стратегию, которая лучше всего соответствует целям организации и операционным предпочтениям.

    Подход Big Bang¶

    Подход Big Bang предполагает полный перенос за один этап. Эта стратегия идеально подходит организациям, стремящимся к быстрому и решительному переходу.

    Преимущества:

    • Сохраняется сильно связанная и взаимозависимая настройка мониторинга
    • Всё переносится сразу с минимальным риском простоя мониторинга
    • Процесс апгрейда экономит время и затраты

    Недостатки:

    • Требуется большой объём координации и коммуникации
    • Рекомендуется проводить тесты в небольших песочницах или предпродакшн-средах

    Поэтапный подход¶

    Поэтапный подход предполагает постепенный переход, позволяющий переносить отдельные компоненты или группы пользователей поэтапно во времени. Он обеспечивает гибкость и минимизирует возможные сбои в работе.

    Преимущества:

    • Изолированные инфраструктурные и организационные настройки можно переносить по одной
    • Метод проб и ошибок с быстрой обратной связью и обучением настройке нового SaaS-тенанта
    • Отсутствие простоя благодаря временной гибридной настройке

    Недостатки:

    • Временная гибридная модель может привести к разрыву Full-Stack Monitoring и нарушению Real User Monitoring
    • Более длительные сроки апгрейда и более высокие затраты из-за более долгих периодов дублирования развёртываний (оборудование и облако)
    • Различия в веб-интерфейсе Dynatrace между Dynatrace Managed и Dynatrace SaaS

    Каталогизация и оценка среды¶

    Нужно начать с обзора того, какие конфигурационные элементы требуется перенести, и определить, можно ли перенести их с помощью автоматизированной утилиты или потребуется ручной труд. Это хорошая возможность провести очистку среды и применить лучшие практики и стандарты. Избыточные или устаревшие конфигурационные элементы можно оставить без переноса. Большинство конфигурационных элементов, таких как теги или management zones, обычно переносятся полностью, тогда как другие, например дашборды, можно просмотреть и перенести выборочно.

    Ниже приведён список конфигураций, которые обычно переносятся:

    • Network zones
    • Management zones
    • Custom services
    • Synthetics
    • Определения приложений
    • Anomaly detection
    • Уведомления и интеграции
    • Токены API

    • OpenTelemetry

    • Группы пользователей

    • Правила тегирования
    • Конфигурация сервисов
    • Cloud automation
    • Настройки глубокого мониторинга
    • Профили оповещений
    • Окна обслуживания
    • Хранилище учётных данных (credentials vault)

    • Интеграции с облачными платформами

    • Настройка SSO

    • Правила именования
    • Extensions
    • Обнаружение групп процессов
    • Мониторинг логов
    • Дашборды
    • SLI/SLO
    • Mainframe volumes
    • Пользовательские метрики

    Каталогизация и оценка развёрнутых компонентов Dynatrace¶

    После документирования конфигурации Dynatrace, которую нужно применить к среде Dynatrace, полезно составить каталог и провести обзор компонентов, настроенных или развёрнутых в среде, и определить предпочтительный подход к их переносу на Dynatrace SaaS. Версии компонентов влияют на возможности развёрнутых OneAgent по подключению к платформе Dynatrace, а также на доступные методы апгрейда.

    Обновление

    Настоятельно рекомендуется обновить компоненты мониторинга Dynatrace до последних версий и применить стандартизированные конфигурации и лучшие практики.

    Компоненты Dynatrace, которые нужно перенести в новую среду Dynatrace:

    • Экземпляры OneAgent (такие как хосты, PaaS и Kubernetes/Openshift)
    • Расширения ActiveGate
    • Расширения OneAgent
    • Внешние или пользовательские источники
    • Экземпляры ActiveGate (маршрутизация трафика, мониторинг облачных сред, мониторинг удалённых технологий с расширениями, синтетические мониторы, модули zRemote)

    Для каждого элемента мониторинга нужно определить версию OneAgent или расширения и выяснить необходимые обновления как с технической, так и с процессной точки зрения.

    • Более новые версии OneAgent предоставляют больше возможностей конфигурации, и версия развёрнутых OneAgent определяет, нужно ли их обновлять или переустанавливать полностью.
    • Поддержка версий OneAgent составляет 9-12 месяцев; нужно проверить самые старые поддерживаемые версии.
    • Миграция RUM-приложений из Dynatrace Managed в Dynatrace SaaS без побочных эффектов в любой из сред требует версии OneAgent 1.289+; перед переносом в Dynatrace SaaS нужно обновить OneAgent.
    • Требуется перезапуск приложений, использующих Full-Stack Monitoring, это нужно учесть в плане апгрейда.

    Обновление OneAgent

    Создание плана обновления¶

    Создание реализуемого плана обновления до SaaS необходимо для максимального успеха всей операции. Рекомендуется начинать планирование и подготовку минимум за 2-3 месяца до фактической фазы миграции. На предыдущих шагах должен был появиться список действий, устраняющих различия между двумя моделями развёртывания, и уточнён объём работ. Теперь пришло время перечислить все действия, необходимые на этапе выполнения. Ниже приведены детали и аспекты, которые нужно учесть при создании плана обновления до SaaS.

    Действия перед обновлением¶

    Если в окружении Managed используется ручное тегирование, нужно преобразовать эти правила в правила автоматического тегирования. Чтобы завершить документирование существующего окружения Dynatrace Managed, следует определить и надёжно задокументировать следующее:

    • Проверить используемые токены API и PaaS и определить их назначение.
    • Надёжно задокументировать учётные данные для облачных и контейнерных платформ, таких как AWS, Azure и Kubernetes.
    • Скоординироваться и подключить все команды, которые затронуты миграцией или владеют её частями.
    • Командам платформы нужно спланировать обновление файлов развёртывания и перезапуски для всей своей инфраструктуры и сервисов.
    • Сетевой команде нужно заранее помочь воспроизвести все правила брандмауэра из предыдущего окружения в новом. Также они должны быть на связи во время миграции на случай проблем.
    • Спланировать каналы коммуникации и сообщения, которые нужно разослать командам приложений. Получить необходимые согласования от руководителей команд и менеджеров на дату, когда возможен перезапуск приложений и сервисов.
    • Создать списки отслеживания. Например, таблицу с основным контактом каждой команды приложения, который будет на связи в выходные для перезапуска своего приложения.
    • Заранее убедиться, что окружение правильно рассчитано по размеру. Связаться с командой Dynatrace минимум за 1 неделю для уведомления, если планируется миграция более 5000 OneAgent.

    Миграция конфигурации¶

    Цель, применить ту же конфигурацию из окружения Dynatrace Managed в новом окружении Dynatrace SaaS. В идеале это делается автоматически там, где возможно, с помощью таких инструментов, как Dynatrace Configuration-as-code.

    Инвентаризация мониторинга¶

    Как было описано ранее (Каталогизация и оценка развёрнутых компонентов Dynatrace), в результате получается список компонентов Dynatrace, развёрнутых в текущем окружении Dynatrace Managed.

    Миграция мониторинга¶

    Все компоненты Dynatrace в окружении Managed нужно перенастроить для отправки данных в новое окружение Dynatrace. Это может также привести к замене существующих развёрнутых компонентов новыми компонентами для Dynatrace, такими как ActiveGate. Подробности см. в разделе Обновление развёртывания.

    Операционализация¶

    Скорректировать существующие процессы и рабочие процессы, включая любые интеграции с ITOM/ITSM, и обновить любые ссылки на окружение Dynatrace Managed на новое окружение Dynatrace.

    Ограничения брандмауэра для RUM¶

    Конфигурация брандмауэров, прокси и веб-серверов потребует корректировки, поскольку имена cookie имеют суффиксы, специфичные для окружения. Подробности см. в разделе Требования к сквозному прохождению инфраструктуры для RUM Classic.

    Шаг 3, Проектирование¶

    На этом шаге создаётся целевой дизайн платформы Dynatrace SaaS. Помимо создания нового окружения Dynatrace SaaS, также нужно рассмотреть и определить окружающую техническую архитектуру (например, сетевой дизайн, влияние на производительность и эксплуатацию), чтобы обеспечить возможность выполнения шагов предоставления ресурсов и развёртывания на последующих этапах.

    Основные цели этого шага:

    • Проверить зависимости, которые могут повлиять на обновление
    • Проверить ограничения, которые могут повлиять на обновление
    • Определить целевую архитектуру

    Результатом этого шага может быть одна диаграмма, документ архитектурного дизайна высокого или низкого уровня, либо письменное краткое описание спецификации, в зависимости от управления, процессов и требований.

    На шаге проектирования нужно учесть следующее:

    Консолидированное окружение SaaS¶

    В Dynatrace Managed есть возможность создавать окружения по мере необходимости. Однако в Dynatrace SaaS важно определить целевой дизайн окружений. Размер и конфигурация окружений потребуются в качестве входных данных при предоставлении окружения SaaS, это будет ценным вкладом при рассмотрении оптимизации конфигурации. Также можно консолидировать несколько кластеров Dynatrace в единый tenant, что даёт преимущества, но и несколько моментов для учёта:

    • Единый источник для конфигурации, дашбордов и окон обслуживания значительно снижает административную нагрузку.
    • Единый домен сбоев позволяет Dynatrace Davis AI предоставлять более полный анализ, а также иметь единое место для просмотра полных трасс из мультирегиональных приложений.
    • Больше приложений, сервисов, процессов и хостов управляются в одном месте. Метаданные, тегирование, management zones, network zones и лучшие практики именования станут более критичными для обеспечения правильной группировки сущностей и данных, а также контроля доступа.

    Обратите внимание, что консолидация окружений может потребовать дополнительной проверки лимитов SaaS, чтобы новое консолидированное окружение оставалось в пределах защитных ограничений платформы.

    Пример архитектуры консолидации окружений Dynatrace¶

    Вот пример того, как может выглядеть план консолидации окружений.

    • 14 окружений Dynatrace Managed на двух кластерах были консолидированы в три окружения SaaS
    • Environment сгруппированы в SaaS по стадии, а не по приложению
    • Снижена сложность интеграций, конфигурации и пользовательского доступа к данным
    • Полная сквозная трассировка транзакций без конфигурации между окружениями

    Network zones¶

    Dynatrace позволяет определять network zones, представляющие архитектуру сети. Network zones используются для маршрутизации телеметрических данных от OneAgent к кластеру Dynatrace оптимизированным способом и позволяют избежать лишнего сетевого трафика между дата-центрами и сетевыми регионами. Подробнее см. Network zones.

    Информацию о соображениях по ActiveGate см. в разделе Dynatrace ActiveGate.

    Развёртывание и определение размера ActiveGate¶

    ActiveGate позволяет оптимизировать и минимизировать использование пропускной способности за счёт развёртывания нужного количества экземпляров ActiveGate в каждом дата-центре и обеспечения использования network zones. Для миграции следует подготовить новые экземпляры ActiveGate, а существующие экземпляры ActiveGate вывести из эксплуатации после завершения миграции. Это обеспечивает бесперебойность мониторинга Dynatrace. Сюда входит рассмотрение того, как развернуть и настроить:

    • Environment ActiveGate, трафик OneAgent (SSL-сертификат)
    • Environment ActiveGate, Cloud & Virtualization
    • Environment ActiveGate, Synthetics
    • Environment ActiveGate, Extensions
    • Environment ActiveGate, приём логов (SSL-сертификат)
    • Environment ActiveGate, мейнфрейм z/OS

    При определении размера ActiveGate нужно убедиться, что есть достаточная ёмкость на случай сбоев. Лучшие практики Dynatrace предписывают минимум два ActiveGate на дата-центр, с добавлением дополнительных экземпляров исходя из требований к ёмкости.

    Проверка зависимостей и ограничений, которые могут повлиять на обновление¶

    Переход с Dynatrace Managed на развёртывание SaaS может потребовать изменений в настройке сети, а также пересмотра и перенастройки внешних систем, интегрированных с Dynatrace (например, для уведомлений о проблемах см. Уведомления о проблемах). Проверка текущей настройки сети и интегрированных систем формирует основу для дальнейших архитектурных решений; проверить и спроектировать компоненты Dynatrace (ActiveGate) с использованием network zones.

    Определение целевой архитектуры¶

    Целевой дизайн для платформы Dynatrace создан. Помимо настройки и конфигурации новой среды Dynatrace, нужно рассмотреть и специфицировать окружающую техническую архитектуру, например, дизайн сети, объём ресурсов и влияние на эксплуатацию, чтобы обеспечить возможность выполнения шагов подготовки и развёртывания на последующих этапах.

    Результатом фазы дизайна может быть одна диаграмма, документ высокоуровневой или низкоуровневой архитектуры или письменная сводка спецификации, в зависимости от процессов управления, процедур и требований. Ниже приведён пример диаграммы целевой архитектуры для базового развёртывания Dynatrace Managed. Использованы следующие допущения:

    • Единый кластер Dynatrace Managed в DC-1 переносится в единую среду SaaS (ID: vre43162)
    • OneAgent и внешняя автоматизация используют единый Environment ActiveGate для передачи данных от кластера и к кластеру
    • Agentless, Mobile RUM, Synthetic и OneAgent, расположенные вне DC-1, используют ActiveGate кластера для передачи данных в кластер
    • Корпоративный фаервол блокирует прямой сетевой доступ к Dynatrace SaaS.
    • Зелёные стрелки и компоненты обозначают изменения, необходимые при обновлении
