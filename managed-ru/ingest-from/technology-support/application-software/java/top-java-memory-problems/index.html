    Главные проблемы памяти Java¶

    Главные проблемы памяти Java¶

    • Чтение: 15 мин
    • Опубликовано 29 марта 2019 г.

    В этой статье рассматриваются часто встречающиеся проблемы с памятью: утечки памяти, высокое потребление памяти, проблемы с загрузчиком классов и настройка GC.

    Утечки памяти¶

    Самая распространённая проблема утечек памяти, это растущие утечки, то есть постоянное увеличение числа утекающих объектов. Такие утечки легко отслеживаются по трендовым или гистограммным дампам.

    Утечки одиночных объектов, напротив, обсуждают реже. Пока памяти достаточно, такие утечки редко создают серьёзную проблему. Однако время от времени одиночный утекающий объект занимает значительный объём памяти и становится проблемой. Хорошая новость в том, что крупные одиночные утечки легко выявляются инструментами анализа heap.

    Ниже перечислены некоторые причины утечек памяти:

    Thread-local переменные

    Переменные ThreadLocal используются для привязки переменной или состояния к потоку. У каждого потока свой экземпляр переменной. Хотя переменные ThreadLocal могут быть полезны, они также могут быть опасны. Их применяют для отслеживания состояния, например текущего идентификатора транзакции, но иногда в них хранят и более обширную информацию.

    Переменная ThreadLocal ссылается на свой поток, и её жизненный цикл привязан к потоку. В большинстве серверов приложений потоки переиспользуются через пулы и никогда не собираются сборщиком мусора. Если код приложения не очищает переменную ThreadLocal аккуратно, это приводит к неприятной утечке памяти. Такие утечки легко обнаружить с помощью heap dump, просмотрев ThreadLocalMap и прокрутив цепочку ссылок.

    На скриншоте выше heap dump показывает, что более 4 тыс. объектов общим объёмом около 10 МБ удерживаются переменными ThreadLocal. Имя потока подсказывает, какая часть приложения отвечает за утечку.

    Изменяемые статические поля и коллекции

    Самая частая причина утечек памяти, это неправильное использование статиков. Статическая переменная удерживается своим классом и, соответственно, его загрузчиком классов. Хотя класс может быть собран сборщиком мусора, в течение жизни приложения это происходит крайне редко. Часто статики используют, чтобы хранить кэш или разделять состояние между потоками. Если делать это небрежно, легко получить утечку памяти. По этой причине изменяемых статических коллекций нужно избегать любой ценой. Хорошее архитектурное правило: вообще не использовать изменяемые статические объекты, а выбирать более удачные альтернативы.

    Циклические и сложные двунаправленные ссылки

    Рассмотрим следующий пример:

    org.w3c.dom.Document doc = readXmlDocument();



org.w3c.dom.Node child = doc.getDocumentElement().getFirstChild();



doc.removeNode(child);



doc = null;

    В конце фрагмента предполагается, что DOM Document будет собран сборщиком мусора. Однако это не так. Объект DOM Node всегда принадлежит Document. Даже после удаления из Document объект Node по-прежнему ссылается на свой документ-владельца. Пока удерживается child-объект, ни Document, ни все содержащиеся в нём узлы не будут собраны сборщиком мусора.

    Утечки памяти JNI

    JNI используется для вызова нативного кода из Java. Этот нативный код может обрабатывать, вызывать и создавать Java-объекты. Каждый Java-объект, созданный нативным методом, начинает свою жизнь как local reference, то есть остаётся доступным по ссылке до возврата из нативного метода. Нативный метод ссылается на Java-объект, поэтому проблем нет, если только нативный метод не выполняется бесконечно. В некоторых случаях создаваемый объект нужно сохранить и после завершения нативного вызова. Для этого можно либо обеспечить, чтобы на него ссылался какой-то другой Java-объект, либо превратить local reference в global reference. Глобальная ссылка является корнем GC и никогда не будет собрана сборщиком мусора, пока её явно не удалит нативный код. Единственный способ обнаружить такую утечку, это использовать инструмент анализа heap, который явно показывает глобальные нативные ссылки. Если применять JNI приходится, ссылайтесь на такие объекты обычным образом и совсем откажитесь от глобальных ссылок.

    Этот тип утечки можно найти, когда инструмент анализа heap dump явно помечает GC Root как нативную ссылку, иначе разобраться будет тяжело.

    Неправильная реализация equals/hashcode

    Если методы equals/hashcode нарушают контракт equals, это приводит к утечкам памяти при использовании объекта в качестве ключа в map. HashMap использует hashcode, чтобы найти объект, а затем проверяет совпадение методом equals. Если два объекта равны, у них должен быть одинаковый hashcode, но не наоборот. Это не так, если hashcode явно не реализован. Стандартный hashcode основан на идентичности объекта. Поэтому при использовании в map объекта без корректной реализации hashcode элементы можно добавлять, но найти их потом не получится. Если попытаться добавить такой же объект снова, он не перезапишет старый элемент, а добавит новый, что приведёт к утечке памяти. Утечку видно по росту, но первопричину определить сложно, если не помнить про эту особенность.

    Самый простой способ избежать такой утечки, это использовать модульные тесты и один из доступных фреймворков, который проверяет контракт equals для ваших классов.

    Утечки загрузчика классов

    Утечки загрузчика классов встречаются в серверах приложений и контейнерах OSGi. Классы удерживаются их загрузчиком и обычно не собираются сборщиком мусора до тех пор, пока сам загрузчик не будет собран. Однако это происходит только при выгрузке приложения сервером приложений или контейнером OSGi. Существуют две формы утечек загрузчика классов:

    • Первая форма: когда объект, чей класс принадлежит загрузчику, продолжает удерживаться кэшем, локальным потоком или каким-либо иным образом. В этом случае весь загрузчик классов, то есть всё приложение целиком, не может быть собран сборщиком мусора. Это происходит в контейнерах OSGi и серверах приложений JEE. Такие утечки памяти встречаются нечасто, потому что возникают только при выгрузке или повторном развёртывании приложения.
    • Вторая форма появилась благодаря фреймворкам манипуляции байт-кодом, таким как BCEL и ASM. Эти фреймворки позволяют динамически создавать новые классы. Если о созданных классах забывают, ответственный код может создавать новые классы для одной и той же цели многократно. Класс остаётся в текущем загрузчике, из-за чего может возникнуть утечка памяти, ведущая к нехватке памяти в permanent generation. Большинство инструментов анализа heap эту проблему не подсвечивают, поэтому её приходится разбирать вручную. Эта форма утечек известна из-за инцидента в старой версии hibernate и использования им CGLIB.

    Высокое использование памяти¶

    Чрезмерное потребление памяти, это всё более частая и критичная проблема в современных корпоративных приложениях. Хотя у среднего сервера часто 10, 20 и более ГБ памяти, высокая степень параллелизма и недостаточное внимание к ресурсам приводят к её нехватке. Ещё одна проблема: в современных JVM можно использовать несколько гигабайт памяти, но побочный эффект, это очень длинные паузы GC. В некоторых случаях увеличение памяти может стать обходным решением для утечек или плохо написанного ПО. Однако чаще такой подход в долгосрочной перспективе ухудшает ситуацию, а не улучшает её. Ниже перечислены самые распространённые причины высокого использования памяти:

    HTTP-сессия как кэш

    Антипаттерн «кэширование в сессии» означает неправильное использование HTTP-сессии в качестве кэша данных. HTTP-сессия предназначена для хранения данных пользователя или состояния, которые должны пережить один HTTP-запрос. Это так называемое состояние диалога, оно встречается в веб-приложениях с нетривиальными пользовательскими сценариями. С HTTP-сессией связано несколько проблем.

    Во-первых, пользователей может быть много, поэтому на одном веб-сервере одновременно бывает довольно много активных сессий. Поэтому важно держать их небольшими. Вторая проблема в том, что приложение не освобождает их в конкретный момент. Вместо этого у веб-серверов есть тайм-аут сессии, который часто делают довольно большим ради комфорта пользователя. Уже это, если учесть число параллельных пользователей, легко приводит к большим требованиям к памяти. На практике HTTP-сессии обычно весят несколько мегабайт.

    Такие кэши возникают потому, что разработчику проще и удобнее добавить объект в сессию, чем продумать другие решения, например кэш. Часто это делается по принципу «отправил и забыл»: данные оттуда никогда не удаляются.

    Пример: хранение данных, которые отображаются в HTML-полях выбора, например списков стран. Эти полустатичные данные часто занимают несколько килобайт и хранятся в heap у каждого пользователя, если положить их в сессию. Лучше держать их в одном общем кэше. Другой пример: неправильное использование hibernate-сессии для управления состоянием диалога. Hibernate-сессию помещают в HTTP-сессию, чтобы обеспечить быстрый доступ к данным. Это означает хранение гораздо большего объёма состояния, чем нужно, и уже при нескольких пользователях потребление памяти резко возрастает. В современных Ajax-приложениях состояние диалога можно перенести на клиента. В идеальном случае это приводит к серверному приложению без состояния или с минимальным состоянием, которое гораздо лучше масштабируется.

    Репликация длинных HTTP-сессий, это ещё одна сопутствующая проблема.

    Неправильное использование кэша

    Кэши применяют, чтобы повысить производительность и масштабируемость, загружая данные лишь однажды. Однако чрезмерное использование кэшей быстро приводит к проблемам производительности Java. В дополнение к типичным проблемам кэшей, таким как промахи и высокая ротация, кэш также может приводить к высокому потреблению памяти и избыточной работе GC. Эти проблемы обычно вызваны слишком большим кэшем. Однако иногда причина лежит глубже.

    Soft reference, это особая форма ссылки на объект. Хотя soft reference могут быть освобождены сборщиком мусора в любой момент по его усмотрению, фактически их освобождают только для того, чтобы избежать ошибки нехватки памяти. В этом они существенно отличаются от weak reference, которые никогда не препятствуют сборке мусора объекта. Поэтому soft reference очень популярны в реализациях кэша.

    Разработчик кэша справедливо предполагает, что данные кэша будут освобождены в случае нехватки памяти. Если кэш настроен неверно, он будет расти быстро и без ограничений, пока память не закончится. При запуске GC все soft reference в кэше очищаются, а их объекты собираются сборщиком мусора. Потребление памяти возвращается к базовому уровню, после чего снова начинает расти. Это явление легко принять за неправильно настроенный young generation. Кажется, что объекты слишком рано переводятся в tenured только для того, чтобы быть собранными следующим major GC. Такая проблема часто запускает упражнение по тюнингу GC, которое не может увенчаться успехом.

    Помочь определить первопричину может только корректный мониторинг метрик кэша или анализ heap dump.

    Темпы оттока и высокий транзакционный расход памяти

    Поколенческий GC спроектирован под большое количество очень короткоживущих объектов. Если транзакционный расход памяти слишком высок, это быстро ведёт к проблемам производительности или стабильности. Однако такие проблемы видны только под нагрузочным тестом и могут быть легко упущены во время разработки.

    Если за слишком короткое время создаётся слишком много объектов, это приводит к учащённым GC в young generation, которые дешёвы лишь тогда, когда большинство объектов умирает. Если много объектов переживает GC, такой сбор фактически дороже, чем при тех же условиях GC в old generation. Поэтому высокие потребности отдельных транзакций в памяти могут не быть проблемой в функциональном тесте, но быстро привести к GC thrashing под нагрузкой. Если нагрузка растёт ещё больше, такие транзакционные объекты переводятся в old generation, поскольку young generation становится слишком малым. Если увеличить размер young generation, проблема отодвинется, но в итоге приведёт к ещё более длинным паузам GC.

    Ошибка из-за высокого транзакционного расхода памяти, это наихудший возможный сценарий. Если памяти и так в обрез, более высокая транзакционная нагрузка может попросту исчерпать доступный heap. Сложность в том, что когда срабатывает OutOfMemory, транзакции, которые пытались выделить объекты и не смогли, прерываются. После этого освобождается много памяти, и мусор собирается. Поскольку большинство инструментов смотрят на память Java каждые несколько секунд, они могут вообще не показать 100% памяти ни в один момент времени.

    Начиная с Java 6 можно настроить создание heap dump при возникновении OutOfMemory, что покажет первопричину. Если OutOfMemory не возникает, для выявления классов, число объектов которых колеблется сильнее всего (обычно это те, что много раз выделяются и тут же собираются), помогают трендовые или гистограммные дампы. Последнее средство, это полноценный анализ выделений.

    Крупные временные объекты

    Временные объекты тоже вызывают ошибки нехватки памяти или повышенную активность GC. Это происходит, когда нужно прочитать и обработать большие документы.

    byte tmpData[] = new byte[1024];



int offs = 0;



do



{



int readLen = bis.read(tmpData, offs, tmpData.length - offs);



if(readLen == -1)



break;



offs += readLen;



if(offs == tmpData.length) {



byte newres[] = new byte[tmpData.length + 1024];



System.arraycopy(tmpData, 0, newres, 0,tmpData.length);



tmpData = newres;



}



} while(true);

    Опытному разработчику очевидно, что обработка нескольких мегабайт таким кодом приводит к плохой производительности из-за множества лишних выделений и постоянно растущих операций копирования. Часто такая проблема не замечается во время тестирования и проявляется только при определённом уровне параллельности, когда количество GC и объём временной памяти становятся проблемой.

    Поэтому при работе с большими документами важно оптимизировать логику обработки и не допускать, чтобы документ целиком держался в памяти.

    Проблемы памяти, связанные с ClassLoader¶

    В современных корпоративных приложениях требования к памяти для загруженных классов могут быстро достигать нескольких сотен МБ и вносить вклад в проблемы с памятью. В Hotspot JVM классы располагаются в permanent generation (PermGen). Это отдельная область памяти, размер которой нужно настраивать отдельно. Если эта область заполнена, новые классы загрузить нельзя и возникает нехватка памяти в PermGen. У других JVM нет permanent generation, но это не решает проблему. Проблемы с загрузчиком классов относятся к одним из самых сложных для обнаружения. Большинство разработчиков никогда с ними не сталкиваются, и поддержка инструментов в этой области тоже слабее всего. Ниже перечислены некоторые из таких проблем:

    Крупные классы

    Важно не увеличивать размер классов без необходимости. Особенно это касается ситуаций, когда классы содержат много строковых констант, например в GUI-приложениях. Там все строки хранятся в константах. Это хороший подход к проектированию, но не стоит забывать, что эти константы тоже требуют места в памяти. В случае Hotspot JVM строковые константы являются частью PermGen, которая после этого может быстро оказаться слишком маленькой.

    Несколько экземпляров одного класса в памяти

    У серверов приложений и контейнеров OSGi бывают проблемы из-за слишком большого числа загруженных классов и связанного с этим потребления памяти. Серверы приложений позволяют загружать разные приложения или их части изолированно друг от друга. Одна из возможностей состоит в том, что несколько версий одного класса могут быть загружены для разных приложений внутри одной JVM. Из-за неверной конфигурации это может быстро удвоить или утроить объём памяти, нужный для классов.

    Такие проблемы лучше всего диагностируются heap dump или трендовым дампом (jmap -histo). Если класс загружен несколько раз, его экземпляры тоже считаются несколько раз. Если один и тот же класс встречается несколько раз с разным числом экземпляров, проблема выявлена. Ответственный загрузчик классов можно определить в heap dump простой трассировкой ссылок. Переменные загрузчика классов раскрывают ссылку на модуль приложения и файл .jar. Это позволяет понять, не загружается ли один и тот же файл .jar несколько раз разными модулями приложения.

    Один и тот же класс загружается повторно

    При повторной загрузке одного и того же класса он, на первый взгляд, не присутствует в памяти дважды. Многие забывают, что классы тоже собираются сборщиком мусора во всех трёх крупных JVM. В Hotspot JVM это происходит только во время major GC, тогда как и IBM, и JRockit могут делать это при каждом GC. Поэтому, если класс используется лишь недолго, он может быть тут же удалён из памяти. Загрузка класса не быстрая операция и обычно не оптимизирована под конкурентность. Если один и тот же класс загружают несколько потоков, Java их синхронизирует.

    В случае Hotspot JVM эта проблема проявится только под нагрузкой и при дефиците памяти, поскольку требует major GC, тогда как в IBM JVM или JRockit это возможно уже при умеренной нагрузке. Класс может даже не пережить первой сборки мусора.
