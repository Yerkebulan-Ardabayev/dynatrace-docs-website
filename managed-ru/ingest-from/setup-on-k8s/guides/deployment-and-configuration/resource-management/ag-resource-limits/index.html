    Руководство по подбору размеров Dynatrace ActiveGate для сценария мониторинга Kubernetes¶

    Руководство по подбору размеров Dynatrace ActiveGate для сценария мониторинга Kubernetes¶

    • Справочный материал
    • Чтение 5 мин
    • Обновлено 15 июня 2026 г.

    Правильно заданные запросы на ресурсы (и лимиты, если нужно) обеспечивают стабильность и предсказуемость экземпляров Dynatrace ActiveGate. В этом руководстве описаны методы подбора размеров в зависимости от масштаба и нагрузки.

    Стабильный, здоровый ActiveGate обеспечивает непрерывное поступление данных мониторинга без пропусков.

    Понимание факторов, влияющих на размер¶

    Фактически необходимые ресурсы растут вместе со следующими параметрами:

    • Количество подов, основной фактор подбора размера, это количество отслеживаемых подов. Потребление ресурсов (CPU и памяти) компонентами Dynatrace ActiveGate масштабируется с ростом числа подов, в первую очередь из-за увеличения объёма обработки данных и потребностей в хранении. По мере роста числа отслеживаемых подов ActiveGate обрабатывает больше данных сущностей, событий и метрик, что приводит к росту нагрузки на CPU при приёме и обработке данных, а также к увеличению потребления памяти на кэширование информации о подах. Это основной фактор подбора размера, потребление растёт пропорционально количеству подов.
    • Объём метрик Prometheus, количество подов с аннотациями Prometheus напрямую влияет на рост требований к ресурсам Dynatrace ActiveGate, прежде всего через увеличение потребления CPU. По мере роста числа аннотированных подов увеличивается объём собираемых метрик, что требует больше циклов CPU на сбор, агрегацию и пересылку данных. Влияние на память вторично, поскольку метрики пересылаются в тенант Dynatrace без долговременного хранения на ActiveGate, хотя оно тоже растёт пропорционально пиковой скорости приёма данных.
    • Количество узлов, потребление ресурсов (CPU и памяти) компонентами Dynatrace ActiveGate масштабируется с ростом числа узлов, в первую очередь из-за увеличения накладных расходов на мониторинг и нагрузки от системных подов уровня узла. По мере роста числа узлов ActiveGate приходится обрабатывать больше данных сбора на системном уровне, обработки сущностей и приёма событий, что приводит к росту вычислительных требований. Это второстепенный фактор по сравнению с количеством подов, но он пропорционально влияет на общую потребность в ресурсах, особенно в крупных кластерах, где мониторинг на уровне узлов создаёт накопительную нагрузку.
    • Количество отслеживаемых кластеров, один ActiveGate может отслеживать несколько кластеров, однако такая схема развёртывания не рекомендуется. Мониторинг нескольких кластеров одним ActiveGate пропорционально увеличивает требования к CPU и памяти за каждый дополнительный кластер, поскольку ActiveGate приходится обрабатывать больше данных, управлять большим числом сущностей и принимать больше событий по всем кластерам. Это может привести к конкуренции за ресурсы, повышенному риску троттлинга или перезапусков из-за нехватки памяти (OOM), а также к возможным пропускам в данных мониторинга.

    Рекомендуется схема, при которой один контейнеризованный ActiveGate отслеживает один кластер Kubernetes.

    Это обеспечивает оптимальную производительность и надёжность. Такой подход упрощает устранение неполадок, исключает конкуренцию за ресурсы и обеспечивает предсказуемое масштабирование, поскольку каждый ActiveGate обрабатывает нагрузку только одного кластера.

    Признаки нездорового ActiveGate¶

    Эти симптомы указывают на исчерпание ресурсов и возможную потерю данных.

    Пропуски в данных мониторинга¶

    ActiveGate собирает разные типы данных независимо друг от друга (например, метрики Prometheus, события Kubernetes, сущности). Если сбор одного типа данных занимает более 1 минуты, пропуск возникает только для этого типа данных в данном временном окне. Остальные задачи сбора продолжают работать в штатном режиме.

    • В метриках будет отсутствовать точка данных за соответствующую минуту.
    • События за соответствующие интервалы сбора вообще недоступны.
    • Сущности могут не отражать последние обновления или вовсе отсутствовать, если они недолговечны.

    Сильный троттлинг CPU¶

    Устойчиво высокий троттлинг означает нехватку CPU. Сильный троттлинг может приводить к пропускам данных. Незначительный троттлинг обычно безвреден. Если троттлинг затрагивает под, обслуживающий мониторинг ActiveGate, это может вызвать пропуски данных.

    Перезапуски из-за нехватки памяти (OOM)¶

    Если ActiveGate принудительно завершён из-за нехватки памяти (OOM-kill), данные становятся недоступны до его перезапуска. После перезапуска повторные OOM-завершения весьма вероятны.

    Мониторинг и проверка¶

    Для оценки работоспособности ActiveGate можно отслеживать следующие показатели.

    • Использование CPU относительно запросов, если использование CPU стабильно превышает 85%, нужно увеличить запрос на CPU.
    • Троттлинг CPU (container_cpu_cfs_throttled_periods_total / periods), если троттлинг стабильно превышает 10%, нужно увеличить запрос на CPU.
    • Рабочий набор памяти относительно запросов, если использование стабильно превышает 80%, нужно увеличить запрос на память.
    • Количество перезапусков ActiveGate, после перезапуска из-за OOM нужно незамедлительно повысить настроенный объём памяти, чтобы избежать повторения.
    • Длительность обработки / время цикла, если время выполнения конвейеров стабильно превышает 50–60 секунд, нужно увеличить запрос на CPU. Время выполнения конвейера также зависит от объёма принимаемых данных и других факторов.
    • Время сборки мусора, рост времени сборки мусора, это явный признак недостаточного выделения ресурсов ActiveGate.

    Увеличение ресурсов¶

    Начать нужно с рекомендуемых значений ниже.

    1. Проверить работоспособность ActiveGate с помощью показателей мониторинга.

    2. При необходимости скорректировать ресурсы:

    3. увеличивать память шагами по 2Gi;

    4. увеличивать CPU шагами по 500m.

    5. Сначала корректируется запрос, и только на следующем шаге, лимиты.

    6. Что касается CPU: лимиты использовать только если это требуется политикой.

    Использование выделенного ActiveGate для мониторинга платформы Kubernetes¶

    Рекомендуется разделить обязанности ActiveGate на две группы: одна группа отвечает за всё, что связано с мониторингом платформы Kubernetes, включая KSPM, а другая, за маршрутизацию трафика Agent, приём телеметрии и расширения. Такое разделение даёт несколько преимуществ:

    • Изоляция, конкуренция за ресурсы в одном модуле не влияет на другой. Всплеск трафика OneAgent не замедлит сбор метрик Kubernetes, и наоборот.

    • Независимое масштабирование, пересылка трафика и мониторинг платформы имеют принципиально разные характеристики масштабирования:

    • Мониторинг платформы Kubernetes может масштабироваться только вертикально, при росте размера кластера нужно увеличивать ресурсы CPU и памяти для соответствующего ActiveGate.

    • Маршрутизация трафика OneAgent может масштабироваться горизонтально, при росте трафика OneAgent дополнительные реплики маршрутизирующего ActiveGate помогают распределить и разделить нагрузку.

    Раздельные ActiveGate позволяют масштабировать каждое измерение независимо, не выделяя избыточные ресурсы. Например, можно добавить ещё 3 реплики маршрутизации для обработки возросшего трафика приложений, не увеличивая при этом без необходимости ресурсы для мониторинга платформы. * Упрощённое устранение неполадок, при возникновении проблем можно сразу определить, связаны ли они с мониторингом платформы или с трафиком OneAgent, что сокращает время диагностики.

    Подбор размера ActiveGate¶

    Сценарии определяются на основе количества подов, которые отслеживает один ActiveGate. Необходимые ресурсы можно взять из следующих таблиц.

    • Сценарий S: менее 1000 подов
    • Сценарий M: 1000–5000 подов
    • Сценарий L: 5000–20000 подов

    Данное руководство не охватывает среды с более чем 20 000 подов. Из-за большого числа переменных дать точные рекомендации для таких масштабных конфигураций невозможно. В качестве отправной точки можно использовать рекомендации для сценария L и постепенно увеличивать ресурсы, пока не установится стабильный мониторинг без пропусков. За индивидуальными рекомендациями советуем обратиться в службу поддержки Dynatrace, чтобы обеспечить оптимальную конфигурацию для вашей среды.

    Второстепенные факторы¶

    Количество узлов¶

    По мере роста числа узлов в Kubernetes кластере ActiveGate требует больше ресурсов CPU и памяти для обработки дополнительной нагрузки мониторинга. Сюда входит обработка данных от системных компонентов и событий на уровне узлов, объём которых растёт пропорционально. Для рекомендаций по размеру ресурсов количество узлов группируется в следующие категории:

    • До 25 узлов
    • До 100 узлов
    • До 500 узлов

    Если в кластере больше 100 узлов, нужно увеличить выделение ресурсов, чтобы учесть дополнительную нагрузку и обеспечить стабильный мониторинг без пробелов. Для кластеров с числом узлов более 500 нужно обратиться в Dynatrace Support за индивидуальными рекомендациями.

    Объём собираемых метрик Prometheus¶

    Dynatrace поддерживает до 1000 pod-экспортеров, каждый из которых может предоставлять до 1000 метрик. Если среда приближается к этим лимитам, нужно увеличить ресурсы, выделенные ActiveGate, для обеспечения надёжной работы.

    Рекомендация для больших масштабов

    Для сбора большого объёма данных Prometheus, а также для новых развёртываний рекомендуется использовать OpenTelemetry Collector. Он масштабируется горизонтально на Kubernetes за счёт развёртывания на основе Target Allocator и передаёт метрики Prometheus в Dynatrace по протоколу OTLP.

    ActiveGate для мониторинга платформы Kubernetes¶

    Все перечисленные ниже сценарии использования предъявляют одинаковые требования к ActiveGate, который берёт на себя часть мониторинга платформы Kubernetes.

    • Только мониторинг платформы Kubernetes
    • Мониторинг платформы Kubernetes + Application observability
    • Мониторинг платформы Kubernetes + Full-stack observability
    Количество pod'ов Ресурс CPU Ресурс памяти
    < 1000 (Small) requests: 200m(limits: 1000m) requests: 6Gi limits: 6Gi
    < 5000 (Medium) requests: 1000m(limits: 2000m) requests: 10Gi limits: 10Gi
    < 20000 (Large) requests: 2000m(limits: 4000m) requests: 12Gi limits: 12Gi

    Рекомендуется запускать ActiveGateы без лимитов CPU.

    ActiveGate для маршрутизации и проксирования трафика OneAgent¶

    Второй ActiveGate не участвует активно в мониторинге платформы Kubernetes, а используется как маршрутизатор/прокси для потоков данных, исходящих от OneAgent. Рекомендуется использовать отдельные ActiveGateы для пересылки трафика и мониторинга платформы. Пересылка трафика масштабируется горизонтально путём добавления реплик. Сам мониторинг Kubernetes горизонтально не масштабируется, вместо этого нужно увеличивать ресурсы.

    Такая настройка требуется в следующих сценариях использования:

    • Мониторинг платформы Kubernetes + Application observability
    • Мониторинг платформы Kubernetes + Full-stack observability
    Количество pod'ов Ресурс CPU Ресурс памяти реплики
    < 1000 (Small) requests: 250m(limits: 1000m) requests: 2Gi limits: 2Gi 3
    < 5000 (Medium) requests: 500m(limits: 2000m) requests: 4Gi limits: 4Gi 3
    < 20000 (Large) requests: 1000m(limits: 4000m) requests: 6Gi limits: 6Gi 6

    Рекомендуется запускать ActiveGateы без лимитов CPU.

    Начало работы¶

    Следующие примеры применяют принцип разделения ActiveGateов на модули с раздельными зонами ответственности. Один ActiveGate отвечает за мониторинг платформы Kubernetes и за управление уровнем защищённости Kubernetes (security posture management), тогда как второй ActiveGate отвечает за маршрутизацию трафика агентов, приём телеметрии и расширения. Нужно скорректировать requests (и, при необходимости, limits) под свою среду.

    Следующие манифесты включают два ресурса DynaKube для настройки ActiveGateов:

    • k8s-monitoring, настраивает ActiveGate, выделенный для мониторинга платформы Kubernetes.
    • agents, настраивает ActiveGate, который поддерживает OneAgent, приём телеметрии и дополнительные функции.

    Можно применить оба манифеста сразу или применить только нужный DynaKube.

    Лимиты CPU закомментированы. Рекомендуется задавать только requests, чтобы ActiveGate мог использовать дополнительный CPU при его наличии. Если лимиты необходимы, их нужно задавать равными или выше значений requests.

    Этот фрагмент включает конфигурацию для Kubernetes Security Posture Management. Это дополнительная, опциональная функция безопасности наряду с мониторингом платформы Kubernetes. Этот фрагмент также включает конфигурации для мониторинга логов, расширений и приёма телеметрии. Эти разделы считаются опциональными, каждый по отдельности.
