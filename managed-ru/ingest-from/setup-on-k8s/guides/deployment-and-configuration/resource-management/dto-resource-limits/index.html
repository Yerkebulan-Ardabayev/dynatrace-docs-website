    Настройка ограничений ресурсов для компонентов Dynatrace Operator¶

    Настройка ограничений ресурсов для компонентов Dynatrace Operator¶

    • 2 минуты на чтение
    • Обновлено 27 февраля 2026 г.

    Правильно настроенные ограничения ресурсов обеспечивают оптимальную производительность и стабильность компонентов Dynatrace Operator и предотвращают конкуренцию за ресурсы в кластере Kubernetes. Это руководство помогает понять, как задать подходящие ограничения ресурсов в зависимости от размера окружения и характера нагрузки.

    Базовый уровень ограничений ресурсов по умолчанию¶

    Предоставленные ограничения ресурсов по умолчанию проверены нагрузочным тестированием. Эти значения по умолчанию хорошо показали себя в следующем окружении:

    • 25 узлов (тип узла e2-standard-32 на Google Kubernetes Engine)
    • 20 DynaKube
    • 2500 пространств имён
    • 5000 подов

    Факторы потребления ресурсов¶

    Следующие пять ключевых показателей влияют на потребление ресурсов разными компонентами Dynatrace Operator:

    Показатель Dynatrace Operator Webhook CSI driver
    Пространства имён Применимо Применимо
    Узлы Применимо
    DynaKube Применимо Применимо
    Поды Применимо Применимо
    Количество версий OneAgent Применимо

    Понимание показателей влияния¶

    • Пространства имён: чем больше пространств имён, тем выше нагрузка на Operator и webhook, так как им нужно отслеживать и управлять ресурсами во всех пространствах имён.

    • Влияние:

      • Увеличивает использование CPU/памяти Operator
      • Увеличивает использование CPU webhook
      • Узлы: дополнительные узлы требуют больше ресурсов, так как Operator хранит список всех доступных узлов в кластерах Kubernetes и проверяет их соответствие доступным хостам на сервере Dynatrace.

    • Влияние:

      • Увеличивает использование CPU/памяти Operator
      • DynaKube: каждый ресурс DynaKube представляет собой отдельное развёртывание Dynatrace, которое нужно управлять индивидуально.

    • Влияние:

      • Увеличивает использование CPU/памяти Operator
      • Увеличивает использование CPU/памяти webhook
      • Увеличивает использование CPU/памяти provisioner в CSI driver
      • Поды: webhook обрабатывает запросы допуска (admission requests) для каждого пода, а CSI driver отвечает за монтирование томов для подов, использующих OneAgent.

    • Влияние:

      • Увеличивает использование CPU/памяти server/liveness-probe/registrar в CSI driver
      • Версии OneAgent: CSI driver должен управлять разными версиями OneAgent и предоставлять к ним доступ, что требует дополнительных ресурсов хранения и обработки.

    • Влияние:

      • Увеличивает использование CPU/памяти provisioner в CSI driver

    Минимизация влияния большого количества узлов¶

    По умолчанию Dynatrace Operator отслеживает доступность хостов в кластере, чтобы обнаруживать ожидаемое удаление подов хоста OneAgent, особенно в сценариях масштабирования. В бессерверных окружениях такой мониторинг не нужен.

    Когда отключать обнаружение доступности хостов¶

    Эта функциональность присутствует во всех версиях Operator до 1.6.0, и её нельзя отключить.

    Её можно отключить в версиях 1.6.3 и 1.7.3 или в более новых версиях Operator.

    applicationMonitoring

    Можно снизить потребление ресурсов Operator, отключив обнаружение доступности хостов, если DynaKube:

    • используют только режимы мониторинга уровня приложения и
    • не используют никакие функции мониторинга на уровне хоста.

    Как отключить обнаружение доступности хостов¶

    Чтобы отключить обнаружение доступности хостов, задайте следующее значение Helm:

    operator:



hostAvailabilityDetection: false

    Эта оптимизация может снизить использование CPU и памяти Operator, особенно в крупных кластерах с большим количеством узлов.

    Настройка на уровне кластера: operator.hostAvailabilityDetection влияет на все DynaKube, управляемые Operator. Отключать это нужно только если есть уверенность, что ни одному из DynaKube не требуется мониторинг на уровне хоста. Отключение при необходимости хостов OneAgent может привести к ложным предупреждениям об отсутствии хоста во время масштабирования узлов или других операций, связанных с узлами.

    Настройка ограничений ресурсов¶

    Хотя ограничения ресурсов по умолчанию должны быть достаточными для большинства случаев использования, их можно настроить в соответствии с конкретными потребностями.

    Компоненты provisioner и job CSI driver не имеют заданных по умолчанию ограничений ресурсов. Это позволяет им использовать дополнительные ресурсы, когда они доступны, повышая производительность.

    Компонент job используется только с node-image-pull.

    Обратите внимание: если ограничения заданы слишком низкими, это может привести к увеличению времени запуска подов из-за CPU throttling или к OOM kill в CSI driver из-за нехватки памяти, что может помешать запуску приложений.

    Масштабирование ограничений ресурсов для разных окружений¶

    Ограничения ресурсов по умолчанию (requests и limits) рассчитаны на окружения среднего масштаба. Используйте следующие рекомендации, чтобы скорректировать ограничения в зависимости от размера окружения. Это стартовые рекомендации: всегда отслеживайте фактическое использование ресурсов в своём окружении и корректируйте их соответственно.

    Классы качества обслуживания подов (Pod Quality of Service Classes): у некоторых компонентов limits и requests заданы равными, чтобы обеспечить гарантированный класс качества обслуживания пода (Guaranteed Pod Quality of Service)﻿. При масштабировании ограничений таких компонентов всегда пропорционально масштабируйте и requests.

    Пропорциональная справедливость: низкие запросы CPU (CPU requests) у контейнера могут приводить к throttling из-за политики управления CPU на узле. Requests служат минимальной гарантией. Поэтому на сильно загруженном узле контейнеры с меньшими requests будут ограничиваться сильнее по сравнению с контейнерами с большими requests, независимо от их limit. При масштабировании ограничений всегда учитывайте и масштабирование requests.

    Запросы и ограничения ресурсов для job CSI driver не нужно масштабировать в зависимости от размера окружения. Они не зависят от количества узлов, подов и DynaKube. Однако можно скорректировать запрос CPU, чтобы контролировать, как быстро завершается job, что определяет, насколько скоро CSI driver будет готов монтировать тома:

    Запрос CPU Приблизительное время завершения
    100m ~1 мин
    200m (по умолчанию) ~30 сек
    300m ~25 сек

    Это влияет только на продолжительность состояния Running job. Это не влияет на время ContainerCreation или PodScheduling.

    Начиная с версии 1.9.0 Dynatrace Operator, помимо ограничений ресурсов можно настраивать replicas webhook в Helm. Это позволяет горизонтально масштабировать Webhook в крупных окружениях, обеспечивая более высокую производительность и доступность.

    Приведённая ниже рекомендация по ресурсам рассчитана на две реплики webhook. Если используется другое количество реплик, скорректируйте ограничения ресурсов соответственно.

    • Пример: если нужно использовать 3 реплики вместо 2, можно снизить запросы/ограничения CPU/памяти примерно на 30%, сохраняя общую производительность, но получая более высокую доступность.

    Подробнее о настройке replicas webhook и связанных значений см. в разделе Настройка высокой доступности.

    Крупные окружения (> 50 узлов, > 10 000 подов)¶

    Увеличьте requests/limits по умолчанию на 50–100%:

    • Dynatrace Operator:

    • Requests: CPU 100m, память 128Mi

    • Limits: CPU 200m, память 256Mi

    • Webhook:

    • Requests: CPU 600m, память 256Mi

    • Limits: CPU 600m, память 256Mi

    • CSI driver provisioner:

    • Requests: CPU 600m, память 200Mi

    • CSI driver server:

    • Requests: CPU 100m, память 200Mi

    • Limits: CPU 100m, память 200Mi

    • CSI driver liveness-probe:

    • Requests: CPU 30m, память 50Mi

    • Limits: CPU 30m, память 50Mi

    • CSI driver registrar:

    • Requests: CPU 30m, память 50Mi

    • Limits: CPU 30m, память 50Mi

    Окружения корпоративного уровня (> 100 узлов, > 25 000 подов)¶

    Увеличьте requests/limits по умолчанию на 100–200%:

    • Dynatrace Operator: CPU 400m, память 512Mi

    • Requests: CPU 200m, память 256Mi

    • Limits: CPU 400m, память 512Mi

    • Webhook:

    • Requests: CPU 1000m, память 512Mi

    • Limits: CPU 1000m, память 512Mi

    • CSI driver provisioner:

    • Requests: CPU 900m, память 300Mi

    • CSI driver server:

    • Requests: CPU 150m, память 300Mi

    • Limits: CPU 150m, память 300Mi

    • CSI driver liveness-probe:

    • Requests: CPU 50m, память 70Mi

    • Limits: CPU 50m, память 70Mi

    • CSI driver registrar:

    • Requests: CPU 50m, память 70Mi

    • Limits: CPU 50m, память 70Mi
