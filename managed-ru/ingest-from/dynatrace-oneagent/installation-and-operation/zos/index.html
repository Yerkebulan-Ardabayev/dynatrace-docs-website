    Dynatrace для z/OS¶

    Dynatrace для z/OS¶

    • Чтение: 8 мин
    • Обновлено 15 ноября 2022 г.

    С Dynatrace вы получаете полные транзакционные инсайты по своим нагрузкам от мобильного фронтенда до программ на мейнфрейме и всего, что находится между ними, чтобы устранять аномалии на уровне кода. Кроме того, Dynatrace может сопровождать вас на пути к гибридному облаку благодаря сквозной наблюдаемости от мейнфрейма до облака.

    Узнайте, как Dynatrace решает наиболее типичные задачи на мейнфрейме:

    Является ли мейнфрейм частью проблемы?

    Изоляция домена отказов на основе ИИ точно определяет первопричину проблем и оценивает их влияние на пользователей, чтобы вы могли расставлять приоритеты в стратегиях устранения и сокращать среднее время восстановления.

    Сценарий использования z/OS

    Все мониторируемые LPAR, регионы и приложения участвуют в этой изоляции домена отказов.

    Сценарий использования z/OS

    Кто и как часто обращается к мейнфрейму?

    Прослеживайте транзакции через Service backtrace, чтобы понимать нагрузки на мейнфрейм и пользоваться возможными скидками IBM (см., например, скидки IBM на mobile и public cloud нагрузки, чтобы снизить ежемесячное пиковое значение MSU по скользящему 4-часовому среднему).

    Service backtrace ниже показывает, как транзакция CICS взаимодействует с мобильным и веб-приложением. Чётко видно, как часто эти приложения вызывают транзакцию CICS, а также какие из их запросов завершились с ошибкой.

    Сценарий использования z/OS

    Какие транзакции дорогие или медленные?

    Анализируйте производительность своих транзакций через service flow, чтобы проверить, выполняют ли они заданные SLO с метриками уровня сервиса. Например, количество запросов может указать, что транзакция вызывается слишком часто из открытой системы, что может привести к дополнительным затратам.

    Сценарий использования z/OS

    Используйте инсайты на уровне кода через распределённые трассировки PurePath, чтобы оптимизировать свои программы.

    Сценарий использования z/OS

    Модернизировать программы на z/OS для гибридного облака?

    Используйте Dynatrace, чтобы обеспечить непрерывность бизнеса для традиционных окружений через мониторинг таких технологий middleware, как сервисные шины предприятия или очереди сообщений. Одновременно трансформируйте свои программы на мейнфрейме, делая их доступными для облачных функций через z/OS Connect EE под мониторингом Dynatrace.

    Настройка мониторинга¶

    Dynatrace предоставляет кодовые модули для технологий CICS, IMS и z/OS Java, чтобы вы могли достичь бесшовной наблюдаемости с инсайтами по трассировкам и метрикам. Чтобы узнать больше о поддерживаемых технологиях, см. Поддержка технологий мейнфрейма.

    Архитектура мониторинга z/OS

    Модули CICS, IMS и z/OS Java взаимодействуют с подсистемой Dynatrace z/OS Data Collection (zDC) через объект общей памяти (SMO) внутри LPAR. Подсистема zDC управляет этим SMO, в который модули записывают данные мониторинга.

    zLocal, размещённый в окружении z/OS Unix System Services (USS), работает как часть zDC. Он управляет TCP/IP-соединением с модулем zRemote, читает данные мониторинга из SMO и передаёт их в zRemote.

    Модуль zRemote обрабатывает данные мониторинга, полученные от zLocal, и направляет эти данные, сжатые и зашифрованные, через локальный ActiveGate в Dynatrace. Таким образом, модуль zRemote снимает с модулей значительную часть работы по обработке, возникающей при инструментировании подсистем и приложений, перенося её на открытую систему.

    Чтобы начать работу, см. Обзор установки z/OS.

    Лицензирование¶

    Использование модулей CICS, IMS и z/OS Java тарифицируется по миллионам единиц обслуживания (MSU).

    Dynatrace Platform Subscription, см. Mainframe Monitoring.

    Классическое лицензирование Dynatrace, см. Mainframe Monitoring на IBM z/OS.

    Часто задаваемые вопросы¶

    Кто должен участвовать в типичной установке Dynatrace для z/OS?

    Чтобы узнать процедуру и состав участников, см. Обзор установки z/OS.

    Можно ли использовать группы хостов для организации нескольких LPAR?

    Да, можно организовать несколько LPAR через группы хостов. Подробнее см. Определение групп хостов для организации нескольких LPAR.

    Что означает volatile-сервис и как решать связанные с ним проблемы?

    Volatile CICS или volatile IMS создаётся Dynatrace автоматически, когда превышается максимальное число идентификаторов сервисов, которое можно сгенерировать в регионе (процессе). Чтобы увеличить лимит генерируемых идентификаторов сервисов, обратитесь к продуктовому эксперту Dynatrace через live chat в вашем окружении Dynatrace.

    Модули CICS и IMS¶

    Как работает инструментирование модулей CICS и IMS?

    Модуль CICS использует хуки для инструментирования терминальных и прикладных регионов CICS, создавая события интереса.

    Модуль IMS использует средства логирования для инструментирования управляющих и обрабатывающих сообщения регионов IMS, создавая события интереса из разбора бинарных логов.

    Оба модуля используют хуки для инструментирования IBM MQ, Db2 и DL/I.

    Инструментирование байт-кода модулями CICS и IMS не применяется.

    Сколько времени GCP потребляют модули CICS и IMS при инструментировании приложений?¶

    Модули CICS и IMS потребляют некоторое количество времени центрального процессора общего назначения (GCP) при инструментировании приложений на IBM Z, но эти накладные расходы, как правило, очень низкие (порядка 1%-2%, в зависимости от типа мониторируемых транзакций). См. примеры ниже.

    • Значения накладных расходов GCP вычислены относительно адресных пространств, в которых работают модули. Если же сравнивать накладные расходы GCP относительно LPAR, эти значения будут ещё ниже.
    • Например, 2% накладных расходов GCP в адресных пространствах CICS соответствуют только 1% накладных расходов GCP по LPAR, если нагрузки CICS потребляют только 50% общего времени GCP на этом LPAR по сравнению с другими нагрузками (такими как задания и системные задачи).

    Могут ли модули CICS и IMS захватывать динамические SQL-операторы?

    Нет, модули CICS и IMS могут захватывать только статические SQL-операторы.

    Модуль z/OS Java¶

    Можно ли мониторить Servants моего WebSphere Application Server модулем z/OS Java?

    WebSphere Application Server на z/OS позволяет динамически разворачивать Servants в зависимости от нагрузки.

    В Dynatrace для различения разных Servants можно использовать process identifier на каждой метрике, как показано на изображении ниже.

    Идентификатор процесса в метриках

    Однако Servants нельзя включить в обнаружение групп процессов WebSphere Application Server, потому что при разворачивании нового Servant создаётся новая сущность процесса и текущий контекст мониторинга теряется.

    Какие атрибуты учитываются при обнаружении групп процессов WebSphere Application Server?

    Dynatrace использует следующие атрибуты для обнаружения и создания сущностей процессов WebSphere Application Server:

    • Имя сервера
    • Имя узла
    • Имя ячейки

    Dynatrace группирует все сущности процессов, принадлежащие одному кластеру WebSphere Application Server, в одну группу процессов.

    Можно ли объединить группы процессов, созданные модулем z/OS Java, в одну группу процессов?

    Нет, группы процессов, созданные модулем z/OS Java, нельзя изменять или объединять.

    В качестве альтернативы можно организовать группы процессов через определение метаданных или определение тегов на основе переменных окружения. Оба подхода применимы также к z/OS Java. Учтите, что переменные окружения можно определить только на уровне процесса, а не на уровне хоста.

    Можно ли тегировать процессы, созданные модулем z/OS Java?

    Да. Процессы, созданные модулем z/OS Java, можно тегировать через определение метаданных или определение тегов на основе переменных окружения. Учтите, что переменные окружения можно определить только на уровне процесса, а не на уровне хоста.

    Можно ли определять пользовательские сервисы через модуль z/OS Java?

    Модуль z/OS Java не поддерживает пользовательские сервисы исключительно через конфигурацию. Вместо этого можно создавать пользовательские трассировки через интероперабельность с OpenTelemetry модуля z/OS Java.

    Можно ли использовать атрибуты спанов, захваченные модулем z/OS Java, как атрибут запроса?

    Да. Чтобы узнать, как настроить атрибут запроса для любого захваченного атрибута спана, см. Определение атрибута запроса для атрибутов спанов.

    Linux на Z¶

    С Dynatrace вы получаете Full-Stack Monitoring с Host monitoring (DPS) для Linux на Z через OneAgent на Linux. Чтобы узнать больше о поддерживаемых технологиях на архитектуре s390, см. Поддержка технологий.
