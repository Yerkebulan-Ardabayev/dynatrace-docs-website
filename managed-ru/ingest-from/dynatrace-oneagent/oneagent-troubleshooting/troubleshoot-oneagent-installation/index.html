    Устранение проблем установки OneAgent¶

    Устранение проблем установки OneAgent¶

    • Troubleshooting
    • 13-min read
    • Updated on Oct 17, 2025

    Узнайте, как устранять проблемы установки OneAgent на AIX, Linux и Windows.

    Общее устранение неполадок¶

    Почему OneAgent не начинает мониторинг процесса Apache после перезапуска?

    После установки OneAgent веб-сервер Apache необходимо полностью перезапустить для включения мониторинга. Для правильного выполнения этого важно понять разницу между «частичным» и «полным» перезапусками. При частичном перезапуске главный процесс Apache повторно читает конфигурационные файлы, повторно открывает файлы лога и затем перезапускает рабочие процессы. OneAgent, однако, требует полного перезапуска веб-сервера Apache, при котором все рабочие процессы и, что наиболее важно, главный процесс Apache полностью останавливаются, а затем перезапускаются.

    Дополнительную информацию о различных типах доступных перезапусков смотрите в разделе Stopping and Restarting Apache HTTP Server.

    Как выполнить полный перезапуск¶

    Linux и AIX

    Возможно, вы привыкли перезапускать Apache командой apachectl restart. Однако эта команда приводит лишь к частичному перезапуску Apache.

    Для полного перезапуска Apache и включения глубокого мониторинга с Dynatrace OneAgent необходимо сначала выполнить полную остановку командой apachectl stop. Только после этого шага можно перезапустить сервер командой apachectl start.

    На системах Ubuntu допустимо использовать service apache2 restart. Обратите внимание, что какие бы команды вы ни использовали, вам, скорее всего, потребуются права суперпользователя (sudo).

    Windows

    На Windows вы можете использовать встроенное управление службами Windows или Apache Service Monitor (httpd.exe) для перезапуска служб Apache. Перезапуск службы Apache с помощью управления службами Windows гарантирует полный перезапуск. С помощью httpd.exe вы, возможно, привыкли перезапускать Apache командой httpd.exe -k restart -n "Apache2.4". Однако эта команда приводит лишь к частичному перезапуску Apache.

    Для полного перезапуска Apache и включения глубокого мониторинга с OneAgent необходимо сначала выполнить полную остановку командой httpd.exe -k stop -n "Apache2.4". Только после этого шага можно перезапустить сервер командой httpd.exe -k start -n "Apache2.4".

    Что делать, если OneAgent блокирует нужный порт?

    Устарело

    Начиная с версии OneAgent 1.301, OneAgent не использует TCP-порты для собственной межпроцессной коммуникации. Если OneAgent занимает порты ваших приложений, обновите OneAgent до версии 1.301+.

    OneAgent состоит из различных процессов, которые взаимодействуют с watchdog через TCP-порт. При запуске watchdog OneAgent пытается открыть первый доступный порт в диапазоне от 50000 до 50100. В некоторых случаях этот порт может понадобиться для приложений, запускаемых после OneAgent. В таких случаях вы можете изменить диапазон портов, используемый watchdog OneAgent, с помощью интерфейса командной строки OneAgent.

    Порт прослушивания watchdog можно изменить с помощью set-watchdog-portrange через инструмент командной строки oneagentctl:

    Например, для изменения диапазона портов на 50005:50105 перейдите в директорию oneagentctl и выполните следующую команду:

    • На Linux или AIX:
      ./oneagentctl --set-watchdog-portrange 50005:50105
    • На Windows:
      .\oneagentctl.exe --set-watchdog-portrange 50005:50105
      Перезапустите службу OneAgent для применения изменений.

    Информацию о портах, используемых Dynatrace, смотрите в разделе Какие сетевые порты использует Dynatrace Server?.

    Сбой проверки сертификата сервера

    OneAgent поставляется с доверенными SSL-сертификатами Dynatrace, которые используются для проверки успешного подключения OneAgent к серверу Dynatrace или ActiveGate.

    Если в вашем окружении используется прокси (что требует обновления SSL-сертификата удалённого сервера) или у вас есть Environment ActiveGate с собственным пользовательским сертификатом, во время начальной проверки подключения может появиться сообщение Server certificate check failed.

    Для устранения этой проблемы смотрите раздел Безопасность OneAgent.

    Процессы не обнаруживаются?

    Могло произойти одно из следующего:

    • Процесс не поддерживается технологией мониторинга. Вы всегда можете проверить, какие типы процессов поддерживает Dynatrace.
    • Процесс не работает на сервере. Убедитесь, что серверы запущены и процессы функционируют.
    • Задержка в коммуникации между Dynatrace и OneAgent. В таком случае подождите несколько минут и повторите попытку.
    • OneAgent не работает должным образом. Перейдите в Settings > Monitoring > Monitoring overview, чтобы убедиться, что мониторинг включён для хоста, на котором работает ваше ПО.

    Если вам по-прежнему не удаётся решить проблему, обратитесь к специалисту Dynatrace через онлайн-чат в вашем окружении. Также рассмотрите вариант установки OneAgent на другой машине.

    Проблемы с OneAgent начались после значительного обновления ОС хоста.

    Мы не поддерживаем существенные изменения операционной системы на хосте, где установлен OneAgent.

    Изменения ОС, влияющие на OneAgent:

    • Патч ядра системы
    • Крупное обновление ОС
    • Любое другое изменение конфигурации системы, приводящее к значительному обновлению или модификации ОС

    Крупные изменения ОС могут привести к следующим проблемам:

    • Проблемы мониторинга OneAgent
    • Перезапуск или удаление службы OneAgent
    • Удаление OneAgent

    Чтобы выполнить крупные изменения ОС на хосте OneAgent:

    1. Удалите OneAgent
    2. Примените изменения ОС
    3. Переустановите OneAgent
      Для переустановки, возможно, потребуется указать параметры подключения в установщике. Однако часть конфигурации OneAgent сохранится после удаления, например идентификация хоста.

    Эта информация применима ко всем операционным системам, на которых поддерживается полная установка OneAgent.

    Инициализация SDK и обработка ошибок

    Если заглушка SDK сталкивается с проблемами при загрузке или инициализации модуля OneAgent (особенно если onesdk_initialize или onesdk_initalize_2 возвращает код ошибки), включите логирование для заглушки SDK для диагностики проблемы.

    Используйте один из следующих вариантов для включения логирования:

    • Установите переменную окружения DT_LOGLEVELSDK={level} (самый простой вариант).
    • Вызовите функцию onesdk_stub_set_logging_level(ONESDK_LOGGING_LEVEL_{LEVEL}).
    • Если ваша программа передаёт аргументы командной строки в SDK (onesdk_stub_process_cmdline_args), используйте аргумент командной строки --dt_loglevelsdk={level}.

    Какой бы вариант вы ни выбрали, обязательно примените его перед вызовом onesdk_initialize или onesdk_initalize_2.

    По умолчанию после включения логирования вывод лога заглушки направляется в stderr. Если вам нужен альтернативный метод обработки сообщений лога заглушки, смотрите документацию функции onesdk_stub_set_logging_callback.

    Если инициализация завершается с ошибкой, наиболее часто встречающийся код ошибки: ONESDK_ERROR_LOAD_AGENT (числовой код 2952658951, -1342308345 или 0xaffe0007, сообщение об ошибке "Could not load agent.").

    Две основные причины этой проблемы:

    • Причина: OneAgent не установлен на хосте, где выполняется программа.

    Решение: Установите OneAgent и перезапустите программу. * Причина: Программа запущена с отладчиком, поэтому OneAgent не выполнит инъекцию.

    Решение: Запустите программу без отладчика. Отладчик можно подключить позже, после запуска программы.

    Устранение проблем SDK после инициализации

    После успешной инициализации SDK могут возникать другие проблемы, например отсутствующие пути в интерфейсе или неожиданные коды ошибок, такие как ONESDK_INVALID_HANDLE. В таких случаях:

    • Проверьте сообщения от callback-функций логирования OneAgent. Смотрите документацию для onesdk_agent_set_warning_callback и onesdk_agent_set_verbose_callback.
    • Изучите файлы лога OneAgent.

    Точные расположения файлов лога смотрите на следующих страницах:

    • Безопасность OneAgent на Windows
    • Безопасность OneAgent на Linux

    Уровень лога OneAgent можно повысить, установив переменную окружения DT_LOGLEVELFILE={level} или передав аргумент командной строки --dt_loglevelfile={level} в SDK.

    В качестве альтернативы используйте DT_LOGLEVELCON={level} или --dt_loglevelcon={level}, если вы хотите получать вывод лога OneAgent через stderr. * В определённых сценариях onesdk_agent_get_current_state может предоставить дополнительную информацию.

    Устранение неполадок, специфичных для ОС¶

    Linux¶

    OneAgent, установленный на развёртываниях Chef Habitat, не выполняет инъекцию в процессы

    Несмотря на то что OneAgent можно успешно установить на машинах с сервисами, развёрнутыми через Chef Habitat, OneAgent не сможет выполнить инъекцию в процессы. В таких развёртываниях Chef Habitat использует собственный glibc супервизора хоста, а не системный glibc, на который опирается OneAgent.

    Обходное решение¶

    В качестве обходного решения создайте файл ld.so.preload для каждой версии glibc, установленной Chef, содержимое которого указывает на Process Module OneAgent на хосте супервизора Chef Habitat. Выполните следующую команду от имени root:

    [ -d /hab/pkgs/core/glibc ] && for v in $(find /hab/pkgs/core/glibc -type d -name etc); do sudo echo "/opt/dynatrace/oneagent/agent/bin/current/linux-x86-64/liboneagentproc.so" > $v/ld.so.preload && echo "Installed workaround in '$v'"; done

    Эту команду необходимо выполнять каждый раз при обновлении версии glibc Chef Habitat. Её также можно запускать как задание cron. В таком случае убедитесь, что оно выполняется до запуска сервиса, который вы хотите мониторировать. В противном случае для включения инъекции OneAgent потребуется перезапустить сервис.

    Operation not permitted

    Если вы видите ошибку Operation not permitted в консоли Linux или логах установки, убедитесь, что установка OneAgent не блокируется антивирусным программным обеспечением на хосте.

    Проблемы связи OneAgent при включённом SELinux

    OneAgent поддерживает SELinux только при загруженной целевой политике; политика многоуровневой безопасности не поддерживается. При попытке установить OneAgent на систему с SELinux в режиме многоуровневой безопасности вы получите следующее сообщение об ошибке: Installation with SELinux loaded in multi-level security mode is not supported. Dynatrace OneAgent may not work correctly.

    Если вы используете систему с SELinux в принудительном режиме и инъектированные OneAgent не могут установить соединение, хотя связь модуля ОС OneAgent работает нормально, попробуйте следующие действия. Обратите внимание, что пример ниже основан на процессе httpd, но то же самое может происходить с NGINX и другими процессами.

    1. Проверьте /var/log/audit/audit.log или journalctl на наличие запретов, например:

    # grep type=AVC /var/log/audit/audit.log



# journalctl --utc -a -t "audit"
    2. Если вы обнаружили запрет для соответствующего процесса, например:

    type=AVC msg=audit(1535366769.867:209537): avc:  denied  { name_connect } for  pid=8348 comm="httpd" dest=9999 scontext=unconfined_u:system_r:httpd_t:s0 tcontext=system_u:object_r:jboss_management_port_t:s0 tclass=tcp_socket`

    сначала проверьте, разрешает ли SElinux коммуникацию с помощью следующей команды:

    # sesearch -AC -s httpd_t -t jboss_management_port_t

    Для интерпретации вывода команды смотрите Using SELinux booleans. 3. Чтобы узнать, не разрешена ли коммуникация, выполните следующую команду:

    # setsebool -P httpd_can_network_connect on

    Команда постоянно (с сохранением после перезагрузки хоста) включит булеву переменную SELinux httpd_can_network_connect, позволяя OneAgent выполнить инъекцию в процесс httpd для установления соединения с ActiveGate. 4. Перезапустите процесс и убедитесь, что связь работает.

    OneAgent на NFS-дисках

    Поступали сообщения о нестабильной работе OneAgent на Linux при развёртывании на некачественных NFS-дисках. Для правильной работы автоматической инъекции и автоматических обновлений убедитесь, что развёртывание OneAgent соответствует следующим рекомендациям:

    Пользовательский путь установки¶

    Настройте установку OneAgent так, чтобы она не находилась в NFS-директориях (расположение OneAgent по умолчанию: /opt/dynatrace). Используйте параметр OneAgent INSTALL_PATH. Подробнее смотрите в разделе Настройка установки OneAgent на Linux.

    Путь среды выполнения¶

    Убедитесь, что путь среды выполнения /var/lib/dynatrace/oneagent не находится в NFS-директориях.

    Доступность файловой системы¶

    Доступность файловой системы критически важна не только для мониторинга OneAgent, но и для запуска любых процессов на хосте. Даже если вы настроили установку OneAgent, он всё равно создаёт символические ссылки на /opt/dynatrace для модулей глубокого мониторинга и автоматической инъекции. Убедитесь, что /opt/dynatrace доступен при запуске системы как можно раньше. OneAgent запускается относительно рано, поэтому /opt/dynatrace должен быть доступен как можно раньше в процессе загрузки.

    Остановка процессов для обновления OneAgent¶

    Если при наличии NFS вы наблюдаете проблемы обновления OneAgent, убедитесь, что перед началом обновления OneAgent остановлены все процессы, у которых могут быть включены модули глубокого мониторинга кода OneAgent.

    FUSE не поддерживается¶

    Любая файловая система, основанная на FUSE, не поддерживается.

    Мы работаем над устранением проблем, связанных с NFS-развёртыванием, поэтому данные рекомендации будут обновляться со временем.

    Несовместимость со Splunk

    Компонент splunkd Splunk версии 8.2+ аварийно завершается при включённой автоматической инъекции OneAgent.

    Проблема¶

    Согласно проблеме Splunk SPL-207550 (внешняя ссылка), Splunk не запускается после установки на Linux при наличии Dynatrace OneAgent с ошибкой ERROR: pid XXXX terminated with signal 4 (core dumped), поскольку возникает конфликт между watchdog Splunk и библиотеками Dynatrace OneAgent.

    Обходное решение¶

    Установите следующее в server.conf, раздел [watchdog]:
    usePreloadedPstacks = false

    Совместимость с антивирусным программным обеспечением

    Блокировка мьютекса в ядре Linux может приводить к блокировке OneAgent продуктом CrowdStrike Falcon при чтении данных процесса из /proc, который содержит по одной поддиректории для каждого запущенного процесса.

    • Когда OneAgent пытается прочитать /proc/<pid>, CrowdStrike Falcon блокирует мьютекс в ядре при создании директории идентификатора процесса. OneAgent окажется в непрерываемом состоянии, из которого можно выйти только перезагрузкой сервера или ожиданием ответа ввода-вывода.
    • Установка OneAgent может быть затронута в любой момент из-за отсутствия единственного правила, вызывающего проблему.

    Oracle Database Server 19c не отвечает

    Проблема¶

    Автоматическая инструментация OneAgent невозможна при установленном Oracle Database Server 19c из-за несовместимости с Process Module OneAgent.

    Подробности¶

    Process Module OneAgent требует базовой функциональности от системной библиотеки libc для выполнения автоматической инструментации. Когда другой продукт переопределяет функциональность (в данном случае функцию __errno_location), Process Module не может различить символ, предоставленный libc, и символ продукта. Вызов символа продукта приводит к сбою, поскольку он ещё не инициализирован на тот момент.

    Область применения¶

    Хосты Linux с Oracle Database Server 19c

    Решение¶

    Достаточно любого из следующих вариантов:

    • Использовать Oracle Database Server 21c или новее
    • Отключить инъекцию Process Agent через oneagentctl --set-auto-injection-enabled false
    • Выполнить следующий сниппет, заменив [PATH-TO-DATABASE-EXECUTABLE] на путь к исполняемому файлу Oracle Database 19c:

    unshare -m -- sh -c 'mount --bind /dev/null /etc/ld.so.preload && [PATH-TO-DATABASE-EXECUTABLE]'
    * Установить переменную окружения LD_AUDIT так, чтобы она применялась к Oracle Database при запуске. Подробнее, включая конкретные шаги, смотрите в разделе Preventing loading of the process module on Linux

    Windows¶

    Совместимость с антивирусным программным обеспечением

    Для бесперебойной работы OneAgent и минимизации накладных расходов рекомендуем исключить все файлы в директории установки OneAgent из антивирусного сканирования.

    Также рекомендуем настроить антивирусное ПО так, чтобы оно считало процесс OneAgent доверенным и безопасным. Обратитесь к документации вашего антивирусного решения, чтобы узнать, как это сделать.

    При использовании McAfee могут наблюдаться накладные расходы CPU. Для устранения проблемы переведите McAfee в Exploit Prevention Compatibility Mode:

    1. Отключите Self-Protection и Exploit Prevention в консоли ENS.

    2. Установите следующие значения реестра как DWORD:

    3. HKEY_LOCAL_MACHINE\SOFTWARE\McAfee\Endpoint\Ips\BO\dwBOCompatibilityMode=1

    4. HKEY_LOCAL_MACHINE\SOFTWARE\Wow6432Node\McAfee\EndPoint\Ips\BO\dwBOCompatibilityMode=1
    5. Повторно включите Self-Protection и Exploit Prevention в консоли ENS.

    Процессы не обнаруживаются?

    Могло произойти одно из следующего:

    • Процесс не поддерживается технологией мониторинга. Вы можете проверить, какие типы процессов поддерживает Dynatrace.
    • Процесс не работает на сервере. Убедитесь, что серверы запущены и процессы функционируют.
    • Задержка в коммуникации между Dynatrace и OneAgent. В таком случае подождите несколько минут и повторите попытку.
    • OneAgent не работает должным образом. Перейдите в Settings > Monitoring > Monitoring overview, чтобы убедиться, что мониторинг включён для хоста, на котором работает ваше ПО.

    Если вам по-прежнему не удаётся решить проблему, обратитесь к специалисту Dynatrace через онлайн-чат в вашем окружении. Также рассмотрите вариант установки OneAgent на другой машине.

    Как восстановить установку OneAgent?

    Установщик OneAgent для Windows не поддерживает операции изменения и восстановления. Переустановить OneAgent с помощью той же версии установщика, что и текущая установка, невозможно.

    Для переустановки OneAgent на Windows либо удалите его и переустановите, либо установите более новую версию поверх существующей.

    Сбой обновления OneAgent из-за отсутствия пакета MSI в кеше Windows Installer

    Установщик OneAgent для Windows использует кеш Windows Installer, который по умолчанию находится в C:\Windows\Installer. В нём хранятся важные файлы, необходимые для удаления и обновления продукта. Если в логе установки (расположение по умолчанию: C:\ProgramData\dynatrace\oneagent\log\installer\installation_msiexec_*.log) встречаются записи, подобные следующим:

    MSI (s) (C0:E4) [09:27:14:308]: Warning: Local cached package 'C:\Windows\Installer\312c0.msi' is missing.



...



Error 1714. The older version of Dynatrace OneAgent cannot be removed.  Contact your technical support group.  System Error 1612.



MSI (s) (C0:54) [09:27:56:489]: Product: Dynatrace OneAgent -- Error 1714. The older version of Dynatrace OneAgent cannot be removed.  Contact your technical support group.  System Error 1612.

    Попробуйте следующие шаги для устранения проблемы:

    1. Загрузите и распакуйте пакет MSI из установщика текущей установленной версии, следуя инструкциям Get MSI Package.
    2. Скопируйте пакет MSI в C:\Windows\Installer и переименуйте его, чтобы имя совпадало с именем, указанным в логе (в данном примере 312c0.msi).

    Дополнительную информацию смотрите в разделе Missing Windows Installer cache requires a computer rebuild.

    Папка AI_RecycleBin заполняет дисковое пространство

    Это известная проблема Advanced Installer. В качестве обходного решения установщик OneAgent очищает AI_RecycleBin в конце установки. Однако эта очистка может не работать, если установка завершается с ошибкой на раннем этапе, например в случае отсутствия пакета MSI в кеше Windows Installer. Подробнее смотрите в обсуждении на форумах Advanced Installer.

    AIX¶

    Инъекция не работает из-за ручной настройки предыдущей версии

    Если вы использовали OneAgent для AIX до версии 1.137 и настраивали его через JAVA_OPTS с помощью скрипта dynatrace-java-env.sh, необходимо удалить эту настройку перед использованием унифицированных скриптов мониторинга dynatrace-agentXX.sh.

    • Убедитесь, что dynatrace-java-env.sh не вызывается нигде в вашей оболочке при использовании скрипта dynatrace-agentXX.sh. dynatrace-java-env.sh устарел и должен использоваться только как запасной вариант.
    • Проверьте и удалите следующий параметр из командной строки Java или скриптов запуска (конкретная директория может отличаться):

    -agentpath:/opt/dynatrace/oneagent/agent/lib64/liboneagentloader.so

    LDR_PRELOAD64: параметр не задан

    Эта ошибка может возникать при использовании dynatrace-agentXX.sh в скрипте оболочки.

    Info: using DT_HOME: /opt/dynatrace/oneagent



.profile[33] LDR_PRELOAD64: parameter not set

    Это происходит при использовании set -u для обработки незаданных переменных и параметров как ошибок. Скрипт dynatrace-agentXX.sh экспортирует переменные, которые, хотя они могут ещё не существовать в вашем скрипте, необходимы для правильной работы. Для решения проблемы вызовите set +u перед скриптом dynatrace-agentXX.sh.

    # avoid error



set +u



export DT_HOME=/opt/dynatrace/oneagent
