    События журналов (Logs Classic)¶

    События журналов (Logs Classic)¶

    • Учебное руководство
    • Чтение: 5 мин
    • Обновлено 08 октября 2025 г.

    Log Monitoring Classic

    Dynatrace Log Monitoring позволяет создавать события журналов на основе данных журналов и использовать их при обнаружении проблем.

    Ценообразование на события журналов основано на модели Davis data units (DDU). Ознакомьтесь с разделом DDU для пользовательских событий Davis, чтобы оценить и отслеживать потребление DDU для событий журналов.

    Когда Dynatrace принимает данные журналов, он применяет запрос, указанный в определении события журнала. Каждое совпадение запускает событие журнала, которое можно настроить на создание отдельной проблемы для каждого события или объединение в одну проблему.

    Создание события журнала¶

    1. Перейдите в Settings > Log Monitoring > Events extraction и нажмите Add log event.
    2. Введите Summary. Summary выступает отображаемым именем конфигурации события журнала и должен быть уникальным.
    3. Введите Matcher. Введите запрос Log Monitoring для фильтрации данных журнала. Подробнее: Просмотрщик журналов.

    4. Настройте Event template.

    5. Укажите Title запускаемого события. Если это событие журнала создаёт проблему, этот заголовок станет также заголовком проблемы.

    6. Укажите Description события. В описании можно использовать заполнители (подробнее ниже).
    7. Выберите Event Type. Типы событий указывают на серьёзность события.
    8. Выберите, разрешить ли Allow merge. Если запускаются два или более события, Dynatrace может объединить их в одну проблему. Эта опция позволяет отключить данное поведение.

    Свойство dt.event.allow_davis_merge не разделяет проблемы для одной и той же конфигурации события журнала — оно только предотвращает слияние с проблемами других конфигураций. Для разделения проблем одной конфигурации свойство event.unique_identifier должно присутствовать в фактических данных журнала. 6. Добавьте Properties. Свойство — это пара ключ/значение, устанавливаемая для каждого запускаемого события. В качестве значения можно использовать заполнители для извлечения значений из данных журнала. Например, свойство с Key PGI и Value заполнителя {dt.entity.process_group_instance} извлечёт значение экземпляра группы процессов из данных журнала при срабатывании события.

    Установка тайм-аута для события журнала¶

    По умолчанию события журналов имеют тайм-аут 15 минут. Тайм-аут определяет, как часто источник событий должен обновлять событие журнала для его поддержания в активном состоянии. Максимальное время для события журнала — шесть часов.

    • Событие журнала остаётся активным, если источник события отправляет обновление до истечения тайм-аута (по умолчанию: 15 минут).
    • Событие автоматически закрывается, если в течение тайм-аута не поступает обновление.

    Установка пользовательского тайм-аута

    1. При создании или редактировании события журнала выберите Add property.
    2. Установите Key в dt.event.timeout.
    3. Установите Value в количество минут (например, 12).

    Заполнители¶

    Заполнители — это атрибуты записей журнала, которые можно использовать для извлечения фактических значений из данных журнала.

    • Можно использовать любой атрибут, отображаемый в просмотрщике журналов для данной записи, в качестве заполнителя.
    • Заключайте значения заполнителей в фигурные скобки (например, {dt.process.name}).

    Пример¶

    В этом примере создаётся событие журнала на основе принятых данных журналов. Событие запускается при совпадении записи журнала с условием в поле Matcher: статус error на хосте 555f5555-555a-5dd5-55f555a5b55d.

    1. Перейдите в Settings > Log Monitoring > Events extraction и нажмите Add log event.

    2. Задайте следующие настройки:

    3. Summary: syslog-agent log event

    4. Matcher: status="error" AND host.name="555f5555-555a-5dd5-55f555a5b55d"
    5. Event template - Title: [Log] log events demo
    6. Event template - Description: {content}
    7. Event template - Event type: Custom alert
    8. Отключите Allow merge.

    9. Для Properties добавьте два свойства:

    10. Key K8 Id с value {dt.kubernetes.config.id}

    11. Key Process Name с value The process name is -> {dt.process.name}
    12. Сохраните изменения и дождитесь приёма данных журналов.

    Экран настроек конфигурации событий журналов.

    Журналы могут содержать атрибуты с длинными значениями до 32 КБ. Такой атрибут можно добавить в раздел Event template. В таком случае события создаются, но значения обрезаются до 4096 символов.

    При обнаружении совпадения событие журнала создаёт проблему для каждого срабатывания.

    Проблема, созданная по событиям журналов.

    Проблема, созданная в результате срабатывания события журнала, содержит информацию из конфигурации события. Заголовок проблемы — это Event title, а раздел влияния отражает настройки Event type и дополнительные сведения из описания и свойств события журнала.

    В примере настроено два свойства (K8 Id и Process Name), однако на странице проблемы отображается только одно — потому что только заполнитель {dt.process.name} имел значение в принятых данных журнала.
