    Как определить окно обслуживания¶

    Как определить окно обслуживания¶

    • Чтение: 7 мин
    • Обновлено 12 января 2026 г.

    Dynatrace предоставляет два способа определения окон обслуживания:

    • Configuration API
    • Веб-интерфейс Dynatrace

    В одной среде мониторинга можно настроить до 2000 окон обслуживания. Это ограничение распространяется на все окна обслуживания независимо от их состояния.

    На этой странице описан подход через пользовательский интерфейс. Справочник по API см. в разделе Maintenance windows API.

    Каждое настраиваемое окно обслуживания имеет имя и описание, которые можно использовать для предоставления контекстной информации о назначении окна.

    Чтобы определить окно обслуживания:

    1. Перейдите в Settings > Maintenance windows > Monitoring, alerting, and availability.
    2. Выберите Create maintenance window.
    3. Задайте Name (имя) окна обслуживания.
    4. Необязательно: укажите Description (описание) назначения окна обслуживания.
    5. В списке Maintenance type выберите Planned (плановое) или Unplanned (внеплановое).
    6. Используйте предоставленные поля для определения расписания повторений (единовременно, ежедневно, еженедельно или ежемесячно), временного диапазона повторений и часового пояса.

    7. В списке Problem detection and alerting укажите, что должен делать Dynatrace при возникновении проблемы в отслеживаемом компоненте во время окна обслуживания:

    8. Detect problems and alert — Dynatrace автоматически обнаруживает и сообщает обо всех проблемах в обычном режиме и отображает значок окна обслуживания на каждой проблеме, обнаруженной в период обслуживания.

    9. Detect problems but don't alert — Dynatrace обнаруживает проблемы, но не отправляет по ним оповещения. Каждая проблема отображается на странице Problems со значком окна обслуживания.
    10. Disable problem detection — Dynatrace не обнаруживает проблемы и не отправляет оповещения. Проблемы, возникающие в плановые окна обслуживания, не включаются на страницу Problems и оповещения не отправляются.
    11. Необязательно: Отключить выполнение синтетических мониторов в течение окна обслуживания.
    12. При необходимости определите область применения нового окна обслуживания. В противном случае оно применяется ко всей среде.

    После создания окно обслуживания включено по умолчанию. Его можно отключить или удалить на странице настроек Maintenance windows.

    Страница окон обслуживания

    Окна обслуживания могут исключаться из расчётов доступности Synthetic Monitoring с помощью глобальной настройки.

    Отключение выполнения синтетических мониторов¶

    Можно выбрать Disable synthetic monitor execution (отключение выполнения синтетических мониторов) во время окна обслуживания. В течение окна обслуживания HTTP-мониторы и браузерные мониторы в пределах области применения окна не выполняются.

    Чтобы отключить выполнение синтетических мониторов во время окна обслуживания, они должны находиться в пределах области применения окна. Выполнение будет отключено только для подмножества синтетических мониторов, включённых в окно обслуживания.

    Обратите внимание, что браузерные и HTTP-мониторы, связанные с сервисами и веб-, мобильными и пользовательскими приложениями, включёнными в область окна обслуживания, также не выполняются.

    В приведённом ниже примере следующие объекты синтетического мониторинга, явно добавленные в окно обслуживания, отключаются:

    1. Конкретный HTTP-монитор VSP - Test
    2. Все браузерные мониторы (тип сущности Synthetic monitor)

    Синтетические мониторы, явно включённые в область окна обслуживания

    Однако если определить окно обслуживания с одним фильтром для приложения easyTravel (показано на изображении ниже), любые браузерные и HTTP-мониторы, связанные с этим приложением, отключаются, даже если они явно не включены в область окна.

    Определение области применения окна обслуживания¶

    По умолчанию окно обслуживания применяется ко всей среде. Можно сузить его область до конкретных отслеживаемых сущностей, не затрагивая остальные.

    Область окна обслуживания определяется указанием одного или нескольких Filters (фильтров) (Add filter).

    Каждый фильтр применяется независимо, то есть сущность, соответствующая любому фильтру, добавляется в окно обслуживания. Если добавить фильтр для браузерных мониторов с тегами easytravel и frontend, а другой — для HTTP-мониторов с тегом easytravel-api, окно обслуживания будет включать браузерные мониторы с обоими тегами easytravel и frontend, а также HTTP-мониторы с тегом easytravel-api.

    Фильтр определяется указанием значения для одного или нескольких условий. Сущность должна соответствовать всем условиям фильтра, чтобы войти в область окна обслуживания.

    • Можно указать Entity type (тип сущности) — выберите тип сущности из предложений автозаполнения. В один фильтр можно добавить только один Entity type.

    Тип сущности

    Если не добавлять других условий, все сущности указанного типа сопоставляются с окном обслуживания. Область можно сузить с помощью условий Entity tags или Management zones. * Можно ограничить фильтр конкретной Entity (сущностью) — выберите тип сущности, а затем выберите сущность из предоставленного списка имён и идентификаторов. В один фильтр можно добавить только одну Entity по идентификатору.

    Идентификатор и имя сущности * Укажите один или несколько Entity tags (тегов сущности) в виде пар «ключ-значение» (Add tag). Дополнительно можно указать тип сущности. Можно указать более одного тега — сущности должны соответствовать всем тегам для включения в окно обслуживания.

    Примечание: теги сущностей чувствительны к регистру.

    Условие тега для окон обслуживания * Укажите одну или несколько Management zones (зон управления) (Add item > выберите зону управления из предоставленного списка). Можно указать более одной зоны управления — сущности должны соответствовать всем зонам управления для включения в окно обслуживания.

    В Dynatrace есть отслеживаемые сущности (например, тип сущности Disk), не поддерживающие зоны управления. При добавлении такой сущности в окно обслуживания и указании условия зоны управления используется зона управления родительской сущности. Так, если добавить тип сущности Disk в окно обслуживания и указать зону управления, все диски хостов, принадлежащих этой зоне, сопоставляются с окном обслуживания. Если родительской зоны управления нет, отслеживаемая сущность сопоставляется с окном обслуживания только при условии, что зона управления не указана.

    Отношения родитель-потомок и группировки¶

    При определении области окна обслуживания Dynatrace учитывает отношения родитель-потомок и группировки сущностей.

    Если родительская сущность включена в область окна обслуживания, её непосредственные дочерние сущности также считаются находящимися на обслуживании. Например, если окно обслуживания настроено для хоста, все его диски также считаются находящимися на обслуживании. Если проблема возникает на диске, а родительский хост находится на обслуживании, затронутый дочерний диск также считается находящимся на обслуживании.

    Если группа сущностей включена в область окна обслуживания, её непосредственные участники также считаются находящимися на обслуживании. Например, если группа хостов включена в окно обслуживания, хосты-участники также считаются находящимися на обслуживании. Однако дочерние диски этих хостов не считаются находящимися на обслуживании, если только хосты не включены явно в область окна.

    Лучшие практики и рекомендации¶

    Фильтры применяются независимо, и в пределах каждого фильтра сущность должна соответствовать всем условиям для включения в область окна обслуживания.

    • Если указать Entity type без других условий, все сущности этого типа добавляются в окно обслуживания.
    • Создавайте отдельный фильтр для каждого типа сущностей, который нужно добавить в окно обслуживания. (Если создать фильтр с одним Entity type и тегом, всегда применяемым к другому типу сущностей, ни одна сущность не будет соответствовать окну обслуживания.)
    • Чтобы добавить сущность по идентификатору в фильтр, не указывайте других условий — это может привести к тому, что ни одна сущность не будет сопоставлена с окном обслуживания.
    • Чтобы добавить все сущности с заданным тегом в окно обслуживания, настройте фильтр с одним условием тега, не указывая других условий. Настройте отдельный фильтр для каждого такого тега.
    • Чтобы добавить все сущности в пределах заданной зоны управления в окно обслуживания, настройте фильтр с одним условием зоны управления, не указывая других условий. Настройте отдельный фильтр для каждой такой зоны управления.
