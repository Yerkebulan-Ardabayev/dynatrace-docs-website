    Глоссарий Dynatrace¶

    Глоссарий Dynatrace¶

    • 26-min read
    • Updated on Jan 28, 2026

    Получите более глубокое понимание терминологии и концепций Dynatrace.

    A¶

    Adaptive Traffic Management (ATM)

    Интеллектуальный механизм выборки трейсов в Dynatrace для Full-Stack Monitoring.

    Подробнее см. в разделе Adaptive Traffic Management для Dynatrace Managed.

    Apdex

    Стандарт измерения производительности, отражающий соответствие зафиксированных показателей производительности ожиданиям реальных пользователей. Apdex предоставляет единый способ измерить, насколько производительность соответствует ожиданиям пользователей. Для получения полной информации о стандарте Apdex посетите Apdex.org. В контексте Dynatrace Apdex используется для быстрой и простой оценки удовлетворённости конечных пользователей вашего приложения. Рейтинги Apdex в Dynatrace основаны на специфичных для каждого приложения пороговых значениях. Рейтинг Apdex равный 1 означает идеальную производительность. Рейтинг Apdex ниже 0.5 означает низкую производительность. Подробнее о применении Apdex в Dynatrace см. в разделе Рейтинги Apdex

    application (приложение)

    Веб-приложение состоит из веб-страниц, обслуживаемых веб-серверами и веб-контейнерами, например Tomcat. Веб- или мобильное приложение построено на основе сервисов, обрабатывающих запросы, таких как веб-запросы, вызовы веб-сервисов и обмен сообщениями. В Dynatrace термин «приложение» относится к фронтенд-части, доступной через браузер или мобильное приложение.

    Dynatrace отслеживает все сервисы, процессы и инфраструктуру вашего приложения. Оценивая все компоненты совместно, Dynatrace точно определяет вклад каждого сервиса в производительность приложения. Например, при мониторинге Java хост, JVM и процессы рассматриваются как единое целое.

    Dynatrace отслеживает все отдельные компоненты, на которых построены приложения: веб-запросы, запросы к базам данных, процессы, сервисы и многое другое. Именно совместная работа этих отдельных компонентов формирует то, что конечные пользователи воспринимают как полноценное приложение. Dynatrace идентифицирует и отслеживает каждый из этих компонентов на детальном уровне. Например, Dynatrace может показать, какой из ваших веб-сервисов работает на сервере приложений Tomcat и как на производительность Tomcat влияет его хост или даже сервер VMware vCenter. Таким образом, Dynatrace обеспечивает наблюдаемость производительности вашего уникального стека приложений, включая все его компоненты и связи.

    Application Security

    Функция Dynatrace, позволяющая обнаруживать, визуализировать, анализировать, отслеживать и устранять уязвимости в открытом и стороннем программном обеспечении в производственных и предпроизводственных средах во время выполнения.

    Application Security unit

    Способ лицензирования Dynatrace Application Security. Количество единиц Application Security, потребляемых средой, зависит от объёма оперативной памяти отслеживаемого сервера и количества часов мониторинга этих единиц.

    B¶

    browser clickpath (браузерный кликпуть)

    Имитируемый визит пользователя, отслеживающий критически важные для бизнеса рабочие процессы вашего приложения. Используйте рекордер Dynatrace для записи точной последовательности кликов и действий пользователя, которые вас интересуют для мониторинга доступности и производительности. После захвата кликов мыши и дополнительных пользовательских действий, включённых в browser clickpath, ваш синтетический монитор запускается автоматически через регулярные интервалы для проверки доступности и функциональности сайта.

    browser monitor (браузерный монитор)

    Аналог имитируемого посещения вашего приложения пользователем через современный веб-браузер. Синтетические браузерные мониторы с одним URL можно настроить для запуска из любой глобальной или частной локации с частотой до одного раза в пять минут. Браузерные мониторы сигнализируют, когда ваше приложение недоступно из Интернета или когда базовая производительность значительно ухудшается.

    bucket (корзина)

    Корзина — это термин для описания определённого объёма данных, хранящегося в течение заданного периода. Внутри корзины необходимо определить, что именно хранится (область охвата), с помощью специфичных для совпадений DQL-запросов. Также необходимо указать, как долго должны храниться данные. Это концепция, общая для решений на основе Grail, таких как Log Management and Analytics powered by Grail и Business Observability.

    C¶

    candidate (кандидат)

    См. monitoring candidates

    Cluster ActiveGate

    Компонент Dynatrace, обеспечивающий безопасную и надёжную связь с облачными узлами кластера Dynatrace. Он загружается из вашего кластера Dynatrace Managed и устанавливается внутри вашей инфраструктуры. Подобно прокси-серверу, ActiveGate снижает потребление пропускной способности за счёт шифрования и сжатия сетевого взаимодействия. Он также обеспечивает мониторинг виртуальной инфраструктуры. Остановка или отключение ActiveGate снижает возможности мониторинга хостов, использующих Cluster ActiveGate для связи с кластерами Dynatrace.

    Cluster Management Console (CMC)

    Веб-интерфейс управления отдельным кластером Dynatrace Managed. CMC позволяет:

    • просматривать статус развёртывания и настраивать кластер Dynatrace Managed;
    • управлять узлами кластера и ActiveGate;
    • управлять средами мониторинга;
    • создавать учётные записи пользователей и группы;
    • просматривать информацию о лицензировании и статус потребления;
    • просматривать журнал аудита и события кластера.

    crash analysis (анализ сбоев)

    Определение первопричин сбоя процесса. Dynatrace ускоряет анализ сбоев, предоставляя запись о сбое процесса для каждого затронутого процесса и отслеживаемого хоста. Доступные для анализа сбоев данные могут включать дамп ядра для C/C++, дамп ядра Java или аномальное завершение программы из-за исключений. Dynatrace также предоставляет доступ к дополнительным артефактам сбоев, например файлам hs_err_pid для сбоев Java, текстовым файлам с анализом дампов ядра Linux и Windows, а также файлам, содержащим исключения .NET, Java или Node.js, потенциально ответственные за сбои.

    credential vault (хранилище учётных данных)

    Централизованное место для безопасного хранения и управления учётными данными, используемыми для аутентификации и авторизации в синтетических мониторах. Поддерживаемые типы учётных данных: пары «имя пользователя/пароль», сертификаты и токены. Учётные данные хранятся в зашифрованном виде с использованием Advanced Encryption Standard (AES-256). Доступ к данным шифруется с использованием TLS 1.2.

    Хранилище учётных данных доступно в разделе Settings > Web and mobile monitoring > Credential vault.

    D¶

    data center (центр обработки данных)

    С точки зрения картографирования топологии среды Dynatrace Smartscape центр обработки данных — это:

    • Группировка виртуальных машин, работающих в экземпляре облака Amazon. Dynatrace сопоставляет зоны доступности с вашим центром обработки данных.
    • Или набор виртуальных хостов на основе vCenter, передающих данные в Dynatrace через один ActiveGate. С одним центром обработки данных может быть связано несколько серверов vCenter или автономных хостов ESXi.

    На каждом виртуальном хосте в центре обработки данных должен быть установлен Dynatrace OneAgent. В представлении Smartscape будет отображаться как минимум один центр обработки данных, если у вас установлен ActiveGate. Если в вашей среде нет ActiveGate, Smartscape не может создать центр обработки данных для карты топологии.

    Обратите внимание, что Dynatrace предоставляет реальные географические сведения о местоположении центров обработки данных — ищите эту информацию в Smartscape и на страницах Host (в панели Properties).

    Davis

    AI-движок причинно-следственного анализа Dynatrace, анализирующий все взаимосвязи и зависимости в сложных ИТ-средах для предоставления ответов, а не просто данных. Davis сообщает о наличии проблемы, её влиянии на бизнес и первопричине, что позволяет быстро устранять её, включая автоматическое исправление через интеграцию с инструментами CI/CD.

    Davis data units (DDU)

    Некоторые возможности Dynatrace (например, приём пользовательских метрик, Log Monitoring и приём событий) лицензируются и потребляются на основе «валюты», называемой Davis data units. Вы приобретаете определённый объём DDU, и доступная квота расходуется с течением времени в зависимости от объёма мониторинга в вашей среде.

    Digital Experience Monitoring

    Digital Experience Monitoring (DEM) расширяет Application Performance Monitoring (APM), включая внешнюю перспективу для обеспечения доступности, функциональности и производительности приложений и сервисов во всех каналах цифрового взаимодействия с клиентами в режиме реального времени. Инструменты мониторинга объединяют данные о производительности приложений, поведении реальных пользователей, синтетический мониторинг и более глубокие аналитические данные об опыте (например, записи сессий) для точного выявления проблем цифрового взаимодействия и понимания их точного влияния на ключевые показатели эффективности бизнеса.

    DEM units

    Возможности Dynatrace Synthetic Monitoring, Real User Monitoring и Session Replay лицензируются и потребляются на основе единиц Digital Experience Monitoring, известных как DEM-единицы. Необходимое количество DEM-единиц зависит от количества синтетических мониторов и количества пользовательских сессий, которые необходимо отслеживать.

    Dynatrace Cluster

    «Кластер» — это группа связанных компьютеров, работающих совместно для выполнения задач. Dynatrace Cluster содержит один или несколько узлов, настроенных с помощью установщика Dynatrace Managed. После успешной установки на каждом узле кластера Dynatrace Managed запускается сервер Dynatrace, ActiveGate и один экземпляр Cassandra, Elasticsearch и NGINX. Каждый Dynatrace Cluster также предоставляет веб-интерфейс управления, называемый Cluster Management Console.

    E¶

    endpoint (конечная точка)

    API сервиса определяется конечными точками, предоставляющими информацию о том, какие операции предоставляются сервисами.

    • Каждый вызов сервиса происходит на одной конечной точке.
    • Dynatrace автоматически определяет конечные точки из отдельных вызовов сервисов на основе встроенных правил обнаружения конечных точек.

    environment (среда)

    См. monitoring environment

    Environment ActiveGate

    ActiveGate, привязанный к вашей среде и поэтому обрабатывающий только трафик от экземпляров OneAgent, принадлежащих той же среде, что и ActiveGate. Environment ActiveGate загружается из вашей среды Dynatrace и устанавливается внутри вашей инфраструктуры.

    Environment ActiveGate обрабатывает только трафик от OneAgent, но не от других ActiveGate. Environment ActiveGate может связываться с сервером Dynatrace напрямую или через Cluster ActiveGate.

    environment topology (топология среды)

    Визуализация Smartscape хостов, процессов, сервисов, приложений и центров обработки данных в вашей среде мониторинга Dynatrace и взаимосвязей между этими сущностями.

    Без ручной настройки Smartscape автоматически обнаруживает все компоненты и зависимости вашего полного технологического стека и визуализирует их в виде удобной карты топологии среды.

    event (событие)

    Ручное действие, выполненное на сервере в вашей среде. События фиксируются Dynatrace и, где это применимо, коррелируются с обнаруженными проблемами. Примеры событий: перезагрузка машины, завершение работы системы, перезапуск процессов и новые развёртывания кода. Любое ручное действие на сервере является событием, даже если оно является регулярно запланированным.

    external service (внешний сервис)

    Любой сервис, вызываемый вашим приложением или другим сервисом и не отслеживаемый напрямую Dynatrace OneAgent, считается внешним сервисом. Dynatrace отслеживает внешние сервисы, когда они отправляют или получают запросы от ваших отслеживаемых сервисов и приложений. Однако Dynatrace не получает данные о производительности внешних сервисов. Исключением являются сервисы баз данных. Если Dynatrace OneAgent установлен на машине, с которой отправляются JDBC-вызовы к вашей базе данных, Dynatrace предоставит полный спектр анализа производительности для вызываемого сервиса базы данных (даже если Dynatrace OneAgent не установлен на самом сервере базы данных).

    H¶

    host unit (хост-единица)

    Размер хоста для целей лицензирования (на основе объёма оперативной памяти хоста). В качестве мерки для определения размера хоста (то есть количества хост-единиц, которое он составляет) используется объём оперативной памяти отслеживаемого сервера. Преимущество этого подхода — его простота: технологические факторы (например, количество JVM или микросервисов, размещённых на сервере) не принимаются во внимание. Не имеет значения, основан ли хост на .NET, Java или чём-либо ещё. У вас может быть 10 JVM или 1000 JVM; такие факторы не влияют на объём потребляемого средой мониторинга.

    Подробнее см. в разделе Расчёт потребления мониторинга.

    host unit hours (хост-единицо-часы)

    Показатель потребления хост-единицы за период времени. 1 хост-единицо-час равен 1 хост-единице, потребляемой в течение 1 часа. Хост с 16 ГБ оперативной памяти (то есть 1 хост-единица), работающий полный день, потребляет 24 хост-единицо-часа.

    Подробнее см. в разделе Расчёт потребления мониторинга.

    HTTP monitor (HTTP-монитор)

    Тип синтетического монитора, состоящий из простых HTTP-запросов для проверки доступности конечных точек API или выполнения простых HTTP-проверок доступности одного ресурса. Как и браузерные мониторы, HTTP-мониторы запускаются автоматически через регулярные интервалы из наших глобальных общедоступных или частных локаций. HTTP-мониторы, выполняемые ActiveGate из частной синтетической локации, можно использовать для проверки доступности внутренних ресурсов, недоступных из-за пределов вашей сети.

    Hyperlyzer

    Многомерный визуальный интерфейс Dynatrace, позволяющий анализировать несколько измерений трёх основных показателей использования вашего приложения. С помощью Hyperlyzer можно видеть подробности о том, где находятся ваши пользователи, какие версии браузеров они используют, их операционные системы и количество различных пользовательских действий, полученных вашим приложением. Интерфейс колеса Hyperlyzer позволяет быстро сфокусироваться на конкретных интересующих вас измерениях.

    Данные Hyperlyzer основаны на правилах обнаружения, которые можно настраивать отдельно для каждого приложения.

    K¶

    key request (ключевой запрос)

    Ключевые запросы — это запросы, требующие особого внимания либо потому, что они являются критически важным показателем успеха вашего бизнеса (например, запрос на вход или оформление заказа в корзине покупок), либо потому, что они обеспечивают жизненно важную техническую функциональность, на которую опирается ваше приложение.

    Подробнее см. в разделе Мониторинг ключевых запросов.

    key performance metric (ключевая метрика производительности)

    Ключевые метрики производительности позволяют определять цели производительности, наилучшим образом соответствующие переменным потребностям каждого приложения, отслеживаемого через RUM или Synthetic Monitoring. Для RUM-приложений и синтетических браузерных мониторов можно выбирать разные ключевые метрики производительности для действий загрузки, XHR-действий и любых пользовательских действий. По умолчанию для действий загрузки и XHR задано значение Visually complete.

    В ситуациях, когда необходимо настроить ключевую метрику производительности для разных функций одного веб-приложения, можно выбрать разные ключевые метрики производительности для различных ключевых пользовательских действий приложения.

    L¶

    Log Monitoring

    Интеллектуальная оценка записей журнала данных, создаваемых сетевыми устройствами, операционными системами, приложениями и другими интеллектуальными или программируемыми устройствами.

    Dynatrace Log Monitoring предоставляет прямой доступ к содержимому журналов всех критически важных процессов вашей системы и позволяет быстро выполнять поиск конкретных сообщений журнала. Вы можете фильтровать содержимое журналов по ключевым словам или периоду анализа, а также одновременно анализировать несколько файлов журналов — даже если они хранятся на разных хостах.

    Что особенно важно, искусственный интеллект Dynatrace автоматически коррелирует соответствующие сообщения журнала с обнаруженными в вашей среде проблемами. Все соответствующие сообщения журнала, связанные с проблемой, учитываются при анализе её первопричины.

    log record (запись журнала)

    В Dynatrace Log Monitoring запись журнала — это единичная запись журнала, содержащая допустимую временну́ю метку. Запись журнала может содержать несколько строк данных журнала. Количество записей журнала играет важную роль в потреблении DDU для Log Monitoring.

    M¶

    management zone (зона управления)

    Зоны управления — это мощный механизм разделения информации, который способствует совместной работе и обмену актуальными данными в вашей среде Dynatrace, одновременно обеспечивая безопасный контроль доступа.

    Каждая настраиваемая зона управления включает набор отслеживаемых сущностей или размерных данных (например, метрик) в вашей среде, определённых правилами на основе механизма тегирования Dynatrace и других критериев. Доступ к зоне управления регулируется разрешениями пользователей и групп.

    Зоны управления могут пересекаться, как часто пересекаются обязанности команд. Пользователям может быть предоставлен доступ ко всей среде, конкретной зоне управления или подмножеству связанных зон управления.

    merged services (объединённые сервисы)

    Объединённый сервис — это сервис, агрегирующий несколько веб-запросных сервисов одной группы процессов, выполняющих идентичные функции на отдельных узлах кластера. Эти веб-запросные сервисы фактически идентичны с точки зрения мониторинга производительности. Объединённый сервис отображается в Dynatrace как единый сервис, содержащий все данные всех агрегированных сервисов.

    Предположим, у вас есть веб-сервер Apache с несколькими определениями виртуальных хостов (например, dynatrace.com, dynatrace.at и dynatrace.pl). С точки зрения Apache это независимые виртуальные хосты. Dynatrace поэтому обнаруживает их как отдельные сервисы. Однако для целей мониторинга вы можете захотеть просматривать эти сервисы как единый объединённый сервис под названием Dynatrace web page.

    Dynatrace автоматически определяет объединяемые сервисы и отображает их на странице Merged service monitoring (Settings > Server-side service monitoring > Merged service monitoring). Объединить можно только сервисы из этого списка.

    Подробнее см. в разделе Объединённые сервисы.

    Million Service Units (MSUs)

    Единица измерения объёма вычислительной работы, которую компьютер может выполнить за один час. Мониторинг модулей кода OneAgent, работающих на IBM z/OS (CICS, IMS и Java), осуществляется на основе лицензирования и потребления MSU, а не хост-единиц.

    Mission Control

    Проактивная поддержка Dynatrace Managed, автоматически решающая многие распространённые задачи обслуживания и поддержки. Проактивная поддержка Mission Control предоставляет полностью автоматизированные возможности управления, обеспечивающие безопасность, надёжность и актуальность сервера Dynatrace — без административных задач, таких как обновления и устранение неполадок.

    Выделенная команда поддержки 24x7 проактивно проводит проверки работоспособности, проверки масштабирования, обновления и устранение неполадок. В частности:

    • анализирует использование оборудования и оповещает при необходимости дополнительных ресурсов;
    • устраняет проблемы или несовместимости с вашим сервером Dynatrace, избавляя вас от необходимости отслеживать системные события и реагировать на них;
    • оптимизирует настройки Dynatrace Managed для обеспечения оптимальной производительности и стабильности;
    • собирает необходимые файлы журналов для решения проблем.

    Подробнее см. в разделе Как работает проактивная поддержка Mission Control?

    monitoring candidate (кандидат на мониторинг)

    Хост, взаимодействующий с отслеживаемыми хостами в вашей среде, но не имеющий установленного Dynatrace OneAgent. Рекомендуется устанавливать Dynatrace OneAgent на все кандидаты на мониторинг для обеспечения полной видимости и полных возможностей мониторинга.

    В Smartscape кандидат отображается пунктирными кругами и стандартным значком хоста. Кандидаты показаны связанными с соответствующими центрами обработки данных. Каждый кандидат на мониторинг также указан на странице Hosts в категории Unmonitored.

    Неактивные кандидаты на мониторинг (те, что не взаимодействовали с хостом более двух часов) не включаются в Smartscape и не отображаются на странице Hosts.

    monitoring environment (среда мониторинга)

    Включает все дашборды, диаграммы, отчёты и другие инструменты, предоставляемые Dynatrace для анализа вашего приложения или приложений. Среда мониторинга аналогична серверу анализа, предоставляющему все функции анализа производительности приложений Dynatrace.

    Можно настроить несколько сред мониторинга для группировки связанных сущностей и отдельного анализа. Например, можно настроить одну среду мониторинга для отслеживания и анализа производительности производственных кластеров, вторую — специально для производительности машин разработчиков, и третью — для серверов промежуточной среды. Сегментация сред мониторинга полностью определяется вашими потребностями.

    multidimensional baselining (многомерное базирование)

    Существует ряд переменных (или измерений), влияющих на производительность приложений: прежде всего, географический регион, ОС, браузер, тип подключения и пользовательские действия. Это пять измерений, используемых Dynatrace для оценки работоспособности приложения. Каждый подмножество когорт подвержен уникальному набору этих переменных, поэтому Dynatrace измеряет работоспособность и отчитывается по каждому из этих сегментов пользователей отдельно.

    multidimensional analysis (многомерный анализ)

    Представление Multidimensional analysis позволяет анализировать веб-запросы ваших сервисов с тонкой настройкой фильтрации, чтобы сосредоточить анализ на наиболее важных измерениях. Это представление легко настраивается и служит удобной точкой входа для углублённого анализа ваших сервисов.

    Подробнее см. в разделе Многомерный анализ.

    O¶

    OneAgent

    Dynatrace OneAgent отслеживает производительность хостов в вашей среде развёртывания приложений, а также всех связанных процессов. На основе настроек среды Dynatrace OneAgent обеспечивает детальный мониторинг производительности и Real User Monitoring ваших приложений и их сервисов. Изменения в ваших приложениях, а также проблемы с инфраструктурой и производительностью передаются в Dynatrace для мониторинга производительности.

    OneAgent DaemonSet

    С помощью OneAgent DaemonSet можно настроить OneAgent на Kubernetes вручную. DaemonSet — это функция, гарантирующая, что при гибели копии пода на узле эта копия воссоздаётся, а при добавлении узлов в кластер также добавляются копии пода.

    OneAgent log module

    С помощью модуля журналов OneAgent (см. матрицу поддержки платформ и возможностей OneAgent) можно настроить OneAgent для приёма и обработки данных журналов. После включения модуль журналов OneAgent собирает данные журналов и передаёт их в Dynatrace для анализа функцией Log Monitoring. С помощью командной строки (см. настройку OneAgent через интерфейс командной строки) можно проверить, включён ли модуль мониторинга журналов в вашем OneAgent.

    OneAgent for Android

    Поддержка мобильных приложений Dynatrace для Android. Используйте плагин Dynatrace Android Gradle для настройки мониторинга пользовательского опыта для проекта Android-приложения, который автоматически добавляет OneAgent в ваше приложение и выполняет автоинструментирование без изменения исходного кода. OneAgent for Android позволяет отслеживать посетителей, действия и распределённые трейсы PurePath®, инициируемые вашим Android-приложением.

    OneAgent for iOS

    Поддержка мобильных приложений для iOS в Dynatrace позволяет отслеживать поведение пользователей на мобильных веб-страницах. OneAgent for iOS помогает выявлять проблемы с мобильным пользовательским опытом до того, как они затронут ваших пользователей. Он также позволяет получать аналитические данные о бизнесе, такие как географическое распределение посетителей веб-страниц и средняя продолжительность сессий.

    Dynatrace позволяет настроить OneAgent for iOS с помощью Swift Package Manager или вручную через Dynatrace для включения UEM-мониторинга iOS-приложений.

    oneagentmon device

    «oneagentmon device» появляется в вашей системе Windows при установке Dynatrace OneAgent. Он используется Dynatrace для глубокого мониторинга процессов. Он работает как драйвер мониторинга и позволяет Dynatrace OneAgent добавлять собственную библиотеку между операционной системой и отслеживаемыми процессами.

    oneagentmon driver

    Специальное устройство, появляющееся в вашей системе Windows при установке Dynatrace OneAgent. Оно необходимо для глубокого мониторинга, поскольку позволяет Dynatrace OneAgent добавлять собственную библиотеку между системой и отслеживаемыми процессами.

    opaque service

    Сервис, обнаруженный со стороны вызывающего объекта Dynatrace, для которого недоступна видимость на уровне кода.

    Dynatrace может обнаруживать запросы непрозрачных сервисов и определять, какими процессами они обрабатываются, однако не может отслеживать непрозрачные сервисы напрямую.

    P¶

    prediction-based anomaly detection (обнаружение аномалий на основе прогнозирования)

    Оценка аномалий трафика и нагрузки строится на других предположениях, чем оценка аномалий производительности. Dynatrace использует ряд мер и методологий для выявления аномалий производительности, влияющих на клиентский опыт и требующих вашего внимания. Однако аномалии трафика и нагрузки основаны на суточных, сезонных и деловых циклах, обусловленных бизнес-моделью приложения, маркетинговыми усилиями и социологическими факторами. Это достаточно предсказуемые циклы, влияющие на трафик и нагрузку по мере использования приложения клиентами. Примеры предсказуемых циклов: выходные/рабочие дни, рабочее время/вечерние часы, годовые циклы активности клиентов, такие как Чёрная пятница. В связи с этим Dynatrace использует методологию, основанную на прогнозировании, для обнаружения аномалий в трафике приложений и нагрузке на сервисы.

    problem (проблема)

    Логическая группа информации в Dynatrace, включающая AI-анализ, контекст среды, анализ первопричин и другие детали, предоставленные для одного или нескольких инцидентов в вашей среде. Проблемы могут проявляться в среде как снижение производительности, неправильная функциональность или отсутствие доступности. Проблема может быть результатом одного «события» или нескольких событий.

    process (процесс)

    Процесс — это экземпляр выполняемой компьютерной программы. Процессы служат контейнерами, размещающими сервисы. При просмотре процессов вы видите топологическую информацию, тогда как сервисы дают вам наблюдаемость на уровне кода. Например, у вас может быть процесс Tomcat, размещающий веб-приложение в виде серверного сервиса. Хотя процессы являются хост-ориентированными — связанными с одной машиной в вашей среде — сервисы являются запросо-ориентированными и поэтому обычно охватывают несколько машин в центре обработки данных.

    PurePath® distributed traces (распределённые трейсы PurePath®)

    Распределённый трейс — это полный многомерный трейс транзакции в вашей среде мониторинга с видимостью на уровне кода, топологической информацией и метаданными для обеспечения максимальной детализации и точности данных.

    Технология PurePath® сводит данные, захваченные OneAgent на различных уровнях вашей среды, в единый трейс, обеспечивая полную видимость начиная с каждого отслеживаемого пользовательского действия и охватывая все компоненты вашего приложения.

    process group (группа процессов)

    Логический кластер процессов, принадлежащих одному приложению или единице развёртывания и выполняющих одинаковую функцию на нескольких хостах. Группы процессов являются ключевыми строительными блоками большинства современных веб-приложений.

    process group instance (экземпляр группы процессов)

    Единичный экземпляр группы процессов, выполняющийся на конкретном хосте. Каждый экземпляр группы процессов идентифицируется и отслеживается индивидуально, что позволяет Dynatrace предоставлять детальную информацию о производительности и поведении каждого экземпляра в контексте группы процессов в целом.

    process responsiveness (отзывчивость процесса)

    Оценка работоспособности процесса на основе метрик времени отклика, измеряющих время реакции на запросы от других процессов или клиентов (независимо от потребления ресурсов).

    Dynatrace измеряет отзывчивость процесса на уровне TCP, поэтому метрика отзывчивости доступна для любого процесса, принимающего TCP-запросы (например, прослушивающего открытый порт) и отвечающего на них. Типичные примеры: процессы веб-серверов, прокси-серверов и SQL-серверов. Эта метрика всегда измеряема. Перезапуск процессов или специальная инструментация не требуются.

    R¶

    Real User Monitoring

    Dynatrace Real User Monitoring анализирует опыт взаимодействия ваших клиентов с приложением. Real User Monitoring — это сочетание веб-аналитики и измерений производительности, создающих картину того, насколько хорошо работают компоненты, обращённые к клиентам.

    request (запрос)

    Запрос — это вызов от одного сервиса к другому на одной конечной точке (вызов сервиса). Запросы формируют метрики для вызываемого сервиса, такие как время отклика, пропускная способность и частота сбоев. Dynatrace определяет следующие типы запросов:

    Dynamic request (динамический запрос) : Запрос является динамическим, когда сервер компилирует или изменяет информацию после её получения. Примеры динамических запросов: запросы, обрабатываемые скриптами PHP (index.php), Java Servlets или обработчиками Node.js express.

    Resource (static) request (ресурсный (статический) запрос) : Запрос является статическим, когда на стороне сервера не требуется никаких дополнительных действий после его доставки (например, запросы на получение JavaScript, изображений или CSS-ресурсов).

    Full-service call (полный вызов сервиса) : Запрос является полным вызовом сервиса, когда базовое приложение или процесс полностью отслеживается Dynatrace. Полные вызовы сервисов — это, например, запросы к веб-запросным сервисам. Dynatrace имеет полное представление о запросе и может формировать серверные, внутренние и клиентские span'ы.

    External service call (вызов внешнего сервиса) : Запрос является вызовом внешнего (или непрозрачного) сервиса, если он направлен к внешнему ресурсу (например, https://MyDomainName.my.salesforce.com/) или к непрозрачному сервису, не передающему распределённые трейсы в Dynatrace (например, запрос к процессу Redis). Dynatrace видит только вызывающую сторону запроса, но не то, что происходит внутри него.

    response time (время отклика)

    Время, необходимое сервису для выполнения и завершения задачи.

    root cause analysis (анализ первопричин)

    Управляемый AI процесс Dynatrace для понимания причин возникновения обнаруженных проблем и предоставления руководства по устранению их первопричин.

    Dynatrace выявляет проблемы производительности до того, как они затрагивают ваших клиентов, и расставляет приоритеты на основе воздействия на клиентов, предоставляя мгновенную информацию о серьёзности проблемы и её влиянии на пользовательский опыт. При обнаружении проблем производительности Dynatrace предоставляет единое уведомление о проблеме, определяющее её первопричину, — вам не нужно вручную интерпретировать десятки источников данных.

    Поскольку проблемы производительности редко бывают изолированными (как правило, они являются симптомами более серьёзных неполадок), искусственный интеллект Dynatrace может анализировать миллиарды событий в вашей ИТ-среде для выявления причин проблем, а не их симптомов. Анализ первопричин Dynatrace также упрощает исправление ошибок с помощью углублённого анализа исходного кода и операторов баз данных, что позволяет сократить среднее время восстановления на 90% и более.

    S¶

    server-side service (серверный сервис)

    Веб- и мобильные приложения построены на основе сервисов, обрабатывающих запросы, таких как веб-запросы, вызовы веб-сервисов и обмен сообщениями. Например, веб-запросы, отправляемые на конкретный сервер Tomcat, являются примером серверного сервиса. Такие сервисы могут быть различных типов: веб-сервисы, веб-контейнеры, запросы к базам данных, пользовательские сервисы и другие. Сервисы могут, в свою очередь, вызывать другие сервисы, такие как веб-сервисы, удалённые сервисы и сервисы баз данных.

    Процессы, по сути, являются контейнерами, размещающими сервисы. При просмотре процессов вы видите топологическую информацию, тогда как сервисы могут предоставлять наблюдаемость на уровне кода. Например, у вас может быть процесс Tomcat, размещающий веб-приложение в виде серверного сервиса. Хотя процессы являются хост-ориентированными — связанными с одной машиной в вашей среде — сервисы являются запросо-ориентированными и поэтому обычно охватывают несколько машин в центре обработки данных.

    service (сервис)

    Серверный код, выполняемый внутри группы процессов.

    service backtrace (обратная трассировка сервиса)

    Визуализация всех сервисов, отправляющих запросы к определённому бэкенд-сервису. Помимо отображения того, какие сервисы напрямую вызывают конкретный бэкенд-сервис, Service backtrace показывает последовательность вызовов сервисов, ведущую к каждому запросу, вплоть до клика в браузере, инициировавшего эту последовательность вызовов.

    Session Replay

    Dynatrace Session Replay — это мощный инструмент для захвата и визуального воспроизведения пользовательских сессий. Можно записывать все взаимодействия клиентов с вашим приложением и воспроизводить каждый клик и действие в виде, подобном просмотру фильма, что позволяет восстановить полный цифровой опыт каждого пользователя, понять точные действия, приведшие к ошибкам, решать жалобы и проводить анализ удобства использования.

    Настройки Session Replay позволяют защищать конфиденциальные данные пользователей с помощью правил маскировки данных и средств управления правами доступа к воспроизведению записанных сессий.

    Smartscape

    Используя запатентованную технологию Smartscape, Dynatrace автоматически обнаруживает весь стек приложений — от центров обработки данных, хостов и процессов, составляющих инфраструктуру и сеть, через сервисы, составляющие приложения, вплоть до самих приложений. Dynatrace также обнаруживает и картографирует все зависимости этих компонентов в режиме реального времени, а затем представляет результаты в интерактивной карте топологии Smartscape. Можно нажать в любом месте визуализации Smartscape для просмотра статистики производительности каждой обнаруженной сущности в вашей среде.

    Synthetic location (синтетическая локация)

    Общедоступная или частная локация, из которой выполняются синтетические мониторы.

    Глобальные облачные общедоступные локации Dynatrace Synthetic Monitoring поддерживают как браузерные, так и HTTP-мониторы. Также можно создавать частные синтетические локации в собственной сетевой инфраструктуре на Synthetic-совместимых ActiveGate. Частные синтетические локации поддерживают браузерные и HTTP-мониторы и позволяют отслеживать как внутренние, так и внешние ресурсы.

    Synthetic Monitoring

    Доступность и производительность приложений могут восприниматься по-разному разными клиентами в разных точках мира в любой момент времени. Dynatrace Synthetic Monitoring упрощает мониторинг доступности и производительности приложений с точки зрения клиентов в любой точке мира и в любое время суток. Синтетический мониторинг проактивно имитирует посещения пользователей даже тогда, когда на сайте нет реальных посетителей. Synthetic Monitoring обеспечивает круглосуточную глобальную видимость веб-приложений за счёт запуска реальных браузерных сессий с полной поддержкой HTML5/Ajax.

    T¶

    timeframe selector (выбор периода анализа)

    Инструмент выбора периода анализа, расположенный в правом верхнем углу всех страниц и представлений Dynatrace. Выбор периода анализа позволяет фильтровать данные мониторинга по определённому временному диапазону. Кнопки Previous и Next позволяют перемещаться вперёд и назад по времени с заранее определёнными шагами (30 minutes, 1 hour, 1 day, 1 week и т.д.). Выбранные периоды сохраняются при переходе между страницами и представлениями.

    U¶

    user action (пользовательское действие)

    Действия, выполняемые вашими клиентами в приложении. Пользовательские действия соответствуют обычным видам активности пользователей: выполнение поиска, просмотр баланса счёта, просмотр корзины покупок или оформление заказа. Пользовательские действия варьируются в зависимости от функций вашего приложения.

    С точки зрения Dynatrace каждое пользовательское действие содержит всю клиентскую активность, происходящую при переходе из одного состояния в другое. Каждое пользовательское действие содержит как минимум один веб-запрос, поскольку при переходе пользователя на страницу браузер загружает несколько ресурсов и вызывает обработчик JavaScript onload, который в свою очередь может выполнять XHR-запросы и другие действия.

    Продолжительность пользовательского действия называется длительностью действия. Она представляет время, которое пользователь должен ждать перед тем, как продолжить работу. Поэтому низкая длительность действия лучше высокой. Подробнее см. в статье Что такое пользовательские действия?.

    user session (пользовательская сессия)

    Пользовательская сессия, также известная как «визит», «путешествие» или «кликпуть», — это последовательность пользовательских действий, выполняемых одним пользователем в вашем приложении за ограниченный период времени. Одна сессия, как правило, включает несколько загрузок страниц или представлений, запросов к сторонним ресурсам, запросов к сервисам и пользовательских действий, таких как клики или касания. Каждая пользовательская сессия включает как минимум одно пользовательское действие.

    Подробнее см. в разделе Пользовательские сессии.
