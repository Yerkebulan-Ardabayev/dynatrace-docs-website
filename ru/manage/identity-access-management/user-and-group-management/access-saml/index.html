    SAML

    • Справочная информация

    Dynatrace поддерживает аутентификацию через поставщика удостоверений (IdP) вашей организации. Если вы хотите использовать корпоративные учётные данные для входа в Dynatrace, вы можете настроить SAML для делегирования аутентификации вашему IdP.

    • Для определения незнакомых терминов см. глоссарий.
    • Для получения справочной информации, такой как обзор типов федерации, варианты использования и алгоритмы, см. Концепции федерации.

    Требования и спецификации¶

    Поставщик удостоверений (IdP)¶

    Ваш IdP должен соответствовать базовым требованиям спецификации SAML и требованиям безопасности для совместимости с Dynatrace SSO:

    • Всё сообщение SAML должно быть подписано (подписи только утверждений SAML недостаточно — это приводит к ответу 400 Bad Request).
    • Версия протокола SAML: urn:oasis:names:tc:SAML:2.0:protocol.
    • Формат NameID: urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress.
    • Временные метки утверждения NotBefore и NotOnOrAfter в ответе IdP должны учитывать рассинхронизацию системных часов и должны быть установлены как минимум на 1 минуту до и 1 минуту после текущего времени (это особенно касается настроек AD FS по умолчанию).
    • Код статуса ответа IdP должен быть urn:oasis:names:tc:SAML:2.0:status:Success.
    • Алгоритм SignatureMethod: http://www.w3.org/2001/04/xmldsig-more#rsa-sha256.
    • Алгоритм DigestMethod: http://www.w3.org/2001/04/xmlenc#sha256.
    • X509Certificate, добавляемый к метаданным федерации, должен быть подписан одним из следующих алгоритмов: SHA256withRSA, SHA384withRSA или SHA512withRSA.
    • Шифрование утверждений не используется.

    Поставщик услуг (SP) — Dynatrace SSO¶

    • Поддерживается подготовка Just-in-Time.
    • Время ожидания сеанса составляет 1 час и не настраивается.

    Метаданные SSO¶

    • Для глобальной федерации метаданные SP Dynatrace SSO доступны по адресу https://sso.dynatrace.com/sso/metadata. Если ваш IdP требует ручной настройки и у вас не установлены расширения XML-парсера для Chrome, рекомендуем просматривать метаданные в Firefox.
    • Для федерации уровня учётной записи или среды вы загружаете уникальные метаданные SP в ходе настройки SAML, как описано ниже.

    В зависимости от типа IdP необходимо настроить следующие конечные точки.

    Глобальная федерация¶

    • https://sso.dynatrace.com:443/saml2/login — для Entity ID (Audience Restriction).
    • https://sso.dynatrace.com:443/saml2/sp/consumer — для Single Sign On URL, Destination URL, Recipient URL.
    • https://sso.dynatrace.com:443/saml2/sp/logout — для Single Logout Service URL.

    Если экран настройки вашего IdP содержит параметр привязок SAML для входа или выхода, установите его в urn:oasis:names:tc:SAML:2.0:bindings:HTTP-POST.

    Федерация уровня учётной записи или среды¶

    • https://sso.dynatrace.com/identity-federation/federation/<configuration-UUID> — для Entity ID.

    В файле метаданных XML найдите md:EntityDescriptor и значение entityID. * https://sso.dynatrace.com/identity-federation/sp/consumer/account/<account-UUID>/federation/<configuration-UUID> — для Assertion Consumer Service (ACS).

    В файле метаданных XML найдите md:AssertionConsumerService и значение Location. * https://sso.dynatrace.com/identity-federation/sp/logout/account/<account-UUID>/federation/<configuration-UUID> — для Logout URL.

    В файле метаданных XML найдите md:SingleLogoutService и значение Location.

    Шаг 1 Создайте резервную учётную запись¶

    Резервная учётная запись должна быть нефедеративной учётной записью пользователя, принадлежащей группе с разрешением Просмотр и управление пользователями и группами, которая не охватывается федеративным входом.

    • При глобальной федерации, когда пользователь входит в систему, Dynatrace проверяет доменную часть корпоративного адреса электронной почты, чтобы определить, настроен ли SAML для этого домена. Если есть совпадение, вход перенаправляется на IdP вашей компании для аутентификации. В качестве резервного варианта вам нужен адрес электронной почты, который не будет перенаправляться таким образом.

    При использовании глобальной федерации необходимо создать резервную учётную запись, чтобы не потерять доступ при возникновении проблем с конфигурацией. * Для входа без использования федерации уровня учётной записи или среды откройте https://sso.dynatrace.com в режиме инкогнито и войдите с использованием учётных данных. (Однако если вы входите с использованием учётных данных, в вашей сессии не будет назначена группа SAML.)

    Однако если пользователь принадлежит к какой-либо глобальной федерации, он будет перенаправлен на IdP глобальной федерации, а не войдёт с помощью своих учётных данных.

    Для создания резервной учётной записи

    1. Перейдите в Account Management > ваша учётная запись > Identity & access management > User management и выберите Invite user.
    2. На шаге User email введите нефедеративный адрес электронной почты (адрес с другим доменом, не тем, для которого вы настраиваете SAML) и выберите Next.
    3. На шаге Add to groups добавьте пользователя в группу с разрешением Просмотр и управление пользователями и группами и выберите Next.
    4. На шаге Confirm permissions убедитесь, что вы добавили правильные разрешения, и выберите Invite.

    Шаг 2 Подтвердите право собственности на домен¶

    Прежде чем настраивать домен для SAML, необходимо подтвердить право собственности на него.

    1. Перейдите в Account Management > ваша учётная запись > Identity & access management > Domain verification.
    2. В разделе Add domain введите домен (например, mycompanyname.com), для которого вы хотите настроить SAML, и выберите Add, чтобы добавить его в таблицу Pending domains.

    Если пользователи вашей организации используют несколько доменов для входа (например, @mycompanyname.com и @uk.mycompanyname.com), вы можете добавить дополнительные домены в отдельные строки и начать их проверку одновременно. Каждый домен вводится в отдельной строке. 3. Для каждого проверяемого домена в таблице Pending domains выберите Copy (в столбце Value) и добавьте скопированную TXT-запись ресурса в настройки DNS вашего домена. 4. Для каждого проверяемого домена в таблице Pending domains выберите Actions > Verify, чтобы Dynatrace мог проверить добавление записи в DNS вашего домена.

    Обычно запись распространяется в системе DNS в течение нескольких минут до того, как станет доступной для проверки Dynatrace. В некоторых случаях это может занять до 24 часов. 5. Каждый подтверждённый домен добавляется в таблицу Verified domains.

    Шаг 3 Настройте SAML¶

    Перейдите в Account Management > ваша учётная запись > Identity & access management > SAML configuration для просмотра всех конфигураций SAML для выбранной учётной записи. Это отправная точка для управления конфигурациями SAML.

    Что можно делать на странице настройки SAML?

    После создания резервной учётной записи и подтверждения права собственности на домен перейдите в Identity & access management > SAML configuration и выберите New configuration для создания новой конфигурации SAML.

    1. Federation type: выберите один из трёх типов федерации и нажмите Next.

    После создания конфигурации SAML:

    • Нельзя изменить тип с Global на Account или Environment, или с Account или Environment на Global.
    • Можно изменить тип с Account на Environment или с Environment на Account.

    Федерация уровня учётной записи — описание

    Федерация уровня учётной записи осуществляет федерацию удостоверения пользователя на основе доменной части адреса электронной почты. Область действия ограничена вашей учётной записью. Ваша конфигурация не влияет на другие учётные записи вашей компании, использующие тот же домен. Она действует только в рамках вашей учётной записи и среды.

    Федерация уровня среды — описание

    Федерация уровня среды также осуществляет федерацию удостоверения пользователя на основе доменной части электронной почты, но область действия ограничена одной или несколькими средами. Например, можно включить федерацию удостоверений для среды мониторинга разработки с использованием экземпляра IdP для разработки.

    Глобальная федерация — описание

    Глобальная межучётная федерация также осуществляет федерацию удостоверения пользователя на основе доменной части адреса электронной почты. Её область действия является глобальной: она влияет на все другие учётные записи в Dynatrace, использующие ту же доменную часть электронной почты пользователя. Если у вашей компании несколько учётных записей Dynatrace и вы используете один и тот же домен для входа во все учётные записи, другие учётные записи автоматически наследуют конфигурацию федерации и используют ваш поставщик удостоверений для аутентификации.

    Подробнее см. Типы федерации. 2. SAML metadata: настройте метаданные SAML и нажмите Next.

    Федерация уровня учётной записи

    Федерация уровня учётной записи осуществляет федерацию удостоверения пользователя на основе доменной части адреса электронной почты. Область действия ограничена вашей учётной записью, и ваша конфигурация не будет влиять на другие учётные записи вашей компании, использующие тот же домен. Она действует только в рамках вашей учётной записи и среды.

    Убедитесь, что выбран правильный тип федерации. Если необходимо изменить его, нажмите Previous для возврата к шагу Federation type, а затем Next для возврата к этому шагу.

    Затем укажите один из следующих вариантов:

    Выбор доменов для конфигурации

    Выберите этот вариант для указания каждого домена, с которого вы хотите разрешить аутентификацию.

    При выборе этого варианта:

    • Если был выбран параметр Allow users from all other domains to authenticate via your IdP, он будет снят.
    • Только пользователи, чей адрес электронной почты совпадает с одним из выбранных доменов, будут аутентифицированы через ваш IdP.

    Разрешить пользователям из всех остальных доменов проходить аутентификацию через ваш IdP

    Выберите этот вариант, если вы хотите аутентифицировать пользователей из доменов, отличных от выбранных в Select domains for configuration.

    При выборе этого варианта:

    • Любой выбор в Select domains for configuration будет снят.
    • Все пользователи, входящие в вашу учётную запись или среду, всегда будут проходить аутентификацию через ваш IdP, независимо от домена в их адресе электронной почты. Это может быть полезно, например, если вы управляете внешними подрядчиками в вашем поставщике удостоверений, у которых нет выбранного адреса электронной почты для входа.

    Федерация уровня среды

    Федерация уровня среды также осуществляет федерацию удостоверения пользователя на основе доменной части электронной почты, но область действия ограничена одной или несколькими средами. Например, можно включить федерацию удостоверений для среды мониторинга разработки с использованием экземпляра IdP для разработки.

    Для каждой добавляемой федерации среды

    1. Выберите Add federation.

    2. Определите федерацию.

      • Select environment: выберите среду
      • Select domains: выберите один или несколько доменов
      • Activation: активируйте конфигурацию.

    Включите Enable SSO и выберите Complete configuration.

    Шаг 4 Проверьте конфигурацию¶

    Для проверки конфигурации

    1. Откройте новый экземпляр браузера и новое окно инкогнито.
    2. Войдите в одну из ваших сред Dynatrace.

    Для проверки федерации уровня учётной записи или среды сначала перейдите по URL-адресу тенанта, так как именно так Dynatrace SSO получает контекст (например, с https://abc12345.live.dynatrace.com).

    Устранение неполадок:

    • При возникновении проблем используйте резервную учётную запись, созданную ранее, для входа и изменения конфигурации или отключения федерации.
    • Ответы на распространённые вопросы см. в разделе FAQ ниже.

    Авторизация SAML¶

    Можно настроить автоматическую синхронизацию членства в группах между вашим IdP и Dynatrace. Для этого необходимо определить сопоставление между локально определёнными группами Dynatrace и входящими групповыми утверждениями из вашего IdP. Эти сопоставления используются во время входа пользователя для установления членства в соответствующих группах.

    Для создания сопоставления необходимо указать значение атрибута группы SAML в конфигурации сопоставления атрибутов, содержащее группы/роли пользователя из вашего IdP, а затем сопоставить группы из вашего IdP с группами в Dynatrace.

    В Dynatrace все разрешения групп пользователей объединяются. Пользователю будут предоставлены разрешения из всех типов групп: LOCAL, SAML и SCIM. Ответственность за решение о том, нужно ли и как изолировать разрешения пользователей, лежит на менеджере учётной записи.

    1. Перейдите в Account Management > ваша учётная запись > Identity & access management > Group management и добавьте новую группу или найдите существующую группу для сопоставления. Список можно фильтровать по имени и разрешениям.

    2. Настоятельно рекомендуем сначала создать новую группу (выберите Create group), чтобы проверить, работает ли авторизация SAML для этой группы.

    3. Переключение группы LOCAL на SAML удалит все назначения пользователей в эту группу.
    4. Убедитесь, что у вас есть нефедеративный пользователь с разрешением manage groups, как обсуждалось ранее.

    5. Разверните панель Edit группы для настройки сопоставления.

    6. Group name — убедитесь, что оно соответствует группе, которую вы хотите изменить.

    7. SAML Group Attribute Value — список из одного или нескольких имён федеративных групп, возвращаемых вашим IdP и сопоставленных с этой группой Dynatrace. Выберите SAML Group Attribute Value, если необходимо сопоставить с входящим групповым утверждением SAML. Обычно это не отображаемое имя группы. Например, это может быть LDAP ID.

      Обратите внимание, что при добавлении значений SAML Group Attribute Values в локальную группу:

      • Все существующие пользователи из этой группы будут удалены.
      • Источник группы изменится на SAML, что означает, что назначение пользователей в эту группу автоматически управляется через правильное сопоставление входящих SAML group claims с определёнными значениями SAML Group Attribute Values.
      • Account permissions Разрешения, связанные с учётной записью, для членов этой группы.
      • Environment permissions Разрешения, связанные со средой, для членов этой группы.
      • Management zone permissions Разрешения, связанные с Management Zone, для членов этой группы.
      • Выберите Save. Примечание: не выходите из Dynatrace.
      • Откройте новый экземпляр браузера и новое окно инкогнито и войдите в систему.
      • Перейдите в Account Management и убедитесь, что вы по-прежнему видите Identity & and access management > User management и Identity & and access management > Group management. Если вы не видите их, вы потеряли права администратора Dynatrace. Используйте нефедеративную учётную запись для изменения конфигурации при возникновении проблем.
      • После успешного входа федеративного пользователя Dynatrace сверяет входящие group claims со SAML Group Attribute Values, определёнными во всех группах SAML. При совпадении пользователь добавляется в соответствующую группу Dynatrace.

    Примечание:

    • При использовании аутентификации SAML вам не нужно приглашать пользователей в Dynatrace. При входе пользователь, которого ещё нет в Dynatrace, создаётся автоматически.

    FAQ¶

    Конфигурация SAML и атрибуты¶

    Как устранить ошибку Missing Destination Attribute при настройке PingFederate?

    Если вы используете PingFederate в качестве IdP и получаете ошибку Missing Destination Attribute value

    1. Перейдите в PingFederate и выберите Identity Provider на левой боковой панели.
    2. Выберите вкладку Signature Policy.
    3. Снимите флажок ALWAYS SIGN ASSERTION, как показано в следующем примере.

    Каков правильный формат NameIdFormat?

    Формат NameIdFormat должен быть urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress.

    Какие правила утверждений нужны в AD FS для доверяющей стороны Dynatrace?

    Вам нужны два правила преобразования выдачи в следующем порядке:

    • Первое использует шаблон правила Send LDAP Attributes as Claims и создаёт исходящий тип утверждения E-Mail Address из атрибута LDAP в хранилище атрибутов, содержащего имя пользователя Dynatrace (доменный адрес электронной почты пользователя).
    • Второе использует шаблон правила Transform an Incoming Claim и создаёт исходящий тип утверждения Name ID с форматом исходящего идентификатора имени Email и включённым параметром Pass through all claim values.

    Поддерживает ли Dynatrace создание пользователей Just-in-Time при входе?

    Да, Dynatrace поддерживает создание пользователей JIT при использовании федерации SAML. Федеративные пользователи создаются или обновляются на лету после успешной аутентификации.

    Нужна ли TXT-запись ресурса проверки домена постоянно?

    Нет, её можно удалить после того, как Dynatrace успешно подтвердит право собственности на домен.

    Разрешена ли отправка полного DN в атрибуте группы?

    Нет, полный DN содержит запятые, и они интерпретируются как отдельные имена групп. IdP должен отправлять имя группы или ID группы.

    Можно ли настроить конкретный URL-адрес среды Dynatrace как URL-адрес назначения по умолчанию?

    Да. Когда URL-адрес среды Dynatrace передаётся в параметре RelayState, пользователь перенаправляется на указанный URL-адрес после входа. Когда значение RelayState отсутствует, пользователь перенаправляется на последнюю страницу тенанта или учётной записи Dynatrace.

    В зависимости от IdP, настройка RelayState в конфигурации SAML IdP обычно влияет на всех пользователей IdP. Dynatrace SSO не поддерживает назначение URL-адреса назначения по умолчанию для каждого пользователя.

    Может ли значение атрибута группы SAML содержать список групп, разделённых запятыми?

    Да. Если значение атрибута группы SAML содержит список групп, разделённых запятыми, значение будет разделено по запятым и интерпретировано как список групп.

    Если группе SAML в Dynatrace назначено несколько групповых утверждений, нужны ли пользователю все они для добавления в группу при входе через SAML?

    Нет, пользователю нужно только одно из групповых утверждений безопасности для добавления в группу.

    Поддерживаются ли вложенные группы LDAP (например, 'CN=Admin,OU=SSO Team,DC=dynatrace,DC=com')?

    Да. Например, если ваш IdP возвращает группы как путь LDAP: CN=Admin,OU=SSO Team,DC=dynatrace,DC=com Dynatrace интерпретирует это как список из четырёх групп, разделённых запятыми: CN=Admin, OU=SSO Team, DC=dynatrace и DC=com При сопоставлении групп следует использовать одно из этих значений.

    На шаге проверки я получаю ошибку "All certificates from SAML metadata are expired or not yet valid", хотя мои метаданные не изменились.

    Сертификат подписи в метаданных IdP истёк с момента настройки конфигурации SAML. Метаданные всегда проверяются в начале процесса верификации.

    Для решения этой проблемы необходимо ротировать сертификат подписи на стороне вашего IdP, сохранить (или скопировать) обновлённые метаданные IdP и загрузить их в конфигурацию федерации в Dynatrace.

    Аутентификация и выход из системы¶

    Почему я выхожу из своих сервисов при запросе выхода из Dynatrace?

    При выходе инициируется глобальный выход, включая выход из вашего IdP, который затем передаётся другим сервисам. В противном случае вы выйдете из Dynatrace, но для повторной аутентификации достаточно будет ввести только электронную почту.

    Если вы хотите отключить это поведение (не рекомендуется с точки зрения безопасности), отредактируйте метаданные, удалите все теги SingleLogoutService и загрузите обновлённые метаданные.

    Поддерживаются ли несколько доменов и поддоменов для аутентификации?

    Да, но необходимо выполнить проверку домена и создать конфигурацию для каждого домена отдельно. Каждая конфигурация домена может использовать одни и те же метаданные и настройки IdP.

    Поддерживается ли вход, инициированный IdP, или RelayState?

    Да, поддерживаются как вход, инициированный IdP, так и RelayState.

    Как настроить аварийный доступ к учётной записи при активной федерации SAML?

    Если ваша политика безопасности требует экстренного доступа к Dynatrace в случае проблем с вашим IdP, рекомендуем пригласить нефедеративного пользователя в среду с разрешениями Account manager и использовать его в качестве аварийной учётной записи.

    • Если это невозможно, можно настроить дополнительный федеративный домен для другого IdP и использовать его для резервного входа при возникновении проблем с основным IdP.
    • Настройка многофакторной аутентификации (MFA) не поддерживается Dynatrace. Вы можете настроить аутентификацию и MFA через IdP вашей организации.

    Как настроить многофакторную аутентификацию (MFA)?

    Настройка многофакторной аутентификации (MFA) не поддерживается Dynatrace. Вы можете настроить аутентификацию и MFA через IdP вашей организации.

    Подписи и сертификаты¶

    Что нужно подписывать в SAML?

    Необходимо подписывать целые сообщения, включая запросы и ответы на выход. Подписи только части утверждения недостаточно.

    Могут ли метаданные содержать несколько сертификатов подписи?

    Да, метаданные IdP клиента могут содержать несколько сертификатов подписи. Dynatrace SSO проверяет, что SAML-сообщения от IdP клиента подписаны одним из них.

    Как ротировать сертификаты SAML IdP в Dynatrace?

    Метаданные SAML могут содержать более одного сертификата, и не все из них должны быть действительными. Это очень удобно при ротации сертификатов, когда срок действия текущего истекает.

    Для смены сертификатов

    1. Создайте новый сертификат, соответствующий нашим требованиям.

    Внимание

    Пока не изменяйте его на стороне вашего IdP! Если вы это сделаете, вы потеряете доступ к среде Dynatrace.

    Внимание

    Если ваш IdP — Azure Entra ID, обратите внимание, что новый неактивный сертификат будет автоматически использован сразу после истечения срока действия старого. Источник: Руководство: Управление сертификатами для федеративного единого входа. 2. Добавьте новый сертификат к вашим метаданным в Dynatrace.

    • Если ваш IdP позволяет включать более одного сертификата, вы можете добавить сертификат в IdP, а затем загрузить в Dynatrace обновлённые метаданные, сгенерированные вашим IdP.

      Для обеспечения непрерывного доступа к Dynatrace во время ротации сертификата проверьте записи между тегами <KeyDescriptor>, чтобы убедиться, что метаданные XML содержат два разных сертификата, как показано в примере ниже.

      <KeyDescriptor use="signing">


<KeyInfo xmlns="http://www.w3.org/2000/09/xmldsig#">


<X509Data>


<X509Certificate>(...) Certificate 1 (...)</X509Certificate>


</X509Data>


</KeyInfo>


</KeyDescriptor>


<KeyDescriptor use="signing">


<KeyInfo xmlns="http://www.w3.org/2000/09/xmldsig#">


<X509Data>


<X509Certificate>(...) Certificate 2 (...)</X509Certificate>


</X509Data>


</KeyInfo>


</KeyDescriptor>

      Обратите внимание, что метаданные XML Azure могут не содержать новый сертификат сразу после его создания. Если сертификат отсутствует, подождите несколько минут, чтобы Azure обработал новый сертификат, а затем снова загрузите метаданные XML. * В противном случае скопируйте полный элемент <KeyDescriptor use="signing"> (...) </KeyDescriptor> в метаданных, вставьте его под текущим и, в копии, замените значение сертификата новым значением сертификата, расположенным между тегами <ds:X509Certificate> и </ds:X509Certificate> в только что вставленном элементе <KeyDescriptor>. 3. Обновите метаданные только на стороне Dynatrace. 4. Проверьте конфигурацию. 5. Если процесс проверки завершился без ошибок, можно обновить сертификат на стороне вашего IdP. 6. Снова запустите тест конфигурации. 7. Необязательно Для дополнительной проверки конфигурации выйдите из системы и войдите снова. При возникновении проблем можно вернуться к старому сертификату в вашем IdP и повторить попытку. 8. Необязательно Удалите старый сертификат из метаданных вашего IdP в Dynatrace.

    Устранение неполадок¶

    Каковы возможные причины появления страницы с техническими трудностями Dynatrace?

    Если после аутентификации через IdP вы перенаправляетесь на страницу Dynatrace с сообщением We've run into technical difficulties... или 400 Bad Request, скорее всего, в конфигурации есть проблема.

    Наиболее распространённые причины:

    • Ваш IdP не принимает запрос authN с нашим форматом NameId и возвращает ответ с ошибкой с кодом статуса InvalidNameIdPolicy. Формат NameIdFormat должен быть urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress. Возможно, потребуется создать правило для формирования NameId в нашем формате.
    • Вы используете <saml:Attribute/> для возврата имени пользователя Dynatrace, но атрибут не был распознан Dynatrace; или мы не смогли найти его с помощью NameID, так как не распознали его формат. Правильный формат см. в предыдущем вопросе (Какие правила утверждений нужны в AD FS для доверяющей стороны Dynatrace?).
    • Dynatrace не доверял сертификату подписи SAML вашего IdP. Загруженные вами метаданные SAML IdP не содержали сертификат для подписи, соответствующий сертификату в утверждении, отправленном вашим IdP. Убедитесь, что сертификат находится в загруженном в Dynatrace файле метаданных XML. Если вы используете URL для загрузки метаданных, проверьте содержимое, генерируемое URL. В некоторых организациях сертификат подписи SAML необходимо запрашивать отдельно и вставлять вручную в метаданные, сохраняя содержимое URL в файл, добавляя сертификат подписи в файл и загружая файл к нам.
    • Ответ от IdP не полностью подписан (подпись утверждения может присутствовать, но этого недостаточно).
    • Ваш IdP не принимает некоторые объекты или атрибуты SAML, которые спецификация SAML 2.0 описывает как необязательные — свяжитесь с экспертом Dynatrace через онлайн-чат.

    Как устранить ошибку AADSTS50105 в конфигурации Azure?

    Если Error AADSTS50105 - The signed in user is not assigned to a role for the application возникает при федеративной аутентификации с Microsoft Entra ID, это означает, что пользователю не был предоставлен доступ к приложению в Entra ID.

    Подробнее см. в документации Microsoft:

    • Error AADSTS50105 - The signed in user is not assigned to a role for the application
    • Quickstart: Create and assign a user account

    Как устранить ошибку AADSTS76026 в конфигурации Azure?

    Если Error AADSTS50105 - The request has expired. Try to submit new request. возникает при федеративной аутентификации с Microsoft Entra ID, то, вопреки описанию ошибки, это означает, что пользователь попытался войти в Dynatrace через вход, инициированный IdP (через M365 app launcher), когда в приложении Azure была включена проверка подписи. Это ограничение Entra ID, согласно документации Microsoft о проверке подписи SAML-запросов:

    Включение Require Verification certificates не позволит проверять запросы на аутентификацию, инициированные IdP (например, функция тестирования SSO, MyApps или M365 app launcher), поскольку IdP не будет иметь тех же закрытых ключей, что и зарегистрированное приложение.

    Для решения этой проблемы выберите один из следующих подходов:

    • Пользователи должны использовать URL-адрес среды Dynatrace для входа в Dynatrace
    • Проверка подписи должна быть отключена

    Подробнее см. в документации Microsoft.
