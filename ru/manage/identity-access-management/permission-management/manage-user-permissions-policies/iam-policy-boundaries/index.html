    Границы политик

    • Latest Dynatrace

    С помощью границ политик вы можете назначать разрешения с детальными ограничениями на уровне данных.

    Границы политик позволяют ограничивать разрешения пользователей на уровне записей и ресурсов и предназначены для эффективной работы с политиками Dynatrace по умолчанию.

    Что такое границы политик?¶

    Границы политик позволяют объединять ограничения на уровне записей и/или ресурсов для использования в назначении разрешений. Это упрощает управление разделами на уровне данных и обеспечивает возможность повторного использования. Сами по себе границы ничего не ограничивают — они всегда используются вместе с политиками в процессе назначения политик группам пользователей.

    Когда граница выбрана в сочетании с политикой безопасности, она дополнительно ограничивает существующую политику, что также может привести к отсутствию доступа для пользователя.

    Границы политик поддерживают все условия, доступные в справочнике IAM.

    Границы и политики по умолчанию¶

    Сценарий использования:

    Как администратор учётной записи я хочу ограничить политику по умолчанию Read logs для 20 групп пользователей, чтобы они видели только логи из пространства имён Kubernetes для разработки.

    Решение:

    1. Создайте границу K8s DEV со следующим содержимым:

    2. storage:k8s.namespace.name = "DEVELOPMENT";

    3. Выберите эту границу при назначении политики по умолчанию Read Logs вашим группам пользователей.

    4. Граница автоматически применяется ко всем разрешениям, к которым она подходит, и тем самым ограничивает доступ к логам.

    Если вам когда-либо понадобится изменить границу из-за изменения требований доступа, например, при добавлении нового этапа в процесс разработки, вам нужно лишь изменить границу следующим образом:

    storage:k8s.namespace.name IN ("DEVELOPMENT","HARDENING");

    Технические подробности о границах¶

    Границы позволяют разделить «Что» и «Где» политики.

    Другими словами, позволяют вынести условия в отдельный объект для упрощения управления и повторного использования.

    ALLOW <permissions> WHERE <conditions>

    • Что — это часть разрешений политики, определяющая, какие API вам разрешено использовать
    • Где — это детальные разрешения на уровне записей/ресурсов, связанные с сервисом, выраженные условиями политики.

    Этот механизм чрезвычайно полезен в сочетании с политиками по умолчанию.

    Политики по умолчанию определяют набор разрешений для доступа к функциям и данным. Поскольку они являются общими, они не ограничивают доступ к конкретным записям и ресурсам.

    Пример:

    Конструкция storage:k8s.namespace.name IN ("DEV","PREPROD") является повторяющимся шаблоном в ваших политиках и может быть вынесена в границу политики.

    ALLOW storage:logs:read WHERE storage:k8s.namespace.name IN ("DEV","PREPROD");


ALLOW storage:metrics:read WHERE storage:k8s.namespace.name IN ("DEV","PREPROD");

    Граница K8s-dev-preprod:

    storage:k8s.namespace.name IN ("DEV","PREPROD");

    Конечно, это базовый пример, и ваша граница может содержать больше условий, точно определяющих, к каким записям и ресурсам имеют доступ ваши пользователи.

    Работа с границами политик¶

    Все перечисленные ниже операции управления границами выполняются на страницах Account Management.

    1. Перейдите в Account Management. Если у вас несколько учётных записей, выберите учётную запись, которой хотите управлять.

    Откроется страница https://myaccount.dynatrace.com/, которую можно добавить в закладки для быстрого доступа к управлению учётной записью. 2. Перейдите в Identity & access management > Policy management. 3. На экране управления политиками в верхней части вы найдёте две вкладки: Policies и Boundaries. 4. Выберите Boundaries, чтобы открыть экран управления границами.

    Создание новой границы политики¶

    1. Выберите Boundary и укажите:

    2. Boundary Name — название границы

    3. Boundary query — запрос границы

      • Здесь вы указываете условия в текстовом формате. По одному условию на строку, логические операторы не допускаются.
      • Выберите Save, чтобы создать границу.

    Редактирование границы политики¶

    1. Найдите границу политики, которую хотите отредактировать, в списке.
    2. В столбце Actions выберите > Edit boundary.

    3. Измените границу в соответствии с вашими требованиями и нажмите Save.

    4. Изменение границы немедленно повлияет на разрешения пользователей, назначенные с этой границей.

    Удаление границы политики¶

    1. Найдите границу политики, которую хотите удалить, в списке.
    2. В столбце Actions выберите > Delete boundary.
    3. В диалоговом окне подтверждения нажмите Cancel для отмены или Delete для подтверждения.

    Удаление используемых границ

    Граница, используемая в привязках политик, не может быть удалена. Перед удалением границы необходимо изменить или удалить все привязки, в которых она используется.

    Назначение границ политик при назначении разрешений¶

    1. Для назначения границы настройте разрешения группы, как описано в Управление группами.

    2. Выберите одну или несколько границ, которые должны быть применены к вашему назначению разрешений.

    3. Чтобы узнать, как границы применяются к разрешениям вашей политики, см. раздел Как применяются границы политик? ниже.

    Назначение границ

    Границы политик могут быть назначены только политикам, они несовместимы с разрешениями на основе ролей.

    Как применяются границы политик?¶

    Границы работают для всех сервисов, охваченных политиками безопасности и задокументированных в справочнике политик IAM.

    Когда привязка политики содержит границы, эффективная политика вычисляется по следующим правилам:

    • Только условия, добавленные в конфигурацию сервиса, будут применены к соответствующим разрешениям. Это правило не распространяется на глобальные условия. Глобальные условия будут применены ко всем разрешениям.
    • Когда граница содержит повторяющиеся имена условий, каждый оператор политики умножается на такие условия. Каждое из повторяющихся условий появляется отдельно в каждом применимом операторе.
    • Когда к политике применяется более одной границы, эффективные операторы вычисляются для каждой границы отдельно.

    В определённых конфигурациях это может привести к непреднамеренному безусловному доступу. Например, когда эта политика:

    "ALLOW storage:logs:read, storage:entities:read;"

    привязана к первой границе:

    storage:host.name="myHost"

    и второй границе:

    storage:dt.security_context="mySC"

    результат будет:

    ALLOW storage:entities:read;


ALLOW storage:entities:read WHERE storage:dt.security_context = "mySC";


ALLOW storage:logs:read WHERE storage:host.name = "myHost";


ALLOW storage:logs:read WHERE storage:dt.security_context = "mySC" `

    что фактически предоставит безусловный доступ на чтение к сущностям хранилища, поскольку условие первой границы storage:host.name="myHost" не применяется к разрешению storage:entities:read.

    • Когда оператор политики содержит несколько разрешений, такой оператор будет разделён на операторы с одним разрешением.
    • Границы применяются к операторам политики без оценки условий оператора. Граница будет применена независимо от того, содержит ли оператор уже такое же условие.
    • Границы не применяются к операторам DENY.

    Выражение зон управления внутри границ политик¶

    Чтобы использовать ваши зоны управления внутри границ, вам нужно лишь указать их в качестве условия в границе политики.

    Граница Kubernetes:

    environment:management-zone startsWith "[Kubernetes]";


storage:k8s.namespace.name IN ("DEV","PREPROD");

    Эта граница включает все зоны управления, начинающиеся с [Kubernetes], а также все записи, хранящиеся в Grail для пространств имён Kubernetes DEV и PREPROD.

    Учтите, что зоны управления, указанные в ваших границах, применяются только к операторам политик, поддерживающим зоны управления.

    Grail не поддерживает зоны управления, а использует поля storage: для контроля доступа на уровне записей. Подробнее см. В чём разница между зонами управления и новыми политиками IAM?

    Ограничения и лимиты границ политик¶

    • Не более 10 ограничений на границу для удобства управления. Если вам нужно больше ограничений, создайте дополнительные границы и назначьте несколько границ политике.
    • Границы совместимы только с политиками безопасности. Поддержка на основе ролей отсутствует.

    • Границы не поддерживают оператор AND — каждая строка границы может содержать только одно условие.

    • Если вам нужны логические операторы и при этом вы хотите повторно использовать определения политик, обратите внимание на шаблонизацию политик.
