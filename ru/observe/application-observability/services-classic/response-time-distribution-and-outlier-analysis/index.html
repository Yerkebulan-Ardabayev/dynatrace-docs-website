    Распределение времени отклика и анализ выбросов¶

    • Classic
    • Практическое руководство
    • Чтение: 5 мин
    • Опубликовано 19 июля 2017

    Для оценки производительности ваших приложений крайне важно иметь возможность отслеживать время отклика каждого запроса в рамках каждой сквозной транзакции, выполняемой вашими приложениями. Dynatrace предоставляет такую функциональность в виде распределения времени отклика и анализа выбросов. Понимая распределение времени отклика по всем запросам, вы можете сосредоточиться на тех запросах, которые имеют наибольшее время отклика. Выбросы времени отклика (запросы с необычно высоким или необычно низким временем отклика) значительно влияют на общее время отклика транзакций.

    Распределение времени отклика¶

    Чтобы просмотреть диаграмму Response time для сервиса

    1. Перейдите в Services Classic.
    2. Выберите сервис, который хотите проанализировать.
    3. На странице обзора сервиса нажмите кнопку View (например, View requests, View dynamic requests или View resource requests).
    4. Выберите вкладку Response time.

    Диаграмма Response time показывает, как были распределены значения времени отклика запросов, инициированных этим сервисом, за выбранный период (см. пример ниже). Эта диаграмма также показывает среднее количество запросов за время, а также минимальное/максимальное время отклика каждого экземпляра сервиса. Для анализа времени отклика вы можете просматривать перцентили Median, Slowest 10% или Slowest 5%.

    Наведите курсор на любой временной интервал на диаграмме, чтобы отобразить элементы управления интервалом и конкретные измерения для этого интервала, включая количество Requests, Response time, Min/max instance response time и дату/время.

    Чтобы настроить временную шкалу по оси X, выберите другой временной интервал из выпадающего списка под диаграммой. Или используйте кнопки +/- для увеличения/уменьшения масштаба периода. Стрелки влево/вправо позволяют перемещаться вперёд и назад по временным интервалам. Эти кнопки видны, если навести курсор на диаграмму.

    Анализ выбросов¶

    Для более глубокого понимания того, как время отклика варьируется по запросам за выбранный период, нажмите Analyze outliers для перехода на страницу Response time distribution.

    Выберите любую полосу перцентиля распределения, чтобы увидеть количество запросов, попадающих в этот диапазон времени отклика. Как видно в разделе Top web requests под диаграммой, самые медленные запросы (с наибольшим временем отклика) могут выполняться значительно дольше, чем самые быстрые. Такие выбросы могут оказывать большое влияние на общее время отклика сервиса.

    Вы можете подробнее рассмотреть запросы в каждом выбранном диапазоне времени отклика, нажав Zoom in.

    Пример ниже показывает, что есть около 600 запросов с временем отклика от 70,5 мс до 439 мс, и что веб-запрос orange.jsf вносит наибольший вклад в общее время отклика (491 запрос за период анализа).

    Выбрав полосу на диаграмме, нажмите Analyze backtrace или View service flow, чтобы увидеть, как эти запросы вписываются в общую последовательность транзакции.

    Пример Service flow ниже показывает, что запросы, инициированные сервисом Customer Frontend service, имеют время отклика от 70 мс до 439 мс. Сервис JourneyService вносит наибольший вклад во время отклика в этом примере. Всего этот сервис получил 585 запросов.

    Нажмите View response time hotspots для просмотра подробной информации о запросах easyTravel Customer Frontend в этом диапазоне времени отклика. Это даёт вам видимость на уровне кода для данного диапазона времени отклика. Выберите раздел Service execution инфографики, чтобы увидеть разбивку времени выполнения сервиса. В разделе Breakdown of service execution time выберите Code execution для просмотра связанных Method hotspots contributing to code execution.

    Анализ медленных запросов¶

    Возвращаясь на страницу Response time distribution, вы можете видеть, что есть несколько запросов с временем отклика более 3,4 с. Выбор этого столбца на диаграмме раскрывает дополнительные детали (см. ниже). Этот последний выбор выявил важное наблюдение — в этом диапазоне времени отклика есть только один запрос с временем отклика 3,77 с (видно ниже в разделе Top web requests). Этот единственный выброс исказил общее время отклика сервиса. Теперь пора копнуть глубже, чтобы найти первопричину этой проблемы.

    Посмотрите на горячую точку времени отклика для этого единственного запроса. Причина этого выброса — RMI-поиск JourneyBean.

    Вы также можете посмотреть Service flow для этого запроса. Service flow показывает всю транзакцию от начала до конца. Он показывает, что этот единственный запрос привёл к не менее чем 93 вызовам базы данных!

    Как видите, всего за несколько кликов Dynatrace позволяет пролить свет на мельчайшие детали каждого запроса, вплоть до уровня кода.

    Корреляция ошибок и времени отклика¶

    Ещё один важный аспект распределения времени отклика — возможность коррелировать частоту ошибок с временем отклика. Ошибочные запросы часто имеют особенно быстрое или медленное время отклика (они либо быстро завершаются с ошибкой, либо в конечном итоге завершаются по таймауту). Пример ниже показывает, что 38% всех запросов быстрее 27,6 мс завершаются с ошибкой. Более детальный просмотр таблицы запросов показывает, что это два конкретных типа запросов, которые всегда завершаются ошибкой в этом диапазоне времени отклика. Вы можете проанализировать эти запросы, нажав View details of failures.

    Анализ ошибок показывает, что все рассматриваемые запросы завершаются ошибкой из-за HTTP 500 в сервисе PHP on FPM pool www, что можно дополнительно проанализировать, нажав Details.

    Как видите, анализ выбросов, обеспечиваемый анализом распределения времени отклика Dynatrace, показывает, какие типы запросов оказывают наибольшее влияние на общее время отклика каждого сервиса. Анализ выбросов также помогает понять корреляцию между конкретными ошибками и временем отклика.
