    Захват бизнес-событий

    • 14 мин чтения

    Чтобы начать работу с бизнес-событиями, сначала необходимо определить область данных, которые вы хотите захватывать. Выбранный подход зависит от источника бизнес-событий.

    Существует пять источников бизнес-событий.

    • OneAgent

    Настройте в Dynatrace правила захвата, триггеры, поля данных и многое другое. * Web и мобильный RUM

    Получайте бизнес-события RUM, используя специализированный метод JavaScript API RUM, OneAgent для мобильных устройств или OpenKit. * Внешние источники

    Настройте внешние бизнес-системы или ИТ-системы для отправки бизнес-событий в формате JSON в API бизнес-событий (конечная точка REST). * Логи

    Используйте логи как дополнительный источник бизнес-событий через OpenPipeline. * Рабочие процессы

    Используйте действие Ingest business event в рабочих процессах для генерации бизнес-событий из автоматизированных задач.

    Получение бизнес-событий через OneAgent¶

    OneAgent версии 1.253+

    Режим мониторинга OneAgent Full-Stack является обязательным для хостов, на которых вы хотите захватывать бизнес-события.

    Чтобы захватывать бизнес-события с помощью OneAgent, сначала необходимо включить эту функцию.

    1. Перейдите в Settings > Preferences > OneAgent features.
    2. Включите функцию бизнес-событий OneAgent для технологий, соответствующих вашей среде.

    Перед захватом бизнес-событий из процесса необходимо перезапустить процесс приложения.

    Конфигурация требует одного или нескольких правил захвата, состоящих из триггеров, обязательных полей данных и необязательных полей данных события.

    В таблице ниже приведены примеры обязательных (event.type, event.provider) и необязательных (event.category) полей данных.

    Поле Тип Описание Примеры
    event.category string Стандартная категоризация на основе значимости события согласно стандарту управления событиями ITIL Availability
    event.type string Уникальный идентификатор типа данного события buy-asset, sell-asset, login
    event.provider string Источник события, например, имя компонента или системы, сгенерировавшей событие OneAgent, easyTrade.com, easyTravel.com

    Настройка источников бизнес-событий на OneAgent¶

    OneAgent версии 1.253+

    Чтобы настроить источники бизнес-событий на OneAgent

    1. Перейдите в Settings > Business Observability > OneAgent Business Event Sources.
    2. Перейдите на вкладку Incoming или Outgoing.
    3. Выберите Add new capture rule и назовите своё правило.
    4. Выберите Add trigger, чтобы определить условие, которое будет запускать бизнес-событие.

    Определите источник данных для вашего триггера, например, тело запроса (Body), Path или HTTP Header. Затем выберите Operator и определите Value. Это позволяет сопоставить содержимое, полученное из источника, со значением, которое вы определили. Когда OneAgent находит совпадение с триггером, генерируется бизнес-событие.

    • В Summary отображается всё правило триггера (например, Request - Path starts with '/api/trade/BuyAssets).
    • По умолчанию Value не чувствительно к регистру. Включите Case sensitive, если вы хотите, чтобы ваш триггер учитывал регистр источника.
    • Триггеры соединяются логикой AND внутри правила; если вы задали несколько триггеров, все они должны выполняться для захвата бизнес-события.
    • Первое совпадающее правило выполняется на каждый запрос OneAgent.
    • Рекомендуется задавать конкретные триггеры. Если триггер слишком общий и приводит к нескольким одинаковым совпадениям правил, вы получите несколько бизнес-событий. Например, если у вас есть условие триггера contains api, и термин api используется во многих ваших приложениях, данные могут быть захвачены там, где этого не следует.

    Не создавайте триггеры, значение которых равно просто /; это может привести к перегрузке и отключению вашей прикладной среды. 5. Выберите источник и значение Event provider.

    Это описывает источник события, например www.easytrade.com. Источником данных для этого поля может быть фиксированное значение, которое вы предоставляете, или оно может быть извлечено из события. 6. Выберите источник и значение Event type.

    Это описывает тип события, отправленного поставщиком события, например Asset purchase. 7. Необязательно Выберите источник и значение Event category, чтобы добавить полезный контекст к событию (например, добавьте название фондовой биржи, например NASDAQ).

    На этом завершается настройка бизнес-события, которое будет генерироваться каждый раз, когда критерии триггера совпадают. Этого может быть достаточно, если всё, что вам нужно, — это подсчёт количества совпадающих событий (например, чтобы ответить на вопрос, сколько покупок активов было совершено). Однако в большинстве случаев вы захотите добавить атрибуты событий для более детального анализа (описано на шаге ниже). Атрибуты — это поля данных, извлечённые из полезной нагрузки события JSON или XML. 8. Выберите Add data field в Event data. Укажите имя поля, затем укажите источник данных и значение из полезной нагрузки JSON или XML. Это описывает пару «атрибут-значение», которая будет связана с событием, например accountId, amount, instrumentId или price. Добавление таких пар может помочь вам ответить на более сложные вопросы, например: сколько аккаунтов купили определённый актив, какие активы покупаются чаще всего, или какие аккаунты совершают наибольшие покупки активов.

    Пример JSON-файла запроса buy-asset

    {


"accountId":6,


"amount":10,


"instrumentId":1,


"price":157.025


}
    9. Выберите Save changes. 10. Убедитесь, что правило имеет статус Enabled.

    Правила захвата OneAgent также можно определять на уровне хоста или группы хостов, чтобы ограничить область захвата.

    Примеры извлечения данных из полезных нагрузок XML¶

    Извлечение данных из полезных нагрузок XML работает с тем же синтаксисом, что и для JSON.

    Пример 1: Извлечение атрибутов и значений из базового XML¶

    <LogoutRequest id="102030AF">


<accountId>100</accountId>


</LogoutRequest>
    Имя поля Источник Путь Результат Описание
    LogoutRequest id Request → Body LogoutRequest.@id 102030AF Захватывает атрибут тега XML.
    accountId Request → Body LogoutRequest.accountId 100 Захватывает значение тега XML.

    Пример 2: Извлечение атрибутов и значений из XML с пространствами имён¶

    <?xml version="1.0" encoding="UTF-8"?>


<s:Envelope xmlns:s="http://schemas.xmlsoap.org/soap/envelope/">


<s:Body>


<Tagh1 xmlns="http://tempuri.org/">


<Tagh2 cusType="A" xmlns:a="http://schemas.datacontract.org/ex" xmlns:i="http://www.w3.org/2001/XMLSchema-instance">


<a:Code i:nil="true" />


<a:Name>MyCustomer1</a:Name>


<a:TotalValue>45</a:TotalValue>


</Tagh2>


</Tagh1>


</s:Body>


</s:Envelope>
    Имя поля Источник Путь Результат Описание
    CustomerName Request → Body s:Envelope.s:Body.Tagh1.Tagh2.a:Name MyCustomer1 Захватывает значение тега XML.
    CustomerType Request → Body s:Envelope.s:Body.Tagh1.Tagh2.@cusType A Захватывает значение атрибута тега XML.

    Примеры захвата данных из запросов и ответов¶

    Захват данных из запросов и ответов доступен как для JSON, так и для XML.

    В следующей таблице показаны примеры извлечения данных из запросов, основанные на примере полезной нагрузки JSON выше.

    Имя поля Источник Путь Результат Описание
    ContentType Request → HTTP Header Content-Type application/json Захватывает определённый заголовок запроса.
    Action Request → Query String parameters action addItems Захватывает определённый параметр строки запроса.

    Пример захвата с использованием подстановочного знака в заголовке

    • Имя поля — RequestHeaders
    • Источник — Request → HTTP header
    • Путь — *
    • Описание — захват всех атрибутов заголовка
    • Результат — все атрибуты заголовка в виде строки, разделённой пробелами

    Пример захвата с использованием подстановочного знака в строке запроса

    • Имя поля — QueryParameters
    • Источник — Request → Query string parameters
    • Путь — *
    • Описание — захват всех параметров строки запроса
    • Результат — все параметры строки запроса в виде строки, совместимой с JSON

    Получение бизнес-событий из RUM¶

    Бизнес-события доступны для всех технологий Dynatrace RUM (web RUM, mobile RUM и OpenKit). Бизнес-события RUM можно получать, используя специализированный метод RUM JavaScript, OneAgent для мобильных устройств или OpenKit.

    Отправка бизнес-событий через RUM JavaScript

    let attributes = {


"event.name": "Confirmed Booking",


"page": "booking-confirmation",

    Бизнес-события захватываются только для отслеживаемых сессий. Если RUM JavaScript отключён с помощью специального метода или из-за управления стоимостью и трафиком, бизнес-события для таких сессий не отправляются. Обратите внимание, что это поведение может измениться в будущем, что потенциально позволит отправлять бизнес-события в Dynatrace независимо от мониторинга сессий.

    Из логов в бизнес-события¶

    Используйте OpenPipeline для преобразования входящих логов в бизнес-события. Это полезно, если логи содержат бизнес-значимую информацию или нет другого пути приёма бизнес-событий.

    Ознакомьтесь со следующим примером лога для начала работы:

    {


"content": "{\"user\": \"009494\", \"ordervalue\": 1000}"


}
    1. Перейдите в Settings > Process and contextualize > OpenPipeline > Logs > Pipelines.
    2. У вас может быть несколько конвейеров. Выберите тот, который используется для обработки ваших строк логов.
    3. Перейдите на вкладку Data extraction.
    4. Откройте список Processor слева и выберите Business event, чтобы добавить новый процессор бизнес-событий.

    5. Заполните поля следующими данными:

    6. Name: OrderBizEventFromLog

    7. Matching condition: matchesPhrase(content,"ordervalue")
    8. Определите Event type как Static string и введите: biz.fromlog.order
    9. Определите Event provider как Static string и введите: customlog
    10. Установите Field extraction на поля, которые необходимо извлечь: Extract all fields
    11. Выберите Save.

    Генерация бизнес-событий из рабочих процессов¶

    Вы можете генерировать бизнес-события из рабочих процессов для захвата и сохранения бизнес-данных. Примеры использования включают:

    • Ежедневный приём курсов конвертации валют для использования в расчёте затрат.
    • Приём сторонней информации ежедневно или по расписанию для захвата бизнес-данных из системы, не отслеживаемой Dynatrace.
    • Приём агрегированной бизнес-информации для базирования, прогнозирования и обнаружения аномалий.

    Вы можете генерировать бизнес-события как действие рабочего процесса, выбрав тип действия Ingest business event.

    Действие Ingest business event может принимать данные либо из предоставленных пользователем данных JSON, либо путём выполнения DQL. Например:

    • Выберите JSON для приёма данных JSON из ответа вызова API в предыдущем шаге рабочего процесса или для предоставления пользовательского блока данных.
    • Выберите DQL для запроса к Grail на поля, которые нужно сохранить в бизнес-событии. Это позволяет запрашивать и агрегировать бизнес-метрики из других источников данных в Grail.

    В дополнение к источнику Business event data и атрибуту Ingest from, следует предоставить значения по умолчанию для атрибутов Event provider, Event type и Event category для генерации бизнес-события. Эти значения могут быть переопределены данными, предоставленными в операторе JSON или DQL. Например, если ваш запрос DQL возвращает значение для event.type, это значение будет записано в бизнес-событие вместо значения по умолчанию.

    Если значения по умолчанию не указаны, атрибуту event.provider и event.type будет присвоено значение Unknown при генерации события, а атрибуту event.category — значение Other.

    Business event data может содержать одно или несколько событий для генерации. Это применимо как к DQL, так и к JSON:

    • Запрос DQL, возвращающий несколько строк данных, сгенерирует столько же бизнес-событий.
    • Данные JSON с несколькими элементами сгенерируют столько же бизнес-событий. Например, следующая полезная нагрузка JSON генерирует три отдельных события, связанных с валютой:
    [


{


"currency.code":"USD",


"currency.name":"US dollar",


"exchange.rate":"1.1718"


},


{


"currency.code":"EUR",


"currency.name":"Euro",


"exchange.rate":"1.0000"


},


{


"currency.code":"GBP",


"currency.name":"Pound sterling",


"exchange.rate":"0.86320"


}


]

    Вы можете использовать упомянутую выше полезную нагрузку JSON, параметризировав входные данные Business event data (JSON) следующим образом:

    {{ result("stepname")| to_json }}

    Необходимо использовать инструкцию | to_json для преобразования ответа предыдущего шага в корректно отформатированный JSON. Это включит всё содержимое полезной нагрузки, включая JSON, тело и заголовки. Чтобы опустить тело и заголовки и принять только содержимое JSON, используйте следующее:

    {{ result("stepname")["json"]}}

    После успешного выполнения рабочего процесса и действия Ingest business event бизнес-события будут сгенерированы согласно вашим параметрам. Ниже приведён пример успешного выполнения:

    Использование DQL

    Шаги рабочего процесса имеют ограниченное временное окно выполнения. Рекомендуется использовать отдельный шаг рабочего процесса Execute DQL Query для длительных сложных запросов DQL. После этого вы можете использовать возвращённые данные в Ingest business event с поддержкой параметризации ({{ result("stepname" }}).

    Для этого действия требуется разрешение storage:events:write, определённое в настройках авторизации рабочего процесса. Подробнее см. Разрешения пользователей для рабочих процессов

    Если вы используете DQL, исполнитель вашего рабочего процесса должен иметь разрешения на выполнение оператора DQL для необходимых бакетов.

    Обработка данных ответа JSON¶

    В некоторых случаях использования требуется взять данные JSON, возвращённые вызовом API из предыдущего действия рабочего процесса, и преобразовать их в корректный JSON для использования с Ingest business event. Например, предположим, что вы хотите ежедневно принимать курсы валют из внешнего сервиса и сохранять их как бизнес-события для использования в других запросах. Для этого можно определить рабочий процесс с двумя шагами:

    1. Отправляет HTTP-запрос к любому API.
    2. Принимает бизнес-событие.

    HTTP-запрос возвращает содержимое полезной нагрузки JSON в следующем формате:

    {


"success": true,


"timestamp": 1754946555,


"base": "EUR",


"date": "2025-08-11",


"rates": {


"USD": 1.161388,


"JPY": 171.953923,


"AUD": 1.78907,


"CAD": 1.600567


}


}

    В нашем случае мы хотим сохранить только данные о курсах.

    Далее необходимо настроить действие Ingest business event. Для этого

    1. Выберите опцию Ingest from > JSON.
    2. Определите Event provider (default), Event type (default) и Event category (default).
    3. В редакторе Business event data (JSON) определите данные для отправки следующим образом:
    {"rates": {{result("step1")["json"]["rates"]| to_json }}}

    Для удобства редактирования синтаксиса можно выбрать Maximize для максимизации редактора. Также можно выбрать Preview для предварительного просмотра результатов выполнения действия.

    Это гарантирует, что в сгенерированном бизнес-событии будут сохранены только данные раздела rates из полезной нагрузки JSON.

    После выполнения рабочего процесса вы можете запросить бизнес-событие и получить следующие данные:

    Для извлечения отдельных курсов из события используйте следующий DQL:

    fetch bizevents


| filter event.type == "currency"


| parse rates, "JSON:rates"


| fields aud = rates[AUD], usd=rates[USD]
