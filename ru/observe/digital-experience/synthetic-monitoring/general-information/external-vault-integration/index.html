    Интеграция с внешним хранилищем секретов

    • How-to guide
    • 22-min read

    Учётные данные Synthetic Monitoring типа имя пользователя-пароль и токен в хранилище учётных данных Dynatrace могут быть синхронизированы с внешним хранилищем — Azure Key Vault, HashiCorp Vault или CyberArk Vault (только учётные данные типа имя пользователя-пароль). Синхронизированные учётные данные содержат ключи внешних пар ключ-значение, которые хранят необходимые значения.

    При настройке синхронизированных учётных данных в хранилище учётных данных Dynatrace автоматически создаёт HTTP-мониторы специально для целей синхронизации. Вы также можете использовать методы api.saveCredential() или api.saveToken() в пре- и пост-скриптах выполнения для создания собственных мониторов синхронизации.

    Автоматически созданные мониторы синхронизации именуются по идентификатору учётных данных синхронизированных учётных данных и по умолчанию выполняются ежечасно из публичной синтетической локации Amazon US East (Северная Вирджиния). Обратите внимание, что тела запросов и ответов, а также заголовки мониторов синхронизации автоматически скрываются из деталей выполнения (Анализ деталей выполнения).

    Другие синтетические мониторы могут вызывать и использовать эти синхронизированные учётные данные для тестирования эндпоинтов API и веб-сайтов. Мониторы, которые вызывают эти учётные данные, используют синхронизированные значения, полученные из внешних хранилищ. Частота синхронизации определяет, как часто эти учётные данные ротируются в синтетических мониторах, которые их вызывают.

    Процесс синхронизации считывает учётные данные из внешнего хранилища и сохраняет копию в хранилище Dynatrace.

    Azure Key Vault¶

    Учётные данные типа имя пользователя-пароль или токен для использования в синтетических мониторах могут быть синхронизированы с парами ключ-значение Azure Key Vault, содержащими имя пользователя, пароль или значение токена.

    Предварительные требования¶

    Перед настройкой учётных данных, синхронизированных с Azure Key Vault, необходимо определить требуемый идентификатор клиента (приложения) и секрет клиента как учётные данные типа токен, хранящиеся в хранилище учётных данных Dynatrace. Мы рекомендуем называть такие вспомогательные токены так, чтобы их было легко идентифицировать как сопутствующие учётные данные для синхронизации. Если ваше хранилище не содержит токенов, к которым у вас есть доступ, вы увидите предупреждение.

    Настройка синхронизированных учётных данных¶

    1. В хранилище учётных данных создайте учётные данные типа Пользователь и пароль или Токен. Вы также можете перезаписать существующие учётные данные.
    2. Для Области учётных данных выберите Synthetic.
    3. Включите Синхронизацию с внешним хранилищем.
    4. Выберите Azure Key Vault (по умолчанию) в качестве Источника учётных данных.
    5. Мы рекомендуем изменить Имя учётных данных по умолчанию для простой идентификации ваших новых учётных данных.
    6. Введите URL для доступа к хранилищу (URL хранилища) и Идентификатор арендатора (каталога).
    7. Выберите сопутствующие токены, созданные ранее, для Идентификатора клиента (приложения) и Секрета клиента.
    8. Введите имя ключа Azure Key Vault.

    Учётные данные типа имя пользователя-пароль

    Учётные данные типа токен

    • В Имя секрета для имени пользователя введите имя ключа Azure Key Vault, сопоставленного со значением имени пользователя; не вводите фактическое имя пользователя.

    • В Имя секрета для пароля введите имя ключа Azure Key Vault, сопоставленного со значением пароля; не вводите фактический пароль.

    • В Имя секрета для токена введите имя ключа Azure Key Vault, сопоставленного со значением токена; не вводите фактическое значение токена.

    • Выберите Локацию для синхронизации — вы можете выбрать любую публичную или приватную синтетическую локацию для выполнения монитора синхронизации. Вы можете искать локацию, введя имя локации в поле.
    • Необязательно: укажите Описание для учётных данных.
    • По умолчанию для учётных данных установлен Доступ только для владельца. (Подробнее о владении учётными данными.)
    • Сохраните ваши учётные данные.

    См. также Лучшие практики и что происходит при редактировании или удалении синхронизированных и сопутствующих учётных данных.

    Мониторы синхронизации Azure Key Vault¶

    Когда вы настроили синхронизированные учётные данные типа имя пользователя-пароль или токен, Dynatrace автоматически создаёт и выполняет HTTP-монитор, который синхронизирует учётные данные с Azure Key Vault. Этот монитор автоматически связывается с синхронизированными учётными данными типа имя пользователя-пароль или токен.

    См. также Лучшие практики и что происходит при редактировании или удалении учётных данных синхронизации.

      * Заголовок авторизации содержит токен доступа, полученный в первом запросе.

      * Значение имени пользователя возвращается в теле ответа. Пост-скрипт выполнения сохраняет значение в глобальной переменной.

    4. URL запроса ссылается на URL хранилища как атрибут синхронизированных учётных данных; URL хранилища не отображается. URL запроса также ссылается на ключ, сопоставленный со значением пароля в Azure Key Vault.

    HashiCorp Vault¶

    Учётные данные типа имя пользователя-пароль или токен для использования в синтетических мониторах могут быть синхронизированы с парами ключ-значение HashiCorp Vault, содержащими имя пользователя, пароль или значение токена. Вы можете использовать аутентификацию на основе AppRole или аутентификацию по сертификату.

    Предварительные требования¶

    • Перед использованием аутентификации на основе AppRole необходимо определить требуемый секретный идентификатор как учётные данные типа токен, хранящиеся в хранилище учётных данных Dynatrace; не используйте повторно другие токены в качестве секретного идентификатора. Если ваше хранилище не содержит токенов, к которым у вас есть доступ, вы увидите предупреждение.
    • Перед использованием аутентификации по сертификату необходимо сохранить требуемый TLS-сертификат в хранилище учётных данных Dynatrace. Если ваше хранилище не содержит сертификатов, к которым у вас есть доступ, вы увидите предупреждение.

    Мы рекомендуем называть такие вспомогательные токены и сертификаты так, чтобы их было легко идентифицировать как сопутствующие учётные данные для синхронизации.

    Настройка синхронизированных учётных данных¶

    1. В хранилище учётных данных создайте учётные данные типа Пользователь и пароль или Токен. Вы также можете перезаписать существующие учётные данные.
    2. Для Области учётных данных выберите Synthetic.
    3. Включите Синхронизацию с внешним хранилищем.
    4. Выберите HashiCorp Vault в качестве Источника учётных данных.
    5. Мы рекомендуем изменить Имя учётных данных по умолчанию для простой идентификации ваших новых учётных данных.
    6. Введите URL для доступа к хранилищу (URL хранилища) и Путь к учётным данным (папки должны быть разделены прямым слэшем).

    URL хранилища HashiCorp для аутентификации по сертификату может отличаться от URL, используемого для аутентификации на основе AppRole. 7. Введите имя ключа HashiCorp Vault.

    • В Имя секрета для имени пользователя введите имя ключа HashiCorp Vault, сопоставленного со значением имени пользователя; не вводите фактическое имя пользователя.

    • В Имя секрета для пароля введите имя ключа HashiCorp Vault, сопоставленного со значением пароля; не вводите фактический пароль.

    Учётные данные типа имя пользователя-пароль

    Учётные данные типа токен

    2. URL запроса ссылается на URL хранилища как атрибут синхронизированных учётных данных; URL хранилища не отображается. URL запроса также содержит метод аутентификации cert.

      * Запрос использует TLS-сертификат для аутентификации.

      * Токен клиента возвращается в теле ответа. Пост-скрипт выполнения сохраняет токен в глобальной переменной.

      2. Второй запрос (GET) получает значения имени пользователя и пароля. Он также использует api.saveCredential() в пост-скрипте выполнения для записи полученных значений в синхронизированные учётные данные типа имя пользователя-пароль, определённые выше. Детали конфигурации запроса

    3. URL запроса ссылается на URL хранилища и путь к учётным данным как атрибуты синхронизированных учётных данных; URL хранилища и путь к учётным данным не отображаются.

      * Заголовок запроса содержит токен клиента, полученный в первом запросе.

      * Значения имени пользователя и пароля возвращаются в JSON-ответе. Пост-скрипт выполнения сохраняет значения в глобальных переменных. Он также использует api.saveCredential() для записи полученных значений в синхронизированные учётные данные типа имя пользователя-пароль.

    4. Первый запрос (POST) получает токен клиента. Детали конфигурации запроса

    5. URL запроса ссылается на URL хранилища как атрибут синхронизированных учётных данных; URL хранилища не отображается. URL запроса также содержит метод аутентификации cert.

      * Запрос использует TLS-сертификат для аутентификации.

      * Токен клиента возвращается в теле ответа. Пост-скрипт выполнения сохраняет токен в глобальной переменной.

    6. URL запроса ссылается на URL хранилища и путь к учётным данным как атрибуты синхронизированных учётных данных; URL хранилища и путь к учётным данным не отображаются.

    CyberArk Vault¶

    Учётные данные типа имя пользователя-пароль для использования в синтетических мониторах могут быть синхронизированы с парами ключ-значение CyberArk Vault, содержащими имя пользователя или пароль. Вы можете использовать аутентификацию по имени пользователя и паролю или аутентификацию на основе хостов для CyberArk Vault. Аутентификация на основе хостов позволяет предварительно определить хосты (Разрешённые машины) в CyberArk Vault, которым разрешён доступ к учётным данным. Эти хосты являются публичными или приватными синтетическими локациями, которые вы выбираете для выполнения монитора синхронизации и определяются по имени хоста или IP-адресу. Аутентификация на основе хостов позволяет мониторам синхронизации обходить получение токена доступа от CyberArk Vault.

    Предварительные требования¶

    • Перед использованием аутентификации по имени пользователя и паролю необходимо определить учётные данные для аутентификации в CyberArk Vault — пару имя пользователя-пароль и, необязательно, учётные данные типа сертификат, хранящиеся в хранилище учётных данных Dynatrace. Мы рекомендуем называть такие вспомогательные учётные данные так, чтобы их было легко идентифицировать как сопутствующие учётные данные для синхронизации.
    • Перед использованием аутентификации на основе хостов необходимо определить Разрешённые машины по имени хоста или IP-адресу в деталях приложения CyberArk Vault. Это хосты, которым разрешён доступ к синхронизированным учётным данным в CyberArk Vault, и это публичные или приватные синтетические локации, которые вы выбираете для выполнения монитора синхронизации. Обратите внимание, что при определении разрешённых машин в CyberArk Vault идентификатор приложения должен совпадать с указанным при настройке синхронизированных учётных данных в Dynatrace. Необязательно вы также можете определить учётные данные типа сертификат, хранящиеся в хранилище учётных данных Dynatrace, для аутентификации в CyberArk Vault. Если ваше хранилище не содержит сертификатов, к которым у вас есть доступ, вы увидите предупреждение.

    Мы рекомендуем называть любые вспомогательные учётные данные так, чтобы их было легко идентифицировать как сопутствующие учётные данные для синхронизации.

    Настройка синхронизированных учётных данных¶

    1. В хранилище учётных данных создайте учётные данные типа Пользователь и пароль. Вы также можете перезаписать существующие учётные данные.
    2. Для Области учётных данных выберите Synthetic.
    3. Включите Синхронизацию с внешним хранилищем.
    4. Выберите CyberArk Vault в качестве Источника учётных данных.
    5. Мы рекомендуем изменить Имя учётных данных по умолчанию для простой идентификации ваших новых учётных данных.
    6. Введите URL центрального провайдера учётных данных (HTTPS) для доступа к хранилищу.
    7. Выполните шаги, связанные с Методом аутентификации.

    Имя пользователя и пароль

    Разрешённые машины

    1. Выберите Имя пользователя и пароль для Метода аутентификации.
    2. Выберите сопутствующие учётные данные типа имя пользователя-пароль, созданные ранее, для аутентификации CyberArk из списка Имя пользователя и пароль для центрального провайдера учётных данных.

    1. Выберите Разрешённые машины (локация) для Метода аутентификации.

      Вы уже должны были зарегистрировать имя хоста или IP-адрес вашей выбранной Локации для синхронизации в списке разрешённых машин CyberArk — см. Предварительные требования выше.

    8. Рекомендуется: выберите Сертификат для аутентификации в приложении CyberArk из предоставленного списка. 9. Введите дополнительные поля для идентификации пары ключ-значение CyberArk Vault.

    • Идентификатор приложения — должен совпадать с идентификатором приложения при определении разрешённых машин в CyberArk Vault — см. Предварительные требования выше.
    • Имя сейфа
    • Имя учётной записи — имя объекта, который хранит имя пользователя и пароль для извлечения и синхронизации с хранилищем учётных данных Dynatrace; это не имя учётной записи, залогиненной в CyberArk Central Credential Provider.
    • Имя папки (необязательно) — имя папки, где хранятся учётные данные в CyberArk Vault; имя папки по умолчанию — Root.
    • Выберите Локацию для синхронизации — вы можете выбрать любую публичную или приватную синтетическую локацию для выполнения монитора синхронизации. Вы можете искать локацию, введя имя локации в поле.
    • Необязательно: укажите Описание для учётных данных.
    • По умолчанию для учётных данных установлен Доступ только для владельца. (Подробнее о владении учётными данными.)
    • Сохраните ваши учётные данные.

    См. также Лучшие практики и что происходит при редактировании или удалении синхронизированных и сопутствующих учётных данных.

    Когда вы настроили синхронизированные учётные данные, Dynatrace автоматически создаёт и выполняет HTTP-монитор, который синхронизирует учётные данные с CyberArk Vault.

    Мониторы синхронизации CyberArk Vault (аутентификация по имени пользователя-паролю)¶

    Автоматически созданный HTTP-монитор содержит два запроса и автоматически связывается с синхронизированными учётными данными, определёнными выше.

      * Кроме того, первый запрос содержит любой сертификат аутентификации, указанный в Сертификат для аутентификации в CyberArk.

      * Токен возвращается в теле ответа. Пост-скрипт выполнения сохраняет токен в глобальной переменной.

      2. Второй запрос (GET) получает значения имени пользователя и пароля из CyberArk Vault. Детали конфигурации запроса

    3. URL запроса ссылается на URL центрального провайдера учётных данных как атрибут синхронизированных учётных данных; URL центрального провайдера учётных данных не отображается. URL запроса также ссылается (но не отображает) на Идентификатор приложения, Имя сейфа, Имя учётной записи и Имя папки.

      * Второй запрос также содержит любой сертификат аутентификации и токен доступа, полученный в первом запросе, в заголовке авторизации.

      * Значения имени пользователя и пароля возвращаются в теле ответа. Пост-скрипт выполнения сохраняет значения в глобальных переменных. Он также использует api.saveCredential() для записи полученных значений в синхронизированные учётные данные типа имя пользователя-пароль.

    Мониторы синхронизации CyberArk Vault (аутентификация на основе хостов)¶

    Поскольку автоматически созданный HTTP-монитор может обходить получение токена доступа, он содержит один GET-запрос для получения значений имени пользователя и пароля из CyberArk Vault. Он также использует api.saveCredential() в пост-скрипте выполнения для записи полученных значений в синхронизированные учётные данные типа имя пользователя-пароль, определённые выше. Монитор синхронизации автоматически связывается с синхронизированными учётными данными.

    Детали конфигурации запроса

    • URL запроса ссылается на URL центрального провайдера учётных данных как атрибут синхронизированных учётных данных; URL центрального провайдера учётных данных не отображается. URL запроса также ссылается (но не отображает) на Идентификатор приложения, Имя сейфа, Имя учётной записи и Имя папки.

    • Запрос также содержит любой сертификат аутентификации, указанный при настройке учётных данных.

    • Значения имени пользователя и пароля возвращаются в теле ответа. Пост-скрипт выполнения сохраняет значения в глобальных переменных. Он также использует api.saveCredential() для записи полученных значений в синхронизированные учётные данные типа имя пользователя-пароль.

    Лучшие практики и предостережения¶

    При создании монитора синхронизации вручную обязательно выберите Не сохранять и не отображать тела запросов и ответов, а также значения заголовков в деталях выполнения в любых запросах, которые получают токены клиента или значения учётных данных из внешних хранилищ. В противном случае конфиденциальная информация будет раскрыта при Анализе деталей выполнения в деталях HTTP-монитора.

    • Автоматически созданные мониторы синхронизации могут быть отредактированы. Для редактирования автоматически созданного монитора синхронизации вы должны иметь доступ к учётным данным, на которые ссылается монитор. Вам может потребоваться внести правки, если поставщик внешнего хранилища вносит изменения. Например, вам может потребоваться отредактировать URL запросов, если Microsoft изменит версию API для получения токенов клиента из Azure Key Vault.

    • В общем случае, однако, мы рекомендуем ограничить ваши изменения частотой выполнения или локациями.

    • При смене локации будьте осторожны, чтобы не выбрать приватные синтетические локации, которые не имеют доступа к внешней сети.
    • При смене локации на приватную синтетическую локацию убедитесь, что конфигурация прокси не блокирует доступ к необходимым ресурсам.
    • Мы рекомендуем изменить имена по умолчанию для синхронизированных учётных данных, сопутствующих учётных данных (например, TLS-сертификатов для HashiCorp Vault) и мониторов синхронизации для простой идентификации.
    • Мы не рекомендуем повторно использовать сопутствующие учётные данные (например, токен секретного идентификатора HashiCorp), необходимые для мониторов синхронизации, в других синтетических мониторах для целей тестирования.

    Редактирование или удаление синхронизированных и сопутствующих учётных данных¶

    • После создания синхронизированные учётные данные больше не могут быть отредактированы кем-либо; их можно только перезаписать. Для перезаписи синхронизированных учётных данных необходимо предоставить новые данные синхронизации; не предоставляйте фактические значения имени пользователя, пароля или токена.

    • При перезаписи синхронизированных учётных данных мониторы синхронизации, созданные Dynatrace, автоматически обновляются.

    • Обязательно поддерживайте одинаковое владение для всех учётных данных в мониторе синхронизации (то есть синхронизированных учётных данных и сопутствующих учётных данных).
    • Вы не можете удалить сопутствующие учётные данные, на которые ссылается монитор синхронизации, если не отключите или не удалите монитор синхронизации.

    • Если вы удалите синхронизированные учётные данные, автоматически созданный монитор синхронизации также будет удалён.

    • Если существует более одного монитора синхронизации, необходимо удалить или отключить такие мониторы, прежде чем вы сможете удалить синхронизированные учётные данные.

    • Любой синтетический монитор, который использует (удалённые) синхронизированные учётные данные для тестирования, отключается.
