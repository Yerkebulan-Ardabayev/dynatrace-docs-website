    Обнаружение групп процессов

    • 10 мин чтения

    Dynatrace определяет, какие процессы являются частью одних и тех же групп процессов, используя набор правил обнаружения по умолчанию.

    Вы можете изменить структуру групп процессов по умолчанию, изменив логику обнаружения групп процессов в:

    • Settings > Processes and containers > Process group detection, который включает следующие страницы:

    • На странице Built-in detection rules вы можете включить или отключить отдельные переключатели обнаружения групп процессов. Наведите указатель мыши на значок info справа от каждого переключателя для получения подробной информации.

    • На страницах Simple detection rules и Advanced detection rules вы можете добавить собственные правила обнаружения групп процессов, которые переопределят правила по умолчанию.
    • На странице Декларативная группировка процессов вы можете отслеживать определённые процессы технологии, неизвестной Dynatrace.

    • Settings > Processes and containers > Containers > Cloud application and workload detection, где вы можете определить правила для объединения похожих рабочих нагрузок Kubernetes в группы процессов.

    • Настройки и правила обнаружения групп процессов требуют перезапуска процессов, чтобы повлиять на то, как процессы идентифицируются и группируются.

    • Настройки и правила обнаружения групп процессов влияют только на состав групп процессов. Если вы хотите изменить именование группы процессов, используйте правила именования групп процессов.
    • Также можно использовать группы хостов для разделения кластеров на разные группы процессов.

    Простые правила обнаружения¶

    Простые правила обнаружения групп процессов позволяют адаптировать логику обнаружения по умолчанию для процессов с глубоким мониторингом через системные свойства Java или переменные окружения. Вы можете создать простое правило обнаружения с помощью веб-интерфейса Dynatrace или Settings API — см. Пример JSON-нагрузки для простого правила обнаружения.

    Простые правила обнаружения групп процессов действуют только при установленном OneAgent на ваших хостах или образах для процессов, поддерживающих глубокий мониторинг.

    Эта функция может только разделить группу процессов на несколько частей. Используйте её, если у вас есть разные развёртывания в одной группе процессов.

    Чтобы создать простое правило обнаружения в веб-интерфейсе Dynatrace

    1. Перейдите в Settings.
    2. Выберите Processes and containers > Simple detection rules.
    3. Выберите Add item.
    4. Установите Property source в значение Java system property или Environment variable.
    5. Установите Group identifier в значение, которое Dynatrace будет использовать для идентификации групп процессов.
    6. Необязательно: Установите Instance identifier в значение, которое Dynatrace будет использовать для идентификации конкретных узлов кластера внутри группы процессов.

    Эта настройка полезна, если в вашей группе процессов есть специфические имена для каждого узла. Если вы не уверены, оставьте поле пустым. По умолчанию используется один узел на хост. 7. Необязательно: Установите Restrict this rule to specific process types на тип процесса, к которому вы хотите применить правило. 8. Выберите Save changes.

    Системные свойства Java¶

    С помощью этой опции вы можете создать более детализированные группы Java-процессов. Системное свойство Java должно быть частью командной строки Java, чтобы быть обнаруженным OneAgent. Это может быть существующее системное свойство, которое ваше приложение уже использует (например, три разных значения jetty.home для трёх разных кластеров Solr), или вы можете добавить новое системное свойство. Пока системное свойство доступно в командной строке, Dynatrace может его использовать.

    Переменные окружения¶

    Эта опция охватывает как Java, так и не-Java процессы, такие как NGINX, Apache HTTPserver, FPM/PHP, Node.js, IIS и .NET. Переменные окружения, выбранные в качестве идентификаторов групп процессов, должны существовать в области видимости обнаруженных процессов и быть доступны при запуске приложения.

    • Для WebSphere это можно сделать в консоли WebSphere в разделе JVM.
    • Для Tomcat и других просто определите переменную окружения в скрипте запуска.

    Мы рекомендуем создавать переменные окружения, предназначенные специально для обнаружения процессов. Переменные окружения, используемые в других целях, такие как DT_TAGS или DT_CUSTOM_PROP, могут вызвать некорректные или непреднамеренные разделения, поскольку все значения переменных окружения учитываются при обнаружении групп процессов.

    Идентификаторы также служат именем по умолчанию для обнаруженных групп процессов. Подробнее о том, как определить переменную окружения для IIS или служб Windows, см. Определение собственных метаданных группы процессов.

    Пример:

    Предположим, у вас есть два почти идентичных развёртывания Apache HTTP-сервера, которые находятся в одном каталоге развёртывания, но на разных хостах. По умолчанию Dynatrace не может различить эти два развёртывания, поскольку у них нет уникальных характеристик для идентификации. Рассмотрим следующее правило: Apache process identified by environment variable MY_PG_NAME.

    Любой процесс Apache HTTP, включающий переменную окружения MY_PG_NAME в свою область видимости, будет использовать содержимое "MY_PG_NAME" в качестве идентификатора и имени по умолчанию. В этом сценарии вы можете заставить Dynatrace отдельно идентифицировать и именовать каждое развёртывание, назначив переменную окружения MY_PG_NAME=dynatrace.com-production одному развёртыванию и MY_PG_NAME=dynatrace.com-staging — другому.

    Расширенные правила обнаружения¶

    Расширенные правила обнаружения групп процессов позволяют создать группу процессов путём объединения процессов из разных групп и адаптировать логику обнаружения для процессов с глубоким мониторингом, используя свойства, автоматически обнаруженные OneAgent при запуске процесса. Вы можете создать расширенное правило обнаружения с помощью веб-интерфейса Dynatrace или Settings API — см. Пример JSON-нагрузки для расширенного правила обнаружения.

    Расширенные правила обнаружения групп процессов действуют только при установленном OneAgent на ваших хостах или образах для процессов, поддерживающих глубокий мониторинг.

    Чтобы создать расширенное правило обнаружения в веб-интерфейсе Dynatrace

    1. Перейдите в Settings.
    2. Выберите Processes and containers > Advanced detection rules.
    3. Выберите Add item.
    4. В разделе Process group detection определите, к каким процессам должно применяться это правило. Например, к Java JAR-файлу, содержащему ws-server.jar.
    5. В разделе Process group extraction выберите, какое значение свойства должно использоваться при обнаружении группы процессов.
    6. Выберите, должно ли правило быть автономным (standalone).

    Эта опция не рекомендуется в контейнеризированных средах, так как обнаружение по умолчанию должно справиться со всем. Подробнее см. Автономные правила. 7. В разделе Process instance extraction выберите, должны ли определённые свойства использоваться для извлечения отдельных экземпляров процессов (узлов). 8. Выберите Save changes.

    Автономные правила¶

    Когда включать эту опцию¶

    Предположим, у вас есть группа процессов с несколькими процессами. Каждый процесс одновременно выполняет одну и ту же функцию для разных клиентов, использующих ваше приложение одновременно. Хотя каждый экземпляр процесса имеет одно и то же имя, каждый работает с уникальной конфигурацией, специфичной для клиента, о которой Dynatrace не имеет информации. Поэтому Dynatrace объединяет все связанные процессы в одну группу для упрощения мониторинга.

    В случаях, когда такая группировка недостаточна, вы можете определить правила обнаружения, учитывающие клиентоспецифичные детали. Такие детали могут быть извлечены из вашей уникальной схемы развёртывания. Если ваша структура каталогов включает идентификатор клиента (например, /opt/MyCustomerBasedApp/<CustomerId>/Service/MyService) и структура каталогов одинакова на всех хостах, вы можете создать правило обнаружения, специфичное для клиента, которое работает для всех экземпляров процессов.

    Пример:

    Вы можете создать правило, которое применяется к процессам с путями исполняемых файлов, содержащими фразу MyCustomerBasedApp. Для процессов, соответствующих этому требованию, строка между /MyCustomerBasedApp/ и /Service в Executable path извлекается и используется для уникальной идентификации каждого экземпляра процесса.

    Когда отключать эту опцию¶

    Вы можете отключить опцию Standalone rules, когда в одной среде хотите различать отдельные сущности (например, рабочая среда и тестовая). В этом случае вы можете использовать обнаружение по умолчанию Dynatrace, но дополнить его собственными знаниями о схеме развёртывания.

    Вы можете определить второе свойство, идентифицирующее конкретные экземпляры процессов (или узлы кластера) внутри группы процессов. Это полезно, если у вашей группы процессов есть специфические имена для каждого экземпляра. Если вы не уверены, оставьте поле пустым. По умолчанию используется один узел на хост.

    Пример:

    • Если ни один из вышеуказанных вариантов обнаружения не работает, вы можете использовать переменную окружения DT_CLUSTER_ID для группировки всех процессов с одинаковым значением этой переменной. Все процессы в среде мониторинга, имеющие одинаковый идентификатор кластера, считаются членами одной группы процессов и разделяются только по хостам, на которых они работают (например, кластеры веб-серверов Apache, принадлежащие друг другу и обслуживающие один сайт). Убедитесь, что вы устанавливаете переменную DT_CLUSTER_ID только на уровне отдельных процессов, а не на уровне всей системы! Если вы установите эту переменную глобально, все процессы могут быть объединены в одну группу для мониторинга. Это нежелательно и не поддерживается.
    • Для идентификации узлов в кластере процессов, работающих на одном хосте, используйте переменную окружения DT_NODE_ID. Она сообщает Dynatrace, какие процессы должны рассматриваться как отдельные экземпляры группы процессов.

    Декларативная группировка процессов¶

    Dynatrace автоматически отслеживает группы процессов известных типов технологий или те, которые потребляют значительные ресурсы. С помощью декларативной группировки процессов вы можете автоматически отслеживать дополнительные технологии, переопределяя поведение по умолчанию и настраивая группировку процессов в группы процессов (PG) и экземпляры групп процессов (PGI). Подробнее см. Декларативная группировка процессов.

    Удобная настройка с помощью Settings API¶

    С помощью Settings API вы можете легко:

    • Изменить определение групп процессов
    • Настроить простое или расширенное правило обнаружения
    • Сконфигурировать декларативное обнаружение групп процессов

    Для использования Settings API

    1. Создайте API-токен и включите разрешение Write settings (settings.write).
    2. Используйте эндпоинт Получение схемы, чтобы узнать формат JSON, необходимый для отправки конфигурации.

    Пример JSON-нагрузки для простого правила обнаружения

    • SchemaID: builtin:process-group.simple-detection-rule

      [


{


"schemaId": "builtin:process-group.simple-detection-rule"


"scope": "environment"


"value": {


"enabled": true,


"ruleType": "env",


"groupIdentifier": "MY_PG_NAME",


"instanceIdentifier": "MY_INSTANCE_NAME",


"processType": "PROCESS_TYPE_APACHE_HTTPD"


}


}


]

    Пример JSON-нагрузки для расширенного правила обнаружения

    • SchemaID: builtin:process-group.advanced-detection-rule

      [


{


"schemaId": "builtin:process-group.advanced-detection-rule"


"scope": "environment"


"value": {


"enabled": true,


"processDetection": {


"property": "JBOSS_SERVER_NAME",


"containedString": "MyJBossServer",


"restrictToProcessType": "PROCESS_TYPE_JBOSS"


},


"groupExtraction": {


"property": "COMMAND_LINE_ARGS",


"delimiter": {


"from": "-environment=",


"to": "-",


"removeIds": true


},


"standaloneRule": false


},


"instanceExtraction": {}


}


}


]

    Пример JSON-нагрузки для конфигурации декларативной группы процессов

    • SchemaID: builtin:declarativegrouping

      [


{


"schemaId": "builtin:declarativegrouping"


"scope": "environment"


"value": {


"name": "keepalived",


"detection": [


{


"id": "keepalived",


"processGroupName": "keepalived",


"rules": [


{


"property": "executable",


"condition": "$eq(keepalived)"


},


{


"property": "executablePath",


"condition": "$prefix(/usr/sbin/keepalived)"


},


{


"property": "commandLine",


"condition": "$eq(-d)"


}


]


}


]


}


}


]
      3. Используйте эндпоинт Создание объекта для отправки конфигурации.

    Добавление конфигурации в Extensions 2.0¶

    Вы также можете прикрепить текущую конфигурацию к расширению Extensions 2.0, чтобы ваше пользовательское расширение поставлялось с предопределёнными правилами группировки процессов. Добавьте определение в файл Extension YAML, как в этом примере:

    ---


name: custom:my-extension


version: 1.0.0


minDynatraceVersion: "1.218"


author:


name: Joe Doe


processes:


- name: keepalived


detection:


- id: ext.keepalived


processGroupName: keepalived


rules:


- property: executable


condition: "$eq(keepalived)"


- property: executablePath


condition: "$prefix(/usr/sbin/keepalived)"


- property: commandLine


condition: "$eq(-d)"
