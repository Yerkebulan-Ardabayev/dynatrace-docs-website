    Мониторинг OS-сервисов

    Dynatrace предоставляет готовый мониторинг доступности OS-сервисов.

    Вы можете отслеживать хосты в режиме полного стека мониторинга или использовать облегчённые режимы мониторинга. Подробнее см. в разделе Режимы мониторинга OneAgent.

    Требования¶

    • На отслеживаемом хосте требуется Linux systemd версии 230+ для мониторинга связей между процессами и OS-сервисами.
    • На отслеживаемом хосте требуется Linux systemd версии 250+ для повышенной производительности обнаружения OS-сервисов за счёт менее частых обновлений.

    Варианты оповещения для OS-сервисов¶

    В зависимости от требований мониторинга можно выбрать базовое или расширенное оповещение для OS-сервисов. Режим Discovery поддерживает только базовое оповещение, тогда как режимы Full-Stack и Infrastructure Monitoring также поддерживают расширенное оповещение.

    1. Базовое оповещение (статус сервиса)¶

    Базовое оповещение предоставляет информацию только о статусе сервиса. Система будет отслеживать текущий статус OS-сервиса и уведомлять вас при его изменении с «выполняется» на «ошибка».

    С помощью свойства статуса сервиса в Smartscape можно запросить текущий статус сервиса.

    Пример¶

    fetch `dt.entity.os:service`


| fieldsAdd status

    Если оповещение включено, события и проблемы создаются при изменении статуса сервиса, например при переходе сервиса из состояния «выполняется» в «ошибка». Подробнее см. в разделе Доступность хоста.

    Оповещение о сбое сервиса¶

    Расширенное оповещение (метрика доступности сервиса)¶

    Расширенное оповещение предоставляет доступ к статусу сервиса и метрике доступности сервиса. В дополнение к изменениям статуса OS-сервиса вы получаете более детальную информацию о доступности сервиса с интервалом в одну минуту. Это позволяет реализовывать более сложную бизнес-логику, например получать уведомление, если сервис остановлен более 10 минут. Можно создавать правила оповещения для мониторинга доступности и запуска оповещения при необходимости.

    Пример¶

    timeseries count(dt.osservice.availability),


by:{dt.osservice.display_name, dt.osservice.status}


| filter dt.osservice.display_name=="apache2"

    Мониторинг сервиса¶

    Для мониторинга OS-сервиса выполните следующие шаги.

    1. Доступ к мониторингу OS-сервисов

    В Dynatrace перейдите в OS services monitoring для уровня, который вы настраиваете.

    • Параметры определяются на уровне среды.
    • Уровень группы хостов наследует все параметры среды. Кроме того, можно добавлять конфигурации для конкретных групп хостов.
    • Уровень хоста наследует все параметры группы хостов. Кроме того, можно добавлять конфигурации для конкретных хостов.

    Уровень среды¶

    Перейдите в Settings > Collect and capture > Infrastructure > OS > OS services monitoring.

    Уровень группы хостов¶

    1. Перейдите в Deployment Status > OneAgents.
    2. На странице OneAgent deployment отключите Show new OneAgent deployments.
    3. В поле Filter by введите Host group и выберите нужную группу хостов из раскрывающегося списка.

    Список хостов теперь отфильтрован по выбранной группе. У каждого хоста в списке есть ссылка Host group: <group name>, где <group name> — это название группы хостов, которую вы хотите настроить.

    Свойство Host group не отображается, если выбранный хост не принадлежит ни одной группе хостов. 4. Выберите название группы хостов в любой строке.

    Поскольку вы отфильтровали по группе хостов, все отображаемые хосты принадлежат одной группе.

    1. В настройках группы хостов выберите OS services monitoring.

    Уровень хоста¶

    1. Перейдите в Hosts Classic.
    2. Найдите и выберите хост для отображения страницы обзора хоста.

    3. В правом верхнем углу страницы обзора хоста выберите More (…) > Settings.

    4. В настройках хоста выберите OS services monitoring.

    5. Добавление политики мониторинга сервиса

    На основе состояния сервиса и правил политика мониторинга сервиса определяет способ мониторинга Dynatrace вашего сервиса. По умолчанию Dynatrace поставляется с политиками Auto-start Windows OS Services и Auto-start Linux OS Services для автоматически запускаемых сервисов Windows и Linux со статусом «ошибка».

    Ограничения

    Обратите внимание, что лимит по умолчанию составляет 100 000 сущностей OS-сервисов на кластер. В больших средах с множеством хостов рекомендуется создавать точные правила, соответствующие только важным сервисам вашей инфраструктуры. Создание слишком общих правил (например, соответствующих всем сервисам на тысячах хостов) может привести к достижению лимита (взрыв сущностей), что приведёт к исчезновению OS-сервисов из Dynatrace. Кроме того, Auto-start Windows OS Services и Auto-start Linux OS Services можно использовать как отправную точку для дальнейшего уточнения политик.

    Порядок политик мониторинга сервисов важен. Политики, стоящие выше в списке, обрабатываются раньше тех, что стоят ниже, пока они не будут выполнены. Это позволяет создавать избирательные оповещения или мониторинг с минимальным количеством политик. Например, если вы хотите отслеживать все автоматически запускаемые сервисы, а не только созданные Microsoft, необходимо добавить политику с отключённым оповещением и/или мониторингом, которая проверяет, является ли производитель Microsoft.

    1. В разделе OS services monitoring для настраиваемого уровня выберите Add policy и определите политику — набор правил.
    2. System: выберите вашу операционную систему.
    3. Rule name: введите название, которое будет отображаться в поле Summary.
    4. Monitor: решите, хотите ли вы отслеживать доступность сервиса с помощью метрики OS service availability (builtin:osservice.availability). При наличии метрика отправляет статус сервиса каждые 10 секунд. Статус передаётся измерением Service status (dt.osservice.status). Обратите внимание, что метрика потребляет точки данных. Подробнее см. в разделе Metrics powered by Grail.
    5. Alert: решите, хотите ли вы оповещение для вашей политики.
    6. OneAgent версии 1.257+ Alert if service is not installed: получать ли оповещения об OS-сервисах, не установленных на хосте.
    7. Service status: установите статус сервиса, при котором должно срабатывать оповещение.

    Можно использовать логические операции для мониторинга статуса сервиса. Например, $eq(running) отслеживает состояние работающего сервиса.

    Доступные логические операции:

    • $not($eq(paused)) — соответствует сервисам в состоянии, отличном от «приостановлен».
    • $or($eq(paused),$eq(running)) — соответствует сервисам в состоянии «приостановлен» или «выполняется».

    Это статусы сервисов, которые можно отслеживать. Используйте одно из следующих значений в качестве параметра для этого условия:

    • running
    • stopped
    • start_pending
    • stop_pending
    • continue_pending
    • pause_pending
    • paused

    Можно использовать логические операции для мониторинга статуса сервиса. Например, $eq(active) отслеживает активное состояние сервиса.

    Доступные логические операции:

    • $not($eq(active)) — соответствует сервисам с состоянием, отличным от «active».
    • $or($eq(inactive),$eq(failed)) — соответствует сервисам в состоянии «inactive» или «failed».

    Это статусы сервисов, которые можно отслеживать. Используйте одно из следующих значений в качестве параметра для этого условия:

    • reloading
    • activating
    • deactivating
    • failed
    • inactive
    • active
    • Необязательно OneAgent версии 1.257+ Alerting delay: количество 10-секундных циклов измерения, в течение которых сервис должен находиться в настроенном состоянии, прежде чем будет создано событие. Не применяется к оповещениям о незаданных сервисах.

    1. Выбор сервисов для мониторинга

    2. Нажмите Add rule.

    3. Необязательно Rule scope: выберите OS Service или Host. По умолчанию выбран вариант OS Service.

    Если выбран Host:

    • OneAgent версии 1.277+ Custom metadata для сопоставления:

    • Key — ключ метаданных для сопоставления.

    • Condition — условие, в котором можно определить строку:

      • Dynatrace версии 1.310+ $match(ver*_1.2.?) — соответствует строке с подстановочными знаками. Используйте * для любого количества символов (включая ноль) и ? ровно для одного символа.
      • $contains(production) — соответствует, если «production» встречается в значении метаданных хоста.
      • $eq(production) — соответствует, если «production» точно совпадает со значением метаданных хоста.
      • $prefix(production) — соответствует, если «production» является префиксом значения метаданных хоста.
      • $suffix(production) — соответствует, если «production» является суффиксом значения метаданных хоста.

      Доступные логические операции:

      • $not($eq(production)) — соответствует, если значение метаданных хоста отличается от «production».
      • $and($prefix(production),$suffix(main)) — соответствует, если значение метаданных хоста начинается с «production» и заканчивается на «main».
      • $or($prefix(production),$suffix(main)) — соответствует, если значение метаданных хоста начинается с «production» или заканчивается на «main».

      При включении специальных символов, таких как скобки ( и ), в выражения для сопоставления экранируйте эти символы тильдой ~. Например, для сопоставления значения метаданных, содержащего скобки, например my(amazing)property, следует написать $eq(my~(amazing~)property).

      Условия нечувствительны к регистру.

    Дополнительная информация о Display name, Path, Manufacturer и Service Name

    С помощью этих свойств определяются сервисы для мониторинга на основе:

    • Отображаемого имени, видимого системному пользователю
    • Пути к бинарному файлу сервиса
    • Производителя сервиса
    • Имени сервиса — имени или идентификатора, под которым распознаётся OS-сервис

    • Condition — условие, в котором можно определить строку:

    • Начинается с — используйте квалификатор $prefix, например $prefix(ss).

    • Заканчивается на — используйте квалификатор $suffix, например $suffix(hd).
    • Равно — используйте квалификатор $eq, например $eq(sshd).
    • Содержит — используйте квалификатор $contains, например $contains(ssh).
    • Dynatrace версии 1.310+ Соответствует — используйте квалификатор $match, например $match(ip?tables*), где * соответствует любому количеству символов (включая ноль), а ? — ровно одному символу.

    Доступные логические операции:

    • $not($eq(sshd)) — соответствует, если значение свойства сервиса отличается от sshd.
    • $and($prefix(ss),$suffix(hd)) — соответствует, если значение свойства сервиса начинается с ss и заканчивается на hd.
    • $or($prefix(ss),$suffix(hd)) — соответствует, если значение свойства сервиса начинается с ss или заканчивается на hd.

    При включении специальных символов, таких как скобки ( и ), в выражения для сопоставления экранируйте эти символы тильдой ~. Например, для сопоставления значения свойства, содержащего скобки, например my(amazing)property, следует написать $eq(my~(amazing~)property).

    Условия нечувствительны к регистру.

    • Равно — используйте квалификатор $eq, например $eq(manual).

    • $not($eq(auto)) — соответствует сервисам с типом запуска, отличным от Automatic.

    • disabled для Disabled

    Управление отслеживаемыми OS-сервисами¶

    Для управления OS-сервисами

    1. В Dynatrace перейдите в OS services monitoring для уровня, который вы настраиваете.

    Уровень хоста

    1. Перейдите в Hosts Classic.
    2. Найдите и выберите хост для отображения страницы обзора хоста.

    3. В правом верхнем углу страницы обзора хоста выберите More (…) > Settings.

    4. В настройках хоста выберите OS services monitoring.

    Уровень группы хостов

    1. Перейдите в Deployment Status > OneAgents.
    2. На странице OneAgent deployment отключите Show new OneAgent deployments.

    3. В поле Filter by введите Host group и выберите нужную группу хостов из раскрывающегося списка.

      Список хостов теперь отфильтрован по выбранной группе. У каждого хоста в списке есть ссылка Host group: <group name>, где <group name> — это название группы хостов, которую вы хотите настроить.

      Свойство Host group не отображается, если выбранный хост не принадлежит ни одной группе хостов. 4. Выберите название группы хостов в любой строке.

      Поскольку вы отфильтровали по группе хостов, все отображаемые хосты принадлежат одной группе.

    4. В настройках группы хостов выберите OS services monitoring.

    Уровень среды

    Перейдите в Settings > Monitoring > OS services monitoring. 2. Отслеживаемые OS-сервисы отображаются в таблице под кнопкой Add policy.

    • Чтобы прекратить мониторинг сервиса из списка, отключите параметр Enabled.
    • Чтобы удалить сервис из таблицы, нажмите кнопку удаления в столбце Delete.
    • Для просмотра и редактирования деталей выберите элемент управления раскрытием в столбце Details.

    Мониторинг доступности сервиса¶

    Dynatrace версии 1.243+

    OneAgent версии 1.243+

    Страница обзора хоста содержит раздел OS services analysis со списком OS-сервисов, для которых выполняется любая политика (с активным оповещением или мониторингом).

    1. Перейдите в Hosts Classic.
    2. Выберите любой хост для перехода на его страницу обзора.
    3. В разделе OS services analysis выберите название сервиса для открытия страницы Service overview.

    Подробнее см. в разделе Анализ OS-сервисов.

    Настройка в масштабе с помощью Settings API¶

    Вы можете использовать Settings API для настройки мониторинга доступности сервисов в масштабе.

    1. Чтобы изучить схему, используйте GET a schema с builtin:os-services-monitoring в качестве schemaId.
    2. На основе схемы builtin:os-services-monitoring создайте объект конфигурации.
    3. Для создания конфигурации используйте POST an object.

    Ограничения¶

    На Linux поддерживаются OS-сервисы systemd со следующими типами запуска:

    • enabled
    • enabled-runtime
    • static
    • disabled
    • indirect
