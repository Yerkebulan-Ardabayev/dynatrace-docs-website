    Лимиты Log Monitoring по умолчанию (Logs Classic)¶

    • Classic
    • Чтение: 1 мин
    • Обновлено 2 фев. 2026

    Log Monitoring Classic

    На этой странице перечислены лимиты по умолчанию для последней версии Dynatrace Log Monitoring.

    Текущие ограничения применяются как к загрузке файлов логов, так и к загрузке логов общего назначения через API.

    Лимиты загрузки логов¶

    В таблице ниже приведены наиболее важные лимиты по умолчанию, связанные с загрузкой логов. Все представленные лимиты относятся к данным в кодировке UTF-8.

    Тип Лимит Описание
    Содержимое 8 КБ Максимальный размер тела записи лога
    Ключ атрибута 100 байт Ключ значения атрибута
    Длина значения атрибута 250 байт Максимальная длина значения атрибута
    Количество атрибутов лога 50 Максимальное количество атрибутов, которые может содержать лог
    Событий лога в минуту 1M/мин Максимальное количество событий лога в минуту
    Возраст лога 24 часа Максимальный возраст записей лога при загрузке
    Логи с будущими датами Без ограничений1 Насколько далеко в будущее могут простираться записи лога
    Значений на атрибут 32 значения Максимальное количество отдельных значений, которые может содержать атрибут
    Размер запроса 10 МБ Максимальный размер полезной нагрузки
    Количество записей лога 50 000 записей Максимальное количество записей лога на запрос
    Вложенные объекты 5 уровней Максимальное количество уровней для вложенных объектов

    1

    Ограничений на загрузку записей лога с будущими временными метками нет, но для записей с временными метками, опережающими текущее время более чем на 10 минут, временная метка устанавливается равной моменту загрузки.

    Неподдерживаемые сценарии автообнаружения¶

    Сценарии, которые не поддерживаются в процессе автообнаружения ротируемых логов, включают:

    • Генерация ротируемых логов со сменой директории. Этот процесс может привести к созданию множества неагрегированных и/или неполных логов, а также к чрезмерному потреблению ресурсов.
    • Генерация ротируемых логов с немедленным сжатием, когда приложение обращается к файлу с тем же именем. Если критерий ротации выполнен (например, достигнут требуемый размер файла), файл перемещается в другое место и немедленно сжимается. Пример: /var/log/application.log -> /var/log/application.log.1.gz -> /var/log/application.log.2.gz -> /var/log/application.log.3.gz. Этот процесс также может привести к созданию неполных логов.

    Ограничения автообнаружения логов при использовании OneAgent¶

    Файлы логов в OneAgent:

    • не могут быть удалены ранее чем через минуту после создания.
    • должны дополняться (старое содержимое не обновляется).
    • должны содержать текст.
    • должны быть открыты постоянно (а не только на короткие периоды добавления записей лога).
    • должны быть открыты в режиме записи.
    • не должны быть меньше настроенного порога размера (по умолчанию: 500 байт) для проверки на бинарное содержимое.

    Максимальное количество источников логов на экземпляр группы процессов по умолчанию равно 200. Это значение настраивается через параметр Maximum number of log sources per process group instance в Settings > Log Monitoring > Advanced log settings.

    В стандартных средах модуль логов OneAgent поддерживает до 10 000 файлов в одной директории с логами и 200 МБ нового содержимого логов в минуту. Если у вас больше данных, особенно на порядок больше, высока вероятность, что модуль логов OneAgent поддержит это. Обратитесь в службу поддержки Dynatrace для предварительной проверки вашей конфигурации.

    В особых случаях, таких как очень низкая производительность оборудования, ограничения модуля логов OneAgent могут быть более строгими.

    Допустимый временной диапазон записей лога¶

    Следующие правила применяются ко всем источникам событий лога, таким как OneAgent и API загрузки логов общего назначения.

    Атрибуты с высокой кардинальностью¶

    Уникальные атрибуты данных лога (атрибуты с высокой кардинальностью), такие как span_id и trace_id, генерируют избыточно длинные списки фасетов, которые могут повлиять на производительность средства просмотра логов. По этой причине они не отображаются в фасетах средства просмотра логов. Вы по-прежнему можете использовать их в расширенном поисковом запросе средства просмотра логов.

    Объекты запросов API загрузки логов¶

    В дополнение к общим ограничениям API Dynatrace, к API загрузки логов применяются следующие специфические ограничения:

    • Объект LogMessagePlain plain текстовый. Длина сообщения ограничена 8 192 символами. Любое содержимое, превышающее лимит, обрезается.

    • Объект LogMessageJson JSON. Объект должен содержать следующие группы ключей (возможные значения ключей перечислены ниже):

    • Временная метка. Поддерживаются следующие форматы: миллисекунды UTC, RFC3339 и RFC3164. Если не задана, используется текущая временная метка.

    • Серьёзность. Если не задана, используется NONE.
    • Содержимое. Если ключ содержимого не задан, весь JSON анализируется как содержимое.
    • Атрибуты. Поддерживаются строковые, числовые, логические значения или массив этих типов. Числа и логические значения преобразуются в строковое представление. Вложенные объекты разворачиваются (например, host: {name: "host1"} становится host.name). Семантические атрибуты и пользовательские атрибуты индексируются и могут использоваться в запросах. Они также отображаются в агрегациях (фасетах). Если встречается неподдерживаемый ключ, он не индексируется и не может использоваться в запросах и агрегациях. Подробнее см. в разделе Семантические атрибуты (Logs Classic).

    Длина значения ограничена для всех атрибутов. Любое содержимое, превышающее лимит, обрезается. Лимиты по умолчанию следующие:

    • Содержимое: 8 192 символа.
    • Значение атрибута: 250 символов.

    Ограничения маскирования конфиденциальных данных¶

    Обратите внимание на следующие ограничения маскирования конфиденциальных данных:

    • Если процесс маскирования занимает слишком много времени, затронутый файл лога блокируется до перезапуска OneAgent или любого изменения конфигурации, и вы получите сообщение File not monitored - incorrect sensitive data masking rule.

    Пропускная способность ActiveGate¶

    Если вы используете SaaS-эндпоинт, вам не нужно беспокоиться о пропускной способности ActiveGate. Пропускная способность такая же, как и для Grail. Если вы используете Environmental ActiveGate, пропускная способность составляет 3,3 ГБ/мин при RTT <= 200 мс.
