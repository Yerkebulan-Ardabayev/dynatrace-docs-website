    Маскировка конфиденциальных данных (Logs Classic)¶

    Маскировка конфиденциальных данных (Logs Classic)¶

    • 11 мин чтения
    • Обновлено 18 января 2023 г.

    Log Monitoring Classic

    Dynatrace версии 1.253+ OneAgent версии 1.243+

    Ваши данные логов содержат информацию, которая может считаться конфиденциальной. Определённые сообщения логов могут включать имена пользователей, адреса электронной почты, параметры URL и другую информацию, которую вы, возможно, не хотите раскрывать. Функция мониторинга логов позволяет маскировать любую информацию путём изменения файла конфигурации на каждом OneAgent, обрабатывающем информацию, которую вы считаете конфиденциальной.

    Для новейшей версии Dynatrace см. Маскировка конфиденциальных данных в OneAgent.

    Вы можете выбрать данные, которые нуждаются в защите, применив набор правил маскировки. В каждом правиле вы можете определить, что необходимо скрыть и чем заменить скрытое содержимое. Если необходимо обработать только определённые атрибуты, такие как предопределённые контейнеры, источники логов или группы процессов, вы можете добиться этого, добавив матчеры к правилам.

    Иерархия правил¶

    Выполнение правил маскировки происходит в заранее определённой иерархии, сверху вниз. Каждое последующее правило применяется к результату предыдущего правила. Иерархия следующая:

    1. Правила конфигурации хоста
    2. Правила конфигурации группы хостов
    3. Правила конфигурации среды

    Правила конфигурации хоста¶

    Правила конфигурации хоста доступны через Host settings для конкретного хоста.

    1. Перейдите в Hosts Classic.
    2. Найдите и выберите ваш хост для отображения страницы обзора хоста.

    3. В правом верхнем углу страницы обзора хоста выберите More (...) > Settings.

    4. В настройках хоста перейдите в Log Monitoring > Sensitive data masking.

    5. Настройте маскировку данных, добавив правила с набором матчеров, идентифицирующих ваши конфиденциальные данные.

    Правила конфигурации группы хостов¶

    Правила конфигурации группы хостов доступны через страницу Host.

    1. Перейдите в Hosts Classic и выберите интересующий вас хост.
    2. На странице обзора хоста выберите Properties and tags.
    3. На панели Properties and tags найдите свойство Host group, чтобы увидеть имя группы хостов, к которой принадлежит выбранный хост.

    Свойство Host group не отображается, если выбранный хост не принадлежит ни одной группе хостов. 4. Выберите имя группы хостов, чтобы увидеть список всех хостов в этой группе. Откроется страница OneAgent deployment, отфильтрованная по выбранной группе хостов. У каждого хоста в списке есть ссылка Host group: <group name>, где <group name> — имя группы хостов, которую вы хотите настроить. 5. Выберите имя группы хостов в любой строке.

    1. В настройках группы хостов выберите Log Monitoring > Sensitive data masking.
    2. Настройте маскировку данных, добавив правила с набором матчеров, идентифицирующих ваши конфиденциальные данные.

    Правила конфигурации среды¶

    Область тенанта доступна в меню настроек.

    1. Перейдите в Settings > Log Monitoring > Sensitive data masking.
    2. Настройте маскировку данных, добавив правила с набором матчеров, идентифицирующих ваши конфиденциальные данные.

    Создание правила¶

    Вы можете настроить маскировку конфиденциальных данных на уровне хоста, группы хостов или среды.

    1. Выберите Create new rule, чтобы начать настройку правила.
    2. Name: Отображаемое имя для вашей конфигурации.
    3. Search expression: Регулярное выражение для поиска строки, которую вы хотите замаскировать. Используйте формат регулярных выражений.

    4. Masking type: Вы можете заменить данные строкой или алгоритмом безопасного хеширования SHA-1.

    5. Если вы выберете replace with string, установите Replacement на строку, которая должна заменить конфиденциальные данные.

    6. Если вы выберете SHA-1, ваши данные будут заменены 40-символьной хеш-строкой.
    7. Выберите Add matcher, чтобы создать специфическое совпадение для этого правила и сузить область его применения. Вы можете включить несколько матчеров в одно правило. Например, правило маскировки может применяться к логам из определённого контейнера, пространства имён или источника логов.
    8. Выберите атрибут совпадения.
    9. Выберите Add value и из списка Values выберите обнаруженные элементы данных логов (файлы логов или группы процессов, содержащие данные логов). К выбранному оператору можно добавить несколько значений. У вас может быть один матчер, указывающий источник логов, который совпадает со значениями /var/log/syslog и Windows Application Log.
    10. Выберите Save changes.

    Определённые правила можно переупорядочить, и они выполняются в том порядке, в котором они отображаются на странице Sensitive data masking.

    Переключатель Active

    Начиная с OneAgent версии 1.249, вы можете активировать/деактивировать правила, включая/выключая переключатель Active. Для эффективного управления правилами мы рекомендуем обновить OneAgent до версии 1.249. Если у вас есть правила, установленные на хосте с OneAgent версии ниже 249, вы не сможете их деактивировать; в этом случае необходимо удалить такие правила, выбрав Delete на уровне правила или через REST API.

    REST API¶

    Вы можете использовать Settings API для управления конфигурацией маскировки конфиденциальных данных:

    • Просмотр схемы
    • Список сохранённых объектов конфигурации
    • Просмотр отдельного объекта конфигурации
    • Создание, редактирование или удаление объекта конфигурации

    Чтобы проверить текущую версию схемы для конфигурации маскировки конфиденциальных данных, перечислите все доступные схемы и найдите идентификатор схемы builtin:logmonitoring.sensitive-data-masking-settings. Объекты конфигурации маскировки конфиденциальных данных доступны для настройки в следующих областях:

    • tenant — объект конфигурации влияет на все хосты в данной среде.
    • host_group — объект конфигурации влияет на все хосты, назначенные данной группе хостов.
    • host — объект конфигурации влияет только на данный хост.

    Чтобы создать конфигурацию маскировки конфиденциальных данных с помощью API

    1. Создайте токен доступа с разрешениями Write settings (settings.write) и Read settings (settings.read).
    2. Используйте эндпоинт GET a schema, чтобы узнать формат JSON, необходимый для отправки конфигурации. Идентификатор схемы (schemaId) конфигурации маскировки конфиденциальных данных — builtin:sensitive-data-masking-settings. Вот пример JSON-нагрузки с конфигурацией маскировки конфиденциальных данных:
    [



{



"schemaId":"builtin:logmonitoring.sensitive-data-masking-settings",



"scope":"tenant",



"value":{



"config-item-title":"Added from REST API",



"masking":{



"expression":"run (\\d+?)",



"type":"STRING",



"replacement":"testing"



},



"matchers":[



{



"attribute":"log.source",



"operator":"MATCHES",



"values":[



"/var/log/syslog"



]



}



]



}



}



]

    Примеры SHA-1¶

    Вы можете маскировать такие данные, как номер кредитной карты или номер телефона, с указанием или без указания группы захвата.

    Маскировка номера кредитной карты¶

    В этом примере вы настроите правило маскировки конфиденциальных данных, нацеленное на номер кредитной карты в следующей записи лога:

    Username: John Doe, CreditCardNumber: 1234-1234-1234-1234

    Правило дополнительно ограничено файлами c:\inetpub\logs\LogFiles\ex_*.log в двух группах процессов: IIS (PROCESS_GROUP-3D9D854163F8F07A) и IIS (PROCESS_GROUP-4A7B47FDB53137AE).

    Перейдите в Settings > Log Monitoring > Sensitive data masking

    1. Выберите Create new rule и укажите имя конфигурации.
    2. Введите регулярное выражение для номера кредитной карты, например CreditCardNumber: (\d{4}-\d{4}-\d{4}-\d{4}).
    3. Выберите SHA-1 для типа маскировки.
    4. Выберите Add matcher.
    5. Из списка Attribute выберите Process group.
    6. В поле Value введите IIS и выберите IIS (PROCESS_GROUP-3D9D854163F8F07A) из списка предложений.
    7. В поле Value снова введите IIS и выберите вторую группу процессов из списка предложений: PROCESS_GROUP-4A7B47FDB53137AE.
    8. Снова выберите Add matcher.
    9. Выберите атрибут совпадения Log Source.
    10. Выберите Add value и введите c:\inetpub\logs\LogFiles\ex_*.log.
    11. Save changes.

    Маскируется только содержимое, найденное в группе захвата, и оно преобразуется в следующее:

    Username: John Doe, CreditCardNumber: 7e938e089861f3975b38cff3a93cc3aa659f7779

    Маскировка номера телефона¶

    В этом примере вы настроите правило маскировки конфиденциальных данных, нацеленное на все номера телефонов в следующей записи лога для всех файлов логов.

    Username: John Doe, PhoneNumber: +48123010100

    Перейдите в Settings > Log Monitoring > Sensitive data masking.

    1. Выберите Create new rule и укажите имя конфигурации.
    2. Введите регулярное выражение для номера телефона. Например, PhoneNumber: [0-9\-\+]{9,15}.
    3. Выберите SHA-1 для типа маскировки.
    4. Выберите Add matcher.
    5. Save changes.

    Группа захвата не указана, поэтому всё выражение целиком считается одной группой захвата и маскируется, преобразуясь в следующее во всех файлах логов:

    Username: John Doe, 011897d555c81e88f286cbb74c59f4ad99ec2f8d

    Примеры регулярных выражений¶

    Распространённые форматы регулярных выражений для конфиденциальных данных включают:

    Неподдерживаемые регулярные выражения¶

    Маскировка данных происходит в рамках всего выражения или группы захвата. Выражение должно соответствовать синтаксису механизма регулярных выражений и не может:

    • Быть частью более чем одной группы захвата
    • Содержать утверждение нулевой длины lookbehind в группе захвата
    • Содержать утверждение нулевой длины backreference в группе захвата
    • Содержать жадные квантификаторы (такие как x?, x* или x+) или сверхжадные квантификаторы (такие как x?+, x*+ или xx++). Используйте ленивые/неохотные квантификаторы (такие как x?? и x+?) вместо них.

    Часто задаваемые вопросы¶

    Где происходит маскировка конфиденциальных данных?

    Вы можете выполнять маскировку конфиденциальных данных в вашей среде, чтобы конфиденциальные данные не покидали вашу инфраструктуру без защиты. Если вы импортируете данные в Dynatrace через общий приём, вам необходимо маскировать конфиденциальные данные на уровне источника, до приёма. Альтернативно вы можете маскировать конфиденциальные данные во время обработки логов. Однако, если вы выберете маскировку данных во время обработки логов, ваши данные покинут вашу среду, поскольку обработка логов происходит на стороне Dynatrace. Поэтому безопаснее маскировать данные внутри вашей среды.

    Сколько групп захвата поддерживается?

    Одна. Если ни одна не указана, то вся область регулярного выражения считается одной группой захвата.

    Ограничения маскировки конфиденциальных данных¶

    Обратите внимание на следующие ограничения маскировки конфиденциальных данных:

    • Если процесс маскировки занимает слишком много времени, затронутый файл лога блокируется до перезапуска OneAgent или любого изменения конфигурации, после чего вы получите сообщение File not monitored - incorrect sensitive data masking rule.
