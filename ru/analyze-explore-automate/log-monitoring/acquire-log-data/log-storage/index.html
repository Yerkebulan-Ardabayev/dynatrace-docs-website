    • Tutorial

    Мониторинг журналов Classic

    Dynatrace версия 1.252+ OneAgent версия 1.243+

    Для самой новой Dynatrace версии см. Правила ингеста журналов.

    Если вы используете OneAgent версию ранее 1.243 и Dynatrace версию кластера ранее 1.252, перейдите к Источники журналов и хранилище.

    Dynatrace позволяет включать и исключать конкретные источники журналов для вашего анализа. Используя Dynatrace фреймворк управления идентификацией и доступом (IAM), вы можете контролировать, какой пользователь может изменять конфигурации в каком объеме.

    Конфигурация основана на правилах, которые используют сопоставители для иерархии, пути журнала и групп процессов. Эти правила определяют, какие файлы журнала среди тех, которые обнаружены OneAgent, либо автоматически, либо определены как пользовательские источники журналов, включаются.

    Правила ингеста журналов¶

    1. Перейдите в Настройки и выберите Мониторинг журналов > Правила ингеста журналов.
    2. Выберите Добавить правило и введите название для вашей конфигурации.
      По умолчанию кнопка Включить в хранилище включена, что указывает на то, что элементы, настроенные этим правилом, будут храниться в Dynatrace. Альтернативно вы можете выбрать тип правила Исключить из хранилища.
    3. Разверните Детали нового правила и выберите Добавить сопоставитель, чтобы создать конкретное совпадение для этого правила.
      В одно правило можно включать несколько сопоставителей.

    Кроме атрибута Источник журнала в Windows (из-за того, что пути к файлам не чувствительны к регистру), сопоставители чувствительны к регистру. 4. Выберите атрибут сопоставления:

    OneAgent версия 1.295+

    2

    Атрибут уровня записи журнала, преобразованный OneAgent, отличается от атрибута статуса журнала, преобразованного сервером Dynatrace.

    3

    OneAgent версия 1.273+

    4

    OneAgent версия 1.309+

    5

    Теги, применяемые вручную или автоматически, не видны OneAgent.

    6

    OneAgent версия 1.289+

    7

    Dynatrace Оператор версия 1.4.2+

    8

    OneAgent версия 1.305+

    Поддерживается использование символа подстановки для любого значения атрибута и его можно использовать несколько раз в одном значении. Однако некоторые атрибуты, например, Группа процессов, имеют ограниченный, предопределенный список возможных значений, которые выбираются из автозаполнения списка.

    Если в значении не используется символ подстановки, то сопоставитель ищет точное совпадение с значением. Если используется символ подстановки, сопоставитель ищет точное совпадение. Например, значение INFO приводит к отправке только данных журнала, имеющих точную строку INFO, но значение *INFO* (с использованием символов подстановки) соответствует данным журнала, содержащим строку INFO в своем содержании.

    1. Выберите Добавить значение и из Значений выберите обнаруженные элементы данных журнала (файлы журнала или группы процессов, содержащие данные журнала). К одному атрибуту можно добавить несколько значений. Вы можете иметь один сопоставитель, который указывает источник журнала и соответствует значениям /var/log/syslog и Windows Application Log.
    2. Сохранить изменения.

    Определенные правила можно переставлять и выполняются в порядке, в котором они появляются на странице Хранилище журналов.

    1. Чтобы активировать правило, включите переключатель Активно.

    Переключатель Активно

    Начиная с OneAgent версии 1.249, вы можете активировать/деактивировать свои правила, включая/выключая переключатель Активно. Чтобы эффективно управлять своими правилами, мы рекомендуем обновить OneAgent до версии 1.249. Если у вас есть какие-либо правила, установленные на хосте с OneAgent версии ранее 249, вы не сможете их деактивировать, в этом случае вам нужно удалить такие правила, выбрав Удалить на уровне правила или через REST API.

    Сопоставление списка правил с данными журнала¶

    Сопоставление происходит в предопределенной иерархии, и правила выполняются сверху вниз. Это означает, что если правило выше в списке соответствует определенным данным журнала, то нижние правила будут пропущены. Элементы, соответствующие более высоким конфигурациям, перезаписываются в более низких конфигурациях, если они соответствуют одинаковым данным журнала. Если не найдено ни одного правила, файл не отправляется. Иерархия сопоставления следующая:

    1. Правила конфигурации хоста
    2. Правила конфигурации группы хостов
    3. Правила конфигурации арендатора

    Объемы конфигурации¶

    Поддерживаются три объема иерархии: хост, группа хостов и арендатор. Объем с наименьшим возможным набором правил имеет приоритет над более крупными наборами.

    1. Правила хранилища журналов, настроенные для хоста, имеют приоритет над правилами хранилища журналов, настроенными для группы хостов.
    2. Правила хранилища журналов, настроенные для группы хостов, имеют приоритет над правилами хранилища журналов, настроенными для арендатора.

    Объем хоста¶

    Объем хоста можно получить через Настройки хоста для конкретного хоста.

    1. Перейдите в Хосты Classic.
    2. Найдите и выберите свой хост, чтобы отобразить страницу обзора хоста.

    3. В правом верхнем углу страницы обзора хоста выберите Еще (…) > Настройки.

    4. Из настроек хоста перейдите в Мониторинг журналов > Правила ингеста журналов.

    5. Настройте загрузку хранилища, добавив правила с набором атрибутов, соответствующих данным журнала, которые будут храниться Dynatrace.

    Объем группы хостов¶

    Объем группы хостов можно получить через страницу Хост.

    1. Перейдите в Хосты Classic и выберите хост, который вас интересует.
    2. На странице обзора хоста выберите Свойства и теги.
    3. На панели Свойства и теги найдите свойство Host Group, чтобы увидеть имя группы хостов, к которой принадлежит выбранный хост.

    Свойство Host Group не отображается, когда выбранный хост не принадлежит ни к одной группе хостов. 4. Выберите имя группы хостов, чтобы перечислить все хосты в этой группе хостов. Это отображает страницу OneAgent развертывания, отфильтрованную по выбранной группе хостов. Каждый перечисленный хост имеет ссылку Host Group: <имя группы>, где <имя группы> — имя группы хостов, которую вы хотите настроить. 5. Выберите имя группы хостов в любой строке.

    1. В настройках группы хостов выберите Мониторинг журналов > Правила ингеста журналов.
    2. Настройте загрузку хранилища, добавив правила с набором атрибутов, соответствующих данным журнала, которые будут храниться Dynatrace.

    Объем арендатора¶

    Объем арендатора доступен в меню настроек.

    1. Перейдите в Настройки и выберите Мониторинг журналов > Правила ингеста журналов.
    2. Настройте загрузку хранилища, добавив правила с набором атрибутов, соответствующих данным журнала, которые будут храниться Dynatrace.

    Список хостов и групп хостов с переопределенными правилами¶

    Таблица на Настройки > Мониторинг журналов > Правила ингеста журналов перечисляет все правила хранилища журналов, которые вы установили на уровне арендатора. Однако вы можете захотеть увидеть, где вы установили правила хранилища журналов для хостов и групп хостов, которые переопределяют правила уровня арендатора.

    Чтобы перечислить все сущности (хосты и группы хостов), для которых применены более конкретные правила хранилища журналов

    1. Перейдите в Настройки > Мониторинг журналов > Правила ингеста журналов.
    2. В правом верхнем углу страницы Правила ингеста журналов выберите Еще (…) > Иерархия и переопределения. Поисковая панель Иерархия и переопределения перечисляет все сущности (хосты и группы хостов), для которых вы установили правила хранилища журналов, которые переопределяют правила уровня арендатора, перечисленные на Настройки > Мониторинг журналов > Правила ингеста журналов.
    3. Выберите имя сущности, чтобы перейти на страницу Правила ингеста журналов этой сущности.

    Пример загрузки¶

    В этом примере мы настраиваем загрузку хранилища арендатора для файлов c:\inetpub\logs\LogFiles\ex_*.log в двух группах процессов: IIS (PROCESS_GROUP-3D9D854163F8F07A) и IIS (PROCESS_GROUP-4A7B47FDB53137AE). Правило хранилища журналов состоит из двух сопоставителей: первый сопоставитель находит группы процессов, а второй сопоставитель соответствует только определенным источникам журнала.

    1. Перейдите в Настройки > Мониторинг журналов > Правила ингеста журналов.
    2. Выберите Добавить правило и введите название для вашей конфигурации.
    3. Выберите Добавить сопоставитель. Это первый сопоставитель для соответствия двум указанным группам процессов.
    4. Из списка Атрибут выберите Группа процессов.
    5. Выберите Добавить значение и введите IIS, затем из списка предложений выберите IIS (PROCESS_GROUP-3D9D854163F8F07A).
    6. Выберите Добавить значение еще раз, введите IIS и выберите вторую группу процессов из списка предложений: IIS (PROCESS_GROUP-4A7B47FDB53137AE).
    7. Выберите Добавить сопоставитель еще раз. Это второй сопоставитель для соответствия указанным данным журнала.
    8. Из списка Атрибут выберите Источник журнала.
    9. Выберите Добавить значение и введите c:\inetpub\logs\LogFiles\ex_*.log в качестве значения.
    10. Сохранить изменения.

    Пример исключения¶

    В этом примере мы настраиваем загрузку хранилища арендатора для всех источников журналов, кроме файлов c:\inetpub\logs\LogFiles\ex_*.log в группе процессов IIS (PROCESS_GROUP-4A7B47FDB53137AE).

    1. Перейдите в Настройки и выберите Мониторинг журналов > Правила приема журналов.
    2. Выберите Добавить правило и предоставьте название для вашей конфигурации.
    3. Отключите Отправить в хранилище.
    4. Выберите Добавить сопоставитель. Это первый сопоставитель для сопоставления указанной группы процессов.
    5. Из списка Атрибут выберите Группа процессов.
    6. Выберите Добавить значение и введите IIS, затем из списка предложений выберите IIS (PROCESS_GROUP-3D9D854163F8F07A).
    7. Выберите Добавить сопоставитель еще раз. Это второй сопоставитель для исключения указанного источника данных журнала.
    8. Из списка Атрибут выберите Источник журнала.
    9. Выберите Добавить значение и введите c:\inetpub\logs\LogFiles\ex_*.log в качестве значения.
    10. Сохраните изменения.

    REST API¶

    Вы можете использовать Настройки API, чтобы управлять своими правилами приема журналов:

    • Просмотреть схему
    • Перечислить сохраненные объекты конфигурации
    • Просмотреть один объект конфигурации
    • Создать новый, изменить или удалить существующий объект конфигурации

    Чтобы проверить текущую версию схемы для правил приема журналов, перечислите все доступные схемы и найдите идентификатор схемы builtin:logmonitoring.log-storage-settings.

    Правила приема журналов можно настроить для следующих областей:

    • tenant – объект конфигурации влияет на все хосты в данном арендаторе.
    • host_group – объект конфигурации влияет на все хосты, назначенные данной группе хостов.
    • host – объект конфигурации влияет только на данный хост.

    Чтобы создать правило приема журналов с помощью API:

    1. Создайте токен доступа с разрешениями Запись настроек (settings.write) и Чтение настроек (settings.read).
    2. Используйте GET-схему конечную точку, чтобы узнать формат JSON, необходимый для отправки вашей конфигурации. Идентификатор схемы конфигурации хранилища журналов (schemaId) равен builtin:logmonitoring.log-storage-settings. Вот пример полезной нагрузки JSON с конфигурацией хранилища журналов:

"schemaId": "builtin:logmonitoring.log-storage-settings",


"schemaVersion": "0.1.0",


"scope": "tenant",


"value": {


"config-item-title": "Добавлено из REST API",


"send-to-storage": true,

{


"attribute": "dt.entity.process_group",

"values": [


"PROCESS_GROUP-05F00CBACF39EBD1"

{

"Windows System Log",


"Windows Security Log"


]

]
    3. Используйте POST-объект конечную точку, чтобы отправить вашу конфигурацию.

    Пример 1: Несколько правил¶

    В этом примере есть два правила:

    • Правило 1 – исключающее правило и имеет два сопоставителя: атрибут группы процессов равен Apache, а атрибут источника журнала равен access.log).
    • Правило 2 – включающее правило и имеет один сопоставитель: атрибут группы процессов равен Apache.

    Будут ли работать более старые OneAgent с этим решением?

    Версии OneAgent, более ранние чем 1.243, не будут отправлять какие-либо данные; они получат пустой список разрешений в ответ.

    Почему я не вижу никакой конфигурации на глобальной странице после миграции с точки зрения хостов?

    Все конфигурации с точки зрения хостов мигрируются в соответствующий объем хоста.

    Являются ли правила ингеста журналов одинаковыми с/частью процесса автоподбора?

    Нет. Автоподбор — это механизм OneAgent, который обнаруживает журналы, но это не означает, что файлы журнала автоматически отправляются на хранилище. Страница конфигурации для автоподбора запланирована для будущего выпуска. Чтобы узнать больше об автоподборе, см. [Автоподбор содержимого журнала (Классические журналы).

    Важен ли порядок элементов конфигурации?

    Да, элементы конфигурации сопоставляются сверху вниз, то есть верхнее значение является наиболее важным.

    Как долго мне нужно ждать, чтобы конфигурация была применена к хосту?

    Она применяется в течение 90 секунд.

    Уменьшает ли добавление сопоставителя содержимого количество событий журнала, отправляемых в Dynatrace?

    Да. Сопоставитель содержимого сужает объем событий журнала (записей журнала) в соответствии с заданными критериями (например, поиск только ошибок журнала).

    Где выполняется фильтрация, в Dynatrace и/или в OneAgent?

    • Фильтрация (сужение объема в соответствии с заданными критериями) выполняется в OneAgent.
    • Установка ограничений (например, ограничение событий журнала в минуту или ограничение значений атрибутов) проводится в Dynatrace.

    Уменьшает ли фильтрация содержимого стоимость DDU и/или использование сети?

    Да. Фильтрация содержимого, проводимая на OneAgent, уменьшает как стоимость DDU, так и использование сети. Вы можете рассчитать снижение стоимости и использования сети, определив общее потребление данных и вычитая размер данных в гигабайтах, которые были отфильтрованы.
