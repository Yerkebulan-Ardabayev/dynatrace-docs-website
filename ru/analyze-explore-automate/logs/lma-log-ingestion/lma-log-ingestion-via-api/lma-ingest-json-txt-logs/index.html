    Приём логов в формате JSON и TXT¶

    • Объяснение

    API приёма логов принимает логи в форматах JSON, TXT и OTLP. На этой странице мы опишем форматы JSON и текстовый. Документацию по OTLP см. в разделе форматов OTLP.

    API приёма логов отвечает за сбор данных и их пакетную пересылку в Dynatrace.

    • Эндпоинты Dynatrace: https://{your-environment-id}.live.dynatrace.com/api/v2/logs/ingest. Эндпоинт API приёма логов доступен в вашей среде Dynatrace.
    • Эндпоинты ActiveGate среды: https://{your-activegate-domain}:9999/e/{your-environment-id}/api/v2/logs/ingest. API приёма логов автоматически включается после установки ActiveGate.

    Подробности о поддерживаемых полезных нагрузках, аутентификации, параметрах и объектах тела запроса см. в разделе Log Monitoring API v2 - POST ingest logs.

    Подробности об ограничениях см. в разделе Ограничения по умолчанию Log Management and Analytics.

    Преобразование данных и автоматический разбор JSON¶

    API приёма логов собирает и пытается автоматически преобразовать данные логов. Каждая запись лога из принятого пакета сопоставляется с одной записью лога Dynatrace, которая содержит три специальных атрибута: timestamp, loglevel, content и атрибуты типа ключ-значение. Эти четыре свойства устанавливаются на основе ключей, присутствующих во входном JSON-объекте, следующим образом.

    Временная метка¶

    • timestamp устанавливается на основе значения первого найденного ключа из следующего списка, оцениваемого в указанном порядке, без учёта регистра: timestamp, @timestamp, _timestamp, eventtime, date, published_date, syslog.timestamp, time, epochSecond, startTime, datetime, ts, timeMillis, @t.
    • Поддерживаемые форматы: миллисекунды UTC, RFC3339 и RFC3164.

    Для неподдерживаемых форматов временных меток используется текущая временная метка, а значение неподдерживаемого формата сохраняется в атрибуте unparsed_timestamp. * Записи логов старше лимита возраста лога отбрасываются. Временные метки, опережающие текущее время более чем на 10 минут, заменяются текущим временем. * Если в записи лога нет поддерживаемого ключа временной метки, значением по умолчанию является текущая временная метка. * Если в временной метке нет часового пояса, часовой пояс по умолчанию — UTC.

    Уровень логирования¶

    • loglevel устанавливается на основе значения первого найденного ключа из следующего списка, оцениваемого в указанном порядке, без учёта регистра: loglevel, status, severity, level, syslog.severity.
    • Значение по умолчанию — NONE.

    Содержимое¶

    • content устанавливается на основе значения первого найденного ключа из следующего списка, оцениваемого в указанном порядке, без учёта регистра: content, message, payload, body, log, _raw (поддерживается только в модели необработанных данных).
    • Значение по умолчанию и обработка зависят от модели данных, используемой для обработки входных данных.

    Атрибуты¶

    • Атрибуты лога содержат все остальные ключи из входного JSON-объекта, кроме тех, которые используются для timestamp, loglevel и content.
    • Атрибуты первого уровня предпочтительно должны соответствовать семантическим атрибутам, чтобы Dynatrace мог сопоставить их с контекстом. Все атрибуты могут использоваться в запросах, хотя Semantic Dictionary помогает ИИ Dynatrace Intelligence в интерпретации логов. Подробнее см. Semantic Dictionary.
    • Автоматический атрибут. Атрибут dt.auth.origin автоматически добавляется к каждой записи лога, принятой через API. Этот атрибут является публичной частью API-ключа, которым источник лога авторизуется для подключения к универсальному API приёма логов.

    Обработка атрибутов различается в зависимости от типа тенанта и среды:

    • Логи в Grail с пользовательской обработкой OpenPipeline (Dynatrace версия 1.295+, ActiveGate среды версия 1.295+): Поддерживает расширенные типы данных, позволяя использовать разнообразные атрибуты в запросах. Ключи чувствительны к регистру.
    • Логи в Grail с OpenPipeline, направленные в Classic Pipeline: Все ключи атрибутов приводятся к нижнему регистру, а все значения атрибутов преобразуются в строки. Все атрибуты могут использоваться в запросах.

    Модели данных¶

    Существуют две модели данных, определяющие способ обработки структурированных логов эндпоинтами приёма логов: необработанная (raw) и выровненная (flattened). Разница между ними заключается в способе преобразования атрибутов с объектными значениями.

    Если этот параметр конфигурации не указан, поведение по умолчанию зависит от времени создания вашей среды.

    • Для версии Dynatrace 1.331+: Необработанная (Raw).
    • Для версий Dynatrace 1.330 и ранее: Выровненная (Flattened).

    Экранирование в примерах вывода используется только для визуализации. \" считается одним символом при тарификации.

    Модель необработанных данных (Raw)¶

    Модель необработанных данных сохраняет исходную структуру и контекст лога, обеспечивая целостность данных. Это обеспечивает удобное взаимодействие и выполнение запросов, поскольку представление записи лога в Dynatrace остаётся таким же, как в источнике.

    Мы рекомендуем использовать этот подход для глубоко вложенных JSON-логов, так как он сохраняет семантический смысл и связи между точками данных.

    При использовании средств пересылки логов, таких как Fluentbit, Fluentd или Logstash, избегайте использования JSON-парсеров на стороне средства пересылки и позвольте Dynatrace выполнить разбор JSON самостоятельно. Этот подход снижает нагрузку на средство пересылки логов и обеспечивает единообразное поведение разбора.

    Модель необработанных данных преобразует содержимое структурированных логов, как описано в разделах ниже.

    Атрибуты с непримитивными типами¶

    Типы объектных атрибутов сохраняются как JSON-строки. Последующие этапы приёма Dynatrace (OpenPipeline, приложение Logs) поддерживают этот формат для удобной обработки и анализа логов.

    Типы массивов сохраняются как массивы, но содержащиеся типы приводятся к единому типу.

    • Сложные значения (такие как массивы или объекты) сопоставляются со строковыми значениями JSON.
    • Если какое-либо значение в массиве является строкой, или если какое-либо значение должно быть преобразовано в строку (например, объект или массив), целевым типом всего массива является строка.
    • Если все значения в исходном массиве числовые, целевой тип массива — числовой.
    • Значения null считаются совместимыми с любым типом.

    Обработка содержимого¶

    Выбранное входное поле содержимого всегда выбирается независимо от его типа и при необходимости преобразуется в строковый тип.

    Первый атрибут из списка поддерживаемых атрибутов содержимого выбирается для выходного поля content.

    Атрибут _raw используется как content, только если нет атрибута содержимого с более высоким приоритетом.

    Модель выровненных данных (Flattened)¶

    Модель выровненных данных обеспечивает прямой доступ к значениям атрибутов через простые пути к ключам.

    Этот подход предоставляется для обратной совместимости. Он также может подходить для определённых случаев использования, например, когда все вложенные значения JSON должны быть доступны на корневом уровне.

    Атрибуты с непримитивными типами¶

    В модели выровненных данных вложенные объекты в атрибутах лога преобразуются в плоские пары значений.

    Когда атрибут лога содержит объект, каждое вложенное свойство становится отдельным атрибутом. Этот процесс работает для атрибутов до пятого уровня вложенности, а атрибуты глубже этого уровня пропускаются.

    Типы массивов сохраняются как массивы, но содержащиеся типы приводятся к единому типу.

    • Сложные значения (такие как массивы или объекты) сопоставляются со строковыми значениями JSON.
    • Если какое-либо значение в массиве является строкой, или если какое-либо значение должно быть преобразовано в строку (например, объект или массив), целевым типом всего массива является строка.
    • Если все значения в исходном массиве числовые, целевой тип массива — числовой.
    • Значения null считаются совместимыми с любым типом.

    Поведение, связанное с содержимым¶

    Приведённые ниже правила определяют, как выбирается и формируется поле content.

    Поддерживаемый атрибут содержимого не найден¶

    Если поддерживаемый атрибут содержимого не найден, полное JSON-представление события лога устанавливается как поле content выходной записи лога. Исходный JSON сохраняется как есть.

    Поле _raw не входит в число поддерживаемых полей содержимого для этой модели данных.

    Сложные значения в поддерживаемых атрибутах содержимого¶

    Любой атрибут, являющийся объектом, включая content, обрабатывается как стандартный атрибут.

    Обработка атрибутов API приёма логов¶

    API приёма логов дополнительно принимает атрибуты лога через:

    • Параметры запроса
    • Специальный заголовок: X-Dynatrace-Attr

    Эти атрибуты объединяются с атрибутами, предоставленными в теле записи лога, в соответствии с правилами, описанными ниже.

    Атрибуты параметров запроса¶

    • Все параметры запроса, переданные эндпоинту API приёма логов, добавляются к атрибутам тела записи лога.
    • Если ключ параметра встречается несколько раз, все значения фиксируются как атрибут-массив.
    • Ключи и значения следуют тем же правилам разбора атрибутов, что и атрибуты тела.
    • Определённые параметры обрабатываются API для внутренних целей и никогда не появляются как атрибуты записи лога, даже если они явно указаны (например, используемые в заголовке X-Dynatrace-Options). Полный список зарезервированных имён параметров и их поведение при обработке см. в документации API.

    Пример¶

    Атрибуты на основе заголовка (X-Dynatrace-Attr)¶

    API поддерживает специальный заголовок для передачи дополнительных атрибутов:

    X-Dynatrace-Attr: region=eu-central-1&team=core

    Правила:

    • Ключи и значения следуют тем же правилам разбора атрибутов, что и параметры запроса.
    • Поддержка многозначности также работает внутри атрибутов заголовка.
    • Применяются те же ограничения на зарезервированные имена параметров.

    Правила приоритета атрибутов¶

    Когда атрибуты появляются в нескольких местах, API приёма логов применяет приоритет атрибутов, сохраняя при этом значения тела для возможности аудита. Атрибуты применяются в следующем порядке:

    • Параметры запроса (наивысший приоритет)
    • Заголовок X-Dynatrace-Attr
    • Тело записи лога (наименьший приоритет; существующий путь приёма)

    Поведение при переопределении¶

    Когда атрибуты из параметров запроса или заголовка переопределяют атрибуты тела:

    • Окончательное значение атрибута устанавливается в соответствии с правилами приоритета источников атрибутов.
    • Значения, уже присутствующие в теле лога, сохраняются и дублируются под overwrittenN.<attribute_key>. Где N — возрастающее целое число (1, 2, ...) в зависимости от того, сколько значений из тела нужно было сохранить. Это обеспечивает уникальность даже при множественных конфликтах.
    • Только значения, происходящие из тела лога, сохраняются под ключами overwrittenN.*. Атрибуты, переопределённые источниками с более высоким приоритетом, не генерируют перезаписанных копий.

    Поведение при тарификации¶

    Атрибуты, предоставленные через параметры запроса или заголовки, включаются в расчёт тарификации.

    Для многозначных атрибутов ключ атрибута учитывается в тарификации только один раз, независимо от количества присутствующих значений.
