    Маскирование конфиденциальных данных в OneAgent

    • Latest Dynatrace
    • 13 мин. чтения

    Ваши данные логов содержат информацию, которая может считаться конфиденциальной. Определённые сообщения логов могут включать имена пользователей, адреса электронной почты, параметры URL и другую информацию, которую вы, возможно, не хотите раскрывать. Функция мониторинга логов позволяет маскировать любую информацию путём изменения файла конфигурации на каждом OneAgent, обрабатывающем информацию, которую вы считаете конфиденциальной.

    Маскирование выполняется непосредственно в OneAgent, гарантируя, что конфиденциальные данные никогда не будут приняты системой.

    Вы можете выбрать данные, которые необходимо защитить, применяя набор правил маскирования. В каждом правиле вы можете определить, что скрывать, и чем заменить скрытый контент. Если вам нужно обработать только определённые атрибуты, такие как предопределённые контейнеры, источники логов или группы процессов, вы можете добиться этого, добавив условия соответствия в ваши правила.

    Правила по умолчанию¶

    Модуль логов OneAgent поставляется со встроенными правилами маскирования, которые обрабатывают конфиденциальные данные в логах, такие как номера платёжных карт и адреса электронной почты.

    • OneAgent версии 1.330 и ниже: данные маскируются строкой <masked>.
    • OneAgent версии 1.331+: строка маскирования указывает тип данных. Например, значения параметров URL-запросов маскируются строкой <masked-value-log>.

    По умолчанию эти правила деактивированы во всех платных (не пробных) средах Dynatrace. Вы можете активировать и деактивировать правила маскирования в соответствии с вашими требованиями.

    Создание правила¶

    Вы можете настроить маскирование конфиденциальных данных на уровне хоста, группы хостов или среды.

    1. Перейдите в Settings > Log Monitoring > Sensitive data masking и выберите Add rule для начала настройки правила.
    2. Имя правила: имя, отображаемое для вашей конфигурации.
    3. Выражение поиска: регулярное выражение для поиска строки, которую вы хотите маскировать. Используйте формат регулярных выражений.
    4. Выберите Проверить регулярное выражение. Введите образцы логов для проверки регулярного выражения и выберите Проверить для просмотра результата.

    5. Тип маскирования: вы можете заменить данные строкой или хешем Secure Hash Algorithm 256 (SHA-256) (SHA-1 устарел).

    6. Если вы выберете SHA-256, ваши данные будут заменены 40-символьной хеш-строкой.

    7. Если вы выберете заменить строкой, задайте Замену — строку, которая заменит ваши конфиденциальные данные.
    8. Выберите Добавить условие для создания конкретного соответствия для этого правила и сужения области действия правила. Вы можете включить несколько условий соответствия в одно правило. Например, правило маскирования может применяться к логам из определённого контейнера, пространства имён или источника логов.
    9. Выберите атрибут соответствия.

    1

    Минимальная требуемая версия OneAgent — 1.295.

    2

    Теги, применённые вручную или автоматически, не видны для OneAgent.

    3

    Минимальная требуемая версия OneAgent — 1.289.

    4

    Минимальная требуемая версия модуля логов OneAgent — 1.309.

    5

    Минимальная требуемая версия Dynatrace Operator — 1.4.2. 8. Выберите Добавить значение и выберите обнаруженные элементы данных логов из списка Значения (файлы логов или группы процессов, содержащие данные логов). К выбранному оператору можно добавить несколько значений. Вы можете создать одно условие соответствия, указывающее источник логов, и сопоставить значения /var/log/syslog и Windows Application Log. 9. Выберите Сохранить изменения.

    Определённые правила можно переупорядочивать, и они выполняются в порядке, в котором они отображаются на странице Маскирование конфиденциальных данных.

    Ограничения конфигурации¶

    Вы можете маскировать максимум 256 объектов на область. Все правила соответствия для источника логов делят общий лимит времени выполнения в 10 секунд. При превышении этого лимита источник логов отключается до перезапуска агента логов или обновления конфигурации.

    Иерархия правил¶

    Выполнение правил маскирования происходит в заданной иерархии, сверху вниз. Каждое последующее правило применяется к результату предыдущего правила. Иерархия следующая:

    1. Правила конфигурации хоста
    2. Правила конфигурации группы хостов
    3. Правила конфигурации среды

    Правила конфигурации хоста¶

    Правила конфигурации хоста доступны через Настройки хоста для конкретного хоста.

    1. Перейдите в Hosts Classic.
    2. Найдите и выберите ваш хост для отображения страницы обзора хоста.

    3. В правом верхнем углу страницы обзора хоста выберите Ещё (...) > Настройки.

    4. В настройках хоста перейдите в Log Monitoring > Sensitive data masking.

    5. Настройте маскирование данных, добавив правила с набором условий соответствия, идентифицирующих ваши конфиденциальные данные.

    Правила конфигурации группы хостов¶

    Правила конфигурации группы хостов доступны через страницу Host.

    1. Перейдите в Hosts Classic и выберите интересующий вас хост.
    2. На странице обзора хоста выберите Свойства и теги.
    3. На панели Свойства и теги найдите свойство Группа хостов, чтобы увидеть имя группы хостов, к которой принадлежит выбранный хост.

    Свойство Группа хостов не отображается, если выбранный хост не принадлежит ни к одной группе хостов. 4. Выберите имя группы хостов, чтобы отобразить все хосты в этой группе. Откроется страница Развёртывание OneAgent, отфильтрованная по выбранной группе хостов. Каждый перечисленный хост имеет ссылку Группа хостов: <имя группы>, где <имя группы> — имя группы хостов, которую вы хотите настроить. 5. Выберите имя группы хостов в любой строке.

    1. В настройках группы хостов выберите Log Monitoring > Sensitive data masking.
    2. Настройте маскирование данных, добавив правила с набором условий соответствия, идентифицирующих ваши конфиденциальные данные.

    Правила конфигурации среды¶

    Область среды доступна в меню настроек.

    1. Перейдите в Settings и выберите Log Monitoring > Sensitive data masking.
    2. Настройте маскирование данных, добавив правила с набором условий соответствия, идентифицирующих ваши конфиденциальные данные.

    REST API¶

    Вы можете использовать Settings API для управления конфигурацией маскирования конфиденциальных данных:

    • Просмотр схемы
    • Получение списка сохранённых объектов конфигурации
    • Просмотр отдельного объекта конфигурации
    • Создание, редактирование или удаление объекта конфигурации

    Чтобы проверить текущую версию схемы для конфигурации маскирования конфиденциальных данных, получите список всех доступных схем и найдите идентификатор схемы builtin:logmonitoring.sensitive-data-masking-settings. Объекты конфигурации маскирования конфиденциальных данных доступны для настройки в следующих областях:

    • tenant — объект конфигурации влияет на все хосты в данной среде.
    • host_group — объект конфигурации влияет на все хосты, назначенные данной группе хостов.
    • host — объект конфигурации влияет только на данный хост.

    Для создания конфигурации маскирования конфиденциальных данных с помощью API:

    1. Создайте токен доступа с разрешениями Write settings (settings.write) и Read settings (settings.read).
    2. Используйте эндпоинт GET схемы для изучения формата JSON, необходимого для отправки конфигурации. Идентификатор схемы (schemaId) конфигурации маскирования конфиденциальных данных — builtin:sensitive-data-masking-settings. Пример JSON-тела с конфигурацией маскирования конфиденциальных данных:
    [


{


"schemaId":"builtin:logmonitoring.sensitive-data-masking-settings",


"scope":"tenant",


"value":{


"config-item-title":"Added from REST API",


"masking":{


"expression":"run (\\d+?)",


"type":"STRING",


"replacement":"testing"


},


"matchers":[


{


"attribute":"log.source",


"operator":"MATCHES",


"values":[


"/var/log/syslog"


]


}


]


}


}


]

    Примеры SHA-256¶

    Вы можете маскировать такие данные, как номер кредитной карты или номер телефона, с указанием или без указания группы захвата.

    Маскирование номера кредитной карты¶

    В этом примере вы настроите правило маскирования конфиденциальных данных, нацеленное на номер кредитной карты в следующей записи лога:

    Username: John Doe, CreditCardNumber: 1234-1234-1234-1234

    Правило дополнительно сужено до файлов c:\inetpub\logs\LogFiles\ex_*.log в двух группах процессов: IIS (PROCESS_GROUP-3D9D854163F8F07A) и IIS (PROCESS_GROUP-4A7B47FDB53137AE).

    Перейдите в Settings и выберите Log Monitoring > Sensitive data masking

    1. Выберите Создать новое правило и укажите имя конфигурации.
    2. Укажите регулярное выражение для номера кредитной карты, например CreditCardNumber: (\d{4}-\d{4}-\d{4}-\d{4}).
    3. Выберите SHA-256 для типа маскирования.
    4. Выберите Добавить условие.
    5. Из списка Атрибут условия выберите Группа процессов.
    6. В поле Значение введите IIS и выберите IIS (PROCESS_GROUP-3D9D854163F8F07A) из списка подсказок.
    7. В поле Значение снова введите IIS и выберите вторую группу процессов из списка подсказок: PROCESS_GROUP-4A7B47FDB53137AE.
    8. Выберите Добавить условие снова.
    9. Выберите атрибут условия Источник логов.
    10. Выберите Добавить значение и введите c:\inetpub\logs\LogFiles\ex_*.log.
    11. Сохраните изменения.

    Маскируется только содержимое, найденное в группе захвата, и оно преобразуется в следующее:

    Username: John Doe, CreditCardNumber: 7e938e089861f3975b38cff3a93cc3aa659f7779

    Маскирование номера телефона¶

    В этом примере вы настроите правило маскирования конфиденциальных данных, нацеленное на все номера телефонов в следующей записи лога для всех файлов логов.

    Username: John Doe, PhoneNumber: +48123010100

    Перейдите в Settings и выберите Log Monitoring > Sensitive data masking.

    1. Выберите Создать новое правило и укажите имя конфигурации.
    2. Укажите регулярное выражение для номера телефона. Например, PhoneNumber: [0-9\-\+]{9,15}.
    3. Выберите SHA-256 для типа маскирования.
    4. Выберите Добавить условие.
    5. Сохраните изменения.

    Группа захвата не указана, поэтому всё выражение рассматривается как одна группа захвата и маскируется, преобразуясь в следующее во всех файлах логов:

    Username: John Doe, 011897d555c81e88f286cbb74c59f4ad99ec2f8d

    Расширенные примеры SHA-256¶

    В приведённых ниже примерах вы можете увидеть, как маскируются различные комбинации конфиденциальных данных. Вы можете использовать указанные JSON-файлы в REST API или вводить указанные правила маскирования, условия, регулярные выражения и атрибуты непосредственно при создании правил через веб-интерфейс Dynatrace.

    Маскирование номеров кредитных карт и электронной почты¶

    Для маскирования всех номеров кредитных карт и адресов электронной почты в вашем контенте необходимо создать два отдельных правила, каждое с отдельным условием:

    {


"masking": {


"expression": "(\\d{4}-\\d{4}-\\d{4}-\\d{4})",


"type": "STRING",


"replacement": "MaskedCreditCardNumber"


},


"matchers": [


],


"enabled": true


},


{


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


],


"enabled": true


}

    Маскирование логов Apache¶

    Для маскирования логов, написанных Apache И имя файла логов которых error.log, вы можете создать одно правило с двумя условиями:

    {


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "log.source",


"values": [


"/path/to/error.log"


]


},


{


"attribute": "dt.entity.process_group",


"values": [


"PROCESS_GROUP-APACHEID"


]


}


],


"enabled": true


}

    Для маскирования логов, написанных Apache ИЛИ имя файла логов которых error.log, необходимо создать два правила с одним условием каждое:

    {


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "log.source",


"values": [


"/path/to/error.log"


]


}


],


"enabled": true


},


{


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "dt.entity.process_group",


"values": [


"PROCESS_GROUP-APACHEID"


]


}


],


"enabled": true


}

    Для маскирования логов, написанных Apache, имя файла которых начинается с error И заканчивается на log, необходимо создать одно правило с тремя условиями, каждое с одним значением.

    {


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "log.source",


"values": [


"/path/to/error*"


]


},


{


"attribute": "log.source",


"values": [


"*log"


]


},


{


"attribute": "dt.entity.process_group",


"values": [


"PROCESS_GROUP-APACHEID"


]


}


],


"enabled": true


}

    Для маскирования логов, написанных Apache, имя файла которых начинается с error ИЛИ заканчивается на log, необходимо создать одно правило с двумя условиями, одно с группой процессов, а второе с двумя значениями содержимого, /path/to/error* и *log:

    {


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "log.source",


"values": [


"/path/to/error*", "*log"


]


},


{


"attribute": "dt.entity.process_group",


"values": [


"PROCESS_GROUP-APACHEID"


]


}


],


"enabled": true


}

    Маскирование логов Apache или MySQL¶

    Для маскирования логов, написанных Apache или MySQL, необходимо создать либо два правила, либо одно правило с одним условием, имеющим два значения.

    Сценарий с двумя правилами:

    {


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "dt.entity.process_group",


"values": [


"PROCESS_GROUP-MYSQL"


]


}


],


"enabled": true


},


{


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "dt.entity.process_group",


"values": [


"PROCESS_GROUP-APACHEID"


]


}


],


"enabled": true


}

    Сценарий с одним правилом с условием, имеющим два значения:

    {


"masking": {


"expression": "email: (.*),",


"type": "SHA256"


},


"matchers": [


{


"attribute": "dt.entity.process_group",


"values": [


"PROCESS_GROUP-APACHEID", "PROCESS_GROUP-MYSQL"


]


}


],


"enabled": true


}

    Примеры регулярных выражений¶

    Распространённые форматы регулярных выражений для конфиденциальных данных включают:

    Неподдерживаемые регулярные выражения¶

    Маскирование данных происходит в рамках всего выражения или группы захвата. Выражение должно соответствовать синтаксису движка регулярных выражений и не может:

    • Быть частью более чем одной группы захвата
    • Содержать утверждение нулевой длины lookbehind в группе захвата
    • Содержать утверждение нулевой длины backreference в группе захвата
    • Содержать жадные квантификаторы (такие как x?, x* или x+) или ревностные квантификаторы (такие как x?+, x*+ или xx++). Вместо этого используйте ленивые/нежадные квантификаторы (такие как x?? и x+?).

    Часто задаваемые вопросы¶

    Где происходит маскирование конфиденциальных данных?

    Вы можете выполнять маскирование конфиденциальных данных в вашей среде, чтобы конфиденциальные данные не покидали вашу инфраструктуру незащищёнными. Если вы импортируете данные в Dynatrace через универсальный приём данных, вам необходимо маскировать конфиденциальные данные на уровне источника перед приёмом. В качестве альтернативы вы можете маскировать конфиденциальные данные во время обработки логов. Однако, если вы выберете маскирование данных во время обработки логов, ваши данные покинут вашу среду, поскольку обработка логов происходит на стороне Dynatrace. Поэтому безопаснее маскировать данные в вашей среде.

    Сколько групп захвата поддерживается?

    Одна. Если группа не указана, вся область регулярного выражения рассматривается как одна группа захвата.

    Ограничения маскирования конфиденциальных данных¶

    Учитывайте следующие ограничения маскирования конфиденциальных данных:

    • Если процесс маскирования занимает слишком много времени, затронутый файл логов блокируется до перезапуска OneAgent или любого изменения конфигурации, после чего вы получите сообщение File not monitored - incorrect sensitive data masking rule.
