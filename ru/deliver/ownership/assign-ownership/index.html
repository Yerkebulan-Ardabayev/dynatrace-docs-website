    Назначение команд-владельцев отслеживаемым сущностям¶

    Назначение команд-владельцев отслеживаемым сущностям¶

    • Classic
    • Практическое руководство
    • Время чтения: 9 мин
    • Обновлено 16 сентября 2025 г.

    Вы можете назначать команды-владельцев любой отслеживаемой сущности в Dynatrace, используя основной идентификатор команды или дополнительные идентификаторы (определяемые при настройке команды). Это позволяет мгновенно найти ответственную команду и связаться с ней при возникновении проблем с сущностью, например, если сущность затронута уязвимостью.

    Вы можете назначать владение несколькими способами, такими как метки и аннотации Kubernetes, метаданные хоста, переменные окружения и теги (в том числе через API пользовательских тегов Dynatrace).

    Поддерживаемые методы назначения владения¶

    Поддерживаются следующие методы назначения владения и типы сущностей.

    • Метаданные развёртывания или конфигурации Рекомендуется
    Метод Типы сущностей
    Метки и аннотации Kubernetes Все объекты Kubernetes
    Метаданные хоста через oneagentctl или hostcustomproperties.conf Хосты
    Переменные окружения Процессы
    * Теги (ручные, автоматические и через API) — все отслеживаемые сущности

    Мы настоятельно рекомендуем использовать предпочтительный метод назначения владения в зависимости от типа сущности, как описано в разделах ниже.

    • Метки и аннотации Kubernetes для объектов Kubernetes
    • Метаданные для хостов
    • Переменные окружения для процессов
    • Теги (ручные, автоматические и через API) для всех остальных сущностей

    Наши рекомендации основаны на наиболее эффективных методах обеспечения адекватного покрытия владением сущностей. Подробнее см. в разделе Лучшие практики владения сущностями.

    Формат применения информации о владении¶

    Независимо от используемого метода, информация о владении применяется к сущностям в виде пар ключ-значение. Ключи по умолчанию owner и dt.owner доступны в каждой среде мониторинга — см. Settings > Ownership > Configuration. Однако вы можете изменить или удалить ключи по умолчанию и определить собственные.

    Значением всегда является уникальный идентификатор команды, указанный при создании команды. Вы также можете использовать дополнительные идентификаторы. (Подробнее об определении этих значений см. в разделе Создание и управление командами для владения сущностями.)

    Сущность может иметь более одной команды-владельца. Для этого можно применить несколько пар ключ-значение, как показано в разделах ниже.

    Пользовательские ключи для информации о владении¶

    Вы можете определить до пяти ключей для применения информации о владении.

    1. Перейдите в Settings > Ownership > Configure.
    2. Выберите Add key.
    3. Введите ключ (по умолчанию Enabled).
    4. Save changes.

    Когда вы отключаете или удаляете ключ, сущности, назначенные командам через такие ключи, больше не будут отображать или содержать метаданные владения. Однако такие ключи будут отображаться как обычные теги, применённые к сущностям.

    Дополнительные требования к ключам владения¶

    • Вы можете определить максимум пять и минимум один ключ.
    • Вы можете использовать любой из определённых ключей в качестве префикса в парах ключ-значение. Например, для ключей owner и dt.owner можно использовать owner-1 и dt.owner-test.

    Метки и аннотации Kubernetes¶

    Вы можете указать владение командой для различных объектов Kubernetes, таких как Deployments, Pods, Services или пространства имён. Чтобы обеспечить адекватное покрытие владением объектов Kubernetes, что особенно важно для кратковременных сущностей, таких как Pods, предоставляйте информацию о владении в виде меток или аннотаций Kubernetes с парами ключ-значение в файле спецификации развёртывания, например deployment.yaml.

    Мы рекомендуем определять владение для Deployment и всех других объектов, для которых вы хотите обеспечить покрытие владением. См. также Лучшие практики владения сущностями.

    • Установка идентификатора команды в качестве метки или аннотации для Deployment, CronJob, Job, DaemonSet или StatefulSet предоставляет информацию о владении командой для соответствующих сущностей CLOUD_APPLICATION. Обратите внимание, что это владение не распространяется на CLOUD_APPLICATION_INSTANCE.

    В этом примере двойное владение устанавливается через две метки для Deployment. Каждая метка имеет уникальный ключ. Уникальные ключи являются обязательным требованием в метках и аннотациях Kubernetes.

    apiVersion: apps/v1



kind: Deployment



metadata:



name: demo



labels:



dt.owner-1: my-team-1 # Dual team ownership defined for the Deployment



dt.owner-2: my-team-2



spec:

    Пример кода ниже показывает аннотацию для владения на уровне Deployment.

    apiVersion: apps/v1



kind: Deployment



metadata:



name: demo



annotations:



dt.owner: my-team # Ownership defined for the Deployment
    * Установите метки для Pods, чтобы указать владение для соответствующей сущности CLOUD_APPLICATION_INSTANCE. Укажите владение для каждого Pod.

    apiVersion: apps/v1



kind: Deployment



metadata:



name: demo



spec:



replicas: 3



selector:



matchLabels:



app: demo



template:



metadata:



labels:



app: demo



dt.owner: my-team # Ownership defined for the Pod



spec:

    Пример кода ниже показывает манифест для Pod с аннотацией dt.owner: myTeam.

    apiVersion: v1



kind: Pod



metadata:



name: annotations-demo



annotations:



imageregistry: "https://hub.docker.com/"



dt.owner: my-team



spec:



containers:



- name: nginx



image: nginx:1.14.2



ports:



- containerPort: 80
    * Для процесса укажите владение с помощью пар ключ-значение через переменную окружения DT_CUSTOM_PROP, определённую для контейнера.

    В переменных окружения можно указать несколько значений для одного ключа. В этом примере двойное владение определяется с помощью ключа owner.

    apiVersion: apps/v1



kind: Deployment



metadata:



name: demo



spec:



replicas: 3



selector:



matchLabels:



app: demo



template:



spec:



containers:



- name: demo



image: demo:1.0.0



env:



- name: DT_CUSTOM_PROP ## Environment variable



value: "owner=team-automation owner=team-dev" # Dual ownership for the process; team IDs are team-automation and team-dev.
    * Метки Kubernetes для Service

    apiVersion: v1



kind: Service



metadata:



name: my-service



labels:



dt.owner: team-a # Ownership defined for the Service



spec:



selector:



app.kubernetes.io/name: MyApp



ports:



- protocol: TCP



port: 80



targetPort: 9376
    * Метки Kubernetes для пространства имён

    apiVersion: v1



kind: Namespace



metadata:



name: my-namespace



labels:



dt.owner: team-a # Ownership defined for the namespace

    Метаданные хоста¶

    Для хостов мы рекомендуем использовать метаданные хоста в качестве основного метода назначения владения.

    Подробнее о определении тегов и метаданных для хостов и командном интерфейсе OneAgent oneagentctl.

    Вы также можете назначить владение хостам с помощью тегов.

    Через oneagentctl¶

    OneAgent версии 1.189+ Для хостов с OneAgent версии 1.189+ используйте командный интерфейс oneagentctl после установки для задания свойства метаданных для отдельного хоста. Запускайте oneagentctl с правами администратора или root из следующих расположений по умолчанию.

    • Windows: %PROGRAMFILES%\dynatrace\oneagent\agent\tools
    • Все Unix-подобные системы: /opt/dynatrace/oneagent/agent/tools

    Этот пример для Unix-подобных систем использует --set-host-property для установки владения с парой ключ-значение owner-1=team-automation, где team-automation — это основной идентификатор команды или дополнительный идентификатор.

    ./oneagentctl --set-host-property owner-1=team-automation

    Через hostcustomproperties.conf¶

    OneAgent версии 1.187 и ранее Для хостов с OneAgent 1.187 и ранее создайте или отредактируйте файл конфигурации OneAgent hostcustomproperties.conf в следующих расположениях.

    • Windows: %PROGRAMDATA%\dynatrace\oneagent\agent\config
    • Все Unix-подобные системы: /var/lib/dynatrace/oneagent/agent/config

    Этот пример для Unix-подобных систем устанавливает владение хостом с парой ключ-значение dt.owner-1=team-automation, где team-automation — это основной идентификатор команды или дополнительный идентификатор.

    cat hostcustomproperties.conf dt.owner-1=team-automation

    Переменные окружения процессов¶

    Для процессов мы рекомендуем указывать владение в парах ключ-значение через переменную окружения DT_CUSTOM_PROP.

    Мы не рекомендуем использовать теги для назначения владения процессам или группам процессов.

    Подробнее об определении метаданных групп процессов, например, настройке переменной окружения DT_CUSTOM_PROP для IIS и Windows.

    export DT_CUSTOM_PROP dt.owner=DemoTeam

    См. раздел Формат применения информации о владении для получения информации обо всех существующих вариантах пар ключ-значение или для создания собственных.

    Теги¶

    Используйте теги для назначения владения только тем сущностям, которые не покрываются описанными выше методами. Обычно это такие сущности, как фронтенд-приложения, которые стабильны и немногочисленны.

    Методы тегирования¶

    Вы можете использовать теги для назначения владения в парах ключ-значение любой отслеживаемой сущности — подробнее о тегировании.

    • Ручные теги с использованием определённых ключей или префиксов ключей (таких как owner и dt.owner) легко применять через веб-интерфейс.
    • Вы также можете настроить правила автоматического тегирования через веб-интерфейс.
    • Мы рекомендуем использовать API пользовательских тегов вместо правил автоматического тегирования для назначения владения сущностям. API пользовательских тегов позволяет удобно применять теги к большой группе сущностей в рамках одного вызова API, выполняемого немедленно.

    Обратите внимание, что теги, применённые вручную, могут быть удалены вручную. Автоматически применённые теги нельзя удалить вручную с отдельных сервисов, групп процессов, экземпляров групп процессов, приложений или хостов.

    См. Лучшие практики владения сущностями для получения дополнительной информации о преимуществах и ограничениях тегирования для назначения владельцев сущностям. См. также Лучшие практики масштабирования правил тегирования и зон управления.

    Разрешения¶

    Для добавления, изменения или удаления владения через API пользовательских тегов вам требуются все следующие разрешения.

    • Разрешение токена entities.write
    • Разрешение токена settings.read или политика IAM ALLOW settings:objects:read WHERE settings:schemaId = "builtin:ownership.teams";

    Для добавления, изменения или удаления тегов через веб-интерфейс вам необходимо разрешение Manage monitoring settings на уровне среды или зоны управления.

    Просмотр информации о владении в веб-интерфейсе¶

    Информация о владении доступна только для унифицированных страниц анализа.

    Для хостов и всех сущностей Kubernetes выберите Owners на странице деталей сущности для просмотра информации о владении.

    В этом примере показана рабочая нагрузка Kubernetes, соответствующая сущности CLOUD_APPLICATION. Разверните название команды для просмотра её деталей. Выберите в карточке Ownership для редактирования деталей команды в Settings.

    В этом примере показана информация о владении для Pod Kubernetes.

    На странице Hosts можно искать хосты с владением — фильтрация по Tags с определёнными вами префиксами ключей, например owner и dt-owner. Обратите внимание, что пары ключ-значение владения должны быть применены хотя бы к одному хосту, чтобы ключ был доступен в фильтре Tags. Этот метод поиска владения доступен на всех страницах групп сущностей, которые можно фильтровать по тегам.

    В примере страницы деталей хоста ниже у хоста три команды-владельца. Один из владельцев отмечен как Unknown team identifier. Это означает, что идентификатор команды был применён к хосту (например, через oneagentctl или ручной тег) в паре ключ-значение, но команда не существует. Invalid team identifier означает, что идентификатор команды или дополнительный ID был применён к хосту в неправильном формате.

    Выберите Properties and tags для просмотра всех тегов, применённых к хосту.

    Выберите рядом с неизвестной командой, а затем выберите Add team для определения информации о команде в Settings. Сущность будет автоматически обновлена определением команды.
