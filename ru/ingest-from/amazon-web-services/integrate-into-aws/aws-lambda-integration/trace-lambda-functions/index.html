    Отслеживание функций Lambda¶

    • Руководство по настройке
    • 8-минутное чтение

    Dynatrace предоставляет вам специальный слой AWS Lambda, содержащий расширение Dynatrace для AWS Lambda. Вам необходимо добавить общедоступный слой для вашего времени выполнения и региона к вашей функции. Затем, в зависимости от метода конфигурации, Dynatrace предоставляет шаблон или конфигурацию для вашей функции AWS Lambda.

    Возможности¶

    Dynatrace предоставляет обширные возможности мониторинга Python, Node.js и Java:

    • Автоматическое распределенное трассирование через сервисы AWS, такие как API Gateway, SQS и просмотр доступных метрик."), SNS и просмотр доступных метрик."), EventBridge и бесшовная интеграция с другими сервисами AWS. Для получения более подробной информации см. Интеграция AWS Lambda.

    • Поддержка OpenTelemetry для трассировки и метрики.

    • Родная ингестия журналов из функций Lambda. Dynatrace поддерживает ингестию журналов напрямую через API телеметрии AWS Lambda, снижая зависимость от CloudWatch. Для получения более подробной информации см. Сбор журналов AWS Lambda.
    • Обнаружение и оптимизация холодного запуска.
    • Поддержка Infrastructure-as-Code (Terraform, AWS SAM, Serverless Framework).

    Смотрите нашу матрицу поддерживаемых технологий для получения информации о поддерживаемых фреймворках и временах выполнения для поддерживаемых AWS рантаймов.

    Входящие вызовы¶

    Для вызовов функций AWS Lambda Dynatrace предлагает общую поддержку всех типов триггеров. OneAgent может захватить конкретную информацию или подключить трассу к любому родительскому элементу и, таким образом, добавить дополнительную информацию только для вызовов, сделанных через:

    • API вызова AWS SDK Lambda
    • API-шлюз
    • URL функции Lambda
    • AWS SQS
    • AWS SNS
    • AWS EventBridge
    • AWS Application Load Balancer

    Для других типов вызовов OneAgent не может захватить никакой конкретной информации или подключить трассу к любому родительскому элементу. Вызовы через AWS SDK требуют инструмента Dynatrace для подключения трассы.

    Чтобы начать

    1. В Dynatrace Hub, выберите AWS Lambda.

    2. Выберите Настройка.

    3. Следуйте инструкциям, чтобы включить мониторинг функций AWS Lambda.

    Выбор метода конфигурации¶

    Dynatrace OneAgent распространяется как слой Lambda, который можно включить и настроить вручную или с помощью известных решений Infrastructure as Code (IaC). Слой Lambda хранится в учетной записи Dynatrace AWS 585768157899.

    Мастер предоставляет вам различные варианты конфигурации и фрагменты кода, которые можно использовать в автоматизации развертывания по вашему выбору.

    Укажите конечную точку API Dynatrace¶

    Необязательно

    Это необязательный шаг, который позволяет вам указать конечную точку API Dynatrace, на которую будут отправляться данные мониторинга.

    Типичный сценарий - развернуть Dynatrace ActiveGate в непосредственной близости (той же области) от функций Lambda, которые вы хотите отслеживать, чтобы уменьшить сетевую задержку, которая может повлиять на выполнение и холодный старт ваших функций Lambda для (обычно одного) сетевого запроса OneAgent на вызов Lambda (который происходит в конце вызова). См. раздел <#monitoring-overhead> для типичных показателей накладных расходов.

    Включить Мониторинг Реальных Пользователей¶

    Необязательно

    Это необязательный шаг для использования Мониторинга Реальных Пользователей (RUM), который предоставляет вам глубокую информацию о действиях пользователей и производительности через браузер или в мобильных приложениях.

    Настройте AWS API Gateway¶

    • Если входящие (не-XHR) запросы к вашим Lambda-функциям не связаны с вызывающим приложением, настройте API Gateway для передачи тега Dynatrace. Для этого включите Use Lambda Proxy Integration на странице конфигурации Integration Request в API Gateway.
    • Если API Gateway настроен со страницы конфигурации Lambda, эта настройка будет включена по умолчанию. Дополнительную информацию смотрите в Enable CORS on a resource using the API Gateway console.

    AWS Lambda также поддерживает non-proxy integration, которая — без дополнительной конфигурации — предотвращает Dynatrace от:

    • Отслеживания вызовов из других отслеживаемых приложений
    • Обнаружения RUM (web и mobile)

    Чтобы отслеживание вызовов из других отслеживаемых приложений/обнаружение RUM работало в этом сценарии, создайте пользовательский шаблон сопоставления в конфигурации integration requests.

    1. В AWS API Gateway Console перейдите в Resources и выберите метод запроса (например, GET).
    2. Выберите Mapping Templates и затем выберите Add mapping template.
    3. Добавьте следующее содержимое в шаблон:
    {


"path": "$context.path",


"httpMethod": "$context.httpMethod",


"headers": {


#foreach($param in ["x-dynatrace", "traceparent", "tracestate", "x-dtc", "referer", "host", "x-forwarded-proto", "x-forwarded-for", "x-forwarded-port"])


"$param": "$util.escapeJavaScript($input.params().header.get($param))"


#if($foreach.hasNext),#end


#end    },


"requestContext": {


"stage": "$context.stage"


}


}

    Заголовок x-dtc специфичен для отслеживания сценариев RUM, в то время как остальные заголовки обычно необходимы для связывания трассировок вместе и извлечения соответствующей информации, такой как метаданные веб-запросов. 4. Выберите Save, чтобы сохранить вашу конфигурацию. 5. Повторно разверните ваш API.

    Настройте AWS¶

    Убедитесь, что заголовок x-dtc разрешен в настройках CORS ваших отслеживаемых Lambda-функций.

    RUM для Lambda-функций требует отправки определенного заголовка (x-dtc) с XHR-запросами в AWS. Чтобы включить его, настройки CORS вашего развертывания AWS должны разрешать заголовок x-dtc во время предварительного запроса (OPTIONS). Чтобы настроить CORS и разрешить заголовок x-dtc для вашей конкретной настройки, обратитесь к Enable CORS on a resource using the API Gateway console в документации AWS.

    Настройте Dynatrace¶

    Чтобы настроить заголовок x-dtc для вызовов ваших Lambda-функций:

    1. Перейдите в Web, Mobile, Frontend или Custom Applications, в зависимости от типа вашего приложения.
    2. Выберите приложение, которое вы хотите связать со своей Lambda-функцией.
    3. Выберите меню просмотра (â€¦) в правом верхнем углу и выберите Edit.
    4. Выберите Capturing > Async web requests and SPAs.
    5. Убедитесь, что выбранный вами фреймворк включен. Если вашего фреймворка нет в списке, включите Capture XmlHttpRequest (XHR) для универсальной поддержки XHR.
    6. Выберите Capturing > Advanced setup.
    7. Прокрутите вниз до раздела Enable Real User Monitoring for cross-origin XHR calls и введите шаблон, который соответствует URL-адресу ваших Lambda-функций. Например: TheAwsUniqueId.execute-api.us-east-1.amazonaws.com
    8. Выберите Save. Через несколько минут заголовок будет прикреплен ко всем вызовам вашей Lambda-функции, и запросы из вашего браузера будут связаны с бэкендом.

    Неудачные запросы

    Если после включения этой опции запросы начнут завершаться с ошибками, просмотрите свои настройки CORS. Чтобы узнать, как настроить CORS, обратитесь к Enable CORS on a resource using the API Gateway console в документации AWS.

    Развертывание¶

    Скопируйте фрагменты конфигурации в ваше развертывание и используйте метод развертывания по вашему выбору, чтобы включить слой и задать конфигурацию для ваших функций Lambda.

    Варианты конфигурации¶

    Получение токена из AWS Secrets Manager¶

    OneAgent версия 1.295+

    Вместо явного указания токена аутентификации в конфигурации вы можете настроить OneAgent, чтобы получить токен, хранящийся в AWS Secrets Manager.

    Предварительные условия

    • Убедитесь, что вы предоставили разрешение secretsmanager:GetSecretValue для секрета ARN токена аутентификации функции Lambda, отслеживаемой OneAgent. Для подробностей см. Аутентификация и контроль доступа для AWS Secrets Manager в документации AWS Secrets Manager.

    • Убедитесь, что значение секрета содержит только открытый текстовый значение токена аутентификации (без кавычек). Обратите внимание, что

    • Секреты со структурой JSON не поддерживаются. Для подробностей см. Создание секрета AWS Secrets Manager в документации AWS Secrets Manager.

    • Когда вы извлекаете значение секрета, Secrets Manager по умолчанию возвращает только текущую версию секрета (AWSCURRENT метка). Для подробностей см. Что такое секрет в Secrets Manager? в документации AWS Secrets Manager.

    Чтобы получить токен для подключения трассировки, задайте секрет ARN токена либо в переменную среды DT_CONNECTION_AUTH_TOKEN_SECRETS_MANAGER_ARN, либо в свойство JSON Connection.AuthTokenSecretsManagerArn.

    Этот вариант всегда переопределяет DT_CONNECTION_AUTH_TOKEN (Connection.AuthToken). Если получение токена не удалось, OneAgent не сможет экспортировать данные трассировки.

    Получение токена происходит только один раз, во время фазы инициализации функции Lambda, что вызывает увеличение времени холодного запуска функции Lambda.

    Для подробностей о получении токена для сбора журналов см. Получение токена из AWS Secrets Manager.

    Перегрузка мониторинга¶

    Включение мониторинга неизбежно вызывает перегрузку выполнения отслеживаемой функции. Перегрузка зависит от нескольких факторов, таких как технология времени выполнения функции, конфигурация и конкретные характеристики функции, такие как размер кода или продолжительность и сложность выполнения.

    Объем памяти, настроенный для функции, напрямую влияет на вычислительные ресурсы, назначенные экземпляру функции. Для подробностей см. Память и вычислительная мощность.

    Худший сценарий измеренной перегрузки - функция с пустым обработчиком функции и минимальной конфигурацией памяти.

    Перегрузка холодного запуска¶

    • Для Node.js перегрузка холодного запуска составляет примерно 900 мс.
    • Для Java перегрузка холодного запуска составляет примерно 1500 мс.
    • Для Python перегрузка холодного запуска составляет примерно 1000 мс.

    Для процесса бенчмаркинга холодного запуска были протестированы функции "hello-world" (возвращающие только ответ) с 512 МБ выделенной памяти. Обратите внимание, что наблюдаемая перегрузка может варьироваться в зависимости от нескольких факторов:

    • Настроенная память: Функция Lambda назначается CPU пропорционально настроенной памяти, что может повлиять на производительность холодного запуска. Функции с более высокой памятью, как правило, демонстрируют более быстрые времена инициализации из-за увеличения выделения CPU.
    • Реализация функции: Сложность фактической реализации функции, включая внешние зависимости, логику инициализации и среду выполнения, может существенно повлиять на продолжительность холодного запуска.
    • Версия среды выполнения: Конкретная версия среды выполнения или контейнерного образа, используемого, также может повлиять на время холодного запуска.

    При проведении оценок производительности мы рекомендуем учитывать эти факторы, поскольку они могут повлиять на результаты бенчмаркинга в реальных сценариях.

    Задержка времени ответа¶

    Задержка зависит от реализации функции, но обычно составляет менее 10%. Это означает, что время, необходимое для получения ответа вызывающим функцию Lambda, может увеличиться на 10% при добавлении слоя OneAgent, по сравнению с ситуацией, когда OneAgent не активен/не присутствует.

    Интеграция Dynatrace AWS¶

    Хотя это не обязательно, мы рекомендуем настроить интеграцию Dynatrace Amazon CloudWatch. Это позволяет объединить данные, полученные через интеграцию AWS, с данными, собранными расширением Dynatrace AWS Lambda.

    Для получения более подробной информации обратитесь к Amazon CloudWatch Metric Streams.

    Фильтрация холодных запусков¶

    Одной из важных метрик для Lambda является частота холодных запусков. Холодный запуск происходит, когда вызывается новый экземпляр функции Lambda. Такие холодные запуски занимают больше времени и добавляют задержку к вашим запросам.

    Высокая частота холодных запусков может указывать на ошибки или неравномерный шаблон нагрузки, который можно смягчить с помощью запланированной одновременности. Dynatrace отчеты о таких холодных запусках как свойство в распределенном трассировании.

    Чтобы проанализировать холодные запуски

    1. Выберите Просмотреть все запросы на странице деталей службы Lambda.

    2. В фильтре запросов выберите Холодный запуск функции в разделе Свойство запроса.

    3. Отфильтруйте вызовы, содержащие Только холодный запуск или Нет холодного запуска.

    Известные ограничения¶

    • Времена ожидания подключения HTTP, времена ожидания сброса и пороги отступления не настраиваются.
    • OneAgent, работающий на AWS Lambda, не поддерживает функции, требующие перенастройки OneAgent через интерфейс пользователя на стороне сервера.
    • OneAgent для AWS Lambda использует только локальные настройки, определенные через переменные окружения или файлы конфигурации, вместо получения конфигурации из кластера Dynatrace. В результате любые настройки, определенные на уровне кластера, игнорируются, и применяются значения по умолчанию, если они не переопределены явно.
    • Большинство расширений Dynatrace AWS Lambda не захватывают IP-адреса исходящих запросов HTTP. Это приводит к неотслеживаемым хостам, если вызываемая служба не отслеживается с помощью Dynatrace.
    • .NET: ASP.NET Core, оценка атак и уязвимостей на уровне кода, а также несколько метрик не поддерживаются в AWS Lambda.
    • Датчики расширения Node.js AWS Lambda (инструментирование) не поддерживают модули ECMAScript. Это означает, что расширение не будет правильно отслеживать исходящие вызовы (например, запросы HTTP или AWS SDK).
    • Обработчики Node.js, упакованные с помощью esbuild в CommonJS, не могут быть автоматически инструментированы OneAgent. Это связано с тем, что процесс упаковки может изменить способ, которым функция обработчика представлена, что предотвращает автоматическое обнаружение и инструментирование OneAgent.

    Как включить автоматическое инструментирование для обработчиков Node.js, упакованных с помощью esbuild?

    Вы можете использовать простой скрипт-обертку, чтобы включить автоматическое инструментирование.

    Вот пример обработчика AWS Lambda, написанного на TypeScript (index.ts):

    import { Context, APIGatewayProxyResult, APIGatewayEvent } from 'aws-lambda';


export const handler = async (event: APIGatewayEvent, context: Context): Promise<APIGatewayProxyResult> => {


console.log(`Event: ${JSON.stringify(event, null, 2)}`);


console.log(`Context: ${JSON.stringify(context, null, 2)}`);


return {


statusCode: 200,


body: JSON.stringify({


message: 'hello world',


}),


};


};

    1. Упакуйте обработчик с помощью esbuild:

      esbuild index.ts --bundle --minify --sourcemap --platform=node --target=es2020 --outfile=dist/index.js
      2. Создайте файл-обертку (esbuild_wrap.js) для представления обработчика в виде, который может быть инструментирован OneAgent:

      const esbuildDist = require("./dist/index");


exports.handler = esbuildDist.handler;
      3. В конфигурации среды выполнения AWS Lambda установите обработчик на esbuild_wrap.handler.

    OneAgent теперь сможет обнаружить и инструментировать функцию правильно. * Экземпляры AWS Lambda в управляемом режиме не поддерживаются. Этот новый режим размещения позволяет развертывать функции Lambda на кластерах экземпляров EC2, управляемых AWS. Расширение Dynatrace Lambda и модульные слои в настоящее время не поддерживают этот режим размещения. * Офлайн-кластеры Managed не поддерживаются.
