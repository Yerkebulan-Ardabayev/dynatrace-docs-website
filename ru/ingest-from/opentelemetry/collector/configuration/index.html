    Настройка OpenTelemetry Collector¶

    Настройка OpenTelemetry Collector¶

    • Latest Dynatrace
    • Практическое руководство
    • Чтение: 4 мин
    • Обновлено 5 декабря 2025 г.

    Для успешной настройки экземпляра Collector необходимо настроить каждый компонент (приемник, необязательный процессор и экспортер) отдельно в YAML-файле конфигурации Collector и включить их через дополнительные объекты конвейеров.

    Dynatrace Collector

    В рамках настройки Dynatrace Collector обратите внимание, что вы можете настраивать только компоненты, поставляемые с Dynatrace Collector.

    Полный список всех поддерживаемых компонентов Collector см. в Dynatrace OTel Collector.

    Пример конфигурации¶

    Ниже приведен пример YAML-файла базовой конфигурации Collector, которую можно использовать для экспорта трассировок, метрик и логов OpenTelemetry в Dynatrace.

    receivers:



otlp:



protocols:



grpc:



endpoint: 0.0.0.0:4317



http:



endpoint: 0.0.0.0:4318



processors:



cumulativetodelta:



max_staleness: 25h



exporters:



otlp_http:



endpoint: "${env:DT_ENDPOINT}"



logs_endpoint: "${env:DT_ENDPOINT}/v1/logs?structure=flattened"



headers:



Authorization: "Api-Token ${env:DT_API_TOKEN}"



service:



pipelines:



traces:



receivers: [otlp]



processors: []



exporters: [otlp_http]



metrics:



receivers: [otlp]



processors: [cumulativetodelta]



exporters: [otlp_http]



logs:



receivers: [otlp]



processors: []



exporters: [otlp_http]

    Рекомендация по процессору cumulativetodelta

    Рекомендуется установить параметр max_staleness процессора cumulativetodelta на значение, превышающее частоту получения метрик Collector (например, как часто метрики поступают через OTLP или каков интервал сбора Prometheus). Это гарантирует, что ссылки на заброшенные потоки метрик не будут накапливаться в памяти со временем.

    В этом YAML-файле мы настраиваем следующие компоненты:

    • OTLP-приемник (otlp), который может получать данные по gRPC и HTTP.
    • Процессор для преобразования метрик с кумулятивной временной агрегацией в дельта-временную агрегацию (подробнее см. Дельта-метрики).

    • OTLP HTTP-экспортер (otlp_http), настроенный с:

    • Эндпоинтом Dynatrace, см. Базовые URL.

    • Эндпоинтом логов, настроенным для выравнивания структурированных логов, см. <#structured-logs>.
    • API-токеном, см. Аутентификация.

    В примере конфигурации выше токен Dynatrace должен иметь разрешения Ingest OpenTelemetry traces (openTelemetryTrace.ingest), Ingest metrics (metrics.ingest) и Ingest logs (logs.ingest).

    В разделе service каждый компонент определяется отдельно.

    • Расширения можно включить в отдельном разделе, а приемники, процессоры и экспортеры группируются в разделе pipeline.
    • Конвейеры могут быть типа traces, metrics или logs.
    • Каждый приемник/процессор/экспортер может использоваться в нескольких конвейерах. Для процессоров, на которые ссылаются несколько конвейеров, каждый конвейер получает отдельный экземпляр процессора. В отличие от этого, для приемников/экспортеров, на которые ссылаются несколько конвейеров, используется только один экземпляр. Также обратите внимание, что порядок процессоров определяет порядок обработки данных.
    • Вы также можете определить одни и те же компоненты более одного раза. Например, вы можете иметь два разных приемника или даже два и более отдельных сегмента конвейера.
    • Даже если компонент правильно настроен в своем разделе, он не будет включен, пока не будет определен в разделе service.

    Валидация конфигурации¶

    Важно убедиться, что используемая конфигурация Collector синтаксически и семантически корректна. Например, YAML использует пробелы (не табуляцию) для отступов, определения иерархии документа, и необходимо использовать правильный уровень отступов для каждого раздела и компонента. Collector предоставляет встроенную команду validate для проверки корректности конфигурации, ее компонентов и сервисов.

    dynatrace-otel-collector validate --config=[PATH_TO_YOUR_CONFIGURATION_FILE]

    Если вы запускаете контейнерный экземпляр Collector, вы также можете использовать следующую команду Docker для запуска валидации непосредственно из контейнера.

    docker run -v $(pwd):$(pwd) -w $(pwd) ghcr.io/dynatrace/dynatrace-otel-collector/dynatrace-otel-collector:0.44.0 validate --config=[YOUR_CONFIGURATION_FILE]

    Дельта-метрики¶

    Dynatrace требует, чтобы данные метрик отправлялись с дельта-временной агрегацией, а не с кумулятивной.

    Если ваше приложение не позволяет настроить дельта-временную агрегацию, вы можете использовать процессор cumulativetodelta, чтобы экземпляр Collector преобразовывал кумулятивные значения в дельта-значения. Пример конфигурации выше показывает, как настроить и подключить процессор в конфигурации Collector.

    Связанные и балансируемые Collector¶

    При использовании нескольких экземпляров Collector важно обеспечить стабильное распространение значений между всеми экземплярами.

    Это особенно важно при отправке OTLP-запросов через разные экземпляры Collector (например, при балансировке нагрузки), поскольку каждый экземпляр Collector отслеживает собственное дельта-смещение, что может нарушить данные, отправляемые в бэкенд Dynatrace.

    В таких сценариях рекомендуется маршрутизировать OTLP-запросы через один исходящий экземпляр Collector, который пересылает данные в бэкенд Dynatrace и выполняет дельта-преобразование. Остальные экземпляры Collector должны использовать кумулятивную агрегацию для обеспечения стабильного и согласованного распространения значений.

    Настройка обработки структурированных логов¶

    При экспорте структурированных логов в Dynatrace система обрабатывает данные логов одним из двух способов. Она либо:

    • Сохраняет исходную вложенную структуру из источника логов ("raw").
    • Выравнивает ее, чтобы все значения атрибутов логов были доступны по ключевым путям ("flattened").

    Для настройки этого поведения используйте параметр logs_endpoint, как показано в примере конфигурации, строка 16. Убедитесь, что указан полный путь API-эндпоинта, включая /v1/logs (допускаются переменные окружения), с указанием желаемого поведения.

    • Для сохранения исходной вложенной структуры используйте structure=raw.
    • Для выравнивания данных логов используйте structure=flattened.

    Если этот параметр конфигурации не указан, поведение по умолчанию зависит от даты создания среды.

    • Для Dynatrace версии 1.331+ и сред, созданных после 1 февраля 2026 г.: Raw.
    • Для сред, созданных до 1 февраля 2026 г.: Flattened.
