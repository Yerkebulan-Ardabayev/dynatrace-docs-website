    Загрузка логов OTLP¶

    • Latest Dynatrace
    • 3 мин. чтения

    OpenTelemetry поддерживает атрибуты на различных уровнях в запросе логов OpenTelemetry, таких как уровень ресурса, уровень области (scope) и уровень записи.

    API загрузки логов собирает и пытается автоматически преобразовать данные логов.

    Преобразование данных¶

    Каждая запись лога из загруженного пакета сопоставляется с одной записью лога Dynatrace, которая содержит три специальных атрибута: timestamp, loglevel, content, а также набор других атрибутов в формате ключ-значение. Эти свойства устанавливаются на основе ключей, присутствующих во входном объекте, следующим образом.

    Временная метка¶

    • Устанавливается на основе поля Timestamp входной записи лога.

    • Если timestamp не может быть установлен на основе поля Timestamp, timestamp определяется на основе записи лога OTLP. Подробнее о различиях между моделями данных см. в разделе Обработка API загрузки логов ниже.

    • Поддерживаемые форматы временных меток: UTC milliseconds, RFC3339 и RFC3164.

    • Значение по умолчанию --- текущая временная метка, часовой пояс по умолчанию --- UTC, если он не указан во временной метке.
    • События логов, старше лимита Log Age, отбрасываются. Временные метки, опережающие текущее время более чем на 10 минут, заменяются текущим временем. Подробности см. в разделе Лимиты загрузки.

    Уровень лога¶

    • Устанавливается на основе поля SeverityText (первый приоритет) или поля SeverityNumber (второй приоритет) входной записи лога.

    • Если loglevel не может быть установлен на основе поля SeverityText или SeverityNumber, loglevel определяется на основе записи лога OTLP. Подробнее о различиях между моделями данных см. в разделе Обработка API загрузки логов ниже.

    • Значение по умолчанию --- NONE.

    Содержимое¶

    • Содержимое устанавливается на основе поля Body входной записи лога.

    • Если поле Body не является строковым типом, значение преобразуется в строку. В случае сложных типов оно преобразуется в JSON-строку. Для типа kvlist_value см. различия между моделями данных в разделе Обработка API загрузки логов ниже.

    Атрибуты¶

    • Содержат все остальные атрибуты из атрибутов входной записи, содержащихся в разделах: Resource, InstrumentationScope и Attributes.
    • Атрибуты TraceID и SpanID сопоставляются с полями trace_id и span_id, а их значения преобразуются в шестнадцатеричное представление (например, 0xCAFEBABE).
    • Автоматический атрибут. Атрибут dt.auth.origin автоматически добавляется к каждой записи лога, загруженной через API. Этот атрибут представляет собой публичную часть API-ключа, которым источник логов авторизуется для подключения к общему API загрузки логов.

    Все атрибуты предпочтительно должны соответствовать семантическим атрибутам, чтобы Dynatrace мог правильно их интерпретировать.

    • Логи в Grail: Все атрибуты могут использоваться в запросах, хотя Semantic Dictionary помогает Dynatrace Intelligence AI в интерпретации логов. Подробнее см. в Semantic Dictionary.
    • Log Monitoring Classic: Подробнее см. в Семантические атрибуты (Logs Classic).

    Типы данных¶

    Dynatrace поддерживает типы данных OpenTelemetry, описанные в разделах ниже.

    Скалярное значение¶

    Скалярные значения преобразуются следующим образом:

    • Логи в Grail с пользовательской обработкой OpenPipeline (Dynatrace версии 1.295+, Environment ActiveGate версии 1.295+): Поддерживаются все типы данных JSON (строка, число, булево значение, null). Все атрибуты могут использоваться в запросах. Ключи чувствительны к регистру.
    • Логи в Grail с маршрутизацией OpenPipeline в Classic Pipeline: Все ключи атрибутов приводятся к нижнему регистру, все значения атрибутов преобразуются в строки. Все атрибуты могут использоваться в запросах.
    • Log Monitoring Classic: Все ключи атрибутов приводятся к нижнему регистру, все значения атрибутов преобразуются в строки. Пользовательские атрибуты и семантические атрибуты, как правило, могут использоваться в запросах.

    Массив байтов¶

    Массивы байтов преобразуются в строки в кодировке base64. Например, следующий массив

    [0x68, 0x65, 0x6c, 0x6c, 0x6f, 0x20, 0x77, 0x6f, 0x72, 0x6c, 0x64]

    преобразуется и загружается как aGVsbG8gd29yZA==.

    Массив¶

    Значения атрибутов-массивов преобразуются в массивы единого типа. Целевой тип выбирается в соответствии со следующими правилами:

    • Сложные значения, такие как массивы или объекты, преобразуются в строковые значения JSON.
    • Если какое-либо значение в массиве является строкой, или если какое-либо значение должно быть преобразовано в строку (например, объект или массив), целевой тип всего массива --- строка.
    • Если все значения в исходном массиве являются числовыми, целевой тип массива --- числовой.
    • Значения null считаются совместимыми с любым типом.

    Словарь (Map)¶

    Обработка словарей зависит от используемой модели данных. Подробнее см. в разделе Обработка API загрузки логов ниже.

    Лимиты загрузки¶

    Лимиты, применяемые к загруженным запросам логов, их атрибутам и значениям атрибутов, см. в Лимиты по умолчанию Log Management and Analytics и Лимиты по умолчанию Log Monitoring (Logs Classic).

    Обработка API загрузки логов¶

    Существуют две модели данных, определяющие обработку структурированных логов конечными точками загрузки логов: raw и flattened. Различие между ними заключается в способе преобразования атрибутов со значениями типа объект/словарь.

    Если этот параметр конфигурации не указан, поведение по умолчанию зависит от времени создания вашей среды.

    • Для Dynatrace версии 1.331+: Raw.
    • Для Dynatrace версии 1.330 и ранее: Flattened.

    Экранирование в примерах вывода приведено только для целей визуализации. \" учитывается как один символ при тарификации.

    Модель данных Raw¶

    Модель данных raw преобразует содержимое структурированных логов, как описано в разделах ниже. Все следующие примеры применимы к конечным точкам API загрузки логов, доступным на Environment ActiveGate .

    При использовании средств пересылки логов, таких как Fluentbit, Fluentd или Logstash, избегайте использования JSON-парсеров на стороне средства пересылки и позвольте Dynatrace обрабатывать JSON-парсинг самостоятельно. Такой подход снижает нагрузку на обработку на стороне средства пересылки логов и обеспечивает согласованное поведение парсинга.

    Словари и массивы в атрибутах¶

    Для модели данных raw значения атрибутов типа словарь преобразуются в JSON-строку, а значения атрибутов типа массив преобразуются в массив единого типа.

    Словарь в теле¶

    В этом случае поле Body входной записи лога преобразуется в JSON-строку.

    Тело в виде массива¶

    В этом случае массив в теле преобразуется в строку.

    Временная метка¶

    Если timestamp не может быть установлен на основе поля Timestamp, timestamp определяется на основе атрибутов записи лога OTLP и устанавливается по значению первого найденного ключа из следующего списка, проверяемого в указанном порядке, без учета регистра: timestamp, @timestamp, _timestamp, eventtime, date, published_date, syslog.timestamp, time, epochSecond, startTime, datetime, ts, timeMillis, @t.

    Уровень лога¶

    Если loglevel не может быть установлен на основе поля SeverityText или SeverityNumber, loglevel определяется на основе атрибутов записи лога OTLP и устанавливается по значению первого ключа из следующего списка, проверяемого в указанном порядке, без учета регистра: loglevel, status, severity, level, syslog.severity.

    Конфликты имен¶

    В модели данных raw атрибуты ресурса OTLP и атрибуты OTLP копируются в атрибуты верхнего уровня Dynatrace, а тело OTLP копируется в содержимое.

    Dynatrace добавляет к дублирующимся атрибутам префикс с инкрементным счетчиком, например overwritten1.myattribute. Значение счетчика указывает, сколько раз атрибут был обнаружен как дубликат. Это предотвращает коллизии имен, когда атрибуты на разных уровнях OTLP имеют одинаковое имя.

    Модель данных Flattened¶

    В модели данных flattened содержимое структурированных логов преобразуется, как описано в разделах ниже. Все следующие примеры применимы к конечным точкам API загрузки логов, доступным на Environment ActiveGate .

    Словари и массивы в атрибутах¶

    В этом случае значения атрибутов типа словарь сглаживаются (flatten), т.е. заменяются ключами, объединенными точкой (.), до достижения простого значения в иерархии, а значения атрибутов типа массив преобразуются в пользовательскую строку.

    Сглаживание выполняется до максимального уровня вложенности, определенного лимитом Nested objects. Структуры, вложенные глубже этого уровня, заменяются строковым значением <truncated due to nesting limit>. Подробности см. в разделе Лимиты загрузки.

    Словарь в теле¶

    Если поле Body имеет тип kvlist_value (список пар ключ-значение), структура обрабатывается так же, как атрибуты записи лога, включая сглаживание и разрешение конфликтов.

    Атрибуты, найденные в Body, также могут использоваться для установки атрибутов timestamp, loglevel и content записи лога, как описано ниже.

    Тело в виде массива¶

    В этом случае массив в теле преобразуется в строку.

    Временная метка¶

    Если timestamp не может быть установлен на основе поля Timestamp, timestamp определяется на основе одного из следующих источников, проверяемых в указанном порядке:

    1. Содержимое тела (если тело является словарем).
    2. Атрибуты записи лога OTLP.

    Если временная метка берется из тела или записи лога OTLP, она устанавливается по значению первого найденного ключа из следующего списка, проверяемого в указанном порядке, без учета регистра: timestamp, @timestamp, _timestamp, eventtime, date, published_date, syslog.timestamp, time, epochSecond, startTime, datetime, ts, timeMillis, @t.

    Уровень лога¶

    Если loglevel не может быть установлен на основе поля SeverityText или SeverityNumber, loglevel определяется на основе одного из следующих источников, проверяемых в указанном порядке:

    1. Содержимое тела (если тело является словарем).
    2. Атрибуты записи лога OTLP.

    Если loglevel берется из тела или записи лога OTLP, он устанавливается по значению первого ключа из следующего списка, проверяемого в указанном порядке, без учета регистра: loglevel, status, severity, level, syslog.severity.

    Содержимое¶

    Если поле Body имеет тип kvlist_value (список пар ключ-значение), content устанавливается по значению первого найденного ключа в Body из следующего списка, проверяемого в указанном порядке: content, message, payload, body, log.

    Если ни один атрибут не найден среди поддерживаемых ключей содержимого, content устанавливается в пустую строку.

    Конфликты имен¶

    Когда атрибуты сохраняются в сглаженном виде на стороне Dynatrace, могут возникать коллизии имен, если атрибуты на разных уровнях имеют одинаковое имя. Dynatrace решает эту проблему, добавляя к дублирующимся атрибутам префикс overwritten[COUNTER].. Значение счетчика указывает, сколько раз имя атрибута уже было обнаружено как дубликат.

    Например, если у вас есть три атрибута с одинаковым именем my.attribute на уровнях ресурса, области и лога:

    • атрибут ресурса загружается как my.attribute
    • атрибут области загружается как overwritten1.my.attribute
    • атрибут лога загружается как overwritten2.my.attribute

    Дополнительная обработка атрибутов¶

    API загрузки логов дополнительно принимает атрибуты логов через:

    • Параметры запроса
    • Специальный заголовок: X-Dynatrace-Attr

    Эти атрибуты объединяются с атрибутами, предоставленными в запросе лога OpenTelemetry, в соответствии с правилами, описанными ниже.

    Атрибуты из параметров запроса¶

    • Все параметры запроса, переданные конечной точке API загрузки логов, добавляются к атрибутам записи лога.
    • Если ключ параметра встречается несколько раз, все значения сохраняются как атрибут-массив.
    • Ключи и значения следуют тем же правилам разбора атрибутов, что и атрибуты запроса лога.
    • Некоторые параметры обрабатываются API для внутренних целей и никогда не отображаются как атрибуты записи лога, даже если указаны явно (например, используемые в заголовке X-Dynatrace-Options). Полный список зарезервированных имен параметров и их поведение обработки см. в документации API.

    Пример¶

    Атрибуты на основе заголовка (X-Dynatrace-Attr)¶

    API поддерживает специальный заголовок для передачи дополнительных атрибутов:

    otlphttp:


endpoint: /api/v2/otlp


headers:


X-Dynatrace-Attr: region=eu-central-1&team=core

    Правила:

    • Ключи и значения следуют тем же правилам разбора атрибутов, что и параметры запроса.
    • Поддержка множественных значений также доступна внутри атрибутов заголовка.
    • Применяются те же ограничения зарезервированных имен параметров.

    Правила приоритета атрибутов¶

    Когда атрибуты присутствуют в нескольких местах, API загрузки логов применяет приоритет атрибутов, сохраняя при этом значения тела для обеспечения аудита. Атрибуты применяются в следующем порядке:

    • Параметры запроса (наивысший приоритет)
    • Заголовок X-Dynatrace-Attr
    • Запрос лога OpenTelemetry (наименьший приоритет; существующий путь загрузки)

    Поведение при переопределении¶

    Когда атрибуты из параметров запроса или заголовка переопределяют атрибуты запроса лога:

    • Итоговое значение атрибута устанавливается в соответствии с правилами приоритета источников атрибутов.
    • Значения, уже присутствующие в запросе лога, сохраняются и дублируются с ключом overwrittenN.<attribute_key>. Где N --- инкрементное целое число (1, 2, ...) в зависимости от того, сколько значений из запроса лога пришлось сохранить. Это обеспечивает уникальность, даже при множественных конфликтах.
    • Только значения, происходящие из запроса лога, сохраняются под ключами overwrittenN.*. Атрибуты, переопределенные источниками с более высоким приоритетом, не генерируют перезаписанных копий.

    Поведение тарификации¶

    Атрибуты, предоставленные через параметры запроса или заголовки, включаются в расчеты тарификации.

    Для атрибутов с множественными значениями ключ атрибута учитывается в тарификации только один раз, независимо от количества значений.
