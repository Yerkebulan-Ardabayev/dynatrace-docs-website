    Мгновенное реагирование на вторжения

    • Latest Dynatrace

    Время играет решающую роль при работе с инцидентами безопасности. На этой странице показано, как вы можете использовать Dynatrace для ускорения реагирования на инциденты в два этапа:

    • При обнаружении атаки Dynatrace может автоматизировать уведомление и анализ, чтобы нужные команды были проинформированы и получили действенную информацию для начала работы над проблемой.
    • При расследовании проблемы каждое новое открытие направляет следующие шаги в процессе реагирования на инциденты. Благодаря бессхемным запросам Grail, Dynatrace позволяет оставаться гибким в этом критически важном по времени процессе.

    Целевая аудитория¶

    Эта страница предназначена для команд безопасности, анализирующих инциденты безопасности, таких как команда реагирования на инциденты.

    Сценарий¶

    Далее мы рассмотрим сценарий, в котором выявление атаки, исследование её масштаба, определение ответственных владельцев объектов и устранение атаки занимают часы, а иногда даже дни.

    • Такие инструменты, как AWS GuardDuty, независимо генерируют события при обнаружении подозрительной активности.
    • Обработка входящей нагрузки при одновременном повышении уровня безопасности невозможна.
    • Вся корреляция и взаимодействие выполняются вручную, что вызывает разочарование из-за разрозненности.
    • Было создано несколько внутренних интеграционных инструментов, сложных в управлении; они полезны, но не обеспечивают надёжную и безопасную автоматизацию.
    • Типичный подозрительный инцидент требует дней для эскалации и квалификации.

    Запрос¶

    Команда хочет быстро:

    • Определить, когда происходит возможная атака.
    • Исследовать масштаб и определить, что затронуто: например, затрагивает ли атака только одну изолированную непроизводственную систему или угрожает критически важной части среды.
    • Определить ответственных владельцев объектов.
    • Устранить атаку.

    Цели¶

    • Эффективность: Команда должна иметь возможность реагировать на атаки значительно быстрее.
    • Гибкость: Команда должна иметь больше гибкости в реагировании на инциденты безопасности.

    Результат¶

    Сочетая возможности автоматизации Dynatrace с аналитикой данных, связанных с безопасностью, наше решение помогает командам безопасности быстрее реагировать на атаки. Команда автоматически сканирует все полученные логи на наличие паттернов, которые могут указывать на возможные атаки. Поскольку каждая атака уникальна, они используют бессхемные запросы с мгновенными ответами для быстрого определения масштаба атаки, тем самым сокращая необходимое время с дней до минут.

    Как это работает¶

    Контекст¶

    Логи из вашей мониторируемой Dynatrace среды поступают в Grail через приём логов. При обнаружении атаки создаётся проблема Dynatrace.

    1. Автоматизация уведомлений о вторжении¶

    Рабочий процесс автоматически запускается при возникновении этого типа проблемы. Рабочий процесс собирает, обрабатывает и обогащает данные контекстом, а затем преобразует полученную информацию в уведомления по выбранным вами каналам.

    Пример настройки автоматизации уведомлений об атаках см. в разделе Автоматизация уведомлений о вторжении.

    2. Мгновенные запросы¶

    На основе полученной информации вы можете немедленно реагировать на обнаруженные факты и выполнять дополнительные расследования, запуская последовательность DQL-запросов в Notebooks, адаптированных к типу атаки.

    Подробнее см. Мгновенные запросы.

    Предварительные требования¶

    • Dynatrace версии 1.283+
    • Настройка приёма логов (приём инцидентов безопасности в Grail).
    • Настройка команд владения (позволяет рабочему процессу назначать инциденты безопасности на основе владения затронутым объектом).
    • Настройка Jira Connector (позволяет рабочему процессу преобразовывать результаты в тикеты Jira).
    • Настройка Slack Connector (позволяет рабочему процессу отправлять результаты в каналы Slack).

    Хотя в текущем сценарии используются Slack и Jira в качестве каналов уведомлений, доступны и другие интеграции. Подробнее см. Интеграции Workflows. * Базовые знания о том, как:

    • Использовать Dynatrace Query Language (DQL)
    • Создавать рабочие процессы

    • Убедитесь, что включены следующие разрешения.

    • Grail: storage:logs:read. Инструкции см. в разделе Назначение разрешений в Grail.

    • Workflows: Разрешения на доступ, просмотр, запись и выполнение рабочих процессов. Подробнее см. Авторизация.

    2. Мгновенные запросы¶

    После получения уведомления команда безопасности может немедленно реагировать на обнаруженные факты и мгновенно выполнять дополнительные DQL-запросы в Notebooks, не зная заранее, где находится нужная информация. В экстренной ситуации это имеет решающее значение, так как быстрый ответ может гарантировать сдерживание атаки.

    Ниже приведены примеры запросов к Grail в случае веб-атаки.

    Логи с IP-адреса атакующего¶

    Пример запроса:

    fetch logs, scanLimitGBytes:-1


| filter log.source == "/var/log/nginx/access.log"


| filter net.peer.ip == "<<IP ADDRESS>>"


| sort timestamp desc

    Успешные веб-запросы с IP-адреса атакующего¶

    Пример запроса:

    fetch logs, scanLimitGBytes:-1


| filter log.source == "/var/log/nginx/access.log"


| filter net.peer.ip == "<<IP Address>>"


| filter http.status_code == "200"


| sort timestamp desc

    Обзор кодов статуса и кодов ответов¶

    Пример запроса:

    fetch logs, scanLimitGBytes:-1


| filter log.source == "/var/log/nginx/access.log"


| filter net.peer.ip == "<<IP Address>>"


| summarize requests=count(), by:{http.status_code, http.user_agent}


| sort http.status_code

    Полезные нагрузки, используемые с IP-адреса атакующего¶

    Пример запроса:

    fetch logs, scanLimitGBytes:-1


| filter log.source == "/var/log/nginx/access.log"


// Successful requests only


| filter http.status_code == "200"


| filter net.peer.ip == "<<IP Address>>"


| summarize requests=count(), by:{http.target}


| sort requests DESC

    Успешные запросы с заданного IP-адреса, отсортированные по размеру ответа¶

    Пример запроса:

    fetch logs, scanLimitGBytes:-1


| filter log.source == "/var/log/nginx/access.log"


| fields timestamp, net.peer.ip, http.target, http.status_code, http.response_content_length, http.user_agent


| filter http.status_code == "200"


// Filter for a specific IP address


| filter net.peer.ip == "<<IP Address>>"


| sort toLong(http.response_content_length) DESC

    Полезные нагрузки в журналах доступа (как успешные, так и неуспешные)¶

    Пример запроса:

    fetch logs, scanLimitGBytes:-1


| filter log.source == "/var/log/nginx/access.log"


| fields payload = "<<PAYLOAD>>", timestamp, net.peer.ip, http.method, http.target, http.status_code, http.request.header.referrer, http.response_content_length, http.user_agent, content


| filter contains(content, payload)

    Успешные SSO-входы с IP-адреса атакующего¶

    Пример запроса:

    fetch logs, scanLimitGBytes: -1


// Search for logins


| filter log.source == "/var/log/sso.log"


// Search for successful logins from a given IP address


| filter contains(content, "user login successful") and contains(content, "<<IP address>>")


| sort timestamp desc

    В какие среды атакующий выполнил вход¶

    Пример запроса:

    fetch logs, scanLimitGBytes: -1


// Search for logins


| filter log.source == "/var/log/sso.log"


// Search for logins from a given account id


| filter contains(content, "<<Account ID>>") and contains(content, "tenant: ")


| sort timestamp desc

    Опережайте угрозы¶

    Вы можете использовать приведённые выше инструкции как строительные блоки для автоматизации типичных шагов в процессе реагирования на инциденты. Они помогут вам быстрее реагировать на инциденты безопасности, тем самым снижая их влияние.
