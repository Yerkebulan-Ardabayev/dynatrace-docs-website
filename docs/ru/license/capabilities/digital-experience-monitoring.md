## Real User Monitoring (RUM)

Dynatrace RUM — это процесс мониторинга производительности, который собирает подробные данные о взаимодействии каждого пользователя с вашим приложением, будь то веб-приложение в браузере или мобильное приложение для Android или iOS.
Real User Monitoring собирает данные по множеству метрик.
Например, данные, собираемые по действиям загрузки, могут включать метрики начала навигации, начала запроса и индекса скорости.

## Real User Monitoring с Session Replay

Session Replay помогает фиксировать и визуально воспроизводить полный цифровой опыт ваших конечных пользователей.
Взаимодействия пользователей с вашим приложением записываются и могут быть воспроизведены в формате, похожем на видео.
Session Replay упрощает воспроизведение производственных проблем и ошибок, а также понимание таких проблем, как некорректное отображение страниц и бесконечные индикаторы загрузки.

## Свойства Real User Monitoring

Dynatrace собирает обширную информацию о производительности ваших приложений. Эта информация может быть дополнена ценными метаданными и преобразована в свойства действий пользователей и пользовательских сессий. Эти свойства позволяют выполнять мощные запросы, сегментацию и агрегацию по собранным метаданным.

Любое свойство RUM может быть настроено для приложения.

## Browser Monitor или Clickpath

Мониторинг браузера имитирует посещение пользователем вашего приложения с помощью современного обновлённого веб-браузера для мониторинга критически важных бизнес-процессов вашего приложения.
Мониторы браузера могут быть настроены для запуска из любых публичных или частных локаций Dynatrace с частотой до одного раза в пять минут и отправляют оповещения, когда приложение становится недоступным или когда производительность существенно снижается.
Кроме того, они могут использоваться для проверки доступности внутренних ресурсов, недоступных извне вашей сети.

Рекордер Dynatrace используется для захвата точной последовательности кликов и пользовательских вводов, доступности и производительности.
В качестве альтернативы, для достижения той же цели можно создать скриптовую последовательность действий.

## Интеграция сторонних синтетических API

Сторонние эндпоинты Synthetic API позволяют отправлять сторонние синтетические данные и события в Dynatrace.
API также позволяет корректировать результаты, ранее отправленные в Dynatrace.

## HTTP Monitor

Синтетические HTTP-мониторы могут быть созданы для проверки доступности ресурсов — веб-сайтов или эндпоинтов API.
Кроме того, они могут использоваться для проверки доступности внутренних ресурсов, недоступных извне вашей сети.

## Синтетический NAM-мониторинг

Мониторы доступности сети (NAM) не имеют отдельной строки в прайс-листе Dynatrace. Вместо этого тарификация основана на [количестве точек данных метрик](metrics.md "Узнайте, как рассчитывается потребление Dynatrace Metrics на базе Grail в модели подписки Dynatrace Platform Subscription."), генерируемых при каждом выполнении теста NAM. Для получения дополнительной информации обратитесь к вашему менеджеру по работе с клиентами Dynatrace.

Расчёт точек данных метрик

Следующие детали относятся к точкам данных метрик:

* Точки данных метрик, связанные с выполнением мониторов и шагов, не тарифицируются.
* На тарификацию влияет только потребление метрик, создаваемых на уровне запросов.
* Каждое выполнение запроса в тестах ping генерирует 6 точек данных метрик.

  + Количество пакетов, используемых в тесте ping, не влияет на количество создаваемых метрик или тарификацию.
  + Количество пакетов не влияет на стоимость.
* Каждое выполнение запроса в тестах TCP/DNS генерирует 3 точки данных метрик.
* Стоимость остаётся одинаковой независимо от того, создаёте ли вы несколько тестов с одним запросом или один тест с множеством запросов для одного и того же набора хостов или устройств.
