## Начало работы

Site Reliability Guardian -- это приложение Dynatrace, которое автоматизирует анализ влияния изменений для проверки доступности, производительности и целей по ёмкости сервисов в различных системах. Оно позволяет инженерам DevOps-платформ принимать правильные решения о релизах и предоставляет SRE-инженерам возможность применять цели уровня обслуживания (SLO) для критически важных сервисов.

## Концепции

Site Reliability Guardian основан на следующих концепциях:

### 1. Guardian

Guardian -- это группировка объективов. Он строится вокруг набора сущностей, отражающих сервис или приложение, которое вы хотите защитить.

Guardian предоставляет вам [рабочий процесс автоматизации](#automation) по умолчанию, который выполняет [валидацию](#validation) объективов. В результате guardian всегда представляет последний результат валидации, полученный из объективов.

Мы поддерживаем два типа guardian.
Хотя эти два типа не отличаются с концептуальной точки зрения, существуют технические и семантические различия, которые их разделяют.

### Lifecycle guardian (события SDLC)

* Читает и записывает события SDLC, которые затем можно использовать для генерации аналитики.") как результаты валидации.
* Соответствует спецификации событий валидации в Semantic Dictionary.").
* Предназначен для использования в контексте жизненного цикла разработки программного обеспечения.

  + В качестве шлюза качества в сценариях прогрессивной доставки
  + В качестве индикатора производительности после нагрузочного тестирования
  + В качестве непрерывного монитора состояния для сервисов и компонентов

### Business guardian (бизнес-события)

* Читает и записывает бизнес-события как результаты валидации
* Предназначен для использования на бизнес-уровне и получения аналитики поведения приложений

Поскольку эти два типа guardian имеют различные источники данных (`bizevents` и `events`) и различные структуры данных событий, вам необходимо адаптировать DQL-запросы, нацеленные на события валидации guardian в Notebooks или Dashboards, при переключении с одного типа на другой.
Подробнее о структурных различиях см. в разделе Структура событий Site Reliability Guardian.

Вы можете создать максимум 1000 guardian.

### 2. Объектив

Объективы -- это средства измерения производительности, доступности, ёмкости и безопасности ваших сервисов. Объективы измеряются индикаторами. Вы можете определить объектив для вашего guardian, который будет валидироваться по запросу или автоматически.

Вы можете создать максимум 50 объективов для каждого guardian.

### 3. Индикатор

Индикатор -- это значение, по которому проверяются пороги предупреждения и отказа с использованием оператора сравнения. Для получения значения индикатора используйте DQL.

### 4. Статические пороги

[Статические пороги предупреждения и отказа](https://dt-url.net/ta5s0pxa) определяют, соответствует ли измеренное значение индикатора объективу, близко к нарушению объектива или нарушает объектив.

Пороги предупреждения и отказа необязательны; результат валидации объектива может варьироваться:

* Если установлены оба порога -- предупреждения и отказа, валидация объектива может вернуть предупреждение, отказ или успех.
* Если установлен только порог предупреждения, валидация объектива может вернуть предупреждение или успех.
* Если пороги не установлены, валидация объектива не возвращает статус, а используется в информационных целях.

### 5. Автоадаптивные пороги

[Автоадаптивные пороги](https://dt-url.net/c55q0p1u) -- это динамические лимиты, которые корректируются со временем на основе предыдущих валидаций. Если поведение объектива изменяется, порог адаптируется автоматически.

* Автоадаптивные пороги доступны только при получении данных через DQL.
  Анализатор порогов Dynatrace Intelligence требует минимум 5 валидаций для вступления автоадаптивных порогов в силу. Во время этой фазы обучения валидация объектива возвращает информационный статус. Все последующие валидации будут использовать автоадаптированные пороги, влияя на общий результат валидации.
* Поддерживается переключение со статических на автоадаптивные пороги.

### 6. Оператор

Оператор сравнения определяет, выполнен ли объектив: индикатор меньше или равен (Меньшее значение лучше для моего результата) или больше или равен (Большее значение лучше для моего результата) порогу предупреждения и отказа.

### 7. Теги

Для организации ваших guardian вы можете назначать им теги. Теги используют формат `key:value`, при этом значение является необязательным.

Чтобы назначить тег вашему guardian, укажите его в разделе **Add tags to your guardian** при создании guardian или добавьте тег позже в режиме редактирования.

Чтобы отфильтровать список всех guardian по тегу, введите тег в поле **Search by name or tag** -- страница автоматически обновится и покажет только guardian с соответствующими тегами.

Этот DQL-запрос показывает первое бизнес-событие `guardian.validation.objective` с определённым идентификатором guardian и анализирует поле тегов guardian для извлечения конкретного значения тега из JSON события.

```
fetch bizevents |


filter event.type == "guardian.validation.objective" AND guardian.id == "vu9U3hXa3q0AAAABADFhcHA6ZHluYXRyYWNlLnNpdGUucmVsaWFiaWxpdHkuZ3VhcmRpYW46Z3VhcmRpYW5zAAZ0ZW5hbnQABnRlbmFudAAkMWNiZDVkYWYtZThhNi0zMDkxLWFkOGQtMmU5NDNmNWJmZWJmvu9U3hXa3q0" |


limit 1 |


parse guardian.tags, "JSON:parsed_guardian_tags"
```

Этот DQL-запрос показывает все бизнес-события `guardian.validation.finished` от guardian с тегом `tagkey:my-tagged-guardian`.

```
fetch bizevents


| filter event.type == "guardian.validation.finished"


| expand guardian.tags


| filter contains(guardian.tags, "my-tagged-guardian")
```

### 8. Действие рабочего процесса Guardian

Вы можете автоматизировать выполнение guardian через Workflows, привязывая выполнение guardian к [событию](../analyze-explore-automate/workflows/trigger/event-trigger.md#event-trigger "Руководство по созданию триггеров автоматизации рабочих процессов на основе событий в Dynatrace Workflows.") или вызову API.

### Добавление действия guardian в существующий рабочий процесс или создание нового

Одни и те же финальные шаги применяются независимо от того, добавляете ли вы guardian в существующий рабочий процесс или создаёте новый.

#### Редактирование существующего рабочего процесса

1. Перейдите в **Workflows** ![Workflows](https://dt-cdn.net/images/workflows-1024-b5708f3cf9.webp "Workflows") и откройте ваш рабочий процесс.
2. Перейдите к последней задаче, которая должна быть предшественником действия валидации guardian, и выберите ![Add](https://dt-cdn.net/images/dashboards-app-menu-plus-7e9b7c3547.svg "Add"), чтобы просмотреть доступные действия.

#### Создание нового рабочего процесса

1. Перейдите в **Workflows** ![Workflows](https://dt-cdn.net/images/workflows-1024-b5708f3cf9.webp "Workflows") и выберите ![Add](https://dt-cdn.net/images/dashboards-app-menu-plus-7e9b7c3547.svg "Add") **Workflow**.
2. Выберите триггер.
3. На узле триггера выберите ![Add](https://dt-cdn.net/images/dashboards-app-menu-plus-7e9b7c3547.svg "Add"), чтобы просмотреть доступные действия.

#### Настройка действия валидации guardian

1. Найдите `Site Reliability Guardian` на панели **Choose action**.
2. На вкладке **Input** у вас есть два варианта выбора необходимого guardian:

* Выберите guardian из списка.
* Используйте выражение для извлечения guardian из триггерного события или предыдущего действия рабочего процесса.
* Настройте временной интервал валидации.

Подробнее см. также Валидация Site Reliability Guardian, Автоматизация валидации релизов, Наблюдаемость тестового пайплайна

### Создание рабочего процесса со страницы всех guardian

Вы можете запустить guardian автоматически с помощью рабочего процесса.

Чтобы создать рабочий процесс для этого guardian, выполните следующие шаги:

1. Перейдите к вашему guardian.
2. Чтобы автоматизировать триггер для вашего guardian, на странице **All guardians** наведите курсор на ваш guardian или откройте его, а затем выберите **Automate**. **Workflows** ![Workflows](https://dt-cdn.net/images/workflows-1024-b5708f3cf9.webp "Workflows") откроется в новой вкладке браузера. Вы также можете получить доступ к **Automate** из деталей валидации.

   Этот шаг создаёт новый рабочий процесс для вашего guardian с триггером события и действием **run validation**.

   При создании рабочего процесса таким образом настраиваются следующие параметры; однако убедитесь, что вы адаптировали их по мере необходимости.

Вы можете создать новый рабочий процесс, выбрав ![Workflows](https://dt-cdn.net/images/workflows-1024-b5708f3cf9.webp "Workflows") **Automate**. При создании рабочего процесса таким образом настраиваются следующие параметры, но обязательно адаптируйте их по мере необходимости.

* Создаётся новый рабочий процесс с триггером события и действием **run validation**.
* В зависимости от типа guardian -- Lifecycle или Business -- **тип события** и **запрос фильтрации** устанавливаются соответствующим образом.

  + Для **Lifecycle guardian** (события SDLC) **тип события** устанавливается как `events`, а **запрос фильтрации** по умолчанию: `event.type == "validation.triggered" AND event.kind == "SDLC_EVENT"`.
  + Для **Business guardian** (бизнес-события) **тип события** устанавливается как `bizevents`, а **запрос фильтрации** по умолчанию: `event.type == "guardian.validation.triggered"`.
  + Если у guardian определены теги, они используются как дополнительные фильтры в **запросе фильтрации** триггера.
* Единственным действием рабочего процесса является соответствующее действие валидации guardian.

Действие валидации guardian генерирует следующий вывод и передаёт его последующим действиям рабочего процесса.

Чтобы узнать больше о рабочих процессах для guardian, выберите  >  **Get started with Automation**.

### 9. Валидация

Если рабочий процесс создан, ваш guardian может быть валидирован автоматически в зависимости от выбранного триггера. Вы также можете выполнить валидацию вручную.

### Обзор валидации

По умолчанию страница **All guardians** отображает список всех guardian.

Подробнее о странице **All guardians** см. в разделе Список и работа с вашими guardian.

### Автоматическая валидация

Подписки на события в рабочем процессе определяют, когда валидация guardian запускается автоматически.

### Ручная валидация

Вы можете выполнить валидацию guardian, нажав кнопку  **Validate** на экране обзора или на экране деталей валидации.

* Выберите временной интервал валидации.
* Нажмите кнопку  **Validate**.

### Результат отдельного объектива

Для каждого объектива валидация возвращает полученное значение и классификацию. Серьёзность варьируется от наивысшей (1) до наименьшей (5).

| Серьёзность | Название | Описание |
| --- | --- | --- |
| 1 | **Error** | Объектив не удалось валидировать из-за ошибки при получении индикатора. |
| 2 | **Fail** | Значение нарушает порог отказа; объектив не выполнен. |
| 3 | **Warning** | Значение находится в диапазоне предупреждения; объектив выполнен, но близок к отказу. |
| 4 | **Pass** | Значение находится в целевом диапазоне; объектив выполнен. |
| 5 | **Info** | Нет классификации, но значение объектива может использоваться в информационных целях. |

### Общий результат валидации

После завершения валидации каждого объектива guardian использует наиболее серьёзный из индивидуальных результатов валидации в качестве общего результата. Примеры использования этого результата включают:

* Принятие решения о релизе в вашем пайплайне доставки.
* Отчётность о текущем состоянии вашего сервиса.

### 10. Сегменты

Используйте Segments в объективах на основе DQL для логической структуризации и удобной фильтрации данных наблюдаемости.
