# Назначение команд-владельцев отслеживаемым сущностям

* Classic
* Практическое руководство
* Время чтения: 9 мин
* Обновлено 16 сентября 2025 г.

Вы можете назначать [команды-владельцев](ownership-teams.md "Определение команд с идентификаторами, описаниями, обязанностями и маршрутной информацией для владения сущностями.") **любой отслеживаемой сущности** в Dynatrace, используя основной идентификатор команды или дополнительные идентификаторы (определяемые при [настройке команды](ownership-teams.md "Определение команд с идентификаторами, описаниями, обязанностями и маршрутной информацией для владения сущностями.")). Это позволяет мгновенно найти ответственную команду и связаться с ней при возникновении проблем с сущностью, например, если сущность затронута уязвимостью.

Вы можете назначать владение несколькими способами, такими как метки и аннотации Kubernetes, метаданные хоста, переменные окружения и теги (в том числе через [API пользовательских тегов](../../dynatrace-api/environment-api/custom-tags.md "Управление пользовательскими тегами отслеживаемых сущностей через API Dynatrace.") Dynatrace).

## Поддерживаемые методы назначения владения

Поддерживаются следующие методы назначения владения и типы сущностей.

* **Метаданные развёртывания или конфигурации** Рекомендуется

  | Метод | Типы сущностей |
  | --- | --- |
  | Метки и аннотации Kubernetes | Все объекты Kubernetes |
  | Метаданные хоста через `oneagentctl` или `hostcustomproperties.conf` | Хосты |
  | Переменные окружения | Процессы |
* **Теги** (ручные, автоматические и через API) — все отслеживаемые сущности

Мы настоятельно рекомендуем использовать предпочтительный метод назначения владения в зависимости от типа сущности, как описано в разделах ниже.

* [Метки и аннотации Kubernetes для объектов Kubernetes](#kubernetes)
* [Метаданные для хостов](#host-metadata)
* [Переменные окружения для процессов](#process-env-variables)
* [Теги (ручные, автоматические и через API) для всех остальных сущностей](#tags)

Наши рекомендации основаны на наиболее эффективных методах обеспечения адекватного покрытия владением сущностей. Подробнее см. в разделе [Лучшие практики владения сущностями](best-practices.md "Советы и лучшие практики для обеспечения адекватного покрытия владением сущностей").

## Формат применения информации о владении

Независимо от используемого метода, информация о владении применяется к сущностям в виде **пар ключ-значение**. Ключи по умолчанию `owner` и `dt.owner` доступны в каждой среде мониторинга — см. **Settings** > **Ownership** > **Configuration**. Однако вы можете изменить или удалить ключи по умолчанию и [определить собственные](#custom-keys).

Значением всегда является уникальный идентификатор команды, указанный при создании команды. Вы также можете использовать дополнительные идентификаторы. (Подробнее об определении этих значений см. в разделе [Создание и управление командами для владения сущностями](ownership-teams.md "Определение команд с идентификаторами, описаниями, обязанностями и маршрутной информацией для владения сущностями.").)

Сущность может иметь более одной команды-владельца. Для этого можно применить несколько пар ключ-значение, как показано в разделах ниже.

### Пользовательские ключи для информации о владении

Вы можете определить до пяти ключей для применения информации о владении.

1. Перейдите в **Settings** > **Ownership** > **Configure**.
2. Выберите **Add key**.
3. Введите ключ (по умолчанию **Enabled**).
4. **Save changes**.

![Пользовательские ключи для владения](https://dt-cdn.net/images/ownership-custom-keys-1482-1a7f6383f7.png)

Когда вы **отключаете или удаляете ключ**, сущности, назначенные командам через такие ключи, больше не будут отображать или содержать метаданные владения. Однако такие ключи будут отображаться как обычные теги, применённые к сущностям.

### Дополнительные требования к ключам владения

* Вы можете определить максимум пять и минимум один ключ.
* Вы можете использовать любой из определённых ключей в качестве префикса в парах ключ-значение. Например, для ключей `owner` и `dt.owner` можно использовать `owner-1` и `dt.owner-test`.

## Метки и аннотации Kubernetes

Вы можете указать владение командой для различных объектов Kubernetes, таких как Deployments, Pods, Services или пространства имён. Чтобы обеспечить адекватное покрытие владением объектов Kubernetes, что особенно важно для кратковременных сущностей, таких как Pods, предоставляйте информацию о владении в виде меток или аннотаций Kubernetes с [парами ключ-значение](#format) в файле спецификации развёртывания, например `deployment.yaml`.

Мы рекомендуем определять владение для Deployment и всех других объектов, для которых вы хотите обеспечить покрытие владением. См. также [Лучшие практики владения сущностями](best-practices.md "Советы и лучшие практики для обеспечения адекватного покрытия владением сущностей").

* Установка идентификатора команды в качестве метки или аннотации для Deployment, CronJob, Job, DaemonSet или StatefulSet предоставляет информацию о владении командой для соответствующих сущностей `CLOUD_APPLICATION`. Обратите внимание, что это владение не распространяется на `CLOUD_APPLICATION_INSTANCE`.

  В этом примере двойное владение устанавливается через две метки для Deployment. Каждая метка имеет уникальный ключ. **Уникальные ключи являются обязательным требованием в метках и аннотациях Kubernetes**.

  ```
  apiVersion: apps/v1



  kind: Deployment



  metadata:



  name: demo



  labels:



  dt.owner-1: my-team-1 # Dual team ownership defined for the Deployment



  dt.owner-2: my-team-2



  spec:
  ```

  Пример кода ниже показывает аннотацию для владения на уровне Deployment.

  ```
  apiVersion: apps/v1



  kind: Deployment



  metadata:



  name: demo



  annotations:



  dt.owner: my-team # Ownership defined for the Deployment
  ```
* Установите метки для Pods, чтобы указать владение для соответствующей сущности `CLOUD_APPLICATION_INSTANCE`. Укажите владение для каждого Pod.

  ```
  apiVersion: apps/v1



  kind: Deployment



  metadata:



  name: demo



  spec:



  replicas: 3



  selector:



  matchLabels:



  app: demo



  template:



  metadata:



  labels:



  app: demo



  dt.owner: my-team # Ownership defined for the Pod



  spec:
  ```

  Пример кода ниже показывает манифест для Pod с аннотацией `dt.owner: myTeam`.

  ```
  apiVersion: v1



  kind: Pod



  metadata:



  name: annotations-demo



  annotations:



  imageregistry: "https://hub.docker.com/"



  dt.owner: my-team



  spec:



  containers:



  - name: nginx



  image: nginx:1.14.2



  ports:



  - containerPort: 80
  ```
* Для **процесса** укажите владение с помощью пар ключ-значение через **переменную окружения** `DT_CUSTOM_PROP`, определённую для контейнера.

  В переменных окружения можно указать несколько значений для одного ключа. В этом примере двойное владение определяется с помощью ключа `owner`.

  ```
  apiVersion: apps/v1



  kind: Deployment



  metadata:



  name: demo



  spec:



  replicas: 3



  selector:



  matchLabels:



  app: demo



  template:



  spec:



  containers:



  - name: demo



  image: demo:1.0.0



  env:



  - name: DT_CUSTOM_PROP ## Environment variable



  value: "owner=team-automation owner=team-dev" # Dual ownership for the process; team IDs are team-automation and team-dev.
  ```
* Метки Kubernetes для Service

  ```
  apiVersion: v1



  kind: Service



  metadata:



  name: my-service



  labels:



  dt.owner: team-a # Ownership defined for the Service



  spec:



  selector:



  app.kubernetes.io/name: MyApp



  ports:



  - protocol: TCP



  port: 80



  targetPort: 9376
  ```
* Метки Kubernetes для пространства имён

  ```
  apiVersion: v1



  kind: Namespace



  metadata:



  name: my-namespace



  labels:



  dt.owner: team-a # Ownership defined for the namespace
  ```

## Метаданные хоста

Для хостов мы рекомендуем использовать [метаданные хоста](../../observe/infrastructure-observability/hosts/configuration/define-tags-and-metadata-for-hosts.md#host-metadata "Узнайте, как назначать теги и устанавливать дополнительные свойства для отслеживаемого хоста.") в качестве основного метода назначения владения.

Подробнее о [определении тегов и метаданных для хостов](../../observe/infrastructure-observability/hosts/configuration/define-tags-and-metadata-for-hosts.md "Узнайте, как назначать теги и устанавливать дополнительные свойства для отслеживаемого хоста.") и [командном интерфейсе OneAgent `oneagentctl`](../../ingest-from/dynatrace-oneagent/oneagent-configuration-via-command-line-interface.md "Узнайте, как выполнять некоторые задачи настройки OneAgent без необходимости переустановки.").

Вы также можете назначить владение хостам с помощью [тегов](#tags).

### Через `oneagentctl`

OneAgent версии 1.189+ Для хостов с OneAgent версии 1.189+ используйте командный интерфейс `oneagentctl` после установки для задания свойства метаданных для отдельного хоста. Запускайте `oneagentctl` с правами администратора или root из следующих расположений по умолчанию.

* Windows: `%PROGRAMFILES%\dynatrace\oneagent\agent\tools`
* Все Unix-подобные системы: `/opt/dynatrace/oneagent/agent/tools`

Этот пример для Unix-подобных систем использует `--set-host-property` для установки владения с [парой ключ-значение](#format) `owner-1=team-automation`, где `team-automation` — это [основной идентификатор команды или дополнительный идентификатор](ownership-teams.md "Определение команд с идентификаторами, описаниями, обязанностями и маршрутной информацией для владения сущностями.").

```
./oneagentctl --set-host-property owner-1=team-automation
```

### Через `hostcustomproperties.conf`

OneAgent версии 1.187 и ранее Для хостов с OneAgent 1.187 и ранее создайте или отредактируйте файл конфигурации OneAgent `hostcustomproperties.conf` в следующих расположениях.

* Windows: `%PROGRAMDATA%\dynatrace\oneagent\agent\config`
* Все Unix-подобные системы: `/var/lib/dynatrace/oneagent/agent/config`

Этот пример для Unix-подобных систем устанавливает владение хостом с [парой ключ-значение](#format) `dt.owner-1=team-automation`, где `team-automation` — это [основной идентификатор команды или дополнительный идентификатор](ownership-teams.md "Определение команд с идентификаторами, описаниями, обязанностями и маршрутной информацией для владения сущностями.").

```
cat hostcustomproperties.conf dt.owner-1=team-automation
```

## Переменные окружения процессов

Для процессов мы рекомендуем указывать владение в парах ключ-значение через переменную окружения `DT_CUSTOM_PROP`.

Мы **не рекомендуем** использовать теги для назначения владения процессам или группам процессов.

Подробнее об [определении метаданных групп процессов](../../observe/infrastructure-observability/process-groups/configuration/define-your-own-process-group-metadata.md "Настройте собственные метаданные процессов на основе уникальных потребностей вашей организации или среды."), например, настройке переменной окружения `DT_CUSTOM_PROP` для [IIS](../../observe/infrastructure-observability/process-groups/configuration/define-your-own-process-group-metadata.md#iis "Настройте собственные метаданные процессов на основе уникальных потребностей вашей организации или среды.") и [Windows](../../observe/infrastructure-observability/process-groups/configuration/define-your-own-process-group-metadata.md#windows "Настройте собственные метаданные процессов на основе уникальных потребностей вашей организации или среды.").

```
export DT_CUSTOM_PROP dt.owner=DemoTeam
```

См. раздел [Формат применения информации о владении](#format) для получения информации обо всех существующих вариантах пар ключ-значение или для создания собственных.

## Теги

**Используйте теги для назначения владения только тем сущностям, которые не покрываются описанными выше методами**. Обычно это такие сущности, как фронтенд-приложения, которые стабильны и немногочисленны.

### Методы тегирования

Вы можете использовать теги для назначения владения в [парах ключ-значение](#format) любой отслеживаемой сущности — подробнее о [тегировании](../../../common/manage/tags-and-metadata.md "Используйте теги и метаданные для организации данных в вашей среде Dynatrace.").

![Владение через ручной тег](https://dt-cdn.net/images/ownership-manual-tag-1004-f7c90a9b42.png)

* [Ручные теги](../../manage/tags-and-metadata/setup/how-to-define-tags.md#manual "Узнайте, как определять и применять теги вручную и автоматически.") с использованием определённых ключей или префиксов ключей (таких как `owner` и `dt.owner`) легко применять через веб-интерфейс.
* Вы также можете настроить [правила автоматического тегирования](../../manage/tags-and-metadata/setup/how-to-define-tags.md#automatic "Узнайте, как определять и применять теги вручную и автоматически.") через веб-интерфейс.
* Мы рекомендуем использовать [API пользовательских тегов](../../dynatrace-api/environment-api/custom-tags.md "Управление пользовательскими тегами отслеживаемых сущностей через API Dynatrace.") вместо правил автоматического тегирования для назначения владения сущностям. API пользовательских тегов позволяет удобно применять теги к большой группе сущностей в рамках одного вызова API, выполняемого немедленно.

Обратите внимание, что теги, применённые вручную, могут быть удалены вручную. Автоматически применённые теги нельзя удалить вручную с отдельных сервисов, групп процессов, экземпляров групп процессов, приложений или хостов.

См. [Лучшие практики владения сущностями](best-practices.md "Советы и лучшие практики для обеспечения адекватного покрытия владением сущностей") для получения дополнительной информации о преимуществах и ограничениях тегирования для назначения владельцев сущностям. См. также [Лучшие практики масштабирования правил тегирования и зон управления](../../manage/tags-and-metadata/basic-concepts/best-practice-tagging-at-scale.md "Оптимизируйте правила автоматического тегирования и зон управления для ускорения процесса автоматического назначения.").

### Разрешения

Для добавления, изменения или удаления владения через API пользовательских тегов вам требуются **все** следующие разрешения.

* [Разрешение токена](../../manage/identity-access-management/access-tokens-and-oauth-clients/access-tokens.md "Узнайте о концепции токена доступа и его областях.") `entities.write`
* Разрешение токена `settings.read` или [политика IAM](../../manage/identity-access-management/permission-management/manage-user-permissions-policies.md "Работа с политиками") `ALLOW settings:objects:read WHERE settings:schemaId = "builtin:ownership.teams";`

Для добавления, изменения или удаления тегов через веб-интерфейс вам необходимо [разрешение](../../manage/identity-access-management/permission-management/role-based-permissions.md "Разрешения на основе ролей") **Manage monitoring settings** на уровне среды или зоны управления.

## Просмотр информации о владении в веб-интерфейсе

Информация о владении доступна только для [унифицированных страниц анализа](../../ingest-from/extend-dynatrace/extend-ui/unified-analysis.md "Расширьте веб-интерфейс Dynatrace с помощью адаптированных к сущностям унифицированных страниц анализа.").

Для хостов и всех сущностей Kubernetes выберите **Owners** на странице деталей сущности для просмотра информации о владении.

В этом примере показана рабочая нагрузка Kubernetes, соответствующая сущности `CLOUD_APPLICATION`. Разверните название команды для просмотра её деталей. Выберите ![Edit](https://dt-cdn.net/images/edit-icon-a083c62c49.svg "Edit") в карточке **Ownership** для [редактирования деталей команды](ownership-teams.md "Определение команд с идентификаторами, описаниями, обязанностями и маршрутной информацией для владения сущностями.") в **Settings**.

![Владелец рабочей нагрузки Kubernetes](https://dt-cdn.net/images/ownership-k8s-workload-2213-2217e57297.png)

В этом примере показана информация о владении для Pod Kubernetes.

![Владелец Pod Kubernetes](https://dt-cdn.net/images/ownership-k8s-pod-2214-e55133496a.png)

На странице **Hosts** можно искать хосты с владением — фильтрация по **Tags** с определёнными вами префиксами ключей, например `owner` и `dt-owner`. Обратите внимание, что [пары ключ-значение владения](#format) должны быть применены хотя бы к одному хосту, чтобы ключ был доступен в фильтре **Tags**. Этот метод поиска владения доступен на всех страницах групп сущностей, которые можно фильтровать по тегам.

![Фильтрация по тегам владения на страницах групп сущностей](https://dt-cdn.net/images/ownership-hosts-page-2190-8b47f48119.png)

В примере страницы деталей хоста ниже у хоста три команды-владельца. Один из владельцев отмечен как **Unknown team identifier**. Это означает, что идентификатор команды был применён к хосту (например, через [oneagentctl](#oneagentctl) или [ручной тег](#tags)) в паре ключ-значение, но команда не существует. **Invalid team identifier** означает, что идентификатор команды или дополнительный ID был применён к хосту в неправильном [формате](#format).

Выберите **Properties and tags** для просмотра всех тегов, применённых к хосту.

![Владельцы хоста](https://dt-cdn.net/images/ownership-host-2199-95873a776f.png)

Выберите ![Expand row](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Expand row") рядом с неизвестной командой, а затем выберите **Add team** для определения информации о команде в **Settings**. Сущность будет автоматически обновлена определением команды.

![Неизвестная команда](https://dt-cdn.net/images/ownership-unknown-team-909-590ed7fbc0.png)
