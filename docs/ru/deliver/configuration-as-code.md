![Configuration as Code](https://dt-cdn.net/images/configuration-as-code-highresolution-1025-29c909e912.png "Configuration as Code") **Configuration as Code** (CaC) обеспечивает подход Observability as Code и Security as Code для полной автоматизации конфигурации платформы Dynatrace в любом масштабе:

* Автоматизация и стандартизация конфигураций наблюдаемости.
* Добавление наблюдаемости в процесс поставки программного обеспечения.
* Обеспечение стандартов при демократизации наблюдаемости.
* Соответствие требованиям Security-as-Code в процессе подключения сервисов и приложений.

## Обзор

Configuration-as-Code представляет собой подход к управлению данными конфигурации программного обеспечения и приложений, включая системы наблюдаемости и безопасности.

Если коротко, этот подход позволяет конфигурировать Dynatrace декларативно, без необходимости настраивать все через пользовательский интерфейс.

Модель самообслуживания Configuration as Code позволяет командам разработки быстро и эффективно настраивать мониторинг, наблюдаемость и политики безопасности даже для крупномасштабных приложений. Это устраняет необходимость создания собственных решений и сокращает ручную работу команд наблюдаемости.

Подход Dynatrace Configuration as Code позволяет управлять задачами наблюдаемости вашей среды Dynatrace с помощью конфигурационных файлов вместо графического пользовательского интерфейса.

## Сценарии использования

Управляйте любой конфигурацией Dynatrace параллельно с любым исходным кодом из YAML-файлов, организованных в Git-репозиториях. Например:

* Валидация релизов
* Столпы AWS Well-Architected Framework
* Ресурсы IAM
* Мониторинг сервисов / подключение сервисов
* Дашборды SLO
* Автоматизация устранения проблем и уведомлений о проблемах

Вы можете управлять любой конфигурацией Dynatrace параллельно с любым исходным кодом из YAML-файлов, организованных в Git-репозиториях.
Ознакомьтесь с нашими [примерами на GitHub](https://github.com/Dynatrace/dynatrace-configuration-as-code-samples).

Примеры того, что можно настроить в коде:

* Dashboards, Notebooks и Launchpads.
* Управление инцидентами.
* Валидация релизов.
* Включение мониторинга реальных пользователей.
* ![Anomaly Detection - new](https://dt-cdn.net/images/davis-anomalydetection-256-105da91594.png "Anomaly Detection - new") **Anomaly Detection**.
* Синтетический мониторинг.
* Конфигурации приема данных и Open Pipeline.

## Концепции

Подход Dynatrace CaC позволяет управлять задачами наблюдаемости вашей среды Dynatrace с помощью конфигурационных файлов вместо графического интерфейса.

Модель самообслуживания CaC позволяет командам разработки быстро и эффективно настраивать даже для крупномасштабных приложений:

* Мониторинг
* Наблюдаемость
* Политики безопасности

Это устраняет необходимость создания собственных решений и сокращает ручную работу команд наблюдаемости.

CaC позволяет:

* Создавать шаблоны конфигураций для нескольких сред.
* Управлять взаимозависимостями между конфигурациями без необходимости хранить уникальные идентификаторы.
* Применять одну и ту же конфигурацию к сотням сред Dynatrace и обновлять их все одновременно.
* Продвигать конфигурации, специфичные для приложений, между средами после развертывания на каждом этапе.
* Поддерживать все механизмы и лучшие практики рабочих процессов на основе Git, такие как pull-запросы, слияние и утверждения.
* Фиксировать конфигурацию в системе контроля версий и совместно работать над изменениями.

## Зачем использовать Configuration as Code

Причина использования CaC заключается в том, чтобы иметь конфигурационные файлы, которые позволяют

* Создавать,
* Обновлять и
* Управлять конфигурациями наблюдаемости безопасно, последовательно и воспроизводимо.

Они могут быть повторно использованы, версионированы и переданы вашей команде.

Стандартизированный подход к конфигурированию Dynatrace как кода имеет множество преимуществ.
Помимо всех преимуществ, которые дает подход на основе Git, таких как

* Контроль версий
* Воспроизводимость

Применение CaC обеспечивает

* Самообслуживание конфигураций наблюдаемости
* Оптимизацию и стандартизацию процессов подключения
* Синхронизацию конфигураций между различными средами

## Использование разработки на основе наблюдаемости в платформе разработчика

* Сократите время развертывания, интегрируя CaC для оптимизации процесса подключения приложений через Golden Paths.
* Внедрите стандарты наблюдаемости и безопасности в вашей среде, интегрируя их в конвейеры CI/CD, например, через образы контейнеров, обеспечивая согласованность на всех этапах.
* Предоставьте возможности самообслуживания, интегрируя наблюдаемость, автоматизацию и quality gates в ваш SDLC.
  Дополнительную информацию см. в разделе [Platform Engineering](../discover-dynatrace/get-started/platform-engineering.md "Use observability and security to drive analytics and automation at scale.").

## Инструменты

Для настройки и управления Dynatrace с помощью CaC у вас есть два варианта инструментов:

* [Terraform](configuration-as-code/terraform.md "Manage your Dynatrace environment using Dynatrace Configuration as Code via Terraform.") -- стандартный в отрасли программный инструмент CaC.
* [Monaco](configuration-as-code/monaco.md "Manage your Dynatrace environment using Dynatrace Configuration as Code via Monaco.") -- проприетарный CLI-инструмент CaC от Dynatrace.

Выбор инструмента зависит от стека технологий и требований.

Мы рекомендуем Terraform с провайдером Dynatrace Terraform, если:

* Вы уже знакомы с Terraform и чувствуете себя уверенно при работе с ним.
* Вы хотите управлять инфраструктурой и конфигурацией нескольких провайдеров в одном рабочем пространстве.
* Вы хотите воспользоваться преимуществами внешнего управления состоянием, выявления расхождений между планом и реальностью, а также совместной работы через удаленные бэкенды состояний, такие как GitHub.
* Вы планируете использовать динамические конфигурации, задействуя вычисления и условную логику, поддерживаемые выражениями HCL (Hashicorp Configuration Language).

Если вы не хотите или не можете использовать Terraform, мы предлагаем проприетарный CLI-инструмент CaC от Dynatrace -- Monaco.
Monaco предоставляет независимое от сторонних решений, работающее в автономном режиме решение, которое использует нативный JSON для описания конфигураций Dynatrace.
