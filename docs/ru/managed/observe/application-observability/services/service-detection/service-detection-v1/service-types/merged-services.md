# Объединённые сервисы


* Classic
* How-to guide
* 3-min read
* Updated on Oct 04, 2022

**Settings** > **Merged service monitoring** устарело.

* Если в вашей среде существуют объединённые сервисы, вы по-прежнему можете открыть страницу для их просмотра и редактирования.
* Создать новые объединённые сервисы на этой странице настроек невозможно. Для создания или редактирования эквивалента объединённого сервиса через обнаружение на основе правил см. раздел [Создание правила для агрегации сервисов](#create-merged-service) ниже.

Объединённый сервис — это сервис, агрегирующий несколько [сервисов веб-запросов](../../../../../../../observe/application-observability/services/service-detection/service-detection-v1.md#web-request-service "Узнайте, как Dynatrace Service Detection v1 обнаруживает и именует различные типы сервисов.") одной группы процессов, выполняющих одинаковые функции на отдельных узлах кластера. Эти сервисы веб-запросов практически идентичны с точки зрения мониторинга производительности. Объединённый сервис отображается в Dynatrace как единый сервис, содержащий все данные всех агрегированных сервисов.

Предположим, у вас есть веб-сервер Apache с несколькими определениями виртуальных хостов (например, `dynatrace.com`, `dynatrace.at` и `dynatrace.pl`). С точки зрения Apache это независимые виртуальные хосты. Поэтому Dynatrace обнаруживает их как отдельные сервисы. Однако для целей мониторинга вы можете захотеть отображать эти сервисы как единый объединённый сервис с именем `Dynatrace web page`.

Dynatrace автоматически определяет объединяемые сервисы и отображает их на странице **Merged service monitoring** (**Settings** > **Server-side service monitoring** > **Merged service monitoring**). Объединить можно только сервисы из этого списка.

## Характеристики существующих объединённых сервисов

* После объединения данные отдельных объединённых сервисов больше не могут быть разделены — данные мониторинга доступны только в агрегированном виде для всех объединённых сервисов.
* Исходные сервисы объединённого сервиса больше не обновляются как источники данных. Хотя исторические данные остаются доступными (например, для построения графиков), для исходных сервисов новые данные не отслеживаются. Агрегированные данные присваиваются и связываются с объединённым сервисом.
* Объединённые сервисы относятся к одному типу. Например, они принадлежат одной группе процессов, используют одну и ту же базовую технологию или следуют одному шаблону именования. Несколько виртуальных хостов, контекстных корней или портов прослушивания, представляющих одну логическую сущность, могут входить в состав объединённого сервиса.
* Объединяются только почти идентичные, автономные сервисы веб-запросов одной группы процессов — объединённые сервисы не объединяются в другие объединённые сервисы.

## Разделение объединённых сервисов

При разделении сервиса из объединённого сервиса («разъединении» сервиса) исторические данные будут доступны только для объединённого сервиса. Все вновь захваченные данные будут связаны с новым автономным сервисом.

Для разделения сервиса из объединённого сервиса:

1. Перейдите в **Settings** > **Server-side service monitoring** > **Merged service monitoring**.
2. Выберите **Edit** ![Edit](https://dt-cdn.net/images/edit-icon-a083c62c49.svg "Edit") в строке объединённого сервиса, который нужно разделить.
3. Выберите **Remove** ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") в строке сервиса, который нужно удалить.
4. Выберите **Update**.

## Удаление объединённого сервиса

При удалении объединённого сервиса все отдельные объединённые сервисы будут разделены и снова будут рассматриваться как автономные сервисы.

Для удаления объединённого сервиса:

1. Перейдите в **Settings** > **Server-side service monitoring** > **Merged service monitoring**.
2. Выберите **Delete** ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") в строке объединённого сервиса, который нужно удалить.

## Создание правила для агрегации сервисов

Для выполнения задачи, эквивалентной созданию объединённого сервиса, но с помощью обнаружения сервисов на основе правил:

1. Перейдите в **Settings** > **Service Detection**.

   * Выберите **Full web request rules**, если веб-запросы полностью отслеживаются Dynatrace.
   * Выберите **External web request rules** для веб-запросов к внешним ресурсам.
2. Выберите **Add item** и добавьте имя для нового правила.
3. Включите хотя бы один участник идентификатора сервиса из следующих: **Application identifier**, **URL context root** и **Server name**.
4. Необязательно: вы можете изменить правило, выполнив следующие действия:

   * Установите зону управления
   * Добавьте определённые условия
   * Отключите порт
5. Выберите **Save changes**.

Подробнее о создании, редактировании и удалении правил обнаружения сервисов см. в разделе [Управление обнаружением сервисов на основе правил](../../../../../../../observe/application-observability/services/service-detection/service-detection-v1/customize-service-detection.md#manage "Используйте правила обнаружения для настройки и улучшения автоматического обнаружения ваших сервисов.").
