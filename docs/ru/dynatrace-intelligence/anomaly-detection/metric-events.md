# Метрические события


Dynatrace Classic

Мы рекомендуем попробовать [Anomaly Detection](anomaly-detection-app.md "Изучите конфигурации обнаружения аномалий с помощью приложения Anomaly Detection.") ![Anomaly Detection - new](https://dt-cdn.net/images/davis-anomalydetection-256-105da91594.png "Anomaly Detection - new") для более расширенных возможностей конфигурации, таких как:

* Возможность использования запросов DQL в дополнение к записям Grail
* Оповещения на основе данных, таких как логи, спаны и бизнес-события
* Возможность создания расширенных запросов с включением большего количества записей данных

Вы можете легко перенести свои конфигурации метрических событий в ![Anomaly Detection - new](https://dt-cdn.net/images/davis-anomalydetection-256-105da91594.png "Anomaly Detection - new") **Anomaly Detection** и выбрать желаемую стратегию мониторинга с минимальными изменениями в существующей конфигурации. Чтобы узнать больше о возможностях и применении Anomaly Detection, посмотрите видео [Введение в ![Anomaly Detection - new](https://dt-cdn.net/images/davis-anomalydetection-256-105da91594.png "Anomaly Detection - new") **Anomaly Detection** на основе DQL](https://www.youtube.com/watch?v=-GxLlr9oGGA).

Dynatrace Davis® AI автоматически анализирует аномальные ситуации в вашей ИТ-инфраструктуре и пытается определить соответствующее влияние и первопричину. Davis использует широкий спектр источников информации, таких как транзакционное представление ваших сервисов и приложений, а также все события, возникающие на отдельных узлах в вашей топологии Smartscape®. Одним из источников событий в Dynatrace являются метрические события, то есть события, основанные на данных метрик. Они настраиваются в глобальных параметрах вашего окружения и видны всем пользователям Dynatrace в вашем окружении. Существует два типа метрических событий в зависимости от того, как выполняется запрос метрики для оценки события:

* [Ключ метрики](metric-events/metric-key-events.md "Узнайте о метрических событиях на основе ключа метрики."). События на основе ключа метрики оценивают входящие измерения одной метрики. С этим типом запроса можно использовать только статические пороговые значения.
* [Селектор метрик](metric-events/metric-selector-events.md "Узнайте о метрических событиях на основе селектора метрик."). События на основе селектора метрик оценивают сложный запрос, определённый с помощью [селектора](../../dynatrace-api/environment-api/metric-v2/metric-selector.md "Настройте селектор метрик для Metric v2 API."). Этот тип запроса может включать исторические данные и даже [арифметические операции](../../dynatrace-api/environment-api/metric-v2/metric-expressions.md "Используйте метрические выражения для применения арифметических операций в запросе точек данных через Metrics API v2.") с несколькими метриками.

Dynatrace использует три стратегии мониторинга для таких событий:

* [Статическое пороговое значение](static-thresholds.md "Когда использовать статическое пороговое значение для обнаружения аномалий.") — пороговое значение, которое не изменяется со временем.
* [Авто-адаптивное пороговое значение](auto-adaptive-threshold.md "Как Dynatrace адаптирует пороговые значения для нескольких сущностей в рамках конфигурации обнаружения аномалий.") — Dynatrace автоматически вычисляет пороговое значение и динамически адаптирует его к поведению вашей метрики.
* [Сезонная базовая линия](../reference/ai-models/seasonal-baseline.md "Как Dynatrace Intelligence предлагает пороговые значения сезонной базовой линии для области сущностей.") — Dynatrace вычисляет доверительный интервал для метрики с сезонными паттернами.

* Авто-адаптивные пороговые значения и сезонное базовое моделирование доступны только для событий на основе селектора метрик.
* Количество конфигураций метрических событий (как на основе ключа метрики, так и селектора метрик) ограничено **10 000**.

## Обзор метрических событий

Dynatrace предоставляет обзор всех ваших метрических событий с информацией об ограничениях и сбоях метрических событий за последние 24 часа. Для доступа к обзору перейдите в **Settings** > **Anomaly detection** > **Metric events**.

![Обзор метрических событий](https://dt-cdn.net/images/metric-events-overview-1194-c3db36a9d3.png)

Обзор лимитов показывает, какая часть каждого типа лимита была использована. Ссылка для просмотра переводит вас в Data Explorer с предварительно заполненным запросом, настроенным на отображение наиболее ресурсоёмких конфигураций.

Обзор метрических событий включает конфигурации, у которых возникли проблемы за последние 24 часа. Отслеживаемые проблемы включают:

* `THROTTLED`: конфигурация достигла лимитов запросов селектора метрик.
* `QUERY_FAILED`: запрос метрики конфигурации завершился неудачей. Некоторые возможные причины сбоя запроса:

  + Зоны управления или теги больше недоступны.
  + Данные метрики больше недоступны.
  + Запрашиваемые данные недоступны с разрешением в 1 минуту.
* `CONFIG_BLOCKED`: конфигурация создаёт значительную нагрузку и была заблокирована, чтобы избежать негативного влияния на другие пользовательские оповещения.

## Зоны управления в метрических событиях

Если у вас есть разрешения на уровне зон управления, вы можете создавать конфигурации метрических событий, привязанные к этим зонам управления. Такие конфигурации используют только данные, поступающие из указанной зоны управления. Вы также можете видеть все конфигурации зон управления, использующие зону управления, к которой у вас есть доступ.

Для создания метрических событий без привязки к зонам управления необходим доступ администратора.

Зоны управления, настроенные через [правила размерных данных](../../manage/identity-access-management/permission-management/management-zones/management-zone-rules.md#logs-metrics "Определите правила для ограничения сущностей, доступных в зоне управления."), не поддерживаются для метрических событий.

## Осведомлённость о топологии

Осведомлённость о топологии и контекст являются ключевыми темами платформы наблюдаемости Dynatrace. Данные — такие как метрики, трассировки, события и логи — не просто передаются и хранятся на платформе. Такие данные содержат множество ссылок на топологию, откуда они были получены. Например, для метрик процессов каждое измерение ссылается на связанные хосты и процессы. Davis AI использует эту топологическую информацию для автоматического определения первопричины и анализа влияния обнаруженных аномалий. То же самое относится ко всем метрическим событиям в вашем окружении.

Когда конфигурация обнаружения аномалий создаёт событие, Dynatrace автоматически определяет наиболее релевантную сущность для сопоставления с событием. Если обнаружено несколько ссылок на сущности, автоматически выбирается наиболее релевантная. Например, если метрика со ссылкой и на хост, и на процесс приводит к событию, событие создаётся на процессе.

[Приём метрик](../../ingest-from/extend-dynatrace/extend-metrics.md "Узнайте, как расширить наблюдаемость метрик в Dynatrace.") позволяет отправлять все типы метрических измерений, независимо от количества сущностей, к которым они относятся. Существуют следующие сценарии:

### Измерения не связаны ни с одной сущностью

Если вы определите метрическое событие для нетопологической метрики, результирующее событие будет создано на самом окружении мониторинга, а не на конкретной сущности Smartscape.

Пример: показатели выручки, измеренные для всех розничных магазинов по географическим регионам

```
business.revenue,shop=shop111,city=NewYork 234


business.revenue,shop=shop999,city=Atlanta 499
```

### Измерения связаны с одной сущностью

Если вы определите метрическое событие для измерения, связанного с одной сущностью, результирующее событие будет создано на этой сущности.

Пример: выполнения пакетных заданий, измеренные на отслеживаемом хосте, где измерение связано с хостом

```
batchjob.executions,dt.entity.host=HOST-1111111,hostname=hostA,ip=53.43.23.12 23


batchjob.executions,dt.entity.host=HOST-2222222,hostname=hostB,ip=53.43.23.12 23
```

### Измерения связаны с несколькими сущностями

Когда для каждого измерения указано несколько сущностей, Dynatrace выбирает наиболее подходящую сущность, на которой следует создать событие. В случае хоста и процесса измерение предположительно относится к процессу, а не к хосту, поэтому событие создаётся на процессе.

Пример: количество запусков пакетных заданий, измеренное для процесса на отслеживаемом хосте, где измерение связано и с процессом, и с хостом

```
batchjob.executions,dt.entity.host=HOST-1,dt.entity.process_group_instance=PROCESS-GROUP-INSTANCE-1,hostname=hostA,ip=53.43.23.12 23


batchjob.executions,dt.entity.host=HOST-2222222,dt.entity.process_group_instance=PROCESS-GROUP-INSTANCE-2,hostname=hostB,ip=53.43.23.12 23
```
