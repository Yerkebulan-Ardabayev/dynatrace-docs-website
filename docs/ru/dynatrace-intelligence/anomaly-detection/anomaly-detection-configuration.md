# Настройка обнаружения аномалий


* Latest Dynatrace
* Explanation
* 7-min read
* Updated on Feb 04, 2026

Настройка обнаружения аномалий опирается на несколько компонентов:

* Источник данных — временной ряд, который подвергается оценке. Это может быть запрос DQL, извлекающий данные из Grail, или конкретная метрика.
* Тип анализатора и параметры: способ оценки данных.
* Скользящее окно: период, в течение которого оцениваются данные.
* Шаблон события: определяет, какой шаблон срабатывает по результатам настройки.

После настройки и активации конфигурация наблюдает за данными и создаёт событие при выполнении условий. Чтобы убедиться, что конфигурация работает корректно и уведомляет вас о нужных событиях, вы можете предварительно просмотреть результаты её работы:

* Previous Dynatrace Предварительный просмотр метрического события визуально отображает поведение вашего события. Вы можете изменять настройки, чтобы увидеть, как они влияют на конфигурацию.

## Источник данных

Источник данных предоставляет временной ряд, который оценивается Dynatrace Intelligence:

* Previous Dynatrace Метрика определяет временной ряд. Это может быть одна метрика, заданная ключом метрики, или [выражение метрики](../../dynatrace-api/environment-api/metric-v2/metric-expressions.md "Use metric expressions to apply arithmetic operations in a data points query via the Metrics API v2.").

Если ваши данные имеют задержку, необходимо задать смещение в конфигурации через параметр **Query offset**. Укажите значение в минутах.

## Задержка

Параметр **Delay** позволяет пользователям снизить частоту выполнения запросов DQL, выполняемых пользовательским оповещением за час, что помогает сократить затраты на запросы и уменьшить нагрузку на систему.

Хотя по умолчанию период задержки между выполнениями пользовательского оповещения составляет `1 Minute`, вы можете настроить интервал выполнения запросов на более длительные промежутки — например, каждые пять минут — без потери возможности ретроспективной оценки для каждой минуты в пределах выбранного временного окна.

Параметр **Delay** можно настроить с использованием **Minutes** или **Seconds**, но задержка не может превышать `60 Minutes`.

## Тип анализатора и параметры

Параметры анализатора определяют, как Dynatrace Intelligence оценивает данные, предоставленные источником данных. Точный набор параметров зависит от типа анализа:

* [Авто-адаптивный порог](auto-adaptive-threshold.md "How Dynatrace adapts thresholds for multiple entities within the scope of an anomaly detection configuration.") — Dynatrace автоматически рассчитывает порог и динамически адаптирует его к поведению ваших данных.
* [Сезонная базовая линия](../reference/ai-models/seasonal-baseline.md "How Dynatrace Intelligence suggests seasonal baseline thresholds for a scope of entities.") — Dynatrace создаёт доверительный интервал для данных с сезонными паттернами.
* [Статический порог](static-thresholds.md "When to use a static threshold for your anomaly detection.") — порог, который не изменяется со временем.

| Параметр | Описание |
| --- | --- |
| Количество колебаний сигнала | Авто-адаптивный порог состоит из двух компонентов: базовой линии и колебания сигнала. Этот параметр определяет, сколько раз колебание сигнала добавляется к базовой линии. Подробнее см. в разделе [Расчёт порога](auto-adaptive-threshold.md#calculation "How Dynatrace adapts thresholds for multiple entities within the scope of an anomaly detection configuration."). |
| Порог | Этот параметр определяет значение статического порога и, при необходимости, его единицу измерения. Выберите **Suggest values**, чтобы использовать значение, рассчитанное Dynatrace Intelligence на основе предыдущих данных. |
| Допуск | Этот параметр определяет [допуск](../reference/ai-models/seasonal-baseline.md#parameters "How Dynatrace Intelligence suggests seasonal baseline thresholds for a scope of entities.") сезонной модели. Чем выше допуск, тем шире доверительный интервал, что приводит к меньшему количеству срабатываний. |
| Условие оповещения | Этот параметр определяет, когда создаётся событие: если метрика выше, ниже или за пределами порога. |
| Оповещение об отсутствии данных | Этот параметр определяет, активно ли оповещение об отсутствии данных для данной конфигурации. Если оно активно, оно комбинируется с пороговым условием по логике **OR**. Найти его можно в разделе **Advanced properties** конфигурации. |

## Оповещение об отсутствии данных

Dynatrace предоставляет возможность настроить оповещение об отсутствии данных в метрике или запросе DQL. Если оповещение активно, Dynatrace регулярно проверяет, содержит ли скользящее окно конфигурации обнаружения аномалий какие-либо измерения. Например, если скользящее окно установлено как **3 violating samples during any 5 minutes**, Dynatrace создаёт оповещение, если данные отсутствуют в течение 3-минутного периода.

Условие отсутствия данных и пороговое условие комбинируются по логике **OR**.

Рекомендуется отключать оповещения об отсутствии данных для разреженных потоков данных, где измерения не ожидаются через регулярные интервалы, так как это приведёт к лавине оповещений.

Для данных с ожидаемым поздним поступлением (например, метрики облачной интеграции с 5-минутной задержкой) используйте длинные скользящие окна, покрывающие задержки. Для 5-минутной задержки используйте скользящее окно не менее 10 минут.

Плейсхолдер `{missing_data_samples}` в описании события подставляет количество минут без полученных данных.

## Скользящее окно

Скользящее окно конфигурации обнаружения аномалий определяет, сколько одноминутных выборок должны нарушить порог в течение заданного периода. Когда указанное количество нарушений достигнуто, Dynatrace создаёт событие. Цель — избежать чрезмерно агрессивного оповещения при единичных нарушениях, когда каждое измерение, превышающее порог, вызывает событие.

Событие остаётся открытым до тех пор, пока метрика не вернётся в пределы порога для определённого количества одноминутных выборок в рамках того же скользящего окна, после чего Dynatrace закрывает событие. Сохранение события открытым помогает избежать избыточных оповещений, добавляя новые нарушения порога к существующей проблеме вместо создания новой.

Настройки скользящего окна находятся в разделе **Advanced properties** конфигурации. По умолчанию:

* Любые три одноминутные выборки из пяти должны нарушить порог для создания события.
* Пять одноминутных выборок должны вернуться в норму для закрытия этого события.

Вы можете задать скользящее окно продолжительностью до 60 минут для каждого из трёх типов анализатора.

Рассмотрим случай со статическим порогом 90% использования CPU.

![Объяснение скользящего окна и снятия оповещения](https://dt-cdn.net/images/sliding-window-example-2026-39b1adf7bb.png)

Анализ событий начинается с первой нарушающей выборки в скользящем окне. Как только количество нарушающих выборок достигает заданного порога, анализ событий прекращается и создаётся проблема. Несмотря на то что анализ событий остановлен, само событие остаётся открытым до тех пор, пока не будут выполнены критерии снятия оповещения:

* Количество нарушающих выборок должно быть ниже порогового числа для создания проблемы.
* Количество «нормальных» выборок должно быть больше или равно числу выборок для снятия оповещения.

**Оба** критерия должны быть выполнены для закрытия события.

Значения по умолчанию (3 нарушающие выборки в скользящем окне из 5 выборок для создания проблемы, 5 выборок для снятия оповещения для закрытия события) подходят для большинства конфигураций. Однако вам может потребоваться их обновить (например, из-за шума в измерениях).

## Шаблон события

Шаблон события определяет характеристики события, создаваемого при нарушении порога. Необходимо указать как минимум имя и тип события.

* Для быстрого понимания события его имя должно быть коротким, легко читаемым описанием ситуации, например `High network activity` или `CPU saturation`.
* Имя может включать плейсхолдеры, такие как `{threshold}` или `{alert_condition}`. Плейсхолдеры заменяются реальными значениями в фактическом событии. Чтобы увидеть доступные плейсхолдеры, введите `{` в поле ввода. Доступны несколько плейсхолдеров:

  + `{alert_condition}` — условие оповещения (выше/ниже порога).

    Если вы установите **Alert condition** в значение `Alert if the metric is outside`, плейсхолдеры `{threshold}`, `{severity}` и `{baseline}` будут недоступны.
  + `{baseline}` — нарушенное значение базовой линии.
  + `{dims}` — список всех измерений (и их значений) метрики, нарушившей порог. Вы также можете указать конкретное измерение: `{dims:dt.entity.<entity>}`. Чтобы получить список доступных измерений для вашей метрики, выполните запрос через GET metric descriptor.
  + `{entityname}` — имя затронутой сущности.
  + `{metricname}` — имя метрики, нарушившей порог.
  + `{missing_data_samples}` — количество выборок с отсутствующими данными. Доступен только при включённом оповещении об отсутствии данных.

    `{missing_data_samples}` в описании события

    Рекомендуется включать плейсхолдер `{missing_data_samples}` в описание события, чтобы видеть, вызвана ли проблема отсутствующими выборками данных или нарушениями порога.
  + `{severity}` — серьёзность события.
  + `{threshold}` — нарушенное значение порога.

Вы можете указать дополнительные параметры в виде пар «ключ-значение». Список возможных свойств событий см. в [Семантическом словаре](../../../common/semantic-dictionary/model/davis.md#davis-ai-events "Get to know the Semantic Dictionary models related to Davis AI.").
