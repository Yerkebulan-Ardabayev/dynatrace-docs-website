# События по ключу метрики

* Latest Dynatrace
* Руководство
* Чтение: 4 мин
* Обновлено 22 мая 2025

События по ключу метрики основаны на входящих необработанных измерениях одной метрики. Для этого типа событий доступна только стратегия мониторинга со статическим пороговым значением. Вы можете отслеживать все измерения метрики в рамках одной конфигурации (например, можно создать оповещение для 20 000 процессоров в одной конфигурации события метрики). В качестве защитной меры Dynatrace ограничивает эти конфигурации лимитом в 200 одновременных оповещений. Вы можете сузить область действия события до конкретных измерений.

Кроме того, действует ограничение в **10 000** конфигураций (как по ключу метрики, так и по селектору метрики) на одну среду.

## Область действия событий по ключу метрики

Многие метрики Dynatrace поставляются несколькими сущностями; их количество может легко достигать тысяч. Скорее всего, вам не нужно, чтобы ваше событие охватывало все эти сущности одновременно. Для сужения области действия события вы можете указать фильтры на основе правил.

Доступны два типа фильтров:

* **Фильтры сущностей** сужают область действия на основе предоставленных критериев (например, имя сущности или тег). Фактический набор доступных критериев зависит от метрики. Если указано несколько фильтров, применяется логика **AND**.
* **Фильтры измерений** отфильтровывают сущности на основе предоставленных кортежей (уникальных комбинаций метрика-ключ измерения-значение измерения). Если указано несколько фильтров, применяется логика **AND**. Например, вы можете задать фильтр измерений, который выберет только пользовательские действия с устройств iOS для вашего события метрики на основе метрики **Action count**.

## Создание события по ключу метрики

1. Перейдите в **Settings** > **Anomaly Detection** > **Metric events** и выберите **Add metric event**.
2. В поле **Summary** укажите краткое описание события.
3. В разделе **Query definition** настройте запрос метрики:

   1. Выберите тип запроса **Metric key**.
   2. Выберите метрику для вашего события. Вы можете указать ключ или отображаемое имя метрики. Начните вводить, чтобы увидеть список предложений.
   3. Выберите тип агрегации для метрики (если применимо).
   4. Выберите зону управления. Только данные из этой зоны будут оцениваться для события метрики. Оставьте это поле пустым, чтобы использовать все данные, предоставляемые метрикой.
4. Необязательно: В разделе **Advanced query definition** укажите смещение запроса (в минутах).
   Смещение необходимо для метрик с задержкой; в противном случае событие метрики может генерировать ложные оповещения.
5. Необязательно: Добавьте фильтры сущностей на основе правил.
6. Необязательно: Выберите измерения, которые должны учитываться событием.
7. Определите стратегию мониторинга. Для запросов по ключу метрики доступны только статические пороговые значения.

   1. Укажите пороговое значение. Выберите **Use suggested threshold**, чтобы использовать значение на основе предыдущих данных.
   2. Если применимо, выберите единицу измерения порогового значения.
   3. Выберите поведение [оповещения об отсутствующих данных](/docs/dynatrace-intelligence/anomaly-detection/anomaly-detection-configuration#missing-data "Как настроить оповещение об отсутствующих измерениях.").
      Если оповещение об отсутствующих данных включено, оно комбинируется с условием порогового значения по логике **OR**.
   4. Выберите условие оповещения: оповещать, если метрика выше или ниже порогового значения.
   5. Необязательно: В разделе **Advanced model properties** укажите скользящее окно для сравнения.
      Скользящее окно определяет, сколько раз пороговое значение (автоматически рассчитанное или указанное вручную) должно быть нарушено в пределах скользящего временного окна для генерации события (нарушения не обязательно должны быть последовательными). Это помогает избежать чрезмерно агрессивного оповещения при единичных нарушениях. Вы можете задать скользящее окно до 60 минут.
8. Просмотрите предварительный просмотр вашего оповещения и оцените эффективность конфигурации.

   1. Выберите значения измерений, которые хотите видеть в предварительном просмотре.
   2. Выберите временной диапазон предварительного просмотра. Вы можете получать оповещения за один, три или семь дней.
9. Укажите **title** для вашего события. Заголовок должен быть коротким, легко читаемым описанием ситуации, например `High network activity` или `CPU saturation`.
10. В разделе **Description** создайте информативное сообщение о событии. Сообщения о событиях помогают понять природу события. Вы можете использовать следующие заполнители:

    * `{alert_condition}` -- условие оповещения (выше/ниже порогового значения).
    * `{baseline}` -- нарушенное значение базовой линии.
    * `{dims}` -- список всех измерений (и их значений) метрики, нарушившей пороговое значение. Вы также можете указать конкретное измерение: `{dims:dt.entity.<entity>}`. Для получения списка доступных измерений вашей метрики выполните запрос через [GET дескриптор метрики](/docs/dynatrace-api/environment-api/metric-v2/get-descriptor "Просмотр дескриптора метрики через Metrics v2 API.").
    * `{entityname}` -- имя затронутой сущности.
    * `{metricname}` -- имя метрики, нарушившей пороговое значение.
    * `{missing_data_samples}` -- количество выборок с отсутствующими данными. Доступно только при включённом оповещении об отсутствующих данных.

      `{missing_data_samples}` в описании события

      Мы рекомендуем включать заполнитель `{missing_data_samples}` в описание события, чтобы видеть, вызвана ли проблема отсутствующими выборками данных или нарушениями порогового значения.
    * `{severity}` -- уровень серьёзности события.
    * `{threshold}` -- нарушенное значение порога.
11. Выберите тип для генерируемых событий.
12. Определите стратегию **объединения** для генерируемых событий.
    Если объединение разрешено, Davis AI попытается объединить это событие с существующими проблемами; в противном случае при каждом срабатывании создаётся новая проблема.
13. Необязательно: Задайте дополнительные свойства в формате ключ-значение для прикрепления к событию.
14. Выберите **Save changes**.
