# События метрик с селектором метрик


Селектор метрик — это мощный инструмент для указания, какие данные вы хотите считывать для оценки событий метрик. Он предоставляет вам две основные возможности:

* Трансформации метрик для преобразования считываемых данных.
* Выражения метрик для объединения одной или нескольких метрик в другой результат с помощью простой математики.

С помощью селектора метрик Davis может получить доступ к историческим данным метрики и изучить нормальное поведение вашей среды, что позволяет использовать автоадаптивные пороговые значения в событиях метрик. Однако действуют некоторые ограничения:

* **100 000** отслеживаемых измерений метрик на среду
* **10 000** конфигураций событий метрик (как по ключу метрики, так и по селектору метрик) на среду
* **1 000** отслеживаемых измерений на конфигурацию события метрики (статический или автоадаптивный порог)
* **500** отслеживаемых измерений на конфигурацию события метрики (сезонная базовая линия)
* **100** селекторов метрик на стратегию мониторинга. Вы можете иметь 100 конфигураций с автоадаптивным порогом и 100 со статическим порогом.

## Область действия событий селектора метрик

Сам селектор определяет область действия события селектора метрик. Важно понимать последствия при настройке селектора, состоящего из измерений от тысяч отдельных источников. Dynatrace применяет защитные ограничения к обнаружению аномалий по количеству измерений метрик, которые могут наблюдаться в одной среде мониторинга, чтобы избежать эксплуатационных проблем. Чтобы узнать, как сузить область действия вашей конфигурации, см. [**Трансформация фильтра**](../../../dynatrace-api/environment-api/metric-v2/metric-selector.md#filter "Configure the metric selector for the Metric v2 API.").

## Комбинирование метрик

С помощью выражений метрик вы можете реализовать оповещение с обзором ситуации сверху вниз, а не оповещение по каждому компоненту.

Например, вы можете наблюдать паттерны логов на нескольких хостах. Вычисляя общее количество наблюдаемых паттернов логов по всем релевантным файлам логов, Dynatrace может обнаруживать аномалии паттернов в агрегированном потоке логов, а не по отдельным подсчётам в каждом файле логов.
Если есть разрозненные подсчёты по многим сущностям (например, подсчёт ошибок по нескольким процессам одного типа), агрегированное обнаружение аномалий сверху вниз значительно более устойчиво к ложноположительным оповещениям, чем обнаружение по отдельным подсчётам ошибок на процесс.

## Создание события селектора метрик

1. Перейдите в **Settings** > **Anomaly Detection** > **Metric events** и нажмите **Add metric event**.
2. В поле **Summary** укажите краткое информативное описание события.
3. В разделе **Query definition** настройте запрос метрики:

   1. Выберите тип запроса **Metric selector**.
   2. Укажите необходимый селектор метрик.
4. Выберите зону управления. Для события метрики оцениваются только данные из этой зоны. Оставьте это поле пустым, чтобы использовать все данные, запрашиваемые селектором метрик.
5. Необязательно В разделе **Advanced query definition** укажите смещение запроса (в минутах).
   Смещение необходимо для метрик с задержкой; в противном случае событие метрики может генерировать ложные оповещения.
6. Определите стратегию мониторинга

   1. Выберите тип модели:

      * Автоадаптивный порог — Dynatrace автоматически рассчитывает порог и динамически адаптирует его к поведению вашей метрики.
      * Статический порог — порог, который не изменяется со временем.
      * Сезонная базовая линия — Dynatrace создаёт доверительную полосу для метрики с сезонными паттернами.
   2. Для статического порога укажите пороговое значение. Нажмите **Use suggested threshold**, чтобы использовать значение на основе предыдущих данных.
   3. Выберите поведение [оповещения об отсутствующих данных](../anomaly-detection-configuration.md#missing-data "How to set up an alert for missing measurements.").
      Если оповещение об отсутствующих данных включено, оно комбинируется с условием порога по логике **ИЛИ**.
   4. Выберите условие оповещения: оповещать, если метрика выше, ниже или за пределами порога.
   5. Необязательно В разделе **Advanced model properties** укажите [скользящее окно](../anomaly-detection-configuration.md#sliding-window "How to set up an alert for missing measurements.") для сравнения.
      Скользящее окно определяет, как часто порог — будь он рассчитан автоматически или указан вручную — должен быть нарушен в пределах скользящего окна времени для генерации события (нарушения не обязательно должны быть последовательными). Это помогает избежать чрезмерно агрессивного оповещения по единичным нарушениям. Вы можете установить скользящее окно до 60 минут.
7. Проверьте предварительный просмотр вашего оповещения и оцените эффективность вашей конфигурации.

   1. Выберите значения измерений, которые хотите видеть в предварительном просмотре.
   2. Выберите период предварительного просмотра. Вы можете получать оповещения за один, три или семь дней.
8. Укажите **Title** для вашего события. Заголовок должен быть короткой, легко читаемой строкой, описывающей ситуацию, например `High network activity` или `CPU saturation`.
9. В разделе **Description** создайте информативное сообщение события. Сообщения событий помогают понять природу события. Вы можете использовать следующие заполнители:

   * `{alert_condition}` — условие оповещения (выше/ниже порога).
   * `{baseline}` — нарушенное значение базовой линии.
   * `{dims}` — список всех измерений (и их значений) метрики, нарушившей порог. Вы также можете указать конкретное измерение: `{dims:dt.entity.<entity>}`. Чтобы получить список доступных измерений для вашей метрики, запросите его через запрос GET metric descriptor.
   * `{entityname}` — имя затронутой сущности.
   * `{metricname}` — имя метрики, нарушившей порог.
   * `{missing_data_samples}` — количество сэмплов с отсутствующими данными. Доступно только при включённом оповещении об отсутствующих данных.

     `{missing_data_samples}` в описании события

     Рекомендуется включать заполнитель `{missing_data_samples}` в описание события, чтобы видеть, вызвана ли проблема отсутствующими сэмплами данных или нарушениями порога.
   * `{severity}` — серьёзность события.
   * `{threshold}` — нарушенное значение порога.
10. Выберите **Event type** для генерируемых событий.
11. Включите или выключите **Allow merge**, чтобы определить стратегию объединения для генерируемых событий.
    Если **Allow merge** включён, Davis AI попытается объединить это событие с существующими проблемами; если выключен, каждый раз создаётся новая проблема.
12. Необязательно Установите дополнительные свойства ключ-значение для прикрепления к событию.
13. Нажмите **Save changes**.
