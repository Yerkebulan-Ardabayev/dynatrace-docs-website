# Поддерживаемые форматы временных меток (Logs Classic)

* Classic
* 3-min read
* Updated on Sep 18, 2025

Log Monitoring Classic

Временная метка, включающая дату, время, часовой пояс и смещение, ищется в первых 64 символах содержимого журнала (это значение настраивается через **Log Monitoring** > **Timestamp/Splitting patterns**). Если смещение или часовой пояс не найдены, используется локальный часовой пояс хоста.

Для новейшей версии Dynatrace см. [Поддерживаемые форматы временных меток](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-supported-timestamp-format.md "Поддерживаемые временные метки для последней версии Log Management and Analytics.").

При обработке журналов каждая строка, в которой обнаружена поддерживаемая временная метка, начинает новую запись журнала. Строка без временной метки считается продолжением существующей записи журнала и присоединяется к строке, содержащей временную метку.
Если временная метка отсутствует в файле журнала или не распознана из-за неподдерживаемого формата, каждая строка, не начинающаяся с пробельных символов (таких как `space` или `tab`), начинает новую запись журнала. Каждая строка, начинающаяся с пробельных символов (таких как `space` или `tab`), обрабатывается как продолжение записи журнала.
Из-за множества форматов входящих данных журналов Log Monitoring также позволяет задать конкретный формат даты с помощью правил временных меток, определяющих, что следует считать временной меткой в записи журнала. Подробнее об определении формата даты см. в разделе [Настройка временных меток журнала](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-timestamp-configuration.md "Определение конкретного формата даты с помощью правил временных меток, указывающих, что следует считать временной меткой в записи журнала.").

Поддерживаемые [форматы временных меток](https://dt-url.net/6e034iy) включают:

* Формат ISO 8601: `%Y-%m-%d %H:%M:%S`
  Пример: `2022-04-17 11:25:12.345`
* Формат RFC 3339: `%Y-%m-%dT%H:%M:%S`
  Пример: `2022-04-17T11:25:12.345`
* Формат Unix Epoch, указывающий количество миллисекунд, прошедших с 1 января 1970 года
  Пример: `1652088888997`
* Формат RFC 3164: `%b %d %H:%M:%S`
  Пример: `Apr 17 11:25:12`
* Формат Db2 (IBM database 2): `%Y-%m-%d-%H.%M.%S`
  Пример: `2022-05-17-11.25.12.114000-300`
* Формат IIS: `%m/%d/%Y, %H:%M:%S`
  Пример: `04/17/2022, 11:25:12.345`
* Формат W3C (World Wide Web Consortium): `%Y-%m-%d %H:%M:%S`
  Пример: `2022-04-17 11:25:12.345`, указанный в часовом поясе UTC
* Формат Klog и Golang/glog: `[IWEF]%m%d %H:%M:%S`
  Пример: `I0408 06:40:02.634162`
* Другие распространённые форматы:

  + `%d %b %Y %H:%M:%S` (пример: `17 Apr 2022 11:25:12.345`)
  + `%Y %b %d %H:%M:%S` (пример: `2022 Apr 17 11:25:12.345`)
  + `%d/%b/%Y:%H:%M:%S` (пример: `17/Apr/2022:11:25:12.345`)

Файлы JSON поддерживаются только для Docker. Если в JSON-файле обнаружен тег `log`, соответствующее сообщение поступает на обработку и анализ. Внутри сообщения выполняется поиск временной метки одного из поддерживаемых форматов. Если поддерживаемый формат файла не найден, выполняется поиск тега `time`. Например:

```
{



"log":"2020-11-24 11:01:36,484 CRIT Supervisor running as root (no user in config file)\n",



"stream":"stdout",



"time":"2020-11-24T11:01:36.484996713Z"



}



{



"log":"2020-11-24 11:01:36,500 INFO RPC interface 'supervisor' initialized\n",



"stream":"stdout",



"time":"2020-11-24T11:01:36.50065223Z"



}
```

### Примеры допустимых форматов времени в файлах журналов

Большинство поддерживаемых форматов временных меток сопряжены с автоматическим определением формата часового пояса. Распознаются следующие форматы часовых поясов:

* Смещение относительно UTC
  Примеры:

  `2022-04-17 11:25:12.345 +01:00`
  `2022-04-17 11:25:12.345 +0100`
  `2022-04-17 11:25:12.345Z -01:00`
  `2022-04-17 11:25:12.345Z-0100`
  `2022-04-17 11:25:12.345 UTC-01:00`
  `2022-04-17 11:25:12.345 GMT-0100`
* Полное название часового пояса [IANA](https://dt-url.net/jq034g6).
  Пример:
  `2022-04-17 11:25:12.345 Europe/Warsaw`
* Сокращённое название часового пояса
  Примеры:
  `2022-04-17 11:25:12.345 CEST`
  `2022-12-17 11:25:12.345 CET`
  `2022-04-17 11:25:12.345Z`
  `2022-04-17 11:25:12.345UTC+0100`

Если в строке журнала не найден индикатор часового пояса, часовой пояс, используемый при разборе временной метки, по умолчанию устанавливается в значение, настроенное в
**Settings** > **Log Monitoring** > **Advanced log settings** > **Default timezone for agents**.

### Переопределение часового пояса

В случае если автоматическое определение формата часового пояса не работает, вы можете добавить правило переопределения часового пояса на [странице настройки временных меток журнала](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-timestamp-configuration.md#createrule "Определение конкретного формата даты с помощью правил временных меток, указывающих, что следует считать временной меткой в записи журнала.").

## Используемый часовой пояс по умолчанию

При обработке временных меток OneAgent учитывает как шаблоны локального часового пояса, так и шаблоны UTC.

### Шаблоны локального часового пояса

Если в содержимом журнала не указан часовой пояс, следующие форматы временных меток по умолчанию используют **локальный часовой пояс** хоста:

* ISO 8601 (`%Y-%m-%d %H:%M:%S`)
* RFC 3164 (`%b %d %H:%M:%S`)
* IIS (`%m/%d/%Y, %H:%M:%S`)
* Klog и Golang/glog (`[IWEF]%m%d %H:%M:%S`)
* `%d %b %Y %H:%M:%S`
* `%Y %b %d %H:%M:%S`
* `%d/%b/%Y:%H:%M:%S`

### Шаблоны часового пояса UTC

Если в содержимом журнала не указан часовой пояс, следующие форматы временных меток по умолчанию используют **UTC**:

* RFC 3339 (`%Y-%m-%dT%H:%M:%S`)
* Db2 (`%Y-%m-%d-%H.%M.%S`)
* W3C (`%Y-%m-%d %H:%M:%S`)
* Unix time

Чтобы проверить, какой часовой пояс был использован при разборе временной метки, найдите в журналах OneAgent (`oneagent-logmon-detailed.log`) запись `Diagnostic statistics for LGI`.
