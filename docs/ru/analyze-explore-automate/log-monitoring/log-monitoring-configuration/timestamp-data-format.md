# Поддерживаемые форматы временных меток (Logs Classic)

* Classic
* 3-минутное чтение
* Обновлено 18 сентября 2025 г.

Log Monitoring Classic

Временная метка — включая дату, время, часовой пояс и смещение — ищется в первых 64 символах содержимого журнала (это значение можно настроить в **Log Monitoring** > **Timestamp/Splitting patterns**). Если смещение или часовой пояс не найдены, используется локальный часовой пояс хоста.

Для самой новой версии Dynatrace, см. [Поддерживаемые форматы временных меток](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-supported-timestamp-format.md "Поддерживаемые временные метки для последней версии Log Management and Analytics.").

Во время обработки журналов каждая строка, в которой обнаружена поддерживаемая временная метка, начинает новую запись журнала. Строка без временной метки считается продолжением существующей записи журнала и добавляется к строке, содержащей временную метку.
Если временная метка отсутствует в файле журнала или не распознается из-за неподдерживаемого формата, каждая строка, не начинающаяся с пробельных символов, таких как `space` или `tab`, начинает новую запись журнала. Каждая строка, начинающаяся с пробельных символов, таких как `space` или `tab`, рассматривается как продолжение записи журнала.
В связи с множеством форматов входящих данных журнала, Log Monitoring также позволяет определить конкретный формат даты, используя правила временных меток, которые указывают, что следует считать временной меткой в записи журнала. Подробности о настройке формата даты см. в разделе [Конфигурация временных меток журнала](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-timestamp-configuration.md "Определите конкретный формат даты, используя правила временных меток, которые указывают, что следует считать временной меткой в записи журнала.").

Поддерживаемые [форматы временных метокï»¿](https://dt-url.net/6e034iy) включают:

* ISO 8601 формат: `%Y-%m-%d %H:%M:%S`
  Пример: `2022-04-17 11:25:12.345`
* RFC 3339 формат: `%Y-%m-%dT%H:%M:%S`
  Пример: `2022-04-17T11:25:12.345`
* Unix Epoch формат, предоставляющий количество миллисекунд, прошедших с 1 января 1970 г.
  Пример: `1652088888997`
* RFC 3164 формат: `%b %d %H:%M:%S`
  Пример: `Apr 17 11:25:12`
* Db2 (IBM database 2) формат: `%Y-%m-%d-%H.%M.%S`
  Пример: `2022-05-17-11.25.12.114000-300`
* IIS формат: `%m/%d/%Y, %H:%M:%S`
  Пример: `04/17/2022, 11:25:12.345`
* W3C (World Wide Web Consortium) формат: `%Y-%m-%d %H:%M:%S`
  Пример: `2022-04-17 11:25:12.345` указан в часовом поясе UTC
* Klog и Golang/glog формат: `[IWEF]%m%d %H:%M:%S`
  Пример: `I0408 06:40:02.634162`
* Другие распространенные форматы:

  + `%d %b %Y %H:%M:%S` (пример: `17 Apr 2022 11:25:12.345`)
  + `%Y %b %d %H:%M:%S` (пример: `2022 Apr 17 11:25:12.345`)
  + `%d/%b/%Y:%H:%M:%S` (пример: `17/Apr/2022:11:25:12.345`)

JSON файлы поддерживаются только для Docker. Если в JSON файле обнаружен тег `log`, соответствующее сообщение извлекается и анализируется. В сообщении ищется временная метка одного из поддерживаемых форматов. Если поддерживаемый формат файла не найден, выполняется поиск тега `time`. Например:

```
{
"log":"2020-11-24 11:01:36,484 CRIT Supervisor running as root (no user in config file)\n",
"stream":"stdout",
"time":"2020-11-24T11:01:36.484996713Z"
}
{
"log":"2020-11-24 11:01:36,500 INFO RPC interface 'supervisor' initialized\n",
"stream":"stdout",
"time":"2020-11-24T11:01:36.50065223Z"
}
```

### Примеры допустимых форматов времени в файлах журналов

Большинство поддерживаемых форматов временных меток сочетаются с автоматическим определением формата часового пояса. Поддерживаются следующие форматы часовых поясов:

* Смещение к UTC зоне
  Примеры:

  `2022-04-17 11:25:12.345 +01:00`
  `2022-04-17 11:25:12.345 +0100`
  `2022-04-17 11:25:12.345Z -01:00`
  `2022-04-17 11:25:12.345Z-0100`
  `2022-04-17 11:25:12.345 UTC-01:00`
  `2022-04-17 11:25:12.345 GMT-0100`
* Полное [IANAï»¿](https://dt-url.net/jq034g6) имя часового пояса.
  Пример:
  `2022-04-17 11:25:12.345 Europe/Warsaw`
* Аббревиатура названия часового пояса
  Примеры:
  `2022-04-17 11:25:12.345 CEST`
  `2022-12-17 11:25:12.345 CET`
  `2022-04-17 11:25:12.345Z`
  `2022-04-17 11:25:12.345UTC+0100`

Если в строке журнала не указан индикатор часового пояса, часовой пояс, используемый при разборе временной метки, по умолчанию устанавливается в часовой пояс, настроенный в
**Settings** > **Log Monitoring** > **Advanced log settings** > **Default timezone for agents**.

### Переопределение часового пояса

В случае сбоя автоматического определения формата часового пояса можно добавить правило переопределения часового пояса на [странице конфигурации временных меток журнала](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-timestamp-configuration.md#createrule "Определите конкретный формат даты, используя правила временных меток, которые указывают, что следует считать временной меткой в записи журнала.").

## Часовой пояс по умолчанию

При обработке временных меток OneAgent следует как за локальными, так и за UTC шаблонами часовых поясов.

### Локальные шаблоны часовых поясов

Если часовой пояс не указан в содержимом журнала, следующие форматы временных меток по умолчанию используют **локальный часовой пояс** хоста:

* ISO 8601 (`%Y-%m-%d %H:%M:%S`)
* RFC 3164 (`%b %d %H:%M:%S`)
* IIS (`%m/%d/%Y, %H:%M:%S`)
* Klog и Golang/glog (`[IWEF]%m%d %H:%M:%S`)
* `%d %b %Y %H:%M:%S`
* `%Y %b %d %H:%M:%S`
* `%d/%b/%Y:%H:%M:%S`

### UTC шаблоны часовых поясов

Если часовой пояс не указан в содержимом журнала, следующие форматы временных меток по умолчанию используют **UTC**:

* RFC 3339 (`%Y-%m-%dT%H:%M:%S`)
* Db2 (`%Y-%m-%d-%H.%M.%S`)
* W3C (`%Y-%m-%d %H:%M:%S`)
* Unix time

Чтобы убедиться, какой часовой пояс использовался для разбора временной метки, выполните поиск в журналах OneAgent (`oneagent-logmon-detailed.log`) по запросу `Diagnostic statistics for LGI`.
