# Настройка временных меток/разделения (Logs Classic)


Log Monitoring Classic

Dynatrace позволяет определять правила, управляющие временными метками данных логов.

Для новейшей версии Dynatrace см. [Настройка временных меток/разделения](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-timestamp-configuration.md "Определите конкретный формат даты с помощью правил временных меток, указывающих, что следует считать временной меткой в записи лога.").

## Обнаружение временных меток

По умолчанию мониторинг логов автоматически распознаёт только наиболее распространённое и однозначное подмножество поддерживаемых форматов дат. Подробнее см. [Поддерживаемые форматы временных меток (Logs Classic)](timestamp-data-format.md "Поддерживаемые временные метки для последней версии мониторинга логов Dynatrace.").

### Временная метка не обнаружена

Если мониторинг логов не может определить формат времени, каждая строка лога обрабатывается как отдельная запись лога с автоматически назначенной временной меткой (временная метка наблюдения) с минутным разрешением, за исключением строк, начинающихся с пробельных символов (пробел, табуляция), которые считаются продолжением записи.

### Лимит поиска временной метки

Независимо от формата, временная метка обычно находится в пределах первых 64 символов записи лога. Однако временная метка может находиться в другом месте, и в этом случае можно увеличить этот лимит на странице конфигурации OneAgent: **Log Monitoring** > **Timestamp/Splitting patterns**.

### Правила временных меток

Независимо от расположения в записи лога, временная метка может быть записана в различных форматах. Dynatrace поддерживает некоторые форматы временных меток по умолчанию, но иногда несколько форматов могут соответствовать входящим данным лога и сопоставить временную метку с неверным шаблоном.

По этой причине мониторинг логов также позволяет определить конкретный формат даты с помощью правил временных меток, указывающих, что следует считать временной меткой в записи лога. Эти правила содержат шаблон временной метки, часовой пояс и фильтры соответствия.

* **Pattern** -- определяет, что следует считать временной меткой в строке лога.
* **Timestamp search limit** -- указывает количество символов в каждой строке лога, считая от начала строки, в которых выполняется поиск временной метки.
* **Timestamps in indented lines** -- включите эту опцию, если не хотите разбирать временные метки в строках, начинающихся с пробельных символов.
* **Time zone** -- определяет часовой пояс временной метки. Необязательно, если шаблон временной метки включает индикатор часового пояса (`%z`).
* **Matcher** -- сужает область действия правила и применяет шаблон временной метки только к совпадающим записям лога. Поскольку нельзя использовать атрибут `log.content` в фильтрах соответствия шаблона временной метки, максимальная гранулярность -- это источник лога. Гранулярность находится на этом уровне, потому что шаблон временной метки используется для разделения содержимого источника лога на отдельные записи, и поэтому применяется до определения значения атрибута `log.content` (или любых других атрибутов, установленных на уровне отдельной записи лога).

  + Если вы создаёте несколько правил, соответствующих одним и тем же данным лога, выполняется поиск всех определённых форматов времени.
  + Если у вас есть хотя бы одно правило, соответствующее данному логу, предопределённые форматы к нему не применяются.

### Несколько шаблонов временных меток в одном источнике лога

При приёме записей лога OneAgent разбирает запись лога для поиска временной метки.
Для этого используется список шаблонов соответствия.

* Если запись лога содержит одну временную метку, OneAgent проверяет её по списку шаблонов, начиная с первого шаблона в списке. Если временная метка записи соответствует одному из шаблонов, OneAgent использует эту временную метку как начало записи лога.
* Если запись лога содержит две или более временных метки, OneAgent проверяет их по списку шаблонов, начиная с шаблона, совпавшего в предыдущей записи лога.

Поэтому возможна ситуация, когда даже если первая временная метка записи лога соответствует первому шаблону в списке, OneAgent фактически сопоставит вторую временную метку -- потому что предыдущая запись лога совпала со вторым шаблоном.
Пример см. в блоке кода ниже.

```
Log entry 1: Pattern 1, Pattern 2


Log entry 2: Pattern 2


Log entry 3: Pattern 1, Pattern 2
```

Для записи лога 3, даже если "Pattern 1" появляется первым в списке шаблонов, OneAgent фактически сопоставит "Pattern 2".

Чтобы обеспечить проверку OneAgent только нужных шаблонов временных меток, тщательно выбирайте шаблоны в вашем списке соответствия. В приведённом примере, чтобы гарантировать использование Pattern 1, удалите Pattern 2 из списка. Это может привести к тому, что сообщения, содержащие только временные метки Pattern 2, будут отброшены.

Например, если файл лога содержит шаблоны `%FT%T` (2024-01-01T12:30:01) и `%F %T %Z` (2024-01-01 12:30:01 UTC), и OneAgent успешно сопоставляет первый шаблон в строке, он отдаст приоритет этому шаблону для последующих строк.

## Поддерживаемые области действия

Поддерживаются три иерархических области действия: хост, группа хостов и среда.

Иерархические области объединяются в один список в следующем порядке:

1. Правила хоста
2. Правила группы хостов
3. Правила среды

![Диаграмма, показывающая приоритет правил для настройки временных меток.](https://dt-cdn.net/images/timestamp-sopes-955-9292323a57.png)

OneAgent получает объединённый список (объединённые списки из соответствующих хостов, групп хостов и сред) без указания определённых областей.

### Область действия хоста

Область действия хоста доступна через **Host settings** для конкретного хоста.

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic**.
2. Найдите и выберите ваш хост для отображения страницы обзора хоста.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**...**) > **Settings**.

4. В настройках хоста выберите **Log Monitoring** > **Timestamp/Splitting patterns**.
5. Настройте маскирование данных, добавив правила с набором фильтров соответствия, указывающих, что следует считать временной меткой в записи лога.

### Область действия группы хостов

Область действия группы хостов доступна через страницу **Host**.

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic** и выберите интересующий хост.
2. На странице обзора хоста выберите **Properties and tags**.
3. На панели **Properties and tags** найдите свойство **Host group**, чтобы увидеть имя группы хостов, к которой принадлежит выбранный хост.

   Свойство **Host group** не отображается, если выбранный хост не принадлежит ни одной группе хостов.
4. Выберите имя группы хостов, чтобы отобразить все хосты в этой группе. Откроется страница **OneAgent deployment**, отфильтрованная по выбранной группе хостов. Каждый указанный хост имеет ссылку **Host group:** `<имя группы>`, где `<имя группы>` -- это имя группы хостов, которую вы хотите настроить.
5. Выберите имя группы хостов в любой строке.

6. В настройках группы хостов выберите **Log Monitoring** > **Timestamp/Splitting patterns**.
7. Настройте маскирование данных, добавив правила с набором фильтров соответствия, указывающих, что следует считать временной меткой в записи лога.

### Область действия среды

Область действия среды доступна в меню настроек.

1. Перейдите в **Settings** > **Log Monitoring** > **Timestamp/Splitting patterns**.
2. Настройте маскирование данных, добавив правила с набором фильтров соответствия, определяющих конфиденциальные данные.

## Создание правила

Чтобы добавить правило (на уровне хоста, группы хостов или среды), которое интерпретирует временные метки входящих данных логов:

1. Выберите **Add rule**, чтобы начать настройку правила.
2. **Rule name**

   Отображаемое имя вашей конфигурации.
3. **Pattern**

   Введите шаблон, который будет считываться как дата из логов. Подробнее о форматах временных меток см. [Поддерживаемые форматы временных меток (Logs Classic)](timestamp-data-format.md "Поддерживаемые временные метки для последней версии мониторинга логов Dynatrace.") и [библиотеку дат](https://dt-url.net/o8034wt).

   Необходимо указать как минимум год, месяц, день, часы, минуты и секунды, хотя можно использовать альтернативные форматы для них. Можно включить индикатор часового пояса (`%z`) или указать часовой пояс отдельно в определении правила.

   Правила без шаблона могут переопределить часовой пояс только для поддерживаемых по умолчанию временных меток.
4. **Timestamp search limit**

   Используйте это поле для определения количества символов в каждой строке лога, в которых выполняется поиск временной метки. Если вы хотите игнорировать временные метки и разделять логи с использованием [правил по умолчанию](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-one-agent-log-data-format.md#plain-text-logs "В этом разделе перечислены все форматы логов, поддерживаемые Log Management and Analytics"), установите значение `0`. Используйте это поле для перезаписи глобального [лимита поиска временной метки](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-timestamp-configuration.md#timestamp-search-limit "Определите конкретный формат даты с помощью правил временных меток, указывающих, что следует считать временной меткой в записи лога.") (по умолчанию 64 байта).
5. **Time zone**

   Выберите часовой пояс для применения к этому шаблону.
   Этот параметр недоступен, если часовой пояс уже указан в шаблоне временной метки (`%z`).

   Можно выбрать `Local time zone` для использования часового пояса хоста, на котором работает OneAgent.
6. Выберите **Add matcher**, чтобы создать конкретное соответствие для этого правила и сузить область его действия.

   В одном правиле можно включить несколько фильтров соответствия. Например, правило настройки временной метки может применяться к логам из конкретного контейнера, пространства имён или источника лога. Несколько фильтров с одним и тем же атрибутом используют логику AND, а фильтры с несколькими присвоенными значениями используют логику OR.
7. Выберите атрибут соответствия.
8. Выберите **Value** и в списке **Value** выберите обнаруженные элементы данных лога.

   К выбранному атрибуту можно добавить несколько значений. Можно иметь один фильтр соответствия, указывающий `Log source` и соответствующий значениям `/var/log/syslog` и `Windows Application Log`. Используйте звёздочки (`*`) в качестве подстановочных символов для частичного совпадения.
9. Выберите **Save changes**.

Правила выполняются в том порядке, в котором они указаны на странице **Timestamp/Splitting patterns**.

Задержка распространения изменения порядка правил

При изменении порядка правил (для изменения порядка их выполнения) дождитесь двух-трёх минут для распространения между сохранением изменения и его вступлением в силу.

Переключатель **Active**

Начиная с OneAgent версии 1.249 можно активировать/деактивировать правила, включая/выключая переключатель **Active**. Для эффективного управления правилами рекомендуется обновить OneAgent до версии 1.249. Если у вас есть правила, установленные на хосте с OneAgent версии ранее 249, вы не сможете их деактивировать -- в этом случае необходимо удалить такие правила, выбрав **Delete** на уровне правила или через REST API.

Правила выполняются в том порядке, в котором они указаны на странице **Timestamp/Splitting patterns**.

## REST API

Вы можете использовать Settings API для управления конфигурацией временных меток:

* Просмотр схемы
* Список сохранённых объектов конфигурации
* Просмотр отдельного объекта конфигурации
* Создание нового, редактирование или удаление существующего объекта конфигурации

Чтобы проверить текущую версию схемы для конфигурации временных меток, просмотрите список всех доступных схем и найдите идентификатор схемы `builtin:logmonitoring.timestamp-configuration`.

Объекты конфигурации временных меток доступны для настройки в следующих областях:

* `environment` -- объект конфигурации влияет на все хосты в данной среде.
* `host_group` -- объект конфигурации влияет на все хосты, назначенные данной группе хостов.
* `host` -- объект конфигурации влияет только на данный хост.

Чтобы создать конфигурацию временной метки с помощью API:

1. [Создайте токен доступа](../../../dynatrace-api/basics/dynatrace-api-authentication.md#create-token "Узнайте, как пройти аутентификацию для использования Dynatrace API.") с разрешениями **Write settings** (`settings.write`) и **Read settings** (`settings.read`).
2. Используйте конечную точку [GET a schema](../../../dynatrace-api/environment-api/settings/schemas/get-schema.md "Просмотр схемы настроек через Dynatrace API."), чтобы узнать формат JSON, необходимый для отправки конфигурации. Идентификатор схемы конфигурации временных меток (`schemaId`) -- `builtin:logmonitoring.timestamp-configuration`. Вот пример JSON-нагрузки с конфигурацией временной метки:

   ```
   [


   {


   "insertAfter":"uAAZ0ZW5hbnQABnRlbmFudAAkMGUzYmY2ZmYtMDc2ZC0zNzFmLhXaq0",


   "schemaId": "builtin:logmonitoring.timestamp-configuration",


   "schemaVersion": "0.1.0",


   "scope": "tenant",


   "value": {


   "config-item-title": "Added from REST API",


   "date-time-pattern": "%Y-%m-%d %H:%M:%S",


   "timezone": "CET",


   "matchers": [


   {


   "attribute": "dt.entity.process_group",


   "operator": "MATCHES",


   "values": [


   "PROCESS_GROUP-05F00CBACF39EBD1"


   ]


   },


   {


   "attribute": "log.source",


   "operator": "MATCHES",


   "values": [


   "Windows System Log",


   "Windows Security Log"


   ]


   }


   ]


   }


   }


   ]
   ```
3. Используйте конечную точку [POST an object](../../../dynatrace-api/environment-api/settings/objects/post-object.md "Создание или валидация объекта настроек через Dynatrace API.") для отправки вашей конфигурации.
