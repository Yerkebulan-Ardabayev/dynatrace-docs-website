Мониторинг журналов Classic

Для самой новой версии Dynatrace см. Потоковые журналы Kubernetes с помощью модуля журналов Dynatrace.

Мониторинг журналов Dynatrace поддерживает сбор журналов из систем оркестровки контейнеров Kubernetes через OneAgent.

В качестве альтернативы сбору журналов на основе OneAgent, вы можете передавать журналы в Dynatrace через потребление журналов API с интеграцией, такой как Fluent Bit , Fluentd , Logstash или [сборщик Dynatrace](../../../ingest-from/opentelemetry/collector.md#dt-collector-dist "Узнайте о сборщике Dynatrace OTel.").

## Поддерживаемые функции

Мониторинг журналов Dynatrace поддерживает различные платформы контейнеров на основе Kubernetes, такие как Upstream Kubernetes или Red Hat OpenShift, использующие **containerd** или **CRI-O** в качестве среды выполнения контейнеров.

**Docker** не соответствует интерфейсу контейнерного выполнения CRI. По этой причине конфигурации Kubernetes, использующие **Docker**, поддерживаются только частично. Kubernetes прекратил поддержку **Docker** в качестве среды выполнения контейнеров после версии v1.20.

Для получения более подробной информации о поддерживаемых версиях Kubernetes см. Поддержка оператора Dynatrace и известные проблемы.

## Автоматическое обнаружение журналов Kubernetes

OneAgent автоматически обнаруживает журналы, написанные контейнерным приложением в его потоки **stdout**/**stderr**. Движок Kubernetes сохраняет эти потоки журналов в файл на узле Kubernetes. OneAgent автоматически обнаруживает эти файлы журналов и сообщает о журналах контейнеров под источником журнала `Container Output`.

Журналы, написанные напрямую в файловую систему подов, не обнаруживаются OneAgent. В этом случае используйте интеграцию с лог-шиппером, такой как Fluent Bit.

## Обогащение журналов метаданными Kubernetes

Модуль журналов OneAgent украшает принятые журналы следующими метаданными Kubernetes: `k8s.cluster.name`, `k8s.cluster.uid`, `k8s.namespace.name`, `k8s.workload.name`, `k8s.workload.kind`, `dt.entity.kubernetes_cluster`, `k8s.pod.name`, `k8s.pod.uid`, `k8s.container.name`, `dt.entity.kubernetes_node`. Эти метаданные используются для сопоставления журналов с моделью сущностей кластеров Kubernetes, пространств имен, рабочих нагрузок и подов.

Кроме того, любые аннотации подов, начинающиеся с префикса `metadata.dynatrace.com/`, добавляются к записям журнала.

См. обогащение метаданными для Kubernetes для получения более подробной информации.

### Контроль потребления журналов с помощью метаданных Kubernetes

Вы можете контролировать журналы из Kubernetes с помощью правил потребления журналов в Dynatrace. Вы можете настроить эти правила на уровне кластера Kubernetes, чтобы разрешить потребление журналов, специфичных для кластера. Правила используют сопоставители для метаданных Kubernetes и других общих атрибутов записей журнала, чтобы определить, какие журналы должны быть потреблены.
Стандартные функции обработки журналов из OneAgent, включая маскирование конфиденциальных данных, настройку временных меток и автоматическое обогащение записей журнала, также доступны и включены здесь.

Используйте следующие рекомендуемые атрибуты сопоставления при настройке потребления журналов из Kubernetes.

1

Может измениться в будущих версиях OneAgent. Отдельные сопоставители для каждого типа рабочей нагрузки будут доступны. Мы рекомендуем использовать имя рабочей нагрузки Kubernetes и тип рабочей нагрузки Kubernetes вместо этого.

2

Атрибут уровня записи журнала, преобразованный модулем журналов OneAgent, отличается от атрибута статуса журнала, преобразованного сервером Dynatrace. Узнайте больше, перейдя на страницу [Автоматическое обогащение журнала](lm-log-data-transformation.md#transform-all-types-of-logs "Общее потребление журналов автоматически преобразует данные журнала в выходные значения для атрибута уровня журнала.").

## Настройка потребления журналов из Kubernetes

Потребление журналов из Kubernetes требует определения правил потребления журналов. Конфигурация основана на иерархии правил, которые используют сопоставители для атрибутов Kubernetes и других общих атрибутов записей журнала. Эти правила определяют, какие файлы журналов, среди обнаруженных OneAgent, потребляются.

Используйте следующие рекомендуемые атрибуты сопоставления при настройке потребления журналов из Kubernetes.

1

Атрибут уровня записи журнала, преобразованный OneAgent, отличается от атрибута статуса журнала, преобразованного сервером Dynatrace. Узнайте больше, перейдя на страницу [Автоматическое обогащение журнала](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-log-data-transformation-oa.md#transform-all-types-of-logs "Общее потребление журналов автоматически преобразует данные журнала в выходные значения для атрибута уровня журнала.").

2

Минимальная необходимая версия OneAgent — 1.273.

Вы также можете использовать следующие общие атрибуты сопоставления при настройке потребления журналов из Kubernetes.

### Иерархия правил потребления журналов

Правила потребления журналов можно определить в объеме окружения, а также на уровне хоста или группы хостов. Иерархия сопоставления следующая:

1. Правила конфигурации хоста;
2. Правила конфигурации группы хостов;
3. Правила конфигурации арендатора.

Сопоставление происходит в предопределенной иерархии, и правила выполняются сверху вниз. Это означает, что если правило выше в списке соответствует определенным данным журнала, то нижние правила будут пропущены. Элементы, сопоставленные в более высоких уровнях конфигурации, перезаписываются в более низких уровнях конфигурации, если они соответствуют одинаковым данным журнала. Если нет сопоставленного правила, файл не отправляется.

См. [Объемы конфигурации](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-log-storage-configuration.md#configuration-scopes "Включите и исключите конкретные источники журналов, уже известные OneAgent, для хранения и анализа.") для четырех объемов иерархии конфигурации.

## Сценарии использования

Изучите следующие сценарии использования для потребления журналов из сред Kubernetes с помощью Dynatrace. Настройка потребления журналов с помощью различных сопоставителей позволяет контролировать, какие журналы захватываются в системе. Сценарии использования ниже предлагают рекомендации по настройке Dynatrace для захвата журналов на основе ваших конкретных потребностей в мониторинге, будь то из конкретного пространства имен, контейнера или других критериев.

### Потребление всех журналов из конкретного пространства имен

1. Перейдите в **Настройки** и выберите **Мониторинг журналов** > **Правила потребления журналов**.
2. Выберите **Добавить правило** и введите имя конфигурации в поле **Имя правила**. 
   Убедитесь, что кнопка **Включить в хранилище** включена, чтобы журналы, соответствующие этой конфигурации, хранились в Dynatrace.
3. Выберите **Добавить условие**.
4. В выпадающем меню **Атрибут сопоставителя** выберите **Имя пространства имен K8s**.
5. Выберите пространство имен из выпадающего меню внутри поля **Значение** и выберите **Добавить сопоставитель**.
6. Выберите **Сохранить изменения**.

Теперь вы можете проанализировать журналы в просмотрщике журналов или тетрадях после фильтрации соответствующего пространства имен. Вы также можете найти журналы в контексте приложения Kubernetes, выбрав вкладку **Журналы**.

### Потребление журналов из конкретного пространства имен и контейнера

1. Перейдите в **Настройки** и выберите **Мониторинг журналов** > **Правила потребления журналов**.
2. Выберите **Добавить правило** и введите имя конфигурации в поле **Имя правила**. 
   Убедитесь, что кнопка **Включить в хранилище** включена, чтобы журналы, соответствующие этой конфигурации, хранились в Dynatrace.
3. Выберите **Добавить условие**.
4. В выпадающем меню **Атрибут сопоставителя** выберите **Имя пространства имен K8s**.
5. Выберите пространство имен из выпадающего меню внутри поля **Значение** и выберите **Добавить сопоставитель**.
6. Добавьте новый сопоставитель, на этот раз выберите **Имя контейнера K8s**, и введите имя контейнера в поле **Значение**. Вы можете добавить несколько имен контейнеров на этом этапе конфигурации.
7. Выберите **Сохранить изменения**.

Теперь вы можете проанализировать журналы в просмотрщике журналов или тетрадях после фильтрации соответствующего пространства имен и контейнера. Вы также можете найти журналы в контексте приложения Kubernetes, выбрав вкладку **Журналы**.

### Потребление всех журналов Kubernetes, исключая конкретные пространства имен

...

1. Перейдите в **Настройки** и выберите **Мониторинг журналов** > **Правила ингеста журналов**.
2. Выберите **Добавить правило** и введите название вашей конфигурации в поле **Имя правила**. 
   Убедитесь, что кнопка **Включить в хранилище** включена, чтобы журналы, соответствующие этой конфигурации, хранились в Dynatrace.
3. Выберите **Добавить условие**.
4. Из выпадающего списка **Атрибут сопоставления** выберите **Имя пространства имен K8s**.
5. Вставьте символ звездочки (\*) в поле **Значение**, как плейсхолдер для всех доступных пространств имен кластера.
6. Выберите **Добавить сопоставление**.
7. Выберите **Сохранить изменения**.
8. Вернитесь на экран **Правила ингеста журналов**, добавьте еще одно правило и выберите опцию **Исключить из хранилища**.
9. В поле **Значение** добавьте пространства имен, которые вы хотите исключить при ингесте журналов Kubernetes.
10. Выберите **Добавить сопоставление**.
11. Выберите **Сохранить изменения**.

На экране **Правила ингеста журналов** расположите настроенные правила так, чтобы правило исключения пространств имен было вверху, а правило включения всех пространств имен - внизу.

## REST API

Вы можете использовать настройки API, чтобы управлять правилами ингеста журналов:

* Просмотреть схему;
* Перечислить сохраненные объекты конфигурации;
* Просмотреть отдельный объект конфигурации;
* Создать новый, отредактировать или удалить существующий объект конфигурации.

Чтобы проверить текущую версию схемы для правил ингеста журналов, перечислите все доступные схемы и найдите идентификатор схемы `builtin:logmonitoring.log-storage-settings`.

Объекты правил ингеста журналов можно настроить для следующих областей:

* `tenant` – объект конфигурации влияет на все хосты в данном арендаторе.
* `host_group` – объект конфигурации влияет на все хосты, назначенные данной группе хостов.
* `host` – объект конфигурации влияет только на данный хост.

Чтобы создать правило ингеста журналов с помощью API:

1. [Создайте токен доступа](../../../dynatrace-api/basics/dynatrace-api-authentication.md#create-token "Узнайте, как получить доступ к Dynatrace API.") с областями **Запись настроек** (`settings.write`) и **Чтение настроек** (`settings.read`).
2. Используйте конечную точку GET-схемы, чтобы узнать формат JSON, необходимый для отправки вашей конфигурации. Идентификатор схемы правил ингеста журналов (`schemaId`) равен `builtin:logmonitoring.log-storage-settings`. Вот пример полезной нагрузки JSON с правилами ингеста журналов:

## FAQ

Каковы предпосылки для автодисквери и ингеста Kubernetes журналов через OneAgent?

Требования для автодисквери и ингеста Kubernetes журналов следующие:

* Используется контейнерный движок **containerd** или **CRI-O**;
* Процесс, запущенный в контейнере, является важным процессом;
* Журналы записываются в потоки **stdout**/**stderr** контейнера;
* Файл журнала на узле Kubernetes существует не менее одной минуты после завершения выполнения контейнера.

Возможно ли украшать журналы метками и аннотациями Kubernetes?

Нет, OneAgent пока не предлагает такую функциональность, хотя она запланирована в будущих выпусках.

Для получения более подробной информации о ингесте см. страницу [Правила ингеста журналов](../../logs/lma-log-ingestion/lma-log-ingestion-via-oa/lma-log-storage-configuration.md#faq "Включить и исключить определенные источники журналов, уже известные OneAgent, для хранения и анализа.").
