# События логов (Logs Classic)

* Classic
* Руководство
* Чтение: 5 мин.
* Обновлено 8 октября 2025 г.

Log Monitoring Classic

Dynatrace Log Monitoring позволяет создавать события логов на основе данных логов и использовать их при обнаружении проблем.

Ценообразование событий логов основано на модели Davis data units (DDU). Ознакомьтесь с [DDU для пользовательских событий Davis](../../../license/monitoring-consumption-classic/davis-data-units/ddu-events.md "Узнайте, как рассчитать потребление Davis data units и затраты, связанные с пользовательскими настроенными и принятыми событиями."), чтобы узнать, как можно оценить и отслеживать потребление DDU для событий логов.

Когда Dynatrace принимает данные логов, он применяет запрос, указанный в определении события лога. Каждое найденное совпадение вызывает событие лога, которое может быть настроено для индивидуального создания проблемы для каждого вызванного события лога или для объединения нескольких событий в одну проблему.

## Создание события лога

1. Перейдите в **Settings** > **Log Monitoring** > **Events extraction** и выберите **Add log event**.
2. Введите **Summary**.
   Резюме выступает в качестве отображаемого имени конфигурации события лога. Резюме должно быть уникальным для всех конфигураций.
3. Введите **Matcher**.
   Введите запрос Log Monitoring для фильтрации данных логов для вашего события лога. Подробнее см. [Log viewer](log-viewer.md#query-syntax "Узнайте, как использовать средство просмотра логов Dynatrace для анализа данных логов.").

   Я перешёл на Grail

   Если вы перешли на [Dynatrace Grail](../../../platform/grail/dynatrace-grail.md "Grail — это озеро данных Dynatrace, специально разработанное для данных наблюдаемости и безопасности, и являющееся единым хранилищем для логов, метрик, трассировок, событий и многого другого."), вы можете начать использовать функции [DQL](../../../platform/grail/dynatrace-query-language.md "Как использовать Dynatrace Query Language.") в ваших запросах Log Monitoring. Подробнее см. [Обработка логов с классическим конвейером](../../logs/lma-classic-log-processing.md#dql-functions "Используйте правила обработки логов для изменения входящих данных логов для лучшего понимания, анализа или дальнейшего преобразования.").
4. Настройте **Event template**.

   * Укажите **Title** события, которое будет вызвано. Если это событие лога вызывает проблему, этот заголовок также станет заголовком проблемы.
   * Укажите **Description** события. Обратите внимание, что в описании можно использовать один или несколько заполнителей (подробнее см. [Заполнители](#placeholders) ниже).
   * Выберите **Event Type**. Типы событий указывают на серьёзность события. См. [Settings API — таблица схемы событий логов](../../../dynatrace-api/environment-api/settings/schemas/builtin-logmonitoring-log-events.md "Просмотрите таблицу схемы настроек builtin:logmonitoring.log-events вашей среды мониторинга через Dynatrace API.").
5. Выберите, разрешить ли **Allow merge**.
   Если вызываются два или более событий, Dynatrace может объединить эти события в одну проблему. Эта опция позволяет отключить такое поведение, что может привести к увеличению количества отчётных проблем.

   Свойство `dt.event.allow_davis_merge` **не** разделяет проблемы для одной и той же конфигурации событий логов. Оно только предотвращает объединение с проблемами из других конфигураций событий логов или других доменов проблем (таких как пользовательские оповещения).
   Если вы хотите разделить проблемы для одной и той же конфигурации событий логов, свойство `event.unique_identifier` должно **присутствовать в фактических данных логов**. Затем вы можете использовать заполнитель в конфигурации события лога для ссылки на это свойство.
   Например, чтобы создать отдельную проблему для каждого сообщения строки лога, используйте `event.unique_identifier={content}`.
6. Добавьте **Properties**.
   Свойство — это пара «ключ-значение», которая устанавливается для каждого вызванного события. Вы можете использовать один или несколько заполнителей в качестве значения, которые будут извлечены из данных логов. Например, свойство с **Key**, установленным в `PGI`, и **Value** в виде заполнителя `{dt.entity.process_group_instance}` извлечёт значение экземпляра группы процессов из данных логов при вызове события. Если подстановка заполнителя не удалась, ни ключ, ни значение не будут доступны.
   Чтобы получить полный список свойств, перейдите к [Events API v2 — GET all event properties](../../../dynatrace-api/environment-api/events-v2/get-event-properties.md "Получите список всех свойств событий через Dynatrace API."). Поле `description` в теле ответа API объясняет, как работает каждое свойство. Для [примера](../../../dynatrace-api/environment-api/events-v2/get-event-properties.md#exampledesc "Получите список всех свойств событий через Dynatrace API."): в `dt.event.allow_davis_merge` описание гласит: `"Allow Davis AI to merge this event into existing problems (true) or force creating a new problem (false)"`.

### Установка тайм-аута для события лога

У событий логов тайм-аут по умолчанию составляет 15 минут. Тайм-аут определяет, как часто источник событий должен обновлять событие лога, чтобы оно оставалось активным. Максимально допустимое время для события лога — шесть часов.

* Событие лога остаётся активным, если источник события отправляет обновление до истечения тайм-аута (по умолчанию: `15` минут).
* Событие автоматически закрывается, если обновление не отправлено в течение периода тайм-аута.
* Вы можете настроить тайм-аут.

Чтобы установить пользовательский тайм-аут:

1. При создании или редактировании события лога нажмите **Add property**.
2. Установите **Key** в значение `dt.event.timeout`.
3. Установите **Value** в количество минут (например, `12`).

Чтобы проверить, что пользовательский тайм-аут был добавлен для события, вызванного на уровне хоста:

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic** и выберите имя хоста.
2. Прокрутите вниз до **Events** и разверните **Details** для события.

### Заполнители

Заполнители — это атрибуты записей логов, которые можно использовать для извлечения фактического значения из данных логов.

* Вы можете использовать любой атрибут, перечисленный в средстве просмотра логов для заданной записи лога, в качестве заполнителя для извлечения значения этого атрибута.
* Вы можете использовать дополнительные атрибуты записей логов в средстве просмотра логов в качестве заполнителей для значений, которые они представляют в данных логов. Заключайте значения заполнителей в фигурные скобки (например, `{dt.process.name}`).

## Пример

В этом примере мы создаём событие лога на основе принятых данных логов. Это событие лога будет вызвано, когда принятая запись лога совпадёт с вводом в поле **Matcher**. Matcher будет искать статус `error` на хосте `555f5555-555a-5dd5-55f555a5b55d`. Мы добавляем свойства (атрибуты) события лога, которые будут извлекать значения из данных логов и включать их в сводку проблемы.

1. Перейдите в **Settings** > **Log Monitoring** > **Events extraction** и выберите **Add log event**.
2. Установите следующие значения:

   * **Summary:** `syslog-agent log event`
   * **Matcher:** `status="error" AND host.name="555f5555-555a-5dd5-55f555a5b55d"`
   * **Event template** — **Title:** `[Log] log events demo`
   * **Event template** — **Description:** `{content}`
   * **Event template** — **Event type:** `Custom alert`
3. Отключите **Allow merge**.
4. Для **Properties** добавьте следующие два свойства:

   * Ключ `K8 Id` со значением `{dt.kubernetes.config.id}`
   * Ключ `Process Name` со значением `The process name is -> {dt.process.name}`
5. Сохраните изменения и дождитесь приёма данных логов.

Страница примера событий логов

![Экран настроек для конфигурации событий логов.](https://dt-cdn.net/images/screenshot-log-events-settings-641-583e83f210.png)

Логи могут содержать атрибуты с длинными значениями до 32 КБ. Такой атрибут может быть добавлен как часть раздела **Event template**. В этом случае события создаются, но значения обрезаются до 4096 символов.

Если совпадение найдено, это событие лога создаст проблему для каждого вызванного события лога.

![Проблема, созданная определёнными событиями логов.](https://dt-cdn.net/images/screenshot-log-events-problem-845-dc714350b6.png)

Проблема, созданная в результате вызванного события лога, будет содержать информацию, сопоставленную из вашей конфигурации события лога. Заголовок проблемы — это **Event title**, а раздел влияния отражает ваши настройки **Event type** и дополнительную информацию, настроенную в описании и свойствах события лога.

Обратите внимание, что в этом примере, хотя мы настроили два свойства (`K8 Id` и `Process Name`), на странице проблемы отображается только одно. Это потому, что только заполнитель `{dt.process.name}` имел значение в принятых данных логов. Значение `{dt.kubernetes.config.id}` не было найдено в этой конкретной записи лога, и определённое свойство было проигнорировано.
