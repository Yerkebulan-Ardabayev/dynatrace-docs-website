Рекомендуемый способ приёма syslog

Используйте потоковую передачу syslog через Fluentd, если вы уже собираете логи с его помощью или если конкретный сценарий использования требует дополнительного компонента, например, для пересылки логов в различные конечные точки. Если вы хотите воспользоваться преимуществами безопасного, доверенного пограничного компонента с корпоративной поддержкой и управлением жизненным циклом, см. [Приём syslog с помощью ActiveGate](lma-log-ingestion-syslog.md "Приём данных syslog-логов в Dynatrace с использованием ActiveGate и их преобразование в информативные сообщения логов.").

В случае, когда основной задачей является наблюдаемость системного syslog Linux, мы рекомендуем развернуть OneAgent, который [автоматически обнаруживает данные syslog хоста](lma-log-ingestion-via-oa/lma-autodiscovery.md#oneagent-log-configuration-flow "Dynatrace автоматически обнаруживает все новые файлы логов, соответствующие определённым требованиям."), сохраняет контекст топологии и требует минимальной настройки и обслуживания.

Вы можете отправлять Syslog в Dynatrace с помощью Fluentd. Настройте Fluentd для отправки Syslog в API приёма логов Dynatrace.

![Диаграмма, показывающая поток приёма Syslog через Fluentd](https://dt-cdn.net/images/syslog-fluentd-1486-41dd7cc63c.png)

## Возможности

* Syslog — это стандартный протокол для ведения журналов сообщений и управления системными логами. Маршрутизаторы, принтеры, хосты, коммутаторы и другие устройства на различных платформах используют Syslog для регистрации действий пользователей, событий жизненного цикла систем и программного обеспечения, состояния или диагностики.
* Во время мониторинга сети удалённый сервер Syslog прослушивает сообщения логов клиента и консолидирует данные логов, которые затем могут быть обработаны с использованием возможностей Dynatrace Log Management and Analytics на основе Grail и анализа первопричин на базе ИИ Dynatrace.

## Настройка

Настройте поток от источника Syslog через Fluentd до Dynatrace, выполнив следующие шаги:

1. Получите [токен API Dynatrace](../../../dynatrace-api/basics/dynatrace-api-authentication.md "Узнайте, как пройти аутентификацию для использования API Dynatrace.") с областью действия `logs.ingest` (Ingest Logs).
2. Разверните Fluentd в соответствии с вашими предпочтениями.

   * [Развернуть Fluentd](https://dt-url.net/o9034h3). Fluentd также может работать как [DaemonSet в кластере Kubernetes](https://dt-url.net/t2234xz). Встроенная отказоустойчивость обеспечивает полноту и согласованность данных, даже если Fluentd или конечный сервис временно недоступны.
3. Включите в Fluentd приём входящих сообщений Syslog.

   * Плагин ввода [in\_syslog](https://dt-url.net/t00343n) позволяет Fluentd получать записи по протоколу Syslog через UDP или TCP. Он включён в ядро Fluentd, поэтому на этом шаге дополнительная установка не требуется.
4. Используйте [плагин Fluentd для Dynatrace](https://dt-url.net/gb23475) для потоковой передачи логов в кластер Dynatrace. Плагин Fluentd для Dynatrace с открытым исходным кодом использует [API приёма логов](lma-log-ingestion-via-api.md "Потоковая передача данных логов в Dynatrace с помощью API и их преобразование в информативные сообщения логов.") для отправки логов в Dynatrace.
5. Настройте ваши устройства для отправки syslog в Fluentd.

## Отправка syslog на удалённую конечную точку

В приведённых ниже примерах вы можете отправлять syslog на удалённую конечную точку, которой является Fluentd.

### Пример 1

Настройте Rsyslog на Linux Ubuntu для пересылки syslog на удалённый сервер Fluentd.
Вам необходимо добавить следующую строку в файл конфигурации демона syslog `/etc/rsyslog.conf` (протокол UDP):

```
*.* @<fluentd host IP>:5140
```

* `*.*` указывает демону пересылать все сообщения на указанный экземпляр Fluentd, прослушивающий порт 5140. `<fluentd host IP>` должен указывать на IP-адрес Fluentd.
* Если вы используете TCP, укажите два символа `@` (`@@`) следующим образом:

```
*.* @@<fluentd host IP>:5140
```

### Пример 2

Настройте систему F5 BIG-IP для записи логов на удалённый сервер syslog (11.x - 17.x).
Обратитесь к [документации F5 BIG-IP](https://dt-url.net/080348q) для получения инструкций по настройке удалённого Syslog.

## Добавление атрибутов к syslog в Fluentd

Платформа программной аналитики Dynatrace и Dynatrace Intelligence зависят от контекстно насыщенных, качественных данных. Вы можете предоставить контекст для данных, принимаемых через API приёма логов, который поддерживает набор [ключей и семантических атрибутов](../../../dynatrace-api/environment-api/log-monitoring-v2/post-ingest-logs.md#parameters "Отправка пользовательских логов в Dynatrace через Log Monitoring API v2."). Вы также можете предоставить пользовательские атрибуты, не требующие индексации, в [Grail](../../../platform/grail/dynatrace-grail.md "Grail — это хранилище данных Dynatrace, специально разработанное для данных наблюдаемости и безопасности, выступающее единым хранилищем для логов, метрик, трейсов, событий и многого другого.").

### Пример 1

Добавьте атрибут `log.source` на основе источника syslog в Fluentd.
Сообщение syslog часто нуждается в дополнительном контексте для различения источников при анализе. В этом примере в Fluentd настроены две отдельные конечные точки syslog: одна для логов Linux, а другая для логов F5. Это помогает обогатить потоки логов значимым атрибутом `log.source`. Файл конфигурации Fluentd должен выглядеть следующим образом:

```
<source>



@type syslog



port 5140



bind 0.0.0.0



tag system-linux



</source>



<source>



@type syslog



port 5141



bind 0.0.0.0



tag system-f5



</source>
```

Вам необходимо добавить атрибут `log.source` на основе тега Fluentd.

```
<filter system-linux.**>



@type record_transformer



<record>



log.source "linux syslogs"



</record>



</filter>



<filter system-f5.**>



@type record_transformer



<record>



log.source "f5 syslogs"



</record>



</filter>
```

Для получения дополнительной информации обратитесь к [документации плагина фильтра record\_transformer для Fluentd](https://dt-url.net/ac2345m).
