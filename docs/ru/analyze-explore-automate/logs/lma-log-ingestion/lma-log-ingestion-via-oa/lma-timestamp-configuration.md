# Настройка меток времени и разделения записей


* Latest Dynatrace
* Руководство
* Чтение: 17 мин
* Обновлено 24 сентября 2025 г.

Dynatrace позволяет определять правила, управляющие метками времени в данных логов.

## Обнаружение меток времени

По умолчанию мониторинг логов автоматически распознаёт только наиболее распространённые и однозначные форматы дат. Подробности см. в разделе [Поддерживаемые форматы меток времени](lma-supported-timestamp-format.md "Поддерживаемые форматы меток времени для последней версии Log Management and Analytics."). Каждый раз, когда обнаруживается шаблон метки времени, строка считается началом записи лога. Все последующие строки без обнаруженной метки времени считаются продолжением и передаются как одна многострочная запись лога.

Вы также можете управлять обнаружением меток времени, используя следующие параметры из ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Log monitoring** > **Advanced log settings**:

* Определение часовых поясов контейнеров: эта опция включает автоматическое определение часового пояса в логах контейнера, если часовой пояс не задан явно или не настроен.
* Часовой пояс по умолчанию для агентов: эта опция задаёт часовой пояс по умолчанию для агента, если более конкретные настройки не определены.
* Предел поиска метки времени: эта опция определяет количество символов в каждой строке лога (начиная с первого символа строки), в пределах которых выполняется поиск метки времени.

### Метка времени не обнаружена

Если мониторинг логов не может определить формат времени, каждая строка лога считается отдельной записью с автоматически назначенной меткой времени (метка времени наблюдения) с минутным разрешением, за исключением строк, начинающихся с пробелов (пробел, табуляция), которые считаются продолжением записи.

### Предел поиска метки времени

Независимо от формата, метка времени обычно находится в первых 64 символах записи лога. Однако метка времени может располагаться в другом месте — в этом случае вы можете увеличить этот предел на странице настройки OneAgent: **Log Monitoring** > **Timestamp/Splitting patterns**.

### Правила меток времени

Независимо от того, где метка времени находится в записи лога, она может быть записана в нескольких форматах. Dynatrace поддерживает некоторые форматы меток времени по умолчанию, но иногда несколько форматов могут подходить к входящим данным лога и сопоставлять метку времени с неправильным шаблоном.

Поэтому мониторинг логов также позволяет определить конкретный формат даты с помощью правил меток времени, которые задают, что именно следует считать меткой времени в записи лога. Эти правила содержат шаблон метки времени, часовой пояс и условия сопоставления.

* **Шаблон** — определяет, что должно считаться меткой времени в строке лога.
* **Предел поиска метки времени** — задаёт количество символов в каждой строке лога, начиная с начала строки, в пределах которых выполняется поиск метки времени.
* **Граница записи** — необязательное поле. Определяет [границу записи](lma-log-entry-boundary.md "Определение конкретного формата даты с помощью правил меток времени."). Необходимо указать фрагмент текста из первой строки записи. Шаблон воспринимается буквально, то есть поддержка звёздочки (\*) в качестве подстановочного символа отсутствует.
* **Метки времени в строках с отступами** — включите эту опцию, если не хотите разбирать метки времени в строках, начинающихся с пробельных символов.
* **Часовой пояс** — определяет часовой пояс метки времени. Необязательный, если ваш шаблон метки времени включает индикатор часового пояса (`%z`).
* **Условие сопоставления** — сужает область применения правила и применяет шаблон метки времени только к совпадающим записям лога. Поскольку нельзя использовать атрибут `log.content` в условиях сопоставления шаблона метки времени, максимальная гранулярность — это источник лога. Это связано с тем, что шаблон метки времени используется для разделения содержимого источника лога на отдельные записи, поэтому он применяется до определения значения атрибута `log.content` (или любых других атрибутов, устанавливаемых на уровне отдельной записи лога).

  + Если вы создадите несколько правил, сопоставляющих одни и те же данные лога, поиск будет выполняться по всем определённым форматам времени.
  + Если хотя бы одно правило соответствует данному логу, предопределённые форматы к нему не применяются.

### Примеры настройки меток времени

Ознакомьтесь с приведёнными ниже форматами меток времени в качестве примеров конфигурации:

* Метка времени без стандартного разделителя: `%Y-%m-%d-%H.%M.%S`
  Пример: `2024-09-05-12.30.01`
* Использование только меток времени с начала записи лога (%^): `%^%F %T`
  Пример: `2024-09-05 12:30:01`
* Поиск метки времени с именем поля (JSON): `"validTimestamp":"%Y-%m-%dT%H:%M:%S"`
  Пример: `"validTimestamp": "2024-09-05T12:30:01"`
* Метка времени со смещением часового пояса: `%m-%d-%Y %H:%M:%S %z`
  Пример: `09-05-2024 12:30:01 +01:00`
* Метка времени с названием или аббревиатурой часового пояса: `%m-%d-%Y %H:%M:%S %Z`
  Пример: `09-05-2024 12:30:01 UTC`
* Метка времени без года (для вычисления используется текущий год): `%b %t%d-%H:%M:%S`
  Здесь `%t` соответствует нулю или одному пробельному символу.
  Пример: `Apr 4-12:30:01` или `Apr 14-12:30:01`
* Любая метка времени с префиксом `myTime.*`: `myTime%*: %Y-%m-%dT%H:%M:%S`
  Пример: `myTimeOfCreation: 2024-09-05T12:30:01`
  Вы можете переопределить часовой пояс по умолчанию, задав часовой пояс без шаблона метки времени.
* Формат с двузначным годом: `%m/%d/%y %H:%M:%S %Z`
  Пример: `09/05/24 12:30:01 America/Chicago`

### Несколько шаблонов меток времени в одном источнике лога

При приёме записей лога OneAgent анализирует запись на наличие метки времени.
Для этого он использует список шаблонов сопоставления.

* Если запись лога содержит одну метку времени, OneAgent сравнивает её со списком шаблонов, начиная с первого шаблона в списке. Если метка времени записи совпадает с одним из шаблонов, OneAgent использует эту метку времени как начало записи лога.
* Если запись лога содержит две или более меток времени, OneAgent сравнивает метку времени со списком шаблонов, начиная с шаблона, который совпал в предыдущей записи лога.

Таким образом, возможна ситуация, когда, даже если первая метка времени записи соответствует первому шаблону в списке, OneAgent на самом деле сопоставит вторую метку времени — потому что предыдущая запись лога совпала со вторым шаблоном.
Пример приведён в блоке кода ниже.

```
Log entry 1: Pattern 1, Pattern 2


Log entry 2: Pattern 2


Log entry 3: Pattern 1, Pattern 2
```

Для записи лога 3, даже если «Pattern 1» стоит первым в списке шаблонов, OneAgent на самом деле сопоставит «Pattern 2».

Чтобы OneAgent использовал только нужные шаблоны меток времени, тщательно выбирайте шаблоны в списке сопоставления. В приведённом выше примере, чтобы гарантировать использование Pattern 1, удалите Pattern 2 из списка. Это может привести к тому, что сообщения, содержащие только метки времени Pattern 2, будут отброшены.

Например, если ваш файл лога содержит шаблоны `%FT%T` (2024-01-01T12:30:01) и `%F %T %Z` (2024-01-01 12:30:01 UTC), и OneAgent успешно сопоставит первый шаблон в строке, он будет приоритизировать этот шаблон для последующих строк.

## Поддерживаемые области действия

Поддерживаются четыре уровня иерархии: хост, группа хостов и среда.

Уровни иерархии объединяются в один список в следующем порядке:

1. Правила на уровне хоста
2. Правила на уровне кластера Kubernetes
3. Правила на уровне группы хостов
4. Правила на уровне среды

![log-storage](https://dt-cdn.net/images/log-storage-1280-2db879c27f.png)

OneAgent получает объединённый список (списки от соответствующих хостов, групп хостов и сред) без указания того, какие области определены.

### Область действия хоста

Область действия хоста доступна через **Host settings** для конкретного хоста.

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic**.
2. Найдите и выберите нужный хост, чтобы отобразить страницу обзора хоста.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**...**) > **Settings**.

4. В настройках хоста выберите **Log Monitoring** > **Timestamp/Splitting patterns**.
5. Настройте маскирование данных, добавив правила с набором условий сопоставления, которые определяют, что должно считаться меткой времени в записи лога.

### Область действия кластера Kubernetes

Область действия кластера Kubernetes доступна через страницу **Kubernetes**.

1. Перейдите в **Kubernetes** или **Kubernetes Classic** (latest Dynatrace) и выберите интересующий вас кластер.
2. Найдите и выберите нужный кластер, чтобы отобразить страницу обзора кластера.
3. В правом верхнем углу страницы обзора кластера выберите **More** (**...**) > **Settings**.
4. В настройках кластера перейдите в **Log Monitoring** > **Timestamp/Splitting patterns**.
5. Настройте загрузку в хранилище, добавив правила с набором атрибутов, соответствующих данным лога, которые должны храниться в Dynatrace.

### Область действия группы хостов

Область действия группы хостов доступна через страницу **Host**.

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic** и выберите интересующий вас хост.
2. На странице обзора хоста выберите **Properties and tags**.
3. На панели **Properties and tags** найдите свойство **Host group**, чтобы увидеть название группы хостов, к которой принадлежит выбранный хост.

   Свойство **Host group** не отображается, если выбранный хост не принадлежит ни к одной группе хостов.
4. Выберите название группы хостов, чтобы вывести список всех хостов в этой группе. Откроется страница **OneAgent deployment**, отфильтрованная по выбранной группе хостов. Для каждого хоста в списке отображается ссылка **Host group:** `<group name>`, где `<group name>` — название группы хостов, которую вы хотите настроить.
5. Выберите название группы хостов в любой строке.

6. В настройках группы хостов выберите **Log Monitoring** > **Timestamp/Splitting patterns**.
7. Настройте маскирование данных, добавив правила с набором условий сопоставления, которые определяют, что должно считаться меткой времени в записи лога.

### Область действия среды

Область действия среды доступна в меню настроек.

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Timestamp/Splitting patterns**.
2. Настройте маскирование данных, добавив правила с набором условий сопоставления для идентификации конфиденциальных данных.

## Создание правила

Чтобы добавить правило (на уровне хоста, группы хостов или среды), которое интерпретирует метки времени входящих данных лога:

1. Выберите **Add rule**, чтобы начать настройку правила.
2. **Rule name**
   Название, которое будет отображаться для вашей конфигурации.
3. **Pattern**
   Введите шаблон, который будет считываться как дата из логов. Подробности о форматах меток времени см. в разделе [Поддерживаемые форматы меток времени](lma-supported-timestamp-format.md "Поддерживаемые форматы меток времени для последней версии Log Management and Analytics.") и в приведённом ниже списке спецификаторов формата.

   Необходимо указать как минимум месяц, день, часы, минуты и секунды, хотя для них можно использовать альтернативные форматы. Вы можете включить индикатор часового пояса (`%z`) или указать часовой пояс отдельно в определении правила.

   Правила без шаблона могут переопределять часовой пояс только для поддерживаемых по умолчанию меток времени.
4. **Timestamp search limit**

   Используйте это поле для определения количества символов в каждой строке лога, в пределах которых выполняется поиск метки времени. Если вы хотите игнорировать метки времени и разделять логи с использованием [правил по умолчанию](lma-one-agent-log-data-format.md#plain-text-logs "Список всех форматов логов, поддерживаемых Log Management and Analytics"), установите значение `0`. Используйте это поле для переопределения глобального [предела поиска метки времени](lma-timestamp-configuration.md#timestamp-search-limit "Определение конкретного формата даты с помощью правил меток времени.") (по умолчанию 64 байта).
5. **Entry boundary**

   Используйте это поле, чтобы указать фрагмент текста из первой строки записи.
6. **Time zone**
   Выберите часовой пояс для применения к этому шаблону.
   Этот параметр недоступен, если вы уже указали часовой пояс в шаблоне метки времени (`%z`).

   Вы можете выбрать `Local time zone`, чтобы использовать часовой пояс хоста, на котором работает OneAgent.
7. Выберите **Add condition**, чтобы создать конкретное условие сопоставления для этого правила и сузить область его применения.

   В одном правиле можно указать несколько условий сопоставления. Например, правило настройки метки времени может применяться к логам из конкретного контейнера, пространства имён или источника лога. Несколько условий с одним и тем же атрибутом используют логику «И» между условиями, в то время как условия с несколькими значениями используют логику «ИЛИ».

   1

   Минимальная требуемая версия OneAgent — 1.295.

   2

   [Вручную или автоматически применённые теги](../../../../manage/tags-and-metadata/setup/how-to-define-tags.md#automatic "Узнайте, как определять и применять теги вручную и автоматически.") не видны для OneAgent.

   3

   Минимальная требуемая версия OneAgent — 1.289.

   4

   Минимальная требуемая версия модуля логирования OneAgent — 1.309.

   5

   Минимальная требуемая версия Dynatrace Operator — 1.4.2.
8. Выберите атрибут сопоставления.
9. Выберите **Value** и из списка **Value** выберите обнаруженные элементы данных лога.

   Вы можете добавить несколько значений к выбранному атрибуту. Можно создать одно условие, указывающее `Log source`, которое сопоставляет значения `/var/log/syslog` и `Windows Application Log`. Используйте звёздочки (`*`) в качестве подстановочных символов для частичного совпадения.
10. Выберите **Save changes**.

Правила выполняются в том порядке, в котором они отображаются на странице **Timestamp/Splitting patterns**.

Задержка распространения при изменении порядка правил

При изменении порядка правил (для изменения порядка их выполнения) предусмотрите задержку распространения в две-три минуты между сохранением изменения и его вступлением в силу.

Переключатель **Active**

Начиная с версии OneAgent 1.249, вы можете активировать/деактивировать правила с помощью переключателя **Active**. Для эффективного управления правилами мы рекомендуем обновить OneAgent до версии 1.249. Если у вас есть правила, установленные на хосте с версией OneAgent ниже 249, вы не сможете их деактивировать — в этом случае необходимо удалить такие правила, выбрав **Delete** на уровне правила или через REST API.

Правила выполняются в том порядке, в котором они отображаются на странице **Timestamp/Splitting patterns**.

### Ограничения конфигурации

Вы можете добавить максимум 100 правил меток времени для каждой области действия (хост, группа хостов, кластер Kubernetes или среда).

## REST API

Вы можете использовать Settings API для управления конфигурацией меток времени:

* Просмотр схемы
* Список сохранённых объектов конфигурации
* Просмотр отдельного объекта конфигурации
* Создание нового, редактирование или удаление существующего объекта конфигурации

Чтобы проверить текущую версию схемы для конфигурации меток времени, выведите список всех доступных схем и найдите идентификатор схемы `builtin:logmonitoring.timestamp-configuration`.

Объекты конфигурации меток времени доступны для настройки в следующих областях:

* `environment` — объект конфигурации влияет на все хосты в данной среде.
* `host_group` — объект конфигурации влияет на все хосты, назначенные данной группе хостов.
* `host` — объект конфигурации влияет только на данный хост.

Чтобы создать конфигурацию метки времени с помощью API:

1. [Создайте токен доступа](../../../../dynatrace-api/basics/dynatrace-api-authentication.md#create-token "Узнайте, как пройти аутентификацию для использования Dynatrace API.") с разрешениями **Write settings** (`settings.write`) и **Read settings** (`settings.read`).
2. Используйте эндпоинт [GET a schema](../../../../dynatrace-api/environment-api/settings/schemas/get-schema.md "Просмотр схемы настроек через Dynatrace API."), чтобы узнать формат JSON, необходимый для отправки конфигурации. Идентификатор схемы конфигурации меток времени (`schemaId`) — `builtin:logmonitoring.timestamp-configuration`. Пример JSON-полезной нагрузки с конфигурацией метки времени:

   ```
   [
   {
   "insertAfter":"uAAZ0ZW5hbnQABnRlbmFudAAkMGUzYmY2ZmYtMDc2ZC0zNzFmLhXaq0",
   "schemaId": "builtin:logmonitoring.timestamp-configuration",
   "schemaVersion": "0.1.0",
   "scope": "tenant",
   "value": {
   "config-item-title": "Added from REST API",
   "date-time-pattern": "%Y-%m-%d %H:%M:%S",
   "timezone": "CET",
   "matchers": [
   {
   "attribute": "dt.entity.process_group",
   "operator": "MATCHES",
   "values": [
   ```
