# Правила приёма журналов

# Правила приёма журналов

* Последняя версия Dynatrace
* Руководство
* Чтение: 19 мин
* Обновлено 07 июл 2025

Конфигурация приёма журналов Dynatrace позволяет удалённо настраивать установленные OneAgents для включения конкретных источников журналов при пересылке в Dynatrace или их исключения из загрузки. В то время как обнаружение журналов относится к [автоматическому определению](lma-autodiscovery.md "Dynatrace автоматически обнаруживает все новые файлы журналов, отвечающие определённым требованиям.") файлов журналов (что не требует дополнительных усилий по настройке источников журналов в вашей среде), приём журналов включает процесс сбора журналов и отправки необходимых источников журналов в Dynatrace.

Конфигурация приёма журналов основана на правилах, использующих матчеры для привязки к группам процессов, содержимому, уровням журналов, путям журналов и другим атрибутам, описанным в этом документе. Эти правила определяют, какие файлы журналов принимаются среди автоматически обнаруженных OneAgent или определённых как [пользовательские источники журналов](lma-custom-log-source.md "Настройте пользовательские источники журналов для ручного добавления источников данных журналов, которые не были обнаружены автоматически."). Правила приёма журналов — это упорядоченные конфигурации, обрабатываемые сверху вниз. Для более детальной настройки правила приёма журналов можно определить на трёх уровнях: хоста, группы хостов и среды, при этом правила уровня хоста имеют наивысший приоритет.

Правила приёма журналов основаны на фреймворке [Settings 2.0](../../../../manage/settings/settings-20.md "Введение в фреймворк Settings 2.0"), который предоставляет унифицированный инструмент для управления различными конфигурациями в Dynatrace через пользовательский интерфейс и API.
Доступ к настройкам контролируется через [политики IAM](../../../../manage/identity-access-management/permission-management/manage-user-permissions-policies.md "Работа с политиками"). Чтобы узнать, как настроить политики доступа для Settings 2.0, ознакомьтесь с задокументированными [примерами политик](../../../../manage/settings/settings-20.md#permissions-and-access "Введение в фреймворк Settings 2.0").

Для приёма журналов Kubernetes следуйте конфигурации, описанной на странице [Передача журналов Kubernetes с помощью Dynatrace Log Module](lma-logs-from-kubernetes.md "Dynatrace поддерживает сбор данных журналов из систем оркестрации контейнеров Kubernetes через OneAgent Log Module или Kubernetes Log Module.").

## Правило приёма журналов

При настройке правил приёма журналов в Dynatrace учтите, что в тестовой среде по умолчанию включены встроенные правила. Для начала вы можете использовать правило **Ingest all logs** для начала сбора данных журналов в вашей среде.

Выполните следующие шаги для настройки правил приёма журналов:

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Выберите **Add rule** и укажите имя для вашей конфигурации.
   По умолчанию кнопка **Include in storage** включена, что означает, что источники журналов, настроенные этим правилом, будут пересылаться в Dynatrace. Альтернативно вы можете выбрать тип правила **Exclude from storage**.
3. Выберите **Add condition** и укажите **Matcher attribute** для создания конкретного совпадения для этого правила.
   В одно правило можно включить несколько матчеров.

   В отличие от других атрибутов, атрибут **Log source** в Windows (из-за нечувствительности путей к файлам к регистру) является нечувствительным к регистру; остальные матчеры чувствительны к регистру.
4. Выберите атрибут сопоставления:

1

OneAgent версии 1.295+

2

Атрибут уровня записи журнала, преобразованный OneAgent, отличается от атрибута `status` журнала, преобразованного сервером Dynatrace.

3

OneAgent версии 1.273+

4

OneAgent версии 1.309+

5

[Теги, применённые вручную или автоматически](../../../../manage/tags-and-metadata/setup/how-to-define-tags.md#automatic "Узнайте, как определять и применять теги вручную и автоматически."), не видны OneAgent.

6

OneAgent версии 1.289+

7

Dynatrace Operator версии 1.4.2+

8

OneAgent версии 1.305+

Символ подстановки поддерживается для любого значения атрибута и может использоваться несколько раз в одном значении. Однако некоторые атрибуты, например Process Group, имеют ограниченный предопределённый список возможных значений, выбираемых из списка автодополнения.

Если символ подстановки не используется в значении, матчер ищет точное соответствие значению. Если символ подстановки используется, матчер ищет точное совпадение. Например, значение `INFO` приводит к отправке только данных журнала с точной строкой `INFO`, а значение `*INFO*` (с символами подстановки) соответствует данным журнала, содержащим строку `INFO` в своём содержимом.

1. Выберите **Add value** и из раздела **Values** выберите обнаруженные элементы данных журнала (файлы журналов или группы процессов, содержащие данные журнала). К выбранному атрибуту можно добавить несколько значений. Вы можете создать один матчер, указывающий источник журнала, который соответствует значениям **/var/log/syslog** и **Windows Application Log**.
2. **Сохраните изменения**.

Определённые правила можно переупорядочивать; они выполняются в том порядке, в котором появляются на странице **Log storage**.

7. Чтобы активировать правило, включите переключатель **Active**.

## Сопоставление списка правил с данными журнала

Сопоставление происходит в предопределённой иерархии, и правила выполняются сверху вниз. Это означает, что если правило выше в списке соответствует определённым данным журнала, нижние правила будут пропущены. Элементы, сопоставленные в конфигурациях более высокого уровня, перезаписываются в конфигурациях более низкого уровня, если они соответствуют тем же данным журнала. Если ни одно правило не совпало, файл не отправляется. Иерархия сопоставления выглядит следующим образом:

1. Правила конфигурации хоста
2. Правила конфигурации кластера Kubernetes
3. Правила конфигурации группы хостов
4. Правила конфигурации среды

## Области конфигурации

Поддерживаются четыре иерархических области: хост, кластер Kubernetes, группа хостов и среда. Область с наименьшим возможным набором правил имеет приоритет над большими наборами.

![log-storage](https://dt-cdn.net/images/log-storage-1280-2db879c27f.png)

1. Правила приёма журналов, настроенные для хоста, имеют приоритет над правилами, настроенными для группы хостов.
2. Правила приёма журналов, настроенные для группы хостов, имеют приоритет над правилами, настроенными для среды.

### Область хоста

Область хоста доступна через **Host settings** для конкретного хоста.

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic**.
2. Найдите и выберите ваш хост для отображения страницы обзора хоста.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**…**) > **Settings**.

4. В настройках хоста перейдите в **Log Monitoring** > **Log ingest rules**.
5. Настройте загрузку в хранилище, добавив правила с набором атрибутов, соответствующих данным журнала, которые должны храниться в Dynatrace.

### Область кластера Kubernetes

Область кластера Kubernetes доступна через страницу **Kubernetes**.

1. Перейдите в **Kubernetes** или **Kubernetes Classic** (последняя версия Dynatrace) и выберите интересующий вас кластер.
2. Найдите и выберите ваш кластер для отображения страницы обзора кластера.
3. В правом верхнем углу страницы обзора кластера выберите **More** (**…**) > **Settings**.
4. В настройках кластера перейдите в **Log Monitoring** > **Log ingest rules**.
5. Настройте загрузку в хранилище, добавив правила с набором атрибутов, соответствующих данным журнала, которые должны храниться в Dynatrace.

### Область группы хостов

Область группы хостов доступна через страницу **Host**.

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic** и выберите интересующий вас хост.
2. На странице обзора хоста выберите **Properties and tags**.
3. На панели **Properties and tags** найдите свойство **Host group**, чтобы увидеть имя группы хостов, к которой принадлежит выбранный хост.

   Свойство **Host group** не отображается, если выбранный хост не принадлежит ни к какой группе хостов.
4. Выберите имя группы хостов для отображения всех хостов в этой группе хостов. Откроется страница **OneAgent deployment**, отфильтрованная по выбранной группе хостов. У каждого перечисленного хоста есть ссылка **Host group:** `<имя группы>`, где `<имя группы>` — имя группы хостов, которую вы хотите настроить.
5. Выберите имя группы хостов в любой строке.

6. В настройках группы хостов выберите **Log Monitoring** > **Log ingest rules**.
7. Настройте загрузку в хранилище, добавив правила с набором атрибутов, соответствующих данным журнала, которые должны храниться в Dynatrace.

### Область среды

Область среды доступна в меню настроек.

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Настройте загрузку в хранилище, добавив правила с набором атрибутов, соответствующих данным журнала, которые должны храниться в Dynatrace.

### Список хостов и групп хостов с переопределяющими правилами

Таблица на **Settings** > **Log Monitoring** > **Log ingest rules** перечисляет все правила хранения журналов, установленные на уровне среды. Однако вы можете захотеть увидеть, где установлены правила хранения журналов для хостов и групп хостов, переопределяющие правила уровня среды.

Чтобы получить список всех сущностей (хостов и групп хостов), к которым применены более конкретные правила хранения журналов:

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. В правом верхнем углу страницы **Log ingest rules** выберите **More** (**…**) > **Hierarchy and overrides**. Панель **Hierarchy and overrides** с функцией поиска перечисляет все сущности (хосты и группы хостов), для которых установлены правила хранения журналов, переопределяющие правила уровня среды, перечисленные на **Settings** > **Log Monitoring** > **Log ingest rules**.
3. Выберите имя сущности, чтобы перейти на страницу **Log ingest rules** этой сущности.

### Ограничения конфигурации

Вы можете добавить максимум 1000 правил приёма на область с максимум 2000 матчерами.

## Пример загрузки

В этом примере мы настраиваем загрузку хранилища среды для файлов `c:\inetpub\logs\LogFiles\ex_*.log` в двух группах процессов: `IIS (PROCESS_GROUP-3D9D854163F8F07A)` и `IIS (PROCESS_GROUP-4A7B47FDB53137AE)`. Правило хранения журналов состоит из двух матчеров: первый матчер находит группы процессов, а второй сопоставляет только определённый источник журнала.

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Выберите **Add rule** и укажите заголовок для вашей конфигурации.
3. Выберите **Add matcher**. Это первый матчер для сопоставления двух указанных групп процессов.
4. Из списка **Attribute** выберите **Process group**.
5. Выберите **Add value**, введите IIS и затем из списка предложений выберите `IIS (PROCESS_GROUP-3D9D854163F8F07A)`.
6. Снова выберите **Add value**, введите `IIS` и выберите вторую группу процессов из списка предложений: `IIS (PROCESS_GROUP-4A7B47FDB53137AE)`.
7. Снова выберите **Add matcher**. Это второй матчер для сопоставления указанного источника данных журнала.
8. Из списка **Attribute** выберите **Log source**.
9. Выберите **Add value** и введите `c:\inetpub\logs\LogFiles\ex_*.log` в качестве значения.
10. Сохраните изменения.

## Пример исключения

В этом примере мы настраиваем загрузку хранилища среды для всех источников журналов, кроме файлов `c:\inetpub\logs\LogFiles\ex_*.log` в группе процессов `IIS (PROCESS_GROUP-4A7B47FDB53137AE)`.

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Выберите **Add rule** и укажите заголовок для вашей конфигурации.
3. Отключите **Send to storage**.
4. Выберите **Add matcher**. Это первый матчер для сопоставления указанной группы процессов.
5. Из списка **Attribute** выберите `Process group`.
6. Выберите **Add value**, введите IIS и затем из списка предложений выберите `IIS (PROCESS_GROUP-3D9D854163F8F07A)`.
7. Снова выберите **Add matcher**. Это второй матчер для исключения указанного источника данных журнала.
8. Из списка **Attribute** выберите **Log source**.
9. Выберите **Add value** и введите `c:\inetpub\logs\LogFiles\ex_*.log` в качестве значения.
10. Сохраните изменения.

## REST API

Вы можете использовать Settings API для управления правилами приёма журналов:

* Просмотр схемы
* Список сохранённых объектов конфигурации
* Просмотр одного объекта конфигурации
* Создание нового, редактирование или удаление существующего объекта конфигурации

Чтобы проверить текущую версию схемы для правил приёма журналов, перечислите все доступные схемы и найдите идентификатор схемы `builtin:logmonitoring.log-storage-settings`.

Правила приёма журналов можно настраивать для следующих областей:

* `tenant` — объект конфигурации влияет на все хосты в данной среде.
* `host_group` — объект конфигурации влияет на все хосты, назначенные данной группе хостов.
* `host` — объект конфигурации влияет только на данный хост.

Чтобы создать правило приёма журналов с помощью API:

1. [Создайте токен доступа](../../../../dynatrace-api/basics/dynatrace-api-authentication.md#create-token "Узнайте, как пройти аутентификацию для использования Dynatrace API.") с разрешениями **Write settings** (`settings.write`) и **Read settings** (`settings.read`).
2. Используйте конечную точку [GET a schema](../../../../dynatrace-api/environment-api/settings/schemas/get-schema.md "Просмотр схемы настроек через Dynatrace API."), чтобы узнать формат JSON, необходимый для публикации конфигурации. Идентификатор схемы правил приёма журналов (`schemaId`) — `builtin:logmonitoring.log-storage-settings`. Пример полезной нагрузки JSON с правилами приёма журналов:

   ```
   [



   {



   "insertAfter":"uAAZ0ZW5hbnQABnRlbmFudAAkMGUzYmY2ZmYtMDc2ZC0zNzFmLhXaq0",



   "schemaId": "builtin:logmonitoring.log-storage-settings",



   "schemaVersion": "0.1.0",



   "scope": "tenant",



   "value": {



   "config-item-title": "Added from REST API",



   "send-to-storage": true,



   "matchers": [



   {



   "attribute": "dt.entity.process_group",



   "operator": "MATCHES",



   "values": [



   "PROCESS_GROUP-05F00CBACF39EBD1"



   ]



   },



   {



   "attribute": "log.source",



   "operator": "MATCHES",



   "values": [



   "Windows System Log",



   "Windows Security Log"



   ]



   }



   ]



   }



   }



   ]
   ```
3. Используйте конечную точку [POST an object](../../../../dynatrace-api/environment-api/settings/objects/post-object.md "Создание или проверка объекта настроек через Dynatrace API."), чтобы отправить конфигурацию.

## Примеры

Следующие примеры показывают результаты различных комбинаций правил и матчеров.

### Пример 1: Несколько правил

В этом примере есть два правила:

* Правило 1 является правилом исключения и содержит два матчера: атрибут группы процессов — Apache, атрибут источника журнала — `access.log`.
* Правило 2 является правилом включения и содержит один матчер: атрибут группы процессов — Apache.

Результаты: `access.log` не отправляется, `error.log` (Apache) отправляется, а `error.log` (другой PG) не отправляется.

* `access.log`, записанный Apache, соответствует первому правилу с `send-to-storage: false`, поэтому не отправляется.
* `access.log`, не записанный Apache, не соответствует первому правилу (из-за неверной группы процессов) и не соответствует второму правилу, поэтому не отправляется.
* `error.log`, записанный Apache, не соответствует первому правилу (из-за неверного источника), но соответствует второму правилу с `send-to-storage: true`, поэтому отправляется.
* `error.log`, не записанный Apache, не соответствует ни первому правилу (из-за неверной группы процессов и источника журнала), ни второму, поэтому не отправляется.

```
{



"send-to-storage": false,



"matchers": [



{



"attribute": "log.source",



"values": [



"/path/to/access.log"



]



},



{



"attribute": "dt.entity.process_group",



"values": [



"PROCESS_GROUP-APACHEID"



]



}



],



"enabled": true



},



{



"send-to-storage": true,



"matchers": [



{



"attribute": "dt.entity.process_group",



"values": [



"PROCESS_GROUP-APACHEID"



]



}



],



"enabled": true



}
```

## Часто задаваемые вопросы

Является ли конфигурация правил приёма журналов тем же самым или частью процесса автообнаружения?

Нет. Автообнаружение — это механизм OneAgent, который обнаруживает журналы, но это не означает, что файлы журналов автоматически принимаются. Оно относится только к автоматическому определению данных журналов. Подробнее об автообнаружении см. в разделе [Автообнаружение содержимого журналов (Logs Classic)](../../../log-monitoring/acquire-log-data/log-content-auto-discovery-v2.md "Узнайте об автообнаружении содержимого журналов и требованиях для его выполнения.")

Важен ли порядок элементов конфигурации?

Да, элементы конфигурации сопоставляются сверху вниз, то есть верхнее значение является наиболее важным.

Как долго нужно ждать, пока конфигурация применится к хосту?

Она применяется в течение 90 секунд.

Почему журналы не принимаются после настройки?

Если ваши журналы не принимаются, это может быть связано с тем, что [OneAgent Log Enablement](../../../../ingest-from/dynatrace-oneagent/oneagent-configuration-via-command-line-interface.md#log-monitoring "Узнайте, как выполнять некоторые задачи настройки OneAgent без необходимости его переустановки.") отключён, или с тем, что приём журналов из источника предотвращён [правилами безопасности OneAgent](lma-custom-log-source.md#security-rules "Настройте пользовательские источники журналов для ручного добавления источников данных журналов, которые не были обнаружены автоматически.").

Уменьшает ли добавление матчера содержимого количество событий журналов, отправляемых в Dynatrace?

Да. Матчер содержимого сужает область событий журналов (записей журналов) в соответствии с установленными критериями (например, приём только журналов ошибок).

Где выполняется фильтрация — в Dynatrace или в OneAgent?

* Фильтрация (ограничение числа принимаемых записей журналов в соответствии с установленными критериями) выполняется в OneAgent.
* Характеристики приёма журналов [ограничения](../../lma-limits.md "Ограничения по умолчанию для последней версии Dynatrace Log Management and Analytics.") (например, лимит событий журналов в минуту или лимит значений атрибутов) применяются в Dynatrace.

Уменьшает ли фильтрация содержимого затраты DDU и/или использование сети?

Да. Фильтрация содержимого, выполняемая в OneAgent, снижает как затраты DDU, так и использование сети. Вы можете рассчитать снижение затрат и использования сети, определив общее потребление данных и вычтя размер в ГБ данных, которые были отфильтрованы. Подробнее о расчёте затрат DDU см.:

* [Расчёт DDU для Log Monitoring](../../../../license/monitoring-consumption-classic/davis-data-units/log-monitoring-consumption.md "Понять, как рассчитывается объём потребления DDU для Dynatrace Log Monitoring Classic.")
* [Расчёт DDU для Log Management and analytics на базе Grail](../../../../license/monitoring-consumption-classic/davis-data-units/log-management-and-analytics.md "Понять, как рассчитывается объём потребления DDU для Dynatrace Log Management and Analytics.")

Будут ли работать старые версии OneAgent с этим решением?

Версии OneAgent ранее `1.243` не будут отправлять никаких данных.
Если вы используете OneAgent версии ранее 1.243 и Dynatrace Cluster версии ранее `1.252`, перейдите в раздел [Log Sources and Storage](lma-add-log-file-sources.md "Узнайте, как включать и исключать источники журналов для анализа.").

Начиная с версии OneAgent `1.249`, вы можете активировать/деактивировать правила, включая/выключая переключатель **Active**. Для эффективного управления правилами рекомендуется обновить OneAgent до версии `1.249`. Если у вас есть правила, установленные на хосте с версией OneAgent ранее 249, вы не сможете их деактивировать; в этом случае вам нужно удалить такие правила, выбрав **Delete** на уровне правила или через REST API.
