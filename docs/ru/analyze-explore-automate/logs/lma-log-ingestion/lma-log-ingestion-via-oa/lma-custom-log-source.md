# Пользовательский источник логов


* Latest Dynatrace
* Учебное пособие
* Чтение: 6 мин
* Обновлено 07 июля 2025 г.

Настройка пользовательского источника логов позволяет вручную добавлять источники логов, которые не были автоматически обнаружены. Если вы хотите их принимать, вам по-прежнему необходимо настроить это с помощью [конфигурации приёма логов](lma-log-storage-configuration.md "Включение и исключение определённых источников логов, уже известных OneAgent, для хранения и анализа."). Случаи, когда необходимо использовать пользовательские источники логов, включают:

* Автоматическое обнаружение может не идентифицировать источник логов, если файл лога не открыт для записи во время работы процесса.
* Оно также может не найти источники логов, которые не являются частью каких-либо процессов или являются частью кратковременных процессов.

Весь процесс состоит из двух частей:

1. Определение источника (настройка пользовательского источника логов), которое описано на этой странице.
2. Получение логов (добавление логов в хранилище), которое описано в [Правила приёма логов](lma-log-storage-configuration.md "Включение и исключение определённых источников логов, уже известных OneAgent, для хранения и анализа.").

Если вам необходимо хранить пользовательские логи, вам нужно выполнить оба шага.

Каждый путь к пользовательскому источнику логов, который вы добавляете, должен быть проверен OneAgent и соответствовать его правилам безопасности. См. [Правила безопасности](lma-security-rules.md "Настройка правил безопасности для пользовательских источников логов для обеспечения защиты данных.") для файлов конфигурации и примеров.

### Хосты

Настройка пользовательских источников логов на уровне хоста

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic** и выберите ваш хост.
2. Выберите **More** (**…**) > **Settings**, чтобы открыть страницу **Host settings** (доступно только на хостах, назначенных группе хостов).
3. На странице **Host settings** выберите **Log Monitoring** > **Custom log sources**.

### Группы хостов

Настройка пользовательских источников логов на уровне группы хостов

1. Перейдите в ![Hosts](https://dt-cdn.net/images/hosts-512-59f5d2dd7f.png "Hosts") **Hosts Classic** и выберите ваш хост.
2. Разверните раздел **Properties and tags** и выберите **Host group** (доступно только на хостах, назначенных группе хостов).
3. На странице **Host group settings** выберите **Log Monitoring** > **Custom log sources**.

### Среда

Настройка пользовательских источников логов на уровне среды

1. Перейдите в **Settings**.
2. Выберите **Log Monitoring** > **Custom log sources**.

## Настройка пользовательского источника логов

1. Перейдите на страницу **Custom log sources** на уровне хоста, группы хостов или среды, как описано выше.
2. Выберите **Add custom log source** и добавьте **Rule name**.
3. Необязательно: Привяжите ваше правило к группе процессов, выбрав имя группы процессов из выпадающего меню.

Любые правила, привязанные к группам процессов, которые отсутствуют на данном хосте, будут проигнорированы для этого хоста.

4. В разделе **Custom log source paths** выберите **Log source type**. Доступны два типа источников:

   * **Log path**
   * **Windows Event Log**
5. Выберите **Allow binary content**, чтобы разрешить приём содержимого из файлов, содержащих бинарные данные. Нетекстовые данные будут экранированы. Это полезно в случаях, когда файл, состоящий в основном из текстовых записей логов, также содержит некоторые бинарные данные.
6. Необязательно: Выберите метод кодирования мониторируемого файла из выпадающего списка **Encoding**.

Иногда OneAgent не может определить метод кодирования мониторируемого файла. Это может быть связано с отсутствием маркера BOM, определяющего метод кодирования, или с использованием определённой кодировки для диакритических символов. В настоящее время поддерживаются кодировки UTF-8, UTF-16 Little и Big Endian, Shift\_JIS, EUC\_JP (японский язык) и EUC\_KR (корейский язык).

7. Определите полный путь к файлу лога, выбрав **Add custom log source path**, введите полный путь (например, `/var/lib/*.log` или `/var/log/sys.bin`) и выберите **Add Path**. Вы можете добавить до 100 путей к логам для одного правила пользовательского источника логов. Вы можете указать пользовательские журналы событий Windows.
   При настройке пользовательского источника логов следуйте этим правилам:

   Если вы выбрали Windows Event Log, добавьте полное имя (Full Name).

   Как показано в следующем примере, вы можете отобразить имя лога, щёлкнув правой кнопкой мыши по выбранному журналу событий и выбрав **Properties**:

   ![Экран свойств средства просмотра событий Windows.](https://dt-cdn.net/images/windows-event-viewer-properties1-547-41612bfc3c.png)

   ![Свойства лога средства просмотра Windows.](https://dt-cdn.net/images/windows-event-viewer-properties2-1261-03ad4741e4.png)
8. Необязательно: Выберите **Show advanced** для раскрытия панели, позволяющей определить список атрибутов, которые обогатят каждую запись лога из определённых источников логов. Введите ключ атрибута или выберите его из списка, затем введите значение атрибута. Это доступно только если вы выбрали **Log source type** как Log Path.

   Обогащение логов пользовательскими атрибутами.

   При использовании подстановочных символов в пути к логу вам может потребоваться различать пути, сопоставленные подстановочными символами. В таких случаях вы можете определить атрибуты, использующие полный путь к файлу или часть пути, сопоставленную подстановочными символами.

   Чтобы определить такой атрибут, выполните следующие шаги:

   1. Введите ключ или выберите его из списка.
   2. В поле Attribute value используйте токен `${N}`, где `N` обозначает индекс подстановочного символа, на который вы ссылаетесь, начиная с 1. `${0}` имеет специальное значение и раскрывается в полный путь к файлу лога.

   Вы можете использовать несколько токенов `${N}` в одном атрибуте и комбинировать их с другими символами. Например, `worker:${1}-${2}`.

   Если токен `${N}` ссылается на индекс подстановочного символа, превышающий количество подстановочных символов в пути к логу, он не будет заменён, и `${N}` останется в значении атрибута. Ключ атрибута должен содержать только латинские буквенно-цифровые символы (верхнего или нижнего регистра), точки (`.`), подчёркивания (`_`), дефисы (`-`) или двоеточия (`:`). Он не должен начинаться с префикса `dt.` и не должен быть одним из следующих:

   ```
   process.technology, log.source, log.content, timestamp, container.name, winlog.level, winlog.eventid, winlog.provider, winlog.opcode, winlog.task, winlog.keywords, winlog.username, k8s.namespace.name, k8s.container.name, k8s.deployment.name
   ```
9. Выберите **Save changes**.
10. Включите переключатель **Active** для активации вашего правила.

## Требования к разрешениям файлов логов

Для того чтобы OneAgent мог принимать файл лога, ему необходимы разрешения на чтение этого файла.

Способ предоставления этих разрешений зависит от ОС, на которой развёрнут OneAgent.

OneAgent поддерживает три операционные системы:

* Linux
* Windows
* AIX

Краткий обзор случаев, когда необходимо активно предоставлять разрешения:

| Диски | Linux | Windows | AIX |
| --- | --- | --- | --- |
| Локальные диски | Действия не требуются | Может потребоваться действие | Действия не требуются |
| Удалённые диски | Может потребоваться действие | Может потребоваться действие | Действия не требуются |

### Linux

* Файлы логов на локальных дисках: действия не требуются.
  По умолчанию OneAgent имеет соответствующие [возможности Linux](https://man7.org/linux/man-pages/man7/capabilities.7.html) для чтения этих файлов.
* Файлы логов на удалённом диске: необходимо предоставить следующие разрешения для учётной записи, под которой работает OneAgent (по умолчанию это учётная запись `dtuser`):

  + Разрешения на чтение и выполнение `(r+x)` для каждого каталога в пути к файлу лога.
  + Разрешение на чтение `(r)` для самого файла лога — для учётной записи, под которой работает OneAgent (по умолчанию `dtuser`).

    Файлы логов на удалённом диске

    Отсутствие прав доступа — одна из наиболее типичных причин отсутствия приёма логов для удалённых дисков на Linux.

    Если у вас есть доступ к мониторируемому хосту, вы можете быстро проверить, является ли отсутствие прав доступа причиной проблемы.
    Для этого выполните команду `sudo -u dtuser cat /path/to/file.log` с правильным полным путём, при условии, что `dtuser` — это учётная запись, под которой работает OneAgent.

### Windows

OneAgent работает под учётной записью `LocalSystem`.

Следующие разрешения обычно предоставляются автоматически для файлов логов на локальных дисках:

* Разрешение на чтение для каждого файла лога.
* Разрешения на чтение и выполнение для каталогов в пути для этой учётной записи.

Для удалённых дисков они могут быть не установлены. Вам необходимо проверить, установлены ли они, и предоставить разрешения.
Для получения дополнительной информации см. [https://www.ntfs.com/ntfs-permissions-setting.htm](https://www.ntfs.com/ntfs-permissions-setting.htm).

OneAgent не может получить доступ к файлам, требующим авторизации через дополнительные диалоговые окна пользовательского интерфейса, например, всплывающее окно Windows Explorer.

### AIX

По умолчанию OneAgent имеет соответствующие разрешения root для чтения этих файлов.

### Ограничения конфигурации

Вы можете добавить максимум 1000 правил пользовательских источников логов на область, с максимумом 100 путей для каждого.

## Сопоставление файлов логов

При настройке пользовательского источника логов следуйте этим правилам:

* Пользовательские пути к логам должны быть абсолютными; относительные пути отклоняются. Абсолютный путь имеет следующий формат:

  + Для Windows: `любая буква:\`
  + Для Linux: Начинается с `/`
  + Для NFS: Начинается с `//hostname/`
* Путь Windows Event Log в Windows Event System должен быть относительным.
* Пользовательские источники логов могут содержать подстановочные символы: `#` заменяет строку цифр, а `*` заменяет строку любых символов, кроме прямой косой черты (`/`) или обратной косой черты (`\`). В то время как `*` можно использовать как в именах файлов, так и в каталогах, `#` можно использовать только в именах файлов.

Нет необходимости использовать последовательность подстановочных символов вместо одного. Например:

* Можно заменить `/path/to/test###.log` на `/path/to/test#.log`
* Можно заменить `/path/to/test#*.log` на `/path/to/test*.log`
* Можно заменить `/path/to/test#-#-#-#.log` на `/path/to/test*.log`, если только вы не хотите обогатить логи пользовательскими атрибутами, используя числа, разделённые дефисами, по отдельности.

Использование подстановочных символов в каталогах может снизить производительность OneAgent.

Если вы выбрали Windows Event Log, добавьте полное имя (Full Name).

Как показано в следующем примере, вы можете отобразить имя лога, щёлкнув правой кнопкой мыши по выбранному журналу событий и выбрав **Properties**:

## Поддерживаемые области

Поддерживаются три уровня иерархии: хост, группа хостов и среда. Чем уже область, тем выше её приоритет.

* Правила источников логов, настроенные для хоста, имеют приоритет над правилами источников логов, настроенными для группы хостов.
* Правила источников логов, настроенные для группы хостов, имеют приоритет над правилами источников логов, настроенными для среды Dynatrace.
