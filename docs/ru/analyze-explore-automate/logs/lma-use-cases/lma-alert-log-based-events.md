# Настройка оповещений на основе событий, извлечённых из логов


Принятые логи могут служить триггерами для открытия новых проблем Davis.

Используя события Davis на основе логов, вы получите немедленные оповещения, как только будет принята определённая вами запись лога.

Следуйте этому руководству, чтобы узнать больше об извлечении событий из логов.

Если вам нужно задать пороговые значения для оповещений, следуйте инструкциям в разделе Настройка пользовательских оповещений на основе метрик, извлечённых из логов.

## Предварительные требования

Необязательно

* Вы настроили приём логов.
* У вас есть необходимые разрешения для настройки OpenPipeline. Например, разрешения, предоставленные политикой по умолчанию: Data Processing and Storage.

## Шаги

В этом примере мы откроем новую проблему Davis, когда будут приняты определённые записи, содержащие конкретную фразу.

1. Найдите логи, для которых вы хотите настроить оповещения

Вы можете найти оповещения, открыв ![Logs](https://dt-cdn.net/images/logs-256-ae0a9ca67f.png "Logs") **Logs** и используя следующий запрос DQL.

```
fetch logs


| filter matchesPhrase(content, "Dropping data because sending_queue is full")


| sort timestamp desc
```

![Результаты логов](https://dt-cdn.net/images/logs-1228-18a19d138d.png)

Если ваш запрос DQL использует `parse`, `fieldAdd` или другие преобразования, вам следует добавить правило обработки для установки этих полей при приёме.

2. Извлечение события Davis в OpenPipeline

1. Добавьте конфигурацию извлечения данных событий Davis в OpenPipeline.

   1. Откройте ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline** > **Logs** и выберите вкладку **Pipelines**.
   2. Найдите конвейер, который хотите изменить, или добавьте новый.
   3. Выберите > **Edit**.
      Откроется страница конфигурации конвейера.
   4. Выберите вкладку **Data extraction** и добавьте процессор **Davis event**.
2. Задайте DQL-фильтр.
   Фильтр задаёт условие для извлекаемого события.
   Это подмножество условий фильтрации в одном операторе DQL.

   В поле **Matching condition** используйте фильтр, как показано ниже.

   ```
   matchesPhrase(content, "Dropping data because sending_queue is full")
   ```

   Если вы используете сегменты или ваши разрешения установлены на уровне записи, вы должны включить эти условия в фильтр.

   Бывают ситуации, когда фильтр невозможно легко извлечь из оператора DQL.
   В таких случаях вы можете создать оповещения для событий логов или сводных данных логов.
3. Задайте свойства события.

   Свойства события — это метаданные, которые будут содержаться в вашем событии при его срабатывании.
   Вы можете переназначить любое поле из записи лога.

   В нашем примере мы переназначим поле `dt.source_entity`, чтобы оповещения были связаны с сущностями для анализа первопричин Dynatrace Intelligence.

   В разделе **Event template** задайте следующие пары ключ/значение.

   * Задайте `event.type` как `CUSTOM_ALERT`.
   * Задайте `event.description` как `Dropping data because sending_queue is full. Try increasing queue_size.`.
   * Задайте `event.name` как `OpenTelemetry exporter failure`.
   * Задайте `dt.source_entity` как `{dt.source_entity}`.

   ![Извлечение данных конвейера](https://dt-cdn.net/images/pipeline-config-1190-5bd3924a5d.png)

3. Открытие проблемы в Problems

Когда будет извлечено первое событие Davis, откроется новая проблема.
Если в течение периода ожидания, определённого в `dt.davis.event_timeout`, новые события не поступят, проблема будет автоматически закрыта.

Период ожидания по умолчанию составляет 15 минут.

![Проблема в приложении Problems](https://dt-cdn.net/images/problems-app-1048-6fd9c7e554.png)

## Заключение

Извлечение событий Davis из логов идеально подходит для простого оповещения, когда пороговые значения не важны.

* Обеспечивает немедленное/реальное оповещение.
* Дополнительный обзор совпадающих данных с течением времени не требуется.

После настройки извлечения событий вы можете использовать их для запуска автоматизаций с помощью простых рабочих процессов, как описано в разделе Создание простого рабочего процесса в Dynatrace Workflows.
