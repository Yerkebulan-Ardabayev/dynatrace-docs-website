# Обнаружение проблем с помощью логов

* Последняя версия Dynatrace
* Руководство
* Время чтения: 8 мин
* Опубликовано 8 октября 2024 г.

Быстрое обнаружение и решение проблем в вашей среде имеет решающее значение для поддержания стабильного дохода и обеспечения доверия клиентов. Однако ручной анализ устаревших приложений или сторонних приложений, к исходному коду которых у вас нет доступа, может занимать много времени.

Решение проблемы с помощью Dynatrace значительно ускоряет среднее время идентификации (MTTI) критических проблем и увеличивает скорость их устранения до того, как они повлияют на качество обслуживания клиентов, тем самым минимизируя влияние сбоев на ваш бизнес. Благодаря единой платформе наблюдаемости для всех сигналов вы снижаете риск человеческих ошибок при ручной корреляции деталей проблемы.

Использование Dynatrace позволяет избежать просмотра всех существующих записей, показывая вам только строки логов, непосредственно связанные с обнаруженной проблемой. Этот метод также позволяет быстро проверить детали ошибки, такие как сообщение, статус и строку кода (LOC), где произошла ошибка.

![Описание варианта использования обнаружения проблем с помощью логов](https://dt-cdn.net/images/obslab-logalerting-animated-c4b8056092.gif)

## Что вы узнаете

Это руководство проведёт вас через процесс извлечения релевантной информации из логов через OpenPipeline и доступа к логам через приложение Problems. Оно покажет, как использовать запросы DQL для поиска информации, относящейся к вашей проблеме, и получения более глубокого контекстного представления о проблеме с помощью трассировок.

К концу вы будете знать, как:

* Обогащать логи дополнительным контекстом при их сборе через OneAgent или OpenTelemetry
* Настроить OpenPipeline для извлечения релевантной информации из логов и преобразования её в событие
* Использовать приложение Problems для доступа к соответствующим строкам логов
* Найти первопричину с помощью логов

Пример, использованный в этом руководстве, взят из [Dynatrace Observability Lab: Problem Detection with Logs](https://dt-url.net/0jdt0unq). Для полноценного опыта вы можете следовать этой практической демонстрации. В ней объясняется процесс создания проблемы и настройки тестовой среды.

### Шаг 1 Создание нового конвейера для извлечения данных

OpenPipeline — это решение Dynatrace для обработки и приёма данных. Вы можете настроить OpenPipeline для извлечения определённой информации, релевантной для вашего случая, и преобразования её в событие, по которому можно настроить оповещение. Дополнительные сведения о возможностях обработки OpenPipeline см. в разделе [Обработка в OpenPipeline](../../../platform/openpipeline/concepts/processing.md "Изучите основные концепции обработки Dynatrace OpenPipeline.").

Чтобы создать новый конвейер

1. В **Dynatrace** выберите  или нажмите `Ctrl+K`, чтобы найти и выбрать **OpenPipeline**.
2. Перейдите в **Logs**, выберите вкладку **Pipelines** и выберите  **Pipeline**.
3. Выберите  и измените имя конвейера на `Log Errors`.
4. Выберите  для сохранения изменений.
5. На вкладке **Data extraction** выберите  **Processor** и выберите **Davis event**. Это создаст новый процессор.
6. Заполните обязательные поля. Результат должен быть похож на пример, следующий за этой процедурой.

   * Установите **Name** на любое имя по вашему выбору
   * **Matching condition** должно быть установлено в `true`
   * Установите **Event name** следующим образом:

     ```
     [{priority}][{deployment.release_stage}][{deployment.release_product}][{dt.owner}] {alertmessage}
     ```
   * Установите **Event description** в `{supportInfo} - Log line: {content}`
7. Установите следующие **Event properties**:
8. Выберите **Save** для сохранения конвейера.

### Шаг 2 Добавление динамического маршрута для конвейера

Принятые и извлечённые данные необходимо направить в конвейер перед обработкой. Создание маршрута необходимо для того, чтобы ваши данные направлялись в правильный конвейер, особенно в случаях, когда у вас несколько конвейеров. Дополнительные сведения см. в разделе [Маршрутизация](../../../platform/openpipeline/concepts/data-flow.md#routing "Узнайте, как данные передаются в Dynatrace Platform, от приёма до хранения, через Dynatrace OpenPipeline.").

Чтобы добавить новую динамическую маршрутизацию

1. Перейдите в **Logs**, выберите вкладку **Dynamic routing** и выберите  **Dynamic route**.
2. Заполните обязательные поля:

   * Установите **Name** на любое имя по вашему выбору
   * **Matching condition** должно быть установлено в

     ```
     isNotNull(alertmessage) and



     isNotNull(priority) and



     priority == "1"
     ```
3. Выберите **Add** для создания нового динамического маршрута.

Пример создания нового динамического маршрута для конвейера Log Errors

### Шаг 3 Доступ к проблемам через приложение Problems

После обнаружения и записи проблемы в логах вы можете проверить её статус в приложении Problems.

Приложение Problems — это инструмент, предназначенный для помощи операционным командам и командам по обеспечению надёжности сайтов в сокращении среднего времени восстановления (MTTR) путём представления всех аспектов инцидента. Дополнительные сведения см. в разделе [Приложение Problems](../../../dynatrace-intelligence/davis-problems-app.md "Используйте приложение Problems для быстрого выявления первопричины инцидентов в вашей среде.").

Чтобы открыть приложение Problems

1. В **Dynatrace** выберите  или нажмите `Ctrl+K`, чтобы найти и выбрать приложение **Problems**.
2. Выберите идентификатор открытой проблемы, чтобы увидеть запись. Открытые проблемы отображаются со статусом **Status** `Active`.

![Список ошибок в приложении Problems](https://dt-cdn.net/images/problems-app-errors-1918-89ccc3de72.png)

### Шаг 4 Просмотр логов через записи проблем

Запись проблемы показывает количество событий, SLO, затронутых пользователей и затронутых сущностей. По умолчанию запись отображает затронутое развёртывание и график, иллюстрирующий проблему. Вы можете переключаться между **Chart** и **Properties**, а также отображать **Deployment**, **Events** или **Logs**, связанные с проблемой.

![Просмотр записи проблемы](https://dt-cdn.net/images/problems-app-record-1912-8f5daeb578.png)

1. Выберите **Logs** для доступа к логам из записи проблемы. На вкладке **Logs** вы увидите график принятых записей и список рекомендуемых запросов, которые помогут вам быстрее проанализировать проблему.
2. Выберите **Run query** для  `Show x errors`, где `x` — количество ошибок, записанных для вашей проблемы.
3. Выберите запись лога, которую вы хотите развернуть. Развёрнутая запись предоставляет полезные метаданные, такие как

   * Временная метка строки лога
   * `host.name`, соответствующее имени контейнера
   * `loglevel` (например, `ERROR`)
   * `span_id` и `trace_id` OpenTelemetry или OneAgent
   * `dt.owner` — владелец компонента
   * `dt.cost.product` и `dt.cost.costcenter`, соответствующие информации о стоимости

Пример развёрнутой записи лога

![Развёрнутая запись лога с метаданными](https://dt-cdn.net/images/problems-app-show-errors-1646-e64b66e6ee.png)

4. Выберите **Show surrounding logs** для просмотра логов, связанных с проблемой.

Пример просмотра окружающих логов

![Окружающие логи на основе трассировок, показывающие события, связанные с ошибкой с тем же trace_id](https://dt-cdn.net/images/surrounding-logs-traces-1834-bfc676163f.png)

5. Выберите фильтр для окружающих логов:

   * **based on trace** (по умолчанию): отображение всех логов с тем же `trace_id`.
   * **based on topology**: показ ошибки в контексте всех логов для сбойного сервиса на момент ошибки.

Логи, связанные с ошибкой, такие как информационные события, могут содержать дополнительную информацию, которая поможет вам в определении первопричины. Примеры дополнительной информации включают:

* `statuscode` содержит код статуса ошибки. Например, `FailedPrecondition`.
* `detail` содержит сообщение об ошибке и строку кода, где произошла ошибка.

Пример дополнительной информации связанного события

![Связанный информационный лог с дополнительной информацией statuscode и detail](https://dt-cdn.net/images/log-problem-debugging-with-logs-1802-ac039df62f.png)

### Шаг 5 Просмотр деталей в распределённых трассировках

Трассировки предоставляют более глубокое представление и дополнительный контекст для информации, доступной в логах. Чтобы иметь возможность доступа к трассировкам через логи, необходимо связать данные логов с трассировками через OpenTelemetry или OneAgent. Подробнее об обогащении логов трассировками см. в разделе [Понимание и исправление нескольких проблем с помощью логов и трассировок](../../../observe/application-observability/distributed-traces/use-cases/problems-logs-traces.md "Используйте обогащение логов для просмотра связанных записей логов в представлении распределённых трассировок и улучшения возможностей анализа.").

Чтобы получить доступ к трассировкам через логи

1. Выберите `trace_id`, находясь в развёрнутом представлении лога.
2. Перейдите в **Explore**, выберите **Open field with** и выберите **Distributed Traces** во всплывающем окне.

Приложение Distributed Traces отображает хронологический список вызванных функций, который вы можете использовать для пошагового анализа проблемы. Точка сбоя будет отмечена красной линией.

Пример представления приложения Distributed Traces

![Приложение Distributed Traces, отображающее вызванные методы с точкой сбоя, отмеченной красным](https://dt-cdn.net/images/distributed-traces-problem-debug-with-logs-1860-966aefdd42.png)

## Итоги

Если вы выполнили все шаги, вы:

* Создали конвейер, предназначенный для обнаружения ошибок.
* Нашли первопричину с помощью приложения Problems и логов.

Если ваш разработчик предоставил вам runbook, который можно использовать в качестве руководства по устранению ошибок, вы можете следовать ему в качестве следующего шага.

В противном случае следующим шагом должно быть обращение к команде, ответственной за обслуживание сервиса или функции, вызвавшей ошибку. Если вы являетесь частью этой команды, вы можете начать процесс отладки проблемы.
