# Оптимизация производительности и стоимости дашбордов с запросами логов


Логи являются важным сигналом мониторинга.
Многие команды создают собственные дашборды, основанные на логах.

Рассмотрим следующий дашборд, который анализирует логи ошибок, генерируемые рабочими нагрузками Kubernetes в пространстве имён.

![Dashboard to analyze error logs generated by Kubernetes workloads running in a namespace](https://dt-cdn.net/images/k8s-log-dashboard-2884-c3fd02fff9.png)

Этот дашборд использует два типа запросов логов:

* Запросы, агрегирующие записи логов

  + с использованием команды `makeTimeseries`.

    ```
    fetch logs


    | filter k8s.namespace.name == "obslab-log-problem-detection"


    | filter status == "ERROR"


    | makeTimeseries  count = count()
    ```

    Запустить в Playground
  + с использованием команды `summarize`.

    ```
    fetch logs


    | filter k8s.namespace.name == "obslab-log-problem-detection"


    | filter status == "ERROR"


    | summarize count = count(), by: { k8s.deployment.name } | sort count desc
    ```

    Запустить в Playground
* Запрос, извлекающий записи логов в необработанном виде.

  ```
  fetch logs


  | filter k8s.namespace.name == "obslab-log-problem-detection"


  | filter status == "ERROR"
  ```

  Запустить в Playground

Команда хочет знать

* Какие компоненты генерируют больше всего ошибок.
* Как ошибки распределены во времени.
* Какие записи являются примерами.

В этом руководстве вы узнаете различные техники получения этой информации при одновременной оптимизации дашборда по производительности и стоимости.

## Как оптимизировать запросы

Рассмотрим, что можно сделать для оптимизации различных типов запросов.

### Преобразование запросов в метрики на основе логов

Когда вас интересуют агрегированные значения, такие как суммы или подсчёты, разделённые по измерению с низкой кардинальностью, следуйте руководству [Парсинг строк логов и извлечение метрики](../../../platform/openpipeline/use-cases/tutorial-log-processing-pipeline.md "Настройка обработки OpenPipeline для строк логов.").

В нашем примере требуемая конфигурация для извлечения метрики:

* **Имя**: `errors in obslab-log-problem-detection`
* **Условие совпадения**: `k8s.namespace.name == "obslab-log-problem-detection" AND status == "ERROR"`
* **Ключ метрики**: `log.k8s.obslab.errors`
* **Измерения**: `k8s.workload.name`

Затем необходимо скорректировать DQL-выражение на плитке дашборда.

```
timeseries count = count(log.obslab.errors)
```

После корректировки двух плиток генерация стоимости запросов при отрисовке этих двух плиток прекратится.
Метрики на основе логов лицензируются так же, как и любые другие [пользовательские метрики на базе Grail](../../../license/capabilities/metrics.md "Узнайте, как рассчитывается потребление метрик Dynatrace на базе Grail в модели подписки Dynatrace Platform.").

Не всегда возможно извлечь метрики из любого DQL-запроса.
Учитывайте кардинальность измерений и специфические команды, такие как `dedup` или `join`.
Если извлечь метрики не получается, вам потребуется оптимизировать DQL-запрос и конфигурацию дашборда.

### Оптимизация DQL-запросов, которые невозможно преобразовать в метрики

В нашем примере дашборда представлено поле содержимого записи лога.
Это поле с высокой кардинальностью, и поэтому оно не подходит для измерения метрики.

Первое, что вы можете сделать, -- установить временной диапазон, сегмент или бакет, если это возможно.
Это сразу повысит производительность и снизит затраты.
Для этого просто отредактируйте плитку дашборда.

* Установить временной диапазон: включите переключатель **Custom timeframe** и выберите временной диапазон из выпадающего меню.
* Установить сегмент: включите переключатель **Custom segments** и выберите сегмент из выпадающего меню.
* Установить бакет: добавьте следующую строку в DQL-запрос.

  ```
  | filter dt.system.bucket == "default_logs"
  ```

Дополнительные лучшие практики DQL для оптимизации запросов см. в [Лучшие практики DQL](../../../platform/grail/dynatrace-query-language/dql-best-practices.md "Лучшие практики использования Dynatrace Query Language.").

### Оптимизация порядка команд запроса

Глядя на наш запрос, мы также можем фильтровать раньше и устанавливать поля.

```
fetch logs


| filter dt.system.bucket == "default_logs"


| filter k8s.namespace.name == "obslab-log-problem-detection"


| filter status == "ERROR"


| fields timestamp, content, k8s.workload.name
```

Если ваш сценарий это позволяет, вы также можете установить ограничения запроса.
Перейдите в **Edit tile** > **Query limits**.
Затем вы можете настроить

* **Read data limit (GB)**.
* **Record limit**.
* **Result size limit (MB)**.
* **Sampling**.

При изменении лимита чтения данных и лимита записей пользователям, которым нужно больше результатов, следует использовать функцию **Open with** и анализировать логи в ![Logs](https://dt-cdn.net/images/logs-256-ae0a9ca67f.png "Logs") **Logs**.

Уменьшение размера выборки снижает объём сканируемых данных и возвращает более разнообразные образцы в больших наборах данных.
Выборка полезна для:

* Оптимизации производительности: используйте выборку для эффективного анализа больших наборов данных без обработки всех данных.
* Обнаружения аномалий: используйте выборку для быстрого выявления необычных закономерностей или выбросов в реальном времени.
* Анализа трендов: используйте выборку для понимания общих тенденций без изучения каждой точки данных.

В ситуациях, когда вы подсчитываете записи (например, используя команду `summarize`), не забудьте умножить результат на коэффициент выборки, чтобы получить лучшее приближение результата.

Выборка -- не лучшая идея, когда вам важна точность или вы работаете с небольшими наборами данных.

### Настройка бакета для использования Retain with Included Queries

Если многие пользователи часто открывают дашборд и запрашивают короткие временные диапазоны (данные менее 35 дней), рассмотрите использование ценовой модели [Retain with Included Queries](../../../license/capabilities/log-analytics.md#log-retain-included-queries "Узнайте, как рассчитывается потребление Dynatrace Log Analytics в модели подписки Dynatrace Platform.") и установите соответствующие [разрешения IAM](../../../platform/grail/organize-data/assign-permissions-in-grail.md#included-queries "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.").

Подробные инструкции по настройке бакета см. в [Управление стоимостью запросов логов с помощью Retain with Included Queries](lma-e2e-included-log-queries.md "Как использовать возможность Retain with Included Queries для контроля и прогнозирования потребления логов.").

### Отключение автообновления

Когда ваш дашборд автоматически обновляется каждую минуту, все ваши запросы выполняются.
Если вы используете запросы логов, с вас взимается плата за каждое выполнение.

Рассмотрите отключение функции автообновления через выпадающее меню, выбрав **Off**.

## Заключение

Используя описанные выше техники, мы сократили объём сканируемых данных за 24-часовой временной диапазон с 12,3 ГБ до 147 МБ.
Ваши результаты могут отличаться.

Лучшие практики:

* По возможности используйте метрики на основе логов, как описано в [Настройка пользовательских оповещений на основе метрик, извлечённых из логов](lma-alert-log-based-metrics.md "Как создавать и настраивать проблемы Davis и пользовательские оповещения с метриками на основе логов.").
  Метрики быстрее, имеют более длительное и дешёвое хранение, запросы для метрик не тарифицируются, вы можете удалять записи логов, формирующие метрику, и экономить на стоимости хранения, метрики можно использовать для оповещений.

  Для извлечения метрик [парсите строки логов и извлекайте метрику](../../../platform/openpipeline/use-cases/tutorial-log-processing-pipeline.md "Настройка обработки OpenPipeline для строк логов.") с помощью OpenPipeline.
* Если ваш сценарий требует представления содержимого логов (или других данных с высокой кардинальностью) через дашборды, оптимизируйте DQL-запросы, устанавливая временной диапазон, сегменты, бакеты, ограничения запросов и отключая автообновление.
