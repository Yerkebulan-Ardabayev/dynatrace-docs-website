# Запросы и фильтрация логов


* Latest Dynatrace

В ![Logs](https://dt-cdn.net/images/logs-256-ae0a9ca67f.png "Logs") **Logs** вы можете создавать запросы, использовать фильтры, искать определенные строки логов и многое другое.

## Создание запроса в Logs **Logs**

Запрашивайте логи, указывая сегмент и оператор фильтрации с ключами и значениями, содержащими ваши поисковые термины, компараторы и логические операторы.

* **Сегмент**: Общий фильтр для данных наблюдаемости в приложениях на платформе Dynatrace.
* **Ключ**: Поле или атрибут, по которому вы хотите фильтровать.
* **Значение**: Конкретное значение, которое вы ищете.
* **Логический оператор**: Соединяет несколько операторов фильтрации.

  По умолчанию все операторы фильтрации соединены оператором `AND`.
* **Компаратор**: Определяет тип сравнения.

![Поле фильтра в приложении Logs](https://dt-cdn.net/images/untitled-001-1835-80e79a2919.png)

[Фильтрация с помощью фасетов](facets.md "Filter with facets in the Dynatrace Logs app.") позволяет автоматически добавлять ключи и значения в ваш фильтр.

Используйте выбор даты, чтобы задать правильный временной диапазон для вашего запроса.

Нажмите **Run query**, чтобы выполнить запрос.

После того как ваш запрос вернет записи в таблицу результатов, вы можете искать ключевые слова в этих данных. Используйте поле **Search in results** для фильтрации таблицы по вашему ключевому слову. Эта фильтрация не выполняет новый запрос, а только отображает уже возвращенные и загруженные результаты в вашем браузере.

## Использование Segments **Segments**

Применяйте сегментный фильтр к вашему запросу при каждой возможности.

* Сегменты позволяют фильтровать логи и другие данные наблюдаемости с помощью единообразного фильтра.
* Сегменты удобны для ограничения ваших запросов только определенными бакетами Grail, что сокращает объем данных, которые необходимо сканировать для получения релевантных результатов.

  Дополнительные сведения см. в разделах [Сегментация логов по бакетам](../../../manage/segments/use-cases/segments-use-cases-logs-by-bucket.md "Segment logs by bucket with segments") и [лучшие практики для логов](../lma-best-practices.md#use-bucket-filters "Best practices for setting up Log Management and Analytics with Dynatrace.").
* Сегменты позволяют сохранять и повторно использовать часто применяемые фильтры, которые работают в ![Logs](https://dt-cdn.net/images/logs-256-ae0a9ca67f.png "Logs") **Logs** и в других приложениях Dynatrace.

## Обзор недавних фильтров и закрепление фильтров

![Logs](https://dt-cdn.net/images/logs-256-ae0a9ca67f.png "Logs") **Logs** сохраняет ваши недавно использованные фильтры, чтобы вы могли применить их повторно одним нажатием. Вы также можете сохранить фильтры на постоянной основе, закрепив их.

Выберите поле фильтра и проверьте раздел **Recently used filters**. В этом разделе отображаются фильтры, которые вы недавно применяли, начиная с самого последнего. По мере ввода нового оператора фильтрации этот список сужается, показывая похожие операторы из ваших недавно использованных фильтров.

Нажмите  (**Pin filter**), чтобы закрепить любой созданный вами фильтр. Когда поле фильтра пустое, выберите его и прокрутите вниз до раздела **Pinned filters**, чтобы просмотреть ранее закрепленные фильтры. Открепите фильтр, нажав  (**Pin filter**) еще раз.

## Поиск фразы в логах

Если вам нужно найти логи, содержащие определенную фразу, у вас есть несколько вариантов -- от самой широкой до более узкой фильтрации.

### Поиск по всем полям

Используйте `*` вместо имени ключа и `~` в качестве компаратора для поиска фразы по всем полям записи лога, соответствующим вашим фильтрам.

Например, фильтр `* ~ "failed to charge card"` находит логи, содержащие эту фразу в любом поле.

Это эквивалентно использованию команды [`search`](../../../platform/grail/dynatrace-query-language/commands/filtering-commands.md#search "DQL filter and search commands") DQL.

### Поиск по полю `content`

Как правило, исходное содержимое лога сохраняется в поле `content` записи лога. Ограничьте поиск этим полем для повышения производительности запроса.

Например, фильтр `content ~ "failed to charge card"` находит логи, содержащие эту фразу в поле `content`.

Это эквивалентно использованию строковой функции [`matchesPhrase`](../../../platform/grail/dynatrace-query-language/functions/string-functions.md#matchesPhrase "A list of DQL string functions.") DQL.

### Подстановочный знак в значении

Вы также можете указать только часть значения с помощью подстановочного знака, используя `*` в поисковом запросе.

Например, фильтр `content = "*card*"` находит логи, содержащие фразу `card` в поле `content`.

Это эквивалентно использованию строковой функции [`matchesValue`](../../../platform/grail/dynatrace-query-language/functions/string-functions.md#matchesValue "A list of DQL string functions.") DQL.

Полное описание см. в разделе [Поле фильтра](../../../discover-dynatrace/get-started/dynatrace-ui/ui-filter-field.md "The filter field is a powerful tool that allows you to quickly find relevant information or narrow down results within apps.").

## Фильтрация по полю с несколькими значениями

По умолчанию все операторы фильтрации соединены логическим оператором `AND`. Например, `status = ERROR` `status = WARN` не вернет результатов, так как одна запись лога не может иметь два статуса.

Для запроса по полю с различными значениями используйте оператор `in`. Например, `status in (ERROR, WARN)` возвращает логи со статусом `ERROR` или `WARN`. В качестве альтернативы вы можете использовать `OR` для объединения нескольких операторов фильтрации.

## Использование автоматических подсказок для фильтров логов

![Logs](https://dt-cdn.net/images/logs-256-ae0a9ca67f.png "Logs") **Logs** предоставляет подсказки автозаполнения для ключей, компараторов и значений.

* Выберите поле фильтра, чтобы получить список подсказок с наиболее часто используемыми или релевантными полями для фильтрации.
* После выбора поля вы получите список подсказок для компараторов.
* После выбора компаратора вы получите список подсказок значений для данного поля. Для поля `content` это не поддерживается.

Обратите внимание, что подсказки основаны на реальных значениях, запрашиваемых в фоновом режиме из ваших данных логов, при этом контекстно релевантные подсказки не расходуют ресурсы запросов.
