# Лимиты по умолчанию для Log Management and Analytics


* Пояснение

На этой странице перечислены лимиты по умолчанию для последней версии Dynatrace Log Management and Analytics. Текущие ограничения применяются как к приёму логов из файлов, так и к приёму логов через Log Ingestion API.

## Лимиты приёма логов

В таблице ниже перечислены наиболее важные лимиты по умолчанию, связанные с приёмом логов. Все представленные лимиты относятся к данным в кодировке UTF-8.

| Тип | Лимит | Описание |
| --- | --- | --- |
| Содержимое | 10 МБ[1](#fn-1-1-def) | Максимальный размер тела записи лога |
| Ключ атрибута | 100 байт | Ключ значения атрибута |
| Длина значения атрибута | 32 КБ | Максимальная длина значения атрибута |
| Количество атрибутов лога | 500 | Максимальное количество атрибутов, которые может содержать лог |
| Событий лога в минуту | Без ограничений | Максимальное количество событий лога в минуту |
| Возраст лога | 24 часа | Максимальный возраст записей лога при приёме |
| Логи с датами в будущем | Без ограничений[2](#fn-1-2-def) | Насколько далеко в будущее могут быть датированы записи лога |
| Значений на атрибут | 32 значения | Максимальное количество отдельных значений, которые может содержать атрибут |
| Размер запроса [3](#fn-1-3-def) | 10 МБ | Максимальный размер данных запроса |
| Количество записей лога | 50 000 записей | Максимальное количество записей лога на запрос |
| Вложенные объекты | 5 уровней | Максимальное количество уровней для вложенных объектов |
| Извлечённый атрибут лога | 4 096 байт | При добавлении логов в шаблон события атрибуты лога усекаются до 4096 байт |

1

Лимит содержимого ниже (512 КБ) для логов, направляемых в **Classic pipeline**.

2

Ограничений на приём записей лога с временными метками в будущем нет, но записи с временными метками более чем на 10 минут в будущем получают временную метку, равную моменту приёма.

3

Что касается размера запроса, конечные точки Log Ingestion API принимают запросы до 10 МБ. Однако после начальной обработки размер пакета может увеличиться. Если после обработки он превысит 16 МБ, он будет отклонён с ошибкой 413: `Message size limit exceeded after preprocessing on ingest endpoint`. Чтобы избежать этой проблемы, отправляйте меньшие пакеты записей лога, чтобы уложиться в ограничения по размеру.

Размер запроса лога может увеличиться по следующим причинам:

* Отсутствие атрибутов содержимого в принятых записях лога: если лог, принятый через конечную точку Log Ingestion API, не содержит атрибута, подобного содержимому, этот атрибут будет добавлен после приёма.
* Для логов, принятых через конечную точку OTLP, атрибуты ресурса и области копируются в каждую отдельную запись лога.

Проверьте ваш доступ к OpenPipeline в разделе Обработка логов с помощью OpenPipeline.

## Задержка приёма логов

Логи, принятые через OneAgent, обычно готовы к анализу через промежуток от нескольких секунд до 90 секунд (в среднем 30 секунд).

Логи, принятые через API, доступны для анализа в Dynatrace в среднем через 10 секунд.

Иногда более высокая задержка может возникать из-за механизмов предотвращения потери данных, таких как повторные передачи, буферизация или другие факторы, которые могут вызывать задержки.

## Допустимый временной диапазон записей лога

Следующие правила применяются ко всем источникам событий лога, таким как OneAgent и общий API приёма логов.

## Метрики логов

Количество метрик ограничено:

* `100 000` (`1000` на конвейер x `100` конвейеров) для Log Management and Analytics на базе Grail с OpenPipeline
* `1000` для Log Management and Analytics на базе Grail без включённого OpenPipeline
* `50` в остальных случаях.

## Объекты запросов Log Ingestion API

Помимо общих ограничений Dynatrace API (Dynatrace API — лимит доступа) действуют следующие ограничения, специфичные для Log Ingestion API:

* JSON-объект `LogMessageJson`. См. Приём логов в форматах JSON и TXT для полного списка ключей и их описания.
* Объект протокола OpenTelemetry `LogMessageOTLP`. См. Приём логов OTLP.

## Лимиты автообнаружения логов при использовании OneAgent

Файлы логов в OneAgent:

* не могут быть удалены ранее чем через минуту после создания.
* должны дополняться (старое содержимое не обновляется).
* должны содержать текстовые данные.
* должны быть открыты постоянно (а не только на короткие периоды добавления записей).
* должны быть открыты в режиме записи.
* не должны быть меньше настроенного порога размера (по умолчанию: 500 байт) для проверки на двоичное содержимое.

Максимальное количество источников логов на экземпляр группы процессов по умолчанию — 200. Это значение настраивается с помощью параметра **Maximum number of log sources per process group instance** в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Log Monitoring** > **Advanced log settings**.

В стандартных средах модуль логов OneAgent поддерживает до 10 000 файлов в одном каталоге с логами и 200 МБ нового содержимого логов в минуту. Если у вас больше данных, особенно на порядок больше, есть высокая вероятность, что модуль логов OneAgent их поддержит. Свяжитесь с командой поддержки Dynatrace для предварительной проверки вашей конфигурации.

В особых случаях, таких как очень низкая производительность оборудования, ограничения модуля логов OneAgent могут быть более строгими.

## Пропускная способность ActiveGate

Если вы используете конечную точку Dynatrace, вам не нужно беспокоиться о пропускной способности ActiveGate. Пропускная способность такая же, как и для Grail.

Если вы используете Environment ActiveGate, выделенный для приёма логов через Log Ingestion API, результаты тестов производительности показывают следующую устойчивую пропускную способность приёма на экземпляр Environment ActiveGate:

* До 10,3 ГБ/мин в типичных условиях WAN (время кругового обхода до ~200 мс)
* До 12,1 ГБ/мин в условиях низкой задержки (RTT <= 10 мс)

Использовался следующий тестовый профиль:

* Экземпляр хоста: c6i.2xlarge
* JSON-запросы объёмом приблизительно 300 КБ на запрос, каждый содержащий 300 строк лога, отправленные через Log Ingestion API
