# Оповещения по журналам


* Latest Dynatrace
* Explanation
* 3-min read

Эффективное оповещение необходимо для поддержания оптимальной производительности и оперативного устранения проблем.
Различные стратегии оповещения с использованием журналов обеспечивают своевременные уведомления на основе данных журналов.
Каждая стратегия предлагает уникальные преимущества и параметры конфигурации, ориентированные на различные сценарии использования и требования.

Понимание этих подходов поможет вам выбрать наиболее подходящий метод оповещения для обеспечения бесперебойной работы ваших приложений и систем.

## Обзор различных методов

### Оповещения с использованием метрик на основе журналов

Используйте пользовательские оповещения с метриками на основе журналов, когда вам нужно:

* Устанавливать пороговые значения.
* Применять статистические анализаторы для запуска оповещений.

Метрики на основе журналов особенно полезны для обнаружения аномалий в количестве вхождений записей журнала или в значениях, производных от полей журнала, например `http.response_time`.

Учтите, что анализаторы метрик срабатывают каждую минуту, поэтому они не подходят для оповещений в режиме реального времени.

Подробные инструкции см. в разделе Настройка пользовательских оповещений на основе метрик, извлечённых из журналов.

### Оповещения с использованием событий на основе журналов

Для простых сценариев оповещения, где не требуется устанавливать пороговые значения, используйте события Davis, извлечённые из журналов.

Этот метод идеально подходит для случаев очень редких вхождений шаблона в журнале (раз в неделю, раз в месяц), когда метрики не были бы полезны.
Он также обеспечивает оповещение практически в реальном времени и мгновенные уведомления без необходимости дополнительного обзора совпадающих данных во времени.

Метод особенно полезен, когда требуется быстро реагировать на конкретные события в журнале без сложности статистического анализа.

Подробные инструкции см. в разделе Настройка оповещений на основе событий, извлечённых из журналов.

### Использование DQL-запросов в пользовательских оповещениях

Используйте DQL-запросы в пользовательских оповещениях, когда вам нужно определить пользовательские условия оповещения на основе конкретных шаблонов данных журнала, а извлечение метрик или событий невозможно.
Этот подход позволяет гибко и точно выполнять запросы для выявления событий или тенденций в журналах.

Учтите, что эти запросы выполняются каждую минуту, что может увеличить потребление лицензии.
Поэтому используйте только оптимизированные запросы.

Этот метод, как правило, не рекомендуется в качестве основной стратегии оповещения.
Однако он может служить запасным вариантом, когда оповещение с использованием метрик или событий невозможно.

Подробные инструкции см. в разделе Создание оповещений по журналам для события журнала или сводки данных журнала.

## Сравнение методов оповещения

| Аспект | События на основе журналов | Метрики на основе журналов (рекомендуется) | DQL-запросы в пользовательских оповещениях |
| --- | --- | --- | --- |
| Тип оповещения | Простое оповещение без пороговых значений | Оповещение по пороговым значениям с использованием статистических анализаторов | Пользовательские запросы для определения условий оповещения |
| Время реакции | Самое быстрое | Срабатывает каждую минуту | Срабатывает каждую минуту |
| Сложность настройки | Низкая (только извлечение событий) | Высокая (требует настройки извлечения метрик и обнаружения аномалий — новая конфигурация пользовательского оповещения **Anomaly Detection**) | Средняя (требует конфигурации пользовательского оповещения) |
| Сценарий использования | Быстрые реакции на конкретные события журнала при поступлении | Обнаружение аномалий в количестве вхождений записей или значениях, производных от полей журнала | Пользовательские условия оповещения на основе данных журнала |
| Пример | Мгновенное оповещение о конкретной записи журнала | Оповещение об аномалиях в значениях поля `http.response_time`; оповещение, когда совпадающая запись встречается 10 раз | Оповещение по результатам конкретного запроса к журналу с применением статистических анализаторов |
| Стоимость | Зависит от количества сгенерированных событий и их размера | Зависит от количества точек данных и размера метрики | Зависит от сложности запроса и объёмов сканирования |
