# Настройка хранения и срока хранения данных для логов


* Latest Dynatrace

Логи хранятся в бакетах Grail с периодом хранения от 10 дней до 10 лет. По умолчанию данные логов хранятся 35 дней в бакете `default_logs`.

## Результат обучения

После завершения этого руководства вы сможете:

* Хранить данные логов в пользовательском бакете для определённой группы пользователей или с увеличенным сроком хранения до 10 лет.
* Пропускать сохранение данных логов из определённого источника приёма или на основе условий соответствия.
* Управлять тарификацией запросов для бакета логов.

## Целевая аудитория

Это руководство предназначено для инженеров по надёжности сайтов (SRE) и архитекторов, которые хотят настроить параметры хранения и сроки хранения для контроля доступа, оптимизации или обеспечения соответствия нормативным требованиям.

## Предварительные требования

* Разрешения для [управления пользовательскими бакетами Grail](../../platform/grail/organize-data.md#managing-custom-grail-buckets "Информация о модели данных Grail, включающей бакеты, таблицы и представления.").
* Разрешения `openpipeline:configurations:write` и `openpipeline:configurations:read`. Чтобы узнать, как настроить разрешения, см. [Разрешения в Grail](../../platform/grail/organize-data/assign-permissions-in-grail.md "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.").

## Пример 1: Хранение логов в течение трёх лет

Использование бакетов может повысить производительность запросов за счёт сокращения времени выполнения запросов и объёма считываемых данных. С помощью этой процедуры вы создаёте новый бакет с настраиваемым сроком хранения для данных логов. Записи логов, соответствующие маршруту и условиям конвейера, сохраняются в соответствии с выбранным сроком хранения бакета и доступны для чтения пользователям на основе разрешений.

Используя пользовательский бакет для логов, вы можете:

* Хранить данные логов с одинаковым сроком хранения.
* Хранить данные логов, которые необходимо запрашивать и анализировать совместно.
* Хранить данные логов, которые необходимо удалять одновременно.

Для получения дополнительной информации см. [Лучшие практики управления и аналитики логов](lma-best-practices.md "Лучшие практики настройки управления и аналитики логов в Dynatrace.").

### 1. Создание пользовательского бакета

1. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Storage management** > **Bucket storage management** >  **Bucket**.
2. Определите новый бакет.

   1. Введите имя бакета и пользовательский срок хранения в днях.
   2. В выпадающем списке **Bucket table type** выберите **logs**.
3. Нажмите **Create**.
4. Нажмите  (**Refresh**), чтобы обновить список бакетов.

### 2. Назначение данных логов бакету

1. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline** > **Logs** > **Pipelines**.
2. Выберите существующий конвейер или создайте новый.
3. На этапе **Storage** выберите  **Processor** > **Bucket assignment** и определите новый процессор.

   1. Введите имя процессора и условие соответствия.
   2. В выпадающем списке **Storage** выберите бакет, созданный на предыдущем шаге.
4. Нажмите **Save**.
5. Убедитесь, что ваш конвейер получает записи через динамический маршрут.

   1. Перейдите в **Dynamic routing**.
   2. Выберите существующий динамический маршрут или создайте новый.
   3. Определите маршрут, введя имя маршрута, условие соответствия (например, `true`) и имя целевого конвейера.
   4. Нажмите **Save**.

### 3. Назначение разрешений бакета пользователям

1. Перейдите в [**Account Management**](https://myaccount.dynatrace.com/) и выберите одну из ваших учётных записей.
2. Настройте разрешения и [назначьте разрешения бакета](../../platform/grail/organize-data/assign-permissions-in-grail.md "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.") нужным пользователям.

## Пример 2: Пропуск сохранения для выбранных логов

В этом примере вы пропускаете сохранение логов, соответствующих маршруту и условиям конвейера. Записи логов не сохраняются.

Это может быть полезно, когда вы [парсите строки логов и извлекаете метрики](../../platform/openpipeline/use-cases/tutorial-log-processing-pipeline.md "Настройка обработки логов в OpenPipeline."), и доступ к исходным записям не требуется.

Чтобы пропустить сохранение для выбранных записей логов

1. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline** > **Logs** >**Pipelines**; выберите существующий конвейер или создайте новый.
2. На этапе **Storage** выберите  **Processor** > **No storage assignment**.
3. Введите имя процессора и условие соответствия.
4. Нажмите **Save**.
5. Убедитесь, что ваш конвейер получает записи через динамический маршрут.

   1. Перейдите в **Dynamic routing**.
   2. Выберите существующий динамический маршрут или создайте новый.
   3. Определите маршрут, введя имя маршрута, условие соответствия (например, `true`) и имя целевого конвейера.
   4. Нажмите **Save**.

## Пример 3: Управление тарификацией запросов для каждого бакета

Существуют две модели хранения, которые можно настроить для каждого бакета:

* **На основе потребления**: каждое выполнение запроса тарифицируется отдельно.
* **Хранение с включёнными запросами**: данные логов за определённый период включены в стоимость хранения. Запросы к этим данным не приводят к дополнительным расходам.

Для получения дополнительной информации см. [Контроль расходов на запросы логов с помощью Retain with Included Queries](lma-use-cases/lma-e2e-included-log-queries.md "Использование возможности Retain with Included Queries для контроля и прогнозирования потребления логов.").
