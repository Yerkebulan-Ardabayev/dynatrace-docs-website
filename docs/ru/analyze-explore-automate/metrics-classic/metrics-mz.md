# Фильтрация классических метрик по зоне управления

# Фильтрация классических метрик по зоне управления

* Classic
* Чтение: 4 мин.
* Обновлено 12 июля 2023 г.

Вы можете фильтровать данные метрик по [зоне управления](/docs/manage/identity-access-management/permission-management/management-zones "Узнайте о концепциях зон управления, о том, как определять зоны управления и как извлечь из них максимальную пользу.") двумя способами.

* Во-первых, вы можете настроить [правила на основе сущностей](#entity-based-rules) для зоны управления, чтобы сопоставлять метрики с соответствующим типом сущности.
* Во-вторых, вы можете настроить [правила на основе измерений](#dimensional-rules) для зоны управления, чтобы сопоставлять серии метрик по ключам метрик или измерений.

В следующих разделах описывается каждый метод, а также [вопросы безопасности](#security) при фильтрации данных метрик по зоне управления.

## Правила на основе сущностей для фильтрации данных метрик

Вы можете определить [правила через интерфейс](/docs/manage/identity-access-management/permission-management/management-zones/management-zone-rules#ui "Определите правила для ограничения сущностей, доступных в зоне управления.") или [текстовые правила](/docs/manage/identity-access-management/permission-management/management-zones/management-zone-rules#text "Определите правила для ограничения сущностей, доступных в зоне управления."), чтобы предоставить доступ к сущностям в рамках зоны управления. Пользователи зоны управления автоматически получают доступ к метрикам, **тип сущности** которых соответствует сущностям, входящим в область действия зоны управления. Обратите внимание, что тип сущности метрики -- это основная сущность, связанная с этой метрикой. Хотя метрика может иметь несколько **измерений на основе сущностей**, это не то же самое, что тип сущности.

### Тип сущности метрики

Вы можете проверить тип сущности метрики, используя:

* Боковую панель информации о метрике в [Data Explorer](/docs/analyze-explore-automate/explorer#metric-name "Запрашивайте метрики и преобразуйте результаты для получения нужной информации.").

  ![Тип сущности метрики в Data Explorer](https://dt-cdn.net/images/data-explorer-entity-type-1122-6a93ade8f2.png)
* [Браузер метрик](/docs/analyze-explore-automate/dashboards-classic/metrics-browser "Просмотр метрик с помощью браузера метрик Dynatrace.").
* [Конечную точку GET дескриптора метрики в Metrics API](/docs/dynatrace-api/environment-api/metric-v2/get-descriptor "Просмотр дескриптора метрики через Metrics v2 API."); проверьте [элемент `entityType`](/docs/dynatrace-api/environment-api/metric-v2/get-descriptor#response "Просмотр дескриптора метрики через Metrics v2 API.").

В качестве альтернативы вы можете явно настроить тип сущности для **пользовательских метрик, полученных при приёме данных**, установив **Тип исходной сущности** на [странице **Метаданные метрики**](/docs/analyze-explore-automate/dashboards-classic/metrics-browser#configuration-ui "Просмотр метрик с помощью браузера метрик Dynatrace.").

![Модальное окно метаданных метрики с заполненным полем «Тип исходной сущности».](https://dt-cdn.net/images/sourceentitytype-metricmetadata-1270-b154365f51.png)

### Ограничения

Существуют два ограничения при фильтрации данных метрик в зонах управления с помощью правил на основе сущностей.

1. Фильтрация метрик по правилам на основе сущностей работает только для **типа сущности** метрики. Метрика может иметь несколько **измерений на основе сущностей**, таких как `dt.entity.host` и `dt.entity.disk`, но большинство метрик имеют только один тип сущности. Если правило зоны управления на основе сущности ссылается на сущность, которая не является типом сущности метрики, серия метрик не будет видна при применении зоны управления, даже если измерение метрики на основе сущности соответствует правилу.

   Например, если ваше правило зоны управления соответствует сущностям `QUEUE`, начинающимся с определённой строки, метрика `builtin:queue.incoming_requests` не будет видна в зоне управления. Это связано с тем, что хотя метрика имеет измерения на основе сущностей `dt.entity.service` и `dt.entity.queue`, её тип сущности -- `SERVICE`, который не соответствует правилу, ссылающемуся на сущности `QUEUE`.
2. Если метрика имеет **несколько типов сущностей**, фильтрация по зоне управления не применяется. Это ограничение затрагивает метрики с измерениями `dt.entity.monitored_entity` или `dt.entity.device_application`.

## Правила на основе измерений для данных метрик

Если тип сущности метрики не соответствует правилу на основе сущности в зоне управления, вы можете определить правило на основе измерения для сопоставления **измерения метрики на основе сущности**. Например, для метрики `builtin:queue.incoming_requests` необходимо определить правило на основе измерения для измерения `dt.entity.queue`. Обратите внимание, что значением условия измерения должен быть идентификатор сущности (например, `'dt.entity.queue' equals 'QUEUE-43717C10AF1BD1E0'`), а не имя сущности (см. изображение ниже).

## Вопросы безопасности

Ниже приведены вопросы безопасности, связанные с отображением данных метрик, и меры по снижению их воздействия.

### Серии метрик, видимые во всех зонах управления

Фильтрация по зоне управления не работает для метрик с несколькими типами сущностей. Следующие метрики подвержены этому ограничению.

* `builtin:billing.ddu.metrics.byEntity`
* `builtin:billing.ddu.metrics.byEntityRaw`
* `builtin:billing.ddu.log.byEntity`
* `builtin:billing.ddu.events.byEntity`
* `builtin:billing.ddu.serverless.byEntity`
* `builtin:billing.ddu.traces.byEntity`
* Несколько метрик `builtin:apps.other.` с измерением `dt.entity.device_application`

Из-за этого ограничения пользователи, которые не должны иметь доступа к этим сериям метрик, всё равно могут получить к ним доступ независимо от зоны управления.

### Преобразования селектора метрик

[Преобразования селектора метрик](/docs/dynatrace-api/environment-api/metric-v2/metric-selector "Настройка селектора метрик для Metric v2 API.") `parents` и `names` добавляют информацию к запросам данных метрик, что может раскрыть идентификатор родительской отслеживаемой сущности (например, идентификатор хоста, являющегося родителем процесса) или отображаемое имя сущности в случае преобразования `names`.

### Влияние

Хотя эти замечания заслуживают внимания, практическое влияние на безопасность ограничено. Данные [метрик с несколькими типами сущностей](#multiple-entity-types) обычно не являются конфиденциальными. Для сред с повышенными требованиями к безопасности рекомендуется учитывать замечания относительно [преобразований селектора метрик](#metric-selector-transformations) (например, идентификаторы сущностей вроде `HOST-27A71FA663E7F352` и отображение имён хостов) при предоставлении пользователю разрешений на чтение любых данных метрик.

Мы рекомендуем особое внимание при выборе измерений, отправляемых в составе данных для **пользовательских метрик с повышенными требованиями к безопасности**, чтобы обеспечить эффективность заданных ограничений доступа на чтение.
