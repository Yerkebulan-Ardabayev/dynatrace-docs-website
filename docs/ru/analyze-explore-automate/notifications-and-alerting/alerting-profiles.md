# Профили оповещений о проблемах


Профили оповещений о проблемах управляют доставкой уведомлений о проблемах по каналам оповещений вашей организации на основе предопределённых фильтров, учитывающих серьёзность проблемы, её продолжительность, пользовательские события и теги. Профили оповещений о проблемах позволяют точно контролировать, какие условия приводят к отправке уведомлений, а какие нет. Это касается всех push-уведомлений о проблемах, отправляемых через [мобильное приложение Dynatrace](push-notifications-via-the-dynatrace-mobile-app.md "Узнайте, как подключить среды Dynatrace к мобильному приложению Dynatrace для получения уведомлений о проблемах.") и отображаемых в веб-интерфейсе Dynatrace. Профили оповещений также могут использоваться для настройки фильтрованных интеграций уведомлений с системами обмена сообщениями сторонних производителей, такими как [Slack](problem-notifications/slack-integration.md "Настройте интеграцию Slack для уведомлений о проблемах, чтобы быть в курсе всех проблем Dynatrace."), [Opsgenie](problem-notifications/opsgenie-integration.md "Настройте интеграцию Opsgenie с Dynatrace.") и [PagerDuty](problem-notifications/pagerduty-integration.md "Узнайте, как отправлять уведомления о проблемах из Dynatrace в PagerDuty.").

Каждая среда мониторинга имеет профиль оповещений по умолчанию, который определяет уровень серьёзности, необходимый для отправки оповещения при обнаружении проблемы.

Администратор среды может создавать новые профили оповещений, обеспечивающие детальный контроль маршрутизации оповещений для отдельных команд или интеграций уведомлений с системами сторонних производителей, такими как Slack, HipChat и другие.

Чтобы просмотреть профили оповещений вашей среды, перейдите в **Settings** > **Alerting** > **Problem alerting profiles**. Профиль `Default` всегда присутствует в списке. Его можно изменить, но нельзя удалить.

## Область действия профиля оповещений

Профили оповещений предоставляют мощный механизм фильтрации уведомлений о проблемах. Комбинируя критерии фильтрации, вы можете создавать пользовательские профили, которые, например, откладывают уведомления о проблемах в средах разработки, при этом немедленно оповещая о проблемах, обнаруженных в производственных средах. Область действия профиля оповещений определяется следующими параметрами:

* Зона управления.
* Правила серьёзности.
* Фильтры событий.

Эти условия объединяются логикой **И (AND)**. Событие должно соответствовать **всем** условиям, чтобы вызвать уведомление на основе профиля.

Сопоставление тегов

Сопоставление тегов в профилях оповещений выполняется точно. Если вы указываете только ключ, тег будет соответствовать сущностям с данным ключом без значения. Например, если вы укажете ключ `Apps` без значения, он будет соответствовать сущности с ключом `Apps` и пустым значением. Однако он не будет соответствовать сущности `AppsRedis`, которая имеет тот же ключ, но непустое значение.

Для сопоставления тегов «ключ-значение» необходимо указать и ключ, и значение.

### Зона управления

Фильтр по зоне управления сокращает объём данных, которые оценивает профиль оповещений. Вместо проверки всех данных, генерируемых вашей средой, вы можете сосредоточиться только на данных из указанной зоны управления.

По умолчанию новый профиль оповещений использует `All` (все) зоны управления, что означает отсутствие фильтра. В большинстве случаев рекомендуется применять фильтр по зоне управления, чтобы ограничить область профиля зоной ответственности вашей команды. Учитывайте, что зоны управления могут пересекаться. Если проблема обнаружена на сервисе, который определён в нескольких зонах управления, будут применены несколько фильтров.

### Правила серьёзности

Правила серьёзности фильтруют события по уровню серьёзности. Для каждого профиля оповещений можно задать до 100 правил серьёзности. Эти правила объединяются логикой **ИЛИ (OR)**. Событие, соответствующее **любому** из правил, вызывает уведомление на основе профиля.

Доступны следующие критерии:

* [Уровень серьёзности](../../dynatrace-intelligence/root-cause-analysis/event-analysis-and-correlation/event-categories.md "Узнайте о различных категориях событий и поддерживаемых типах событий, их уровнях серьёзности и логике их создания.").
* Время, в течение которого проблема должна быть открыта, прежде чем будет отправлено оповещение — это позволяет избежать оповещений о проблемах низкой серьёзности, которые не влияют на пользовательский опыт и не требуют немедленного внимания.
* Необязательно: мониторируемые сущности, имеющие любые или все указанные теги

Критерии правила объединяются логикой **И (AND)**. **Все** критерии должны быть выполнены для срабатывания правила.

### Фильтры событий

Правила событий фильтруют события по их свойствам. Для каждого профиля оповещений можно задать до 20 правил событий. Особенно для сценариев автоматического исправления полезно запускать определённые действия на основе детальной информации, собранной во время аномальных ситуаций, например, отправлять оповещения в случаях, когда проблемы связаны со сбоями процессов.

Доступны следующие критерии:

* Предопределённые события: тип события
* Пользовательские события: заголовок, описание и свойства события

Каждый критерий может быть инвертирован с помощью опции **negate** (отрицание). Например, это превращает критерий **begins with** (начинается с) в **does not begin with** (не начинается с).

Правила объединяются следующей логикой:

1. Все правила, содержащие инвертированные критерии, группируются логикой **И (AND)**.
2. Все остальные правила группируются логикой **ИЛИ (OR)**.
3. Эти две группы (инвертированные и неинвертированные) объединяются логикой **И (AND)**.

## Создание профиля оповещений

Чтобы создать профиль оповещений:

1. Перейдите в **Settings** > **Alerting** > **Problem alerting profiles** и выберите **Add alerting profile**.
2. Введите имя нового профиля в поле **Create new alerting profile** и нажмите **Create**.
3. Определите фильтр по [зоне управления](../../manage/identity-access-management/permission-management/management-zones.md "Узнайте о концепциях зон управления, их определении и эффективном использовании.").
4. Определите правила [уровня серьёзности](../../dynatrace-intelligence/root-cause-analysis/event-analysis-and-correlation/event-categories.md "Узнайте о различных категориях событий и поддерживаемых типах событий, их уровнях серьёзности и логике их создания.") для профиля.
5. Определите фильтр событий. Он может основываться на одном из следующих вариантов:

   * `Predefined`: конкретный встроенный тип события
   * `Custom`: заголовок, описание и свойства события
6. Просмотрите сводку критериев вашего нового профиля оповещений и при необходимости скорректируйте их.
7. Нажмите **Done** в правом верхнем углу, чтобы сохранить новый профиль оповещений.

## API профилей оповещений

Помимо веб-интерфейса Dynatrace, вы можете управлять профилями оповещений через [Settings API](../../dynatrace-api/configuration-api/alerting-profiles-api.md "Узнайте о возможностях API профилей оповещений Dynatrace."). Ищите схему **Problem alerting profiles** (`builtin:alerting.profile`).
