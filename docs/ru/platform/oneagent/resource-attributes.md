## Атрибуты ресурсов уровня хоста

Все артефакты мониторинга, покидающие данный хост, то есть имеющие хост в качестве своего ресурса, обогащаются атрибутами ресурсов уровня хоста.

Атрибуты ресурсов уровня хоста -- это атрибуты ресурсов мониторируемых хостов. Все события, генерируемые компонентами OneAgent, работающими на данном хосте, и все измерения, поступающие от них, обогащаются этими атрибутами. Затем вы можете использовать их в своих запросах для структурирования и фильтрации данных мониторинга.

Вы также можете использовать некоторые из атрибутов для создания политик управления доступом к данным. См. [Справочник глобальных полей](../../semantic-dictionary/fields.md "Ознакомьтесь со списком глобальных полей, имеющих чётко определённое семантическое значение в Dynatrace и используемых в различных типах мониторинга.") и выполните поиск полей с меткой Permission.

Если у вас есть доступ к хосту с установленным OneAgent, вы можете просмотреть файлы `dt_host_metadata.json` и `dt_host_metadata.properties`, чтобы увидеть объём обогащения атрибутами ресурсов, предоставляемого OneAgent. Для получения дополнительной информации см. [Обогащение принимаемых данных полями, специфичными для Dynatrace](../../ingest-from/extend-dynatrace/extend-data.md "Узнайте, как автоматически обогащать ваши данные телеметрии полями, специфичными для Dynatrace.").

### Пользовательские атрибуты уровня хоста

Вы можете создавать собственные атрибуты, настраивая теги и свойства типа «ключ-значение» через [oneagentctl](../../observe/infrastructure-observability/hosts/configuration/define-tags-and-metadata-for-hosts.md "Узнайте, как присваивать теги и задавать дополнительные свойства для мониторируемого хоста.") или через [удалённое управление конфигурацией OneAgent и ActiveGate](../../ingest-from/bulk-configuration.md "Выполняйте конфигурацию OneAgent и ActiveGate на хостах со страницы состояния развёртывания или массово с использованием API Dynatrace."). Пользовательские теги и свойства, определённые таким образом, передаются как плоские атрибуты ресурсов первого уровня.

Теги-ключи без значений игнорируются.

Теги, назначенные через [автоматизированные правила](../../manage/tags-and-metadata/setup/how-to-define-tags.md#automatic "Узнайте, как определять и применять теги вручную и автоматически."), [переменные окружения](../../manage/tags-and-metadata/setup/define-tags-based-on-environment-variables.md "Узнайте, как Dynatrace позволяет определять теги на основе переменных окружения.") и [API Topology и Smartscape](../../dynatrace-api/environment-api/topology-and-smartscape.md "Узнайте об API Dynatrace Topology и Smartscape."), не включаются.

Поскольку ключи зависят от вашей конфигурации, они не описаны в Semantic Dictionary.

Имена тегов не могут иметь префикс `dt.`, за исключением следующих тегов, которые зависят от вашей конфигурации:

* `dt.security_context` -- атрибут для управления доступом к данным с помощью политик. См. [Предоставление доступа к сущностям с контекстом безопасности](../../manage/identity-access-management/use-cases/access-security-context.md "Предоставление доступа к сущностям с контекстом безопасности").
* `dt.cost.costcenter` -- атрибут для назначения использования центру затрат. См. [Распределение затрат DPS](../../license/cost-allocation.md "Узнайте, как распределять затраты по центрам затрат и продуктам.").
* `dt.cost.product` -- атрибут для назначения использования продукту или идентификатору приложения. См. [Распределение затрат DPS](../../license/cost-allocation.md "Узнайте, как распределять затраты по центрам затрат и продуктам.").

Атрибуты из пользовательских свойств и тегов не переопределяют встроенные обогащения, если имеют одинаковое имя. При создании пользовательских свойств и тегов проверьте [Справочник глобальных полей](../../semantic-dictionary/fields.md "Ознакомьтесь со списком глобальных полей, имеющих чётко определённое семантическое значение в Dynatrace и используемых в различных типах мониторинга."), чтобы убедиться, что ваше имя ещё не используется.

#### Обработка конфликтов имён при объединении пользовательских свойств и тегов

Атрибуты ресурсов уровня хоста могут быть импортированы из пользовательских свойств и тегов хоста. Файлы `hostcustomproperties.conf` и `hostautotag.conf` являются источниками этих атрибутов. Если они следуют шаблону `key=value`, одинаковые ключи могут существовать в обоих файлах, но с разными значениями. При объединении содержимое из `hostautotag.conf` имеет приоритет над содержимым из `hostcustomproperties.conf`, поскольку теги имеют приоритет в случае конфликта имён.

Если конфликт имён возникает при объединении атрибутов ресурсов на разных уровнях, наименьший уровень имеет приоритет.

### Нормализация атрибутов ресурсов

Для обеспечения согласованного и надёжного приёма метрик ко всем соответствующим внутренним ключам и значениям метрик применяется нормализация атрибутов ресурсов. Этот процесс помогает предотвратить отбрасывание метрик из-за недопустимых или неправильно сформированных измерений.

#### Правила для ключей измерений

| Описание правила | Детали |
| --- | --- |
| **Недопустимые символы** | Недопустимый символ или серия недопустимых символов заменяется одним подчёркиванием `_`. Например, `za_$%_` заменяется на `za_`. |
| **Пустые ключи** | Измерения без допустимых символов пропускаются |
| **Ограничение длины ключа** | OneAgent версии 1.317+ Макс. 350 символов (ранее макс. 100 символов) |

#### Правила для значений измерений

| Описание правила | Детали |
| --- | --- |
| **Допустимые символы** | Все не-управляющие символы (ASCII и Unicode) |
| **Управляющие символы** | Не допускаются. Управляющий символ (то есть символ, используемый как инструкция и не отображаемый; например, перенос строки, табуляция) или серия таких символов заменяется одним подчёркиванием `_`. |
| **Ограничение длины значения** | OneAgent версии 1.313+ Макс. 2048 символов (ранее макс. 255 символов) |
| **Значения в кавычках** | Если значение начинается и заканчивается `"`, оно экранируется |

#### Ограничения измерений

Для соответствия текущей спецификации используется определённая иерархия измерений и заданные ограничения для предотвращения предупреждений и отбрасывания метрик, вызванных превышением этих ограничений.

По умолчанию глобальное ограничение измерений равно `100`, а ограничение измерений, определённых пользователем, составляет 40% от глобального ограничения.
