С помощью OpenPipeline вы можете принимать данные на платформу Dynatrace из широкого разнообразия форматов и провайдеров через источники приема. Затем данные маршрутизируются в конвейеры для обработки и сохраняются в бакетах Grail.

Конвейер (Pipeline)

## Область конфигурации

Области конфигурации, такие как логи и события, предоставляют наблюдаемость состояния, производительности и поведения вашей системы, позволяя командам обнаруживать, диагностировать и устранять проблемы. Каждая область конфигурации предлагает уникальную перспективу благодаря своим уникальным характеристикам.

OpenPipeline предоставляет унифицированное решение для настройки приема и обработки данных, обеспечивая при этом гибкость в зависимости от специфики области конфигурации.

## Источники приема

Данные поступают на платформу Dynatrace через различные источники приема, такие как конечные точки API, OneAgent и расширения, которые собирают данные от провайдеров данных. В OpenPipeline они определяются именем и путем (`dt.openpipeline.source`).

После того как записи достигают вашей среды Dynatrace SaaS через источники приема, вы можете маршрутизировать их в конвейер.

Чтобы узнать об источниках приема, доступных в OpenPipeline, см. [Источники приема в OpenPipeline](../reference/api-ingestion-reference.md "Reference ingest sources and APIs for the configuration scopes supported in OpenPipeline.").

### Типы

Источники приема могут быть встроенного, готового или пользовательского типа.

| Технический аспект | Встроенный источник приема | Готовый источник приема | Пользовательский источник приема |
| --- | --- | --- | --- |
| Владелец | OpenPipeline | Расширение | Пользователь или группа пользователей |
| Разрешения | `settings:read` | `settings:read` | `settings:read`, `settings:write` |
| Тип доступа | Только просмотр для всех пользователей | Только просмотр для всех пользователей | Контроль доступа на основе владельца |
| Доступность области конфигурации | Все | События, Логи, Метрики | События (кроме проблем Davis и событий Davis) |
| Вариант маршрутизации | Динамическая | Динамическая, статическая | Динамическая, статическая |
| Предварительная обработка | Не настраивается | Не настраивается | Настраивается |

#### Встроенный источник приема

Встроенные источники приема представляют основные точки входа данных, предоставляемые Dynatrace. Они принадлежат OpenPipeline, что позволяет всем пользователям с разрешениями `settings:read` просматривать их, но не изменять и не удалять. Этот тип всегда присутствует в таблицах OpenPipeline, доступен для всех областей конфигурации и поддерживает только динамическую маршрутизацию.

#### Готовый источник приема

Готовый источник приема автоматически создается при установке [расширения](../../../ingest-from/extensions.md "Learn how to create and manage Dynatrace Extensions."). Помимо источника приема, установка также может генерировать конвейер и активный маршрут. Готовые источники приема принадлежат устанавливающему расширению и доступны только для просмотра всем пользователям с разрешениями `settings:read`. Они остаются неизменными при повторных активациях расширения и удаляются только при удалении расширения. Этот тип доступен для областей конфигурации событий, логов и метрик. Как правило, готовый источник приема маршрутизирует записи в сгенерированный конвейер через статическую маршрутизацию, однако динамическая маршрутизация также поддерживается.

#### Пользовательский источник приема

Пользовательские источники приема создаются пользователями, при этом владение автоматически назначается им. Владельцы имеют доступ на просмотр и редактирование и могут предоставлять доступ на просмотр другим пользователям или группам пользователей в соответствии с разрешениями пользователей. Пользовательские источники приема доступны только в областях конфигурации событий, за исключением проблем Davis и событий Davis. Они поддерживают предварительную обработку и допускают как динамическую, так и статическую маршрутизацию.

### Сценарии использования

* Настройка нескольких конвейеров для одной области конфигурации с использованием инструкций обработки, специфичных для источника приема.

## Предварительная обработка

Необязательная обработка данных, которая происходит после приема и перед маршрутизацией. Настраивая предварительную обработку, вы можете преобразовывать необработанные данные в структурированные форматы сразу после их поступления в вашу среду Dynatrace SaaS. Предварительно обработанные данные затем маршрутизируются в конвейер и доступны для дальнейшей обработки перед сохранением. Обратите внимание, что предварительная обработка доступна только для пользовательских источников приема.

### Сценарии использования

* Применение единой структуры к форматам данных различных провайдеров.

### Лучшие практики

Настройте предварительную обработку, чтобы избежать создания сложных условий сопоставления на основе форматов данных, специфичных для провайдера. Это поможет вам упростить обслуживание маршрутизации и обработки, например, когда вы начинаете принимать данные от нового провайдера.

## Маршрутизация

После приема (и необязательной предварительной обработки) данные маршрутизируются в конвейеры. Порядок маршрутов имеет значение -- позиция в списке определяет порядок выполнения. Если ни один маршрут не соответствует записи, запись маршрутизируется через **маршрут по умолчанию**.

Маршрутизация определяется в соответствии со следующим:

* Область конфигурации: конвейеры специфичны для определенной области конфигурации. Разные области конфигурации маршрутизируются в разные конвейеры.
* Источник приема: вы можете настроить маршрутизацию для каждого источника приема. Несколько источников приема одной области конфигурации могут быть маршрутизированы в один и тот же конвейер.
* Вариант маршрутизации

  + **Динамическая маршрутизация**: данные маршрутизируются на основе условия сопоставления. Условие сопоставления -- это запрос DQL, определяющий набор данных, который вы хотите маршрутизировать.
  + **Статическая маршрутизация**: данные маршрутизируются в определенный конвейер, который остается фиксированным, если его не обновить вручную. Статическая маршрутизация доступна только для [пользовательских источников приема](#ingest-sources).

    Если запись соответствует другому условию, но вы уже настроили статическую маршрутизацию для ее пользовательского источника приема, совпадение пропускается и данные маршрутизируются непосредственно в указанный вами конвейер.

### Сценарии использования

* Маршрутизация данных источника приема в выделенный конвейер.

### Лучшие практики

* Маршрутизируйте как можно больше данных в [пользовательские конвейеры](processing.md#custom-pipeline "Learn the core concepts of Dynatrace OpenPipeline processing."), используя явные условия сопоставления.
* При наличии нескольких вариантов маршрутизации выбирайте в соответствии с размером набора данных. Например, большие наборы данных больше выигрывают от динамической маршрутизации.

## Обработка

Обработка OpenPipeline происходит в конвейерах, содержащих инструкции по структурированию, разделению и хранению ваших данных. Подробнее см. в разделе [Обработка в OpenPipeline](processing.md "Learn the core concepts of Dynatrace OpenPipeline processing.").

## Хранение

База данных Dynatrace Grail предоставляет единое унифицированное решение для хранения всех ваших областей конфигурации. Целевое хранилище OpenPipeline -- это [бакеты Grail](../../grail/organize-data.md "Insights on the Grail data model consisting of buckets, tables, and views."). Вы можете использовать встроенные бакеты и, если это доступно для данной области конфигурации, создавать новые бакеты с пользовательскими сроками хранения. Каждый бакет назначается таблице базы данных DQL. [Назначьте разрешения](../../grail/organize-data/assign-permissions-in-grail.md "Find out how to assign permissions to buckets and tables in Grail.") группам пользователей или отдельным пользователям, чтобы предоставить им доступ к определенным бакетам и таблицам.

По умолчанию OpenPipeline маршрутизирует данные во встроенный конвейер с целевым хранилищем -- встроенный бакет Grail соответствующей области конфигурации. Вы можете настроить назначение хранилища:

* Для пользовательского источника приема, непосредственно определяя его целевое хранилище.
* Для конвейера, на основе условий сопоставления обработки.

Исключения для системных событий

Хранение и срок хранения для системных событий не настраивается.
