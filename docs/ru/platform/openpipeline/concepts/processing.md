# Обработка в OpenPipeline

* Последняя версия Dynatrace
* Объяснение
* Чтение: 4 мин
* Обновлено 28 января 2026 г.

Dynatrace OpenPipeline может преобразовывать входящие данные для лучшего понимания, обработки и анализа. Обработка в OpenPipeline основана на правилах, которые вы создаёте, и предлагает гибкий способ извлечения ценности из необработанных записей.

## Ключевые термины

Источники приёма данных
:   Источник приёма данных для области конфигурации, собирающий данные от поставщика в Dynatrace Platform, например, эндпоинты API или OneAgent.

Маршрутизация
:   Назначение данных конвейеру на основе условий сопоставления (динамическое) или прямого назначения (статическое).

## Конвейер

После приёма и маршрутизации данных обработка OpenPipeline происходит в конвейерах. Каждый конвейер содержит набор инструкций обработки (процессоров), которые выполняются в упорядоченной последовательности этапов и определяют, как Dynatrace должен структурировать, разделять и хранить ваши данные. После обработки записи она отправляется в хранилище и становится доступной для дальнейшего анализа.

Обогащение записей

Обработка основана на доступных записях и не учитывает обогащение записей из внешних сервисов.

### Типы

Конвейеры могут быть пользовательского или встроенного типа.

#### Пользовательский конвейер

Вы можете создавать новые пользовательские конвейеры и модифицировать их для группировки обработки и извлечения данных в соответствии с нужной технологией или командой. Добавляя пользовательские конвейеры для каждой команды, вы можете управлять ими через контроль доступа на основе владельцев.

#### Встроенный конвейер

Встроенные конвейеры предоставляются «из коробки». Они необходимы для работы OpenPipeline и, как правило, не могут быть изменены в рамках OpenPipeline. Доступ к этим конвейерам намеренно ограничен для сохранения их конфигурации.

##### Конвейер по умолчанию

Конвейер **по умолчанию** — это встроенный конвейер, который обрабатывает неназначенные входящие данные для хранения. Он уникален для области конфигурации и гарантирует, что записи будут назначены бакету по умолчанию и не будут случайно отброшены. Он доступен всем пользователям в режиме «только просмотр». Он недоступен для областей конфигурации логов и бизнес-событий, где доступен **классический конвейер**.

Направляйте как можно больше данных в пользовательские конвейеры с помощью явных условий сопоставления; ограничьте использование конвейера по умолчанию мониторингом неназначенных входящих данных.

##### Классический конвейер

**Классический конвейер** — это встроенный конвейер, специфичный для областей конфигурации логов и бизнес-событий. Он представляет и служит точкой входа для правил, которые вы настраиваете в **Settings Classic** для обработки [логов](../../../analyze-explore-automate/logs/lma-classic-log-processing.md "Utilize log processing rules to reshape incoming log data for better understanding, analysis, or further transformation.") или [бизнес-событий](../../../observe/business-observability/bo-event-processing.md "Utilize business event processing rules to reshape incoming business event data for better understanding, analysis, or further transformation.") через классический конвейер.

Если вы используете обработку логов через классический конвейер, перенесите свои правила в пользовательские конвейеры OpenPipeline.

Обработка через классический конвейер и OpenPipeline

Обработка через OpenPipeline предлагает более высокие лимиты и гибкость. В следующей таблице приведены основные технические различия обработки логов через классический конвейер и OpenPipeline.

| Технический аспект | Классический конвейер для логов | OpenPipeline |
| --- | --- | --- |
| Поддержка типов данных | `String` | `String`, `Number` и `Boolean` |
| Лимит поля content | 512 КБ | 10 МБ |
| Чувствительность имён полей к регистру | Нечувствительна к регистру | Чувствительна к регистру[1](#fn-1-1-def) |
| Связывание данных логов с трассировками | Встроенные правила | Автоматически[2](#fn-1-2-def) |
| Парсеры технологий | Встроенные правила | Предустановленные наборы с более широкой поддержкой технологий |
| Язык запросов | LQL, DQL | DQL[3](#fn-1-3-def) |
| Именование измерений метрик | Нет | Да |
| Префикс `log` для ключа метрики | Обязательный | Необязательный |

1

Если логи принимаются через [Log Monitoring API v2 — POST ingest logs](../../../dynatrace-api/environment-api/log-monitoring-v2/post-ingest-logs.md "Push custom logs to Dynatrace via the Log Monitoring API v2."), имена полей автоматически приводятся к нижнему регистру после маршрутизации данных в **классический конвейер**.

2

Обогащение выполняется автоматически, без какого-либо взаимодействия со стороны пользователя.

3

См. также [Конвертация в DQL для логов](../../../analyze-explore-automate/logs/logs-upgrade/lma-dql-conversion.md "Convert your current log monitoring rules to DQL.").

### Сценарии использования

* Подготовка, преобразование и хранение данных в Grail.
* Предоставление доступа к определённым записям.

## Этап

Этап — это фаза в последовательности конвейера, которая сосредоточена на определённой задаче, такой как маскирование, фильтрация, обработка или извлечение. Этапы содержат предопределённый список настраиваемых процессоров, которые определяют задачу этапа.

1

Данные остаются в исходной структурированной форме. Это важно для детального анализа и устранения неполадок, так как гарантирует, что никакая информация не будет потеряна или изменена.

2

Извлечённые метрики отправляются только в Grail, за исключением областей конфигурации событий безопасности (новые) и спанов.

## Процессор

Процессор — это предварительно отформатированная инструкция обработки, сосредоточенная либо на модификации (например, переименование или добавление нового поля), либо на извлечении данных (например, создание события из строки лога или извлечение метрик).

Хотя формат процессора предопределён, он содержит настраиваемый matcher (правило сопоставления) и определение обработки.

* Matcher определяет цель процессора через DQL-запрос. Он сужает доступные данные до конкретного набора, который вы хотите обработать.
* Определение обработки указывает Dynatrace, как преобразовывать или модифицировать данные, отфильтрованные matcher-ом.
