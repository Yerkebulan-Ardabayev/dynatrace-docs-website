# Извлечение метрики для отслеживания системных событий


* Latest Dynatrace
* Руководство
* Чтение: 3 мин.
* Обновлено 23 июня 2025 г.

Dynatrace генерирует системные события для системной активности в вашей среде, например, когда приложение устанавливается, обновляется и удаляется. Вы можете извлечь метрику через OpenPipeline и отслеживать, сколько обновлений приложений произошло в вашей среде.

## Для кого это

Эта статья предназначена для администраторов и пользователей приложений.

## Что вы узнаете

В этой статье вы узнаете, как настроить OpenPipeline для извлечения метрики и мониторинга частоты системных событий.

## Прежде чем начать

Необходимые знания

* [Dynatrace Query Language](../../grail/dynatrace-query-language.md "Как использовать Dynatrace Query Language.")
* [Обработка в OpenPipeline](../concepts/processing.md "Изучите основные концепции обработки в Dynatrace OpenPipeline.")

Предварительные требования

* Среда [Latest Dynatrace](../../../platform.md "Dynatrace — это универсальная платформа, созданная для широкого спектра вариантов использования.")
* [Лицензия Dynatrace](../../../license.md "О Dynatrace Platform Subscription (DPS) — модели лицензирования для всех возможностей Dynatrace.") с возможностями [Обзор метрик на основе Grail (DPS)](../../../license/capabilities/metrics.md "Узнайте, как рассчитывается потребление метрик Dynatrace на основе Grail в модели Dynatrace Platform Subscription.")
* Разрешение пользователя `storage:system:read`

## Шаги

1. Найдите условие для соответствующих записей

1. Перейдите к ![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") **Notebooks** и откройте существующий или новый блокнот.
2. Выберите  > **DQL**.
3. Введите запрос DQL для получения соответствующих записей.

   События жизненного цикла приложений — это события аудита, генерируемые [реестром](https://developer.dynatrace.com/develop/sdks/client-app-engine-registry/) Dynatrace AppEngine. Следующий пример запроса извлекает события жизненного цикла приложений и суммирует результаты по типу события.

   ```
   fetch dt.system.events


   | filter event.kind == "AUDIT_EVENT" AND event.provider == "APP_REGISTRY"


   | summarize by:{event.type}, count()
   ```

   Run in Playground
4. Чтобы отфильтровать только обновления приложений, выберите **app.updated** >  **Filter**.

Вы нашли условие, идентифицирующее обновления приложений (`event.kind == "AUDIT_EVENT" AND event.provider == "APP_REGISTRY" AND event.type == "app.updated"`).

2. Создайте конвейер для извлечения метрик

1. Перейдите к ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline**: > **System events** > **Pipelines**.
2. Чтобы создать новый конвейер, выберите  **Pipeline** и введите имя — например, `App LifeCycle-Updates`.
3. Чтобы настроить извлечение метрик, перейдите к **Metric extraction** >  **Processor** > **Counter metric** и определите процессор, указав:

   * Описательное имя — например, `Frequency`
   * Условие совпадения `event.type == "app.updated"`
   * Ключ новой метрики — например, `apps.updates`
   * Измерения метрики:

     1. Выберите **Pre-defined** и выберите `resource` из [предопределённых измерений](../../../semantic-dictionary/model/dt-system-events.md#audit-event "Познакомьтесь с моделями семантического словаря, связанными с системными событиями."). Это измерение идентифицирует ID приложения, из которого поступает обновление.
     2. Выберите **Custom** и введите:

        + **Field name on record**: Пользовательское измерение, которое дополнительно определяет вашу метрику — например, `details.app.type`.
        + **Dimension name**: Необязательное поле, которое можно использовать в случае вложенных полей или если вы хотите дать измерению псевдоним.
     3. Нажмите **Add**.
4. Нажмите **Save**.

Вы успешно создали новый конвейер для извлечения метрики, содержащей информацию об источнике события и дополнительных деталях приложения. Новый конвейер виден в списке конвейеров.

3. Направьте данные в конвейер

1. Перейдите к ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline**: > **System events** > **Dynamic routing**.
2. Чтобы создать новый маршрут, выберите  **Dynamic route** и укажите:

   * Описательное имя — например, `App LifeCycle`
   * Условие совпадения

     ```
     event.kind == "AUDIT_EVENT" AND event.provider == "APP_REGISTRY"
     ```
   * Конвейер, содержащий инструкции обработки (`App LifeCycle-Updates`)
3. Нажмите **Add**.

Вы успешно создали новый маршрут. Все события жизненного цикла приложений направляются в конвейер, где метрика извлекается только для обновлений приложений. Новый маршрут виден в списке маршрутов.

4. Запросите метрику

1. Перейдите к ![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") **Notebooks** и откройте существующий или новый блокнот.
2. Выберите  > **Metrics** > **Select a metric**.
3. Введите и выберите ключ новой метрики (`dt.system.events.apps.updates`).
4. Нажмите  **Run**.

## Заключение

Вы успешно извлекли метрику для отслеживания частоты обновлений приложений. Все новые события жизненного цикла приложений направляются в новый конвейер. Когда запись является обновлением приложения, новый конвейер извлекает метрику, содержащую ID приложения, из которого поступает обновление, и дополнительные детали приложения. Вы можете запрашивать метрику в Dashboards и Notebooks.
