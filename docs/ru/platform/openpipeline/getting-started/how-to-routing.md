* Актуальная версия Dynatrace

Существует несколько сценариев использования разделения входящих записей на разные потоки, например, отделение данных, не имеющих отношения к производственной среде, или предоставление командам возможности безопасно форматировать только записи приложений и сервисов, которыми они управляют.

В этом руководстве показано, как маршрутизировать журналы нескольких производственных сервисов в выделенные конвейеры.

## Для кого предназначено это руководство

Эта статья предназначена для администраторов, управляющих потоками данных.

## Шаги

1. Определение условия

Вы можете использовать Notebooks ![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") для определения количества необходимых маршрутов и условий их соответствия. Для определения условий значимых производственных сервисов:

1. Получите журналы всех производственных сервисов и сведите записи по атрибуту.

   При получении журналов через DQL в Notebooks вы получаете обзор обнаруженных атрибутов, которые можно использовать для сужения результатов, например, `k8s.deployment.name`.

   Следующий DQL-запрос получает журналы с пространством имён Kubernetes `prod` и сводит результаты по имени развёртывания.

   ```
   fetch logs


   | filter k8s.namespace.name == "prod"


   | summarize by:{k8s.deployment.name}, count()
   ```

   Запустить в Playground
2. Определите пары ключ-значение, идентифицирующие значимые сервисы.

   Пары ключ-значение будут использоваться в качестве условий соответствия. В этом руководстве рассматриваются четыре сервиса, идентифицируемые следующими парами ключ-значение:

   * `k8s.deployment.name == "checkoutservice-*"`
   * `k8s.deployment.name == "currencyservice-*"`
   * `k8s.deployment.name == "emailservice-*"`
   * `k8s.deployment.name == "paymentservice-*"`

Вы определили количество необходимых маршрутов (4) и условия их соответствия (например, `k8s.deployment.name == "checkoutservice-*`).

2. Создание конвейеров

1. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline**: > **Logs** > **Pipelines**.
2. Создайте конвейер для каждого сервиса.

   1. Нажмите **Pipeline** и введите название конвейера — например, `Checkout service pipeline` для сервиса **checkoutservice**.
   2. Нажмите **Save**.

Вы создали пустой конвейер для каждого сервиса.

3. Маршрутизация записей в выделенный конвейер

Создайте маршрут для каждого конвейера.

1. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline**: > **Logs** > **Pipelines** > **Dynamic route**.
2. Задайте условие маршрутизации, указав:

   * Имя
   * Условие соответствия
   * Целевой конвейер

   В следующей таблице приведены примеры условий на основе пространства имён Kubernetes и развёртывания для маршрутизации журналов каждого сервиса в соответствующий конвейер.

Журналы, соответствующие условию маршрутизации, направляются в целевой конвейер. Таблица маршрутизации теперь включает новые маршруты.

## Заключение

Вы настроили маршрутизацию строк журналов для каждого значимого производственного сервиса в выделенный пустой конвейер.

Сообщите командам, что они могут изменять содержимое конвейера и создавать правила обработки для своих сервисов. После загрузки и маршрутизации журналов в один из вновь созданных конвейеров они будут обрабатываться в соответствии с заданными правилами.

Строки журналов производственной среды, не соответствующие ни одному из вновь заданных условий, продолжают направляться в соответствии с маршрутом по умолчанию в классический конвейер. Задайте новые условия для их маршрутизации в другой конвейер.

Чтобы изменить способ обработки журналов, вы можете изменить условие соответствия для исключения или включения других строк журналов, перенаправить строки журналов в другой конвейер обработки или изменить целевое хранилище. Например, можно создать новый конвейер для пропуска хранилища с помощью процессора **No storage assignment** и настроить маршрутизацию всех нерелевантных для производственной среды журналов, соответствующих условию `isNotNull(k8s.namespace.name) and k8s.namespace.name != "prod"`.
