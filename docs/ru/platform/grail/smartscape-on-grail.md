# Smartscape на Grail


* Latest Dynatrace
* Explanation

Smartscape на Grail — это нативное для Grail хранилище, которое автоматически записывает топологические данные, такие как мониторируемые сущности и связи.

Smartscape на Grail использует возможности запросов DQL для:

* Представления сущностей в виде узлов.
* Представления связей в виде рёбер.
* Перехода от одного узла к другому.

Это даёт более глубокое понимание ваших записей данных и позволяет манипулировать и извлекать необходимую информацию для дальнейшего анализа.

## Доступ к Smartscape на Grail

Smartscape на Grail полностью интегрирован в DQL и вводит новые команды и функции, которые вы можете использовать для:

* Запроса узлов и рёбер.
* Обогащения данных мониторинга, таких как логи и спаны, деталями сущностей.
* Навигации по топологии.
* Фильтрации топологии на основе сложных связей.

Стоимость и лицензия DPS

Smartscape на Grail включён в лицензию DPS. Это означает, что данные, возвращаемые запросами Smartscape, не влекут дополнительных затрат.

## Возможности Smartscape на Grail

### Изменяемые данные

Данные, хранящиеся в традиционных бакетах — такие как логи, события, спаны или точки данных метрик — загружаются для определённой метки времени и никогда не изменяются. В отличие от этого, узлы и рёбра Smartscape являются изменяемыми и могут меняться со временем.

### Идентификаторы Smartscape

Узлы Smartscape идентифицируются через ID, представленный типом данных Grail `Smartscape ID`. Идентификаторы Smartscape регулярно обновляются с помощью событий upsert.

Идентификатор Smartscape состоит из типа сущности и 16-символьного числа. Он может быть представлен следующей строкой:

* `ENTITY_TYPE-000000000000007B`

Тип Smartscape полностью совместим со своим строковым представлением, что означает возможность сравнения строки с идентификатором Smartscape.

### Тип

Каждый узел имеет поле типа, описывающее тип сущности и определяющее схему сущности. Семантический словарь содержит схему для каждого типа. По соглашению, типы узлов всегда записываются в верхнем регистре, например `HOST`, `K8S_NAMESPACE` и `AWS_EC2_INSTANCE`.

### Время жизни

Поскольку узлы обновляются регулярно, они не имеют единого поля метки времени. Вместо этого узлы имеют два поля временного диапазона, представляющих время жизни узла:

* `lifetime.start`: время первого обнаружения узла.
* `lifetime.end`: время последнего наблюдения узла.

Поле `lifetime.start` остаётся неизменным, тогда как поле `lifetime.end` постоянно обновляется с каждым входящим событием upsert, пока узел продолжает наблюдаться.

На основе временного диапазона запроса вы увидите только те узлы, время жизни которых пересекается с временным диапазоном запроса. Например, узел с полем `lifetime.end`, содержащим вчерашнюю дату, не будет включён в результаты запроса за последние 2 часа.

### Рёбра

Рёбра — это связи, соединяющие два узла друг с другом. Ребро всегда хранит тип ребра (например, `runs_on`, `calls` или `relates_to`) и два идентификатора двух различных узлов.

Способ хранения ребра на узле зависит от того, является ли ребро статическим или динамическим:

* Статическое: ребро наследует время жизни узла.
* Динамическое: ребро записывается для конкретного момента времени.

Статически хранимые рёбра включаются в результаты, когда временной диапазон запроса пересекается с временем жизни узла, тогда как динамические рёбра включаются в результаты на основе временного диапазона записи ребра.

Статические рёбра в основном основаны на конфигурации, например, когда диск настроен для подключения к определённому хосту. Динамические рёбра обычно основаны на сигналах мониторинга, выявляющих конкретную связь между узлами, например, когда сервис на основе трассировок вызывает другой сервис.

### Ссылки узлов

Все статические рёбра доступны через исходный узел с помощью поля `references`. Хотя это поле скрыто по умолчанию, вы можете отобразить его и использовать в запросах с помощью команды `fieldsAdd`.

Ниже приведены примеры использования поля `references`:

Отображение поля references с помощью fieldsAdd для всех узлов

```
smartscapeNodes "*"


| fieldsAdd references
```

Группировка контейнеров по идентификатору хоста, на котором они работают

```
smartscapeNodes CONTAINER


| summarize by:references[runs_on.host], count()
```

### Хранение данных

Срок хранения данных фиксирован на 35 дней. Это означает, что узлы, чьё поле `lifetime.end` старше 35 дней, будут удалены, включая все статические рёбра. Динамические рёбра также очищаются через 35 дней.

### Связь сигналов с узлами Smartscape

Поля сигнала, начинающиеся с `dt.smartscape.__type__` и содержащие идентификаторы Smartscape, указывают, что сигнал возник в данном узле. Один сигнал может иметь несколько полей `dt.smartscape.__type__`.

Дополнительно сигнал может содержать:

* Поле `dt.smartscape_source.id`, указывающее на точный источник, создавший сигнал.
* Поле `dt.smartscape_source.type`, описывающее тип сущности, создавшей сигнал.

### Теги

Каждый узел имеет специальное поле `tags`, содержащее различные вложенные поля, распознаваемые как теги.

В настоящее время теги могут быть установлены только на уровне источника данных. Это означает, что, например:

* Мониторинг Kubernetes добавляет метки и аннотации в качестве тегов.
* Облачный мониторинг добавляет теги AWS.
* OneAgent добавляет теги агента.

### Контекст безопасности

Как и другие данные, хранящиеся в Grail, узлы Smartscape имеют поле `dt.security_context`, которое может содержать несколько значений.

Поле `dt.security_context` является необязательным полем узла и по умолчанию пустым, поскольку обычные поля разрешений полностью поддерживаются и часто являются достаточными.

### Разрешения полей

Вы можете настраивать и использовать [наборы полей](organize-data/assign-permissions-in-grail.md#grail-permissions-fields "Find out how to assign permissions to buckets and tables in Grail.") с таблицей `smartscape` для применения фильтров ко всем данным, возвращаемым командами Smartscape (`smartscapeNodes` и `smartscapeEdges`). Учтите, что только определённые поля могут быть отфильтрованы для всех типов сущностей одновременно.

Ниже приведён пример конфигурации набора полей:

Определение нового набора полей для фильтрации деталей конфигурации Kubernetes

Чтобы определить новый набор полей для фильтрации деталей конфигурации Kubernetes, хранящихся в поле `k8s.object`, вы можете использовать API со следующим запросом:

```
POST /fieldsets


{


"name": "sensitive-field-k8s-object",


"description": "Make k8s.object sensitive",


"enabled": true,


"scope": "TABLE",


"fields": [


"k8s.object"


],


"tables": [


"smartscape"


]


}
```

Теперь только пользователи с разрешением `ALLOW storage:fieldsets:read WHERE storage:fieldset-name="sensitive-field-k8s-object"` смогут читать данные этого поля.

### Классические идентификаторы сущностей

Типы узлов, используемые в Smartscape на Grail, могут отличаться от классических типов сущностей. Это означает, что идентификатор сущности или узла также может отличаться (например, `CLOUD_APPLICATION_INSTANCE` называется `K8S_POD` в Smartscape на Grail). Во избежание путаницы узлы Smartscape включают поле `id_classic`, содержащее идентификатор сущности соответствующей классической сущности. Классические идентификаторы сущностей доступны для сущностей K8s, базовых сущностей и сервисов.

Если соответствующие классические сущности отсутствуют (например, для облачных платформ), поля `id_classic` на этих узлах отсутствуют.

## Сегменты Smartscape

Узлы Smartscape могут быть отфильтрованы с помощью:

* Сегмента "All data"
* Правил сегмента "Entity"

С помощью сегментов можно фильтровать только узлы Smartscape; рёбра Smartscape не поддерживают фильтрацию сегментами.

* Определение пользовательских типов сущностей, например расширений
