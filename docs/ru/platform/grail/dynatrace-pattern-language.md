# Dynatrace Pattern Language

# Dynatrace Pattern Language

* Последняя версия Dynatrace
* Пояснение
* Обновлено 11 августа 2022 г.

Dynatrace Pattern Language (DPL) -- это язык шаблонов, позволяющий описывать паттерны с помощью сопоставителей (matchers), где сопоставитель -- это мини-паттерн, соответствующий определённому типу данных. Например, `INTEGER` (или `INT`) соответствует целым числам, а `IPADDR` соответствует адресам IPv4 или IPv6. Для обработки всех типов данных доступны различные сопоставители.

## Использование

DPL используется для:

* Разбора поля записи на несколько выходных полей с помощью [команды parse DQL](dynatrace-query-language/commands/extraction-and-parsing-commands.md#parse "Команды извлечения DQL").
* Преобразования входящих данных для лучшего понимания, анализа или дальнейшей обработки в [обработке журналов](../../analyze-explore-automate/logs/lma-log-processing.md "Используйте Dynatrace на базе Grail и DQL для преобразования входящих данных журналов для лучшего понимания, анализа или дальнейшей обработки.").

Для мгновенной обратной связи по эффективности и покрытию ваших паттернов для конкретного сценария использования применяйте [DPL Architect](dynatrace-pattern-language/dpl-architect.md "Извлечение полей с помощью Dynatrace Pattern Language Architect.").

## Структура паттерна

Записанный паттерн интерпретируется слева направо, игнорируя лишние пробелы, переносы строк и комментарии между элементами. Вы можете записать паттерн, описывающий целое число, за которым следует один пробел, IP-адрес и перенос строки, в виде однострочника:

```
INT ' ' IPADDR:ip EOL
```

Или записать тот же паттерн более развёрнуто:

```
/* этот паттерн ожидает целое число и IP-адрес,



разделённые одним пробелом в каждой строке */



INT       //целое число



' '       //за которым следует один пробел



IPADDR:ip //за которым следует адрес IPv4 или IPv6, извлечённый как новое поле `ip`



EOL       //строка завершается символом перевода строки
```

С помощью [DPL Architect](dynatrace-pattern-language/dpl-architect.md "Извлечение полей с помощью Dynatrace Pattern Language Architect.") вы можете использовать [готовые паттерны](dynatrace-pattern-language/dpl-architect.md#preset-patterns "Извлечение полей с помощью Dynatrace Pattern Language Architect.") для наиболее популярных технологий.

## Сопоставление и разбор

Не все элементы данных во входных данных необходимы для анализа. Например, разделители полей или маркеры конца записи в строке журнала полезны только для разбора, но не нужны при выполнении запросов. Все сопоставители в определённом паттерне должны совпасть, но только часть из них может также извлекать (разбирать) данные.

Сопоставитель извлекает данные только тогда, когда ему присвоено [имя экспорта](dynatrace-pattern-language/log-processing-modifiers.md#exportname "Синтаксис DPL для необязательных управляющих элементов (модификаторов).") -- произвольное имя по вашему выбору, которое становится именем поля для использования в выражениях запросов.

В следующем примере паттерн содержит:

* 11 сопоставителей всего
* 4 сопоставителя, извлекающих данные

## Структура сопоставителя

Паттерн DPL состоит из одного или нескольких выражений сопоставителей. Они могут разделяться пробелами, запятыми или переносами строк. Краткий справочник по всем сопоставителям см. на [странице грамматики DPL](dynatrace-pattern-language/log-processing-grammar.md "Полный список грамматики синтаксиса Dynatrace Pattern Language (DPL).").

### Типы сопоставителей

В общем случае выражение сопоставителя может быть одним из следующих:

* Встроенные сопоставители для многих часто используемых типов данных (числовые, временные, сетевые и т.д.)
* [Литеральные выражения](dynatrace-pattern-language/log-processing-literal-expression.md "Синтаксис DPL для работы с литеральными выражениями.")
* [Группы символов](dynatrace-pattern-language/log-processing-lines-strings.md "Синтаксис DPL для работы со строками и строковыми данными.") -- произвольный набор символов для сопоставления (совместимый с регулярными выражениями)
* [Ссылка на другое выражение паттерна](dynatrace-pattern-language/log-processing-macros.md "Синтаксис DPL для создания последовательностей выражений сопоставителей (подпаттернов).") для построения сложных паттернов модульным способом

### Группировка сопоставителей

Выражения сопоставителей могут быть сгруппированы:

* [Группа последовательности](dynatrace-pattern-language/log-processing-sequence-group.md "Синтаксис DPL для группового сопоставления последовательности, где все члены должны совпасть.") -- определяет упорядоченную последовательность сопоставителей
* [Группа альтернатив](dynatrace-pattern-language/log-processing-alternatives-group.md "Синтаксис DPL для группы альтернатив.") -- определяет список сопоставителей для выбора
* [Массив](dynatrace-pattern-language/log-processing-array.md "DPL ARRAY позволяет разбирать повторяющиеся последовательности элементов данных переменной длины.") -- для разбора повторяющихся элементов данных
* [Структура](dynatrace-pattern-language/log-processing-structure.md "Синтаксис DPL для структурирования данных.") -- для захвата разобранных данных как составного типа
* [Группа перечисления](dynatrace-pattern-language/log-processing-enum.md "Конструктор DPL Enum позволяет сопоставлять набор предопределённых строк и преобразовывать их в соответствующие целочисленные значения.") -- для сопоставления строк с числовыми значениями
* [JSON](dynatrace-pattern-language/log-processing-json-object.md "Синтаксис DPL для работы с JSON-объектами.") -- для разбора JSON-структур

### Синтаксис выражения сопоставителя

Выражение сопоставителя состоит из самого сопоставителя и необязательных управляющих элементов (модификаторов), расположенных в следующем порядке (справа налево, квадратные скобки обозначают необязательные элементы):

```
[lookaround] MATCHER_EXPR ['(' configuration ')'] [quantifier] [mod_optional] [':'export_name]
```

Обратите внимание, что пробелы и переносы строк допускаются между элементами. Размещение элементов в другом порядке (например, `mod_optional` после `export_name`) вызовет синтаксическую ошибку.

### Операторы сопоставителей

Выражения сопоставителей имеют `операторы`:

* Некоторые сопоставители допускают [конфигурацию](dynatrace-pattern-language/log-processing-modifiers.md "Синтаксис DPL для необязательных управляющих элементов (модификаторов)."), определяющую их поведение. Например, для сопоставителя временных меток требуется определение ожидаемого формата.
* К большинству сопоставителей и группировок может быть добавлен [квантификатор](dynatrace-pattern-language/log-processing-modifiers.md "Синтаксис DPL для необязательных управляющих элементов (модификаторов)."), указывающий движку, сколько раз следует попытаться выполнить сопоставление.
* Все сопоставители и группировки могут быть объявлены [необязательными](dynatrace-pattern-language/log-processing-modifiers.md "Синтаксис DPL для необязательных управляющих элементов (модификаторов)."). Если элемент в ожидаемой позиции отсутствует, движок выводит NULL в результирующий набор и продолжает со следующим сопоставителем в выражении.
* Всем сопоставителям и группировкам может быть присвоено [имя экспорта](dynatrace-pattern-language/log-processing-modifiers.md "Синтаксис DPL для необязательных управляющих элементов (модификаторов).") -- имя поля, предоставляемого слою запросов.

  Единственная цель сопоставления паттернов -- сделать элементы данных доступными для движка запросов. Однако не все совпавшие элементы нужны для запросов (например, разделители полей в табулированных файлах), поэтому имя экспорта -- это механизм, позволяющий пользователю объявить, какие элементы данных доступны для запросов (одновременно предоставляя имя для полей запроса). Сопоставитель без имени экспорта по-прежнему выполняет свою работу по сопоставлению паттерна, но не виден в запросах.
* Все сопоставители и группировки могут ["просматривать вокруг"](dynatrace-pattern-language/log-processing-modifiers.md "Синтаксис DPL для необязательных управляющих элементов (модификаторов).") (назад или вперёд), главным образом для обеспечения принятия решений (условного ветвления).

## Пример

Ниже приведён пример пошагового сопоставления паттерна.
Предположим, у нас есть запись с разделителями-запятыми (завершаемая символом перевода строки) со следующими полями:

* номер заказа -- целое число
* имя пользователя -- состоит из букв верхнего и нижнего регистра и цифр (но не запятой)
* IPv4-адрес пользователя

```
1,alice,192.168.1.1



2,bob,10.6.24.18



3,mallory,192.168.1.3
```

Эта структура может быть описана следующим выражением паттерна:

```
INT:seq



','



LD:uname



','



IPADDR:user_ip



EOL
```

где:

* строка 1: сопоставитель целых чисел для номера заказа, виден в запросах как 'seq' на строке 2,4 сопоставитель постоянной строки для разделителя полей, не виден в запросах
* строка 3: сопоставитель данных строки, виден в запросах как 'uname'
* строка 5: сопоставитель IP-адреса, виден в запросах как 'user\_ip'
* строка 6: сопоставитель группы символов для символа перевода строки, завершающего запись

Движок сопоставления паттернов пытается применить паттерн, используя сопоставители в том порядке, в котором они определены. Приведённый выше пример начинается с попытки сопоставить `INT:seq` с первым байтом входных данных. Это оказывается `1`. Поскольку он подходит для целочисленного типа, движок переходит к следующему байту.

Далее движок обнаруживает, что байт является запятой. Это не соответствует целому числу, поэтому сопоставитель `INT:seq` завершается преобразованием `1` в целое число, и выбирается следующий сопоставитель в паттерне: `','`. Движок проверяет его для текущей позиции данных и находит совпадение.

Указатель данных перемещается на следующий байт (указывая на первую букву `bob`). Поскольку сопоставитель постоянной строки содержал только один символ, сопоставитель считается завершённым, и движок берёт следующий в паттерне: `[a-zA-Z0-9]*:uname`. [Квантификатор](dynatrace-pattern-language/log-processing-modifiers.md "Синтаксис DPL для необязательных управляющих элементов (модификаторов).") `*` заставляет `[a-zA-Z0-9]*:uname` потреблять переменное количество байтов (ноль или более), поэтому он продолжает сопоставление, пока не найдёт байт, не соответствующий определённым символам. Это происходит на второй запятой (сразу после `bob`), где движок считает сопоставитель `[a-zA-Z0-9]*:uname` завершённым и берёт следующий: `','`. Снова он пытается сопоставить его с байтом в текущей позиции и успешно завершает операцию.

Указатель данных перемещается на следующий байт, указывая на начало `192.168.1.1`. Поскольку `','` завершён, движок берёт `IPV4ADDR:user_ip`. Пробуя его с текущей позиции, совпадение найдено, и указатель данных перемещается вперёд на 11 байтов, теперь указывая на символ новой строки. Движок находит совпадение для него, используя последний сопоставитель в паттерне: `[\n]`.

Теперь указатель данных перемещается на следующий байт, итератор паттерна сбрасывается, и цикл продолжается с попытки применить первый сопоставитель паттерна к текущим данным. Это продолжается до конца входных данных.

Если движок встречает данные, для которых не может найти совпадение, он сбрасывает итератор паттерна, помечает этот байт как несопоставленный и переходит к следующему байту. Это продолжается, пока не будет найдено совпадение или не закончатся данные. В итоге получаются следующие структурированные данные:

| seq | uname | user\_ip |
| --- | --- | --- |
| `1` | `alice` | `192.168.1.1` |
| `2` | `bob` | `10.6.24.18` |
| `3` | `mallory` | `192.168.1.3` |
