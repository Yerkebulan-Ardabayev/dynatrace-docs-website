# Включение данных в сегменты

* Latest Dynatrace
* Explanation
* 5-min read
* Updated on Nov 04, 2025

Сегменты могут логически структурировать данные и служить удобными фильтрами при анализе данных в различных приложениях. Сегменты строятся на основе правил, называемых **Includes** (включениями). Узнайте, как данные различных типов могут быть включены в сегменты.

## Ключевые термины

Include (включение)
:   Единственное правило, ссылающееся на данные, которые должны быть включены в сегмент. Запрос данных, явно не упомянутых ни одним включением выбранного сегмента, приведёт к пустым результатам.

Signal (сигнал)
:   Наблюдаемая точка данных, отправляемая отслеживаемой сущностью. Доступные типы сигналов в Dynatrace: `logs`, `metrics`, `events`, `spans` и другие.

Monitored entity (отслеживаемая сущность)
:   Логический компонент, отслеживаемый Dynatrace, хранящийся в виде узла Smartscape и/или классической сущности.

Smartscape node (узел Smartscape)
:   Объекты топологии узлов, аналогичные сущностям в кластере Dynatrace. Узлы представляют собой совокупность всех видов сущностей независимо от их типа.

Classic entity (классическая сущность)
:   Сущность типа «приложение», «сервис», «процесс», «хост» или «центр обработки данных», хранящаяся в кластере Dynatrace (например, `dt.entity.host`, `dt.entity.service`, `dt.entity.kubernetes_cluster`). Классические сущности привязаны к типу.

## Включения

Сегменты строятся поэтапно с помощью включений, шаг за шагом расширяя область охвата сегмента. Включения ссылаются на определённый тип данных и определяются условием фильтрации для данных этого типа.

В исходном состоянии, логически говоря, сегмент можно считать пустым. Чтобы использовать сегмент для фильтрации, необходимо включить в него данные.

![An example of segments missing data](https://dt-cdn.net/images/segments-includes-new-2076-89db100a5d.png)

В следующем примере были добавлены два блока включений для включения журналов и метрик по заданным условиям. Это иллюстрирует, как включения поэтапно расширяют область охвата сегмента.

![segments query](https://dt-cdn.net/images/segments-4-1478-162bffb583.png)

Порядок блоков включений не имеет значения

Блоки включений объединяются оператором ИЛИ (OR).

Вы можете перетаскивать блоки включений вверх и вниз в веб-интерфейсе, но это влияет только на порядок их отображения в интерфейсе и не влияет на то, какие данные включаются в сегмент.

Логические условия внутри одного включения могут быть, конечно, более сложными, чем простые условия проверки на равенство, как показано выше.

![segments query](https://dt-cdn.net/images/segments-5-1498-f099426644.png)

Условия включений сегмента вычисляются во время запроса, непосредственно влияя на его производительность. Обязательно рассмотрите использование [![OpenPipeline](https://cdn.bfldr.com/B686QPH3/at/rp4vgwhpjx5rv6rm53mk88cc/OpenPipeline.svg?auto=webp&width=72&height=72 "OpenPipeline") OpenPipeline](../../../platform/openpipeline.md "Scale Dynatrace platform data handling with Dynatrace OpenPipeline.") для обработки полей и упрощения сложных условий.

## Типы данных

### Сигналы

Сбор сигналов наблюдаемости является основой Dynatrace, поэтому необходимо понимать их схему и строить сегменты с учётом формы этих сигналов.

Сигналы могут быть включены в сегменты различными способами. Для лучшей эффективности результирующих выражений фильтрации всегда старайтесь ссылаться на них напрямую.

![Segments: data types example: three include blocks added to include logs, metrics, and events](https://dt-cdn.net/images/data-types-1-2248-556d6370c4.png)

Конфигурации, подобные приведённой выше, можно выразить более элегантно с помощью параметра **All data types**:

![Segments: data types example: using the "All data types" option as a more elegant solution](https://dt-cdn.net/images/data-types-5-2248-9558f6aca9.png)

Поскольку контекст имеет значение, а уникальная модель отслеживаемых сущностей в Dynatrace предоставляет альтернативный способ работы с сигналами наблюдаемости, существует второй вариант включения сигналов через связи «сущность — сигнал».

#### Связи «сущность — сущность»

Связи сущностей в сегментах поддерживаются только для обратной совместимости с классическими сущностями.

Ещё одно преимущество сегментов в отношении отслеживаемых сущностей — возможность использовать связи между ними.

Хотя работа с сущностями, хранящимися в виде узлов Smartscape в Grail, имеет множество преимуществ, иногда необходимо строить сегменты для классических сущностей. Например, это позволяет создавать сегменты, включающие рабочие нагрузки Kubernetes или другие отслеживаемые сущности с более высокой кардинальностью, отфильтрованные по связанным с ними кластерам Kubernetes.

![segments entity relationships](https://dt-cdn.net/images/segments-entities-relationships-2842-fb9d9bec55.png)

Сегменты допускают только один шаг обхода связей. Однако можно настроить несколько параллельных связей одной и той же исходной сущности.
