# Переменные в сегментах


* Последняя версия Dynatrace
* Пояснение
* Время чтения: 3 мин
* Обновлено 06 окт. 2025 г.

Сегменты поддерживают переменные для формирования динамических условий блоков включения. Это позволяет избегать дублирования сегментов, охватывая схожие экземпляры в одном сегменте.

## Ключевые термины

Одноразовый сегмент
:   Сегмент, настроенный со статическими условиями для разового сценария.

Динамический сегмент
:   Сегмент, настроенный с динамическими условиями с использованием переменных.

Запрос переменной
:   DQL-запрос, используемый для заполнения значений переменных.

## Использование переменных

В следующем примере переменная `$bucket` имеет значение `default_logs`, поэтому оператор включения, следующий за ней, разрешается в `dt.system.bucket = "default_logs"`.

![segments log buckets](https://dt-cdn.net/images/segments-10-2142-0d7af74bca.png)

## Конфигурация переменных

Значения переменных можно настраивать динамически или статически. В следующем примере DQL-запрос используется для получения динамического списка бакетов Grail, содержащих логи.

![segments variable configuration](https://dt-cdn.net/images/segments-11-2142-785fbf7a08.png)

Аналогичного результата можно достичь с помощью [команды data](../../../platform/grail/dynatrace-query-language/commands/data-source-commands.md#data "Команды источников данных DQL") в DQL для формирования статического списка значений, как показано ниже.

```
data


record(bucket="custom_sen_high_kubernetes_istio_network_logs"),


record(bucket="custom_sen_low_logs_platform_service_shared"),


record(bucket="custom_sen_low_query_frontend_dql_logs")
```

Выбор между динамическим и статическим формированием списков значений переменных зависит от предпочтений и конкретных требований.

Не следует воспроизводить поиск сущностей с помощью переменных сегментов, фильтрующих по идентификаторам сущностей. Переменные следует использовать для перечисления значений измерения, по которому смоделирован сегмент, например `aws-region`.

### Первичные и вторичные переменные

Переменные можно использовать для моделирования сегмента по одному измерению. Это единственное измерение будет **первичной переменной** и определяется первым столбцом результирующего набора запроса переменной. Первичные переменные формируют список предлагаемых значений при использовании сегмента для анализа данных.

Кроме того, **вторичные переменные** можно использовать для учёта вариаций необходимых условий в блоках включения сегментов. Думайте о вторичных переменных как о дополнительных столбцах результирующего набора запроса переменной. Вторичные переменные строго привязаны к выбору значения первичной переменной при использовании сегмента для анализа данных. Это можно представить как выбор полной строки со всеми её столбцами.

Следующий пример создаёт первичную переменную `$name` и вторичную переменную `$key`.

```
data


record(name = "istio_network", key="custom_sen_high_kubernetes_istio_network_logs"),


record(name = "platform_service", key="custom_sen_low_logs_platform_service_shared"),


record(name = "dql_query_frontend", key="custom_sen_low_query_frontend_dql_logs")
```

### Переменные и подстановочные знаки

Ни имена переменных, ни значения переменных не могут содержать символ подстановки `*`. Однако в условиях блоков включения сегментов символ подстановки может следовать за именем переменной для формирования условия `starts-with` (начинается с), например `k8s.cluster.name = $cluster*`.

## Анализ данных с использованием динамических сегментов

Сегменты, настроенные для использования переменных, при использовании для анализа данных в приложениях представляют дополнительный шаг выбора. В нашем примере этот дополнительный шаг позволяет анализировать логи, выбирая один или несколько бакетов из списка.

![segment selection](https://dt-cdn.net/images/segments-12-2142-b2a14173b1.png)

### Выполнение запросов переменных

При использовании динамических сегментов для анализа данных запрос переменной выполняется в контексте активного пользователя. Поэтому пользователям этого сегмента потребуются соответствующие разрешения для доступа к данным, запрашиваемым для заполнения списка значений. Если у пользователей отсутствуют необходимые разрешения, список значений для выбора будет пустым.

Поскольку запросы переменных выполняются при явном взаимодействии с пользователем, они не окажут существенного влияния на потребление лицензии. Тем не менее рекомендуется избегать запросов больших объёмов данных, например `logs`.
