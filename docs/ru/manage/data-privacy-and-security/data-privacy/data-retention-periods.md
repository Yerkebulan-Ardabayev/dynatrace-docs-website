# Сроки хранения данных

# Сроки хранения данных

* Последняя версия Dynatrace
* Чтение: 8 мин
* Обновлено 28 января 2026 г.

Dynatrace хранит различные типы данных мониторинга из ваших сред. Данные мониторинга хранятся в кластере Dynatrace. В следующей таблице показаны общие сроки хранения для данных сервисов (распределённые трассировки), мониторинга реальных пользователей (действия и сессии пользователей), синтетических мониторов, управления и анализа журналов, а также временных рядов метрик.

## Пробные аккаунты

После истечения 15-дневного пробного аккаунта Dynatrace продолжает хранить данные мониторинга аккаунта в течение 30 дней, чтобы гарантировать отсутствие потери данных.

## Приобретённые аккаунты

Для активных аккаунтов Dynatrace по умолчанию установлены следующие сроки хранения:

## Распределённая трассировка на базе Grail

С помощью [распределённой трассировки на базе Grail](/docs/observe/application-observability/distributed-tracing "Трассировка и анализ в реальном времени высокораспределённых систем с помощью Grail.") вы можете загружать, обрабатывать, хранить и анализировать данные трассировок, хранящихся в хранилище данных Grail в SaaS-средах.

При хранении в Grail вам не нужно беспокоиться об управлении производительностью хранилища данных, доступностью или свободным пространством. Выберите желаемый срок хранения для ваших трассировок в [конфигурации корзины](/docs/observe/application-observability/distributed-tracing/storage "Управление хранением данных и сроками хранения для распределённой трассировки на базе Grail."). Для корзин спанов доступный срок хранения составляет от 10 дней до 10 лет с дополнительной неделей.

## Распределённые трассировки Classic

Dynatrace хранит полные данные каждой транзакции в течение 10 дней. Это позволяет анализировать отдельные транзакции и получать все детали, доступные при вашей [инструментации](/docs/observe/application-observability/distributed-traces "Обеспечьте наблюдаемость высокораспределённых облачных архитектур для эффективной трассировки и анализа транзакций в реальном времени.").

Для пробных пользователей применяется дополнительное ограничение на объём хранилища, что может привести к сокращению сроков хранения.

### Аналитика на уровне кода

Аналитика на уровне кода доступна при инструментации OneAgent в течение 10 дней.

По истечении 10 дней данные сессий оптимизируются для агрегированных представлений. Неагрегированные и агрегированные данные на уровне кода дают сопоставимые результаты для длительных временных периодов, тогда как для коротких периодов могут наблюдаться различия.

## Сервисы Classic: Запросы и атрибуты запросов

Краткосрочное хранение данных, связанных с метриками сервисов, используемых в [многомерном анализе](/docs/observe/application-observability/multidimensional-analysis "Настройте представление многомерного анализа и сохраните его как вычисляемую метрику.") и [построении графиков запросов](/docs/observe/application-observability/services-classic#charts "Узнайте о классическом мониторинге сервисов Dynatrace"). Эти данные доступны в течение 35 дней со следующими уровнями гранулярности интервалов:

| Временной период | Гранулярность интервала |
| --- | --- |
| Менее 20 минут | 10 секунд |
| 20-40 минут | 20 секунд |
| 40-60 минут | 30 секунд |
| Более 1 часа | 1 минута |

Анализ на коротком временном периоде обращается к данным на уровне кода, доступным в течение 10 дней.

По истечении 10 дней данные сессий оптимизируются для агрегированных представлений. Неагрегированные и агрегированные данные на уровне кода дают сопоставимые результаты для длительных временных периодов, тогда как для коротких периодов могут наблюдаться различия.

## RUM Classic: Данные действий пользователей

Агрегированные метрики действий пользователей, используемые в таблицах **Top user actions** и **Top JavaScript errors**, доступны в течение 35 дней. По истечении 10 дней данные действий пользователей оптимизируются для агрегированных представлений, и некоторые отдельные действия пользователей становятся недоступными для индивидуального анализа. Однако выборка достаточно велика для статистически корректных агрегаций.

Для [ключевых действий пользователей](/docs/observe/digital-experience/rum-concepts/user-actions#key-user-actions "Узнайте, что такое действия пользователей и как они помогают понять поведение пользователей.") необработанные данные действий также хранятся в течение 35 дней. Срок хранения данных временных рядов для ключевых действий пользователей совпадает со сроками хранения [метрик временных рядов](/docs/manage/data-privacy-and-security/data-privacy/data-retention-periods#metrics-classic "Сроки хранения различных типов данных.").

## RUM Classic: Сессии пользователей

Включает данные Session Replay. Все данные сессий пользователей хранятся в течение 35 дней. Обратите внимание, что данные каскадного анализа и ошибок JavaScript хранятся вместе с [аналитикой на уровне кода и ошибками распределённых трассировок](#purepath).

## RUM Classic: Сбои мобильных приложений

Включает все данные о сбоях и стек-трейсы мобильных и пользовательских приложений. Данные хранятся в течение 35 дней.

Примечание

Данные о сбоях, отображаемые на странице подробной информации о приложении, могут отличаться от данных на странице статистики. Для этих страниц данные извлекаются из разных хранилищ с разными сроками хранения.

## RUM Classic: Session Replay

Минимальный размер требуемого объёма хранилища Session Replay полностью зависит от нагрузки. Максимальный размер не требуется.

Для данных Session Replay используется выделенный диск.

## Новый интерфейс RUM: Пользовательские события и сессии

Срок хранения по умолчанию как для пользовательских событий (включая взаимодействия пользователей), так и для сессий пользователей составляет 35 дней. Вы можете продлить срок хранения данных, присоединившись к [предварительной версии расширенного хранения для RUM и Synthetic](/docs/whats-new/preview-releases#extended-retention-for-rum-and-synthetic "Узнайте о предварительных версиях и о том, как принять в них участие.").

## Управление и анализ журналов

Управление и анализ журналов позволяет загружать, обрабатывать, хранить и анализировать данные журналов, хранящихся в хранилище данных [Grail](/docs/analyze-explore-automate/logs "Управление и анализ журналов обеспечивает единый подход к контролю и изучению данных журналов в Dynatrace.") в SaaS-средах.

При хранении в Grail вам не нужно беспокоиться об управлении производительностью хранилища данных, доступностью или свободным пространством. Выберите желаемый срок хранения для ваших журналов в [конфигурации корзины](/docs/analyze-explore-automate/logs/lma-bucket-assignment "Данные журналов могут храниться в корзинах хранения данных на основе определённых сроков хранения."). Для корзин журналов доступный срок хранения составляет от 1 дня до 10 лет с дополнительной неделей.

## Мониторинг журналов Classic

Мониторинг журналов Classic позволяет хранить все журналы централизованно во внешнем хранилище. Это делает данные журналов доступными независимо от самих файлов журналов.

Файлы журналов хранятся в Amazon Elastic File System в зоне, где расположена ваша среда Dynatrace. Вам не нужно беспокоиться о производительности хранилища, доступности или свободном пространстве. Расходы на хранение дисков включены в подписку на мониторинг журналов Classic.

## Дампы памяти

Дампы памяти немедленно удаляются с диска после загрузки в ActiveGate. Если загрузка невозможна, дампы памяти размером до 20 ГБ хранятся на диске до 2 часов.

## Метрики на базе Grail

Метрики на базе Grail обеспечивают гранулярность интервала по умолчанию в 1 минуту на протяжении 15 месяцев. Метрики с такой гранулярностью и сроком хранения доступны через приложения платформы, такие как [Дашборды и Notebooks](/docs/analyze-explore-automate/dashboards-and-notebooks "Дашборды и Notebooks"). Подробнее см. [Ограничения метрик](/docs/analyze-explore-automate/metrics/limits "Справка по метрикам на базе Grail").

## Метрики Classic

Для создания дашбордов и доступа через API доступны следующие уровни гранулярности интервалов:

| Временной период | Гранулярность интервала |
| --- | --- |
| 0-14 дней | 1 минута |
| 14-28 дней | 5 минут |
| 28-400 дней | 1 час |
| 400 дней - 5 лет | 1 день |

Для обеспечения точных расчётов метрик временных рядов Dynatrace использует [алгоритм P2](https://dl.acm.org/citation.cfm?id=4378) для динамического расчёта квантилей. Этот алгоритм известен хорошими результатами и хорошо работает со значениями в длинных хвостах распределений. Однако алгоритм агрегации не является ни ассоциативным (`(a + b) + c == a + (b + c)`), ни коммутативным (`a + b + c == c + b + a`). Для некоторых метрик, например времени отклика, это может приводить к различным значениям квантилей при каждом запуске алгоритма или при агрегации данных различными способами, например, когда одна метрика разделена по URL, а другая -- по браузеру.

## Диагностика OneAgent и ActiveGate

Диагностика OneAgent и диагностика ActiveGate -- это необязательные функции, позволяющие собирать и анализировать архивы поддержки на предмет аномалий.

Архивы поддержки создаются Dynatrace OneAgent или Dynatrace ActiveGate и хранятся в Cassandra, где автоматически удаляются через 30 дней. Когда вы разрешаете Dynatrace анализировать проблему, дополнительная копия архива поддержки сохраняется в настроенном бакете AWS S3. Результаты анализа проблемы и архив поддержки также автоматически удаляются из бакета AWS S3 через 30 дней. Dynatrace OneAgent и Dynatrace ActiveGate не сохраняют копии созданных архивов поддержки.

Вы можете удалить проблемы диагностики OneAgent или ActiveGate в любое время. При удалении проблемы соответствующий архив поддержки и отчёт об анализе немедленно удаляются из Cassandra и бакета AWS S3. Результат анализа в Dynatrace Health Control удаляется через 30 дней.

## Данные безопасности на базе Grail

В зависимости от источника данных Dynatrace хранит [события безопасности](/docs/secure/threat-observability/concepts#security-events "Основные концепции, связанные с Threat Observability") в выделенных [корзинах Grail](/docs/platform/grail/organize-data#built-in-grail-buckets "Информация о модели данных Grail, состоящей из корзин, таблиц и представлений.") в течение различных периодов времени:

* События безопасности, сгенерированные Dynatrace из вашей мониторируемой среды, хранятся в корзине `default_securityevents_builtin` в течение **трёх лет**.
* События безопасности, загруженные из сторонних источников, хранятся в корзине `default_securityevents` в течение **одного года**.

## Данные безопасности Classic

### Уязвимости

* Открытые [уязвимости сторонних компонентов](/docs/secure/application-security/vulnerability-analytics/third-party-vulnerabilities "Мониторинг, визуализация, анализ и устранение уязвимостей сторонних компонентов.") хранятся, пока они остаются открытыми, независимо от временного периода.
* Срок хранения устранённых уязвимостей сторонних компонентов зависит от момента их устранения:

  + Если уязвимость устранена до истечения 365 дней с момента первого обнаружения, она удаляется по истечении 365 дней.
  + Если уязвимость устранена после 365 дней с момента первого обнаружения, она удаляется в ближайшую годовщину даты первого обнаружения.

  Примеры:

  | Первое обнаружение | Первое устранение | Повторное открытие | Повторное устранение | Дата удаления |
  | --- | --- | --- | --- | --- |
  | 2022-05-12 | 2023-05-06 |  |  | 2023-05-13 |
  | 2022-05-12 | 2023-08-06 |  |  | 2024-05-13 |
  | 2022-05-12 | 2023-08-06 | 2024-01-01 | 2024-02-08 | 2024-05-13 |

### События

[События эволюции уязвимостей сторонних компонентов](/docs/secure/application-security/vulnerability-analytics/third-party-vulnerabilities/manage-third-party-vulnerabilities#evolution "Мониторинг проблем безопасности ваших сторонних библиотек.") хранятся в течение 365 дней и могут быть запрошены только до момента первого обнаружения уязвимости.
