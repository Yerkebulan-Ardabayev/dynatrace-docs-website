# Предоставление доступа к сущностям с контекстом безопасности


* Latest Dynatrace
* Руководство
* Чтение: 4 мин
* Обновлено 17 сентября 2025 г.

В этой статье показано, как настроить контекст безопасности и предоставить доступ к отслеживаемым сущностям с помощью политик.

Не путайте политики IAM с зонами управления. В отличие от зон управления, политика IAM, настроенная для фильтрации сущностей (`allow storage:entities:read`), не будет фильтровать связанные метрики, логи или трейсы.
Эти фильтры сущностей управляют только тем, какие сущности можно запрашивать через DQL.
Для управления доступом к логам, метрикам и спанам необходимо использовать соответствующие разрешения, такие как `allow storage:logs:read`.

## Для кого это

Это руководство предназначено для администраторов аккаунтов Dynatrace, которые отвечают за создание политик для предоставления пользователям доступа к отслеживаемым сущностям, таким как хосты, через [DQL](../../../platform/grail/dynatrace-query-language.md "Как использовать Dynatrace Query Language.").

## Что вы узнаете

В этой статье вы узнаете:

* Как настроить контекст безопасности для отслеживаемых сущностей.
* Как создать политики для предоставления доступа к данным мониторинга, хранящимся в Grail, в контексте отслеживаемых сущностей.
* Как использовать контекст безопасности в сочетании с политиками безопасности IAM для обеспечения гранулярного доступа к вашим сущностям.

## Шаги

Начнём с настройки контекста безопасности, а затем узнаем, как создать политики для предоставления доступа к данным о сущностях.

1. Настройка контекста безопасности для отслеживаемых сущностей

Для настройки контекста безопасности сущностей вы можете выбрать один из следующих вариантов:

#### Вариант 1: Использование существующего свойства сущности

Если вы уже настроили зоны управления, можно сопоставить зоны управления с контекстом безопасности в зависимости от его типа.

1. Перейдите в **Settings** > **Topology model** > **Grail Security Context**.
2. Разверните соответствующий тип сущности.
3. В разделе **Destination property** выберите поле, которое должно быть сопоставлено с контекстом безопасности. Это может быть `managementZones`, `entity.detected_name` или сам `dt.security_context`.

   ![Grail Security Context ](https://dt-cdn.net/images/123-1532-c8aa238683.webp)

`managementZones` является значением по умолчанию для большинства типов сущностей. При этом варианте контекст безопасности сущностей сопоставляется с классическими зонами управления.
Логи, спаны, метрики и события, работающие на Grail и отправляемые сущностью, не наследуют зоны управления этой сущности.
Это означает, что [контроль доступа](../../../platform/grail/organize-data/assign-permissions-in-grail.md "Узнайте, как назначить разрешения для бакетов и таблиц в Grail.") к логам, спанам, метрикам и событиям, работающим на Grail, должен обрабатываться отдельно.
Этот вариант полезен, если у вас уже настроены зоны управления и вы хотите повторно использовать их для управления доступом к сущностям.

#### Вариант 2: Теги и свойства хоста

Вы можете настроить контекст безопасности, используя свойство хоста `dt.security_context`, заданное с помощью автоматизированных правил или свойств хоста, настроенных через [интерфейс командной строки конфигурации OneAgent](../../../ingest-from/dynatrace-oneagent/oneagent-configuration-via-command-line-interface.md "Узнайте, как выполнять некоторые задачи конфигурации OneAgent без необходимости переустановки OneAgent.").

После установки контекста безопасности на хосте он будет автоматически использоваться для определения контекста безопасности всех логов, спанов, метрик и событий, отправляемых с этого хоста.

Кроме того, он установит контекст безопасности для связанных с хостом сущностей, таких как экземпляры групп процессов и диски.

При использовании этого варианта убедитесь, что контекст безопасности сопоставлен с помощью поля `dt.security_context` для интересующих вас типов сущностей, как описано в Варианте 1 выше.

Чтобы установить контекст безопасности с помощью `oneagentctl`:

```
./oneagentctl --set-host-property=dt.security_context=my-security-context
```

#### Вариант 3: Определение правила извлечения для универсальных типов

Для [универсальных типов сущностей](../../../ingest-from/extend-dynatrace/extend-topology/events-entity-extraction.md#extract-map "Извлечение универсальных сущностей из метаданных событий и сопоставление событий с ними.") можно добавить правило извлечения для `dt.security_context` и получить контекст безопасности из любых данных источника.
Это особенно полезно для расширений, где можно задать `dt.security_context` для каждой конфигурации расширения.

Рассмотрим упрощённый пример с расширением PostgreSQL, предполагая конфигурацию с `dt.security_context`, установленным в `TeamA`.
Для этой конфигурации расширение отправляет точки данных метрик с помощью Metrics API, например:

```
postgres.activity.idle,port=5432,dt.security_context="TeamA",dt.entity.sql:postgres_instance="CUSTOM_DEVICE-..." 42


postgres.activity.idle,port=5432,dt.security_context="TeamA",dt.entity.sql:postgres_instance="CUSTOM_DEVICE-..." 45


postgres.activity.idle,port=5432,dt.security_context="TeamA",dt.entity.sql:postgres_instance="CUSTOM_DEVICE-..." 43
```

На основе этого можно добавить правило извлечения к существующим правилам извлечения расширения PostgreSQL, которое устанавливает `dt.security_context`
в значение измерения `dt.security_context` в точках данных метрик (`TeamA`).

Как и в Варианте 2, убедитесь, что контекст безопасности сопоставлен с помощью поля `dt.security_context` для интересующих вас типов сущностей.
Если типы сущностей не указаны в списке, мы рекомендуем создать эти типы сущностей.

#### Вариант 4: Dynatrace API

Этот API устарел.

Установите контекст безопасности через [API отслеживаемых сущностей](../../../dynatrace-api/environment-api/entity-v2/security-context.md "Создание или удаление контекста безопасности через Dynatrace API.").

2. Создание политики

Разрешения на сущности позволяют определять политики, контролирующие доступ к данным сущностей. В отличие от данных мониторинга, разрешения на сущности позволяют фильтрацию только по полю `dt.security_context`.

Доступ ко всем сущностям настраивается через разрешение `storage:entities:read`, которое поддерживает следующие условия:

* `storage:entity.type`

  тип сущности в формате UPPER_SNAKE_CASE (например, `PROCESS_GROUP_INSTANCE`)
* `storage:dt.security_context`

  контекст безопасности данной сущности. Может быть многозначным полем, и `startsWith` будет выполнять проверку для любого совпадающего значения.

Например, следующая политика предоставляет доступ к данным с контекстом безопасности, установленным в `mySecurityContext`.

```
ALLOW


storage:entities:read


WHERE


storage:dt.security_context = "mySecurityContext";
```
