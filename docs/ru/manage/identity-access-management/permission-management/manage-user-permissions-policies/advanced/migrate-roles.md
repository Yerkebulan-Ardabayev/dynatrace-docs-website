# Переход с разрешений на основе ролей на политики IAM Dynatrace

# Переход с разрешений на основе ролей на политики IAM Dynatrace

* Latest Dynatrace
* Время чтения: 11 мин
* Обновлено 20 августа 2025 г.

Dynatrace версии 1.266+

Это руководство определяет и сравнивает управление доступом на основе ролей (RBAC) и управление доступом на основе атрибутов (ABAC), а также содержит рекомендации, которые следует учитывать при планировании перехода с RBAC на ABAC. Переход в данном контексте означает выражение классических разрешений RBAC, нацеленных на предыдущую версию Dynatrace, в виде операторов в политиках IAM. Как будет объяснено далее, переход предоставляет множество преимуществ.

Руководство в первую очередь предназначено для администраторов учётных записей, желающих воспользоваться возможностями фреймворка ABAC в предыдущей версии Dynatrace.

## Что такое ABAC?

ABAC является предпочтительным общеотраслевым фреймворком управления доступом благодаря своему бизнес-ориентированному и современному подходу к определению политик безопасности через логический, понятный язык.

Он определяет доступ к защищённым ресурсам на основе комбинации атрибутов пользователя, ресурса, действия и контекста. Он использует политики безопасности как механизм определения правил доступа к ресурсам.

### Понимание различий: RBAC и ABAC

Роли RBAC обычно создаются для сопоставления с рабочими функциями, такими как отдел, стаж или рабочее задание. Разрешения RBAC обычно предоставляют доступ к ресурсам по принципу «всё или ничего» на основе заранее определённых статических ролей.

Например, назначение разрешения «View Logs» группе в Dynatrace предоставляет пользователям этой группы возможность просматривать все захваченные логи. Нет возможности дополнительно уточнить их доступ к логам.

* Различная информация о пользователях, ресурсах и контексте не учитывается в классическом RBAC.
* Роли RBAC плохо адаптируются к изменяющимся требованиям к разрешениям доступа и динамической природе атрибутов защищаемых ресурсов.

ABAC, с другой стороны, определяет доступ пользователей путём оценки атрибутов ресурса, данных, пользователя и контекста, а также конкретного запрашиваемого действия. Это обеспечивает гибкий фреймворк разрешений, который обрабатывает изменения в требованиях доступа более динамично, а не полагается на статические роли RBAC. Кроме того, предлагаемая гранулярность управления доступом позволяет разрабатывать как типичные, так и сложные требования к доступу.

Фреймворк разрешений IAM Dynatrace использует эти основные принципы ABAC и дополнительно расширяет их такими возможностями, как [шаблонирование политик](iam-policy-templating.md "Шаблонирование политик"), [границы политик](../iam-policy-boundaries.md "Ограничьте политики безопасности границами политик для предоставления адаптированного доступа пользователям.") и [политики по умолчанию](../../default-policies.md#default-policies "Справочник по политикам по умолчанию Dynatrace"), чтобы сделать реализацию ABAC более гибкой и удобной.

### Реализация ABAC в Dynatrace

Фреймворк разрешений IAM Dynatrace в настоящее время поддерживает как классический RBAC, так и более новый ABAC, использующий политики безопасности для определения разрешений. Политики безопасности — это самостоятельные компоненты фреймворка ABAC, позволяющие объединять один или несколько операторов политики для определения условного доступа к ресурсам Dynatrace.

Разрешения доступа, определённые в политике безопасности, вступают в силу, когда эта политика привязана к группе, что непосредственно контролирует разрешения пользователей этой группы. Для простоты далее в руководстве мы будем называть политики безопасности просто политиками.

Следующая диаграмма отображает взаимосвязь между пользователями, группами, политиками и границами политик.

![Пользователи-Группы-Политики-Границы](https://dt-cdn.net/images/user-group-policy-boundary-812-75ea55d16b.png)

[Политики](../../../permission-management.md "Управление разрешениями") используют атрибуты пользователя, ресурса, данных и контекста для настройки доступа к защищённым ресурсам Dynatrace. Вы можете защищать отдельные ресурсы данных, приложения и сервисы, при необходимости указывая бизнес-специфические условия управления доступом через выразительность [синтаксиса операторов политик](../iam-policystatement-syntax.md "Синтаксис операторов политик IAM.").

Один или несколько операторов политики можно объединить в одну политику. Затем политику можно привязать к одной или нескольким группам пользователей, предоставляя пользователям этих групп доступ к ресурсам, определённый политикой.

Например, политика ABAC Dynatrace ниже предоставляет возможность устанавливать или удалять (действие) пользовательские приложения (ресурс), где (условие) идентификатор приложения (атрибут ресурса) начинается с «custom» (значение атрибута):

```
ALLOW app-engine:apps:install, app-engine:apps:delete WHERE shared:app-id startsWith "custom";
```

Следующий пример использует дополнительную контекстную информацию (время суток) для ограничения возможности запуска приложений только в рабочие часы.

```
ALLOW app-engine:apps:run WHERE shared:app-id = "dynatrace.automations" and global:time-of-day > "09:00+01:00" AND global:time-of-day < "17:00+01:00";
```

Классические роли RBAC также можно выразить в политиках. Например, оператор политики ниже предоставляет доступ к среде только для определённой зоны управления.

```
ALLOW environment:roles:viewer WHERE environment:management-zone="mgmt-eu"
```

Возможность включения классических ролей RBAC в операторы политик является очень важной функцией политик ABAC для облегчения перехода с RBAC.

Истинная мощь фреймворка ABAC Dynatrace заключается в его способности динамически нацеливаться на отдельные или группы ресурсов платформы, при необходимости комбинируя доступные атрибуты для дальнейшего сужения параметров доступа с помощью выразительного синтаксиса операторов политик.

### Преимущества перехода на ABAC

Фреймворк разрешений ABAC Dynatrace предлагает следующие преимущества:

#### Безопасность и соответствие требованиям

* Политики позволяют администраторам применять [принцип минимальных привилегий](https://en.wikipedia.org/wiki/Principle_of_least_privilege)
* Политики обеспечивают лучшую основу для соблюдения стандартов ИТ-соответствия

#### Гибкость

* Политики созданы для адаптации к изменениям и будущим сервисам платформы
* [Политики по умолчанию](../../default-policies.md "Справочник по политикам по умолчанию Dynatrace") управляются Dynatrace и автоматически адаптируются к будущим изменениям платформы
* Несколько операторов политики можно объединить в одну политику, позволяя администраторам создавать политики, соответствующие бизнес-требованиям к управлению доступом

#### Масштабируемость

* Политики позволяют лаконично выражать сложные разрешения
* Границы политик обеспечивают дополнительную оптимизацию и обобщение политик
* [Шаблонирование политик](iam-policy-templating.md "Шаблонирование политик") позволяет параметризировать политики через параметры привязки

#### Гранулярность

* Каждый оператор политики может нацеливаться на один или несколько ресурсов платформы
* Условия `WHERE` могут быть добавлены к операторам для дополнительного ограничения доступа

#### Удобство использования

[Границы политик](../iam-policy-boundaries.md "Ограничьте политики безопасности границами политик для предоставления адаптированного доступа пользователям.") можно повторно использовать в нескольких политиках.

* Политики по умолчанию предоставляют отличную отправную точку для типичных сценариев доступа

#### Ориентация на будущее

* ABAC останется и будет продолжать развиваться и расширяться.

## Планирование перехода с RBAC на ABAC

Следующее руководство по переходу рекомендует стратегию и лучшие практики, которые следует учитывать при планировании и выполнении перехода с RBAC на ABAC.

Если вы используете зоны управления для контроля доступа, это руководство предлагает примеры обращения к зонам управления в политиках, чтобы вы могли удалить соответствующие назначения RBAC. Это руководство не охватывает миграцию использования зон управления в разрешения Grail.

### Рекомендации по переходу

Стратегия перехода, которую вы выберете, будет зависеть от вашей текущей конфигурации Dynatrace.

Общая стратегия:

1. Определите разрешения RBAC, которые в настоящее время настроены для ваших групп.
2. Определите соответствие этих разрешений RBAC разрешениям ABAC, которые можно реализовать в политиках.

   * Вы можете повторять это пакетами, пока не охватите все группы.
   * При необходимости рассмотрите использование [политик по умолчанию](../../default-policies.md "Справочник по политикам по умолчанию Dynatrace") вместо создания пользовательских политик.

Вы можете использовать эти примеры [скриптов Dynatrace Notebook или PowerShell](https://community.dynatrace.com/t5/Dynatrace-tips/RBAC-to-ABAC-migration-helper-scripts-Notebook-and-PowerShell/m-p/257807#M1447) для получения начальной оценки и некоторых рекомендаций. Мы будем использовать вывод скрипта из нашей примерной среды Dynatrace для иллюстрации некоторых важных концепций и лучших практик, которые следует учитывать при планировании перехода с RBAC на ABAC.

Обратите внимание, что любые скрипты автоматизации, которые вы создали с использованием разрешений RBAC, также потребуют проверки на предмет возможных изменений в случае полного перехода на ABAC для управления доступом.

### Оценка RBAC в Dynatrace

План перехода должен начинаться с оценки текущей конфигурации RBAC. Это означает выявление существующих ролей RBAC, назначенных вашим группам. Нас интересуют назначения ролей RBAC, нацеленные на среды или зоны управления.

Выполните следующие шаги для начальной оценки. Вы можете использовать портал Account Management для выполнения шагов или [импортировать](../../../../../analyze-explore-automate/dashboards-and-notebooks/notebooks.md "Анализируйте, визуализируйте и делитесь инсайтами из данных наблюдаемости — всё в одном совместном настраиваемом рабочем пространстве.") и использовать предоставленный блокнот. Альтернативно можно использовать предоставленный скрипт PowerShell.

1. Создайте необходимый клиент OAuth в портале [Account Management](../../../../account-management.md "Управляйте лицензией, учётными записями, внедрением платформы и здоровьем среды Dynatrace.") с разрешениями, указанными в блокноте.
2. Импортируйте предоставленный блокнот **RBAC Assessment** в Dynatrace или, если это удобнее, используйте предоставленный скрипт PowerShell.
3. Запустите скрипт.
4. Результатом будет таблица, подобная приведённому ниже примеру:

| tenant-replay-session-with-masking | xyz:mgmt\_na\_west | management-zone | mgmt\_na\_west | group\_na\_supp | `environment:roles:replay-sessions-with-masking` | Dynatrace Professional |
| tenant-replay-session-without-masking | xyz:mgmt\_na\_west | management-zone | mgmt\_na\_west | group\_na\_supp | `environment:roles:replay-sessions-without-masking` | Dynatrace Professional |

### Переход на ABAC

С уже созданным перечнем назначенных разрешений RBAC вы можете сопоставить разрешения RBAC с разрешениями ABAC. Следует отдавать приоритет существующим [политикам по умолчанию](../../default-policies.md "Справочник по политикам по умолчанию Dynatrace"), когда это возможно, вместо создания пользовательских политик.

Мы используем «group\_na\_supp» для иллюстрации некоторых ключевых лучших практик, которые следует учитывать для конфигурации ABAC.

Эта группа в настоящее время имеет следующие разрешения RBAC, ограниченные двумя зонами управления «mgmt\_na\_east» и «mgmt-na\_west»:

| Имя роли RBAC | Имя роли ABAC |
| --- | --- |
| tenant-replay-session-with-masking | `environment:roles:replay-sessions-with-masking` |
| tenant-replay-session-without-masking | `environment:roles:replay-sessions-without-masking` |
| tenant-manage-settings | `environment:roles:manage-settings` |
| tenant-view-sensitive-request-data | `environment:roles:view-sensitive-request-data` |
| tenant-view-security-problems | `environment:roles:view-security-problems` |
| tenant-manage-security-problems | `environment:roles:manage-security-problems` |
| tenant-log-viewer | `environment:roles:logviewer` |

Если бы мы создавали новую пользовательскую политику для назначения этих разрешений, мы могли бы написать следующий оператор политики:

```
ALLOW  environment:roles:replay-sessions-with-masking, environment:roles:replay-sessions-without-masking, environment:roles:manage-settings, environment:roles:view-sensitive-request-data , environment:roles:manage-security-problems, environment:roles:logviewer



WHERE environment:management-zone startsWith "mgmt_na";
```

Затем мы могли бы привязать эту политику к группе `group_na_supp` и удалить старое назначение роли RBAC из группы. Для дальнейшей оптимизации нового разрешения ABAC можно использовать **Dynatrace Professional** вместо создания пользовательской политики. Поскольку [политики по умолчанию](../../default-policies.md "Справочник по политикам по умолчанию Dynatrace") доступны только для чтения, нам необходим механизм для добавления нашего условия `WHERE`.

Здесь в игру вступают [границы политик](../iam-policy-boundaries.md "Ограничьте политики безопасности границами политик для предоставления адаптированного доступа пользователям."). Мы можем сохранить условие `environment:management-zone startsWith "mgmt\_na";` в границе политики и затем использовать его с нашей группой при привязке группы к политике **Dynatrace Professional**, дополнительно ограничивая это назначение разрешений с помощью созданной границы.

Сначала создадим границу политики. В [Account Management](../../../../account-management.md "Управляйте лицензией, учётными записями, внедрением платформы и здоровьем среды Dynatrace."), в разделе **Identity & Access management** > **Policy management**, мы можем создать границу политики, специфичную для зон управления Северной Америки.

![Границы](https://dt-cdn.net/images/boundaries-468-08355e2a83.png)

Далее назначаем политику по умолчанию **Dynatrace Professional** группе `group_na_supp` и применяем только что созданную границу.

![Назначение разрешения](https://dt-cdn.net/images/permissionassign-936-e6e96537be.png)

Как видно из этого примера, границы политик позволяют вынести ваши бизнес-специфические требования к доступу из политики в отдельную самостоятельную единицу, которую можно многократно применять к различным политикам. Границы политик могут быть назначены одной или нескольким привязкам группа-политика, что делает их повторно используемыми. Комбинация политик по умолчанию с границами политик потенциально может ещё больше упростить конфигурацию разрешений.

Лучшая практика

Используйте [политики по умолчанию](../../default-policies.md "Справочник по политикам по умолчанию Dynatrace"), когда это возможно, и определяйте бизнес-специфические условия доступа в границах политик.

### Когда следует создавать пользовательские политики

Вместо создания новых пользовательских политик администраторам настоятельно рекомендуется в первую очередь использовать политики по умолчанию, которые создаются и управляются Dynatrace и уже содержат необходимые операторы политик для типичных сценариев доступа. При назначении этих политик по умолчанию группам вы можете дополнительно ограничить разрешения с помощью границ политик.

В случаях, когда политики по умолчанию не соответствуют вашим конкретным требованиям к разрешениям или когда вы хотите ограничить их определёнными операторами `DENY`, можно создавать пользовательские политики.

Лучшая практика

Создавайте собственные пользовательские политики только тогда, когда предоставленные политики по умолчанию не соответствуют вашим требованиям или когда вы хотите дополнительно ограничить политики по умолчанию.

### Тестирование и валидация

Мы рекомендуем создавать тестовые группы и пользователей и тестировать отдельно, по возможности в изолированной среде.

Также Enterprise API Dynatrace может предоставить полезную информацию о разрешениях для конкретных пользователей или групп. Например, следующий GET-запрос к `https://api.dynatrace.com` вернёт текущие разрешения для конкретного пользователя:

```
`https://api.dynatrace.com/iam/v1/resolutionaccount/:accountIdHere/effectivepermissions?entityType=user\&entityId=:userIdHere`
```

### Развёртывание

Развёртывайте политики поэтапно, чтобы минимизировать нарушения доступа пользователей. Вы можете выполнять развёртывание пакетами групп или другим способом, более подходящим для вашей конфигурации.

Лучшая практика

Рассмотрите развёртывание изменений пакетами, а не всех сразу.

### Мониторинг и настройка

Отслеживайте разрешения ABAC с течением времени, чтобы определить, не повлияли ли они негативно на доступ пользователей, и корректируйте по мере необходимости.

### Поддержка перехода

Обращайтесь к нашей онлайн-документации и форумам Dynatrace Community для получения дополнительной помощи при переходе с RBAC на ABAC. Также используйте каналы поддержки Dynatrace для более целенаправленной помощи.

### Итоговый контрольный список

1. Составьте перечень существующих ролей RBAC и назначений групп

   * Выявите и запланируйте замену любой пользовательской автоматизации, всё ещё использующей RBAC
2. Разработайте политики ABAC:

   * Используйте политики по умолчанию, когда это уместно
   * Создайте границы политик для фиксации вашей бизнес-логики
3. Определите стратегию перехода

   * Рассмотрите поэтапный подход вместо единовременного, внося изменения несколькими пакетами
4. Протестируйте переход

   * Используйте изолированную среду, если она доступна, для тестирования эффектов новой конфигурации только на ABAC
   * Попробуйте сначала перевести выбранный набор групп
5. Отслеживайте эффекты перехода для обнаружения и исправления ошибок конфигурации
