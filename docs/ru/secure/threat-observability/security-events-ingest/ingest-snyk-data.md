## Начало работы

### Обзор

Интеграция Dynatrace с [Snyk](https://dt-url.net/4h03x92) позволяет унифицировать и контекстуализировать результаты анализа уязвимостей различных инструментов и продуктов DevSecOps, обеспечивая централизованную приоритизацию, визуализацию и автоматизацию обработки результатов безопасности.

Продукты Snyk ([Snyk Code](https://dt-url.net/mr23x60), [Snyk Open Source](https://dt-url.net/7k43x64), [Snyk Container](https://dt-url.net/dy63xzt), [Snyk IaC](https://dt-url.net/6283xq1)) генерируют результаты анализа уязвимостей артефактов разработки, таких как код и контейнеры. Платформа Dynatrace наблюдает за соответствующими сущностями среды выполнения, связанными с этими артефактами. Приём и обогащение результатов анализа уязвимостей помогает пользователям лучше сосредоточиться на наиболее критичных рисках, влияющих на их производственные приложения.

## Активация и настройка

1. В Dynatrace найдите **Snyk** и выберите **Install**.
2. Следуйте инструкциям на экране для настройки расширения.
3. Проверьте конфигурацию, выполнив следующие запросы в [![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") **Notebooks**](/docs/analyze-explore-automate/dashboards-and-notebooks/notebooks "Анализируйте, визуализируйте и делитесь выводами на основе данных наблюдаемости в едином настраиваемом рабочем пространстве."):

   * Для журналов аудита:

     ```
     fetch logs



     | filter log.source=="Snyk"
     ```
   * Для событий обнаружения уязвимостей:

     ```
     fetch security.events



     | filter dt.system.bucket == "default_securityevents"



     | filter event.provider=="Snyk"



     AND event.type=="VULNERABILITY_FINDING"
     ```
   * Для событий сканирования:

     ```
     fetch security.events



     | filter dt.system.bucket == "default_securityevents"



     | filter event.provider=="Snyk"



     AND event.type=="VULNERABILITY_SCAN"
     ```
4. После установки и запуска расширения вы можете получить доступ к нему и управлять им в Dynatrace через ![Extensions](https://dt-cdn.net/images/dynatrace-extensions-256-9cb05e0f55.png "Extensions") **Extensions**. Подробнее см. [О расширениях](/docs/ingest-from/extensions/concepts "Узнайте больше о концепции расширений Dynatrace.").

## Подробности

### Как это работает

![Как это работает](https://dt-cdn.net/images/image-20250210-114410-2266-9b907186a6.png)

Интеграция Dynatrace с Snyk — это [расширение](/docs/ingest-from/extensions "Узнайте, как создавать и управлять расширениями Dynatrace."), работающее на Dynatrace ActiveGate. После включения и настройки расширения Dynatrace для Snyk:

1. Оно периодически обращается к продуктам Snyk и получает новые результаты анализа уязвимостей, сканирования и журналы аудита.
2. Полученные данные принимаются в Dynatrace и сопоставляются с [Семантическим словарём Dynatrace](https://dt-url.net/z1c3xsm).
3. Данные хранятся в бакете `default_securityevents` (подробнее см. [Встроенные бакеты Grail](/docs/platform/grail/organize-data#built-in-grail-buckets "Информация о модели данных Grail, состоящей из бакетов, таблиц и представлений.")).

### Лицензирование и стоимость

Информацию о тарификации см. в разделе [Events powered by Grail](/docs/license/capabilities/events "Узнайте, как рассчитывается потребление событий Dynatrace, основанных на Grail, в рамках модели подписки Dynatrace Platform.").

## Наборы функций

При активации расширения с помощью [конфигурации мониторинга](#monitoring-configuration) вы можете ограничить мониторинг одним из наборов функций. Для корректной работы расширение должно собрать хотя бы одну метрику после активации.

В сильно сегментированных сетях наборы функций могут отражать сегменты вашей среды. Тогда при создании конфигурации мониторинга вы можете выбрать набор функций и соответствующую группу ActiveGate, которая может подключаться к данному сегменту.

Все метрики, не отнесённые ни к одному набору функций, считаются набором по умолчанию и передаются всегда.

Метрика наследует набор функций подгруппы, которая, в свою очередь, наследует набор функций группы. Также набор функций, определённый на уровне метрики, переопределяет набор функций на уровне подгруппы, который, в свою очередь, переопределяет набор функций на уровне группы.

## Часто задаваемые вопросы

### Какая модель данных используется для журналов безопасности и событий, поступающих из Snyk?

* [**События обнаружения уязвимостей**](/docs/semantic-dictionary/model/security-events#vulnerability-finding-events "Ознакомьтесь с моделями Семантического словаря, связанными с событиями безопасности.") хранят индивидуальные результаты обнаружения уязвимостей, полученные от различных продуктов Snyk для затронутых артефактов и компонентов.
* [**События сканирования уязвимостей**](/docs/semantic-dictionary/model/security-events#vulnerability-scan-events "Ознакомьтесь с моделями Семантического словаря, связанными с событиями безопасности.") указывают на покрытие сканирования для отдельных артефактов.
* [**Журналы аудита**](/docs/semantic-dictionary/model/log#audit-logs "Ознакомьтесь с моделями Семантического словаря, связанными с Log Analytics.") представляют журналы активности пользователей в продуктах Snyk.

### Какие дополнительные поля расширения добавляются к основным полям событий, принятых из Snyk?

* Пространство имён `container_image` добавляется для результатов обнаружения контейнеров и сканирований для хранения всей информации, связанной с образом контейнера.
* Пространство имён `snyk` добавляется для извлечения нескольких специфичных для Snyk атрибутов поверх исходного JSON, который хранится в поле `event.original_content`.

  Примеры: `snyk.org.name`, `snyk.project.name`, `snyk.project.tags`, `snyk.target.name`, `snyk.target.reference` и другие.

### Какие проблемы Snyk импортируются в Dynatrace?

* Если расширение настроено на приём данных с интервалом `n` часов, то при каждом запуске приёма все открытые проблемы, относящиеся к проектам, для которых был сделан снимок за последние `n` часов, будут приняты (при первом приёме Dynatrace учитывает все проблемы за предыдущие `m` часов, где `m` — интервал первого приёма, настроенный в конфигурации мониторинга).
* Если за последние `n` часов для одного проекта было сделано несколько снимков, набор результатов будет принят **для каждого снимка**.
* Если для проекта не было сделано снимков, результаты не будут приняты, даже если у проекта есть открытые проблемы.

### Почему количество результатов обнаружения уязвимостей в Dynatrace не совпадает с количеством проблем в Snyk?

Как описано [выше](#imported-issues), проблемы принимаются при каждом запуске снимка проекта, что означает, что если проблема не была отмечена как решённая между запусками снимков, Dynatrace примет её дважды.

Для соответствия Семантическому словарю Dynatrace некоторые проблемы Snyk разделяются на несколько результатов обнаружения уязвимостей при приёме в Dynatrace.

Чтобы подсчитать количество проблем Snyk из данных в Dynatrace, вы можете использовать команду `| dedup snyk.issue.id`.

### Какие типы активов Snyk поддерживаются Dynatrace для контекстуализации среды выполнения?

`CONTAINER_IMAGE`: все результаты обнаружения от Snyk Container, полученные при анализе образов контейнеров, сопоставляются со значением `CONTAINER_IMAGE` в поле `object.type`, и добавляется пространство имён `container_image` с соответствующими полями:

* `container_image.digest` устанавливается в дайджест образа контейнера. Это значение может использоваться для сопоставления с контейнерами среды выполнения.

  Для некоторых контейнеров, например импортированных в Snyk из CLI, дайджест недоступен.
* `container_image.repository` представляет имя репозитория контейнера.
* `container_image.registry` представляет имя реестра контейнера.
* `container_image.tags` представляет теги, установленные на образе контейнера.
* `container_image.id` представляет идентификатор образа контейнера.

Для некоторых образов, например импортированных из CLI, Snyk не сообщает дайджест образа контейнера. По умолчанию для контекстуализации среды выполнения используется `container_image.digest`, но в таких случаях может потребоваться использование `container_image.id`.

### Как нормализуется оценка риска для результатов Snyk?

Dynatrace нормализует уровни серьёзности и оценки риска для всех результатов, принятых через текущую интеграцию. Это помогает вам последовательно приоритизировать результаты независимо от их источника.
Подробнее о работе нормализации см. в разделе [Нормализация серьёзности и оценки](/docs/secure/threat-observability/concepts#normalization "Основные концепции, связанные с наблюдаемостью угроз").

Уровни и оценки риска Dynatrace сопоставляются с оригинальными [уровнями серьёзности и оценками Snyk](https://dt-url.net/ek03whg).

* `dt.security.risk.level` берётся из уровня серьёзности Snyk и сопоставляется с оригинальными значениями в `finding.severity`.
* `dt.security.risk.score` берётся из уровня серьёзности Snyk и сопоставляется со статическими оценками. Оценка CVSS (CVSS v3 или v4 для уязвимостей, обнаруженных после 2024 года), предоставляемая Snyk, доступна в `finding.score`; однако она может не всегда совпадать с указанным уровнем серьёзности.

| `dt.security.risk.level` (сопоставление с `finding.severity`) | `dt.security.risk.score` (сопоставление с `dt.security.risk.level`) |
| --- | --- |
| critical -> CRITICAL | 10.0 |
| high -> HIGH | 8.9 |
| medium -> MEDIUM | 6.9 |
| low -> LOW | 3.9 |
