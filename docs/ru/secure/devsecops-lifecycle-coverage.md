# Покрытие жизненного цикла DevSecOps с Snyk

* Latest Dynatrace
* Приложение
* Обновлено 13 декабря 2023 г.

Объединяя аналитику среды выполнения приложений на базе ИИ Dynatrace с информацией о безопасности до развёртывания от Snyk, ![DevSecOps](https://dt-cdn.net/images/dev-sec-ops-logo-c5208e654c.svg "DevSecOps") **DevSecOps Lifecycle Coverage with Snyk** предоставляет целостное представление всего жизненного цикла DevSecOps, позволяя устранять «слепые зоны», снижать риски безопасности и обеспечивать сквозное согласованное покрытие безопасности от разработки до продакшена.

**Возможности:**

* Просмотр в реальном времени и непрерывная отчётность о покрытии ваших контейнерных рабочих нагрузок, выявление случаев, когда контейнеры попали в продакшен, минуя средства контроля безопасности, и возможность улучшить покрытие, уменьшить количество уязвимых контейнеров и исключить неотсканированные.
* Гибкий подход, позволяющий детализировать вашу среду и сосредоточиться на конкретных частях, которые вас интересуют, чтобы получить именно ту информацию, которая вам нужна.
* Детали контейнеров, охватывающие все уровни критически важных приложений вашей организации.

### Требования Snyk

Для отображения данных сканирования образов контейнеров необходимо настроить подключение к Snyk через [![Workflows](https://dt-cdn.net/images/workflows-1024-b5708f3cf9.webp "Workflows") **Workflows**](https://dt-url.net/wl234nw).

1. Перейдите в **DevSecOps Lifecycle Coverage with Snyk**.
2. На странице **Overview** в разделе **Pre-deployment scans are not running** выберите **Adapt workflow**.
3. В диалоговом окне **Open with** выберите **Create a new workflow with tasks and an optional trigger**. Это перенаправит вас к предварительно настроенному рабочему процессу с рекомендуемыми значениями от Dynatrace.
4. Под узлом **Time interval trigger** выберите **Snyk container image coverage ingest**.
5. В боковой панели выберите **Create a new connection**.
6. В диалоговом окне **Open with** выберите **Open settings**.
7. На странице настроек **Snyk connections** выберите **Add item**, затем введите имя подключения и токен учётной записи сервиса Snyk.
8. Выберите **Save changes**.
9. Перейдите к рабочему процессу и выберите **Run** для проверки подключения, затем выберите **Save** для сохранения рабочего процесса.

## Глобальный обзор покрытия безопасности

Страница **Overview** отображает глобальный обзор покрытия контейнерных рабочих нагрузок, образов контейнеров и работающих контейнеров в вашей среде. Контейнерные рабочие нагрузки — это группы работающих контейнеров, которые вместе реализуют приложение. Работающие контейнеры основаны на образах контейнеров и могут принадлежать различным контейнерным платформам, таким как Kubernetes/OpenShift, Cloud Foundry, Docker и другим.

Отображается следующая информация.

* **N container workloads**:

![container-workloads](https://dt-cdn.net/images/n-container-workloads-1387-a55403f265.png)

* Кольцевая диаграмма с процентом покрытия контейнерных рабочих нагрузок в вашей среде по режиму покрытия (`Fully covered`, `Partially covered`, `Not covered`). Для подробностей наведите курсор на диаграмму.
* Количество контейнерных рабочих нагрузок, обнаруженных в вашей среде, работающих контейнеров, формирующих рабочие нагрузки, и образов контейнеров, на которых основаны работающие контейнеры.
* Покрытие контейнерных рабочих нагрузок на протяжении всего жизненного цикла DevSecOps:

  + **Fully covered**: контейнерная рабочая нагрузка считается полностью покрытой, если все её работающие контейнеры отслеживаются с помощью Application Security и все образы контейнеров, на которых основаны работающие контейнеры, были отсканированы Snyk.
  + **Partially covered**: контейнерная рабочая нагрузка считается частично покрытой, если только некоторые из её работающих контейнеров отслеживаются с помощью Application Security или только некоторые образы контейнеров были отсканированы Snyk.
  + **Not covered**: контейнерная рабочая нагрузка считается непокрытой, если ни один из её работающих контейнеров не отслеживается с помощью Application Security и ни один образ контейнера не был отсканирован Snyk.
* Выберите **View container workloads** для перехода на страницу [**Container workloads**](#workloads) с подробной информацией о покрытии контейнерных рабочих нагрузок.

Инструкции по улучшению покрытия рабочих нагрузок или устранению проблем с покрытием см. в разделе [Улучшение покрытия](#improve-coverage).

* **N container images**:

![container-images](https://dt-cdn.net/images/container-images-681-774dfe03f5.png)

* Кольцевая диаграмма с процентом покрытия образов контейнеров в вашей среде по режиму покрытия (`Covered`, `Not covered`). Для подробностей наведите курсор на диаграмму.
* Покрытие образов контейнеров на этапе до развёртывания (разработки) жизненного цикла DevSecOps:

  + **Covered**: образ контейнера считается покрытым, если он был отсканирован Snyk.
  + **Not covered**: образ контейнера считается непокрытым, если он не был отсканирован Snyk.
* **N running containers**:

![running-containers](https://dt-cdn.net/images/2023-11-28-12-15-46-777-22e713116a.png)

* Кольцевая диаграмма с процентом покрытия работающих контейнеров в вашей среде по режиму покрытия (`Covered`, `Not covered`). Для подробностей наведите курсор на диаграмму.
* Покрытие работающих контейнеров на этапе после развёртывания (среда выполнения) жизненного цикла DevSecOps:

  + **Covered**: работающий контейнер считается покрытым, если все его процессы отслеживаются с помощью Application Security.
  + **Not covered**: работающий контейнер считается непокрытым, если не все его процессы отслеживаются с помощью Application Security.
* Выберите **View details** для перехода на страницу [**Running containers**](#running-containers) с подробной информацией о покрытии работающих контейнеров.

### Детализация вашей среды

Помимо глобального обзора покрытия безопасности в вашей среде, вы можете создавать различные представления (виды), позволяющие детально изучить покрытие безопасности в конкретных частях вашей среды. Представление соответствует той части среды, которую вы хотите детализировать.

Для добавления представления

1. На странице **Overview** выберите **Add new view**.
2. Введите имя представления.
3. В поле **Entity selection** введите запрос сущностей для определения контейнерных рабочих нагрузок, которые вы хотите отслеживать.

Рабочая нагрузка — это сущность типа `CONTAINER_GROUP`. Пример запроса: `type("CONTAINER_GROUP"), entityName.startsWith("MyWorkload")`. Подробнее о написании запросов селектора сущностей см. [Entity selector](https://dt-url.net/oq034sp).

4. (Необязательно) Выберите **Add query** для добавления нового запроса. Вы можете добавить столько запросов, сколько нужно, для формирования представления.
5. Выберите **Add view** для сохранения конфигурации.

Представления отображаются в **Overview** > **Your customized views**.
Пользовательское представление показывает информацию о покрытии безопасности выбранных контейнерных рабочих нагрузок и связанных с ними образов контейнеров и работающих контейнеров.

Пользовательское представление видно только пользователю, который его создал.

Вы можете добавить столько представлений, сколько нужно, и выбранные разделы могут перекрываться. Выбор представления сворачивает или разворачивает его.

* Выберите **View container workloads** для перехода на страницу [**Container workloads**](#workloads) с подробной информацией о покрытии контейнерных рабочих нагрузок в вашем пользовательском представлении.
* Чтобы удалить представление, выберите меню для представления, которое хотите удалить, затем выберите **Delete** и подтвердите.

### Фильтрация контейнерных рабочих нагрузок

В разделе **Filter container workloads** вы можете фильтровать контейнерные рабочие нагрузки во всей среде/в выбранном пользовательском представлении по имени или режиму покрытия (`Fully covered`, `Partially covered`, `Not covered`).

* **Fully covered**: фильтрует контейнерные рабочие нагрузки, где все работающие контейнеры отслеживаются с помощью Application Security и все образы контейнеров были отсканированы Snyk.
* **Partially covered**: фильтрует контейнерные рабочие нагрузки, где только некоторые работающие контейнеры отслеживаются или только некоторые образы контейнеров были отсканированы Snyk.
* **Not covered**: фильтрует контейнерные рабочие нагрузки, где ни один работающий контейнер не отслеживается и ни один образ контейнера не был отсканирован Snyk.

Выберите **Open with...** > **View container group** на любой контейнерной рабочей нагрузке для перехода на страницу обзора выбранной рабочей нагрузки.

## Покрытие работающих контейнеров

Страница **Running containers** предоставляет подробную информацию о покрытии ваших работающих контейнеров. Предоставляются два различных представления в зависимости от способа перехода на эту страницу:

* Чтобы увидеть работающие контейнеры в вашей среде, перейдите в **Overview** > **Global** и в разделе **Running containers** выберите **View details**. Это откроет глобальный обзор страницы **Running containers**.
* Чтобы увидеть работающие контейнеры в выбранном пользовательском представлении, перейдите в **Overview** > **Your customized views** и в разделе **Running containers** выберите **View details** для нужного представления. Это откроет страницу **Running containers**, отфильтрованную по выбранному пользовательскому представлению.

### Фильтрация работающих контейнеров

В разделе **Filter running containers** вы можете фильтровать работающие контейнеры во всей среде/в выбранном пользовательском представлении по имени или режиму покрытия (`Covered`, `Not covered`).

* **Covered**: фильтрует работающие контейнеры, где все процессы отслеживаются с помощью Application Security.
* **Not covered**: фильтрует работающие контейнеры, где не все процессы отслеживаются с помощью Application Security.

Список результатов содержит информацию о:

* Контейнерной рабочей нагрузке, к которой принадлежит работающий контейнер
* Образе контейнера, на котором основан работающий контейнер

## Улучшение покрытия

Для улучшения покрытия контейнерных рабочих нагрузок необходимо улучшить покрытие как для [образов контейнеров](#improve-coverage-images), так и для [работающих контейнеров](#improve-coverage-containers).

### Улучшение покрытия для образов контейнеров

Образ контейнера считается покрытым, когда он отсканирован Snyk. Для улучшения покрытия образов контейнеров:

* Убедитесь, что ваши образы контейнеров [сканируются Snyk](https://dt-url.net/23034xd).
* Убедитесь, что ваши образы контейнеров имеют идентификаторы контейнеров, иначе они не сопоставляются и исключаются.
* В ![Workflows](https://dt-cdn.net/images/workflows-1024-b5708f3cf9.webp "Workflows") **Workflows** проверьте статус [подключения Snyk](#workflow) или запустите рабочий процесс вручную. Подробнее см. [Запуск рабочего процесса](https://dt-url.net/q22349v). Если выполнение не удалось, отредактируйте настройки подключения или настройте новое подключение Snyk.

Как настроить новое подключение Snyk

1. Перейдите в ![Workflows](https://dt-cdn.net/images/workflows-1024-b5708f3cf9.webp "Workflows") **Workflows** и выберите ![Add](https://dt-cdn.net/images/dashboards-app-menu-plus-7e9b7c3547.svg "Add") **New workflow**.
2. Укажите название рабочего процесса (например, `DevSecOps Lifecycle Coverage with Snyk`).
3. В боковой панели выберите **Time interval trigger** и установите запуск каждые `15` минут, `every day`.
4. Выберите ![Add](https://dt-cdn.net/images/dashboards-app-menu-plus-7e9b7c3547.svg "Add") на узле триггера.
5. В боковой панели найдите и выберите действие **DevSecOps Lifecycle Coverage with Snyk**.
6. Выберите **Create a new connection**.
7. В диалоговом окне **Open with** выберите **Open settings**.
8. На странице настроек **Snyk connections** выберите **Add item**, затем введите имя подключения и токен учётной записи сервиса Snyk.
9. Выберите **Save changes**.
10. Перейдите к рабочему процессу и выберите **Run** для проверки подключения, затем выберите **Save** для сохранения рабочего процесса.

### Улучшение покрытия для работающих контейнеров

Работающий контейнер считается покрытым, если все его процессы отслеживаются Dynatrace. Для улучшения покрытия работающих контейнеров:

* Убедитесь, что все [требования Dynatrace](#dt-requirements) выполнены.
* [Включите Third-party Vulnerability Analytics](application-security/vulnerability-analytics.md#enable-tpva "Отслеживайте, визуализируйте, анализируйте и устраняйте уязвимости сторонних библиотек и кода, отслеживайте прогресс устранения и создавайте правила мониторинга.") глобально.
* Включите все [поддерживаемые технологии](application-security/vulnerability-analytics.md#tech-tpv "Отслеживайте, визуализируйте, анализируйте и устраняйте уязвимости сторонних библиотек и кода, отслеживайте прогресс устранения и создавайте правила мониторинга."), которые вы хотите, чтобы Dynatrace покрывал.
* Убедитесь, что ваши [правила мониторинга](application-security/vulnerability-analytics/third-party-vulnerabilities/define-monitoring-rules-tpv.md "Создание, изменение и удаление пользовательских правил мониторинга для процессов, отслеживаемых Dynatrace.") не исключают никаких сущностей.

## Часто задаваемые вопросы

### Обнаруживает ли сканирование контейнеров Snyk уязвимости в библиотеках?

***Обнаруживает ли сканирование контейнеров Snyk уязвимости в библиотеках? В приложении DevSecOps Lifecycle Coverage with Snyk каковы различия между уязвимостями, обнаруженными в образах контейнеров Snyk, и обнаруженными Runtime Vulnerability Analytics?***

![DevSecOps](https://dt-cdn.net/images/dev-sec-ops-logo-c5208e654c.svg "DevSecOps") **DevSecOps Lifecycle Coverage with Snyk** отслеживает покрытие ваших контейнерных рабочих нагрузок на этапах разработки и выполнения. Оно не обнаруживает уязвимости.

* Образ контейнера считается покрытым, если он был отсканирован Snyk. Результаты покрытия отображаются в DevSecOps Lifecycle Coverage with Snyk, а результаты сканирования Snyk отображаются [на стороне Snyk](https://dt-url.net/cn03zyw).
* Работающий контейнер считается покрытым, если все его процессы отслеживаются Application Security. Результаты покрытия отображаются в DevSecOps Lifecycle Coverage with Snyk, а результаты мониторинга Application Security отображаются в ![Vulnerabilities](https://dt-cdn.net/images/vulnerabilities-highresolution-1025-9279da9743.png "Vulnerabilities") **Vulnerabilities**.
