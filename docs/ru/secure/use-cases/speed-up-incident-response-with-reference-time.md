# Ускорение реагирования на инциденты с помощью эталонного времени Dynatrace Investigations


* Latest Dynatrace
* Tutorial
* Опубликовано 13 мая 2025 г.

Эффективное реагирование на инциденты и анализ корневых причин зависят от точности временных данных и контекста. С помощью [эталонного времени](../investigations/concepts.md#reference "Key concepts for using Dynatrace Investigations across security, operations, and performance analysis.") в ![Investigations](https://dt-cdn.net/images/security-investigator-256-93f6c187d9.png "Investigations") **Investigations** вы можете улучшить расследования, основанные на данных, получив более глубокое понимание последовательности событий.
В этой статье рассмотрено, как максимально эффективно использовать привязку к временным меткам в [![Investigations](https://dt-cdn.net/images/security-investigator-256-93f6c187d9.png "Investigations") **Investigations**](../investigations.md "Combine Grail functionalities for evidence-driven investigations, including incident resolution, root cause analysis, and threat hunting.").

## Целевая аудитория

Эта статья предназначена для инженеров по безопасности и надёжности (Site Reliability Engineers), которые участвуют в реагировании на инциденты, анализе корневых причин и расследовании угроз с использованием форензических расследований на основе данных.

## Сценарий

Вы получаете уведомление о том, что в вашей производственной среде наблюдается большое количество ошибок HTTP 503. Вам необходимо быстро выяснить, что вызвало эти ошибки. Вы входите в среду Dynatrace и начинаете расследование. К счастью, ваши коллеги уже начали расследование и выполнили некоторые начальные шаги.

## Прежде чем начать

1. Откройте [расследование в режиме только для чтения в Dynatrace Playground](https://wkf10640.apps.dynatrace.com/ui/apps/dynatrace.security.investigator/share/484db0e6-d7fa-48ec-97ce-6612260e0854).
2. Дублируйте расследование, чтобы продолжить сценарий расследования и иметь возможность выполнять запросы. Инструкции см. в разделе [Дублирование расследований](../investigations/case-sharing.md#duplicate "Share, duplicate, and control access to investigations across teams in Dynatrace Investigations.").

## Начало работы

Следующие шаги проведут вас через процесс анализа корневых причин с использованием эталонного времени как одного из инструментов в ![Investigations](https://dt-cdn.net/images/security-investigator-256-93f6c187d9.png "Investigations") **Investigations**.

1. Анализ диаграммы ответов

Откройте дублированное расследование и выберите синий узел для просмотра графического представления распределения кодов ответов во времени.

Вы увидите, что код ответа 503 резко возрастает на некоторое время, а затем возвращается в нормальное состояние.

![chart](https://dt-cdn.net/images/092f92179d3-3279-04ba0d498c.png)

2. Поиск первого появления ошибки

Как указано в [RFC7231](https://datatracker.ietf.org/doc/html/rfc7231#section-6.6.4), код ответа `503 (Service Unavailable)` указывает на то, что сервер в данный момент не может обработать запрос из-за временной перегрузки или планового обслуживания, что, вероятно, будет устранено через некоторое время. Другими словами, запрос, получивший ответ 503, не является причиной проблемы, а лишь указанием на то, что что-то уже нарушило работу сервиса.

Чтобы выяснить, что вызвало проблемы сервиса, давайте рассмотрим события, произошедшие до первого запроса, получившего ответ 503.

1. В поле ввода запроса для синего узла удалите последнюю строку, содержащую команду `makeTimeseries`.
2. Добавьте следующую команду `filter` для получения только запросов с кодом ответа `503`:

   ```
   | filter response_code == 503
   ```
3. Добавьте следующую команду `summarize` для получения минимального значения временной метки из всех результатов:

   ```
   | summarize min(timestamp)
   ```
4. Выполните запрос.

   Будет создан новый узел с единственным значением в поле **timestamp**.
5. Щёлкните правой кнопкой мыши по временной метке в таблице результатов и выберите **Set as reference time**, чтобы определить её как эталонное время.

   Вы увидите блок эталонного времени в верхней части страницы и новый виртуальный столбец **timestamp\_diff**, представляющий смещение между значением временной метки и вашим эталонным временем. Эталонное время равно `00:00:00.000`, поскольку значение в поле **timestamp** и эталонное время совпадают.

   ![reference-time](https://dt-cdn.net/images/2025-05-12-11-28-20-1423-b9a8d6eb0e.png)

   В одном наборе результатов может быть несколько виртуальных полей смещения; вы можете создать виртуальное поле для любого типа `timestamp`.

3. Поиск предшествующего сообщения об ошибке

Используя эталонное время, давайте найдём запрос, который привёл к тому, что наш сервис стал недоступен.

1. Перейдите ко второму (оранжевому) узлу для просмотра журналов Istio. Обратите внимание, что поле смещения эталонного времени присутствует и в этой таблице результатов.
2. В меню блока эталонного времени выберите **Earlier than**. Это [создаст фильтр по временной метке](../investigations/filter-logs.md#reference "Narrow down data to relevant entries in Investigations.") для получения журналов, записанных до времени эталонного значения.

   ![timestamp filter](https://dt-cdn.net/images/2025-05-12-12-19-49-1920-3d859cdb64.png)
3. Переименуйте `timestamp` в `start_time` в команде фильтра по временной метке.

   **Причина**: При создании эталонного времени для фильтрации используется первое поле timestamp. В данном случае используется поле `timestamp`. Для журналов Istio необходимо использовать поле `start_time`; это означает, что вам нужно вручную изменить фильтр и переименовать `timestamp` в `start_time` в операторе фильтра.
4. Добавьте команду `sort` для сортировки результатов в хронологическом порядке.

   Ваш DQL-запрос должен выглядеть примерно так:

   ```
   fetch logs, timeframe: "05:00:00Z/06:00:00Z"


   | filter k8s.cluster.name == "prod.cupid.cluster"


   | filter k8s.container.name == "istio-proxy"


   | parse content, "json{JSONTIMESTAMP:start_time, INT:response_code}(flat=true)"


   | filter start_time < toTimestamp("2025-05-12T05:32:01.000Z")


   | sort timestamp desc
   ```
5. Выполните запрос.

   Первая запись в таблице результатов содержит HTTP-ответ 500, который может указывать на запрос, вызвавший ошибку в вашей системе.

   ![response code 500](https://dt-cdn.net/images/2025-05-12-13-41-43-1920-26f19ab69f.png)
6. Щёлкните правой кнопкой мыши по временной метке записи и выберите **Replace as reference time**, чтобы заменить текущее эталонное время значением из подозрительного запроса.

4. Анализ журналов приложения

Вы нашли запрос, который привёл к ошибке HTTP 500, в журналах Istio. Однако для понимания того, что произошло с вашим приложением, необходимо углубиться в журналы приложения.

1. Перейдите к первому узлу дерева запросов. Здесь вы можете увидеть все контейнеры и процессы, для которых у вас есть журналы.
2. В поле ввода запроса удалите команду `summarize`.
3. Щёлкните правой кнопкой мыши по значению `heartbeat-matcher-service` в столбце `k8s.container.name`, выберите **Filter** и выполните запрос.

   Результаты показывают все журналы приложения и их относительное расстояние от момента возникновения ошибки (то есть смещение относительно нашего эталонного времени).
4. В меню блока эталонного времени выберите `1 min` в разделе **Show surrounding logs**, а затем выполните запрос.

   Это позволяет внимательно изучить релевантные события и отфильтровать только те, которые находятся вокруг вашего эталонного времени.
5. В заголовке столбца timestamp выберите **Sort ascending** для сортировки результатов в хронологическом порядке.

   Просматривая результаты, вы можете увидеть все запросы и зарегистрированную информацию о времени инцидента. Прокручивая дальше, вы обнаружите, что трассировка стека была записана в журналы в точный момент эталонного времени.

   ![stack trace](https://dt-cdn.net/images/2025-05-12-15-47-12-1920-0726580722.png)

## Заключение

С помощью эталонного времени вы можете эффективно навигировать по журналам, сохраняя временной контекст инцидента. Это помогает отслеживать временное смещение между анализируемыми событиями и моментом возникновения инцидента. Это позволяет обнаруживать релевантные потоки и доказательства — даже из журналов и событий, которые изначально могут казаться не связанными.

Эталонное время интегрирует разнородную информацию, обеспечивая согласованный контекст инцидента по всем точкам данных.
