# Настройки спанов

* Latest Dynatrace
* Чтение: 4 мин
* Обновлено 20 декабря 2023 г.

OneAgent версии 1.215+ Dynatrace версии 1.216+

Dynatrace автоматически захватывает все спаны OpenTracing и OpenTelemetry, но вы можете контролировать и адаптировать то, как спаны OpenTelemetry и OpenTracing объединяются с данными OneAgent в распределенные трассировки PurePath.

Настройки спанов доступны в разделе **Settings** > **Server-side service monitoring**. Вы можете определить правила для:

* Хранения и маскирования только определенных атрибутов -- **Захват атрибутов**
* Исключения определенных спанов -- **Захват спанов**
* Определения спанов, которые должны считаться точками входа -- **Точки входа спанов**
* Включения распространения контекста для определенных спанов -- **Распространение контекста спанов**

Подробности см. в разделах ниже.

## Атрибуты

Сенсор OpenTelemetry Span кодового модуля OneAgent автоматически захватывает все атрибуты OpenTelemetry.
Если вы хотите предотвратить случайное хранение персональных данных, вы можете исключить определенные ключи атрибутов, значения которых не должны сохраняться.
Опуская атрибуты, содержащие персональные данные, вы можете выполнить требования вашей организации по конфиденциальности и контролировать объем хранимых данных мониторинга.

Настройка хранения и маскирования атрибутов для вашей среды

1. Перейдите на соответствующую страницу конфигурации:

   * В Latest Dynatrace перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Collect and capture** > **General monitoring settings** > **OneAgent features**.
   * В Dynatrace Classic перейдите в **Settings** > **Preferences** > **OneAgent network modules & integrations**

2. Выберите **Server-side service monitoring** > **Attribute capturing**.
3. Необязательно: чтобы изменить настройку сохранения атрибутов OpenTelemetry по умолчанию, перейдите в **Preferences**.

   * Чтобы сохранять все атрибуты, кроме указанных в списке **Blocked attributes**, выберите **Allow all attributes**
   * Чтобы блокировать все атрибуты, кроме указанных в списке **Allowed attributes**, выберите **Block all attributes**

   Возможна только одна настройка.
4. Добавьте имя атрибута в список атрибутов.

   1. На странице **Attribute capturing** выберите **Blocked attributes** или **Allowed attributes**.

      Список разрешенных атрибутов Dynatrace рекомендует несколько базовых атрибутов для общего включения, таких как `service.name` или `service.version`. Для удобства Dynatrace поставляется с конфигурацией по умолчанию, которую можно настроить.
   2. Выберите **Add item**, чтобы добавить новый ключ в список атрибутов, и введите ключ.
   3. Выберите **Save changes**.
5. Выполните следующие действия для маскирования значения сохраненного атрибута.

   1. На странице **Attribute capturing** выберите **Attribute data masking**.
   2. Выберите **Add item**, чтобы добавить новый ключ в список маскируемых атрибутов.
   3. Введите ключ сохраненного значения и выберите вариант из выпадающего списка **Masking**. Подробнее о вариантах маскирования см. в разделе [Трассировки OpenTelemetry](../../../manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace.md#otel-traces "Узнайте, какие типы данных конечных пользователей могут быть захвачены во время мониторинга Dynatrace и какие методы доступны для маскирования персональных данных конечных пользователей.").
   4. Выберите **Save changes**.

Затем вы можете найти ключ атрибута на странице **Distributed traces** на [вкладке **Summary**](../../../observe/application-observability/distributed-traces/use-cases/segment-request.md#summary-tab "Повысьте производительность распределенной системы, сегментируя запросы с медленным временем отклика через Service flow и анализируя их распределенные трассировки.").

## Захват спанов

Все обнаруженные спаны OpenTelemetry и OpenTracing захватываются по умолчанию. Это означает, что каждый обнаруженный спан добавляется в распределенные трассировки. Это обеспечивает видимость на уровне кода вместе с атрибутами спанов даже для технологий, не поддерживаемых OneAgent из коробки.

Вы можете создавать [атрибуты запроса](../../../observe/application-observability/services/request-attributes.md "Узнайте, что такое атрибуты запроса и как их использовать на всех уровнях представлений анализа сервисов.") на основе спанов для сегментации распределенных трассировок.

Мы рекомендуем исключать спаны для технологий, поддерживаемых из коробки OneAgent для [Java](../../technology-support.md#java "Технические подробности о поддержке Dynatrace для конкретных платформ и фреймворков разработки.") и [Go](../../technology-support.md#go "Технические подробности о поддержке Dynatrace для конкретных платформ и фреймворков разработки.").

Настройка захвата спанов

1. Перейдите в **Settings** > **Server-side service monitoring** > **Span capturing**.
2. Выберите **Add item**.
3. Введите **Rule name**.
4. В списке **Rule action** решите, хотите ли вы **Ignore** (игнорировать) или **Capture** (захватить) спаны, соответствующие критериям, которые вы собираетесь определить.
5. Выберите **Add item**.
6. Вы можете управлять захватом спанов на основе значений следующих источников:

   * **Attribute** (атрибут)
   * **Instrumentation scope name** (имя области инструментирования), `OpenTelemetry`
   * **Instrumentation scope version** (версия области инструментирования)
   * **Span kind** (тип спана) -- например, `server`
   * **Span name** (имя спана)
7. Выберите **Comparison type** (тип сравнения) -- например, **Contains** или **Equals**.
8. Введите значение для указанного ранее источника.
9. По умолчанию поиск по ключу и значению не чувствителен к регистру. Включите **Case sensitive**, если хотите, чтобы правило учитывало регистр ключа и значения.
10. Выберите **Save changes**.

## Точки входа спанов

По умолчанию:

* Типы спанов OpenTelemetry/OpenTracing `server` и `consumer` автоматически начинают новые распределенные трассировки. Это означает, что все сенсоры OneAgent вносят вклад в эту распределенную трассировку, и для нее создается сервис по умолчанию.
* Типы спанов OpenTelemetry/OpenTracing `client`, `internal` и `producer` не начинают новые распределенные трассировки автоматически.

Вы можете выбрать запуск распределенных трассировок на основе типов спанов `client`, `internal` и `producer`, а также отказаться от этого для типов `server` и `consumer` на основе различных деталей спана.

Настройка точек входа спанов

1. Перейдите в **Settings** > **Server-side service monitoring** > **Span entry points**
2. Выберите **Add item**.
3. Введите **Rule name**.
4. В списке **Rule action** укажите **Create entry point** (создать точку входа) или **Do not create entry point** (не создавать точку входа) для спанов, соответствующих критериям, которые вы собираетесь определить.
5. Выберите **Add item**.
6. Вы можете управлять точками входа спанов на основе значений следующих источников:

   * **Attribute** (атрибут)
   * **Instrumentation scope name** (имя области инструментирования), `OpenTelemetry`
   * **Instrumentation scope version** (версия области инструментирования)
   * **Span kind** (тип спана) -- например, `server`
   * **Span name** (имя спана)
7. Выберите **Comparison type** (тип сравнения) -- например, **Contains** или **Equals**.
8. Введите значение для указанного ранее источника.
9. По умолчанию поиск по ключу и значению не чувствителен к регистру. Включите **Case sensitive**, если хотите, чтобы правило учитывало регистр ключа и значения.
10. Выберите **Save changes**.

## Распространение контекста спанов

Распространение контекста позволяет связывать распределенные трассировки через OpenTelemetry/OpenTracing. Вы можете связывать распределенные трассировки через любой протокол и распространять использование inject и extract на контекст распределенных трассировок Dynatrace PurePath.

Чтобы снизить риск конфликтов распространения контекста со встроенными сенсорами, распространение контекста отключено по умолчанию и ограничено спанами, соответствующими критериям по вашему выбору.

Определение правил для включения распространения контекста для определенных спанов

1. Перейдите в **Settings** > **Server-side service monitoring** > **Span context propagation**
2. Выберите **Add item**.
3. Введите **Rule name**.
4. В списке **Rule action** укажите **Propagate** (распространять) или **Do not propagate** (не распространять) контекст спана для спанов, соответствующих критериям, которые вы собираетесь определить.
5. Выберите **Add item**.
6. Вы можете управлять распространением контекста спанов на основе значений следующих источников:

   * **Attribute** (атрибут)
   * **Instrumentation scope name** (имя области инструментирования), `OpenTelemetry`
   * **Instrumentation scope version** (версия области инструментирования)
   * **Span kind** (тип спана) -- например, `server`
   * **Span name** (имя спана)
7. Выберите **Comparison type** (тип сравнения) -- например, **Contains** или **Equals**.
8. Введите значение для указанного ранее источника.
9. По умолчанию поиск по ключу и значению не чувствителен к регистру. Включите **Case sensitive**, если хотите, чтобы правило учитывало регистр ключа и значения.
10. Выберите **Save changes**.
