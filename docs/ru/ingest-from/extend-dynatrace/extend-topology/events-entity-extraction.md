# Event topology extraction and mapping

# Event topology extraction and mapping

* Latest Dynatrace
* 4-min read
* Updated on Jan 28, 2026

Каждое событие, хранящееся в Dynatrace, сопоставляется с отслеживаемой сущностью, на которую оно влияет. Dynatrace Intelligence использует эти топологические знания при автоматическом анализе первопричин. Примером топологического контекста является скачок загрузки CPU, сопоставленный с хостом, на котором он был обнаружен. Однако после начала приёма ваших собственных источников данных в Dynatrace стандартного топологического сопоставления может оказаться недостаточно: может потребоваться сопоставление событий с вашими пользовательскими сущностями.

[Events API v2](../../../dynatrace-api/environment-api/events-v2.md "Узнайте, что можно делать с Dynatrace Events API v2.") позволяет включать метаданные в события, принимаемые в Dynatrace. Благодаря такому обогащению можно включить топологический контекст непосредственно в событие. Dynatrace может извлечь эту информацию и сопоставить входящее событие с сущностью, которой оно принадлежит.

## Сопоставление с предопределёнными типами сущностей

Чтобы сопоставить событие с сущностью предопределённого типа, укажите его в поле **entitySelector**. Обратите внимание, что событие можно сопоставить только с сущностями, которые были активны в течение последних 24 часов. Если ни одна сущность не соответствует вашему селектору или селектор не указан вовсе, событие сопоставляется с уровнем среды. Дополнительная настройка не требуется. Чтобы узнать синтаксис селектора сущностей, см. [Entity selector](../../../dynatrace-api/environment-api/entity-v2/entity-selector.md "Настройте селектор сущностей для конечных точек Environment API.").

### Шаг 1. Определение пользовательского типа сущности

Сопоставлять события можно только с сущностями существующего пользовательски определённого обобщённого типа. Если нужный тип ещё не определён, создайте его. Сведения о том, как это сделать, см. в разделе [**Define new entity type**](custom-topology.md#define-new-entity-type "Узнайте, как создать пользовательскую топологическую модель, подходящую для ваших данных телеметрии.").

Нельзя извлекать сущности предопределённых типов и повторно сопоставлять с ними события.

### Шаг 2. Предоставление метаданных события

Чтобы иметь возможность извлекать обобщённые сущности из событий, необходимо предоставить соответствующую информацию в конфигурации события. Следующие элементы свойств событий управляют этой функцией. Описания всех полей конфигурации событий можно найти в разделе [**POST an event**](../../../dynatrace-api/environment-api/events-v2/post-event.md "Приём события через Dynatrace API.").

Показать пример JSON

```
{



"eventType": "CUSTOM_ALERT",



"title": "Truck fuel low",



"timeout": 5,



"properties": {



"trucknr": "13",



"dt.event.allow_entity_remapping": "true",



"dt.event.preferred_entity_type": "logistics:truck"



}



}
```

### Шаг 3. Определение правил извлечения

Чтобы назначить правило извлечения обобщённому типу сущности

1. Перейдите в **Settings** > **Topology model** > **Generic types**.
2. Разверните обобщённый тип, с которым нужно сопоставить события.
3. Выберите **Add extraction rule**.
4. В поле **Extracted ID pattern** укажите заполнитель свойства события, содержащего идентификатор сущности. Для примера Easy Shipping это будет `{trucknr}`.
5. Необязательно: в поле **Instance name pattern** укажите понятный шаблон имён для извлечённых сущностей. Используйте заполнители для автоматического создания уникальных имён. Для примера Easy Shipping это может быть `Truck {trucknr}`.
6. Выберите **Add source**.
7. Выберите **Events** в качестве типа источника данных.
8. В поле условия используйте условие `$eq()` со значением типа события. Для примера Easy Shipping это будет `$eq(CUSTOM_ALERT)`.
9. Сохраните изменения.

## Устранение неполадок

Если повторное сопоставление завершилось неудачей, можно получить диагностическую информацию на странице обзора события или через запрос [**GET an event**](../../../dynatrace-api/environment-api/events-v2/get-event.md "Просмотр параметров события через Events API v2."). Найдите поле **Entity remapping failure information** (`dt.event.entity_remapping_failure_info`).
