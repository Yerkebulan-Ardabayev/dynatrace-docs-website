# Определение пользовательской топологии

* Последняя версия Dynatrace
* 5 мин. чтения

Мы продолжаем наш [пример логистики Easy Shipping LTD](/docs/ingest-from/extend-dynatrace/extend-topology#custom-topology-model-in-action "Убедитесь, что все поступающие наблюдения обогащены контекстом и анализируются в контексте отслеживаемых сущностей, к которым они относятся."), в котором грузовики и контейнеры, установленные на грузовиках, отправляют данные телеметрии в Dynatrace.

Пример потока данных соответствует [формату протокола метрик Dynatrace](/docs/ingest-from/extend-dynatrace/extend-metrics/reference/metric-ingestion-protocol "Узнайте, как работает протокол приёма данных для Dynatrace Metrics API.").

```
truck.fuel.usage,trucknr=99,model=mac-granite 17



truck.operation.hours,trucknr=99,model=mac-granite 23766



truck.fuel.total,trucknr=12,model=mac-anthem 234



truck.fuel.usage,trucknr=12,model=mac-anthem 10



truck.operation.hours,trucknr=12,model=mac-anthem 13766



container.temperature,containernr=234321,trucknr=99 40



container.temperature.dev,containernr=234321,trucknr=99 0



container.temperature,containernr=111111,trucknr=12 39



container.temperature.dev,containernr=111111,trucknr=12 2.5
```

### Шаг 1. Определение имени сущности

На основе приведённого выше примера определите новый тип сущности **Truck** в среде Dynatrace.

1. Перейдите в **Settings** > **Topology model** > **Generic types**.
2. Выберите **Add type definition**, чтобы начать определение собственного типа сущности `truck`.
3. Задайте **Type name** — уникальное имя типа, которое будет служить идентификатором сущности.

   Чтобы обеспечить уникальность для всей среды Dynatrace, добавьте к **Type name** префикс с доменом вашей топологии (например, `logistics:truck`).
4. Задайте **Type display name** — понятное отображаемое имя.
5. Задайте **Created by** — имя создателя (человека или расширения).

Пример

![Пользовательская топология](https://dt-cdn.net/images/type-definition-1552-1418014aaf.png)

### Шаг 2. Задание правил извлечения

Научите Dynatrace, как новый тип сущности должен автоматически сопоставляться и как идентифицировать уникальные экземпляры сущностей (грузовики).

1. Выберите **Add extraction rule**.
2. Чтобы задать правило извлечения для идентификатора экземпляра нового типа сущности, установите **Extracted ID pattern** в значение `{trucknr}`.
   Необходимо использовать `{placeholder}`, ссылающийся на имя измерения, чтобы идентификатор экземпляра был уникальным. Значение измерения станет идентификатором экземпляра.
3. Чтобы задать правило извлечения для имени экземпляра нового типа сущности, установите **Instance name pattern** в значение `{trucknr}`.
   Значение измерения станет именем экземпляра.

   Пример

   ![Пользовательская топология](https://dt-cdn.net/images/extraction-rules-1243-69772b09cb.png)
4. Чтобы задать правило извлечения для атрибута сущности, выберите **Add attribute extraction rule**. В этом примере данные каждого грузовика передаются с измерением `model`, которое является атрибутом грузовика:

   * **Attribute key**
     Уникальный идентификатор правила для извлечения значения атрибута.
   * **Attribute display name**
     Можно оставить пустым, чтобы использовать ключ в качестве отображаемого имени.
   * **Attribute value extraction pattern**
     Добавьте заполнитель для измерения `{model}`. Значение измерения станет именем атрибута.

   Пример

   ![Пользовательская топология](https://dt-cdn.net/images/attribute-extraction-1250-c9e3cc8800.png)

### Шаг 3. Повторение для дополнительных сущностей

Повторите описанные выше шаги для определения дополнительных сущностей. В данном случае нам ещё необходимо определить тип сущности `container`, задать правила извлечения (`containernr`) и правила извлечения для его атрибутов (`trucknr`).

После настройки собственных типов домена поверх входящих метрик или потоков логов Dynatrace будет автоматически извлекать и создавать новые экземпляры грузовиков и контейнеров всякий раз, когда грузовик или контейнер начинает отправлять данные телеметрии.

## Определение связей между сущностями

Для завершения определения топологии задайте связь между грузовиком и перевозимым контейнером. Без этой связи измерения, связанные с грузовиком и контейнером, будут отображаться Dynatrace отдельно, без топологической связи.

Для извлечения связи между данным типом сущности и другим типом сущности необходимо найти объединённый поток данных. В нашем примере это измерение температуры контейнера, содержащее оба идентифицирующих значения — номер грузовика и номер контейнера. Например:

```
container.temperature,containernr=111111,trucknr=12 39
```

С помощью объединённого потока данных мы теперь можем определить связь между грузовиком и контейнером.

1. Перейдите в **Settings** > **Topology model** > **Generic relationships**.
2. Выберите **Add relationship definition**.
3. Выберите **Add source**.
4. Определите источник:

   * Задайте **Ingest datasource type** — `Metrics`
   * Задайте **Condition** — `$eq(container.temperature)`
     Это фильтр, который будет сопоставляться с именем объединяющей метрики. В этом примере `$eq(container.temperature)` используется для сопоставления с метрикой `container.temperature`.
   * Задайте **Source type name** — `logistics:container`
   * Задайте **Destination type** — `logistics:truck`
   * Задайте **Type of relationship** — `is child of`
     Это определяет связь между **Source type name** (`logistics:container`) и **Destination type** (`logistics:truck`), что означает, что `logistics:container` `is child of` `logistics:truck`.
5. Выберите **Save changes**.

### Необязательные роли

Использование ролей (таких как клиент и сервер или вызывающий и вызываемый) необязательно. В нашем примере нам не нужно указывать эти роли, так как объединяющий источник данных содержит два разных типа сущностей.

Роли необходимы, если вы хотите связать два экземпляра одного и того же типа сущности с помощью объединяющего измерения.

Примером может служить строка измерения, показывающая время отклика сервиса, вызывающего другой сервис. Например:

```
service.responsetime,caller=service1,callee=service2 39
```

В приведённом выше примере два экземпляра сервиса одного и того же типа должны быть извлечены и объединены связью вызова.

В этом случае одно правило извлечения должно быть идентифицировано с ролью (caller), а другое — с другой ролью (callee).

При извлечении связи роли caller и callee затем могут использоваться для указания направления результирующей связи.
