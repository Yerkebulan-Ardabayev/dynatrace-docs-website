# Протокол загрузки метрик


* Latest Dynatrace
* 6-min read
* Updated on Jul 08, 2025

На этой странице описан протокол загрузки метрик в Dynatrace.

## Синтаксис

Общий синтаксис загрузки метрик следующий:

```
metric.key,dimensions payload
```

### Ключ метрики (обязательный)

Ключ метрики, которую вы отправляете. Он состоит из секций, разделённых точками, например `first.second.third`.

Допустимые символы: строчные и прописные буквы, цифры, дефисы (`-`) и символы подчёркивания (`_`). Применяются следующие ограничения:

* Нелатинские буквы (например, `ö`) не допускаются.
* Ключи метрик не могут начинаться с цифры или дефиса (`-`).
* Секции не могут начинаться с дефиса (`-`).
* Длина ключа должна быть от 3 до 255 символов.

Ключ метрики заканчивается либо на первой запятой (если вы указываете измерения), либо на первом пробеле (если вы опускаете измерения).

К предоставленному ключу может быть автоматически добавлен суффикс в зависимости от полезной нагрузки. Подробнее см. [Полезная нагрузка](#payload).

Избегайте секций с неалфавитными символами. При [запросе метрики](../../../../dynatrace-api/environment-api/metric-v2/get-data-points.md "Read data points of one or multiple metrics via Metrics v2 API.") вам потребуется экранировать эти символы в [селекторе метрик](../../../../dynatrace-api/environment-api/metric-v2/metric-selector.md "Configure the metric selector for the Metric v2 API.").

### Измерение (необязательное)

Если вы хотите опустить измерения, укажите полезную нагрузку сразу после ключа метрики, разделив их пробелом.

Измерения указываются в виде пар `key="value"`. Вы можете указать до 50 измерений, разделённых запятыми (`,`).

Допустимые символы для ключа: строчные буквы, цифры, дефисы (`-`), точки (`.`), двоеточия (`:`) и символы подчёркивания (`_`). Специальные буквы (например, `ö`) не допускаются. Ключ может быть в формате `key.key-section`.

Значение измерения передаётся как строка в кавычках. Если вы хотите передать кавычки (`"`) и/или обратные косые черты (`\`) в значении измерения, экранируйте их обратной косой чертой (`\`). Например:

```
workHours,team="devops\\bugfixing",project="\"product\"_improvement" 1000
```

В настоящее время поддерживается только одно значение измерения для каждого ключа измерения. Если один и тот же ключ измерения указан несколько раз в одной полезной нагрузке (например, `ipaddress="192.168.100.1",ipaddress="10.0.0.1"`), полезная нагрузка является валидной, но принимается только одно значение.

#### Зарезервированные измерения Dynatrace

Ключ `dt.entity.<entity_type>` является зарезервированным ключом измерения Dynatrace, который связывает метрику с мониторируемой сущностью, указанной в качестве значения измерения (например, измерение `dt.entity.host=HOST-06F288EE2A930951` сопоставляет точки данных с хостом с идентификатором **HOST-06F288EE2A930951**).

### Полезная нагрузка (обязательная)

Общий формат полезной нагрузки следующий:

```
format,dataPoint timestamp
```

#### Формат (необязательный)

Вы можете указать полезную нагрузку в двух форматах: калибровочное значение (`gauge`) или значение счётчика (`count`). Укажите формат перед точками данных и отделите его от точек данных запятой (например, `gauge,80.6`).

Ключ метрики может ссылаться только на один тип полезной нагрузки в классических метриках. Таким образом, к предоставленному ключу метрики автоматически добавляется суффикс `.count` для типа полезной нагрузки `count`, если ключ ещё не заканчивается на `.count`. Аналогично, если ключ метрики типа gauge заканчивается на `.count`, к нему добавляется суффикс `.gauge`.

Метрики Grail не применяют суффиксы `.count` и `.gauge`.

gauge

count

Для формата gauge вы можете указать следующие статистические сводки:

* `min`
* `max`
* `sum`
* `count` — количество измерений, включённых в точку данных.

Вы можете опустить формат, если используете полезную нагрузку gauge с одним значением. В этом случае предоставленное значение используется для всех сводок, а count устанавливается в `1`.

Например, полезная нагрузка `80.6` эквивалентна `gauge,min=80.6,max=80.6,sum=80.6,count=1`.

Использование формата count автоматически создаёт новую метрику с ключом `your-metric-key.count`. Для указания значения count необходимо предоставить поле delta: `count,delta=500`.

Точки данных типа `count` представляют собой **дельты** между предыдущей и текущей точками данных. Например, если начальная точка данных имеет значение `500`, а вторая — `1000`, фактическое сохранённое значение на метке времени второй точки данных будет `1500`.

#### Точка данных (обязательная)

Точка данных может включать одно или, в случае формата gauge, несколько измерений. Для нескольких измерений укажите их со статистическими сводками. Необходимо указать все сводки.

Точка данных типа `count` — это **дельта** между предыдущей и текущей точками данных.

#### Метка времени (необязательная)

Формат метки времени — миллисекунды UTC. Допустимый диапазон — от **1 часа** в прошлом до **10 минут** в будущем от текущего момента. Точки данных с метками времени за пределами этого диапазона отклоняются.

Если метка времени не указана, используется **текущая метка времени сервера**.

### Метаданные (необязательные)

Вы можете предоставить пользовательские метаданные метрики через протокол загрузки. Протокол загрузки поддерживает только создание метаданных. Если метаданные для одной и той же метрики указаны несколько раз в полезной нагрузке, используется только первое вхождение. Для просмотра или обновления метаданных используйте [Metrics browser](../../../../analyze-explore-automate/dashboards-classic/metrics-browser.md "Browse metrics with the Dynatrace metrics browser.") или Settings API (чтобы узнать, как составить полезную нагрузку API, см. [Set metric metadata](custom-metric-metadata.md#create "Provide metadata for your custom metric.")).

```
#metric.key <payload-format> dt.meta.<property>="<value>"
```

Укажите `gauge` или `count` вместо `<payload-format>`. Использование формата count автоматически создаст новую метрику с ключом `metric-key.count`.

Доступны следующие свойства. Для указания нескольких свойств разделите их запятой (`,`).

## Примеры

Общий синтаксис загрузки метрик следующий:

```
metric.key,dimensions payload


#metric.key <payload-format> dt.meta.<property1>="<value>", dt.meta.<property2>="<value>"
```

### Измерения

Вот пример метрики с несколькими измерениями — `team` и `businessapp`, описывающими передаваемые точки данных.

```
mymetric,team=teamA,businessapp=hr 1000
```

Тот же пример с меткой времени точки данных.

```
mymetric,team=teamA,businessapp=hr 1000 1609459200000
```

### Метрика GAUGE

Gauge — это тип данных по умолчанию, поэтому тип данных можно не указывать при отправке значений gauge:

```
cpu.temperature,hostname=hostA,cpu=1 55


cpu.temperature,hostname=hostA,cpu=2 45
```

Вот пример с явно указанным типом данных `gauge`.

```
cpu.temperature,hostname=hostA,cpu=1 gauge,45
```

Вы также можете предоставить консолидированную информацию о нескольких точках данных, записанных на стороне клиента, перед отправкой в Dynatrace. В приведённом ниже примере 2 точки данных объединены, и минимальное, максимальное, суммарное значение и количество точек данных отправляются в одной строке.

```
cpu.temperature,hostname=hostA,cpu=1 gauge,min=17.1,max=17.3,sum=34.4,count=2
```

Вы также можете связать измерения с существующими сущностями хостов, используя зарезервированный ключ измерения `dt.entity.host`.

```
cpu.temperature,dt.entity.host=HOST-4587AE40F95AD90D,cpu=1 gauge,min=17.1,max=17.3,sum=34.4,count=2
```

Вам не нужно указывать измерение `dt.entity.host` при использовании локальных методов загрузки через OneAgent, то есть [dynatrace\_ingest](../ingestion-methods/oneagent-pipe.md "Learn how to ingest metrics using local scripting integration.") и [local API](../ingestion-methods/oneagent-metric-api.md "Use the Dynatrace API to retrieve the metrics of monitored entities."), поскольку для этих методов загрузки контекст хоста добавляется автоматически.

### Метрика COUNT

Для метрики типа count дельта вычисляется и предоставляется клиентом, отправляющим метрику в Dynatrace. В приведённом ниже примере это новые пользователи по регионам.

```
new_user_count,region=EAST count,delta=50


new_user_count,region=WEST count,delta=150
```

### Создание метаданных

Вот пример предоставления метаданных для метрики `cpu.temperature`.

```
#cpu.temperature gauge dt.meta.unit=count,dt.meta.description="The temperature of the CPU",dt.meta.displayname="CPU temperature"
```
