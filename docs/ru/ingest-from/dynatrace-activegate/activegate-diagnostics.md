# Диагностика ActiveGate

# Диагностика ActiveGate

* Latest Dynatrace
* Время чтения: 9 мин
* Обновлено 20 октября 2025 г.

Вы можете запустить полностью автоматизированное устранение неполадок ActiveGate для сред Dynatrace SaaS и Managed.

Рабочий процесс позволяет вам:

* Автоматически определить проблему, связанную с ActiveGate, в высокодинамичных средах в определённый момент времени
* Легко собрать диагностические данные для конкретной сущности и автоматически получить потенциальные решения для обнаруженных аномалий
* Быстро решить распространённые проблемы самостоятельно, сокращая время, затрачиваемое на диагностику
* Непосредственно предоставить службе поддержки Dynatrace все детали, необходимые для диагностики проблемы

Если ActiveGate не может подключиться к среде Dynatrace или не запускается, вам может потребоваться собрать диагностические данные локально из командной строки. Подробности см. в разделе [Сбор диагностических данных с помощью `agctl`](#collect-diagnostic-data-with-agctl).

## Требования

* Разрешение среды **View sensitive request data**
* Для ActiveGate, настроенного для поддержки нескольких сред, вы можете запустить диагностику ActiveGate только в основной среде, как определено в [конфигурации ActiveGate](configuration/configure-an-environment-activegate-for-multi-environment-support.md "Ознакомьтесь с пошаговой процедурой настройки одного Environment ActiveGate для поддержки нескольких сред.").

## Автоматический анализ

Эта процедура описывает процесс по умолчанию: Dynatrace собирает диагностические данные для ActiveGate и немедленно анализирует их.

Если вы предпочитаете собрать и просмотреть данные перед ручной отправкой их в Dynatrace для анализа, см. [Сбор и локальный просмотр](#collect-and-review-locally).

1. Перейдите в **Deployment Status** > **ActiveGates**.
2. Разверните запись ActiveGate, для которого вы хотите выполнить устранение неполадок, и выберите **Run ActiveGate diagnostics**.
3. На странице **Run Dynatrace ActiveGate diagnostics** кратко опишите, что работает не так, как ожидалось, с вашей точки зрения.
4. Необязательно. По умолчанию для анализа собираются данные за 7 дней. Если вам нужно больше данных, выберите ссылку **Advanced options**, измените количество дней и выберите **Apply**.
5. Выберите **Start analysis**.

### Что происходит далее

Dynatrace выполняет следующие действия:

* Собирает диагностические данные за последние 7 дней (если вы не изменили значение по умолчанию) затронутого ActiveGate
* Сохраняет собранные диагностические данные
* Загружает диагностические данные в корзину S3 в регионе AWS вашей среды для дальнейшего анализа

Столбец **State** описывает текущий этап процесса.

**State** не обновляется автоматически. Выберите **Refresh**, чтобы проверить изменение состояния.

Collecting

Сбор данных выполняется.
Во время сбора данных вы можете:

* **Refresh** — обновить страницу для проверки прогресса.
* **Cancel** — отменить сбор диагностических данных.

Collected

Dynatrace завершил сбор диагностических данных.
После сбора данных вы можете:

* **Analyze** — отправить собранные данные в Dynatrace для анализа.
* **Download** — загрузить собранные данные локально для проверки.
* **Delete** — удалить проблему, включая собранные диагностические данные.

Sending in progress

Диагностические данные передаются в Dynatrace для анализа.
Во время отправки данных вы можете:

* **Refresh** — обновить страницу для проверки прогресса.
* **Download** — загрузить собранные диагностические данные.
* **Delete** — удалить проблему, включая собранные диагностические данные.

Sent to Dynatrace cloud

Диагностические данные были переданы в Dynatrace для анализа.

Analyzing

Dynatrace анализирует диагностические данные.
Во время анализа данных вы можете:

* **Refresh** — обновить страницу для проверки прогресса.
* **Download** — загрузить собранные диагностические данные.
* **Delete** — удалить проблему, включая собранные диагностические данные.

Analyzed

Анализ завершён. Количество связанных оповещений отображается в скобках.
После анализа вы можете:

* **Download** — загрузить собранные диагностические данные.
* **Delete** — удалить проблему, включая собранные диагностические данные.

Delete in progress

Диагностические данные удаляются. Во время удаления данных вы можете:

* **Refresh** — обновить страницу для проверки прогресса.

Deleted

Диагностические данные удалены. Dynatrace сохраняет только небольшой набор информации о том, кто, когда, где и почему собирал диагностические данные.

Canceled

Процесс диагностики был отменён вручную до его завершения.

### Просмотр результатов анализа Dynatrace

Когда анализ завершён, Dynatrace отправляет результаты обратно в вашу среду. Если потенциальное решение найдено, Dynatrace перечисляет его в **разделе оповещений (Alerts).**

## Сбор и локальный просмотр

Эта процедура описывает, как собрать диагностические данные для ActiveGate. Используйте эту опцию, если вы предпочитаете собрать и просмотреть данные перед ручной отправкой их в Dynatrace для анализа.

Если вместо этого вы хотите собрать данные и автоматически отправить их в Dynatrace для анализа, см. [Автоматический анализ](#analyze-automatically).

1. Перейдите в **Deployment Status** > **ActiveGates**.
2. Разверните запись ActiveGate, для которого вы хотите выполнить устранение неполадок, и выберите **Run ActiveGate diagnostics**.
3. На странице **Run Dynatrace ActiveGate diagnostics** кратко опишите, что работает не так, как ожидалось, с вашей точки зрения.
4. Выберите ссылку **Advanced options**.
5. Выберите **and store locally**.

   * Здесь вы также можете изменить количество дней сбора данных (по умолчанию = `7 days`).
6. Выберите **Apply**.
7. Выберите **Start collection**, чтобы собрать диагностические данные и сохранить их локально.

### Что происходит далее

Dynatrace теперь:

* Собирает диагностические данные за последние 24 часа (если вы не изменили значение по умолчанию) затронутого ActiveGate
* Сохраняет собранные диагностические данные

Столбец **State** описывает текущий этап процесса.

**State** не обновляется автоматически. Выберите **Refresh**, чтобы проверить изменение состояния.

Collecting

Сбор данных выполняется.
Во время сбора данных вы можете:

* **Refresh** — обновить страницу для проверки прогресса.
* **Cancel** — отменить сбор диагностических данных.

Collected

Dynatrace завершил сбор диагностических данных.
После сбора данных вы можете:

* **Analyze** — отправить собранные данные в Dynatrace для анализа.
* **Download** — загрузить собранные данные локально для проверки.
* **Delete** — удалить проблему, включая собранные диагностические данные.

### Что делать с собранными данными

Теперь, когда данные собраны, вы можете:

* **Download** — загрузить собранные данные.

  + Вы можете просмотреть данные. См. [Содержимое диагностических данных](#contents) для обзора того, что содержится в загрузке.
  + Вы можете добавить данные к вашему обращению в службу поддержки.
* **Analyze** — проанализировать данные.
* **Delete** — удалить проблему, включая собранные диагностические данные.

### Диагностика ActiveGate в изолированных средах Dynatrace Managed

В изолированной среде Dynatrace Managed:

1. Используйте опцию **Store locally** в **Advanced options**, как описано выше.
2. После сбора диагностических данных вы можете добавить данные к вашему обращению в службу поддержки.
3. Dynatrace может затем получить диагностические данные из вашего обращения в службу поддержки, проанализировать их и предоставить автоматическую обратную связь службе поддержки Dynatrace об обнаруженных аномалиях.

На протяжении всего этого процесса обеспечиваются и регистрируются строгие меры защиты конфиденциальности данных.

## Сбор диагностических данных с помощью `agctl`

ActiveGate версии 1.275+

В случаях, когда ActiveGate не может подключиться к среде Dynatrace или не запускается, вам может потребоваться собрать диагностические данные локально из командной строки на хосте ActiveGate. Для этого используйте инструмент командной строки `agctl`, чтобы собрать все необходимые данные и создать пакет, который можно проанализировать позже.

### Использование `agctl` на Linux или Windows

Linux

Windows

#### Расположение

* `agctl` находится в [каталоге установки ActiveGate](configuration/where-can-i-find-activegate-files.md "Узнайте, где хранятся файлы ActiveGate в системах Windows и Linux."). Если вы установили ActiveGate в пользовательскую папку, путь к `agctl` будет другим.
* Диагностический пакет записывается в текущий рабочий каталог по умолчанию.

  + Чтобы изменить это, используйте параметр `--directory=<path>`.
  + После успешного создания файла диагностических данных `agctl` выводит путь к этому файлу.

#### Разрешения

* `agctl` должен быть запущен от [того же пользователя, что и процесс ActiveGate](installation/linux/linux-customize-installation-for-activegate.md#user-service "Узнайте о параметрах командной строки, которые можно использовать с ActiveGate на Linux.") — по умолчанию `dtuserag`.
* Убедитесь, что `dtuserag` имеет права на запись в целевой каталог.

#### Собираемые данные

* По умолчанию собираются данные за 30 дней. Чтобы изменить это, используйте параметр `--days=<days>`.
* По умолчанию диагностические данные собираются из всех модулей. Чтобы изменить это, используйте параметр `--modules=<modules-list>` со списком модулей через запятую (например, `--modules=zremote,synthetic`).

#### Синтаксис

Чтобы собрать диагностические данные, запустите скрипт `agctl` с параметром `create-support-archive` и любыми необязательными параметрами, описанными выше.

```
sudo -u dtuserag /opt/dynatrace/gateway/agctl create-support-archive [--directory=<path> --days=<days> --modules=<modules-list>]
```

#### Расположение

* `agctl.bat` находится в [каталоге установки ActiveGate](configuration/where-can-i-find-activegate-files.md "Узнайте, где хранятся файлы ActiveGate в системах Windows и Linux."). Если вы установили ActiveGate в пользовательскую папку, путь к `agctl.bat` будет другим.
* Диагностический пакет записывается в текущий рабочий каталог по умолчанию.

  + Чтобы изменить это, используйте параметр `--directory=<path>`.
  + После успешного создания файла диагностических данных инструмент выводит путь к этому файлу.

#### Разрешения

* `agctl.bat` должен быть запущен от [того же пользователя, что и процесс ActiveGate](installation/linux/linux-customize-installation-for-activegate.md#user-service "Узнайте о параметрах командной строки, которые можно использовать с ActiveGate на Linux.") или от имени администратора.

#### Собираемые данные

* По умолчанию собираются данные за 30 дней. Чтобы изменить это, используйте параметр `--days=<days>`.
* По умолчанию диагностические данные собираются из всех модулей. Чтобы изменить это, используйте параметр `--modules=<modules-list>` со списком модулей через запятую (например, `--modules=zremote,synthetic`).

#### Синтаксис

Чтобы собрать диагностические данные, запустите скрипт `agctl.bat` с параметром `create-support-archive` и любыми необязательными параметрами, описанными выше.

```
."C:\Program Files\dynatrace\gateway\agctl.bat" create-support-archive [--directory=<path> --days=<days> --modules=<modules-list>]
```

### Использование `agctl` из контейнеризированного ActiveGate

ActiveGate версии 1.325+

Вы можете использовать `agctl` для сбора диагностических данных из контейнеризированного ActiveGate с помощью параметра `--stdout`, который выводит архив поддержки напрямую и позволяет перенаправить его в файл на вашей локальной машине.

* По умолчанию собираются данные за 30 дней. Чтобы изменить это, используйте параметр `--days=<days>`.
* По умолчанию диагностические данные собираются из всех модулей. Чтобы изменить это, используйте параметр `--modules=<modules-list>` со списком модулей через запятую (например, `--modules=zremote,synthetic`).

Kubernetes

OpenShift

```
kubectl exec -n <namespace> <pod-name> -- /opt/dynatrace/gateway/agctl create-support-archive --stdout > ag-support-archive.zip
```

```
oc exec -n <namespace> <pod-name> [-c <container>] -- /opt/dynatrace/gateway/agctl create-support-archive --stdout > ag-support-archive.zip
```

Содержимое архива поддержки записывается в `stdout`, что позволяет перенаправить его в zip-файл. Остальной вывод отправляется в `stderr` для сохранения целостности архивного файла.

Windows PowerShell не поддерживается

Убедитесь, что используете командную строку (`cmd.exe`) на Windows; PowerShell в настоящее время не поддерживается.

#### Для версий ActiveGate до 1.325

ActiveGate версии от 1.275 до 1.325

Для более ранних версий контейнеризированного ActiveGate используйте `agctl` с параметром `--directory`, а затем перенесите файл с помощью кодировки base64.

* По умолчанию собираются данные за 30 дней. Чтобы изменить это, используйте параметр `--days=<days>`.
* По умолчанию диагностические данные собираются из всех модулей. Чтобы изменить это, используйте параметр `--modules=<modules-list>` со списком модулей через запятую (например, `--modules=zremote,synthetic`).

Kubernetes

OpenShift

1. Выполните команду.

   ```
   kubectl exec <pod-name> -n <namespace> -- /opt/dynatrace/gateway/agctl create-support-archive --directory=<tmp-path> [--days=<days> --modules=<modules-list>]
   ```
2. Перенесите созданный файл на хост. Используйте base64 для создания закодированного файла.

   ```
   kubectl exec <pod-name> -n <namespace> -- base64 --wrap=0 <created-file-path> > <encoded-file-host-path>
   ```

1. Выполните команду.

   ```
   oc exec <pod-name> [-c <container>] -- /opt/dynatrace/gateway/agctl create-support-archive --directory=<tmp-path> [--days=<days> --modules=<modules-list>]
   ```
2. Перенесите созданный файл на хост. Используйте base64 для создания закодированного файла.

   ```
   oc exec <pod-name> [-c <container>] -- base64 --wrap=0 <created-file-path> > <cenoded-file-host-path>
   ```

3. Декодируйте закодированный файл с помощью соответствующей команды Windows или Linux.

   Linux

   Windows

   ```
   base64 --decode -i <encoded-file-path> > <decoded-file-path>
   ```

   ```
   $base64Data = Get-Content <encoded-file-path>



   $zipBytes = [Convert]::FromBase64String($base64Data)



   Set-Content -Value $zipBytes -Encoding Byte -Path <decoded-file-path>
   ```

## Конфиденциальность данных

Для соответствия региональным нормам защиты данных и конфиденциальности Dynatrace автоматически удаляет все диагностические данные через 30 дней после их сбора. Это относится к данным в вашей среде Dynatrace и на кластере Dynatrace.

Вы можете удалить собранные диагностические данные раньше. Для обеспечения прозрачности Dynatrace сохраняет только небольшой набор информации о том, кто, когда, где и почему собирал диагностические данные.

Подробнее о конфиденциальности данных Dynatrace см. [Сроки хранения данных](../../manage/data-privacy-and-security/data-privacy/data-retention-periods.md#diagnostics "Проверьте сроки хранения для различных типов данных.").

## Часто задаваемые вопросы

Могу ли я получить прямой доступ к S3 или использовать собственный S3?

Нет, вы не можете получить прямой доступ к S3 или использовать собственный.

Диагностические данные загружаются в корзину S3 Dynatrace, которая настроена для среды/кластера компанией Dynatrace. Используемая корзина S3 зависит от расположения среды/кластера.
