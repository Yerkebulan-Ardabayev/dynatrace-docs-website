# Миграция с классического полнофункционального режима на облачный полнофункциональный режим

* Latest Dynatrace
* 4-min read
* Updated on Sep 05, 2025

Dynatrace Operator версии 1.0.0+

В этом руководстве описаны шаги, необходимые для миграции развёртывания Dynatrace с классического полнофункционального режима на [облачный полнофункциональный режим (cloud-native full-stack)](/docs/ingest-from/setup-on-k8s/how-it-works#cloud-native "Подробное описание работы развёртывания в Kubernetes.").

## Преимущества

Облачный полнофункциональный режим развёртывания представляет собой значительное улучшение в области безопасности, используя облачные методы для внедрения OneAgent. Этот подход устраняет два ключевых ограничения, присущих традиционному полнофункциональному режиму:

* Облачный полнофункциональный режим предотвращает состояния гонки, которые могут возникать при одновременном запуске подов DaemonSet OneAgent и подов отслеживаемых приложений.
* Используя концепции Kubernetes, такие как admission webhooks и CSI-драйвер для внедрения Code Module, облачный полнофункциональный мониторинг снижает требуемые привилегии для OneAgent.

### Замечания и последствия

* При переключении на облачный полнофункциональный мониторинг ранее развёрнутые OneAgent будут деактивированы и глубокий мониторинг приложений прекратится. Следовательно, перезапуск всех подов приложений, требующих глубокого мониторинга, становится обязательным. Перезапуск этих подов обеспечивает повторное внедрение агентов, что позволяет возобновить глубокий мониторинг.
* В облачном полнофункциональном режиме идентификаторы хостов определяются по-другому, что приводит к временному наличию как новых, так и старых хостов в списках хостов. Старые сущности хостов и связанные с ними данные подчиняются политике хранения данных, определённой Dynatrace, и остаются доступными в течение указанного периода.
* В облачном полнофункциональном режиме правила мониторинга контейнеров игнорируются. Вместо этого для точного управления внедрением OneAgent следует использовать [селекторы меток](/docs/ingest-from/setup-on-k8s/guides/deployment-and-configuration/monitoring-and-instrumentation/annotate "Настройка мониторинга для пространств имён и подов").

## Миграция на облачный полнофункциональный режим

В этом разделе представлена вся информация, необходимая для миграции с классического на облачный полнофункциональный режим.

Использование среды выполнения контейнеров CRI-O

Стандартная процедура миграции, описанная ниже, требует OneAgent версии 1.281 или выше для кластеров Kubernetes, использующих CRI-O в качестве среды выполнения контейнеров, поэтому перед продолжением выполнения описанных ниже шагов необходимо обновить OneAgent.

Если обновление невозможно, следуйте процедуре [Использование CRI-O с OneAgent версий 1.279 и ранее](#running-crio) для альтернативного процесса миграции, а затем вернитесь к шагу 1 данной процедуры.

1. Обновите установку с включённым CSI-драйвером:

   Helm

   Manifest

   ```
   helm upgrade dynatrace-operator oci://docker.io/dynatrace/dynatrace-operator \



   --atomic \



   --csidriver.enabled="true" \ # By default CSI driver is enabled



   --namespace dynatrace
   ```

   **Kubernetes**

   ```
   kubectl apply -f https://github.com/Dynatrace/dynatrace-operator/releases/download/v1.8.1/kubernetes-csi.yaml
   ```

   **OpenShift**

   ```
   oc apply -f https://github.com/Dynatrace/dynatrace-operator/releases/download/v1.8.1/openshift-csi.yaml
   ```
2. Перенастройте (существующий) DynaKube для облачного полнофункционального режима:

   Следующее сравнение показывает, как перенастроить DynaKube CR с классического полнофункционального режима на облачный полнофункциональный мониторинг:

   Классический полнофункциональный мониторинг

   Облачный полнофункциональный мониторинг

   ```
   apiVersion: dynatrace.com/v1beta5



   kind: DynaKube



   metadata:



   name: dynakube



   namespace: dynatrace



   spec:



   apiUrl: https://<environment-id>.live.dynatrace.com/api



   networkZone: <network-zone>



   oneAgent:



   hostGroup: <host-group>



   classicFullStack: {}



   activeGate:



   capabilities:



   - routing



   - kubernetes-monitoring



   - dynatrace-api
   ```

   ```
   apiVersion: dynatrace.com/v1beta5



   kind: DynaKube



   metadata:



   name: dynakube



   namespace: dynatrace



   spec:



   apiUrl: https://<environment-id>.live.dynatrace.com/api



   networkZone: <network-zone>



   oneAgent:



   hostGroup: <host-group>



   cloudNativeFullStack: {}



   activeGate:



   capabilities:



   - routing



   - kubernetes-monitoring



   - dynatrace-api
   ```

   Дополнительную информацию о настройке DynaKube для облачного полнофункционального режима см. в сравнении ниже, [руководстве по развёртыванию](/docs/ingest-from/setup-on-k8s/deployment/full-stack-observability "Развёртывание Dynatrace Operator в облачном полнофункциональном режиме в Kubernetes") или [параметрах DynaKube](/docs/ingest-from/setup-on-k8s/reference/dynakube-parameters#spec-oneagent-cloudnativefullstack "Список доступных параметров для настройки Dynatrace Operator в Kubernetes."). Также вы можете загрузить [пример пользовательского ресурса DynaKube](https://dt-url.net/9n636jg) для облачного полнофункционального режима с GitHub и адаптировать пользовательский ресурс DynaKube в соответствии с вашими требованиями.
3. Примените пользовательский ресурс DynaKube:

   Выполните приведённую ниже команду для применения пользовательского ресурса DynaKube. Вебхук валидации предоставит полезные сообщения об ошибках в случае проблемы.

   ```
   kubectl apply -f dynakube.yaml
   ```

   Это действие приведёт к удалению OneAgent в классическом полнофункциональном режиме и, следовательно, к прекращению глубокого мониторинга подов приложений вскоре после этого.
4. Дождитесь готовности OneAgent:

   Dynatrace Operator обнаружит изменения в пользовательском ресурсе DynaKube и обеспечит доступность новых OneAgent на каждом узле.
5. Перезапустите рабочие нагрузки приложений:

   Незамедлительно перезапустите все рабочие нагрузки приложений для запуска внедрения OneAgent и включения глубокого мониторинга, предотвращая/минимизируя перерывы в мониторинге.

#### Использование CRI-O с OneAgent версий 1.279 и ранее

В этом разделе описана процедура миграции для кластеров Kubernetes, использующих среду выполнения контейнеров CRI-O и OneAgent версии 279 или ранее.

Необходимо удалить хуки CRI-O, установленные и используемые для внедрения OneAgent в классическом полнофункциональном режиме. Дополнительные сведения о хуках CRI-O см. в этой [записи блога Red Hat](https://dt-url.net/fq039v2).

Пошаговые инструкции

Следуйте этим инструкциям для успешной миграции с классического полнофункционального режима:

1. Удалите пользовательский ресурс DynaKube:

   Удалите DynaKube, настроенный в классическом полнофункциональном режиме, выполнив следующую команду:

   ```
   kubectl delete dynakube -n dynatrace <dynakube-name>
   ```

   Это действие приведёт к удалению OneAgent в классическом полнофункциональном режиме и, следовательно, к прекращению глубокого мониторинга подов приложений вскоре после этого. Кроме того, если мониторинг Kubernetes настроен в пользовательском ресурсе DynaKube, мониторинг Kubernetes мгновенно прекратится с удалением ActiveGate.
2. Дождитесь завершения работы подов OneAgent.
3. Следуйте инструкциям в разделе [Очистка узлов](/docs/ingest-from/setup-on-k8s/guides/deployment-and-configuration/updates-and-maintenance/update-uninstall-operator#cleanup-nodes "Процедуры обновления и удаления Dynatrace Operator") для удаления хуков Dynatrace CRI-O со всех узлов Linux.
4. Продолжите с шага 1 [стандартной процедуры миграции](#migrate).
