# Руководство по определению размеров ресурсов для Dynatrace ActiveGate в сценарии мониторинга Kubernetes


* Latest Dynatrace
* 5 мин чтения

Установка подходящих запросов ресурсов (и лимитов при необходимости) обеспечивает стабильную и предсказуемую работу экземпляров Dynatrace ActiveGate. В этом руководстве описаны методы определения размеров в зависимости от масштаба и нагрузки.

Стабильный и исправный ActiveGate обеспечивает непрерывный мониторинг данных без пробелов.

## Факторы, определяющие размеры

Фактические требуемые ресурсы возрастают с увеличением:

* **Количества подов** — Основной фактор определения размеров — количество отслеживаемых подов. Потребление ресурсов (CPU и память) компонентов Dynatrace ActiveGate масштабируется пропорционально количеству подов, главным образом из-за возросших потребностей в обработке и хранении данных. По мере роста количества отслеживаемых подов ActiveGate обрабатывает больше данных о сущностях, событий и метрик, что приводит к повышенной нагрузке на CPU для приёма и обработки данных, а также к увеличению потребления памяти для кэширования информации о подах. Это основной фактор, с потреблением, пропорциональным количеству подов.
* **Объёма метрик Prometheus** — Количество подов с аннотациями Prometheus напрямую коррелирует с увеличением требований к ресурсам для Dynatrace ActiveGate, главным образом через повышенное потребление CPU. С ростом числа аннотированных подов увеличивается объём собираемых метрик, что требует больше циклов CPU для сбора, агрегации и пересылки. Влияние на память вторично, поскольку метрики пересылаются на тенант Dynatrace без долговременного хранения на ActiveGate, хотя и масштабируются пропорционально пиковым скоростям приёма.
* **Количества узлов** — Потребление ресурсов (CPU и память) компонентов Dynatrace ActiveGate масштабируется с количеством узлов, главным образом из-за увеличения накладных расходов на мониторинг и нагрузки от системных подов уровня узла. С ростом числа узлов ActiveGate должен обрабатывать больше данных системного уровня, сущностей и событий, что приводит к повышенным вычислительным требованиям. Это вторичный фактор по сравнению с количеством подов, но он пропорционально влияет на общие потребности в ресурсах, особенно в крупных кластерах, где мониторинг на уровне узлов создаёт кумулятивную нагрузку.
* **Количества отслеживаемых кластеров** — Один ActiveGate может отслеживать несколько кластеров, однако такая схема развёртывания **не рекомендуется**. Мониторинг нескольких кластеров одним ActiveGate увеличивает требования к CPU и памяти пропорционально каждому дополнительному кластеру, поскольку ActiveGate должен обрабатывать больше данных, управлять большим числом сущностей и принимать события со всех кластеров. Это может привести к конкуренции за ресурсы, повышенному риску троттлинга или перезапусков из-за OOM, а также к потенциальным пробелам в данных мониторинга.

**Мы рекомендуем конфигурацию, при которой один контейнеризированный ActiveGate отслеживает один кластер Kubernetes.**

Это обеспечивает оптимальную производительность и надёжность. Такой подход упрощает устранение неполадок, исключает конкуренцию за ресурсы и обеспечивает предсказуемое масштабирование, поскольку каждый ActiveGate обрабатывает нагрузку только одного кластера.

## Признаки неисправного ActiveGate

Эти симптомы указывают на исчерпание ресурсов и потенциальную потерю данных.

### Пробелы в данных мониторинга

ActiveGate собирает различные типы данных независимо (например, метрики Prometheus, события Kubernetes, сущности). Если одна задача сбора занимает более 1 минуты, пробел возникает только для этого типа данных в данном временном окне. Другие задачи сбора продолжают работать нормально.

* Метрики будут иметь отсутствующие точки данных за данную минуту.
* События для данных временных интервалов сбора будут полностью недоступны.
* Сущности могут не отражать последние обновления или отсутствовать полностью, если они короткоживущие.

### Интенсивный троттлинг CPU

Устойчивый высокий троттлинг означает недостаточность CPU. Интенсивный троттлинг может вызвать пробелы в данных. Незначительный троттлинг обычно безвреден.
Если троттлинг затрагивает под, обслуживающий мониторинговый ActiveGate, это может вызвать пробелы в данных.

### Перезапуски из-за нехватки памяти (OOM)

Если ActiveGate завершается из-за OOM, данные становятся недоступными до перезапуска. После перезапуска повторные завершения OOM вероятны.

## Мониторинг и валидация

Следующие индикаторы можно отслеживать для определения исправности ActiveGate.

* **Использование CPU vs запросы** — Если загрузка CPU постоянно превышает 85%, увеличьте запрос CPU.
* **Троттлинг CPU** (container\_cpu\_cfs\_throttled\_periods\_total / periods) — Если троттлинг постоянно превышает 10%, увеличьте запрос CPU.
* **Рабочий набор памяти vs запрос** — Если использование постоянно превышает 80%, увеличьте запрос памяти.
* **Счётчик перезапусков ActiveGate** — После перезапуска из-за OOM немедленно увеличьте настроенный объём памяти для предотвращения повторения.
* **Длительность обработки / время цикла** — Если время выполнения конвейеров постоянно превышает 50-60 секунд, увеличьте запрос CPU. Время выполнения конвейера также зависит от объёма принимаемых данных и других факторов.
* **Время сборки мусора** — Увеличение времени сборки мусора является явным индикатором недостаточно обеспеченного ресурсами ActiveGate.

### Увеличение ресурсов

Начните с рекомендуемых значений ниже.

1. Проверьте исправность ActiveGate с помощью индикаторов мониторинга.
2. При необходимости отрегулируйте ресурсы:

   * Увеличивайте память шагами по 2 ГиБ.
   * Увеличивайте CPU шагами по 500m.
3. Сначала откорректируйте запросы, а на следующем шаге — лимиты.

   * Относительно CPU: используйте лимиты только если это требуется политикой.

## Использование выделенного ActiveGate для мониторинга платформы Kubernetes

Рекомендуется разделить ответственности ActiveGate на две группы: одна отвечает за всё, что связано с мониторингом платформы Kubernetes, включая KSPM, а другая управляет маршрутизацией трафика агентов, приёмом телеметрии и расширениями. Такое разделение обеспечивает ряд преимуществ:

* **Изоляция** — Конкуренция за ресурсы в одном модуле не влияет на другой. Пиковая нагрузка трафика OneAgent не замедлит сбор метрик Kubernetes, и наоборот.
* **Независимое масштабирование** — Пересылка трафика и мониторинг платформы имеют принципиально различные характеристики масштабирования:

  + **Мониторинг платформы Kubernetes** масштабируется только вертикально — при росте кластера увеличивают ресурсы CPU и памяти соответствующего ActiveGate.
  + **Маршрутизация трафика OneAgent** может масштабироваться горизонтально — при увеличении трафика OneAgent дополнительные реплики маршрутизирующего ActiveGate помогают распределить нагрузку.

  Отдельные ActiveGate позволяют масштабировать каждое направление независимо без избыточного выделения ресурсов. Например, можно добавить 3 дополнительных реплики маршрутизации для обработки возросшего трафика приложений без необходимости увеличивать ресурсы для мониторинга платформы.
* **Упрощённое устранение неполадок** — При возникновении проблем вы можете сразу определить, происходят ли они от мониторинга платформы или трафика OneAgent, сокращая время диагностики.

## Определение размеров ActiveGate

Сценарии определяются на основе количества подов, отслеживаемых одним ActiveGate. Необходимые ресурсы можно взять из следующих таблиц.

* **Сценарий S**: <1000 подов
* **Сценарий M**: 1000-5000 подов
* **Сценарий L**: 5000-20000 подов

  Данное руководство не охватывает среды с более чем 20 000 подов. Из-за множества переменных предоставление точных рекомендаций для таких масштабных конфигураций невозможно. В качестве отправной точки вы можете использовать рекомендации для сценария L и постепенно увеличивать ресурсы до достижения стабильного мониторинга без пробелов.
  Для индивидуальных рекомендаций мы советуем обратиться в службу поддержки Dynatrace для обеспечения наилучшей конфигурации вашей среды.

### Вторичные факторы

#### Количество узлов

По мере увеличения количества узлов в кластере Kubernetes ActiveGate требует больше ресурсов CPU и памяти для обработки дополнительной нагрузки мониторинга. Это включает обработку данных от системных компонентов уровня узлов и событий, которые нарастают пропорционально.
Мы группируем количество узлов в следующие категории для рекомендаций по размерам:

* До 25 узлов
* До 100 узлов
* До 500 узлов

Если в вашем кластере более 100 узлов, необходимо увеличить выделение ресурсов с учётом дополнительных требований, обеспечивая стабильный мониторинг без пробелов. Для кластеров с более чем 500 узлами обратитесь в службу поддержки Dynatrace за индивидуальными рекомендациями.

#### Объём собираемых метрик Prometheus

Dynatrace поддерживает до 1000 экспортёров подов, каждый из которых может предоставлять до 1000 метрик. Если ваша среда приближается к этим лимитам, необходимо увеличить ресурсы, выделенные ActiveGate, для обеспечения надёжной производительности.

### ActiveGate для мониторинга платформы Kubernetes

Все следующие сценарии использования фактически имеют одинаковые требования к ActiveGate, отвечающему за мониторинг платформы Kubernetes.

* Только мониторинг платформы Kubernetes
* Мониторинг платформы Kubernetes + наблюдаемость приложений
* Мониторинг платформы Kubernetes + полная наблюдаемость стека

| Количество подов | Ресурсы CPU | Ресурсы памяти |
| --- | --- | --- |
| < 1000 (Small) | requests: 200m (limits: 1000m) | requests: 6Gi limits: 6Gi |
| < 5000 (Medium) | requests: 1000m (limits: 2000m) | requests: 10Gi limits: 10Gi |
| < 20000 (Large) | requests: 2000m (limits: 4000m) | requests: 12Gi limits: 12Gi |

Мы рекомендуем запускать ActiveGate без лимитов CPU.

### ActiveGate для маршрутизации и проксирования трафика OneAgent

Второй ActiveGate не участвует активно в мониторинге платформы Kubernetes, а используется в качестве маршрутизатора/прокси для потоков данных, исходящих от OneAgent.
Мы рекомендуем использовать отдельные ActiveGate для пересылки трафика и мониторинга платформы. Пересылка трафика масштабируется горизонтально путём добавления реплик. Мониторинг Kubernetes не масштабируется горизонтально; вместо этого увеличивайте ресурсы.

Эта конфигурация необходима в следующих сценариях:

* Мониторинг платформы Kubernetes + наблюдаемость приложений
* Мониторинг платформы Kubernetes + полная наблюдаемость стека

| Количество подов | Ресурсы CPU | Ресурсы памяти | Реплики |
| --- | --- | --- | --- |
| < 1000 (Small) | requests: 250m (limits: 1000m) | requests: 2Gi limits: 2Gi | 3 |
| < 5000 (Medium) | requests: 500m (limits: 2000m) | requests: 4Gi limits: 4Gi | 3 |
| < 20000 (Large) | requests: 1000m (limits: 4000m) | requests: 6Gi limits: 6Gi | 6 |

Мы рекомендуем запускать ActiveGate без лимитов CPU.

## Начало работы

Следующие примеры применяют подход разделения ActiveGate на модули с различными зонами ответственности. Один ActiveGate отвечает за мониторинг платформы Kubernetes и управление безопасностью конфигурации Kubernetes, тогда как второй ActiveGate отвечает за маршрутизацию трафика агентов, приём телеметрии и расширения.
Настройте запросы (и лимиты при необходимости) в соответствии с вашей средой.

Следующие манифесты включают два ресурса DynaKube для настройки ActiveGate:

* **k8s-monitoring** — Конфигурирует ActiveGate, выделенный для мониторинга платформы Kubernetes.
* **agents** — Конфигурирует ActiveGate, поддерживающий OneAgent, приём телеметрии и дополнительные функции.

Вы можете применить оба манифеста одновременно или только нужный вам DynaKube.

Лимиты CPU закомментированы. Мы рекомендуем определять только запросы, чтобы ActiveGate мог использовать дополнительный CPU при его доступности. Если лимиты необходимы, установите их равными или выше запросов.

Этот фрагмент включает конфигурацию для управления безопасностью конфигурации Kubernetes (KSPM). Это дополняющая функция безопасности, подключаемая по выбору наряду с мониторингом платформы Kubernetes.
Этот фрагмент также включает конфигурации для мониторинга логов, расширений и приёма телеметрии. Эти разделы являются необязательными для каждого конкретного случая.
