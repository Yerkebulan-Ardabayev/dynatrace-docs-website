# Настройка разрешений Grail для OpenTelemetry

* Latest Dynatrace
* Практическое руководство
* Чтение: 2 мин
* Обновлено 20 ноября 2025 г.

Dynatrace имеет [модель разрешений для Grail](../../platform/grail/organize-data/assign-permissions-in-grail.md "Узнайте, как назначать разрешения для бакетов и таблиц в Grail."). Она применяется ко всем телеметрическим данным, таким как метрики, события, спаны и логи.

Мы рекомендуем настраивать разрешения в соответствии с организационной структурой и областями развёртывания. Подходящими концепциями являются группы хостов, кластеры Kubernetes и пространства имён Kubernetes. Эти атрибуты, как правило, доступны для всех телеметрических данных, собранных методами сбора Dynatrace, такими как OneAgent, OpenTelemetry или оператор Kubernetes. Следовательно, вы можете использовать эти атрибуты для включения [разрешений на уровне записей](../../platform/grail/organize-data/assign-permissions-in-grail.md#grail-permissions-table-record "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.").

Для развёртываний на базе Kubernetes убедитесь, что оператор Dynatrace имеет включённое [обогащение метаданными](../setup-on-k8s/guides/metadata-automation/metadata-enrichment.md "Обогащение метаданными в операторе Dynatrace добавляет контекст к подам Kubernetes, прикрепляя соответствующие метаданные к сущностям, таким как поды, хосты и процессы, для улучшения наблюдаемости.").

Если вам требуется только базовая концепция разрешений, лучшим вариантом является настройка разрешений на уровне бакетов. Затем вы можете направлять данные в нужный бакет в OpenPipeline, сопоставляя одно из упомянутых основных полей Grail, связанных с развёртыванием.

Для более детального контроля в Dynatrace вы можете настроить границы политик с более детальными ограничениями на уровне данных. По умолчанию вы можете использовать следующие атрибуты:

* `dt.host_group.id`
* `k8s.cluster.name`
* `k8s.namespace.name`
* Любой другой атрибут, перечисленный в [модели разрешений](../../platform/grail/organize-data/assign-permissions-in-grail.md#grail-permissions-table-record "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.")

Dynatrace предоставляет комплексную модель разрешений для Grail, применяемую ко всем телеметрическим данным, включая метрики, логи, спаны и события.

## Настройка контекста безопасности

Если разрешений на уровне атрибутов развёртывания или на уровне бакетов недостаточно, Dynatrace позволяет настроить детальные разрешения путём добавления атрибута `dt.security_context` к вашим данным. Этот контекст может отражать вашу архитектуру безопасности и даже быть иерархическим путём кодирования в строку вида `department-A/department-AB/team-C`.

Чтобы использовать `dt.security_context` и другие атрибуты для разрешений, убедитесь, что эти атрибуты присутствуют в ваших телеметрических данных.

Чтобы добавить контекст безопасности в данные OpenTelemetry, обогатите ваши сигналы атрибутом `dt.security_context`. Dynatrace автоматически распространяет значение `dt.security_context` из спанов на сущность сервиса для данных спанов.

Вы можете использовать существующие метки и теги для управления разрешениями в Dynatrace.

### Контекст безопасности через метки или аннотации Kubernetes

Вы можете использовать [метки или аннотации Kubernetes](../setup-on-k8s/guides/metadata-automation/k8s-metadata-telemetry-enrichment.md "Руководства по обогащению телеметрии в Kubernetes") в качестве источника для вашего `dt.security_context`. Это один из наиболее удобных способов сделать это.

В качестве альтернативы вы можете [настроить коллектор OpenTelemetry для обогащения данных в процессе передачи](collector/use-cases/kubernetes/k8s-enrich.md "Настройте коллектор OpenTelemetry для обогащения запросов OTLP данными Kubernetes."). При таком подходе вам может потребоваться сопоставить метаданные Kubernetes с `dt.security_context` в OpenPipeline.

### Контекст безопасности через атрибуты ресурсов OpenTelemetry

Чтобы использовать переменную среды [`OTEL_RESOURCE_ATTRIBUTES`](../../deliver/release-monitoring/version-detection-strategies.md#otel_resource_attributes "Метаданные для обнаружения версий в различных технологиях"), просто задайте `dt.security_context` непосредственно как атрибут ресурса. Вы также можете использовать любой атрибут ресурса в OpenPipeline в качестве источника для вашего `dt.security_context`.

### Контекст безопасности через файл метаданных OneAgent

OneAgent от Dynatrace предоставляет файлы обогащения или переменные среды для [добавления атрибутов непосредственно в код вашего приложения](../extend-dynatrace/extend-data.md#operator-enrichment-directory "Узнайте, как автоматически обогатить ваши телеметрические данные полями, специфичными для Dynatrace.").

Подробнее о вариантах обогащения и настройке см. в разделе [обогащение через переменную среды](../setup-on-k8s/guides/metadata-automation/k8s-metadata-telemetry-enrichment.md#expand--enrich-via-environment-variable--4 "Руководства по обогащению телеметрии в Kubernetes").
