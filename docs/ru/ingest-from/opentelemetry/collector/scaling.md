# Как масштабировать OpenTelemetry Collector

* Latest Dynatrace
* Руководство
* 10 мин. чтения
* Опубликовано 30 сентября 2025

Когда использование CPU или памяти Collector превышает порог, при котором он рискует быть перегруженным в случае всплеска трафика, рекомендуется найти способы либо увеличить ресурсы, выделенные Collector, либо масштабировать обработку на несколько экземпляров Collector. Здесь мы в основном рассмотрим решения, доступные в Kubernetes. Обратите внимание, что рекомендации и примеры в этой документации носят общий характер и могут не обеспечить оптимальную производительность для вашего конкретного случая; вам необходимо проанализировать свои системы, чтобы определить лучший способ их масштабирования.

Для получения более общей информации в документации OpenTelemetry см. страницу [Масштабирование Collector](https://opentelemetry.io/docs/collector/scaling/) на сайте OpenTelemetry.

## Определение необходимости масштабирования

Стоит рассмотреть масштабирование, когда вы начинаете приближаться к лимитам ресурсов, выделенных вашему Collector. Метрики самомониторинга, доступные через Collector, и метрики хостовой среды (например, Kubernetes) будут полезны для отслеживания этого. См. нашу страницу о [самомониторинге Collector](self-monitoring.md "Как мониторить OpenTelemetry Collector с помощью дашбордов Dynatrace.") для получения дополнительной информации о сборе этих данных.
Ниже приведены несколько метрик, на которые стоит обратить внимание:

* `otelcol_processor_refused_spans`: Если у вас включён [Memory Limiter Processor](https://github.com/open-telemetry/opentelemetry-collector/blob/v0.145.0/processor/memorylimiterprocessor), эта метрика (или эквивалент для других сигналов) укажет на то, что Collector нужно больше памяти для продолжения обработки текущей нагрузки.
* `otelcol_exporter_queue_capacity` и `otelcol_exporter_queue_size`: Когда размер очереди экспортёра начинает приближаться к ёмкости очереди или превышать её, это указывает на то, что Collector испытывает трудности с отправкой данных на бэкенд. Это может происходить либо из-за недоступности воркеров для отправки данных, либо из-за перегрузки самого бэкенда. Возможно, потребуется увеличить вычислительную мощность Collector для продолжения обработки такого объёма данных.
* `k8s.resource_quota.used`: Если вы мониторите свой Kubernetes-кластер с помощью [Kubernetes Cluster Receiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/v0.145.0/receiver/k8sclusterreceiver), эту метрику можно использовать для определения объёма квоты CPU/памяти, использованной вашим Collector.
* `container.cpu.usage` и `container.memory.usage`: Если вы мониторите кластер с помощью [Kubelet Stats Receiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/v0.145.0/receiver/kubeletstatsreceiver), они покажут, приближается ли контейнер Collector к своим лимитам квот или достиг их.

## Масштабирование Collector

Kubernetes поставляется с несколькими типами объектов, способных масштабировать Collector в зависимости от потребностей конкретных сценариев. Для простого масштабирования можно использовать Deployments или ReplicaSets для создания пула Collector, которые Kubernetes может планировать без особых предварительных расчётов. Для получения более общей информации об архитектурах развёртывания Collector см. наше руководство по [развёртыванию Dynatrace OTel Collector](deployment.md "Как развернуть Dynatrace OTel Collector.").

Большинство рекомендаций в этом документе относятся к горизонтальному масштабированию Collector путём создания дополнительных экземпляров или распределения экземпляров по машинам. Однако, если ваше текущее развёртывание использует один экземпляр Collector для всей обработки, сначала следует определить, достаточно ли вертикального масштабирования для ожидаемой нагрузки. Вертикальное масштабирование имеет более низкий потолок по вычислительной мощности и памяти, которые можно предоставить Collector, но оно также проще. В Kubernetes это делается путём увеличения лимитов CPU и памяти для пода Collector.

### Масштабирование Collector без состояния

Масштабировать Collector без состояния сравнительно просто: поскольку не имеет значения, какие данные попадают в какой Collector, решение о том, какому Collector отправить данные, может приниматься независимо от содержимого данных. В результате подойдёт любой стандартный балансировщик нагрузки для данного протокола передачи.

Самый простой способ балансировки нагрузки — через объект Kubernetes Service, который указывает на несколько реплик пода Collector, развёрнутого через любой стандартный тип рабочей нагрузки Kubernetes, такой как Deployment, ReplicaSet, StatefulSet или DaemonSet. Для кратковременных соединений это достаточно равномерно распределит нагрузку между Collector, доступными через сервис. Обратите внимание, что долгоживущие соединения, такие как HTTP/2 или gRPC, будут поддерживать соединение с одним Collector и, следовательно, могут привести к неравномерной нагрузке между Collector.

Для более сложных случаев, таких как обработка gRPC-соединений, Service типа `LoadBalancer` может предоставить больше контроля над балансировкой нагрузки. LoadBalancer Services могут использовать отдельный балансировщик нагрузки для определения, к какому Collector направлять соединение. Сервисные сетки, такие как Istio или Linkerd, также могут помочь с балансировкой нагрузки, так как они имеют детальный контроль над сетевыми соединениями внутри кластера.

Для случаев, когда важна топология развёртывания, например, при использовании шлюзовых Collector, развёрнутых через DaemonSet, можно использовать объект Service со специальными настройками маршрутизации для отправки данных только на Collector, работающие на том же узле, что и источник данных. В Kubernetes версии 1.26+ это делается путём настройки сервиса для [приёма только внутреннего трафика узла](https://kubernetes.io/docs/concepts/services-networking/service-traffic-policy/).

### Масштабирование обработки с состоянием без пула Collector

При использовании Collector для обработки с состоянием важно, чтобы одни и те же данные всегда отправлялись на один и тот же Collector. Можно увеличить пропускную способность конвейера, соблюдая это правило, путём назначения определённых Collector для обработки определённых данных. Это можно сделать, выбрав конкретный паттерн развёртывания или назначив источники данных определённым Collector:

* **Sidecar Collector**: Если Collector развёрнут как sidecar и связан с приложением, то все данные этого приложения будут проходить через sidecar Collector и могут обрабатываться с операциями с состоянием.
* **DaemonSet Collector**: Агентский Collector, развёрнутый на узле Kubernetes (например, через DaemonSet), может использоваться для обработки с состоянием, если данное приложение на узле всегда отправляет данные в этот Collector. Обратите внимание, что это предполагает наличие только одного Collector на узле.
* **Единственный Collector**: Если вам нужно запустить только один Collector для определённого набора источников данных, этот Collector можно использовать для обработки с состоянием, так как все данные будут проходить через один Collector. Вы можете выбрать этот вариант, если решите отправлять определённый сигнал или данные из выбранного набора приложений на данный Collector. Обратите внимание, что для высокой доступности резервные экземпляры Collector должны храниться как бэкапы и не получать данные, пока первый Collector не выйдет из строя. Кроме того, обработка будет сброшена при активации резервного Collector.

### Масштабирование пула Collector с состоянием с помощью Load Balancing Exporter

Масштабирование горизонтально масштабированного пула Collector с состоянием, вероятно, требует использования [Load Balancing Exporter](https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/v0.145.0/exporter/loadbalancingexporter). Load Balancing Exporter превращает Collector в OTLP-совместимый балансировщик нагрузки, позволяющий маршрутизировать данные к определённому нижестоящему Collector на основе информации внутри OTLP-полезной нагрузки, такой как имя метрики.

Обратите внимание, что для метрик компонент Load Balancing Exporter имеет [уровень стабильности Development](https://github.com/open-telemetry/opentelemetry-collector/blob/v0.145.0/docs/component-stability.md). На данный момент он не рекомендуется для производственного использования.

#### Процессоры с состоянием

Стоит рассмотреть использование экспортёра, если вы масштабируете и используете любой из следующих компонентов с состоянием. Здесь мы рассматриваем только компоненты, включённые в Dynatrace Collector; вам необходимо самостоятельно определить лучшие настройки для других компонентов с состоянием. Вы также можете настроить, какая часть данных используется для маршрутизации. Лучший ключ зависит от вашего сценария использования, но ниже мы даём рекомендации.

* [Cumulative to Delta Processor](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/v0.145.0/processor/cumulativetodeltaprocessor): Точки данных для одной метрики должны отправляться на один и тот же Collector в течение периода сбора метрики. Поэтому ключ `metric_name` является хорошим значением по умолчанию для маршрутизации.
* [Tail Sampling Processor](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/v0.145.0/processor/tailsamplingprocessor): Для принятия решения о сэмплировании трейса процессор должен видеть все спаны внутри трейса. Поэтому все спаны должны отправляться на один Collector, и мы рекомендуем маршрутизацию по ключу `traceID`.
* [Span Metrics Connector](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/v0.145.0/connector/spanmetricsconnector): Коннектор должен видеть все спаны от сервиса, чтобы генерировать метрики о его производительности. Поэтому мы настоятельно рекомендуем маршрутизацию по ключу `service`.

#### Настройка Load Balancing Exporter

Есть два важных элемента в настройке [Load Balancing Exporter](https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/v0.145.0/exporter/loadbalancingexporter): ключ, используемый для маршрутизации данных, и метод, который экспортёр использует для обнаружения Collector в пуле.

Настройка ключа маршрутизации выполняется через параметр `routing_key`. Значения по умолчанию для каждого сигнала:

* Трейсы: `traceID`
* Метрики: `service`
* Логи: `traceID`, если присутствует, иначе случайный trace ID. Параметр `routing_key` не переопределяет это поведение и не влияет на маршрутизацию логов.

Мы рекомендуем оставить значения по умолчанию или установить их в соответствии с рекомендациями в разделе [Процессоры с состоянием](#stateful-processors) выше.

Другой важный параметр конфигурации — ключ `resolver`, который используется экспортёром для определения доступных Collector для пересылки данных. В Kubernetes мы рекомендуем использовать резолвер `k8s`, так как он является нативным для Kubernetes. В частности, он поддерживает динамическое обновление пула в зависимости от работающих подов Collector и будет добавлять или удалять Collector при изменении количества реплик. Он также удалит нездоровые Collector, обеспечивая выполнение требований высокой доступности, если также настроены повторные попытки через параметр `retry_on_failure`.

Обратите внимание, что настройка резолвера `static` с фиксированным пулом Collector может привести к потере данных, если Collector выйдет из строя и не будет заменён до исчерпания лимита повторных попыток. Collector, настроенные в пуле, фиксируются на время жизни балансирующего Collector.

#### Отказоустойчивость

Load Balancing Exporter поставляется с параметрами отказоустойчивости для снижения риска потери данных. Эти параметры предназначены как для работы с изменяющимся числом нижестоящих Collector, так и для решения проблем с отправкой данных на конкретный Collector. [Документация](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/v0.145.0/exporter/loadbalancingexporter#resilience-and-scaling-considerations) подробно описывает эти параметры и объясняет, как и когда их использовать.

#### Масштабирование балансирующих Collector

Поскольку Load Balancing Exporter использует детерминированный хеш для определения, какому нижестоящему Collector отправить данные, балансирующие Collector можно считать не имеющими состояния и, следовательно, масштабировать с помощью подходов, описанных в разделе [Масштабирование Collector без состояния](#scaling-stateless-collectors). Обратите внимание, что если резолверы балансирующих Collector обновляют свои нижестоящие пулы в разное время, это может привести к тому, что данные, предназначенные для одного Collector, будут временно отправлены на несколько Collector.

## Компоненты

В нашей конфигурации мы используем следующие компоненты.

### Receivers (приёмники)

В разделе `receivers` мы настраиваем [приёмник `otlp`](https://github.com/open-telemetry/opentelemetry-collector/tree/v0.145.0/receiver/otlpreceiver) для приёма данных по gRPC и HTTP.

### Exporters (экспортёры)

В разделе `exporters` мы настраиваем три [`loadbalancing` экспортёра](https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/v0.145.0/exporter/loadbalancingexporter), по одному для каждого сигнала. Все экспортёры настроены на использование резолвера `k8s`, который использует Kubernetes Service для определения пула Collector для отправки данных. Одна из причин разделения дальнейшей обработки по сигналам заключается в том, что каждый сигнал, вероятно, получает различный объём трафика: например, вы можете получать большой объём логов, некоторое количество трейсов и относительно мало метрик. Поэтому пул Collector для обработки логов должен быть больше, чем для обработки метрик; дополнительные Collector, выделенные для обработки меньшего количества метрик, могут расходовать ресурсы впустую.

### Service pipelines (конвейеры сервиса)

В наших конвейерах мы принимаем данные по OTLP и экспортируем их через Load Balancing Exporter для соответствующего сигнала без дополнительной обработки. Поскольку этот Collector предназначен исключительно для балансировки нагрузки, мы хотим выполнять как можно меньше обработки, чтобы он мог обрабатывать как можно больше данных.
