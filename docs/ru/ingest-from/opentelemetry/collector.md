Collector -- это сетевое сервисное приложение, которое можно использовать для пакетной обработки и преобразования телеметрических данных. Он действует как прокси и может принимать OTLP-запросы, а также данные из других источников, преобразовывать эти запросы в соответствии с определенными правилами и перенаправлять их на бэкенд.

![Обзор OpenTelemetry Collector](https://dt-cdn.net/images/collector-1062-e6ec25a6ee.png)

## Преимущества использования Collector

В целом использование Collector вместе с вашим сервисом может быть полезным, поскольку это позволяет вашему сервису быстро выгружать данные и обеспечивает дополнительную обработку, такую как повторные попытки, пакетирование, шифрование или фильтрацию конфиденциальных данных. Он централизует общие задачи обработки вместо их дублирования в каждом приложении.

Вам следует использовать Collector, если:

* Вам необходимо собирать данные из различных источников в разных форматах, и вам нужен простой способ обеспечить доставку всех данных на бэкенд, который иначе был бы несовместим.
* Вам необходима единая обработка атрибутов ваших телеметрических данных.

Collector является относительно легковесным компонентом, поэтому команды могут развертывать собственные экземпляры, чтобы не использовать одну и ту же конфигурацию.

Collector настраивается в одном YAML-файле. Это избавляет от необходимости просматривать несколько файлов и сокращает объем обслуживания. Подробнее о настройке см. Настройка OpenTelemetry Collector.

## Дистрибутивы

Collector поставляется в различных вариантах дистрибутивов с различными возможностями установки и развертывания.

### Dynatrace Collector

Дистрибутив Dynatrace для OpenTelemetry Collector -- это настроенная сборка Collector, предоставляемая Dynatrace и адаптированная для типичных сценариев использования в контексте Dynatrace. Он поставляется с оптимизированным и проверенным набором компонентов Collector.

Преимущества Dynatrace Collector

Dynatrace Collector предлагает следующие преимущества по сравнению с дистрибутивами OpenTelemetry Collector:

* Поддерживается службой поддержки Dynatrace (x86-64 и ARM64)
* Компоненты Collector проверены Dynatrace
* Патчи безопасности независимы от релизов OpenTelemetry Collector

Вы можете загрузить релиз Dynatrace Collector из [нашего репозитория](https://github.com/Dynatrace/dynatrace-otel-collector/releases/v0.44.0).

Он также доступен как контейнерный образ на [Github Packages](https://github.com/Dynatrace/dynatrace-otel-collector/pkgs/container/dynatrace-otel-collector%2Fdynatrace-otel-collector) и может быть загружен с помощью следующей Docker-команды:

```
docker pull ghcr.io/dynatrace/dynatrace-otel-collector/dynatrace-otel-collector:0.44.0
```

Дополнительную информацию о других доступных реестрах контейнеров можно найти в разделе [Развертывание Dynatrace OTel Collector](collector/deployment.md#container-image-registries "How to deploy Dynatrace OTel Collector.").

Dynatrace Collector поставляется с определенными компонентами, описанными в репозитории Dynatrace Collector на GitHub.
Полный список см. в разделе [Components](https://github.com/Dynatrace/dynatrace-otel-collector#components).

Ознакомьтесь с примерами использования OpenTelemetry Collector для получения конкретных примеров сценариев и конфигураций для отдельных компонентов.

### Дистрибутивы OpenTelemetry

Существует два основных дистрибутива OpenTelemetry для Collector:

* **Collector Core**
* **Collector Contrib**

Кроме того, OpenTelemetry также предоставляет **Collector Builder**, который позволяет собрать собственный пользовательский бинарный файл Collector.

Поддержка Dynatrace

Dynatrace предоставляет ограниченную поддержку для конфигураций Core, Contrib и Builder, охватывая только те компоненты и их версии, которые включены в Dynatrace Collector.

Для полностью поддерживаемого дистрибутива Collector см. [Dynatrace Collector](#dt-collector-dist).

#### Collector Core

[Дистрибутив Core](https://github.com/open-telemetry/opentelemetry-collector) содержит базовый прокси-сервис и несколько фундаментальных сервисных компонентов:

* Ресивер для OTLP через HTTP и gRPC
* Процессоры для пакетирования запросов и обеспечения ограничений использования памяти
* Экспортеры для вывода в консоль и OTLP через HTTP и gRPC

#### Collector Contrib

[Дистрибутив Contrib](https://github.com/open-telemetry/opentelemetry-collector-contrib) основан на Core и расширяет его функциональность, поставляя большое количество дополнительных ресиверов, процессоров и экспортеров, предоставленных сторонними разработчиками.

Учитывая, что дистрибутив Contrib является пакетом "все включено" и поставляется со всеми предварительно скомпилированными сервисными компонентами, он может использовать больше системных ресурсов (хранилище и память), чем оптимизированная сборка Collector. В целом мы рекомендуем использовать дистрибутив Contrib для целей тестирования, а для продуктивной среды -- пользовательскую сборку Collector (см. [Builder](#collector-builder)).

#### Collector Builder (ocb)

Помимо двух дистрибутивов, OpenTelemetry также предлагает [Collector Builder](https://github.com/open-telemetry/opentelemetry-collector/tree/v0.145.0/cmd/builder) -- инструмент командной строки, который позволяет собрать собственную пользовательскую версию Collector.

### Какой дистрибутив выбрать?

| Тип | Когда использовать? |
| --- | --- |
| Dynatrace Collector (рекомендуется) | Предпочтительный выбор для большинства сценариев использования. Поставляется с набором проверенных и стабильных компонентов Collector, обычно используемых с конфигурациями Dynatrace. |
| Пользовательская версия Builder | Когда необходимо полностью настроить экземпляр Collector и запускать только те компоненты, которые требуются для вашего сценария использования. |
| Core | Когда вам в первую очередь нужно конвертировать между протоколами OTLP (HTTP -- gRPC), обеспечить ограничения использования памяти или применить пакетирование запросов. |
| Contrib | Идеально для тестовых конфигураций, так как содержит все сторонние компоненты и не требует пользовательской сборки. Как правило, не рекомендуется для использования в продуктивных системах, поскольку обычно потребляет больше ресурсов и может быть менее стабильным, чем оптимизированный экземпляр Builder. |

## Компоненты

### Ресивер

Ресивер -- это компонент, который обеспечивает поступление данных в Collector. Он может принимать данные из нескольких источников. Многие ресиверы имеют настройки по умолчанию и не требуют значительной конфигурации.

Список доступных ресиверов и их базовую конфигурацию см. в официальной [документации OpenTelemetry по ресиверам](https://opentelemetry.io/docs/collector/configuration/#receivers).

### Процессор (необязательный)

Процессор -- это необязательный компонент, который определяет, что делать с данными.

Список доступных процессоров и их базовую конфигурацию см. в официальной [документации OpenTelemetry по процессорам](https://opentelemetry.io/docs/collector/configuration/#processors). OpenTelemetry предоставляет список [рекомендуемых процессоров](https://github.com/open-telemetry/opentelemetry-collector/tree/v0.145.0/processor#recommended-processors), но они являются необязательными.

### Экспортер

Экспортер -- это компонент, который отправляет обработанные данные на один или несколько бэкендов. Экспортеры могут поддерживать более одного источника данных.

Поскольку многие экспортеры требуют дополнительной настройки (например, конечной точки), обязательно проверьте официальную [документацию OpenTelemetry по экспортерам](https://opentelemetry.io/docs/collector/configuration/#exporters) для получения списка доступных экспортеров и их конфигураций.

### Сервисы

Сервисы используются для определения конвейеров, которые направляют данные через Collector. Они определяют, какие компоненты работают вместе для обработки данных OpenTelemetry.
