Полная поддержка предоставляется до окончания срока службы версии Kubernetes или OpenShift. После этого Dynatrace предоставляет поддержку в режиме обслуживания в течение приблизительно одного года. Даты окончания поддержки объявляются в [объявлениях об окончании поддержки](../../whats-new/technology/end-of-support-news.md#dto "Объявления об окончании поддержки технологий, поддерживаемых Dynatrace.").

Основное различие между полной поддержкой и поддержкой в режиме обслуживания заключается в том, что Dynatrace сокращает ежедневные тестовые мероприятия в период поддержки в режиме обслуживания.

В периоды полной поддержки и поддержки в режиме обслуживания каждая обнаруженная ошибка проходит оценку на возможность обратного портирования. В зависимости от серьезности и риска изменений исправление либо портируется обратно и выпускается в виде патча, либо исправляется в следующей версии. Подробности см. в соответствующих примечаниях к выпуску Dynatrace.

## Поддержка Dynatrace Operator

Dynatrace Operator доступен на следующих архитектурах:

* x86
* ARM
* ppc64le
* s390x [1](#fn-2-1-def)

1

Поддерживаются только развертывание cloud-native полного стека, мониторинг приложений и мониторинг хостов.

В случаях, когда проблемы, связанные с Dynatrace Operator, не удается воспроизвести на архитектурах x86 или ARM и они идентифицированы как специфичные для ppc64le, необходимо обратиться по адресу `dt-operator@ibm.com` для получения дополнительной поддержки. Дополнительную информацию можно найти в соответствующем open-source pull request для Dynatrace Operator на [GitHub](https://dt-url.net/ev034k3).

Dynatrace Operator отвечает за развертывание и управление жизненным циклом различных компонентов Dynatrace в Kubernetes и OpenShift (включая ActiveGate и OneAgent). Dynatrace Operator является проектом с открытым исходным кодом, поддерживаемым на [GitHub](https://dt-url.net/d7034gj). Он следует схеме [семантического версионирования](https://semver.org/) `major.minor.patch` с частотой выпуска минорных версий примерно каждые 2-3 месяца.

Три последние версии Dynatrace Operator тестируются с последними версиями Kubernetes и OpenShift. Кроме того, мы проводим оценку на возможность обратного портирования для каждой ошибки или уязвимости, чтобы проанализировать серьезность и риск исправления. Мы рекомендуем использовать последнюю патч-версию, поскольку вновь реализованные функции увеличивают минорную версию. Подробности см. в примечаниях к выпуску Dynatrace Operator.

Все версии Dynatrace Operator, которые не считаются снятыми с поддержки, находятся в режиме обслуживания, который включает наши стандартные процессы поддержки клиентов. Версии в режиме обслуживания не получают исправлений ошибок и обратных портов уязвимостей. См. объявления об окончании поддержки в [объявлениях об окончании поддержки](../../whats-new/technology/end-of-support-news.md#dto "Объявления об окончании поддержки технологий, поддерживаемых Dynatrace.").

## Известные проблемы и решения

Список известных проблем для версий Dynatrace Operator и их влияние на различные компоненты. Эти проблемы присутствуют в выпущенных версиях Dynatrace Operator и могут потребовать обновления минорной версии для их решения!

### Внедрение компонентов Dynatrace

Dynatrace Operator версии 1.7.0 Dynatrace Operator версии 1.7.1 Dynatrace Operator версии 1.7.2

#### Проблема

* Из-за оптимизации внедряемых монтирований (их объединения в `/var/lib/dynatrace`) компоненты Dynatrace больше не могут быть внедрены с OneAgent.

* Компоненты Dynatrace содержат конфигурацию в `/var/lib/dynatrace`, которая скрывается монтированиями, добавленными Webhook-инъекцией.

По умолчанию мониторинг пространства имен Dynatrace Operator (и, следовательно, компонентов Dynatrace) не включен. Проблема может проявиться, если флаг функции `feature.dynatrace.com/ignored-namespaces` используется для переопределения игнорируемых пространств имен и включения пространства имен Dynatrace Operator.

#### Решение

Настройте аннотацию пода `dynatrace.com/split-mounts` (требуется Dynatrace Operator версии 1.8.0+) на затронутых подах.

### Классический полный стек с обогащением метаданных

Dynatrace Operator версии 1.7.0 Dynatrace Operator версии 1.7.1 Dynatrace Operator версии 1.7.2

#### Проблема

* Классический полный стек и обогащение метаданных несовместимы и не могут использоваться для внедрения в одни и те же поды приложений.

И OneAgent, и Webhook-инъекция пытаются добавить монтирование в каталог `/var/lib/dynatrace` пода приложения. Эти монтирования несовместимы и не могут сосуществовать.

#### Решение

При использовании версий Dynatrace Operator ниже 1.7.0:

* С OneAgent версии 1.333 изменений не требуется.
* С OneAgent версии 1.331 или более ранней необходимо настроить отладочный флаг `remountOperatorEnrichment` для OneAgent.

Обновитесь до Dynatrace Operator версии 1.8.0 (когда будет доступна).

### Обнаружение доступности хостов

Dynatrace Operator версии 1.6.0 Dynatrace Operator версии 1.6.1 Dynatrace Operator версии 1.6.2 Dynatrace Operator версии 1.7.0 Dynatrace Operator версии 1.7.1

#### Проблема

* В средах Kubernetes -- особенно использующих автоскейлеры -- существуют трудности с надежным определением того, был ли узел намеренно удален или неожиданно вышел из строя. Эта неопределенность может приводить к большому количеству ложноположительных оповещений "Хост недоступен", влияя на точность мониторинга и качество оповещений.

#### Решение

Обновитесь до Dynatrace Operator версии 1.7.2+ или 1.6.3.

### Переключение на Node image pull с Code modules image

Dynatrace Operator версии 1.5.0 Dynatrace Operator версии 1.5.1 Dynatrace Operator версии 1.6.0 Dynatrace Operator версии 1.6.1 Dynatrace Operator версии 1.6.2

#### Проблема

При переключении с использования CSI-драйвера без `codeModulesImage` на использование с node image pull убедитесь, что файловая система CSI-драйвера не содержит модуль кода для указанного DynaKube. Если он присутствует, CSI-драйвер выйдет из строя и потребует ручного вмешательства для восстановления.

* При возникновении этой проблемы возврат к использованию без `codeModulesImage` восстановит работоспособность CSI-драйвера.
* Вы можете использовать команду `find` для проверки наличия загруженного модуля кода для DynaKube в файловой системе контейнера CSI `server`:

  ```
  > find /data -name latest-codemodule


  /data/_dynakubes/my-dynakube-1/latest-codemodule


  /data/_dynakubes/my-dynakube-2/latest-codemodule
  ```

#### Решение

* Обновитесь до Dynatrace Operator версии 1.7.0+.

Другие способы решения проблемы:

* Удалите DynaKube и создайте его заново. Это приведет к перерывам в мониторинге.

### Несовместимости с конкретными версиями компонентов

Dynatrace Operator версии 1.5.0+

Функция автоматического TLS-сертификата требует ActiveGate версии 1.307.35+.

* Если вы предпочитаете отключить эту функцию, установите флаг функции `feature.dynatrace.com/automatic-tls-certificate: false` в вашей конфигурации DynaKube.
