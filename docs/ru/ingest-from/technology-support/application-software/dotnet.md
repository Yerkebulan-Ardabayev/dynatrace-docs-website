# .NET


* Latest Dynatrace
* Справочник
* Чтение: 3 мин
* Обновлено 23 октября 2025 г.

Dynatrace OneAgent инструментирует ваши приложения .NET, размещая операторы трассировки в стратегических точках кода для трассировки кода, сбора метрик производительности, обнаружения ошибок, отслеживания зависимостей и многого другого.

Не каждое обнаруженное приложение .NET инструментируется по умолчанию. Dynatrace поддерживает набор правил для инструментирования определённых процессов (например, пулов приложений IIS, которые вы можете расширить собственными правилами). Чтобы узнать основы настройки мониторинга групп процессов (автоматический глубокий мониторинг, пользовательские правила мониторинга и встроенные правила мониторинга), см. [Настройка мониторинга групп процессов](../../../observe/infrastructure-observability/process-groups/configuration/pg-monitoring.md "Способы настройки мониторинга групп процессов").

## Обработка логов с помощью парсеров технологических пакетов

С помощью OpenPipeline вы можете использовать и настраивать технологические пакеты. Технологический пакет — это библиотека парсеров (правил обработки), которые обрабатывают логи различных технологий, таких как Java, .NET и Microsoft IIS.

Парсеры помогают улучшить фильтрацию, устранение неполадок, метрики, оповещения и дашборды за счёт эффективного извлечения уровней логов и релевантных атрибутов. Вы также можете использовать технологические пакеты для структурирования логов технологий, которые не поддерживаются Dynatrace из коробки.

![syslog-bundles](https://dt-cdn.net/images/env-syslogbundles-2589-90e4e38b45.png)

Дополнительную информацию см. в разделе [Обработка логов с помощью парсеров технологических пакетов](../../../platform/openpipeline/use-cases/tutorial-technology-processor.md "Настройте конвейер обработки для структурирования технологических логов в соответствии с семантическим словарём Dynatrace.").

## Ограничения

### Method hotspots на Linux

Модуль кода .NET использует сигналы POSIX для захвата стековых трассировок для функции method hotspots.

Поскольку эти сигналы могут прерывать приложение в произвольные моменты, некоторые приложения/библиотеки (в частности, библиотеки .NET с нативными зависимостями) могут работать некорректно при включённой функции method hotspots.

Затронутое приложение может проявлять такие симптомы, как:

* Проблемы с подключением
* Искажённые текстовые данные
* Сбои проверок готовности и работоспособности Kubernetes

Единственное текущее решение — отключить соответствующие функции OneAgent для затронутой группы процессов:

* Capture method hotspot information in PurePaths
* Capture background CPU method hotspot information
* .NET Async Method Hotspots

Method hotspots может некорректно захватывать стековые трассировки на операционных системах на базе musl-libc, таких как Alpine. Это связано с тем, что определённая отладочная информация удаляется библиотекой musl-libc. Переход на операционную систему на базе glibc (Debian/Ubuntu) может устранить эту конкретную проблему.

* Method hotspots .NET недоступны для Linux на ARM64 (AArch64).

### Тримминг

Необязательная функция .NET [тримминг автономных развёртываний и исполняемых файлов](https://docs.microsoft.com/en-us/dotnet/core/deploying/trim-self-contained) была введена для оптимизации размера пакетируемых приложений.
Приложения с триммингом не поддерживаются OneAgent, поэтому эту опцию необходимо отключить. [UiPath](https://www.uipath.com/), например, использует тримминг, что делает невозможным инструментирование с помощью OneAgent.

### Приложения в виде одного файла

.NET SDK поддерживает сборку приложения в виде [одного файла](https://learn.microsoft.com/en-us/dotnet/core/deploying/single-file/overview?tabs=cli), объединяющего все зависимости в один платформозависимый исполняемый файл.

В зависимости от того, используете ли вы [развёртывание, зависящее от фреймворка](https://learn.microsoft.com/en-us/dotnet/core/deploying/#publish-framework-dependent) или [автономные приложения](https://learn.microsoft.com/en-us/dotnet/core/deploying/#publish-self-contained), среда выполнения также может быть включена в исполняемый файл.

Также учитывайте следующую информацию:

* OneAgent не поддерживает инструментирование модулей, включённых в исполняемый файл, что приводит к тому, что приложения, собранные как одиночные файлы и автономные, вообще не поддерживаются.
* Приложения, собранные как одиночный файл и зависящие от фреймворка, могут быть частично инструментированы.
* Сборки .NET, включённые в исполняемый файл одного файла, не могут быть инструментированы (логика трассировки не может быть размещена для этих сборок).
