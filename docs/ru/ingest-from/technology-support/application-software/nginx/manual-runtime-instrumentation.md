Кодовый модуль NGINX основывается на предположениях, сделанных заблаговременно (ahead-of-time), об объявлениях внутренних структур данных NGINX и их расположении в памяти в процессе автоматического инструментирования. Если базовые объявления структур данных были исправлены (то есть исходный код, определяющий эти структуры, был изменён) и заблаговременные предположения больше не действительны, автоматическое инструментирование кодовым модулем может вызвать проблемы в NGINX из-за чтения из или записи в неправильные области памяти.

Чтобы избежать подобной ситуации, кодовый модуль NGINX пытается обнаружить изменённые объявления структур данных NGINX. Если изменённое объявление обнаружено, кодовый модуль отключает автоматическое инструментирование и отображает следующее уведомление на странице процесса в веб-интерфейсе:

Notification

* В этом процессе обнаружены несовместимые модули NGINX. Автоматическое инструментирование было отключено. Чтобы принудительно инструментировать NGINX, см. [Runtime instrumentation](manual-runtime-instrumentation.md "Learn how to force instrumenting patched/non-standard NGINX binaries during runtime.").

## Инструментирование изменённого NGINX

Кодовый модуль NGINX может инструментировать изменённый NGINX для [поддерживаемых версий](../nginx.md#nginx-versions "Learn the details of Dynatrace support for NGINX."), проверяя объявления внутренних структур данных NGINX при запуске (во время выполнения), вместо того чтобы полагаться на заблаговременные предположения.

Примером изменённого NGINX является бинарный файл NGINX, распространяемый вместе с Kong Gateway.

Поскольку исправления могут существенно изменить внутреннюю работу NGINX, не гарантируется, что NGINX, инструментированный с использованием инструментирования во время выполнения, будет работать как ожидается. Примите во внимание следующие ограничения.

Ограничения инструментирования во время выполнения

* Инструментирование во время выполнения зависит от наличия отладочных символов для бинарного файла NGINX, что бывает не всегда.
* Инструментирование во время выполнения добавляет заметную задержку при запуске NGINX (10 секунд и более).
* Инструментирование во время выполнения на Linux ARM64 требует OneAgent версии 1.313+.

  NGINX или Kong Pod в Kubernetes не запускается из-за таймаута

  В облачных развёртываниях скорректируйте таймауты запуска Pod или контейнера, чтобы предотвратить превышение NGINX таймаута при запуске.

  Если Pod NGINX или Kong в Kubernetes не запускается, поищите упоминание о сбое проверок работоспособности (liveness) или готовности (readiness) в логах и установите достаточно высокие значения начальной задержки, таймаута и порога отказов для этих проверок. Точная конфигурация зависит от развёртывания.
* Инструментирование во время выполнения требует больше памяти при запуске NGINX. Такое повышенное пиковое потребление памяти может привести к завершению работы Pod Kubernetes (или другими контейнерными средами выполнения) при строгих ограничениях памяти. Скорректируйте ограничения памяти, чтобы учесть инструментирование во время выполнения.

Чтобы принудительно инструментировать изменённый NGINX во время выполнения:

1. Добавьте переменную окружения `DT_NGINX_FORCE_RUNTIME_INSTRUMENTATION` в ваш NGINX:

```
DT_NGINX_FORCE_RUNTIME_INSTRUMENTATION=on
```

2. Перезапустите NGINX для применения переменной окружения.
