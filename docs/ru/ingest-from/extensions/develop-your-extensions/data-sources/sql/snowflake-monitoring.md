# Конфигурация мониторинга базы данных Snowflake


После определения области конфигурации необходимо указать следующее:

* Базы данных, из которых будут собираться данные
* ActiveGate для выполнения расширения и подключения к устройствам

## Пример тела запроса

Пример тела запроса для активации расширения Snowflake Database:

```
[


{


"value": {


"enabled": true,


"description": "My SnowFlake DB extension",


"version": "0.1.1",


"featureSets": [


"io",


"cpu",


],


"sqlSnowflakeRemote": {


"endpoints": [


{


"host": "sqlserver.org",


"port": 1521,


"databaseName":"SNOWFLAKE_SAMPLE_DATA",


"warehouse":"yourwarehouse",


"schema":"yourschema",


"authentication": {


"scheme": "basic",


"username": "admin",


"password": "password"


}


}


]


}


},


"scope": "ag_group-default"


}


]
```

## Параметры

### Enabled (Включено)

Если задано значение `true`, конфигурация активна и Dynatrace немедленно начинает мониторинг.

### Description (Описание)

Понятное описание особенностей данной конфигурации мониторинга.

### Version (Версия)

Версия данной конфигурации мониторинга. Обратите внимание, что одно расширение может выполнять несколько конфигураций мониторинга.

### Feature sets (Наборы функций)

Добавьте список наборов функций, которые необходимо отслеживать. Чтобы включить отчётность по всем наборам функций, добавьте `all`.

```
"featureSets": [


"cpu",


"io"


]
```

### Endpoints (Конечные точки)

В одной конфигурации мониторинга в разделе `SQLSnowflakeRemote` можно определить до 20 000 конечных точек.

```
"sqlSnowflakeRemote": {


"endpoints": [


{


"host": "your-snowflake.com",


"port": 1521,


"databaseName":"SNOWFLAKE_SAMPLE_DATA",


"warehouse":"yourwarehouse",


"schema":"yourschema",


"authentication": {


"scheme": "basic",


"username": "admin",


"password": "password"


}


}


]


}
```

Для определения сервера базы данных Snowflake добавьте следующие сведения в раздел `endpoints`:

* Хост
* Порт
* Имя базы данных
* Хранилище (Warehouse)
* Схема
* Учётные данные для аутентификации

### Authentication (Аутентификация)

Учётные данные аутентификации, передаваемые в API Dynatrace при активации конфигурации мониторинга, обфусцируются и их невозможно получить обратно.

#### Хранилище учётных данных

Тип аутентификации с хранилищем учётных данных обеспечивает более безопасный подход к использованию расширений за счёт защищённого хранения учётных данных пользователя и управления ими. Для его использования вы должны быть владельцем учётных данных и иметь хранилище учётных данных, отвечающее следующим критериям:

* **Тип учётных данных** — имя пользователя и пароль
* **Область применения учётных данных** — включены области Synthetic (при использовании внешнего хранилища) и Extension authentication
* **Только для владельца** включено только для владельцев учётных данных

```
"authentication": {


"scheme": "basic",


"useCredentialVault": true,


"credentialVaultId": "some-credential-vault-id"


}
```

### Scope (Область применения)

Обратите внимание, что каждому хосту ActiveGate, запускающему расширение, необходим корневой сертификат для проверки подлинности расширения. Подробнее см. в разделе [Подписание расширения](../../sign-extensions.md "Узнайте, как подписать расширение для безопасного распространения в вашей среде с помощью фреймворка Dynatrace Extensions.").

Область применения — это группа ActiveGate, которая будет выполнять расширение. Только один ActiveGate из группы будет запускать данную конфигурацию мониторинга. Если вы планируете использовать один ActiveGate, назначьте его в отдельную группу. Вы можете назначить ActiveGate в группу во время или после установки. Подробнее см. в разделе [Группа ActiveGate](../../../../dynatrace-activegate/activegate-group.md "Основные концепции групп ActiveGate.").

При определении группы ActiveGate используйте следующий формат:

```
"scope": "ag_group-<ActiveGate-group-name>",
```

Замените `<ActiveGate-group-name>` фактическим именем группы.
