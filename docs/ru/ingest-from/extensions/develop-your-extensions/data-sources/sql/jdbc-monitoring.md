# Настройка мониторинга JDBC


* Latest Dynatrace
* How-to guide
* 3-min read

Источник данных SQL расширений Dynatrace позволяет выполнять запросы к любой базе данных, допускающей подключение через драйвер JDBC, помимо всех поставщиков баз данных, поддерживаемых по умолчанию. Для таких баз данных требуется выполнить дополнительные шаги.

## Предварительные требования

Поддерживаются драйверы на базе JDBC 4.0+.

## Загрузка драйвера JDBC на ActiveGate

Необходимо предоставить драйвер выбранного поставщика базы данных, чтобы ActiveGate, выполняющий расширение, мог подключиться к базе данных.

Пример для MariaDB

Для MariaDB драйвер можно загрузить со страницы [Download MariaDB](https://mariadb.com/downloads/connectors/connectors-data-access/java8-connector).

Загрузите коннектор Java 8+, независимый от платформы, а именно файл `mariadb-java-client-3.5.0.jar`.

Загрузите драйвер JDBC на ActiveGate, принадлежащий к группе, назначенной для выполнения вашего расширения:

**Windows**: `C:\ProgramData\dynatrace\remotepluginmodule\agent\conf\userdata\libs`
**Linux**: `/var/lib/dynatrace/remotepluginmodule/agent/conf/userdata/libs/`

Убедитесь, что пользователь `dtuserag` имеет права на чтение драйвера. Например, для Linux установите `CHMOD` на `775`.

## Настройка мониторинга

После определения области конфигурации необходимо указать следующее:

* Базы данных, из которых нужно собирать данные, и их данные аутентификации
* Узлы ActiveGate для выполнения расширения и подключения к вашим устройствам. На таких ActiveGate должен быть загружен соответствующий [драйвер JDBC](#upload).

Пример JSON-запроса для активации расширения JDBC:

```
[


{


"value": {


"enabled": true,


"description": "My JDBC extension",


"version": "0.0.1",


"featureSets": [


"statements"


],


"jdbcRemote": {


"endpoints": [


{


"host": "193.36.194.170",


"port": 3306,


"connectionString": "jdbc:mariadb://193.36.194.170/mysql",


"authentication": {


"scheme": "basic",


"useCredentialVault": false,


"username": "user",


"password": "password"


}


}


]


}


},


"scope": "ag_group-someAgGroup"


}


]
```

Обратите внимание, что необходимо указать как конечную точку (хост и порт), так и соответствующую строку подключения.

Меры безопасности

Строка подключения SQL по своей природе может содержать конфиденциальную информацию, такую как учётные данные пользователей. По возможности избегайте включения секретной информации в строку подключения. Если ваша строка подключения содержит конфиденциальную информацию:

* Ограничьте доступ на чтение и запись к конфигурации мониторинга JDBC. Убедитесь, что только пользователи, имеющие доступ к секретным данным, имеют права на чтение и запись конфигураций.
* В отличие от данных аутентификации, строка подключения не хешируется. Просматривайте и редактируйте конфигурацию только в безопасной среде, где неавторизованные пользователи не могут её видеть.

## Параметры

### Enabled

Если установлено значение `true`, конфигурация активна и Dynatrace немедленно начинает мониторинг.

### Description

Метка конфигурации, которая должна содержать основные сведения о специфике данной конфигурации мониторинга.

### Version

Версия данной конфигурации мониторинга.

### Feature sets

Добавьте список наборов функций, которые вы хотите мониторить.

```
"featureSets": [


"cpu",


"io"


]
```

### Endpoints

В одной конфигурации мониторинга в разделе `jdbcRemote` можно определить до 20 000 конечных точек.

```
"jdbcRemote": {


"endpoints": [


{


"host": "jdbchost",


"port": 3306,


"connectionString": "jdbc:mariadb://193.36.194.170/mysql",


"authentication": {


"scheme": "basic",


"useCredentialVault": false,


"username": "admin",


"password": "password"


}


}


]


}
```

Для определения сервера JDBC-базы данных добавьте следующие сведения в раздел `endpoints`:

* Хост
* Порт
* Строка подключения
* Учётные данные для аутентификации

### Authentication

Данные аутентификации, передаваемые в API Dynatrace при активации конфигурации мониторинга, обфусцируются и не могут быть извлечены.

#### Хранилище учётных данных

Тип аутентификации через хранилище учётных данных обеспечивает более безопасный подход к использованию расширений благодаря надёжному хранению и управлению учётными данными пользователей. Для использования этого способа вы должны быть владельцем учётных данных и иметь хранилище учётных данных, соответствующее следующим критериям:

* **Тип учётных данных** — пользователь и пароль
* **Область учётных данных** — включены области Synthetic (в случае использования внешнего хранилища) и Extension authentication
* **Доступ только для владельца** включён только для владельцев учётных данных

```
"authentication": {


"scheme": "basic",


"useCredentialVault": true,


"credentialVaultId": "some-credential-vault-id"


}
```

### SSL

ActiveGate версии 1.295+

Включите SSL, чтобы источник данных проверял сертификат сервера и использовал SSL-шифрование вместо нативного шифрования.

```
"ssl": true
```

#### Включение SSL без локального хранилища доверенных сертификатов

Когда SSL включён и цепочка сертификатов сервера является публично проверяемой (например, выпущена Azure или другими известными центрами сертификации), нет необходимости вручную создавать хранилище доверенных сертификатов. Система автоматически доверяет сертификату сервера на основе доверенных центров сертификации в среде.

Однако если вам необходимо использовать локальное хранилище доверенных сертификатов для сертификатов, не являющихся глобально признанными, или для дополнительных мер безопасности

1. В каталоге `userdata` на узлах ActiveGate, выполняющих источник данных SQL, вручную создайте хранилище PKCS12 с именем `sqlds_truststore` и паролем `sqlds_truststore`.

   Команда для создания хранилища с помощью keytool:

   ```
   keytool -genkey -keystore sqlds_truststore -storepass sqlds_truststore -keyalg DSA
   ```

   Расположение каталога `userdata`:

   * Windows: `%PROGRAMDATA%\dynatrace\remotepluginmodule\agent\conf\userdata`
   * Unix: `/var/lib/dynatrace/remotepluginmodule/agent/conf/userdata`
2. Добавьте в него сертификат сервера.

   Команда для импорта сертификата с помощью keytool:

   ```
   keytool -import -keystore sqlds_truststore -file .\ora.crt -alias oracle
   ```

#### Проверка SSL-сертификатов

ActiveGate версии 1.269+

Сертификат дополнительно проверяется по имени хоста, что означает, что домен из сертификата должен совпадать с доменом конечной точки, указанным в конфигурации мониторинга.

Включите эту опцию при подключении к базам данных с использованием пользовательских сертификатов.

```
"validateCertificates": true
```

### Scope

Обратите внимание, что на каждом узле ActiveGate, выполняющем ваше расширение, должен быть корневой сертификат для проверки подлинности расширения. Дополнительную информацию см. в разделе Подписание расширения.

Область действия — это группа ActiveGate, которая будет выполнять расширение. Только один ActiveGate из группы будет выполнять данную конфигурацию мониторинга. Если вы планируете использовать один ActiveGate, назначьте его в выделенную группу. Вы можете назначить ActiveGate в группу во время или после установки. Дополнительную информацию см. в разделе Группа ActiveGate.

Используйте следующий формат при определении группы ActiveGate:

```
"scope": "ag_group-<ActiveGate-group-name>",
```

Замените `<ActiveGate-group-name>` фактическим именем.
