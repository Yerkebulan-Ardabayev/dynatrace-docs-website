# Устранение неполадок

* Последняя версия Dynatrace
* Устранение неполадок
* 3 минуты на чтение
* Опубликовано 10 июля 2023 г.

При работе с расширениями Dynatrace, использующими источник данных SNMP, могут возникать проблемы, требующие устранения неполадок.

## Статус конфигурации не OK

При создании или обновлении конфигурации мониторинга может потребоваться несколько минут для её полной активации и начала мониторинга. До этого момента статус конфигурации может изменяться на Warning или Error, пока конфигурация планируется к отправке на конечную точку, ставится в очередь на загрузку, активируется, проверяется и запускается. Подождите не менее 5 минут. Если статус по-прежнему не OK, щёлкните цветную точку рядом с ним — откроется интерфейс просмотра журнала с подробными сведениями.

### Ошибки Fastcheck

Fastcheck — это простой SNMP-запрос Get, цель которого — получить один OID от устройства, представляющий его системное имя. Устройству даётся 18 секунд на ответ, иначе проверка считается неудачной. Это самый первый шаг перед сбором каких-либо других данных с устройства.

Ошибки Fastcheck указывают на проблему со связью с устройством:

* Неверные учётные данные для подключения к устройству
* Сетевые брандмауэры, блокирующие связь
* Неправильно настроенные устройства, не допускающие SNMP-запросы

### GetBulk вернул ошибку

GetBulk — это операция SNMP-запроса, используемая для получения данных с устройства. Когда это сообщение появляется в тексте ошибки, это означает, что устройство доступно (FastCheck прошёл), но данные не удалось получить.

Этот тип ошибки может иметь несколько причин:

* Предоставленные учётные данные (например, строка сообщества) недействительны
* Сеть нестабильна, что вызывает проблемы со связью
* Слишком много данных для получения; попробуйте уменьшить наборы функций или оптимизировать дополнительные настройки

### Ошибки недопустимой конфигурации

Ошибки недопустимой конфигурации указывают на данные, введённые в поля конфигурации мониторинга. Хотя параметры устройства говорят сами за себя, фильтры переменных должны соответствовать синтаксису, указанному в предыдущем разделе.

### Высокая загрузка ЦП

Статус `HIGH_CPU` означает, что на ActiveGate достигнуто максимально допустимое потребление ЦП для модуля источника данных контроллера выполнения расширений (EEC).

* Объём данных не может быть собран и обработан без превышения встроенного лимита ресурсов в 5% ЦП.
* Попробуйте сначала включить меньше наборов функций (что подразумевает меньше метрик и, следовательно, меньше запросов для обработки) или распределите наборы функций по нескольким конфигурациям.

### Журналы расширений

Журналы расширений можно найти в [каталогах ActiveGate](../../../../dynatrace-activegate/configuration/where-can-i-find-activegate-files.md "Узнайте, где хранятся файлы ActiveGate в системах Windows и Linux."). Ищите запись `Extensions configuration, logs` в столбце **Purpose of**.

## Оптимизация для крупных устройств

Конфигурации мониторинга снабжены набором дополнительных настроек, влияющих на то, как данные запрашиваются с устройства через SNMP. Значения по умолчанию работают в большинстве случаев, но их можно скорректировать при возникновении таких проблем, как отсутствие данных.

* Timeout и Retries определяют максимальное время ожидания ответа на SNMP-запрос и количество повторных попыток в случае сбоев.
* Max. repetitions определяет, сколько раз OID (идентификатор метрики в SNMP) может повторяться в рамках одного ответа на SNMP GetBulk-запрос, когда одна и та же метрика собирается для нескольких объектов/экземпляров. Меньшее значение означает большее количество запросов между расширением и устройством для сбора большого набора данных.

  Из-за скорости и ненадёжности протокола SNMP эффективнее использовать меньшее значение (например, 20). Значение по умолчанию = 50.
* Max. OIDs per query определяет максимальное количество OID, которые можно запросить в рамках одного SNMP GetBulk-запроса.

  В очень крупных средах рекомендуется установить это значение равным 5. Это повышает производительность за счёт дополнительного разделения нагрузки на большее количество запросов.
