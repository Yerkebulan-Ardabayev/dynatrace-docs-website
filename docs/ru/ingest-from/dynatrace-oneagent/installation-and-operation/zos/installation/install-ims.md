# Установка модуля IMS

# Установка модуля IMS

* Latest Dynatrace
* Время чтения: 21 мин
* Обновлено 18 нояб. 2025

С помощью модуля IMS вы можете получить наблюдаемость для ваших транзакций и программ IMS, включая вызовы IBM MQ и баз данных.

## Установка

Модуль IMS перехватывает данные о различных событиях обработки транзакций IMS и пересылает данные мониторинга в подсистему zDC через общие буферы.

Для установки модуля Dynatrace IMS:

1. Установите модуль IMS в Control Region каждой системы IMS DB/DC и DCCTL, которую вы хотите мониторить. Этого достаточно для охвата всех регионов обработки сообщений, связанных с Control Region.
   Обратите внимание, что установка модуля IMS в систему, работающую только как DBCTL, не поддерживается.
2. Необходимо добавить exit Dynatrace в каждый IMS Connect, который вы хотите мониторить.
3. Необходимо добавить [модуль z/OS Java](/docs/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/install-zos-java#middleware "Настройка мониторинга Java на z/OS с помощью модуля Java.") в каждый IMS SOAP Gateway, который вы хотите мониторить.

Перезапуск IMS

4. Необходимо повторно выполнить задание инъекции после каждого перезапуска IMS. Подробности см. в разделе [Заметки по инъекции](#injection-notes).

### Обновление ранее установленной версии Dynatrace

Если у вас установлен Dynatrace и вы обновляетесь до более новой версии, вы можете перейти к разделу [Обновление без перезапуска региона](#update) далее в этом документе.

### IMS Connect

Добавьте авторизованный набор данных Dynatrace `<hlq>.SZDTAUTH` в STEPLIB задания IMS Connect, чтобы IMS Connect мог загрузить exit Dynatrace HWSTECL0.

* Если вы используете IMS Connect Extensions, расположите SZDTAUTH после библиотеки IMS Connect Extensions.
* Если вы используете локально разработанный exit HWSTECL0, расположите SZDTAUTH перед набором данных, содержащим локальный exit.

Exit IMS Connect может быть настроен для создания узлов PurePath в распределённой трассировке. Активируйте необходимую [функцию OneAgent](/docs/ingest-from/dynatrace-oneagent/oneagent-features "Управление функциями OneAgent глобально и по группам процессов.") **z/OS IMS Connect**.

* Если exit IMS Connect настроен для создания узлов PurePath, exit будет подключаться к имени подсистемы zDC по умолчанию. Целевое имя подсистемы zDC можно переопределить, указав следующее DDNAME и ключевой параметр в JCL запуска IMS Connect:

```
//ZDTPARMS DD *



ZDCID=<zDC_Id>
```

Exit IMS Connect необходим для поддержки протокола IMS Connect API и для поддержки IMS SOAP Gateway в случаях, когда SOAP Gateway не был настроен для вставки идентификатора трассировки с помощью команды трассировки `iogmgmt`.

### IMS Control region

Необходимо запустить утилиту инъекции для установки модуля IMS в control region.

1. Авторизуйте `<hlq>.SZDTAUTH`, добавив его в список APF.
2. Запустите задание ZDTINST из PDS `<hlq>.SZDTSAMP`.

Подробную информацию о параметрах также можно найти в PDS-члене ZDTINST.

Пример JCL с позиционными параметрами:

```
//S1       EXEC PGM=ZDTIINST,PARM='<IMS_Id>,<zDC_Id>'



//STEPLIB  DD DISP=SHR,DSN=<hlq>.SZDTAUTH
```

Пример JCL с ключевыми параметрами:

```
//S1       EXEC PGM=ZDTIINST



//STEPLIB  DD DISP=SHR,DSN=<hlq>.SZDTAUTH



//SYSIN DD *



IMSID=<IMS_Id>



ZDC=<zDC_Id>
```

Утилита инъекции (ZDTIINST) поддерживает изменение размера таблиц `pathid` без необходимости перезапуска IMS и повторной инъекции модуля. Размер таблицы путей можно изменить, используя код действия `M` с указанием сегмента таблицы путей в качестве шестого параметра и, необязательно, целевого IMSID в качестве седьмого параметра.

Ниже приведён полный список и описание позиционных параметров утилиты инъекции. Все параметры должны быть в верхнем регистре:

```
// ...    PARM='<IMS_Id>,<zDC_Id>,<ActionCode>|<PathTableSize>,



<WaitZdcMin>,<Y|N>,<PathTableSegm>,<PathTableIms>'
```

Параметры утилиты инъекции могут быть указаны как позиционные или ключевые. Есть семь позиционных параметров. Ключевые параметры должны быть указаны в файле SYSIN. Позиционные параметры в операторе JCL EXEC по-прежнему действительны, однако любые новые параметры, добавляемые в будущем, будут добавляться только как ключевые значения SYSIN. Если найдено значение параметра EXEC PARM, параметры SYSIN не учитываются. Рекомендуется использовать файл SYSIN и ключевые параметры.

## Заметки по трассировке транзакций BMP

Автоматическая трассировка для транзакционно-ориентированных BMP (входящее сообщение поступает из очередей сообщений IMS) будет аналогичной существующей трассировке для регионов MPP/IFP: если входящее сообщение отслеживается сенсором control region, путь будет начат для транзакции в BMP-регионе.
Автоматическая трассировка для работы, выполняемой пакетными (нетранзакционными) BMP, возможна только если входящее сообщение поступает через MQGET (см. ограничения ниже). В противном случае поддержка пакетных BMP означает, что агент выполняет службы инициализации для BMP-региона (выделяет рабочие области и размещает сенсоры), но не начинает никаких путей.
Для транзакционно-ориентированных и пакетных BMP путь начинается для BMP-региона всякий раз, когда приложение выполняет MQGET, при условии выполнения всех следующих условий:

1. Инструментирование включено для фильтра z/OS IMS MPR MQ в настройках интерфейса.
2. Другой сенсор ещё не начал путь для BMP-региона.
3. Имя очереди MQ для MQGET включено или не исключено из списка фильтров IMS MQ (**Settings** > **Mainframe** > **IBM MQ Filters**).
4. Вызов DL/I CHKP выполнен перед первым MQGET. Это требуется только для пакетных BMP.

Для транзакционно-ориентированных и пакетных BMP SDK может использоваться для инструментирования приложения, чтобы отправлять сообщения событий для начала и завершения путей.
Нет возможности ограничить трассировку конкретными именами заданий BMP или PSB, а также отслеживать пакетную работу, отличную от IMS BMP.

## Заметки по сборке мусора

При обновлении модуля IMS вы можете освободить хранилище ECSA, используемое старой версией модуля IMS.

После запуска задания инъекции для обновлённого модуля IMS необходима активность Message Region IMS для обновления внутренних управляющих блоков, чтобы они указывали на новую версию модуля IMS. После этого вы можете повторно запустить задание инъекции с кодом действия `G` в параметре номер 3. Это пометит старую версию модуля IMS как кандидата на удаление.

По истечении не менее 3 часов запустите задание инъекции с кодом действия `G` в параметре номер 3 ещё раз. Это освободит хранилище ECSA, содержащее старую версию модуля IMS.

Вы можете многократно запускать задание инъекции с кодом действия `G` в параметре номер 3; никаких негативных побочных эффектов нет. Но для того чтобы процесс сборки мусора освободил неиспользуемое хранилище модуля, процесс должен быть вызван как минимум дважды после IPL, с интервалом не менее 3 часов между первым и вторым вызовом.

## Заметки по инъекции

Запустите zDC хотя бы один раз после IPL системы перед выполнением задания инъекции модуля IMS.

Если целевой zDC не был запущен с момента последнего IPL, задание инъекции ожидает его запуска. Время ожидания по умолчанию составляет 30 минут, но может быть задано параметром `<WaitZdcMin>`. Если zDC всё ещё не запущен по истечении этого времени, инъекция не может быть выполнена и задание инъекции завершается с пользовательским кодом `ABEND` 100.

Если zDC был запущен хотя бы один раз с момента последнего IPL системы, задание инъекции может завершиться даже когда zDC в данный момент не запущен. В этом случае выдаются предупреждающие сообщения о том, что zDC не запущен и сбор данных недоступен, и задание инъекции завершается с `RC=4`.

Запустите и полностью инициализируйте систему IMS перед выполнением задания инъекции модуля IMS. Если вы остановите и запустите систему IMS, задание инъекции необходимо выполнить повторно. Если вы запустите новый зависимый регион IMS или остановите и перезапустите его, никаких действий не требуется.

При обновлении модуля IMS в ранее инъецированном IMS повторно выполните задание инъекции модуля IMS для загрузки обновлённых загрузочных модулей IMS. Не указывайте код действия для задания инъекции в параметре номер 3. IMS **не** требует перезапуска после обновления модуля IMS.

Мы рекомендуем использовать средство автоматизации, такое как IMS Time-Controlled Operations (TCO), для выполнения задания инъекции модуля IMS.

Все сообщения инъекции модуля IMS записываются в журнал задания. Вы можете контролировать уровень сообщений, включая определённые DDNAME в JCL задания инъекции. Если DDNAME не указаны, по умолчанию используется уровень 4 для информационных сообщений. Конкретные DDNAME и уровни сообщений:

* `//DTMSGLV0 DD SYSOUT=*` Для сообщений DEBUG уровня 0 (finest) и выше
* `//DTMSGLV2 DD SYSOUT=*` Для сообщений DEBUG уровня 2 (fine) и выше
* `//DTMSGLV4 DD SYSOUT=*` Для информационных сообщений уровня 4 и выше
* `//DTMSGLV5 DD SYSOUT=*` Для предупреждений уровня 5 и выше

Неудачные задания инъекции обычно приводят к пользовательскому коду `ABEND` 100. В большинстве случаев причину неудачи можно определить по предшествующим диагностическим сообщениям, записанным в SYSPRINT.

В зависимости от параметров вашей системы z/OS, дамп симптомов также может быть записан в журнал задания, но дамп не захватывается, если в JCL задания инъекции не указано одно из следующих DDNAME:

```
//SYSABEND DD SYSOUT=*



//SYSMDUMP DD DSN=DT.IMSAGENT.INJECT.SDUMP,DISP=(OLD,KEEP,KEEP)
```

Оба примера JCL-членов ZDTINST (утилита инъекции) и ZDTIDIAG (диагностическая утилита) содержат завершающий шаг для выполнения процедуры ZDTZLOGC. Функция шага ZDTZLOGC -- копирование вывода spool задания в файл журнала HFS, чтобы он автоматически включался в архив поддержки. Для этой функции предусмотрены новая JCL-процедура (ZDTZLOGC), программа REXX (ZDTZLOGR) и программа ассемблера (ZDTZLOGS). Для реализации выполните следующее:

1. Скопируйте пример JCL-члена ZDTZLOGC в PDS для хранения модифицированного JCL. В JCL PROC необходимо изменить DSN для DD-имён SYSEXEC и SZDTAUTH.
   `//SYSEXEC DD DISP=SHR,DSN=<hlq>.SZDTSAMP` <- измените `DSN` на расположение REXX exec `ZDTZLOGR` (см. шаг 3)
   `//SZDTAUTH DD DISP=SHR,DSN=<hlq>.SZDTAUTH` <- измените `DSN` на расположение программы `ZDTZLOGS`
2. Оба примера ZDTINST и ZDTIDIAG содержат оператор JCLLIB, который необходимо изменить, а шагу процедуры ZDTZLOGC необходимо указать `<ZDC Id>`.
   `JCLLIB ORDER=(<hlq>.SZDTSAMP)` <- измените `DSN` на расположение JCL-процедуры `ZDTZLOGC` (см. шаг 1)
   `//ZDTZLOGC EXEC ZDTZLOGC,ZDC='<ZDC Id>',COND=(EVEN)` <- измените ZDC Id (например, `MEPC` или `*` для zDC по умолчанию)

## Журналирование

Существует два набора журналов модуля IMS.

* Первый набор сообщений IMS поступает из задания инъекции IMS. Это сообщения, возникающие во время инъекции модуля в control region IMS. Эти сообщения появляются только в spool задания инъекции модуля IMS.
* Второй набор сообщений IMS поступает от модуля IMS по мере мониторинга активности IMS. Эти сообщения отправляются в zDC, а затем перенаправляются в zRemote.

Вы можете получить доступ к журналам модуля IMS через [журналы zRemote](/docs/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/install-zremote#logging "Подготовка и установка zRemote для мониторинга z/OS.").

## Обновление без перезапуска региона

Для обновления модуля IMS до более новой версии без перезапуска региона:

1. [Загрузите наборы данных продукта z/OS](/docs/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/zosmf-installer/download-zos-datasets#download-pax "Загрузка и установка наборов данных продукта Dynatrace для z/OS.") и [извлеките их](/docs/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/zosmf-installer/download-zos-datasets#extract-datasets "Загрузка и установка наборов данных продукта Dynatrace для z/OS.").
2. Обновите задание инъекции, чтобы оно указывало на новый `<hlq>.SZDTAUTH`. Если вы [определили псевдоним](/docs/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/zosmf-installer/download-zos-datasets#alias "Загрузка и установка наборов данных продукта Dynatrace для z/OS."), переопределите псевдоним. Например:

   ```
   DELETE 'DT.DYNTRC.SZDTAUTH' NOSCRATCH



   DEFINE ALIAS(NAME('DT.DYNTRC.SZDTAUTH') RELATE('DT.R12710.SZDTAUTH'))
   ```
3. Запустите задание инъекции без кода действия для загрузки обновлённого модуля IMS. Подробности см. в разделе [Заметки по инъекции](#injection-notes).
4. Для освобождения ECSA, используемой старой версией модуля IMS, см. раздел [Заметки по сборке мусора](#gc-notes).

## Часто задаваемые вопросы

Как отключить модуль IMS?

Если в модуле IMS происходит ABEND, процесс восстановления создаёт диагностику ABEND, если это возможно, а затем отключает модуль IMS. Система IMS продолжает работать. При этом серия WTO-сообщений записывается в системный журнал для control region IMS и/или зависимого региона IMS. Пример типичного набора сообщений:

```
ZDTI032W Recovery routine entered.



ZDTI036W ZDTIII15 0000000 20221103 10.51 VER 1.255.0 ABEND at offset 007874.



ZDTI033W Successful ABEND recovery, agent disabled.
```

Другие или дополнительные сообщения могут быть выданы, если процесс восстановления сталкивается с аномальными условиями (например, невозможность получить динамическое хранилище, повторная попытка не разрешена или SDWA не был передан). Все сообщения, связанные с процессом восстановления после ABEND, задокументированы в разделе [Сообщения модулей z/OS](/docs/ingest-from/dynatrace-oneagent/installation-and-operation/zos/operation/zos-code-module-messages "Сообщения, создаваемые модулями Dynatrace z/OS.").

Запись Software (SFT) Error Record с дополнительным описанием ABEND обычно записывается в набор данных z/OS SYS1.LOGREC. Необходимо запустить утилиту z/OS EREP для печати Software (SFT) Error Record, связанной с ABEND.

Опционально может быть создан SVC-дамп во время восстановления, в зависимости от параметра восстановления после ABEND, указанного или принятого по умолчанию при инъекции модуля IMS. Действие по умолчанию -- захват SVC-дампа при запуске восстановления после ABEND. Этот параметр можно указать при начальной инъекции модуля IMS или в сочетании с параметрами функций Modify или Enable для включения или выключения захвата дампа при восстановлении для ранее инъецированного модуля IMS.

Когда модуль IMS отключается в результате процесса восстановления после ABEND, он остаётся отключённым до явного повторного включения с помощью программы инъекции модуля IMS.

Как определить несовместимость версий?

Убедитесь, что версия модуля IMS меньше или равна версии модуля zRemote. Не подключайте более новые модули IMS к более старым модулям zRemote. Ниже приведён пример сообщения в журнале zRemote при несовместимости версии модуля IMS с версией zRemote.

```
severe  [native] IMS14CR1[asid = 108] is trying to initialize with an invalid protocol version number : x.xxx.xx
```
