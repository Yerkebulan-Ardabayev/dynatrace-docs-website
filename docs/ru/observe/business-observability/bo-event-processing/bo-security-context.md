* Latest Dynatrace

Dynatrace имеет [модель разрешений для Grail](../../../platform/grail/organize-data/assign-permissions-in-grail.md "Узнайте, как назначать разрешения для бакетов и таблиц в Grail."). Это относится ко всем данным телеметрии, таким как метрики, события, спаны и логи.

Мы рекомендуем настраивать разрешения в соответствии с организационной структурой и областями развёртывания. Подходящие концепции включают группы хостов, кластеры Kubernetes и пространства имён Kubernetes. Эти атрибуты обычно доступны для всех данных телеметрии, поступающих через методы сбора Dynatrace, такие как OneAgent, OpenTelemetry или Kubernetes operator. Таким образом, вы можете использовать эти атрибуты для включения [разрешений на уровне записей](../../../platform/grail/organize-data/assign-permissions-in-grail.md#grail-permissions-table-record "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.").

Для развёртываний на основе Kubernetes убедитесь, что Dynatrace Operator имеет включённое [обогащение метаданными](../../../ingest-from/setup-on-k8s/guides/metadata-automation/metadata-enrichment.md "Обогащение метаданными в Dynatrace Operator добавляет контекст к подам Kubernetes, присоединяя соответствующие метаданные к сущностям, таким как поды, хосты и процессы, для улучшения наблюдаемости.").

Если вам нужна только базовая концепция разрешений, лучшим вариантом будет настройка разрешений на уровне бакетов. Затем вы можете маршрутизировать данные в нужный бакет в OpenPipeline, сопоставляя одно из упомянутых первичных полей Grail, связанных с развёртыванием.

Для более детального контроля в Dynatrace вы можете настроить границы политик с более точными ограничениями на уровне данных. По умолчанию вы можете использовать следующие атрибуты:

* `dt.host_group.id`
* `k8s.cluster.name`
* `k8s.namespace.name`
* Любой другой атрибут, указанный в [модели разрешений](../../../platform/grail/organize-data/assign-permissions-in-grail.md#grail-permissions-table-record "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.")

Dynatrace предоставляет комплексную модель разрешений для Grail, которая применяется ко всем данным телеметрии, включая метрики, логи, спаны и события.

## Настройка контекста безопасности

Dynatrace позволяет настраивать поступающие данные бизнес-событий, добавляя атрибут `dt.security_context` к определённым записям бизнес-событий. Это позволяет устанавливать дополнительные параметры, например, разрешения для отдельных записей. См. [Разрешения в Grail](../../../platform/grail/organize-data/assign-permissions-in-grail.md "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.").

Вы можете устанавливать эти разрешения для отдельных записей на основе определённых значений атрибутов или автоматически обогащённых сущностей в бизнес-событии (хосты, группы процессов, экземпляры групп процессов), когда OneAgent захватывает данные.

Для базовых сценариев использования лучше всего, чтобы границы разрешений следовали за развёртыванием вашей организационной структуры вокруг хостов и групп процессов или других атрибутов, которые могут определять такие границы, например, географическое расположение данных.

Чтобы создать настройку контекста безопасности для поступающих данных бизнес-событий, необходимо создать правила, которые:

1. Фильтруют записи, к которым будет добавлен атрибут `dt.security_context`.
   Перейдите в ![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") **Notebooks** и создайте DQL-запрос, используя [функцию обработки бизнес-событий](bo-events-processing-matcher.md "Это DQL-сопоставитель для событий в классическом конвейере."). Например:

   ```
   fetch bizevents


   | filter matchesValue(geo.city.name, "Brussels")
   ```

   Это позволяет отфильтровать записи бизнес-событий, к которым вы хотите добавить атрибут `dt.security_context`. Когда результат запроса вас устроит, скопируйте функцию обработки бизнес-событий из DQL-запроса: `matchesValue(geo.city.name, "Brussels")`.
2. Определите правило контекста безопасности бизнес-событий, используя эту функцию, и укажите значение атрибута `dt.security_context`.
   Значение атрибута `dt.security_context` может быть литеральным значением, которое вы предоставляете, или вы можете указать имя другого атрибута, значение которого будет использоваться в качестве значения `dt.security_context`.

### Использование существующих тегов на источнике

Вы можете определить контекст безопасности на источнике через [OneAgent](../../../ingest-from/dynatrace-oneagent/oneagent-security-context.md "Узнайте, как настроить разрешения Grail для OneAgent."), [OpenTelemetry](../../../ingest-from/opentelemetry/opentelemetry-security-context.md "Настройте разрешения Grail для OpenTelemetry.") или [метки и аннотации Kubernetes](../../../ingest-from/setup-on-k8s/guides/metadata-automation/k8s-metadata-telemetry-enrichment.md "Руководства по обогащению телеметрии в Kubernetes"). Это позволяет использовать существующие метки и теги для управления разрешениями в Dynatrace.

## Создание правила контекста безопасности бизнес-событий через OpenPipeline

Рекомендуется (необязательно)

Контекст безопасности бизнес-событий основан на правилах, содержащих сопоставитель и определение атрибута `dt.security_context`, и задаётся как процессор в OpenPipeline. Сопоставитель сужает набор доступных записей бизнес-событий для выполнения конкретного правила, а тип источника значения определяет значение атрибута `dt.security_context`.

Подробнее читайте о [процессорах OpenPipeline](../../../platform/openpipeline/concepts/processing.md#processor "Изучите основные концепции обработки данных в Dynatrace OpenPipeline.").

1. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline** > **Business events**.
2. Перейдите на вкладку **Pipelines** и выберите существующий конвейер или создайте новый с существующим динамическим маршрутом, который будет применяться к входящему бизнес-событию.
3. Выберите вкладку **Permission** для просмотра процессоров безопасности обработки бизнес-событий, действующих в этом конвейере, изменения порядка существующих процессоров и создания новых.
   Процессоры выполняются в порядке, в котором они перечислены, сверху вниз. Этот порядок критически важен, так как выполняется первый пользовательский процессор, который соответствует условию. Процессор безопасности бизнес-событий состоит из следующих компонентов:

   * **Name**: Имя процессора.
   * **Matching condition**: Функция обработки бизнес-событий, которая сужает набор доступных данных бизнес-событий для выполнения конкретного процессора.
   * **Security context value**

     + **Field name**: Значение будет скопировано из поля.
     + **Static string**: В качестве значения будет использована постоянная строковая константа.
     + **Static array**: В качестве значения будет использован массив.
   * **Security context field value**: Значение контекста безопасности в зависимости от выбранного типа.
4. Чтобы добавить новый процессор контекста безопасности, перейдите на вкладку **Permission** и выберите **Processor** > **Set security context** в выбранном конвейере.
5. Добавьте условие сопоставления к вашему процессору, вставив функцию обработки бизнес-событий из вашего DQL-запроса.
6. Выберите **Security context value** и укажите **Security context field value**:

   * Если вы выбрали **Field**, значение должно быть именем поля, используемого в качестве источника для копирования значения в атрибут `dt.security_context`.
   * Если вы выбрали **Literal**, значение должно быть постоянной строковой константой, которая будет использоваться в качестве значения атрибута `dt.security_context`.

### Пример OpenPipeline

Ваши записи бизнес-событий содержат атрибут `geo.city.name` со значением `Brussels`. Только пользователи группы `team_EU` должны иметь к ним доступ, и вы хотите создать правило контекста безопасности бизнес-событий, которое:

* Фильтрует все записи, содержащие `Brussels` в качестве значения атрибута `geo.city.name`.
* Добавляет атрибут `dt.security_context` со значением `team_EU` ко всем отфильтрованным записям бизнес-событий.

Чтобы создать новое правило

1. Перейдите в ![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") **Notebooks** и выполните DQL-запрос, используя [функцию обработки бизнес-событий](bo-events-processing-matcher.md "Это DQL-сопоставитель для событий в классическом конвейере."):

   ```
   fetch bizevents


   | filter matchesValue(geo.city.name, "Brussels")
   ```
2. Скопируйте функцию обработки бизнес-событий из DQL-запроса: `matchesValue(geo.city.name, "Brussels")`.
3. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **OpenPipeline** и перейдите к конвейеру, в который вы хотите добавить процессор.
4. Выберите вкладку **Permission** и выберите **Processor** > **Set security context**.
5. Определите новый процессор:

   * **Name**: `team_EU - Brussels`
   * **Matching condition**: `matchesValue(geo.city.name, "Brussels")`
   * **Security context value**: выберите **Static string**
   * **Security context field value**: `team_EU`
6. Выберите **Save**.

## Создание правила контекста безопасности бизнес-событий через классический конвейер

Необязательно

Если вы ещё не перешли на Grail и OpenPipeline, следуйте этому разделу для обработки через классический конвейер.

Контекст безопасности бизнес-событий основан на правилах, содержащих сопоставитель и определение атрибута `dt.security_context`.

* Сопоставитель сужает набор доступных записей бизнес-событий для выполнения конкретного правила.
* Тип источника значения определяет значение атрибута `dt.security_context`.

Перейдите в **Settings** > **Business Observability** > **Security context** для просмотра правил безопасности обработки бизнес-событий, которые действуют, изменения порядка существующих правил и создания новых. Правила выполняются в порядке, в котором они перечислены, сверху вниз. Этот порядок критически важен, так как выполняется первое пользовательское правило, которое соответствует условию.

Разверните **Details** для просмотра определения правила. Правило безопасности обработки бизнес-событий состоит из следующих компонентов:

* **Rule name**: Имя правила.
* **Matcher**: Функция обработки бизнес-событий, которая сужает набор доступных данных бизнес-событий для выполнения конкретного правила.
* **Select value source type**

  + **Field**: Значение будет скопировано из поля.
  + **Literal**: В качестве значения будет использована константа.
* **Value**: Значение типа источника.

Чтобы добавить правило контекста безопасности бизнес-событий:

1. Выберите **Add rule** на странице **Business event security context**.
2. Добавьте **Matcher** к вашему правилу, вставив [функцию обработки бизнес-событий](bo-events-processing-matcher.md "Это DQL-сопоставитель для событий в классическом конвейере.") из вашего DQL-запроса.
3. Выберите тип источника значения.
4. Укажите **Value** для типа источника.

   * Если вы выбрали **Field**, значение должно быть именем поля, используемого в качестве источника для копирования значения в атрибут `dt.security_context`.
   * Если вы выбрали **Literal**, значение должно быть постоянной строковой константой, которая будет использоваться в качестве значения атрибута `dt.security_context`.

### Пример классической обработки

Ваши записи бизнес-событий содержат атрибут `geo.city.name` со значением `Brussels`. Только пользователи группы `team_EU` должны иметь к ним доступ, и вы хотите создать правило контекста безопасности бизнес-событий, которое:

* Фильтрует все записи, содержащие `Brussels` в качестве значения атрибута `geo.city.name`.
* Добавляет атрибут `dt.security_context` со значением `team_EU` ко всем отфильтрованным записям бизнес-событий.

1. Перейдите на страницу **Logs and events** и выполните DQL-запрос, используя [функцию обработки бизнес-событий](bo-events-processing-matcher.md "Это DQL-сопоставитель для событий в классическом конвейере."):

   ```
   fetch bizevents


   | filter matchesValue(geo.city.name, "Brussels")
   ```
2. Скопируйте функцию обработки бизнес-событий из DQL-запроса: `matchesValue(geo.city.name, "Brussels")`.
3. Перейдите в **Settings** > **Business Analytics** > **Security context** и выберите **Add rule** на странице **Business event security context**.
4. Определите новое правило:

   * **Rule name**: `team_EU - Brussels`
   * **Matcher**: `matchesValue(geo.city.name, "Brussels")`
   * **Select value source type**: выберите **Literal**
   * **Value**: `team_EU`
5. Выберите **Save changes**.

## Проверка контекста безопасности

Необязательно

После поступления новых данных бизнес-событий они обрабатываются правилами контекста безопасности бизнес-событий или процессором (в зависимости от доступности OpenPipeline). Добавляется новый атрибут `dt.security_context` со значением `team_EU`. Вы можете перейти в ![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") **Notebooks** и проверить, что новые записи бизнес-событий обработаны вашим правилом контекста безопасности `team_EU - Brussels`. Используйте DQL для просмотра всех записей бизнес-событий, содержащих значение `team_EU` в атрибуте `dt.security_context`:

```
fetch bizevents


| filter matchesValue(dt.security_context, "team_EU")
```

На основе этого атрибута вы теперь можете создавать политики пользователей и групп, связанные с безопасностью. См. [Разрешения в Grail](../../../platform/grail/organize-data/assign-permissions-in-grail.md "Узнайте, как назначать разрешения для бакетов и таблиц в Grail.").
