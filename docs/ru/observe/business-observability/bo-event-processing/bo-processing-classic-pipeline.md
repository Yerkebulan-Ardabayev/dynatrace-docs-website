# Обработка бизнес-событий через классический конвейер

* Latest Dynatrace
* Tutorial
* 5-min read
* Published Oct 24, 2022

Обработка бизнес-событий через классический конвейер позволяет определить, как ваши данные будут обрабатываться далее.

* Вы можете создать правило обработки, которое трансформирует входящие события.
* Вы также можете указать время хранения для ваших событий, назначив их в бакеты.

Если вы создаёте несколько правил, они выполняются в том порядке, в котором вы их определили.

Мы рекомендуем использовать обработку бизнес-событий с помощью [OpenPipeline](../../../platform/openpipeline.md "Масштабирование обработки данных платформы Dynatrace с помощью Dynatrace OpenPipeline.") как масштабируемое и мощное решение для управления и обработки бизнес-событий. Если у вас нет доступа к OpenPipeline, используйте классический конвейер обработки бизнес-событий.

## Настройка правил обработки бизнес-событий

Чтобы настроить правила обработки бизнес-событий:

1. Перейдите в **Settings** > **Business Observability** > **Ingest Pipeline** > **Processing**.
2. Выберите **Add rule** и назовите ваше правило.
3. Добавьте **Matcher** к вашему правилу, вставив ваш [DQL-запрос для матчера](bo-events-processing-matcher.md "Это матчер DQL для событий в классическом конвейере.").

   Сопоставление на основе предыдущих правил не поддерживается

   Матчер работает с исходным набором данных до применения каких-либо правил обработки. Сопоставление записей, изменённых предыдущими правилами, не поддерживается. Например, изменённое значение атрибута в правиле 1 не может использоваться в качестве значения для сопоставления в правиле 2.
4. Выберите **Add item**, чтобы указать поля, которые будут трансформированы через определение процессора. Определите типы данных и имена для ваших полей. Вы можете указать, является ли правило **Optional**, **Is Array** или **Read-only**.

   * **Optional**: Если поле трансформации отмечено как необязательное, трансформация может произойти, даже если поле не предоставлено. Но если поле трансформации отмечено как обязательное (не optional), трансформация происходит только при наличии поля во входных данных.
   * **Is Array**: Необходимо выбрать, когда поле содержит несколько значений и вы намерены работать с конкретными значениями.
   * **Read-Only**: Не может быть изменено во время трансформации.
5. Добавьте **Processor definition** (определение процессора).
   Определения процессоров — это инструкции, которые указывают Dynatrace, как трансформировать данные, выбранные в DQL-запросе матчера.
   Они создаются с использованием [команд обработки](../../../analyze-explore-automate/logs/lma-classic-log-processing/lma-log-processing-commands.md "Изучите сценарии использования команд обработки логов в Dynatrace на базе Grail."), [функций](../../../analyze-explore-automate/logs/lma-classic-log-processing/lma-log-processing-functions.md "Изучите сценарии использования функций обработки логов в Dynatrace на базе Grail.") и [сопоставления шаблонов](../../../platform/grail/dynatrace-pattern-language.md "Используйте Dynatrace Pattern Language для описания шаблонов с помощью матчеров.") (Dynatrace Pattern Language), которые позволяют добавлять, изменять или удалять входящие записи.
6. В разделе **Rule testing** вставьте пример события и запустите тест.

   Пример входных данных должен быть предоставлен в формате JSON.
7. Выберите **Save changes**.

## Примеры обработки бизнес-событий

В приведённых ниже примерах вы можете увидеть, как обработка через конвейер приёма может использоваться для удовлетворения ваших технических и административных требований.

* Пример 1. Добавление нового вычисляемого поля в конвейер.
  Как пользователю, вам нужно добавить вычисляемое значение объёма торгов в долларах в конвейер.

  ```
  {



  "action":"buy",



  "accountId":6,



  "amount":10,



  "instrumentId":1,



  "price":157.025



  }
  ```

  + **Имя правила**: Add field
  + **Матчер**: `matchesValue(action, "buy")`
  + **Поля трансформации**:

  + **Определение процессора**: `FIELDS_ADD(trading_volume: price*amount)`
* Пример 2. Маскировка ваших данных.
  Вам нужно скрыть поле CVV кредитной карты во входящем JSON-пейлоаде.

  ```
  {



  "action": "payment",



  "creditCardNumber":"5570001112223344",



  "valid":"12/27",



  "cvv":"001"



  }
  ```

  + **Имя правила**: Mask field
  + **Матчер**: `matchesValue(action, "payment")`
  + **Поля трансформации**:

  + **Определение процессора**: `FIELDS_ADD(cvv: SHA1(cvv))`
* Пример 3. Удаление атрибута события.
  Вам нужно удалить поле даты рождения во входящем JSON-пейлоаде.

  ```
  {



  "action": "newUser",



  "firstName":"Frank",



  "lastName": "Underwud",



  "birthDate": "10.01.1967"



  }
  ```

  + **Имя правила**: Drop field
  + **Матчер**: `matchesValue(action, "newUser")`
  + **Поля трансформации**:

  + **Определение процессора**: `FIELDS_REMOVE(birthDate)`
* Пример 4. Парсинг вложенного JSON.
  Вам нужно извлечь атрибуты из вложенного JSON, чтобы они были атрибутами верхнего уровня в Grail.

  ```
  {



  "action":"sell",



  "details":{



  "accountId":6,



  "amount":10,



  "instrumentId":1,



  "price":157.025



  }



  }
  ```

  + **Имя правила**: Parse field
  + **Матчер**: `matchesValue(action, "sell")`
  + **Поля трансформации**:

  + **Определение процессора**: `PARSE(details,"JSON{INTEGER:accountId,INTEGER:amount,INTEGER:instrumentId,DOUBLE:price}(flat=true)") | FIELDS_REMOVE(details)`
* Пример 5. Парсинг сообщения об ошибке.

  В этом примере вам нужно удалить код ваучера из поля сообщения об ошибке ниже, чтобы иметь возможность подсчитать, как часто появляется одно и то же сообщение об ошибке. Решение ниже использует [Dynatrace Pattern Language](../../../platform/grail/dynatrace-pattern-language.md "Используйте Dynatrace Pattern Language для описания шаблонов с помощью матчеров.") для парсинга.

  ```
  {



  "timestamp":"2023-01-18T10:50:23.777000000Z",



  "cartId":"58583939",



  "error.message":"The voucher [XY-892940] is not valid!",



  "error.messageKey":"error.voucher "



  }
  ```

  + **Имя правила**: Parse error message
  + **Матчер**: `matchesValue(error.messageKey, "error.voucher ")`
  + **Поля трансформации**:

  + **Определение процессора**:
    `FIELDS_ADD(final:REPLACE_PATTERN(error.message, "LD:p1 '[' LD:to_be_masked ']' LD:p2 ", "${p1}${p2}"))`
  + **Результат**: `The voucher is not valid!`.
