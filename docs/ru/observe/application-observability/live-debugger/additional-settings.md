# Настройка дополнительных параметров Live Debugging

* Последняя версия Dynatrace
* Практическое руководство
* Чтение займёт 3 минуты
* Обновлено 19 февраля 2026 г.

## Управление включением/отключением (Opt-in/Opt-out)

Предоставляет разрешение на управление включением/отключением мониторинга кода для всей среды или по области действия.

**Пример политики**:

```
ALLOW settings:objects:read, settings:objects:write, settings:schemas:read WHERE settings:schemaId = "builtin:devobs.agent.optin";
```

## Управление точками останова (breakpoints)

Предоставляет разрешение на управление точками останова Live Debugging.

**Пример политики**:

Разрешение на управление точками останова в рамках всей среды:

```
ALLOW dev-obs:breakpoints:manage;
```

## Настройка OneAgent

### Node

#### Транспилированный код

Если код вашего приложения транспилируется или компонуется, необходимо включить карты источников (source maps) — либо встроенные, либо в виде отдельных файлов.

* Webpack
  Используйте значение `inline-source-map` или `source-map` для параметра `devtool` в конфигурационном файле Webpack.
* Babel
  Используйте значение `inline`, `both` или `true` для параметра `sourceMaps` в конфигурационном файле Babel.
* TypeScript
  Установите поля `inlineSourceMap` или `sourceMap` в значение `true`, а также `inlineSources` в значение `true` в конфигурационном файле TypeScript.
* CoffeeScript
  Передайте флаг `-M` (встроенный) или `-m` инструменту командной строки coffee.
* Webpack + TypeScript
  Добавьте `sourceMap` и `inlineSources` со значением `true` (установки `inlineSourceMap` недостаточно в данном случае) в конфигурационный файл TypeScript и используйте значение `inline-source-map` или `source-map` для параметра `devtool` в конфигурационном файле Webpack.

#### Обфускация/минификация

![Live Debugger](https://dt-cdn.net/images/live-debugger-256-b934a2bad5.png "Live Debugger") **Live Debugger** работает с обфусцированным/минифицированным кодом при наличии карт источников. Однако манглинг (изменение имён переменных) не должен применяться. Webpack может автоматически менять имена переменных при использовании в режиме production.

```
const TerserPlugin = require("terser-webpack-plugin");



module.exports = {



// ...



optimization: {



minimizer: \[new TerserPlugin({



terserOptions: {



mangle: false,



},



})],



},



};
```

## Интеграция с системой контроля версий

При отладке в удалённой среде необходимо точно знать, какой исходный код выполняется.

![Live Debugger](https://dt-cdn.net/images/live-debugger-256-b934a2bad5.png "Live Debugger") **Live Debugger** интегрируется с вашим провайдером системы контроля версий для получения корректного исходного кода для каждой среды.

Настройте следующие переменные среды для включения автоматического получения исходного кода:

* `DT_LIVEDEBUGGER_COMMIT` — строка, указывающая ваш git-коммит
* `DT_LIVEDEBUGGER_REMOTE_ORIGIN` — строка, указывающая ваш удалённый источник git

Для нескольких источников используйте переменную среды `DT_LIVEDEBUGGER_SOURCES` для инициализации SDK с информацией об источниках, используемых в вашем приложении.

`DT_LIVEDEBUGGER_SOURCES` — это разделённый точками с запятой список информации о репозитории системы контроля версий и версии, объединённых символом `#`. Например: `DT_LIVEDEBUGGER_SOURCES=https://github.com/myorg/MyRepo#abc123;https://github.com/otherorg/OtherRepo#xyz789`.

В языках JVM можно также указать путь к JAR-файлу, который должен содержать следующие атрибуты в манифесте JAR:

* `DT-LiveDebugger-Repository`
* `DT-LiveDebugger-Revision`

Тогда можно использовать комбинацию URL-адресов и путей к JAR-файлам.

Пример: `DT_LIVEDEBUGGER_SOURCES=https://github.com/myorg/MyRepo#abc123;/path/to/lib.jar`.

## Применение правил маскирования данных

![Live Debugger](https://dt-cdn.net/images/live-debugger-256-b934a2bad5.png "Live Debugger") **Live Debugger** предлагает правила маскирования данных, позволяющие контролировать конфиденциальные данные и указывать, какие данные не следует собирать и отображать пользователям.

Правила маскирования данных можно настроить на странице настроек Live Debugger.

1. Перейдите в **Settings** > **Observability for Developers** > **Sensitive data masking**.
2. Настройте маскирование данных, добавив правила с набором условий, идентифицирующих конфиденциальные данные.
3. Выберите **Add rule**, чтобы начать настройку правила.
4. Укажите **Name** — имя, отображаемое для вашей конфигурации.
5. Выберите тип правила:

   * **Redact by variable name**
     При маскировании по имени переменной всё значение переменной, соответствующей этому имени, будет скрыто. Можно указать, как должно совпадать имя переменной:

     **Equals**: Сопоставляет переменные с именами, точно совпадающими с введённой строкой (например, `foo` соответствует только `foo`).
     **Contains**: Сопоставляет переменные с именами, содержащими введённую строку (например, `foo` соответствует `myfoobar` или `foo123`).
     **Starts with**: Сопоставляет переменные с именами, начинающимися с введённой строки (например, `foo` соответствует `foobar` или `foo123`).
     **Ends with**: Сопоставляет переменные с именами, заканчивающимися введённой строкой (например, `foo` соответствует `myfoo` или `barfoo`).
   * **Redact by regex**
     При маскировании по регулярному выражению все значения переменных, соответствующие этому выражению, будут скрыты.
     Например, добавление правила для значения переменной `\[0-9]+` заменит `nameAndPassword:LordHelmet-12345` на `nameAndPassword:LordHelmet-\*\*\*\*`.

Заданные правила можно переупорядочивать; они выполняются в том порядке, в котором указаны на странице **Sensitive data masking**.

Скрытые данные по умолчанию заменяются строкой `\*\*\*`.

Можно выбрать произвольную строку замены для правила или заменить данные хешем SHA-256. Например, добавление правила для имени переменной `secretKey` с пользовательской заменой `REDACTED` заменит вывод `secretKey:12345` на `secretKey:REDACTED`.
