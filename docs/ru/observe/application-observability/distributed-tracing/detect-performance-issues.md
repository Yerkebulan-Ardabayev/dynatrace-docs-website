Понимание того, какие запросы выполняются медленно и почему это происходит, может быть сложной задачей в современных облачных распределенных системах. Приложение [Distributed Tracing ![Distributed Tracing](https://dt-cdn.net/images/distributed-tracing-4ed13d1274.svg "Distributed Tracing")](https://dt-url.net/vc23u9n) предназначено для быстрого получения ответов на эти вопросы с помощью исследовательского подхода.

Протестируйте приложение в изолированной среде:

* [Playground-среда](https://dt-url.net/al43u2h)

## Целевая аудитория

Эта статья предназначена для SRE-инженеров, разработчиков и архитекторов производительности, которые хотят исследовать проблемы производительности путем анализа распределенных трассировок в приложении Distributed Tracing.

## Что вы узнаете

В этом руководстве вы узнаете, как использовать приложение Distributed Tracing для анализа и устранения проблем с медленными запросами в производственном пространстве имен Kubernetes. Вы научитесь:

* Фильтровать трассировки по релевантности
  Использовать фасеты для сужения области трассировок, фокусируясь на тех, которые соответствуют вашим обязанностям или интересам.
* Увеличивать масштаб для определенных временных интервалов
  Определять релевантные периоды времени с помощью графика временных рядов или селектора временного интервала для просмотра производительности за выбранный период.
* Выявлять аномальные отклонения производительности
  Использовать гистограммы для обнаружения трассировок с увеличенным временем ответа.
* Анализировать затронутые сервисы и запросы
  Использовать функцию группировки для понимания того, какие сервисы и запросы затронуты.
* Детально изучать отдельные трассировки
  Исследовать подробные данные трассировок для конкретных запросов, чтобы выявить коренные причины и определить, что произошло во время событий с медленным откликом.

## Предварительные требования

* Вы знакомы с [концепциями распределенной трассировки](../distributed-traces/concepts.md "Узнайте больше об основных концепциях и терминологии распределенной трассировки.").
* У вас есть подписка Dynatrace Platform Subscription (DPS).
* Вы загружаете данные трассировок через [Dynatrace OneAgent](../../../ingest-from/dynatrace-oneagent.md "Узнайте о важных концепциях, связанных с OneAgent, и о том, как установить и использовать OneAgent на различных платформах.") или [OpenTelemetry](../../../ingest-from/opentelemetry/getting-started.md "Как отправить данные OpenTelemetry в Dynatrace."). Чтобы начать загрузку данных трассировок, перейдите в **Distributed Tracing** > **Traces** > выберите источник и следуйте инструкциям в продукте.
* Убедитесь, что ваши трассировки доступны в приложении [Distributed Tracing ![Distributed Tracing](https://dt-cdn.net/images/distributed-tracing-4ed13d1274.svg "Distributed Tracing")](distributed-tracing-app.md "Откройте для себя функциональные возможности нового приложения Distributed Tracing.").

## Анализ и устранение проблем с производительностью

1. Перейдите в Distributed Tracing ![Distributed Tracing](https://dt-cdn.net/images/distributed-tracing-4ed13d1274.svg "Distributed Tracing").
2. В правом верхнем углу выберите временной интервал **Last 30 minutes**, чтобы увидеть, как изменялись количество запросов и время их ответа за этот период.
3. Используйте фасеты для фильтрации трассировок по определенному пространству имен. Чтобы проанализировать трассировки, проходящие через пространство имен Kubernetes **prod**,

   1. В списке фасетов перейдите в **Kubernetes** > **Kubernetes namespace**.

      Пространства имен, которые чаще всего обнаруживаются в ваших трассировках, отображаются первыми.
   2. Выберите **prod**.

      Запрос в поле фильтра автоматически изменится. Вы можете дополнительно отредактировать запрос фильтра перед обновлением результатов.
   3. Выберите **Update**, чтобы обновить результаты.
4. Изучите графики, чтобы быстро обнаружить аномальные отклонения производительности.

   1. Перейдите к графику **Timeseries**, чтобы получить обзор тенденций.

      ![Detect performance issues timeseries](https://dt-cdn.net/images/traces-example-00001-timeseries-3430-6a591decfd.png)

      Пространство имен Kubernetes **prod** показывает несколько всплесков количества запросов и кластер неудачных запросов.
   2. Для дальнейшего анализа данных выберите **Histogram**.

      ![Detect performance issues histogram](https://dt-cdn.net/images/traces-example-00002-3440-40b39e1573.png)

      Гистограмма обобщает количество запросов для определенного диапазона времени ответа. Существует кластер запросов к среде **prod**, которые занимают около 5 секунд.
   3. Выберите область графика, содержащую затронутый кластер, чтобы сфокусироваться на запросах, попадающих в этот диапазон времени ответа.

      Полоса фильтра обновляется для отображения трассировок для выбранного диапазона времени ответа, а список показывает трассировки с похожими сервисами и конечными точками.
5. Сгруппируйте запросы по сервису и конечной точке, чтобы понять, какие сервисы и конечные точки затронуты.

   1. Перейдите к таблице и выберите **Group by:**.
   2. Выберите атрибуты **Service** и **Endpoint**.

      В табличном представлении (:TableIcon:) вы можете увидеть агрегированный список сервисов и конечных точек, попадающих в выбранный диапазон времени ответа для пространства имен **prod**. Столбцы содержат среднюю и максимальную продолжительность, количество запросов и количество неудачных запросов, включая процент ошибок.
   3. Чтобы узнать больше о главных составляющих агрегата в таблице, выберите .

      Обратив внимание на запрос **/cart/checkout**, вы увидите, что он не только имеет медленное время ответа (**5.82 с**), но и высокий процент ошибок (**28%**), что указывает на возможное узкое место.

      ![distributed tracing cart/checkout request](https://dt-cdn.net/images/wkf10640-apps-dynatrace-com-ui-apps-dynatrace-distributedtracing-explorer-3840-e0a340e0bf.png)
   4. Чтобы сфокусироваться на запросе, выберите имя запроса (**/cart/checkout**) в таблице и отфильтруйте по конечной точке.

      Запрос в поле фильтра и результаты таблицы автоматически обновляются.
6. Чтобы определить узкое место для запроса, соответствующего выбранным фильтрам, перейдите к представлению списка и выберите временную метку одного из запросов.

Перспектива отдельной трассировки открывается в нижней половине страницы. Теперь вы можете проанализировать полную сквозную трассировку. В нашем случае мы видим, что узким местом является конечная точка **GetCart** в сервисе **Cart**, что приводит к медленному времени ответа.

## Заключение

В этом руководстве вы научились эффективно использовать приложение Distributed Tracing ![Distributed Tracing](https://dt-cdn.net/images/distributed-tracing-4ed13d1274.svg "Distributed Tracing") для анализа и устранения проблем с медленными запросами в производственном пространстве имен Kubernetes. Теперь вы умеете фильтровать трассировки по релевантности, увеличивать масштаб для определенных временных интервалов для выявления аномалий, а также анализировать затронутые сервисы и запросы. Эти навыки позволяют вам эффективно диагностировать и решать проблемы с производительностью в вашей среде Kubernetes.

Мы проанализировали данные трассировок и обнаружили кластер медленных и неудачных запросов в производственном пространстве имен Kubernetes. Путем фильтрации, группировки и изучения данных трассировок мы смогли определить узкое место в конечной точке **GetCart** сервиса **Cart**.
