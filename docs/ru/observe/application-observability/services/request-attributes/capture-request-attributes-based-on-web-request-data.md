# Захват атрибутов запросов на основе данных веб-запросов

# Захват атрибутов запросов на основе данных веб-запросов

* Latest Dynatrace
* 4-min read
* Updated on Jul 06, 2023

Чтобы определить атрибут запроса на основе данных веб-запроса:

1. Перейдите в **Settings** > **Server-side service monitoring** > **Request attributes**.
2. Нажмите кнопку **Create new request attribute**.
3. Укажите уникальное **Request attribute name**. Вы можете переименовать атрибут в любой момент в будущем.
4. Укажите **Data type**.
   Вы не сможете изменить **Data type** после настройки атрибута запроса.
5. Если в одном запросе существует несколько значений, укажите, что должно сохраняться в атрибуте запроса для каждого запроса, и выберите способ нормализации текста.
6. Проверьте, должно ли это правило иметь доступ к немаскированным данным и содержит ли атрибут запроса [конфиденциальные данные](../../../../manage/data-privacy-and-security/configuration/configure-global-privacy-settings.md#restrict-view-access "Узнайте, как настроить маскирование данных для конфиденциальности IP-адресов конечных пользователей, геолокаций и имён действий пользователей.").

   Это потенциально обеспечит доступ к персональным данным.
7. Добавьте источник данных. Вы можете определить одно или несколько правил для указания того, как должен извлекаться ваш атрибут.
   Каждому правилу нужен источник. Dynatrace должен знать, откуда он может собирать атрибут запроса. Вы можете определить правило, сначала выбрав, где правило должно применяться (на основе группы процессов, группы хостов, технологии сервиса или группового тега), а затем указав, где находится фактический параметр (**Request attribute source**).
8. После добавления всех источников данных нажмите **Save**, чтобы сохранить правило атрибута запроса.

## Правила атрибутов запросов

Ознакомьтесь с примером правила атрибута запроса ниже. Обратите внимание, что атрибут запроса `destination` может получать своё значение из двух различных источников: параметра `HTTP Post` (`iceform:destination`) или параметра `HTTP GET` (`destination`). Правила выполняются по порядку. Если запрос соответствует критериям обоих правил, его значение будет взято из первого правила.

Каждому правилу нужен источник. В примере ниже источником атрибута запроса является параметр `HTTP GET` веб-запроса (`destination`).

![Пример определения правила атрибутов запроса](https://dt-cdn.net/images/get-2018-10-30-14-26-29-1373-3d3ff74f1c.png)

Этот параметр `GET` будет захватываться на всех мониторируемых процессах, поддерживающих анализ на уровне кода, и будет сообщаться для всех запросов, мониторируемых Dynatrace.

Хотя это удобно, это не всегда то, что нужно. Поэтому вы можете ограничить правила подмножеством групп процессов и сервисов. Для этого выберите имена групп процессов и сервисов из четырёх выпадающих списков выше, чтобы сократить количество групп процессов и сервисов, к которым применяется правило.

Вас может не интересовать захват каждого значения. В других случаях значение может содержать префикс, который вы хотите проверить. Для этого укажите, что назначенный параметр должен использоваться только в том случае, если его значение соответствует определённому значению. Вы также можете использовать не всё значение целиком, а извлечь его часть. Пример ниже настроен так, чтобы учитывать только параметры `HTTP POST` `iceform:destination`, начинающиеся со строки `Journey :`. Этот подход извлечёт всё, что следует за строкой `Journey:`, и сохранит это в атрибуте запроса.

![Пример определения правила POST для атрибутов запроса](https://dt-cdn.net/images/post-2018-10-30-14-15-35-1379-6f82cbbf7c.png)

Запросы могут иметь столько атрибутов, сколько вам нужно.

## Источники атрибутов запросов для веб-запросов

Источники данных атрибутов запросов для веб-запросов включают:

* Технологически-независимые источники, такие как:

  + Параметры HTTP POST
  + [IP-адреса клиентов](../../../digital-experience/rum-concepts/detection-of-ip-addresses-locations-and-user-agents.md#ip-addresses "Dynatrace обнаруживает IP-адреса и геолокации, такие как город, регион и страна, а также браузеры, устройства и операционные системы.")
    Сообщается значение первого совпавшего заголовка.
  + Заголовки HTTP-запросов и ответов
  + URL веб-запроса или один из его компонентов, например путь или параметр запроса

  В последних двух случаях вы также можете выбрать сторону веб-запроса, на которой нужно захватывать и сохранять атрибут.

  OneAgent всегда требуется при захвате.

  1

  Полный вызов сервиса: запрос, полностью мониторируемый Dynatrace.

  2

  Вызов внешнего сервиса: запрос, направленный к внешним ресурсам.
* Технологически-специфичные источники.

  + Приложение может хранить сложный объект в атрибуте, в то время как OneAgent преобразует его в строку при захвате (например, в [атрибуте сессии Java-сервлета](https://dt-url.net/q503soq) и [переменной состояния сессии ASP.NET](https://dt-url.net/qf23stx)). Это может иметь побочные эффекты, поэтому будьте осторожны с тем, что вы захватываете.
  + ASP.NET: когда несколько параметров имеют одинаковое имя, вы увидите только первое захваченное значение.
  + Java-сервлет: атрибуты запроса и атрибуты сессии Java-сервлета захватываются только после входа в сервлет или фильтр.

## Постобработка

В большинстве случаев захваченное значение будет содержать именно то, что вам нужно. Однако вам может не понадобиться всё значение целиком или даже каждое значение. С помощью постобработки вы можете манипулировать захваченным значением.

Разверните параметр **Optionally restrict or process the captured parameter(s) further**, чтобы увидеть шаги обработки. Шаги выполняются в указанном порядке -- каждый шаг применяется к результату предыдущего шага.

Вам не нужно применять все шаги. Каждый шаг становится активным, как только вы указываете для него значение или выбираете соответствующий флажок.

![Параметры постобработки для определения правила атрибута запроса](https://dt-cdn.net/images/post5steps-1353-6e753964a1.png)

Шаг 1 позволяет извлечь что-либо из результирующей строки на основе символов-разделителей.

Шаг 2 может разделить захваченное значение на несколько значений на основе символа-разделителя.

Шаг 3 удаляет пробелы.

Шаг 4 позволяет отфильтровать захваченные значения, которые не соответствуют указанному критерию.

Шаг 5 позволяет извлечь что-либо из результирующей строки на основе [регулярного выражения](../../../../manage/tags-and-metadata/reference/regular-expressions-in-dynatrace.md "Узнайте, как использовать регулярные выражения в контексте Dynatrace.").
