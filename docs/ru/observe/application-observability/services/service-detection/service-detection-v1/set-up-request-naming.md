Вы можете использовать правила именования запросов для корректировки способа отслеживания ваших запросов и определения точек входа и конечных точек сервисов в вашем клиентском рабочем процессе. С такой сквозной трассировкой вы можете легко распознавать и мониторить важные бизнес-транзакции, которые критически важны для успеха вашего цифрового бизнеса.

Используя атрибуты запросов в сочетании с правилами именования, вы можете захватить ещё больше контекста вокруг ваших запросов и использовать эти дополнительные данные для нарезки и анализа ваших данных мониторинга.

Поскольку правила именования запросов создают отдельные запросы сервисов, каждый запрос [независимо базируется](../../../../../dynatrace-intelligence/anomaly-detection/automated-multidimensional-baselining.md "Узнайте, как ИИ Dynatrace автоматически вычисляет базовые линии на основе многомерной схемы.") и мониторится на предмет аномалий производительности. Метрики производительности, захваченные для этих запросов, также доступны отдельно через конечную точку [Timeseries API v1](../../../../../dynatrace-api/environment-api/metric-v1.md "Получение информации о метриках через Timeseries v1 API.").

## Перед началом

* Правила именования запросов поддерживаются для следующих типов сервисов.

  + Веб-запрос (кроме **Requests to unmonitored hosts** и **Requests to public networks**)
  + Web
  + Запрос метода
  + Обмен сообщениями и очереди (кроме [сервисов-слушателей](../../../../infrastructure-observability/queues/queue-concepts.md#listener-service "Основные концепции мониторинга очередей сообщений в Dynatrace."))
  + RMI
  + CICS
  + IMS
  + Корпоративная сервисная шина
  + Удалённый вызов
* Обнаружение ключевых запросов основано на именах. Если правило именования запроса влияет на ключевой запрос и вы хотите, чтобы Dynatrace продолжал его обнаруживать как ключевой запрос, вам необходимо добавить новое имя в [список имён ключевых запросов](../../../services-classic/monitor-key-requests.md#rename "Узнайте, как вести детальный мониторинг запросов, критически важных для вашего бизнеса.").

## Создание правила именования запроса для сервиса

Первый шаг в настройке понятного именования для запросов вашего сервиса (веб) — создание правил именования запросов с условиями, определяющими их отображение в вашей среде.

Чтобы создать правило именования запроса для вашего сервиса

1. Перейдите на страницу настроек именования (веб) запросов:

   Latest Dynatrace, вариант 1

   Latest Dynatrace, вариант 2

   Dynatrace Classic

   1. Перейдите в ![Services](https://dt-cdn.net/hub/logos/services.png "Services") **Services** и выберите сервис, который вы хотите настроить.
   2. В правом верхнем углу панели обзора сервиса выберите (**Actions menu**) > **Settings** и перейдите к **Request naming** или **Web request naming**.

   1. Перейдите в ![Services](https://dt-cdn.net/hub/logos/services.png "Services") **Services** и выберите сервис, который вы хотите настроить.
   2. На вкладке **Overview** прокрутите вниз до раздела **Endpoints**.
   3. В заголовке раздела **Endpoints** выберите (**Actions menu**) > **Configure request naming**.

   1. Перейдите в ![Services Classic](https://dt-cdn.net/images/services-classic-f58502bd22.svg "Services Classic") **Services Classic** и выберите сервис, который вы хотите настроить.
   2. Выберите **More** (**...**) > **Settings** и перейдите к **Request naming** или **Web request naming**.
2. Выберите **Add rule**.
3. Определите набор условий, которые представляют критерии ваших операций сервиса.

   Вы можете использовать всё, от заголовков запроса до значений аргументов методов на уровне кода, для идентификации конкретных запросов. Все запросы, соответствующие указанным критериям, будут именоваться на основе определённого шаблона именования.

   ![Правило именования запроса](https://dt-cdn.net/images/naming-rule-2010-1385de1a99.png)

   В примере выше определены три условия.

   * Путь URL должен содержать строку `orange-booking-payment`.
   * Запрос должен быть **HTTP-методом** типа **GET**.
   * Запрос должен иметь атрибут `easyTravel destination`.

   Все запросы, соответствующие **всем** указанным критериям, будут именоваться на основе определённого шаблона именования `PAYMENT`.

   ![Добавление правила именования запроса — пример](https://dt-cdn.net/images/example-naming-rule-2006-1ef0fc7a46.png)
4. Выберите **Save**.

## Настройка обнаружения ресурсных запросов

Dynatrace автоматически обнаруживает ресурсные запросы на основе расширения файла и отслеживает эти запросы в следующих группах:

* Изображения
* CSS
* JavaScript
* Бинарные файлы

Когда функция **Enhanced endpoints for SDv1** включена, все запросы статических ресурсов группируются в единую конечную точку **Static resources**. См. [Запросы статических ресурсов](enhanced-endpoints-sdv1.md#resource-requests "Используйте функцию Enhanced endpoints for SDv1 для получения более глубокого понимания производительности вашего приложения и улучшения возможностей мониторинга и устранения неполадок взаимодействий сервисов.") для получения подробной информации.

Если Dynatrace не обнаруживает автоматически одно из изображений или бинарных файлов вашего приложения, вы можете добавить недостающие расширения файлов. Вы можете настроить эти параметры для всей среды или для конкретного сервиса.

Среда

1. Перейдите в ![Services](https://dt-cdn.net/hub/logos/services.png "Services") **Services**.
2. Перейдите на вкладку **Explorer** и выберите любой сервис SDv1.
3. В правом верхнем углу панели деталей сервиса выберите (**Actions menu**) > **Settings**.
4. В разделе **Naming** выберите **Web request naming**.
5. Прокрутите вниз до раздела **Resource requests** и перейдите по ссылке **settings** в **Use global resource request detection settings**.
6. В разделе **Web resource requests** введите необходимые расширения файлов для ресурсов изображений и бинарных файлов.

Также вы можете использовать наш API для изменения [глобальных правил именования запросов](#global-rules).

Сервис

1. Перейдите в ![Services](https://dt-cdn.net/hub/logos/services.png "Services") **Services**.
2. Перейдите на вкладку **Explorer**, найдите сервис, для которого вы хотите обновить настройки, и выберите его.
3. В правом верхнем углу панели деталей сервиса выберите (**Actions menu**) > **Settings**.
4. В разделе **Naming** выберите **Web request naming**.
5. Прокрутите вниз до раздела **Resource requests** и отключите **Use global resource request detection settings**.
6. Введите необходимые расширения файлов для ресурсов изображений и бинарных файлов.

## Дополнительные элементы для именования запросов

### Атрибуты запросов и заполнители

Вы можете включить значение атрибутов запросов и заполнителей в шаблоны именования.

* Используйте **атрибут запроса** для создания интуитивных вариантов имён для вашего запроса. Запрос создаёт отдельный отслеживаемый запрос для каждой перестановки соответствующего атрибута запроса.

Пример: Атрибуты запросов в шаблонах именования

Используя атрибут запроса `easyTravel destination` в шаблоне именования, для каждого направления, забронированного клиентами `easyTravel`, создаётся вариант запроса. В результате это правило больше не создаёт один тип запроса, а создаёт отдельный отслеживаемый запрос для каждого забронированного направления.

![Использование атрибутов запросов для настройки правил именования](https://dt-cdn.net/images/request-attributes-2002-61662e499a.png)

Глядя на распределённые трассировки, URL-адреса одинаковы, но каждое имя запроса теперь включает разное значение атрибута города назначения.

![Шаблон именования с атрибутами запросов](https://dt-cdn.net/images/request-attributes-example-2004-cfa8f5d39c.png)

* **Заполнители** могут извлекать значения из атрибутов запросов или URL-адресов.

Пример: Пользовательские заполнители

Если путь URL содержит строку `orange-booking`, бронирование, определённое заполнителем `stage`, извлекается из шаблона именования URL и переводится в верхний регистр. Заполнитель `stage` определяется как текст между `orange-booking-` и `.jsf` в пути URL.

![Извлечение с заполнителем](https://dt-cdn.net/images/extraction-placeholders-2000-0683e7a52f.png)

При использовании этого заполнителя результирующий список имён запросов выглядит следующим образом:

![Результаты именования запросов с заполнителем в списке распределённых трассировок сервиса](https://dt-cdn.net/images/placeholders-request-naming-1130-60e819ce95.png)

Построив точки входа и конечные точки вашего сервиса таким образом для мониторинга Dynatrace, вы можете отслеживать все ключевые операции вашей организации, такие как бизнес-транзакции, на высокогранулярном уровне.

### Правила очистки

Правила и настройки на уровне сервиса, включая правила очистки URL веб-запросов, переопределяют глобальные правила именования запросов.

* Вы можете определить глобальные правила именования запросов через API для очистки URL одного или нескольких сервисов одновременно.
* Сервисы веб-запросов имеют уникальные функции для генерации чистых URL через интерфейс.

В разделе **Service settings** > **Web request naming** вы можете:

* **Remove UUIDs, IP addresses and IBANs from URLs**.

  Это действие нормализует пути URL, содержащие UUID, IP-адреса и IBAN, заменяя конкретные значения статическими строками, связанными с содержимым, такими как `UUID`, `IPv4` и `IBAN`.

  Эта опция недоступна для сред, в которых включена функция [**Enhanced endpoints for SDv1**](enhanced-endpoints-sdv1.md "Используйте функцию Enhanced endpoints for SDv1 для получения более глубокого понимания производительности вашего приложения и улучшения возможностей мониторинга и устранения неполадок взаимодействий сервисов.").
* **Create clean URL rule**.

  Определите [регулярные выражения](../../../../../manage/tags-and-metadata/reference/regular-expressions-in-dynatrace.md "Узнайте, как использовать регулярные выражения в контексте Dynatrace.") для удаления частей пути URL сервиса веб-запросов, таких как идентификаторы.

Пользовательские имена запросов сервисов никогда не маскируются.

### Предварительный просмотр

Вы можете изменить правило именования запроса или комбинировать правила для создания ещё более детализированных имён запросов. **Preview Rule** показывает результат нового правила именования запроса.

![Изменение и предварительный просмотр правила именования запроса](https://dt-cdn.net/images/modify-naming-rule-2012-c9ae3151bf.png)

В приведённом выше примере новое правило `Booking {RequestAttribute:easyTravel destination} - {stage}` объединяет два предыдущих примера. Оно определяет шаблон именования запроса, который включает не только этап бронирования (`{stage}`), но и атрибут направления (`{RequestAttribute:easyTravel destination}`). Теперь мы получаем отдельный запрос для каждого этапа бронирования, плюс дальнейшее разделение на основе атрибута направления.

### Маскирование данных

Вы можете выбрать отображение немаскированных данных для конкретных запросов, установив флажок **Access unmasked data**.

Это потенциально раскроет [персональные данные](../../../../../manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace.md#service-request-monitoring "Узнайте, какие типы данных конечных пользователей могут быть захвачены при мониторинге Dynatrace и какие методы доступны для маскирования персональных данных.") до их сохранения и отображения.

## Анализ сервиса

Полная ценность настройки ваших критически важных для бизнеса запросов таким образом становится очевидной, когда вы углубляетесь в анализ, доступный для каждого отдельного запроса на уровне сервиса.

Атрибуты запросов дают вам абсолютную гибкость в идентификации и именовании ваших запросов на основе ваших бизнес-требований. Dynatrace отслеживает каждый запрос из конца в конец и показывает, как все запросы связаны между собой.

Многоуровневый анализ

Чтобы проиллюстрировать многоуровневый анализ, добавим ещё одно правило именования запроса, которое разделяет запросы бронирования на основе атрибута `Loyalty`.

![Многоуровневый анализ сервиса](https://dt-cdn.net/images/multi-tiered-analysis-2002-c95686fde7.png)

Это добавление приводит к отдельным запросам к этому сервису на основе статуса лояльности.

Теперь, когда мы смотрим на отдельную трассировку, мы можем увидеть, насколько полезны два определённых атрибута запроса. В этом примере мы видим, что `Booking Shizunai` был выполнен клиентом со статусом лояльности `PLATINUM`. Это было достигнуто путём определения правила именования запроса на сервисе `easyTravel Customer Frontend`, которое отслеживает бронирования по направлениям, и отдельного правила именования запроса, которое отслеживает бронирования на бэкенд-сервисе `BookingServiceTom` на основе статуса лояльности.

![Трассировка многоуровневого анализа сервиса](https://dt-cdn.net/images/multi-tiered-analysis-trace-2024-0b4a266f8f.png)

Многомерный анализ

Поскольку правила именования запросов создают отдельные запросы сервисов, у вас есть ещё больше возможностей фильтрации на основе атрибутов новых запросов, которые были определены. В примере ниже разбивка по направлениям комбинируется со статусом лояльности `PLATINUM`.

![Атрибуты ресурсов](https://dt-cdn.net/images/multidim-analysis-2036-7a8488d166.png)

![Атрибуты запросов в многомерном анализе](https://dt-cdn.net/images/multidimen-analysis-request-attributes-2020-b78f22fb23.png)

Вы также можете использовать эту функциональность в сочетании с [мощной иерархической фильтрацией](../../../services-classic/service-flow/service-flow-filtering.md "Узнайте, как работает фильтрация сервисов и как её использовать.").
В этом примере мы анализируем запросы бронирования на этапе `finish`, с направлением `shizunai`, диапазоном времени отклика 1с-2с и статусом лояльности Platinum.

![Фильтрация трассировок по запросу](https://dt-cdn.net/images/multidim-analysis-traces-2014-6fc5fb8941.png)

Хотя это было возможно с использованием только атрибутов запросов, именование запросов делает этот подход ещё более мощным.
