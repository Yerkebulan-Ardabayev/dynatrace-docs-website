# Настройка обнаружения сбоев в Service Detection v2

* Latest Dynatrace
* How-to guide
* Время чтения: 3 мин
* Обновлено 24 ноября 2025 г.

Service Detection v2 (SDv2) позволяет обнаруживать сбои на основе атрибутов span, которые имеют отношение к обнаружению сбоев.
Эти релевантные span включают, например, корневые span [запросов](../../../../../semantic-dictionary/fields.md#request "Get to know the list of global fields that have a well defined semantic meaning in Dynatrace and can be used across different monitoring types.").
Вы можете использовать стандартные правила обнаружения Dynatrace, а также определять собственные пользовательские правила.

## Цель и контекст

На этой странице описывается обнаружение сбоев для SDv2, как использовать стандартные правила обнаружения и как создавать собственные пользовательские правила.

Обнаружение сбоев состоит из:

* Набора правил (ruleset) — DQL-matcher, который определяет общие атрибуты span и ресурсов, подлежащие оценке.
* Условий сбоя, при которых span будет считаться неуспешным.
* Пользовательских правил сбоев, которые используют DQL-matcher для указания дополнительных условий, при которых span будет считаться неуспешным.
* Игнорируемых исключений, при которых можно принудительно считать span успешным, даже если определённые исключения произошли.

### Наборы правил и правила обнаружения сбоев

* Наборы правил и правила оцениваются по порядку сверху вниз.
* Пользовательские наборы правил всегда оцениваются до стандартных.
* Применяется первый совпавший набор правил.
* В рамках применённого набора правил оцениваются все правила.

### Стандартные наборы правил и правила обнаружения сбоев

Dynatrace предоставляет стандартные наборы правил и правила обнаружения сбоев.
Кроме того, вы можете добавлять пользовательские наборы правил и правила, как описано в разделе [Создание нового набора правил](failure-detection-v2.md#create-new-ruleset "Find out how to detect failed requests within services.").

Запрос считается успешным, если ни одно правило обнаружения сбоев не совпало.

## Шаги

Обнаружение сбоев настраивается в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **Services** > **Failure detection**.

Для настройки обнаружения сбоев:

1. Создайте набор правил.
2. Определите условия сбоя.
3. Определите игнорируемые исключения. Необязательно
4. Определите правило(а). Необязательно
5. Определите переопределения кодов состояния HTTP или gRPC. Необязательно

### Создание набора правил

1. В разделе **Failure detection** выберите **Add ruleset**.
2. Заполните следующие обязательные и необязательные поля:

   * **Ruleset name**: Обязательное Пользовательское имя для набора правил.
   * **Description**: Необязательное Читаемое описание правила.
   * **Matching condition**: Обязательное [DQL-matcher](../../../../../platform/openpipeline/reference/dql-matcher-in-openpipeline.md "Examine specific DQL functions and logical operators for log processing."), который определяет атрибуты span и ресурсов, к которым применяется набор правил.
     Если условие совпадения выполняется, набор правил оценивается.
3. Для сохранения выберите **Save changes**.

### Определение условий сбоя

В рамках набора правил эти параметры определяют условия, при которых span будет считаться неуспешным.

* **HTTP status codes**: Один или несколько кодов состояния HTTP.
  Если возвращается один из этих кодов, span будет считаться неуспешным.
  По умолчанию установлено значение `500-599`.
* **gRCP status codes**: Один или несколько кодов состояния gRPC.
  Если возвращается один из этих кодов, span будет считаться неуспешным.
  По умолчанию установлено значение `2,4,12,13,14,15`.
* **Span status code**: Если включено: когда код состояния span указывает на сбой, span будет считаться неуспешным.
  По умолчанию включено.
* **Exceptions**: Если включено: когда происходит необработанное исключение, span будет считаться неуспешным.
  По умолчанию включено.

### Добавление игнорируемых исключений

Необязательно

В рамках набора правил вы можете исключить определённые исключения.
Если span содержит определённый тип и сообщение исключения, это исключение не приведёт к тому, что span будет считаться неуспешным.

Выберите **Add ignored exception** и введите значения для обоих полей:

* **Exception type contains**.
* **Exception message contains**.

Пример игнорируемого исключения

Чтобы игнорировать все NullPointerException, содержащие ожидаемое значение null в сообщении, задайте следующие значения.

* Exception type contains: `NullPointerException`
* Exception message contains: `<Expected null value>`

### Определение пользовательских правил сбоев

Необязательно

Помимо уже определённых условий сбоя, вы можете задать дополнительные пользовательские условия, при которых span будет считаться неуспешным.
Эти условия называются пользовательскими правилами сбоев.
Они определяются с помощью любой пары ключ/значение с использованием DQL-matcher.

1. Для определения правила выберите **Add custom rule**.
2. Заполните следующие обязательные поля.

   * **Rule name**: Описательное имя.
   * **DQL condition**: [DQL-matcher](../../../../../platform/openpipeline/reference/dql-matcher-in-openpipeline.md "Examine specific DQL functions and logical operators for log processing."), состоящий из одной или нескольких функций DQL.
     Если условие совпадения выполняется, правило срабатывает.
     Если условие совпадения не выполняется, правило не срабатывает.

     Независимо от того, выполняется ли условие, span оценивается по последующим правилам.

При срабатывании правила создаётся соответствующая запись в [`dt.failure\_detection.results array](../../../../../semantic-dictionary/model/trace.md#failure-detection-result "Get to know the Semantic Dictionary models related to traces.").

Пример пользовательского правила

Это пример пользовательского правила для обнаружения сбоев бизнес-логики.

* Rule name: `Business Transaction Failure`
* DQL matching rule: `transaction.status == "FAILED"`

### Переопределение обнаружения сбоев

Необязательно

Иногда требуется, чтобы span считался успешным, даже если он содержит индикаторы ошибок.
В рамках набора правил вы можете определить условия, которые принудительно обеспечат успех.

Для этого выберите  > **Override failure detection with success forcing rules**.
Затем определите одно или несколько правил.

* **HTTP status codes**: Коды ошибок HTTP, которые принудительно обеспечат успех на стороне сервера.
* **gRPC error codes**: Коды ошибок gRPC, которые принудительно обеспечат успех на стороне сервера.
* **Span status code**: Если код состояния span равен `OK`, он будет считаться успешным независимо от любых других индикаторов ошибок.
* **Force success on specific exceptions**: Определённые исключения, которые должны принудительно обеспечить успех, см. [Определение игнорируемых исключений](#ignored-exceptions).
* **Custom success forcing rules**: Пользовательские условия, которые должны принудительно обеспечить успех, см. [Определение правил](#define-rules).

Пример обработанного исключения

Чтобы принудительно обеспечить успех для примера span, содержащего обработанное исключение, задайте следующие значения.

* Evaluated expression: `Any(span.events[][span_event.name] == "exception" and span.events[][exception.escaped] != false)`
* Failure detection result: `reason="exception", verdict="success", exception_ids`

Это полезно для приложений, которые используют исключения как часть обычного потока управления, а не как индикаторы ошибок.

Пример пользовательского правила принудительного успеха

Чтобы принудительно обеспечить успех для ожидаемого бизнес-условия, задайте следующие значения.

* Rule name: `Expected Cache Miss`
* DQL matching rule: `cache.result == "MISS" and operation.type == "READ"`

Это полезно для бизнес-специфичных сценариев, где существуют ожидаемые бизнес-условия.

### Редактирование пользовательских наборов правил и правил

Вы можете менять порядок пользовательских наборов правил и правил для изменения приоритета.

Вы также можете редактировать пользовательские наборы правил и правила.
Для этого перейдите к набору правил или правилу и выберите **Details** > .
Отредактируйте необходимые поля и затем выберите **Save changes**.

### Удаление пользовательских наборов правил и правил

Чтобы удалить пользовательский набор правил или правило, перейдите к набору правил или правилу и выберите **Delete** > .
Выберите **Save changes** для окончательного удаления набора правил или правила, или **Discard changes** для сохранения правила.

Удалять можно только пользовательские наборы правил и правила, но не встроенные.
