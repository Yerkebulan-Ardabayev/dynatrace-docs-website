# Настройка сопоставления шаблонов URL-путей в Service Detection v2


* Latest Dynatrace
* How-to guide

Dynatrace версии 1.330+

С помощью функции **сопоставления шаблонов URL** Service Detection v2 (SDv2) может генерировать более качественные имена эндпоинтов, извлекая шаблоны URL-путей из необработанных URL-путей.

Dynatrace может извлекать стабильные, низковолатильные шаблоны URL из необработанных URL-путей и записывать их в атрибут span `url.path.pattern`. Атрибут `url.path.pattern` определяется предварительно и затем используется обнаружением эндпоинтов SDv2 для корректного именования и группировки эндпоинтов.

Функция **сопоставления шаблонов URL** предназначена для ситуаций, когда фреймворки или серверы не предоставляют шаблоны маршрутов (например, обратные прокси или библиотеки без `http.route`), чтобы имена эндпоинтов оставались осмысленными и согласованными, сохраняя корректную гранулярность для критически важных метрик производительности.

## Цель и контекст

На этой странице описывается сопоставление шаблонов URL-путей для SDv2 и способы создания правил сопоставления шаблонов URL-путей, в частности, как правильно определять шаблоны URL-путей.

* SDv2 применяет сопоставление шаблонов URL к типам span `span.kind == server` и `span.kind == consumer`.
* Обнаружение эндпоинтов включает [встроенное правило эндпоинта](endpoint-detection-v2.md#default-endpoint-rules "Find out how to detect endpoints that are entry points into your service.") **HTTP request method and URL path pattern**. Это правило использует `url.path.pattern` и имеет приоритет над правилом **HTTP request method and route** при определении имён эндпоинтов. Таким образом, извлечённый шаблон URL имеет приоритет, когда доступны оба атрибута — `url.path.pattern` и `http.route`.
* Чтобы воспользоваться функцией **сопоставления шаблонов URL**, необходимо [создать правила сопоставления шаблонов URL-путей](#create-rule), указав необходимые шаблоны URL-путей в соответствии с синтаксисом шаблонов.
* Шаблон URL-пути описывает, как необработанный URL-путь (например, `/order/12345`) обобщается в стабильный шаблон (например, `/order/{id}`). Он используется для создания низковолатильных значений атрибута span `url.path.pattern`.

## Логика сопоставления шаблонов и приоритеты

* Обрабатываются только span с `span.kind == server` и `span.kind == consumer`.
* Одно правило сопоставления шаблонов URL-путей может содержать один или несколько шаблонов URL-путей.
* Правила сопоставления шаблонов URL-путей оцениваются по порядку сверху вниз. Применяется только первое совпавшее правило. Первое правило с совпавшим шаблоном URL применяется к span.
* Шаблоны URL-путей в рамках совпавшего правила также оцениваются по порядку сверху вниз. Применяется только первый совпавший шаблон URL.

  Первый совпавший шаблон URL-пути определяет результирующее значение атрибута `url.path.pattern` для span. Когда шаблон URL совпадает, атрибут span `url.path.pattern` основывается на этом совпавшем шаблоне URL.
* Шаблоны URL оперируют сегментами пути; сегменты пути — это части URL между символом косой черты `/`.
* Ведущие косые черты в URL-путях нормализуются перед сопоставлением.

  Перед оценкой шаблонов URL любое количество ведущих косых черт в пути сокращается до одной `/` для обеспечения согласованного и надёжного сопоставления.
* Сопоставление чувствительно к регистру.

## Справочник по синтаксису шаблонов URL

Используйте приведённую ниже информацию для создания осмысленных шаблонов URL-путей для ваших [правил сопоставления шаблонов URL-путей](#create-rule).

Следующие символы или значения допустимы в шаблонах URL-путей. Используйте их для замены высококардинальных частей ваших URL (идентификаторов, номеров версий и глубоких внутренних URL-путей) заполнителями или символами подстановки, сохраняя при этом общую структуру эндпоинта узнаваемой.

* Литеральные сегменты
* Заполнители `{placeholder-name}`
* Переменные сегменты `_`
* Подстановочный символ `*`

### Литеральные сегменты

* Представлены литеральным значением сегмента пути.
* Совпадают с точным сегментом пути.
* Копируются в результат без изменений.
* Используйте для фиксированных частей пути, которые никогда не меняются.

### Заполнитель `{placeholder-name}`

* Представлен именем заполнителя в фигурных скобках, например `{id}` или `{date}`.
* Совпадает ровно с одним сегментом пути.
* Заменяется именем заполнителя в фигурных скобках.
* Используйте для скрытия динамических значений пути, например идентификаторов, UUID или временных меток.

### Переменные сегменты `_`

* Представлены символом подчёркивания `_`.
* Совпадают ровно с одним сегментом пути.
* Заменяются исходным значением сегмента.
* Используйте, когда сегмент должен оставаться видимым, например для версионированных API, где версия должна оставаться видимой.

### Подстановочный символ `*`

* Представлен символом звёздочки `*`.
* Совпадает с нулём или более завершающими сегментами. Должен быть последним токеном в шаблоне URL.
* Заменяется на `*`.
* Используйте для сопоставления с любыми оставшимися сегментами пути.

## Настройка сопоставления шаблонов URL

В разделах ниже описано, как создавать и управлять правилами сопоставления шаблонов URL.

### Создание правила сопоставления шаблонов URL

Для создания правила сопоставления шаблонов URL-путей:

1. Перейдите в **Settings** (Dynatrace Classic) или **Settings Classic** > **Service detection** > **URL path pattern matching**.

Если опция **Settings Classic** недоступна, используйте Settings API для создания правил сопоставления шаблонов URL.

2. Выберите **Add rule**.
3. Заполните следующие обязательные и необязательные поля.

   * **Rule name**: Обязательное

     Пользовательское имя для правила.
   * **Description**: Необязательное

     Читаемое описание правила.
   * **Matching condition**: Обязательное

     DQL-matcher.
     Если условие совпадения выполняется, правило оценивается.
4. Выберите **Add item** и укажите шаблон URL-пути в соответствии с [справочником по синтаксису шаблонов](#syntax-reference). Повторите, если необходимо добавить дополнительные шаблоны путей.
5. При необходимости используйте ![Drag handle](https://dt-cdn.net/images/drag-handle-turquoise-600-1aa0e5ea00.svg "Drag handle") для изменения порядка добавленных шаблонов URL-путей, учитывая, что первый совпавший шаблон используется для определения значения атрибута `url.path.pattern`.
6. Выберите **Save changes**.

### Управление правилами сопоставления шаблонов URL

Вы можете редактировать, отключать или удалять правила сопоставления шаблонов URL.

1. Перейдите в **Settings** (Dynatrace Classic) или **Settings Classic** > **Service detection** > **URL path pattern matching**.

Если опция **Settings Classic** недоступна, используйте Settings API для создания правил сопоставления шаблонов URL.

2. Внесите необходимые изменения в правила сопоставления шаблонов URL-путей:

   * ![Expand row](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Expand row") в столбце **Details** для редактирования правила.
   * ![Toggle icon](https://dt-cdn.net/images/icon-toggle-barista-701-35879d6adf.png "Toggle icon") в столбце **Enabled** для отключения или включения правила.
   * ![Drag handle](https://dt-cdn.net/images/drag-handle-turquoise-600-1aa0e5ea00.svg "Drag handle") для изменения порядка правил.
   * ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") в столбце **Delete** для окончательного удаления правила.
3. Выберите **Save changes** для применения внесённых изменений.

## Сценарии использования и примеры

Функция **сопоставления шаблонов URL** особенно полезна для следующих сценариев:

* Маршрутизация через обратный прокси Nginx
* Версионированные API
* REST-эндпоинты с идентификаторами
* Внутренние API с глубокими путями
* Пути смешанной глубины
