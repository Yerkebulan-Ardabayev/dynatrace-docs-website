# Настройка обнаружения сервисов в Service Detection v2

# Настройка обнаружения сервисов в Service Detection v2

* Classic
* How-to guide
* 2-min read
* Updated on Nov 24, 2025

Service Detection v2 (SDv2) позволяет определять сервисы на основе атрибутов ресурсов span.
Вы можете использовать правила обнаружения Dynatrace по умолчанию, а также определять собственные правила.
Те же правила также применяются к приёму журналов и метрик, что позволяет Dynatrace связывать входные данные из всех источников.

## Цель и контекст

На этой странице описывается обнаружение сервисов для SDv2, использование правил обнаружения по умолчанию и создание собственных правил.

### Правила обнаружения сервисов

* Правила применяются к span-данным, метрикам и журналам.

* Правила оцениваются по порядку, сверху вниз.
* Пользовательские правила всегда оцениваются перед правилами по умолчанию.
* Применяется только первое совпадающее правило.

### Правила обнаружения сервисов по умолчанию

Dynatrace предоставляет несколько правил обнаружения сервисов по умолчанию.
Кроме того, можно создавать пользовательские правила, как описано в разделе [Создание нового правила](/docs/observe/application-observability/services/service-detection/service-detection-v2/service-detection-v2#create-new-rule "Узнайте, как обнаруживать сервисы на основе атрибутов ресурсов OpenTelemetry.").

Любые будущие изменения правил по умолчанию будут применяться по запросу: новые правила будут поставляться в отключённом состоянии; вы можете выбрать, активировать ли их.

Идентификатор сервиса — это уникальный идентификатор, например `SERVICE-649B4E44CBA804F4`, который является результатом хэширования значений атрибутов, используемых в шаблоне имени, дополнительных атрибутах обнаружения сервиса и [атрибутах разделения сервиса](/docs/observe/application-observability/services/service-detection/service-detection-v2/service-splitting-v2 "Узнайте, как разделять обнаруженные сервисы на основе атрибутов ресурсов.") при их наличии.

## Шаги

Правила обнаружения настраиваются в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Process and contextualize** > **Services** > **Service detection**.

### Создание нового правила

1. В разделе **Service detection** выберите **Add rule**.
2. Заполните следующие необязательные и обязательные поля:

   * **Rule name**: Обязательное поле

     Определяемое пользователем имя для правила.
   * **Description**: Необязательное поле

     Понятное описание правила.
   * **Matching condition**: Обязательное поле

     [Матчер DQL](/docs/platform/openpipeline/reference/dql-matcher-in-openpipeline "Изучите конкретные функции DQL и логические операторы для обработки журналов.").
     Если условие совпадения применимо, правило оценивается.
   * **Service name template**: Обязательное поле

     Имя, которое вы хотите присвоить сервису.
     Можно использовать обычный текст или атрибуты ресурсов, заключённые в фигурные скобки (`{}`).
     Чтобы правило применялось, span должен содержать все указанные атрибуты ресурсов.
     Изменение статических частей шаблона или изменение порядка используемых атрибутов не влияет на идентификатор сервиса. Только полное удаление атрибута или добавление нового атрибута также изменит идентификатор сервиса.
   * **Additional service detection attributes**: Необязательное поле

     Дополнительные атрибуты, используемые для обнаружения и разделения сервисов без влияния на сгенерированное имя.
     Каждый атрибут состоит из атрибута ресурса, указанного без фигурных скобок (например, `service.name` или `k8s.workload.name`).
     Добавление или удаление атрибутов в этом разделе изменит результирующий идентификатор сервиса.

     Можно применить до 10 дополнительных атрибутов обнаружения сервиса.
3. Выберите **Save changes**.

### Редактирование пользовательских правил

Вы можете изменять порядок пользовательских правил.

Вы также можете редактировать пользовательское правило.

1. Перейдите к правилу и выберите **Details** > .
2. Отредактируйте поля по мере необходимости.
3. Выберите **Save changes**.

### Удаление пользовательских правил

Для удаления пользовательского правила:

1. Перейдите к правилу и выберите **Delete** > .
2. Выберите **Save changes** для окончательного удаления правила или **Discard changes**, чтобы сохранить правило.

Удалять можно только пользовательские правила, встроенные правила удалить нельзя.

## Часто задаваемые вопросы

### Как удалить атрибут, использовавшийся в имени сервиса, без изменения идентификатора сервиса?

Чтобы удалить атрибут, использовавшийся в имени сервиса, без изменения идентификатора сервиса:

1. Удалите атрибут из поля **Service name template**.
2. Добавьте атрибут в раздел **Additional service detection attributes**.

Таким образом, он по-прежнему будет влиять на идентификатор сервиса, но больше не будет частью имени сервиса.

### Мои **Matching conditions** соответствуют определённым span-данным, но моё правило всё равно не применяется. Почему?

Убедитесь, что все атрибуты, используемые в качестве частей **Service name template** и **Additional service detection attributes**, также существуют в span-данных. Наличие этих атрибутов является неявным условием совпадения.
