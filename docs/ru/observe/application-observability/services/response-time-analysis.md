# Анализ времени отклика

* Latest Dynatrace
* Tutorial
* 3-min read
* Published Jan 12, 2026

Анализ времени отклика в Dynatrace помогает пользователям быстро выявлять ключевые факторы, влияющие на низкую производительность сервиса. Анализируя исходящие вызовы, взаимодействие с базами данных и очередями, а также внутренние процессы сервиса, эта функция предоставляет чёткую разбивку того, на что уходит время. Кроме того, она даёт инфраструктурную перспективу, предоставляя сведения о ключевых метриках связанных инфраструктурных компонентов.

Этот анализ предназначен как для реактивного использования — для расследования конкретных проблем с производительностью, — так и для проактивного — для исследования потенциальных узких мест. Пользователи также могут сравнивать различные временные периоды, чтобы понять, как изменилось время отклика сервисов и их конечных точек.

## Ключевые возможности

## Доступ и навигация

Новый анализ времени отклика доступен в виде отдельной вкладки **Response Time** в [![Services](https://dt-cdn.net/hub/logos/services.png "Services") **Services**](services-app.md "Maintain centralized control over service health, performance, and resources with the Services app."). Он разработан для поддержки как контекстных, так и исследовательских рабочих процессов.

### Анализ времени отклика для конкретного сервиса

При просмотре сведений о конкретном сервисе выберите **Analyze response time** в верхней части графика времени отклика или для определённой конечной точки сервиса. Откроется вкладка **Response Time** с фильтрами на основе выбранного сервиса.

### Контекстный доступ через параметры детализации по проблемам

Когда деградация времени отклика определена как первопричина проблемы, вкладка **Response Time** открывается с фильтрами на основе затронутого сервиса и конечной точки.

### Исследовательский доступ

Чтобы изучить сервисы и определить наиболее медленные из них, вручную перейдите на вкладку **Response Time** в [![Services](https://dt-cdn.net/hub/logos/services.png "Services") **Services**](services-app.md "Maintain centralized control over service health, performance, and resources with the Services app.").

* Настройте фильтры для исследования сбоев по сервисам, конечным точкам и временным периодам.
* Сравните с другими временными периодами.

## Начало работы

Чтобы выяснить, что работает медленно

1. Перейдите в [![Services](https://dt-cdn.net/hub/logos/services.png "Services") **Services**](services-app.md "Maintain centralized control over service health, performance, and resources with the Services app.") и выберите вкладку **Response Time**.
2. Примените [сегменты](../../../manage/segments.md "Use segments to logically structure and conveniently filter observability data across apps.") и фильтры для фокусировки на нужных сервисах и конечных точках.

   Для анализа отдельного сервиса вы также получаете разбивку вклада с классификацией вызовов по:

   * Внутреннему выполнению сервиса
   * Исходящим вызовам
   * Использованию базы данных
   * Обработке сообщений

Чтобы выяснить, почему работает медленно

1. Включите **Analyze details** над таблицей для получения дополнительных сведений, помогающих понять, что в наибольшей степени способствовало времени отклика выбранных сервисов и трассировок.
2. Анализ времени отклика предоставляет детальную разбивку того, на что уходит время при выполнении запросов к сервису. Это включает:

   | Категория | Описание |
   | --- | --- |
   | **Outbound calls** | Показывает время ожидания ответа от зависимых нисходящих компонентов, таких как вызовы API или обработка на стороне клиента. |
   | **Database interactions** | Выделяет время, затраченное на запросы к базе данных и взаимодействие с ней. |
   | **Service internals** | Разбивает время, затраченное внутри самого сервиса, включая выполнение кода и внутреннюю обработку. |
   | **Infrastructure metrics** | Отображает ключевые метрики для связанных инфраструктурных компонентов, таких как использование ЦП, потребление памяти и задержка сети. |

## Сценарии использования

### Расследование медленных сервисов

Когда сервис воспринимается как медленный, используйте анализ времени отклика для:

* Определения того, вызвана ли проблема внутренними процессами сервиса, зависимыми нисходящими компонентами, взаимодействием с базой данных или перегрузками инфраструктуры.
* Детализации в конкретных областях для выявления первопричины медленной работы.
* Просмотра связанных метрик инфраструктуры для определения того, вносят ли аппаратные или сетевые проблемы вклад в возникшую проблему.

### Проактивный мониторинг производительности

Используйте анализ времени отклика проактивно для:

* Исследования потенциальных узких мест в производительности до того, как они начнут влиять на пользователей.
* Мониторинга ключевых метрик для критически важных сервисов и инфраструктурных компонентов.
* Получения более глубокого понимания того, как различные факторы влияют на общее время отклика.
