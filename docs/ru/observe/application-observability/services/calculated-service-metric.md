* Latest Dynatrace

Dynatrace автоматически собирает важные метрики для сервисов без необходимости настройки. Вам могут понадобиться дополнительные бизнес- или технические метрики, специфичные для вашего приложения. Эти метрики могут быть вычислены и получены на основе широкого разнообразия доступных данных в собранной распределённой трассировке. Вы также можете разделить эти метрики по нескольким измерениям, например, по атрибуту запроса или HTTP-методу.

Извлечение метрик OpenPipeline теперь заменяет вычисляемые метрики сервисов в качестве метода создания дополнительных бизнес- или технических метрик сервисов. Подробнее об извлечении метрик см. Извлечение метрик из спанов и распределённых трассировок.
Если вы являетесь существующим клиентом, вы всё ещё можете использовать их до тех пор, пока мы не объявим об их устаревании. Если вы новый клиент, вы больше не сможете создавать вычисляемые метрики сервисов.

## Почему стоит начать использовать OpenPipeline вместо классических вычисляемых метрик сервисов?

* Более гибкое создание метрик из спанов — с OpenPipeline метрики больше не ограничены поддержкой максимум 100 значений измерений («top X»).

  + Вы можете хранить полную кардинальность измерений
  + Вы можете свободно комбинировать измерения
  + Мощные возможности приёма, обработки и трансформации доступны через единое решение (которое работает одинаково для других областей конфигурации, таких как логи и события).
* Лучшая масштабируемость и производительность — приём и потоковая обработка на уровне платформы, созданные для работы с объёмами, превышающими петабайты; больше нет ограничения в 500 вычисляемых метрик сервисов на тенант.
* Безопасность, управление затратами и управление данными — фильтрация и маскирование конфиденциальных или ненужных полей; назначение использования центра затрат для конкретных метрик; маршрутизация данных в определённые бакеты Grail с контролируемым сроком хранения.

Ограничения

* Только новые данные записываются в вычисляемые метрики; ретроспективные данные не включаются.
* Вы можете иметь до 500 включённых вычисляемых метрик на среду и до 100 включённых вычисляемых метрик на сервис.
* Классические вычисляемые метрики поддерживают не более 100 значений измерений. Это называется правилом «top X», так как вы можете выбрать меньше в зависимости от конфигурации. Однако, какие бы top 100 значений измерений вы ни выбрали, оставшиеся измерения агрегируются в один временной ряд, и значение измерения доступно через специальное измерение `remainder`. Условие фильтра [remainder](../../../dynatrace-api/environment-api/metric-v2/metric-selector.md#remainder "Configure the metric selector for the Metric v2 API.") позволяет фильтровать по этому измерению `remainder`.

* Вычисляемые метрики сервисов Grail с кардинальностью выше 2000 в любом 5-минутном окне за последние 2 недели или с момента последнего изменения метрики автоматически отключаются в Grail. Включение таких метрик в Grail не допускается. Если метрика уже включена в Grail, вы будете проинформированы об отклонении метрики через готовый дашборд [**Metric & Dimensions Usage + Rejections**](../../../dynatrace-api/environment-api/metric-v2/best-practices.md#identify-high-cardinality-situations "Best practices for metrics."). Чтобы включить классическую метрику в Grail и продолжить сбор входящих данных в Grail, убедитесь, что кардинальность остаётся ниже лимита.

## Создание вычисляемой метрики сервиса

### Шаги

1. Перейдите в **Settings** > **Server-side service monitoring** > **Calculated service metrics** > **Create new metric**.
2. Введите название метрики.
   Название и префикс `calc:service.` автоматически добавляются к ключу метрики. Обратите внимание, что после [включения метрики в Grail](#grail) префикс автоматически изменяется на `service.`
3. Выберите источник метрики из списка **Metric source**.

   * Если источником является атрибут запроса, выберите необходимую единицу измерения.
   * Необязательно: Чтобы исключить вклад данных приглушённых запросов, включите опцию **Ignore muted requests**.
4. Необязательно: Выберите зону управления. Новая метрика будет ограничена данными из этой зоны.
5. Задайте условия для определения того, какие запросы включаются в расчёт.

   Если вы указываете несколько условий, **все условия должны быть выполнены** для использования метрики.

   1. Выберите **Add condition**.
   2. Выберите атрибут для проверки.
   3. Выберите оператор условия.
   4. При необходимости укажите эталонное значение.

      * Предварительный просмотр классической метрики показывает список сервисов, включённых в пользовательскую метрику, и оценку потребления DDU.

        Предварительный просмотр учитывает только зону управления и условия на основе атрибутов сервиса. Эти атрибуты отмечены как `[Service property]` в списке атрибутов.

      * Метрика Grail: Хотя оценка DDU действительна только для классической метрики, для метрики Grail ожидайте повышенную точность и более полную информацию, так как в Grail хранится полная кардинальность измерений метрик (а не только значения Top X). Это может привести к большему количеству точек данных по сравнению с классической метрикой, что позволяет более детально анализировать данные.
6. Необязательно: Добавьте измерение к новой метрике.

   1. Включите **Split by dimension**.
   2. Выберите заполнители для определения измерения.

      * Выберите доступный заполнитель из **Dimension value pattern**.
      * Создайте [пользовательский заполнитель](#placeholder).
   3. Введите название измерения.
   4. Классическая метрика: Определите лимит значений top *X*.

      Лимит значений top *X* применяется только к классической метрике, а не к метрике Grail.

      1. В поле **Number of top values** укажите количество значений top *X*, которые будут вычислены для метрики.
      2. Из списков **Value sorting** и **Value aggregation** выберите сортировку и агрегацию значений top *X*.
7. Проверьте источник метрики и названия измерений. Они будут использоваться в пользовательском интерфейсе и API. После создания метрики вы не сможете их изменить.
8. Выберите **Save metric**.

## Создание пользовательского заполнителя

Когда заполнитель недоступен, вы можете создать пользовательский заполнитель. Все пользовательские заполнители должны быть использованы в шаблоне значения измерения, в противном случае вы можете удалить неиспользуемые пользовательские заполнители.

Предварительные знания

### Метод извлечения

У вас есть два метода для извлечения значения заполнителя:

* На основе разделителя. В этом случае Dynatrace извлекает значение, сравнивая значение источника с эталонным значением и указанным видом и позицией разделителя.
* На основе регулярных выражений. В этом случае Dynatrace использует извлечение с помощью регулярных выражений. Чтобы узнать, как Dynatrace использует регулярные выражения, см. Регулярные выражения в Dynatrace.

### Метод извлечения для атрибутов запроса

Заполнители, основанные на атрибутах запроса, предоставляют три варианта извлечения значения:

* **First**: Для извлечения значения используется первое вхождение атрибута.
* **Last**: Для извлечения значения используется последнее вхождение атрибута.
* **Count**: Значение равно количеству вхождений атрибута.

### Шаги

Чтобы создать пользовательский заполнитель

1. При создании или редактировании вычисляемой метрики сервиса выберите **Add custom placeholder**.
2. Введите имя для вашего заполнителя. Имя будет использоваться в поле **Dimension value pattern**.
3. Выберите источник для измерения.
4. Выберите [метод извлечения](#prior).

   Если источником является атрибут запроса,

   1. Необязательно: Установите флажок **Use from downstream services**, чтобы использовать дочерние вызовы в качестве источника значения.
   2. Необязательно: Ограничьте дочерние вызовы определённой зоной управления или тегом сервиса.
5. Выберите **Add**.
6. Обязательно: Используйте вновь созданный заполнитель в шаблоне значения измерения.

## Включение метрики в Grail

Предварительные условия

* Убедитесь, что кардинальность каждого измерения составляет менее 2000 в любом 5-минутном окне за последние 2 недели или с момента последнего изменения метрики.
* Проверьте измерение на наличие [неподдерживаемых заполнителей](../../../analyze-explore-automate/metrics/built-in-metrics-on-grail.md#unsupported-placeholders "Get to know the equivalents of the classic built-in metrics supported on Grail.") в Grail.

### Шаги

Чтобы включить метрику в Grail, включите опцию **Enable on Grail**.

### Результат

Префикс автоматически изменяется с `calc:service.` на `service.`. Название метрики и ранее существующее измерение сохраняются. Поддерживаемые заполнители преобразуются в дополнительные измерения, и для метрик, полученных OneAgent, добавляются новые измерения.

Полный список доступных измерений см. в [Встроенные метрики в Grail — Вычисляемые метрики сервисов](../../../analyze-explore-automate/metrics/built-in-metrics-on-grail.md#calc-service "Get to know the equivalents of the classic built-in metrics supported on Grail.").
