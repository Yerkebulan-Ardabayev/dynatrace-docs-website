# Работа с диагностическими сообщениями

# Работа с диагностическими сообщениями

* Classic
* 4-min read
* Updated on Jun 10, 2023

Захваченные распределённые трассировки являются основным источником данных для каузального ИИ Davis® от Dynatrace. Поскольку Davis опирается на высокое качество и точность данных, для анализа собранных данных необходимо соответствие определённым стандартам: все сервисы, методы, временны́е метки, свойства и т. д. должны быть корректно захвачены и переданы. Если этого не происходит, вы получите уведомление о проблеме в виде диагностического сообщения для затронутых трассировок.

* Если правильный сервис удаётся определить, распределённые трассировки с диагностическими сообщениями отображаются на странице **Distributed traces** соответствующего сервиса вместе со всеми другими распределёнными трассировками, связанными с этим сервисом.

  ![Distributed traces diagnostic message - capture error](https://dt-cdn.net/images/distributed-traces-capture-error-3112-189ed8979d.png)
* Если связанный сервис определить невозможно, например, из-за отсутствующего имени или типа сервиса, распределённые трассировки перечислены как **Unexpected services** (Неожиданные сервисы) в списке сервисов процесса, для которого были захвачены распределённые трассировки.

  Неожиданные сервисы могут быть вызваны проблемами сети (данные потеряны при передаче) или внезапным завершением процесса (перезапуск или масштабирование).

## Устранение неполадок диагностических сообщений

По содержанию диагностических сообщений вы можете понять возможные причины отсутствующего захвата или передачи данных, а также в какой мере ИИ Davis® может опираться на собранные данные. Иногда диагностические сообщения являются симптомами проблемы с настройкой, требующей расследования.

Для поддержания высокого качества данных рекомендуем:

* Использовать паттерны сетевых ошибок для определения источника проблемы.

  Например, если все запросы определённого сервиса имеют ошибки, связанные с сетью, причиной может быть кратковременный процесс, который не отправляет данные в Dynatrace.
* Уменьшить количество [атрибутов запросов](../../services/request-attributes.md "Understand what request attributes are and learn how to use them across all levels of all service-analysis views.") или изменить определения [пользовательских сервисов](../../services/service-detection/service-detection-v1/service-types/custom-services.md "Define entry points (a method, class, or interface) for custom services that don't use standard protocols.") для снижения числа запросов.

## Усечение данных трассировки

Трассировка может быть усечена, поскольку данные не полностью получены, скоррелированы или визуализированы. Обычно это происходит при превышении ограничений, установленных для защиты ресурсов вашей среды. При усечении данных трассировки вы получите конкретное диагностическое сообщение.

Если вы настроили [межсредовую трассировку](connect-environments.md "Analyze requests across environment boundaries."), данные, полученные из удалённых сред, не усекаются, а агрегируются.

Хотя ограничения, защищающие ресурсы вашей среды, как правило, не могут быть сняты, вы можете повлиять на усечение, вмешавшись в источник проблемы. Ниже приведены некоторые рекомендации по снижению усечения данных трассировки.

* Уменьшите количество узлов, отправляемых на трассировку, например, изменив пользовательские сервисы и активные функции OneAgent в вашей среде.

  OneAgent может отправить ограниченное количество узлов на трассировку. Как только он сталкивается с ограничениями ресурсов, новые узлы больше не создаются. Трассировка усекается при загрузке, и новые входящие данные не добавляются в трассировку. Это может повлиять на корреляцию объектов и снизить качество данных.
* Исследуйте трассировки, работающие в течение длительного времени.

  Для защиты ресурсов вашей среды принимаются меры, когда трассировки выполняются в течение длительного времени. Например, трассировки, которые перестают быть активными после 90 минут работы, истекают по тайм-ауту, тогда как трассировки, время начала вызова которых слишком далеко в прошлом, усекаются. В обоих случаях новые данные не добавляются в трассировку. Это может повлиять на корреляцию объектов и качество данных.
* Исследуйте трассировки с недостаточным или избыточным объёмом данных.

  + Когда данные трассировки не содержат достаточных метаданных о сетях или не все узлы были получены, корреляция объектов может оказаться невозможной, что приводит к усечению трассировок.
  + Когда вызов имеет слишком много зависимостей в процессе выполнения или трассировка содержит слишком много узлов, такая трассировка усекается. Рекомендуем исследовать связанный сервис, а в случае пользовательских сервисов обратиться к эксперту по продуктам Dynatrace через живой чат в вашей среде.
* Адаптируйте полный вид трассировки для отображения необходимых данных.

  Полный вид трассировки в Dynatrace обычно достаточно велик для отображения наиболее важной информации обо всей трассировке. Однако форматирование можно изменить, чтобы помочь вам визуализировать все необходимые данные.
