## Контекст трассировки между сервисами

Dynatrace обеспечивает непрерывную видимость потоков сервисов, распространяя контекст трассировки по мере перемещения транзакций между сервисами и компонентами. Это распространение необходимо для сквозного мониторинга распределённых приложений.

## Автоматическое распространение с OneAgent

При установке OneAgent он автоматически:

* Внедряет контекст трассировки в исходящие запросы.
* Считывает контекст трассировки из входящих запросов.
* Поддерживает контекст транзакции на границах между сервисами.
* Использует различные механизмы для корреляции распределённых трассировок, позволяя Dynatrace отображать полный поток транзакций через ваше приложение.

## Механизмы распространения

Dynatrace применяет несколько механизмов для поддержания контекста трассировки:

* **x-dynatrace** — для различных протоколов связи. Этот формат является проприетарным для Dynatrace.
* **traceparent** и **tracestate** — стандартный формат W3C, используемый как OneAgent, так и OpenTelemetry.
* **dtdTraceTagInfo** — пользовательское свойство для различных систем обмена сообщениями.

Конкретная реализация зависит от технологии: иногда используются заголовки HTTPS, свойства сообщений SQS или внедрение в полезную нагрузку AWS EventBridge.

## Смешанные среды с OpenTelemetry

В смешанных средах, где используются как OneAgent, так и инструментирование OpenTelemetry, распространение контекста трассировки устраняет разрыв:

* Как OneAgent, так и OpenTelemetry используют формат W3C Trace Context.
* Контекст должен последовательно распространяться по каналу связи.
* Для HTTP это стандартизировано в заголовках.
* Для других протоколов, таких как обмен сообщениями и события, необходимо обеспечить согласованность размещения между OneAgent и OpenTelemetry.

## Включение W3C trace context

Существует несколько причин включить W3C trace context в Dynatrace:

* Соответствие отраслевому стандарту — соответствие спецификации W3C.
* Независимая от поставщика трассировка — работает в гетерогенных средах с несколькими решениями для мониторинга.
* Перспективность — соответствует отраслевому направлению развития распределённой трассировки.

### Особенности использования W3C trace context

* Проблемы совместимости — несмотря на то что это стандарт, реальные реализации могут различаться.
* Поведение браузера и приложений — некоторые клиенты могут многократно отправлять один и тот же traceId, что влияет на качество трассировки.
* Конфликты инструментов — несколько APM-инструментов в одном процессе могут перезаписывать контекст друг друга.

### Настройка W3C trace context

Чтобы включить W3C trace context:

**Latest Dynatrace**

1. Перейдите в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Collect and capture** > **General monitoring settings** > **OneAgent features**.
2. Включите **Send W3C Trace Context HTTP headers** и **Send W3C Trace Context gRPC headers**.

**Dynatrace Classic**

1. Перейдите в **Settings** > **Preferences** > **OneAgent features**.
2. Включите **Send W3C Trace Context HTTP headers** и **Send W3C Trace Context gRPC headers**.

Хотя стандарт W3C формально описывает распространение через HTTP, Dynatrace и отрасль в целом применяют эти концепции к другим протоколам связи.
