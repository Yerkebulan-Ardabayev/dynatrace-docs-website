# Основные запросы к базам данных

* Classic
* Практическое руководство
* Чтение: 4 мин
* Обновлено 13 сентября 2022

Представление **Top database statements** позволяет понять общую активность базы данных, а не только активность одной конкретной базы данных. Это представление также может использоваться для более качественного анализа сквозной активности баз данных за определенные временные интервалы.

Чтобы получить доступ к странице **Top database statements**

1. Перейдите в **Multidimensional Analysis** или ![Databases Services Classic](https://dt-cdn.net/images/databases-512-6aa6fff194.png "Databases Services Classic") **Database Services Classic**.
2. Выберите **Top database statements**.

## Анализ запросов к базам данных

На странице **Top database statements** отображается график, показывающий основные SQL- и NoSQL-запросы в вашей среде с течением времени. На графике также выделены несколько основных запросов по количеству выполнений. Таблица под графиком содержит 100 **основных измерений**.

![Страница Top database statements](https://dt-cdn.net/images/top-database-statements-3564-3bd08ea1b1.png)

Вы можете настроить представление, изменяя **Metric** и другие параметры или добавляя фильтр.

* Различные метрики, такие как **Fetch count**, **Response time** и **Row count**, дают разный порядок запросов. В зависимости от временных затрат на запрос, частоты выполнения запроса или уровня ошибок, на странице сначала перечисляются запросы, которые выполняются чаще или являются более ресурсоемкими, от наибольшего значения выбранной метрики к наименьшему.
* Вы можете использовать **Filter requests** для сужения списка запросов к базам данных.

Примеры фильтрации запросов

Следующий пример показывает представление, настроенное с метрикой **Response time** и фильтром по диапазону значений `>=100ms`. В списке отображаются все запросы к базам данных, время отклика которых составило не менее 100 мс.

![Фильтр по времени отклика в анализе основных запросов к БД](https://dt-cdn.net/images/response-time-filter-in-top-database-statement-analysis-3684-0962b8fba9.png)

Представление в этом примере настроено с метрикой **Fetch count** и фильтром по диапазону значений `>=5`. В списке отображаются запросы к базам данных с как минимум 5 обращениями к базе данных на одно выполнение.

![Количество обращений в анализе основных запросов к БД](https://dt-cdn.net/images/fetch-count-in-top-database-statements-analysis-3548-bdb184cbd0.png)

* Вы можете экспортировать данные таблицы в файл формата CSV (значения, разделенные запятыми).

  1. В правом нижнем углу страницы выберите **Show export menu** ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More").

     ![Show export menu](https://dt-cdn.net/images/show-export-menu-107-2a8a76c9a2.png)
  2. Выберите **Export visible data** или **Export table data**.
* На графике используются [данные трассировок и запросов](/docs/observe/application-observability/multidimensional-analysis#data-source "Configure a multidimensional analysis view and save it as a calculated metric."), которые имеют разные периоды хранения данных. Для временных интервалов, содержащих данные старше 10 дней, вы можете включить переключатель **Show data retention**, чтобы лучше понять, какие данные доступны за какой период, непосредственно из графика.

## Анализ отдельных SQL-запросов

Чтобы получить доступ к нескольким представлениям анализа конкретных запросов, включая **Service backtrace** и анализ **Outliers**, выберите **More** (**...**).

### Детали запроса к базе данных

Вы можете напрямую перейти к анализу отдельных запросов, выбрав **Filter**. Альтернативно выберите **More** (**...**) > **Statement details**.

![Распределенные трассировки отдельного запроса к БД](https://dt-cdn.net/images/distributed-traces-top-database-statements-1593-b6fb81c79e.png)

Выберите запрос, чтобы просмотреть все варианты выбранного запроса. Dynatrace агрегирует операции вставки, обновления и удаления на уровне таблицы.

![Детали отдельного запроса к БД](https://dt-cdn.net/images/details-individual-statements-1564-56ba5e5697.png)

### Обратная трассировка сервиса базы данных

Чтобы получить доступ к анализу обратной трассировки для выбранного запроса, выберите **Service backtrace** в меню **Analyze**. Страница **Backtrace** предоставляет информацию о пользовательском взаимодействии, которое вызвало выбранную перестановку SQL-запроса, и о пользовательском взаимодействии или запросе сервиса, из которого произошел запрос. Выберите любой из входящих запросов для визуализации обратной трассировки.

![Обратная трассировка для основных запросов к БД](https://dt-cdn.net/images/backtrace-for-top-database-statements-1699-76085f0963.png)

### Распределение времени отклика

Чтобы понять распределение времени отклика команды, выберите **Outliers** в меню **Analyze**. На странице **Response time distribution** показано количество запросов, попавших в различные диапазоны времени отклика. Выберите любой столбец на графике для анализа конкретных запросов, попавших в данный диапазон времени отклика.

![Распределение времени отклика запросов сервиса](https://dt-cdn.net/images/response-time-distribution-database-statements-1800-8ff28dc6c7.png)

### Просмотр отдельных выполнений

Чтобы просмотреть отдельные выполнения каждой команды, выберите **More** (**...**) > **Distributed traces** > **Trace**.

В столбце **Fetches** показано количество обращений к базе данных на одно выполнение SQL. Поскольку большинство драйверов баз данных используют постраничную выдачу результатов, обращения обычно выполняются наборами по **Rows returned** на одно обращение. Поэтому, если набор результатов содержит 200 строк и размер выборки составляет 50 строк на обращение, потребуется 4 обращения для получения всех данных. Большее количество обращений означает более медленное время отклика. Однако слишком большой размер выборки также может привести к неэффективному использованию памяти, и обычно вы все равно не будете считывать весь набор данных целиком.

Выберите любое выполнение в списке для доступа к его распределенной трассировке.

![Трассировка PurePath выполнения](https://dt-cdn.net/images/purepath-database-statements-1597-d6db847c88.png)

## Пример: анализ SQL для потока сервиса

Хотя анализ SQL доступен в нескольких местах рабочего процесса анализа сервисов, он был добавлен в качестве основного представления анализа для баз данных в представлении [**Service flow**](/docs/observe/application-observability/services-classic/service-flow "Find out how Dynatrace can help you trace the sequence of service calls that are triggered by each service request in your environment."). Следующий пример показывает поток сервиса, который уже отфильтрован для фокусировки на определенной цепочке вызовов. Видно, что каждый раз, когда сервис **Customer Frontend** вызывает **JourneyService**, **58%** **вклада во время отклика** может быть отнесено к SQL-сервису **EasyTravel**.

![От потока сервиса к запросу к БД](https://dt-cdn.net/images/from-service-flow-to-database-statement-1748-bff9188259.png)

Действие по умолчанию изменено и теперь отображает **View database statements**.

Вы можете выполнить несколько типов анализа:

1. Чтобы увидеть запросы к базам данных, которые были выполнены выбранным потоком за период анализа, выберите **View database statements**. Это может быть чрезвычайно полезно, поскольку показывает, почему база данных вносит определенный вклад во время отклика.
2. Чтобы определить, какие запросы имеют наибольшее общее время отклика, отсортируйте список **Top dimensions** по **Count/min**.

![Пример многомерного анализа SQL-сервиса](https://dt-cdn.net/images/example-of-sql-service-multidimensional-analysis-3574-7c039d0068.png)

3. Выберите **More** (**...**) и затем:

* Выберите **Outliers** для просмотра распределения времени отклика.
* Выберите **Distributed traces** для понимания эволюции каждого отдельного SQL-запроса с течением времени, а также среднего значения **Rows returned**.
* Выберите **Service backtrace**, чтобы увидеть, какие действия пользователей привели к самым медленным выполнениям запроса.
