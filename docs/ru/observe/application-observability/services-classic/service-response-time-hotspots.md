# Горячие точки времени отклика сервиса

* Classic
* Практическое руководство
* Чтение: 5 мин
* Опубликовано 19 июля 2017

При включённом глубоком мониторинге процессов Dynatrace анализирует время отклика каждого сервиса, работающего в каждом процессе. Это применимо к Java, .NET, Node.js, PHP, веб-серверу Apache, IIS, NGINX и другим технологиям.

Горячие точки времени отклика показывают, какие действия потребляют больше всего времени для конкретного сервиса. Каждая страница **Service** предоставляет обзор связанных горячих точек времени отклика в разделе **Current hotspot** и включает:

* Основные находки, такие как запросы с медленным временем отклика, высокой частотой ошибок или высоким потреблением CPU.
* Запросы с высоким потреблением ресурсов запроса. Процент показывает долю запроса в общем времени отклика сервиса. В примере ниже `85%` времени отклика `JourneyService` тратится на выполнение запроса `findJourneys`.

![Hotspots 0](https://dt-cdn.net/images/hotspots0-1309-9b8dfdbea1.png)

Чтобы просмотреть анализ горячих точек

1. Перейдите в ![Services Classic](https://dt-cdn.net/images/services-classic-f58502bd22.svg "Services Classic") **Services Classic**.
2. Выберите сервис, который хотите проанализировать.
3. Выполните одно из следующих действий:

   * Чтобы проанализировать основную горячую точку, выберите находку в разделе **Current hotspots** для отображения деталей сервиса с фильтрацией по проблемному запросу.
   * Чтобы проанализировать все горячие точки времени отклика сервиса, на странице обзора сервиса выберите один из вариантов: **View requests**, **View dynamic requests** или **View resource requests**. Dynatrace отобразит общие детали сервиса.
4. Выберите **View response time hotspots** для перехода на страницу **Response time analysis**.

![Hotspots 1](https://dt-cdn.net/images/hotspots1-1309-752c2e2bd8.png)

На странице **Response time analysis** вы видите среднее время отклика, наблюдаемое за анализируемый период. В левой части инфографики, в разделе **Distribution**, можно увидеть, сколько времени приходится на вызовы других сервисов, вызовы баз данных и выполнение кода. В правой части, в разделе **Top findings**, перечислены самые значительные горячие точки, выявленные Dynatrace. Вы можете выбрать любую из этих записей для просмотра дополнительных деталей.

## Вызовы других сервисов

Выберите **Interaction with services and queues** для отображения деталей о том, как эти вызовы влияют на общее время отклика.

![Hotspots 2](https://dt-cdn.net/images/hotspots2-1889-e6dec0cd21.png)

В левой части вы видите, как часто анализируемый сервис вызывает другие сервисы и в какой степени эти вызовы влияют на время отклика. В правой части эта информация разбита более подробно. В примере выше видно, что большую часть времени вносит сервис `Easytravel Customer Frontend`. Выберите название сервиса для просмотра следующего уровня иерархии. Когда вы достигнете самого нижнего уровня (уровня запроса), вы увидите полные детали времени отклика для выбранного запроса.

![Hotspots 3](https://dt-cdn.net/images/hotspots3-1887-d7ea5fdd55.png)

В примере выше мы видим, что `EasytravelWebserver:8079` вызывает URL `/orange.jsf` на `easytravel Customer Frontend` только в 7,46% от общего числа инициируемых запросов — ровно один раз каждый. Мы видим, что время отклика `/orange.jsf` составляет в среднем `255 мс`. Однако, поскольку этот вызов выполняется редко, он вносит лишь `19 мс` в среднее время отклика `EasytravelWebserver:8079`.

Если вы выберете кнопку со стрелкой на левой границе раздела **Interaction with services and queues**, вы можете снова подняться вверх по иерархии.

## Запросы к базам данных

Выберите раздел **Database usage** инфографики для доступа к деталям запросов к базам данных анализируемого сервиса и для просмотра того, как они влияют на время отклика. Верхняя часть предоставляет ту же навигацию и информацию, что и раздел **Interaction with services and queues**. Ниже вы можете просмотреть все запросы к базам данных.

![Hotspots 4](https://dt-cdn.net/images/hotspots4-1252-be6d5ddef6.png)

Выбрав выпадающий список в верхнем правом углу, вы можете отсортировать SQL-выражения по **среднему времени выполнения**, **проценту вызовов**, **количеству вызовов** и **вкладу**. По умолчанию выражения, вносящие наибольший вклад во время отклика, располагаются в верхней части списка. Выберите SQL-выражение для просмотра значений этих четырёх атрибутов.

![Hotspots 5](https://dt-cdn.net/images/hotspots5-1228-0f1d8e37b5.png)

## Видимость на уровне кода

Выберите **Service execution** для просмотра того, сколько времени сервис потратил на выполнение собственного кода.

![Service execution time](https://dt-cdn.net/images/service-execution-time-1445-780d85d724.png)

Длина полос представляет количество времени, проведённого в каждой конкретной области. Тёмно-синий цвет обозначает время CPU. Выберите любое имя строки, отображаемое как ссылка, чтобы просмотреть соответствующие горячие точки на уровне методов.

![Code-level visibility](https://dt-cdn.net/images/hotspots-1446-6c931f44ae.png)

Нажмите кнопку **View method hotspots** и проверьте все классы и методы, которые были выполнены для работы сервиса, в разделе **Call tree**. Столбец **Stacktrace samples** показывает, сколько раз класс/метод был выполнен для сервиса за выбранный период. Столбец **Contribution** показывает долю потребления, которую класс/метод вносит в общее выполнение. Это позволяет определить, какой класс и/или метод потребляет больше всего времени выполнения, и впоследствии оптимизировать код.

### Загрузка кода

Предварительные требования

Убедитесь, что у вас есть [разрешение](../../../manage/identity-access-management/permission-management/role-based-permissions.md "Role-based permissions") **View sensitive request data** в вашем окружении.

Ограничения

Загрузка кода поддерживается для Java, .NET и Node.js.

Чтобы загрузить код для выполнения,

1. Перейдите в **Actions** и выберите  > **Source code**.
2. Выберите процесс, из которого хотите загрузить код.
3. Нажмите **Download** для интересующего вас выполнения.
4. Откройте исходный код.

   Node.js

   Java

   .NET

   Исходный код непосредственно виден в Dynatrace.

   Для преобразования кода в исходный код используйте внешний декомпилятор, например [Bytecode Viewer](https://bytecodeviewer.com).

   Выберите декомпилятор, наиболее подходящий для конкретной версии Java, с которой вы работаете.

   Для преобразования кода в исходный код используйте внешний декомпилятор, например [dotPeak](https://www.jetbrains.com/decompiler/).

   Выберите декомпилятор, наиболее подходящий для конкретной версии .NET, с которой вы работаете.

## Автоматические горячие точки

Раздел **Top findings** инфографики **Response time** перечисляет горячие точки, которые являются ключевыми вкладчиками в анализируемый запрос.

Если запрос медленный, вы увидите основную причину в разделе **Top findings**. Выберите находку для просмотра дополнительных деталей. В этом примере вы можете видеть, что 226 мс из 639 мс были потрачены на вызов `PaymentLogic.DoPay`!

![Hotspots 7](https://dt-cdn.net/images/hotspots7-1254-e581fe35b2.png)

## Распределение времени отклика

Выберите круговую часть **Response time** инфографики, чтобы открыть диаграмму, показывающую распределение времени отклика всех анализируемых запросов. Это показывает, имеет ли ваш сервис диапазон значений времени отклика или работает равномерно.

![Hotspots 8](https://dt-cdn.net/images/hotspots8-1248-286099cfc1.png)

Как правило, сервисы веб-запросов имеют широкий диапазон значений времени отклика. Анализ отдельного запроса часто показывает более однородную картину.
