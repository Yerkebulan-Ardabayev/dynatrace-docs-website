Kubernetes является неотъемлемой частью инфраструктуры многих организаций. При большом количестве узлов и подов в типичной инфраструктуре Kubernetes важно иметь масштабируемое решение для оповещений.

## Реальный сценарий

Одним из основных применений оповещений Kubernetes является немедленное обнаружение аномалий в вашей инфраструктуре.

Например, рассмотрим ситуацию, когда поды не могли запуститься в производственной среде, тогда как в промежуточной среде всё работало без проблем. Несмотря на то что прямого влияния на производство не было (трафик не направлялся к неготовым подам), Dynatrace выдал оповещение о развёртывании, которое не могло продвинуться вперёд (зависшее развёртывание), что означало невозможность выкатить последнюю версию.

Используя комбинацию событий Kubernetes и логов приложений в Dynatrace, было установлено, что первопричиной являлась проблема с DNS. Путём корректировки сетевых политик Kubernetes проблема была устранена.

## Преимущества

Встроенные (OOTB) оповещения Kubernetes легко настраиваются в рамках глобальных настроек обнаружения аномалий. С помощью этой функции вы можете добиться следующего:

* **Быстрая настройка**: OOTB-оповещения, в том числе для Kubernetes, поставляются преднастроенными, что обычно позволяет выполнить настройку мониторинга менее чем за 5 минут.
* **Оптимизированный мониторинг**: OOTB-оповещения автоматизируют контроль множества метрик, централизуя процесс мониторинга и снижая необходимость в частых ручных проверках.
* **Адаптивная реакция**: Функция корректирует пороги оповещений на основе актуальных нагрузок кластера, обеспечивая релевантный мониторинг и минимизируя возможные ошибки человека.
* **Прямая навигация**: Переходите к настройкам, связанным с вашим пространством имён, непосредственно из Dynatrace и адаптируйте всё под свои нужды без необходимости во внешних конфигурациях.
* **Конфигурации по умолчанию**: Настраивайте конфигурации оповещений по умолчанию для всех активных и будущих кластеров Kubernetes, пространств имён и рабочих нагрузок, обеспечивая согласованный мониторинг по мере роста инфраструктуры.
* **Детальная кастомизация**: Настраивайте параметры оповещений на различных уровнях, позволяя по-разному обрабатывать оповещения для производственных и разрабатываемых кластеров и корректировать оповещения узлов внутри каждого из них.
* **Автоматизация с помощью Dynatrace API**: Используйте Dynatrace API для автоматизации конфигураций, гарантируя, что ваша система оповещений плавно эволюционирует вместе с изменениями инфраструктуры. Кроме того, с помощью Dynatrace API вы можете применять подход [Configuration as Code](/docs/deliver/configuration-as-code "Используйте конфигурацию Dynatrace как код с помощью Monaco или Terraform.") для настройки оповещений, интегрируя их в рабочий процесс GitOps.

## Начало работы

Оповещения о типичных проблемах Kubernetes можно настроить на трёх уровнях:

* **Окружение**: Настройки применяются ко всем кластерам, узлам, пространствам имён или рабочим нагрузкам в среде Kubernetes.
* **Кластер**: Настройки, специфичные для отдельных кластеров.
* **Пространство имён**: Настройки, специфичные для отдельных пространств имён.

Чтобы настроить эти параметры

1. Перейдите в **Settings** > **Anomaly detection**.
2. В разделе **Kubernetes** выберите любой вариант.

Полные сведения об этой функции см. в [Оповещения о типичных проблемах Kubernetes/OpenShift](/docs/observe/infrastructure-observability/container-platform-monitoring/kubernetes-monitoring/alert-on-kubernetes-issues "Настройка оповещений на уровне кластера Kubernetes/OpenShift, узла, пространства имён или рабочей нагрузки.").
