# Результаты отчётности HTTP-мониторов


* Последняя версия Dynatrace
* Справочник
* 15 минут чтения
* Обновлено 03 ноября 2025 г.

Страница результатов отчётности HTTP-мониторов предоставляет обзорную панель результатов выполнения монитора, визуализации результатов и свойств монитора. Панель сведений отображает информацию и графики с первого взгляда, а также ссылки и фильтры для быстрого перехода к деталям проблем и редактирования настроек монитора.

Для доступа к странице результатов отчётности HTTP-мониторов:

1. Перейдите в ![Synthetic](https://dt-cdn.net/images/synthetic-new-256-1ddb35ac79.png "Synthetic") **Synthetic**.
2. Необязательно Выберите **HTTP** слева для фильтрации списка по HTTP-мониторам.
3. Из списка HTTP-мониторов выберите монитор, который вы хотите изучить.
4. Выберите **View details** на панели предварительного просмотра для доступа к странице сведений.

## Визуализации метрик

Верхняя панель показывает общую [доступность](#availability) и [производительность](#performance) монитора в виде инфографики за выбранный период времени.

Используйте панель фильтров в верхней части страницы для фильтрации всех сведений об HTTP по одной или нескольким локациям.

В правом верхнем углу страницы вы можете получить доступ к [**Analyze executions**](#analyze-executions), [**On-demand execution**](on-demand-executions.md "Узнайте о том, как выполнять выполнения по требованию.") и другим настройкам HTTP-монитора:

* **Edit**
* для дополнительных параметров (**Disable** и **Delete**)

### Анализ выполнений

Лицензия DPS

Если вы не являетесь пользователем DPS, см. раздел [Результаты отчётности HTTP-мониторов](../../synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic.md#analyze-last-execution "Узнайте о странице сведений Synthetic для HTTP-мониторов.").

При доступе к функциональности **Analyze executions** вам отображается раздел **Select execution** с диаграммой рассеяния, в которой можно перемещать окно просмотра к следующим или предыдущим 50 выполнениям. Для удобства чтения вся диаграмма ограничена 100 точками. Содержимое диаграммы рассеяния может быть разделено по результатам (успех/неудача), локациям или всем этим измерениям. Поэтому вы можете анализировать ваши выполнения тремя способами:

* **Split by status** — вид по умолчанию показывает **Healthy** и **Failed** выполнения отдельно. Под графиком находится легенда, в которой можно выбрать любой статус для фокусировки на нём. Выберите значок для дополнительных параметров.
* **Split by location** — показывает выполнения из разных локаций. Под графиком находится легенда, в которой можно выбрать конкретную локацию для фокусировки на ней. Выберите значок для дополнительных параметров.
* **Split by both** — объединяет виды **Split by status** и **Split by location**.

  Обратите внимание, что если для локации имеется более одного статуса выполнения, легенда под графиком отображает локацию более одного раза. Например, `Las Vegas (Healthy)` и `Las Vegas (Failed)`.

Вы можете не только просматривать сведения о последнем неудачном или успешном выполнении, но и сведения обо всех выполнениях за период хранения.

![Вид анализа выполнений](https://dt-cdn.net/images/analyze-executions-3022-c395fceee3.png)

  2. Выберите второе выполнение для сравнения с первым. Наведите курсор на его данные во всплывающей подсказке и выберите **Compare execution**.

## Доступность

Раздел доступности отображает доступность монитора за выбранный период времени с подробностями о времени простоя и затронутых локациях, если таковые имеются. Панель содержит информацию обо всех проблемах, связанных с данным монитором. Инфографика продолжительности простоя и затронутых локаций отображается, если монитор недоступен в одной или нескольких локациях.

Кроме того, на панели отображаются причины отклонённых тестов. Полоса над временным диапазоном указывает на время простоя. Наведите курсор на полосу, чтобы узнать, почему тесты были отклонены. Возможные причины:

* **Maintenance window** — выполнение монитора было намеренно [деактивировано](../../../../analyze-explore-automate/notifications-and-alerting/maintenance-windows/define-maintenance-window.md#disable-synthetic "Создание окон обслуживания и определение их области действия.") из-за технического обслуживания. Подробнее см. в окне обслуживания.

  Как просмотреть окно обслуживания и что происходит, если выполнение монитора включено/выключено

  Для просмотра окна обслуживания:

  1. Выберите полосу, чтобы закрепить всплывающую подсказку **Maintenance window**.
  2. Наведите курсор на всплывающую подсказку **Maintenance window**.
  3. Выберите **View maintenance window settings**.
  4. Выберите **Settings Classic**.

  + Когда параметр **Disable synthetic monitor execution** включён в окне обслуживания, тесты подавляются в период обслуживания, поэтому результаты тестов за этот период отсутствуют.
  + Когда параметр **Disable synthetic monitor execution** включён в окне обслуживания, запланированные автоматические тесты подавляются в период обслуживания, но [тесты по требованию](../../synthetic-monitoring/general-information/on-demand-executions.md "Выполнение синтетических мониторов по требованию из публичных или частных локаций") не подавляются.
  + Когда параметр **Disable synthetic monitor execution** выключен в окне обслуживания, тесты выполняются в период обслуживания, но из-за технического обслуживания результаты тестов могут отличаться от наблюдаемых в обычных условиях.

* **No result** — выполнение началось, но результаты не были доставлены. Обратитесь в службу поддержки Dynatrace.
* **No capability** — вы пытаетесь выполнить мониторы, которые не поддерживаются локацией (например, NAM ICMP-мониторы из локации с неактивным ICMP).
* **Discarded, not executed** — вероятная причина — перегруженная частная локация. Проверьте метрики ёмкости. Если с метриками нет проблем, обратитесь в службу поддержки Dynatrace.
* **Http monitor local outage** — простой произошёл в одной или нескольких локациях, из которых монитор выполнял тесты.
* **Http monitor global outage** — простой произошёл во всех локациях, из которых монитор выполнял тесты.

  Локальный и глобальный простой — просмотр сведений о проблеме

  Для просмотра сведений о проблеме, вызывающей локальный или глобальный простой:

  1. Выберите полосу со значком, чтобы закрепить всплывающую подсказку **Http monitor local outage** или **Http monitor global outage**.
  2. Наведите курсор на всплывающую подсказку.
  3. Выберите **View problem details**.
  4. Выберите один из доступных вариантов:

     + **Problems Classic**
     + **Problems** в приложении проблем
     + **Problems** в текущем приложении для просмотра сведений о проблеме.

Монитор может быть недоступен в одной или всех локациях, даже если не заданы [пороговые значения простоя (глобальные или локальные)](create-and-configure-an-http-monitor.md#outage-and-performance "Узнайте, как настроить и отредактировать HTTP-монитор для проверки производительности и доступности вашего сайта."). Продолжительность простоя — это сумма всех периодов простоя в выбранном временном диапазоне без учёта перекрывающихся периодов простоя.

Карточка **Availability** показывает общую доступность во всех локациях монитора с аннотациями для глобальных/локальных простоев и глобальных/локальных отсутствующих данных (например, когда монитор неактивен).

### Расчёт доступности

Метрики доступности рассчитываются путём деления количества успешных выполнений («up») на общее количество выполнений. Таким образом, для расчёта доступности:

1. Определите количество успешных выполнений за период времени.

   Это общее количество выполнений за период времени минус количество неудачных («down») выполнений за период времени.
2. Определите доступность.

   Разделите количество успешных выполнений на общее количество выполнений, затем умножьте на 100, чтобы получить процент.

**Пример**

Предположим, у нас 5 неудачных выполнений за 35-минутный период времени.

* Всего выполнений = `35`
* Неудачных выполнений = `5`
* Успешных выполнений = `35-5 = 30`
* Доступность = `(30/35) \* 100 = 0.8571 \* 100` = `85,71%`

#### Исключение выполнений периода обслуживания из расчётов метрик

Вы можете настроить [окно обслуживания](../../../../analyze-explore-automate/notifications-and-alerting/maintenance-windows.md "Узнайте, когда использовать окно обслуживания. Ознакомьтесь с поддерживаемыми типами окон обслуживания.") так, чтобы [выполнения тестов не подавлялись](../../../../analyze-explore-automate/notifications-and-alerting/maintenance-windows/define-maintenance-window.md#disable-synthetic "Создание окон обслуживания и определение их области действия.") в периоды обслуживания. Тогда, если в период обслуживания возникнут простои, неудачные выполнения будут включены в расчёты метрик.

В приведённом ниже примере доступность составляет менее 100%, поскольку неудачные выполнения включены в расчёт метрик по умолчанию.

```
timeseries avg(dt.synthetic.http.availability), by:{dt.entity.http_check, dt.maintenance_window_ids, interpolated}


| filter dt.entity.http_check == "HTTP_CHECK-2F280898D4FCB1A8"
```

Запустить в Playground

![Доступность с включёнными выполнениями периода обслуживания](https://dt-cdn.net/images/a40863a4-4a5d-4380-8ef8-7d8353639756-1179-34988a102e.png)

Для исключения выполнений периода обслуживания из расчёта метрик:

1. Добавьте измерение `dt.maintenance_window_ids` в запрос.
2. Отфильтруйте выполнения, не охваченные окном обслуживания, с помощью `isNull(dt.maintenance_window_ids)`.

В приведённом ниже примере у нас есть неудачные выполнения, обнаруженные в период обслуживания, но они не были включены в расчёт метрики, поскольку мы использовали измерение `dt.maintenance_window_ids` и условие фильтра `isNull(dt.maintenance_window_ids)`.

```
timeseries av = avg(dt.synthetic.http.availability), by: {dt.entity.http_check, dt.maintenance_window_ids}


| filter dt.entity.http_check == "HTTP_CHECK-2F280898D4FCB1A8"


| filter isNull(dt.maintenance_window_ids)


| fields avgAV=arrayAvg(av)
```

Запустить в Playground

![Пример использования dt.maintenance_window_ids](https://dt-cdn.net/images/dem-3738-1179-248a06b401.png)

#### Интерполяция

Описанный выше подход требует, чтобы все выполнения происходили с одинаковой частотой. Однако в реальной среде это не всегда так, поскольку частота выполнения монитора может изменяться или могут запускаться дополнительные выполнения по требованию.

Для повышения точности расчёта вводится механизм интерполяции: общее время мониторинга делится на точки данных с точностью до минуты. Все точки данных считаются выполнениями, хотя они не обязательно совпадают с фактическими выполнениями.

На экране ниже синие точки данных совпадают с фактическими выполнениями, а белые точки — нет.

![Интерполированные точки данных up/down](https://dt-cdn.net/images/screenshot-2025-06-05-142009-1552-ea3432b75b.png)

Представим, что монитор настроен на выполнение тестов каждые пять минут. Таким образом, каждая пятая точка данных (синяя) совпадает с фактическим выполнением. Все точки данных, следующие за первым неудачным выполнением и предшествующие первому успешному выполнению, считаются «неудачными». Таким образом, при расчёте метрик доступности количество неудачных выполнений рассчитывается следующим образом:

Неудачные выполнения = первое фактическое неудачное выполнение + все последующие «неудачные» точки данных.

Предопределённые запросы являются примерами работы с деталями выполнения синтетического монитора с использованием нашего языка запросов Dynatrace. Вы можете использовать эти запросы как шаблоны и адаптировать их под свои нужды.

* Запрос 3 — возвращает полные сведения о выполнении HTTP-монитора, включая весь запрос и ответ.

## Производительность

Раздел производительности отображает производительность HTTP-монитора — время ответа для суммы всех запросов по всем локациям за выбранный период времени. Мы не выполняем суммирование для выполнений из разных локаций.

Дополнительные метрики фиксируются для каждого запроса и отображаются в [карточке запросов](#http-requests).

Карточка **Performance** показывает линии тренда для времени ответа за время выполнения всего монитора с заштрихованной областью, представляющей разницу между двумя значениями в любой момент времени. Обратите внимание, что если ваш монитор запускается из одной локации, линии минимума и максимума совпадают.

Необязательно Вы можете просматривать линии тренда производительности HTTP-монитора.

### Отображение проблем в разделе производительности

* Красная полоса над графиком указывает на текущую проблему.
* Серая полоса над графиком указывает на закрытую проблему.

Проблемы доступности и другие проблемы не отмечаются в этом разделе.

Если монитор нарушает [пороговое значение производительности](create-and-configure-an-http-monitor.md#outage-and-performance "Узнайте, как настроить и отредактировать HTTP-монитор для проверки производительности и доступности вашего сайта."), будь то для суммы всех запросов или отдельного запроса, над графиком производительности на протяжении всей проблемы появляется сплошная красная линия. Выберите сплошную красную полосу для отображения ссылки на страницу обзора проблемы.

Просмотр сведений о проблеме

1. Наведите курсор на полосу проблемы, чтобы увидеть сведения о текущей/закрытой проблеме в виде всплывающей подсказки.
2. Выберите полосу проблемы и наведите курсор на всплывающую подсказку для отображения дополнительных параметров.
3. Выберите **View problem details** для открытия сведений о проблеме в Problems ![Приложение Problems — новое](https://dt-cdn.net/images/dynatrace-davis-new-256-340162f8c6.webp "Приложение Problems — новое").

## Проблемы

Карточка **Problems** показывает проблемы производительности (нарушение порогового значения) и доступности (локальный или глобальный простой) при включении соответствующих пороговых значений в [настройках монитора](create-and-configure-an-http-monitor.md#outage-and-performance "Узнайте, как настроить и отредактировать HTTP-монитор для проверки производительности и доступности вашего сайта.").

Информацию об определении правил производительности и пороговых значений простоя доступности см. в разделе [Создание и настройка HTTP-монитора](create-and-configure-an-http-monitor.md "Узнайте, как настроить и отредактировать HTTP-монитор для проверки производительности и доступности вашего сайта."). Информацию о рабочем процессе и концепциях оповещений, включая настройку профилей уведомлений и шаблонов, см. в разделе [Обзор оповещений Synthetic](../../synthetic-monitoring/analysis-and-alerting/synthetic-alerting-overview.md "Узнайте о концепциях и рабочем процессе оповещений Synthetic.").

Существует три основных типа проблем для HTTP-мониторов:

* Глобальный простой (доступность)
* Локальный простой (доступность)
* Нарушение порогового значения производительности (проблема производительности для суммы всех запросов или отдельных запросов)

Проблемы производительности могут включать нарушения пороговых значений как на уровне монитора, так и на уровне запросов.

В разделе проблем перечислены все проблемы за выбранный период времени — как открытые, так и закрытые. Кроме того, вы можете:

* Просматривать сведения о проблеме в виде таблицы с датой начала и продолжительностью.
* Разворачивать каждую проблему для просмотра затронутых объектов и профилей оповещений.

Проблемы вместе с составляющими их [событиями](#events) и соответствующими [кодами статуса](#http-requests) дают полную картину количества и масштаба проблем вашего монитора.

## HTTP-запросы

HTTP-монитор может состоять из одного или нескольких HTTP-запросов. Карточка **HTTP requests** предоставляет обзор всех выполненных запросов, их порядок, имя, URL запроса и использованный HTTP-метод. Для каждого запроса карточка **HTTP requests** разбивает производительность (**Response time**) по следующим метрикам (подробнее в разделе [Метрики HTTP-монитора](../../synthetic-monitoring/http-monitors-classic/http-monitor-metrics-classic.md "Узнайте о метриках производительности, собираемых для HTTP-мониторов.")):

* **DNS lookup time**
* **TCP connect time**
* **TLS handshake time**
* **Waiting**
* **Download**

**Response time** — это сумма этих различных метрик.

Разверните запрос из списка для просмотра всех метрик производительности на одном графике. Выберите метрику в легенде для удаления/добавления её на график производительности. Выберите **Edit request** для перехода к настройкам монитора из этой карточки.

Вкладка **Status codes** показывает временную шкалу возвращённых HTTP-кодов статуса для конкретного запроса. (Карточка [HTTP status codes](#status-codes) показывает возвращённые HTTP-коды статуса для вашего HTTP-монитора в целом.)

Если запрос нарушает [специфический для события пороговый значение производительности](create-and-configure-an-http-monitor.md#outage-and-performance "Узнайте, как настроить и отредактировать HTTP-монитор для проверки производительности и доступности вашего сайта."), он выделяется красным. Разверните запрос для просмотра временных параметров производительности и нарушенного порогового значения. Над стековым графиком на протяжении всей проблемы появляется сплошная красная линия; пороговое значение запроса отображается в виде пунктирной красной линии. Выберите сплошную красную полосу для отображения ссылки на страницу обзора проблемы.

## Размер ответа

Карточка **Response size** показывает линии тренда для минимального и максимального размера ответа для суммы всех запросов с заштрихованной областью, представляющей разницу между двумя значениями в любой момент времени. Линии тренда могут совпадать, например, когда ваш монитор запускается из одной локации. Однако размер ответа может варьироваться, например, когда разные ответы отправляются в зависимости от локации.

Для отслеживания таких различий вы можете использовать **Show response size breakdown per location**. Обратите внимание, что если вы фильтруете всю страницу сведений по каким-либо локациям, переключатель становится недоступным.

## События

Карточка **Events** показывает все события, составляющие проблемы. В списке и на временной шкале отображаются события как для активных, так и для разрешённых проблем.

Наведите курсор на временной интервал на временной шкале событий, чтобы увидеть тип и количество событий, сгенерированных в этом интервале. Выберите временной интервал для просмотра всплывающей подсказки с событиями, произошедшими в нём.

Выберите тип события, например **HTTP monitor location slowdown**, для просмотра списка событий. Одно событие замедления всегда создаётся для каждой локации, в которой ваш монитор нарушает пороговые значения производительности на уровне запроса или монитора. Выберите отдельное событие для просмотра сведений.

[Метод `api.fail()`](../../synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic.md#end "Узнайте, как применять скрипты до и после выполнения запросов.") можно использовать для определения пользовательского **сообщения об ошибке**, которое отображается в карточке Events в случае неудачи. Пользовательские сообщения журнала также отображаются в атрибуте `customLogs` в [сведениях о выполнении HTTP-монитора](#analyze-last-execution).

## Изменения

Карточка **Changes** показывает журнал истории, в котором отображается:

* Когда монитор был создан и кем.
* Кто изменил свойства монитора, когда было внесено изменение и какие свойства были изменены в выбранном периоде времени.

## Ошибки

Карточка **Errors** отображает ошибки, обнаруженные в выбранном периоде времени.

На карточке вы можете:

* Просматривать сведения об ошибке в виде таблицы с шагом и количеством вхождений.
* Выбрать **Analyze errors**.

## Фронтенды

Карточка **Frontends** отображает фронтенды, которым назначен монитор.

## Локации

Карточка **Locations** показывает статус последнего выполнения монитора для каждой локации. Статус отображается цветом:

* Зелёный — локация была доступна во время последнего выполнения монитора.
* Красный — во время последнего выполнения монитора был обнаружен простой.
* Серый — данные о выполнениях для этой локации отсутствуют.

## Свойства и теги

Эта информационная карточка показывает количество шагов, количество и список локаций, частоту выполнения монитора и применённые теги.

Выберите **Edit** для редактирования или применения дополнительных тегов.

## Сервисы

Карточка **Services** отображает любые мониторируемые сервисы, автоматически связанные с HTTP-монитором. Эта карточка появляется, когда мониторируемая конечная точка HTTP размещена на хосте, мониторируемом OneAgent, и URL конечной точки обрабатывается одним из обнаруженных сервисов. Связь между HTTP-монитором и сервисом устанавливается на основе данных, предоставленных OneAgent. Поэтому перед тем, как сервис можно будет связать с монитором, необходимо завершить хотя бы одно выполнение монитора.

Выберите ссылку на отображаемый сервис для просмотра страницы обзора сервиса, отфильтрованной по HTTP-монитору.

### Просмотр трассировок

В HTTP-мониторах со связанными сервисами вы можете видеть:

* Полный список трассировок для всего монитора за определённый период времени
* Полный список трассировок для конкретного сервиса за определённый период времени
* Трассировки для конкретных синтетических выполнений

#### Трассировки для всего монитора

Для просмотра полного списка трассировок для всего монитора выберите **View traces** в любом из следующих мест:

* В верхней части страницы сведений о мониторе
* В заголовке карточки **Services**

#### Трассировки для конкретного сервиса

Для просмотра полного списка трассировок для конкретного сервиса:

1. В карточке **Services** найдите сервис, который вы хотите изучить.
2. Выберите значок в столбце **View traces**.

![Трассировки для конкретного сервиса](https://dt-cdn.net/images/screenshot-2025-05-28-104143-910-bad89d821e.png)

#### Трассировки для конкретного выполнения

Для просмотра бэкенд-трассировок для конкретного выполнения:

1. Выберите **Analyze executions** в верхней части страницы сведений о мониторе.
2. На графике выберите выполнение, чтобы закрепить его всплывающую подсказку.
3. Наведите курсор на всплывающую подсказку и выберите **View traces**.

![Трассировки для конкретного выполнения](https://dt-cdn.net/images/screenshot-2025-05-27-165215-910-851ffdee13.png)
