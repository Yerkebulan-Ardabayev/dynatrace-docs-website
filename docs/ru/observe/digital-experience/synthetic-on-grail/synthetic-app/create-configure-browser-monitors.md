# Создание и настройка браузерного монитора

# Создание и настройка браузерного монитора

* Latest Dynatrace
* Практическое руководство
* Обновлено 16 фев. 2026

Вы можете создавать синтетические браузерные мониторы для проверки доступности ваших веб-сайтов. Браузерные мониторы можно запускать из наших [глобальных публичных](/docs/observe/digital-experience/synthetic-monitoring/general-information/public-synthetic-locations "Узнайте обо всех доступных публичных локациях синтетического мониторинга.") или [приватных](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/private-locations "Узнайте, как управлять приватными локациями в приложении Synthetic.") локаций синтетического мониторинга.

Перейдите в ![Synthetic Classic](https://dt-cdn.net/images/synthetic-512-83ec796e54.png "Synthetic Classic") **Synthetic** > **New monitor** > **Browser**. Параметры конфигурации разделены на следующие разделы:

* [Общие настройки](#general)
* [Шаги](#steps)
* [Частота и локации](#frequency-and-locations)
* [Сбои и производительность](#outage-and-performance)
* [Расширенные настройки](#advanced-setup)
* [Сводка](#summary)

Пройдите все разделы по порядку и выполните все необходимые настройки в каждом из них.

## Общие настройки

Для начала настройте общие параметры.

1. **Назовите этот монитор** -- введите название (до 500 символов) для синтетического монитора. Это название должно в целом описывать все запросы в данном браузерном мониторе.

2. Нажмите **Add tag**, чтобы вручную применить теги к монитору.
3. Нажмите **Assign frontends**, чтобы привязать монитор к фронтенд-приложению. Вы можете привязать монитор к нескольким фронтенд-приложениям, а фронтенд-приложение может иметь несколько привязанных мониторов. Это позволяет отслеживать доступность и производительность фронтенд-приложения. Обнаруженные проблемы автоматически ассоциируются с вашим фронтенд-приложением. Если выполнение монитора завершается неудачей из-за проблем с фронтенд-приложением, отчётный показатель доступности фронтенд-приложения уменьшается в заданном временном интервале.

   Браузерный монитор может быть автоматически привязан к фронтенд-приложению, если URL, проверяемый монитором, совпадает с шаблоном, определённым в [![Experience Vitals](https://dt-cdn.net/images/experience-vitals-256-9999590b55.png "Experience Vitals") **Experience Vitals**](/docs/observe/digital-experience/new-rum-experience/experience-vitals "Приложение Experience Vitals предоставляет точку входа для мониторинга веб- и мобильных фронтендов.").

Расширенные настройки

4. Необязательно. Укажите **Device profile**. Вы можете выбрать из:

   * **Существующих устройств** с предустановленными характеристиками. Вы можете изменить только ориентацию -- горизонтальную или вертикальную.
   * **Пользовательское устройство** -- добавьте новое устройство с пользовательскими характеристиками, такими как ориентация (горизонтальная или вертикальная), высота экрана и ширина экрана, и укажите **Custom name** нового устройства.
5. Необязательно. Укажите **Bandwidth profile**. Вы можете выбрать из:

   * **Существующего профиля** с готовыми к использованию предустановками.
   * **Пользовательского профиля** -- добавьте новый профиль пропускной способности с пользовательскими характеристиками, такими как скорость загрузки, скорость отдачи и задержка, и укажите **Custom name** нового профиля.
6. Необязательно. Укажите **User agent** для настройки HTTP-заголовка user agent, отправляемого браузером во время выполнения монитора. При выборе **Device profile** из **Existing devices** Dynatrace автоматически заполняет **User agent**.

   Стандартный user agent Dynatrace при записи и локальном воспроизведении имеет формат `Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/{version} Safari/537.36 RuxitSynthetic/1.0 v0 t0 cfeatureHash=7efgijmoqtvx caes=1 ccux=1 sia=1 smf=1`, где:

   * `{version}` -- текущая версия браузера, используемая для записи.
   * `v0` и `t0` идентифицируют трафик синтетического мониторинга.
   * `sia=1` указывает на ускоренное внедрение RUM JavaScript (значение может быть `1` или `0`).
   * `smf=1` указывает на мониторинг страниц во фреймах (если параметр **Capture performance metrics for pages loaded in frames** включён в [**Расширенных настройках**](#other); значение `0`, если не включён).
   * `cfeatureHash=<value>` появляется, когда пользовательские настройки RUM JavaScript включены в [**Расширенных настройках**](#custom-rum-js-settings).
   * Стандартная строка user agent для выполнения браузерного монитора из публичных или приватных локаций синтетического мониторинга: `Mozilla/5.0 (X11; Ubuntu; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/{version} Safari/537.36 RuxitSynthetic/1.0 v{id} t{id} ath{id} alt{id} cfeatureHash=7efgijmoqtvx caes=1 ccux=1 sia=1 smf=1`, где:

     + `{id}` представляет длинные идентификаторы, используемые Dynatrace для идентификации выполнения монитора.
     + Остальные параметры описаны выше.

     Обратите внимание, что даже при определении пользовательского user agent Dynatrace всегда автоматически добавляет `RuxitSynthetic/1.0 v{id} t{id} ath{id} alt{id}`, `sia`, `smf` и, при необходимости, `cfeaturehash` и любые пары ключ-значение, начинающиеся с `c`, к user agent для обеспечения идентификации трафика синтетического мониторинга.

## Шаги

Определите шаги. При создании браузерного монитора у вас есть два варианта:

* Настроить монитор для проверки одной веб-страницы на доступность и производительность. Для этого выполните шаги 1-7 и шаг 9 в руководстве ниже (пропустите шаг 8).
* Настроить монитор для выполнения тестового сценария, включающего пользовательский путь -- например, когда пользователь открывает главную страницу в браузере, задаёт критерии поиска, выполняет поиск и выбирает вариант из результатов поиска. Для этого выполните все шаги, перечисленные ниже (1-9).

1. **Name** -- укажите название шага.
2. **URL** -- укажите URL веб-страницы, которую вы хотите проверить на доступность и производительность.
3. Необязательно. Укажите **Context setup**. Вы можете задать:

   * Cookie-файлы, которые будут отправляться с запросами, выполняемыми монитором.
   * Заголовки запросов, которые будут установлены для всех запросов, выполняемых монитором.
   * Запросы с определёнными шаблонами, которые будут блокироваться во время выполнения монитора.

   Хотя вы можете указать cookie-файлы, заголовки и запросы с определёнными шаблонами позже в [Расширенных настройках](#advanced-setup), в некоторых случаях целесообразно сделать это здесь, чтобы обеспечить плавную [запись кликпути](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/clickpath-synthetic-app-browser-monitor "Узнайте, как записать кликпуть браузера для браузерного монитора").
4. Необязательно. Включите **Use HTTP authentication**, если веб-сайт, который вы хотите проверить, защищён HTTP-аутентификацией (не обрабатывается аутентификацией через веб-форму).

   Мы поддерживаем базовую **HTTP authentication** и аутентификацию **Kerberos**. Если вы выбираете **Kerberos** в качестве аутентификации, также укажите имя области (realm) и список разрешённых серверов аутентификации.

   Вы можете выбрать способ хранения учётных данных.

   * **Credential Vault** -- рекомендуемый вариант, обеспечивающий повышенную безопасность. Выберите ранее созданные учётные данные или добавьте новые.

     Вы можете добавить новые учётные данные как в ![Synthetic Classic](https://dt-cdn.net/images/synthetic-512-83ec796e54.png "Synthetic Classic") **Synthetic** (**Add credential**), так и в предыдущем интерфейсе Dynatrace (**Credential vault (classic)**).
   * **Monitor script** -- укажите имя пользователя и пароль.
5. Необязательно. Выберите **Condition to consider this step as done**, чтобы [определить время ожидания между шагами](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/steps-waiting-time-validation-thresholds#steps-waiting-time "Узнайте, как настроить время ожидания между шагами, валидировать контент и устанавливать пороговые значения производительности.").
6. Необязательно. В разделе **Validate content** настройте [правила валидации контента](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/steps-waiting-time-validation-thresholds#validate-content "Узнайте, как настроить время ожидания между шагами, валидировать контент и устанавливать пороговые значения производительности.").
7. Необязательно. Установите [пороговое значение производительности](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/steps-waiting-time-validation-thresholds#performance-thresholds "Узнайте, как настроить время ожидания между шагами, валидировать контент и устанавливать пороговые значения производительности.") для загрузки страницы, указанной в **URL**.
8. Запишите кликпуть или добавьте шаги вручную. Используйте эту функцию, если хотите, чтобы монитор выполнял тестовый сценарий, включающий пользовательский путь -- например, когда пользователь открывает главную страницу в браузере, задаёт критерии поиска, выполняет поиск и выбирает вариант из результатов поиска. Подробнее см. [Запись кликпути браузера](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/clickpath-synthetic-app-browser-monitor "Узнайте, как записать кликпуть браузера для браузерного монитора").

   ![Browser monitor steps](https://dt-cdn.net/images/screenshot-2025-08-29-151144-1417-d6dc411680.png)
9. Нажмите **Continue** для продолжения настройки монитора.

## Частота и локации

Расписание мониторинга определяется двумя факторами -- частотой запуска браузерного монитора и количеством локаций, из которых он выполняется.

Dynatrace предоставляет глобальную сеть [публичных локаций синтетического мониторинга](/docs/observe/digital-experience/synthetic-monitoring/general-information/public-synthetic-locations "Узнайте обо всех доступных публичных локациях синтетического мониторинга."). Вы также можете [создавать приватные локации синтетического мониторинга](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/private-locations "Узнайте, как управлять приватными локациями в приложении Synthetic.") в инфраструктуре вашей собственной сети. Как публичные, так и приватные локации отображаются на этой странице настроек.

Частота и количество локаций определяют количество выполнений монитора в час. Например, запуск монитора из 3 локаций каждые 15 минут приводит к 12 выполнениям в час (4 раза в час из каждой из 3 локаций). Выполнения монитора равномерно распределяются в пределах выбранного интервала. То есть для монитора, запускаемого из 3 локаций каждые 15 минут, выполнения запускаются с интервалом в 5 минут.

Вы можете выбрать частоту каждые **5**, **10**, **15** или **30** минут; или **1**, **2** или **4** часа. Вы также можете настроить монитор для выполнения [**только по запросу**](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/on-demand-executions "Узнайте о выполнении по запросу."). Вы можете выбрать несколько глобальных локаций, из которых будет выполняться ваш браузерный монитор.

Обратите внимание, что все публичные локации синтетического мониторинга используют всемирное координированное время (UTC). Если ваш скрипт монитора требует локального времени или часового пояса, вы можете использовать [метод `api.getContext()`](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/browser-clickpath-steps#retrieve-data "Узнайте о типах шагов, создаваемых при записи кликпути браузера") и системные часы для реализации условной логики.

## Сбои и производительность

Вы можете настроить глобальные и локальные параметры **обработки сбоев** и **пороговые значения производительности** для суммы всех запросов.

* **Создавать проблему и отправлять оповещение, когда этот монитор недоступен во всех настроенных локациях (глобальный сбой).**

  Эта настройка включена по умолчанию для вновь созданных мониторов. Она оповещает о глобальных сбоях доступности, то есть когда все локации одновременно фиксируют неудачу.

  По умолчанию проблема глобального сбоя создаётся, когда все локации фиксируют неудачу один раз. Однако вы можете указать количество последовательных неудач (от 1 до 5) для глобального сбоя, то есть сколько раз подряд все локации должны зафиксировать неудачу для создания проблемы глобального сбоя.
* **Создавать проблему и отправлять оповещение, когда этот монитор недоступен в течение одного или нескольких последовательных запусков в любой локации. Создание проблемы локального сбоя доступно только при назначении не менее двух локаций.**

  Это позволяет создавать проблему при последовательных неудачах в одной или нескольких локациях. На уровне среды вы можете выбрать количество неудач. На уровне монитора вы также можете определить количество локаций монитора, которые должны зафиксировать неудачу для создания проблемы локального сбоя.

**Автоматический повтор при ошибке** включён по умолчанию (вы можете отключить его при необходимости), что означает, что неудачные выполнения монитора немедленно повторяются один раз для предотвращения ложных срабатываний, и учитывается только второй результат. При отключении используется первый результат.

Для **пороговых значений производительности** вы можете включить:

* **Создавать проблему и отправлять оповещение при нарушении пороговых значений производительности.**

  Эта настройка предоставляет возможность установить **пороговое значение для суммы всех запросов (в секундах)**. Если пороговое значение превышает указанное вами время, вы получите уведомление.

  Также вы можете установить пороговое значение производительности для [отдельных шагов](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/steps-waiting-time-validation-thresholds#performance-thresholds "Узнайте, как настроить время ожидания между шагами, валидировать контент и устанавливать пороговые значения производительности.").

### Использование клиентских сертификатов

Dynatrace версии 1.272+

ActiveGate версии 1.271+

Вы можете настроить аутентификацию по сертификату для браузерных мониторов, работающих в любой [публичной локации](/docs/observe/digital-experience/synthetic-monitoring/general-information/public-synthetic-locations "Узнайте обо всех доступных публичных локациях синтетического мониторинга.") и [приватных локациях](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/private-locations "Узнайте, как управлять приватными локациями в приложении Synthetic.") с ActiveGate на базе Linux (как контейнеризированных, так и неконтейнеризированных).

Перед первой записью кликпути на веб-сайте, требующем аутентификации по сертификату, убедитесь, что необходимый сертификат установлен в Chrome. Затем при переходе на веб-сайт в окне записи нативный диалог браузера автоматически выберет правильный сертификат.

Если в цепочке есть хотя бы один просроченный сертификат, вся цепочка может быть отклонена. Это приводит к неудачным выполнениям монитора. Результаты могут отличаться при использовании той же цепочки сертификатов с HTTP-монитором или при ручном запуске команды curl с хоста.

После записи кликпути необходимо указать сертификат для выполнения браузерного монитора.

1. Нажмите **Add certificate**.
2. Введите **Domain**, для которого действителен сертификат.
3. Выберите учётные данные из списка отображаемых сертификатов. Или нажмите **Create new credential**, чтобы загрузить и использовать новый клиентский сертификат. Любые создаваемые вами учётные данные сертификата автоматически обозначаются как [только для владельца](/docs/manage/credential-vault#work-with-credentials "Хранение и управление учётными данными в хранилище учётных данных.") и сохраняются в [хранилище учётных данных](/docs/manage/credential-vault "Хранение и управление учётными данными в хранилище учётных данных.").

Вы можете указывать и загружать файлы сертификатов в форматах PFX, P12 или PEM.

### Установка cookie-файлов

Cookie-файлы позволяют хранить информацию о состоянии браузера на стороне клиента, чтобы каждое выполнение монитора основывалось на одном и том же состоянии и вы могли точно отслеживать базовый уровень производительности.

Cookie-файлы, установленные в **Расширенных настройках**, действуют на протяжении всего выполнения монитора. Если вы хотите установить cookie-файлы только для определённой части кликпути, используйте [шаг Cookie](/docs/observe/digital-experience/synthetic-on-grail/synthetic-app/create-configure-browser-monitors/browser-clickpath-steps#cookie "Узнайте о типах шагов, создаваемых при записи кликпути браузера").

Нажмите **Add cookie**, укажите **Name** и **Value** cookie-файла. Каждый cookie-файл должен быть уникальным в списке.

Следующие символы не допускаются в значении cookie-файла: `;,\"`. Укажите **Domain** cookie-файла и, при необходимости, **Path** к cookie-файлу.

### Добавление дополнительных HTTP-заголовков

Монитор создаётся с минимальным набором заголовков, необходимых по протоколу.

1. Нажмите **Add header**.
2. Введите имя **Header** и **Value**.

Вы можете указать несколько HTTP-заголовков. Они будут установлены для всех запросов, выполняемых монитором.

Если вы хотите установить заголовки только для определённых запросов, выберите **Only apply headers to requests matching a pattern**, а затем определите **Pattern**. После этого заголовок устанавливается только для запросов, соответствующих определённому вами шаблону.

### Блокировка определённых запросов

Вы можете заблокировать один или несколько запросов, указав полные URL-адреса или шаблоны совпадения. Строка URL или регулярного выражения не должна превышать 90 символов и нечувствительна к регистру.

Блокировка запросов позволяет проанализировать, как эти запросы влияют на производительность вашего приложения, и оптимизировать его соответствующим образом.

Нажмите **Add pattern**, а затем укажите полный URL-адрес.

Вы также можете определять шаблоны, например `http:*://*/*.png` или `https*://*/*.png`, чтобы заблокировать все запросы к PNG-изображениям.

Совпадающие запросы блокируются на протяжении всего выполнения монитора и не зависят от событий.

Если вам не удаётся заблокировать определённые запросы, обратитесь к статье Dynatrace Community [Невозможно заблокировать определённые запросы (из-за ограничений длины регулярных выражений)](https://dt-url.net/w0i2xhk).

### Игнорирование определённых кодов состояния

Это позволяет исключить HTTP-коды состояния 400-599 из создания сбоев доступности/ошибок при их обнаружении в запросе основного документа. Как правило, это базовая страница в действии загрузки. Если эта настройка включена, и базовая страница действия загрузки сталкивается с HTTP-ошибками 400-599, сбой доступности не создаётся.

Вы можете указать точный код состояния, диапазон или маску класса состояния. Используйте запятые для разделения нескольких значений; используйте дефис (`-`) без пробелов для диапазона, например `404, 405-410, 5xx`. Вы можете применить правило только к запросам документов, соответствующим определённому регулярному выражению (**Only apply to document request matching this regex**).

* Эта настройка не применяется к XHR-запросам или запросам документов во фреймах (iframe); браузерные мониторы не завершаются с ошибкой при неудаче XHR-запроса или запроса документа в iframe.
* Эта настройка применяется к будущим выполнениям, то есть к выполнениям, которые происходят после включения настройки.

### Пользовательские настройки RUM JavaScript

**Capture XmlHttpRequest (XHR)** и **Capture fetch() requests** включены по умолчанию. Вы можете их отключить.

### Другое

**Bypass Content Security Policy (CSP) of monitored page** -- если у вас настроена политика безопасности контента (Content Security Policy), она, вероятно, препятствует отправке браузером данных мониторинга в кластер Dynatrace. В качестве первого и предпочтительного метода для обхода CSP отслеживаемых страниц в вашем браузерном мониторе с одним URL или кликпутью включите эту опцию. Если по какой-либо причине вы не можете использовать эту опцию, обратитесь к расширенным решениям для обхода CSP в статье Dynatrace Community [Браузерные мониторы: проблемы с Content Security Policy](https://dt-url.net/ycs2x56).

**Capture performance metrics for pages loaded in frames** -- включите эту опцию для начала сбора данных о производительности страниц в iframe или фреймсетах.
