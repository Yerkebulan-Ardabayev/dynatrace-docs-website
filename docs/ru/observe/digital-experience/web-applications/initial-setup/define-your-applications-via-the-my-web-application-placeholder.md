После установки [OneAgent](../../../../ingest-from/dynatrace-oneagent.md "Узнайте важные концепции, связанные с OneAgent, и как установить и управлять OneAgent на различных платформах.") в режиме полного мониторинга на хосте он мониторит все приложения, работающие на этом хосте. В качестве начальной точки все данные мониторинга объединяются в приложение-заглушку под названием **My web application**. Мы предлагаем это приложение-заглушку для большей гибкости — именно вы решаете, как организовать ваши приложения.

Dynatrace предлагает следующие подходы к определению вашего приложения для RUM.

| Подход к обнаружению приложений | Тип внедрения RUM JavaScript | Когда использовать |
| --- | --- | --- |
| [Рекомендуемый подход](#automated-approach) | Автоматическое внедрение | Когда ваши веб-приложения можно легко идентифицировать по доменам и вы хотите создать приложение из уже захваченного трафика RUM-мониторинга. |
| [Подход с правилами обнаружения приложений](#application-detection-rules) | Автоматическое внедрение | Когда вы хотите создать новые приложения или когда разделения трафика по доменам недостаточно. Используйте правила обнаружения приложений для определения более сложных шаблонов группировки трафика RUM-мониторинга в приложения. |
| [Ручной подход](#alternative-approach) (также известный как «безагентный мониторинг») | Ручная вставка | Когда у вас нет доступа к хосту вашего веб-приложения, но есть доступ к коду приложения. |

Вы можете выбрать формат внедрения как для автоматически внедряемых, так и для вручную вставляемых приложений. Подробнее см. [Выбор формата сниппета](snippet-formats.md "Выберите формат сниппета RUM JavaScript, наиболее подходящий для вашего конкретного случая использования").

Приложения на основе Java Servlet

Если вы используете RUM для приложения на основе Java Servlet, обязательно перезапустите процесс после завершения начальной настройки. Сенсор RUM не активируется до перезапуска.

## Рекомендуемый подход

Приложение-заглушка **My web application** агрегирует трафик со всех обнаруженных доменов. Это приложение может служить вашей отправной точкой для сопоставления идентифицированных доменов с отдельными приложениями в вашей среде.

1. Перейдите в **Web**.
2. Выберите приложение-заглушку **My web application**.
3. Прокрутите вниз, чтобы найти панель **Top 3 included domains**.
   Эта панель показывает домены с наибольшим количеством действий пользователей, обнаруженных OneAgent в вашей среде.
4. Выберите **View full details**.
5. В списке **Top domains** нажмите кнопку со стрелкой в столбце **Transfer domain**, чтобы развернуть запись домена.
6. Выполните один из следующих вариантов:

   * Чтобы добавить новое приложение, выберите **Create new application**. Ваше приложение будет создано и отображено на странице **Applications**.
   * Чтобы добавить домен к существующему приложению, выберите **Transfer**.

Возможно, вы захотите дать более осмысленное имя вашему приложению — замените автоматически сгенерированное имя на пользовательское имя приложения по вашему выбору.

Переименование приложения

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **General settings** > **Application name**.
5. Введите предпочтительное имя в поле вверху страницы. Обратите внимание, что имена приложений должны быть уникальными.

### Возможные проблемы рекомендуемого подхода

**Неинструментированный компонент перезаписывает URL**. При использовании рекомендуемого подхода для вашего нового или обновлённого приложения автоматически создаётся правило обнаружения приложения. Это правило сопоставляет действия пользователей, захваченные на домене, с вашим приложением. OneAgent использует часть хоста в URL для определения домена. Если неинструментированный компонент между браузером и первым инструментированным уровнем перезаписывает хост, вам необходимо [настроить определение имени хоста](#rum-appdetection-uninstrumentedcomponent). В противном случае OneAgent может не применить ваше правило обнаружения приложения.

**Действия по-прежнему сопоставляются с приложением-заглушкой**. Если действия пользователей по-прежнему сопоставляются с **My web application** вместо вашего приложения, обратитесь к разделу [Как быстро вступают в силу изменения правил обнаружения приложений?](#rum-appdetection-rule-changes).

## Подход с правилами обнаружения приложений

Если вы хотите создать больше приложений, изменить существующие сопоставления приложений или если вам нужно определить более сложные правила, основанные не только на доменах, но и на URL, вы можете использовать страницу настроек **Application detection**.

URL-адреса, используемые для обнаружения приложений, имеют структуру `scheme://host:port/path?query`, где строка запроса является необязательной, а порты по умолчанию `80` для HTTP и `443` для HTTPS опускаются. URL не включает возможный идентификатор фрагмента, как в `scheme://host:port/path?query#fragment`, поскольку правила обнаружения приложений оцениваются на стороне сервера приложения, а идентификатор фрагмента используется только браузером и не добавляется к веб-запросам.

Добавление правила обнаружения приложения

1. Перейдите в **Settings** > **Web and mobile monitoring** > **Application detection**. Отображается список правил обнаружения приложений.
   Для каждого приложения, определённого с помощью [рекомендуемого подхода](#automated-approach), автоматически генерируется правило обнаружения и добавляется в конец списка. Правила применяются последовательно, причём правила в верхней части списка имеют приоритет над правилами, расположенными ниже.
2. Выберите **Add detection rule**.
3. Используйте опции, предложенные на странице, для создания соответствующего правила обнаружения для вашего приложения.
   Вы можете применять правила к новым или существующим приложениям, а также создавать правила на основе URL или домена (хоста) вашего приложения.
4. Выберите **Save**, чтобы создать правило обнаружения приложения.
   Правило помещается в самый конец списка правил обнаружения приложений.
5. Необязательно: выберите и удерживайте **Drag row** ![Ручка перетаскивания](https://dt-cdn.net/images/drag-handle-turquoise-600-1aa0e5ea00.svg "Ручка перетаскивания") рядом с именем правила и переместите правило вверх или вниз по списку, чтобы изменить его приоритет.

Вы можете создать до 1 000 правил обнаружения приложений на одну среду.

### Возможные проблемы подхода с правилами обнаружения приложений

**Неинструментированный компонент перезаписывает URL**. Если неинструментированный компонент между браузером и первым инструментированным уровнем перезаписывает URL, необходимо учесть особые обстоятельства (подробнее см. [Что делать, если неинструментированный компонент перезаписывает части URL?](#rum-appdetection-uninstrumentedcomponent)). В противном случае OneAgent может не применить ваши правила обнаружения приложений.

**Действия по-прежнему сопоставляются с предыдущим приложением**. Если действия пользователей по-прежнему сопоставляются с предыдущим приложением вместо требуемого, обратитесь к разделу [Как быстро вступают в силу изменения правил обнаружения приложений?](#rum-appdetection-rule-changes).

### Часто задаваемые вопросы по рекомендуемому подходу и подходу с правилами обнаружения

Как Dynatrace применяет ваши правила обнаружения приложений?

Правила обнаружения приложений оцениваются на стороне сервера вашего приложения с помощью OneAgent на первом инструментированном уровне — ближайшем к браузеру. Результат передаётся последующим OneAgent с помощью заголовка `x-dynatrace-application`. При таком подходе любое перезаписывание URL, которое может происходить между инструментированными серверными уровнями вашего приложения, не влияет на результат обнаружения приложения.

OneAgent добавляет параметр `app` в RUM JavaScript, внедряемый в HTML-код вашего приложения. Этот параметр содержит идентификатор обнаруженного приложения.

![Параметр app в RUM JavaScript](https://dt-cdn.net/images/injected-app-parameter-2505-7ee0111d11.png)

При отправке запросов маячков с захваченными действиями пользователей RUM JavaScript включает этот параметр `app` в строку запроса.

![Параметр app, добавленный в строку запроса](https://dt-cdn.net/images/app-parameter-in-beacon-query-1922-30c91adb9b.png)

На кластере Dynatrace параметр `app` в строке запроса маячка используется для назначения действий пользователей приложениям.

Проверка идентификатора приложения для сравнения с параметром `app`

1. В Dynatrace перейдите в **Web**.
2. Выберите ваше приложение.
3. Проверьте URL страницы, на которой вы сейчас находитесь. Этот URL содержит параметр `uemapplicationId`, например `uemapplicationId=APPLICATION-4CC1C5694BF9B3AB`. Его значение без префикса `APPLICATION-` — это идентификатор приложения, который вы ищете.

   ![Проверка идентификатора приложения в веб-интерфейсе Dynatrace](https://dt-cdn.net/images/application-id-2559-b6b9f31629.png)

Что делать, если неинструментированный компонент перезаписывает части URL?

Как объясняется в разделе [Как Dynatrace применяет ваши правила обнаружения приложений?](#rum-appdetection-appparam), ваши правила обнаружения приложений оцениваются OneAgent на первом инструментированном уровне вашего приложения. Однако между браузером и первым инструментированным уровнем может находиться неинструментированный компонент, например обратный прокси-сервер или балансировщик нагрузки, который перезаписывает URL. В результате URL, используемый для обнаружения приложения, отличается от URL, изначально запрошенного браузером. В этом сценарии необходимо учесть особые обстоятельства.

### Хост перезаписывается

Вы можете настроить компоненты, такие как прокси-серверы и балансировщики нагрузки, для передачи исходной информации о хосте на бэкенд в заголовке запроса, например в заголовке [`X-Forwarded-Host`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/X-Forwarded-Host) или [`Forwarded`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Forwarded). OneAgent может получить исходный хост из одного из этих заголовков или из любого проприетарного заголовка, имеющего тот же формат, что и `X-Forwarded-Host`.

![Передача исходной информации о хосте в заголовке X-Forwarded-Host](https://dt-cdn.net/images/host-name-determination-1093-9025d73d9b.png)

Использование исходного имени хоста при обнаружении приложения

1. Настройте ваш неинструментированный компонент для добавления заголовка, передающего исходное имя хоста (если он не делает этого по умолчанию).
   Например, если исходное имя хоста, запрошенное браузером, было `www.example.com`, а ваш неинструментированный компонент перенаправил запрос на внутренний хост `backend.com`, то компонент должен добавить заголовок запроса, передающий `www.example.com`, как показано на диаграмме последовательности выше.
2. В Dynatrace перейдите в **Settings** > **Web and mobile monitoring** > **Host name determination**.
3. Если заголовок, добавленный вашим компонентом, отсутствует в списке, выберите **Add HTTP header**.
4. Заголовки запросов в списке обрабатываются последовательно, причём заголовки в верхней части списка имеют приоритет. Переместите вашу новую запись выше заголовка `Host`, который является частью каждого запроса HTTP/1.1.

В качестве альтернативы вы также можете использовать внутреннее имя хоста в правиле обнаружения приложения. Однако OneAgent использует результат определения имени хоста не только для обнаружения приложения, но и для [автоматического определения домена cookie](../additional-configuration/configure-the-cookie-domain.md#automatic-cookie-domain-determination "Узнайте, когда и как настроить домен cookie."). Поэтому, если вы не настроите определение имени хоста, как описано выше, вам необходимо настроить домен cookie вручную, как описано в разделе [Ручная настройка домена cookie](../additional-configuration/configure-the-cookie-domain.md#manual-cookie-domain-config "Узнайте, когда и как настроить домен cookie."). Рекомендуемый выбор — [eTLD+1](https://web.dev/same-site-same-origin/) домена, который OneAgent определяет автоматически.

### Путь URL перезаписывается

В отличие от части хоста URL, нет возможности определить исходный путь URL, поскольку компоненты редко устанавливают заголовки, передающие исходный путь последующим уровням. Поэтому вы не можете использовать часть пути URL, которая удаляется при перезаписи URL на неинструментированном компоненте, в вашем правиле обнаружения приложения.

Как быстро вступают в силу изменения правил обнаружения приложений?

Когда вы изменяете правила обнаружения приложений, обновлённые правила обычно передаются в OneAgent в течение минуты. OneAgent классифицирует входящие запросы согласно вашим правилам обнаружения приложений и устанавливает параметр `app` во внедрённом RUM JavaScript соответственно. Затем RUM JavaScript отправляет свои маячки с использованием обновлённого параметра `app`, определяющего категоризацию захваченных действий пользователей. Подробнее см. [Как Dynatrace применяет правила обнаружения приложений?](#rum-appdetection-appparam).

Следующие факторы могут задержать вступление в силу обновлённых правил обнаружения приложений:

* HTML-код доставляется через CDN или другой кеш. В этом случае OneAgent может внедрить RUM JavaScript только когда HTML-код будет вытеснен из кеша и снова запрошен с инструментированного исходного сервера.
* Приложение указывает длительное время истечения для HTML-документа с помощью заголовка ответа `Expires` или директивы `max-age` заголовка ответа `Cache-Control`. Кеш может обслуживать его в течение указанного в этих заголовках времени, включая внедрённый параметр `app`. Изменение правил обнаружения приложений не может быть отражено немедленно, поскольку кеш не отправляет запрос к исходному серверу, пока документ не истёк.
* Если приложение не указывает явное время истечения, это не означает, что страница не может быть кеширована (см. [RFC7234: Calculating heuristic freshness](https://datatracker.ietf.org/doc/html/rfc7234#section-4.2.2)). Кеш может использовать страницу без повторной валидации столько, сколько сочтёт нужным, если ответ не содержит директив, предотвращающих это, например `Cache-Control: no-cache`.
* Одностраничные приложения динамически перезаписывают текущую страницу вместо загрузки новой страницы с сервера. Если HTML-код был загружен во вкладку браузера до изменения правил обнаружения приложений, то категоризация захваченных XHR-действий пользователей ещё не отражает обновлённую конфигурацию. Это изменится, как только страница будет обновлена.

## Подход безагентного RUM

Когда у вас нет доступа к вашему веб-серверу — и поэтому вы не можете установить OneAgent на хосте — но у вас есть доступ к коду вашего приложения, выберите [безагентный RUM](set-up-agentless-real-user-monitoring.md "Настройте безагентный мониторинг для ваших веб-приложений.").

Для некоторых технологий [автоматическое внедрение RUM JavaScript](rum-injection.md "Настройте автоматическое внедрение RUM JavaScript на страницы ваших приложений") не поддерживается даже при установленном OneAgent. Например, хотя OneAgent может мониторить серверную сторону приложения Heroku, он не может внедрить RUM JavaScript на страницы приложения. В таком случае вам необходимо использовать безагентный RUM и вручную добавить RUM JavaScript на страницы вашего приложения.

Список технологий и серверов, поддерживающих автоматическое внедрение RUM JavaScript, см. в разделе [Поддержка технологий - мониторинг реальных пользователей - веб-серверы и приложения](../../../../ingest-from/technology-support.md#rum-auto-injection "Узнайте технические подробности о поддержке Dynatrace для конкретных платформ и фреймворков разработки.").

## Лучшие практики и рекомендации

* Определяйте ваши приложения на основе принадлежности команде, чтобы вы могли легко использовать [зоны управления](../../../../manage/identity-access-management/permission-management/management-zones.md "Узнайте о концепциях зон управления, как определить зоны управления и как максимально их использовать.") для ограничений доступа.
* Может иметь смысл определять приложения на основе их технологического стека, чтобы применялись правильные настройки и управление конкретными настройками было проще. Например, активация поддержки определённых XHR-фреймворков обычно требуется только для конкретных частей большого приложения и для конкретных правил именования действий пользователей. Может помочь разделение таких больших приложений на более мелкие на основе используемых технологий или принадлежности команде.
* Приложение-заглушка **My web application**

  + Избегайте переименования приложения-заглушки **My web application**, поскольку оно включает все действия пользователей на всех доменах, которые не включены в правило обнаружения приложения. Если вы переименуете **My web application**, может быть трудно отличить его от ваших других приложений.
  + Невозможно удалить это приложение-заглушку.
  + Если вы не можете найти приложение-заглушку **My web application**, возможно, кто-то его переименовал. Чтобы найти его, откройте любое другое приложение и замените идентификатор приложения в URL на `EA7C4B59F27D43EB`. Вы будете перенаправлены на **My web application**.
  + В новых средах вы увидите это приложение-заглушку только после того, как начнёт поступать трафик от автоматически внедряемого приложения.
* Разделение приложений по доменам работает лучше всего, поскольку Dynatrace не может коррелировать действия пользователей между доменами с конкретными пользовательскими сессиями. Эта корреляция выполняется через cookie и поэтому работает только если cookie может быть установлен на одном домене. Например, действия пользователей для `www.dynatrace.com` и `blog.dynatrace.com` могут быть захвачены в одном приложении, поскольку cookie может быть установлен на `dynatrace.com`. Однако трафик для `www.dynatrace.com` и `www.internal-dynatrace.com` не может быть захвачен в одной пользовательской сессии. Вы всё ещё можете разделить действия пользователей по домену, но пользовательские сессии не могут включать действия пользователей из нескольких доменов. Подробнее см. [Пользовательские сессии](../../rum-concepts/user-session.md "Узнайте, как определяется пользовательская сессия, когда она начинается или заканчивается, как рассчитывается продолжительность и т.д.").
* Группируйте приложения с низким трафиком. Если вы создаёте приложение на основе домена, имеющего менее 10 действий в минуту, Dynatrace не будет автоматически обнаруживать аномалии для этого вновь созданного приложения. Dynatrace зависит от стабильного трафика приложения для корректного [обучения многомерных базовых линий и автоматического создания отчётов о проблемах приложения](../../../../dynatrace-intelligence/anomaly-detection/static-thresholds-infrastructure.md "Узнайте о фиксированных порогах, используемых Dynatrace для определения, когда обнаруженное замедление или увеличение частоты ошибок оправдывает генерацию нового события проблемы."). Вы можете изменить настройки обнаружения аномалий приложения на странице **Anomaly detection** в настройках вашего приложения. Хотя эта рекомендация противоречит упомянутой выше, может иметь смысл объединить приложения с низким трафиком.
* Правила обнаружения приложений обрабатываются последовательно для каждого запроса. Больше правил означает больше времени обработки, и поскольку правила обрабатываются внутри OneAgent, вы должны стремиться размещать правила для приложений с наибольшим трафиком вверху списка. Как только одно правило совпадает, дальнейшая обработка не выполняется.
* Более специфичные правила обнаружения приложений должны быть определены первыми, а более общие правила должны быть внизу списка правил обнаружения приложений.
  Предположим, что вы хотите создать приложение `A`, на которое будут сопоставлены следующие два домена:

  `http://www.mydomain.com/cms/directory/shop/index.html`
  `http://shop.differentdomain.de/index.html`

  и другое приложение `B` для следующего домена:

  `http://newdomain.co.uk/hello/shop.html`

  Если вы создадите общее правило группировки на основе значения `shop`, все три домена будут сгруппированы в одно веб-приложение для мониторинга. Поэтому сначала следует определить более специфичное правило, например «Если URL заканчивается на `shop.html`», чтобы только третий URL был сопоставлен с приложением `B`. Затем вы можете безопасно определить общее правило на основе значения `shop`, поскольку третий URL уже будет сопоставлен с предыдущим приложением и поэтому не будет включён в приложение `A`.
* В зависимости от ваших требований вы можете настроить потребление мониторинга и соответствующим образом сконфигурировать мониторинг реальных пользователей.
* Для мониторинга трафика в одном приложении вы можете использовать [свойства действий пользователей и сессий](../additional-configuration/define-user-action-and-session-properties.md "Определите пользовательские строковые, числовые и свойства дат для ваших отслеживаемых веб-приложений."). Для мониторинга отдельных приложений и получения полной информации о потреблении на приложение вы можете настроить отдельные приложения, использовать теги для разделения метрик и определить зоны управления.
