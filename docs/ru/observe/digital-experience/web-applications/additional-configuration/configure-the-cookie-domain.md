# Настройка домена cookie RUM для веб-приложений


* How-to guide
* 4-min read

Dynatrace Real User Monitoring использует HTTP-cookie для группировки пользовательских действий в сессии и связывания пользовательских действий с соответствующими [распределёнными трассировками](../../../application-observability/distributed-traces.md "Получите наблюдаемость в высокораспределённых облачных архитектурах для эффективной трассировки и анализа транзакций в реальном времени."). Браузеры включают cookie в заголовок запроса `Cookie` только в том случае, если URL запроса находится в области действия cookie, которая определяется атрибутами cookie `Domain` и `Path`.

Dynatrace всегда устанавливает атрибут `Path` cookie RUM в значение `/`, чтобы область действия cookie охватывала все пути URL в домене. Атрибут `Domain` определяется автоматически по умолчанию, но вы также можете [настроить его вручную для каждого приложения](#configure-cookie-domain).

## Автоматическое определение домена cookie

Если вы не [настроили домен cookie](#configure-cookie-domain), Dynatrace определяет его автоматически. Dynatrace выбирает [эффективный домен верхнего уровня плюс один (eTLD+1)](https://web.dev/same-site-same-origin/) URL запроса. Например, Dynatrace выбирает домен cookie `example.com` для домена `www.example.com` или домен cookie `example.co.uk` для домена `www.example.co.uk`. При таком выборе Dynatrace может фиксировать непрерывную сессию, даже когда ваши пользователи посещают несколько поддоменов при взаимодействии с вашими приложениями, например, при переходе с `www.example.com` на `shop.example.com`.

Домен cookie определяется либо на стороне сервера с помощью OneAgent, либо на стороне клиента с помощью RUM JavaScript, в зависимости от того, какой из них первым зафиксирует взаимодействие пользователя с вашим приложением. Если RUM JavaScript внедряется автоматически, обычно домен cookie определяет OneAgent. Однако когда браузер берёт страницу из кэша, домен cookie определяет RUM JavaScript.

### Возможные проблемы при автоматическом определении домена cookie

OneAgent использует результат определения имени хоста в качестве отправной точки при определении домена cookie. Если существует неинструментированный компонент, который перезаписывает хостовую часть URL, крайне важно для автоматического определения домена cookie, чтобы определение имени хоста было настроено правильно. Подробности см. в разделе [Что делать, если неинструментированный компонент перезаписывает части URL?](../initial-setup/define-your-applications-via-the-my-web-application-placeholder.md#rum-appdetection-uninstrumentedcomponent "Узнайте, как определять ваши приложения с помощью предложенного, ручного подхода или правил обнаружения приложений.")

## Ручная настройка домена cookie

В большинстве сценариев нет необходимости настраивать домен cookie вручную. Однако существуют два сценария, когда это необходимо:

* Вам нужно разделить пользовательские действия разных поддоменов в отдельные пользовательские сессии.
* Автоматическое определение домена cookie не работает, потому что неинструментированный компонент перезаписывает хостовую часть URL и не пересылает информацию об исходном хосте в заголовке запроса. Предпочтительным решением является настройка компонента на добавление такого заголовка (см. инструкции в разделе [Что делать, если неинструментированный компонент перезаписывает части URL?](../initial-setup/define-your-applications-via-the-my-web-application-placeholder.md#rum-appdetection-uninstrumentedcomponent "Узнайте, как определять ваши приложения с помощью предложенного, ручного подхода или правил обнаружения приложений.")). Однако ручная настройка домена cookie также является возможным обходным решением.

### Ручная настройка домена cookie

Для настройки домена cookie

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Injection** > **Cookie**.
5. В поле **Domain to be used for cookie placement** введите нужный домен.

### Возможные проблемы при ручной настройке

#### Настроенный домен cookie является публичным суффиксом

Браузеры не позволяют устанавливать cookie с атрибутом `Domain`, который выходит за границы одной организации. Поэтому установка домена cookie в значение [публичного суффикса](https://publicsuffix.org/list/) не работает.

#### Несколько доменов без общего eTLD+1 привязаны к настроенному приложению

Поскольку домен cookie настраивается для каждого приложения, все домены, привязанные к вашему приложению, должны иметь хотя бы общий eTLD+1 для возможности определения домена cookie.

Например, вы не можете вручную настроить домен cookie для приложения, к которому привязаны как `www.example.com`, так и `www.example.co.uk`. Если вы выберете домен cookie `example.com`, браузер отклонит cookie RUM при запросах к `www.example.co.uk`.

#### Перекрывающиеся домены cookie

Если вы настраиваете домен cookie для всех или некоторых ваших приложений вручную, вы должны убедиться, что домены cookie ваших приложений не перекрываются.

Рассмотрим следующий пример. Домен `www.example.com` привязан к приложению **Example app**, а домен `shop.example.com` привязан к приложению **Shopping app**. По умолчанию домен cookie `example.com` определяется автоматически для обоих доменов. Если вы установите домен cookie `shop.example.com` для приложения **Shopping app** и ваши пользователи будут переходить между двумя приложениями, возникнут неоднозначные ситуации, поскольку cookie RUM для **Example app** имеют домен cookie `example.com` и, следовательно, также применимы к **Shopping app**, которое имеет другой набор cookie RUM с доменом `shop.example.com`. При такой конфигурации Dynatrace может случайным образом разбивать собранные данные RUM на короткие пользовательские сессии и пользовательские действия, а распределённые трассировки могут быть связаны не так, как ожидалось.
