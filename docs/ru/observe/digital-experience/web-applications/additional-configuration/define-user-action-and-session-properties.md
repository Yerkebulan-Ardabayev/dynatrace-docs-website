# Определение свойств пользовательских действий и сессий для веб-приложений

# Определение свойств пользовательских действий и сессий для веб-приложений

* Практическое руководство
* Время чтения: 8 мин
* Обновлено 12 февраля 2026 г.

Dynatrace собирает большой объём информации о производительности вашего веб-приложения. Вы можете обогатить эту информацию ценными метаданными и превратить их в свойства пользовательских действий и сессий.

Свойства действий и сессий представляют собой пары ключ-значение, которые можно фильтровать в нескольких представлениях анализа Dynatrace. Эти свойства полезны, когда вам нужно создавать мощные запросы, сегментации и агрегации на основе захваченных метаданных. Вы можете использовать свойства действий и сессий для создания вычисляемых метрик приложения. Вы также можете видеть их в представлении многомерного анализа, на странице **User sessions** и на странице **User sessions query**. Чтобы получить более глубокое понимание того, как использовать эти свойства, посетите [Использование свойств пользовательских действий и сессий для веб-приложений](../analyze-and-use/action-and-session-properties.md "Свойства пользовательских действий и сессий, которые представляют собой пары ключ-значение метаданных, обеспечивают дополнительную видимость и более глубокий анализ опыта ваших конечных пользователей. Используя эти свойства для ваших веб-приложений, вы можете фильтровать пользовательские сессии, добавлять вычисляемые метрики, создавать графики и многое другое.").

Ниже вы найдёте информацию о том, как настроить такие свойства, а также связанные примеры конфигурации. Чтобы использовать свойства действий и сессий, вам сначала нужно [добавить свойства в веб-интерфейсе Dynatrace](#add-properties), а затем начать передачу необходимых метаданных в Dynatrace.

## Добавление пользовательского свойства

Используйте пользовательские свойства для настройки строковых, числовых и свойств типа даты для ваших мониторируемых [пользовательских действий](../../rum-concepts/user-actions.md "Узнайте, что такое пользовательские действия и как они помогают понять, что пользователи делают с вашим приложением.") и [пользовательских сессий](../../rum-concepts/user-session.md "Узнайте, как определяется пользовательская сессия, когда она начинается или заканчивается, как рассчитывается её продолжительность и многое другое."). Dynatrace затем захватывает значения свойств как часть пути каждого пользователя. Вы можете использовать значения свойств для непревзойдённой видимости всех деталей взаимодействия ваших пользователей с вашим приложением. Вы можете определить пользовательские свойства действий и сессий, специфичные для каждого приложения.

### Типы выражений

Dynatrace может захватывать метаданные из следующих источников (**типы выражений**):

* CSS-селектор
* Переменная JavaScript
* JavaScript API
* Функция JavaScript
* Meta-тег
* Значение cookie

  Требования для значения cookie

  Если вы хотите использовать `Cookie value` как источник данных, убедитесь, что cookies не имеют атрибута `HttpOnly`. В противном случае JavaScript RUM не сможет прочитать значения cookie, поскольку cookies с `HttpOnly` недоступны для JavaScript.
* Строка запроса
* Серверный атрибут запроса
* Заголовок ответа XHR/fetch (доступен начиная с версии JavaScript RUM 1.253 для XHR/fetch и начиная с версии JavaScript RUM 1.257 для Angular)

  Требования для заголовка ответа XHR/fetch

  Если вы хотите использовать `XHR/fetch response header` как источник данных, убедитесь в следующем:

  + В настройках вашего приложения включите опции **Capture fetch() requests**, **Capture XmlHttpRequest (XHR)** или **Angular**. Подробные инструкции см. в разделе [Активация поддержки универсальных фреймворков JavaScript](../initial-setup/configure-dynatrace-real-user-monitoring-to-capture-xhr-actions.md#enable-generic-js-frameworks "Узнайте, почему необходимо активировать конкретные фреймворки JavaScript для поддержки XHR-действий и как настроить мониторинг реальных пользователей для XHR-действий.") или [Активация поддержки Angular](../initial-setup/configure-dynatrace-real-user-monitoring-to-capture-xhr-actions.md#enable-specific-js-frameworks "Узнайте, почему необходимо активировать конкретные фреймворки JavaScript для поддержки XHR-действий и как настроить мониторинг реальных пользователей для XHR-действий.").

    Захват метаданных из заголовков ответа XHR/fetch не поддерживается для AngularJS.
  + Для CORS и XHR-запросов Dynatrace может захватывать метаданные только из [заголовков ответа, включённых в безопасный список CORS](https://developer.mozilla.org/en-US/docs/Glossary/CORS-safelisted_response_header). Если вы хотите извлекать данные из других заголовков, используйте заголовок [`Access-Control-Expose-Headers`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Access-Control-Expose-Headers).

    - `Access-Control-Expose-Headers: Content-Encoding, Destination, Basket-value` для раскрытия указанных заголовков
    - `Access-Control-Expose-Headers: *` для раскрытия всех заголовков, кроме заголовка `Authorization`, для запросов без учётных данных
    - `Access-Control-Expose-Headers: *, Authorization` для раскрытия всех заголовков для запросов без учётных данных

### Типы данных

Dynatrace может сохранять захваченные метаданные в одном из следующих типов данных:

| Тип данных | Примечание |
| --- | --- |
| `String` | Свойства действий и сессий типа `String` ограничены 1000 символами до применения правила очистки. |
| Number | `Double` или `Long` |
| `Date` | Доступен только для [типа выражения](#expression-types) `JavaScript API` — метаданные должны быть захвачены через [JavaScript API RUM](customize-rum.md "Узнайте, как настроить мониторинг реальных пользователей с помощью JavaScript API."). |

### Типы хранения

Вы можете сохранять метаданные на разных «уровнях хранения».

* **Свойство пользовательского действия**: Данные сохраняются в определённом свойстве на уровне пользовательского действия для каждого действия, где JavaScript RUM может получить метаданные.
* **Свойство пользовательской сессии**: Последнее захваченное значение сохраняется в определённом свойстве на уровне сессии.
* **Оба варианта**: Данные сохраняются в определённом свойстве как на уровне пользовательского действия, так и на уровне сессии.

### Определение пользовательского свойства

Чтобы определить пользовательское свойство

1. Перейдите в **Web**.
2. Выберите приложение, которое вы хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **Session and action properties**.
5. Выберите **Add property** > **Custom-defined property**.
6. Настройте свойство:

   * Выберите [**Expression type**](#expression-types). Если применимо, также выберите [**Data type**](#data-types) и **String length**.
   * Введите обозначение типа выражения, который вы хотите использовать, например, атрибут CSS-селектора или имя meta-тега. Для типа **Server-side request attribute** выберите [имя атрибута запроса](../../../application-observability/services/request-attributes.md "Узнайте, что такое атрибуты запроса и как их использовать на всех уровнях представлений анализа сервисов.").
   * Необязательно. Определите **Display name** — имя свойства, которое используется в веб-интерфейсе Dynatrace, например, на странице деталей сессии или странице деталей пользовательского действия.
   * Укажите **Key** — имя свойства, которое используется для идентификации и последующего поиска свойства в [USQL](../../session-segmentation/custom-queries-segmentation-and-aggregation-of-session-data.md "Узнайте, как получить доступ к данным пользовательских сессий и выполнять запросы на основе ключевых слов, синтаксиса, функций и многого другого.") и на странице **User sessions**.

     **Key**, который использовался ранее, не может быть повторно использован [до тех пор, пока данные хранятся](../../../../manage/data-privacy-and-security/data-privacy/data-retention-periods.md "Проверьте сроки хранения для различных типов данных.") в вашей среде. Это связано с тем, что захваченные данные по-прежнему ссылаются на старую конфигурацию свойства.
   * Необязательно. Отключите **Comply with "Do Not Track" browser settings** только в том случае, если свойство не содержит персональных данных.

     Эта опция доступна только для определённых типов выражений и если вы включили параметр **[Comply with "Do Not Track" browser settings](configure-real-user-monitoring-according-to-gdpr.md#do-not-track-gdpr "Узнайте о настройках конфиденциальности, которые Dynatrace предоставляет для обеспечения соответствия ваших веб-приложений требованиям конфиденциальности данных вашего региона.")** в разделе конфиденциальности данных настроек вашего приложения.
   * Выберите хотя бы один [тип хранения](#storage-levels) — свойство пользовательского действия, свойство сессии или оба. Для типа хранения свойства сессии выберите один из типов агрегации.
   * Необязательно. Чтобы ограничить захватываемые значения, включите **Apply cleanup rule** и укажите [регулярное выражение](../../../../manage/tags-and-metadata/reference/regular-expressions-in-dynatrace.md "Узнайте, как использовать регулярные выражения в контексте Dynatrace.").

     Регулярное выражение применяется к захваченному значению: если вы пытаетесь захватить строку длиной, например, 1000 символов, захватываются только первые 100 символов, и регулярное выражение применяется к этим 100 символам.

## Добавление пакета свойств

Используйте пакеты свойств для связывания данных аналитики с информацией о производительности. Вы можете сделать это, интегрируя инструменты, такие как веб-аналитика и мониторинг производительности, в Dynatrace. Доступен набор пакетов свойств, например, Adobe, Google, Intercom и Tealeaf.

Чтобы добавить свойство из пакета свойств

1. Перейдите в **Web**.
2. Выберите приложение, которое вы хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **Session and action properties**.
5. Выберите **Add property** > **Property packs**, а затем выберите нужный пакет свойств из выпадающего списка.
6. В столбце **Configure properties** выберите **Add** для свойств, которые вы хотите добавить, а затем выберите **Next**.
7. Выберите хотя бы один [тип хранения](#storage-levels) — свойство пользовательского действия, свойство сессии или оба — для каждого из добавленных свойств.
8. Выберите **Create property**.

  ![Инструменты разработчика](https://dt-cdn.net/images/userid-devtools-2544-09723cd81f.png)

  Чтобы указать, что строку необходимо разобрать как JSON, используйте символ `$` перед именем свойства (даже если имя свойства начинается с `$`, добавьте ещё один `$` перед ним).
  В этом примере строка, хранящаяся в `sessionStorage.user`, будет получена, и символ `$` указывает, что строку необходимо разобрать как JSON, а затем будет захвачено значение `userid`.

## Ограничения

* Вы можете определить максимум 200 свойств на приложение.
* Вы можете определить максимум 20 свойств действий на приложение.
* Свойства действий и сессий типа `String` ограничены 1000 символами до применения правила очистки.
* Вы можете использовать до 20 свойств на приложение бесплатно. Дополнительные свойства потребляют единицы DEM.
  См. [Единицы DEM](../../../../license/monitoring-consumption-classic/digital-experience-monitoring-units.md#dem-units "Узнайте, как рассчитывается потребление мониторинга цифрового опыта Dynatrace на основе единиц DEM.") (обратите внимание на записи **Session property** и **Action property** в таблице) и [Бесплатный уровень свойств действий и сессий](../../../../license/monitoring-consumption-classic/digital-experience-monitoring-units.md#free-action-and-session-properties "Узнайте, как рассчитывается потребление мониторинга цифрового опыта Dynatrace на основе единиц DEM.") для получения дополнительной информации.

## Примечания

* Dynatrace начинает захватывать свойства действий и сессий только после того, как вы [определите эти свойства в настройках вашего приложения](#add-properties).
* Свойства сессий недоступны для текущих активных сессий. Свойства становятся доступными только после завершения сессии.
* Метаданные захватываются в конце пользовательского действия.
  Если вы хотите захватить метаданные в начале пользовательского действия, выберите **JavaScript API** в качестве [типа выражения](#expression-types) при [добавлении пользовательского свойства](#add-properties), а затем используйте [JavaScript API RUM](customize-rum.md "Узнайте, как настроить мониторинг реальных пользователей с помощью JavaScript API.") для передачи необходимых значений, в частности, методы [addActionProperties](https://www.dynatrace.com/support/doc/javascriptapi/doc/types/dtrum.html#addactionproperties) и [sendSessionProperties](https://www.dynatrace.com/support/doc/javascriptapi/doc/types/dtrum.html#sendsessionproperties).
* Вы можете проверить, сколько свойств вы уже используете и сколько из них ещё можно добавить.

  1. Перейдите в **Web**.
  2. Выберите приложение, которое вы хотите настроить.
  3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
  4. В настройках приложения выберите **Capturing** > **Session and action properties**.
  5. Прокрутите вниз до раздела **Property usage quotas**.
