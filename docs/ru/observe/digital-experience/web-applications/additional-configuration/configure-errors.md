# Настройка обнаружения ошибок для веб-приложений

* Classic
* Практическое руководство
* Чтение: 8 мин
* Обновлено 12 июля 2022 г.

Чтобы настроить способ захвата ошибок, отредактируйте ваше приложение и разверните раздел **Errors**. Выберите нужный тип ошибки, а затем настройте захват ошибки, включение в [расчёты Apdex](../../rum-concepts/scores-and-ratings/apdex-ratings.md "Узнайте, как Dynatrace использует Apdex для измерения удовлетворённости пользователей производительностью приложения.") или учёт при [обнаружении и анализе проблем Davis AI](../../../../dynatrace-intelligence.md "Ознакомьтесь с возможностями Dynatrace Intelligence.").

## Типы ошибок

Dynatrace классифицирует [ошибки](../../rum-concepts/user-and-error-events.md#error "Узнайте о событиях пользователей и ошибок, а также о типах событий пользователей и ошибок, захватываемых Dynatrace.") на следующие типы.

* **Ошибки запросов** обнаруживаются браузером и OneAgent на ваших серверах.
* **Пользовательские ошибки** инициируются непосредственно в вашем приложении через [RUM JavaScript API](customize-rum.md "Узнайте, как настроить мониторинг реальных пользователей с помощью JavaScript API.").
* **Ошибки JavaScript** — это обнаруженные исключения JavaScript, генерируемые браузером.

## Настройка ошибок запросов

По умолчанию Dynatrace считает все HTTP-коды состояния `4xx` и `5xx` ошибками запросов. Нарушения CSP и ошибки с неудачными запросами изображений также захватываются как ошибки запросов. Кроме того, RUM JavaScript также может использовать [пользовательские коды состояния](#custom-status-codes) для информирования вас о проблемах фреймворка.

Вы можете изменить настройки по умолчанию, настроив правила ошибок запросов.

### Добавление правила ошибки запроса

Чтобы добавить правило ошибки запроса:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Errors** > **Request errors**.
5. Выберите **Add request error rule**.
6. Используйте как минимум один из следующих вариантов:

   * **Match by error code**. Укажите либо отдельный HTTP-код ошибки, например `404`, либо диапазон кодов ошибок, например `400-499`.
   * **Match by errors that have failed image requests**
   * **Match by errors that have CSP violations**
7. Необязательно: чтобы применить правило ошибки запроса только к определённым URL-адресам, измените параметры в разделе **Filter settings**.
8. Укажите, хотите ли вы **Capture this error**. Также вы можете **Include error in Apdex calculations** и **Include error in Davis AI problem detection and analysis**.

   ![Настройка правила ошибки запроса](https://dt-cdn.net/images/configure-request-error-1188-fed713c886.png)

   Если вы выберете включение ошибки в анализ Davis AI, Davis может сообщить об этой ошибке как о новой открытой проблеме.

Правила ошибок запросов выполняются в порядке их расположения в списке ошибок. Выберите и удерживайте **Drag row** ![Ручка перетаскивания](https://dt-cdn.net/images/drag-handle-turquoise-600-1aa0e5ea00.svg "Ручка перетаскивания") рядом с именем правила и переместите правило вверх или вниз в списке, чтобы изменить его приоритет.

Вы также можете включить **Ignore request errors in Apdex calculations**, чтобы переопределить настройки Apdex для отдельных правил ошибок запросов.

### Управление пользовательскими кодами состояния

Помимо захвата стандартных HTTP-кодов ошибок, RUM JavaScript также может использовать пользовательские коды состояния для сигнализации об обнаруженной проблеме фреймворка.

Для приложений, созданных с Dynatrace версии 1.238+, RUM JavaScript сообщает о некоторых ошибках запросов с использованием пользовательских кодов состояния `970`-`979`, когда реальный HTTP-код состояния не может быть захвачен. Обратите внимание, что эти пользовательские коды состояния не являются реальными HTTP-кодами состояния; они лишь означают, что RUM JavaScript обнаружил ошибку, вызванную используемым вами фреймворком.

Влияют ли такие ошибки запросов на ваших пользователей, зависит от вашего приложения. Например, если вы отправляете запрос к сервису отслеживания и запрос отменяется, такая ошибка едва ли влияет на пользователей вашего приложения. Однако неудачный запрос на оплату определённо создаст неудобства для них.

RUM JavaScript использует следующие пользовательские коды состояния:

| Код | Название | Объяснение |
| --- | --- | --- |
| 970 | Error | В используемом фреймворке произошла неуточнённая ошибка. |
| 971 | Canceled | Запрос был отменён. |
| 972 | Timeout | Время ожидания запроса истекло. |
| 973 | Parse | Не удалось выполнить разбор ответа. Сообщается модулями XMLHttpRequest, jQuery и MooTools. |
| 974 | Setup | Ошибка произошла при настройке запроса. Сообщается только модулем MooTools. |
| 979 | Unknown | Произошла неизвестная ошибка фреймворка. Сообщается только модулями jQuery и MooTools. |

#### Приложения, созданные начиная с Dynatrace версии 1.238

Для приложений, созданных начиная с Dynatrace версии 1.238, эти пользовательские коды состояния обнаруживаются автоматически. Они включаются в расчёты Apdex, но не учитываются при анализе Davis AI.

Если вы хотите игнорировать эти пользовательские коды состояния, удалите или отключите правило ошибки запроса **HTTP 970-979**.

Чтобы удалить или отключить правило ошибки запроса **HTTP 970-979**:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Errors** > **Request errors**.
5. Найдите правило ошибки **HTTP 970-979** и выполните одно из следующих действий:

   * Выберите **Delete row** ![Удалить](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Удалить"), чтобы полностью удалить это правило ошибки запроса.
   * Разверните правило и отключите **Capture this error**.

#### Приложения, созданные до Dynatrace версии 1.238

Для приложений, созданных с версиями Dynatrace ранее 1.238, пользовательские коды состояния не используются. Если вы хотите захватывать эти коды, [добавьте правило ошибки запроса](#configure-request-errors) со следующими настройками:

* **Match by error code**: `970-979`
* **Capture this error**: `Enabled`
* **Include error in Apdex calculations**: `Enabled` или `Disabled` в зависимости от ваших потребностей
* **Include error in Davis AI problem detection and analysis**: `Enabled` или `Disabled` в зависимости от ваших потребностей

## Настройка пользовательских ошибок

Пользовательские ошибки позволяют обнаруживать ваши собственные ошибки. Примером является ошибка, которая срабатывает при валидации поля формы.

Для настройки пользовательской ошибки укажите, как Dynatrace должен обрабатывать ошибку, а затем инициируйте ошибку в вашем приложении с помощью [RUM JavaScript API](customize-rum.md "Узнайте, как настроить мониторинг реальных пользователей с помощью JavaScript API.").

Чтобы добавить правило пользовательской ошибки:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Errors** > **Custom errors**.
5. Выберите **Add custom error rule**.
6. Укажите условия, которые Dynatrace должен использовать для идентификации пользовательской ошибки.

   * Используйте **Match key** и **Key pattern** для `key` пользовательской ошибки.
   * Используйте **Match value** и **Value pattern** для `value` пользовательской ошибки.

     Правила пользовательских ошибок нечувствительны к регистру. Например, значения `mykey` и `MyKeY` обрабатываются одинаково.
7. Укажите, хотите ли вы **Capture this error**. Также вы можете **Include error in Apdex calculations** и **Include error in Davis AI problem detection and analysis**.

   ![Настройка правила пользовательской ошибки](https://dt-cdn.net/images/configure-custom-error-1188-d01572ccac.png)

   Если вы выберете включение ошибки в анализ Davis AI, Davis может сообщить об этой ошибке как о новой открытой проблеме.

Правила пользовательских ошибок выполняются в порядке их расположения в списке ошибок. Выберите и удерживайте **Drag row** ![Ручка перетаскивания](https://dt-cdn.net/images/drag-handle-turquoise-600-1aa0e5ea00.svg "Ручка перетаскивания") рядом с именем правила и переместите правило вверх или вниз в списке, чтобы изменить его приоритет.

Вы также можете включить **Ignore custom errors in Apdex calculations**, чтобы переопределить настройки Apdex для отдельных правил пользовательских ошибок.

После добавления или исключения пользовательской ошибки в Dynatrace вы должны инициировать ошибку в вашем приложении с помощью метода `dtrum.reportCustomError()` из [RUM JavaScript API](customize-rum.md "Узнайте, как настроить мониторинг реальных пользователей с помощью JavaScript API.").

Например, ошибка поля формы потребует следующих параметров метода JavaScript:

| Параметр | Объяснение | Пример |
| --- | --- | --- |
| `key` | Имя поля формы. | `custom_buying_form_number_of_travelers_field` |
| `value` | Ошибка валидации, сгенерированная валидатором. | `availability exceeded - e3434` |
| `hint` | Фактический ввод пользователя. | `1000` |

Только `key` и `value` могут использоваться для группировки и анализа пользовательских ошибок; `hint` — необязательная информация.

Для получения дополнительной информации о том, как сообщать о пользовательских ошибках, см. [RUM JavaScript API - reportCustomError](https://www.dynatrace.com/support/doc/javascriptapi/doc/types/dtrum.html#reportcustomerror).

## Настройка ошибок JavaScript

Браузеры обнаруживают ошибки JavaScript автоматически, поэтому вам не нужно добавлять эти ошибки. Если вы настроите захват ошибок JavaScript, они автоматически включаются в расчёты Apdex и анализ Davis.

Чтобы игнорировать ошибки JavaScript на индивидуальном уровне, выберите **Ignore this JavaScript error** на соответствующей странице деталей ошибки JavaScript.

![Игнорирование ошибки JavaScript](https://dt-cdn.net/images/ignore-js-error-1217-3c39f37cfa.png)

Чтобы игнорировать все ошибки JavaScript в расчётах Apdex:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Errors** > **JavaScript errors**.
5. Включите **Ignore JavaScript errors in Apdex calculations**.

## Создание дополнительных оповещений об ошибках для Davis

Если встроенное оповещение по ошибкам запросов, пользовательским ошибкам и ошибкам JavaScript недостаточно чувствительно для вас, или если вы хотите сфокусироваться на отдельной ошибке, вы можете создать дополнительные оповещения об ошибках.

Чтобы создать дополнительные оповещения об ошибках:

1. Перейдите в **Web** и выберите приложение для отображения страницы обзора приложения.
2. На инфографике выберите **Errors**.
3. В разделе **Errors by type** выберите **Analyze by type**.
4. В разделе **Detail analysis for selected timeframe** установите фильтры в поле **Filter by...**. Доступные фильтры в категории **Errors**: `Custom error name`, `Custom error type`, `Error type`, `Errors`, `Request error code`, `Request error resource` и `Request error type`.
5. Выберите **Create metric**.
6. В открывшейся панели настройте параметры метрики и выберите **Create metric**.
   Подробнее о создании и построении графиков вычисляемых метрик см. [Создание вычисляемых метрик для веб-приложений](rum-calculated-metrics-web.md "Создавайте вычисляемые метрики и пользовательские графики на основе вычисляемых метрик для ваших веб-приложений.").
7. Выберите **Create alert**.
8. Настройте параметры оповещения. По завершении выберите **Create custom event for alerting**.

   Подробнее о создании оповещений см. [Метрические события](../../../../dynatrace-intelligence/anomaly-detection/metric-events.md "Узнайте о метрических событиях в Dynatrace").

   Используйте статический порог `1`, если хотите, чтобы Davis всегда создавал проблему при возникновении определённой ошибки. Если вас интересуют только аномалии для этой ошибки, используйте [автоматическое определение базовых линий](../../../../dynatrace-intelligence/anomaly-detection/automated-multidimensional-baselining.md "Узнайте, как Dynatrace AI автоматически рассчитывает базовые линии на основе схемы многомерного определения базовых линий.").
