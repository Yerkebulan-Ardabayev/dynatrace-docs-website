# Определение свойств действий пользователя и сессий для пользовательских приложений

Dynatrace собирает большой объём информации о производительности ваших приложений. Вы можете обогатить эту информацию ценными метаданными, а затем преобразовать метаданные в свойства действий пользователя и сессий.

Свойства действий и сессий — это пары ключ-значение метаданных, по которым вы можете фильтровать данные в аналитических представлениях Dynatrace. Эти свойства полезны, когда вам нужно создавать мощные запросы, сегментации или агрегации на основе собранных метаданных. Вы можете использовать эти свойства на страницах **User sessions** и **User sessions query**. Для более глубокого понимания того, как использовать эти свойства, см. [Использование свойств действий пользователя и сессий для пользовательских приложений](../analyze-and-use/action-and-session-properties-custom.md "Свойства действий пользователя и сессий, которые представляют собой пары ключ-значение метаданных, обеспечивают дополнительную видимость и более глубокий анализ опыта ваших конечных пользователей. Используя эти свойства для ваших приложений, вы можете фильтровать сессии пользователей, добавлять вычисляемые метрики, создавать диаграммы и многое другое.").

Ниже вы найдёте информацию о настройке таких свойств, а также соответствующие примеры конфигурации. Для использования свойств действий и сессий вам сначала нужно отправить необходимые метаданные в Dynatrace, а затем [добавить свойства в веб-интерфейсе Dynatrace](#add-properties).

## Отправка метаданных в Dynatrace

Прежде чем определять свойства действий и сессий, необходимо начать передачу необходимых метаданных в Dynatrace. Это можно сделать двумя способами:

* [Отправка пользовательских значений через SDK](#report-value-sdk)
* [Определение атрибутов запроса](#add-request-attribute)

### Отправка значения через SDK

Вы можете отправлять пары ключ-значение метаданных в исходном коде вашего пользовательского приложения через вызов API. Подробности см. в [Методы API Dynatrace OpenKit | Отправка значения](../../../../ingest-from/extend-dynatrace/openkit/dynatrace-openkit-api-methods.md#report-values "Узнайте, как Dynatrace OpenKit может использоваться с точки зрения разработчика.").

Чтобы проверить, доходят ли данные, отправляемые из кода вашего приложения, до Dynatrace, перейдите к деталям действия пользователя, которое должно содержать эти данные, и прокрутите до раздела **Reported values**.

![Страница деталей действия пользователя с отправленными через SDK значениями](https://dt-cdn.net/images/user-action-details-with-reported-values-2048-b44e8bca3e.png)

### Определение атрибута запроса

Дополнительно вы можете добавить необходимые метаданные к серверным запросам, определить атрибуты запросов, а затем использовать эти атрибуты для создания свойств действий и сессий.

Атрибуты запроса извлекаются из URL-адресов веб-запросов, заголовков HTTP-запросов и других метаданных запросов. Эти атрибуты представляют собой пары ключ-значение, по которым вы можете фильтровать данные в различных аналитических представлениях Dynatrace и представлениях распределённой трассировки.

Подробнее см. [Атрибуты запроса](../../../application-observability/services/request-attributes.md "Узнайте, что такое атрибуты запроса и как использовать их на всех уровнях всех представлений анализа сервисов.").

## Добавление свойств действий и сессий

После того как вы отправили значения через код приложения или определили атрибуты запроса, информация, собираемая Dynatrace, обогащается ценными метаданными. Вы можете «повысить» эти метаданные до свойств действий и сессий в настройках вашего приложения.

Вы можете сохранить метаданные на разных «уровнях хранения»:

* **Свойство действия пользователя**: данные сохраняются в определённом свойстве на уровне действия пользователя для каждого действия, из которого Dynatrace может извлечь метаданные.
* **Свойство сессии пользователя**: последнее полученное значение сохраняется в определённом свойстве на уровне сессии.
* **Оба варианта**: данные сохраняются в определённом свойстве на обоих уровнях — действия пользователя и сессии.

Чтобы определить свойство действия или сессии

1. Перейдите в **Frontend** и выберите приложение, которое вы хотите настроить.
2. Выберите **More** (**...**) > **Edit**.
3. В настройках приложения выберите **Session and user action properties** и нажмите **Add property**.
4. Выберите **Expression type** и, при необходимости, **Data type**.

   * Если вы добавили метаданные к запросам и [определили атрибуты запроса](#add-request-attribute), выберите **Server-side request attribute**.
   * Если вы [отправили пользовательские значения через SDK](#report-value-sdk), выберите **SDK reported value**.
5. Укажите **Name** или **Request attribute name**, **Display name** и **Key**.

   | Параметр | Описание |
   | --- | --- |
   | Name Request attribute name | Для типа значения, отправленного через SDK, это имя отправленного значения из кода вашего приложения. **Name** чувствителен к регистру, поэтому должен точно совпадать с записью в вашем коде. Для типа серверного атрибута запроса это имя атрибута запроса, заданное в **Settings** > **Server-side service monitoring** > **Request attributes**. |
   | Display name | Имя свойства, отображаемое в веб-интерфейсе Dynatrace, например на странице деталей сессии или странице деталей действия пользователя. |
   | Key | Имя свойства, используемое для идентификации и последующего поиска свойства в [USQL](../../session-segmentation/custom-queries-segmentation-and-aggregation-of-session-data.md "Узнайте, как обращаться к данным сессий пользователей и выполнять запросы на основе ключевых слов, синтаксиса, функций и многого другого."). |
6. Выберите хотя бы один [тип хранения](#storage-levels) — свойство действия пользователя, свойство сессии или оба.
7. Для типа хранения «свойство сессии» выберите один из типов агрегации.
8. Необязательно: для ограничения собираемых значений включите **Apply cleanup rule** и укажите [регулярное выражение](../../../../manage/tags-and-metadata/reference/regular-expressions-in-dynatrace.md "Узнайте, как использовать регулярные выражения в контексте Dynatrace.").

## Примеры свойств действий и сессий

Ниже приведены примеры нескольких свойств действий и сессий, которые мы настроили для нашего демонстрационного веб-приложения easyTravel.

Journey ID

Member status

Total purchase amount

Credit card

![Journey ID](https://dt-cdn.net/images/property-journey-id-2136-1e20f89c85.png)

Свойство **Journey ID**, содержащее значение типа данных `long`, добавляется к каждому действию пользователя и сессии пользователя в приложении easyTravel. Для сессии пользователя сохраняется значение последнего действия пользователя.

![Свойство Member status](https://dt-cdn.net/images/property-member-status-2136-1665ea176c.png)

Свойство **Member status** фиксирует статус участия в программе лояльности для всей сессии. В данном случае нас не интересуют отдельные действия пользователя, так как статус лояльности обычно одинаков на протяжении всей сессии.

![Свойство Purchase amount](https://dt-cdn.net/images/property-purchase-amount-2136-8a92eff1b1.png)

Свойство **Total purchase amount** представляет общую стоимость всех поездок, забронированных через наше приложение easyTravel. Обратите внимание, что это свойство сессии фактически является суммой значений `purchase_amount` из нескольких действий пользователя.

![Свойство Credit card type](https://dt-cdn.net/images/property-credit-card-type-2136-8a271f7d47.png)

**Credit card type** — это пример того, как мы «повысили» серверный атрибут запроса до свойств действия и сессии.

## Ограничения

* Вы можете определить максимум 200 свойств на приложение.
* Вы можете определить максимум 20 свойств действий на приложение.
* Свойства действий и сессий типа данных `String` ограничены 100 символами после применения правила очистки.
* Вы можете использовать до 20 свойств на приложение бесплатно. Дополнительные свойства потребляют единицы DEM.
  См. [Единицы DEM](../../../../license/monitoring-consumption-classic/digital-experience-monitoring-units.md#dem-units "Узнайте, как рассчитывается потребление Dynatrace Digital Experience Monitoring на основе единиц DEM.") (обратите внимание на записи **Session property** и **Action property** в таблице) и [Бесплатный уровень свойств действий и сессий](../../../../license/monitoring-consumption-classic/digital-experience-monitoring-units.md#free-action-and-session-properties "Узнайте, как рассчитывается потребление Dynatrace Digital Experience Monitoring на основе единиц DEM.") для получения дополнительной информации.

## Примечания

* Dynatrace начинает собирать свойства действий и сессий только после того, как вы [определите эти свойства в настройках вашего приложения](#add-properties).
* Вы можете проверить, сколько свойств вы уже используете и сколько ещё можете добавить. Перейдите в настройки вашего приложения > **Session and user action properties** и прокрутите до **Property usage quotas**.
