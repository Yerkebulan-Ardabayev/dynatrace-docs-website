* Latest Dynatrace

После настройки нового фронтенда с автоматическим внедрением графики в [![Experience Vitals](https://dt-cdn.net/images/experience-vitals-256-9999590b55.png "Experience Vitals") **Experience Vitals**](../../experience-vitals.md "Приложение Experience Vitals предоставляет точку входа для мониторинга веб- и мобильных фронтендов.") должны начать отображать данные в течение десяти минут, если ваш фронтенд получает трафик. Если этого не происходит, ваша настройка может потребовать дополнительных шагов конфигурации. Это руководство проведёт вас через серию проверок, которые помогут определить, нужна ли дополнительная настройка.

## Проверьте, не сопоставлены ли данные RUM с другим фронтендом

Фронтенд, с которым сопоставляются захваченные данные, определяется правилами обнаружения фронтенда, которые вы указали при настройке. Если заданные правила не могут сопоставить запросы HTML-документов с предполагаемым фронтендом, захваченные данные назначаются другому фронтенду с соответствующим правилом или — если ни одно правило не совпадает — [фронтенду-перехватчику](set-up-auto-injected-frontend.md#deploy-oneagent "Узнайте, как настроить веб-фронтенд с автоматическим внедрением в новом интерфейсе RUM.").

### Определите, были ли захвачены данные

Чтобы проверить, были ли данные захвачены и сопоставлены с другим фронтендом

1. Перейдите в ![Notebooks](https://dt-cdn.net/images/notebooks-768-046137830a.webp "Notebooks") **Notebooks** и откройте существующий блокнот или создайте новый, выбрав **Notebook**.
2. Выберите > **DQL**.
3. Выполните следующий запрос DQL, заменив `https://example.com/path` на URL страницы, которую вы инструментируете. Используйте URL в том виде, как он запрашивается браузером, без серверных перезаписей.

```
fetch user.events


| filter matchesPhrase(page.url.full, "https://example.com/path")


| filter matchesValue(dt.openpipeline.source, "rumagent")


| summarize by: {frontend.name}, count()
```

Если запрос возвращает пустой ответ, пользовательские события для этой страницы не были захвачены. В этом случае перейдите к разделу [Проверьте, что JavaScript RUM внедрён](#injection).

### Определите фронтенд, к которому были сопоставлены данные

Запрос возвращает количество [пользовательских событий](../../concepts/data-model.md#user-events "Ознакомьтесь с моделью данных, лежащей в основе нового интерфейса RUM."), захваченных для страницы, и включает поле `frontend.name`, которое указывает фронтенд, к которому были сопоставлены события. Чтобы определить, какому фронтенду соответствует возвращённое значение `frontend.name`

1. Перейдите в ![Experience Vitals](https://dt-cdn.net/images/experience-vitals-256-9999590b55.png "Experience Vitals") **Experience Vitals** > **Overview**.
2. Выберите **Web** для просмотра всех веб-фронтендов.
3. Выберите для отображения столбца **frontend.name**. Теперь вы можете найти значение `frontend.name` из результатов запроса в таблице.

### Определите причину

Когда данные RUM сопоставляются с другим фронтендом, чем ожидалось, это обычно вызвано одной из следующих причин:

* Использование идентификатора фрагмента в шаблоне правила, например, часть `#overview` в `https://example.com/docs#overview`. Для получения дополнительной информации о том, какие части URL поддерживаются, см. [Узнайте, как применяются правила обнаружения фронтенда](set-up-auto-injected-frontend.md#learn-application-of-detection-rules "Узнайте, как настроить веб-фронтенд с автоматическим внедрением в новом интерфейсе RUM.").
* Неучёт неинструментированных компонентов при настройке. Для получения дополнительной информации см. [Учёт неинструментированных компонентов](set-up-auto-injected-frontend.md#account-for-uninstrumented-components "Узнайте, как настроить веб-фронтенд с автоматическим внедрением в новом интерфейсе RUM.").
* Неправильный порядок правил обнаружения фронтенда. Для получения дополнительной информации см. [Установка порядка правил обнаружения фронтенда](set-up-auto-injected-frontend.md#set-order-of-rules "Узнайте, как настроить веб-фронтенд с автоматическим внедрением в новом интерфейсе RUM.").

## Проверьте, что JavaScript RUM внедрён

Чтобы убедиться, что JavaScript RUM внедрён в HTML вашей страницы, найдите элемент script, содержащий атрибуты `data-config`, `data-envconfig` и `data-appconfig`. Если вы нашли такой элемент, JavaScript RUM был внедрён, включая конфигурацию для нового интерфейса RUM. В этом случае перейдите к разделу [Проверьте, что код мониторинга загружается успешно](#monitoring-code).

Если элемент script с этими атрибутами отсутствует, проверьте наличие элемента script, содержащего строку `data-dtconfig`. Если вы его нашли, внедрение прошло успешно, но новый интерфейс RUM не активен. См. Включение нового интерфейса RUM для ваших веб-приложений RUM Classic, чтобы узнать, как его активировать.

![Внедрённый JavaScript RUM](https://dt-cdn.net/images/injected-rum-javascript-1046-e0d0d44dec.png)

Если вы не можете найти ни один такой элемент script, автоматическое внедрение не произошло.

### RUM отключён для браузеров с поддержкой «Do Not Track»

Если вы включили параметр **Turn Real User Monitoring off for "Do Not Track"-enabled browsers** в [настройках конфиденциальности данных](../additional-configuration/data-privacy-web.md#comply-with-dnt "Узнайте о доступных настройках, которые помогут вашим веб-фронтендам соответствовать требованиям конфиденциальности данных.") и ваш браузер настроен на включение заголовка `DNT`, OneAgent не внедряет JavaScript RUM. При проверке настройки сконфигурируйте ваш браузер так, чтобы он не включал заголовок `DNT`.

### Некорректный HTML

В некоторых случаях OneAgent может быть не в состоянии выполнить внедрение с помощью встроенных правил. Обычно это связано с некорректным HTML, хотя другие редкие условия также могут помешать набору правил определить подходящую точку внедрения. Для решения этой проблемы создайте [пользовательское правило внедрения](configure-auto-injection.md#custom-injection-rule "Настройте автоматическое внедрение JavaScript RUM на страницы ваших фронтендов в новом интерфейсе RUM."), которое внедряет код до или после определённого HTML-шаблона.

## Проверьте, что код мониторинга загружается успешно

Если вы не выбрали формат сниппета [встроенный код](snippet-formats.md#inline-code "Узнайте, как выбрать формат сниппета JavaScript RUM, который лучше всего подходит для вашего конкретного случая в новом интерфейсе RUM."), код мониторинга загружается как отдельный файл. В средах, где не настроен [пользовательский префикс имени файла кода мониторинга](configure-monitoring-code-source.md#configure-custom-monitoring-code-filename-prefix "Настройте источник кода мониторинга реальных пользователей в новом интерфейсе RUM для соответствия вашим конкретным требованиям."), имя файла начинается с одного из следующих префиксов:

* `ruxitagentjs_` — если запрос обрабатывается OneAgent, что является поведением по умолчанию.
* `ruxitagent_` — если вы настроили запрос кода мониторинга [из CDN Dynatrace](configure-monitoring-code-source.md#request-rum-monitoring-code-from-cdn "Настройте источник кода мониторинга реальных пользователей в новом интерфейсе RUM для соответствия вашим конкретным требованиям.").

Если инструменты разработчика браузера показывают ответ `200 OK` для этого запроса, загрузка прошла успешно и вы можете перейти к разделу [Проверьте, что RUM-маяки отправляются в Dynatrace](#beacons). В противном случае требуется дополнительная настройка.

![Успешная загрузка кода мониторинга RUM](https://dt-cdn.net/images/rum-monitoring-code-successful-download-1348-feb695c704.png)

### Нарушения правил CSP

Если вы видите нарушения правил CSP в консоли браузера, связанные с загрузкой кода мониторинга, скорректируйте правила CSP, определённые в вашем приложении. Подробности см. в разделе [Разрешение загрузки внешнего кода мониторинга](adapt-csp-rules.md#allow-external-monitoring-code "Узнайте, как адаптировать правила CSP для нового интерфейса RUM.").

### Нет приложения, развёрнутого в корне

Когда OneAgent обрабатывает запросы кода мониторинга, они часто [отправляются в корень веб- или app-сервера](configure-monitoring-code-source.md#default-behavior-auto-injection "Настройте источник кода мониторинга реальных пользователей в новом интерфейсе RUM для соответствия вашим конкретным требованиям."). Эти запросы могут возвращать коды ответа, такие как `404 Not Found` или `503 Service Unavailable`, если в корне не развёрнуто приложение.

В этом случае настройте путь URL, как описано в разделе [Изменение пути URL кода мониторинга](configure-monitoring-code-source.md#modify-rum-monitoring-code-url-path "Настройте источник кода мониторинга реальных пользователей в новом интерфейсе RUM для соответствия вашим конкретным требованиям.").

### Запросы блокируются компонентами инфраструктуры

Запросы кода мониторинга могут блокироваться компонентами инфраструктуры, такими как межсетевые экраны и прокси-серверы.

В зависимости от конфигурации компонента, часто можно предотвратить блокировку запросов, выбрав другую конфигурацию источника кода мониторинга. В противном случае убедитесь, что ваша инфраструктура пропускает запросы кода мониторинга; см. Ограничения межсетевого экрана для RUM.

## Проверьте, что RUM-маяки отправляются в Dynatrace

JavaScript RUM отправляет маяки (beacons), содержащие захваченные данные, на конечную точку маяков. По умолчанию конечная точка маяков для фронтенда с автоматическим внедрением — это OneAgent, который инструментирует веб- или app-сервер, на котором размещено приложение. Если новый интерфейс RUM активен, маяки в новом формате отправляются вместе с маяками RUM Classic на ту же конечную точку.

Чтобы определить RUM-маяки в инструментах разработчика вашего браузера, ищите запросы, у которых последний сегмент пути URL имеет префикс `rb_`. Маяки в новом формате включают параметр строки запроса `pv=4`.

![RUM-маяки в новом формате](https://dt-cdn.net/images/rum-beacon-new-format-1410-749d442eca.png)

### Маяки не отправляются

Если инструменты разработчика браузера не показывают маяки, проверьте следующие аспекты вашей настройки:

* Если вы включили [режим сбора данных и согласия](../additional-configuration/data-privacy-web.md#data-collection-and-opt-in-mode "Узнайте о доступных настройках, которые помогут вашим веб-фронтендам соответствовать требованиям конфиденциальности данных.") во время настройки, ваш код должен вызвать [`dtrum.enable()`](https://www.dynatrace.com/support/doc/javascriptapi/doc/types/dtrum.html#enable) из JavaScript API после того, как пользователь примет вашу политику конфиденциальности данных. Только тогда JavaScript RUM начнёт отправлять данные.
* Если вы выбрали формат сниппета [встроенный код](snippet-formats.md#inline-code "Узнайте, как выбрать формат сниппета JavaScript RUM, который лучше всего подходит для вашего конкретного случая в новом интерфейсе RUM."), консоль вашего браузера может отображать нарушение правил CSP, указывающее, что встроенный скрипт был заблокирован. В этом случае скорректируйте правила CSP, как описано в разделе [Разрешение выполнения встроенного кода](adapt-csp-rules.md#allow-the-execution-of-inline-code "Узнайте, как адаптировать правила CSP для нового интерфейса RUM.").

### Маяки заблокированы из-за нарушений правил CSP

Если вы видите нарушения правил CSP в консоли, указывающие, что соединение с конечной точкой маяков было заблокировано, скорректируйте правила CSP в вашем приложении. Подробности см. в разделе [Разрешение отправки RUM-маяков](adapt-csp-rules.md#allow-sending-rum-beacons "Узнайте, как адаптировать правила CSP для нового интерфейса RUM.").

### Нет приложения, развёрнутого в корне

Когда OneAgent обрабатывает RUM-маяки, они часто [отправляются в корень веб- или app-сервера](configure-beacon-endpoint.md#default-endpoint-auto-injection "Узнайте, как настроить конечную точку маяков для веб-фронтендов в соответствии с вашими конкретными требованиями."). Эти запросы могут возвращать коды ответа, такие как `404 Not Found` или `503 Service Unavailable`, если в корне не развёрнуто приложение.

В этом случае настройте путь URL, как описано в разделе [Изменение пути URL конечной точки маяков](configure-beacon-endpoint.md#modify-beacon-endpoint-url-path "Узнайте, как настроить конечную точку маяков для веб-фронтендов в соответствии с вашими конкретными требованиями.").

### Запросы блокируются компонентами инфраструктуры

Запросы маяков могут блокироваться компонентами инфраструктуры, такими как межсетевые экраны и прокси-серверы. Вы можете определить, где был обработан запрос маяка, просмотрев тело ответа — запросы маяков, которые не были обработаны конечной точкой маяков, возвращают JSON-ответ, содержащий поля `status`, `handler`, `errorCode` и `errorReason`, тогда как маяки, перехваченные межсетевым экраном или аналогичным компонентом, не содержат этого JSON-ответа.

В зависимости от конфигурации этих компонентов, вы часто можете обеспечить доставку запросов до конечной точки маяков, выбрав другую конфигурацию конечной точки маяков. В противном случае убедитесь, что ваша инфраструктура пропускает RUM-маяки; см. Ограничения межсетевого экрана для RUM.
