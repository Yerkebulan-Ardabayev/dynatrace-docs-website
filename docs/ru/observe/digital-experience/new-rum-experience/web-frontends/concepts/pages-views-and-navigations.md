# Страницы, представления и навигации в New RUM Experience


* Latest Dynatrace
* Описание
* Обновлено 08 января 2026 г.

New RUM Experience использует концепции страниц, представлений и навигаций, чтобы предоставить вам детальную видимость поведения вашего приложения и помочь повысить производительность — независимо от того, работаете ли вы с традиционным веб-сайтом или одностраничным приложением (SPA).

## Страницы

В New RUM Experience концепция страницы соответствует техническому определению браузера: она представляет запрос документа, отображаемое содержимое и все события, происходящие в течение её жизненного цикла — до следующего запроса документа.

Для традиционного веб-сайта страницы позволяют анализировать каждую полностраничную навигацию. Для SPA страницы фиксируют начальную загрузку, когда пользователь впервые открывает приложение, тогда как [представления](#views) дают полную картину при динамическом изменении контента.

### Страницы и экземпляры страниц

Каждое посещение веб-сайта создаёт **экземпляр страницы**. Собранные данные агрегируются в [событие сводки страницы](../../../../../semantic-dictionary/model/rum/user-events/navigation-related.md#page-summary), которое идентифицируется по его [`page.instance_id`](../../../../../semantic-dictionary/model/rum/user-events.md#page-attributes "Пользовательские события обеспечивают глубокую видимость и аналитику опыта, поведения, производительности и ошибок ваших клиентов и конечных пользователей в режиме реального времени.").

**Страницы** группируют все посещения в одну агрегированную запись. Для анализа одной и той же страницы по всем посещениям экземпляры страниц группируются по `page.name`. Атрибут `page.name` предназначен для фильтрации динамических частей URL для лучшей группировки. Он формируется из `page.detected_name`, который RUM JavaScript определяет на основе URL.

**Загрузка страницы** каждого нового экземпляра страницы фиксируется как событие со следующими тремя [характеристиками](../../../../../semantic-dictionary/model/rum/user-events.md#user-event-characteristics "Пользовательские события обеспечивают глубокую видимость и аналитику опыта, поведения, производительности и ошибок ваших клиентов и конечных пользователей в режиме реального времени."):

* `characteristics.has_request`
* `characteristics.has_navigation`
* `characteristics.has_w3c_navigation_timings`

### Google Web Vitals

[Google Web Vitals](https://web.dev/articles/vitals) критически важны для понимания пользовательского опыта на странице. Эти метрики фиксируются с каждым событием сводки страницы. Подробнее см. [атрибуты Web Vitals](../../../../../semantic-dictionary/model/rum/user-events/navigation-related.md#page-summary-web-vitals-attributes) в Semantic Dictionary. Поскольку официальная спецификация Google Web Vitals основана на страницах, встроенные метрики Web Vitals в New RUM Experience базируются на значениях в сводках страниц, хотя Web Vitals также фиксируются в [событиях сводки представлений](#views).

Встроенные метрики для Web Vitals

### Пример запроса DQL

Следующий запрос DQL получает 75-й перцентиль для метрики Largest Contentful Paint (LCP) Web Vital по страницам.

```
fetch user.events


| filter characteristics.has_page_summary


| filter isNotNull(web_vitals.largest_contentful_paint)


| summarize {


percentile(web_vitals.largest_contentful_paint, 75)


}, alias:page_LCP, by:{ page.name }


| sort page_LCP desc
```

## Представления

На традиционном веб-сайте каждая навигация инициирует полную загрузку страницы, что позволяет легко отслеживать отдельные страницы и их Web Vitals, предоставляя данные, необходимые для оптимизации сайта. Однако SPA работают иначе. Обычно они включают одну начальную загрузку страницы, а затем обновляют контент динамически без перезагрузки всей страницы. Это требует другого подхода к сегментации и агрегации данных для оптимизации отдельных частей приложения.

New RUM Experience автоматически интерпретирует мягкие навигации (также называемые сменами маршрутов) как переходы к новому представлению. Представление — это отображаемый контент и все события, происходящие между двумя навигациями. Эта концепция предоставляет значимые данные о производительности для SPA.

### Представления и экземпляры представлений

Каждая навигация создаёт новый **экземпляр представления**, будь то мягкая навигация или жёсткая навигация с полной загрузкой страницы. Начало нового экземпляра страницы всегда означает начало нового экземпляра представления.

Собранные данные для экземпляра представления агрегируются в [событие сводки представления](../../../../../semantic-dictionary/model/rum/user-events/navigation-related.md#view-summary), идентифицируемое по его [`view.instance_id`](../../../../../semantic-dictionary/model/rum/user-events.md#view-attributes "Пользовательские события обеспечивают глубокую видимость и аналитику опыта, поведения, производительности и ошибок ваших клиентов и конечных пользователей в режиме реального времени.").

**Представления** группируют все посещения в одну агрегированную запись. Для анализа одного и того же представления по нескольким посещениям экземпляры представлений группируются по `view.name`. Этот атрибут предназначен для фильтрации динамических частей URL для лучшей группировки. Он формируется из `view.detected_name`, который RUM JavaScript определяет на основе URL.

В течение жизненного цикла страницы атрибут `view.sequence_number` увеличивается с каждой навигацией к новому представлению.

Концептуально, если страница загружается и до следующей загрузки страницы не происходит мягких навигаций, значения в событии сводки представления и событии сводки страницы идентичны. Когда одна или несколько мягких навигаций происходят в течение жизненного цикла экземпляра страницы, значения в сводках страниц и представлений расходятся.

### Пример экземпляров страниц и представлений

Следующая диаграмма показывает экземпляры страниц и представлений, создаваемые во время посещения веб-сайта, включающего как жёсткие, так и мягкие навигации.

![Страницы и экземпляры страниц](https://dt-cdn.net/images/page-instances-4210-69ed8aa0e8.png)

* Когда пользователь посещает стартовую страницу `/` веб-сайта, фиксируются следующие атрибуты:

  + `page.instance_id` — случайное значение, идентифицирующее данный экземпляр страницы.
  + `page.detected_name` — имя, определённое RUM JavaScript.
  + `page.name` — имя, используемое бэкендом Dynatrace для группировки всех экземпляров.
* Когда пользователь переходит на страницу товаров `/products` с помощью жёсткой навигации:

  + `page.instance_id` и `view.instance_id` — вновь сгенерированные случайные значения.
  + `view.sequence_number` сбрасывается до 1.
* Когда пользователь просматривает два отдельных товара с помощью мягких навигаций:

  + `page.instance_id` остаётся прежним.
  + `view.sequence_number` увеличивается с каждой мягкой навигацией.
  + `view.instance_id` генерируется заново для каждой мягкой навигации.
  + `view.detected_name` содержит меняющийся идентификатор (`12345` и `56789`).
  + `view.name` заменяет этот динамический идентификатор на `:id:` для лучшей группировки.
* Когда пользователь возвращается на стартовую страницу `/` с помощью жёсткой навигации:

  + `page.instance_id` и `view.instance_id` — вновь сгенерированные случайные значения.
  + `view.sequence_number` сбрасывается до 1.

### Google Web Vitals

New RUM Experience не просто фиксирует Web Vitals для страниц, но также фиксирует их для представлений внутри страницы. Это особенно важно для SPA.

Например, представьте страницу с галереей изображений. Начальное представление показывает обзор с маленькими миниатюрами. Когда пользователь переходит к детальному просмотру изображения, начинается новое представление.

* Первая сводка представления содержит значение Interaction to Next Paint (INP) для начальной загрузки страницы и для кликов по маленьким миниатюрам.
* Последующие события сводки представлений содержат значение INP для детальных представлений, где пользователь взаимодействует с более крупными изображениями.
* Событие сводки страницы содержит значение INP для всех кликов, происходящих по всем навигированным представлениям.

### Пример запроса DQL

Следующий запрос получает 75-й перцентиль для LCP по представлениям.

```
fetch user.events


| filter characteristics.has_view_summary


| filter isNotNull(web_vitals.largest_contentful_paint)


| summarize {


percentile(web_vitals.largest_contentful_paint, 75)


}, alias:view_LCP, by:{ view.name }


| sort view_LCP desc
```

## Навигации

[События навигации](../../../../../semantic-dictionary/model/rum/user-events/navigation-related.md#navigation) генерируются при смене представлений и/или страниц. Способ их генерации зависит от того, является ли навигация жёсткой (полная загрузка страницы) или мягкой (смена маршрута SPA).

### Жёсткие навигации

Для жёсткой навигации или полной загрузки страницы событие навигации включает запрос документа и имеет, среди прочего, следующие характеристики:

* `characteristics.has_navigation`
* `characteristics.has_w3c_navigation_timings`

Эти события также содержат полные данные [W3C Navigation Timing](https://www.w3.org/TR/navigation-timing-2/) из поддерживающих браузеров.

### Мягкие навигации

Для мягкой навигации генерируется отдельное событие навигации с характеристикой `characteristics.has_navigation`. Оно включает основную информацию о навигации, например:

* `navigation.type`, который может быть `hard` или `soft`.
* Информация о предыдущем представлении, такая как `view.source.name` и `view.source.url.full`.
* Информация о новом представлении, такая как `view.name` и `view.url.full`.

### Пример жёстких и мягких навигаций

Приведённая ниже диаграмма иллюстрирует, как сводки представлений и страниц генерируются во время жёстких и мягких навигаций, а также соответствующие события навигации.

![Страницы, представления и навигации](https://dt-cdn.net/images/pages-views-navigations-3296-b049928613.png)

* Когда загружается новая страница (жёсткая навигация), отправляется событие навигации с характеристикой `has_w3c_navigation_timing`.
* Когда происходит мягкая навигация, отправляются сводка предыдущего представления и событие навигации для нового представления.
* Когда страница покидается, сводка представления и сводка страницы отправляются вместе.

### Пример запроса DQL

Следующий запрос DQL получает количество жёстких и мягких навигаций.

```
fetch user.events


| filter characteristics.has_navigation


| summarize count(), by: navigation.type
```
