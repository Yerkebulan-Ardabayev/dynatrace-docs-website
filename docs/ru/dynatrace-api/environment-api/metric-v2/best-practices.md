# Metrics API v2 — Рекомендации


На этой странице приведены рекомендации по настройке и пересмотру метрик.

## Обдумайте необходимые измерения

Каждая метрика поддерживает измерения для обогащения метрики контекстом, например указанием места измерения. Затем эти измерения можно фильтровать и агрегировать во время запроса.

Свои измерения можно просматривать при использовании [протокола приёма метрик](../../../ingest-from/extend-dynatrace/extend-metrics/reference/metric-ingestion-protocol.md "Узнайте, как работает протокол приёма данных для Dynatrace Metrics API.").

### Что такое кортеж измерений?

Комбинация измерений называется **кортежем измерений**.

### Примеры кортежей измерений

В следующем примере создаётся метрика с двумя измерениями: `host.name` и `owner`.

```
cpu.temperature, host.name="Alice's PC",           owner="Alice"  gauge,50 1699625420


cpu.temperature, host.name="Alice's Raspberry Pi", owner="Alice"  gauge,40 1699625420


cpu.temperature, host.name="Bob's PC",             owner="Bob"    gauge,60 1699625420
```

Приведённые выше данные создают следующие кортежи измерений.

```
(host.name="Alice's PC",           owner="Alice")


(host.name="Alice's Raspberry Pi", owner="Alice")


(host.name="Bob's PC",             owner="Bob")
```

## Избегайте нестабильных измерений

Выбор правильных измерений для метрики необходим для максимальной эффективности использования Dynatrace. Нестабильные измерения могут давать нежелательные результаты, увеличивать затраты и снижать производительность запросов.

### Что такое нестабильное измерение?

Кратковременные идентификаторы и метки времени являются нестабильными измерениями. Если значение вашего измерения является UUID, меткой времени эпохи или числовым значением, подумайте, действительно ли оно необходимо для вашего сценария использования.

### Пример нестабильного измерения

Следующий пример представляет собой распространённую ошибку конфигурации с использованием синтаксиса из протокола приёма метрик:

```
cpu.temperature, host.name="Alice's PC", measured_at="1699625420"  gauge,50 1699625420


cpu.temperature, host.name="Alice's PC", measured_at="1699625421"  gauge,50 1699625421


cpu.temperature, host.name="Alice's PC", measured_at="1699625422"  gauge,49 1699625422


cpu.temperature, host.name="Alice's PC", measured_at="1699625423"  gauge,49 1699625423
```

Измерение `measured_at` использует метку времени эпохи для отслеживания момента измерения данных.

* Это излишне, поскольку метка времени уже предоставлена; `measured_at` лишь дублирует её.
* Поскольку с каждой новой точкой данных создаётся новый кортеж измерений, будет достигнут предел мощности после 1 миллиона секунд работы — всего за 11 дней.

## Избегайте высокой мощности измерений

Для смягчения негативных последствий нестабильных измерений Dynatrace устанавливает ограничения мощности.

### Что такое ограничение мощности?

Ограничение мощности — это предел на количество кортежей измерений, которые можно принять.

В Dynatrace классические метрики имеют несколько ограничений кортежей измерений:

Ограничения применяются к кортежам измерений за последние 30 дней (скользящее окно 30 дней). При достижении ограничения точки данных, имеющие новые и ещё не встречавшиеся кортежи измерений, отклоняются. Ограничения применяются к тенанту.

### Выявление ситуаций высокой мощности

Dynatrace предоставляет метрики самомониторинга для получения сведений о текущем использовании пространства кортежей измерений. Эти метрики отображаются на готовой панели управления **Metric & Dimension Usage + Rejections**.

В приведённом ниже примере обратите внимание на следующие плитки:

* **Custom metric dimensions usage** (`0.87%`)

  Это приблизительно 435 000 кортежей измерений пользовательских метрик за последние 30 дней, так как предел кортежей измерений пользовательских метрик составляет 50 миллионов.
* **Builtin metric dimensions usage** (`1.68%`)

  Встроенные метрики ограничены 20 миллионами кортежей, поэтому 1.68% означает наличие около 336 000 кортежей измерений.
* **Highest metric dimensions usage** (нет данных в примере)

  На этой плитке отображаются отдельные метрики, превышающие 50% предела для одной метрики.
* **High-cardinality dimensions** (нет данных в примере)

  На этой плитке отображается таблица измерений с наибольшей мощностью для ваших метрик за последние 30 дней. Включаются только измерения с мощностью, равной или превышающей 10% предела. Если ваша пользовательская метрика достигает предела измерений, рассмотрите удаление или оптимизацию наиболее нестабильных измерений.

### Расследование метрик с высокой мощностью

Если вы близки к достижению пределов мощности:

1. Проверьте, является ли объём принятых данных разумным.

   Не перегружает ли задание cron тестовой среды или другой ресурс процесс приёма? Возможно, первопричина — долго работающий ресурс, который так и не был отключён.
2. Проанализируйте, какие метрики отклоняются.

   Актуальна ли метрика по-прежнему и соответствует ли её размерность сценариям использования?
3. Оцените относительную важность частей вашей среды.

   Вы лучше всех знаете свою среду. Не все части среды одинаково важны. Опять же, соответствует ли размерность метрики сценариям использования?
