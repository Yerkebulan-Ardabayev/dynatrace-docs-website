# Заметки о выпуске Dynatrace Managed версии 1.314

* Заметки о выпуске
* Updated on Jul 28, 2025

Начало развёртывания: May 12, 2025

На этой странице представлены новые функции, изменения и исправления ошибок в Dynatrace Managed версии 1.314.

## Новые функции и улучшения

Dynatrace Cluster

### Обеспечение соответствия требованиям с помощью Software Bill of Materials (SBOM) для Dynatrace Managed

Теперь можно обеспечить соответствие нормативным требованиям, сканируя SBOM Dynatrace Managed на наличие уязвимостей.

SBOM содержит все сторонние библиотеки, используемые Dynatrace Managed, и их версии.

* Новый SBOM предоставляется с каждой версией. Он входит в состав установщика Managed и соответствует стандарту CycloneDX.
* Имя файла SBOM:

  `/opt/dynatrace-managed/installer/dynatrace-managed-<version>/dynatrace-managed-sbom.cdx.json`

Dynatrace Cluster

### Уменьшение использования диска за счёт смены алгоритма сжатия данных временных рядов

В рамках данного выпуска формат сжатия данных временных рядов в Cassandra будет переключён с LZ4 на ZSTD. По результатам нагрузочного тестирования это может значительно снизить требуемое дисковое пространство.

Примечание: снижение использования диска не будет мгновенным, а будет происходить постепенно в течение месяца, так как новый алгоритм сжатия применяется только к новым данным.

Вмешательство пользователя или время простоя не требуются; переключение происходит автоматически в рамках обычных обновлений версии.

Dynatrace Cluster

### Ускоренное назначение тегов, управленческих зон и правил именования

Новый периодический рабочий процесс применяет теги, управленческие зоны и правила именования к недавно созданным отслеживаемым сущностям быстрее. Этот рабочий процесс выполняется непрерывно и учитывает все отслеживаемые сущности, созданные за последние 30 минут и обнаруженные за последние 2 часа.

По умолчанию он автоматически включён для всех окружений Dynatrace Managed. Дополнительная настройка не требуется.

Dynatrace Cluster

### Обновление JDK до версии 21 для Dynatrace Cluster

В рамках данного выпуска узлы Dynatrace Cluster Server будут обновлены для работы с JDK 21 вместо JDK 17.

Вмешательство пользователя или время простоя не требуются; обновление должно выполняться посредством поэтапного (rolling) обновления в рамках обычных обновлений версии.

Внутреннее тестирование показало значительное сокращение времени, затрачиваемого на сборку мусора Java, поэтому обновление может привести к улучшению производительности процесса Dynatrace Cluster Server.

Digital Experience | Synthetic

### Отчётность о недействительных сертификатах включена по умолчанию для новых HTTP-мониторов

Значение свойства `Accept any SSL certificate` для запроса HTTP-монитора по умолчанию изменено на `false`, то есть SSL-сертификат будет проверяться для новых мониторов.

Application Observability

### Улучшенное управление вычисляемыми метриками сервисов

Управление и просмотр вычисляемых метрик сервисов теперь проще и эффективнее — особенно для окружений с большим количеством метрик. Новые возможности в настройках вычисляемых метрик сервисов:

* **Фильтрация по статусу**: Быстро определяйте включённые или отключённые вычисляемые метрики сервисов.
* **Удобная сортировка**: Сортируйте метрики по их статусу или включению Grail.
* **Просмотр всех метрик**: Пагинация позволяет просматривать все вычисляемые метрики сервисов без ограничений.

Эти улучшения экономят время, обеспечивают лучшую видимость вычисляемых метрик сервисов и делают управление ими более удобным.

Platform | Davis

### Изменение задержки анализа первопричин

События, не запускающие полный анализ первопричин, больше не будут ждать 3 минуты перед уведомлением. Основные случаи — это события с пользовательским запуском, например `AVAILABILITY_EVENT` через REST API.

Кроме того, `dt.davis.analysis_trigger_delay` теперь приоритетнее внутреннего трёхминутного значения по умолчанию для некоторых событий типа доступности. Если `dt.davis.analysis_trigger_delay` установлен, только он учитывается для анализа первопричин.

Digital Experience

### Синтетические локации без браузера снижают аппаратные требования для NAM- и HTTP-мониторов

В общем случае рекомендуется развёртывание полных синтетических приватных локаций для поддержки выполнения всех типов синтетических мониторов (HTTP, браузерных, NAM).

Если не требуется запуск браузерных мониторов, можно рассмотреть развёртывание локации в режиме без браузера. В этом режиме локация (или ActiveGate, принадлежащий ей) развёртывается без браузера, что снижает аппаратные требования. Однако браузерные мониторы не могут выполняться на локации без браузера.

Рассматривайте локации без браузера как альтернативу стандартным синтетическим приватным локациям, когда вы сосредоточены исключительно на:

* Сетевых и инфраструктурных сценариях (с использованием NAM-мониторов)
* Мониторинге API (с использованием HTTP-мониторов)
