# Планирование обновления развёртывания

* Published Apr 24, 2023

На этапе планирования вы собираете требования, адаптируетесь к различиям, определяете объём работ и выбираете подход к переходу на SaaS. Результатом этапа планирования должен стать детальный план обновления до развёртывания SaaS и хорошо проработанная архитектура целевого решения в рамках платформы Dynatrace.

## Шаг 1 — Анализ

Основные цели этого шага:

* Оценить текущую ситуацию
* Определить проблемы и собрать требования
* Учесть требования к безопасности, конфиденциальности и соответствию нормативным требованиям
* Оценить осуществимость перехода на SaaS

В рамках данных целей необходимо рассмотреть следующие темы:

### Оценка текущих потребностей

Как описано в разделе [Информация перед обновлением](/managed/upgrade/up-information "Get essential information about upgrading to Dynatrace SaaS and related concepts."), необходимо ознакомиться с [различиями между SaaS и Managed](/managed/upgrade/up-information#saas-versus-managed-differences "Get essential information about upgrading to Dynatrace SaaS and related concepts.") до перехода на развёртывание SaaS. Для каждого различия, требующего дополнительных действий, создайте пункт в документе плана обновления.

### Анализ требований к безопасности, конфиденциальности и соответствию нормативным требованиям

Для обеспечения ценности для клиентов Dynatrace собирает большие объёмы данных, зачастую включающих конфиденциальную и персональную информацию пользователей. Dynatrace понимает риски, связанные с безопасностью и конфиденциальностью данных, и может обобщить свой подход следующим образом:

#### Безопасность данных

Подход, ориентированный на конкретные сценарии использования, позволяет настраивать Dynatrace в соответствии с вашими требованиями соответствия нормативным требованиям, включая соответствие GDPR и CCPA. Вы можете исключить конфиденциальные данные, включая персональные, ещё на этапе захвата — через простую настройку и надёжные параметры по умолчанию для автоматического исключения данных, включая IBAN и номера платёжных карт для соответствия PCI-DSS.

Можно также выбрать обработку конфиденциальных данных в кластере для получения первоначального анализа без хранения данных в кластере — данные исключаются на этапе хранения, что обеспечивается простой настройкой и полезными параметрами по умолчанию.

Кроме того, вы имеете детальный контроль доступа к данным и возможность автоматизировать прозрачность событий доступа через API журнала аудита, что позволяет интегрировать события журнала аудита с внутренними решениями SIEM (управление информацией и событиями безопасности), а также через API управления разрешениями для автоматизации ограничений доступа в ответ на сигналы SIEM.

Для изучения деталей ознакомьтесь со следующими страницами:

* [Меры безопасности данных](/managed/manage/data-privacy-and-security/data-security/data-security-controls "Learn about data security and operational security controls.")
* [Меры безопасной разработки](/managed/manage/data-privacy-and-security/data-security/secure-development-controls "Learn how we ensure complete security for all Dynatrace software components and development practices.")

#### Конфиденциальность данных

Принципы privacy-by-design и privacy-by-default позволяют настраивать Dynatrace для максимальной ценности при соблюдении требований законодательства. Изучите и составьте список действий, которые необходимо предпринять в отношении следующих аспектов:

* Конфиденциальность данных
* API конфиденциальности данных
* Настройки на уровне среды
* Session Replay
* Real User Monitoring

Информация по каждой из этих тем доступна в [Центре доверия Dynatrace](https://www.dynatrace.com/company/trust-center/). В частности, рекомендуется ознакомиться со следующими темами:

* [Персональные данные, собираемые Dynatrace](/managed/manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace "Find out what types of end-user data may be captured during Dynatrace monitoring and the methods that are available for masking personal end-user data.")
* [Уровни защиты данных](/managed/manage/data-privacy-and-security/data-privacy/levels-of-data-protection "Learn how Dynatrace protects end-user information by applying situation-dependent levels of protection.")

#### Периоды хранения данных

Развёртывание Dynatrace Managed обеспечивает гибкость настройки периодов хранения при соответствующих затратах на инфраструктуру. В Dynatrace SaaS стандартные защитные параметры хранения оптимизированы и применяются по умолчанию, однако каждый из них может быть настроен индивидуально.

### Защитные параметры платформы

Если ранее в вашей среде Dynatrace Managed были увеличены какие-либо лимиты, детальная проверка периодов хранения и лимитов поможет выявить защитные параметры, которые могут превышать настройки по умолчанию в вашей среде Dynatrace SaaS.

Для каждого защитного параметра Dynatrace определяет значение по умолчанию и жёсткий лимит. В SaaS настройки могут быть увеличены выше значения по умолчанию, вплоть до жёсткого лимита, однако такие увеличения могут повлечь дополнительные лицензионные расходы в связи с более высоким потреблением хранилища или вычислительных ресурсов.

Если необходимо увеличить лимиты хранения данных выше стандартного лимита SaaS, рекомендуется заблаговременно согласовать это с командой Dynatrace и прояснить возможные последствия запроса на увеличение лимитов.

Лимиты, как правило, не могут быть увеличены выше жёсткого лимита. Если требуется превысить жёсткий лимит, обратитесь в [службу поддержки Dynatrace](https://www.dynatrace.com/services-support/).

### Переменные привязки SQL

Переменные привязки требуют высокой пропускной способности сети и больших объёмов хранилища, поэтому их захват не включён по умолчанию. Информацию об условиях доступности в развёртывании Dynatrace SaaS и о начале работы см. в разделе [Поддержка переменных привязки SQL](/managed/observe/infrastructure-observability/databases/database-services-classic/support-for-sql-bind-variables "Learn how you can enable Dynatrace OneAgent to capture the values of bind variables.").

### Лимиты API

Dynatrace API можно использовать для автоматизации задач мониторинга и экспорта различных типов данных в сторонние инструменты отчётности и анализа. Различий в обработке лимитов API-запросов между Dynatrace Managed и SaaS нет. См. раздел [API Dynatrace — лимиты доступа](/managed/dynatrace-api/basics/access-limit "Find out about payload limits and request throttling that may affect your use of the Dynatrace API."). Если регулирование API-запросов является критичным или кластер Dynatrace Managed специально рассчитан на обработку большого объёма API-вызовов, обратитесь к команде Dynatrace.

### Обновления программного обеспечения Dynatrace

В Dynatrace SaaS все обновления программного обеспечения кластера управляются Dynatrace, автоматически обеспечивая наиболее безопасную и оптимизированную версию Dynatrace, а также доступ ко всем последним достижениям, включая функции раннего доступа. Обновления SaaS выполняются автоматически без перерывов или простоев в мониторинге и разработаны так, чтобы быть незаметными. Кластеры Dynatrace SaaS обновляются примерно раз в две недели. При автоматическом обновлении кластера вы сохраняете полный контроль над [настройками обновления OneAgent](/managed/ingest-from/dynatrace-oneagent/oneagent-update "Learn how to update OneAgent.") и [ActiveGate](/managed/ingest-from/dynatrace-activegate/operation/update-activegate "Learn how to find out which version of ActiveGate you have installed and how you can download and install the latest version.").

### Adaptive Traffic Management

Одной из ключевых возможностей Dynatrace является автоматический захват распределённых трейсов. Dynatrace защищает производительность и доступность платформы, проактивно управляя частотой захвата трафика. Метод Adaptive Traffic Management в Dynatrace SaaS отличается от Dynatrace Managed: он адаптируется и настраивается автоматически с меньшей зависимостью от архитектуры отдельного отслеживаемого приложения. Защитные параметры применяются к общему объёму, передаваемому в среду Dynatrace, позволяя лимитам масштабироваться с лицензией и разрешая точкам входа с высоким трафиком передавать больше данных. Для получения подробной информации ознакомьтесь с разделом [Переход на Dynatrace SaaS](https://docs.dynatrace.com/docs/shortlink/up-upgrade) и обратите внимание на указанные лимиты.

### Предотвращение перегрузки RUM

Вы можете контролировать количество захватываемых пользовательских действий в веб-, мобильных и OpenKit-приложениях, задав максимальное количество пользовательских действий в минуту. По достижении этого лимита Dynatrace автоматически ограничивает количество захватываемых пользовательских сессий до возврата числа пользовательских действий в рамки лицензии. Запланируйте настройку предотвращения перегрузки RUM в соответствии с вашими требованиями в среде Dynatrace SaaS. Подробнее см. в статье [RUM: что означает сообщение 'Max. user actions per minute exceeded'?](https://dt-url.net/h92389d) в сообществе Dynatrace.

При необходимости дополнительной информации для обеспечения охвата аспектов безопасности, конфиденциальности, соответствия нормативным требованиям и управления данными обратитесь в отдел продаж Dynatrace.

## Шаг 2 — Стратегия

Основные цели этого шага:

* Оценить объём перехода на Dynatrace SaaS
* Понять трудозатраты, необходимые для перехода на Dynatrace SaaS
* Оценить свои возможности и компетенции
* Принять решение о самостоятельном обновлении или с привлечением Dynatrace ACE Services
* Разработать план перехода на Dynatrace SaaS

В ходе этого шага необходимо рассмотреть следующие темы:

### Миграция данных

Dynatrace хранит три основных типа данных:

* Данные конфигурации
* Учётные записи пользователей и разрешения
* Исторические данные мониторинга

Миграция данных конфигурации, таких как дашборды, приложения и теги, важна для подготовки новой среды SaaS к приёму данных мониторинга из всех источников, ранее передававших данные в кластер Dynatrace Managed. Независимо от того, выполняете ли вы миграцию конфигурации или начинаете с нуля, необходимо создать новые учётные записи пользователей и разрешения для среды SaaS. Объём работ зависит от количества пользователей, групп, зон управления и политик IAM, которые планируется мигрировать.

В связи с техническими ограничениями перенос исторических данных мониторинга (таких как метрики, трейсы, проблемы, события и логи) из одной среды в другую невозможен. Однако данные по-прежнему доступны в кластере Dynatrace Managed. Кластеры Dynatrace Managed могут эксплуатироваться параллельно при необходимости доступа к историческим данным. Это может потребовать параллельной работы лицензий Dynatrace Managed и SaaS.

Также может быть полезно настроить [межсредовую трассировку](/managed/observe/application-observability/distributed-traces/analysis/connect-environments "Analyze requests across environment boundaries.") и [создать многосредовые дашборды Dynatrace](/managed/analyze-explore-automate/dashboards-classic/dashboards/dashboards-multi-environment "Create dashboards that display data from multiple Dynatrace environments.") для связи сред Dynatrace Managed и SaaS.

### Параллельная эксплуатация лицензий Dynatrace Managed и SaaS

По запросу Dynatrace может предоставить дополнительные лицензии Dynatrace SaaS без взимания платы на период обновления. Это даёт достаточно времени для накопления долгосрочных трендовых данных в среде SaaS, а также гарантирует, что целевая среда SaaS будет работоспособна до переключения. Лицензии SaaS могут использоваться только для переносимых рабочих нагрузок и настроек и предоставляются только на период обновления — до шести месяцев. По завершении обновления лицензии Dynatrace Managed будут деактивированы, и будут выставляться счета за лицензии SaaS. Для получения подробной информации обратитесь к вашему менеджеру по продажам Dynatrace.

### Оценка трудозатрат и выбор подхода

Оцените трудозатраты, навыки и ресурсы, необходимые для выполнения обновления. Начните с каталогизации и оценки среды, а также определения элементов конфигурации, подлежащих миграции. Затем каталогизируйте и проверьте все экземпляры OneAgent и ActiveGate, развёрнутые в вашей среде. Это поможет определить предпочтительный подход к переходу на SaaS.

#### Какие варианты подхода к обновлению доступны?

У каждой организации свои потребности. Dynatrace предлагает два принципиально разных подхода к обновлению:

* Подход «Большой взрыв»  
  Рекомендуется для клиентов с высоко взаимосвязанной и взаимозависимой инфраструктурой.
* Поэтапный подход  
  Рекомендуется для организаций с изолированными командами и инфраструктурными единицами.

Каждый метод имеет свои преимущества и аспекты, которые необходимо учитывать. Выберите стратегию, наилучшим образом соответствующую целям организации и операционным предпочтениям.

#### Подход «Большой взрыв»

Подход «Большой взрыв» предполагает комплексную миграцию за один раз. Эта стратегия идеальна для организаций, стремящихся к быстрому и решительному переходу.

Преимущества:

* Высокосвязанная и взаимозависимая настройка мониторинга сохраняется
* Всё переносится одновременно с минимальным риском простоя мониторинга
* Это экономичный по времени и затратам процесс обновления

Недостатки:

* Требует высоких усилий по координации и коммуникации
* Рекомендуется предварительное тестирование в меньших sandbox-средах или средах предпродакшена

#### Поэтапный подход

Поэтапный подход предполагает постепенный переход, позволяя мигрировать отдельные компоненты или группы пользователей в разное время. Он обеспечивает гибкость и минимизирует возможные перебои в работе.

Преимущества:

* Изолированная инфраструктура и организационные структуры могут мигрировать по одной.
* Итеративная миграция с быстрой обратной связью и обучением по настройке нового SaaS-тенанта
* Отсутствие простоя благодаря временной гибридной конфигурации

Недостатки:

* Временная гибридная модель может привести к разрыву в Full-Stack Monitoring и перебоям в Real User Monitoring
* Расширенные сроки обновления и более высокие затраты из-за длительных периодов дублированного развёртывания (оборудование и облако)
* Различия в веб-интерфейсе Dynatrace Managed и Dynatrace SaaS

#### Каталогизация и оценка среды

Начните с анализа того, какие элементы конфигурации необходимо мигрировать, и определите, можно ли выполнить миграцию с помощью автоматизированного инструмента или потребуются ручные усилия. Это возможность для очистки среды и применения лучших практик и стандартов. Избыточные или устаревшие элементы конфигурации можно не переносить. Большинство элементов конфигурации, таких как теги или зоны управления, обычно мигрируются полностью, тогда как другие, например дашборды, можно выборочно проверить и мигрировать.

#### Каталогизация и оценка развёрнутых компонентов Dynatrace

После документирования конфигурации Dynatrace, которую необходимо применить к среде Dynatrace, полезно каталогизировать и проанализировать компоненты, настроенные или развёрнутые в вашей среде, и определить предпочтительный подход к их переносу в Dynatrace SaaS. Версии компонентов влияют на возможности развёрнутых OneAgent по подключению к платформе Dynatrace, а также на доступные методы обновления.

Обновление

Настоятельно рекомендуется обновить компоненты мониторинга Dynatrace до последних версий и применить стандартизированные конфигурации и лучшие практики.

Компоненты Dynatrace, которые необходимо перенести в новую среду Dynatrace:

* Экземпляры OneAgent (хосты, PaaS и Kubernetes/OpenShift)
* Расширения ActiveGate
* Расширения OneAgent
* Внешние или пользовательские источники данных
* Экземпляры ActiveGate (маршрутизация трафика, мониторинг облачных сред, мониторинг удалённых технологий с расширениями, Synthetic-мониторы, модули zRemote)

Для каждого элемента мониторинга определите версию OneAgent или расширения и установите необходимые обновления как с технической, так и с процессной точки зрения.

* Более новые версии OneAgent предоставляют больше возможностей настройки; версия развёрнутых OneAgent определяет, нужно ли обновить или полностью переустановить OneAgent.
* Поддержка версий OneAgent осуществляется 9–12 месяцев; проверьте самые старые поддерживаемые версии.
* Для миграции RUM-приложений с Dynatrace Managed на Dynatrace SaaS без побочных эффектов на обе среды требуется OneAgent версии 1.289+; обновите OneAgent до переноса на Dynatrace SaaS.
* Перезапуск приложений с Full-Stack Monitoring обязателен и должен быть включён в план обновления.

### Создание плана обновления

Разработка реалистичного плана перехода на SaaS необходима для максимизации успеха операции. Рекомендуется начинать планирование и подготовку не менее чем за 2–3 месяца до фактической фазы миграции. На предыдущих шагах вы должны были составить список действий, решающих различия между двумя моделями развёртывания и уточняющих объём работ. Теперь пришло время перечислить все действия, необходимые в фазе выполнения. Ниже приведены детали и аспекты, которые следует учитывать при составлении плана обновления до SaaS.

* Группы пользователей
* Пользователи

* Маппинг разрешений — зона управления

#### Инвентаризация конфигурации

Анализ существующей конфигурации Dynatrace Managed даёт список конфигураций, которые необходимо применить к новой среде Dynatrace SaaS. Для выполнения чек-листа готовности к обновлению перейдите в раздел [Оценка готовности к обновлению](/managed/upgrade/up-plan/feasibility-checklist "Pre-upgrade checklist to identify potential upgrade warnings or challenges.").

#### Действия до обновления

Если в вашей среде Managed используется ручная маркировка, преобразуйте эти правила в правила автоматической маркировки. Для завершения документирования существующей среды Dynatrace Managed необходимо идентифицировать и надёжно задокументировать следующее:

* Проверьте используемые токены API и PaaS и определите их назначение.
* Надёжно задокументируйте учётные данные для облачных и контейнерных платформ, таких как AWS, Azure и Kubernetes.
* Координируйте работу и привлеките все команды, затронутые миграцией или имеющие ответственность за её части.
* Платформенные команды должны запланировать обновления файлов развёртывания и перезапуски всей инфраструктуры и сервисов.
* Сетевая команда должна заблаговременно помочь скопировать все правила фаервола из предыдущей среды в новую. Она также должна быть доступна во время миграции на случай возникновения проблем.
* Запланируйте каналы коммуникации и сообщения, которые необходимо направить командам приложений. Получите необходимые согласования от руководителей команд и менеджеров на дату, когда возможен перезапуск приложений и сервисов.
* Создайте списки отслеживания. Например, таблицу с основным контактным лицом каждой команды приложений, которое будет на связи во время миграции для перезапуска своих приложений.
* Убедитесь, что среда правильно настроена заблаговременно. Уведомите команду Dynatrace не менее чем за 1 неделю, если планируется миграция более 5000 OneAgent.

#### Миграция конфигурации

Цель — применить ту же конфигурацию вашей среды Dynatrace Managed к новой среде Dynatrace SaaS. В идеале это выполняется автоматически при наличии такой возможности с помощью инструментов, таких как Dynatrace Configuration as Code.

#### Инвентаризация мониторинга

Как описано ранее (в разделе [Каталогизация и оценка развёрнутых компонентов Dynatrace](#catalog-and-evaluate-dynatrace-components)), результатом является список компонентов Dynatrace, развёрнутых в вашей среде Dynatrace Managed.

#### Миграция мониторинга

Все компоненты Dynatrace в среде Managed необходимо перенастроить для передачи данных в новую среду Dynatrace. Это может потребовать замены существующих развёрнутых компонентов, например ActiveGate, на новые. Подробнее см. в разделе [Выполнение обновления развёртывания](/managed/upgrade/up-execute-upgrade "Execute specific steps to migrate your Managed configurations to Dynatrace.").

#### Операционализация

Скорректируйте существующие процессы и рабочие потоки, включая любые интеграции с ITOM/ITSM, и обновите все ссылки на среду Dynatrace Managed для работы с новой средой Dynatrace.

#### Ограничения фаервола для RUM

Конфигурацию фаерволов, прокси и веб-серверов необходимо скорректировать, поскольку имена файлов cookie содержат суффиксы, специфичные для среды. Подробнее см. в разделе [Ограничения фаервола для RUM](/managed/observe/digital-experience/web-applications/initial-setup/firewall-constraints-for-rum#cookies "Find out how to make sure that Real User Monitoring data passes through your firewall.").

## Шаг 3 — Проектирование

На этом шаге создаётся целевой дизайн платформы Dynatrace SaaS. Помимо настройки и создания новой среды Dynatrace SaaS, необходимо рассмотреть и определить окружающую техническую архитектуру (например, сетевой дизайн, ёмкость и операционное влияние) для обеспечения последующих шагов подготовки и развёртывания.

Основные цели этого шага:

* Анализ зависимостей, которые могут повлиять на обновление
* Анализ ограничений, которые могут повлиять на обновление
* Определение целевой архитектуры

Результатом этого шага может быть отдельная диаграмма, высокоуровневый или детальный архитектурный документ, или сводная письменная спецификация — в зависимости от требований управления, процессов и ваших требований.

На шаге проектирования необходимо рассмотреть следующие аспекты:

### Консолидированная среда SaaS

В Dynatrace Managed вы можете создавать среды по мере необходимости. Однако в Dynatrace SaaS важно определить целевой дизайн сред. Параметры масштабирования и конфигурации сред понадобятся при подготовке среды SaaS — это будет важным вводом данных при рассмотрении оптимизаций конфигурации. Также можно объединить несколько кластеров Dynatrace в один тенант, что даёт преимущества, но и ряд аспектов для рассмотрения:

* Единый источник конфигурации, дашбордов и окон обслуживания значительно сокращает административные накладные расходы.
* Единый домен отказов позволяет Davis AI Dynatrace обеспечивать более полный анализ, а также иметь единое место для просмотра полных трейсов многорегиональных приложений.
* Большее количество приложений, сервисов, процессов и хостов для управления в одном месте. Передовые практики работы с метаданными, тегированием, зонами управления, сетевыми зонами и именованием станут более важными для корректной группировки сущностей и данных, а также контроля доступа.

Обратите внимание, что консолидация сред может потребовать дополнительной проверки лимитов SaaS, чтобы убедиться, что новая консолидированная среда остаётся в рамках защитных параметров платформы.

* Снижена сложность интеграций, конфигурации и управления доступом пользователей к данным
* Полная сквозная трассировка транзакций без конфигурации межсредовых соединений

### Сетевые зоны

Dynatrace позволяет определять сетевые зоны, отражающие архитектуру вашей сети. Сетевые зоны используются для оптимизированной маршрутизации телеметрических данных от OneAgent к кластеру Dynatrace и предотвращения излишнего сетевого трафика между дата-центрами и сетевыми регионами. Подробнее см. в разделе [Сетевые зоны](/managed/manage/network-zones "Find out how network zones work in Dynatrace.").

Информацию об аспектах, связанных с ActiveGate, см. в разделе [Dynatrace ActiveGate](/managed/ingest-from/dynatrace-activegate "Understand the basic concepts related to ActiveGate.").

### Развёртывание и масштабирование ActiveGate

ActiveGate позволяет оптимизировать и минимизировать использование пропускной способности за счёт развёртывания правильного количества экземпляров ActiveGate в каждом дата-центре и принудительного использования сетевых зон. Для миграции следует подготовить новые экземпляры ActiveGate, а существующие вывести из эксплуатации после завершения миграции. Это гарантирует бесперебойный мониторинг. Необходимо рассмотреть развёртывание и настройку следующих компонентов:

* Environment ActiveGate — трафик OneAgent (SSL-сертификат)
* Environment ActiveGate — Cloud & Virtualization
* Environment ActiveGate — Synthetic
* Environment ActiveGate — Extensions
* Environment ActiveGate — Log Ingest (SSL-сертификат)
* Environment ActiveGate — z/OS Mainframe

При масштабировании ActiveGate убедитесь в наличии достаточного резерва на случай сбоев. Передовые практики Dynatrace предписывают не менее двух ActiveGate на дата-центр с добавлением дополнительных экземпляров в зависимости от требований к ёмкости.

### Анализ зависимостей и ограничений, влияющих на обновление

Переход с Dynatrace Managed на развёртывание SaaS может потребовать изменений в сетевой конфигурации, а также повторного анализа и перенастройки внешних систем, интегрированных с Dynatrace (например, для уведомлений о проблемах см. раздел [Уведомления о проблемах](/managed/analyze-explore-automate/notifications-and-alerting/problem-notifications "Learn how to integrate third-party problem notification systems with Dynatrace.")). Анализ текущей сетевой конфигурации и интегрированных систем является основой для дальнейших архитектурных соображений; проанализируйте и спроектируйте компоненты Dynatrace (ActiveGate) с использованием сетевых зон.

### Определение целевой архитектуры

Создаётся целевой дизайн платформы Dynatrace. Помимо настройки и развёртывания новой среды Dynatrace, необходимо рассмотреть и определить окружающую техническую архитектуру — например, сетевой дизайн, ёмкость и операционное влияние — для обеспечения последующих шагов подготовки и развёртывания.

Результатом фазы проектирования может быть отдельная диаграмма, высокоуровневый или детальный архитектурный документ или сводная письменная спецификация — в зависимости от требований управления, процессов и ваших требований. Ниже вы найдёте образец диаграммы целевой архитектуры для базового развёртывания Dynatrace Managed. В качестве допущений использованы следующие условия:

* Один кластер Dynatrace Managed в DC-1 мигрирован в одну среду SaaS (ID: vre43162)
* OneAgent и внешняя автоматизация используют один Environment ActiveGate для передачи данных в кластер
* Безагентный мониторинг, Mobile RUM, Synthetic и OneAgent за пределами DC-1 используют Cluster ActiveGate для передачи данных в кластер
* Корпоративный фаервол запрещает прямой сетевой доступ к Dynatrace SaaS.
* Зелёными стрелками и компонентами обозначены необходимые корректировки в процессе обновления
