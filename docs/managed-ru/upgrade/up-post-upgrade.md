Действия после обновления с Managed на SaaS

После переноса мониторинга в новую среду SaaS анализ и совершенствование использования Dynatrace в среде клиента гарантируют, что сервисы наблюдаемости в организации остаются оптимальными, стабильно обеспечивая максимальную ценность и открывая возможности для дальнейшего расширения применения наблюдаемости. В конечном счёте это позволяет высвободить ресурсы, освободившиеся в результате перехода с локального сервиса, работавшего на инфраструктуре клиента, на полностью облачный сервис под управлением Dynatrace.

На этой странице описаны три шага, которые необходимо выполнить при эксплуатации Dynatrace SaaS после обновления с Dynatrace Managed.

## Шаг 1 — Внедрение

Основные цели этого шага:

* Пригласить пользователей в новые среды SaaS
* Настроить права доступа пользователей
* Оповестить пользователей о переходе на Dynatrace SaaS
* Обучить пользователей работе с новой платформой

### Подключение пользователей

Dynatrace позволяет управлять правами пользователей на основе членства учётных записей в группах пользователей. Этими учётными записями и группами можно управлять локально или через поставщика удостоверений по вашему выбору с помощью протокола [SAML 2.0](https://docs.dynatrace.com/docs/shortlink/access-saml) или [SCIM](https://docs.dynatrace.com/docs/shortlink/access-scim). После подготовки первой среды SaaS вы получите доступ к сервису Account Management. Затем можно приступить к настройке параметров управления доступом и настройке прав пользователей вручную или автоматически с помощью [Account Management API](/managed/dynatrace-api/account-management-api "Explore endpoints of the Account Management API.").

Сообщите об операционных отличиях, которые могут повлиять на дизайн сервисов для пользователей Dynatrace. Для большинства пользователей переход с Dynatrace Managed на SaaS не должен быть заметен.

### Пересмотр ролей и контроля доступа

Пересмотрите роли администраторов, пользователей и заинтересованных сторон Dynatrace и выявите возможности для дальнейшей оптимизации взаимодействия и процессов. Определите, нужно ли пересмотреть доступ на основе ролей, и внедрите новые политики доступа для максимизации охвата и ценности, которую Dynatrace может принести.

Пересмотрите структуру, политики и процедуры контроля доступа с учётом необходимых корректировок в части управления безопасностью и конфиденциальностью данных, а также доступа к платформе SaaS. Используйте SSO для интеграции и автоматизации процедур доступа, максимально продвигая практику Monitoring-as-a-Service.

## Шаг 2 — Расширение

Основные цели этого шага:

* Изучить возможности использования новых функций Dynatrace
* Оптимизировать использование Dynatrace

### Расширение сервисов наблюдаемости

Рекомендуется ознакомиться с [Grail](https://docs.dynatrace.com/docs/shortlink/grail) — базой данных, разработанной специально для данных наблюдаемости. Это единое унифицированное решение для хранения логов, метрик, трейсов, событий и других данных. Все данные в Grail взаимосвязаны в рамках модели реального времени, отражающей топологию и зависимости в отслеживаемой среде. Запланируйте применение следующих возможностей:

* [Log Management and Analytics](https://docs.dynatrace.com/docs/shortlink/log-management-and-analytics)
* [Business Observability](https://docs.dynatrace.com/docs/shortlink/business-observability-hub)

## Шаг 3 — Оптимизация

Основные цели этого шага:

* Пересмотреть и оптимизировать внутренние процессы
* Вывести из эксплуатации старую инфраструктуру Managed

### Пересмотр и оптимизация внутренних процессов

Переход с Dynatrace Managed на SaaS снижает нагрузку по обслуживанию инфраструктуры для организаций, однако также требует корректировки операционных процедур. Пересмотрите и синхронизируйте операционные процедуры организации в части расписаний выпуска обновлений Dynatrace, уведомлений и высокой доступности.

### Свёртывание Dynatrace Managed

После миграции всех источников мониторинга на платформу SaaS оставшиеся компоненты Dynatrace Managed можно вывести из эксплуатации.

Из развёртывания Dynatrace Managed могут остаться следующие типы компонентов Dynatrace:

#### Вывод из эксплуатации ActiveGate

Общая рекомендация при обновлении — развернуть новые экземпляры ActiveGate для новой среды Dynatrace SaaS и вывести из эксплуатации неиспользуемые ActiveGate после завершения обновления.

#### Вывод из эксплуатации узлов кластера Dynatrace Managed

Узлы кластера обеспечивают вычислительные мощности и хранилище данных для Dynatrace Managed. Эти узлы можно временно сохранить для доступа к устаревшим данным и конфигурации Managed. После того как они перестанут быть нужны, следует вывести их из эксплуатации. Для получения дополнительной информации о параллельной эксплуатации Dynatrace Managed и SaaS обратитесь к вашему менеджеру по продажам Dynatrace.

Рекомендуется разработать план вывода из эксплуатации инфраструктуры Dynatrace Managed. В качестве примера предлагаем образец плана, составленного с учётом гипотетических требований:

* В начале кластер Dynatrace Managed был рассчитан на 12 узлов
* Отслеживаемый трафик постепенно переключается на целевую среду SaaS на протяжении всего периода обновления
* На середине процесса кластер Dynatrace Managed масштабируется до половины исходного размера
* Оставляем 3 узла для обеспечения высокой доступности в ходе фазы валидации
* Оставляем 1 узел кластера для соответствия политике хранения данных. По истечении срока хранения кластер полностью выводится из эксплуатации.
