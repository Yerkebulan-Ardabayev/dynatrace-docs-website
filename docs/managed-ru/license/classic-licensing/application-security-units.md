# Мониторинг Application Security (ASU)

# Мониторинг Application Security (ASU)

* Чтение 6 мин
* Опубликовано 30 нояб. 2021 г.

## Application Security

[Application Security](/managed/secure/application-security "Access the Dynatrace Application Security functionalities.") помогает визуализировать, анализировать и отслеживать уязвимости безопасности в среде, связанные со сторонними библиотеками во время выполнения.

### Единицы Application Security

Лицензирование Dynatrace Application Security основано на потреблении единиц Application Security (ASU). Количество единиц Application Security, потребляемых средой, зависит от серверов, на которых выполняются приложения, отслеживаемые с помощью Application Security.

На потребление ASU влияет несколько факторов:

* объём RAM на отслеживаемом сервере (см. таблицу весов ниже);
* количество часов работы сервера;
* возможности Application Security, включённые на сервере.

### Возможности Application Security

В настоящее время Application Security предоставляет две возможности:

* Runtime Vulnerability Analytics
* Runtime Application Protection

### Влияние возможностей на потребление при мониторинге

Каждая возможность потребляет 1 ASU в час, умноженный на весовой коэффициент RAM (подробности см. в таблице весов).

Runtime Application Protection (RAP) опирается на Runtime Vulnerability Assessment (RVA) для оценки уязвимости, на которой основана атака. Поэтому сервер с включённым Runtime Application Protection всегда потребляет ASU и для RVA, и для RAP.

| Размер хоста (по объёму RAM в ГиБ) | Runtime Vulnerability Analytics (единиц Application Security в час) | Runtime Vulnerability Analytics и Runtime Application Protection (единиц Application Security в час) |
| --- | --- | --- |
| 1,6 ГиБ | 0,10 | 0,20 |
| 4 ГиБ | 0,25 | 0,50 |
| 8 ГиБ | 0,50 | 1 |
| 16 ГиБ | 1 | 2 |
| 32 ГиБ | 2 | 4 |
| 48 ГиБ | 3 | 6 |
| 64 ГиБ | 4 | 8 |
| 80 ГиБ | 5 | 10 |
| N × 16 | N | N × 2 |

#### Пример

Допустим, среда состоит из:

* 1 сервера с 32 ГиБ RAM, на котором работают RVA и RAP;
* 1 сервера с 32 ГиБ RAM, на котором работает RVA;
* 1 сервера с 4 ГиБ RAM, на котором работает RVA;
* 1 сервера с 32 ГиБ RAM для обработки пиковых нагрузок, на котором работают RVA и RAP.

Если первые три сервера из списка работают круглосуточно и без выходных, они потребят 54 750 единиц Application Security в год. Когда среда перестаёт справляться с нагрузкой, для обработки пиков запускается дополнительный сервер с 32 ГиБ RAM, на котором работают RVA и RAP. Этот сервер работает в общей сложности 250 часов в течение года, поэтому потребление увеличивается на 1000 ASU.

Расчёт выполняется следующим образом:

* Один из серверов с 32 ГиБ RAM работает с RVA и RAP и потребляет 35 040 ASU в год.  
  `4 ASU за RVA и RAP для хоста 32 ГиБ × 24 (часа) × 365 (дней)`
* Другой сервер с 32 ГиБ RAM работает только с RVA и потребляет 17 520 ASU в год.  
  `2 ASU за RVA для хоста 32 ГиБ × 24 (часа) × 365 (дней)`
* Сервер с 4 ГиБ RAM работает только с RVA и потребляет 2190 ASU в год.  
  `0,25 ASU за RVA для хоста 4 ГиБ × 24 (часа) × 365 (дней)`
* Когда среда перестаёт справляться с нагрузкой, для обработки пиков запускается дополнительный сервер с 32 ГиБ RAM, на котором работают RVA и RAP, но он работает в общей сложности лишь 250 часов в течение года. Поэтому потребление увеличивается на 1000 ASU.  
  `4 ASU за RVA и RAP для хоста 32 ГиБ × 250 (часов)`

### Совмещение мониторинга Application Security с Full-Stack и Infrastructure Monitoring

Единицы Application Security потребляются одновременно с единицами хостов как для Full-Stack, так и для Infrastructure Monitoring. Например, можно отслеживать безопасность хоста, на котором работает сервер Tomcat, отслеживаемый с помощью Dynatrace Infrastructure Monitoring, а не Full-Stack Monitoring. Такой подход даёт данные Dynatrace Application Security, но не даёт преимуществ улучшенной приоритизации на основе топологии или более глубоких сведений о производительности, которые предоставляет режим Full-Stack Monitoring.

Выделение единиц Application Security применимо только к хостам, на которых работают поддерживаемые технологии. Чтобы узнать подробнее, свяжитесь с экспертом по продуктам Dynatrace через онлайн-чат.

## Runecast Security Posture Management (SPM)

Перечисленные здесь лицензии не связаны с ASU мониторинга Application Security; для потребления Security Posture Management (SPM) не требуется предварительное лицензирование ASU мониторинга Application Security. Подробнее о ценообразовании DPS SPM см. в разделе [Application Security (DPS)](/managed/license/capabilities/application-security "Learn how Dynatrace Application Security monitoring consumption is calculated using the Dynatrace Platform Subscription model.").

Runecast® Security Posture Management (SPM) обеспечивает непрерывный мониторинг и автоматизированную оценку сред VMware® и облачных сред. Благодаря данным о конфигурации, проблемах соответствия требованиям и оценке рисков подверженности угрозам, клиенты могут поддерживать надёжный уровень защищённости.

### Возможности Runecast Security Posture Management (SPM)

Runecast Security Posture Management (SPM) предоставляет следующие возможности:

* VMware SPM (VSPM)
* Cloud SPM (CSPM)

### Единицы измерения Runecast Security Posture Management (SPM)

Лицензирование Runecast Security Posture Management (SPM) основано на SKU. Количество SKU, потребляемых средой, зависит от следующих единиц измерения:

* единица измерения для VSPM, это процессорные сокеты в год (сокеты процессоров лицензированных хостов VMware® ESXi), независимо от количества ядер или виртуальных машин;
* единица измерения для CSPM, это хосты в год, где хост в контексте CSPM применяется к вычислительным ресурсам, ресурсам баз данных и функциональным ресурсам. CSPM можно включить в облаке для каждой учётной записи отдельно.

Лицензии структурированы как параллельные годовые единицы. Неиспользованные лицензии не переносятся на следующие периоды. Чтобы расширить количество лицензированных процессорных сокетов и (или) хостов, нужно приобрести дополнительные лицензии.

#### Пример расчёта VSPM

Допустим, приобретена лицензия на 20 процессорных сокетов, начиная с 1 января. Проверки VSPM запускаются и выполняются непрерывно. Через шесть месяцев в среду добавляется ещё 10 процессорных сокетов и приобретается дополнительная лицензия на эти новые сокеты с датой начала действия 1 июля.

Первоначальная лицензия на 20 процессорных сокетов истекает через 12 месяцев после первоначальной покупки, 31 декабря того же года.

* 20 процессорных сокетов остаются без лицензии и не могут проверяться до возобновления лицензии.
* Вторая лицензия продолжает покрывать оставшиеся 10 процессорных сокетов до истечения её срока действия 30 июня следующего года.

#### Пример расчёта CSPM

Допустим, приобретена подписка на 20 тарифицируемых ресурсов AWS (инстансы EC2 и функции Lambda), начиная с 1 января. Проверки CSPM запускаются и выполняются непрерывно.
Через шесть месяцев принимается решение добавить в среду ещё 10 хостов и приобрести дополнительную лицензию на эти новые хосты с датой начала действия 1 июля.
В этом случае потребуется применить новую лицензию на 30 тарифицируемых ресурсов, заменяющую текущую подписку.

Лицензия на первоначальные 20 хостов истекает через 12 месяцев после первоначальной покупки, 31 декабря.

* 20 хостов остаются без лицензии и не могут проверяться до возобновления лицензии.
* Вторая лицензия продолжает покрывать оставшиеся 10 хостов до истечения её срока действия 30 июня следующего года.
