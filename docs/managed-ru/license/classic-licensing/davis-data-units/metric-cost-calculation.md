# DDU для метрик

* 17 мин чтения
* Опубликовано 30 марта 2021 г.

На этой странице объясняется, как рассчитывается DDU для метрик, что такое концепция пользовательских метрик и как можно оценить и отслеживать потребление DDU для метрик.

## Какие типы метрик потребляют DDU?

* Пользовательские метрики
* Некоторые встроенные метрики (см. ниже)

## Что такое пользовательская метрика?

Пользовательские метрики, это метрики, которые определяет или устанавливает пользователь. Они расширяют возможности мониторинга Dynatrace, позволяя интегрировать сторонние источники данных, рассчитывать пользовательские метрики, импортировать метрики через API, использовать extensions и многое другое.

Ниже приведён неполный список типов пользовательских метрик:

* Все [метрики, отправленные через API](/managed/ingest-from/extend-dynatrace/extend-metrics "Узнайте, как расширить наблюдаемость метрик в Dynatrace.")
* Calculated service metrics, custom DEM metrics и [метрики логов](/managed/analyze-explore-automate/log-monitoring/analyze-log-data/log-metrics "Узнайте, как создавать и использовать метрики логов Dynatrace для анализа данных логов.")

  + Например, calculated service metrics, созданные на основе атрибутов запроса
* Все метрики, предоставляемые вручную установленными [Dynatrace Monitoring Extensions](/managed/ingest-from/extensions "Узнайте, как создавать Extensions Dynatrace и управлять ими.")

Подробную информацию о настройке и приёме пользовательских метрик в Dynatrace см. в разделе [Приём метрик](/managed/ingest-from/extend-dynatrace/extend-metrics "Узнайте, как расширить наблюдаемость метрик в Dynatrace.").

## Какие встроенные метрики потребляют DDU?

К метрикам, потребляющим DDU, относятся:

* Метрики, предоставляемые некоторыми [встроенными extensions](/managed/ingest-from/extensions#out-of-the-box "Узнайте, как создавать Extensions Dynatrace и управлять ими."), которые не включены по умолчанию

  + Переключатели в веб-интерфейсе Dynatrace для включения/отключения таких метрик помечены, чтобы показать, влияют ли они на потребление DDU.
* Все метрики, принимаемые Extensions Public Cloud ([AWS](/managed/ingest-from/amazon-web-services/integrate-with-aws/aws-all-services "Мониторинг всех облачных сервисов AWS с помощью Dynatrace и просмотр доступных метрик."), [Azure](/managed/ingest-from/microsoft-azure-services/azure-integrations/azure-cloud-services-metrics "Мониторинг сервисов Azure с помощью Dynatrace и просмотр доступных метрик.") и [Google Cloud](/managed/ingest-from/google-cloud-platform "Мониторинг Google Cloud с помощью Dynatrace.")).

  + Например, метрики, полученные из облачных сервисов.

## Как рассчитывается потребление DDU для метрик?

Потребление происходит при приёме точек данных метрики. Точка данных метрики, это единичное значение измерения,
сохранённое в Dynatrace. Значение принадлежит метрике, которая идентифицируется ключом метрики и
имеет временную метку. Метрика временного ряда, это последовательность таких точек данных, например загрузка CPU для заданного хоста на протяжении периода анализа.

Каждая принятая точка данных списывает 0.001 DDU с доступной квоты.

Метрики, предоставляемые Dynatrace, обычно записываются раз в минуту. Это даёт следующее типичное годовое потребление DDU для одной метрики:

`1 точка данных метрики × 60 мин × 24 ч × 365 дней × 0.001 = 525.6 DDU на метрику в год`

### Как измерения метрики влияют на потребление DDU?

Точки данных метрики могут иметь связанные измерения, которые идентифицируются ключами и значениями измерений. Например, используя синтаксис
[протокола приёма метрик](/managed/ingest-from/extend-dynatrace/extend-metrics/reference/metric-ingestion-protocol "Узнайте, как работает протокол приёма данных для API Metrics Dynatrace.") два значения можно принять следующим образом:

```
my_cpu_utilization,hostname=hostA 55 1609459200000



my_cpu_utilization,hostname=hostB 45 1609459200000
```

Для одной и той же временной метки (`1609459200000`) Dynatrace сохраняет два разных значения измерения для одной метрики на двух разных хостах.
Это достигается с помощью ключа измерения `hostname` и значений измерения `hostA` и `hostB`. `55` и `45`, это измеренные проценты загрузки CPU для этой метрики на двух хостах.

Обратите внимание, что это можно рассматривать как две метрики: один график можно создать для загрузки CPU hostA, а другой для загрузки CPU hostB.
Если точки данных периодически принимаются для обоих хостов, конечный эффект, это удвоенное количество принятых точек данных, потому что здесь каждое измерение по хосту
приводит к отдельному значению метрики, которое нужно сохранять.

Это удвоение принятых точек данных также отражается на соответствующем потреблении DDU: теперь потребление DDU фактически составляет 0.002 в минуту, потому что
принимаются данные с двух хостов.
Поэтому, чтобы рассчитать потребление DDU на основе ключей метрик,
нужно учитывать, какие данные принимаются, и принимать во внимание измерения этих данных.

Если добавляется больше измерений и количество точек данных увеличивается, то потребление DDU также увеличится.
Вот пример:

```
my_cpu_utilization,hostname=hostA,cpu=1 55 1609459200000



my_cpu_utilization,hostname=hostA,cpu=2 11 1609459200000



my_cpu_utilization,hostname=hostB,cpu=1 45 1609459200000



my_cpu_utilization,hostname=hostB,cpu=2 45 1609459200000
```

В этом примере будет потребляться 0.004 DDU в минуту из-за двух хостов и двух CPU.

Однако дополнительное потребление вызвано не увеличением количества измерений, а
увеличением количества принятых точек данных, которое обычно сопровождает увеличение количества измерений.

Если измерения добавляются, но количество точек данных остаётся прежним, то потребление DDU не меняется.
Так происходит, когда добавляются чисто описательные измерения, значения которых определяются другими уже существующими измерениями:

```
my_cpu_utilization,hostname=hostA,host_ip="127.0.0.23" 55 1609459200000



my_cpu_utilization,hostname=hostB,host_ip="127.0.0.42" 45 1609459200000
```

Эти принятые точки данных дают такое же потребление DDU, как если бы они были отправлены без измерения `host_ip`.
Формально говоря, если добавляются измерения, значения которых
[функционально определяются﻿](https://en.wikipedia.org/wiki/Functional_dependency) другими уже существующими измерениями, то
потребление DDU не изменится.

### Включённые метрики на host unit

Если хосты мониторятся с помощью OneAgent и потребляют host units, то предоставляется определённое количество «метрик, включённых для хоста», в зависимости от
потребления host units мониторимых хостов. Точнее, для каждой заданной минуты хост, потребляющий host units, имеет бюджет включённых точек данных метрик для хоста.
Когда принимаются точки данных метрик «о хосте» (например, с формальным измерением хоста), сначала расходуется включённый бюджет точек данных для хоста,
прежде чем это повлияет на квоту DDU. Одна включённая метрика хоста определяется как одна точка данных метрики в минуту, с учётом предыдущего раздела
об измерениях принятых точек данных.

1000 метрик на host unit включены на каждом мониторимом хосте (в режиме Full-Stack Monitoring). Эти метрики включены в стоимость Full-Stack
host units и, соответственно, не потребляют DDU.

* Каждый хост, мониторимый OneAgent с включённым Full-Stack Monitoring, включает 1000 метрик на Host Unit (подробности в таблице ниже).
* Хосты, мониторимые OneAgent с включённым Infrastructure Monitoring, всегда включают 200 метрик, которые не потребляют DDU.
* Все метрики, превышающие количество включённых метрик хоста, потребляют DDU.
* Если все DDU в пуле Metrics израсходованы, то включённые метрики на host unit также становятся недоступны.
* Calculated service metrics, custom DEM metrics и метрики логов не учитываются в этих включённых метриках на host unit.

Обратите внимание, что включённые метрики привязаны к конкретным хостам, мониторимым через OneAgent. Это значит, что учитываются только метрики, закреплённые либо за мониторимым хостом, либо за процессом мониторимого хоста. Сюда также входят метрики, закреплённые за сущностями Amazon EC2, Azure VM и виртуальными машинами VMware (если эти виртуальные машины мониторятся OneAgent). Обратите внимание, что для метрик, принятых через API, это применяется только к метрикам, закреплённым за мониторимым хостом.

Оценка того, превышает ли хост количество включённых метрик, основана на общем количестве точек данных метрик, которые сообщаются с
частотой раз в минуту (для этих метрик нет скользящего временного окна). Когда количество принятых точек данных превышает количество включённых метрик хоста,
за разницу списываются DDU.

Если хост лицензируется на основе host-unit hours, а не host units, расчёт работает так же: можно потреблять до количества включённых метрик
хоста на host-unit hour. Каждая точка данных сверх этого количества потребляет DDU.

| Размер хоста | Режим Full-Stack | Режим Full-Stack | Режим Infrastructure | Режим Infrastructure |
| --- | --- | --- | --- | --- |
| На основе RAM в ГиБ | Host units | Включённые метрики | Host units | Включённые метрики |
| 1.6 ГиБ | 0.1 | 200 | 0.03 | 200 |
| 4 ГиБ | 0.25 | 250 | 0.075 | 200 |
| 8 ГиБ | 0.5 | 500 | 0.15 | 200 |
| 16 ГиБ | 1 | 1,000 | 0.3 | 200 |
| 32 ГиБ | 2 | 2,000 | 0.6 | 200 |
| 48 ГиБ | 3 | 3,000 | 0.9 | 200 |
| 64 ГиБ | 4 | 4,000 | 1.0 | 200 |
| N × 16 | N | N [1](#fn-1-1-def) × 1,000 | 1.0 | 200 |

1

Минимальное количество включённых метрик всегда 200, даже если N меньше 0.2

### Пример того, как включённые метрики влияют на потребление DDU

Рассмотрим следующий пример.

* У хоста 16 ГиБ RAM, другими словами, 1 host unit
* Хост мониторится через OneAgent в режиме Full-Stack

(1 host unit = 1 000 host-included metrics)

* 1 500 метрик от различных расширений и локального приёма API (например, JMX и RabbitMQ) учитываются на этом OneAgent-мониторимом хосте
* 300 репортящихся метрик через публичный API без связи с OneAgent-мониторимым хостом

В этом сценарии 1 000 метрик/мин включены за счёт host unit, а хост репортит избыток в 500 метрик, которые будут потреблять DDU. 300 метрик, которые репортятся через API и не связаны с OneAgent-мониторимым хостом, тоже будут потреблять DDU. Итого получается 800 метрик/мин, потребляющих DDU.

Вот ещё несколько простых сценариев (все значения ниже основаны на метриках, отправляемых с интервалом в 1 мин).

| Режим | Host units | Included metrics на host unit [1](#fn-2-1-def) | Метрики, репортящиеся для хоста | Платные метрики | Списанные DDU/мин |
| --- | --- | --- | --- | --- | --- |
| Full-Stack | 0.5 | 500 | 300 | 0 | 0 |
| Full-Stack | 1 | 1 000 | 1 500 | 500 | 0.5 |
| Full-Stack | 1 | 1 000 | 500 | 0 | 0 |
| Full-Stack | 4 | 4 000 | 5 000 | 1 000 | 1 |
| Infrastructure | 0.6 | 200 | 150 | 0 | 0 |
| Infrastructure | 1 | 200 | 1 000 | 800 | 0.8 |

1

Included metrics привязаны к отдельным хостам.

## Reported DDU и consumed DDU

Reported DDU определяются как все DDU до применения каких-либо учётов host-included. Consumed DDU отражают общее количество «оплачиваемых» DDU, которые вычитаются из баланса DDU.

Можно сравнить reported DDU для конкретного хоста с числом [host-included metrics](#metrics-per-host-unit), создав диаграмму в [Data Explorer](/managed/analyze-explore-automate/explorer "Query for metrics and transform results to gain desired insights.") с использованием следующих метрик. Это позволяет получить представление о количестве доступных и потреблённых DDU для конкретного хоста.

* `builtin:billing.ddu.includedMetricDduPerHost`
  Эта метрика разбита по host ID. Она показывает host-included DDU. Например, если у хоста 1 000 [included metrics](#metrics-per-host-unit), значение этой метрики будет равно 1 (другими словами, 1 000 × 0.001)
* `builtin:billing.ddu.metrics.byEntityRaw`
  Метрика `byEntityRaw` разбита по мониторимой сущности и отражает общее количество reported DDU до применения каких-либо учётов host-included.
* `builtin:billing.ddu.metrics.byEntity`
  Аналогично метрике `byEntityRaw`, метрика `byEntity` разбита по мониторимой сущности. Она отражает общее количество consumed DDU (другими словами, всё потребление сверх количества included metrics на host unit). Это эквивалентно информации, показанной на [странице обзора потребления DDU](/managed/license/classic-licensing/davis-data-units#consumption-per-environment "Understand how Dynatrace monitoring consumption is calculated based on Davis data units (DDU).") (см. пример ниже).

* Поскольку биллинг DDU рассчитывается на поминутной основе (подробнее в [примере потребления DDU для метрик](#consumption-overview)), рекомендуется работать в диаграмме также с разрешением в 1 минуту.

Приведённый выше пример запроса Data Explorer показывает следующую информацию:

* Фиолетовая линия отражает host-included DDU. В примере выше это 0.25 DDU (другими словами, 250 метрик) на поминутной основе.
* Жёлтая линия отражает метрику `byEntityRaw`. Она показывает общее количество reported DDU для мониторимой сущности, отфильтрованной по хосту. Любое потребление DDU сверх host-included бюджета превышает порог, обозначенный фиолетовой линией.
* Бирюзовая линия отражает информацию из метрики `byEntity`. Она показывает общее количество потреблённых метрик. В примере выше host-included лимит превышается примерно каждые 15 минут между 09:30 и 10:45, а начиная с 10:45, превышается непрерывно. Это приводит к consumed DDU, которые представлены бирюзовой линией на диаграмме.

Обратите внимание, что точки данных метрики `byEntity` могут задерживаться до нескольких минут.

### Страница обзора Davis data units

Чтобы посмотреть, сколько DDU потребила ваша среда

1. Перейдите в [**Account Management**﻿](https://myaccount.dynatrace.com/) > **License** / **Subscription** > **Overview**.

   * В таблице **Consumption by DDU pool** проверьте строку **Metrics**.
   * В разделе **DDU consumption details** проверьте вкладку **Metrics**.

Там можно посмотреть потребление DDU и определить основных участников потребления DDU.

## FAQ

Какие метрики потребляют DDU?

Все метрики, отправляемые через стандартный OneAgent (например, метрики хостов), включены бесплатно и не потребляют DDU. Метрики, поступающие через расширение и привязанные к конкретным хостам (или процессам), сначала расходуют все доступные включённые метрики, прежде чем начать потреблять дополнительные DDU. Метрики, поступающие через API, расходуют все доступные включённые метрики, привязанные к конкретным хостам (не процессам), прежде чем начать потреблять дополнительные DDU.

Следующие типы метрик потребляют DDU:

| Тип метрики | Потребляет DDU | Включена в host unit |
| --- | --- | --- |
| Built-in extensions | Да | Да |
| Rest API | Да | Да[1](#fn-3-1-def) |
| JMX | Да | Да |
| Python | Да | Нет |
| Calculated service metrics | Да | Нет |
| Calculated log metrics | Да | Нет |
| Calculated DEM metrics | Да | Нет |
| AWS metrics | Да | Да[2](#fn-3-2-def) |
| Azure metrics | Да | Да[2](#fn-3-2-def) |
| Google Cloud metrics | Да | Да[2](#fn-3-2-def) |
| Cloud Foundry | Нет | Нет |
| Kubernetes Cluster metrics | Нет | Нет |
| VMware metrics | Нет | Нет |

1

Если использовать REST API для отправки метрик на хост, отслеживаемый OneAgent, эти метрики будут засчитываться в счёт выделенных включённых метрик host unit. Если метрики отправляются на другие сущности, они будут потреблять DDU.

2

Метрики облачных сервисов, привязанные к хостам, отслеживаемым OneAgent, вычитаются из выделенных включённых метрик host unit. Например, метрика инстанса EC2 засчитывается в счёт включённых метрик host unit этого инстанса EC2. Все остальные метрики облачных сервисов подлежат потреблению DDU.

Какие метрики считаются включёнными метриками host unit?

* Любая метрика, зарегистрированная на отслеживаемом хосте или экземпляре process group instance, поскольку их можно сопоставить с конкретным хостом. Обратите внимание, что для метрик, поступающих через API, это применимо только к метрикам, зарегистрированным на отслеживаемом хосте.
* Начиная с версии Dynatrace 1.203, ингест метрик может регистрироваться непосредственно на отслеживаемых хостах, что позволяет лучше использовать включённые метрики хостов.
* В одном из будущих релизов расширения ActiveGate также смогут регистрировать метрики на отслеживаемых хостах и таким образом использовать преимущества включённых метрик хостов, как это уже делают расширения OneAgent.

DDU используются только для расширений?

Нет, для расширений любая метрика, зарегистрированная на хосте или процессе, потребляет DDU. Для ингеста через API сначала расходуются включённые метрики на host unit (то есть 1000 включённых метрик для Full-Stack Monitoring), прежде чем начинают потребляться DDU. Подробности см. в разделе [Included metrics per host unit](#metrics-per-host-unit).

Каков вес (стоимость) одной точки данных?

Каждая точка данных потребляет `0.001 DDU`.

Сколько DDU потребляет одна метрика с частотой 1 минута за год?

`1 точка данных × 60 мин × 24 часа × 365 дней × 0.001 DDU веса = 525.6 DDU`

Почему разбивка потребления по отслеживаемой сущности отличается от разбивки по метрике?

Разбивка потребления DDU, показанная на странице **Monitored entities** и **Metrics**, а также метрики `byEntity` и `byMetric` в API, могут немного отличаться по следующим причинам:

* Разбивка по отслеживаемой сущности учитывает включённые DDU на host unit, чего разбивка по метрике не делает.
* Потребление по отслеживаемым сущностям может отставать до 10 минут по сравнению с метриками `byEntity` и `byMetric` в API.

Поэтому для целей лицензирования и биллинга учитывается только разбивка по отслеживаемой сущности (разбивка по метрике предоставляется исключительно в информационных целях).

Для метрик, не связанных с отслеживаемым хостом, потребление по отслеживаемой сущности и потребление по метрике идентичны.

У нас работает много расширений OneAgent. Покрываются ли эти метрики включёнными метриками? Только в режиме Full-Stack Monitoring?

Да, метрики через расширение OneAgent покрываются включёнными метриками. Для режима Full-Stack Monitoring это означает 1000 включённых метрик. Для режима Infrastructure это означает 200 включённых метрик.

Верно ли, что метрики, привязанные к хосту или процессу, засчитываются в счёт включённых метрик на host unit?

Да. Обратите внимание, что для метрик, поступающих через API, это применимо только к метрикам, зарегистрированным на отслеживаемом хосте.

У нас есть расширения, которые используют custom device API для ингеста метрик. Засчитывается ли это в счёт включённых метрик?

Нет, у custom devices нет включённых метрик. Включённые метрики доступны только для хостов и процессов, отслеживаемых OneAgent.
Следующие расширения используют custom devices для ингеста метрик: F5, Windows Server, Apigee Edge, SAP, Juniper Network, IBM I, IBM DataPower, Citrix Netscaler и другие.

Что произойдёт, если я буду отслеживать удалённые источники с помощью расширений ActiveGate? Будут ли эти метрики засчитываться в счёт включённых метрик host unit?

Расширения ActiveGate не потребляют включённые метрики host unit. Включённые метрики доступны только для метрик хостов и процессов, отслеживаемых OneAgent.

Что произойдёт, если я изменю конфигурацию с Full-Stack на Infrastructure Monitoring? Адаптируются ли включённые метрики немедленно?

Изменение конфигурации должно занять всего несколько секунд, без негативного влияния на потребление метрик.

Что произойдёт, если я буду отправлять пользовательские метрики с более высокой частотой, например каждые 10 секунд?

Рекомендуется отправлять пользовательские метрики с частотой 1 минута. При ингесте большего количества точек данных метрик в минуту потребляется больше DDU.
Например, пользовательская метрика, отправляемая с частотой 10 секунд, эквивалентна `6 точек данных × 60 мин × 24 часа × 365 дней × 0.001 DDU веса = 3153.6 DDU`

Потребляют ли облачные сервисы для AWS и Azure DDU, или они являются частью включённых метрик на хост?

Облачные сервисы для AWS и Azure потребляют DDU. Все метрики, относящиеся к хосту, отслеживаемому OneAgent, либо являются частью базового пакета (это не метрики), либо засчитываются в счёт включённых метрик, которые предоставляются с каждым host unit. Все остальные метрики расширений или интеграций потребляют DDU.

Могу ли я выбирать, какие сервисы AWS или Azure отслеживаются, чтобы контролировать потребление DDU?

Да, можно выбирать, какие отдельные сервисы AWS и Azure отслеживаются. Все «cloud services» доступны для отслеживания на основе opt-in. Кроме того, используя фильтрацию по тегам, можно отказаться от отслеживания отдельных «Built-in services» (сервисов, которые отслеживаются автоматически).

Потребляют ли метрики Kubernetes (использование/лимит/ресурсы CPU) DDU, даже если OneAgent не установлен?

Узлы Kubernetes автоматически отслеживаются при установленном OneAgent; эти метрики являются частью базового предложения OneAgent и включены на host unit.
