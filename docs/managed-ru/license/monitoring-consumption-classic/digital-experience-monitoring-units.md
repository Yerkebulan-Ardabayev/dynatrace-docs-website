# Digital Experience Monitoring (единицы DEM)

# Digital Experience Monitoring (единицы DEM)

* 7-min read
* Published Mar 30, 2021

В дополнение к [мониторингу приложений и инфраструктуры](/managed/license/monitoring-consumption-classic/application-and-infrastructure-monitoring "Understand how Dynatrace application and infrastructure monitoring consumption is calculated based on host units."), предоставляемому OneAgent, вам также могут потребоваться Dynatrace Synthetic Monitoring, Real User Monitoring и Session Replay.
Эти возможности тарифицируются на основе единиц Digital Experience Monitoring, иначе называемых единицами DEM.

## Единицы DEM

Необходимое количество единиц DEM зависит от числа запускаемых синтетических мониторов и числа пользовательских сессий, которые необходимо мониторировать.
В таблице ниже указана скорость потребления единиц DEM для каждого типа возможности и единицы измерения.

| Единица измерения | Возможность | Потребление на единицу измерения |
| --- | --- | --- |
| Синтетическое действие | Browser monitors, browser clickpaths | 1,0 DEM |
| Синтетический запрос | HTTP monitor | 0,1 DEM |
| Пользовательская сессия на приложение | Real User Monitoring (без воспроизведения Session Replay) | 0,25 DEM |
| Пользовательская сессия на приложение | Сессия Real User Monitoring с захватом Session Replay | 1,0 DEM |
| Свойство сессии | Real User Monitoring | 0,01 DEM |
| Свойство действия | Real User Monitoring | 0,01 DEM |
| Результат стороннего синтетического теста | Приём данных через Third-party synthetic API | 0,1 DEM |

Пользовательские сессии тарифицируются на приложение, даже если сессия охватывает несколько приложений одного домена. Учитываются только сессии реальных пользователей. Сессии синтетических пользователей и «роботов» не включаются в потребление пользовательских сессий.

После примерно 200 пользовательских действий в сессии создаётся новая сессия, и все последующие действия включаются в неё. За дополнительные сессии, создаваемые Dynatrace после таких разделений, дополнительная плата не взимается.

Типы данных для свойств влияют на тарификацию: короткие строки (менее 100 символов) и числовые типы (длинные строки, double или даты) считаются одним свойством. Длинные строки считаются за 1 свойство на каждые 100 символов.

## Real User Monitoring

Единственная [сессия Real User Monitoring](/managed/observe/digital-experience/rum-concepts/rum-overview "Learn about Real User Monitoring, key performance metrics, mobile app monitoring, and more.") (иначе называемая «пользовательской сессией») определяется как последовательность взаимодействий пользователя с браузерным веб-приложением или нативным мобильным приложением на iOS или Android в рамках временного интервала с не менее чем двумя пользовательскими действиями.
Пользовательское действие — это клик мышью, касание пальцем или запуск приложения, инициирующий веб-запрос (например, загрузку страницы или навигационный переход).
Взаимодействия, включающие только одно пользовательское действие, считаются «отскочившими» и не учитываются.
Пользователь, одновременно взаимодействующий с несколькими приложениями, потребляет по одной сессии для каждого, кроме случаев «отскока».
Взаимодействия с гибридными мобильными приложениями, включающими веб и мобильное приложение, считаются единой сессией.

Тарифицируемая сессия заканчивается при [завершении сессии пользователя](/managed/observe/digital-experience/rum-concepts/user-session#user-session-end "Learn how a user session is defined, when a user session starts or ends, how user session duration is calculated, and more.") или после 60 минут непрерывного взаимодействия с приложением.

При наличии годовой квоты сессий RUM использование сбрасывается ежегодно.

### Пример потребления DEM для Real User Monitoring

Например, пользователь взаимодействовал с веб-приложением или мобильным приложением в течение четырёх непрерывных часов.
С точки зрения лицензии сессия заканчивается после 60 минут непрерывного взаимодействия, после чего возобновляется новая сессия.
Таким образом, четырёхчасовая сессия эквивалентна четырём лицензированным сессиям:

* Без данных Session Replay: `4 × 0,25 = 1 единица DEM`.
* С данными Session Replay: `4 × 1 = 4 единицы DEM`.

## Синтетические действия и запросы

* «Синтетическое действие» — это взаимодействие с синтетическим браузером, инициирующее веб-запрос: загрузку страницы, навигационное событие или действие, вызывающее XHR- или Fetch-запрос:
  + Browser monitors выполняют одно синтетическое взаимодействие (например, измерение производительности и доступности одного URL). Browser monitors потребляют одно синтетическое действие на выполнение. XHR- или Fetch-запросы, не инициированные напрямую пользовательским вводом, не потребляют синтетические действия.
  + Clickpath monitors — последовательности предварительно записанных синтетических действий. Clickpath monitors потребляют одно синтетическое действие на каждое взаимодействие, инициирующее веб-запрос. Прокрутка, нажатия клавиш или клики, не инициирующие веб-запросы, не считаются действиями.
* «Синтетический запрос» состоит из одного или нескольких HTTP(S)-запросов (например, GET, POST, HEAD). Каждый запрос, выполненный HTTP(S)-монитором, потребляет синтетический запрос.

Синтетическое действие/запрос тарифицируется сразу после его выполнения — независимо от результата. Если действие/запрос завершается ошибкой и последующие не выполняются, они не тарифицируются.

`# синтетических действий/запросов на монитор = (# синтетических действий/запросов в мониторе) × (# запусков в час) × (# локаций) × # часов`

### Пример расчёта синтетических действий/запросов

Например, записанный браузерный clickpath, проходящий через две страницы и нажимающий одну кнопку, инициирующую XHR или Fetch-запрос, потребляет три синтетических действия.
При запуске такого монитора каждые 15 минут из двух локаций в течение одного дня clickpath потребляет 576 синтетических действий в день.

`3 (синтетических действия) × 4 (запуска в час) × 2 (локации) × 24 (часа в день) = 576 (синтетических действий)`

Подробнее см. в [Synthetic Monitoring](/managed/observe/digital-experience/synthetic-monitoring "Learn about Synthetic Monitoring and how to create a single-URL browser monitor, a browser clickpath, or an HTTP monitor.").

## Превышения Digital Experience Monitoring (опционально)

Если для вашего аккаунта настроены превышения DEM (то есть разрешено превышать максимальный лимит единиц DEM), превышаемые единицы потребляются так же, как обычные. Каждая дополнительная сессия или синтетический тест-превышение увеличивает потребление единиц DEM аккаунта.
Для добавления или удаления превышений [свяжитесь с отделом продаж Dynatrace](https://www.dynatrace.com/contact/).

## Бесплатный уровень свойств действий и сессий

Для обогащения сессии или пользовательского действия можно настраивать дополнительные определённые свойства.
В настоящее время предоставляется бесплатный уровень из 20 определённых свойств на приложение.
Как показано в таблице, стоимость DEM на сессию увеличивается на 0,01 единицы DEM за каждое дополнительное определённое свойство.

Например, 100 сессий с 25 определёнными свойствами потребляют `100 × (25 - 20) × 0,01 = 5 единиц DEM` за дополнительные свойства.
Суммарная стоимость в DEM в данном случае составит 30 единиц:

`5 единиц DEM (дополнительные свойства) + 25 единиц DEM (100 сессий; 1 сессия = 0,25 DEM) = 30 единиц DEM`
