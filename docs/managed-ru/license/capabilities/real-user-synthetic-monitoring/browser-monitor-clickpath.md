# Понимание и управление потреблением для Browser Monitor или Clickpath (DPS)

* Пояснение
* Чтение за 5 минут
* Обновлено 11 мая 2026 г.

Browser Monitor, это возможности синтетического мониторинга, которые симулируют взаимодействия пользователей с веб-приложениями.

На этой странице описано, как рассчитывается потребление и как можно оценивать, управлять и оптимизировать использование синтетического мониторинга в Dynatrace.

## Как рассчитывается потребление

Browser Monitor потребляет ресурсы на основе количества **выполненных синтетических действий**, используя позицию прайс-листа **Browser Monitor or Clickpath**.

Синтетические действия запускаются шагами внутри browser monitor, как описано ниже.

### Ключевые определения

Browser monitor
:   Симулированный тест браузера, настраиваемый в Dynatrace.
    Каждый browser monitor состоит из одного или нескольких шагов.

Шаг (Step)
:   Отдельная единица внутри browser monitor, представляющая одно симулированное взаимодействие пользователя.
    Примеры шагов: «перейти на главную страницу», «выбрать кнопку входа» или «отправить форму».

Синтетическое действие (Synthetic action)
:   Синтетическое действие, это конкретный шаг, подлежащий тарификации.
    Шаги тарифицируются только тогда, когда они инициируют веб-запрос, например загрузку страницы, переходы и действия пользователя, явно инициирующие запросы XHR или Fetch.

### Правила подсчёта

* Ни один шаг никогда не считается более чем одним синтетическим действием.
* Каждый раз при выполнении синтетического теста Dynatrace подсчитывает действия, выполненные в ходе этого запуска.
* Синтетические действия учитываются как для публичных, так и для приватных синтетических выполнений.
* Если шаг инициирует один или несколько веб-запросов, он засчитывается как одно синтетическое действие.
  Шаг, инициирующий, например, три запроса XHR, тарифицируется как одно синтетическое действие.
  Потребление определяется количеством шагов, которые формируют хотя бы один веб-запрос, а не общим числом веб-запросов.
* Синтетическое действие тарифицируется сразу после выполнения шага, независимо от того, прошло действие успешно или нет.
* Если шаг не инициирует ни одного веб-запроса, он засчитывается как ноль синтетических действий.
  Следующие взаимодействия не потребляют синтетические действия:

  + Шаги, состоящие только из прокрутки, нажатий клавиш или кликов, не инициирующих веб-запрос.
  + Запросы XHR или Fetch, автоматически инициируемые браузером после загрузки страницы, если они явно не инициированы действием пользователя.
  + Любые шаги, происходящие после неудавшегося действия, поскольку они никогда не выполняются.

### Пример потребления

В этом примере рассмотрим browser monitor с четырьмя шагами.

| Шаг | Действие пользователя | Что происходит | Тарифицируемые синтетические действия |
| --- | --- | --- | --- |
| 1 | Перейти на главную страницу | Инициирует загрузку страницы | 1 |
| 2 | Выбрать **Login** | Инициирует запрос XHR к сервису API авторизации, который затем запускает ещё два дополнительных запроса XHR для профиля пользователя и настроек | 1 |
| 3 | Прокрутить страницу вниз | Веб-запрос не инициируется | 0 |
| 4 | Выбрать **Submit order** | Инициирует запрос Fetch | 1 |

В итоге за это единственное выполнение browser monitor тарифицируются **три синтетических действия**, хотя browser monitor содержит четыре шага, а шаг 2 сформировал три веб-запроса.

## Оценка стоимости

Можно оценить месячную стоимость, рассчитав общее количество тарифицируемых шагов (то есть шагов, инициирующих хотя бы один веб-запрос).
Затем использовать формулу ниже.

### Формула потребления

```
(# тарифицируемых шагов за выполнение)



× (# выполнений в час)



× (# локаций)



× (# часов)



= Общее количество синтетических действий
```

### Пример расчёта стоимости

Для простоты в этом примере предполагается, что:

* Расчёты основаны на прайс-листе $0.0045 за синтетическое действие, см. [цены Dynatrace﻿](https://www.dynatrace.com/pricing/).
  Эта цена может отличаться от вашего прайс-листа.
* Один месяц эквивалентен 30 дням, или 720 часам.

Если есть browser monitor, который:

* Имеет четыре шага, из которых три инициируют хотя бы один запрос.
* Выполняется каждые 15 минут.
* Отслеживается из трёх локаций.

Месячное потребление можно рассчитать по формуле ниже.

```
3 тарифицируемых шага за выполнение



×   4 выполнения в час



×   3 локации



× 720 часов



= 25 920 синтетических действий
```

Умножив на стоимость единицы $0.0045 за результат, общая стоимость синтетических действий в этом примере составляет **$116.64 в месяц**.

## Отслеживание потребления

Dynatrace предоставляет различные варианты, помогающие понимать и анализировать потребление Browser Monitor или Clickpath в организации.

### Аналитика через Account Management

Менеджеры лицензий могут просматривать использование и затраты в Account Management.
Перейдите в [**Account Management**﻿](https://myaccount.dynatrace.com/) > **Subscription** > **Overview** > **Cost and usage details** > **Usage summary** и выберите возможность **Browser Monitor and Clickpath**.

Дополнительную информацию см. в разделе [Overview (Dynatrace Platform Subscription)](/managed/manage/account-management/license-subscription/subscription-overview-dps "View your Dynatrace Platform Subscription (DPS) budget summary and cost analysis.").

### Аналитика через события использования для выставления счетов

События использования для выставления счетов (BUE, `billing_usage_event`), это системные события, отправляемые Dynatrace в пространство данных `dt.system.events`.
Можно использовать DQL для запроса BUE и анализа использования и стоимости возможностей Real User Monitoring без повторного применения правил тарификации или логики подсчёта сессий.

BUE представляют уже рассчитанное, тарифицируемое использование для возможностей DPS (а не какую-либо конфигурацию или потенциальное использование) и соответствуют тому, что отображается в Account Management и в счетах.
Поэтому они являются рекомендуемым источником данных для понимания соответствующего потребления.

События использования для выставления счетов содержат:

* Какая возможность DPS была использована.
* Объём использования, учитываемый при тарификации.
* Временной интервал, к которому относится использование.
* Контекст сущности, к которой отнесено использование (например, приложение).

#### Запрос событий использования для выставления счетов с помощью DQL

Можно использовать события использования для выставления счетов как авторитетный источник при построении представлений распределения затрат, анализа использования или прозрачности стоимости.

Например, агрегировать по приложениям, чтобы понять, какие приложения больше всего влияют на использование и стоимость.

Ниже приведены примеры запросов DQL для различных сценариев использования.
Эти запросы можно использовать как есть или изменять под свои нужды.

1. Общее использование во времени

   ```
   fetch dt.system.events



   | filter event.kind == "BILLING_USAGE_EVENT" and event.type == "Browser Monitor and Clickpath"



   | dedup event.id



   | summarize totalUsage = sum(billed_sessions), by:{bin(timestamp, 1d)}
   ```

## Часто задаваемые вопросы

Потребляются ли синтетические действия при запуске тестов по требованию?

Да.
Выполнения по требованию из назначенных локаций учитываются так же, как и запланированные выполнения.

Потребляют ли неудавшиеся действия ресурсы?

Да.
Действия учитываются с момента начала их выполнения.
Однако любые шаги, пропущенные из-за сбоя, ресурсы не потребляют.

Учитывает ли Dynatrace внутренние сетевые вызовы браузера как действия?

Нет.
Учитываются только веб-запросы, инициированные непосредственно синтетическими действиями пользователя (такие как загрузка страницы или явные вызовы XHR).

Потребляют ли ресурсы отключённые мониторы?

Нет.
Отключённые мониторы прекращают все выполнения и не создают никакого потребления.
