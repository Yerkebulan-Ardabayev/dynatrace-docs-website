# Диагностика ActiveGate

# Диагностика ActiveGate

* 9-min read
* Updated on Feb 24, 2026

Вы можете выполнить полностью автоматизированную диагностику ActiveGate для окружений Dynatrace SaaS и Managed.

Этот рабочий процесс позволяет:

* Автоматически выявлять проблемы, связанные с ActiveGate, в высокодинамичных окружениях в определённый момент времени
* Легко собирать диагностические данные для конкретного объекта и автоматически получать возможные решения обнаруженных аномалий
* Быстро самостоятельно устранять типичные проблемы, сокращая время диагностики
* Предоставлять специалистам Dynatrace Support все необходимые данные для диагностики

Если ActiveGate не может подключиться к окружению Dynatrace или не запускается, может потребоваться сбор диагностических данных локально из командной строки. Подробнее смотрите в разделе [Сбор диагностических данных с помощью `agctl`](#collect-diagnostic-data-with-agctl).

## Требования

* Разрешение окружения **View sensitive request data**
* Для ActiveGate с поддержкой нескольких окружений диагностику можно запустить только для основного окружения, определённого в [конфигурации ActiveGate](/managed/ingest-from/dynatrace-activegate/configuration/configure-an-environment-activegate-for-multi-environment-support "Пошаговая процедура настройки одного Environment ActiveGate для поддержки нескольких окружений.").

## Автоматический анализ

Эта процедура описывает стандартный процесс: Dynatrace собирает диагностические данные для ActiveGate и немедленно анализирует их.

Если вы предпочитаете сначала собрать и просмотреть данные, а затем вручную отправить их в Dynatrace для анализа, смотрите раздел [Сбор и просмотр локально](#collect-and-review-locally).

1. Перейдите в **Deployment Status** > **ActiveGates**.
2. Разверните запись нужного ActiveGate и выберите **Run ActiveGate diagnostics**.
3. На странице **Run Dynatrace ActiveGate diagnostics** кратко опишите, что не работает должным образом.
4. Необязательно: по умолчанию для анализа собираются данные за 7 дней. Если нужно больше данных, выберите ссылку **Advanced options**, измените количество дней и нажмите **Apply**.
5. Выберите **Start analysis**.

### Что происходит далее

Dynatrace выполняет следующее:

* Собирает диагностические данные за последние 7 дней (если значение по умолчанию не изменено) для затронутого ActiveGate
* Сохраняет собранные диагностические данные
* Загружает диагностические данные в корзину S3 в регионе AWS вашего окружения для дальнейшего анализа

Столбец **State** описывает текущую фазу процесса.

**State** не обновляется автоматически. Нажмите **Refresh** для проверки изменения состояния.

Collecting (Сбор данных)

Идёт сбор данных.
Во время сбора данных вы можете:

* **Refresh** для обновления прогресса.
* **Cancel** для отмены сбора диагностических данных.

Collected (Данные собраны)

Dynatrace завершил сбор диагностических данных.
После сбора данных вы можете:

* **Analyze** для отправки данных в Dynatrace на анализ.
* **Download** для локальной загрузки данных.
* **Delete** для удаления проблемы вместе с диагностическими данными.

Sending in progress (Отправка данных)

Диагностические данные передаются в Dynatrace для анализа.
Во время отправки вы можете:

* **Refresh** для обновления прогресса.
* **Download** для загрузки диагностических данных.
* **Delete** для удаления проблемы вместе с диагностическими данными.

Sent to Dynatrace cloud (Отправлено в облако Dynatrace)

Диагностические данные переданы в Dynatrace для анализа.

Analyzing (Анализ)

Dynatrace анализирует диагностические данные.
Во время анализа вы можете:

* **Refresh** для обновления прогресса.
* **Download** для загрузки диагностических данных.
* **Delete** для удаления проблемы вместе с диагностическими данными.

Analyzed (Анализ завершён)

Анализ завершён. Количество связанных оповещений указано в скобках.
После анализа вы можете:

* **Download** для загрузки диагностических данных.
* **Delete** для удаления проблемы вместе с диагностическими данными.

Delete in progress (Удаление)

Диагностические данные удаляются. Во время удаления вы можете:

* **Refresh** для обновления прогресса.

Deleted (Удалено)

Диагностические данные удалены. Dynatrace сохраняет только минимальный набор сведений: кто, когда, где и зачем собирал данные.

Canceled (Отменено)

Процесс диагностики был отменён вручную до завершения.

### Просмотр результатов анализа Dynatrace

После завершения анализа Dynatrace отправляет результаты обратно в ваше окружение. При обнаружении возможного решения Dynatrace отображает его в разделе **Alerts section**.

## Сбор и просмотр локально

Эта процедура описывает, как собирать диагностические данные для ActiveGate. Используйте этот вариант, если хотите сначала собрать и просмотреть данные перед отправкой в Dynatrace для анализа.

Если вы хотите собирать данные и отправлять их в Dynatrace автоматически, смотрите раздел [Автоматический анализ](#analyze-automatically).

1. Перейдите в **Deployment Status** > **ActiveGates**.
2. Разверните запись нужного ActiveGate и выберите **Run ActiveGate diagnostics**.
3. На странице **Run Dynatrace ActiveGate diagnostics** кратко опишите, что не работает должным образом.
4. Выберите ссылку **Advanced options**.
5. Выберите **and store locally**.

   * Здесь же можно изменить количество дней для сбора данных (по умолчанию: `7 дней`).
6. Выберите **Apply**.
7. Выберите **Start collection** для сбора диагностических данных и их локального сохранения.

### Что происходит далее

Dynatrace выполняет следующее:

* Собирает диагностические данные за последние 24 часа (если значение по умолчанию не изменено) для затронутого ActiveGate
* Сохраняет собранные диагностические данные

Столбец **State** описывает текущую фазу процесса.

**State** не обновляется автоматически. Нажмите **Refresh** для проверки изменения состояния.

Collecting (Сбор данных)

Идёт сбор данных.
Во время сбора данных вы можете:

* **Refresh** для обновления прогресса.
* **Cancel** для отмены сбора диагностических данных.

Collected (Данные собраны)

Dynatrace завершил сбор диагностических данных.
После сбора данных вы можете:

* **Analyze** для отправки данных в Dynatrace на анализ.
* **Download** для локальной загрузки данных.
* **Delete** для удаления проблемы вместе с диагностическими данными.

### Что делать с собранными данными

После сбора данных вы можете:

* **Download** — загрузить собранные данные.

  + Просмотрите данные. Обзор содержимого смотрите в разделе [Содержимое диагностических данных](#contents).
  + Приложите данные к заявке в службу поддержки.
* **Analyze** — проанализировать данные.
* **Delete** — удалить проблему вместе с диагностическими данными.

### Диагностика ActiveGate в Dynatrace Managed air-gapped окружениях

В Dynatrace Managed air-gapped окружении:

1. Используйте параметр **Store locally** в разделе **Advanced options**, как описано выше.
2. После сбора диагностических данных добавьте их в заявку в службу поддержки.
3. Dynatrace может получить диагностические данные из заявки, проанализировать их и предоставить автоматизированную обратную связь специалистам Dynatrace Support об обнаруженных аномалиях.

На протяжении всего процесса применяются и фиксируются строгие меры защиты персональных данных.

## Сбор диагностических данных с помощью `agctl`

ActiveGate версии 1.333+

В случаях, когда ActiveGate не может подключиться к окружению Dynatrace или не запускается, может потребоваться локальный сбор диагностических данных из командной строки на хосте ActiveGate. Для этого используйте инструмент командной строки `agctl`, чтобы собрать все необходимые данные и создать пакет для последующего анализа.

Подробнее об `agctl` смотрите в разделе [agctl — интерфейс командной строки для ActiveGate](/managed/ingest-from/dynatrace-activegate/agctl-command-line-interface "Узнайте, как использовать agctl для настройки и управления ActiveGate из командной строки").

### Использование команды `agctl support-archive`

Используйте команду `agctl support-archive` для генерации архива поддержки с диагностическими данными.

Подробнее о команде `support-archive` смотрите в разделе [Support archive](/managed/ingest-from/dynatrace-activegate/agctl-command-line-interface#support-archive "Узнайте, как использовать agctl для настройки и управления ActiveGate из командной строки").

Для сбора диагностических данных выполните команду `support-archive` с операцией `create`.

```
agctl support-archive create [--directory=<PATH> --days=<NUMBER> --modules=<MODULES> --stdout]
```

* По умолчанию диагностический пакет записывается в текущий рабочий каталог.

  + Для изменения целевого каталога используйте параметр `--directory=<PATH>`.
  + После успешного создания файла `agctl` выводит путь к нему.
  + Убедитесь, что у вас есть права на запись в целевой каталог.
* По умолчанию собираются данные за 30 дней. Для изменения используйте параметр `--days=<NUMBER>`.
* По умолчанию диагностические данные собираются из всех модулей. Для изменения используйте параметр `--modules=<MODULES>` с разделённым запятыми списком модулей (например, `--modules=zremote,synthetic`).

### Использование команды `agctl support-archive` из контейнерного ActiveGate

Команду `support-archive` можно использовать для сбора диагностических данных из контейнерного ActiveGate с помощью параметра `--stdout`, который выводит архив поддержки напрямую и позволяет перенаправить его в файл на локальном компьютере.

Kubernetes

OpenShift

```
kubectl exec -n <namespace> <pod-name> -- agctl support-archive create --stdout 1 > ag-support-archive.zip 2 > output-logs.txt
```

```
oc exec -n <namespace> <pod-name> [-c <container>] -- agctl support-archive create --stdout 1 > ag-support-archive.zip 2 > output-logs.txt
```

Содержимое архива поддержки записывается в `stdout`, что позволяет перенаправить его в ZIP-файл. Прочий вывод отправляется в `stderr` для сохранения целостности архивного файла.

Windows PowerShell не поддерживается

На Windows используйте командную строку (`cmd.exe`); PowerShell не поддерживается.

### Сбор диагностических данных с помощью `agctl` для более ранних версий

ActiveGate версии 1.331 и ниже

Для более ранних версий ActiveGate используйте `agctl` с устаревшей командой `create-support-archive`.

#### Использование устаревшей команды `agctl create-support-archive` на Linux или Windows

Linux

Windows

`agctl` находится в [директории установки ActiveGate](/managed/ingest-from/dynatrace-activegate/configuration/where-can-i-find-activegate-files "Узнайте, где хранятся файлы ActiveGate на Windows и Linux.").

`agctl` должен запускаться [от того же пользователя, что и процесс ActiveGate](/managed/ingest-from/dynatrace-activegate/installation/linux/linux-customize-installation-for-activegate#user-service "Узнайте о параметрах командной строки, которые можно использовать с ActiveGate на Linux.") (по умолчанию: `dtuserag`).

**Синтаксис**

```
sudo -u dtuserag /opt/dynatrace/gateway/agctl create-support-archive [--directory=<PATH> --days=<NUMBER> --modules=<MODULES>]
```

`agctl.bat` должен запускаться с правами администратора.

**Синтаксис**

```
"C:\Program Files\dynatrace\gateway\agctl.bat" create-support-archive [--directory=<PATH> --days=<NUMBER> --modules=<MODULES>]
```

* По умолчанию диагностический пакет записывается в текущий рабочий каталог.
* По умолчанию собираются данные за 30 дней. Для изменения используйте `--days=<NUMBER>`.
* По умолчанию данные собираются из всех модулей. Для изменения используйте `--modules=<MODULES>` с разделённым запятыми списком модулей.

#### Использование устаревшей команды `agctl create-support-archive` из контейнерного ActiveGate

ActiveGate версии 1.325+

Вы можете использовать параметр `--stdout` для вывода архива поддержки напрямую с перенаправлением в файл.

Kubernetes

OpenShift

```
kubectl exec -n <namespace> <pod-name> -- /opt/dynatrace/gateway/agctl create-support-archive --stdout 1 > ag-support-archive.zip 2 > output-logs.txt
```

```
oc exec -n <namespace> <pod-name> [-c <container>] -- /opt/dynatrace/gateway/agctl create-support-archive --stdout 1 > ag-support-archive.zip 2 > output-logs.txt
```

ActiveGate версии 1.323 и ниже

Используйте `agctl` с параметром `--directory`, затем перенесите файл с помощью кодировки base64.

Kubernetes: выполните команду, затем перенесите созданный файл на хост через base64.

OpenShift: аналогично Kubernetes, но с командой `oc exec`.

После переноса декодируйте файл (Linux: `base64 --decode`, Windows: через PowerShell `[Convert]::FromBase64String`).

### Что делать с собранными данными

После сбора данных:

* Просмотрите данные. Обзор содержимого смотрите в разделе [Содержимое диагностических данных](#contents).
* Добавьте данные в заявку в службу поддержки.

Рекомендуется удалить файл после использования.

## Содержимое диагностических данных

Все собранные диагностические данные сжимаются в ZIP-файл `SupportArchive<ID number>` (для `agctl`: `support_archive_<timestamp>`), который содержит следующие папки и файлы:

| Папка или файл | Описание |
| --- | --- |
| `details.txt` (файл) | Содержит общую информацию о том, когда и где были собраны диагностические данные, и статистику архива. |
| `config` (папка) | Содержит снимок [директории конфигурации ActiveGate](/managed/ingest-from/dynatrace-activegate/configuration/where-can-i-find-activegate-files "Узнайте, где хранятся файлы ActiveGate на Windows и Linux."). |
| `debugui` (папка) | Содержит снимок внутренней конфигурации окружения, связанной с ActiveGate. |
| `log` (папка) | Содержит снимок [директории логов ActiveGate](/managed/ingest-from/dynatrace-activegate/configuration/where-can-i-find-activegate-files "Узнайте, где хранятся файлы ActiveGate на Windows и Linux."). |
| `autoupdater` (папка) | Содержит логи службы AutoUpdater и модули установки RPM, Synthetic, zRemote. |
| `install` (папка) | Содержит логи установщика ActiveGate. |
| `zremote` (папка) | Содержит логи zRemote, watchdog и файлы конфигурации агента. |
| `synthetic` (папка) | Содержит файлы конфигурации и логи синтетических мониторов ActiveGate. |
| `remotepluginmodule` (папка) | Содержит логи расширений и агентов, их выполняющих. |

## Конфиденциальность данных

Для соблюдения региональных норм защиты данных и конфиденциальности Dynatrace автоматически удаляет все диагностические данные через 30 дней после их сбора. Это применяется как к данным в вашем окружении Dynatrace, так и на кластере Dynatrace.

Вы можете удалить собранные диагностические данные раньше. Для обеспечения прозрачности Dynatrace сохраняет только минимальный набор сведений: кто, когда, где и зачем собирал диагностические данные.

Подробнее о конфиденциальности данных Dynatrace смотрите в разделе [Сроки хранения данных](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#diagnostics "Проверьте сроки хранения для различных типов данных.").

## Устранение неполадок

Автоматический анализ:

* Status: «Collecting of the diagnostic data wasn't possible within 20 minutes.»
* «State» appears to be frozen.

agctl:

* Message: «Unexpected response status code = 500, cannot create SupportArchive file.»

## FAQ

Могу ли я получить прямой доступ к S3 или использовать собственный S3?

Нет, прямой доступ к S3 и использование собственного S3 недоступны.

Диагностические данные загружаются в корзину S3 Dynatrace, настроенную для окружения/кластера компанией Dynatrace. Используемая корзина S3 зависит от расположения окружения/кластера.
