# Помощь в разработке

* Пояснение
* Чтение: 10 мин
* Опубликовано 16 июня 2025 г.

**Dynatrace Extensions** использует все возможности редактора VS Code, чтобы помочь создавать более качественные расширения быстрее.

Читайте дальше, чтобы узнать, как максимально эффективно использовать эти функции.

## Обзор функций

| Тип функции | Описание |
| --- | --- |
| [Дополнение кода](#code-completions) | Предложения по автодополнению слов или полей манифеста |
| [Действия с кодом](#code-actions) | Команды для создания или редактирования содержимого манифеста расширения |
| [Code lens](#code-lens) | Дополнительные команды для взаимодействия с данными расширения |
| [Режим быстрой разработки](#fast-development-mode) | Рабочий процесс, ориентированный на скорость доставки |
| [Диагностика](#diagnostics) | Дополнительная диагностика помимо базовой проверки схемы |

## Дополнение кода

Дополнение кода или подсказки срабатывают в ключевых точках манифеста расширения. В автоматическом режиме или по запросу (Ctrl + Space) предлагаются известные значения, которые можно вставить в позицию курсора.

### Реализованные триггеры

| Ключевое слово-триггер | Эффекты |
| --- | --- |
| При `fromType:` или `toType:` внутри элементов списка `topology.relationships` | Просмотр встроенных и пользовательских типов сущностей |
| При `sourceAttribute:` внутри элементов списка `mappingRules` топологических связей | Просмотр атрибутов сущности (сущность должна присутствовать в атрибуте `fromType`) |
| При `destinationAttribute:` внутри элементов списка `mappingRules` топологических связей | Просмотр атрибутов сущности (сущность должна присутствовать в атрибуте `toType`) |
| При `entityType:` и `entityTypes:` | Просмотр соответствующих типов сущностей |
| При `key:` (атрибутов внутри свойств экрана) | Атрибуты предлагаются из топологии и встроенных значений |
| При `entitySelectorTemplate:` | Используйте Ctrl + Space для вызова дополнений при построении селектора или выберите один из готовых селекторов (из связей, видимых в манифесте) |
| При `iconPattern:` (в правилах топологии) или `icon:` (в заголовке staticContent) | Просмотр доступных кодов иконок Barista |
| При `key:` (карточек в `screens`, как в макете, так и в отдельных списках) | Просмотр ключей определённых, но ещё не используемых карточек |
| При `value:` для метрик и измерений расширения Prometheus | Просмотр метрик и измерений, собранных с помощью Prometheus code lens |
| При `description:` (в разделе `metrics` манифеста) | Для метрик, собранных с помощью Prometheus code lens, добавляет описание из собранных данных |

## Действия с кодом

Действия с кодом срабатывают на ключевых строках манифеста расширения. Редактор автоматически отображает значок лампочки
когда действия применимы к выбранной строке. Как правило, они создают и вставляют содержимое в
манифест расширения или исправляют проблемы, выделенные **Dynatrace Extensions**.

### Текущие реализованные триггеры

| Триггер действия | Эффекты |
| --- | --- |
| Внутри `propertiesCard` при нажатии на свойства | Автоматически добавляет свойства для атрибутов и связей сущности |
| Внутри `entitiesListCards` при нажатии на столбцы | Автоматически добавляет столбцы для атрибутов и связей перечисленной сущности |
| Внутри `chartsCards` и `entitiesListCards` при нажатии на диаграммы внутри карточки | Автоматически добавляет диаграммы для метрик, которых ещё нет в карточке |
| Внутри `graphChartConfig` при нажатии на метрики | Добавляет дополнительные метрики в диаграмму, которые не используются в окружающей карточке |
| Внутри `screens` при нажатии на `chartsCards` | Автоматически добавляет карточки с диаграммами для целых наборов функций метрик |
| Внутри `screens` при нажатии на `entitiesListCards` | Автоматически добавляет карточки для перечисления данной сущности и связанных с ней |
| При нажатии на `metrics` в источнике данных Prometheus | Автоматически добавляет сведения из [собранного эндпоинта Prometheus](#prometheus) |
| На `screens` | Автоматически создаёт полные экраны унифицированного анализа для сущностей |
| Внутри `entitiesListCards` при нажатии на `filtering` | Вставляет целые блоки фильтрации с фильтром по умолчанию по имени |
| Внутри `entitiesListCards` и внутри `filtering` при нажатии на `filters` | Вставляет отдельный фильтр для атрибутов сущности |
| Внутри `screens` при нажатии на `actions` | Вставляет глобальные действия для настройки расширения |
| Внутри `actions` при нажатии на `actions` | Вставляет выражение действия для настройки расширения |

## Code lens

Code lens: применимая контекстная информация, встроенная в код. В **Dynatrace Extensions** они помогают запускать
действия, связанные с кодом, в среде Dynatrace или на других внешних эндпоинтах.

### Селекторы метрик

Lens появляется автоматически везде, где встречается `metricSelector`, и отображает:

* Query data: действие, которое выполняет запрос селектора метрик и визуализирует его результаты
* Validate selector: действие для проверки селектора в среде Dynatrace
* Значок, показывающий статус последней проверки данного селектора. Наведите курсор для получения подробностей

Результаты селектора метрик отображаются в отдельной панели метрик.

Любые ошибки при запросе среды отображаются в панели вывода Dynatrace (по умолчанию внизу редактора).

### Селекторы сущностей

Lens появляется автоматически везде, где упоминается `entitySelector`, и отображает:

* Query data: действие, которое выполняет запрос селектора сущностей и визуализирует его результаты
* Validate selector: действие для проверки селектора в среде Dynatrace
* Значок, показывающий статус последней проверки данного селектора. Наведите курсор для получения подробностей

Результаты селектора сущностей, а также любые ошибки отображаются в панели вывода Dynatrace (по умолчанию внизу редактора).

### Prometheus

Lens появляется автоматически, когда определён источник данных Prometheus, и отображает:

* Scrape data: действие, которое подключается к эндпоинту сборщика Prometheus и собирает его данные
* Edit config: действие, позволяющее вносить изменения в подключённый эндпоинт Prometheus
* Текст, показывающий статус последнего сбора данных: временную метку и количество собранных метрик

После сбора данных их можно использовать с [действиями с кодом](#code-actions) для:

* вставки определений метрик в раздел `prometheus` манифеста
* вставки измерений в раздел `prometheus` манифеста
* вставки метаданных метрик в раздел `metrics` манифеста

### Запросы Windows Management Interface (WMI)

Lens появляется автоматически над запросами внутри определения источника данных WMI и отображает:

* Run WMI Query: действие, которое выполняет запрос к локальной машине Windows
* Текст с сводкой результатов запроса (то есть количество найденных экземпляров объектов)

Полные результаты выполнения запроса WMI отображаются в отдельной панели. После выполнения запроса результаты можно использовать с [дополнением кода](#code-completions).

### Экраны унифицированного анализа

Lens появляется автоматически над определениями типов сущностей внутри элемента `screens` манифеста
и отображает:

* Open List View: действие для открытия браузера с [представлением списка унифицированного анализа](/managed/ingest-from/extend-dynatrace/extend-ui/unified-analysis#page-types "Расширяйте веб-интерфейс Dynatrace с помощью страниц унифицированного анализа, адаптированных для конкретных сущностей.") данной сущности
* Open Details view: действие для открытия браузера с [представлением сведений унифицированного анализа](/managed/ingest-from/extend-dynatrace/extend-ui/unified-analysis#page-types "Расширяйте веб-интерфейс Dynatrace с помощью страниц унифицированного анализа, адаптированных для конкретных сущностей.") данной сущности

Код ответа 404 ожидается, если тип сущности ещё не существует.

## Режим быстрой разработки

Режим быстрой разработки: рабочий процесс, позволяющий опытным разработчикам получать немедленную обратную связь о текущем состоянии расширения и сокращать количество шагов и время, необходимое для просмотра обновлений в подключённой среде Dynatrace.

### Как это работает?

При включении каждый раз при сохранении манифеста расширения версия расширения автоматически увеличивается, расширение упаковывается, подписывается и загружается в подключённую среду. Рабочий процесс **полностью автоматизирован**: при достижении максимального количества версий расширения одна из них будет удалена автоматически, чтобы загрузка прошла успешно.

Сопутствующая строка состояния подтверждает активность режима и отображает статус последней попытки сборки.

В этом рабочем процессе предварительная проверка перед загрузкой пропускается в пользу скорости, а все проблемы немедленно сообщаются через канал вывода.

При разработке статических ресурсов, например экранов унифицированного анализа, этот режим является самым быстрым способом
перебрать различные изменения перед нахождением оптимальной конфигурации.

## Диагностика

Нередко абсолютно корректный YAML создаёт манифест, который вызывает проблемы при загрузке
расширения в Dynatrace или при его последующем выполнении. Многие из таких ситуаций можно выявить заблаговременно, и **Dynatrace Extensions** стремится сделать их явными, чтобы можно было исправить и сократить количество
неудачных попыток развёртывания.

### Как работает пользовательская диагностика?

При каждом редактировании манифеста расширения набор диагностических инструментов запускает обновление всех диагностических
элементов, применимых к данному файлу. По результатам проверки соответствующее содержимое выделяется в манифесте, а
наведение курсора на выделенный фрагмент предоставляет подробности о проблеме.

Для некоторых проблем в составе информации при наведении курсора может присутствовать ссылка **Quick fix**. При её наличии
она инициирует изменение содержимого документа для устранения выделенной проблемы.

Уровни серьёзности диагностики:

* `Error`: эти результаты выделяются красным и представляют проблемы, которые нарушат работу расширения при развёртывании. [Команда Build](/managed/ingest-from/extensions/develop-your-extensions/addon-for-vscode/commands#build "Обзор всех команд, доступных в Dynatrace Extensions") не будет работать, если в манифесте есть диагностика уровня `Error`.
* `Warning`: эти результаты выделяются жёлтым и представляют проблемы, которые не остановят работу расширения, но могут вызывать нежелательное поведение.

### Текущие реализованные диагностики

| Код | Серьёзность | Что это означает? |
| --- | --- | --- |
| DED001 | `Error` | У расширения нет имени, которое является обязательным. |
| DED002 | `Error` | Имя расширения должно содержать менее 50 символов. |
| DED003 | `Error` | Имя расширения недопустимо. Оно должно содержать только строчные буквы, цифры, дефисы, символы подчёркивания или точки. |
| DED004 | `Error` | Имя расширения должно начинаться с custom:, но это условие не выполнено. |
| DED005 | `Warning` | Имена внутренних расширений Dynatrace не должны начинаться с custom:. |
| DED006 | `Warning` | Метрики типа count должны иметь ключи, оканчивающиеся на **.count** или **\_count**. |
| DED007 | `Warning` | Метрики типа gauge не должны иметь ключи, оканчивающиеся на **.count** или **\_count**. |
| DED008 | `Error` | Этот ключ карточки используется в макете экрана, но не имеет определения. |
| DED009 | `Warning` | Карточка определена, но не используется в макете экрана. |
| DED010 | `Warning` | Отсутствуют онлайн-данные об этом идентификаторе объекта (OID). Рекомендуется проверить его. |
| DED011 | `Error` | Этот OID недоступен для чтения. Разрешения доступа ([MAX-ACCESS](https://learn.microsoft.com/en-us/windows/win32/wmisdk/access-and-max-access-clauses)) не допускают это. |
| DED012 | `Error` | Этот OID возвращает строку, но используется как значение метрики. |
| DED013 | `Warning` | Этот OID возвращает Counter, но используется как метрика Gauge. |
| DED014 | `Warning` | Этот OID возвращает Gauge, но используется как метрика Counter. |
| DED015 | `Error` | Недопустимый синтаксис OID. OID не должен начинаться или заканчиваться на '.' и может содержать только точки и цифры. |
| DED016 | `Error` | Недопустимый синтаксис OID. OID не должен заканчиваться на '.0', если в подгруппе для 'table' задано значение 'true'. |
| DED017 | `Error` | Недопустимый синтаксис OID. OID должен заканчиваться на '.0', если в подгруппе для 'table' задано значение 'false'. |
| DED018 | `Error` | Онлайн-данные указывают, что этот OID является статическим, но он используется внутри подгруппы 'table'. |
| DED019 | `Error` | Онлайн-данные сопоставляют этот OID с записями таблицы, но он не используется внутри подгруппы 'table'. |

Код сам по себе ничего не означает: он лишь предоставляет уникальный идентификатор среди всех диагностических сообщений, которые могут появиться
в VS Code. Префикс **DED** выбран как сокращение от Dynatrace Extensions Diagnostic.
