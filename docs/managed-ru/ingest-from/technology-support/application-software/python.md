# Python

* Справочник
* Чтение: 2 мин
* Обновлено 5 ноября 2025 г.

Вы можете отправлять данные из вашего [Python](https://python.org)-приложения в Dynatrace. Python, это универсальный высокоуровневый язык программирования, известный своей читаемостью и простотой; он часто используется для веб-разработки, анализа данных, искусственного интеллекта и научных вычислений благодаря обширным библиотекам и поддержке сообщества.

## Возможности

* Автоматически внедряется в процессы Python и поддерживаемые контейнеры. Правила внедрения позволяют исключать или включать определённые процессы и контейнеры.
* Отслеживает использование CPU, потребление памяти и отзывчивость процессов Python в реальном времени.
* Мониторит рабочие нагрузки Python, выполняющиеся внутри контейнеров, захватывая специфичные для контейнеров метрики производительности и метаданные о технологии и окружении контейнера.
* Предоставляет метрики среды выполнения, включая активность сборки мусора (GC), метрики потоков и версии загруженных библиотек.
* Предоставляет удалённые обновления конфигурации, позволяющие настраивать параметры мониторинга удалённо через OneAgent. Изменения применяются во время выполнения (без перезапуска процесса, когда это возможно).
* Генерирует диагностические данные для устранения неполадок, включая архивы поддержки, доступные через интерфейс [OneAgent Diagnostics](/managed/ingest-from/dynatrace-oneagent/oneagent-troubleshooting/oneagent-diagnostics "Узнайте, как запускать диагностику OneAgent").
* Полное сквозное трассирование транзакций для запросов к веб-сервисам, удалённым сервисам, очередям сообщений, базам данных и другим. Подробнее см. в разделе [Сервисы](/managed/observe/application-observability/services "Узнайте, как мониторить и анализировать сервисы, задавать и использовать атрибуты запросов и многое другое.").
* Обнаружение уязвимостей в стороннем коде. Подробнее см. в разделе [Runtime Vulnerability Analytics](/managed/secure/application-security/vulnerability-analytics "Мониторьте, визуализируйте, анализируйте и устраняйте уязвимости в стороннем коде и на уровне кода, отслеживайте прогресс устранения и создавайте правила мониторинга.").
* [Поддержка OpenTelemetry](https://github.com/open-telemetry/opentelemetry-python/) для захвата трассировок. Подробнее см. в разделе [Инструментирование Python-приложения с OpenTelemetry](/managed/ingest-from/opentelemetry/walkthroughs/python "Узнайте, как инструментировать Python-приложение с помощью OpenTelemetry и Dynatrace.").
* [OneAgent SDK](/managed/ingest-from/extend-dynatrace/extend-tracing/oneagent-sdk "Dynatrace OneAgent SDK позволяет вручную инструментировать приложение, чтобы расширить сквозную видимость для фреймворков и технологий, для которых ещё нет готового модуля кода.") для пользовательской трассировки.

  Если вы уже используете [OneAgent SDK for Python](https://github.com/Dynatrace/OneAgent-SDK-for-Python) или [OneAgent SDK Python auto-instrumentation](https://github.com/dynatrace-oss/OneAgent-SDK-Python-AutoInstrumentation), мы рекомендуем перейти на автоматическое инструментирование с помощью OneAgent for Python.

Подробности о поддерживаемых технологиях, используемых вместе с Python, см. в [матрице поддерживаемых технологий](/managed/ingest-from/technology-support#python "Технические подробности о поддержке Dynatrace для конкретных платформ и фреймворков разработки.").

## Включение мониторинга Python

Чтобы активировать мониторинг Python

1. Перейдите в **Settings** > **Monitoring** > **Monitored technologies**.
2. Найдите **Python** в списке поддерживаемых технологий и выберите **Edit**.
3. Включите **Monitor Python**.
4. [Создайте правило мониторинга процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-monitoring#rules "Способы настройки мониторинга групп процессов"), чтобы включить глубокий мониторинг выбранных процессов.

## Совместимость модуля кода OneAgent Python и OneAgent SDK for Python

Модуль кода OneAgent Python поддерживает приложения, инструментированные OneAgent SDK for Python. Оба могут работать вместе в одном приложении.  
[Dynatrace OneAgent SDK for Python](https://github.com/Dynatrace/OneAgent-SDK-for-Python), это обёртка над Dynatrace OneAgent SDK for C/C++.

Когда установлен модуль кода Python, он внутренне заменяет Dynatrace OneAgent SDK for C/C++ и собирает данные, создаваемые OneAgent SDK for Python. Это означает, что трассировки, создаваемые SDK, будут связаны с трассировками, создаваемыми самим модулем кода Python.

## Горячие точки методов

Модуль кода Python может захватывать трассировки стека Python в фоновом режиме для [функции горячих точек методов](/managed/observe/application-observability/profiling-and-optimization/cpu-profiling "Узнайте, как использовать Dynatrace для расширенного анализа кода.").

### Предварительные требования для горячих точек методов

* Требуется Python версии 3.12+.
* В следующих случаях модуль кода OneAgent Python должен быть переключён в режим без поддержки fork (non-forkable):

  + Горячие точки методов на уровне процесса (все версии OneAgent)
  + Горячие точки методов на уровне сервиса (только OneAgent версии 1.326 и более ранних)

  Чтобы переключиться в режим без поддержки fork, задайте переменной окружения `DT_PYTHON_FORKABLE` значение `0`. После установки переменной окружения перезапустите приложение. Также обязательно ознакомьтесь с [разделом ограничений ниже](#method-hotspot-limitations).

### Ограничения для горячих точек методов {#method-hotspot-limitations}

* После переключения в режим без поддержки fork приложение будет многопоточным даже во время форка. Убедитесь, что форки не выполняют код Python без предварительного вызова системного вызова `exec`.
* Сэмплы берутся асинхронно, и выполнение программы не останавливается. Поэтому стеки могут быть неполными или даже содержать некорректные ассоциации вызовов.

## Специальные функции связывания трассировок

Некоторые подключаемые функции агента Python, связанные со связыванием трассировок, рассчитаны на конкретные сценарии использования, и агент не определяет автоматически, соответствует ли фактическое использование этим сценариям.

Это функции Python queue, Python threading и Python subprocess.

### Python queue

Используйте эту функцию для трассировки потока задач между производителями и потребителями с помощью стандартной библиотеки Python `queue.Queue`.

Включите её, если применимо следующее:

* Вы хотите использовать `queue.Queue` для распределения задач или похожих единиц работы между производителями и потребителями.
* Вы хотите продолжить трассировку от производителя (где объект помещается) к потребителю (где он извлекается с помощью `get`).

Не включайте её, если:

* Потребитель выполняет несвязанную работу после извлечения элемента из очереди.
* Вы или используемая вами библиотека используете очереди нестандартным образом.
* Вы используете API очередей, не документированные в стандартной библиотеке Python для `queue.Queue`.

### Python threading

Используйте эту функцию для продолжения трассировок между потоками в приложениях Python. Когда новый поток запускается с помощью `Thread.start`, эта функция создаёт связь от вызывающего потока к вновь созданному.

Включите её, если применимо следующее:

* Вы хотите обработать трассируемый запрос, и нужно, чтобы трассировка продолжилась в новом потоке.
* Вы хотите продолжить трассировку в новый поток, запускаемый в рамках обработки исходного запроса.

Не включайте её, если:

* Вы можете использовать альтернативные средства, такие как `concurrent.futures`, для установления связи трассировки.
* Новый поток выполняет работу, не связанную с исходным запросом.

### Python subprocess

Используйте эту функцию для создания связи трассировки между родительским процессом и запущенным из него подпроцессом. Это гарантирует, что контекст трассировки сохраняется, когда подпроцесс запускается во время обработки трассируемого запроса.

Включите её, если применимо следующее:

* Вы хотите обработать трассируемый запрос в родительском процессе.
* Вы хотите продолжить трассировку в подпроцесс.

Не включайте её, если:

* Вы можете использовать альтернативные средства, такие как процессы `concurrent.futures` в Python, для установления связи трассировки.
* Подпроцесс выполняет работу, не связанную с исходным запросом.

Когда мониторируемое приложение не соответствует требованиям функции во всех случаях использования соответствующих API, к типичным проблемам относятся: связывание несвязанных трассировок вместе, привязка несвязанных внутренних фоновых операций к трассировкам, чрезмерно большой размер трассировок или неожиданно большая их длительность. Кроме того, функции OpenTelemetry или Python SDK могут неожиданно захватывать или подавлять некоторые спаны.

## Ограничения

* Поддерживается только стандартный интерпретатор CPython (реализация с [Python.org](https://python.org)).
* Сборки без GIL (nogil) не поддерживаются. Имена бинарных файлов версии nogil имеют суффикс `t`, например `Python3.13t`.
* [PEP 703 - Making the Global Interpreter Lock Optional in Python](https://peps.python.org/pep-0703/) не поддерживается.
* Мониторинг процессов с установленным пакетом [Gevent](https://www.gevent.org/) версий ниже 20.9.0 не поддерживается.
* Для приложений, создающих [форкнутые](https://docs.python.org/3/library/os.html#os.fork) дочерние процессы: модуль кода использует фоновые потоки. Чтобы поддерживать приложения, использующие форк процессов,
  модуль кода обычно гарантирует, что его фоновые потоки останавливаются перед форком и перезапускаются после него. Действуют следующие ограничения:

  + В Python 3.12 и новее интерпретатор Python записывает в лог предупреждение вида
    `DeprecationWarning: This process (pid=12345) is multi-threaded, use of fork() may lead to deadlocks in the child.`
    Причина в том, что после форка и до того, как Python проверит, является ли процесс многопоточным, модуль кода уже вынужден перезапустить свои потоки.
    Если предупреждение появляется только с модулем кода Python и исчезает после его удаления,
    это ложное срабатывание, и его можно игнорировать.
    В противном случае оно может указывать на проблему в коде приложения, и к нему следует отнестись серьёзно.
  + При форке без вызова [обработчиков форка Python](https://docs.python.org/3/library/os.html#os.register_at_fork)
    модуль кода не останавливает свои потоки.
    Это делает родительский процесс многопоточным со всеми обычными последствиями этого для форка.
    Для типичных приложений это не имеет значения, но затронуты будут пользовательские нативные модули расширения, вызывающие
    низкоуровневые API форка напрямую.
