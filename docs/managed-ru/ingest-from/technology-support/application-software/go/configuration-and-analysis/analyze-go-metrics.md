# Анализ метрик Go

* Чтение: 6 мин
* Опубликовано: 15 апреля 2019

Dynatrace OneAgent отслеживает множество метрик, связанных с Go.

Вкладки **Go metrics** и **Go HTTP metrics** на странице обзора процесса содержат самые важные метрики, специфичные для Go, и HTTP-метрики. Вкладки **Go managed memory**, **Heap details** и **Scheduling**, доступные через вкладку **Further details**, дают ещё больше ценной информации о производительности вашего Go-приложения.

## Ключевые метрики

Вкладка **Go metrics** даёт обзор следующих важных метрик Golang:

* **Suspension**: процентная доля сборщика мусора Go по сравнению с общим временем CPU приложения.
* **Runtime heap**: объём памяти, используемой или выделенной под кучу Go, а также время сборки мусора Go.
* **Goroutines**: количество горутин, созданных приложением и инфраструктурой среды выполнения Go.

## HTTP-метрики

На вкладке **Go HTTP metrics** можно изучить следующие HTTP-метрики:

* **Total requests**: количество всех запросов, отражающее общий поток трафика.
* **Number of HTTP 5xx responses**: количество ответов, которые указывают на постоянно падающие приложения или проблемы с ответами приложений.
* **Number of HTTP bad gateways**: количество ответов, которые указывают на некорректные ответы сервиса, формируемые приложением.
* **Response latency**: среднее время ответа приложения клиентам.

## Управляемая память Go

Вкладка **Go managed memory** разбивает метрики памяти на различные категории:

* **Heap**: объём памяти, используемой или выделенной под кучу среды выполнения Go.
* **OffHeap**: объём памяти, используемой или выделенной под внутренние структуры среды выполнения Go, которые не выделяются из памяти кучи. Примером памяти OffHeap являются структуры данных, используемые в реализации кучи Go.
* **Overall**: сумма памяти Heap, OffHeap и Stack.
* **Stack**: объём памяти, используемой или выделенной под динамические стеки Go. Стеки Go используются для выполнения горутин и растут динамически.

## Детали кучи

Вкладка **Heap details** позволяет глубже изучить устройство кучи Go.

* **Heap allocated Go objects count**: количество объектов Go, выделенных в куче Go.
* **Heap idle size**: объём памяти, не назначенной куче или стеку. Простаивающая память может быть возвращена операционной системе или удержана средой выполнения Go для последующего переназначения куче или стеку.
* **Heap live size**: объём памяти, считающейся «живой» сборщиком мусора Go. Эта метрика накапливает память, удержанную последним запуском сборщика мусора и выделенную с того момента.
* **Garbage collector invocation count**: количество запусков сборщика мусора Go.

## Планирование

О планировании горутин

Базовое понимание внутренних принципов планирования поможет читать метрики планирования и выявлять потенциальные аномалии.

Реализация планирования горутин оперирует тремя центральными типами объектов: M (Machine), P (Processor) и G (Goroutine). В исходном коде среды выполнения Go можно найти множество экземпляров этих типов объектов. Для простоты Dynatrace использует следующие альтернативные термины:

| Термин среды выполнения Go | Термин Dynatrace |
| --- | --- |
| M (Machine) | Worker thread |
| P (Processor) | Scheduling context |
| G (Goroutine) | Goroutine |

Go выполняет горутины в контексте рабочих потоков, которые берутся из пула нативных потоков операционной системы. Горутина может быть в любой момент назначена другому рабочему потоку, если только для закрепления привязки к рабочему потоку не используется `runtime.LockOSThread`.

Обычно к одному рабочему потоку назначается несколько горутин. За кооперативное планирование горутин отвечает контекст планирования. Компилятор Go добавляет в пролог каждой функции Go код, который проверяет, израсходовала ли выполняемая в данный момент горутина свой 10-миллисекундный квант времени выполнения. Сам механизм остроумно использует защитные метки стека (stack guards) для принудительного перепланирования. Если квант времени превышен, контекст планирования назначает на выполнение следующую горутину. До Go 1.14 планирование было полностью кооперативным, что приводило к проблеме с горутинами, которые не вызывали функции Go и поэтому не перепланировались. В Go 1.14 появился механизм, позволяющий вытесняющее планирование горутин для обработки этого особого случая.

Каждый набор горутин выполняется рабочим потоком. Выполнением управляет контекст планирования. Когда горутина записывает большой объём данных на диск или блокируется в ожидании входящего соединения, горутина блокируется в системном вызове, и никакая другая горутина не планируется. В такой ситуации контекст планирования с другими горутинами назначается другому рабочему потоку, припаркованному или только что созданному. Эти горутины не заблокированы системным вызовом и могут продолжать выполнение. Поэтому общее число рабочих потоков больше числа объектов контекста планирования. После возврата из блокирующего вызова горутина снова назначается контексту планирования. Если назначение не удаётся, горутина отправляется в глобальную очередь выполнения горутин. Тот же принцип применяется к вызовам cgo (из Go в язык C).

Число контекстов планирования является единственным параметром, который можно настроить в алгоритме планирования Go. Числом рабочих потоков управлять нельзя. Именно поэтому запись большого объёма данных на диск или ожидание входящих соединений не блокирует другие горутины, изначально назначенные тому же рабочему потоку. Эти горутины продолжают выполнение на своих вновь назначенных рабочих потоках.

Вкладка **Scheduling** даёт уникальную информацию о планировании горутин.

* **Go runtime system call count**: количество системных вызовов, выполненных средой выполнения Go. В это число не входят системные вызовы, выполненные пользовательским кодом.
* **Go to C language (cgo) call count**: количество вызовов cgo.
* **Parked worker thread count**: количество рабочих потоков, припаркованных средой выполнения Go. Припаркованный рабочий поток не потребляет циклы CPU, пока среда выполнения Go не выведет его из парковки.
* **Out-of-work worker thread count**: количество рабочих потоков, у связанного контекста планирования которых не осталось горутин для выполнения. Когда это происходит, рабочий поток пытается «украсть» горутины из другого контекста планирования или из глобальной очереди выполнения. Если кража не удаётся, рабочий поток через некоторое время паркуется. Тот же механизм применяется в сценарии высокой нагрузки. Когда существует простаивающий контекст планирования, среда выполнения Go выводит из парковки припаркованный рабочий поток и связывает его с простаивающим контекстом планирования. Выведенный из парковки рабочий поток теперь находится в состоянии «out of work» и начинает кражу горутин.
* **Worker thread count**: количество потоков операционной системы, созданных для выполнения горутин. Go не завершает рабочие потоки, а сохраняет их в припаркованном состоянии для повторного использования в будущем.
* **Global Goroutine run queue size**: количество горутин в глобальной очереди выполнения. Горутины помещаются в глобальную очередь выполнения, если рабочий поток, использованный для выполнения блокирующего системного вызова, не может получить контекст планирования. Контексты планирования периодически забирают горутины из глобальной очереди выполнения.
* **Idle scheduling context count**: количество контекстов планирования, у которых не осталось горутин для выполнения и для которых не удалось получить горутины из глобальной очереди выполнения или из других контекстов планирования.
