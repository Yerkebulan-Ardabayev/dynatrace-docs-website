# Сборщик мусора G1 – Java 9

* Чтение: 4 мин
* Обновлено 04 мая 2026 г.

В Hotspot VM Oracle Java 9 в качестве сборщика мусора по умолчанию используется Garbage First (G1) GC. Этот GC, впервые представленный в Java 7, может эффективно и параллельно работать с очень большими кучами. Его также можно настроить так, чтобы он не превышал заданное максимальное время паузы.

Большинство современных GC классифицируют объекты кучи на молодое (young generation) или старое (old generation) поколения. Это сделано в основном потому, что исследования реальных Java-приложений показали: более 90% объектов не переживают свою первую сборку мусора. Объекты постарше, пережившие несколько циклов сборки, как правило, остаются живыми и имеют 98% шанс выжить. Java GC дополнительно делят объекты молодого поколения на пространство выживших (survivor) и пространство Eden.

Вновь выделенные объекты всегда размещаются в пространстве Eden. После того как объект переживает первую сборку мусора, он перемещается в старшее поколение. Это сделано для того, чтобы:

* к новым объектам можно было применить алгоритм с эффективным временем выполнения; время выполнения зависит только от числа выживших объектов, но тратит половину размера кучи.
* к старому поколению можно было применить алгоритм с эффективным использованием памяти; время выполнения зависит от размера кучи, но используется доступная память максимально эффективно.

Куча Java

По сравнению с большинством сборщиков мусора G1 имеет два главных преимущества:

* он может работать параллельно, не останавливая потоки приложения
* он использует несмежные пространства, что позволяет G1 эффективно работать с очень большими кучами

Благодаря тому, как G1 использует доступную кучу, он может собирать молодое и старое поколения одновременно. Вместо разделения кучи на 3 пространства (Eden, survivor и old) он делит кучу на множество маленьких регионов размером по умолчанию 2 МБ. Каждый регион назначается какому-либо пространству.

Куча G1 в Java

Разбиение кучи на маленькие регионы позволяет G1 выбирать небольшую группу регионов для сборки и быстро её завершать. Если регион запланирован к сборке, все выжившие объекты копируются из собираемого региона в неназначенный регион. Если собираемый регион принадлежал пространству Eden, неназначенный регион, содержащий все выжившие объекты, становится регионом survivor. Идеально, если регион полностью заполнен мусором и не содержит ни одного выжившего объекта: его можно объявить «неназначенным» (unassigned), и работа над ним не производится.

Чтобы собрать всю кучу, G1 может выбрать любое число или сочетание регионов. Чтобы оптимизировать время сборки, он всегда выбирает регионы, полностью или почти полностью заполненные мусором, минимизируя объём работы по освобождению места в куче для последующих выделений. Другие GC всегда собирают всё поколение целиком, поэтому сложность их выполнения зависит от общего размера кучи. В случае G1 она зависит от числа живых объектов, поскольку память можно освободить без обработки всего поколения. Идеально, когда куча достаточно велика и часть регионов всегда полностью заполнена мусором, что упрощает их сборку.

Кроме того, G1 может выполнять большую часть работы параллельно. В мире Java мы уже знакомы с параллельной сборкой по сборщику Concurrent Mark & Sweep (CMS). Однако CMS способен собирать параллельно только старое поколение, и для сборки молодого поколения по-прежнему требуется останавливать приложение. Процесс проходит в следующих фазах:

1. **Initial mark**: G1 останавливает приложение только в начале сборки мусора, чтобы быстро провести служебные операции, затем приложение возобновляется.
2. **Concurrent mark**: Пока приложение выполняется, GC обходит все ссылки и помечает живые объекты.
3. **Final mark**: Приложение снова приостанавливается, и проводится финальная очистка.
4. **Evacuation**: Выбирается несколько регионов, и они собираются.

Поскольку фаза **Evacuation** проходит быстро, особенно для больших куч, G1 обычно превосходит другие GC по времени приостановки выполняемого приложения.

Недостаток в том, что G1 плохо работает с маленькими кучами, поэтому учитывайте конкретное приложение. При необходимости всегда можно вернуться к сборщику по умолчанию, задав флаг `-XX:+UseParallelOldGC`. Если у G1 очень мало доступной кучи, в журнале GC будут появляться **Full GC**. **Full GC** выполняется только когда G1 определяет, что обычный режим работы больше невозможен. В этом случае вся куча собирается медленным, но экономичным по памяти алгоритмом, что улучшает условия для следующей сборки. Если случаются только Full GC, стоит рассмотреть увеличение размера кучи и переход на другой GC.

Благодаря малым регионам G1 можно настроить так, чтобы ограничить максимальное время паузы, задав `-XX:MaxGCPauseMillis=n`. Тогда G1 учтёт предыдущие сборки и объём обнаруженного мусора и оценит максимальное число регионов, которые он может собрать за один раз, не превышая этот лимит.

G1 пока далёк от настоящего сборщика реального времени, но работает чуть лучше других сборщиков, ограниченных своей жёсткой структурой кучи.
