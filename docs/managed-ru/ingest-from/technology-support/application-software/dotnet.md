# .NET

# .NET

* Справочник
* Чтение: 3 мин
* Обновлено 23 октября 2025 г.

Dynatrace OneAgent инструментирует ваши .NET-приложения, размещая операторы трассировки в стратегических местах кода для трассировки кода, метрик производительности, обнаружения ошибок, отслеживания зависимостей и многого другого.

Не каждое обнаруженное .NET-приложение инструментируется по умолчанию. Dynatrace поддерживает набор правил для инструментирования определённых процессов (например, пулов приложений IIS, которые можно дополнить собственными правилами). Чтобы узнать основы настройки мониторинга групп процессов (автоматический глубокий мониторинг, пользовательские правила мониторинга и встроенные правила мониторинга), см. [Настройка мониторинга групп процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-monitoring "Способы настройки мониторинга групп процессов").

## Ограничения

### Горячие точки методов в Linux

Модуль кода .NET использует сигналы POSIX, чтобы захватывать трассировки стека для функции горячих точек методов.

Поскольку эти сигналы могут прерывать приложение в произвольные моменты, некоторые приложения и библиотеки (в частности, .NET-библиотеки с нативными зависимостями) могут работать некорректно при включённых функциях горячих точек методов.

У затронутого приложения могут проявляться такие симптомы, как:

* Проблемы с подключением
* Искажённые текстовые данные
* Сбои проб готовности (readiness) и живучести (liveness) в Kubernetes

Единственное доступное сейчас решение состоит в том, чтобы отключить соответствующие функции OneAgent для затронутой группы процессов:

* Capture method hotspot information in PurePaths
* Capture background CPU method hotspot information
* .NET Async Method Hotspots

Горячие точки методов могут некорректно захватывать трассировки стека в операционных системах на базе musl-libc, таких как Alpine. Причина в том, что библиотека musl-libc удаляет часть отладочной информации. Переход на операционную систему на базе glibc (Debian/Ubuntu) может смягчить эту конкретную проблему.

* Горячие точки методов .NET недоступны для Linux на ARM64 (AArch64).

### Trimming

Необязательная функция .NET [усечённые автономные развёртывания и исполняемые файлы](https://docs.microsoft.com/en-us/dotnet/core/deploying/trim-self-contained) была введена для оптимизации размера упакованных приложений.
Усечённые приложения не поддерживаются OneAgent, поэтому этот параметр необходимо отключить. Например, [UiPath](https://www.uipath.com/) использует усечение (trimming), что делает невозможным его инструментирование с помощью OneAgent.

### Однофайловые приложения

.NET SDK поддерживает сборку приложения в виде [одного файла](https://learn.microsoft.com/en-us/dotnet/core/deploying/single-file/overview?tabs=cli), который объединяет все зависимости в один платформозависимый исполняемый файл.

В зависимости от того, используете ли вы [развёртывание с зависимостью от платформы](https://learn.microsoft.com/en-us/dotnet/core/deploying/#publish-framework-dependent) или [автономные приложения](https://learn.microsoft.com/en-us/dotnet/core/deploying/#publish-self-contained), среда выполнения также будет включена в исполняемый файл.

Также учтите следующую информацию:

* OneAgent не поддерживает инструментирование модулей, включённых в исполняемый файл, из-за чего приложения, собранные как однофайловые и автономные, не поддерживаются вообще.
* Приложения, собранные как однофайловые и зависящие от платформы, могут быть инструментированы частично.
* Сборки .NET, включённые в однофайловый исполняемый файл, не могут быть инструментированы (логику трассировки невозможно разместить для таких сборок).
