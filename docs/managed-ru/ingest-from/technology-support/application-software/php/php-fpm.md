# PHP-FPM monitoring

# PHP-FPM monitoring

* Чтение: 4 мин
* Обновлено: 17 декабря 2025

Мониторинг PHP-FPM в Dynatrace предоставляет информацию о соединениях, медленных запросах и процессах. Если PHP-FPM работает недостаточно эффективно или возникает проблема, Dynatrace немедленно уведомляет вас и показывает, какие хосты затронуты.

## Предварительные условия

* Linux или Windows
* PHP версии 7.1.0+
* Страница статуса **PHP-FPM** должна быть включена на всех узлах, которые требуется мониторить, и страница статуса PHP-FPM должна быть доступна с `localhost` по HTTP(S). Полные инструкции см. в [документации PHP](https://www.php.net/manual/en/fpm.status.php), но учтите, что путь URL по умолчанию теперь `/status` (а не `/fpm-status`, как показано в их примере).

## Включение мониторинга PHP-FPM глобально

Когда мониторинг PHP-FPM включён глобально, Dynatrace автоматически собирает метрики PHP-FPM при каждом обнаружении в вашем окружении нового хоста с PHP-FPM.

1. Перейдите в **Settings**.
2. Выберите **Monitoring** > **Monitored technologies**.
3. На вкладке **Supported technologies** найдите и разверните строку **PHP-FPM**.
4. Введите **Status page URI**.

   Чтобы мониторить несколько пулов, введите URI отдельных страниц статуса PHP-FPM (разделённые пробелами) в поле **Status page URI**. У всех экземпляров PHP-FPM должна быть правильная ссылка на URI страницы статуса.
5. Выберите **Save**.
6. Включите **PHP-FPM**.

## Включение мониторинга PHP-FPM на отдельных хостах

Dynatrace позволяет включать мониторинг PHP-FPM для конкретных хостов, а не глобально.

1. Если глобальный мониторинг PHP-FPM сейчас включён, отключите его: перейдите в **Settings** > **Monitoring** > **Monitored technologies** и отключите **PHP-FPM**.
2. Перейдите в **Hosts**.
3. В таблице **All hosts** найдите и выберите хост, который требуется настроить.  
   Откроется страница хоста.
4. Выберите **More** (**…**) > **Settings**.
5. В таблице **Monitored technologies** найдите **PHP-FPM** и включите его.

## Мониторинг PHP-FPM

После включения мониторинга PHP-FPM он начинает работать.

1. Перейдите в **Technologies & Processes**.
2. Выберите плитку **PHP**.
3. Чтобы просмотреть метрики кластера, найдите **PHP-FPM** в таблице **Process group** под плитками и выберите **Details**, чтобы отобразить сведения о группе процессов PHP-FPM.  
   На диаграмме отображается выбранная метрика группы процессов во времени. Можно выбрать другую метрику из списка.
4. Откройте вкладку **Technology-specific metrics**, чтобы отобразить метрики, специфичные для PHP-FPM.

   * Выберите другой интервал времени в селекторе временного диапазона на верхней панели меню, чтобы сосредоточиться на другом диапазоне.
   * Выберите тип метрики из выпадающего списка метрик под временной шкалой, чтобы сравнить значения всех узлов в сортируемой табличной форме.
5. Чтобы отобразить метрики конкретного узла, выберите узел из списка **Process** внизу страницы.
6. Откройте вкладку **PHP-FPM**, чтобы отобразить число **Accepted connections** (соединений, принятых пулом) и счётчик **Slow requests**.

   **Accepted connections** иногда ошибочно принимают за число запросов, но эта метрика измеряет именно то, что следует из её названия: число соединений, принятых пулом.

## Метрики PHP-FPM

| **Метрика** | **Описание** |
| --- | --- |
| Accepted connections | Число соединений, принятых пулом. |
| Slow requests | Число запросов, превысивших значение `request_slowlog_timeout`. |
| Waiting connections | Число запросов в очереди ожидающих соединений. |
| Max number of waiting connections | Размер очереди сокета ожидающих соединений. |
| Active processes | Число активных процессов. |
| Total processes | Сумма простаивающих и активных процессов. |

## Метрики мониторинга узлов PHP-FPM

Дополнительные метрики мониторинга PHP-FPM доступны на страницах отдельных процессов.

Чтобы просмотреть диаграммы **Requests**, **Input buffering** и **Processes**, откройте вкладку **Further details**.

Когда общее число активных процессов достигает лимита **Total processes**, запуск новых скриптов блокируется до завершения проблемных процессов. Максимальное число **Waiting connections** определяет максимальное число соединений, которые будут поставлены в очередь. При достижении этого лимита последующие запросы отклоняются или игнорируются.
