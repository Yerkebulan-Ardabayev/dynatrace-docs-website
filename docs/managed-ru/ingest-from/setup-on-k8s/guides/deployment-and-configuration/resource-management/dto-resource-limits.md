# Задание лимитов ресурсов для компонентов Dynatrace Operator

* Чтение: 2 мин
* Обновлено 27 февраля 2026 г.

Правильно настроенные лимиты ресурсов обеспечивают оптимальную производительность и стабильность компонентов Dynatrace Operator, предотвращая конкуренцию за ресурсы в вашем кластере Kubernetes. Это руководство помогает понять, как задавать подходящие лимиты ресурсов в зависимости от размера вашего окружения и характера использования.

## Базовые лимиты ресурсов по умолчанию

Предоставленные лимиты ресурсов по умолчанию проверены нагрузочным тестированием. Эти значения по умолчанию хорошо показали себя в следующем окружении:

* 25 узлов (тип узла e2-standard-32 в Google Kubernetes Engine)
* 20 DynaKube
* 2500 пространств имён
* 5000 подов

## Факторы потребления ресурсов

Следующие пять ключевых индикаторов влияют на потребление ресурсов различными компонентами Dynatrace Operator:

| **Индикатор** | **Dynatrace Operator** | **Webhook** | **CSI driver** |
| --- | --- | --- | --- |
| Пространства имён | Применимо | Применимо |  |
| Узлы | Применимо |  |  |
| DynaKube | Применимо |  | Применимо |
| Поды |  | Применимо | Применимо |
| Число версий OneAgent |  |  | Применимо |

### Что означают индикаторы влияния

* **Пространства имён**: больше пространств имён увеличивают нагрузку на Operator и webhook, так как им нужно отслеживать ресурсы во всех пространствах имён и управлять ими.

  + Влияние:

    - Увеличивает использование CPU/памяти Operator
    - Увеличивает использование CPU webhook
* **Узлы**: дополнительные узлы требуют больше ресурсов, так как Operator хранит список всех доступных узлов в кластерах Kubernetes и проверяет их соответствие доступным хостам на сервере Dynatrace.

  + Влияние:

    - Увеличивает использование CPU/памяти Operator
* **DynaKube**: каждый ресурс DynaKube представляет отдельное развёртывание Dynatrace, которое требует индивидуального управления.

  + Влияние:

    - Увеличивает использование CPU/памяти Operator
    - Увеличивает использование CPU/памяти webhook
    - Увеличивает использование CPU/памяти компонента provisioner CSI driver
* **Поды**: webhook обрабатывает запросы admission для каждого пода, а CSI driver выполняет монтирование томов для подов, использующих OneAgent.

  + Влияние:

    - Увеличивает использование CPU/памяти компонентов server/liveness-probe/registrar CSI driver
* **Версии OneAgent**: CSI driver должен управлять разными версиями OneAgent и предоставлять к ним доступ, что требует дополнительных ресурсов хранения и обработки.

  + Влияние:

    - Увеличивает использование CPU/памяти компонента provisioner CSI driver

### Минимизация влияния большого числа узлов

По умолчанию Dynatrace Operator отслеживает доступность хостов в вашем кластере, чтобы обнаруживать ожидаемое удаление подов хостового OneAgent, особенно в сценариях масштабирования. Этот мониторинг не нужен в бессерверных окружениях.

#### Когда отключать обнаружение доступности хостов

Эта функциональность присутствует во всех версиях Operator до 1.6.0 и не может быть отключена.

Её можно отключить в 1.6.3 и 1.7.3 или в более новых версиях Operator.

applicationMonitoring

Снизить потребление ресурсов Operator можно, отключив обнаружение доступности хостов, если ваши DynaKube:

* используют **только** режимы мониторинга приложений и
* **не** используют ни одну из функций мониторинга на основе хостов.

#### Как отключить обнаружение доступности хостов

Чтобы отключить обнаружение доступности хостов, задайте следующее значение Helm:

```
operator:



hostAvailabilityDetection: false
```

Эта оптимизация может снизить использование CPU и памяти Operator, особенно в больших кластерах с множеством узлов.

**Настройка на уровне всего кластера**: `operator.hostAvailabilityDetection` влияет на все DynaKube, управляемые Operator. Отключайте это, только если вы уверены, что **ни один** из ваших DynaKube не требует мониторинга на основе хостов. Отключение при необходимости хостовых OneAgent может вызывать ложноположительные предупреждения об отсутствии хостов во время масштабирования узлов или других операций, связанных с узлами.

## Настройка лимитов ресурсов

Хотя лимитов ресурсов по умолчанию должно быть достаточно для большинства сценариев использования, их можно настроить под конкретные потребности.

Измените `values.yaml`, чтобы задать лимиты ресурсов для Dynatrace Operator, webhook или CSI driver.

* **Dynatrace Operator**

  ```
  operator:



  requests:



  cpu: 50m



  memory: 64Mi



  limits:



  cpu: 100m



  memory: 128Mi
  ```
* **Webhook**

  ```
  webhook:



  requests:



  cpu: 300m



  memory: 128Mi



  limits:



  cpu: 300m



  memory: 128Mi
  ```
* **CSI driver**

  ```
  csidriver:



  csiInit:



  resources:



  requests:



  cpu: 50m



  memory: 100Mi



  limits:



  cpu: 50m



  memory: 100Mi



  server:



  resources:



  requests:



  cpu: 50m



  memory: 100Mi



  limits:



  cpu: 50m



  memory: 100Mi



  provisioner:



  resources:



  requests:



  cpu: 300m



  memory: 100Mi



  job:



  resources:



  requests:



  cpu: 200m



  memory: 30Mi



  registrar:



  resources:



  requests:



  cpu: 20m



  memory: 30Mi



  limits:



  cpu: 20m



  memory: 30Mi



  livenessprobe:



  resources:



  requests:



  cpu: 20m



  memory: 30Mi



  limits:



  cpu: 20m



  memory: 30Mi
  ```

Для компонентов `provisioner` и `job` драйвера CSI driver не заданы лимиты ресурсов по умолчанию. Это позволяет им использовать дополнительные ресурсы при их доступности, повышая производительность.

Компонент `job` используется только с [node-image-pull](/managed/ingest-from/setup-on-k8s/guides/deployment-and-configuration/node-image-pull "Настройка извлечения образа на узле").

Обратите внимание, что если лимиты заданы слишком низкими, это может привести к увеличению времени запуска подов из-за троттлинга CPU или к завершениям по OOM в CSI driver из-за нехватки памяти, что может помешать запуску приложений.

### Масштабирование лимитов ресурсов для разных окружений

Запросы и лимиты ресурсов по умолчанию рассчитаны на окружения среднего масштаба. Используйте следующие рекомендации, чтобы скорректировать лимиты в зависимости от размера вашего окружения. Это начальные рекомендации: всегда отслеживайте фактическое использование ресурсов в вашем окружении и корректируйте соответственно.

**Классы Quality of Service подов**: у некоторых компонентов лимиты и запросы заданы равными, чтобы обеспечить [гарантированный класс Quality of Service пода](https://kubernetes.io/docs/concepts/workloads/pods/pod-qos/#guaranteed). При масштабировании лимитов таких компонентов всегда пропорционально масштабируйте и запросы.

**Пропорциональная справедливость:** низкие запросы CPU у контейнера могут вызывать троттлинг из-за политики управления CPU на узле. Запросы служат **минимальной гарантией**. Поэтому на сильно загруженном узле контейнеры с меньшими запросами будут троттлиться сильнее по сравнению с контейнерами с большими запросами, независимо от их лимита. При масштабировании лимитов всегда рассматривайте и масштабирование запросов.

Запросы и лимиты ресурсов компонента `job` для CSI driver не нужно масштабировать в зависимости от размера окружения. Они не зависят от числа узлов, числа подов и числа DynaKube. Тем не менее можно скорректировать запрос CPU, чтобы управлять тем, насколько быстро завершается job, что определяет, как скоро CSI driver будет готов монтировать тома:

| Запрос CPU | Примерное время завершения |
| --- | --- |
| 100m | ~1 мин |
| 200m (по умолчанию) | ~30 сек |
| 300m | ~25 сек |

Это влияет только на длительность состояния `Running` для job. Это не влияет на время `ContainerCreation` или `PodScheduling`.

Начиная с версии Dynatrace Operator 1.9.0, параметр `replicas` webhook можно настраивать в Helm в дополнение к его лимитам ресурсов. Это позволяет масштабировать Webhook горизонтально в более крупных окружениях, обеспечивая лучшую производительность и доступность.

Рекомендация по ресурсам ниже рассчитана на две реплики webhook. Если вы решите использовать другое число реплик, скорректируйте лимиты ресурсов соответственно.

* Пример: если требуется использовать 3 реплики вместо 2, можно снизить запросы/лимиты CPU/памяти примерно на 30%, сохранив общую производительность, но получив более высокую доступность.

Подробнее о настройке параметра `replicas` webhook и связанных значений см. [Настройка высокой доступности](#high-availability-helm).

#### Крупные окружения (> 50 узлов, > 10 000 подов)

Увеличьте запросы/лимиты по умолчанию на 50–100%:

* **Dynatrace Operator**:

  + Запрос: CPU 100m, память 128Mi
  + Лимиты: CPU 200m, память 256Mi
* **Webhook**:

  + Запросы: CPU 600m, память 256Mi
  + Лимиты: CPU 600m, память 256Mi
* **provisioner CSI driver**:

  + Запросы: CPU 600m, память 200Mi
* **server CSI driver**:

  + Запросы: CPU 100m, память 200Mi
  + Лимиты: CPU 100m, память 200Mi
* **liveness-probe CSI driver**:

  + Запросы: CPU 30m, память 50Mi
  + Лимиты: CPU 30m, память 50Mi
* **registrar CSI driver**:

  + Запросы: CPU 30m, память 50Mi
  + Лимиты: CPU 30m, память 50Mi

#### Окружения корпоративного масштаба (> 100 узлов, > 25 000 подов)

* **Dynatrace Operator**: CPU 400m, память 512Mi

  + Лимиты: CPU 400m, память 512Mi
* **Webhook**:

  + Запросы: CPU 1000m, память 512Mi
  + Лимиты: CPU 1000m, память 512Mi
* **provisioner CSI driver**:

  + Запросы: CPU 900m, память 300Mi
* **server CSI driver**:

  + Запросы: CPU 150m, память 300Mi
  + Лимиты: CPU 150m, память 300Mi
* **liveness-probe CSI driver**:

  + Запросы: CPU 50m, память 70Mi
  + Лимиты: CPU 50m, память 70Mi
* **registrar CSI driver**:

  + Запросы: CPU 50m, память 70Mi
  + Лимиты: CPU 50m, память 70Mi
