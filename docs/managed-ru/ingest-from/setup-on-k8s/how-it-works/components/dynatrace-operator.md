## Оператор

Оператор Dynatrace управляет автоматическим развёртыванием, конфигурацией и жизненным циклом компонентов Dynatrace. Он использует пользовательские ресурсы вида [DynaKube](/managed/ingest-from/setup-on-k8s/reference/dynakube-parameters "List the available parameters for setting up Dynatrace Operator on Kubernetes.") и [EdgeConnect](/managed/upgrade/unavailable-in-managed "Your selection is unavailable in Dynatrace Managed.").

Окна обновления в настоящее время не применяются в средах Kubernetes.

Функциональность Оператора Dynatrace привязана к пользовательским ресурсам. Эти ресурсы определяют, какие функции включены. При создании пользовательского ресурса Оператор Dynatrace запускает начальную сверку (reconciliation) для обеспечения достижения желаемого состояния. Процесс сверки регулярно отслеживает изменения в кластере и соответствующим образом корректирует состояние. Частота сверки снижается после начального развёртывания, поскольку изменений обычно вносится меньше.

При запуске Оператор Dynatrace регистрирует [webhook](#webhook) в Kubernetes API. В каждом цикле сверки Оператор Dynatrace проверяет изменения в пользовательских ресурсах, чтобы:

* Обновить статус пользовательских ресурсов
* Убедиться, что начальный запуск управляемых компонентов соответствует требованиям
* Управлять рабочими нагрузками
* Регистрировать компоненты на сервере Dynatrace

Оператор Dynatrace сверяет свои собственные пользовательские ресурсы только в пределах пространства имён своего развёртывания.

Пространства имён, отслеживаемые Оператором Dynatrace, нужно настроить следующим образом:

* Пометить пространства имён метками в соответствии с `namespaceSelector`, чтобы webhook мог изменять поды.
* Убедиться, что в пространствах имён, куда выполняется инъекция, существуют необходимые предварительные условия.

### Требования к ресурсам

Конфигурация по умолчанию:

* 1 реплика на кластер

Одной реплики Оператора Dynatrace обычно достаточно благодаря выбору лидера (leader election). Дополнительные реплики активируются только при завершении работы текущего лидера и избрании нового.

Контейнер Оператора Dynatrace имеет предустановленные значения запросов и лимитов CPU и памяти. Чтобы изменить эти значения при развёртывании с помощью Helm, нужно отредактировать файл `values.yaml`.

Конфигурация ресурсов с помощью Helm

```
operator:



requests:



cpu: 50m



memory: 64Mi



limits:



cpu: 100m



memory: 128Mi
```

## Webhook

Webhook Dynatrace изменяет определения подов для инъекции модулей кода Dynatrace для наблюдаемости приложений (Application observability). Он также проверяет определения DynaKube и преобразует DynaKube с более старыми версиями API.

Конфигурации webhook управляются Оператором Dynatrace и периодически обновляются. Эти обновления обеспечивают возможность Kubernetes API продолжать взаимодействие с webhook.

### Основные функции

* Изменение подов (Pod mutation), webhook изменяет поды, редактируя их определения для включения метаданных, необходимых для наблюдаемости приложений (Application observability).

  + Присоединяет Init-контейнер для загрузки (если CSI-драйвер не используется) и настройки модулей кода при запуске пода.
  + Присоединяет тома CSI, если они настроены, чтобы CSI-драйвер мог обрабатывать создание томов.
  + Изменяет определения подов для обогащения метаданных.
* Изменение пространств имён (Namespace mutation), webhook изменяет пространства имён, чтобы включить мониторинг подов внутри этих пространств имён.

  + Добавляет метки к пространствам имён, чтобы webhook мог отслеживать события `CREATE` подов.

  Конфигурации webhook поддерживают только статический селектор пространства имён. Требуемая метка добавляется к вновь созданным пространствам имён.
* Проверка конфигурации

  + Проверяет, что пользовательские ресурсы, такие как DynaKube, содержат корректную конфигурацию при их создании или обновлении

    - Разная проверка в зависимости от полей, некоторые обязательны, а некоторые должны соответствовать более специфичным правилам
  + Выдаёт полезные предупреждения/сообщения об ошибках при неудачной проверке

  Дополнительную информацию по каждому полю смотри в [параметрах DynaKube](/managed/ingest-from/setup-on-k8s/reference/dynakube-parameters "List the available parameters for setting up Dynatrace Operator on Kubernetes.") и [параметрах EdgeConnect](/managed/upgrade/unavailable-in-managed "Your selection is unavailable in Dynatrace Managed.")!
* Преобразование между версиями

  + Обеспечивает совместимость между разными версиями определений пользовательских ресурсов
  + Преобразует более старые версии пользовательского ресурса в версию хранения, понятную Оператору Dynatrace

### Требования к ресурсам

Конфигурация по умолчанию:

* 2 реплики на кластер (можно масштабировать для повышения доступности)

Контейнер Dynatrace Webhook имеет заданные по умолчанию запросы и лимиты. Если нужно задать другие запросы или лимиты ресурсов, это можно сделать при развёртывании Оператора Dynatrace с помощью Helm.

Конфигурация ресурсов с помощью Helm

Чтобы задать лимиты ресурсов, нужно отредактировать `values.yaml`. Конфигурация по умолчанию приведена ниже.

```
webhook:



requests:



cpu: 300m



memory: 128Mi



limits:



cpu: 300m



memory: 128Mi
```

## CSI-драйвер

CSI-драйвер Оператора Dynatrace предоставляет модули кода Dynatrace для подов приложений, минимизируя использование хранилища и нагрузку на среду Dynatrace. Кроме того, он предоставляет доступное для записи хранилище томов для OneAgent, используя [эфемерные локальные тома﻿](https://dt-url.net/j9027w2).

* Для конфигураций `applicationMonitoring` он предоставляет подам на каждом узле необходимый бинарный файл OneAgent для мониторинга приложений в виде монтирования **только для чтения**.
* Для конфигураций `hostMonitoring` он предоставляет доступную для записи папку для конфигураций OneAgent, когда используется файловая система хоста, доступная только для чтения.
* Для `cloudNativeFullStack` он предоставляет оба варианта выше.

Минимизирует количество загрузок, загружая модули кода один раз на узел и сохраняя их в файловой системе узла.

* С CSI-драйвером инъекция 100 подов, распределённых по трём узлам, приведёт всего к 3 загрузкам модулей кода.
* Без CSI-драйвера каждому поду потребуется загружать собственные модули кода, поэтому инъекция 100 подов приведёт к загрузке 100 модулей кода.

Минимизирует использование хранилища, сохраняя модули кода в файловой системе узла и создавая монтирование [OverlayFs﻿](https://dt-url.net/hf036vi) для каждого пода, в который выполнена инъекция.

* С CSI-драйвером инъекция 100 подов, распределённых по трём узлам, приведёт к сохранению только 3 модулей кода.
* Без CSI-драйвера каждый под сохраняет собственный модуль кода, поэтому инъекция 100 подов приведёт к сохранению 100 модулей кода.

Подробности о том, где на узлах Kubernetes хранятся файлы и журналы OneAgent (с CSI-драйвером и без него), смотри в разделе [требования к хранилищу](/managed/ingest-from/setup-on-k8s/reference/storage "A comprehensive overview of the storage requirements for different Dynatrace Operator deployment mode in Kubernetes environments").

### Требования к ресурсам

Конфигурация по умолчанию:

* 1 реплика на узел (развёрнута через DaemonSet)

Провизионер CSI-драйвера не имеет предустановленных лимитов ресурсов. Однако, если нужно задать лимиты ресурсов или изменить лимиты ресурсов для других контейнеров CSI-драйвера, это можно сделать при развёртывании Оператора Dynatrace с помощью Helm.

Конфигурация ресурсов с помощью Helm

Чтобы задать лимиты ресурсов для контейнера `provisioner`, нужно отредактировать `values.yaml`. Пример конфигурации приведён ниже.

* Увеличить CPU для ускорения распаковки и установки модулей кода (важно для быстрого запуска и инициализации узла)
* Увеличение памяти не влияет на производительность

```
provisioner:



resources:



requests:



cpu: 300m



memory: 100Mi



limits:



cpu: 300m



memory: 100Mi



server:



resources:



requests:



cpu: 50m



memory: 100Mi



limits:



cpu: 50m



memory: 100Mi
```

### Разрешения и привилегии

CSI-драйверу Оператора Dynatrace требуются повышенные разрешения для создания и управления монтированиями в хостовой системе. В частности, для тома, где CSI-драйвер хранит модули кода, требуется разрешение [mountPropagation: Bidirectional﻿](https://dt-url.net/90236h3). Это разрешение доступно только для привилегированных контейнеров.

Полный список ресурсов, к которым обращается CSI-драйвер, можно найти в [документации по безопасности](/managed/ingest-from/setup-on-k8s/reference/security "This page provides an overview of the Dynatrace components, their default configurations, and the permissions they require").

Входящий и исходящий трафик описан в [документации по сетевому трафику](/managed/ingest-from/setup-on-k8s/reference/network "Network traffic requirements for the Dynatrace Operator components in a Kubernetes cluster.").
