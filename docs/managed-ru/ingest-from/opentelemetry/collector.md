OTel Collector (или просто «Collector»), это сетевое сервисное приложение, которое используется для группировки и преобразования данных телеметрии. Он работает как прокси и может принимать OTLP-запросы, а также данные из других источников, преобразовывать эти запросы согласно заданным правилам и передавать их в бэкенд.

На следующей диаграмме показаны разные компоненты, которые Collector может использовать для приёма, обработки и экспорта данных телеметрии в Dynatrace.

## Преимущества использования Collector

В целом использование Collector вместе с сервисом даёт преимущество, поскольку позволяет сервису быстро сгружать данные и берёт на себя дополнительную обработку, такую как повторные попытки, группировку, шифрование или фильтрацию чувствительных данных. Он централизует типовые задачи обработки вместо их дублирования в каждом приложении.

Collector стоит использовать, если:

* Нужно собирать данные из разных источников в разных форматах, и нужен простой способ заставить их все доставлять данные в бэкенд, с которым они иначе были бы несовместимы.
* Нужна общая обработка атрибутов данных телеметрии.

Collector, это относительно лёгкий компонент, поэтому команды могут разворачивать собственные экземпляры, не деля общую конфигурацию.

Collector настраивается в едином файле YAML. Это устраняет необходимость просматривать несколько файлов и упрощает сопровождение. Подробнее о конфигурации см. в разделе [Настройка OTel Collector](/managed/ingest-from/opentelemetry/collector/configuration "How to configure the OpenTelemetry Collector.").

## Дистрибутивы

Collector поставляется в разных вариантах дистрибутива с разными вариантами настройки и развёртывания.

| Тип | Когда использовать? |
| --- | --- |
| Dynatrace OTel Collector Рекомендуется | Предпочтительный выбор для большинства сценариев использования. Поставляется с набором проверенных и стабильных компонентов Collector, обычно применяемых в связке с Dynatrace. |
| Core | Когда в первую очередь нужно преобразование между протоколами OTLP (например, между HTTP и gRPC, см. [Преобразование OTLP gRPC в HTTP с помощью OTel Collector](/managed/ingest-from/opentelemetry/collector/use-cases/grpc "Configure the OpenTelemetry Collector to transform a gRPC OTLP request to HTTP.")), нужно применять ограничения использования памяти или группировку запросов. |
| Contrib | Идеально подходит для тестовых сред, поскольку содержит все сторонние компоненты и не требует индивидуальной сборки. Как правило, не рекомендуется для промышленных систем, поскольку обычно потребляет больше ресурсов и может быть менее стабильным, чем оптимизированный экземпляр Builder. |
| Collector Builder | Когда нужно полностью настроить экземпляр Collector и запускать только компоненты, необходимые для конкретного сценария использования. |

### Dynatrace OTel Collector

Дистрибутив Dynatrace для OpenTelemetry (OTel) Collector, это индивидуальная сборка Collector, предоставляемая Dynatrace. Она адаптирована под типовые сценарии использования в контексте Dynatrace и поставляется с оптимизированным и проверенным набором компонентов Collector.

#### Преимущества Dynatrace OTel Collector

Dynatrace OTel Collector предлагает следующие преимущества по сравнению с другими дистрибутивами Collector.

* Покрывается поддержкой Dynatrace, см. [Поддержка Dynatrace OTel Collector](#support).
* Компоненты Collector проверены Dynatrace.
* Патчи безопасности независимы от релизов OTel Collector.

Dynatrace OTel Collector сохраняет стандартную модель конфигурации и семантику компонентов.
Избегая проприетарных абстракций или скрытых значений по умолчанию, он позволяет как людям, так и ИИ-агентам надёжно переиспользовать стандартные ссылки и примеры OpenTelemetry с меньшим числом ошибок.

#### Поддержка Dynatrace OTel Collector

Сборки x86‑64 и ARM64 дистрибутива Dynatrace OTel Collector поддерживаются командой поддержки Dynatrace в соответствии с [политикой поддержки Dynatrace﻿](https://support.dynatrace.com/).

Для полного покрытия поддержкой нужно обращаться в Dynatrace через официальные каналы поддержки.
Проблемы, сообщённые через GitHub, обрабатываются по мере возможности; договоры поддержки и SLA к ним не применяются.

Каждая минорная версия поддерживается три месяца.
Исправления предоставляются либо в виде патч-релиза для последней поддерживаемой минорной версии, либо в составе следующего минорного релиза.

### Дистрибутивы OpenTelemetry

Dynatrace предоставляет ограниченную поддержку для конфигураций Core, Contrib и Builder.
Эта поддержка охватывает только компоненты и их версии, которые входят в состав Dynatrace OTel Collector.

Полностью поддерживаемый дистрибутив Collector см. в разделе [Dynatrace OTel Collector](#dt-collector-dist).

#### Collector Core

[Дистрибутив Core﻿](https://github.com/open-telemetry/opentelemetry-collector) содержит базовый прокси-сервис и несколько основных сервисных компонентов.
Это включает:

* Ресивер для данных протокола OpenTelemetry (OTLP) по HTTP и gRPC.
* Процессоры для группировки запросов и обеспечения ограничений использования памяти.
* Экспортёры для логирования в консоль и OTLP по HTTP и gRPC.

#### Collector Contrib

[Дистрибутив Contrib﻿](https://github.com/open-telemetry/opentelemetry-collector-contrib) построен поверх Core и расширяет его функциональность, поставляясь с большим числом дополнительных ресиверов, процессоров и экспортёров, предоставленных сторонними разработчиками.

Поскольку дистрибутив Contrib представляет собой единый пакет «всё в одном» и поставляется со всеми сервисными компонентами, скомпилированными заранее, он может использовать больше системных ресурсов (хранилище и память), чем оптимизированная сборка Collector.

Dynatrace рекомендует использовать дистрибутив Contrib только в тестовых целях.
Его не следует использовать в промышленных средах.

#### Collector Builder

Помимо двух дистрибутивов, OpenTelemetry также предлагает [OpenTelemetry Collector Builder (OCB)﻿](https://github.com/open-telemetry/opentelemetry-collector/tree/v0.156.0/cmd/builder), инструмент командной строки, который позволяет собрать собственную индивидуальную версию Collector.

## Компоненты

### Receiver

Ресивер, это компонент, который позволяет данным поступать в Collector. Он может принимать данные из нескольких источников. Многие ресиверы поставляются с настройками по умолчанию и не требуют значительной настройки.

Список доступных ресиверов и их базовую настройку см. в официальной [документации OpenTelemetry по ресиверам﻿](https://opentelemetry.io/docs/collector/configuration/#receivers).

### Processor Опционально

Процессор, это опциональный компонент, который преобразует, фильтрует или обогащает данные перед экспортом.

Список доступных процессоров и их базовую настройку см. в официальной [документации OpenTelemetry по процессорам﻿](https://opentelemetry.io/docs/collector/configuration/#processors). У OpenTelemetry есть список [рекомендуемых процессоров﻿](https://github.com/open-telemetry/opentelemetry-collector/tree/v0.156.0/processor#recommended-processors), но они опциональны.

### Exporter

Экспортёр, это компонент, который отправляет обработанные данные в один или несколько бэкендов. Экспортёры могут поддерживать более одного источника данных.

Поскольку многим экспортёрам требуется дополнительная настройка (например, конечная точка), список доступных экспортёров и их конфигураций нужно уточнять в официальной [документации OpenTelemetry по экспортёрам﻿](https://opentelemetry.io/docs/collector/configuration/#exporters).

### Services

Services определяют конвейеры, по которым данные проходят через Collector. Они определяют, какие компоненты работают совместно для обработки данных OpenTelemetry.
