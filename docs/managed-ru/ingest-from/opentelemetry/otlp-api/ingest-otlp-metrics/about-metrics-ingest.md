На этой странице представлена информация о том, как Dynatrace принимает и обогащает метрики OpenTelemetry.

Миграция с Dynatrace OpenTelemetry metrics exporter

Если вы ещё используете Dynatrace OpenTelemetry metrics exporter, рекомендуется перейти на OTLP metrics exporter.
Дополнительные сведения см. в разделе [Migrating from the Dynatrace OTel metrics exporter to standard OTLP metrics exporter](https://community.dynatrace.com/t5/Open-Q-A/Migrating-from-the-Dynatrace-OTel-metrics-exporter-to-standard/m-p/286986/thread-id/37689#M37690).

## Сопоставление с типами метрик Dynatrace

Dynatrace сопоставляет отдельные инструменты OpenTelemetry со следующими типами метрик Dynatrace:

| Инструмент | с темпоральностью | сопоставляется с типом Dynatrace |
| --- | --- | --- |
| Counter | Дельта | Counter |
| Counter | Кумулятивная | N/A |
| Gauge | N/A | Gauge |
| Explicit bucket histogram [1](#fn-1-1-def) | Дельта | Histogram |
| Exponential histogram [2](#fn-1-2-def) | Дельта | Exponential histogram |
| UpDownCounter | Дельта | Counter |
| UpDownCounter | Кумулятивная | Gauge |
| Summary | N/A | N/A |

1

Explicit bucket histograms поддерживаются в Dynatrace Managed версии 1.324+ и ActiveGate версии 1.321+.

2

Для экспоненциальных гистограмм Dynatrace принимает `min|max|sum|count` гистограммы, но не принимает интервалы.

## Пределы и проверки API

При приёме метрик OpenTelemetry применяются пределы и проверки, описанные в таблицах ниже.

| Сущность | Предел | Описание |
| --- | --- | --- |
| Длина ключа метрики, символы | Мин: 2, Макс: 255 | Длина ключа метрики. |
| Длина ключа измерения, символы | Мин: 1, Макс: 100 | Длина ключа измерения. Если превышена максимальная длина, ключ усекается до верхнего предела. |
| Длина значения измерения, символы | Мин: 1, Макс: 255 | Длина значения измерения. Если превышена максимальная длина, значение измерения усекается до верхнего предела. |
| Количество измерений на точку данных метрики | 50 | Максимальное общее количество измерений в одной точке данных метрики. Если число измерений превышено, вся точка данных отбрасывается. |
| Общее количество возможных ключей метрик на среду | 100 000 | Максимальное количество ключей метрик, которые могут быть активны одновременно. |
| Количество кортежей в месяц на метрику | 1 000 000 | Максимальное количество кортежей (уникальных комбинаций ключ-значение измерение-метрика) для каждого ключа метрики за последние 30 дней. |
| Количество кортежей в месяц для всех пользовательских метрик | 50 000 000 | Максимальное количество кортежей (уникальных комбинаций ключ-значение измерение-метрика) для всех пользовательских метрик за последние 30 дней. |
| Единица инструмента, символы | 63 | Максимальная общая длина единицы измерения инструмента. Если превышена максимальная длина, единица измерения отбрасывается. |
| Описание инструмента, символы | 1 023 | Максимальная общая длина описания инструмента. Если превышена максимальная длина, описание инструмента усекается до лимита символов. |
| Размер запроса | 4 МБ | Максимальный несжатый размер OTLP-запроса с полезной нагрузкой метрик. Если лимит превышен, весь запрос отбрасывается. |
| Точки данных метрик | 15 000 | Максимальное количество точек данных метрик в OTLP-запросе с полезной нагрузкой метрик. Если лимит превышен, весь запрос отбрасывается. |

### Приём атрибутов

OpenTelemetry поддерживает атрибуты на разных уровнях в запросе метрик OpenTelemetry: точки данных, области видимости и ресурсы.
Так как атрибуты сохраняются на стороне Dynatrace в плоском виде, при совпадении имён атрибутов на разных уровнях могут возникать коллизии имён.

Для разрешения таких конфликтов имён Dynatrace применяет следующий приоритетный порядок выбора принимаемого атрибута:

1. Атрибуты точки данных
2. Атрибуты области видимости
3. Атрибуты ресурса

Например, если точка данных и атрибут области видимости имеют одинаковое имя, приоритет отдаётся значению точки данных.
Аналогично, если атрибут области видимости и атрибут ресурса имеют одинаковое имя, Dynatrace примет значение атрибута области видимости.

### Aggregation temporality

Бэкенд Dynatrace работает исключительно с дельта-значениями и требует соответствующей темпоральности агрегации.
Убедитесь, что ваш exporter метрик настроен соответствующим образом, или задайте переменной окружения [`OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE`](https://opentelemetry.io/docs/specs/otel/metrics/sdk_exporters/otlp/) значение `DELTA`.

Примеры установки темпоральности для конкретных языков программирования см. в разделе [пошаговые руководства по интеграции](/managed/ingest-from/opentelemetry/walkthroughs "Узнайте, как интегрировать и принимать данные OpenTelemetry (трассировки, метрики и логи) в Dynatrace.").

### Ключи метрик

* Ключ метрики состоит из секций, разделённых точками (например, `dt.metrics`).
* Ключ метрики может содержать строчные и прописные буквы, цифры, дефисы (`-`) и символы подчёркивания (`_`).
* Ключ метрики должен начинаться с буквенного символа.
* Ключ метрики не должен содержать не-латинские символы (например, `ä`, `ö` и `ü`).
* К ключу метрики может автоматически добавляться суффикс в зависимости от полезной нагрузки (например, `.count` для счётчиков и `.gauge` для gauge).

Если используются символы, недопустимые согласно приведённым правилам, они будут заменены символами подчёркивания.
Если ключ метрики не содержит ни одного допустимого символа, точка данных будет отброшена.

### Ключи измерений

* Измерения сопоставимы с атрибутами спана или ресурса.
* Ключ измерения может содержать только строчные буквы (не прописные), цифры, дефисы (`-`), точки (`.`) и символы подчёркивания (`_`).
* Ключ измерения должен начинаться с буквенного символа.
* Ключ измерения не должен содержать не-латинские символы (например, `ä`, `ö` и `ü`).
* Ключ измерения может иметь формат `key.key-section`.
* Можно задать до 50 измерений.
* Если один и тот же ключ измерения указан несколько раз в одной полезной нагрузке, принимается только первое значение.

Если используются символы, недопустимые согласно приведённым правилам, они будут заменены символами подчёркивания.
Если ключ измерения не содержит ни одного допустимого символа, ключ будет отброшен.

Значения измерений должны передаваться как строка, булево значение или целое число.

Dynatrace не поддерживает измерения нестрокового типа и при приёме преобразует булевы значения и целые числа в строки.

Если используется любой другой тип, всё измерение будет отброшено.

### Гистограммы

Для экспоненциальных гистограмм Dynatrace принимает `min|max|sum|count` гистограммы, но не принимает интервалы.

Для explicit bucket histogram следует учитывать несколько аспектов:

* Dynatrace добавляет к ключу метрики гистограммы суффикс `.histogram`.
* Метрика гистограммы тарифицируется как метрика count или gauge; отдельные интервалы не тарифицируются отдельно.
* Хранится не более 12 интервалов на точку данных гистограммы.
* Отрицательные границы интервалов не поддерживаются.

Если происходит любое из нижеперечисленного, OTLP ingest API возвращает ответ `400` или `200 with partial success`.

* Кумулятивные гистограммы не принимаются (аналогично кумулятивным счётчикам).
* Точки данных гистограммы без суммы не принимаются. Это происходит при записи отрицательных значений.
* Интервалы гистограммы не отсортированы.
* Граничные значения интервалов гистограммы `NaN` или `Infinite` недопустимы.

Dynatrace OpenTelemetry ingest API возвращает HTTP `400` только в том случае, если все метрики в OTLP-запросе недопустимы.

### Сводки (Summary)

Dynatrace не поддерживает метрики типа Summary.

Метрики Summary существуют в протоколе OpenTelemetry (OTLP) только для совместимости с другими форматами.
Приложения, использующие официальные SDK OpenTelemetry, не могут создавать метрики Summary.
