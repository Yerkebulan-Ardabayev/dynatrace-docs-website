# Установка модуля IMS

* Чтение: 21 мин
* Обновлено 18 ноября 2025 г.

С помощью модуля IMS можно получить наблюдаемость для транзакций и программ IMS, включая IBM MQ и обращения к базам данных.

| Наблюдаемость для | Включая |
| --- | --- |
| Транзакции  * IMS * Fast Path * [BMP](https://www.ibm.com/docs/en/ims/15.1.0?topic=bmps-batch-message-processing-transaction-oriented) | Транзакции, инициируемые через  * IBM MQ Bridge и Trigger Monitor * IMS TM Resource Adapter, IMS SOAP Gateway и IMS Connect * терминал 3270 |
| Обращения к базам данных | Обращения к базам данных с SQL-операторами из IMS к Db2 и IMS DB через  * метод доступа DL/I * методы доступа Fast Path |

## Установка

Модуль IMS перехватывает данные о различных событиях обработки транзакций IMS и передаёт данные мониторинга в подсистему zDC через разделяемые буферы.

Чтобы установить модуль IMS Dynatrace:

1. Установите модуль IMS в контрольный регион каждой системы IMS DB/DC и DCCTL, которую нужно мониторить. Этого достаточно, чтобы охватить все регионы обработки сообщений, связанные с контрольным регионом.
   Установка модуля IMS только в систему DBCTL не поддерживается.
2. Нужно добавить Dynatrace exit в каждый IMS Connect, который требуется мониторить.
3. Нужно добавить [модуль z/OS Java](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/install-zos-java#middleware "Настройка мониторинга Java на z/OS с помощью модуля Java.") в каждый IMS SOAP Gateway, который требуется мониторить.

Перезапуск IMS

4. После каждого перезапуска IMS необходимо повторно запустить задание внедрения. Подробнее см. раздел [Замечания по внедрению](#injection-notes).

### Обновление ранее установленной версии Dynatrace

Если Dynatrace уже установлен и выполняется обновление до новой версии, перейдите к разделу [Обновление без перезапуска региона](#update) в этом документе.

### IMS Connect

Добавьте авторизованный набор данных Dynatrace `<hlq>.SZDTAUTH` в STEPLIB задания IMS Connect, чтобы разрешить IMS Connect загружать Dynatrace exit HWSTECL0.

* Если используется IMS Connect Extensions, добавьте SZDTAUTH после библиотеки IMS Connect Extensions.
* Если используется локально разработанный exit HWSTECL0, добавьте SZDTAUTH перед набором данных, содержащим локальный exit.

Exit IMS Connect можно включить для создания узлов PurePath в распределённой трассировке. Активируйте требуемую [возможность OneAgent](/managed/ingest-from/dynatrace-oneagent/oneagent-features "Управление возможностями OneAgent глобально и для каждой группы процессов.") **z/OS IMS Connect**.

* Если exit IMS Connect настроен на создание узлов PurePath, он будет подключаться к имени подсистемы zDC по умолчанию. Имя целевой подсистемы zDC можно переопределить, указав следующие DDNAME и ключевой параметр в JCL запуска IMS Connect:

```
//ZDTPARMS DD *



ZDCID=<zDC_Id>
```

Exit IMS Connect необходим для поддержки протокола IMS Connect API и поддержки IMS SOAP Gateway в случаях, когда SOAP Gateway не настроен на вставку идентификатора отслеживания с помощью команды `iogmgmt`.

### Контрольный регион IMS

Для установки модуля IMS в контрольный регион необходимо запустить утилиту внедрения (Injection Utility).

1. Авторизуйте `<hlq>.SZDTAUTH`, добавив его в список APF
2. Запустите задание ZDTINST из PDS `<hlq>.SZDTSAMP`.

Подробная информация о параметрах также содержится в элементе PDS ZDTINST.

Пример JCL с позиционными параметрами:

```
//S1       EXEC PGM=ZDTIINST,PARM='<IMS_Id>,<zDC_Id>'



//STEPLIB  DD DISP=SHR,DSN=<hlq>.SZDTAUTH
```

Пример JCL с ключевыми параметрами:

```
//S1       EXEC PGM=ZDTIINST



//STEPLIB  DD DISP=SHR,DSN=<hlq>.SZDTAUTH



//SYSIN DD *



IMSID=<IMS_Id>



ZDC=<zDC_Id>
```

Утилита внедрения (ZDTIINST) поддерживает изменение размера таблиц `pathid` без перезапуска IMS и повторного внедрения модуля. Размер таблицы путей можно изменить, используя код действия `M` с указанием сегмента таблицы путей в качестве шестого параметра и, при необходимости, целевого IMSID в качестве седьмого параметра.

Ниже приведён полный список и описание позиционных параметров утилиты внедрения. Все параметры должны быть в верхнем регистре:

```
// ...    PARM='<IMS_Id>,<zDC_Id>,<ActionCode>|<PathTableSize>,



<WaitZdcMin>,<Y|N>,<PathTableSegm>,<PathTableIms>'
```

Параметры утилиты внедрения можно задавать как позиционными, так и ключевыми параметрами. Позиционных параметров семь. Ключевые параметры должны задаваться в файле SYSIN. Позиционные параметры в операторе EXEC JCL по-прежнему допустимы, однако новые параметры в будущем будут добавляться только как ключевые значения SYSIN. Если задан параметр EXEC PARM, параметры SYSIN не учитываются. Рекомендуемый метод: использовать файл SYSIN и ключевые параметры.

| Символ | Позиция | Ключевое слово | Описание | Обязательный |
| --- | --- | --- | --- | --- |
| `<IMS_Id>` | 1 | IMSID= | Четырёхсимвольный IMSID контрольного региона.  Параметр не нужен при использовании функции сборки мусора для освобождения памяти ECSA, содержащей неиспользуемые модули IMS. | Обязательный |
| `<zDC_Id>` | 2 | ZDC= | Четырёхсимвольный zDCID, используемый в параметре SUBSYSTEM_ID() целевого zDC.  Параметр не нужен, если целевой zDC является zDC ПО УМОЛЧАНИЮ. В этом случае введите звёздочку (`*`) или опустите параметр. | Обязательный |
| `<ActionCode>|<PathTableSize>` | 3 | ACTION= | SIZE= | Однобуквенный код действия или значение размера таблицы `pathid`. Код действия Коды действий задают действие над ранее внедрённым модулем IMS. Код действия неприменим к первоначальному внедрению модуля IMS и является взаимоисключающим со значением размера таблицы `pathid`.  Если задан допустимый код действия, программа внедрения не выполняет повторное внедрение модуля IMS, а выполняет указанное действие над уже внедрённым модулем. Это означает, что новые версии модулей IMS загружены не будут.  Коды действий:  * **E**: Enable (включить) внедрённый модуль IMS, который был ранее отключён. * **D**: Disable (отключить) внедрённый модуль IMS. Модуль IMS больше не вызывается. Для повторного включения используйте код действия `E`. * **G**: Garbage collection (сборка мусора). Если модуль IMS был обновлён и внедрён, эта функция освобождает память ECSA, занятую старыми (неиспользуемыми) модулями IMS. Подробнее см. раздел [Замечания по сборке мусора](#gc-notes). * **M**: Изменить параметр восстановления внедрённого модуля IMS при использовании в сочетании с пятым параметром `<Y|N>`.    Либо изменить размер ранее выделенных таблиц путей при использовании в сочетании с шестым параметром `<PathTableSegm>` и, при необходимости, седьмым параметром `<PathTableIms>`.    Либо изменить размер ранее выделенного объекта разделяемой памяти Fast Path SMO (Shared Memory Object) при использовании в сочетании с оператором `FPATHSIZE=`. * **F**: Освободить все блокировки ресурсов и сбросить указатели таблицы `pathid`. Этот код действия следует использовать только в том случае, если действие `E` завершилось с ошибкой из-за блокировки ресурса модуля IMS или по указанию Dynatrace Support. Значение размера таблицы Значение размера таблиц `pathid` используется при первоначальном внедрении модуля IMS и является взаимоисключающим с кодами действий. Если модуль IMS уже внедрён, значение размера таблицы игнорируется.  Значение размера таблицы задаётся как SEGM=nnnn, где nnnn: число сегментов по 1 МБ (до 4 цифр).  Фактическое количество доступных записей может быть больше, поскольку память выделяется сверх указанной квоты. Количество записей будет округлено вверх для использования всей выделенной памяти.  Если параметр не указан при первоначальном внедрении модуля IMS, используется значение по умолчанию: четыре сегмента (4 МБ). Это обеспечивает место для 8 191 записи (приблизительно 2 048 записей на сегмент).  Количество выделенных записей должно соответствовать ожидаемому количеству распределённых трассировок IMS в течение любой одной минуты. При превышении числа трассируемых транзакций IMS относительно числа доступных слотов таблицы новые распределённые трассировки не записываются. Третий параметр не задан Если третий параметр не задан, программа внедрения внедряет модуль IMS.  Этот вариант также можно использовать для обновления модуля IMS. В этом случае существующий размер таблицы остаётся неизменным. | Необязательный |
| `<WaitZdcMin>` | 4 | ZDCWAIT= | Время ожидания инициализации целевого zDC, в минутах (максимум две цифры).  Значение по умолчанию: 30 минут. Целевой zDC должен быть инициализирован хотя бы один раз с момента последнего IPL системы, прежде чем модуль IMS сможет завершить инициализацию. | Необязательный |
| `<Y|N>` | 5 | DUMP= | Указывает, должен ли модуль IMS захватывать SVC дамп при срабатывании процедуры восстановления ABEND.  Параметр можно задавать при первоначальном внедрении модуля IMS или в сочетании с кодами действий `M` или `E` (параметр номер 3) для включения/выключения захвата дампа при восстановлении ранее внедрённого модуля IMS.  Значение по умолчанию: `Y`, то есть захват дампа включён. | Необязательный |
| `<PathTableSegm>` | 6 | SEGM= | Значение для изменения размера таблиц `pathid`.  Параметр применяется только в сочетании с кодом действия `M` (параметр номер 3). Формат значения: `SEGM=nnnn`, где `nnnn`: число сегментов по 1 МБ (до 4 цифр). | Необязательный |
| `<PathTableIms>` | 7 | PATHIMS= | Значение (до 4 символов), задающее IMSID, для которого изменяется размер таблицы pathid.  Параметр применяется только в сочетании с кодом действия `M` (параметр номер 3) и значением `<PathTableSegm>` (параметр номер 6).  Если IMSID указан, изменяется размер таблицы pathid только для этого IMS.  Если параметр опущен, изменяется размер таблицы `pathid` только для локального IMS (параметр номер 1).  Если указана звёздочка (`*`), изменяется размер таблицы `pathid` для локального IMS **и** таблиц `pathid`, выделенных локально для всех удалённых IMSID. | Необязательный |
| n/a | n/a | REMOTESEGM= | Задаёт число сегментов для выделения удалённых таблиц pathid. Формат значения: `REMOTESEGM=nnnn`, где `nnnn`: число сегментов по 1 МБ (до 4 цифр).  Параметр необязательный, допустим только как параметр `SYSIN` и игнорируется при указании ключевого слова `ACTION=`. Если параметр опущен, используется значение по умолчанию: `4` МБ.  Этот параметр не изменяет размер уже существующих удалённых таблиц pathid. Чтобы изменить выделение существующей удалённой таблицы pathid, необходимо использовать ключевые слова `ACTION=M`, `SEGM=` и `PATHIMS=`. | Необязательный |

Пример 1. Внедрение модуля IMS в IMS IB01 для zDC ZDC1.

Принять значение `PathTableSize` по умолчанию и ждать инициализации ZDC1 не более 20 минут.

Третий позиционный параметр не задан.

### Трассировка транзакций Fast Path

| Символ | Позиция | Ключевое слово | Описание | Обязательный |
| --- | --- | --- | --- | --- |
| n/a | n/a | FPATH= | Указывает, следует ли активировать трассировку транзакций Fast Path:  * **Y**: Внедрить и включить хуки Fast Path. Включает хуки, которые внедрены, но отключены. * **N** (по умолчанию): Не внедрять хуки Fast Path. Отключает хуки, которые внедрены и включены.  Допустимо только как параметр SYSIN и только для IMS версии 15 и выше.  Если параметр опущен или задан неверно, никаких действий не выполняется. | Необязательный |
| n/a | n/a | FPATHSIZE= | Размер объекта разделяемой памяти Fast Path SMO (Shared Memory Object) для трассировки транзакций Fast Path, в сегментах по 1 МБ. Диапазон: `1`-`9999`. Значение по умолчанию: `4`.  Значение можно задавать при первоначальном внедрении модуля IMS или в сочетании с оператором изменения (`ACTION=M`) для изменения размера Fast Path SMO.  Допустимо только как параметр SYSIN и только для IMS версии 15 и выше. | Необязательный |

### Трассировка транзакций BMP

IMS module версии 1.259+

| Символ | Позиция | Ключевое слово | Описание | Обязательный |
| --- | --- | --- | --- | --- |
| n/a | n/a | BMP= | Указывает, следует ли активировать трассировку транзакций для регионов BMP:  * **Y**: Включает трассировку транзакций для регионов BMP. * **N** (по умолчанию): Отключает трассировку транзакций для регионов BMP.  Допустимо только как параметр SYSIN.  Если параметр опущен или задан неверно, трассировка транзакций BMP будет отключена. | Необязательный |

## Замечания по трассировке транзакций BMP

Автоматическое отслеживание транзакционно-ориентированных BMP (входные сообщения поступают из очередей сообщений IMS) будет аналогично тому, что существует для регионов MPP/IFP: если входное сообщение отслеживается сенсором контрольного региона, для транзакции в регионе BMP будет запущен путь трассировки.
Автоматическое отслеживание работы пакетных (нетранзакционно-ориентированных) BMP возможно только при поступлении входного сообщения через MQGET (см. ограничения ниже). В противном случае поддержка пакетных BMP означает, что агент выполняет инициализационные сервисы для региона BMP (выделяет рабочие области и размещает сенсоры), но не запускает пути трассировки.
Для транзакционно-ориентированных и пакетных BMP путь трассировки запускается для региона BMP при каждом вызове MQGET приложением, при соблюдении всех следующих условий:

1. Инструментирование включено для фильтра z/OS IMS MPR MQ в настройках UI Settings.
2. Другой сенсор ещё не запустил путь трассировки для региона BMP.
3. Имя очереди MQ для MQGET включено или не исключено по своему наличию или отсутствию в списке фильтров IMS MQ (**Settings** > **Mainframe** > **IBM MQ Filters**).
4. Вызов DL/I CHKP выполнен перед первым MQGET. Это требуется только для пакетно-ориентированных BMP.

Для транзакционно-ориентированных и пакетных BMP SDK можно использовать для инструментирования приложения с целью отправки событийных сообщений для запуска и завершения путей трассировки.
Не предусмотрена возможность ограничить отслеживание конкретными именами заданий BMP, PSB или отслеживать пакетные задания, отличные от IMS BMP.

Пример 13: Включение трассировки транзакций BMP.

С ключевым параметром:

```
//S1       EXEC PGM=ZDTIINST



//STEPLIB  DD DISP=SHR,DSN=<hlq>.SZDTAUTH



//SYSIN DD *



IMSID=IMSA



ZDC=ZDC1



BMP=Y
```

Пример 14: Отключение трассировки транзакций BMP.

С ключевым параметром:

```
//S1       EXEC PGM=ZDTIINST



//STEPLIB  DD DISP=SHR,DSN=<hlq>.SZDTAUTH



//SYSIN DD *



IMSID=IMSA



ZDC=ZDC1



BMP=N
```

## Замечания по сборке мусора

При обновлении модуля IMS можно освободить память ECSA, занятую старой версией модуля IMS.

После запуска задания внедрения обновлённого модуля IMS активность IMS Message Region необходима для обновления внутренних управляющих блоков, чтобы они указывали на новую версию модуля IMS. После этого можно повторно запустить задание внедрения с кодом действия `G` в параметре номер 3. Это помечает старую версию модуля IMS как кандидата на удаление.

Не менее чем через 3 часа запустите задание внедрения с кодом действия `G` в параметре номер 3 повторно. Это освобождает память ECSA, содержащую старую версию модуля IMS.

Задание внедрения с кодом действия `G` в параметре номер 3 можно запускать многократно без нежелательных последствий. Однако для того, чтобы процесс сборки мусора освободил неиспользуемую память модуля, он должен быть вызван не менее двух раз после IPL с интервалом не менее 3 часов между первым и вторым вызовом.

## Замечания по внедрению

Запустите zDC хотя бы один раз после IPL системы перед выполнением задания внедрения модуля IMS.

Если целевой zDC не запускался с момента последнего IPL, задание внедрения ожидает его запуска. Время ожидания по умолчанию составляет 30 минут, но его можно задать параметром `<WaitZdcMin>`. Если zDC по-прежнему не запущен по истечении этого времени, внедрение невозможно, и задание завершается с пользовательским кодом `ABEND` 100.

Если zDC запускался хотя бы один раз с момента последнего IPL системы, задание внедрения может завершиться даже при незапущенном zDC. В этом случае выдаются предупреждающие сообщения о том, что zDC не запущен и сбор данных недоступен, а задание завершается с `RC=4`.

Запустите и полностью инициализируйте систему IMS перед выполнением задания внедрения модуля IMS. При останове и последующем запуске системы IMS задание внедрения необходимо запустить повторно. При запуске нового зависимого региона IMS или его останове с последующим запуском никаких действий не требуется.

При обновлении модуля IMS в ранее внедрённом IMS повторно запустите задание внедрения, чтобы загрузить обновлённые загрузочные модули. Не указывайте код действия для задания внедрения в параметре номер 3. IMS **не требует** перезапуска после обновления модуля IMS.

Для выполнения задания внедрения модуля IMS рекомендуем использовать средство автоматизации, например IMS Time-Controlled Operations (TCO).

Все сообщения о внедрении модуля IMS записываются в журнал задания. Уровень сообщений можно регулировать, включая в JCL задания внедрения конкретные DDNAMEs. Если DDNAMEs не включены, по умолчанию используется уровень 4 для информационных сообщений. Конкретные DDNAMEs и уровни сообщений:

* `//DTMSGLV0 DD SYSOUT=*` Для сообщений уровня 0 DEBUG (детальный) и выше
* `//DTMSGLV2 DD SYSOUT=*` Для сообщений уровня 2 DEBUG и выше
* `//DTMSGLV4 DD SYSOUT=*` Для информационных сообщений уровня 4 и выше
* `//DTMSGLV5 DD SYSOUT=*` Для предупреждающих сообщений уровня 5 и выше

Сбои задания внедрения, как правило, приводят к пользовательскому коду `ABEND` 100. В большинстве случаев причину сбоя можно определить по предшествующим диагностическим сообщениям, записанным в SYSPRINT.

В зависимости от системных параметров z/OS в журнал задания также может быть записан симптомный дамп, однако дамп не захватывается, если в JCL задания внедрения не включён один из следующих DDNAMEs:

```
//SYSABEND DD SYSOUT=*



//SYSMDUMP DD DSN=DT.IMSAGENT.INJECT.SDUMP,DISP=(OLD,KEEP,KEEP)
```

Оба образца элементов JCL: ZDTINST (утилита внедрения) и ZDTIDIAG (диагностическая утилита) содержат финальный шаг для выполнения процедуры ZDTZLOGC. Функция шага ZDTZLOGC: скопировать вывод спула задания в файл журнала HFS, чтобы он автоматически включался в архив поддержки. Для этой функции предусмотрены новая JCL-процедура (ZDTZLOGC), программа REXX (ZDTZLOGR) и программа ассемблера (ZDTZLOGS). Для реализации выполните следующее:

1. Скопируйте образец элемента JCL ZDTZLOGC в PDS для размещения изменённого JCL. В JCL PROC необходимо изменить DSN для DD-имён SYSEXEC и SZDTAUTH.
   `//SYSEXEC DD DISP=SHR,DSN=<hlq>.SZDTSAMP` <- замените `DSN` на путь к `REXX` exec `ZDTZLOGR` (см. шаг 3)
   `//SZDTAUTH DD DISP=SHR,DSN=<hlq>.SZDTAUTH` <- замените `DSN` на путь к программе `ZDTZLOGS`
2. Оба образца элементов ZDTINST и ZDTIDIAG содержат оператор JCLLIB, который необходимо изменить, а шаг процедуры ZDTZLOGC должен содержать `<ZDC Id>`.
   `JCLLIB ORDER=(<hlq>.SZDTSAMP)` <- замените `DSN` на путь к JCL-процедуре `ZDTZLOGC` (см. шаг 1)
   `//ZDTZLOGC EXEC ZDTZLOGC,ZDC='<ZDC Id>',COND=(EVEN)` <- замените ZDC Id (например, `MEPC` или `*` для zDC по умолчанию)

## Журналирование

Существует два набора журналов модуля IMS.

* Первый набор сообщений IMS поступает от задания внедрения IMS. Это сообщения, возникающие в процессе внедрения модуля в контрольный регион IMS. Данные сообщения отображаются только в спуле задания внедрения модуля IMS.
* Второй набор сообщений IMS поступает от модуля IMS в процессе мониторинга активности IMS. Эти сообщения направляются в zDC, а затем перенаправляются в zRemote.

Доступ к журналам модуля IMS можно получить через [журналы zRemote](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/install-zremote#logging "Подготовка и установка модуля zRemote для мониторинга z/OS.").

## Обновление без перезапуска региона

Чтобы обновить модуль IMS до новой версии без перезапуска региона:

1. [Загрузите наборы данных продукта z/OS](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/zosmf-installer/download-zos-datasets#download-pax "Загрузка и установка наборов данных продукта Dynatrace для z/OS.") и [распакуйте их](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/zosmf-installer/download-zos-datasets#extract-datasets "Загрузка и установка наборов данных продукта Dynatrace для z/OS.").
2. Обновите задание внедрения, указав новый `<hlq>.SZDTAUTH`. Если вы [определили псевдоним](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/zosmf-installer/download-zos-datasets#alias "Загрузка и установка наборов данных продукта Dynatrace для z/OS."), переопределите его. Например:

   ```
   DELETE 'DT.DYNTRC.SZDTAUTH' NOSCRATCH



   DEFINE ALIAS(NAME('DT.DYNTRC.SZDTAUTH') RELATE('DT.R12710.SZDTAUTH'))
   ```
3. Запустите задание внедрения без кода действия, чтобы загрузить обновлённый модуль IMS. Подробнее см. раздел [Замечания по внедрению](#injection-notes).
4. Для освобождения памяти ECSA, занятой старой версией модуля IMS, см. раздел [Замечания по сборке мусора](#gc-notes).

## Вопросы и ответы

Как деактивировать модуль IMS?

При возникновении ABEND в модуле IMS процесс восстановления по возможности формирует диагностику ABEND, а затем отключает модуль IMS. Система IMS продолжает функционировать. При этом в системный журнал контрольного региона IMS и/или зависимого региона IMS записывается серия WTO-сообщений. Ниже приведён пример типичного набора сообщений:

```
ZDTI032W Recovery routine entered.



ZDTI036W ZDTIII15 0000000 20221103 10.51 VER 1.255.0 ABEND at offset 007874.



ZDTI033W Successful ABEND recovery, agent disabled.
```

Дополнительные или отличающиеся сообщения могут выдаваться при обнаружении аномальных условий в процессе восстановления (например, при невозможности получить динамическую память, запрете повторной попытки или отсутствии SDWA). Все сообщения, связанные с процессом восстановления ABEND, задокументированы в разделе [Сообщения модулей z/OS](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/operation/zos-code-module-messages "Сообщения, создаваемые модулями Dynatrace z/OS.").

Запись об ошибке программного обеспечения (SFT), подробно описывающая ABEND, как правило, записывается в набор данных SYS1.LOGREC системы z/OS. Для вывода записи об ошибке программного обеспечения (SFT), связанной с ABEND, следует запустить служебную программу EREP z/OS.

При желании во время восстановления может быть снят SVC дамп в зависимости от параметра восстановления ABEND, заданного или принятого по умолчанию при внедрении модуля IMS. Действие по умолчанию: захватывать SVC дамп при срабатывании процедуры восстановления ABEND. Этот параметр можно задать при первоначальном внедрении модуля IMS или в сочетании с параметрами функций Modify или Enable для включения/выключения захвата дампа при восстановлении ранее внедрённого модуля IMS.

При отключении модуля IMS в результате процесса восстановления ABEND он остаётся отключённым до тех пор, пока не будет явно включён снова с помощью программы внедрения модуля IMS.

Как обнаружить несовместимость версий?

Убедитесь, что версия модуля IMS меньше или равна версии модуля zRemote. Не подключайте более новые модули IMS к более старым модулям zRemote. Ниже приведён пример сообщения в журнале zRemote при несовместимости версии модуля IMS с версией zRemote.

```
severe  [native] IMS14CR1[asid = 108] is trying to initialize with an invalid protocol version number : x.xxx.xx
```
