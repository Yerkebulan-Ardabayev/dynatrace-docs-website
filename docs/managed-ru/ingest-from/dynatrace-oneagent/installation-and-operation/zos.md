# Dynatrace для z/OS

* Чтение: 8 мин
* Обновлено 15 ноября 2022 г.

С Dynatrace вы получаете полные транзакционные инсайты по своим нагрузкам от мобильного фронтенда до программ на мейнфрейме и всего, что находится между ними, чтобы устранять аномалии на уровне кода. Кроме того, Dynatrace может сопровождать вас на пути к гибридному облаку благодаря сквозной наблюдаемости от мейнфрейма до облака.

Узнайте, как Dynatrace решает наиболее типичные задачи на мейнфрейме:

Является ли мейнфрейм частью проблемы?

Изоляция домена отказов на основе ИИ точно определяет первопричину проблем и оценивает их влияние на пользователей, чтобы вы могли расставлять приоритеты в стратегиях устранения и сокращать среднее время восстановления.

## Настройка мониторинга

Dynatrace предоставляет кодовые модули для технологий CICS, IMS и z/OS Java, чтобы вы могли достичь бесшовной наблюдаемости с инсайтами по трассировкам и метрикам. Чтобы узнать больше о поддерживаемых технологиях, см. [Поддержка технологий мейнфрейма](/managed/ingest-from/technology-support/mainframe-technology-support "Узнайте, какие технологии Dynatrace поддерживает для мониторинга мейнфрейма.").

![Архитектура мониторинга z/OS](https://dt-cdn.net/images/zos-architecture-1745-8d165d1510.png)

Архитектура мониторинга z/OS

Модули CICS, IMS и z/OS Java взаимодействуют с подсистемой Dynatrace z/OS Data Collection (zDC) через объект общей памяти (SMO) внутри LPAR. Подсистема zDC управляет этим SMO, в который модули записывают данные мониторинга.

zLocal, размещённый в окружении z/OS [Unix System Services](https://www.ibm.com/docs/en/zos/2.5.0?topic=zos-unix-system-services) (USS), работает как часть zDC. Он управляет TCP/IP-соединением с модулем zRemote, читает данные мониторинга из SMO и передаёт их в zRemote.

Модуль zRemote обрабатывает данные мониторинга, полученные от zLocal, и направляет эти данные, сжатые и зашифрованные, через локальный ActiveGate в Dynatrace. Таким образом, модуль zRemote снимает с модулей значительную часть работы по обработке, возникающей при инструментировании подсистем и приложений, перенося её на открытую систему.

Чтобы начать работу, см. [Обзор установки z/OS](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation "Обзор установки модулей Dynatrace z/OS.").

## Часто задаваемые вопросы

Кто должен участвовать в типичной установке Dynatrace для z/OS?

Чтобы узнать процедуру и состав участников, см. [Обзор установки z/OS](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation "Обзор установки модулей Dynatrace z/OS.").

Можно ли использовать группы хостов для организации нескольких LPAR?

Да, можно организовать несколько LPAR через группы хостов. Подробнее см. [Определение групп хостов для организации нескольких LPAR](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/installation/install-zremote/customize-zremote#host-groups "Настройка модуля zRemote под ваши задачи.").

Что означает volatile-сервис и как решать связанные с ним проблемы?

**Volatile CICS** или **volatile IMS** создаётся Dynatrace автоматически, когда превышается максимальное число идентификаторов сервисов, которое можно сгенерировать в регионе (процессе). Чтобы увеличить лимит генерируемых идентификаторов сервисов, обратитесь к продуктовому эксперту Dynatrace через live chat в вашем окружении Dynatrace.

### Модули CICS и IMS

Как работает инструментирование модулей CICS и IMS?

Модуль CICS использует хуки для инструментирования терминальных и прикладных регионов CICS, создавая события интереса.

Модуль IMS использует средства логирования для инструментирования управляющих и обрабатывающих сообщения регионов IMS, создавая события интереса из разбора бинарных логов.

Оба модуля используют хуки для инструментирования IBM MQ, Db2 и DL/I.

Инструментирование байт-кода модулями CICS и IMS не применяется.

### Сколько времени GCP потребляют модули CICS и IMS при инструментировании приложений?

Модули CICS и IMS потребляют некоторое количество времени центрального процессора общего назначения (GCP) при инструментировании приложений на IBM Z, но эти накладные расходы, как правило, очень низкие (порядка 1%-2%, в зависимости от типа мониторируемых транзакций). См. примеры ниже.

| Отрасль клиента | Страна | Кодовый модуль | Год измерения | Метод измерения | Накладные расходы GCP |
| --- | --- | --- | --- | --- | --- |
| Финансы (банк) | Италия | CICS | 2023 | HIS profiling[1](#fn-1-1-def) | < 1.9 % |
| Финансы (банк) | Испания | CICS | 2020 | HIS profiling[1](#fn-1-1-def) | < 1.0 % |
| Страхование | Германия | CICS | 2020 | собственные тесты | < 1.0 % |
| Страхование | Германия | IMS | 2020 | HIS profiling[1](#fn-1-1-def) | < 1.0 % |
| Финансы (банк) | Германия | CICS | 2019 | собственные тесты | < 1.0 % |
| Страхование | Германия | IMS | 2017 | HIS profiling[1](#fn-1-1-def) | < 1.61 % |
| Страхование | Германия | IMS | 2017 | HIS profiling[1](#fn-1-1-def) | < 0.33 % |
| Финансы (банк) | Австрия | CICS | 2015 | HIS profiling[1](#fn-1-1-def) | < 2.04 % |

1

С использованием [Hardware Instrumentation Services](https://www.ibm.com/docs/en/zos/2.1.0?topic=aids-hardware-instrumentation-services) от IBM.

* Значения накладных расходов GCP вычислены относительно адресных пространств, в которых работают модули. Если же сравнивать накладные расходы GCP относительно LPAR, эти значения будут ещё ниже.
* Например, 2% накладных расходов GCP в адресных пространствах CICS соответствуют только 1% накладных расходов GCP по LPAR, если нагрузки CICS потребляют только 50% общего времени GCP на этом LPAR по сравнению с другими нагрузками (такими как задания и системные задачи).

Могут ли модули CICS и IMS захватывать динамические SQL-операторы?

Нет, модули CICS и IMS могут захватывать только статические SQL-операторы.

### Модуль z/OS Java

Можно ли мониторить Servants моего WebSphere Application Server модулем z/OS Java?

WebSphere Application Server на z/OS позволяет динамически разворачивать [Servants](https://www.ibm.com/docs/en/was-zos/9.0.5?topic=zos-websphere-application-server-terminology) в зависимости от нагрузки.

В Dynatrace для различения разных Servants можно использовать **process identifier** на каждой метрике, как показано на изображении ниже.

![Идентификатор процесса в метриках](https://dt-cdn.net/images/was-process-id-1607-06f91781b9.png)

Идентификатор процесса в метриках

Однако Servants нельзя включить в обнаружение групп процессов WebSphere Application Server, потому что при разворачивании нового Servant создаётся новая сущность процесса и текущий контекст мониторинга теряется.

Какие атрибуты учитываются при обнаружении групп процессов WebSphere Application Server?

Dynatrace использует следующие атрибуты для обнаружения и создания сущностей процессов WebSphere Application Server:

* Имя сервера
* Имя узла
* Имя ячейки

Dynatrace группирует все сущности процессов, принадлежащие одному кластеру WebSphere Application Server, в одну группу процессов.

Можно ли объединить группы процессов, созданные модулем z/OS Java, в одну группу процессов?

Нет, группы процессов, созданные модулем z/OS Java, нельзя изменять или объединять.

В качестве альтернативы можно организовать группы процессов через [определение метаданных](/managed/observe/infrastructure-observability/process-groups/configuration/define-your-own-process-group-metadata#variables "Настройте собственные метаданные процессов с учётом уникальных потребностей вашей организации или окружения.") или [определение тегов](/managed/manage/tags-and-metadata/setup/define-tags-based-on-environment-variables "Узнайте, как Dynatrace позволяет определять теги на основе переменных окружения.") на основе переменных окружения. Оба подхода применимы также к z/OS Java. Учтите, что переменные окружения можно определить только на уровне процесса, а не на уровне хоста.

Можно ли тегировать процессы, созданные модулем z/OS Java?

Да. Процессы, созданные модулем z/OS Java, можно тегировать через [определение метаданных](/managed/observe/infrastructure-observability/process-groups/configuration/define-your-own-process-group-metadata#variables "Настройте собственные метаданные процессов с учётом уникальных потребностей вашей организации или окружения.") или [определение тегов](/managed/manage/tags-and-metadata/setup/define-tags-based-on-environment-variables "Узнайте, как Dynatrace позволяет определять теги на основе переменных окружения.") на основе переменных окружения. Учтите, что переменные окружения можно определить только на уровне процесса, а не на уровне хоста.

Можно ли определять пользовательские сервисы через модуль z/OS Java?

Модуль z/OS Java не поддерживает пользовательские сервисы исключительно через конфигурацию. Вместо этого можно создавать пользовательские трассировки через [интероперабельность с OpenTelemetry](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/monitoring/zos-opentelemetry "Используйте OpenTelemetry, чтобы закрыть пробелы в наблюдаемости ваших Java-приложений на z/OS.") модуля z/OS Java.

Можно ли использовать атрибуты спанов, захваченные модулем z/OS Java, как атрибут запроса?

Да. Чтобы узнать, как настроить атрибут запроса для любого захваченного атрибута спана, см. [Определение атрибута запроса для атрибутов спанов](/managed/ingest-from/dynatrace-oneagent/installation-and-operation/zos/monitoring/zos-opentelemetry#request-attribute "Используйте OpenTelemetry, чтобы закрыть пробелы в наблюдаемости ваших Java-приложений на z/OS.").
