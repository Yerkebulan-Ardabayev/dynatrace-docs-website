# Диагностика OneAgent

# Диагностика OneAgent

* Чтение: 7 мин
* Опубликовано 22 октября 2020 г.

Полностью автоматизированное устранение неполадок OneAgent доступно для окружений Dynatrace SaaS и Managed начиная с версии Dynatrace 1.208.

Рабочий процесс позволяет:

* Автоматически выявлять связанные с OneAgent проблемы в высокодинамичных окружениях в конкретный момент времени
* Легко собирать диагностические данные для конкретной сущности и автоматически получать возможные решения для обнаруженных аномалий
* Быстро самостоятельно устранять типичные проблемы, сокращая время диагностики
* Напрямую передавать в Dynatrace Support все необходимые для диагностики подробности

Диагностика через командную строку

Если доступ к Dynatrace отсутствует или нужно автоматизировать сбор диагностических данных, используйте команду `oneagentctl` для сбора подмножества полных диагностических данных прямо на хосте, где установлен OneAgent. Подробнее см. раздел [Создать архив поддержки](/managed/ingest-from/dynatrace-oneagent/oneagent-configuration-via-command-line-interface#create-support-archive "Узнайте, как выполнять задачи настройки OneAgent без переустановки.").

## Требования

* Окружение Dynatrace SaaS: Dynatrace версии 1.202+
* Окружение Dynatrace Managed: Dynatrace версии 1.208+
* Разрешение окружения **View sensitive request data**

## Автоматический анализ

Данная процедура описывает стандартный сценарий: Dynatrace собирает диагностические данные для хоста или процесса и немедленно анализирует их.

Если требуется сначала собрать и проверить данные перед тем, как вручную передать их в Dynatrace для анализа, см. раздел [Собрать и проверить локально](#collect-and-review-locally).

1. Перейдите в **Hosts**.
2. Выберите хост, для которого нужно выполнить диагностику.
3. Запустите диагностику на уровне хоста или процесса.

   * Уровень хоста: на странице хоста откройте меню (**...**) и выберите **Run OneAgent diagnostics**.
     Пример выбора пункта меню

     ![OneAgent diagnostics: menu item](https://dt-cdn.net/images/diagnostics-menu-item2-950-0243641987.png)

     OneAgent diagnostics: пункт меню
   * Уровень процесса: на странице хоста откройте страницу процесса, затем выберите **Run OneAgent diagnostics** из меню (**...**) на странице процесса.
4. На странице **Run Dynatrace OneAgent diagnostics** кратко опишите, что именно работает не так, как ожидается.

   Пример страницы диагностики OneAgent

   ![OneAgent diagnostics: run](https://dt-cdn.net/images/run-diagnostics-1548-2f9988d21a.png)

   OneAgent diagnostics: запуск
5. Необязательно. По умолчанию для анализа собираются данные за последние 24 часа (1 день). Если нужно больше данных, нажмите ссылку **Advanced options**, измените количество дней и нажмите **Apply**.
6. Нажмите **Start analysis**.

### Что происходит далее

Dynatrace выполняет следующее:

* Собирает диагностические данные за последние 24 часа (если значение по умолчанию не изменялось) для затронутого хоста или процесса
* Сохраняет собранные диагностические данные
* Загружает диагностические данные в корзину S3 в регионе AWS вашего окружения для дальнейшего анализа

Столбец **State** отражает текущую фазу процесса.

**State** не обновляется автоматически. Нажмите **Refresh**, чтобы проверить изменение статуса.

Collecting

Сбор данных выполняется.
Во время сбора данных доступны следующие действия:

* **Refresh** для обновления хода выполнения.
* **Cancel** для отмены сбора диагностических данных.

Collected

Dynatrace завершил сбор диагностических данных.
После сбора данных доступны следующие действия:

* **Analyze** для передачи собранных данных в Dynatrace на анализ.
* **Download** для локальной загрузки собранных данных.
* **Delete** для удаления задачи вместе с собранными диагностическими данными.

Sending in progress

Диагностические данные передаются в Dynatrace для анализа.
Во время передачи данных доступны следующие действия:

* **Refresh** для обновления хода выполнения.
* **Download** для загрузки собранных диагностических данных.
* **Delete** для удаления задачи вместе с собранными диагностическими данными.

Sent to Dynatrace cloud

Диагностические данные переданы в Dynatrace для анализа.

Analyzing

Dynatrace выполняет анализ диагностических данных.
Во время анализа данных доступны следующие действия:

* **Refresh** для обновления хода выполнения.
* **Download** для загрузки собранных диагностических данных.
* **Delete** для удаления задачи вместе с собранными диагностическими данными.

Analyzed

Анализ завершён. Количество связанных оповещений указано в скобках.
После анализа доступны следующие действия:

* **Download** для загрузки собранных диагностических данных.
* **Delete** для удаления задачи вместе с собранными диагностическими данными.

Delete in progress

Диагностические данные удаляются. Во время удаления доступно следующее действие:

* **Refresh** для обновления хода выполнения.

Deleted

Диагностические данные удалены. Dynatrace сохраняет только минимальный набор сведений о том, кто, когда, где и зачем собирал диагностические данные.

Canceled

Процесс диагностики был отменён вручную до его завершения.

По завершении анализа Dynatrace отправляет результаты обратно в ваше окружение. Если выявлено возможное решение, Dynatrace отображает его в разделе **Alerts section.**

## Сбор и локальная проверка данных

Данная процедура описывает, как собирать диагностические данные для хоста или процесса локально. Используйте этот вариант, если нужно сначала собрать и проверить данные перед тем, как вручную передать их в Dynatrace для анализа.

Если нужно собрать данные и автоматически передать их в Dynatrace для анализа, см. раздел [Автоматический анализ](#analyze-automatically).

1. Перейдите в **Hosts**.
2. Выберите хост, для которого нужно выполнить диагностику.
3. Запустите диагностику на уровне хоста или процесса.

   * Уровень хоста: на странице хоста откройте меню (**...**) и выберите **Run OneAgent diagnostics**.
     Пример

     ![OneAgent diagnostics: menu item](https://dt-cdn.net/images/diagnostics-menu-item2-950-0243641987.png)

     OneAgent diagnostics: пункт меню
   * Уровень процесса: на странице хоста откройте страницу процесса, затем выберите **Run OneAgent diagnostics** из меню (**...**) на странице процесса.
4. На странице **Run Dynatrace OneAgent diagnostics** кратко опишите, что именно работает не так, как ожидается.

   Пример страницы диагностики OneAgent

   ![OneAgent diagnostics: run](https://dt-cdn.net/images/run-diagnostics-1548-2f9988d21a.png)

   OneAgent diagnostics: запуск
5. Нажмите ссылку **Advanced options**.
6. Выберите **and store locally**.

   * Здесь же можно изменить количество дней, за которые собираются данные (по умолчанию: `1 day`).
7. Нажмите **Apply**.
8. Нажмите **Start collection** для сбора диагностических данных и их хранения локально.

### Что происходит далее

Теперь Dynatrace:

* Собирает диагностические данные за последние 24 часа (если значение по умолчанию не изменялось) для затронутого хоста или процесса
* Сохраняет собранные диагностические данные

Столбец **State** отражает текущую фазу процесса.

**State** не обновляется автоматически. Нажмите **Refresh**, чтобы проверить изменение статуса.

Collecting

Сбор данных выполняется.
Во время сбора данных доступны следующие действия:

* **Refresh** для обновления хода выполнения.
* **Cancel** для отмены сбора диагностических данных.

Collected

Dynatrace завершил сбор диагностических данных.
После сбора данных доступны следующие действия:

* **Analyze** для передачи собранных данных в Dynatrace на анализ.
* **Download** для локальной загрузки собранных данных.
* **Delete** для удаления задачи вместе с собранными диагностическими данными.

### Что делать с собранными данными

После сбора данных доступны следующие действия:

* **Download** для загрузки собранных данных.

  + Просмотрите данные. Обзор содержимого загружаемого архива приведён в разделе [Содержимое диагностических данных](#contents).
  + Добавьте данные к вашему обращению в службу поддержки.
* **Analyze** для анализа данных.
* **Delete** для удаления задачи вместе с собранными диагностическими данными.

### Устранение неполадок OneAgent в изолированных окружениях Dynatrace Managed

В изолированном окружении Dynatrace Managed:

1. Используйте параметр **Store locally** в разделе **Advanced options**, как описано выше.
2. После сбора диагностических данных добавьте их к вашему обращению в службу поддержки.
3. Затем Dynatrace извлечёт диагностические данные из вашего обращения, проанализирует их и предоставит Dynatrace Support автоматизированный отчёт об обнаруженных аномалиях.

На протяжении всего процесса применяются и журналируются строгие средства защиты конфиденциальности данных.

## Обзор диагностики OneAgent

Для просмотра всех запусков диагностики OneAgent в вашем окружении:

1. Перейдите в **Hosts**.
2. Откройте меню (**...**) и выберите **OneAgent diagnostics overview**.

Страница **Dynatrace OneAgent diagnostics overview** отображает всю диагностическую активность OneAgent в вашем окружении. Разверните любую запись для просмотра подробностей. Если данные не были удалены (все диагностические данные удаляются автоматически через 30 дней), их можно загрузить или удалить.

## Содержимое диагностических данных

Все собранные диагностические данные сжимаются в ZIP-файл `SupportArchive<ID number>`, который включает следующие папки и файлы:

| Папка или файл | Описание |
| --- | --- |
| `host` (папка) | Содержит снимок топологической информации об объекте хоста, включая связи с другими хостами. |
| `monitored_entities` (папка) | Содержит снимок топологической информации обо всех задействованных группах процессов, экземплярах групп процессов, службах и экземплярах служб. |
| `agent_registration_entries` (JSON) | Содержит информацию о том, какие кодовые модули OneAgent подключены к Dynatrace. |
| `archive` (JSON) | Содержит информацию о том, кто, когда, где и зачем собирал диагностические данные. |
| `monitoring_state` (JSON) | Содержит информацию о состоянии мониторинга процессов и связанных проблемах. |
| `support_archive` (ZIP) | Содержит локальную конфигурацию OneAgent, установленного на хосте или в процессе, для которых выполнялась диагностика, а также связанные с OneAgent файлы журналов. |
| `diagnostic_files` ZIP | Содержит информацию об обнаружении групп процессов, проблемах автоматической инъекции и конфигурации расширений. |

## Конфиденциальность данных

В целях соответствия региональным требованиям по защите данных и конфиденциальности Dynatrace выполняет следующее:

* Маскирует часть персональных данных перед сохранением архива поддержки в Cassandra и загрузкой его в корзину AWS S3. Например, IBAN и учётные данные URI заменяются на `<masked>`. При этом часть персональных данных может не быть замаскирована.
* Записывает сообщения журнала аудита при создании, анализе, обращении и удалении архивов поддержки для обеспечения прозрачности их использования.
* Предоставляет доступ к архивам поддержки OneAgent только пользователям с [разрешением окружения](/managed/manage/identity-access-management/permission-management/role-based-permissions#environment "Разрешения на основе ролей") **View sensitive request data**.
* Автоматически удаляет все диагностические данные через 30 дней после их сбора. Этот [период хранения данных](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#diagnostics "Проверьте сроки хранения различных типов данных.") настраивается.

  Это относится к данным на кластере Dynatrace. Собранные диагностические данные также можно удалить раньше срока.

Дополнительные сведения, связанные с диагностикой OneAgent, см. на следующих страницах:

* [Персональные данные, собираемые Dynatrace](/managed/manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace#oneagent-diagnostics "Узнайте, какие типы данных конечных пользователей могут собираться при мониторинге Dynatrace и какие методы маскирования персональных данных доступны.")
* [Периоды хранения данных](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#diagnostics "Проверьте сроки хранения различных типов данных.")

## Устранение неполадок

Статус: `Collecting of the diagnostic data wasn't possible within 20 minutes.`

* Если Dynatrace не может собрать диагностические данные в течение 20 минут, процесс автоматически повторяется.
* Если повторная попытка также завершается неудачно, рекомендуем обратиться к специалисту по продуктам Dynatrace через чат.

Статус `State` кажется застывшим.

**State** не обновляется автоматически. Нажмите **Refresh**, чтобы проверить изменение статуса.

## Вопросы и ответы

Можно ли получить прямой доступ к S3 или использовать собственную корзину S3?

Нет, прямой доступ к S3 и использование собственной корзины не поддерживаются.

Диагностика OneAgent загружает диагностические данные в корзину S3 Dynatrace, настроенную для окружения/кластера компанией Dynatrace. Используемая корзина S3 зависит от местоположения окружения/кластера.
