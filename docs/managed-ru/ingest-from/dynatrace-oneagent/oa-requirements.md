## Требования к CPU OneAgent для архитектуры x86-64

В этом разделе описаны требования к CPU для запуска Dynatrace OneAgent. Соответствие системы этим требованиям необходимо для надёжной и эффективной работы.

### Минимальный поддерживаемый уровень микроархитектуры: **x86-64-v2**

Бинарные артефакты Dynatrace OneAgent зависят от поддержки инструкций, входящих в уровень микроархитектуры x86-64-v2.

Версия Dynatrace OneAgent, в которой это будет применено, будет объявлена в будущем выпуске.

#### Intel

Nehalem была первой архитектурой Intel, полностью поддерживающей базовый уровень x86-64-v2. Для получения дополнительной информации обратитесь к [спецификации продуктов Intel](https://www.intel.com/content/www/us/en/products/overview.html).

**Примеры процессоров Intel:**

* **Intel Core i7-920** — Desktop (LGA 1366)
* **Intel Xeon X5550** — Server (LGA 1366)
* **Intel Core i5-750** — Mainstream desktop (LGA 1156)
* **Intel Xeon L5520** — Low-power server CPU

#### AMD

Bulldozer и его преемники поддерживают базовый уровень x86-64-v2. Для некоторых CPU поддержка была добавлена в определённом степпинге. Для получения дополнительной информации обратитесь к [спецификации продуктов AMD](https://www.amd.com/en/products/specifications.html).

**Примеры процессоров AMD:**

* **AMD FX-4100** — Desktop (AM3+)
* **AMD Opteron 6200 Series** — Server (Socket G34)
* **AMD A8-3870K** — APU (FM1)
* **AMD FX-8350** — High-performance desktop (AM3+)

* **x86-64-v2** — базовый уровень, включающий поддержку инструкций SSE3, SSSE3, SSE4.1, SSE4.2, POPCNT и CMPXCHG16B.
* Системы, не соответствующие базовому уровню x86-64-v2, не смогут запускать Dynatrace OneAgent.
* В виртуализированных окружениях необходимо убедиться, что сквозная передача функций CPU включена для соответствия требованиям к микроархитектуре.

## Требования к памяти OneAgent

Глубокий мониторинг приложения с помощью Dynatrace OneAgent влечёт увеличение потребления памяти на уровне каждого приложения по сравнению с выполнением без Dynatrace OneAgent. Помимо памяти, необходимой для загрузки бинарного кода кодового модуля OneAgent в процесс приложения, память также используется для хранения информации о состоянии мониторируемого приложения, коммуникационных буферов и т. д.

Потребление памяти не является постоянным числом или пропорцией от требований к памяти приложения, а зависит от технологии, конфигурации мониторинга, свойств приложения и выполняемой нагрузки. Подробнее смотрите в разделе [О вариабельности потребления памяти](#about-memory-demand-variance) ниже.

### Требование к памяти для кодового модуля OneAgent

Как описано выше, потребление памяти при мониторинге зависит от множества факторов. Для упрощения планирования ресурсов рекомендуем закладывать дополнительный бюджет памяти в **200 МБ** для мониторируемых процессов приложения. Этого достаточно для мониторинга огромного числа приложений. Эмпирические наблюдения показывают, что для большинства приложений потребление памяти значительно ниже **100 МБ**.

Потребление памяти при мониторинге относится к размеру резидентного набора (RSS) или эквивалентному показателю на операционных системах, отличных от Linux. RSS является ключевым показателем для применения ограничений памяти к процессам.

#### Ограничения памяти на облачных платформах

Kubernetes и другие облачные платформы предусматривают определение ограничений памяти для рабочих нагрузок. Заданные ограничения применяются (примерно) к RSS, и рабочие нагрузки автоматически завершаются при превышении заданного ограничения.

Поскольку кодовые модули Dynatrace OneAgent увеличивают потребление памяти мониторируемых приложений, ограничения памяти необходимо корректировать соответствующим образом.

### О вариабельности потребления памяти

Потребление памяти при глубоком мониторинге кодовым модулем OneAgent не может быть точно выражено постоянным числом или пропорцией памяти, потребляемой процессом приложения. Это сумма памяти, необходимой для базовой работы кодового модуля OneAgent (например, коммуникационных буферов), и динамических данных мониторинга, собираемых кодовыми модулями OneAgent. Потребление памяти для динамических данных мониторинга зависит от настроек конфигурации, базовой технологии приложения и самого приложения.

#### Зависящее от приложения потребление памяти

Видимость на уровне кода и анализ горячих точек требуют записи времени и частоты выполнения функций. Поэтому количество функций в приложении и их выполнение определяют количество элементов данных и, в конечном счёте, объём памяти, необходимый для измерения производительности функций. То же самое относится к информации о распределённых трассировках — потребление памяти для сбора информации о распределённых трассировках зависит от количества одновременных запросов, обрабатываемых приложением, и сложности (то есть длины PurePath) выполнений, инициируемых этими запросами.

#### Зависящее от конфигурации потребление памяти

Мониторинг пользовательских сервисов является примером конфигурационно-зависимого потребления памяти. Определение пользовательского сервиса увеличивает базовый бюджет памяти для выбранной инструментации функций и динамический бюджет памяти для расширенного сбора данных о распределённых трассировках для вызовов пользовательского сервиса. В технологии .NET инструментирование дополнительных сборок для пользовательского сервиса может существенно увеличить потребление памяти при запуске (смотрите ниже).

#### Зависящее от технологии потребление памяти

Кодовые модули OneAgent оптимизированы для эффективного использования памяти и освобождения ресурсов, когда они больше не нужны, чтобы как можно меньше влиять на выполнение приложения. Поэтому потребление памяти может варьироваться в процессе выполнения приложения.

В зависимости от кодового модуля OneAgent потребление памяти может достигать пика при запуске приложения. Это особенно характерно для технологии .NET. Подготовка .NET-сборок для мониторинга вызывает пик потребления памяти, поскольку код сборки временно находится в памяти в двух экземплярах. После завершения процесса внедрения среда выполнения .NET сохраняет в памяти как исходные сборки, так и инструментированные версии логики приложения. Это известная проблема технологии Microsoft .NET, которую Dynatrace OneAgent не может устранить.
