## Атрибуты ресурса на уровне хоста

Все данные мониторинга, покидающие определённый хост, то есть имеющие этот хост в качестве ресурса, обогащаются атрибутами ресурса на уровне хоста.

Атрибуты ресурса на уровне хоста, это атрибуты ресурса отслеживаемых хостов. Все события, поступающие от компонентов OneAgent, работающих на определённом хосте, и измерения, поступающие от них, обогащаются этими атрибутами. Затем их можно использовать в запросах для структурирования и фильтрации данных мониторинга.

Некоторые атрибуты также можно использовать для создания политик управления доступом к данным. См. [Недоступно в Dynatrace Managed](/managed/upgrade/unavailable-in-managed "Ваш выбор недоступен в Dynatrace Managed.") и найдите поля с тегом Permission

Если есть доступ к хосту с установленным OneAgent, можно посмотреть `dt_host_metadata.json` и `dt_host_metadata.properties`, чтобы увидеть область обогащения атрибутами ресурса, предоставляемую OneAgent. Подробнее см. [Обогащение поступающих данных специфичными для Dynatrace полями](/managed/ingest-from/extend-dynatrace/extend-data "Узнайте, как автоматически обогащать данные телеметрии специфичными для Dynatrace полями.").

### Пользовательские атрибуты уровня хоста

Можно создавать собственные атрибуты, настраивая теги ключ-значение и свойства, заданные через [oneagentctl](/managed/observe/infrastructure-observability/hosts/configuration/define-tags-and-metadata-for-hosts "Узнайте, как тегировать и задавать дополнительные свойства для отслеживаемого хоста."), либо через [удалённое управление конфигурацией OneAgent и ActiveGate](/managed/ingest-from/bulk-configuration "Выполняйте настройку OneAgent и ActiveGate на хостах со страницы Deployment status или массово с помощью Dynatrace API."). Пользовательские теги и свойства, заданные таким образом, передаются как плоские атрибуты ресурса первого уровня.

Теги-ключи без значения игнорируются.

Теги, назначенные через [автоматизированные правила](/managed/manage/tags-and-metadata/setup/how-to-define-tags#automatic "Узнайте, как определять и применять теги вручную и автоматически."), [переменные окружения](/managed/manage/tags-and-metadata/setup/define-tags-based-on-environment-variables "Узнайте, как Dynatrace позволяет определять теги на основе переменных окружения.") и [Topology and Smartscape API](/managed/dynatrace-api/environment-api/topology-and-smartscape "Узнайте о Dynatrace Topology and Smartscape API."), не включаются.

### Общие атрибуты уровня хоста

| Атрибут | Описание |
| --- | --- |
| Пользовательские теги хоста | См. [Пользовательские атрибуты](#custom) |
| `dt.host_group.id` | Обратите внимание на разницу между `dt.entity.host_group` и `dt.host_group.id`:  * `dt.host_group.id`, это имя [группы хостов](/managed/observe/infrastructure-observability/hosts/configuration/organize-your-environment-using-host-groups "Узнайте, как Dynatrace позволяет организовывать хосты, процессы и сервисы с помощью групп хостов."). Это удобочитаемый идентификатор группы хостов, например `MyHostGroup`. * `dt.entity.host_group`, это идентификатор сущности группы хостов в модели Dynatrace. Идентификатор сущности не совпадает с удобочитаемым идентификатором группы хостов. Он создаётся автоматически, и его формат, например, `HOST_GROUP-008786529C8EE446`.  Подробнее см. [Организация окружения с помощью групп хостов](/managed/observe/infrastructure-observability/hosts/configuration/organize-your-environment-using-host-groups "Узнайте, как Dynatrace позволяет организовывать хосты, процессы и сервисы с помощью групп хостов."). |
| `dt.entity.host` | ID сущности типа HOST. |
| `dt.entity.host_group` | ID сущности типа `HOST_GROUP`. |
| `host.name` | Имя хоста, определяемое в зависимости от источника данных, например OneAgent, Extensions или OpenTelemetry. Имя хоста может изменяться на основе правил именования.  Обратите внимание, что это не ID сущности типа `HOST` (`dt.entity.host`). |

### Нормализация атрибутов ресурса

Для обеспечения согласованного и надёжного приёма метрик ко всем соответствующим внутренним ключам и значениям метрик применяется нормализация атрибутов ресурса. Этот процесс помогает предотвратить отбрасывание метрик из-за недопустимых или некорректно сформированных измерений (dimensions).

#### Правила для ключей измерений

| Описание правила | Подробности |
| --- | --- |
| **Недопустимые символы** | Недопустимый символ или серия недопустимых символов заменяется одним символом подчёркивания `_`. Например, `zaó$%ć` заменяется на `za_`. |
| **Пустые ключи** | Измерения без допустимых символов пропускаются |
| **Ограничение длины ключа** | OneAgent версии 1.317+ Макс. 350 символов (ранее макс. 100 символов) |

#### Правила для значений измерений

| Описание правила | Подробности |
| --- | --- |
| **Допустимые символы** | Все неуправляющие символы (ASCII и Unicode) |
| **Управляющие символы** | Не допускаются. Управляющий символ (то есть символ, используемый как инструкция и не отображаемый; например, перенос строки, табуляция) или серия таких символов заменяется одним символом подчёркивания `_`. |
| **Ограничение длины значения** | OneAgent версии 1.313+ Макс. 2048 символов (ранее макс. 255 символов) |
| **Значения в кавычках** | Если значение начинается и заканчивается на `"`, оно экранируется |

#### Ограничения измерений

Для соответствия текущей спецификации используется определённая иерархия измерений и заданные лимиты, чтобы предотвратить предупреждения и отбрасывание метрик из-за превышения этих лимитов.

По умолчанию глобальный лимит измерений равен `100`, а лимит измерений, определяемых заказчиком, составляет 40% от глобального лимита.
