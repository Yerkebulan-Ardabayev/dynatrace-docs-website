# Топ веб-запросов

* How-to guide
* 4-min read
* Updated on Sep 13, 2022

Гибкое представление **Top web requests** позволяет анализировать топ веб-запросов по всем вашим сервисам и искать запросы, исходящие с определённых URL-адресов или направленные к ним. Это представление легко настраивается и служит удобной точкой входа для углублённого анализа ваших сервисов.

Чтобы перейти на страницу анализа **Top web requests**

1. Перейдите в **Multidimensional Analysis**.
2. Нажмите плитку **Top web requests**.

![Страница топ веб-запросов](https://dt-cdn.net/images/top-web-requests-page-3518-5c4cc7fb02.png)

Страница топ веб-запросов

## Настройка представления

На странице **Top web requests** перечислены все веб-запросы, выполненные за выбранный период и в пределах выбранной зоны управления. В разделе **Configure view** можно настроить несколько возможностей фильтрации. Представление обновляется автоматически при изменении параметров.

| Параметр | Описание |
| --- | --- |
| Metric | Метрика для анализа — по умолчанию установлено **Request count**. |
| Aggregation | Способ агрегации значений метрики — доступные агрегации зависят от выбранной метрики. |
| Split mode | Способ обработки измерений, указанных в **Split by dimension**.  * **Split by services** — каждое измерение отображается отдельно для каждого сервиса. * **Merge by services** — одинаковые измерения из разных сервисов объединяются в одно. |
| Split by dimension | Список измерений, по которым разбиваются запросы. По умолчанию установлено измерение имени запроса (`{Request:Name}`).  Можно указать несколько измерений. Поместите курсор в поле ввода, чтобы увидеть доступные параметры. Запросы разбиваются по измерениям в указанном порядке. |
| Filter requests | Фильтрация запросов для включения в представление. По умолчанию установлены следующие критерии:  * **Service type**: `Web service` * **Service type**: `Web request service`  Можно добавить дополнительные критерии. Поместите курсор в поле ввода, чтобы увидеть доступные параметры.  Критерии одного типа объединяются по логике OR. Критерии разных типов объединяются по логике AND. |

После настройки представление можно сохранить для быстрого доступа в будущем. Просто выберите **Save view** и укажите имя.

Также можно сохранить конфигурацию как вычисляемую метрику сервиса и использовать её как любую другую метрику в Dynatrace (например, для [оповещений](/managed/dynatrace-intelligence/anomaly-detection/metric-events "Узнайте о событиях метрик в Dynatrace")).

Данные таблицы можно экспортировать в файл CSV.

1. В правом нижнем углу страницы выберите **Show export menu** ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More").

   ![Меню Show export](https://dt-cdn.net/images/show-export-menu-107-2a8a76c9a2.png)

   Меню Show export
2. Выберите **Export visible data** или **Export table data**.

   | Параметр | Экспортируемые данные | Поля | Количество записей |
   | --- | --- | --- | --- |
   | **Export visible data** | Текущая видимая область таблицы с учётом применённых фильтров | Только видимые данные | До 100 топовых измерений |
   | **Export table data** | Все данные таблицы | Все доступные данные, связанные с топовыми измерениями | До 100 топовых измерений |

## Представление

* На диаграмме страницы **Top web requests** показаны 15 топовых измерений (все остальные измерения агрегируются в одно), а таблица ниже содержит до 85 дополнительных измерений, что в сумме составляет до 100 отображаемых на диаграмме измерений. Представление мгновенно адаптируется к изменениям в панели **Configure view**.
* Диаграмма использует [данные трассировок и запросов](/managed/observe/application-observability/multidimensional-analysis#data-source "Настройте представление многомерного анализа и сохраните его как вычисляемую метрику.") с разными периодами хранения. Для временных периодов, содержащих данные старше 10 дней, можно включить переключатель **Show data retention**, чтобы лучше понять, какие данные доступны за какой период непосредственно на диаграмме.
* В столбце **Actions** таблицы можно выбрать:

  + **Filter** ![Filter](https://dt-cdn.net/images/filter-icon-41ddd02d66.svg "Filter") — для фильтрации представления по указанному измерению.
  + **More** (**…**) — для доступа к дополнительным параметрам анализа из [меню **Analyze**](/managed/observe/application-observability/services-classic/context-specific-drill-down#analyze-menu "Узнайте об удобной навигации и фильтрации для анализа сервисов.").

## Примеры использования

Ниже приведены некоторые сценарии использования представления **Top web requests**.

### Поиск конкретного URL

С помощью представления **Top web request** можно сосредоточить анализ на запросах, поступающих с определённого URL-адреса или направленных к нему.

1. Укажите `{URL:Host}{Relative-URL}` в поле **Split by dimension**, чтобы разбить запросы по URL-адресам.
2. Теперь отфильтруйте запросы по конкретному URL. В поле **Filter requests** выберите тип фильтра `Web request URL` и укажите шаблон URL. Обратите внимание, что этот фильтр использует оператор **CONTAINS**.

### Поиск запросов с кодами ответа об ошибке

Можно легко сузить список запросов до тех, которые вернули код ответа об ошибке (то есть любой код ошибки в диапазоне **4xx**). После настройки интересующих измерений поместите курсор в поле **Filter requests**, выберите тип фильтра **HTTP response code** (можно начать вводить текст, чтобы сузить предложения). Затем выберите критерий **4xx** и дождитесь завершения обработки данных.

### Поиск HTTP-запросов POST

Существует два способа быстро просмотреть запросы POST (или любой другой HTTP-метод): (1) задать измерение HTTP-метода или (2) отфильтровать запросы по HTTP-методу.

#### Измерение

В этом случае все запросы будут сгруппированы по HTTP-методу и по любому другому указанному измерению. Просто укажите `{HTTP-Method}` в поле **Split by dimension**.

#### Фильтр

В этом случае все запросы, не являющиеся POST, отфильтровываются, и можно сосредоточить анализ. Просто укажите `HTTP method` и выберите значение `POST` в поле **Filter requests**.
