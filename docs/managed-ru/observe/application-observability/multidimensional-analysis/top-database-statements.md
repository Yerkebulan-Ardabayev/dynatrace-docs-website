# Топ операторов БД

# Топ операторов БД

* How-to guide
* 4-min read
* Updated on Sep 13, 2022

Представление **Top database statements** позволяет понять общую активность базы данных, а не только активность одной базы данных. Это представление также можно использовать для более детального анализа сквозной активности базы данных за конкретные временные периоды.

Чтобы перейти на страницу **Top database statements**

1. Перейдите в **Multidimensional Analysis** или **Database Services**.
2. Выберите **Top database statements**.

## Анализ операторов БД

На странице **Top database statements** отображается диаграмма с топ SQL- и NoSQL-операторами в вашем окружении за определённый период времени. Диаграмма также выделяет несколько топовых операторов по количеству выполнений. В таблице после диаграммы перечислены 100 **топ измерений**.

![Страница топ операторов БД](https://dt-cdn.net/images/top-database-statements-3564-3bd08ea1b1.png)

Страница топ операторов БД

Настроить представление можно, изменив **Metric** и другие параметры или добавив фильтр.

* Различные метрики, такие как **Fetch count**, **Response time** и **Row count**, дают разный порядок операторов. В зависимости от времени выполнения на оператор, частоты выполнения оператора или частоты ошибок страница сначала отображает операторы, которые выполняются чаще или требуют больше ресурсов — от наибольшего значения выбранной метрики к наименьшему.
* Для сужения списка операторов БД можно использовать **Filter requests**.

Примеры фильтрации запросов

В следующем примере показано представление, настроенное с метрикой **Response time** с фильтрацией по диапазону значений `≥100ms`. В списке отображаются все операторы БД, время отклика которых составляло не менее 100 мс.

![Фильтр по времени отклика в анализе топ операторов БД](https://dt-cdn.net/images/response-time-filter-in-top-database-statement-analysis-3684-0962b8fba9.png)

Фильтр по времени отклика в анализе топ операторов БД

Представление в этом примере настроено с метрикой **Fetch count** с фильтрацией по диапазону значений `≥5`. В списке отображаются операторы БД, выполняющие не менее 5 выборок за одно выполнение.

![Fetch count в анализе топ операторов БД](https://dt-cdn.net/images/fetch-count-in-top-database-statements-analysis-3548-bdb184cbd0.png)

* Данные таблицы можно экспортировать в файл CSV.

  1. В правом нижнем углу страницы выберите **Show export menu** ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More").

     ![Меню Show export](https://dt-cdn.net/images/show-export-menu-107-2a8a76c9a2.png)

     Меню Show export
  2. Выберите **Export visible data** или **Export table data**.

     | Параметр | Экспортируемые данные | Поля | Количество записей |
     | --- | --- | --- | --- |
     | **Export visible data** | Текущая видимая область таблицы с учётом применённых фильтров | Только видимые данные | До 100 топовых измерений |
     | **Export table data** | Все данные таблицы | Все доступные данные, связанные с топовыми измерениями | До 100 топовых измерений |
* Диаграмма использует [данные трассировок и запросов](/managed/observe/application-observability/multidimensional-analysis#data-source "Настройте представление многомерного анализа и сохраните его как вычисляемую метрику.") с разными периодами хранения. Для временных периодов, содержащих данные старше 10 дней, можно включить переключатель **Show data retention**, чтобы лучше понять, какие данные доступны за какой период непосредственно на диаграмме.

## Анализ отдельных SQL-операторов

Чтобы получить доступ к нескольким представлениям анализа для конкретных операторов, включая **Service backtrace** и анализ **Outliers**, выберите **More** (**…**).

### Детали оператора БД

Перейти к анализу отдельных операторов можно напрямую, выбрав **Filter**. Либо выберите **More** (**…**) > **Statement details**.

![Распределённые трассировки отдельного оператора БД](https://dt-cdn.net/images/distributed-traces-top-database-statements-1593-b6fb81c79e.png)

Распределённые трассировки отдельного оператора БД

Выберите оператор, чтобы просмотреть все вариации выбранного запроса. Dynatrace агрегирует операции вставки, обновления и удаления в разбивке по таблицам.

![Детали отдельного оператора БД](https://dt-cdn.net/images/details-individual-statements-1564-56ba5e5697.png)

Детали отдельного оператора БД

### Обратная трассировка сервиса БД

Чтобы перейти к анализу обратной трассировки для выбранного оператора, выберите **Service backtrace** из меню **Analyze**. Страница **Backtrace** предоставляет сведения о пользовательском взаимодействии, инициировавшем выбранную вариацию SQL-оператора, а также о пользовательском взаимодействии или запросе сервиса, породившем этот оператор. Выберите любой из входящих запросов для визуализации обратной трассировки.

![Обратная трассировка для топ операторов БД](https://dt-cdn.net/images/backtrace-for-top-database-statements-1699-76085f0963.png)

Обратная трассировка для топ операторов БД

### Распределение времени отклика

Чтобы понять распределение времени отклика для команды, выберите **Outliers** из меню **Analyze**. На странице **Response time distribution** показано количество запросов, попавших в различные диапазоны времени отклика. Выберите любой столбец на диаграмме для анализа конкретных запросов, попавших в данный диапазон времени отклика.

![Распределение времени отклика запросов сервиса](https://dt-cdn.net/images/response-time-distribution-database-statements-1800-8ff28dc6c7.png)

Распределение времени отклика запросов сервиса

### Просмотр отдельных выполнений

Чтобы просмотреть отдельные выполнения каждой команды, выберите **More** (**…**) > **Distributed traces** > **Trace**.

В столбце **Fetches** отображается количество обращений к базе данных за одно выполнение SQL. Поскольку большинство драйверов баз данных разбивают наборы результатов на страницы, обращения обычно выполняются наборами строк **Rows returned** за одно обращение. Таким образом, если результирующий набор содержит 200 строк, а размер выборки составляет 50 строк за обращение, для получения всех данных потребуется 4 обращения. Больше обращений означает более медленное время отклика. Однако слишком большой размер выборки также может привести к неэффективному использованию памяти, и в большинстве случаев весь набор данных всё равно не читается целиком.

Выберите любое выполнение в списке для перехода к его распределённой трассировке.

![Трассировка PurePath® выполнения](https://dt-cdn.net/images/purepath-database-statements-1597-d6db847c88.png)

Трассировка PurePath® выполнения

## Пример: понимание SQL для потока сервисов

Хотя анализ SQL доступен в нескольких местах рабочего процесса анализа сервисов, он добавлен как основное представление анализа для баз данных в представлении [**Service flow**](/managed/observe/application-observability/services-classic/service-flow "Узнайте, как Dynatrace помогает отслеживать последовательность вызовов сервисов, инициируемых каждым запросом в вашем окружении."). В следующем примере показан поток сервисов, уже отфильтрованный для фокусировки на конкретной цепочке вызовов. Видно, что всякий раз, когда сервис **Customer Frontend** вызывает **JourneyService**, **58%** от **вклада в время отклика** приходится на сервис SQL **EasyTravel**.

![От потока сервисов к оператору БД](https://dt-cdn.net/images/from-service-flow-to-database-statement-1748-bff9188259.png)

От потока сервисов к оператору БД

Действие по умолчанию изменилось и теперь отображает **View database statements**.

Можно выполнить несколько типов анализа:

1. Чтобы просмотреть операторы БД, выполненные в рамках выбранного потока за период анализа, выберите **View database statements**. Это очень полезно, так как показывает, почему база данных вносит определённый вклад в общее время.
2. Чтобы выявить операторы с наибольшим суммарным временем отклика, отсортируйте список **Top dimensions** по **Count/min**.

![Пример многомерного анализа сервиса SQL](https://dt-cdn.net/images/example-of-sql-service-multidimensional-analysis-3574-7c039d0068.png)

Пример многомерного анализа сервиса SQL

3. Выберите **More** (**…**), затем:

* **Outliers** — для просмотра распределения времени отклика.
* **Distributed traces** — для понимания динамики каждого отдельного SQL-оператора со временем, а также среднего количества **Rows returned**.
* **Service backtrace** — для просмотра того, какие действия пользователей привели к наиболее медленным выполнениям оператора.
