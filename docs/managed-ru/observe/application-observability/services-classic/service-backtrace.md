# Обратная трассировка сервиса

* How-to guide
* 4-min read
* Published Jul 19, 2017

Помимо знания о том, какой сервис непосредственно вызывает ваш сервис, полезно знать последовательность вызовов сервисов, ведущую к каждому запросу, — вплоть до нажатия в браузере, инициировавшего эту последовательность вызовов. **Service-level backtrace** в Dynatrace позволяет просматривать такие цепочки вызовов.

Предположим, вы анализируете сервис базы данных SQL и хотите понять последовательность предшествующих запросов к сервисам, в итоге инициировавших входящие запросы к SQL-сервису. С помощью обратной трассировки на уровне сервисов можно, например, выяснить, что сервис базы данных SQL вызывается `Service1`, `Service1` вызывается `Service2`, который, в свою очередь, был инициирован нажатием кнопки входа в систему.

Чтобы просмотреть обратную трассировку на уровне сервисов

1. Перейдите в **Services**.
2. Выберите сервис для анализа.
3. На странице обзора сервиса в разделе **Understand dependencies** выберите **Analyze backtrace** для просмотра обратной трассировки запросов к данному сервису.

Обратите внимание, что в данном примере большинство вызовов поступает от веб-приложения `easyTravel`, и вызовы проходят по конкретной цепочке сервисов (`EasyTravelWebServer` вызывает `easyTravel Customer Frontend`, который вызывает `CouchDB_ET`). Выберите любой элемент цепочки, чтобы увидеть имена отдельных запросов. На изображении ниже все запросы к `CouchDB_ET`, поступающие из приложения `easytravel.com`, приходят из детального списка действий пользователя.

![Действия пользователя в обратной трассировке на уровне сервисов](https://dt-cdn.net/images/user-actions-service-level-backtrace-1606-34253e59f0.png)

При выборе сервиса или приложения в иерархии обратной трассировки доступен дальнейший анализ этого сервиса или приложения в двух отдельных разделах.

![Обратная трассировка на уровне сервисов](https://dt-cdn.net/images/service-level-backtrace-1598-04ecfb12fc.png)

Обратная трассировка на уровне сервисов

В первом разделе для приложения можно просмотреть список [действий пользователя](/managed/observe/digital-experience/rum-concepts/user-actions "Узнайте, что такое действия пользователя и как они помогают понять, что пользователи делают с вашим приложением.") данного конкретного приложения за [выбранный временной период](/managed/analyze-explore-automate/dashboards-classic/dashboards/dashboard-timeframe "Узнайте о настройках временного периода и зоны управления на дашборде Dynatrace."). Для сервисов можно просмотреть типы запросов, которые данный конкретный сервис совершал к следующему сервису в потоке обратной трассировки.

Следующий раздел содержит аналитические данные о количестве запросов и любых произошедших сбоях. Дополнительную информацию можно получить, выбрав вкладки:

* **Reasons for failed requests**
  Перечисляет причины, по которым определённые запросы завершились с ошибкой.
* **Stacktrace**
  Показывает, в какой части кода был выполнен запрос.
* **Referring pages**
  Показывает HTTP-реферреры, которые способствовали конкретному потоку обратной трассировки.
* **Proxies**
  Предоставляет сведения о прокси-серверах или балансировщиках нагрузки, через которые прошёл запрос.
* **Analyze**
  Предлагает различные варианты анализа как для выбранного сервиса (в левом столбце), так и для анализируемого сервиса в обратной трассировке (правый столбец).

  + [View service flow](/managed/observe/application-observability/services-classic/service-flow "Узнайте, как Dynatrace помогает отслеживать последовательность вызовов сервисов, инициируемых каждым запросом в вашем окружении.")
  + [View details of failure](/managed/observe/application-observability/services-classic/response-time-distribution-and-outlier-analysis#correlate-errors-with-response-times "Получите представление о распределении времени отклика по всем запросам, включая аномально высокие и аномально низкие.")
  + [View distributed traces](/managed/observe/application-observability/distributed-traces/use-cases/segment-request "Улучшайте производительность распределённой системы, сегментируя запросы с медленным временем отклика через поток сервисов и анализируя их распределённые трассировки.")
  + [View response time](/managed/observe/application-observability/services-classic/service-response-time-hotspots "Определяйте действия, потребляющие наибольшую часть времени отклика для каждого сервиса.")
  + [View outliers](/managed/observe/application-observability/services-classic/response-time-distribution-and-outlier-analysis "Получите представление о распределении времени отклика по всем запросам, включая аномально высокие и аномально низкие.")
  + [View method hotspots](/managed/observe/application-observability/services-classic/service-response-time-hotspots#code-level-visibility "Определяйте действия, потребляющие наибольшую часть времени отклика для каждого сервиса.")
  + [View web requests](/managed/observe/application-observability/multidimensional-analysis/top-web-requests "Узнайте, как анализировать все веб-запросы по всем вашим сервисам с помощью Dynatrace.")

Кроме того, можно отображать только те транзакции, которые содержат текущую цепочку вызовов. Просто выберите [Filter service backtrace](/managed/observe/application-observability/services-classic/service-flow/service-flow-filtering "Поймите, как работает фильтрация сервисов и как ею можно воспользоваться.").

![Фильтрация цепочки обратной трассировки на уровне сервисов](https://dt-cdn.net/images/filter-service-level-backtrace-chain-1589-94fc599a8b.png)

Фильтрация цепочки обратной трассировки на уровне сервисов

Если в обратной трассировке выбран кластеризованный сервис, откройте вкладку **Instances** для просмотра аналогичного анализа для каждого [экземпляра сервиса](/managed/observe/application-observability/services-classic/analyze-individual-service-instances "Узнайте, как выполнить анализ экземпляра сервиса.") (см. изображение выше).

## Примеры анализа обратной трассировки

Ниже приведены примеры использования анализа обратной трассировки:

Анализ влияния ошибок

Функция обратной трассировки становится ещё более полезной при анализе ошибок. Рассмотрим пример ниже. Он показывает **Failure rate** `100%` для запроса `AdsForBlog`.

![Обратная трассировка 6](https://dt-cdn.net/images/backtrace06-1354-edc72085b5.png)

Обратная трассировка 6

Чтобы узнать первопричины обнаруженных сбоев, выберите **Analyze failure rate degradation**. После этого вы увидите сообщение и трассировку стека (см. изображение ниже). Чтобы оценить значимость сбоев и то, повлияли ли они на пользователей, выберите **Analyze backtrace**.

![Обратная трассировка 7](https://dt-cdn.net/images/backtrace07-1587-9980999988.png)

Обратная трассировка 7

Dynatrace показывает, как этот неудачный запрос повлиял на другие сервисы и пользователей. Видно, что все запросы поступали от одного действия пользователя — загрузки блога. Также видно, что запрос `/blog` на веб-сервере не был затронут неудачным запросом `Ads`. Ни один запрос `/blogs` не завершился с ошибкой — ошибка была обработана корректно.

Понимание влияния сторонних сервисов

Dynatrace позволяет видеть, какие сервисы вызывают какие сторонние сервисы, а также какие нажатия в браузере инициируют такие цепочки вызовов. На изображении ниже обратите внимание, что все запросы к `fedex.com` завершаются с ошибкой. Обратная трассировка сервиса показывает, какие сервисы и какие действия пользователя привели к этому вызову стороннего сервиса.
