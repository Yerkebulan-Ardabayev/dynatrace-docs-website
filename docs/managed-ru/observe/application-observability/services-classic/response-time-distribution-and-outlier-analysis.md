# Распределение времени отклика и анализ выбросов

# Распределение времени отклика и анализ выбросов

* How-to guide
* 5-min read
* Published Jul 19, 2017

Для оценки производительности приложений крайне важно отслеживать время отклика каждого запроса в каждой сквозной транзакции, выполняемой приложениями. Dynatrace предоставляет такую функциональность в виде анализа распределения времени отклика и выбросов. Понимая распределение времени отклика по всем запросам, можно сосредоточиться на тех запросах, которые имеют наибольшее время отклика. Выбросы времени отклика (запросы с аномально высоким или аномально низким временем отклика) существенно влияют на общее время отклика транзакций.

## Распределение времени отклика

Чтобы просмотреть диаграмму **Response time** для сервиса

1. Перейдите в **Services**.
2. Выберите сервис для анализа.
3. На странице обзора сервиса выберите кнопку **View** (например, **View requests**, **View dynamic requests** или **View resource requests**).
4. Выберите вкладку **Response time**.

Диаграмма **Response time** показывает, как распределялось время отклика запросов, инициированных данным сервисом, в течение [выбранного временного периода](/managed/analyze-explore-automate/dashboards-classic/dashboards/dashboard-timeframe "Узнайте о настройках временного периода и зоны управления на дашборде Dynatrace.") (см. пример ниже). Эта диаграмма также показывает среднее количество запросов за единицу времени, а также минимальное/максимальное время отклика каждого [экземпляра сервиса](/managed/observe/application-observability/services-classic/analyze-individual-service-instances "Узнайте, как выполнить анализ экземпляра сервиса."). Для анализа времени отклика доступны перцентили `Median`, `Slowest 10%` и `Slowest 5%`.

![Выброс 1](https://dt-cdn.net/images/outlier01-1650-3919f9c0c9.png)

Выброс 1

Наведите курсор на любой временной интервал на диаграмме, чтобы отобразить элементы управления интервалом и конкретные измерения для этого интервала, включая количество **Requests**, **Response time**, **Min/max instance response time** и дату/время.

Для изменения временной шкалы по оси X выберите другой временной интервал из раскрывающегося списка под диаграммой. Или используйте кнопки **+/-** для масштабирования по временному периоду. Стрелки влево/вправо позволяют переходить вперёд и назад по временным интервалам. Эти кнопки видны при наведении курсора на диаграмму.

## Анализ выбросов

Для более глубокого понимания того, как время отклика варьируется между запросами в течение выбранного периода, выберите **Analyze outliers** для перехода на страницу **Response time distribution**.

Выберите любой столбец распределения по перцентилям, чтобы просмотреть количество запросов, попавших в данный диапазон времени отклика. Как видно в разделе **Top web requests** под диаграммой, самые медленные запросы (с наибольшим временем отклика) могут выполняться значительно дольше, чем самые быстрые. Такие выбросы могут существенно влиять на общее время отклика сервиса.

![Выброс 2](https://dt-cdn.net/images/outlier02-1675-11a21ba4a2.png)

Выброс 2

Можно подробнее рассмотреть запросы в каждом выбранном диапазоне времени отклика, выбрав **Zoom in**.

![Выброс 3](https://dt-cdn.net/images/outlier03-1539-4d0ab0a038.png)

Выброс 3

В примере ниже показано, что около 600 запросов имеют время отклика от `70.5 мс` до `439 мс` и что веб-запрос `orange.jsf` вносит наибольший вклад в общее время отклика (491 запрос за период анализа).

![Выброс 4](https://dt-cdn.net/images/outlier04-1539-ca5706f5dd.png)

В примере **Service flow** ниже показано, что запросы, инициированные `Customer Frontend service`, имеют время отклика от `70 мс` до `439 мс`. В данном примере наибольший вклад в время отклика вносит `JourneyService`. Всего этот сервис получил `585` запросов.

Это даёт видимость на уровне кода для данного диапазона времени отклика. Выберите раздел **Service execution** на инфографике, чтобы увидеть разбивку по времени выполнения базовых сервисов. ## Анализ медленных запросов

Вернувшись на страницу **Response time distribution**, можно увидеть, что несколько запросов имеют время отклика более `3.4 с`. Выбор этого столбца на диаграмме открывает дополнительные сведения (см. ниже). Этот последний выбор выявил важную особенность: в данном диапазоне времени отклика существует только один запрос со временем отклика `3.77 с` (видно ниже в разделе **Top web requests**). Этот единственный выброс исказил общее время отклика сервиса. Теперь пришло время копнуть глубже для выявления первопричины проблемы.

## Корреляция ошибок с временем отклика

Ещё один важный аспект распределения времени отклика — возможность сопоставления частоты отказов с временем отклика. Неудачные запросы часто имеют особенно быстрое или медленное время отклика (они либо быстро завершаются с ошибкой, либо в итоге выходят по таймауту). В примере ниже показано, что 38% всех запросов быстрее `27.6 мс` завершаются с ошибкой. Более детальный взгляд на таблицу запросов показывает, что это на самом деле два конкретных типа запросов, которые всегда завершаются с ошибкой в данном диапазоне времени отклика. Эти запросы можно проанализировать, выбрав **View details of failures**.

![Выброс 10](https://dt-cdn.net/images/outlier10-1548-9fbe76f395.png)

Выброс 10

Анализ сбоев показывает, что все рассматриваемые запросы завершаются с ошибкой из-за ошибки `HTTP 500` в сервисе `PHP on FPM pool www`, которую можно дополнительно проанализировать, выбрав **Details**.

![Выброс 11](https://dt-cdn.net/images/outlier11-1237-d52e611821.png)

Выброс 11

Как видно, анализ выбросов, обеспеченный функцией анализа распределения времени отклика Dynatrace, позволяет выявить, какие типы запросов оказывают наибольшее влияние на общее время отклика каждого сервиса. Анализ выбросов также помогает понять корреляцию между конкретными ошибками и временем отклика.
