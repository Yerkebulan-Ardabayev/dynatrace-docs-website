# Фильтрация потока сервисов

# Фильтрация потока сервисов

* 5-min read
* Published Jul 19, 2017

Современные веб-приложения, как правило, имеют сложные сервисные архитектуры, способные обрабатывать миллионы различных типов запросов. Поскольку каждый уникальный запрос ведёт себя немного по-своему и инициирует слегка отличающийся поток сервисов, анализ производительности и поведения отдельных запросов может быть весьма непростой задачей.

Функции фильтрации помогают ориентироваться в сложности сервисной архитектуры приложения — позволяя найти иголку в стоге сена. **Service flow** в Dynatrace позволяет анализировать подмножества запросов, инициируемых данным сервисом.

Общая процедура фильтрации **Service flow** выглядит следующим образом:

1. Перейдите в **Services**.
2. Выберите сервис для анализа.
3. На странице обзора сервиса в разделе **Understand dependencies** выберите **View service flow**.
4. В **Service flow** выберите вызываемый сервис для определения последовательности сервисов для анализа.
   [Панель справа](/managed/observe/application-observability/services-classic/service-flow/service-flow-metrics#side-pane "Узнайте о метриках потока сервисов, измеряющих производительность вызовов сервисов, инициируемых каждым запросом в вашем окружении.") немедленно откроется на вкладке **Passing transactions**.
5. Чтобы создать фильтр для выбранной последовательности сервисов, выполните одно из следующих действий:

   * Выберите **Filter service flow** в верхней плитке.
   * Выберите **Filter** ![Filter](https://dt-cdn.net/images/filter-icon-41ddd02d66.svg "Filter") над выбранным сервисом.
6. Добавьте дополнительные критерии к фильтру:

   1. В фильтре выберите сервис, к которому хотите применить дополнительную фильтрацию.
   2. В списке **Filtered by** выберите критерий.
   3. Задайте пороговое значение или правила соответствия для критерия.
   4. Выберите **Apply**.
   5. При необходимости добавьте дополнительные критерии к фильтру.
   6. Когда закончите, выберите **Apply**.

Подробное описание приведено ниже.

## Фильтрация запросов на основе конкретных цепочек вызовов

Фильтры цепочек вызовов доступны в большинстве представлений анализа сервисов, но наиболее наглядны в **Service flow**. Как видно в примере ниже, только **5,9%** запросов к сервису `easyTravel Customer Frontend` также вызывали сервис `JourneyService`. Далее **99%** из них вызвали сервис базы данных `easyTravel-Business`.

![Поток сервисов — обзор](https://dt-cdn.net/images/service-flow-overview-1779-2ec7d38f7d.png)

Поток сервисов — обзор

Чтобы сфокусировать **Service Flow** на этих вызовах

1. Перейдите в **Services**.
2. Выберите сервис для анализа.
3. На странице обзора сервиса в разделе **Understand dependencies** выберите **View service flow**.
4. В **Service flow** выберите вызываемый сервис для определения последовательности сервисов для анализа. В нашем примере это `easyTravel-Business`.
   Вкладка **Passing transactions** появится в правой части страницы.
5. Чтобы создать фильтр для выбранной последовательности сервисов, выполните одно из следующих действий:

   * Выберите **Filter service flow** в верхней плитке.
   * Выберите **Filter** ![Filter](https://dt-cdn.net/images/filter-icon-41ddd02d66.svg "Filter") над выбранным сервисом.

Вернёмся к нашему примеру. Создан фильтр, чтобы сосредоточить анализ только на тех запросах от сервиса `easyTravel Customer Frontend`, которые вызывают сервис `JourneyService` и последовательно вызывают базу данных `easyTravel-Business`.

Обратите внимание, что количество анализируемых запросов к `easyTravel Customer Frontend` уменьшилось до **8,73k** с **156k**, поскольку теперь учитывается только подмножество вызовов. Соответственно, среднее время отклика теперь представляет только те запросы к `easyTravel Customer Frontend`, которые вызывают `JourneyService`.

Теперь внимательно посмотрим на узел `JourneyService`. Обратите внимание, что **35%** времени отклика сервиса `easyTravel Customer Frontend` занимает `JourneyService`. **Service flow** также обнаруживает неожиданное — некоторые из выбранных запросов инициируют `RMI server`.

![Поток сервисов — фильтр по вызываемому сервису](https://dt-cdn.net/images/service-flow-filter-1621-b66825047a.png)

## Многоаспектные фильтры

Каждый фильтр может содержать несколько цепочек вызовов. Это означает, что можно создавать сложные, многоаспектные фильтры цепочек вызовов в соответствии с конкретными потребностями анализа сервисов. **Service Flow** будет фокусироваться только на вызовах, соответствующих всем критериям.

Чтобы добавить новые последовательности к существующему фильтру

1. Пока текущий фильтр активен, выберите дополнительную цепочку вызовов в **Service flow**.
2. На вкладке **Passing transactions** [панели справа](/managed/observe/application-observability/services-classic/service-flow/service-flow-metrics#side-pane "Узнайте о метриках потока сервисов, измеряющих производительность вызовов сервисов, инициируемых каждым запросом в вашем окружении.") выберите **Filter service flow**.

В примере ниже критерии фильтра расширены вызовами от сервиса `easyTravel Customer Frontend`, которые вызывают сервис `RMI server` и последовательно вызывают базу данных `easyTravel-Business`. Теперь количество анализируемых запросов к `easyTravel Customer Frontend` уменьшилось до **528**. Также сервис `JourneyService` теперь отвечает за **39%** времени отклика.

## Расширение фильтров дополнительными критериями

После сужения количества вызовов на основе задействованных сервисов можно добавить дополнительные критерии к фильтру.

1. В фильтре выберите сервис, к которому хотите применить дополнительную фильтрацию.
2. В списке **Filtered by** выберите критерий.
3. Задайте пороговое значение или правила соответствия для критерия.

   ![Фильтр потока сервисов 8](https://dt-cdn.net/images/service-filter-8-699-1e8e0e114f.png)

   Фильтр потока сервисов 8
4. Выберите **Apply**.
5. При необходимости добавьте дополнительные критерии к фильтру.
6. Когда закончите, выберите **Apply**.

В нашем примере мы применили пороговое значение **Response time** в **200 мс** для `JourneyService`. Теперь отображаются только вызовы со временем отклика более **200 мс** — их всего **6**. Мы уже очень близки к нахождению иголки в стоге сена!

## Анализ цепочек вызовов с разных ракурсов

Истинная сила фильтрации цепочек вызовов становится очевидной при начале анализа проблемной цепочки вызовов. В приведённом выше примере **Service flow** теперь показывает нам, что **40%** времени отклика сервиса `easyTravel Customer Frontend` занимает `JourneyService`.

Следующий логичный шаг — анализ времени отклика сервиса `JourneyService` в контексте выбранной цепочки вызовов. Для этого выберите `JourneyService` в **Service flow**. Правая панель показывает все варианты анализа, которые можно выполнить для выбранного сервиса — всё в контексте отфильтрованной цепочки вызовов. Все фильтры, созданные в **Service flow**, также будут применены к дополнительному анализу.

Подробнее о дополнительных параметрах анализа в темах, перечисленных ниже.

* [Distributed traces](/managed/observe/application-observability/distributed-traces/use-cases/segment-request#pp-analysis "Улучшайте производительность распределённой системы, сегментируя запросы с медленным временем отклика через поток сервисов и анализируя их распределённые трассировки.")
  Анализируйте детальную цепочку вызовов на уровне методов.
* [Analyze backtrace](/managed/observe/application-observability/services-classic/service-backtrace "Отследите последовательность вызовов сервисов вплоть до нажатия в браузере, инициировавшего эту последовательность.")
  Исследуйте последовательность вызовов сервисов, ведущую к конкретному запросу к сервису.
* [View response time](/managed/observe/application-observability/services-classic/service-response-time-hotspots "Определяйте действия, потребляющие наибольшую часть времени отклика для каждого сервиса.")
  Просматривайте распределение времени отклика по различным функциям сервиса (например, использование базы данных и выполнение кода).
* [Analyze outliers](/managed/observe/application-observability/services-classic/response-time-distribution-and-outlier-analysis "Получите представление о распределении времени отклика по всем запросам, включая аномально высокие и аномально низкие.")
  Просматривайте распределение времени отклика запросов к сервису в конкретном временном периоде.
