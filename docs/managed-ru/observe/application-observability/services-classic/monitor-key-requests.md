# Мониторинг ключевых запросов

# Мониторинг ключевых запросов

* How-to guide
* 5-min read
* Updated on Jan 21, 2026

*Ключевые запросы* — это запросы, требующие особого внимания: либо потому что они являются важными показателями успеха вашего бизнеса (например, запрос входа в систему или запрос оформления заказа в корзине), либо потому что они обеспечивают жизненно важную техническую функциональность, от которой зависит ваше приложение.

* Ключевые запросы имеют [специальные метрики](/managed/analyze-explore-automate/metrics-classic/built-in-metrics#key-requests "Ознакомьтесь с полным списком встроенных метрик Dynatrace."), которыми можно управлять через веб-интерфейс или [API](#manage-api). Можно создавать специальные плитки дашборда для построения диаграмм ключевых запросов с прямым доступом с дашборда и анализировать долгосрочную историю метрик ключевых запросов в [диаграммах запросов](/managed/observe/application-observability/services-classic#request-charting "Узнайте о классическом мониторинге сервисов в Dynatrace.").
* Оповещения для ключевых запросов всегда включены, даже если они составляют менее 1% от пропускной способности. Они также поддерживают пользовательские пороговые значения.
* Периоды хранения данных ключевых запросов поддерживаются следующим образом:

  | Тип данных | Период хранения |
  | --- | --- |
  | [Детализированные данные уровня кода](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#purepath "Проверьте сроки хранения для различных типов данных.") | 10 дней |
  | [Агрегированные данные уровня кода](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#request-attributes "Проверьте сроки хранения для различных типов данных.") | 35 дней |
  | [Долгосрочная история метрик](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#metrics-classic "Проверьте сроки хранения для различных типов данных.") | 5 лет |

Ключевые запросы выделены в разделе **Key requests/endpoints** на каждой странице обзора сервиса. Это особенно ценно для запросов с малым объёмом, но высокой важностью, которые в противном случае оказались бы внизу списка **Top requests**.

Количество ключевых запросов ограничено:

* 500 ключевых запросов на окружение по всем сервисам.
* 100 ключевых запросов на сервис.

При достижении этого лимита рассмотрите использование [вычисляемых метрик сервисов](/managed/observe/application-observability/services/calculated-service-metric "Узнайте, как создать вычисляемую метрику на основе веб-запросов."), предоставляющих более гибкий подход.

## Создание ключевого запроса (через веб-интерфейс)

Чтобы пометить конкретный запрос как ключевой

1. Перейдите в **Services**.
2. Выберите соответствующий сервис из списка.
3. На странице обзора сервиса выберите кнопку **View** (например, **View requests**, **View dynamic requests** или **View resource requests**).
4. Прокрутите вниз до **Top requests** и выберите запрос, который нужно пометить как ключевой.
5. На странице обзора запроса выберите **More** (**…**) > **Mark as key request** или **Pin to dashboard**.

   ![Установка ключевого запроса](https://dt-cdn.net/images/key-request-1000-c04070fc96.png)

   Установка ключевого запроса

После ручного указания ключевого запроса его линии тренда сохраняются бессрочно.

## Отображение ключевых запросов на дашборде

Чтобы создать плитку дашборда для конкретного запроса

1. Перейдите в **Services**.
2. Выберите соответствующий сервис из списка.
3. На странице обзора сервиса выберите **View** (**View requests**, **View dynamic requests** или **View resource requests**).
4. Прокрутите вниз до **Key requests/endpoints** и выберите запрос для отображения на дашборде.
5. На странице обзора запроса выберите **More** (**…**) > **Pin to dashboard**.  
   На дашборд добавляется новая плитка для конкретного запроса, показывающая наиболее важные метрики для этого запроса.

Плитки дашборда содержат только данные, собранные после того, как запрос был помечен как ключевой.

## Переименование ключевых запросов

Обнаружение ключевых запросов основано на именах. При применении [правила именования запросов](/managed/observe/application-observability/services/service-detection/service-detection-v1/set-up-request-naming "Настройте именование запросов и определите операции, предлагаемые вашими сервисами.") оно может повлиять на ключевые запросы. Если вы хотите, чтобы Dynatrace продолжал обнаруживать переименованные запросы как ключевые, необходимо добавить новое имя в список имён ключевых запросов.

1. Перейдите в **Services** и выберите сервис для настройки.
2. Выберите **More** (**…**) > **Settings**.
3. На странице **Service settings** перейдите на вкладку **Key requests** и выберите **Add item**, чтобы добавить имя, к которому применяются правила именования запросов.

## Обнаружение аномалий с ключевыми запросами

Dynatrace предполагает, что запросы с малым объёмом менее важны, чем запросы с высоким объёмом и ключевые запросы. Это означает, что запросы, составляющие менее 1% от общей нагрузки сервиса, не будут генерировать оповещения, если только их воздействие не достаточно значительно, чтобы повлиять на общее время отклика или частоту отказов сервиса. Поскольку такая обработка по умолчанию не подходит для всех запросов с малым объёмом, следует вручную помечать любые важные запросы с малым объёмом как ключевые, чтобы обеспечить для них стандартные пороги оповещений.

### Пороги оповещений для конкретных запросов

Поскольку определённые запросы могут иметь специфические паттерны времени отклика и частоты отказов, тогда как другие могут иметь строгие пороги SLA, Dynatrace позволяет определять пользовательские пороги оповещений при обнаружении аномалий, связанных с производительностью ключевых запросов. Если заданы, пороги на уровне ключевых запросов переопределяют пороги на уровне сервиса. Сведения о настройке порогов на уровне запросов см. в разделе [**Thresholds for a specific web request**](/managed/dynatrace-intelligence/anomaly-detection/adjust-sensitivity-anomaly-detection/adjust-sensitivity-services#key-request "Узнайте, как настроить чувствительность обнаружения проблем для сервисов.").

## Вычисляемая метрика сервиса

В качестве альтернативного способа фокусировки на конкретных запросах можно создать [вычисляемую метрику сервиса](/managed/observe/application-observability/services/calculated-service-metric "Узнайте, как создать вычисляемую метрику на основе веб-запросов.") на основе нужных запросов. Этот подход обеспечивает большую гибкость в оповещениях — вычисляемую метрику можно использовать так же, как любую встроенную метрику Dynatrace.

## Управление ключевыми запросами через Settings API

Конфигурациями ключевых запросов можно управлять через [Settings API](/managed/dynatrace-api/environment-api/settings "Узнайте, что предлагает Dynatrace Settings API.").

Для использования API необходим токен доступа с областями **Read settings** (`settings.read`) и **Write settings** (`settings.write`). Сведения о его получении см. в разделе [Создание токена доступа](/managed/dynatrace-api/basics/dynatrace-api-authentication#create-token "Узнайте, как пройти аутентификацию для использования Dynatrace API.").

### Создание конфигурации ключевого запроса

Выполните приведённые ниже шаги для создания новой конфигурации ключевого запроса. Обратите внимание, что эта процедура перезаписывает существующую конфигурацию. Сведения об изменении существующей конфигурации см. в разделе [**Обновление конфигурации ключевого запроса**](#update-api) ниже.

1. Чтобы узнать формат объекта настроек, запросите его схему через вызов [GET a schema](/managed/dynatrace-api/environment-api/settings/schemas/get-schema "Просмотрите схему настроек через Dynatrace API."). Идентификатор схемы ключевых запросов — `builtin:settings.subscriptions.service`.
2. Создайте объект JSON для ваших настроек.  
   Обратите внимание, что область ключевого запроса всегда является сервисом. Необходимо указать сервис по его идентификатору сущности Dynatrace. Чтобы узнать идентификатор сущности вашего сервиса, используйте запрос [GET entities list](/managed/dynatrace-api/environment-api/entity-v2/get-entities-list "Просмотрите список отслеживаемых сущностей через Dynatrace API.").
   Пример JSON

   ```
   [



   {



   "schemaId": "builtin:settings.subscriptions.service",



   "scope": "SERVICE-123456789",



   "value": {



   "keyRequestNames": [



   "/cart/checkout"



   ]



   }



   }



   ]
   ```
3. Используйте конечную точку [POST an object](/managed/dynatrace-api/environment-api/settings/objects/post-object "Создайте или проверьте объект настроек через Dynatrace API.") для отправки конфигурации.

### Обновление конфигурации ключевого запроса

1. Чтобы узнать формат объекта настроек, запросите его схему через вызов [GET a schema](/managed/dynatrace-api/environment-api/settings/schemas/get-schema "Просмотрите схему настроек через Dynatrace API."). Идентификатор схемы ключевых запросов — `builtin:settings.subscriptions.service`.  
   Обратите внимание, что область ключевого запроса всегда является сервисом. Необходимо указать сервис по его идентификатору сущности Dynatrace. Чтобы узнать идентификатор сущности вашего сервиса, используйте запрос [GET entities list](/managed/dynatrace-api/environment-api/entity-v2/get-entities-list "Просмотрите список отслеживаемых сущностей через Dynatrace API.").
2. Запросите текущую конфигурацию через вызов [GET objects](/managed/dynatrace-api/environment-api/settings/objects/get-objects "Просмотрите несколько объектов настроек через Dynatrace API.").
3. Создайте JSON для обновления.

   1. Используйте значение **updateToken** из предыдущего шага.
   2. При необходимости измените список запросов в массиве **keyRequestNames**.
      Пример JSON

      ```
      {



      "updateToken": "vu9U3hXY3q0ATAAkMG",



      "value": {



      "keyRequestNames": [



      "/cart/checkout",



      "/cart"



      ]



      }



      }
      ```
4. Используйте конечную точку [PUT an object](/managed/dynatrace-api/environment-api/settings/objects/put-object "Редактируйте объект настроек через Dynatrace API.") для отправки конфигурации.
