# Анализ отдельных экземпляров сервисов

* How-to guide
* 2-min read
* Published Jul 19, 2017

Dynatrace предлагает возможность [обнаружения кластеризованных сервисов](/managed/observe/application-observability/services/service-detection/service-detection-v1 "Узнайте, как Dynatrace Service Detection v1 обнаруживает и именует различные типы сервисов."). Кластеризованные сервисы — это сервисы, обслуживаемые несколькими процессами. Dynatrace позволяет анализировать распределение нагрузки, распределение времени отклика и частоту отказов для каждого отдельного экземпляра сервиса.

Чтобы просмотреть экземпляры сервисов

1. Перейдите в **Services**.
2. Выберите кластеризованный сервис для анализа.
3. На странице обзора сервиса выберите кнопку **View** (например, **View requests**, **View dynamic requests** или **View resource requests**).
4. Прокрутите вниз и разверните раздел **Service instances** для просмотра сведений.
5. Выберите экземпляр сервиса для анализа. Подробности приведены ниже.

## Анализ метрик экземпляра сервиса

Раздел **Service instances** содержит запись для каждого существующего в настоящее время экземпляра этого сервиса (или существовавшего в течение [временного периода анализа](/managed/analyze-explore-automate/dashboards-classic/dashboards/dashboard-timeframe "Узнайте о настройках временного периода и зоны управления на дашборде Dynatrace.")). Для каждого экземпляра можно увидеть узел кластера, на котором запущен экземпляр (указан под именем экземпляра сервиса), **Total time consumption** экземпляра, а также **Median response time**.

![Экземпляры 2](https://dt-cdn.net/images/instances2-1607-eadbcf75cd.png)

Экземпляры 2

В приведённом примере обратите внимание на различие в суммарном потреблении времени и медианном времени отклика между экземплярами сервисов. Это означает, что нагрузка не распределена равномерно по всем узлам кластера.

Выберите отдельный экземпляр сервиса для отображения специфичных для этого экземпляра показателей **Response time**, **Failure rate**, потребления **CPU** и **Throughput**. Это позволяет понять, наблюдается ли всплеск в общем времени отклика сервиса на всех экземплярах или только на одном. Обратите внимание, что имя конкретного экземпляра отображается как критерий фильтрации.

![Экземпляры 0](https://dt-cdn.net/images/instances0-1653-c78bd33bd0.png)

Экземпляры 0

Выберите **View response time hotspots** для дальнейшего изучения различий во времени отклика между узлами. В примере ниже показан [анализ времени отклика](/managed/observe/application-observability/services-classic/service-response-time-hotspots "Определяйте действия, потребляющие наибольшую часть времени отклика для каждого сервиса.") для одного экземпляра. Обратите внимание, что имя узла экземпляра отображается под заголовком страницы в виде фильтра.

## Определение экземпляров, совершающих вызовы к другим сервисам

Ещё одна полезная возможность анализа экземпляров сервисов — сведения о конкретных экземплярах, из которых исходят определённые вызовы. Просто выберите **Analyze backtrace** на любой странице экземпляра сервиса. [Анализ обратной трассировки](/managed/observe/application-observability/services-classic/service-backtrace "Отследите последовательность вызовов сервисов вплоть до нажатия в браузере, инициировавшего эту последовательность.") предоставляет разбивку по экземплярам для каждого сервиса.
