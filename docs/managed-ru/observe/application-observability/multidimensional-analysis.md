# Многомерный анализ

# Многомерный анализ

* How-to guide
* 4-min read
* Updated on Sep 13, 2022

Представление **Multidimensional analysis** позволяет анализировать веб-запросы ваших сервисов с точной настройкой фильтрации, чтобы сосредоточить анализ на наиболее важных измерениях. Это представление легко настраивается и служит удобной точкой входа для углублённого анализа ваших сервисов.

С помощью многомерного анализа можно исследовать пользовательские представления: **Top web requests**, **Database statements** и **Exception analysis**.

Чтобы перейти на страницу **Multidimensional analysis**

1. Перейдите в **Multidimensional Analysis**.
2. Выберите один из вариантов анализа для исследования или **Create analysis view**.

## Источник данных

Многомерный анализ использует данные трассировок и запросов, включающие информацию о [распределённых трассировках](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#traces-grail "Проверьте сроки хранения для различных типов данных.") и [запросах](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#request-attributes "Проверьте сроки хранения для различных типов данных.").

* Для временных периодов короче двух часов используется более высокое разрешение (менее 1 минуты).
* Для больших объёмов данных, особенно для длительных временных периодов или нефильтрованного анализа, применяется выборка: используется лишь часть данных трассировок и запросов.
* Чтобы просмотреть информацию о данных, используемых для анализа, и рекомендации по улучшению точности, наведите курсор на **Refine**.
* Чтобы включить в анализ больше данных, выберите **Refine**.

Многомерный анализ и диаграммы

В отличие от [Data Explorer](/managed/analyze-explore-automate/explorer#limitations "Запрашивайте метрики и преобразуйте результаты для получения нужных аналитических данных."), многомерный анализ использует данные трассировок и запросов, а не данные метрик, поэтому значения на диаграммах многомерного анализа могут отличаться от значений на пользовательских диаграммах.

## Настройка представления

В разделе **Configure view** можно настроить несколько возможностей фильтрации. Представление обновляется автоматически при изменении параметров.

| Параметр | Описание |
| --- | --- |
| Metric | Метрика для анализа. |
| Aggregation | Способ агрегации значений метрики. Доступные агрегации зависят от выбранной метрики. |
| Split mode | Способ обработки измерений, указанных в **Split by dimension**.  * **Split by services** — каждое измерение отображается отдельно для каждого сервиса. * **Merge by services** — одинаковые измерения из разных сервисов объединяются в одно. |
| Split by dimension | Список измерений, по которым разбиваются запросы.  Можно указать несколько измерений. Поместите курсор в поле ввода, чтобы увидеть доступные параметры. Запросы разбиваются по измерениям в указанном порядке. |
| Filter requests | Фильтрация запросов для включения в представление. Поместите курсор в поле ввода, чтобы увидеть доступные параметры.  Критерии одного типа объединяются по логике **OR**. Критерии разных типов объединяются по логике **AND**. |

Данные таблицы можно экспортировать в файл CSV.

1. В правом нижнем углу страницы выберите **Show export menu** ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More").

   ![Меню Show export](https://dt-cdn.net/images/show-export-menu-107-2a8a76c9a2.png)

   Меню Show export
2. Выберите **Export visible data** или **Export table data**.

   | Параметр | Экспортируемые данные | Поля | Количество записей |
   | --- | --- | --- | --- |
   | **Export visible data** | Текущая видимая область таблицы с учётом применённых фильтров | Только видимые данные | До 100 топовых измерений |
   | **Export table data** | Все данные таблицы | Все доступные данные, связанные с топовыми измерениями | До 100 топовых измерений |

## Представление

![Страница топ операторов БД](https://dt-cdn.net/images/top-database-statements-3564-3bd08ea1b1.png)

Страница топ операторов БД

На диаграмме показаны 15 топовых измерений (все остальные измерения агрегируются в одно), а таблица ниже содержит до 85 дополнительных измерений, что в сумме составляет до 100 отображаемых на диаграмме измерений. Представление мгновенно адаптируется к изменениям в панели **Configure view**.

В столбце **Actions** таблицы можно выбрать:

* **Filter** ![Filter](https://dt-cdn.net/images/filter-icon-41ddd02d66.svg "Filter") — для фильтрации представления по указанному измерению.
* **More** (**…**) — для доступа к дополнительным параметрам анализа из [меню **Analyze**](/managed/observe/application-observability/services-classic/context-specific-drill-down#analyze-menu "Узнайте об удобной навигации и фильтрации для анализа сервисов.").

Для временных периодов, содержащих данные старше 10 дней, можно включить **Show data retention**, чтобы лучше понять, какие данные доступны за какой период непосредственно на диаграмме.

После настройки представления его можно сохранить для быстрого доступа в будущем. Просто выберите **Save view** и укажите имя.

## Вычисляемая метрика сервиса

Настроенное представление можно сохранить как вычисляемую метрику сервиса, которую затем можно использовать как любую другую метрику Dynatrace, например для [построения диаграмм](/managed/analyze-explore-automate/explorer "Запрашивайте метрики и преобразуйте результаты для получения нужных аналитических данных.") или [экспорта данных через API](/managed/dynatrace-api/environment-api/metric-v2/get-data-points "Считывайте точки данных одной или нескольких метрик через Metrics v2 API.").

Вычисляемые метрики включают только новые данные; ретроспективные данные не включаются.

Ограничения

* В вычисляемые метрики записываются только новые данные; ретроспективные данные не включаются.
* Можно иметь до 500 включённых вычисляемых метрик на окружение и до 100 включённых вычисляемых метрик на сервис.
* Классические вычисляемые метрики поддерживают не более 100 значений измерений. Это называется правилом «топ X», так как можно выбрать меньшее количество в зависимости от конфигурации. При выборе 100 топовых значений измерений оставшиеся измерения агрегируются в один временной ряд, и значение измерения доступно через специальное измерение `remainder`. Условие фильтра [remainder](/managed/dynatrace-api/environment-api/metric-v2/metric-selector#remainder "Настройте селектор метрик для Metric v2 API.") позволяет фильтровать по этому измерению `remainder`.

Чтобы создать вычисляемую метрику сервиса из представления многомерного анализа

1. Перейдите в **Multidimensional Analysis**.
2. Выберите **Create analysis view**.
3. Необязательно: выберите зону управления. Новая метрика будет ограничена данными из этой зоны.
4. Настройте представление. Описание доступных параметров см. в разделе [**Configure view**](#configure) выше.
5. На панели **View** выберите **Create metric**.

   ![Создание вычисляемой метрики](https://dt-cdn.net/images/service-calculated-metric-create-328-e5859c1ea9.png)

   Создание вычисляемой метрики
6. Укажите имя метрики. Имя автоматически добавляется к ключу метрики.
7. Необязательно: при необходимости настройте ключ метрики.

   После создания метрики изменить её ключ невозможно.
8. Необязательно: чтобы более точно настроить метрику, выберите **Advanced options** для настройки дополнительных параметров. Подробности см. в разделе [Вычисляемые метрики для сервисов](/managed/observe/application-observability/services/calculated-service-metric "Узнайте, как создать вычисляемую метрику на основе веб-запросов.").
9. Выберите **Create metric**.
