# Непрерывный анализ потоков

# Непрерывный анализ потоков

* How-to guide
* 5-min read
* Published Aug 30, 2019

Многопоточные архитектуры легко масштабируются, поскольку распределение работы позволяет процессорам оставаться активными и продолжать выполнять код. Однако когда системам необходимо координировать работу между несколькими потоками, блокировки учащаются и может возникать снижение скорости выполнения кода.

В Dynatrace такое поведение автоматически и непрерывно обнаруживается OneAgent. При непрерывном анализе потоков данные постоянно доступны с историческим контекстом и не требуют запуска при возникновении проблемы. Вы можете получать оповещения о тенденциях, анализировать дампы потоков и напрямую приступать к улучшению кода там и тогда, где и когда это необходимо, для предотвращения узких мест.

## Перед началом

Чтобы начать использование непрерывного анализа потоков

1. Перейдите в **Settings** > **Preferences** > **OneAgent features**.
2. Найдите и включите следующие функции OneAgent для технологии, которую нужно отслеживать:

   | Технология для мониторинга | Функции OneAgent для включения |
   | --- | --- |
   | Java | **Continuous thread analysis** |
   | Node.js | * **Node.js worker threads monitoring** * **Node.js code module preloading** |

Поддерживаемые версии Java

Из-за известной проблемы с Java 11 непрерывный анализ потоков JVM доступен только для Java 8 и Java 17+ (на основе [поддерживаемых версий](/managed/ingest-from/technology-support/application-software/java/support-for-jvms "Узнайте о поддерживаемых Dynatrace основных JVM и JDK.")). Потоки приложений и агента не затрагиваются.

## Анализ потоков

Чтобы начать анализ дампов потоков

1. Перейдите в **Profiling & Optimization** > **Continuous CPU profiling**.
2. Для группы процессов, которую нужно проанализировать, выберите **More** (**…**) > **Threads**.
3. На странице **Thread analysis** можно:

   * Анализировать разбивку потоков по состояниям потоков или по расчётному времени CPU.
   * Включать или исключать данные по ожидающим потокам в любое время, установив соответствующий флажок.
   * Фильтровать данные

     + По типу запроса.  
       Выберите **Filter requests** > **Type** и выберите параметр.
     + По имени группы потоков запроса.  
       Выберите имя группы потоков. Сегментация групп потоков учитывает тот факт, что они обычно выполняют разные функции, и обеспечивает быстрый способ идентификации потребителей CPU.
   * Анализировать горячие точки методов группы потоков.  
     В столбце **Actions** группы потоков выберите **More** (**…**) > **Method hotspots**.

## Сценарии использования

Потоки являются источником масштабируемости, так как позволяют приложениям выполнять несколько задач одновременно. В определённых ситуациях это может стать источником узких мест в производительности. Например:

* В системах с высокой нагрузкой могут возникать проблемы с блокировкой потоков, [препятствующие масштабированию приложения](#scalability).
* Если количество активных потоков слишком велико, [ресурсы CPU могут расходоваться впустую](#cpu) из-за чрезмерной загрузки или принуждения операционной системы планировать тысячи потоков на ограниченном наборе ядер.

Пример: блокировки препятствуют масштабируемости приложения

В этом примере группа процессов `idea*.exe` показывает всплеск потребления CPU. Исследуем проблему на её странице **Thread analysis**, начиная с фильтрации списка групп потоков по состоянию потока **Locking**.

Время блокировки в группе процессов Java

Видно, что система не находится в плохом состоянии, но поведение блокировки всё же затрагивает `23` группы потоков. Выбираем первую и наиболее затронутую группу потоков `JobScheduler FJ pool` для продолжения анализа.

![Анализ времени блокировки в группе процессов Java](https://dt-cdn.net/images/locking-usercase-thread-analysis-2-1562-55c0abba1f.png)

Анализ времени блокировки в группе процессов Java

Для группы потоков `JobScheduler FJ pool` значительная доля времени (почти половина: `46.1%`) постоянно тратится на блокировку и распределена по `15` потокам. Это указывает на общую блокировку, затрагивающую все эти потоки. Такого поведения следует избегать, поскольку оно ограничивает как скорость, так и способность повышать пропускную способность путём добавления ресурсов. В конечном счёте это может привести к состоянию, при котором система не сможет обрабатывать больше данных, даже если добавить оборудование.

Чтобы установить первопричину, выбираем **More** (**…**) > **Method hotspots**.

![Анализ горячих точек методов для времени блокировки в группе процессов Java](https://dt-cdn.net/images/locking-usercase-thread-analysis-3-1158-1e89cbdec0.png)

Анализ горячих точек методов для времени блокировки в группе процессов Java

На основе полученной информации можно определить место блокировки и проверить различные способы написания кода для повышения его производительности.

## Часто задаваемые вопросы

Связан ли период хранения данных для непрерывного анализа потоков с периодом хранения данных уровня кода?

Да, период хранения данных для непрерывного анализа потоков совпадает с периодом хранения для [распределённых трассировок, детализации уровня кода и ошибок](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods#purepath "Проверьте сроки хранения для различных типов данных.").
