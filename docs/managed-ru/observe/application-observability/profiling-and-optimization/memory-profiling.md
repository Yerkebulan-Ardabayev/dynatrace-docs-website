# Профилирование памяти

* How-to guide
* 7-min read
* Updated on Aug 30, 2022

Для использования этой функции необходима JVM с поддержкой Low-Overhead Heap Profiling, представленной в [JEP 331](https://openjdk.java.net/jeps/331). Обратите внимание, что эта функция доступна только для процессов Java.

Профилирование памяти позволяет понять поведение выделения памяти и сборки мусора в приложениях со временем. Оно помогает выявлять вызовы методов в контексте, в котором выделяется большая часть памяти, и сочетать эту информацию с количеством выделенных объектов. Перспектива Survivor помогает понять контекст, в котором создаются долгоживущие объекты (объекты, пережившие несколько циклов сборки мусора).

Для доступа к профилированию памяти доступны следующие варианты:

* **Через диагностические инструменты**

  1. Перейдите в **Profiling & Optimization** и выберите **Continuous CPU profiling**.
  2. В столбце **Actions** нужной группы процессов выберите **More** (**…**) > **Memory allocation**.
* **Через сведения о процессе**

  1. Перейдите в **Hosts** и выберите нужный хост.
  2. На странице хоста перейдите в **Process analysis** и выберите нужный процесс.  
     При использовании [классического дизайна страницы хоста](/managed/observe/infrastructure-observability/hosts/monitoring/host-monitoring "Мониторинг хостов с Dynatrace.") для поиска списка процессов перейдите в **Processes**.
  3. Выполните одно из следующих действий:

     + Выберите **More** (**…**) > **Memory profiling**, чтобы открыть вкладку **All allocations**.
     + Откройте вкладку **JVM metrics** и выберите **Analyze suspension** для непосредственного открытия вкладки **Survivors**.

Можно выбрать любой временной период для анализа, независимо от глобального временного периода. Рекомендуется [исключить сторонние библиотеки](#exclude-3rd-party) из анализа, чтобы сосредоточиться на коде под вашим контролем.

## Все выделения

В представлении **All allocations** показывается, сколько раз выполнялась сборка мусора и сколько памяти выделяется. В нижней части страницы можно найти различные параметры анализа, такие как **Allocation hotspots**, **Flame graph** и **Hotspot APIs**.

На вкладке **Allocation hotspots** таблица показывает методы анализируемого процесса, потребляющие наибольшее количество памяти.

![Топ 100 горячих точек выделения памяти](https://dt-cdn.net/images/memory-profiling-allocation-hotspots-3492-d7820b4c3e.png)

Топ 100 горячих точек выделения памяти

* Чтобы увидеть вызывающих объектов метода, разверните метод.  
  Можно увидеть объём выделенной памяти, количество выделенных объектов и выделенную память по типам в методе.
* Чтобы просмотреть последовательность методов, вызванных анализируемым методом, выберите **Called Methods** в столбце **Actions**.

Столбец **Allocated types** показывает объекты, выделенные только в самом методе. Они не применяются к последующим вызываемым методам.

Можно [исключить сторонние API](#exclude-3rd-party) из этого представления, выбрав **More** (**…**) > **API detection rules** в столбце **Actions**.

Flame graph — тип столбчатой диаграммы, отображающей иерархические данные. Flame graph повышает читаемость трассировок стека с помощью макета сверху вниз и помогает выявлять горячие точки выделения памяти в коде приложения.

* Ось X охватывает объём памяти, выделенной для метода. Более широкая полоска означает, что для метода было выделено больше памяти.
* Ось Y представляет глубину стека. Трассировка стека отображается в виде столбца полосок, где каждая полоска представляет метод.
* Цвета зависят от настроенных определений API.

Выбрав полоску метода, можно скрыть верхние уровни и сосредоточить анализ, начиная с выбранного метода.

[Измените настроенные определения API](#allocation-hotspot-apis), чтобы повысить уровень детализации flame graph.

На вкладке **Hotspot APIs** в представлении **All allocations** перечислены выделенная память и выделенные объекты для каждого API. Цвета, связанные с каждым API, отображаются в flame graph.

Можно изменить список API, сделав его более специфичным для вашего приложения:

* За счёт дополнительной сегментации разбивки API. Узнайте, как [добавлять пользовательские API](/managed/observe/application-observability/services/customize-api-definitions "Установите правила обнаружения для настройки API в вашем окружении.").
* Путём [исключения сторонних библиотек](#exclude-3rd-party).

### Вызванные методы

Выбрав метод на вкладке **Allocation hotspots** или **Flame graph**, можно проанализировать трассировку стека, выбрав **Called methods** в правом верхнем углу вкладки.

* Вкладка **Overview** отображает методы, наиболее влияющие на потребление памяти, и flame graph для общего обзора.
* Вкладка **Details** отображает детальный и динамический flame graph для дальнейшего анализа трассировки стека.

## Survivor

Выяснив, какие методы потребляют больше всего памяти, можно проверить их долговечность в представлении **Survivors**. Представление показывает:

* Объекты, пережившие одну или несколько сборок мусора.
* Время, затраченное на сборку мусора.
* Объём памяти, оставшейся после всех сборок мусора.

Обратите внимание, что объекты, пережившие несколько сборок мусора, вносят вклад в объём уцелевшей памяти несколько раз.

В нижней части страницы можно найти различные параметры анализа, такие как **Survivor hotspots**, **Flame graph** и **Hotspot APIs**.

## Исключение сторонних библиотек

Чтобы сосредоточить анализ на вашем коде, рекомендуется определить API сторонних библиотек. Для этого необходимо пометить [пользовательский API](/managed/observe/application-observability/services/customize-api-definitions "Установите правила обнаружения для настройки API в вашем окружении.") как стороннюю библиотеку.

1. Перейдите в **Settings** > **Server-side service monitoring** > **API detection rules**.
2. Выберите **Expand** ![Развернуть строку](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Развернуть строку") для редактирования нужного API.
3. Включите **This API defines a third party library**.
4. Сохраните изменения.

* Можно использовать предварительно заполненную версию страницы **API detection rules** из таблиц горячих точек на странице **Memory profiling**. В строке выбранного метода выберите **More** (**…**) > **API detection rules** в столбце **Action**.
* Можно исключить библиотеку без ручной настройки правил из списка **Built-in APIs**. Создайте пользовательское правило для библиотеки и пометьте его как стороннее. Узнайте, как это сделать, в разделе [Пользовательские определения API](/managed/observe/application-observability/services/customize-api-definitions "Установите правила обнаружения для настройки API в вашем окружении.").

Страница **Called method** настроена для отображения всех методов, включая сторонние API. Когда вы определяете API сторонних библиотек, они не исключаются из представлений **Called method**.
