# Сегментация запросов для улучшения деградации времени отклика

# Сегментация запросов для улучшения деградации времени отклика

* Tutorial
* 3-min read
* Updated on May 17, 2024

В вашем окружении существуют тысячи запросов, каждый со своими взаимосвязями и контекстом. Чтобы определить первопричину неэффективности, необходимо сузить анализ до только релевантных запросов. Вы можете сегментировать запросы, [фильтруя поток сервисов](/managed/observe/application-observability/services-classic/service-flow/service-flow-filtering "Узнайте, как работает фильтрация потока сервисов и как её использовать.") или с помощью [анализа выбросов](/managed/observe/application-observability/services-classic/response-time-distribution-and-outlier-analysis "Получите представление о распределении времени отклика по всем запросам, включая аномально высокие и аномально низкие.").

## Сценарий

Сервис `easyTravel Customer Frontend` получил 249 000 запросов в течение выбранного 2-часового периода. В этом примере нужно выявить запросы с медленным временем отклика для данного сервиса.

## Шаги

1. Начните с сегментации запросов через **Service flow**.

   1. Чтобы перейти к потоку сервисов

      1. Перейдите в **Services**.
      2. Выберите сервис, который нужно проанализировать.
      3. На странице обзора сервиса в разделе **Understand dependencies** выберите **View service flow**.

      Нас интересуют именно запросы от `easyTravel Customer Frontend`, которые сначала вызывают `AuthenticationService`, а затем `easyTravel-Business`. **94%** запросов `easyTravel Customer Frontend`, вызывающих `AuthenticationService`, также вызывают `VerificationService`.
   2. Чтобы сосредоточиться на подмножестве запросов

      1. Выберите вызываемый сервис > **Apply filter** ![Filter](https://dt-cdn.net/images/filter-icon-41ddd02d66.svg "Filter").

         ![Сегментация транзакций через поток сервисов](https://dt-cdn.net/images/transactions-manual-segmentation-1225-210c1e1ad2.png)

         Сегментация транзакций через поток сервисов
      2. Чтобы добавить сервис в качестве второго параметра фильтрации, выберите нужный сервис.

         ![Уточнение цепочки вызовов](https://dt-cdn.net/images/service-flow-chain-of-calls-1221-12f6c53a07.png)

         Уточнение цепочки вызовов
2. Чтобы перейти к списку распределённых трассировок, отфильтрованных по заданным параметрам, выберите вызывающий сервис (`easyTravel Customer Frontend`) > **Distributed traces** ![Distributed traces](https://dt-cdn.net/images/purepaths-icon-790bea38ba.svg "Distributed traces").

   В списке **Most recent traces** отображаются запросы, инициированные `easyTravel Customer Frontend` и соответствующие критериям фильтра. Список можно фильтровать или сортировать по **Start time**, **Name**, **Response time**, **Processing time**, **HTTP method** или **Response code**.
3. Чтобы отобразить только запросы `easyTravel Customer Frontend` со временем отклика 80 мс и более

   1. Выберите узел **easyTravel Customer Frontend**.
   2. Из списка **Filter requests** выберите **Response time**.
   3. Выберите **greater than or equal to ≥**, введите `80` в поле ввода и выберите **Apply**.
   4. Выберите **Apply**.

   ![Фильтрация распределённых трассировок](https://dt-cdn.net/images/filter-distributd-traces-1621-2d6b34444e.png)

   Фильтрация распределённых трассировок

   Только **3** запроса из первоначальных 249 000 требуют углублённого анализа распределённой трассировки.

   ![Обнаружение медленных запросов](https://dt-cdn.net/images/slower-requests-detection-1589-e1fc1d5a66.png)

   Обнаружение медленных запросов
4. Выберите трассировку из уточнённого списка для перехода к анализу на уровне кода.

## Заключение

Сегментировав запросы в потоке сервисов `easyTravel Customer Frontend` и детализировав только те, что соответствуют нашим критериям, мы сузили необходимый углублённый анализ с 249 000 до 3 запросов за выбранный 2-часовой период.

Вы можете расширить анализ:

* Чтобы узнать, откуда поступил запрос, выберите **More** (**…**) > [**Service backtrace**](/managed/observe/application-observability/services-classic/service-backtrace "Отследите последовательность вызовов сервисов вплоть до нажатия в браузере, инициировавшего эту последовательность.").
* Чтобы увидеть всю трассировку от первой полностью отслеживаемой группы процессов, выберите **Show full trace**.  
  Каждая распределённая трассировка отслеживает запрос от начала до конца. Это означает, что трассировки всегда начинаются с первой полностью отслеживаемой группы процессов. С помощью этой опции можно изменить перспективу и сосредоточиться только на одном сервисе.
