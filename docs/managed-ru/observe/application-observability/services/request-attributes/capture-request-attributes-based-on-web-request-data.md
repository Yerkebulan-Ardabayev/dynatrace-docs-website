# Захват атрибутов запросов на основе данных веб-запросов

# Захват атрибутов запросов на основе данных веб-запросов

* 4-min read
* Updated on Jul 06, 2023

Чтобы определить атрибут запроса на основе данных веб-запроса:

1. Перейдите в **Settings** > **Server-side service monitoring** > **Request attributes**.
2. Нажмите кнопку **Create new request attribute**.
3. Укажите уникальное **Request attribute name**. Атрибут можно переименовать в любой момент в будущем.
4. Укажите **Data type**.
   Изменить **Data type** после настройки атрибута запроса невозможно.
5. Если в одном запросе существует несколько значений, укажите, что должно храниться в атрибуте запроса для каждого запроса, и выберите способ нормализации текста.
6. Проверьте, должно ли это правило обращаться к немаскированным данным и содержит ли атрибут запроса [конфиденциальные данные](/managed/manage/data-privacy-and-security/configuration/configure-global-privacy-settings#restrict-view-access "Узнайте, как настроить маскировку персональных данных для IP-адресов конечных пользователей, геолокаций и имён действий пользователей.").

   Это потенциально может обращаться к персональным данным.
7. Добавьте источник данных. Можно определить одно или несколько правил, указывающих, как должен извлекаться атрибут.
   Каждому правилу нужен источник. Dynatrace необходимо знать, откуда можно собирать атрибут запроса. Правило можно определить, сначала выбрав, где оно должно применяться (на основе группы процессов, группы хостов, технологии сервиса или тега группы), а затем указав, где именно находится параметр (**Request attribute source**).
8. После добавления всех источников данных выберите **Save** для сохранения правила атрибута запроса.

## Правила атрибутов запросов

Рассмотрим пример правила атрибута запроса ниже. Обратите внимание, что атрибут запроса `destination` может получать своё значение из двух разных источников: из параметра `HTTP Post` (`iceform:destination`) или параметра `HTTP GET` (`destination`). Правила выполняются по порядку. Если запрос соответствует критериям обоих правил, его значение будет взято из первого правила.

Каждому правилу нужен источник. В примере ниже источником атрибута запроса является параметр `HTTP GET` веб-запроса (`destination`).

![Пример определения правила атрибутов запросов](https://dt-cdn.net/images/get-2018-10-30-14-26-29-1373-3d3ff74f1c.png)

Пример определения правила атрибутов запросов

Этот параметр `GET` будет захватываться во всех отслеживаемых процессах, поддерживающих видимость на уровне кода, и будет сообщаться для всех запросов, отслеживаемых Dynatrace.

Хотя это удобно, это не всегда то, что требуется. Именно поэтому правила можно ограничить подмножеством групп процессов и сервисов. Для этого выберите имена групп процессов и сервисов из четырёх раскрывающихся списков выше, чтобы уменьшить количество групп процессов и сервисов, к которым применяется правило.

Может не потребоваться захватывать каждое значение. В других случаях значение может содержать префикс, по которому нужно проверить соответствие. Для этого укажите, что обозначенный параметр должен использоваться только в том случае, если его значение соответствует определённому значению. Также можно не использовать всё значение целиком, а извлечь его часть. Пример ниже настроен так, чтобы учитывать только параметры `HTTP POST` `iceform:destination`, начинающиеся со строки `Journey :`. Этот подход извлечёт всё, что следует за строкой `Journey:`, и сохранит это в атрибуте запроса.

![Пример определения правила атрибутов запросов POST](https://dt-cdn.net/images/post-2018-10-30-14-15-35-1379-6f82cbbf7c.png)

Пример определения правила атрибутов запросов POST

## Источники атрибутов запросов для веб-запросов

Источники данных атрибутов запросов для веб-запросов включают:

* Независимые от технологии источники, такие как:

  + HTTP POST-параметры
  + [IP-адреса клиентов](/managed/observe/digital-experience/rum-concepts/detection-of-ip-addresses-locations-and-user-agents#ip-addresses "Dynatrace определяет IP-адреса и геолокации (город, регион, страну), а также браузеры, устройства и операционные системы.")
    Сообщается значение первого соответствующего заголовка.
  + Заголовки HTTP-запросов и ответов
  + URL веб-запроса или один из его компонентов, например путь или параметр запроса

  В двух последних случаях также можно выбрать сторону веб-запроса, на которой захватывается и хранится атрибут.

  На стороне захвата всегда требуется OneAgent.

  | Опция | Тип запроса | Сторона захвата | Сторона хранения |
  | --- | --- | --- | --- |
  | **Capture on the client side of a web request service and store on the calling service** | Полный вызов сервиса[1](#fn-1-1-def) | Клиент | Сервер |
  | **Capture on server side of a web request service** | Полный вызов сервиса | Сервер | Сервер |
  | **Capture on the client side of a web request service** | Вызов внешнего сервиса[2](#fn-1-2-def) | Клиент | Клиент |
  | **Capture from both client and server side and store where found** | Полный вызов сервиса | Сервер | Сервер |
  | **Capture from both client and server side and store where found** | Вызов внешнего сервиса | Клиент | Клиент |

  1

  Полный вызов сервиса: запрос, полностью отслеживаемый Dynatrace.

  2

  Вызов внешнего сервиса: запрос, обращающийся к внешним ресурсам.
* Источники, зависящие от технологии.

  + Приложение может хранить сложный объект в атрибуте, тогда как OneAgent преобразует его в строку при захвате (например, в [атрибуте сессии Java servlet](https://dt-url.net/q503soq) и [переменной состояния сессии ASP.NET](https://dt-url.net/qf23stx)). Это может иметь побочные эффекты, поэтому следует тщательно выбирать, что захватывать.
  + ASP.NET При наличии нескольких параметров с одинаковым именем будет отображаться только первое захваченное значение.
  + Java servlet Атрибуты запросов и атрибуты сессий Java servlet захватываются только после входа в сервлет или фильтр.

## Постобработка

В большинстве случаев захваченное значение будет содержать именно то, что нужно. Однако может не потребоваться всё значение или каждое значение. С помощью постобработки можно обработать захваченное значение.

Разверните опцию **Optionally restrict or process the captured parameter(s) further**, чтобы увидеть шаги обработки. Шаги выполняются в указанном порядке — каждый шаг применяется к результату предыдущего.

Применять все шаги необязательно. Каждый шаг становится активным после указания значения или выбора флажка.

![Опции постобработки для определения правила атрибута запроса](https://dt-cdn.net/images/post5steps-1353-6e753964a1.png)

Опции постобработки для определения правила атрибута запроса

Шаг 1 позволяет извлечь часть результирующей строки на основе символов-разделителей.

Шаг 2 может разбить захваченное значение на несколько значений на основе символа-разделителя.

Шаг 3 удаляет пробельные символы.

Шаг 4 позволяет отфильтровать захваченные значения, не соответствующие указанному критерию.

Шаг 5 позволяет извлечь часть результирующей строки на основе [регулярного выражения](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Узнайте, как использовать регулярные выражения в контексте Dynatrace.").
