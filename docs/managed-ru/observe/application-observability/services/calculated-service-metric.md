# Вычисляемые метрики для сервисов

# Вычисляемые метрики для сервисов

* How-to guide
* 5-min read
* Published Mar 30, 2021

Dynatrace автоматически собирает важные метрики для сервисов без необходимости настройки. Могут потребоваться дополнительные бизнес- или технические метрики, специфичные для вашего приложения. Эти метрики могут быть вычислены и получены на основе широкого разнообразия доступных данных в перехваченной распределённой трассировке. Также можно разбивать эти метрики по нескольким измерениям, например по атрибуту запроса или HTTP-методу.

## Почему стоит использовать OpenPipeline вместо классических вычисляемых метрик сервисов?

* Более гибкое создание метрик из span'ов — с OpenPipeline метрики больше не ограничены поддержкой максимум 100 значений измерений («top X»).

  + Можно хранить полную кардинальность измерений.
  + Можно свободно комбинировать измерения.
  + Доступны мощные возможности приёма, обработки и преобразования через единое решение (работает одинаково для других областей настройки, таких как логи и события).
* Лучшая масштабируемость и производительность — приём и потоковая обработка данных на уровне платформы, превосходящие петабайтный масштаб; больше нет ограничения в 500 вычисляемых метрик сервисов на тенант.
* Безопасность, управление затратами и управление данными — фильтрация и маскировка конфиденциальных или ненужных полей; назначение использования центра затрат конкретным метрикам; маршрутизация данных в определённые бакеты Grail с контролируемыми сроками хранения.

Ограничения

* В вычисляемые метрики записываются только новые данные; ретроспективные данные не включаются.
* Можно иметь до 500 включённых вычисляемых метрик на окружение и до 100 включённых вычисляемых метрик на сервис.
* Классические вычисляемые метрики поддерживают не более 100 значений измерений. Это называется правилом «top X», поскольку можно выбрать меньше в зависимости от конфигурации. Однако независимо от выбора топ 100 значений измерений, оставшиеся измерения агрегируются в один временной ряд, а значение измерения доступно через специальное измерение `remainder`. Условие фильтрации [remainder](/managed/dynatrace-api/environment-api/metric-v2/metric-selector#remainder "Настройте селектор метрик для Metric v2 API.") позволяет фильтровать по этому измерению `remainder`.

## Создание вычисляемой метрики сервиса

### Шаги

1. Перейдите в **Settings** > **Server-side service monitoring** > **Calculated service metrics** > **Create new metric**.
2. Введите имя для метрики.
   Имя и префикс `calc:service.` автоматически добавляются к ключу метрики.
3. Выберите источник метрики из списка **Metric source**.

   * Если источником является атрибут запроса, выберите необходимую единицу измерения.
   * Необязательно Чтобы исключить вклад данных [заглушённых запросов](/managed/observe/application-observability/services/service-detection/service-detection-v1/service-monitoring-mute "Отключайте мониторинг определённых запросов сервисов, чтобы сосредоточиться на производительности запросов, влияющих на ваших клиентов."), включите **Ignore muted requests**.
4. Необязательно Выберите зону управления. Новая метрика будет ограничена данными из этой зоны.
5. Укажите условия для определения того, какие запросы включаются в расчёт.

   Если указано несколько условий, **все условия должны быть выполнены** для использования метрики.

   1. Выберите **Add condition**.
   2. Выберите атрибут для проверки.
   3. Выберите оператор условия.
   4. При необходимости укажите эталонное значение.

      В предварительном просмотре отображается список сервисов, включаемых в пользовательскую метрику, и оценка [потребления DDU](/managed/license/monitoring-consumption-classic/davis-data-units/metric-cost-calculation "Поймите, как рассчитать потребление Davis Data Unit и расходы, связанные с отслеживаемыми метриками.").

      Предварительный просмотр учитывает только зону управления и условия на основе атрибутов сервиса. Эти атрибуты помечены `[Service property]` в списке атрибутов.
6. Необязательно Добавьте измерение к новой метрике.

   1. Включите **Split by dimension**.
   2. Выберите заполнители для определения измерения.

      * Выберите доступный заполнитель в **Dimension value pattern**.
      * Создайте [пользовательский заполнитель](#placeholder).
   3. Введите имя измерения.
   4. Определите ограничение значений топ *X*.

      1. В поле **Number of top values** укажите количество значений топ *X* для расчёта метрики.
      2. В списках **Value sorting** и **Value aggregation** выберите сортировку и агрегацию значений топ *X*.
7. Проверьте названия источника метрики и измерений. Они будут использоваться в пользовательском интерфейсе и API. После создания метрики их нельзя изменить.
8. Выберите **Save metric**.

## Создание пользовательского заполнителя

Если нужный заполнитель недоступен, можно создать пользовательский. Все пользовательские заполнители должны использоваться в шаблоне значений измерения; в качестве альтернативы можно удалить неиспользуемые пользовательские заполнители.

Предварительные знания

### Метод извлечения

Доступны два метода извлечения значения для заполнителя:

* На основе разделителя. В этом случае Dynatrace извлекает значение, проверяя значение источника по эталонному значению и указанному виду и позиции разделителя.
* На основе регулярного выражения. В этом случае Dynatrace использует извлечение с помощью регулярного выражения. Об использовании регулярных выражений в Dynatrace см. раздел [Регулярные выражения в Dynatrace](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Узнайте, как использовать регулярные выражения в контексте Dynatrace.").

### Метод извлечения для атрибутов запросов

Заполнители, основанные на атрибутах запросов, предоставляют три варианта извлечения значений:

* **First**: для извлечения значения используется первое вхождение атрибута.
* **Last**: для извлечения значения используется последнее вхождение атрибута.
* **Count**: значение равно количеству вхождений атрибута.

### Шаги

Чтобы создать пользовательский заполнитель

1. При создании или редактировании вычисляемой метрики сервиса выберите **Add custom placeholder**.
2. Введите имя для заполнителя. Имя будет использоваться в поле **Dimension value pattern**.
3. Выберите источник для измерения.
4. Выберите [метод извлечения](#prior).

   Если источником является атрибут запроса:

   1. Необязательно Установите флажок **Use from downstream services** для использования дочерних вызовов как источника значения.
   2. Необязательно Ограничьте дочерние вызовы определённой зоной управления или тегом сервиса.
5. Выберите **Add**.
6. Обязательно Используйте только что созданный заполнитель в шаблоне значений измерения.
