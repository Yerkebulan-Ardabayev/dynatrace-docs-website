# Мониторинг сторонних сервисов

# Мониторинг сторонних сервисов

* How-to guide
* 2-min read
* Updated on Feb 21, 2023

Dynatrace захватывает и отслеживает все исходящие запросы от отслеживаемых серверных сервисов. Это включает веб-запросы, которые ваши серверные компоненты отправляют через интернет.

* Большинство из них, вероятно, не важны для вас, поэтому Dynatrace собирает их в специальный сервис под названием **Requests to public networks**.
* Однако некоторые из них, вероятно, *важны* для вас — например, если ваше приложение зависит от производительности и доступности сторонних поставщиков услуг. По этой причине Dynatrace предоставляет способ мониторинга сторонних сервисов с важными HTTP-запросами как самостоятельных сервисов.

Обратите внимание, что в настоящее время функция не поддерживает gRPC-запросы.

## Мониторинг стороннего сервиса как отдельного сервиса

Чтобы настроить Dynatrace для мониторинга стороннего поставщика услуг как отдельного сервиса

1. Перейдите в **Services**.
2. Чтобы найти сервис **Requests to public networks**, поместите курсор в поле **Filter by** и выберите **External services** > **Third party services**.

   ![Сторонний сервис 1](https://dt-cdn.net/images/3rd-party-service-1-1075-9e270ec960.png)

   Сторонний сервис 1
3. На странице **Requests to public networks** выберите **View requests** для получения сведений об агрегированных запросах.
4. Прокрутите вниз до таблицы **top domains** и выберите домен, который нужно отслеживать как отдельный сервис.
   Страница теперь отфильтрована по этому доменному имени.
5. В правом верхнем углу страницы выберите **More** (**…**) > **Monitor as separate service**.

   ![Сторонний сервис 2](https://dt-cdn.net/images/3rd-party-service-2-1178-b6dda96df9.png)

   Сторонний сервис 2

   В нижней части страницы отображается сообщение с подтверждением.

Запросы к данному домену отслеживаются как самостоятельный сервис. Этот сервис:

* Отображается на странице **Services** как самостоятельный сервис.
* Отображается в Smartscape, Service flow и других местах как самостоятельный сервис. Это позволяет лучше отслеживать время отклика.
* Может быть закреплён на дашбордах в виде выделенной плитки.
* Генерирует оповещения при возникновении проблем с производительностью сервисов.

Только новые данные отображаются в самостоятельном сервисе. Существующие данные остаются нетронутыми в сервисе **Requests to public networks**.

## Возврат к статусу стороннего сервиса

Если больше не нужен перекатегоризированный сторонний сервис в качестве самостоятельного, его можно вернуть к статусу стороннего.

1. Перейдите в **Services**.
2. Найдите и откройте нужный сервис.
3. Выберите **More** (**…**) > **Revert to 3rd party service status**.

   ![Сторонний сервис 3](https://dt-cdn.net/images/3rd-party-service-3-842-21ac60233d.png)

   Сторонний сервис 3

## API правил обнаружения сервисов

В качестве альтернативы можно настроить пользовательское правило обнаружения сервисов, которое будет сопоставлять определённые запросы с самостоятельным сервисом. Дополнительные сведения см. в разделе [API правил обнаружения сервисов](/managed/dynatrace-api/configuration-api/service-api/detection-rules "Узнайте, что предлагает Dynatrace API конфигурации правил обнаружения сервисов."). Вам потребуется тип правил **opaque web requests**.
