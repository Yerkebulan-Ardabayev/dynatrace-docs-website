# Настройка обнаружения сбоев сервиса

* How-to guide
* 9-min read
* Updated on Mar 30, 2026

Обнаружение сбоев Dynatrace автоматически выявляет подавляющее большинство условий ошибок в вашем окружении, включая лежащие в их основе [первопричины](/managed/dynatrace-intelligence/root-cause-analysis/concepts "Ознакомьтесь с концепциями анализа первопричин."). Такой подход позволяет Dynatrace давать ответы при возникновении проблем или снижении производительности приложения.

Когда Dynatrace обнаруживает ошибку сервиса, это не обязательно означает, что запрос следует помечать как неудачный. Если настройки обнаружения ошибок сервиса по умолчанию не соответствуют вашим потребностям, можно настроить параметры обнаружения сбоев, как описано на этой странице.

По умолчанию Dynatrace обнаруживает:

* Программные исключения (Java, .NET, Node.js и PHP) как причину неудачных запросов, когда исключения приводят к прерыванию вызовов сервисов.
* Страницы ошибок, предоставляемые многими веб-контейнерами для обработанных исключений.
* Коды ошибок HTTP `500`–`599` для веб-запросов, интерпретируемых как ошибки на стороне сервера.
* Коды ошибок HTTP `400`–`599` для веб-запросов, интерпретируемых как ошибки на стороне клиента.

Почему коды 5xx включены в клиентскую сторону?

Обнаруживаемые коды ошибок зависят от перспективы:

* С точки зрения сервера, только код `5xx` является ошибкой, поскольку код `4xx` означает ошибку клиента.
* С точки зрения клиента, `5xx` всё равно означает ошибку, даже если это не вина клиента.

## Настройки обнаружения сбоев

Обнаружение сбоев можно настроить глобально или для отдельных сервисов.

* При настройке параметров обнаружения сбоев на уровне сервиса они переопределяют глобальные настройки.
* Правила обнаружения сбоев оцениваются сверху вниз; применяется первое совпавшее правило. Если несколько правил обнаружения сбоев имеют одинаковые условия, применяется только первое совпавшее правило.

Следующие правила обнаружения сбоев не применяются к типам сервисов:

* Span service
* Unified service

  Подробнее об обнаружении сбоев для unified service см. в разделе [Unified services](/managed/observe/application-observability/services/service-detection/service-detection-v1/service-types/unified-service "Определяйте сервисы на основе сигналов наблюдаемости, принятых через Trace ingest API.").

Глобальные настройки

Настройки на уровне сервиса

Чтобы настроить обнаружение сбоев сервиса глобально

1. Перейдите в **Settings** и разверните **Server-side service monitoring**.
2. Чтобы добавить новое правило обнаружения сбоев, перейдите в **Failure detection rules** > **Add failure detection rule**.

   Каждое правило может иметь несколько условий на основе определённых параметров обнаружения сбоев. Для правил обнаружения сбоев можно использовать существующие параметры или создавать новые.

   * Чтобы добавить новый параметр обнаружения сбоев, перейдите в **Failure detection parameters** > **Add failure detection parameters**. Подробнее см. в разделах [Параметры](#parameters), [HTTP-параметры](#http) и [Общие параметры](#general) ниже.
   * Используйте переключатель **Enabled** для включения или отключения правила.
   * Dynatrace пересчитывает соответствие глобальным правилам каждые 10 минут.

Чтобы настроить обнаружение сбоев для конкретного сервиса

1. Перейдите в **Services** и выберите сервис, для которого нужно адаптировать обнаружение сбоев.
2. Выберите **More** (**…**) > **Settings**.
3. Выберите **Failure detection**, а затем **HTTP parameters** или **General parameters** в зависимости от параметров, которые нужно настроить.
4. Включите **Override global failure detection settings** (см. **1** на рисунке).
   Если применяется правило обнаружения сбоев, отсюда можно получить доступ как к правилу (см. **3** на рисунке), так и к параметрам (см. **2** на рисунке).

## Параметры

Параметры обнаружения сбоев включают HTTP-специфические параметры и общие параметры (связанные с исключениями, пользовательскими ошибками и обнаружением сбоев span). Хотя общие параметры всегда доступны как глобально, так и на уровне сервиса, HTTP-параметры обнаружения сбоев на уровне сервиса могут быть недоступны, поскольку они видны только для определённых сервисов, таких как веб-запросы и веб-сервисы. Их можно настроить после включения **Override global failure detection settings** (**1** на рисунке), даже если ни одно глобальное правило не соответствует сервису.

![Обнаружение сбоев — HTTP-параметры](https://dt-cdn.net/images/failure-detection-http-parameters-1791-26675df666.png)

Обнаружение сбоев — HTTP-параметры

### HTTP-параметры

* **HTTP response codes**

  Коды ответов `HTTP-4XX` обычно указывают на ошибки на стороне клиента, а не на стороне сервера. Можно указать, какие отсутствующие HTTP-коды ответов должны считаться ошибками на стороне сервера, а какие — на стороне клиента. Можно определить несколько диапазонов, разделённых запятыми (например, `400-402, 405-417`).

  В зависимости от приложения отсутствующие коды ответов могут указывать на вызов «fire and forget», который вообще не вернул ответ, на тайм-аут или на ситуацию ошибки. Dynatrace считает отсутствующие коды ответов особыми случаями и не сообщает о них по умолчанию. Это можно изменить, включив **Treat missing HTTP response code as server side errors** или **Treat missing HTTP response code as client side errors**.
* **HTTP 404 - broken link configuration**

  Когда веб-сервер не может найти определённую страницу, он возвращает код ответа `HTTP 404`. Обычно это указывает на проблему на вызывающей стороне. Если вызывающая сторона принадлежит тому же веб-сайту, это считается неработающей ссылкой.

  Поскольку большинство пользователей не считают неработающие ссылки проблемой на их сервере, Dynatrace классифицирует их как проблемы на стороне клиента, а не как сбои на стороне сервера автоматически. Однако можно включить переключатель **Consider 404 HTTP response codes as failures** для классификации неработающих ссылок как сбоев на стороне сервера. После этого можно связать дополнительные хосты на других доменах с приложением, добавив имя хоста в разделе **Add other application domain**.

### Общие параметры

* **Success forcing exceptions**

  Эти исключения указывают на то, что вызов сервиса не должен считаться неудачным, например, потому что клиент прервал операцию. Хотя они являются техническими ошибками, в принципе они не считаются неудачными запросами, поскольку не вызваны неисправностями сервиса. Если запрос встречает такое исключение в корневом вызове сервиса, Dynatrace считает запрос успешным, независимо от кода ошибки HTTP или любой другой информации. Можно выбрать **Add exception** для добавления классов исключений, указывающих на такие ситуации.
* **Ignored exceptions**

  Бывают ситуации, когда код (или сторонний код, который не контролируется) возвращает исключения, указывающие на определённый ответ, а не на ошибку. Например, клиент Thrift для Cassandra возвращает ответ `NotFoundException`, когда строка не найдена. Это не ошибка, а просто код ответа.

  Можно выбрать **Add exception** для настройки Dynatrace таким образом, чтобы не считать такие исключения индикаторами неудачных запросов. Дополнительно можно определить строку, которая должна быть найдена в сообщении исключения для его игнорирования. Если код HTTP-ответа для того же вызова показывает ошибку, Dynatrace считает запрос неудачным. Чтобы считать запрос успешным независимо от кода ошибки HTTP или любой другой информации, см. **Success forcing exceptions**.
* **Custom handled exceptions**

  Бывают ситуации, когда код приложения корректно обрабатывает исключения способом, который Dynatrace не обнаруживает автоматически. В таких случаях Dynatrace не обнаруживает неудачные запросы и не уведомляет об ошибках.

  Такие ситуации можно исправить, указав класс исключения, который должен приводить к неудачному запросу. При необходимости можно определить строку, которая должна быть найдена в сообщении исключения. Если эта строка не найдена, исключение не приведёт к неудачному запросу.
  Если Dynatrace обнаружит определённое исключение (и опционально определённое сообщение исключения) в запросе, он пометит соответствующий запрос к сервису как неудачный.
  Обратите внимание, что это не работает, если класс исключения исключён из захвата в [настройках глубокого мониторинга процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-monitoring#deep-monitoring "Способы настройки мониторинга группы процессов").
* **Ignore all exceptions**

  Когда **Ignore all exceptions** включено, Dynatrace игнорирует **Success forcing exceptions**, **Ignored Exceptions** и **Custom handled exceptions** для сервисов, к которым применяются параметры — конкретного сервиса, если переключатель включён на уровне сервиса, или сервисов, соответствующих глобальному правилу.
  Поскольку исключения всё равно отслеживаются, они появляются в распределённых трассировках, но оповещений о них не поступает и запросы не помечаются как неудачные.
* **Custom errors via request attributes**

  Пользовательские ситуации ошибок могут запускаться исключениями, но некоторые обнаруживаются только через возвращаемое значение или другие средства. Для поддержки таких случаев можно определить [атрибут запроса](/managed/observe/application-observability/services/request-attributes "Поймите, что такое атрибуты запросов, и узнайте, как использовать их на всех уровнях всех представлений анализа сервисов."), захватывающий необходимые данные. Затем можно определить пользовательское правило ошибки на основе атрибута запроса, которое проверяет его значение для определения того, завершился ли запрос неудачей.

  Пример: использование атрибутов запросов для обнаружения ошибок бизнес-логики

  Запросы могут завершаться с ошибкой по причинам, связанным с бизнес-логикой. Такие ситуации часто не обнаруживаются через исключения или HTTP-коды ответов. Тем не менее они указывают на проблемы и могут быть даже важнее ситуаций, обнаруженных через исключения и коды ответов. Например, может быть бизнес-функция в Java-коде, указывающая на ошибку через возвращаемое значение, или собственная функциональность обработки ошибок, которая при вызове указывает на функциональную бизнес-ошибку.

  Такие ситуации можно захватывать через [атрибуты запросов](/managed/observe/application-observability/services/request-attributes "Поймите, что такое атрибуты запросов, и узнайте, как использовать их на всех уровнях всех представлений анализа сервисов."), которые можно использовать как индикаторы ситуаций ошибок.

  Чтобы создать пользовательское правило ошибки

  1. Перейдите в **Services**.
  2. Выберите сервис, для которого нужно адаптировать обнаружение сбоев.
  3. Выберите **More** (**…**) > **Settings**.
  4. Выберите **Failure detection** > **General parameters**.
  5. В разделе **Custom error rules** выберите **Add custom error rule**.
  6. Выберите атрибут запроса из отображаемого списка.
  7. Определите условие для правила, например **contains** и значение.

  В примере ниже значение `-1` в атрибуте `Amount of recommendations` указывает на ошибку. Согласно этому правилу, если Dynatrace обнаружит такую ошибку, он пометит соответствующий запрос к сервису как неудачный и объяснит, что совпадение с правилом является причиной сбоя.

  ![Пользовательское правило ошибки](https://dt-cdn.net/images/custom-error-rule-2908-978f284947.png)

  Пользовательское правило ошибки
* **Span failure**

  Обнаружение сбоев span специфично для OpenTelemetry. По умолчанию Dynatrace обнаруживает сбои span, но в некоторых конкретных случаях может потребоваться изменить эти настройки. Чтобы игнорировать обнаружение сбоев span, включите **Ignore span failure detection**.

## Сведения о схеме

На уровне сервиса можно визуализировать **Schema ID**, выбрав **More** (**…**) > **Schema info** в правом верхнем углу страницы **HTTP parameters** или **General parameters**.

## Справочник по Settings API

Также можно настроить параметры обнаружения сбоев через [Settings 2.0 API](/managed/dynatrace-api/environment-api/settings "Узнайте, что предлагает Dynatrace Settings API."). Используйте следующие два идентификатора схемы:

| ID схемы | Описание |
| --- | --- |
| `builtin:failure-detection.environment.parameters` | Определяет наборы параметров обнаружения сбоев (HTTP-коды ответов, неработающие ссылки, правила исключений). |
| `builtin:failure-detection.environment.rules` | Определяет правила, назначающие наборы параметров сервисам на основе условий. |

### Аутентификация

Необходим токен доступа со следующими областями:

| Область | Операции |
| --- | --- |
| `settings.read` | Список и просмотр объектов настроек. |
| `settings.write` | Создание, обновление и удаление объектов настроек. |

Об получении и использовании токена доступа см. в разделе [Dynatrace API — токены и аутентификация](/managed/dynatrace-api/basics/dynatrace-api-authentication "Узнайте, как пройти аутентификацию для использования Dynatrace API.").

### Наборы параметров обнаружения сбоев

Settings 2.0 API поддерживает четыре типа операций для наборов параметров обнаружения сбоев: `GET`, `POST`, `PUT` и `DELETE`.

Получение списка всех наборов параметров

Для получения полного списка всех определённых наборов параметров используйте следующий запрос.

```
GET /api/v2/settings/objects?schemaIds=builtin:failure-detection.environment.parameters&scopes=environment
```

Просмотр набора параметров

Для чтения содержимого одного набора параметров используйте запрос `GET` с его уникальным `objectId`.

```
GET /api/v2/settings/objects/{objectId}
```

Идентификатор должен быть UUID по RFC 4122, назначенным данному объекту настроек набора параметров.

Создание набора параметров

Можно добавить новый набор параметров с помощью запроса `POST` с телом, указывающим необходимые значения.

```
POST /api/v2/settings/objects
```

Пример тела запроса:

```
[



{



"schemaId": "builtin:failure-detection.environment.parameters",



"scope": "environment",



"value": {



"name": "Ignore known database errors",



"httpResponseCodes": {



"serverSideErrors": "500-599",



"failOnMissingResponseCodeServerSide": false,



"clientSideErrors": "401-402,405-499",



"failOnMissingResponseCodeClientSide": false



},



"brokenLinks": {



"http404NotFoundFailures": false



},



"exceptionRules": {



"ignoreAllExceptions": false,



"successForcingExceptions": [],



"ignoredExceptions": [



{



"classPattern": "psycopg2.errors.UniqueViolation",



"messagePattern": "duplicate key value violates unique constraint \"run_once_at_unique_identifier_and_timestamp\""



}



],



"customHandledExceptions": [],



"customErrorRules": [],



"ignoreSpanFailureDetection": false



}



}



}



]
```

Для пользовательских правил ошибок каждая запись включает [атрибут запроса](/managed/observe/application-observability/services/request-attributes "Поймите, что такое атрибуты запросов, и узнайте, как использовать их на всех уровнях всех представлений анализа сервисов.") и `condition`:

```
{



"requestAttribute": "<request-attribute-object-id>",



"condition": {



"compareOperationType": "STRING_EQUALS",



"textValue": "error",



"caseSensitive": false



}



}
```

Тело ответа возвращает новый `objectId` и может выглядеть следующим образом:

```
[



{



"code": 200,



"objectId": "vu9U3hXa3q0AAAABADBidWlsdGluOmZhaWx1cmUtZGV0ZWN0aW9uLmVudmlyb25tZW50LnBhcmFtZXRlcnMABnRlbmFudAAGdGVuYW50ACRlNWEwYjAyYi0zMDNlLTM2ZTEtOTMyNS0wMjM1YWNhMDc5MmO-71TeFdrerQ"



}



]
```

Затем этот ID можно использовать напрямую для создания или обновления правила обнаружения сбоев.

Обновление набора параметров

Для изменения существующего набора параметров используйте тело запроса, аналогичное показанному в разделе **Create**. В URL укажите действительный ID объекта настроек.

```
PUT /api/v2/settings/objects/{objectId}
```

Используйте ту же структуру `value`, что показана в разделе **Create**.

Удаление набора параметров

Для удаления существующего набора параметров сначала получите его `objectId`, затем выполните запрос `DELETE`, указывающий этот ID.

```
DELETE /api/v2/settings/objects/{objectId}
```

Дополнительные сведения об опциональных и обязательных параметрах, а также подробную информацию о типах см. в [таблице схемы параметров обнаружения сбоев Settings API](/managed/dynatrace-api/environment-api/settings/schemas/builtin-failure-detection-environment-parameters "Просмотрите таблицу схемы настроек builtin:failure-detection.environment.parameters вашего окружения мониторинга через Dynatrace API.").

### Правила обнаружения сбоев

Правила обнаружения сбоев оцениваются сверху вниз; применяется первое совпавшее правило.

Получение списка всех правил

Для получения всех текущих определённых правил обнаружения сбоев используйте следующий запрос.

```
GET /api/v2/settings/objects?schemaIds=builtin:failure-detection.environment.rules&scopes=environment
```

Просмотр правила

Используйте запрос `GET` с `objectId` правила как параметром пути для чтения одного правила обнаружения сбоев.

```
GET /api/v2/settings/objects/{objectId}
```

Создание правила

Можно добавить новое правило обнаружения сбоев с помощью запроса `POST` с телом, указывающим необходимые атрибуты.

```
POST /api/v2/settings/objects
```

Поле `parameterId` должно содержать ID объекта существующего набора параметров. По умолчанию новые правила добавляются в конец списка правил. Чтобы вставить правило на определённую позицию, установите `insertAfter` в ID объекта предшествующего правила или в пустую строку для размещения в начале списка правил.

Пример тела запроса:

```
[



{



"schemaId": "builtin:failure-detection.environment.rules",



"scope": "environment",



"value": {



"name": "Ignore known database errors for automation-server",



"enabled": true,



"parameterId": "<object-id-of-parameter-set>",



"conditions": [



{



"attribute": "SERVICE_NAME",



"predicate": {



"predicateType": "STRING_EQUALS",



"textValues": [



"automation-server"



],



"caseSensitive": true



}



}



]



}



}



]
```

Обновление правила

Для изменения существующего правила используйте ту же структуру тела запроса, что показана в разделе **Create**, и укажите ID объекта настроек в URL.

```
PUT /api/v2/settings/objects/{objectId}
```

Используйте ту же структуру `value`, что показана в разделе **Create**.

Удаление правила

Для удаления существующего правила сначала получите его `objectId`, затем используйте запрос `DELETE`, указывающий этот ID.

```
DELETE /api/v2/settings/objects/{objectId}
```

Изменение порядка правил

Для изменения порядка оценки правил обнаружения сбоев обновите каждое правило, установив поле `insertAfter` или `insertBefore` в ID объекта правила, которое должно предшествовать или следовать за ним. Чтобы разместить правило в начале списка, оставьте `insertAfter` пустым. Чтобы переместить его в конец списка, добавьте пустой атрибут `insertBefore`.

```
[



{



"schemaId": "builtin:failure-detection.environment.rules",



"scope": "environment",



"value": {



"name": "New top rule",



"...": "...",



"parameterId": "<object-id-of-a-parameter-set>",



"...": "..."



},



"insertAfter": ""



}



]
```

Дополнительные сведения о допустимых типах и доступных параметрах, включая поддерживаемые атрибуты условий, см. в [таблице схемы правил обнаружения сбоев Settings API](/managed/dynatrace-api/environment-api/settings/schemas/builtin-failure-detection-environment-rules "Просмотрите таблицу схемы настроек builtin:failure-detection.environment.rules вашего окружения мониторинга через Dynatrace API.").

### Атрибуты условий

Каждое правило обнаружения сбоев содержит список условий. Для применения правила должны быть выполнены все условия. Каждое условие проверяет атрибут сервиса или группы процессов с помощью предиката.

| Атрибут | Описание | Типы предикатов |
| --- | --- | --- |
| `SERVICE_NAME` | Соответствует обнаруженному имени сервиса. | `STRING_EQUALS`, `STARTS_WITH`, `ENDS_WITH`, `CONTAINS` |
| `SERVICE_TYPE` | Соответствует типу сервиса. | `SERVICE_TYPE_EQUALS` |
| `PG_NAME` | Соответствует имени группы процессов. | `STRING_EQUALS`, `STARTS_WITH`, `ENDS_WITH`, `CONTAINS` |
| `PG_TAG` | Соответствует тегам группы процессов. | `TAG_EQUALS`, `TAG_KEY_EQUALS` |
| `SERVICE_MANAGEMENT_ZONE` | Соответствует зонам управления сервисом. | `MANAGEMENT_ZONES_CONTAINS_ALL` |
| `SERVICE_TAG` | Соответствует тегам сервиса. | `TAG_EQUALS`, `TAG_KEY_EQUALS` |

Для окружений с полным опытом работы с последней версией Dynatrace атрибуты условий `SERVICE_MANAGEMENT_ZONE` и `SERVICE_TAG` больше недоступны, а правила обнаружения сбоев, использующие их, игнорируются во время выполнения.

Кроме того, атрибут условия `PG_TAG` больше не оценивается. Вместо этого это поле можно использовать для сопоставления на основе основных тегов, определённых в развёртывании приложения.
Входные данные, начинающиеся с `primary_tags.`, сопоставляются с основными тегами входных данных.
