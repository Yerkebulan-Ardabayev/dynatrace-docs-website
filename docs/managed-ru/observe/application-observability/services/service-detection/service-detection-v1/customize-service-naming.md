# Правила именования сервисов

* How-to guide
* 4-min read
* Updated on Apr 09, 2026

Dynatrace [автоматически обнаруживает и именует серверные сервисы приложений](/managed/observe/application-observability/services/service-detection/service-detection-v1 "Узнайте, как Dynatrace Service Detection v1 обнаруживает и именует различные типы сервисов.") на основе базовых свойств развёртывания и конфигурации приложения. Встроенные правила определяют эти имена. Эти свойства и результирующие имена сервисов должны быть интуитивно понятны, поскольку отражают ваш ландшафт сервисов. Тем не менее может потребоваться настройка этих имён. Пользовательские правила именования сервисов позволяют улучшить автоматическое именование сервисов.

## Встроенные правила

Встроенные правила именования сервисов определяют, как именование сервисов работает «из коробки». Встроенные правила не настраиваются. К ним можно обратиться для ознакомления. Перейдите в **Settings** > **Server-side service monitoring** > **Service naming rules** и прокрутите до раздела **Built-in rules**. Здесь можно развернуть каждое правило для просмотра его конфигурации.

## Пользовательские правила

Пользовательские правила именования сервисов переопределяют встроенные правила, позволяя создавать собственные стандарты именования. Для доступа к пользовательским правилам перейдите в **Settings** > **Server-side service monitoring** > **Service naming rules**.

Пользовательские правила оцениваются сверху вниз, и применяется первое совпавшее правило, поэтому убедитесь, что правило размещено на правильной позиции в списке.

Чтобы определить пользовательское правило именования сервисов

1. Перейдите в **Settings** > **Server-side service monitoring** > **Service naming rules**.
2. Выберите **Add a new rule**.
3. Введите **Rule name**.
4. Определите **Service name format**, включая любую статическую текстовую строку для описания именуемого сервиса. Можно использовать заполнители для динамического включения конкретных свойств сервиса и группы процессов в автоматическое имя сервиса.
5. Необязательно Рассмотрите ограничение этого правила именования конкретной группой процессов, технологией или типом сервиса.
   Хотя этот шаг необязателен, он обеспечивает быстрый способ уменьшения количества сервисов, к которым применяется правило. Для дальнейшего сужения списка применимых сервисов можно добавить **Conditions** к новому правилу.

   Если в формате имени есть ссылка на свойство, которое не существует в сервисе, к которому применяется правило, заполнитель будет заменён пустой строкой. Чтобы убедиться, что правило применяется только к сервисам, где свойство существует, добавьте условие `exists` для нужного свойства.
6. Выберите **Save**.

## Формат имени сервиса

Формат имени сервиса позволяет строить сложные стандарты именования для сервисов в вашем окружении. Для построения имени на основе свойств сервиса можно использовать заполнители. Заполнители будут заменены фактическими значениями в имени сервиса. Если указанное значение отсутствует, заполнитель будет преобразован в пустую строку. Поместите курсор в поле ввода **Service name format** для просмотра списка доступных заполнителей.

Можно использовать [регулярные выражения](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Узнайте, как использовать регулярные выражения в контексте Dynatrace.") для извлечения частей имени сервиса, созданного встроенным правилом. Добавьте регулярное выражение перед закрывающей фигурной скобкой `}` заполнителя. Например, для `{ProcessGroup:DetectedName}` используйте `{ProcessGroup:DetectedName/REGEX}`.
