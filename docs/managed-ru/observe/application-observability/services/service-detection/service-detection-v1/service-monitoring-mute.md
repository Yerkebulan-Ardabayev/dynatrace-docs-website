# Отключение мониторинга запросов сервисов

* How-to guide
* 4-min read
* Published Jan 29, 2021

Dynatrace отслеживает все серверные запросы от начала до конца. При снижении производительности или успешности конкретного запроса Dynatrace поднимает оповещение. Однако не все высоконагруженные запросы важны и не все медленные запросы должны вызывать оповещения.

Хорошим примером таких запросов являются heartbeat-запросы. Они обычно приходят в большом объёме и с высокой частотой. Эти запросы могут искажать медианное время отклика сервисов. Такие запросы могут вызывать оповещения о времени отклика, отвлекающие от производительности важных запросов, действительно влияющих на пользовательский опыт приложений.

Для лучшей фокусировки на производительности запросов, влияющих на клиентов, можно исключить неважные запросы из мониторинга, заглушив их. После заглушения запроса:

* Автоматические оповещения для заглушённого запроса больше не будут генерироваться.
* Заглушённый запрос будет исключён из [диаграмм сервисов](/managed/observe/application-observability/services-classic#charts "Узнайте о классическом мониторинге сервисов в Dynatrace.").

Заглушение запросов или конечных точек для unified services не поддерживается.

## Заглушение запроса

1. Перейдите в **Services**.
2. На странице **Services** выберите сервис, получающий запрос для заглушения.
3. Выберите **View requests**.
4. Выберите запрос для заглушения.
5. На странице **Details** запроса выберите **More** (**…**) > **Mute**.

На странице **Details** запроса отображается сообщение **This request has been muted.** В дальнейшем Dynatrace не будет генерировать оповещения для этого запроса и будет записывать его данные отдельно.

Заглушённые запросы сервиса остаются доступными (и даже остаются видимыми в списках запросов ограниченный период времени), но их времена отклика больше не учитываются в общих метриках и диаграммах.

## Настройка референсных периодов для оповещений

В некоторых случаях заглушение высоконагруженного запроса может привести к ложным оповещениям о деградации времени отклика. Это происходит только в редких случаях, а причиной является отображаемый сдвиг медианы и соответствующее изменение в базовом распределении запросов. В таком случае лучше сбросить референсный период для оповещений по этому сервису. Это позволит Dynatrace установить новые базовые показатели производительности, исключающие заглушённый запрос.

Чтобы настроить референсные периоды оповещений для конкретного сервиса

1. Перейдите в **Services**.
2. На странице **Services** выберите сервис, получающий заглушённый запрос.
3. На странице сервиса выберите **More** (**…**) > **Edit**.
4. Выберите **Anomaly detection**.
5. В разделе **Reference period** выберите **Reset**.

## Список и отмена заглушения запросов

Данные заглушённых запросов по-прежнему записываются, но в отдельном месте.

Чтобы вывести список всех заглушённых запросов

1. Перейдите в **Services**.
2. На странице **Services** выберите сервис, получающий заглушённый запрос.
3. В разделе **Muted requests** выберите **View muted requests**.
   Здесь доступны все ранее заглушённые запросы.

Чтобы отменить заглушение запроса

1. Выберите заглушённый запрос.
2. На странице **Details** выберите **More** (**…**) > **Unmute**.

Запрос удаляется из списка заглушённых, новые данные снова записываются в обычном режиме, и оповещения для этого запроса повторно включаются.

## Заглушение запроса через Settings API

Заглушёнными запросами можно управлять через [Settings API](/managed/dynatrace-api/environment-api/settings "Узнайте, что предлагает Dynatrace Settings API.").

Для использования API необходим токен доступа с областями **Read settings** (`settings.read`) и **Write settings** (`settings.write`). О его получении см. в разделе [Создание токена доступа](/managed/dynatrace-api/basics/dynatrace-api-authentication#create-token "Узнайте, как пройти аутентификацию для использования Dynatrace API.").

### Создание конфигурации заглушённого запроса

Эта процедура перезаписывает любую существующую конфигурацию. Если нужно изменить существующую конфигурацию, см. раздел [**Обновление конфигурации заглушённого запроса**](#muted-request-api-update) ниже.

1. Чтобы узнать формат объекта настроек, запросите его схему через вызов [GET a schema](/managed/dynatrace-api/environment-api/settings/schemas/get-schema "Просмотрите схему настроек через Dynatrace API."). ID схемы заглушённых запросов — `builtin:settings.mutedrequests`.
2. Создайте объект JSON для ваших настроек.
   Поскольку заглушённые запросы всегда привязаны к сервису, необходимо указать сервис по его идентификатору сущности Dynatrace. Чтобы узнать идентификатор сущности вашего сервиса, используйте запрос [GET entities list](/managed/dynatrace-api/environment-api/entity-v2/get-entities-list "Просмотрите список отслеживаемых сущностей через Dynatrace API.").

Пример JSON

```
[



{



"schemaId": "builtin:settings.mutedrequests",



"scope": "SERVICE-123456789",



"value": {



"mutedRequestNames": [



"/health"



]



}



}



]
```

3. Используйте конечную точку [POST an object](/managed/dynatrace-api/environment-api/settings/objects/post-object "Создайте или проверьте объект настроек через Dynatrace API.") для отправки конфигурации.

### Обновление конфигурации заглушённого запроса

1. Чтобы узнать формат объекта настроек, запросите его схему через вызов [GET a schema](/managed/dynatrace-api/environment-api/settings/schemas/get-schema "Просмотрите схему настроек через Dynatrace API."). ID схемы заглушённых запросов — `builtin:settings.mutedrequests.service`.
   Поскольку заглушённые запросы всегда привязаны к сервису, необходимо указать сервис по его идентификатору сущности Dynatrace. Чтобы узнать идентификатор сущности вашего сервиса, используйте запрос [GET entities list](/managed/dynatrace-api/environment-api/entity-v2/get-entities-list "Просмотрите список отслеживаемых сущностей через Dynatrace API.").
2. Запросите текущую конфигурацию через вызов [GET objects](/managed/dynatrace-api/environment-api/settings/objects/get-objects "Просмотрите несколько объектов настроек через Dynatrace API.").
3. Создайте JSON для обновления.

   1. Используйте значение **updateToken** из предыдущего шага.
   2. При необходимости измените список запросов в массиве **mutedRequestNames**.
      Пример JSON

      ```
      {



      "updateToken": "vu9U3hXY3q0ATAAkMG",



      "value": {



      "mutedRequestNames": [



      "/health"



      ]



      }



      }
      ```
4. Используйте конечную точку [PUT an object](/managed/dynatrace-api/environment-api/settings/objects/put-object "Редактируйте объект настроек через Dynatrace API.") для отправки конфигурации.
