# Объединённые сервисы

* Практическое руководство
* Чтение: 3 мин
* Обновлено 4 октября 2022 г.

Раздел **Settings** > **Merged service monitoring** устарел.

* Если в вашем окружении есть объединённые сервисы, вы по-прежнему можете открыть эту страницу для их просмотра и редактирования.
* Создать новые объединённые сервисы на этой странице настроек нельзя. Чтобы создать или изменить эквивалент объединённого сервиса с помощью обнаружения на основе правил, см. раздел [Создание правила для агрегации сервисов](#create-merged-service) ниже.

Объединённый сервис, это сервис, который агрегирует несколько [сервисов веб-запросов](/managed/observe/application-observability/services/service-detection/service-detection-v1#web-request-service "Узнайте, как обнаружение сервисов Dynatrace v1 обнаруживает и именует разные типы сервисов.") одной группы процессов, которые выполняют идентичные функции на отдельных узлах кластера. С точки зрения мониторинга производительности эти сервисы веб-запросов фактически идентичны. В Dynatrace объединённый сервис отображается как единый сервис, содержащий все данные всех агрегированных сервисов.

Допустим, у вас есть веб-сервер Apache с несколькими определениями виртуальных хостов (например, `dynatrace.com`, `dynatrace.at` и `dynatrace.pl`). С точки зрения Apache это независимые виртуальные хосты. Поэтому Dynatrace обнаруживает их как отдельные сервисы. Однако для целей мониторинга вам может потребоваться рассматривать эти сервисы как единый объединённый сервис под названием `Dynatrace web page`.

Dynatrace автоматически определяет сервисы, которые можно объединить, и отображает их на странице **Merged service monitoring** (**Settings** > **Server-side service monitoring** > **Merged service monitoring**). Объединить можно только сервисы из этого списка.

## Характеристики существующих объединённых сервисов

* После объединения данные отдельных объединённых сервисов больше нельзя различить: данные мониторинга доступны только в агрегированном виде для всех объединённых сервисов.
* Исходные сервисы объединённого сервиса больше не обновляются как источники данных. Хотя исторические данные остаются доступными (например, для построения графиков), новые данные для исходных сервисов не отслеживаются. Агрегированные данные присваиваются объединённому сервису и связываются с ним.
* Объединённые сервисы относятся к одному типу. Например, они принадлежат одной группе процессов, используют одну и ту же базовую технологию или следуют одному шаблону именования. В объединение сервисов могут входить несколько виртуальных хостов, контекстных корней или портов прослушивания, представляющих одну логическую сущность.
* Объединяются только почти идентичные автономные сервисы веб-запросов одной группы процессов: объединённые сервисы не объединяются с другими объединёнными сервисами.

## Разделение объединённых сервисов

При отделении сервиса от объединённого сервиса («разъединении» сервиса) исторические данные будут доступны только для объединённого сервиса. Все новые собранные данные будут связаны с новым автономным сервисом.

Чтобы отделить сервис от объединённого сервиса

1. Откройте **Settings** > **Server-side service monitoring** > **Merged service monitoring**.
2. Нажмите **Edit** ![Edit](https://dt-cdn.net/images/edit-icon-a083c62c49.svg "Edit") в строке объединённого сервиса, который нужно разделить.
3. Нажмите **Remove** ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") в строке сервиса, который нужно удалить.
4. Нажмите **Update**.

## Удаление объединённого сервиса

При удалении объединённого сервиса все отдельные объединённые сервисы будут разделены и снова станут рассматриваться как автономные сервисы.

Чтобы удалить объединённый сервис:

1. Откройте **Settings** > **Server-side service monitoring** > **Merged service monitoring**.
2. Нажмите **Delete** ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") в строке объединённого сервиса, который нужно удалить.

## Создание правила для агрегации сервисов

Чтобы выполнить задачу, эквивалентную созданию объединённого сервиса, но через обнаружение сервисов на основе правил

1. Откройте **Settings** > **Service Detection**.

   * Выберите **Full web request rules**, если веб-запросы полностью отслеживаются Dynatrace.
   * Выберите **External web request rules** для веб-запросов к внешним ресурсам.
2. Нажмите **Add item** и задайте имя для нового правила.
3. Включите хотя бы один источник идентификатора сервиса из следующих: **Application identifier**, **URL context root** и **Server name**.
4. Необязательно. Правило можно изменить:

   * Назначить management zone
   * Добавить определённые условия
   * Отключить порт
5. Нажмите **Save changes**.

Подробнее о создании, редактировании и удалении правил обнаружения сервисов см. в разделе [Управление обнаружением сервисов на основе правил](/managed/observe/application-observability/services/service-detection/service-detection-v1/customize-service-detection#manage "Используйте правила обнаружения для настройки и улучшения автоматического обнаружения сервисов.").
