# Определение пользовательских сервисов

* Практическое руководство
* 9 минут чтения
* Обновлено 14 июля 2026 г.

Пользовательский сервис, это сервис, у которого точка входа задана вручную, например метод, класс или интерфейс. Пользовательские сервисы можно определять для Java, .NET, Go и PHP. Каждый пользовательский сервис может содержать несколько точек входа.

Методы конкретного класса или реализации интерфейса могут служить точками входа для пользовательского сервиса. Каждый нерекурсивный вызов такого метода представляет собой один запрос к пользовательскому сервису.

Если сервисы приложения не построены на стандартных технологиях, они могут не распознаваться из коробки. Такие технологии можно отслеживать, но их нужно определить как пользовательские сервисы с конкретными точками входа сервиса.

Пользовательские сервисы также можно использовать для микросервисов, которые применяются более крупным приложением и не предоставляются через коммуникационные технологии.

Dynatrace позволяет определить любой метод, класс или интерфейс как точку входа отслеживаемого сервиса. При этом инструментированный метод должен завершиться до остановки всего процесса. Если выход из метода не происходит никогда, данные не передаются.

Разрозненные транзакции

Если транзакции не связаны между собой, создание пользовательского сервиса **не поможет** их сшить. Пользовательский сервис создаст новую точку входа.

## Поведение пользовательского сервиса в SDv1 и SDv2

В [Service Detection v1 (SDv1)](/managed/observe/application-observability/services/service-detection/service-detection-v1 "Find out how Dynatrace Service Detection v1 detects and names different types of services."), каждый пользовательский сервис, это отдельная сущность сервиса.

В [Service Detection v2 (SDv2)](/managed/observe/application-observability/services/service-detection/service-detection-v2 "Find out how to detect, name, and split services from OpenTelemetry and OneAgent spans."), определение пользовательского сервиса не создаёт новую сущность сервиса. Вместо этого выбранный класс или метод становится спаном на существующем сервисе, в рамках которого выполняется код. Так и задумано: SDv2 построен на модели «один сервис на процесс», поэтому внутренняя операция это часть сервиса, а не отдельный сервис.

## Делегирование методов

OneAgent понимает концепцию делегирования методов и записывает только первый вызов в цепочке делегирования. Например, если есть несколько методов, вызывающих друг друга, все эти методы можно спокойно пометить как точки входа, случаи, когда один метод напрямую вызывает другой, записываются как единые запросы.

### Подавление делегирования

OneAgent подавляет делегирование методов **в пределах сервиса**. То есть любые вызовы между двумя методами одного сервиса не записываются. Если нужно захватывать такие вызовы, для этого нужно создать отдельный пользовательский сервис.

## Сервисы Java, .NET и Go

Инструментирование методов Java

В Java метод входа должен завершиться до остановки процесса. Если метод входа, например `main()` или `execute()`, запускающий процесс, завершается только вместе с завершением процесса, OneAgent не может финализировать и отправить данные.

Для Java, .NET и Go можно использовать поиск для выбора точки входа и методов или задать их вручную. Чтобы создать новый пользовательский сервис Java, .NET или Go:

Найти точку входа

Задать точку входа вручную

1. Перейти в **Settings** > **Service detection** > **Custom service detection**.
2. Выполнить одно из следующих действий:

   * Java Выбрать вкладку **Java services** и выбрать **Define Java services**.
   * .NET Выбрать вкладку **.NET services** и выбрать **Define .NET services**.
   * Go Выбрать вкладку **Go services** и выбрать **Define Go services**.
3. Дать сервису понятное название.
4. Выбрать **Find entry point**.
5. Найти и выбрать группу процессов, содержащую точку входа.
6. Выбрать процесс, содержащий точку входа, и выбрать **Continue**.
7. Найти класс, который нужно инструментировать. Ввести название или часть названия для поиска.
8. Выбрать нужный класс и выбрать **Continue**.
9. Определить, как инструментировать класс. Доступны два варианта:

   * **Use the selected class** для инструментирования методов только выбранного класса.
   * Java .NET **Use an implemented interface or superclass** для инструментирования методов в любом интерфейсе или суперклассе в иерархии классов. В этом случае выбрать **Load inheritance**, чтобы загрузить все доступные суперклассы и интерфейсы, а затем выбрать нужный.
10. Выбрать методы, которые нужно инструментировать, и выбрать **Finish**.  
    На странице **Define custom service** отображаются добавленная точка входа и методы.
11. При необходимости добавить ещё точки входа.
12. При необходимости ограничить новый пользовательский сервис определёнными группами процессов. См. раздел [**Restrict a custom service to specific process groups**](#restrict) ниже.
13. Проверить точку входа и методы, которые будут инструментированы.
14. В нижнем правом углу страницы выбрать **Save**.

1. Перейти в **Settings** > **Service detection** > **Custom service detection**.
2. Выполнить одно из следующих действий:

   * Java Выбрать вкладку **Java services** и выбрать **Define Java services**.
   * .NET Выбрать вкладку **.NET services** и выбрать **Define .NET services**.
   * Go Выбрать вкладку **Go services** и выбрать **Define Go services**.
3. Дать сервису понятное название.
4. Выбрать **Define entry point manually**.
5. В поле **Fully qualified class** ввести имя класса, содержащего методы, которые нужно инструментировать.
6. Указать методы для инструментирования одним из двух способов:

   * Ввести полное имя метода вручную.
   * Найти методы в классе и выбрать их:

     1. Выбрать **Search for methods**.
     2. Выбрать группу процессов, содержащую точку входа.
     3. Выбрать методы, которые нужно инструментировать, и выбрать **Finish**.
7. При необходимости добавить ещё методы.
8. При необходимости добавить ещё точки входа.
9. Проверить точку входа и методы, которые будут инструментированы.
10. При необходимости ограничить новый пользовательский сервис определёнными группами процессов. См. раздел [**Restrict a custom service to process groups**](#restrict) ниже.
11. В нижнем правом углу страницы выбрать **Save**.

Неподдерживаемый сценарий

Обратите внимание, что некоторые сценарии не работают в Dynatrace, например сценарий Queue/Worker. Всё, что помещается в `java.util.Queue` и обрабатывается другим рабочим потоком, не видно в PurePath. См. [Oracle java.util Interface Queue﻿](https://dt-url.net/xy03uja).

## Сервисы PHP

Для PHP можно использовать поиск для выбора точки входа и методов или задать их вручную. Чтобы создать новый пользовательский сервис PHP:

Найти точку входа

Задать точку входа вручную

1. Перейти в **Settings** > **Process and contextualize** > **Services** > **Custom service detection**.
2. Выбрать вкладку **PHP services** и выбрать **Define PHP service**.
3. Дать сервису понятное название.
4. Выбрать **Find entry point**.
5. Найти и выбрать группу процессов, содержащую точку входа.
6. Выбрать процесс, содержащий точку входа, а затем выбрать **Continue**.
7. Найти класс, который нужно инструментировать. Ввести название или часть названия для поиска.
8. Выбрать нужный класс, а затем выбрать **Continue**.
9. Выбрать методы, которые нужно инструментировать, и выбрать **Finish**.  
   На странице **Define custom service** отображаются добавленная точка входа и методы.
10. При необходимости добавить ещё точки входа.
11. При необходимости ограничить новый пользовательский сервис определёнными группами процессов. См. раздел [**Restrict a custom service to specific process groups**](#restrict) ниже.
12. Проверить точку входа и методы, которые будут инструментированы.
13. В нижнем правом углу страницы выбрать **Save**.

1. Перейти в **Settings** > **Process and contextualize** > **Services** > **Custom service detection**.
2. Выбрать вкладку **PHP services**, а затем выбрать **Define PHP service**.
3. Дать сервису понятное название.
4. Выбрать **Define entry point manually**.
5. В поле **File name** ввести полный путь к файлу PHP, содержащему методы, которые нужно инструментировать.
6. В поле **Fully qualified class name** ввести имя класса, содержащего методы, которые нужно инструментировать.
7. В поле **Methods** ввести имя метода, который нужно инструментировать.
8. При необходимости выбрать **Add method manually** и указать дополнительные методы для инструментирования.
9. При необходимости ограничить новый пользовательский сервис определёнными группами процессов. См. раздел [**Restrict a custom service to specific process groups**](#restrict) ниже.
10. После добавления всех методов выбрать **Save** в верхнем правом углу страницы.

### Пример: определение пользовательского сервиса PHP через файл

Следующий пример показывает, как определить пользовательский сервис на основе файла `.php`. Также показано, как результат обнаружения пользовательского сервиса отображается в **Distributed Traces**.

```
<?php



class InternalExample {



public function say($what) {



echo $what;



}



}



class Example {



public $internalExample;



public function __construct() {



$this->internalExample = new InternalExample();



}



public function run() {



$this->internalExample->say("Hello world!\n");



}



}



$example = new Example();



$example->run();
```

В приведённом выше скрипте объект класса `Example` создаёт объект класса `InternalExample` и вызывает его, чтобы напечатать `Hello world!`. В этом примере нужно трассировать печать `Hello world!` и создание класса `Example`.

Теперь воспользуемся файлом `index.php`, чтобы определить пользовательский сервис, следуя шагам из раздела [Указать точку входа вручную](#manual-php-service) выше. В файле можно инструментировать конкретные функции следующим образом:

* Можно инструментировать функцию `say` из класса `InternalExample`:

  ![Instrument the say function](https://dt-cdn.net/images/php-custom-service-example-1-1091-b7b92d31ae.png)

  Инструментирование функции say
* Можно инструментировать функцию `run` из класса `Example`. Дополнительно инструментируем функцию `__construct` класса `Example`, чтобы показать его создание в трассировке.

  ![Instrument the construct function](https://dt-cdn.net/images/php-custom-service-example-2-1087-65aa6d790c.png)

  Инструментирование функции construct

В обоих примерах в поле **File name** используется опция **Match**, чтобы найти файл `index.php` по локальному пути. Использование локальных путей в определениях пользовательских сервисов полезно, поскольку некоторые PHP-фреймворки изменяют рабочую директорию выполнения скрипта. При указании имён классов и методов важен регистр, но перечислять параметры методов не нужно.

## Определение пользовательского сервиса Go через API

Чтобы определить пользовательский сервис Go через API:

1. Подготовить HTTP-запрос

1. Определить область видимости (`className`), пакет или тип, в котором определён метод.

   В примерах ниже область видимости, это `main.(*CronJob)`, потому что функция определена как метод типа `CronJob` в пакете `main`.
2. Определить полное имя метода (`methodName`), имя метода с префиксом области видимости.

В примере тела запроса ниже полное имя метода, это `main.(*CronJob).Run`. Пример содержит все обязательные параметры. Чтобы использовать его, нужно скорректировать значения под свой запрос.

```
{



"name": "Cron Job Service",



"enabled": true,



"rules": [



{



"enabled": true,



"className": "main.(*CronJob)",



"methodRules": [



{



"methodName": "main.(*CronJob).Run",



"returnType": ""



}



]



}



],



"queueEntryPoint": false



}
```

2. Отправить POST-запрос на свой endpoint

* Использовать `go` в качестве технологии.
* Использовать свой access token или personal access token для аутентификации.

```
curl --request POST 'https://{your-environment-id}.live.dynatrace.com/api/config/v1/service/customServices/go' \



--header 'Authorization: Api-Token dt0s01.ST2EY72KQINMH574WMNVI7YN.G3DFPBEJYMODIDAEX454M7YWBUVEFOWKPRVMWFASS64NFH52PX6BNDVFFM572RZM' \



--header 'Content-Type: application/json' \



--data '{



"name": "Cron Job Service",



"enabled": true,



"rules": [



{



"enabled": true,



"className": "main.(*CronJob)",



"methodRules": [



{



"methodName": "main.(*CronJob).Run",



"returnType": ""



}



]



}



],



"queueEntryPoint": false



}'
```

При успехе возвращается `id` вновь созданного пользовательского сервиса, при ошибке возвращается описательное сообщение об ошибке. С этим `id` можно редактировать пользовательский сервис через API.

```
{



"id": "114ff497-be8a-44c7-a990-35bb4267b677",



"name": "Cron Job Service"



}
```

## Замыкания в Go

Замыкания, это анонимные функции, которым компилятор Go присваивает специальные имена.

Сгенерированные имена замыканий, определённых в области видимости, нумеруются по порядку появления: `func1`, `func2` и так далее. Например, замыкание в функции `main` получает метку `func1`:

```
package main



func main() {



go func() {



// ...



}()



}
```

## Приоритет пользовательских сервисов

Если определено несколько пользовательских сервисов, оценка идёт сверху вниз, применяется первое совпадающее правило. Если по какой-то причине один и тот же класс и метод определены в нескольких пользовательских сервисах, нужно расставить приоритеты сервисов соответствующим образом.

## Заголовок x-dynatrace

Для HTTP-запросов Dynatrace использует дополнительный HTTP-заголовок `x-dynatrace` для сшивания транзакций. Этот заголовок устанавливается OneAgent между веб-серверами, чтобы связывать их друг с другом. Технология PurePath® для распределённой трассировки в значительной степени полагается на такие заголовки, поэтому важно убедиться, что сетевые компоненты, такие как файрволы и маршрутизаторы, никогда не настроены на удаление этих заголовков. Неправильная настройка сетевых компонентов может привести к разрыву pure path. Кроме того, некоторые сетевые компоненты полностью блокируют такие запросы (и возвращают ошибку HTTP 403), поскольку дополнительный заголовок может считаться небезопасным. В таких случаях нужно настроить эти компоненты на приём заголовка `x-dynatrace`.

## Редактирование пользовательского сервиса

Пользовательский сервис можно редактировать в любой момент. Чтобы изменения вступили в силу, нужно перезапустить затронутые процессы, если не активированы обновления в реальном времени для Java и PHP. Для .NET процесс нужно перезапустить обязательно.

Чтобы отредактировать пользовательский сервис, нужно выбрать кнопку **Edit** сервиса в списке сервисов.

Можно активировать/деактивировать существующие точки входа, добавлять/удалять точки входа, добавлять/удалять методы в точках входа.

Также можно ограничить пользовательский сервис определёнными группами процессов. См. раздел [**Ограничение пользовательского сервиса определёнными группами процессов**](#restrict) ниже.

### Обновления в реальном времени

Обновления пользовательских сервисов Java и PHP можно применять почти в реальном времени, без перезапуска процессов. Чтобы активировать эту функцию, нужно перейти в **Settings** > **Server-side service detection** > **Deep monitoring** > **Real-time updates to Java and PHP services** и включить соответствующие переключатели.

С этой функцией возможности мониторинга сервисов и request attributes добавляются во время пауз существующей сборки мусора (Garbage Collection) в приложении, а не только при запуске процесса. Это может увеличить паузы Garbage Collection и замедления. Рекомендуется ограниченное использование в чувствительных ко времени средах, поскольку Dynatrace не управляет паузами Garbage Collection, предоставляемыми и реализуемыми виртуальными машинами.

## Ограничение пользовательского сервиса определёнными группами процессов

Использование любого пользовательского сервиса можно ограничить определёнными группами процессов. Правила пользовательских сервисов будут применяться только в указанных группах процессов и будут игнорироваться в остальных группах процессов. Пользовательский сервис можно ограничить при создании или отредактировать позже.

Чтобы ограничить пользовательский сервис:

1. На странице **Define custom service** или **Edit service** нужно раскрыть **Optionally restrict custom service rules by process groups**.
2. Выбрать **Add process group**.
3. Выбрать группу процессов, в которой нужно применить пользовательский сервис.
4. Выбрать **Add**.
5. В нижнем правом углу страницы **Define custom service** или **Edit service** выбрать **Save**.
