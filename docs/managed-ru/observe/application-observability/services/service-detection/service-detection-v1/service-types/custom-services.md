# Определение пользовательских сервисов

# Определение пользовательских сервисов

* How-to guide
* 9-min read
* Updated on Sep 17, 2025

Пользовательский сервис — это сервис с вручную определённой точкой входа, такой как метод, класс или интерфейс. Пользовательские сервисы можно определять для Java, .NET, Go и PHP. Каждый пользовательский сервис может содержать несколько точек входа.

Методы конкретного класса или реализации интерфейса могут служить точками входа для пользовательского сервиса. Каждый нерекурсивный вызов такого метода представляет собой один запрос к пользовательскому сервису.

Если сервисы приложения построены не на стандартных технологиях, они могут не распознаваться «из коробки». Такие технологии можно отслеживать, но необходимо определить их как пользовательские сервисы с конкретными точками входа.

Пользовательские сервисы также можно использовать для микросервисов, применяемых более крупным приложением и не предоставляемых через технологии коммуникации.

Dynatrace позволяет определить любой метод, класс или интерфейс в качестве точки входа для отслеживаемого сервиса. Однако инструментируемый метод должен завершиться прежде, чем весь процесс завершит работу. Если метод никогда не выходит, данные не сообщаются.

Несвязанные транзакции

Если транзакции не связаны, создание пользовательского сервиса **не поможет** их объединить. Пользовательский сервис создаёт новую точку входа.

## Делегирование методов

OneAgent понимает концепцию делегирования методов и записывает только первый вызов в цепочке делегирования. Например, если есть несколько методов, вызывающих друг друга, можно безопасно пометить все эти методы как точки входа — случаи прямого вызова одного метода другим записываются как единичные запросы.

### Подавление делегирования

OneAgent подавляет делегирование методов **на уровне сервиса**. То есть вызовы между двумя методами одного сервиса не записываются. Если нужно захватывать такие вызовы, необходимо создать для них отдельный пользовательский сервис.

## Сервисы Java, .NET и Go

Инструментирование методов Java

В Java входной метод должен завершиться до завершения процесса. Если входной метод — например, `main()` или `execute()`, запускающий процесс, — завершается только при завершении процесса, OneAgent не может финализировать и отправить данные.

Для Java, .NET и Go можно использовать поиск для выбора точки входа и методов или указать их вручную. Чтобы создать новый пользовательский сервис Java, .NET или Go:

Найти точку входа

Указать точку входа вручную

1. Перейдите в **Settings** > **Service detection** > **Custom service detection**.
2. Выполните одно из следующих действий:

   * Java Выберите вкладку **Java services** и выберите **Define Java services**.
   * .NET Выберите вкладку **.NET services** и выберите **Define .NET services**.
   * Go Выберите вкладку **Go services** и выберите **Define Go services**.
3. Дайте сервису понятное имя.
4. Выберите **Find entry point**.
5. Найдите и выберите группу процессов, содержащую точку входа.
6. Выберите процесс, содержащий точку входа, и нажмите **Continue**.
7. Найдите класс для инструментирования. Введите имя или его часть для поиска.
8. Выберите нужный класс и нажмите **Continue**.
9. Определите способ инструментирования класса. Доступны два варианта:

   * **Use the selected class** — для инструментирования только методов выбранного класса.
   * Java .NET **Use an implemented interface or superclass** — для инструментирования методов любого интерфейса или суперкласса в иерархии классов. В таких случаях выберите **Load inheritance** для загрузки всех доступных суперклассов и интерфейсов, затем выберите нужный.
10. Выберите методы для инструментирования и нажмите **Finish**.
    На странице **Define custom service** отображается только что добавленная точка входа и методы.
11. При необходимости добавьте дополнительные точки входа.
12. При необходимости ограничьте новый пользовательский сервис определёнными группами процессов. См. раздел [**Ограничение пользовательского сервиса определёнными группами процессов**](#restrict) ниже.
13. Проверьте точку входа и методы для инструментирования.
14. В правом нижнем углу страницы выберите **Save**.

1. Перейдите в **Settings** > **Service detection** > **Custom service detection**.
2. Выполните одно из следующих действий:

   * Java Выберите вкладку **Java services** и выберите **Define Java services**.
   * .NET Выберите вкладку **.NET services** и выберите **Define .NET services**.
   * Go Выберите вкладку **Go services** и выберите **Define Go services**.
3. Дайте сервису понятное имя.
4. Выберите **Define entry point manually**.
5. В поле **Fully qualified class** введите имя класса, содержащего методы для инструментирования.
6. Укажите инструментируемые методы одним из двух способов:

   * Введите полное имя метода вручную.
   * Найдите методы в классе и выберите их:

     1. Выберите **Search for methods**.
     2. Выберите группу процессов, содержащую точку входа.
     3. Выберите методы для инструментирования и нажмите **Finish**.
7. При необходимости добавьте дополнительные методы.
8. При необходимости добавьте дополнительные точки входа.
9. Проверьте точку входа и методы для инструментирования.
10. При необходимости ограничьте новый пользовательский сервис определёнными группами процессов. См. раздел [**Ограничение пользовательского сервиса группами процессов**](#restrict) ниже.
11. В правом нижнем углу страницы выберите **Save**.

Неподдерживаемый сценарий

Обратите внимание, что некоторые сценарии не работают в Dynatrace, например сценарий Queue/Worker. Всё, что помещается в `java.util.Queue` и обрабатывается другим рабочим потоком, не видно в Pure Path. См. [Oracle java.util Interface Queue](https://dt-url.net/xy03uja).

## PHP-сервисы

Для PHP можно использовать поиск для выбора точки входа и методов или указать их вручную. Чтобы создать новый пользовательский PHP-сервис:

Найти точку входа

Указать точку входа вручную

1. Перейдите в **Settings** > **Process and contextualize** > **Services** > **Custom service detection**.
2. Выберите вкладку **PHP services** и выберите **Define PHP service**.
3. Дайте сервису понятное имя.
4. Выберите **Find entry point**.
5. Найдите и выберите группу процессов, содержащую точку входа.
6. Выберите процесс, содержащий точку входа, и нажмите **Continue**.
7. Найдите класс для инструментирования. Введите имя или его часть для поиска.
8. Выберите нужный класс и нажмите **Continue**.
9. Выберите методы для инструментирования и нажмите **Finish**.
   На странице **Define custom service** отображается только что добавленная точка входа и методы.
10. При необходимости добавьте дополнительные точки входа.
11. При необходимости ограничьте новый пользовательский сервис определёнными группами процессов. См. раздел [**Ограничение пользовательского сервиса определёнными группами процессов**](#restrict) ниже.
12. Проверьте точку входа и методы для инструментирования.
13. В правом нижнем углу страницы выберите **Save**.

1. Перейдите в **Settings** > **Process and contextualize** > **Services** > **Custom service detection**.
2. Выберите вкладку **PHP services** и затем **Define PHP service**.
3. Дайте сервису понятное имя.
4. Выберите **Define entry point manually**.
5. В поле **File name** введите полный путь к PHP-файлу, содержащему методы для инструментирования.
6. В поле **Fully qualified class name** введите имя класса, содержащего методы для инструментирования.
7. В поле **Methods** введите имя метода для инструментирования.
8. При необходимости выберите **Add method manually** и укажите дополнительные методы для инструментирования.
9. При необходимости ограничьте новый пользовательский сервис определёнными группами процессов. См. раздел [**Ограничение пользовательского сервиса определёнными группами процессов**](#restrict) ниже.
10. После добавления всех методов выберите **Save** в правом верхнем углу страницы.

### Пример: определение пользовательского PHP-сервиса через файл

Следующий пример показывает, как определить пользовательский сервис на основе `.php`-файла, а также как результирующее обнаружение пользовательского сервиса отображается в **Distributed Traces**.

```
<?php



class InternalExample {



public function say($what) {



echo $what;



}



}



class Example {



public $internalExample;



public function __construct() {



$this->internalExample = new InternalExample();



}



public function run() {



$this->internalExample->say("Hello world!\n");



}



}



$example = new Example();



$example->run();
```

В приведённом скрипте объект класса `Example` создаёт и вызывает объект класса `InternalExample` для вывода `Hello world!`. В этом примере нужно трассировать вывод `Hello world!` и создание класса `Example`.

Теперь используем файл `index.php` для определения пользовательского сервиса, следуя шагам в разделе [Указать точку входа вручную](#manual-php-service) выше. В файле можно инструментировать конкретные функции следующим образом:

* Можно инструментировать функцию `say` из класса `InternalExample`:

  ![Инструментирование функции say](https://dt-cdn.net/images/php-custom-service-example-1-1091-b7b92d31ae.png)

  Инструментирование функции say
* Можно инструментировать функцию `run` из класса `Example`. Дополнительно инструментируем функцию `__construct` класса `Example` для отображения его создания в трассировке.

  ![Инструментирование функции construct](https://dt-cdn.net/images/php-custom-service-example-2-1087-65aa6d790c.png)

  Инструментирование функции construct

В обоих примерах в поле **File name** используется опция **Match** для указания локального пути к файлу `index.php`. Использование локальных путей в определениях пользовательских сервисов удобно, поскольку некоторые PHP-фреймворки изменяют каталог выполнения скрипта. При указании имён классов и методов важна чувствительность к регистру, но перечислять параметры методов не нужно.

Результирующие pure paths можно увидеть в **Distributed Traces**, объединёнными в одну трассировку:

## Определение пользовательского Go-сервиса через API

Чтобы определить пользовательский Go-сервис через API

1. Подготовка HTTP-запроса

1. Определите область (`className`) — пакет или тип, в котором определён метод.

   В следующих примерах область — `main.(*CronJob)`, поскольку функция определена как метод типа `CronJob` в пакете `main`.
2. Определите полностью квалифицированное имя метода (`methodName`) — имя метода с префиксом области.

В примере тела запроса ниже полностью квалифицированное имя метода — `main.(*CronJob).Run`. Пример содержит все обязательные параметры. Для использования скорректируйте значения в соответствии со своим запросом.

```
{



"name": "Cron Job Service",



"enabled": true,



"rules": [



{



"enabled": true,



"className": "main.(*CronJob)",



"methodRules": [



{



"methodName": "main.(*CronJob).Run",



"returnType": ""



}



]



}



],



"queueEntryPoint": false



}
```

2. Отправка POST-запроса к конечной точке

* Используйте `go` в качестве технологии.
* Используйте токен доступа или персональный токен доступа для аутентификации.

```
curl --request POST 'https://{your-environment-id}.live.dynatrace.com/api/config/v1/service/customServices/go' \



--header 'Authorization: Api-Token dt0s01.ST2EY72KQINMH574WMNVI7YN.G3DFPBEJYMODIDAEX454M7YWBUVEFOWKPRVMWFASS64NFH52PX6BNDVFFM572RZM' \



--header 'Content-Type: application/json' \



--data '{



"name": "Cron Job Service",



"enabled": true,



"rules": [



{



"enabled": true,



"className": "main.(*CronJob)",



"methodRules": [



{



"methodName": "main.(*CronJob).Run",



"returnType": ""



}



]



}



],



"queueEntryPoint": false



}'
```

При успехе возвращается `id` только что созданного пользовательского сервиса; при сбое возвращается описательное сообщение об ошибке. С помощью этого `id` можно редактировать пользовательский сервис через API.

```
{



"id": "114ff497-be8a-44c7-a990-35bb4267b677",



"name": "Cron Job Service"



}
```

## Замыкания в Go

Замыкания — это анонимные функции, которым компилятор Go присваивает специальные имена.

Генерируемые имена замыканий, определённых в области видимости, нумеруются в порядке их появления: `func1`, `func2` и т.д. Например, замыкание в функции `main` получает метку `func1`:

```
package main



func main() {



go func() {



// ...



}()



}
```

## Приоритет пользовательских сервисов

При наличии нескольких определённых пользовательских сервисов оценка происходит сверху вниз, применяется первое совпавшее правило. Если один и тот же класс и метод определены в нескольких пользовательских сервисах, убедитесь, что сервисы расставлены в правильном приоритете.

## Заголовок x-dynatrace

Для HTTP-запросов Dynatrace использует дополнительный HTTP-заголовок `x-dynatrace` для объединения транзакций. Этот заголовок устанавливается OneAgent между веб-серверами для их связывания. Технология PurePath® для распределённых трассировок в значительной мере опирается на такие заголовки, поэтому важно убедиться, что сетевые компоненты, такие как брандмауэры и маршрутизаторы, никогда не настроены на удаление этих заголовков. Неправильная настройка сетевых компонентов может привести к обрывам pure path. Кроме того, некоторые сетевые компоненты полностью блокируют такие запросы (возвращая ошибку HTTP 403), поскольку дополнительный заголовок может считаться небезопасным. В таких случаях необходимо настроить эти компоненты на приём заголовка `x-dynatrace`.

## Редактирование пользовательского сервиса

Пользовательский сервис можно редактировать в любое время. Для вступления изменений в силу необходимо перезапустить затронутые процессы, если только не активированы обновления в реальном времени для Java и PHP. Для .NET процесс необходимо перезапустить.

Для редактирования пользовательского сервиса выберите кнопку **Edit** для этого сервиса в списке сервисов.

Можно активировать/деактивировать существующие точки входа, добавлять/удалять точки входа, добавлять/удалять методы в точках входа.

Также можно ограничить пользовательский сервис определёнными группами процессов. См. раздел [**Ограничение пользовательского сервиса определёнными группами процессов**](#restrict) ниже.

### Обновления в реальном времени

Обновления пользовательских сервисов Java и PHP можно применять почти в реальном времени без перезапуска процессов. Для активации этой функции перейдите в **Settings** > **Server-side service detection** > **Deep monitoring** > **Real-time updates to Java and PHP services** и включите соответствующие переключатели.

С этой функцией возможности мониторинга для сервисов и атрибутов запросов добавляются во время существующих пауз сборки мусора приложения, а не только при запуске процесса. Это может увеличить паузы сборки мусора и замедления. Рекомендуется ограниченное использование в среде, чувствительной к задержкам, поскольку Dynatrace не управляет паузами сборки мусора, предоставляемыми и реализованными виртуальными машинами.

## Ограничение пользовательского сервиса определёнными группами процессов

Можно ограничить использование любого пользовательского сервиса определёнными группами процессов. Правила пользовательских сервисов будут применяться только в указанных группах процессов и будут игнорироваться в других. Ограничение можно задать при создании или отредактировать позднее.

Чтобы ограничить пользовательский сервис:

1. На странице **Define custom service** или **Edit service** разверните **Optionally restrict custom service rules by process groups**.
2. Выберите **Add process group**.
3. Выберите группу процессов, в которой должен применяться пользовательский сервис.
4. Выберите **Add**.
5. В правом нижнем углу страницы **Define custom service** или **Edit service** выберите **Save**.
