# Настройка именования запросов

# Настройка именования запросов

* How-to guide
* 7-min read
* Updated on Feb 24, 2026

Правила именования запросов позволяют настроить отслеживание запросов и определить точки входа и конечные точки сервисов в рабочем процессе, ориентированном на клиентов. Благодаря такой сквозной трассировке можно легко распознавать и отслеживать важные бизнес-транзакции, критически важные для успеха вашего цифрового бизнеса.

Используя атрибуты запросов в сочетании с правилами именования, можно захватывать ещё больше контекста о запросах и применять эти дополнительные сведения для детального анализа данных мониторинга.

Поскольку правила именования запросов порождают отдельные запросы к сервисам, каждый запрос [базируется независимо](/managed/dynatrace-intelligence/anomaly-detection/automated-multidimensional-baselining "Узнайте, как искусственный интеллект Dynatrace автоматически рассчитывает базовые показатели на основе многомерной схемы базирования.") и отслеживается на предмет аномалий производительности. Метрики производительности, захваченные для этих запросов, также доступны отдельно через конечную точку [Timeseries API v1](/managed/dynatrace-api/environment-api/metric-v1 "Получайте информацию о метриках через Timeseries v1 API.").

## Перед началом

* Правила именования запросов поддерживаются для следующих типов сервисов:

  + Web request (кроме **Requests to unmonitored hosts** и **Requests to public networks**)
  + Web
  + Method request
  + Messaging and queing (кроме [listener services](/managed/observe/infrastructure-observability/queues/queue-concepts#listener-service "Основные концепции мониторинга очередей сообщений в Dynatrace."))
  + RMI
  + CICS
  + IMS
  + Enterprise service bus
  + Remote call
* Обнаружение ключевых запросов основано на именах. Если правило именования запросов затрагивает ключевой запрос и вы хотите, чтобы Dynatrace продолжал его обнаруживать как ключевой, необходимо добавить новое имя в [список имён ключевых запросов](/managed/observe/application-observability/services-classic/monitor-key-requests#rename "Узнайте, как тщательно отслеживать запросы, критически важные для вашего бизнеса.").

## Создание правила именования запросов для сервиса

Первый шаг в настройке чёткого именования запросов к сервису — создание правил именования запросов с условиями, определяющими их отображение в вашем окружении.

Чтобы создать правило именования запросов для вашего сервиса

1. Перейдите на страницу настройки именования (веб-)запросов:

   1. Перейдите в **Services** и выберите сервис для настройки.
   2. Выберите **More** (**…**) > **Settings** и перейдите в **Request naming** или **Web request naming**.
2. Выберите **Add rule**.
3. Задайте набор условий, отражающих критерии операций вашего сервиса.

   Для идентификации конкретных запросов можно использовать всё — от заголовков запросов до значений аргументов методов на уровне кода. Все запросы, соответствующие указанным критериям, будут именоваться в соответствии с определённым шаблоном именования.

   ![Правило именования запросов](https://dt-cdn.net/images/naming-rule-2010-1385de1a99.png)

   Правило именования запросов

   В примере выше определены три условия:

   * Путь URL должен содержать строку `orange-booking-payment`.
   * Запрос должен быть **HTTP method** типа **GET**.
   * Запрос должен иметь атрибут `easyTravel destination`.

   Все запросы, соответствующие **всем** указанным критериям, будут именоваться в соответствии с шаблоном именования `PAYMENT`.

   ![Добавление правила именования запросов — пример](https://dt-cdn.net/images/example-naming-rule-2006-1ef0fc7a46.png)

   Добавление правила именования запросов — пример
4. Выберите **Save**.

## Глобальные правила именования запросов

[API именования запросов сервисов](/managed/dynatrace-api/configuration-api/service-api/request-naming-api "Узнайте, что предлагает Dynatrace Configuration API для именования запросов.") позволяет консолидировать или уточнять запросы в нескольких сервисах. Кроме того, эти правила можно синхронизировать между несколькими окружениями Dynatrace. Чтобы узнать, как создать глобальное правило именования запросов через API, см. [данный пример использования](/managed/dynatrace-api/configuration-api/service-api/request-naming-api/create-a-new-request-naming-rule "Узнайте, как создать правило именования запросов через Dynatrace API.").

## Настройка обнаружения ресурсных запросов

Dynatrace автоматически обнаруживает ресурсные запросы по расширению файлов и отслеживает их в следующих группах:

* Images
* CSS
* JavaScript
* Binaries

Если Dynatrace автоматически не обнаруживает изображения или двоичные файлы вашего приложения, можно добавить отсутствующие расширения имён файлов. Эти настройки можно задать для всего окружения или для конкретного сервиса.

Окружение

1. Перейдите в **Settings** > **Server-side service monitoring** > **Request naming**.
2. Прокрутите вниз до раздела **Web resource requests** и введите необходимые расширения файлов для ресурсов изображений и двоичных файлов.

Либо используйте API для изменения [глобальных правил именования запросов](#global-rules).

Сервис

1. Перейдите в **Services** и выберите сервис, для которого необходимо обновить настройки.
2. В правом верхнем углу панели сведений о сервисе выберите ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More") (**More**) > **Settings**.
3. Перейдите в **Web request naming** и прокрутите вниз до раздела **Resource requests**.
4. Выключите **Use global resource request detection settings**.
5. Введите необходимые расширения файлов для ресурсов изображений и двоичных файлов.

## Дополнительные элементы именования запросов

### Атрибуты запросов и заполнители

Значения атрибутов запросов и заполнителей можно включать в шаблоны именования.

* Используйте **Request attribute** для создания интуитивно понятных вариантных имён запросов. Запрос создаёт отдельный отслеживаемый запрос для каждой перестановки соответствующего атрибута запроса.

Пример: Атрибуты запросов в шаблонах именования

Используя атрибут запроса `easyTravel destination` в шаблоне именования, для каждого направления, забронированного клиентами `easyTravel`, создаётся вариантный запрос. В результате это правило создаёт не один тип запроса, а отдельный отслеживаемый запрос для каждого забронированного направления.

![Использование атрибутов запросов для настройки правил именования запросов](https://dt-cdn.net/images/request-attributes-2002-61662e499a.png)

Использование атрибутов запросов для настройки правил именования запросов

Просматривая распределённые трассировки, можно видеть, что URL-адреса одинаковы, но каждое имя запроса теперь включает другое значение атрибута направления.

![Шаблон именования с атрибутами запросов](https://dt-cdn.net/images/request-attributes-example-2004-cfa8f5d39c.png)

Шаблон именования с атрибутами запросов

* **Placeholder(s)** (заполнители) позволяют извлекать значения из атрибутов запросов или URL-адресов.

Пример: Пользовательские заполнители

Если путь URL содержит строку `orange-booking`, бронирование, определённое заполнителем `stage`, извлекается из шаблона именования URL и преобразуется в верхний регистр. Заполнитель `stage` определяется как текст между `orange-booking-` и `.jsf` в пути URL.

![Извлечение с заполнителем](https://dt-cdn.net/images/extraction-placeholders-2000-0683e7a52f.png)

Извлечение с заполнителем

С использованием этого заполнителя результирующий список имён запросов выглядит следующим образом:

![Результаты именования запросов с заполнителем в списке распределённых трассировок сервиса](https://dt-cdn.net/images/placeholders-request-naming-1130-60e819ce95.png)

Результаты именования запросов с заполнителем в списке распределённых трассировок сервиса

Выстраивая таким образом точки входа и конечные точки сервисов для мониторинга Dynatrace, можно отслеживать все ключевые операции организации, такие как бизнес-транзакции, на высоко детализированном уровне.

### Правила очистки

Правила и настройки на уровне сервиса, включая правила очистки URL для сервисов веб-запросов, переопределяют глобальные правила именования запросов.

* Через API можно определять глобальные правила именования запросов для очистки URL одного или нескольких сервисов одновременно.
* Для сервисов веб-запросов предусмотрены специальные функции генерации чистых URL через веб-интерфейс.

В разделе **Service settings** > **Web request naming** можно:

* **Remove UUIDs, IP addresses and IBANs from URLs** — нормализовать пути URL, содержащие UUID, IP-адреса и IBAN, заменив конкретные значения связанными статическими строками, такими как `UUID`, `IPv4` и `IBAN`.
* **Create clean URL rule** — определить [регулярные выражения](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Узнайте, как использовать регулярные выражения в контексте Dynatrace.") для удаления частей пути URL сервиса веб-запросов, например идентификаторов.

Пользовательские имена запросов к сервисам никогда не маскируются.

### Предварительный просмотр

Правило именования запросов можно изменять или комбинировать несколько правил для создания ещё более детализированных имён запросов. **Preview Rule** отображает результат нового правила именования запросов.

![Изменение и предварительный просмотр правила именования запросов](https://dt-cdn.net/images/modify-naming-rule-2012-c9ae3151bf.png)

Изменение и предварительный просмотр правила именования запросов

В приведённом выше примере новое правило `Booking {RequestAttribute:easyTravel destination} - {stage}` объединяет два предыдущих примера. Оно определяет шаблон именования запросов, включающий не только этап бронирования (`{stage}`), но и атрибут направления (`{RequestAttribute:easyTravel destination}`). Теперь создаётся отдельный запрос для каждого этапа бронирования плюс дополнительные разбивки на основе атрибута направления.

### Маскировка данных

Можно отобразить немаскированные данные для конкретных запросов, установив флажок **Access unmasked data**.

Это потенциально раскрывает [персональные данные](/managed/manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace#service-request-monitoring "Узнайте, какие типы данных конечных пользователей могут захватываться во время мониторинга Dynatrace и какие методы доступны для маскировки персональных данных конечных пользователей.") до их сохранения и отображения.

## Анализ сервисов

Полная ценность такой настройки бизнес-критических запросов становится очевидной при углублении в анализ, доступный для каждого отдельного запроса на уровне сервиса.

Атрибуты запросов обеспечивают абсолютную гибкость при идентификации и именовании запросов в соответствии с бизнес-требованиями. Dynatrace отслеживает каждый запрос от начала до конца и показывает, как все запросы связаны между собой.

Многоуровневый анализ

Для иллюстрации многоуровневого анализа добавим ещё одно правило именования запросов, которое разбивает запросы на бронирование по атрибуту `Loyalty`.

![Многоуровневый анализ сервисов](https://dt-cdn.net/images/multi-tiered-analysis-2002-c95686fde7.png)

Многоуровневый анализ сервисов

Это добавление приводит к созданию отдельных запросов к данному сервису в зависимости от статуса лояльности.

Просматривая отдельную трассировку, можно видеть, насколько полезны два определённых атрибута запроса. В данном примере `Booking Shizunai` было выполнено клиентом со статусом лояльности `PLATINUM`. Это было достигнуто путём определения правила именования запросов на сервисе `easyTravel Customer Frontend`, отслеживающего бронирования по направлениям, и отдельного правила именования запросов, отслеживающего бронирования на бэкенд-сервисе `BookingServiceTom` по статусу лояльности.

![Трассировка многоуровневого анализа сервисов](https://dt-cdn.net/images/multi-tiered-analysis-trace-2024-0b4a266f8f.png)

Трассировка многоуровневого анализа сервисов

Многомерный анализ

Поскольку правила именования запросов порождают отдельные запросы к сервисам, появляются дополнительные возможности фильтрации на основе атрибутов новых определённых запросов. В примере ниже разбивка по направлениям сочетается со статусом лояльности `PLATINUM`.

![Атрибуты ресурсов](https://dt-cdn.net/images/multidim-analysis-2036-7a8488d166.png)

В этом примере анализируются запросы на бронирование на этапе `finish` с направлением `shizunai`, диапазоном времени отклика 1–2 с и статусом лояльности Platinum.
