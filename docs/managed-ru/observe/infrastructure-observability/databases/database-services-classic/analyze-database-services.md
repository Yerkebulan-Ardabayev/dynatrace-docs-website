# Анализ сервисов баз данных (классическая страница)

* Explanation
* 6-min read
* Updated on Jun 20, 2023

Начиная с Dynatrace 1.269 страница обзора базы данных была полностью переработана.

* Чтобы перейти на [новую страницу обзора базы данных](/managed/observe/infrastructure-observability/databases/database-services-classic/analyze-database-services-new "Анализируйте сервисы баз данных с помощью Dynatrace (новая страница)."), просто включите **Try it out** в верхней части страницы обзора базы данных. При желании можно вернуться к классическому дизайну.
* Документация ниже описывает классический дизайн.

Dynatrace мониторит время ответа, частоту ошибок, пропускную способность и другие параметры каждого оператора БД, выполняемого вашими приложениями.

Для анализа производительности базы данных:

1. Перейдите в **Database Services**.
2. Чтобы перейти на страницу обзора сервиса базы данных, выберите базу данных, которую нужно проанализировать.

![Обзор базы данных](https://dt-cdn.net/images/database-overview-1297-02a737c3d3.png)

Обзор базы данных

Инфографика на странице обзора сервиса базы данных даёт представление о различных аспектах производительности базы данных, включая **SQL queries or procedures**, **SQL modifications**, **SQL transactions**, обнаруженные проблемы или проблемы с доступностью, «горячие точки» и многое другое. Точный список включённых плиток зависит от технологии базы данных, которая мониторится. Каждая плитка содержит метрики **Response time**, **Failure rate** и **Throughput**.

Чтобы проанализировать отдельные операторы, составляющие оператор БД, на странице обзора базы данных выберите **View database statements**.

## Сведения о базе данных

В нижней части страницы **Details** находится список всех **операторов БД**, выполненных в течение выбранного временного интервала. По умолчанию они отсортированы по медианному времени ответа, однако список можно сортировать по общему времени, времени ответа или по 10% самых медленных, а также фильтровать по оператору, имени базы данных, поставщику, процессу или имени хоста.

![Страница сведений о базе данных](https://dt-cdn.net/images/database-details-page-1293-85e42be938.png)

Страница сведений о базе данных

Выберите любой оператор для просмотра полных сведений. Для каждого оператора Dynatrace показывает количество **Executions**/мин, **Total executions**, **Total time**, медианное **Response time** и 10% самых медленных выполнений, а также **Failure rate**.

Ограничения операторов

По умолчанию длина операторов БД ограничена 4 КБ (килобайтами). Этот параметр не настраивается. Операторы БД, превышающие это ограничение, усекаются до 4 КБ.

### Обратная трассировка сервиса базы данных

Чтобы получить доступ к анализу обратной трассировки для определённого оператора:

1. В списке операторов БД перейдите к строке оператора, который нужно проанализировать.
2. Выберите **More** (**…**) > **Service backtrace**.

![Обратная трассировка сервиса базы данных — приложение](https://dt-cdn.net/images/database-service-level-backtrace-application-1490-89f24b7b0f.png)

Обратная трассировка сервиса базы данных — приложение

Страница **Backtrace** предоставляет сведения о кликах, которые инициировали выбранную вариацию SQL-оператора, и позволяет увидеть, какая цепочка сервисов и какой код приводит к конкретному оператору. Выбрав конкретное приложение в обратной трассировке, можно найти все действия пользователя из этого приложения, которые инициируют анализируемый оператор БД.

Пример

На изображении выше в столбце **Incoming requests** видно, что **1 200** запросов к базе данных на `easyTravelBusiness` исходят от **668** действий пользователя.

Выбрав `www.easyTravelBusiness.com`, выделяется строка, указывающая, что **668** действий пользователя привели к **1 150** запросам на `easyTravel Customer Frontend`. При таком выборе видно, что три различных действия пользователя из `www.easytravel.com` инициируют этот оператор. Также видна цепочка сервисов, которая его инициирует. Для дальнейшего изучения увеличения количества запросов в цепочке выберите любой из последующих операторов в цепочке вызовов сервисов.

![Обратная трассировка сервиса базы данных — цепочка](https://dt-cdn.net/images/database-service-level-backtrace-chain-1490-b807b8f624.png)

Обратная трассировка сервиса базы данных — цепочка

При новом выборе (`easyTravel Customer Frontend` на изображении выше) нижний раздел страницы обратной трассировки адаптируется, и теперь видно, что **668** действий пользователя из `www.easyTravelBusiness.com` вызывают два запроса на `easyTravel Customer Frontend` (`/orange-booking-payment.jsf` и `/orange-booking-finish.jsf`). Если некоторые из этих запросов завершились неудачей, их можно проанализировать на соответствующей вкладке **Reasons for failed requests**. В данном примере больший интерес представляет код, приводящий к вызовам базы данных, поэтому следует выбрать вкладку **Stacktrace** для просмотра выполненного кода в контексте.

### Распределение времени ответа

Чтобы понять распределение времени ответа команды, на странице **Details** выберите **Analyze outliers**.

Страница **Response time distribution** показывает количество запросов, попавших в различные диапазоны времени ответа. Выберите любой столбец на графике для анализа запросов, попавших в данный конкретный диапазон времени ответа. Это позволяет определить, вызываются ли определённые операторы часто и вносят ли относительно значительный вклад во время ответа.

![Выбросы — распределение времени ответа](https://dt-cdn.net/images/database-response-time-distribution-1309-417be68a11.png)

Выбросы — распределение времени ответа

## Анализ деградации производительности

С помощью мониторинга баз данных Dynatrace можно анализировать деградацию производительности. Операторы, ответственные за обнаруженное замедление, выделяются в списке операторов в нижней части страницы **Details**.

![База данных — деградация времени ответа](https://dt-cdn.net/images/database-4-1090-03b012cb2e.png)

База данных — деградация времени ответа

В примере ниже обнаруженная деградация времени ответа была вызвана двумя операторами (`Aggregations in BookingCollection` и `Aggregations in JourneyCollection`).

Чтобы узнать, какой сервис выполнил конкретный оператор, приведший к замедлению базы данных, перейдите к обратной трассировке оператора.

![Медленные операторы БД](https://dt-cdn.net/images/mysql-7-966-2c43e07703.png)

Медленные операторы БД

В данном случае `easyTravel-Business` — это сервис, выполняющий оператор, вызывающий замедление.

![Сервис базы данных — медленные операторы](https://dt-cdn.net/images/database-6-1286-28bedc7573.png)

## Анализ ошибок базы данных

Каждая плитка на странице обзора сервиса базы данных содержит график **Failure rate** (1).

![База данных — анализ ошибок](https://dt-cdn.net/images/database-analyze-failures-1755-bca922ddef.png)

База данных — анализ ошибок

Выберите график **Failure rate** (1), чтобы открыть вкладку ошибок на странице **Details**.

Таблица неудачных **операторов БД** в нижней части страницы немедленно показывает, какие операторы завершились неудачей, вместе с их частотой ошибок.

1. Выберите **View details of failures**, чтобы понять первопричину ошибок.  
   Dynatrace определяет причины высокой частоты ошибок, предоставляя информацию для устранения проблем с базой данных и предотвращения будущих ошибок.

   ![Частота ошибок — сведения о базе данных](https://dt-cdn.net/images/failure-rate-database-details-1753-9ba03aabd4.png)

   Частота ошибок — сведения о базе данных
2. Для дальнейшего анализа проблемы выберите **Distributed traces** ![Distributed traces](https://dt-cdn.net/images/purepaths-icon-790bea38ba.svg "Распределённые трассировки") или **Backtrace** ![Backtrace](https://dt-cdn.net/images/backtrace-icon-69e2c66211.svg "Обратная трассировка").

   ![Анализ ошибок](https://dt-cdn.net/images/failure-analysis-1925-81fd9dc157.png)

   Анализ ошибок

### Быстрый обзор проблемы

На странице обзора сервиса базы данных раздел **Current hotspots** (2) предлагает быстрый обзор проблемы, выделяя оператор с наибольшей частотой ошибок. Выбрав оператор в этом разделе (2), можно напрямую перейти к вкладке ошибок на его странице **Details**.

## Поток сервисов для анализа SQL

[**Service flow**](/managed/observe/application-observability/services-classic/service-flow "Узнайте, как Dynatrace помогает трассировать последовательность вызовов сервисов, инициированных каждым запросом к сервису в вашей среде.") визуализирует последовательность вызовов сервисов, инициированных каждым запросом к сервису в вашей среде. С помощью потока сервисов можно видеть поток вызовов сервисов, включая вызовы сервисов баз данных, с точки зрения одного сервиса, запроса или их отфильтрованного подмножества. Хотя анализ SQL доступен в нескольких местах рабочего процесса анализа сервисов, он является основным представлением анализа баз данных в представлении **Service flow**.

Чтобы просмотреть поток сервисов, инициированный конкретным сервисом базы данных:

1. Перейдите в **Services**.
2. Выберите сервис, который нужно проанализировать.
3. На странице обзора сервиса в разделе **Understand dependencies** выберите **View service flow**.

![Поток сервисов — анализ SQL](https://dt-cdn.net/images/service-flow-sql-analysis-1759-f706b49d27.png)

Поток сервисов — анализ SQL

Пример

В примере выше поток сервисов базы данных отфильтрован для фокусировки на конкретной цепочке вызовов. При каждом обращении `easyTravel Customer Frontend` к `JourneyService` **0,01%** **вклада во время ответа** можно отнести к сервису MongoDB `EasyTravelBusiness`.

Чтобы просмотреть операторы БД, выполненные в рамках выбранного потока в течение временного интервала анализа, выберите **View database statements**. Это может быть крайне ценно, поскольку помогает понять, почему база данных вносит определённый вклад во время.

![Многомерный анализ топ-измерений](https://dt-cdn.net/images/multidimesional-analysis-sql-analysis-1752-aa2f7307ab.png)

Многомерный анализ топ-измерений

Для одного из **топ-измерений** выберите **More** (**…**), а затем:

* Выберите **Outliers**, чтобы понять разброс времени ответа.
* Выберите **Statement details**, чтобы понять динамику каждого SQL-оператора во времени вместе со средним количеством **Rows returned**.
* Выберите **Service backtrace**, чтобы увидеть, какие клики пользователя привели к наиболее медленным выполнениям данного оператора MongoDB.

Как видно, представления анализа операторов БД Dynatrace обеспечивают мощный анализ использования баз данных.
