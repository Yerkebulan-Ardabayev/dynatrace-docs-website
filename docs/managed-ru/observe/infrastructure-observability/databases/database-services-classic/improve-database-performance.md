# Улучшение производительности баз данных

# Улучшение производительности баз данных

* How-to guide
* 6-min read
* Published Mar 20, 2019

Базы данных — это сложные приложения, а доступ к базе данных является ключевой функцией многих приложений. Чтобы избежать сбоев или низкой производительности, важно, чтобы базы данных были надёжно размещены и имели достаточно ресурсов для оптимальной работы.

Оптимизировать базы данных можно с помощью:

* Серверных данных, поддерживающих мониторинг работоспособности хоста
* Метрик гипервизора и виртуальных машин, поддерживающих мониторинг уровня виртуализации
* Данных приложений, оптимизирующих доступ к базе данных
* Сетевых данных, дающих представление о сетевом воздействии коммуникаций с базой данных

С помощью следующих нескольких шагов простой настройки производительности базы данных можно значительно ускорить большинство приложений.

## Шаг 1 Проверьте работоспособность базы данных

Первый шаг — убедиться, что хост, обслуживающий процесс базы данных, имеет достаточно ресурсов: CPU, памяти и дискового пространства.

### CPU

* Оснастите каждый хост минимум двумя ядрами CPU. Соответствие количества CPU на хосте позволяет:

  + Обеспечить отзывчивость хоста, поскольку серверы баз данных создают непрерывную базовую нагрузку на машины.
  + Предотвратить излишние расходы или аппаратные ограничения, поскольку лицензирование серверов баз данных зависит от количества CPU.
* При мониторинге виртуальных машин мониторьте также хост, на котором они работают. Это даёт более полную картину, чем метрики CPU отдельных виртуальных машин, которые отображают только доступность CPU на соответствующем уровне.

  ![Метрики CPU](https://dt-cdn.net/images/cpu-1198-a20fe73396.png)

  Метрики CPU

### Память

* Помимо мониторинга метрики **Memory usage**, мониторьте **Page faults** в секунду, чтобы узнать, сколько дополнительной памяти требуется. Тысячи ошибок страниц в секунду указывают на нехватку памяти на хосте.

  ![Использование памяти хоста](https://dt-cdn.net/images/host-memory-analysis-page-faults-2372-02f34bdaa5.png)

  Использование памяти хоста

### Дисковое пространство

* Убедитесь, что доступное хранилище для сервера базы данных превышает дисковое пространство, требуемое для данных.

  Из-за индексов и других оптимизаций производительности базы данных могут использовать больше дискового пространства, чем требуется для самих данных. Например, базы данных NoSQL (такие как Cassandra и MongoDB) потребляют значительно больше дискового пространства, чем ожидается. По сравнению с обычными базами данных SQL, базы данных MongoDB могут потреблять меньше оперативной памяти, но больше дискового пространства.
* Убедитесь, что база данных работает на выделенных жёстких дисках, чтобы уменьшить фрагментацию диска, вызванную другими процессами.
* Проверьте **Disk latency**.

  В зависимости от нагрузки на жёсткий диск задержка диска может увеличиваться, что приводит к снижению производительности базы данных. Предотвратить высокую задержку диска можно путём максимального использования механизмов кэширования приложения и базы данных.

  ![Задержка диска](https://dt-cdn.net/images/host-disk-latency-2060-80803a8edc.png)

  Задержка диска
* Если результаты вышеперечисленных мер неудовлетворительны, следует рассмотреть следующее.

  1. Добавьте дополнительные жёсткие диски.

     Производительность чтения можно увеличить простым зеркалированием жёстких дисков. Производительность записи выигрывает от использования RAID 1 или RAID 10 вместо RAID 6.
  2. Попробуйте твердотельные накопители.

     Убедитесь, что выбрана модель, предназначенная для использования с базами данных, поскольку базы данных выполняют больше циклов чтения/записи в хранилище, чем большинство обычных приложений. Твердотельные накопители дороже традиционных жёстких дисков, но обеспечивают существенный прирост производительности.

## Шаг 2 Поймите, как осуществляется доступ к базе данных

Как только база данных размещена на исправном оборудовании, стоит изучить приложения, обращающиеся к ней. Если известно приложение или сервис с плохой производительностью базы данных, не следует считать, что именно это приложение влияет на производительность базы данных — причиной может оказаться совсем другое приложение или сервис.

![Обзор базы данных | Unified analysis](https://dt-cdn.net/images/database-ua-overview-3502-2a520ae771.png)

Обзор базы данных | Unified analysis

Снижение производительности базы данных может затронуть всю базу данных или отдельного клиента.

* Если все клиенты испытывают снижение производительности, проверьте состояние хоста. В большинстве случаев причиной является оборудование, не способное справиться с нагрузкой.
* Если только один сервис испытывает низкое время ответа, углубитесь в метрики сервиса для поиска первопричины проблемы.

## Шаг 3 Понимайте нагрузку и индивидуальное время ответа каждого экземпляра сервиса

Когда сервис имеет низкую производительность базы данных, можно проанализировать его взаимодействие с базой данных через **операторы БД**. Можно получить аналитику о количестве выполненных запросов, частоте выполнения запросов на запрос, количестве строк, возвращаемых каждым запросом, и т.д.

![Страница сведений о базе данных](https://dt-cdn.net/images/database-details-page-1293-85e42be938.png)

Страница сведений о базе данных

* При наличии нескольких **экземпляров сервиса** проверьте, затронуты ли все экземпляры, а не только один.

* Проверьте, как часто запросы вызываются на один запрос. Возможно, удастся сократить количество запросов к базе данных, улучшив кэш базы данных сервиса. Если один запрос выполняется более одного раза за запрос, можно повысить производительность, применив интеллектуальные стратегии кэширования.

## Шаг 4 Проверьте количество соединений с базой данных

Низкая производительность базы данных может сохраняться даже при правильной настройке запросов. В таких случаях следует убедиться, что пул соединений с базой данных приложения правильно настроен.

При настройке пула соединений следует учитывать:

* Максимальное количество соединений, которые может обработать база данных
* Правильный размер пула соединений для приложения

Поскольку приложение может быть не единственным клиентом, подключённым к базе данных, следует убедиться, что размер пула соединений не установлен на максимум. Если приложение занимает все соединения, сервер базы данных не будет работать в ожидаемом режиме.

Как определить максимальное количество соединений

Максимальное количество соединений с базой данных зависит от ресурсов базы данных. Чтобы найти максимальное количество соединений, следует постепенно увеличивать нагрузку и количество разрешённых соединений с базой данных.

При этом необходимо следить за метриками сервера базы данных: CPU, памятью и производительностью диска. Как только один из этих показателей достигнет максимума, ограничение будет достигнуто. Если доступного количества соединений недостаточно для приложения, следует рассмотреть обновление оборудования.

Дополнительные сведения о размере пула соединений с базой данных см. в статье [About Pool Sizing](https://github.com/brettwooldridge/HikariCP/wiki/About-Pool-Sizing).

Как определить правильный размер пула соединений для приложения

Количество разрешённых одновременных соединений с базой данных эквивалентно объёму параллельной нагрузки, которую приложение создаёт на сервер базы данных. Существуют определённые инструменты для определения правильного числа.

Увеличение нагрузки приводит к более высокому времени ответа транзакций, даже если сервер базы данных работает исправно. Следует измерить сквозное время ответа транзакции, чтобы проверить, увеличивается ли время **Connection acquisition** при высокой нагрузке. Если это так, пул соединений может быть исчерпан. Если нет, нужно проверить метрики сервера базы данных для определения максимального количества соединений, которые база данных может обработать.

Размер пула соединений должен быть постоянным. Поэтому минимальный и максимальный размеры пула следует устанавливать равными.

## Шаг 5 Проверьте сеть

Физические ограничения виртуализированной инфраструктуры могут влиять на производительность базы данных: кабели могут выходить из строя, маршрутизаторы — ломаться. Сетевые метрики позволяют выявить невиртуальные проблемы. Например, если проблемы появляются после месяцев или даже лет безупречной работы, инфраструктура может испытывать физические неполадки. Следует проверить маршрутизаторы, кабели и сетевые интерфейсы.

![Анализ процессов хоста](https://dt-cdn.net/images/host-process-analysis-1922-6b4f5157a1.png)

Анализ процессов хоста

Чаще всего перегруженные процессы начинают отбрасывать пакеты при исчерпании ресурсов. Если сетевая проблема не связана с оборудованием, видимость на уровне процессов может помочь выявить неисправный компонент.
