# Host anomaly detection

# Обнаружение аномалий хостов

* How-to guide
* 4-min read
* Updated on Mar 11, 2024

Dynatrace автоматически обнаруживает аномалии производительности, связанные с инфраструктурой, такие как высокое насыщение CPU и нехватка памяти.

Можно настроить обнаружение аномалий хостов, включая пороги проблем и событий, в соответствии с конкретными потребностями. Например, если у вас система с:

* Низким потреблением ресурсов и вы хотите получать оповещения немедленно при превышении порогов.
* Высоким потреблением ресурсов и вы не хотите получать постоянные оповещения.

## Доступ к обнаружению аномалий

В Dynatrace обнаружение аномалий можно настраивать на нескольких уровнях — среды, хоста, группы хостов или для конкретных дисков.

На уровне хоста

На уровне группы хостов

На уровне среды

1. Перейдите в раздел **Hosts**.
2. Найдите и выберите ваш хост для отображения страницы обзора хоста.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**…**) > **Settings**.

4. В настройках хоста выберите **Anomaly detection** > **Infrastructure**.

1. Перейдите в **Deployment Status** > **OneAgents**.
2. На странице **OneAgent deployment** отключите **Show new OneAgent deployments**.
3. В поле **Filter by** введите **Host group** и выберите нужную группу хостов из раскрывающегося списка.

   Список хостов теперь отфильтрован по выбранной группе хостов. Каждый перечисленный хост имеет ссылку **Host group:** `<имя группы>`, где `<имя группы>` — имя группы хостов, которую требуется настроить.

   Свойство **Host group** не отображается, если выбранный хост не принадлежит ни одной группе хостов.
4. Выберите имя группы хостов в любой строке.

   Так как фильтр применён по группе хостов, все отображаемые хосты относятся к одной группе.

5. В настройках группы хостов выберите **Anomaly detection** > **Infrastructure**.

Перейдите в **Settings** > **Anomaly detection** > **Infrastructure** > **Hosts**.

## Установка порогов создания проблем

Можно настроить чувствительность обнаружения и включить оповещения для хостов и сетей.

### Хосты

1. Перейдите в **Settings** > **Anomaly detection** > **Infrastructure** > **Hosts**.
2. Включите или отключите доступные параметры для каждой настройки на странице или нажмите **Use defaults** в правом верхнем углу страницы.

   * При выборе **Automatic** из раскрывающегося меню будут использоваться встроенные настройки.
   * При выборе **Based on custom settings** настройте **Alerting event thresholds** и **Dealerting event thresholds**. Подробнее см. в разделе [Установка порогов событий](#event-thresholds).
3. Нажмите **Save changes**.

### Сети

1. Перейдите в **Settings** > **Anomaly detection** > **Infrastructure** > **Networks**.
2. Включите или отключите доступные параметры для каждой настройки на странице или нажмите **Use defaults** в правом верхнем углу страницы.

   * При выборе **Automatic** из раскрывающегося меню будут использоваться встроенные настройки.
   * При выборе **Based on custom settings** настройте **Alerting event thresholds** и **Dealerting event thresholds**. Подробнее см. в разделе [Установка порогов событий](#event-thresholds).
3. Нажмите **Save changes**.

## Установка порогов событий

Можно задать пороги оповещающих/деоповещающих событий для настроек обнаружения аномалий.

### Пороги оповещающих событий

* **Violating samples** — количество нарушающих 10-секундных выборок, при котором формируется оповещение. Значение должно быть выше числа выборок.
* **Evaluation window size for violating samples** — количество 10-секундных выборок, образующих скользящее окно оценки для обнаружения нарушающих выборок.

Примеры

* Окно 1 минута

  + **Violating samples**: 3
  + **Evaluation window size for violating samples**: 6
* Окно 5 минут

  + **Violating samples**: 15
  + **Evaluation window size for violating samples**: 30
* Окно 10 минут

  + **Violating samples**: 30
  + **Evaluation window size for violating samples**: 60

### Пороги деоповещающих событий

* **Dealerting samples** — количество ненарушающих 10-секундных выборок, при котором оповещение деактивируется. Значение должно быть ниже числа выборок.
* **Evaluation window size for dealerting samples** — количество 10-секундных выборок, образующих скользящее окно оценки для обнаружения деактивированных выборок.

Пороги событий недоступны для настроек **Detect host or monitoring connection lost problems** и **Detect high retransmission rate**.

Примеры

* Окно 1 минута

  + **Dealerting samples**: 3
  + **Evaluation window size for dealerting samples**: 6
* Окно 5 минут

  + **Dealerting samples**: 15
  + **Evaluation window size for dealerting samples**: 30
* Окно 10 минут

  + **Dealerting samples**: 30
  + **Evaluation window size for dealerting samples**: 60

## Установка порогов для конкретных дисков

Серверные оповещения о дисках для новых тенантов

Начиная с SaaS версии 1.308, серверные оповещения о дисках для новых тенантов отключены по умолчанию. Вместо них рекомендуется использовать [Disk Edge alerting](#disk-edge-alerting). Disk Edge alerting позволяет создавать более сложные и специфичные правила с использованием:

* Метрик для оповещений (доступное дисковое пространство, файловая система только для чтения, время чтения, время записи и доступные inodes)
* Операционной системы, к которой должна применяться политика
* Фильтров имён дисков
* Условий пользовательских метаданных хоста
* Пользовательских свойств, прикреплённых к сгенерированному событию

Обратите внимание, что для Disk Edge alerting требуется OneAgent version 1.293+.

Davis автоматически обнаруживает аномалии дисков, такие как недостаточное дисковое пространство или медленные диски. На хосте могут быть разные виды дисков: загрузочный, диск для хранения журналов или диск для хранения бизнес-данных. Оповещение о нехватке дискового пространства не имеет смысла для загрузочного диска с фиксированным размером образа, но вполне оправдано для диска с критически важными бизнес-данными.

С помощью пользовательских правил обнаружения дисков можно задать точные правила для отдельных групп дисков (группы определяются по шаблонам имён дисков и/или тегам хостов). Пороги на уровне диска переопределяют глобальные пороги для соответствующих дисков, тогда как глобальные настройки по-прежнему применяются к остальным дискам.

Чтобы изменить настройки порогов для группы дисков:

1. Перейдите в **Settings** > **Anomaly detection**.
2. В разделе **Infrastructure** выберите **Custom disk-detection rules**.
3. Нажмите **Add item**.
4. Выберите метрику для отслеживания и задайте понятное имя для правила.
5. Укажите порог для метрики и количество выборок, которые должны нарушить порог для создания оповещения.
6. Необязательно: укажите шаблон имени диска.

   Отдельные правила не связаны логически и применяются независимо. Например, если одно правило соответствует всем дискам, не содержащим `A`, а другое — всем дискам, не содержащим `B`, то каждый диск будет соответствовать первому, второму или обоим правилам одновременно. Добавление нескольких значений, подстановочных знаков или регулярных выражений в один фильтр правила не поддерживается.
7. Необязательно: для дополнительного уточнения использования диска перечислите теги, которые должен иметь хост.
8. Нажмите **Save changes**.

## Disk Edge alerting

OneAgent version 1.293+

Используйте **Disk Edge** для настройки оповещений автоматического обнаружения аномалий производительности, связанных с дисковой инфраструктурой.

Disk Edge обеспечивает автоматическое обнаружение аномалий производительности дисковой инфраструктуры. Эти настройки позволяют адаптировать чувствительность обнаружения к конкретному имени диска и/или пользовательским метаданным. Определение пользовательских свойств может помочь при последующей обработке события.

Политики можно определять на уровне хоста, группы хостов и среды.

1. Перейдите в **Settings** > **Anomaly detection** > **Infrastructure** > **Disk Edge**.
2. Нажмите **Add policy**.
3. Определите политику.

   * **Policy name**: имя, под которым будет отображаться политика.
   * **Operating system**: правила обнаружения можно задавать для выбранных операционных систем (если ни одна не выбрана, настройки не будут применяться).
   * **Alerts**: одно правило может содержать до семи оповещений — по одному для каждого типа события (см. таблицу ниже).
   * **Disk name filters**: правила можно фильтровать по имени диска.
   * **Host custom metadata conditions**: правила можно фильтровать по [пользовательским метаданным хоста](/managed/observe/infrastructure-observability/hosts/configuration/define-tags-and-metadata-for-hosts#host-metadata "Learn how to tag and set additional properties for a monitored host.").
   * **Properties**: к отправляемому событию можно добавить свойства с использованием поддерживаемых заменителей:

     + `disk.all_mountpoints`
     + `disk.device_name`
     + `disk.mountpoint`
     + `dt.entity_host`
     + `dt.host_group.id`
     + `host.name`

| Событие | Серьёзность | Связанная метрика OneAgent |
| --- | --- | --- |
| Available disk space (%) below | Меньшее значение — более серьёзное | DiskStats object field: `availPercentage`   Mintv2 metric: `dt.host.disk.free`   Timeseries: `builtin:host.disk.free` |
| Available disk space (MiB) below | Меньшее значение — более серьёзное | DiskStats object field: `avail`   Mintv2 metric: `dt.host.disk.avail`   Timeseries: `builtin:host.disk.avail` |
| Available inodes (%) below | Меньшее значение — более серьёзное | DiskStats object field: `availINodesPercentag`   Mintv2 metric: `dt.host.disk.inodes_avail`   Timeseries: `builtin:host.disk.inodesAvail` |
| Available inodes (number) below | Меньшее значение — более серьёзное | Calculated from Diskstats: `totalINodes` \* `availINodesPercentage`   Mintv2 metric: `dt.host.disk.inodes_avail` \* `dt.host.disk.inodes_total`   Timeseries: `builtin:host.disk.inodesTotal` \* `builtin:host.disk.inodesAvail` |
| Is read only file system | N/A | Disk object field: `readOnly`   Mintv2 metric: N/A   Timeseries: N/A |
| Read time (ms) exceeding | Большее значение — более серьёзное | Disk object field: `readTime`   Mintv2 metric: `dt.host.disk.read_time`   Timeseries: `builtin:host.disk.readTime` |
| Write time (ms) exceeding | Большее значение — более серьёзное | Disk object field: `writeTime`   Mintv2 metric: `dt.host.disk.write_time`   Timeseries: `builtin:host.disk.writeTime` |

## Иерархия конфигурации обнаружения аномалий

Правила и политики обнаружения аномалий можно настраивать на нескольких уровнях — хоста, группы хостов и среды.

При наличии нескольких правил, влияющих на одну и ту же сущность, более специфичное правило имеет приоритет над более общими.

### Правила для дисков

Пользовательские правила оцениваются сверху вниз, и применяется первое совпавшее правило — убедитесь, что правило размещено в правильной позиции в списке.

Диск назначается первой политике, которой он соответствует (на основе имени диска и/или метаданных), в соответствии с иерархией политик.

### Политики Disk Edge

Для Disk Edge порядок политик также оценивается от более специфичной области (с более высоким приоритетом) к более общей (с более низким приоритетом).
