# Настройки на уровне хоста

* How-to guide
* 2-min read
* Updated on Mar 28, 2024

Во многих случаях параметры мониторинга можно настраивать на уровне окружения, группы хостов или хоста.

* Настройки на уровне хоста переопределяют настройки на уровне группы хостов и окружения.
* Настройки на уровне группы хостов переопределяют настройки на уровне окружения.

Для настройки параметров на уровне хоста:

1. Перейдите в **Hosts**.
2. Найдите и выберите хост, чтобы перейти на страницу его обзора.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**…**) > **Settings**.

Здесь можно выбирать элементы на левой панели для навигации по страницам настроек на уровне хоста. В данном примере отображается страница **General** параметров HOST-001.

![Раздел General в настройках хоста](https://dt-cdn.net/images/select-host-settings-286-abb08ad1d1.png)

Раздел General в настройках хоста

Навигация по иерархии

Изменения параметров на уровне хоста переопределяют настройки на уровне группы хостов и окружения.

* Если на уровне хоста не было изменено ни одной настройки, информационное окно предлагает ссылку от настроек уровня хоста к эквивалентным настройкам на уровне группы хостов. Если для группы хостов также не настроены параметры, аналогичное информационное окно предлагает ссылку от настроек группы хостов к настройкам окружения.
* Если настройки на уровне хоста были изменены, информационное окно сообщает, что «Эти настройки переопределяют настройки [имя группы хостов] (Host group)». Можно выбрать имя группы хостов для перехода к этим настройкам или выбрать **Remove override**, чтобы удалить настройки уровня хоста и использовать настройки группы хостов.

## Общие настройки

На странице **General** в разделе **Host settings** отображается таблица мониторируемых технологий.

1. На странице **Host settings** уровня хоста выберите **General**.

   В таблице **Monitored technologies** перечислены технологии мониторинга на выбранном хосте:

   * **Technology** — название технологии
   * **Type** — тип мониторинга, например `JMX monitoring`, `OneAgent extension`, `Custom extension` и `Service insights`
   * **Configuration** — текущий уровень настройки для выбранного хоста.
   * **Monitoring** — состояние мониторинга этой технологии на выбранном хосте.
2. Выберите в столбце **Edit** для просмотра параметров конфигурации:

   * **Use host configuration** определяет, использовать ли настройки мониторинга на уровне хоста, переопределяя эквивалентные настройки окружения и группы хостов.
   * При включении **Use host configuration** становятся доступными настройки, специфичные для технологии.

Для управления настройками мониторинга хоста перейдите в [Host monitoring](#host-monitoring).

## Мониторинг хоста

OneAgent автоматически мониторит хост и его процессы, сервисы и приложения, однако можно отключить мониторинг, мониторинг полного стека или авто-инжекцию на уровне хоста.

1. На странице **Host settings** выберите **Host monitoring**.
   На этой странице есть три вкладки:

   * **Monitoring**
   * **Monitoring Mode**
   * **Advanced Settings**
2. Перейдите в **Monitoring** и установите **Monitor this host** для включения или отключения мониторинга выбранного хоста.
3. Перейдите в **Monitoring Mode** и установите **Full-Stack**, **Infrastructure** или **Discovery** для включения или отключения выбранного режима мониторинга на выбранном хосте.

   * Подробности см. в разделе [Режимы мониторинга OneAgent](/managed/platform/oneagent/monitoring-modes/monitoring-modes "Подробнее о доступных режимах мониторинга при использовании OneAgent.").
   * Режим мониторинга OneAgent будет автоматически перезаписывать этот параметр при каждом его изменении через oneagentctl или при подключении OneAgent.
4. Перейдите в **Advanced settings** и установите **ProcessAgent injection** или **CodeModule injection** для включения или отключения автоматической инжекции на выбранном хосте.
5. Выберите **Save changes**.

## Технологии контейнеров

Dynatrace OneAgent автоматически мониторит все процессы, запущенные на ваших мониторируемых хостах. В контейнерных средах (например, Kubernetes, OpenShift, Cloud Foundry или Docker) OneAgent автоматически внедряет кодовые модули в контейнеризованные процессы для обеспечения готовой к использованию полной стековой видимости приложений, работающих внутри контейнеров.

1. На странице **Host settings** выберите **Container technologies**.
2. Для каждого типа контейнера включите или отключите **Enabled**, чтобы определить, будут ли кодовые модули автоматически внедряться.

   * При включении авто-инжекция обеспечивает глубокий мониторинг всех процессов внутри контейнеров на уровне запросов и распределённых трассировок для выбранного хоста.
   * При отключении OneAgent не будет выполнять инжекцию в контейнер выбранного типа на выбранном хосте.

## Параметры дисков

OneAgent автоматически обнаруживает и мониторит все точки монтирования, однако можно создавать правила исключений для удаления определённых дисков из списка мониторинга.

1. На странице **Host settings** выберите **Disk options**.
2. При необходимости задайте специфичные для хоста параметры дисков.

   * **Show all NFS disks** Linux

     При отключении OneAgent пытается дедублировать NFS-диски. По умолчанию отключено.
   * **Exclude disks**

     Можно создавать правила исключений для удаления дисков из мониторинга.

     Подробности см. в разделе [Исключение дисков и сетевого трафика из мониторинга хоста](/managed/observe/infrastructure-observability/hosts/configuration/exclude-disks-and-network-traffic#disk-options "Узнайте, как исключить выбранные диски и сетевой трафик из мониторинга хоста.").

## Расширение Disk Analytics

Linux

Установите расширение Disk Analytics для получения более детальной видимости локальных хранилищ данных и их томов, разделов и RAID-экземпляров на хостах Linux.

1. На странице **Host settings** выберите **Disk Analytics Extension**.
2. Включите или отключите **Enable Disk Analytics data collection**, чтобы определить, собираются ли данные Disk Analytics на выбранном хосте.

   При включении сбора данных без добавления расширения данные видны только в [Data Explorer](/managed/analyze-explore-automate/explorer "Запрашивайте метрики и преобразуйте результаты для получения нужных аналитических данных.").

## Трафик NetTracer

Linux

NetTracer — это инструмент с открытым исходным кодом для отслеживания TCP-событий и сбора метрик сетевых соединений в Linux.

1. На странице **Host settings** выберите **NetTracer traffic**.
2. Включите или отключите **Enable NetTracer traffic network monitoring**, чтобы определить, будет ли NetTracer мониторить сетевой трафик на выбранном хосте.
3. Выберите **Save changes**.

Подробности см. в разделе [Расширенный мониторинг сети](/managed/observe/infrastructure-observability/networks-classic/network-monitoring-with-nettracer "Расширение мониторинга сети с помощью метрик сетевого трафика на контейнеризованных хостах Linux с использованием NetTracer.").

## Исключение сетевого трафика

OneAgent автоматически обнаруживает и мониторит весь сетевой трафик, однако можно исключить трафик на определённых сетевых интерфейсах или хостах из мониторинга.

1. На странице **Host settings** выберите **Exclude network traffic**.

   На этой странице есть два раздела:

   * **Exclude NIC** — в этом разделе перечислены все сетевые интерфейсы, исключённые из мониторинга сетевого трафика на конкретном интерфейсе.

     + Для добавления записи выберите **Add item**
     + Для удаления записи выберите ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") **Delete row**
     + Для редактирования записи разверните ![Expand row](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Expand row") **Details**
   * **Exclude IP** — в этом разделе перечислены все IP-адреса хостов, исключённые при вычислении связности (другие метрики по-прежнему будут вычисляться).

     + Для добавления записи выберите **Add item**
     + Для удаления записи выберите ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") **Delete row**
     + Для редактирования записи разверните ![Expand row](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Expand row") **Details**
2. Выберите **Save changes**.

Подробности см. в разделе [Исключение дисков и сетевого трафика из мониторинга хоста](/managed/observe/infrastructure-observability/hosts/configuration/exclude-disks-and-network-traffic#exclude-network-traffic "Узнайте, как исключить выбранные диски и сетевой трафик из мониторинга хоста.").

## Обнаруженные процессы

Страница **Detected processes** — это таблица процессов, обнаруженных на выбранном хосте, доступная только для чтения.

* Для управления настройками мониторинга процессов выберите ссылку **Process group monitoring**.
* Для включения или отключения глубокого мониторинга для определённых групп процессов на хосте выберите ссылку **Process group deep monitoring**.

Подробности см. в разделе [Глубокий мониторинг процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-monitoring "Способы настройки мониторинга группы процессов").

## Мониторинг группы процессов

Для включения или отключения глубокого мониторинга для определённой группы процессов на хосте:

1. На странице **Host settings** выберите **Process group monitoring**.

   В таблице перечислены группы процессов, имеющие специфичные для хоста настройки мониторинга.

   * Для добавления записи выберите **Add process group**, выберите **Process group** и установите **Monitoring state** для выбранной группы процессов.
   * Для удаления записи выберите ![Remove](https://dt-cdn.net/images/remove-icon-105c6a04c2.svg "Remove") **Delete row**
   * Для редактирования записи разверните ![Expand row](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Expand row") **Details** и измените настройки.
2. Выберите **Save changes**.

Подробности см. в разделе [Глубокий мониторинг процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-monitoring "Способы настройки мониторинга группы процессов").

## Флаги обнаружения групп процессов

На странице **Built-in detection rules** перечислены настройки флагов обнаружения групп процессов для выбранного хоста.

Для включения или отключения флагов обнаружения групп процессов:

1. На странице **Host settings** выберите **Process group detection flags**.

   Если настройки на уровне хоста не заданы, информационное окно предлагает ссылку на эквивалентные настройки на уровне группы хостов и, далее, на уровень окружения.
2. Включите или отключите правила обнаружения по необходимости для установки специфичных для хоста флагов обнаружения групп процессов.

   Наведите указатель на значок информации для правила, чтобы просмотреть его детали.
3. Выберите **Save changes**.

Подробности см. в разделе [Обнаружение групп процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-detection "Способы настройки обнаружения групп процессов").

## Декларативное группирование процессов

Dynatrace автоматически мониторит группы процессов известных типов технологий или потребляющих значительные ресурсы. С помощью декларативного группирования процессов можно автоматически мониторить дополнительные технологии.

На странице **Declarative process grouping** перечислены все группы процессов, определённые через декларативное группирование.

Для добавления декларативной группы процессов в таблицу:

1. На странице **Host settings** выберите **Declarative process grouping**.
2. Выберите **Add process group**.
3. Введите имя, идентификатор и настройку отчётности.
4. Выберите **Add detection rule**, чтобы описать, как обнаруживать группу процессов: свойство и условие. Для помощи с условиями обратитесь к инструкциям на экране.
5. Выберите **Save changes**.

Подробности см. в разделе [Обнаружение групп процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-detection#declare "Способы настройки обнаружения групп процессов").

## Доступность процессов

Используйте **Process availability** для мониторинга наличия хотя бы одного процесса, соответствующего указанному правилу мониторинга, на хосте. Если ни один из процессов, запущенных на хосте, не соответствует правилу, вы получите оповещение. При дополнительном включении [снимков экземпляров процессов](#process-instance-snapshots) будет доступен подробный отчёт об активности наиболее ресурсоёмких процессов, а также о последней активности процессов, соответствующих правилу.

Для добавления правила мониторинга процессов:

1. На странице **Host settings** выберите **Process availability**.
2. Выберите **Add monitoring rule**.
3. Введите имя правила.
4. Выберите **Add detection rule**, чтобы описать, как обнаруживать процесс: свойство и условие. Для помощи с условиями обратитесь к инструкциям на экране.
5. Выберите **Add property**, чтобы определить дополнительные свойства ключ-значение для прикрепления к инициированному событию.
6. Выберите **Save changes**.

Подробности см. в разделе [Доступность процессов](/managed/observe/infrastructure-observability/hosts/monitoring/process-availability "Мониторинг доступности и производительности ключевых процессов на ваших хостах.").

## Снимки экземпляров процессов

При включении снимков экземпляров процессов Dynatrace анализирует наиболее ресурсоёмкие процессы, запущенные на хосте, и процессы, мониторируемые через [Process availability](#process-availability). При возникновении инициирующего события метрики, зафиксированные за 10 минут до и 10 минут после события для этих процессов, отправляются в кластер.

Для включения снимков экземпляров процессов:

1. На странице **Host settings** выберите **Process instance snapshots**.
2. Включите **Enable Process instance snapshots**.
3. Установите **Reported processes limit** — максимальное количество процессов, которые будет отправлять хост.
4. Выберите **Save changes**.

## Обнаружение аномалий

Используйте страницы обнаружения аномалий на уровне хоста для настройки чувствительности обнаружения, установки порогов оповещений или отключения оповещений для выбранного хоста.

Для настройки обнаружения аномалий для хоста:

1. На странице **Host settings** выберите **Anomaly detection** > **Infrastructure**.
2. При необходимости скорректируйте специфичные для хоста пороговые значения.
3. Выберите **Save changes**.

Для настройки обнаружения аномалий для дисков хоста:

1. На странице **Host settings** выберите **Anomaly detection** > **Disks**.
2. При необходимости скорректируйте специфичные для хоста пороговые значения.
3. Выберите **Save changes**.

Подробности см. в разделе [Настройка чувствительности обнаружения аномалий для инфраструктуры](/managed/dynatrace-intelligence/anomaly-detection/adjust-sensitivity-anomaly-detection/adjust-sensitivity-infastructure "Настройка чувствительности обнаружения проблем для инфраструктуры.").

## Обновления OneAgent

Используйте **OneAgent updates** для настройки поведения обновления OneAgent на выбранном хосте.

* Автоматические обновления при первой возможности
* Автоматические обновления в периоды обновления
* Без автоматических обновлений

Обновления окружения, запущенные вручную, переопределяют индивидуальные настройки обновления хоста. Дополнительные сведения об обновлениях окружения см. в разделе [Обновления OneAgent](/managed/ingest-from/dynatrace-oneagent/oneagent-update#oneagent-environment-settings "Узнайте, как обновить OneAgent.").

Для настройки автоматического обновления OneAgent на выбранном хосте:

1. На странице **Host settings** выберите **OneAgent updates**.
2. Установите **Update mode**.
3. Выберите **Save changes**.

Для ручного обновления OneAgent на выбранном хосте:

1. Установите **Update OneAgent to a specific version** — версию OneAgent, которую нужно установить на выбранный хост.
2. Выберите **Update now**.

## Мониторинг сервисов ОС

Используйте **OS services monitoring** для настройки оповещений о сервисах ОС в нежелательных состояниях.

1. На странице **Host settings** выберите **OS services monitoring**.
2. Выберите **Add policy**.
3. Установите **System** (Linux или Windows).
4. Установите **Rule name**.
5. Определите условия оповещения.
6. Выберите **Save changes**.

Подробности см. в разделе [Мониторинг сервисов ОС](/managed/observe/infrastructure-observability/hosts/monitoring/os-services "Улучшение видимости инфраструктуры с помощью мониторинга доступности сервисов операционной системы.").

## Extension Execution Controller

Используйте **Extension Execution Controller** для настройки Extension Execution Controller (EEC) для развёртывания OneAgent.

1. На странице **Host settings** выберите **Extension Execution Controller**.
2. Настройте параметры EEC:

   * Установите **Enable Extension Execution Controller**.
   * Установите **Enable local HTTP Metric, Log and Event Ingest API**.
   * Установите **Enable Dynatrace StatsD**.
   * Установите **Performance profile**. Подробности см. в разделе [Профиль производительности — потребление ресурсов](#resource-consumption).
3. Выберите **Save changes**.

Подробности об EEC см. в разделе [О расширениях](/managed/ingest-from/extensions/concepts#eec "Подробнее о концепции расширений Dynatrace.").

## Мониторинг журналов

Используйте страницу **Log Monitoring** для настройки специфичных для хоста параметров мониторинга журналов.

1. На странице **Host settings** выберите **Log Monitoring**.
2. Откройте страницу настройки для параметров, которые нужно сконфигурировать на этом хосте.

   * Конфигурация OneAgent

* Конфигурация пользовательского источника журналов — подробности см. в разделе [Недоступно в Dynatrace Managed](/managed/upgrade/unavailable-in-managed "Ваш выбор недоступен в Dynatrace Managed.")
* Конфигурация хранилища журналов — подробности см. в разделе [Недоступно в Dynatrace Managed](/managed/upgrade/unavailable-in-managed "Ваш выбор недоступен в Dynatrace Managed.")
* Маскирование конфиденциальных данных — подробности см. в разделе [Недоступно в Dynatrace Managed](/managed/upgrade/unavailable-in-managed "Ваш выбор недоступен в Dynatrace Managed.")
* Конфигурация временной метки — подробности см. в разделе [Недоступно в Dynatrace Managed](/managed/upgrade/unavailable-in-managed "Ваш выбор недоступен в Dynatrace Managed.")
* Выберите **Save changes**.

## Аналитика дампов аварийного завершения

Используйте **Crash dump analytics** для управления автоматическим обнаружением аварийных завершений приложений.

1. На странице **Host settings** выберите **Crash dump analytics**.
2. Включите или отключите **Crash dump analytics**, чтобы определить, будет ли Dynatrace автоматически обнаруживать аварийные завершения приложений на выбранном хосте.
3. Выберите **Save changes**.

Подробности об аварийных дампах Linux и Windows см. в разделе [Анализ аварийных завершений](/managed/observe/application-observability/profiling-and-optimization/crash-analysis "Узнайте, как Dynatrace помогает получить представление об аварийных завершениях процессов.").
