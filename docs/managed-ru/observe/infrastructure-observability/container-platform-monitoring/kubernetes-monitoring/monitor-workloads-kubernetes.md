# Monitor Kubernetes/OpenShift workloads

# Мониторинг нагрузок Kubernetes/OpenShift

* 5-min read
* Updated on Nov 02, 2023

Dynatrace version 1.232+

При развёртывании только для приложений OneAgent отслеживает только память, диск, CPU и сеть процессов внутри контейнера. Метрики хоста не отслеживаются.

## Предварительные требования

* ActiveGate с включённым мониторингом Kubernetes API
* [Последний образ OneAgent с Docker Hub с тегом 1.38.1000+](https://hub.docker.com/r/dynatrace/oneagent)
* В Dynatrace перейдите на страницу настроек кластера Kubernetes и убедитесь, что параметр **Monitor Kubernetes namespaces, services, workloads, and pods** включён.

## Быстрый обзор среды Kubernetes

После включения поддержки мониторинга нагрузок Kubernetes можно легко просматривать, сколько ресурсов кластера было выделено через нагрузки, работающие в кластере.

Для отображения нагрузок Kubernetes перейдите в раздел **Kubernetes Workloads**.

## Единый аналитический вид для нагрузок, пространств имён и подов

Единый аналитический вид позволяет просматривать все данные, связанные с пространством имён, на обзорной странице конкретного пространства имён Kubernetes, все данные, связанные с нагрузкой, на обзорной странице конкретной нагрузки Kubernetes, и все данные, связанные с подом, на обзорной странице конкретного пода Kubernetes.

Настройка единого аналитического вида

Для настройки информации, отображаемой на странице единого анализа, выберите **More** (**…**) в правом верхнем углу любого раздела. Различные элементы навигации на странице единого анализа (**View all namespaces**, **View all workloads**, **View all pods** и т. д.) позволяют перейти непосредственно к любому разделу или подразделу, который требуется настроить.

### Пространства имён

Организации, использующие Kubernetes, часто разбивают приложения по пространствам имён для изоляции различных бизнес-единиц. Например, группа управления персоналом может размещать приложения в пространстве имён `hr`, а финансовая группа — в пространстве имён `finance`.

Страница единого анализа пространства имён предоставляет ценный обзор для таких бизнес-единиц, позволяя отслеживать объём выделенных ресурсов и сравнивать его с показателями использования.

На странице единого анализа пространства имён можно просматривать свойства, возможные проблемы, запросы и лимиты ресурсов, анализ нагрузок, квоты и события, а также все нагрузки, принадлежащие данному пространству имён (со ссылками на них). Пространства имён можно фильтровать по фильтрам измерений метрик.

Для отображения страницы единого анализа пространства имён в Dynatrace перейдите в раздел **Kubernetes Workloads** и выберите пространство имён.

### Нагрузки

Нагрузка (workload) состоит из одного или нескольких подов. Это способ описания типа микросервиса, составляющего приложение. Например, приложение может иметь frontend-нагрузку и backend-нагрузку, состоящие из десятка подов в кластере Kubernetes.

Страница единого анализа нагрузки предоставляет сведения об использовании ресурсов, обнаружении проблем, [уязвимостях](/managed/observe/infrastructure-observability/container-platform-monitoring/kubernetes-monitoring/monitor-vulnerabilities-kubernetes "Keep track of vulnerabilities in Kubernetes/OpenShift."), количестве подов в соответствующей нагрузке, количестве сервисов, передающих трафик на поды, и событиях для всех подов данной нагрузки. Эта информация ценна для анализа общей производительности микросервиса без необходимости изучать конкретные проблемы в отдельном экземпляре пода.

Для отображения страницы единого анализа нагрузки в Dynatrace перейдите в раздел **Kubernetes Workloads** и выберите нагрузку.

При детальном изучении приложений, развёрнутых в одном из пространств имён, можно ознакомиться с наиболее важными метриками использования ресурсов. Представление нагрузок охватывает нагрузки типов `Deployment`, `DeploymentConfig`, `ReplicaSet`, `DaemonSet`, `StatefulSet`, `StaticPod` и `ReplicationController`.

Метрика тротлинга CPU показывает, как долго приложение было ограничено, что позволяет определить, где требовалось больше процессорного времени. Обычно это происходит, когда контейнерам не хватает ресурсов CPU (лимитов) в определении нагрузки. Это может влиять на производительность процессов и приложений, работающих внутри контейнеров.

Также можно просматривать количество работающих подов по сравнению с желаемым количеством для каждого облачного приложения.

### Поды

Поды — наименьшая единица в Kubernetes и фактические экземпляры нагрузки. Страница единого анализа пода — это место, где анализируются конкретные проблемы при сбоях или замедлении пода из-за насыщения памяти или CPU.

На странице единого анализа пода можно просматривать свойства, возможные проблемы, использование ресурсов и события, а также контейнер, которому принадлежит под (со ссылкой на него).

Чтобы просмотреть обзорную страницу пода Kubernetes:

1. Перейдите в раздел **Kubernetes Workloads** и выберите нагрузку.
2. Выберите **Pods**.
3. Выберите нужный под.

### Ограничения при получении аннотаций

ActiveGate version 1.277 Dynatrace version 1.277

Аннотации Kubernetes получаются ActiveGate и отображаются на страницах сведений о сущностях для узлов, пространств имён, нагрузок, подов и сервисов Kubernetes. Однако следует учитывать ряд ограничений:

* Аннотация `kubectl.kubernetes.io/last-applied-configuration` не принимается.
* Максимальное количество аннотаций на сущность — 100.
* Максимальная длина принимаемого значения аннотации — 1024 символа. Значения аннотаций, превышающие эту длину, усекаются.

## Проверка наличия достаточных ресурсов CPU для приложений

Помимо возможностей автоматического обнаружения и трассировки, OneAgent собирает низкоуровневые метрики контейнеров, отражающие эффект лимитов ресурсов контейнеров.
Общие метрики ресурсов для всех поддерживаемых сред выполнения контейнеров на Linux доступны в пользовательских диаграммах и сгруппированы в разделах **Containers** > **CPU** и **Containers** > **Memory**.
Метрики количества работающих и желаемых подов также доступны в разделе **Cloud Platform**.

![Kubernetes dashboard](https://dt-cdn.net/images/kubernetes-dashboard-1920-97799f0ea5.png)

Панель мониторинга Kubernetes

Метрики времени тротлинга CPU и процента использования памяти показывают, правильно ли заданы лимиты ресурсов в спецификациях подов Kubernetes. Если использование памяти достигает 100%, контейнеры или приложения завершат работу (out of memory) и потребуют перезапуска.

## Детальное управление видимостью пространств имён и нагрузок через зоны управления

Зоны управления можно использовать для контроля доступа пользователей к данным мониторинга конкретных объектов Kubernetes в вашей среде. Например, можно ограничить доступ к конкретным нагрузкам и пространствам имён для определённых групп пользователей. Этот подход позволяет контролировать доступ пользователей к конкретным страницам Dynatrace Kubernetes, пользовательским диаграммам и панелям мониторинга.
