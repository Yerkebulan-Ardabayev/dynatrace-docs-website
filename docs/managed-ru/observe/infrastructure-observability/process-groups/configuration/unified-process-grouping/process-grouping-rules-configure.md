# Настройка правил группировки процессов

# Настройка правил группировки процессов

* Практическое руководство
* Чтение за 5 минут
* Опубликовано 24 марта 2026 г.

Правила группировки процессов позволяют объединить несколько групп процессов в одну или разделить группу процессов на отдельные, более детализированные группы.

## Перед началом работы

Необходимые знания

* [Правила группировки процессов: часто задаваемые вопросы](#faq)

Предварительные требования

* окружение Dynatrace с включёнными **правилами группировки процессов**.
* Права доступа: **администратор Environment** или права **process group configuration**.

## Доступ к настройкам правил группировки процессов

1. Перейти в **Settings** > **Processes and containers** > **Process group detection** > **Process grouping rules**.

## Первоначальная настройка

1. Выбрать **Add rule**.
2. Опционально Выбрать **Custom technology name**. Это имя будет отображаться на экране экземпляра группы процессов.
3. Выбрать **Extract process groups**.

## Определение свойств группы

1. Опционально Выбрать **Process group display name**.

   Если оставить это поле пустым, Dynatrace автоматически сгенерирует имя для группы.
2. Выбрать режим передачи данных, указав опцию в **Report process group**.

   Варианты передачи данных

   * **Always:** процесс всегда помечается как важный, и его данные всегда отправляются на сервер. Доступно как для процессов с глубоким мониторингом, так и без него.
   * **Automatic:** на сервер отправляются только процессы, которые OneAgent считает важными. Такие процессы удовлетворяют хотя бы одному из двух условий:

     + процессы с глубоким мониторингом;
     + процессы с высоким потреблением ресурсов.
   * **Never:** процесс никогда не передаётся.
3. Опционально Выбрать **Type of captured processes**.

   Рекомендации по использованию

   * Эту опцию можно использовать, чтобы ограничить созданное правило конкретными типами процессов и избежать смешения свойств глубокого мониторинга, что может привести к запутанным результатам.
   * Если планируется мониторинг более одной технологии, оставить это поле пустым.

## Определение правил обнаружения

1. Выбрать **Add detection rule**.

   Несколько условий

   * Можно добавить несколько условий. Добавленные условия отображаются в разделе **Summary** ниже.
   * Для срабатывания правила должны выполняться все условия.
2. Выбрать **Property** для обнаружения.
3. Создать условие обнаружения с использованием одной из функций сопоставления.

   Функции сопоставления

   * `$contains(svc)` — срабатывает, если `svc` встречается в любом месте значения свойства процесса.
   * `$eq(svc.exe)` — срабатывает, если `svc.exe` точно совпадает со значением свойства процесса.
   * `$prefix(svc)` — срабатывает, если `svc` совпадает с началом значения свойства процесса.
   * `$suffix(svc.py)` — срабатывает, если `svc.py` совпадает с концом значения свойства процесса.

   Например, `$suffix(svc.py)` обнаружит процессы с именами `loyaltysvc.py` и `paymentssvc.py`.
4. Опционально Выбрать **Case sensitive**. Если опция включена, условия сопоставления учитывают регистр. Если выключена, условия сопоставления регистр не учитывают.

## Определение правил группировки

Правило группировки определяет, как обнаруженные процессы будут разделены или объединены.

1. Выбрать, должно ли правило быть [автономным правилом](#standalone-rules).
2. Выбрать **ID type**, чтобы определить, какое свойство идентифицирует группу процессов.

   Пользовательский параметр расчёта ID

   Ввести **Process group identifier**. Этот идентификатор используется Dynatrace для распознавания данной группы процессов. Все процессы, обнаруженные правилом обнаружения, получают один и тот же seed для расчёта ID, в результате чего все обнаруженные процессы объединяются в единую группу процессов.

   Существующее свойство процесса

   1. Выбрать **Property**. Процессы динамически разделяются или объединяются на основе значения выбранного свойства:

      * процессы с одинаковым значением свойства объединяются в одну группу;
      * процессы с разными значениями разделяются на отдельные группы.
   2. Опционально Развернуть **Advanced settings**, чтобы задать разделители:

      * **Delimit from** — с какого места начинается значимая часть значения свойства.
      * **Delimit to** — где заканчивается значимая часть значения свойства.
3. Опционально Определить свойство **Process ID source** для идентификации отдельных процессов.
4. Опционально Добавить **Advanced settings** для задания разделителей.

   Расширенные настройки

   * **Delimit from** — с какого места начинается значимая часть значения свойства.
   * **Delimit to** — где заканчивается значимая часть значения свойства.

## Примеры

Разделение групп процессов

Есть процессы `server.js`, порождающие воркеры `worker.js`, все под управлением менеджера процессов PM2:

```
NodeJS v18.20.6 PM2 Process Manager



\_ node /my/path/server.js



|   \_ /bin/node /my/path/worker.js



\_ node /my/path/server.js



|   \_ /bin/node /my/path/worker.js



\_ node /my/path/server.js



|   \_ /bin/node /my/path/worker.js



\_ node /my/path/server.js



\_ /bin/node /my/path/worker.js
```

Все эти процессы автоматически группируются вместе. Менеджер процессов PM2 запускает процессы в контейнере особого типа, что вызывает подмену командной строки. Из-за этого командную строку нельзя использовать для разделения. Чтобы разделить их на группы серверных процессов и воркеров, нужно использовать свойство `Node.js script name`.

1. В **Type of captured processes** выбрать `Node.js`.
2. Добавить правило обнаружения с **Property**, установленным в `Node.js script name`, и условием `$eq(worker.js)`.
3. Добавить правило обнаружения с **Property**, установленным в `Node.js script name`, и условием `$eq(server.js)`.
4. В **Define grouping rules** включить **Standalone rule**.
5. Установить **ID type** в **Existing process property** и выбрать `Node.js script name`.

Это разделяет обнаруженные процессы на отдельные группы на основе имени скрипта: одна группа для `server.js` и одна для `worker.js`.

Объединение групп процессов

Есть два процесса:

```
\_ node /my/path/server.js



|



\_ python /my/path/server.py
```

Dynatrace автоматически группирует их раздельно, поскольку серверные процессы используют разные технологии. Чтобы объединить их в одну группу, нужно найти общее свойство, присущее только нужным процессам. Например:

* переменная Environment;
* путь к исполняемому файлу;
* любое общее свойство, характерное только для нужных процессов. Если свойство не уникально для этих процессов, другие процессы могут быть непреднамеренно сгруппированы вместе.

## Часто задаваемые вопросы

Что такое автономные правила?

Правила группировки процессов дают возможность определять группы процессов и экземпляры групп процессов с помощью автономных правил. Чтобы определить автономные правила, нужно включить эту функцию при создании нового правила. Это приводит к следующему:

* правила разделения Dynatrace по умолчанию отключаются для обнаруженного процесса;
* автономное правило становится единственным правилом, используемым для создания группы процессов.

Автономные правила действительны только для процессов с глубоким мониторингом.

В чём разница между Command line и Command line - single argument?

Два распространённых свойства, используемых для определения правил обнаружения, это `Command line` и `Command line - single argument`. Несмотря на схожие названия, это разные свойства с разным поведением сопоставления:

* Свойство **Command line** представляет собой всю командную строку целиком, включая исполняемый файл.
* Свойство **Command line — single argument** рассматривает каждый аргумент отдельно.

Для процесса вида:

```
myProgram.exe --instance=10 --port=9090 --arg=32
```

**Command line** сопоставляется с полной строкой, поэтому применимы следующие условия:

* `$contains(--port=9090)`
* `$suffix(=32)`
* `$prefix(my)`
* `$eq(myProgramm.exe --instance=10 --port=9090 --arg=32)`

**Command line — single argument** сопоставляется с каждым аргументом по отдельности, поэтому применимы следующие условия:

* `$contains(--port=9090)`
* `$suffix(exe)`, `$suffix(=10)`, `$suffix(9090)`, `$suffix(arg=32)`
* `$prefix(my)`, `$prefix(--instance)`, `$prefix(--p)`, `$prefix(--arg=)`
* `$eq(myProgramm.exe)`, `$eq(--instance=10)`, `$eq(--port=9090)`, `$eq(--arg=32)`

Что такое устаревшие переменные группировки?

Следующие переменные окружения являются устаревшими и их следует использовать только в крайнем случае, поскольку они могут приводить к ошибкам. Эти переменные доступны как переменные окружения, а также как пользовательские переменные ID кластера Dynatrace и пользовательские переменные ID узла Dynatrace.

Пользовательские переменные ID кластера Dynatrace:

* `DT_CLUSTER_ID`
* `RUXIT_CLUSTER_ID`

Пользовательские переменные ID узла Dynatrace:

* `DT_NODE_ID`
* `RUXIT_NODE_ID`

Когда требуется перезапуск процесса с глубоким мониторингом?

Процессы с глубоким мониторингом нужно перезапускать после каждого изменения конфигурации.

Чтобы помочь с этим, Dynatrace отображает предупреждение **Restart required** для каждого экземпляра группы процессов, когда обнаруживает, что загруженная конфигурация устарела.

Обратите внимание, что изменения технологии или имени группы в правиле не вызывают появления этого предупреждения, поэтому в интерфейсе всё ещё может отображаться старая технология или имя группы.

Какие процессы нельзя мониторить с помощью правил группировки процессов?

Следующие процессы нельзя мониторить с помощью правил группировки процессов:

* `conhost.exe`
* `cmd.exe`

Это связано с тем, что для них невозможно рассчитать ID. Другие процессы могут порождать тысячи таких процессов, поэтому OS Agent пропускает для них определённые данные, такие как рабочий каталог, командная строка и свойства контейнера, которые критически важны для расчёта ID.
