# Process deep monitoring

# Глубокий мониторинг процессов

* How-to guide
* 6-min read
* Updated on Aug 07, 2023

Установка OneAgent предоставляет возможности мониторинга групп процессов:

* Автоматический мониторинг всех групп процессов, обнаруженных в среде, после перезапуска всех процессов, выполнявшихся во время установки OneAgent.
* Полная видимость запросов всех автоматически обнаруженных серверных сервисов, включая сервисы баз данных.
* Полностью автоматизированный процесс мониторинга без необходимости конфигурации, что упрощает мониторинг крупных сред с сотнями хостов и тысячами процессов, где ручная настройка мониторинга всех сущностей нецелесообразна.

Дополнительно можно настраивать правила мониторинга для выборочного указания процессов, которые Dynatrace отслеживает. Например, рассмотрим следующие распространённые сценарии:

* У вас есть ряд неважных или кратковременных процессов, которые вы не хотите отслеживать на уровне кода.
* Вы не можете выполнять глубокий мониторинг приложений клиентов, которые не находятся под вашим контролем.
* Вы хотите лучше контролировать, какие процессы отслеживаются.
* Вы хотите выполнять глубокий мониторинг процессов .NET и Go (Dynatrace не выполняет его автоматически, поскольку на них опираются многие произвольные процессы). Например, вы хотите отслеживать все приложения ASP.NET и все приложения Go и .NET core, работающие в Cloud Foundry или Kubernetes.

Состояния мониторинга можно настроить в **Settings** > **Processes and containers** > **Process group monitoring**.

## Включение автоматического глубокого мониторинга

* По умолчанию автоматический глубокий мониторинг установлен в значение **On**, что позволяет Dynatrace OneAgent выполнять глубокий мониторинг всех обнаруженных процессов (если не заданы исключения для конкретных процессов или правила с исключениями). Отключайте этот параметр только если это требуется политиками компании.
* Установите **Off**, если вы хотите, чтобы Dynatrace OneAgent выполнял глубокий мониторинг только явно заданных процессов или процессов, охватываемых правилом глубокого мониторинга. После этого можно вручную включить мониторинг на уровне процесса или группы процессов либо определить правила мониторинга.

Отключение автоматического глубокого мониторинга:

1. Перейдите в **Settings**.
2. Выберите **Processes and containers** > **Process group monitoring**.
3. Отключите **Enable automatic deep monitoring**.
4. Выберите **Save changes**.

Применение правил мониторинга процессов

**Enable automatic deep monitoring** не имеет приоритета над [индивидуальными правилами мониторинга процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-detection "Ways to customize process-group detection"). Если правило мониторинга процессов указывает, что Dynatrace должен отслеживать определённый процесс, а **Enable automatic deep monitoring** выключен, индивидуальное правило будет иметь приоритет и Dynatrace будет отслеживать соответствующий процесс. Таким образом, **каждое правило мониторинга процессов является исключением из общей политики мониторинга**.

## Определение пользовательских правил мониторинга процессов

Пользовательские правила мониторинга процессов предоставляют детальный контроль над тем, какие процессы OneAgent отслеживает, — с подходом, легко масштабируемым в крупных средах. Изменения системной конфигурации не требуются, а несколько правил могут охватывать тысячи процессов.

Добавление пользовательского правила мониторинга:

1. Перейдите в **Settings**.
2. Выберите **Processes and containers** > **Custom process monitoring rules**.
3. Выберите **Add item**.
4. Установите **Mode** для определения основного условия:

   * **Monitor** — отслеживать процесс при выполнении условия.
   * **Do not monitor** — не отслеживать процесс при выполнении условия.
5. Определите **Condition**:

   * Объект условия (см. [правила обнаружения групп процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-detection "Ways to customize process-group detection")).
   * Оператор условия (например, `contains`).
   * Значение условия.
6. Выберите **Save changes**, чтобы сохранить конфигурацию и добавить новое правило в список пользовательских правил мониторинга.

Например, можно создать правило, запрещающее инъекцию OneAgent в любой процесс в пространствах Cloud Foundry, содержащих строку `customer`.

Редактирование существующего правила:

1. Выберите правило для настройки.
2. Выберите **Details** для редактирования.
3. Выберите **Save changes**.

Удаление правила:

1. Выберите правило для удаления.
2. Выберите **Delete**.
3. Выберите **Save changes**.

## Включение или отключение встроенных правил мониторинга процессов

Встроенные правила применяются к процессам, отслеживаемым Dynatrace по умолчанию:

* Все приложения .NET и Go в Kubernetes
* Все приложения .NET и Go в Cloud Foundry
* Все приложения .NET и Go, развёрнутые в контейнерах Docker
* Приложения ASP.NET Core, запускаемые IIS
* Основные компоненты Cloud Foundry, написанные на Go
* Caddy — веб-сервер, написанный на Go
* InfluxDB — база данных временных рядов, написанная на Go

Просмотр всех встроенных правил:

1. Перейдите в **Settings**.
2. Выберите **Processes and containers** > **Built-in process monitoring rules**.

Все встроенные правила включены по умолчанию. Их можно отключить, но нельзя редактировать.

Встроенные правила не охватывают ваши собственные приложения .NET и Go, если они не развёрнуты в контейнерах, Cloud Foundry или Kubernetes. В противном случае для ваших приложений .NET и Go следует добавить собственные [пользовательские правила мониторинга](/managed/observe/infrastructure-observability/process-groups/configuration/pg-detection "Ways to customize process-group detection").

Dynatrace не выполняет автоматически глубокий мониторинг **всех** процессов .NET и Go. Многие популярные приложения, такие как Microsoft Office, используют .NET, а многие распространённые компоненты инфраструктуры написаны на Go, поэтому Dynatrace выполняет глубокий мониторинг процессов .NET и Go только при явном включении или при наличии охватывающих правил мониторинга.

## Настройка состояний мониторинга на уровне группы хостов

Состояния мониторинга групп процессов можно задавать на уровне группы хостов.

1. Перейдите в **Hosts** и выберите интересующий хост.
2. На странице обзора хоста выберите **Properties and tags**.
3. На панели **Properties and tags** найдите свойство **Host group**, чтобы увидеть имя группы хостов, к которой принадлежит выбранный хост.

   Свойство **Host group** не отображается, если выбранный хост не принадлежит ни одной группе хостов.
4. Выберите имя группы хостов, чтобы просмотреть список всех хостов в ней. Откроется страница **OneAgent deployment**, отфильтрованная по выбранной группе. Каждый перечисленный хост имеет ссылку **Host group:** `<имя группы>`, где `<имя группы>` — имя группы хостов для настройки.
5. Выберите имя группы хостов в любой строке.

6. В настройках группы хостов раскройте **Process monitoring** и выберите один из доступных вариантов для настройки правил мониторинга.

Настройки группы процессов на уровне группы хостов переопределяют настройки на уровне среды.

## Настройка состояний мониторинга на уровне хоста

Можно добавить группу процессов и определить её состояния мониторинга на уровне хоста.

1. Перейдите в **Hosts**.
2. Найдите и выберите хост для отображения страницы его обзора.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**…**) > **Settings**.

4. В настройках хоста выберите **Process group monitoring**.
5. Выберите **Add process group** и выберите группу процессов из раскрывающегося списка.
6. Установите **Monitoring state** (`Monitor`, `Do not monitor` или `Default`).

   | Состояние мониторинга | Описание |
   | --- | --- |
   | **Monitor** | Настройки группы процессов на уровне хоста переопределяют соответствующие настройки на уровне группы хостов и среды. |
   | **Do not monitor** | Настройки группы процессов на уровне хоста переопределяют соответствующие настройки на уровне группы хостов и среды. |
   | **Default** | Состояние мониторинга наследуется от настроек предыдущего уровня. |

## Ограничения

* Правила глубокого мониторинга влияют только на мониторинг сервисов и уровня кода.
* Правила глубокого мониторинга эффективны только при установке OneAgent на хостах или образах.
* Интеграции только для приложений без полной установки OneAgent не поддерживают правила мониторинга. Однако в таких ситуациях сами интеграции эффективно обеспечивают тот же уровень контроля над настройкой мониторинга процессов.
* Правила могут работать на более ранних версиях OneAgent, но официально поддерживаются только для OneAgent version 1.151+.

## Включение или отключение мониторинга кратковременных процессов

Кратковременные процессы — это процессы, которые не обнаруживаются OneAgent в его стандартном 10-секундном цикле. Их можно частично отслеживать и назначать конкретным группам процессов, используя информацию о родительском процессе.

Включение или отключение мониторинга кратковременных процессов через мониторинг хоста OneAgent:

1. Перейдите в **Settings** > **Processes and containers** > **Built-in detection rules**.
2. Включите или отключите **Monitor short lived processes**.

Кроме того, кратковременные процессы, которые часто перезапускаются и внедряются автоматической инъекцией OneAgent, могут испытывать чрезмерную нагрузку из-за накладных расходов инъекции. Это может привести к задержкам в этих процессах при ограниченных преимуществах сбора данных из-за их короткого времени работы.

Если часто запускающиеся кратковременные процессы внедряются инъекцией, отключите для них глубокий мониторинг, чтобы избежать задержек, вызванных логикой инъекции.
