# Концепции очередей

# Концепции очередей

* Explanation
* 4-min read
* Updated on Jan 26, 2023

Очереди сообщений обеспечивают лёгкое хранилище для сообщений. Как правило, они реализованы с клиент-серверной архитектурой. В такой архитектуре приложения подключаются к очередям или топикам через клиенты обмена сообщениями, тогда как сами очереди и топики управляются сервером обмена сообщениями (например, менеджером очередей или брокером).

Очереди сообщений имеют форму очереди или топика. Они предоставляют конечные точки, позволяющие приложениям отправлять им сообщения, и конечные точки, позволяющие приложениям асинхронно получать сообщения от них или подписываться на топики.

* Очередь: одно сообщение получает ровно один потребитель, даже если к очереди подключено несколько потребителей (**модель «точка-точка»**).
* Топик: одно сообщение публикуется всем подписчикам топика (**модель «публикация-подписка»**).

## Предварительные требования

OneAgent должен работать в режиме [Full-Stack Monitoring](/managed/platform/oneagent/monitoring-modes/monitoring-modes "Подробнее о доступных режимах мониторинга при использовании OneAgent.") для обнаружения очередей и топиков в составе распределённых трассировок. Только в режиме Full-Stack Monitoring Dynatrace создаёт непрерывный поток сервисов через подключённые сервисы-производители и потребители.

## Типы объектов очередей: очереди и топики

OneAgent автоматически обнаруживает очереди и топики, когда мониторируемые приложения взаимодействуют с ними через инструментирование совместимых клиентов обмена сообщениями. Если очереди и топики не используются приложениями, OneAgent не может получить к ним доступ, даже если они доступны на сервере обмена сообщениями. Для проверки совместимых клиентов см. раздел [Поддержка технологий](/managed/ingest-from/technology-support "Технические сведения о поддержке Dynatrace конкретных платформ и фреймворков разработки.").

Dynatrace создаёт объекты **Queue** для всех обнаруженных очередей и топиков, являющихся частью распределённых трассировок. Эти объекты отображаются в таблице **Queues and topics** на странице **Queues**.

| Объект | Тип | Схема именования |
| --- | --- | --- |
| Queue | Queue | `<имя-очереди>` |
| Queue | Topic | `<имя-топика>` |
| Queue | IBM MQ queue | `<имя-менеджера-очередей>.<имя-очереди>` |
| Queue | IBM MQ topic | `<имя-менеджера-очередей>.<имя-топика>` |

Ограничение

Расширения Dynatrace могут обнаруживать очереди и топики, доступные на сервере обмена сообщениями, однако они не образуют объекты **Queue** в Dynatrace. Следовательно, они не видны в таблице **Queues and topics**. Расширения Dynatrace могут только добавлять технологически специфичные метрики к объектам, созданным OneAgent.

## Сервисы обмена сообщениями

### Сервисы-производители и потребители

* Сервис-производитель представляет приложение, отправляющее сообщения в очередь или топик через клиент обмена сообщениями.
* Сервис-потребитель представляет:

  + приложение, асинхронно получающее сообщения из очереди через [слушатель](#listener-service),
  + или приложение, подписавшееся на топик через [слушатель](#listener-service).
* В JMS-приложении (Java message service) может также существовать синхронный сервис-потребитель. В этом сценарии клиент может синхронно запрашивать следующее сообщение от `MessageConsumer` с использованием одного из его [методов receive](https://docs.oracle.com/javaee/7/api/javax/jms/MessageConsumer.html) (например, клиент может опрашивать или ожидать следующее сообщение).

Для предоставления непрерывного представления потоков сервисов Dynatrace использует следующие идентификаторы для трассировки сообщений через очереди и топики:

| Идентификатор | Тип |
| --- | --- |
| `traceparent` и `tracestate` | HTTP-заголовок запроса |
| `x-dynatrace` | Пользовательский HTTP-заголовок |
| `dtdTraceTagInfo` | Пользовательское свойство сообщения |
| Уникальный ключ (генерируется на основе свойств сообщения) | - |

### Сервисы-слушатели

Сервис-слушатель, или сервис-слушатель очереди, представляет слушатель очереди или слушатель топика. Он подсчитывает количество извлечённых из очереди сообщений, однако не даёт представления о самой обработке сообщений.

Dynatrace автоматически обнаруживает обработчик очереди сообщений на основе событий по его имени класса и создаёт для него сервис-слушатель очереди. Сервис-слушатель всегда сопровождается сервисом-потребителем, который даёт представление о деталях обработки сообщений.

Если нужно только мониторить очередь или топик, не вникая в детали обработки сообщений, сервис-слушатель может существовать самостоятельно.

Из-за своих свойств:

* Сервисы-слушатели не видны на страницах аналитики, доступных со страницы **Queues**, однако детали можно найти на страницах [**Service analysis**](/managed/observe/application-observability/services-classic "Узнайте о классическом мониторинге сервисов Dynatrace"), [**Service flow**](/managed/observe/application-observability/services-classic/service-flow "Узнайте, как Dynatrace помогает трассировать последовательность вызовов сервисов, инициированных каждым запросом к сервису в вашей среде.") и [**Distributed traces**](/managed/observe/application-observability/distributed-traces "Получите наблюдаемость в высокораспределённых, облачных нативных архитектурах для эффективной трассировки и анализа транзакций в реальном времени.").
* Запросы сервисов-слушателей нельзя переименовывать или закреплять на дашборде.

  Следует учитывать, что за сервисом-слушателем всегда следует сервис обмена сообщениями, над которым можно выполнять такие действия. Например, можно переименовывать запросы сервиса обмена сообщениями через (глобальные) [правила именования запросов](/managed/observe/application-observability/services/service-detection/service-detection-v1/set-up-request-naming "Настройка именования запросов и определение операций, предлагаемых сервисами."), используя имя очереди сообщений в качестве заполнителя, а затем [закрепить запрос на дашборде](/managed/analyze-explore-automate/dashboards-classic/charts-and-tiles/available-tiles#tile-service-or-request "Узнайте, как настроить дашборд для отслеживания критически важных для бизнеса действий пользователей и целей конверсии.").
