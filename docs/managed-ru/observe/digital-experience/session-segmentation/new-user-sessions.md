# Анализ сессий пользователей

* How-to guide
* 21-min read
* Updated on Oct 12, 2023

Dynatrace версии 1.224+

Анализ отдельных сессий пользователей бывает полезен в конкретных ситуациях, однако зачастую он не даёт полной картины. Пользователи вашего приложения ведут себя непредсказуемо, преследуют разные цели, находятся в разных географических регионах и используют самые разные комбинации устройств, операционных систем и браузеров.

Dynatrace поддерживает сегментацию сессий пользователей с помощью мощного механизма фильтрации. Анализ сессий пользователей Dynatrace позволяет разбивать сессии вашего приложения на значимые сегменты по общим характеристикам: операционной системе, типу браузера, местоположению или пользовательскому тегу. Например, вы можете сегментировать анализ по типам браузеров: настольный, мобильный или синтетический. Это позволяет детально изучить агрегированные результаты и выявить проблемы производительности, затрагивающие лишь небольшую часть пользователей.

## Анализ сессии пользователя

Для анализа сессии пользователя:

1. Перейдите в **Session Segmentation**.
2. Нажмите на текстовое поле в верхней части страницы (см. **1** в примере ниже) и выберите один из доступных атрибутов фильтрации.
   После выбора атрибута отобразится список доступных значений для этого фильтра.

   В фильтрах оператор тильды (`~`) не работает как ключевое слово `LIKE` в [USQL](/managed/observe/digital-experience/session-segmentation/custom-queries-segmentation-and-aggregation-of-session-data "Узнайте, как выполнять запросы к данным сессий пользователей с помощью ключевых слов, синтаксиса, функций и многого другого.").
3. Выберите интересующее вас значение атрибута. Некоторые атрибуты предоставляют текстовые поля для произвольного поиска. Вы также можете выбрать несколько значений одного атрибута — это работает как оператор `OR` для данного атрибута.
4. Повторите процесс для нужных атрибутов. После определения фильтра нажмите в любом месте за пределами текстового поля.

   Результат отображает первые 500 сессий, упорядоченных по времени начала (от новых к старым). Для изменения порядка сортировки кликните по заголовку столбца.
5. Выберите временную метку сессии (см. **3** в примере ниже) для перехода на страницу сведений о сессии. Или выберите имя пользователя (см. **2** в примере ниже) для перехода на [страницу этого пользователя](/managed/observe/digital-experience/session-segmentation/analyze-all-sessions-of-a-single-user "Узнайте о поведении пользователей, анализируя профиль пользователя и активность сессий.").

Используйте [выбор временного интервала](/managed/analyze-explore-automate/dashboards-classic/dashboards/dashboard-timeframe "Узнайте о настройках временного интервала и зоны управления дашборда Dynatrace.") в строке меню для настройки анализируемого периода.

Сессии пользователей, которые являются [текущими](/managed/observe/digital-experience/rum-concepts/user-session#live-vs-completed-sessions "Узнайте, как определяется сессия пользователя.") в пределах выбранного интервала, отображаются в списке сессий. Поиск по атрибуту **Analysis over time**, напротив, учитывает только сессии, начавшиеся в пределах заданного интервала.

Например, если интервал задан с 12:00 до 12:05, а сессия, начавшаяся в 11:55, продолжается в этот период, она будет показана в списке, но не будет учтена в поиске **Analysis over time**, поскольку начата до заданного интервала.

## Детализация с помощью анализа

Панель анализа расположена слева на странице **User sessions**. Панель содержит готовые результаты анализа и различные визуализации для разных атрибутов. Например, выберите категорию **Application versions**, чтобы увидеть, какая версия вашего приложения имеет больше сессий, или категорию **Applications**, чтобы просмотреть агрегированные данные по каждому приложению.

Вы можете выбрать любой из результатов анализа и применить его как фильтр, нажав **Apply selection as a filter** в нижнем левом углу страницы.

## Фокус на сессиях отдельного пользователя

Вы можете сфокусироваться на сессиях конкретного пользователя. Выберите пользователя в столбце **User**, чтобы перейти на [страницу обзора этого пользователя](/managed/observe/digital-experience/session-segmentation/analyze-all-sessions-of-a-single-user "Узнайте о поведении пользователей, анализируя профиль пользователя и активность сессий.").

Для поиска сессий конкретного пользователя выберите [**User tag**](/managed/observe/digital-experience/rum-concepts/user-and-error-events#user-tagging "Узнайте о событиях пользователей и ошибках, а также о типах событий, фиксируемых Dynatrace.") в поле **Filter by** и выберите нужного пользователя. Например, для отображения сессий пользователя `Zara` добавьте фильтр **User tag:** `Zara`. Затем выберите **Zara** в столбце **User** для перехода на страницу обзора этого пользователя.

Для настройки тегирования пользователей в зависимости от типа приложения см.:

* [Веб-приложения](/managed/observe/digital-experience/web-applications/additional-configuration/identify-individual-users-for-session-analysis "Тегируйте отдельных пользователей через JavaScript API для анализа сессий.")
* Мобильные приложения:
  + [Android](/managed/observe/digital-experience/mobile-applications/instrument-android-app/instrumentation-via-oneagent-sdk/oneagent-sdk-for-android#tag-specific-users "Узнайте, как расширить возможности мониторинга мобильного пользовательского опыта на Android с помощью OneAgent SDK.")
  + [iOS](/managed/observe/digital-experience/mobile-applications/instrument-ios-app/customization/oneagent-sdk-for-ios#tag-specific-users "Расширьте возможности мониторинга мобильного пользовательского опыта с помощью OneAgent SDK для iOS.")
* [Пользовательские приложения: OpenKit](/managed/ingest-from/extend-dynatrace/openkit/dynatrace-openkit-api-methods#tag-specific-users "Узнайте, как использовать Dynatrace OpenKit с точки зрения разработчика.")

## Просмотр сведений об ошибках

Вы также можете использовать анализ сессий для получения подробной информации об [ошибках](/managed/observe/digital-experience/rum-concepts/user-and-error-events#error "Узнайте о событиях пользователей и ошибках, а также о типах событий, фиксируемых Dynatrace.") в вашем приложении.

Для просмотра сведений об ошибке:

1. Перейдите в **Session Segmentation**.
2. В поле **Filter by** задайте **Error type** одно из следующих значений в зависимости от типа приложения:

   | Тип ошибки | Описание | Web | Mobile | Custom |
   | --- | --- | --- | --- | --- |
   | Request error | Обнаружена браузером и OneAgent на серверах | Применимо | Применимо | Применимо |
   | Reported error | Сообщается вручную через специальный метод API | Не применимо | Применимо | Применимо |
   | Custom error | Сообщается вручную через RUM JavaScript API | Применимо | Не применимо | Не применимо |
   | JavaScript error | Исключения JavaScript в браузере | Применимо | Не применимо | Не применимо |

3. Выберите интересующую вас сессию для открытия страницы сведений о ней.
4. В разделе **Events and actions** разверните действие пользователя, содержащее ошибку, и выберите **Perform waterfall analysis**.
5. Выполните одно из следующих действий в зависимости от типа приложения:

   * Веб-приложения: на странице **Waterfall analysis** выберите плитку **Error**, а затем выберите ошибку. Откроется страница сведений об ошибке.
   * Мобильные и пользовательские приложения: прокрутите до раздела **Web request errors** или **Reported errors** и выберите ошибку. Откроется страница сведений об ошибке.

Страница сведений об ошибке предоставляет подробную информацию: предполагаемое количество ошибок, провайдер (для request error), технология (для reported error) и другое. На странице также перечислены затронутые сессии и действия пользователей. Разбивка по распределению отображает сведения об относительной частоте операционных систем, версий ОС, версий приложений и устройств, а разбивка по странам показывает все затронутые страны и соответствующий уровень ошибок.

## Проверка сессий с rage events

Когда приложение не реагирует достаточно быстро, текстовая метка выглядит как кнопка или переключатель скрыт под другим элементом, пользователи могут многократно нажимать на затронутый элемент управления. Dynatrace обнаруживает такое поведение как [rage event](/managed/observe/digital-experience/rum-concepts/user-and-error-events#rage-event "Узнайте о событиях пользователей и ошибках, а также о типах событий, фиксируемых Dynatrace."): **rage click** или **rage tap**.

Для просмотра сессий с rage events:

1. Перейдите в **Session Segmentation**.
2. Задайте **Filter by** значение **Rage click count:** `>= 1` или **Rage tap count:** `>= 1`.
3. Выберите интересующую вас сессию для открытия страницы сведений о ней.
4. Прокрутите до раздела **Events and actions** и разверните rage click или rage tap, чтобы просмотреть подробности.

## Анализ сбоев

Мобильные и пользовательские приложения

Когда сессия пользователя заканчивается [сбоем](/managed/observe/digital-experience/rum-concepts/user-and-error-events#crash "Узнайте о событиях пользователей и ошибках, а также о типах событий, фиксируемых Dynatrace."), вы можете использовать анализ сессий для просмотра полной последовательности действий пользователя, предшествовавших сбою. Также вы можете открыть отчёт о сбое для получения информации на уровне кода и быстрого выявления первопричин.

1. Перейдите в **Session Segmentation**.
2. Задайте следующие фильтры в поле **Filter by**:

   * **Application type:** `Mobile` для мобильных сессий или **Application type:** `Custom` для пользовательских приложений
   * **Crashes:** `Yes` для сессий, завершившихся сбоем
   * Мобильные приложения: **Session Replay:** `Yes` для сессий, записанных с Session Replay on crashes для [Android](/managed/observe/digital-experience/session-replay/session-replay-android "Настройте Session Replay для Android-приложений.") или [iOS](/managed/observe/digital-experience/session-replay/session-replay-ios "Предварительные условия и процедура включения Session Replay для iOS.")
3. Выберите интересующую вас сессию для открытия её страницы сведений.
4. Мобильные приложения: для просмотра записи Session Replay перейдите на вкладку **Session Replay** и нажмите **Play**. Последнее событие сессии является сбоем, отображаемым красной точкой на временной шкале. Используйте элементы управления Session Replay для детального анализа сбоя.
5. Для просмотра всех действий и событий пользователя, предшествовавших сбою, прокрутите до раздела **Events and actions**.
6. Для просмотра отчёта о сбое разверните событие сбоя и выберите **Open crash details**. Отчёт предоставляет информацию об устройстве пользователя и трассировку стека. Также вы можете анализировать группы сбоев для [мобильных](/managed/observe/digital-experience/mobile-applications/analyze-and-use/crash-reports-mobile#crash-groups "Проверьте последние отчёты о сбоях для мобильных приложений.") и [пользовательских](/managed/observe/digital-experience/custom-applications/analyze-and-use/crash-reports-custom#crash-groups "Проверьте последние отчёты о сбоях для пользовательских приложений.") приложений или загрузить трассировку стека.

## Анализ сессии в нескольких доменах

По техническим и соображениям безопасности анализ одной сессии пользователя в разных доменах невозможен.

Если пользователь посещает два совершенно разных домена в рамках одной «сессии», а оба домена инструментированы Dynatrace, вы увидите две отдельные сессии:

1. Первая сессия начинается с загрузки страницы первого домена и заканчивается, когда пользователь переходит по ссылке на страницу другого домена.
2. Вторая сессия начинается с первой загрузки страницы на втором домене и заканчивается при выполнении любого из [критериев завершения сессии](/managed/observe/digital-experience/rum-concepts/user-session "Узнайте, как определяется сессия пользователя.").

Причины: куки не могут быть общими для разных доменов (за исключением поддоменов) и это является ограничением безопасности браузеров.

## Анализ в реальном времени

* Доступ к текущим сессиям через плитку **Live user activity** на дашборде.
* Перейдите на одну из страниц проблем и выберите **See user sessions sample**.
* Затронутые сессии можно также открыть непосредственно со страницы проблемы.
* Отказы и конверсии доступны для завершённых сессий.

## Примеры

Фильтрация сессий по длительности

Вы можете фильтровать сессии по длительности: дольше или короче определённого значения или в определённом диапазоне.

Поиск всех сессий, содержащих хотя бы одно из указанных действий пользователя

Вы можете искать все сессии пользователей, которые содержат хотя бы одно из нескольких указанных действий в своих clickpaths.

Создание сложных фильтров

Пример фильтра, охватывающего сессии, соответствующие следующим критериям:

* **Application type: Web** и **Browser Type: Desktop Browser**. Пользователь посетил веб-приложение через настольный браузер.
* **Duration 60s**. Длительность сессии не менее 1 минуты.
* **Action count = 5**. Пользователь выполнил пять действий в ходе сессии.
* **User action name: test**. В сессии присутствовало действие пользователя `test`.

Просмотр категорий на панели анализа

Для каждой категории на панели анализа имеется отдельный раздел с визуализациями и результатами для этой категории. Например, **Application type** отображает текущее распределение, распределение во времени и географическое распределение различных типов приложений.

Вы можете выбрать любой результат и применить его как фильтр, нажав **Apply selection as a filter** в нижнем левом углу страницы.
