# Мониторинг доступности сети

* Updated on Aug 23, 2024

Dynatrace Managed версии 1.296+ ActiveGate версии 1.295+

Мониторинг доступности сети (NAM) позволяет отслеживать доступность удалённых хостов или служб по сети, когда HTTP/HTTPS-эндпоинт недоступен.

NAM можно использовать для задач, связанных с инфраструктурой, или для углублённого анализа первопричин проблем HTTP-мониторов и браузерных мониторов.

Вы можете создавать синтетические мониторы доступности сети типа **ICMP**, **TCP** или **DNS**. Подробности о создании NAM-мониторов см. в разделе [Настройка NAM-монитора](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed "Узнайте, как создать NAM-монитор и управлять им для проверки производительности и доступности сайта.").

## Типы NAM

Существует три типа мониторов доступности сети.

* ICMP — отправляет пинги с настраиваемым количеством пакетов или размером для проверки наличия сетевого соединения с хостом или устройством. Также проверяет качество этого соединения.
* TCP — устанавливает TCP-соединение с определённым портом. Проверяет, открыт ли порт и принимает ли он TCP-соединения. Также проверяет доступность хоста по сети.
* DNS — проверяет возможность разрешения имени хоста в IP-адрес.

Подробности о типах NAM см. в разделе [доступных свойств конфигурации скрипта](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#script-properties "Узнайте, как создать NAM-монитор и управлять им для проверки производительности и доступности сайта.").

## Отчётность

Выполнение монитора по запросу

Выполнение монитора возможно через [API](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-monitor-execution "Просматривайте результаты выполнений синтетических мониторов через API Synthetic v2."). Период хранения данных о выполнении монитора составляет 6 часов.

Для отображения обзора монитора выберите **Network availability monitors** в **Synthetic Classic**. Появится список всех ваших NAM-мониторов; нажмите на имя монитора для открытия страницы обзора (отчётности).

Страницы отчётности по умолчанию включают:

* Metrics (метрики)
* Request metrics (метрики запросов)
* Метрики TCP, ICMP или DNS
* Monitored hosts (мониторируемые хосты, отображаются только если монитор определён с выражением фильтра)
* Events (события)

### Metrics

В разделе **Metrics** отображаются:

* столбчатая диаграмма с количеством выполнений монитора;
* линейный график со средним временем выполнения монитора;
* линейный график со средней доступностью монитора.

Для каждого из перечисленного можно нажать ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More") для **Show in data explorer**, **Create metric event**, **Pin to dashboard**.

### Request metrics

В разделе **Request metrics** отображаются:

* столбчатая диаграмма со средним количеством выполнений запросов;
* линейный график со средней доступностью запросов.

Для каждого из перечисленного можно нажать ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More") для **Show in data explorer**, **Create metric event**, **Pin to dashboard**.

### Метрики TCP, ICMP и DNS

В этом разделе отображается диаграмма с одной из следующих метрик в зависимости от типа монитора:

* TCP — диаграмма времени соединения по запросу;
* ICMP — диаграммы процента успешных запросов по запросу, отправленных пакетов по запросу, полученных пакетов по запросу и времени кругового рейса по запросу;
* DNS — диаграмма времени разрешения.

Для каждого из перечисленного можно нажать ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More") для **Show in data explorer**, **Create metric event**, **Pin to dashboard**.

### Monitored hosts

В этом разделе отображается список мониторируемых хостов с возможностью сортировки по имени. При выборе конкретного имени хоста открывается его детальное представление в разделе **Hosts**.

### Events

В этом разделе отображается столбчатая диаграмма событий. Если в списке есть сбой, нажмите ![Expand row](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Expand row") для просмотра его деталей.

## Проблемы

Проблемы создаются на уровне монитора.

1. При наличии проблемы с монитором вы увидите **Problems** под именем монитора. Нажмите **Problems**, затем выберите **Go to Problems**.
2. В разделе **Problems** отображаются мониторы, затронутые проблемами; каждая проблема содержит хотя бы один из следующих элементов:

   * Affected steps — разверните **Step Id** для просмотра хотя бы одного из следующих элементов: идентификатор запроса, тип запроса, причина сбоя и сообщение о статусе.
   * Affected requests — разверните **Request target** для просмотра адреса цели запроса, идентификатора запроса, типа запроса и причин сбоя.
   * Affected locations.
   * Дата первого неудачного выполнения.

## Ограничения

При использовании мониторов доступности сети существуют определённые ограничения, подробно описанные ниже.

### Количество запросов

Максимальное количество сетевых операций, выполняемых одним монитором доступности сети, составляет 1000. Сетевая операция — это один DNS-запрос, один TCP-запрос или один ICMP-пакет. Dynatrace может использовать несколько пакетов в рамках одного ICMP-запроса, если это настроено.

Если монитор использует [фильтр цели](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#target-filter-managed "Узнайте, как создать NAM-монитор и управлять им для проверки производительности и доступности сайта."), точное количество запросов может быть неизвестно до выполнения (например, при мониторинге целой группы хостов или подсети с широким диапазоном IP). В таких случаях ограничение применяется при разрешении фильтра цели в фактический список адресов перед выполнением монитора.

Количество запросов зависит от числа хостов, соответствующих фильтру. Например, при мониторинге целой группы хостов или **фильтрации мониторируемых хостов** с использованием подсети с широким диапазоном IP.

### Количество мониторов

Максимальное количество NAM-мониторов на среду составляет 5000. Другие типы мониторов не учитываются в этом ограничении.

### Тип развёртывания

Мониторы доступности сети поддерживаются только в [частных расположениях Synthetic](/managed/observe/digital-experience/synthetic-monitoring/private-synthetic-locations/create-a-private-synthetic-location "Узнайте, как создать частное расположение для синтетического мониторинга.").

Контейнеризованные расположения

Мониторы доступности сети поддерживаются в [контейнеризованных](/managed/observe/digital-experience/synthetic-monitoring/private-synthetic-locations/containerized-locations#nam-monitors-on-containerized-locations "Развёртывайте контейнеризованные автоматически масштабируемые частные расположения Synthetic на Kubernetes/RedHat OpenShift и управляйте ими.") развёртываниях ActiveGate с поддержкой Synthetic, однако для ICMP-тестов требуются дополнительные разрешения.

Включение типа запроса ICMP для выполнения NAM

1. В **Settings** выберите и разверните **Web and mobile monitoring**.
2. В разделе **Web and mobile monitoring** выберите **Private Synthetic Locations**.
3. Нажмите **Add Kubernetes location**.
4. Настройте расположение и убедитесь, что включён параметр **Enable ICMP request type for Network Availability Monitors execution**.

ICMP-мониторы используют исполняемый файл `ping`, которому требуется возможность `CAP_NET_RAW` для контейнера, выполняющего запросы (`synthetic-vuc`). Кроме того, свойство `allowPrivilegeEscalation` в `securityContext` этого контейнера должно быть установлено в `true`, поскольку процесс, запускающий `ping`, по умолчанию не имеет необходимых привилегий.

Полный `securityContext` для контейнера `synthetic-vuc` с включёнными мониторами доступности сети должен выглядеть следующим образом.

```
securityContext:



readOnlyRootFilesystem: true



privileged: false



allowPrivilegeEscalation: true



runAsNonRoot: true



capabilities:



drop: ["all"]



add: ["NET_RAW"]
```
