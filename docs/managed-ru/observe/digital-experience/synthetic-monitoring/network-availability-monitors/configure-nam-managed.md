# Настройка NAM-монитора

* Updated on Jan 19, 2026

С помощью мониторов доступности сети (NAM) можно проверять доступность хостов, устройств и служб.

Существует три типа NAM-мониторов: ICMP, TCP и DNS. Подробности о них см. в разделе [Типы NAM](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/nam-for-managed#NAM-types).
Создавать и управлять NAM-мониторами можно через [API](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#nam-api "Узнайте, как создать NAM-монитор и управлять им.") или [веб-интерфейс](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#nam-web-ui "Узнайте, как создать NAM-монитор и управлять им.").

## Настройка NAM

NAM-мониторы могут включать более одного шага.

* Каждый шаг может содержать один или несколько запросов.
* Как и в HTTP-мониторах и браузерных мониторах, шаги выполняются последовательно.
* В отличие от HTTP-мониторов и браузерных мониторов, NAM-мониторы могут содержать несколько запросов в рамках одного шага. Все запросы, назначенные конкретному шагу, выполняются параллельно. Сбой одного запроса не влияет на выполнение других запросов в рамках этого шага.
* Шаг пропускается, если ничто не соответствует его определению в конфигурации монитора. Например, если в списке [фильтров цели](#target-filter) указан тег, но ничто ему не соответствует.
* Концепция параллельного выполнения запросов существует только для NAM-мониторов.

Например, если вы хотите отслеживать группу из 4 хостов с помощью ICMP-теста, вы хотите применить одинаковые условия (например, частоту, расположение, выполняющее тест, и количество пакетов) для каждого хоста из вашей группы.

NAM предлагает несколько способов решить эту задачу:

* Можно определить 4 отдельных теста — по одному на хост. Преимущество: Dynatrace создаёт отдельную проблему для каждого хоста, и для каждого можно настроить отдельные уведомления. Также можно независимо настраивать параметры тестирования для каждого хоста.
* Можно определить один тест с 4 запросами (в рамках 1 шага). Выполняются те же ICMP-проверки, но отчётность и оповещения будут отличаться. Количество или процент недоступных хостов отображается в метрике «Процент успешных запросов» (Requests Success rate). Можно настроить пользовательский порог для завершения всего монитора с ошибкой. Например, если допустимо, что 1 из 4 хостов недоступен из-за обновления, можно задать порог `>=75%`. Для монитора всегда создаётся одна проблема, однако она содержит подробную информацию о недоступных хостах. Ещё одно преимущество — более простое обслуживание (изменение одного параметра для всех 4 хостов).

  Фильтры позволяют определять тесты для динамически изменяющейся структуры. Например, если вы хотите проводить ICMP-тесты для определённой группы хостов, вам не нужно обновлять конфигурацию NAM-монитора при изменении группы хостов.

Для каждого монитора необходимо определить ограничения: условия, которые должны быть выполнены для успешного выполнения монитора. Ограничение **Success rate** обязательно. Подробности см. в разделе [Ограничения на уровне шага](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#step-level-constraints "Узнайте, как создать NAM-монитор и управлять им.").

## Управление NAM-мониторами через API

Управление NAM-мониторами через REST API

1. Найдите и выберите **Dynatrace API**.
2. В поле **Select a definition** выберите **Synthetic - Network availability monitors**.
3. Аутентифицируйтесь с помощью API-токена.
4. Выполните одно из следующих действий.

   | Задача | Раздел **Synthetic - Network availability monitors** |
   | --- | --- |
   | Получить все синтетические мониторы | **GET /synthetic/monitors** |
   | Создать определение синтетического монитора | **POST /synthetic/monitors** |
   | Удалить определение синтетического монитора по идентификатору | **DELETE /synthetic/monitors/{monitorId}** |
   | Получить определение синтетического монитора по идентификатору | **GET /synthetic/monitors/{monitorId}** |
   | Обновить определение синтетического монитора по идентификатору | **PUT /synthetic/monitors/{monitorId}** |

Подробности см. в разделе [Synthetic monitors API v2](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-network-availability-monitors "Создавайте, читайте, обновляйте и удаляйте синтетические мониторы. В настоящее время только браузерные мониторы и мониторы доступности сети.").

## Управление NAM-мониторами через веб-интерфейс

Managed версии 1.302+

Хотя рекомендуется управлять NAM через API, для тестирования возможностей NAM можно использовать простой веб-интерфейс. Создание NAM-монитора через веб-интерфейс позволяет быстро настраивать отдельные мониторы без прямого взаимодействия с API.

Интерфейс создания NAM в Managed использует пары ключ-значение. Для полной поддержки веб-интерфейса мониторов доступности сети требуется Dynatrace SaaS.

Помимо создания NAM-мониторов через UI, можно также удалять мониторы или изменять их конфигурацию.

### Создание монитора доступности сети

Создание NAM-монитора через веб-интерфейс

1. В среде Dynatrace Managed перейдите в **Digital Experience** > **Synthetic**.
2. Под количеством синтетических мониторов вверху страницы нажмите **List of Network Availability Monitors is available here**.
3. Нажмите **Create a Network Availability Monitor**.

Для каждого шага конфигурации NAM необходимо нажать **Save changes**. В противном случае изменения будут отменены.

1. Monitor name (имя монитора)

1. На вкладке **Monitor name** введите **Monitor name**.
2. Нажмите **Save changes**.

2. Configuration (конфигурация)

1. Необязательно На вкладке **Configuration** введите **Monitor description**.
2. Нажмите **Add step** и задайте **Step name**.
3. Выберите **Request type** из списка, например `ICMP`.
4. Нажмите **Add IP address or DNS name** для определения цели монитора, например `1.1.1.1` или `community.dynatrace.com`. Для добавления другого адреса цели нажмите **Add IP address or DNS name** ещё раз.
5. Необязательно Задайте **Target filter**. Подробности о доступных фильтрах цели см. в разделе [Target filter](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#target-filter "Узнайте, как создать NAM-монитор и управлять им.").
6. Необязательно Нажмите **Add configuration property** и заполните пару ключ-значение **Configuration properties**. Подробности см. в разделе [Configuration properties](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#script-properties "Узнайте, как создать NAM-монитор и управлять им."). Для добавления другого свойства нажмите **Add configuration property** ещё раз.

   Пример пары ключ-значение свойства конфигурации

   * Ключ свойства:
     `ICMP_PACKET_SIZE`
   * Значение свойства:
     `32`
7. Обязательно Нажмите **Add step-level constraint** и установите **Constraint type** в `SUCCESS_RATE_PERCENT`. Ограничение на уровне шага применяет условие успешного выполнения монитора на уровне шага и включает все запросы в рамках шага. Подробности см. в разделе [Ограничения на уровне шага](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#step-level-constraints "Узнайте, как создать NAM-монитор и управлять им.").
8. Дважды нажмите **Add property** для создания двух полей ключей свойств и заполните их по следующему примеру:

   | Ключ свойства | Значение свойства |
   | --- | --- |
   | **operator** | `=` |
   | **value** | `100` |

   Ключи **operator** и **value** чувствительны к регистру и должны быть записаны в нижнем регистре. В противном случае конфигурация не будет работать.

   Creating NAM with Managed web UI, step-level value property example
9. Обязательно Нажмите **Add request-level configuration**.

Шаги с **10** по **12** обязательны только для ICMP- и DNS-мониторов. Эти шаги можно пропустить и оставить **Request configuration** пустым для других типов мониторов.

10. Нажмите **Add constraint**.
11. Установите **Constraint type** в одно из доступных ограничений на уровне запроса, например `ICMP_SUCCESS_RATE_PERCENT`. Подробности см. в разделе [Ограничения на уровне запроса](/managed/observe/digital-experience/synthetic-monitoring/network-availability-monitors/configure-nam-managed#request-level-constraints "Узнайте, как создать NAM-монитор и управлять им.").

12. Дважды нажмите **Add property** для создания двух полей ключей свойств и заполните их по следующему примеру:

13. Нажмите **Save changes**.

3. Frequency and locations (частота и расположения)

NAM-мониторы поддерживаются только в частных расположениях.

1. На вкладке **Frequency and locations** выберите нужные **Frequency** и **Location** из доступных вариантов.
2. Нажмите **Save changes**.

4. Outage handling (обработка сбоев)

1. На вкладке **Outage handling** выберите, следует ли генерировать проблему и отправлять оповещение при недоступности монитора во всех расположениях. Затем в **Alert if all locations are unable to access my target address** выберите один из доступных вариантов. Отключите **Generate a problem and send an alert when the monitor is unavailable at all configured locations**, если оповещение не нужно.
2. Включите **Generate a problem and send an alert when the monitor is unavailable for one or more consecutive runs at any location**, если хотите получать оповещения о недоступности монитора в каком-либо из расположений. Затем выберите количество расположений и количество раз, в течение которых монитор должен быть недоступен.
3. Нажмите **Save changes**.

5. Performance thresholds (пороговые значения производительности)

1. Необязательно На вкладке **Performance thresholds** включите **Generate a problem and send an alert on performance threshold violations**, если хотите получать оповещения о проблемах производительности.
2. Необязательно Нажмите **Add threshold**.
3. Необязательно Оставьте настройки по умолчанию или задайте порог и тип агрегации.
4. Нажмите **Save changes**.

### Удаление монитора доступности сети

Managed версии 1.308+

Для удаления NAM-монитора необходимы права на редактирование всех схем, связанных с монитором.

Удаление NAM-монитора через веб-интерфейс

1. В среде Dynatrace Managed перейдите в **Digital Experience** > **Synthetic**.
2. Под количеством синтетических мониторов вверху страницы нажмите **List of Network Availability Monitors is available here**.
3. Выберите монитор, который хотите удалить.
4. Нажмите **Delete** в верхнем правом углу.
5. Появится сообщение «Are you sure you want to delete network availability monitor `имя монитора`?». Нажмите **Delete**.

   * При успешном выполнении появится сообщение «Network availability monitor `имя монитора` successfully deleted.»
   * При сбое появится сообщение «Failed to delete network availability monitor `имя монитора`.» Сбой может произойти, если монитор уже был удалён другим пользователем.

## Target filter (фильтр цели)

Target filter предоставляет возможность фильтровать **хосты, мониторируемые Dynatrace**, или **пользовательские устройства с IP-адресами**. С помощью этого фильтра можно выбирать эти два типа целей на основе:

* типа (`type`)
* тегов (`tag`)
* идентификатора хоста (`hostId`) (устарел, работает только для хостов; используйте entity ID)
* идентификатора сущности (`entityId`)
* групп хостов (`hostGroup`)
* зон управления (`managementZone`)
* IP-маски (`ipMask`)
* диапазона IP (`ipRange`)
* экземпляра группы процессов (`processGroupInstance`)
* сетевых интерфейсов конкретного пользовательского устройства (`interfacesOf`)
* имени расширения (`extensionName`)

Диапазон IP и IP-маска — это фильтры для хостов или устройств, известных серверу Dynatrace, а не возможность сканировать сеть.

### Синтаксис

* Логические операторы: `AND` и `OR` (без учёта регистра)
* Круглые скобки
* Операторы выражений: `==` и `!=`
* Имена и значения тегов
* Отрицание («не»): `!=`.
* Символ подстановки: `*` (выбирает все хосты, мониторируемые Dynatrace)

### Примеры

* `tag == tagname or hostGroup == group1`
* `(tag == tagname1:tagvalue1 or tag == tagname1:tagvalue2) and (hostGroup == group1 or managementZone == zone1)`
* `tag != tagname1 and tag != tagname2:tagvalue`
* `tag == tagname:tagvalue and (managementZone == zone1 or managementZone == zone2)`
* `tag == "[tagwithbrackets and spaces]":"value, with, commas, and, spaces"`
* `ipMask == 127.0.0.1/24`
* `hostId == HOST-000123`
* `type == CUSTOM_DEVICE and ipMask == 172.17.0.2/24`
* `entityId == HOST-045BFCDA3F507D30 or entityId == CUSTOM_DEVICE-13081D4B74B3E2C8`
* `type == HOST and processGroupInstance == PROCESS_GROUP_INSTANCE-07611353BB98908C`
* `type == CUSTOM_DEVICE and interfacesOf == CUSTOM_DEVICE-E1A88946BF04D5E7`
* `type == CUSTOM_DEVICE and extensionName == "Docker devices"`

## Доступные свойства конфигурации скрипта

Необязательно

Мониторы состоят из запросов, всегда одного типа (ICMP, TCP или DNS).

Если значение не указано, применяются значения по умолчанию.

Примеры формата длительности

* 100 миллисекунд = `PT0.1S`
* 500 миллисекунд = `PT0.5S`
* 1 секунда = `PT1S`
* 10 секунд = `PT10S`
* 90 секунд = `PT1M30S`
* 1 минута = `PT1M`
* 2 минуты = `PT2M`
* 5 минут = `PT5M`

### ICMP

| Имя | Тип | Описание | Значения | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `EXECUTION_TIMEOUT` | string:duration | Тайм-аут выполнения одного запроса. | Допустимая длительность, например `PT1S`. Диапазон = `0`–`PT2M` | `ICMP_NUMBER_OF_PACKETS` × `ICMP_TIMEOUT_FOR_REPLY` + `1s` |
| `ICMP_NUMBER_OF_PACKETS` | integer | Количество эхо-запросов. Аналог параметров `-c` (Linux) и `-n` (Windows) команды ping. | Диапазон = `1`–`10` | `1` |
| `ICMP_PACKET_SIZE` | integer | Длина данных. Аналог параметров `-s` (Linux) и `-l` (Windows) команды ping. | Диапазон = `0`–`65500` | `32` |
| `ICMP_TIME_TO_LIVE` | integer | Время жизни (TTL). Аналог параметров `-t` (Linux) и `-i` (Windows) команды ping. | Диапазон = `1`–`255` |  |
| `ICMP_TYPE_OF_SERVICE` | integer | Тип обслуживания. Аналог параметров `-Q` (Linux) и `-v` (Windows) команды ping. | Диапазон = `0`–`255` |  |
| `ICMP_DO_NOT_FRAGMENT_DATA` | boolean | Не фрагментировать. Аналог параметров `-M do` (Linux) и `-f` (Windows) команды ping. | `true` или `false` |  |
| `ICMP_TIMEOUT_FOR_REPLY` | string:duration | Ожидание эхо-ответа. Аналог параметров `-W` (Linux) и `-w` (Windows) команды ping. | Допустимая длительность, например `PT1S`. В Windows допустимы значения вроде `PT1.1S` (точность до миллисекунды). В Linux требуются значения вроде `PT1S` (точность до секунды). Диапазон = `PT1S`–`PT2S` | `PT1S` |

Интервал передачи пакетов жёстко задан равным `200ms`, что соответствует минимальному допустимому значению параметра `-i` команды ping в Linux без требования повышенных привилегий.

### TCP

| Имя | Тип | Описание | Значения | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `TCP_PORT_RANGES` | string | Разделённый запятыми список диапазонов портов. Один диапазон может быть либо единственным номером порта, либо диапазоном портов, заданным двумя номерами через дефис. Итоговый список выполняемых запросов — это произведение всех заданных портов и целевых хостов. Например, если шаг имеет два целевых хоста (`1.1.1.1` и `2.2.2.2`) и диапазон из двух портов (`80-81`), в рамках шага выполняются четыре запроса: запрос к `1.1.1.1` на порт `80`, запрос к `1.1.1.1` на порт `81`, запрос к `2.2.2.2` на порт `80`, запрос к `2.2.2.2` на порт `81`. | Примеры: `8080`, `8000-9000`, `80,443,90-100` |  |
| `EXECUTION_TIMEOUT` | string:duration | Тайм-аут соединения | Допустимая длительность, например `PT1S`. Диапазон = `0`–`PT2M` | `1s` |

### DNS

| Имя | Тип | Описание | Значения | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `DNS_RECORD_TYPES` | string | Разделённый запятыми список типов DNS-записей. Итоговый список выполняемых запросов — это произведение всех заданных типов записей и целевых хостов. Например, если шаг имеет два типа записей (`A,AAAA`) и два целевых хоста (`host1.domain.com` и `host2.domain.com`), выполняются четыре запроса: запрос содержимого записи `A` для `host1.domain.com`, запрос содержимого записи `AAAA` для `host1.domain.com`, запрос содержимого записи `A` для `host2.domain.com`, запрос содержимого записи `AAAA` для `host2.domain.com`. | Примеры: `A`, `A,AAAA`, `A,AAAA,CNAME` |  |
| `EXECUTION_TIMEOUT` | string:duration | Тайм-аут соединения | Допустимая длительность, например `PT1S`. Диапазон = `0`–`PT2M` | `2s` |
| `DNS_SERVER` | string | Адрес DNS-сервера для запроса с опциональным портом. Если указано имя хоста, оно разрешается в IP-адрес с использованием системного DNS-сервера по умолчанию. | Допустимый адрес, например: `1.1.1.1`, `8.4.4.8:53`, `dns.google`, `dns9.quad9.net:53` | Если указан только хост, используется порт `53` по умолчанию. Если значение не указано, используются системный DNS-сервер и порт по умолчанию. |
| `DNS_FORCE_TCP` | boolean | По умолчанию DNS-сервер запрашивается по UDP-соединению, если только сообщение не слишком велико для UDP-датаграммы. Этот параметр позволяет указать TCP-соединение. | `true` или `false` | `false` |

## Ограничения конфигурации скрипта

Ограничения — это условия, которые должны быть выполнены для успешного выполнения монитора. Ограничения необходимо определить для каждого монитора.

### Ограничения на уровне шага

#### `SUCCESS_RATE_PERCENT`

Обязательно

Это обязательное ограничение для всех мониторов по проценту успешных запросов. Значение по умолчанию: `>=80%`.

Процент успешных запросов в рамках шага.

Фактический процент успеха = отношение количества запросов, которые не завершились с ошибкой, к общему количеству запросов.

Например, если 1 из 6 запросов завершился с ошибкой, соотношение составляет (6-1)/6 = 83,33%.

| Свойство | Тип | Описание | Значение | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `operator` | string | Оператор сравнения для сравнения фактического и ожидаемого значений | `>=`, `>`, `<=`, `<`, `=` или `!=` | `>=` |
| `value` | integer | Ожидаемое значение процента успеха для сравнения | Диапазон = `0`–`100` | `80` |

```
{



"type": "SUCCESS_RATE_PERCENT",



"properties": {



"value": "80",



"operator": ">"



}



}
```

Для запросов с единственной целью рекомендуется устанавливать значения ограничений равными `100`.

### Ограничения на уровне запроса

#### Ограничения ICMP-запросов

##### `ICMP_SUCCESS_RATE_PERCENT`

Обязательно

Это обязательное ограничение для ICMP-мониторов по проценту успешных запросов. Значение по умолчанию: `>=80%`.

Процент успешных пингов (эхо-запросов) в рамках запроса.

Фактический процент успеха = отношение количества полученных пакетов к количеству отправленных.

Например, если было отправлено 5 пакетов и получено 4, соотношение составляет 4/5 = 80,00%.

| Свойство | Тип | Описание | Значение | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `operator` | string | Оператор сравнения для сравнения фактического и ожидаемого значений | `>=`, `>`, `<=`, `<`, `=` или `!=` | `>=` |
| `value` | integer | Ожидаемое значение процента успеха для сравнения | Диапазон = `0`–`100` | `80` |

```
{



"type": "ICMP_SUCCESS_RATE_PERCENT",



"properties": {



"value": "80",



"operator": ">"



}



}
```

#### Ограничения TCP-запросов

##### `TCP_PORT_UNREACHABLE`

Необязательно

Это необязательное ограничение для TCP-мониторов.
Это специальное ограничение, которое инвертирует [статус выполнения](#execution-statuses) для TCP-запросов. Его следует применять, если ожидается, что проверяемый порт недоступен.

Это ограничение не имеет свойств конфигурации.

При применении этого ограничения:

* Статусы `Execution timeout (12033)` и `TCP socket connection error (22000)`, обнаруженные после попытки соединения, интерпретируются как `HEALTHY (0)`.
* Статус соединения `HEALTHY (0)` интерпретируется как `CONSTRAINT_VIOLATED (1401)`, поскольку ожидался сбой.
* Статусы `UNEXPECTED_ERROR (-1)` и `UNKNOWN_HOST (12013)` сохраняются и отображаются как есть.

```
{



"type": "TCP_PORT_UNREACHABLE",



"properties": {}



}
```

#### Ограничения DNS-запросов

Необходимо хотя бы одно

Ограничения DNS являются **необязательными**, однако хотя бы одно должно быть определено для указания, что считается успешным запросом.

##### `DNS_STATUS_CODE`

По умолчанию `DNS_STATUS_CODE` равен `1`; код статуса `=0` является простейшим примером успешного запроса.

| Свойство | Тип | Описание | Значение | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `operator` | string | Оператор сравнения для сравнения фактического и ожидаемого значений | `=` или `!=` | `=` |
| `statusCode` | integer | Числовой код, указывающий статус DNS-ответа. Исключает `status` | Допустимый код, например `0`. Диапазон = `0`–`65535` | `0` |
| `status` | string | Мнемоника, представляющая код статуса DNS. Исключает `statusCode` | Допустимый статус, например `NOERROR` | `NOERROR` |

```
{



"type": "DNS_STATUS_CODE",



"properties": {



"operator": "=",



"statusCode": "0"



}



}
```

```
{



"type": "DNS_STATUS_CODE",



"properties": {



"operator": "=",



"status": "NOERROR"



}



}
```

##### `DNS_IP_ADDRESS`

Проверяет значение IP-адреса, возвращённого в записях A/AAAA.

| Свойство | Тип | Описание | Значение | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `quantifier` | string | Сколько записей должны соответствовать условию для выполнения ограничения | `any` (хотя бы одна) или `all` | `any` |
| `operator` | string | Оператор сравнения | `=`, `!=` или `in` | `=` для адреса, `in` для подсети |
| `recordType` | string | Тип DNS-записи | `A` или `AAAA` | `A` для адреса/подсети IPv4; `AAAA` для адреса/подсети IPv6 |
| `address` | string | IPv4-адрес в точечной нотации или IPv6-адрес в нотации с двоеточием. Только с операторами `=`, `!=`. Исключает `subnet`. | Допустимый адрес, например `192.168.0.1`, `2001:db8::2:1` |  |
| `subnet` | string | Подсеть IPv4 в точечной нотации или подсеть IPv6 в нотации с двоеточием с длиной префикса сети. Только с оператором `in`. Исключает `address`. | Допустимая подсеть, например `172.22.80.0/20`, `2001:db8:85a3::0/48` |  |

```
{



"type": "DNS_IP_ADDRESS",



"properties": {



"quantifier": "any",



"recordType": "A",



"operator": "=",



"address": "54.208.9.71"



}



}
```

```
{



"type": "DNS_IP_ADDRESS",



"properties": {



"quantifier": "all",



"recordType": "A",



"operator": "in",



"subnet": "10.102.44.0/24"



}



}
```

##### `DNS_RECORD_VALUE`

Проверяет необработанное значение записей заданного типа; формат значения зависит от типа записи.

Ограничения обрабатывают все записи в ответе, независимо от раздела, к которому они принадлежат (`answer`, `authority` или `additional`).

| Свойство | Тип | Описание | Значение | Значение по умолчанию |
| --- | --- | --- | --- | --- |
| `quantifier` | string | Сколько записей должны соответствовать условию для выполнения ограничения | `any` (хотя бы одна) или `all` | `any` |
| `operator` | string | Оператор сравнения. `contains` — проверяет, содержит ли значение записи заданную последовательность. `matches` — проверяет, соответствует ли значение записи заданному шаблону. | `contains` или `matches` | `contains` |
| `recordType` | string | Тип DNS-записи | Допустимый тип записи, например `TXT` |  |
| `sequence` | string | Последовательность символов, которую должно содержать значение записи. Только с оператором `contains`. Исключает `pattern`. | Непустая последовательность, например `ms71815323` |  |
| `pattern` | string | Шаблон для сопоставления со значением записи, где `*` обозначает 0 или более символов. Только с оператором `matches`. Исключает `sequence`. | Непустой шаблон, например `"v=spf1 include:*` |  |

```
{



"type": "DNS_RECORD_VALUE",



"properties": {



"quantifier": "any",



"recordType": "TXT",



"operator": "contains",



"sequence": "ms71815323"



}



}
```

```
{



"type": "DNS_RECORD_VALUE",



"properties": {



"quantifier": "any",



"recordType": "TXT",



"operator": "matches",



"pattern": "\"v=spf1 include:*"



}



}
```
