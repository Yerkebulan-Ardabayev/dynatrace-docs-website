# On-demand synthetic monitor executions for CI/CD

# On-demand synthetic monitor executions for CI/CD

* Explanation
* 17-min read
* Updated on Jul 03, 2024

ActiveGate версии 1.233+

Помимо планирования выполнений синтетических мониторов через регулярные интервалы (в настройках **Frequency and locations**), можно также запускать выполнения по требованию для [браузерных](/managed/observe/digital-experience/synthetic-monitoring/browser-monitors/configure-browser-monitors "Узнайте о настройке браузерных мониторов и clickpath-ов.") и [HTTP](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic "Узнайте о настройке HTTP-мониторов.")-мониторов. Можно даже запускать выполнения по требованию для [автоматически создаваемых HTTP-мониторов для синхронизации учётных данных](/managed/observe/digital-experience/synthetic-monitoring/general-information/external-vault-integration "Синхронизируйте учётные данные Synthetic Monitoring с внешними хранилищами."). В [веб-интерфейсе Dynatrace](#ui) нажмите **On-demand execution** в верхней части страницы деталей монитора.

![Trigger on-demand executions](https://dt-cdn.net/images/on-demand-execution-ui-2174-b5b18ea2b7.png)

Trigger on-demand executions

Также можно использовать мощный [API **Synthetic - On-demand monitor executions**](#api) для запуска выполнений нескольких мониторов и получения результатов. Запросы на выполнение по требованию имеют приоритет над запланированными выполнениями и помещаются в начало очереди запросов.

Выполнения по требованию можно запускать из **любого** назначенного или не назначенного расположения [через веб-интерфейс](#ui) или [API](#api).

При создании и редактировании синтетического монитора также можно настроить выполнение монитора только по требованию: выберите частоту **On demand only**.

![On-demand frequency](https://dt-cdn.net/images/frequencyondemandonly-617-ae2bdb37f6.jpg)

On-demand frequency

Выполнения по требованию с их мощными функциями являются ценным инструментом в стратегии CI/CD:

* Мониторы можно выполнять по требованию из любого расположения: публичного или частного.
* Результаты выполнения по требованию можно включать или исключать из общих результатов монитора.
* Выполнения можно включать или исключать из обнаружения проблем.
* Выполнения можно запускать через веб-интерфейс или API.

Эти выполнения могут проверять успешность нового развёртывания программного обеспечения и могут интегрироваться с инструментами сборки, такими как Jenkins или [Keptn](https://v1.keptn.sh/docs).

Выполнения по требованию также очень ценны как быстрые тесты мониторов, например, для тестирования монитора после разработки сложного скрипта. Аналогично, такие выполнения могут проверять эффективность исправления проблемы, обнаруженной Dynatrace, например, после добавления памяти хосту или возврата к более старой версии приложения.

Благодаря возможности включения или исключения из общих результатов монитора и обнаружения проблем выполнения по требованию предлагают значительно больше вариантов, чем [локальное воспроизведение браузерных clickpath-ов](/managed/observe/digital-experience/synthetic-monitoring/browser-monitors/record-a-browser-clickpath#playback "Узнайте, как записать браузерный clickpath для мониторинга доступности и производительности приложения.").

## Выполнения по требованию через веб-интерфейс

Кнопка **On-demand execution** на страницах деталей синтетических мониторов позволяет запускать выполнения для выбранного монитора из всех или выбранного расположения. Также отображается список всех выполнений по требованию для монитора за последние шесть часов, запущенных любым пользователем через веб-интерфейс или [API](#api).

![On-demand execution page](https://dt-cdn.net/images/on-demand-executions-http-monitor-2207-0d9244213b.png)

On-demand execution page

### Запуск выполнения

Любой пользователь с разрешением **View environment** на уровне окружения или зоны управления может запускать выполнения через веб-интерфейс.

Даже если вы [отключаете запланированное выполнение синтетического монитора в течение окон обслуживания](/managed/analyze-explore-automate/notifications-and-alerting/maintenance-windows/define-maintenance-window#disable-synthetic "Создавайте окна обслуживания и определяйте их область."), вы всё равно можете выполнять мониторы по требованию.

1. Нажмите **On-demand execution** в верхней части страницы деталей монитора.
2. Вы перейдёте на страницу **On-demand execution**. Выберите **All assigned locations** или любое назначенное или не назначенное публичное или частное расположение.
3. Выберите **Processing mode** (обратите внимание: это применяется ко всем выполнениям при выборе всех расположений).

   * **Standard** (по умолчанию): выполнения учитываются в обнаружении проблем и статистике доступности и производительности. Для браузерных мониторов данные точки можно просматривать на [странице **Multidimensional analysis**](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/multidimensional-analysis-for-browser-monitors "Узнайте, как анализировать точки данных браузерных мониторов."). Для HTTP-мониторов результаты выполнения доступны при [**Analyze execution details**](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic "Узнайте о странице Synthetic details для HTTP-мониторов.").
   * **Disable problem detection**: выполнения учитываются в статистике доступности и производительности, но не в обнаружении проблем. Для браузерных мониторов точки данных появляются на странице **Multidimensional analysis**; для HTTP-мониторов результаты выполнения можно просматривать при **Analyze execution details**.
   * **Execution details only**: выполнения не учитываются в статистике доступности и производительности, а также в обнаружении проблем. Для HTTP-мониторов результаты всё равно доступны при **Analyze execution details**; для браузерных мониторов эти точки данных появляются на странице **Multidimensional analysis**.

   Обратите внимание: для всех режимов обработки выполнения видны в списке выполнений по требованию в веб-интерфейсе и доступны через [API](#api) выполнений по требованию. Сводная и подробная информация о выполнении доступна в течение шести часов через API.
4. **Fail on performance threshold violation**: по умолчанию выполнения по требованию завершаются неуспешно при нарушении порогового значения производительности, поскольку их основная цель — проверка новых версий программного обеспечения в CI/CD-конвейере. Однако можно отключить этот параметр, чтобы выполнения по требованию вели себя как регулярно запланированные (то есть нарушения пороговых значений производительности не вызывают сбоев мониторов).
5. Только для HTTP-мониторов: **Fail on missing or expiring SSL certificate**: завершать монитор неуспешно, если один или несколько запросов обнаруживают истёкший, отсутствующий или истекающий SSL-сертификат.

   Этот параметр работает только если **SSL expiration date verification** уже включена для HTTP-запроса в [настройках монитора](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic "Узнайте о настройке HTTP-мониторов."); он использует настройку на уровне запроса для проверки действительности SSL-сертификата в течение указанного количества дней.

   Если монитор содержит несколько запросов с разными настройками действительности SSL-сертификата, **Fail on missing or expiring SSL certificate** проверяет и учитывает каждую настройку.
6. Только для браузерных мониторов: **Take screenshot on successful execution**: делать скриншоты при успешном выполнении из публичных или частных расположений; этот параметр отключён по умолчанию. При включении этого параметра скриншоты при успехе делаются даже если монитор превышает пороговое значение производительности. При запуске выполнений из всех расположений скриншоты при успехе делаются из первого расположения. Скриншоты при успехе можно также делать для выполнений, запущенных через API.

   Обратите внимание: скриншоты автоматически делаются при неудаче.
7. Нажмите **Trigger now**: появляется диалоговое окно с перечнем расположений выполнения; в списке выполнений появляются новые записи для запущенных выполнений. **Execution stage** изначально имеет значение `Triggered`.

   Любые изменения скрипта (конфигурации) монитора немедленно становятся доступны для выполнений по требованию в публичных или частных расположениях.

   Обратите внимание: выполнения из разных расположений могут начинаться не одновременно — из публичных расположений они могут начинаться дольше, чем из частных. Дополнительную информацию об ограничении запросов ([throttling](#throttling)) см. ниже.

   ![Triggered execution](https://dt-cdn.net/images/on-demand-exec-triggered-2185-f3ccf8cf28.png)

   Triggered execution

### Ограничение запросов и лимиты

* Между последовательными выполнениями по требованию одного монитора из одного расположения одним пользователем обязателен промежуток в 60 секунд (как при запуске через веб-интерфейс, так и через API).

  Примеры ограничения запросов

  Все примеры предполагают выполнение из одного расположения.

  + Пользователь A запускает монитор через API в течение 60 секунд после запуска того же монитора через веб-интерфейс — запрос ограничен.
  + Пользователь B запускает монитор через API в течение 60 секунд после запуска того же монитора пользователем A через веб-интерфейс — ограничений нет.
  + Пользователь B запускает монитор через веб-интерфейс в течение 60 секунд после запуска того же монитора пользователем A через веб-интерфейс — ограничений нет.
  + Пользователь B запускает монитор через API в течение 60 секунд после запуска того же монитора пользователем A через API — ограничений нет.

* При запуске нескольких выполнений по требованию через [API](/managed/observe/digital-experience/synthetic-monitoring/general-information/on-demand-executions#api "Выполняйте синтетические мониторы по требованию из публичных или частных расположений.") лимит составляет 100 выполнений на пакет.
* Также существует лимит 5000 выполнений по требованию в минуту для окружения Dynatrace.
* Можно задавать до 64 пар ключ-значение [метаданных на пакет](/managed/observe/digital-experience/synthetic-monitoring/general-information/on-demand-executions#api "Выполняйте синтетические мониторы по требованию из публичных или частных расположений."), где каждый ключ и значение могут содержать до 1024 символов.
* При указании [повторных выполнений по требованию на расположение](/managed/observe/digital-experience/synthetic-monitoring/general-information/on-demand-executions#api "Выполняйте синтетические мониторы по требованию из публичных или частных расположений.") максимальное количество выполнений равно 10.

### Неудача при запуске

Выполнения по требованию могут не быть успешно запущены по различным причинам, например, если монитор отключён, расположение недоступно или действует ограничение запросов для монитора в данном расположении.

Существует ряд дополнительных причин, по которым выполнения по требованию могут не быть успешно запущены через [API](#api): удаление монитора, неправильное указание идентификаторов монитора или расположения, неправильное указание идентификаторов связанных сервисов или приложений, удаление расположения, проблемы с публичной инфраструктурой Synthetic или проблемы с окружением мониторинга Dynatrace.

Если выполнение не может быть успешно запущено через веб-интерфейс, причины отображаются в **Triggering status summary** после нажатия **Trigger now**. Значение **Execution stage** в списке выполнений — `Not triggered`. Сведения о выполнениях, которые не были запущены через [API](#api), возвращаются в параметрах ответа `triggeringProblemsCount` и `triggeringProblemsDetails` для POST-запроса.

![Failure to be triggered](https://dt-cdn.net/images/on-demand-exec-not-triggered-2190-97500e2d2a.png)

Failure to be triggered

Выполнения, которые не удалось запустить, отличаются от выполнений, которые были запущены, но не могут быть выполнены — подробности см. в разделе [API](#api) ниже.

### Список выполнений

В списке выполнений отображаются все выполнения по требованию (запущенные любым пользователем через веб-интерфейс или API) для данного монитора за последние шесть часов.

![On-demand execution list](https://dt-cdn.net/images/on-demand-exec-list-2179-8c66facb90.png)

On-demand execution list

* **Execution ID** — уникальный идентификатор, присвоенный каждому выполнению; при запуске выполнений из всех расположений у выполнения каждого расположения свой идентификатор.
* Столбец **Triggered** показывает время начала выполнения в часовом поясе вошедшего в систему пользователя.
* **Source** показывает, было ли выполнение запущено через веб-интерфейс (`UI`) или `API`.
* Столбец **User** показывает идентификатор пользователя Dynatrace, запустившего выполнение.
* Столбец **Location** показывает имя публичного или частного расположения, из которого выполнялся монитор.
* **Execution stage** определяет различные стадии выполнения по требованию. Начальное значение — `Triggered` (или `Not triggered`). После завершения выполнения значение меняется на `Executed`. Основные результаты, такие как длительность и HTTP-код статуса, доступны на этой стадии. Счётчик загрузки продолжает отображаться на стадии `Executed` до тех пор, пока не станут доступны подробные результаты и значение не изменится на `Data retrieved`. При запуске нескольких последовательных выполнений на расположение первое выполнение помечается как `Triggered`; остальные помечаются как `Waiting`.

* Столбец **Result** отображает, было ли выполнение `Success` (успешным) или `Failure` (неуспешным) с сопровождающей **Failure reason** (причиной неудачи).
* Можно **повторно запустить** ![Refresh](https://dt-cdn.net/images/refresh-turquoise-500-64f72fec6c.svg "Refresh") выполнение с точно определённой конфигурацией, например, с метаданными, заданными через API. Помимо метаданных, реплицируются следующие параметры выполнения: [режим обработки](#trigger-ui), неудача при нарушении порогового значения производительности, неудача из-за проблемы с SSL-сертификатом и создание скриншота при успехе.
* Выберите кнопку развертывания ![Expand row](https://dt-cdn.net/images/expand-row-icon-9c4950fc2e.svg "Expand row") в столбце **Details** для просмотра длительности выполнения, метаданных пакета, таких как пары ключ-значение (доступны только через [API](#api)), и ссылки на результаты.

### Результаты

Когда значение **Execution stage** в [списке выполнений](#list) изменяется на `Data retrieved`, можно перейти по ссылке **execution details** для просмотра подробных результатов выполненного браузерного или HTTP-монитора.

Для браузерных мониторов вы переходите на страницу **Multidimensional analysis** с выбранной точкой данных на точечном графике. Выполнения по требованию идентифицируются по форме в [точечном графике](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/multidimensional-analysis-for-browser-monitors#scatter-plot "Узнайте, как анализировать точки данных браузерных мониторов."), а также по аннотации в списке [точек данных](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/multidimensional-analysis-for-browser-monitors#data-points "Узнайте, как анализировать точки данных браузерных мониторов.").

![Scatter plot](https://dt-cdn.net/images/scatterplotondemandexecution-2164-393f41dda4.jpg)

Scatter plot

Сбор скриншотов для выполнений браузерных мониторов по требованию аналогичен [запланированным выполнениям](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-browser-monitors#screenshots "Анализируйте результаты браузерных мониторов и clickpath-ов на странице Synthetic details."). Однако можно включить сбор скриншотов при успехе [через веб-интерфейс](#trigger-ui) и POST-запросами [через API](#api).

Для HTTP-мониторов вы переходите на вкладку **On-demand execution** страницы **Analyze execution details**. Также можно получить доступ к этой вкладке, выбрав **Analyze execution details** на [странице деталей HTTP-монитора](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic "Узнайте о странице Synthetic details для HTTP-мониторов."). Выберите выполнение по требованию **Execution** из раскрывающегося списка для просмотра его деталей.

![On-demand executions tab](https://dt-cdn.net/images/analyze-last-execution-filters2-1600-0085726604.png)

On-demand executions tab

Вкладка **On-demand execution** перезаписывается при каждом выполнении по требованию. Если выполнения по требованию находятся в режимах **Standard** или **Disable problem detection**, детали также записываются на вкладки последнего неуспешного/успешного выполнения. Обратите внимание: в этих режимах, если монитор завершается неуспешно из-за нарушения порогового значения производительности, выполнение появляется на вкладках успешных и выполнений по требованию.

## API: Synthetic - On-demand monitor executions

Информацию об ограничениях выполнений по требованию через веб-интерфейс и API см. в разделе [ограничение запросов](#throttling) выше.

Дополнительную информацию см. в документации по [API выполнений синтетических мониторов v2](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-monitor-execution "Просматривайте результаты выполнений синтетических мониторов через Synthetic v2 API."). Можно [запускать пакетные выполнения](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-monitor-execution/post-batch-execution "Запускайте пакетные выполнения синтетических мониторов через Dynatrace API."), [перечислять выполнения по требованию](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-monitor-execution/get-all-executions "Перечисляйте все выполнения синтетических мониторов по требованию через Dynatrace API."), проверять [основные](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-monitor-execution/get-execution-basic "Просматривайте основную информацию о выполнении синтетического монитора по требованию через Dynatrace API.") и [подробные результаты](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-monitor-execution/get-execution-full "Просматривайте полную информацию о выполнении синтетического монитора по требованию через Dynatrace API.") одного выполнения, а также [результаты пакетного выполнения](/managed/dynatrace-api/environment-api/synthetic-v2/synthetic-monitor-execution/get-batch-summary "Просматривайте сводку пакетного выполнения синтетических мониторов через Dynatrace API.").

API-выполнения по требованию обеспечивают большую гибкость и масштабируемость, чем веб-интерфейс. API **Synthetic - On-demand monitor executions** предлагает:

* Возможность выполнять несколько мониторов по требованию (**пакетное выполнение**), указав идентификаторы мониторов и, при желании, идентификаторы расположений. Если расположения не указаны, монитор выполняется из всех назначенных расположений. Эта возможность доступна только через API.
* Возможность выполнять мониторы из **любого расположения**. В POST-запросе для запуска выполнений по требованию каждый указанный монитор выполняется из указанных расположений (или из всех назначенных расположений, если расположения не указаны).
* ActiveGate версии 1.259+: возможность указывать **количество выполнений для повторных выполнений на расположение**, что удобно для контроля качества и нагрузочного тестирования. Эта возможность доступна только через API. Параметр `executionCount` позволяет указывать количество выполнений на расположение для каждого монитора; максимальное значение — 10; если значение не указано, монитор выполняется по одному разу на расположение. Выполнения можно запускать последовательно (`repeatMode` равен `SEQUENTIAL`) или параллельно (`repeatMode` равен `PARALLEL`). Режим по умолчанию — последовательный.

  В последовательном режиме каждое выполнение автоматически ссылается на следующее (при наличии) через параметр `nextExecutionId`. При запуске нескольких последовательных выполнений на расположение первое выполнение находится на стадии `Triggered`, а остальные помечены как `Waiting`. Выполнения на каждом расположении независимы друг от друга.

  + Если последовательное выполнение превышает тайм-аут на расположении, последующие выполнения на этом расположении не запускаются.
  + Обратите внимание: идентификаторы выполнений в `nextExecutionId` являются последовательными для публичных расположений и генерируются случайно для частных расположений.
  + Изменения скрипта монитора или учётных данных после запуска пакета последовательных выполнений не влияют на оставшиеся выполнения. Таким образом, если другой пользователь изменяет последовательно выполняемый скрипт, изменения не отражаются в оставшихся выполнениях.
* ActiveGate версии 1.259+: возможность **частично переопределять скрипт монитора** и предоставлять пользовательские значения параметров специально для выполнения по требованию, что упрощает перенастройку и тестирование синтетических мониторов. Эта возможность доступна только через API. Параметр `customizedScript` позволяет перечислять `requests` (для HTTP-мониторов) или `events` (для браузерных мониторов) с настройками на уровне запроса или события. Для HTTP-мониторов параметры с пользовательскими значениями видны при [**Analyze execution details**](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic#analyze-last-execution "Узнайте о странице Synthetic details для HTTP-мониторов.").

  Следующие параметры можно настраивать для HTTP-мониторов

  + `url`
  + `requestBody`
  + `validation`
  + `preProcessingScript`
  + `postProcessingScript`
  + `requestTimeout`
  + `authentication`
  + `configuration`

    - `requestHeaders`
    - `acceptAnyCertificate`
    - `followRedirects`

  Следующие типы событий можно настраивать для браузерных мониторов

  + Navigate
  + Click
  + JavaScript
  + Keystroke
  + Tap
  + Select option
  + Cookie

  При желании можно указывать `sequenceId` для порядка запроса или события в скрипте монитора. Например, используйте `"sequenceId": "3"` для указания изменений третьего события в браузерном мониторе. Если `sequenceId` не указан, изменения применяются к первому запросу или событию, на которое ещё не ссылается идентификатор последовательности в API-запросе. См. примеры ниже.

  + Такие изменения скрипта предназначены только для выполнения по требованию и не сохраняются после него.
  + Нельзя изменять или переупорядочивать количество запросов или событий.
  + Нельзя изменять **Name** запросов или событий (`description` в режиме скрипта).
  + Нельзя изменять тип запроса (например, GET) или тип события (например, Navigate).
  + Нельзя добавлять скрипт предварительной или последующей обработки в HTTP-монитор, не имеющий такого скрипта.

  Пример: HTTP-монитор с пользовательским URL для одного запроса

  Поскольку идентификатор последовательности не указан, переопределение URL в этом примере применяется к первому HTTP-запросу.

  ```
  {



  "monitors": [



  {



  "monitorId": "HTTP_CHECK-C608F75BF82E5B22",



  "customizedScript": {



  "requests": [



  {



  "url": "https://www.yourdomain.com"



  }



  ]



  }



  }



  ]



  }
  ```

  Пример: HTTP-монитор с переопределениями для двух запросов

  Поскольку идентификатор последовательности не указан, переопределение URL и скрипт предварительного выполнения с методом `api.fail()` применяются к первому HTTP-запросу. Обратите внимание: скрипт предварительной или последующей обработки можно переопределять только для монитора, в котором такой скрипт уже определён. Для третьего запроса, для которого указан идентификатор последовательности, добавлено правило проверки.

  ```

  {



  "monitorId": "HTTP_CHECK-6349B98E1CD87352",



  "customizedScript": {



  "requests": [



  {



  "url": "https://www.somepage.org",



  "preProcessingScript": "if (response.getResponseBody().includes(\"error\")) {api.fail(\"HTTP failing monitor.\");}"



  },



  {



  "sequenceId": "3",



  "validation": {



  "rules": [



  {



  "value": "=201",



  "passIfFound": "true"



  }

  }



  }



  ]



  }

  Пример: Браузерный монитор с пользовательским URL для события Navigate

  Поскольку идентификатор последовательности не указан, переопределение URL в этом примере применяется к первому событию Navigate.

  ```
  {



  "takeScreenshotsOnSuccess": true,



  "monitors": [



  {



  "monitorId": "SYNTHETIC_TEST-114F1C18CF07CD1D",



  "customizedScript": {



  "events": [



  {



  "type": "navigate",



  "url": "www.yourdomain.com"

  ]

* Возможность выполнять группу мониторов по требованию, указав общие **теги и/или идентификаторы связанных сервисов или приложений**. Обратите внимание: все указанные условия должны совпадать для выполнения монитора по требованию. Эта возможность доступна только через API.

  Если в POST-запросе указаны три монитора по идентификатору и один тег, будут выполнены все три монитора плюс все мониторы, соответствующие тегу.
* Автоматическое присвоение **идентификатора пакета** всем выполнениям метода POST, как для одного монитора, так и для нескольких. Это позволяет также получать результаты по идентификатору пакета.
* Возможность определять **пользовательские пары ключ-значение**, например, для идентификации версий приложения, в `metadata` пакета. Эта возможность доступна только через API. Можно определять до 64 пар на пакет, где каждый ключ и значение могут содержать до 1024 символов. Метаданные ключ-значение доступны в результатах отдельных и пакетных выполнений, получаемых через API, и отображаются в веб-интерфейсе.
* Создание **скриншотов при успехе** для браузерных мониторов из публичных или частных расположений путём установки `"takeScreenshotsOnSuccess": true` в POST-запросе (значение по умолчанию — `false`). Скриншоты при успехе делаются из первого расположения, указанного для каждого идентификатора монитора. Если расположение не указано (например, при использовании тегов для определения списка мониторов), скриншоты делаются из любого из расположений, назначенных монитору.

  Обратите внимание: скриншоты автоматически делаются при неудаче. Однако если монитор завершается неуспешно из-за нарушения порогового значения производительности, он всё ещё считается доступным и скриншоты не делаются; можно включить `takeScreenshotsOnSuccess`.
* Возможность завершить выполнение HTTP-монитора по требованию неуспешно при отсутствующем, истёкшем или истекающем **SSL-сертификате** (параметр `failOnSslWarning`).
* Возможность **останавливать запуск всех выполнений в пакете** при возникновении проблемы с запуском любого выполнения (параметр `stopOnProblem`), например, если идентификатор монитора неверен/отсутствует или монитор удалён. Эта возможность доступна только через API.
* Получение **списка** выполнений по требованию (запущенных через веб-интерфейс или API всеми пользователями окружения) за предыдущие шесть часов. Список включает идентификаторы выполнений для каждого выполнения. Список можно фильтровать по:

  + Меткам времени выполнения и доставки данных
  + Идентификаторам пакетов
  + Идентификаторам мониторов
  + Идентификаторам расположений
  + Идентификаторам пользователей
* Получение как основных, так и более подробных **результатов** выполнения (для которого указывается идентификатор выполнения).

  Основные результаты лучше всего подходят для CI/CD-целей и включают количество выполненных запросов/событий, результат успеха/неудачи и некоторые ключевые метрики. Например, основные результаты для HTTP-мониторов сообщают общий размер запроса (для всех запросов вместе взятых), время до первого байта, время TLS-рукопожатия, время TCP-соединения, время DNS-поиска и итоговый HTTP-код статуса.

  Подробные результаты более подходят для устранения неполадок. Для HTTP-мониторов это полный набор результатов, отображаемых в веб-интерфейсе при выборе [**Analyze execution details**](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic#analyze-last-execution "Узнайте о странице Synthetic details для HTTP-мониторов.").
* **Пакетное получение результатов** при указании идентификатора пакета.

  Результаты пакета включают информацию о запущенных выполнениях и выполнениях, давших результат.

  Обратите внимание: `executedCount` — это количество выполнений, завершившихся успехом или неудачей. `failedToExecuteCount` — это количество запущенных выполнений, для которых результаты недоступны по техническим причинам (например, из-за проблемы с ActiveGate или Synthetic-движком, или результаты не удалось отправить в кластер Dynatrace). `executedCount` + `failedToExecuteCount` = `triggeredCount`.
* **Детальные разрешения** для запуска и получения данных о выполнениях по требованию.

  Для запуска выполнений (POST) необходимы права на создание мониторов (область токена `ExternalSyntheticIntegration`) или область токена `syntheticExecutions.write`, которая позволяет запускать выполнения, но не создавать новые мониторы.

  Для получения данных (GET) необходима любая из областей токена: `ExternalSyntheticIntegration`, `ReadSyntheticData` или `syntheticExecutions.read`.

### Лучшие практики

* При переопределении учётных данных в скрипте монитора, например, для заголовка Authorization, убедитесь, что у вас есть доступ к обновлённым учётным данным.
* При переопределении URL в событии Navigate браузерного монитора рекомендуется не включать создание скриншотов при успехе, так как эталонные скриншоты для события будут перезаписаны скриншотами с пользовательского URL.
