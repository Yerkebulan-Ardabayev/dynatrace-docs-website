# Настройка HTTP-мониторов

* How-to guide
* 16-min read
* Updated on Apr 01, 2026

Dynatrace позволяет легко настраивать HTTP-мониторы как при первоначальном создании, так и в любое время после него.

В процессе создания HTTP-монитора параметры настройки появляются после выбора **Create an HTTP monitor**. Это подмножество полного набора параметров, доступных в режиме редактирования (описан ниже) после развёртывания монитора. Если вы создаёте новый HTTP-монитор, см. раздел [Создание HTTP-монитора](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/create-an-http-monitor-classic "Узнайте, как создать HTTP-монитор для проверки производительности и доступности сайта.").

HTTP-мониторы можно запускать из глобальных [публичных](/managed/observe/digital-experience/synthetic-monitoring/general-information/public-synthetic-locations "Узнайте обо всех доступных публичных расположениях Synthetic Monitoring.") или [частных расположений Synthetic](/managed/observe/digital-experience/synthetic-monitoring/private-synthetic-locations/create-a-private-synthetic-location "Узнайте, как создать частное расположение для синтетического мониторинга."), а также из кластерных расположений в Dynatrace Managed. Подробности об использовании ActiveGate для Synthetic Monitoring см. в разделе [Создание частного расположения Synthetic](/managed/observe/digital-experience/synthetic-monitoring/private-synthetic-locations/create-a-private-synthetic-location "Узнайте, как создать частное расположение для синтетического мониторинга."). Подробности о поддерживаемых версиях Windows и Linux см. в разделе [Требования для частных расположений Synthetic](/managed/observe/digital-experience/synthetic-monitoring/private-synthetic-locations/system-and-hardware-requirements-for-private-synthetic "Поддерживаемые ОС, версии Chromium и требования к оборудованию для частных расположений.").

## Настройка существующего HTTP-монитора

Настройка существующего HTTP-монитора

1. Перейдите в **Synthetic Classic**.
2. Выберите нужный HTTP-монитор (откроется [страница деталей](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic "Узнайте о странице Synthetic details для HTTP-мониторов.")).
3. Нажмите **Edit** в быстрых ссылках в верхнем левом углу для перехода к настройкам монитора.

   Кроме того, в **Synthetic Classic** в режиме списка можно установить флажок рядом с нужным монитором и нажать **Edit** в нижнем левом углу.
4. Просматривайте вкладки **Monitor settings** слева для настройки параметров (подробные описания ниже; подмножество этих параметров доступно при первоначальном создании HTTP-монитора):

   * [General](#setup)
   * [HTTP requests](#http-requests)
   * [Frequency and locations](#frequency-locations)
   * [Outage handling](#outage-handling)
   * [Performance thresholds](#performance-thresholds)
5. При внесении изменений в нижнем левом углу появляются кнопки **Discard changes** и **Save changes**. Обязательно сохраняйте изменения по завершении редактирования монитора.

Кто может редактировать монитор с привязанными учётными данными?

* Если монитор связан с публичными учётными данными, любой член команды может включить/выключить, удалить или отредактировать монитор.

* Если HTTP-монитор связан с ограниченными учётными данными (только для владельца или общедоступными для нескольких пользователей), любой пользователь может вносить изменения в определённые поля, даже не имея доступа к использованным учётным данным. Можно редактировать имя монитора, расположения, валидацию, пороговые значения и, конечно, менять сертификат или пару UID/пароль. Можно редактировать и менять учётные данные в URL, значении заголовка или теле запроса. В режиме редактирования можно создать учётные данные прямо в настройках монитора. Потребуется заменить все учётные данные в мониторе на те, к которым у вас есть доступ. Замена чужих учётных данных своими необратима.

  Элементы управления, которые нельзя редактировать, такие как URL запроса, HTTP-метод, скрипт предвыполнения, скрипт пост-выполнения, HTTP-заголовки, тело запроса, параметр следования редиректам, а также добавление/удаление HTTP-запросов, будут неактивны или покажут сообщение об ошибке при попытке сохранить изменения, как в режиме скрипта, так и в режиме UI.

* Вы можете включить/выключить или удалить синтетический монитор, защищённый учётными данными другого пользователя с правами только для владельца.

Подробности о правах доступа к учётным данным см. в разделе [Credential vault для синтетических мониторов](/managed/manage/credential-vault "Храните учётные данные и управляйте ими в хранилище credential vault.").

## General

На вкладках **General** можно:

* Просматривать и изменять **Имя монитора** (до 500 символов). Имя должно в общих чертах описывать все запросы данного HTTP-монитора.
* Настраивать **Cookies** для хранения информации о состоянии или для указания серверу не отправлять определённые виды информации. Cookie добавляются в заголовки, а значения задаются перед выполнением первого HTTP-запроса. Этот параметр доступен только в режиме редактирования.

  В режиме редактирования включите **Set cookies**, затем укажите **Имя** и **Значение** cookie. Каждый cookie должен быть уникальным в пределах списка.

  В значении cookie недопустимы следующие символы: `;,\"`. Укажите **Домен** cookie и, при необходимости, **Путь** к нему. Нажмите **Save**.

  Выберите **Add cookie** для определения дополнительных cookie.

  Значения cookie сохраняются на клиенте до получения от сервера заголовка `Set-Cookie`, изменения значения cookie в [скрипте предвыполнения](#pre-execution) или до завершения выполнения монитора.

### Assigned applications

Этот параметр доступен только в режиме редактирования.

Если данный синтетический монитор связан с одним из ваших отслеживаемых приложений, вы можете назначить монитор приложению для отслеживания его доступности и производительности. Обнаруженные проблемы будут автоматически связаны с вашим приложением. Если монитор недоступен, связанное приложение также считается недоступным.

Нажмите **Assign to application** и выберите приложение из списка. Монитор можно назначить нескольким приложениям, а одному приложению могут быть назначены несколько мониторов.

HTTP-монитор можно назначить веб-, мобильному или пользовательскому приложению.

Обратите внимание: нельзя заблокировать трафик Synthetic Monitoring для RUM-приложений путём [исключения ботов, пауков или IP-адресов расположений Synthetic](/managed/observe/digital-experience/web-applications/additional-configuration/exclude-browsers-robots-and-spiders-from-monitoring "Отключите Real User Monitoring для определённых IP-адресов, браузеров, ботов и пауков.").

## HTTP requests

Вы можете редактировать и добавлять HTTP-запросы, отправляемые монитором на ваш сервер.

В **Visual mode**:

* Нажмите **Add HTTP request** для добавления ещё одного HTTP-запроса к HTTP-монитору.
* Выберите существующий HTTP-запрос для отображения и редактирования его параметров.
* При необходимости можно удалить запросы из монитора, нажав **x** в столбце **Delete** для соответствующего запроса.
* Используйте стрелки **Move up/down** ![Move up](https://dt-cdn.net/images/sorter-move-up-6275b6459e.svg "Move up") ![Move down](https://dt-cdn.net/images/sorter-move-down-710c5d6229.svg "Move down") для изменения порядка запросов.

Режимы не ограничены: вы можете переключаться между интерфейсным режимом и режимом скрипта, нажимая **Visual mode** или **Script mode**. Подробности редактирования HTTP-монитора в формате JSON см. в разделе [Script mode для настройки HTTP-мониторов](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/script-mode-for-http-monitor-configuration-classic "Создавайте и редактируйте HTTP-мониторы в формате JSON.").

### Базовые параметры запроса

Эти параметры доступны для [запросов авторизации OAuth2](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-authentication#oauth2 "Узнайте, как настроить методы аутентификации для мониторинга веб-приложений и API-эндпоинтов в Synthetic Monitoring."), а также для всех [HTTP-методов](#http-methods) в HTTP-запросах:

* **URL HTTP-запроса**

  Для повышения безопасности синтетических мониторов Dynatrace блокирует отправку мониторами запросов на локальный хост (например, `localhost` или `127.0.0.1`).

  Вы можете добавить [токен-учётные данные](/managed/manage/credential-vault#token "Храните учётные данные и управляйте ими в хранилище credential vault.") в **URL HTTP-запроса**: начните вводить `{cr` для просмотра списка подсказок учётных данных. В этом списке отображаются только доступные вам учётные данные: публичные или созданные вами с доступом только для владельца.

  + В полях, где можно вручную вставить ссылку на учётные данные (например, в URL запроса), скопировав идентификатор из хранилища, используйте формат `{<credential ID>|token}`, `{<credential ID>|username}` или `{<credential ID>|password}` в зависимости от типа вставляемых учётных данных. Копировать и вставлять можно только учётные данные, к которым у вас есть доступ.
  + Кто может редактировать монитор с привязанными учётными данными?

    - Если монитор связан с публичными учётными данными, любой член команды может включить/выключить, удалить или отредактировать монитор.

    - Если HTTP-монитор связан с ограниченными учётными данными (только для владельца или общедоступными для нескольких пользователей), любой пользователь может вносить изменения в определённые поля, даже не имея доступа к использованным учётным данным. Можно редактировать имя монитора, расположения, валидацию, пороговые значения и менять сертификат или пару UID/пароль. Потребуется заменить все учётные данные в мониторе на те, к которым у вас есть доступ. Замена чужих учётных данных своими необратима.

      Элементы управления, которые нельзя редактировать, такие как URL запроса, HTTP-метод, скрипт предвыполнения, скрипт пост-выполнения, HTTP-заголовки, тело запроса, параметр следования редиректам, а также добавление/удаление HTTP-запросов, будут неактивны или покажут сообщение об ошибке при попытке сохранить изменения.

    - Вы можете включить/выключить или удалить синтетический монитор, защищённый учётными данными другого пользователя с правами только для владельца.

    Подробности о правах доступа к учётным данным см. в разделе [Credential vault для синтетических мониторов](/managed/manage/credential-vault "Храните учётные данные и управляйте ими в хранилище credential vault.").
* **Add authorization data to**

  Это поле доступно только для типа запроса **OAuth2 authorization request**: подробности см. в разделе [Поддерживаемые методы аутентификации в Synthetic Monitoring](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-authentication#oauth2 "Узнайте, как настроить методы аутентификации для мониторинга веб-приложений и API-эндпоинтов в Synthetic Monitoring.").

* **HTTP-метод**

  Для каждого создаваемого или редактируемого запроса начните с выбора **HTTP-метода**, поскольку от него зависят доступные параметры запроса. Поддерживаемые HTTP-методы:

  + GET
  + POST
  + PUT
  + DELETE
  + HEAD
  + PATCH
  + OPTIONS
* Опциональный [**User agent**](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/User-Agent)

  User agent по умолчанию имеет формат `DynatraceSynthetic/{version}`, где `{version}` — текущая версия движка Synthetic, выполняющего монитор. Даже при определении пользовательского user agent Dynatrace автоматически добавляет к нему `DynatraceSynthetic/{version}`, чтобы трафик синтетического мониторинга всегда можно было идентифицировать. **Не используйте** `DynatraceSynthetic/` в своём user agent: это зарезервировано для Dynatrace.
* **Response status code verification**

  Выберите условие **Fail**/**Pass** для монитора на основе возвращённого HTTP-кода статуса запроса.

  Можно указать точный код статуса, знак неравенства (`<`, `<=`, `>`, `>=`, `!=`), диапазон или маску класса статуса (только `xx`). Для диапазона используйте знак минус (`-`) без пробелов; для оператора «не равно» используйте `!=`. (Обратите внимание: `=>` и `=<` не поддерживаются.) Для разделения нескольких значений используйте запятые, например `3xx, 404, 406-410, >=500`. По умолчанию при включении монитор завершается с ошибкой, если возвращённый код статуса `>=400`.

  Валидация HTTP-монитора

  При валидации HTTP-монитора в разделе `validation.rules` в [режиме скрипта](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/script-mode-for-http-monitor-configuration-classic "Создавайте и редактируйте HTTP-мониторы в формате JSON.") оцениваются следующие типы правил:

  + Валидация кода статуса ответа
  + Текстовая валидация: первые 50 КБ тела ответа проверяются на наличие строки текста или регулярного выражения
  + Валидация срока действия SSL-сертификата (ActiveGate версии 1.235+)

  В мониторе с несколькими валидациями все правила оцениваются. Однако возвращается единый статус выполнения по приоритету: валидация HTTP-кода статуса имеет наивысший приоритет, затем — текстовая и регулярная, и в последнюю очередь — валидация срока действия SSL-сертификата.

### Дополнительные параметры

При необходимости настройте **Additional options**.

#### Включение скрипта предвыполнения

[Скрипты предвыполнения и пост-выполнения](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic "Узнайте, как применять скрипты пред- и пост-выполнения к запросам.") позволяют добавлять пользовательскую логику между запросами HTTP-монитора: например, анализировать ответ, изменять URL запроса или пропускать запросы при определённых условиях.

При включении этой опции появляется поле для ввода скрипта, выполняемого непосредственно перед запуском запроса. Скрипты предвыполнения основаны на пользовательском JavaScript-коде и могут использоваться для формирования HTTP-запроса или добавления логики, которую нельзя или нежелательно реализовывать в UI.

Подробности и справочник по методам см. в разделе [Скрипты предвыполнения и пост-выполнения для HTTP-мониторов](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#methods "Узнайте, как применять скрипты пред- и пост-выполнения к запросам."). Обратите внимание: некоторые методы доступны только в [скриптах предвыполнения](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#pre-execution "Узнайте, как применять скрипты пред- и пост-выполнения к запросам.") или [скриптах пост-выполнения](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#post-execution "Узнайте, как применять скрипты пред- и пост-выполнения к запросам.").

#### Включение скрипта пост-выполнения

При включении этой опции появляется поле для ввода скрипта, выполняемого после завершения запроса. Скрипты пост-выполнения основаны на пользовательском JavaScript-коде и могут использоваться для обработки ответов на запрос.

[Метод `api.fail()`](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#end "Узнайте, как применять скрипты пред- и пост-выполнения к запросам.") можно использовать для задания пользовательского **Сообщения об ошибке**, которое отображается в карточке Events на [странице деталей HTTP-монитора](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic "Узнайте о странице Synthetic details для HTTP-мониторов.").

![Custom failure message](https://dt-cdn.net/images/httpdetailseventsfailuremsg-1102-2317af2250.png)

Custom failure message

Подробности и справочник по методам см. в разделе [Скрипты предвыполнения и пост-выполнения для HTTP-мониторов](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic "Узнайте, как применять скрипты пред- и пост-выполнения к запросам."). Обратите внимание: некоторые методы доступны только в [скриптах предвыполнения](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#pre-execution "Узнайте, как применять скрипты пред- и пост-выполнения к запросам.") или [скриптах пост-выполнения](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#post-execution "Узнайте, как применять скрипты пред- и пост-выполнения к запросам.").

#### Настройка аутентификации/авторизации

Обратите внимание: в типе запроса **OAuth2 authorization request** этот параметр называется **Set token request authentication**.

Включите эту опцию и укажите пару имя пользователя/пароль для автоматизации входа в защищённые паролем сайты через аутентификацию **Basic**, **NTLM** или **Kerberos**. Dynatrace автоматически генерирует необходимый заголовок `Authorization` с указанными данными. (Подробности см. в разделе [Поддерживаемые методы аутентификации в Synthetic Monitoring](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-authentication "Узнайте, как настроить методы аутентификации для мониторинга веб-приложений и API-эндпоинтов в Synthetic Monitoring.").)

Аутентификация Kerberos не поддерживается для мониторов, назначенных на частные расположения Synthetic, где ActiveGate, обрабатывающий выполнения мониторов, использует прокси для подключения к тестируемым ресурсам.

Dynatrace хранит все учётные данные Synthetic Monitoring в хранилище credential vault и управляет ими. Учётные данные имеют контроль доступа и могут быть назначены как только для владельца, так и публичными.

Можно выбрать существующие учётные данные (**Select credentials**). В этом списке отображаются только доступные вам учётные данные: публичные или созданные вами с доступом только для владельца.

Можно **Создать новые учётные данные** (**Create new credentials**), введя **Username** и **Password**. Укажите **Credential name** и нажмите **Save to vault**.

Формат `<domain>\<username>` не поддерживается в качестве имени пользователя в HTTP-мониторах: укажите только `<username>`.

![Create a credential in an HTTP monitor](https://dt-cdn.net/images/httpmonitorcreatecredentialscv1-533-2baa3b4622.png)

Обратите внимание: создать учётные данные таким способом можно в новом или существующем HTTP-мониторе, даже не имея прав доступа к управлению учётными данными в [хранилище credential vault](/managed/manage/credential-vault "Храните учётные данные и управляйте ими в хранилище credential vault.").

#### Добавление дополнительных HTTP-заголовков

Монитор создаётся с минимальным набором заголовков, необходимых протоколу. Включите эту опцию для создания пользовательских/дополнительных заголовков:

2. Введите **имя** заголовка (**Header**) и **значение** (**Value**).
3. При необходимости нажмите **Add another header**.

HTTP-заголовки можно использовать для реализации bearer-аутентификации или аутентификации по токену: подробности см. в разделе [Поддерживаемые методы аутентификации в Synthetic Monitoring](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-authentication#bearer-token "Узнайте, как настроить методы аутентификации для мониторинга веб-приложений и API-эндпоинтов в Synthetic Monitoring.").

Вы можете добавить токен-учётные данные в **Значение** заголовка: начните вводить `{cr` для просмотра списка подсказок учётных данных. В этом списке отображаются только доступные вам учётные данные.

* В полях, где можно вручную вставить ссылку на учётные данные (например, в значении заголовка), скопировав идентификатор из хранилища, используйте формат `{<credential ID>|token}`, `{<credential ID>|username}` или `{<credential ID>|password}` в зависимости от типа вставляемых учётных данных. Копировать и вставлять можно только учётные данные, к которым у вас есть доступ.
* Заголовки HTTP по умолчанию

  Это заголовки по умолчанию, которые Dynatrace создаёт для каждого запроса.

  | Имя | Значение |
  | --- | --- |
  | `Accept` | `*/*` |
  | `Connection` | `close` |
  | `Host` | `<hostname>` |
  | `User-Agent` | `DynatraceSynthetic/{version}`[1](#fn-1-1-def) |

  1

  `{version}` — текущая версия движка Synthetic, выполняющего монитор. Даже при [определении пользовательского user agent](#basic-settings) Dynatrace всегда автоматически добавляет `DynatraceSynthetic/{version}` к user agent, чтобы трафик синтетического мониторинга можно было идентифицировать.

  Dynatrace также создаёт заголовки `x-Dynatrace-Tenant` и `x-Dynatrace-Test` для внутренних административных целей.

#### Тело запроса

Вы можете отправлять полезную нагрузку с запросами **POST**, **PUT**, **DELETE** и **PATCH**. В тело запроса можно добавить токен-учётные данные: начните вводить `{cr` для просмотра списка подсказок учётных данных. В этом списке отображаются только доступные вам учётные данные. Если это поле содержит [токен только для владельца](/managed/manage/credential-vault#owner-shared-public "Храните учётные данные и управляйте ими в хранилище credential vault."), другие пользователи не смогут его редактировать.

Укажите формат и содержимое тела запроса:

* **x-www-form-urlencoded**: нажмите **Add property** и введите одну или несколько пар ключ-значение свойств.
* **raw**: введите тело запроса в поле редактора.

В полях, где можно вручную вставить ссылку на учётные данные (например, в **Значении** пары ключ-значение в теле запроса), скопировав идентификатор из хранилища, используйте формат `{<credential ID>|token}`, `{<credential ID>|username}` или `{<credential ID>|password}` в зависимости от типа вставляемых учётных данных. Копировать и вставлять можно только учётные данные, к которым у вас есть доступ.

#### Установка правила валидации ответа

Этот параметр недоступен для типа запроса **OAuth2 authorization request**.

Валидация ответа позволяет задать условия прохождения/провала монитора на основе ожидаемого содержимого в первых 50 КБ тела ответа.

* **Pass if** или **Fail if** определяет базовое условие теста.
* **Text contains** задаёт текст для поиска в ответе. Строка чувствительна к регистру. Если строка содержит не-ASCII символы, экранируйте их в [режиме скрипта](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/script-mode-for-http-monitor-configuration-classic "Создавайте и редактируйте HTTP-мониторы в формате JSON.").
* **Interpret content match as regular expression** обрабатывает указанный текст как [регулярное выражение](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Узнайте об использовании регулярных выражений в Dynatrace.").

Несколько правил текстовой валидации можно настроить в [режиме скрипта](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/script-mode-for-http-monitor-configuration-classic "Создавайте и редактируйте HTTP-мониторы в формате JSON.").

Валидация HTTP-монитора

При валидации HTTP-монитора в разделе `validation.rules` в [режиме скрипта](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/script-mode-for-http-monitor-configuration-classic "Создавайте и редактируйте HTTP-мониторы в формате JSON.") оцениваются следующие типы правил:

* Валидация кода статуса ответа
* Текстовая валидация: первые 50 КБ тела ответа проверяются на наличие строки текста или регулярного выражения
* Валидация срока действия SSL-сертификата (ActiveGate версии 1.235+)

В мониторе с несколькими валидациями все правила оцениваются. Однако возвращается единый статус выполнения по приоритету: валидация HTTP-кода статуса имеет наивысший приоритет, затем — текстовая и регулярная, и в последнюю очередь — валидация срока действия SSL-сертификата.

#### Следование редиректам

По умолчанию HTTP-монитор следует за не более чем 10 редиректами от исходного запроса до достижения конечного адреса. Отключите этот параметр, чтобы мониторить только первый ответ цепочки редиректов, например, для проверки кодов статуса ответов редиректов.

#### Принять любой SSL-сертификат

По умолчанию HTTP-мониторы завершаются с ошибкой при недействительных SSL-сертификатах. Включите параметр **Accept any SSL certificate**, чтобы монитор принимал любые SSL-сертификаты независимо от их действительности.

Причины недействительности SSL-сертификата

В зависимости от настройки HTTP-монитор может завершиться успешно или с ошибкой, если SSL-сертификат недействителен по одной из следующих причин:

* Самоподписанный сертификат
* Истёкший сертификат
* Неактивный сертификат
* Срок действия сертификата превышает 398 дней
* Отсутствует имя хоста
* Недействительная/неполная цепочка сертификатов
* Недоверенный центр сертификации
* Небезопасный алгоритм подписи

См. также раздел [Проверка даты истечения SSL-сертификата](#certificate-expiry) ниже.

#### Проверка даты истечения SSL-сертификата

Включите этот параметр для генерации проблемы **SSL certificate expiration problem**, если SSL-сертификат отсутствует, истёк или истечёт в течение заданного количества дней (не более 100). Обратите внимание: эта проблема генерируется без завершения монитора с ошибкой и без влияния на доступность монитора.

Проблемы истечения SSL-сертификата имеют уровень серьёзности **Custom** и отображают количество дней до истечения. Это значение обновляется при каждом выполнении затронутого монитора.

![SSL certificate expiration problem](https://dt-cdn.net/images/sslcertexpirationproblem-1009-680712510b.png)

SSL certificate expiration problem

Для частных расположений этот параметр требует ActiveGate версии 1.235+.

Для частных расположений с ActiveGate ниже версии 1.235

При включении **проверки даты истечения SSL-сертификата** в частных расположениях с ActiveGate ниже версии 1.235:

* HTTP-монитор завершится с ошибкой, генерируя проблему сбоя, если до истечения SSL-сертификата осталось меньше дней, чем указано в настройках.
* Следующие статусы истечения SSL-сертификата отображаются в деталях проблемы сбоя и в карточках **Events** и **Failed requests** в [деталях HTTP-монитора](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic "Узнайте о странице Synthetic details для HTTP-мониторов."):

  + `SSL certificate expired` — если сертификат истёк
  + `SSL certificate expires within N days`, где `N` = `10`, `30`, `50` или `100`

  Например, если указано, что до истечения сертификата должно оставаться более 20 дней, а сертификат действителен ещё 15 дней, HTTP-запрос завершится с ошибкой и статусом `SSL certificate expires within 30 days`.

Следующие примеры показывают, как применять параметры **SSL certificate expiration date verification** и [**Accept any SSL certificate**](#ssl-accept).

* Чтобы настроить монитор, принимающий самоподписанные сертификаты с оповещением при истечении:

  + Включите **Accept any SSL certificate**.
  + Включите **SSL certificate expiration date verification** и задайте пороговое значение для генерации проблемы.
* Если не принимать любые SSL-сертификаты и SSL-сертификат истёк, HTTP-монитор завершится с ошибкой и создаст проблему сбоя с сообщением `SSL certificate expired` в деталях проблемы и в карточках **Events** и **Failed requests** в [деталях HTTP-монитора](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic "Узнайте о странице Synthetic details для HTTP-мониторов.").
* Если включены оба параметра и SSL-сертификат истёк, HTTP-монитор создаст проблему истечения SSL-сертификата. В карточке **Events** отобразится пользовательское оповещение об истечении SSL-сертификата.
* Если включены оба параметра и задано пороговое значение `n` дней до истечения SSL-сертификата, и сертификат истекает в течение `n` дней, HTTP-монитор создаст проблему истечения SSL-сертификата. В карточке **Events** отобразится пользовательское оповещение об истечении SSL-сертификата.

Валидация HTTP-монитора

При валидации HTTP-монитора в разделе `validation.rules` в [режиме скрипта](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/script-mode-for-http-monitor-configuration-classic "Создавайте и редактируйте HTTP-мониторы в формате JSON.") оцениваются следующие типы правил:

* Валидация кода статуса ответа
* Текстовая валидация: первые 50 КБ тела ответа проверяются на наличие строки текста или регулярного выражения
* Валидация срока действия SSL-сертификата (ActiveGate версии 1.235+)

В мониторе с несколькими валидациями все правила оцениваются. Однако возвращается единый статус выполнения по приоритету: валидация HTTP-кода статуса имеет наивысший приоритет, затем — текстовая и регулярная, и в последнюю очередь — валидация срока действия SSL-сертификата.

#### Не хранить и не отображать тела запросов и ответов, значения заголовков и данные однорангового сертификата в деталях выполнения

Включите этот параметр для скрытия потенциально конфиденциальной информации (например, в теле ответа) в [деталях выполнения (**Analyze last execution**)](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-details-for-http-monitors-classic#analyze-last-execution "Узнайте о странице Synthetic details для HTTP-мониторов.").

Это необходимо сделать для каждого запроса, для которого вы хотите ограничить отображение. Тела запросов и ответов, значения заголовков запросов и ответов, а также данные однорангового сертификата будут заменены текстом-заглушкой.

Этот параметр автоматически включается для запросов OAuth 2.0 и [мониторов синхронизации с внешним хранилищем](/managed/observe/digital-experience/synthetic-monitoring/general-information/external-vault-integration "Синхронизируйте учётные данные Synthetic Monitoring с внешними хранилищами."). Пользователи с доступом к любым учётным данным, содержащимся в мониторе, могут его отключить.

#### Настройка тайм-аута запроса

Включите этот параметр для изменения тайм-аута запроса. Обратите внимание: тайм-аут запроса по умолчанию составляет 10 секунд, даже если этот параметр не включён. При превышении заданного/дефолтного тайм-аута монитор завершится с ошибкой. Обратите внимание: время выполнения монитора — это сумма времён выполнения отдельных запросов, а тайм-аут монитора составляет 60 секунд.

## Frequency and locations

Расписание мониторинга определяется двумя факторами:

* **Frequency**: выберите частоту выполнения HTTP-монитора из каждого расположения: каждые `1`, `2`, `5`, `10`, `15`, `30` или `60` минут. Также можно настроить запуск монитора [**по запросу**](/managed/observe/digital-experience/synthetic-monitoring/general-information/on-demand-executions "Выполняйте синтетические мониторы по запросу из публичных или частных расположений.") (**On demand only**).
* **Location** определяет расположения, из которых выполняется монитор.

  HTTP-мониторы можно запускать из глобальных [публичных](/managed/observe/digital-experience/synthetic-monitoring/general-information/public-synthetic-locations "Узнайте обо всех доступных публичных расположениях Synthetic Monitoring.") или [частных расположений Synthetic](/managed/observe/digital-experience/synthetic-monitoring/private-synthetic-locations/create-a-private-synthetic-location "Узнайте, как создать частное расположение для синтетического мониторинга."), а также из кластерных расположений в Dynatrace Managed.

Вместе эти факторы определяют количество выполнений HTTP-монитора в час. Например, запуск монитора из 3 расположений каждые 15 минут даёт 12 выполнений в час (4 раза в час из каждого из 3 расположений). Выполнения равномерно распределяются в рамках выбранного интервала. То есть для монитора, запускаемого из 3 расположений каждые 15 минут, выполнения запускаются с интервалом 5 минут.

Обратите внимание: все публичные расположения Synthetic используют Coordinated Universal Time (UTC). Если скрипт монитора требует местного времени или часового пояса, можно использовать [метод `api.getContext()`](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#retrieve-data "Узнайте, как применять скрипты пред- и пост-выполнения к запросам.") и системные часы для реализации условной логики.

## Outage handling

Параметры обработки сбоев определяют действия при неудачных выполнениях монитора (перебои доступности). Поведение по умолчанию можно задать на уровне среды для всех браузерных мониторов или всех HTTP-мониторов: перейдите в **Settings** > **Web and mobile monitoring** и выберите соответствующую вкладку **Outage handling**. Можно использовать значения по умолчанию Dynatrace (**Use defaults**) для задания обработки сбоев на уровне среды. При включении эти значения применяются ко всем браузерным или HTTP-мониторам, у которых нет переопределённых настроек обработки сбоев на уровне монитора.

На уровне монитора этот параметр доступен только в режиме редактирования. Параметры обработки сбоев на уровне среды и на уровне монитора одинаковы. Вы можете переопределить значения по умолчанию, установленные для среды, на уровне монитора. Также можно восстановить значения по умолчанию уровня среды (**Remove override**).

Генерацию проблем при глобальных и локальных сбоях можно отключить, если вы тестируете нестабильный сайт или у вас запланированное техническое обслуживание, о котором не нужно получать оповещения.

* **Generate a problem and send an alert when this monitor is unavailable at all configured locations**

  По умолчанию этот параметр включён для вновь создаваемых мониторов. Он оповещает о глобальных перебоях доступности, то есть когда все расположения одновременно сталкиваются со сбоем.

  По умолчанию проблема глобального сбоя генерируется при однократном отказе всех расположений. Однако можно указать количество последовательных отказов (от 1 до 5) для генерации проблемы глобального сбоя, то есть сколько раз подряд должны отказать все расположения.
* **Generate a problem and send an alert when this monitor is unavailable for one or more consecutive runs at any location**

  Позволяет создавать проблему при последовательных отказах в одном или нескольких расположениях. На уровне среды можно выбрать количество отказов. На уровне монитора также можно определить количество расположений, которые должны отказать для генерации проблемы локального сбоя.

  В приведённом ниже примере монитор настроен на `4` расположения, и проблема будет сгенерирована, если `3` из этих `4` расположений не смогут получить доступ к сайту в течение `2` или более последовательных выполнений.

![Outage handling](https://dt-cdn.net/images/outagehttp-914-2e08ea40f8.png)

Outage handling

Проблема сбоя считается решённой, когда количество последовательных успешных выполнений равно настроенному количеству отказов для генерации проблемы. Успешные выполнения должны произойти на количестве расположений = общее количество расположений — количество расположений, необходимых для генерации проблемы + 1.

Обратите внимание: при решении проблемы глобального сбоя в одном или нескольких расположениях могут по-прежнему наблюдаться неудачи монитора. Настройте правила локальных сбоев для получения оповещений о них.

Подробности см. в разделе [Расчёты Synthetic](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-calculations "Узнайте о расчётах метрик Synthetic Monitoring."):

* Разница между [решением сбоя и тайм-аутами](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-calculations#availability-problems "Узнайте о расчётах метрик Synthetic Monitoring.").
* [Исключение выполнений синтетического монитора в окнах обслуживания из расчётов доступности](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-calculations#m-windows-availability "Узнайте о расчётах метрик Synthetic Monitoring.").

## Performance thresholds

Пороговые значения производительности позволяют проактивно реагировать на задержки сайта.

1. Включите **Performance thresholds enabled**.
2. Нажмите **Add new performance threshold**.
3. Выберите один из запросов в вашем мониторе или **Sum of all requests**.
4. Задайте **Threshold value** (пороговое время в секундах). Для помощи в установке порога можно просмотреть среднюю производительность за последние 24 часа до текущего момента. Затем нажмите **Add threshold**.

   Можно задать несколько пороговых значений производительности: для монитора в целом и для отдельных запросов. Пороговые значения производительности определяются как **Время отклика** монитора или отдельных запросов.

Dynatrace генерирует проблему производительности, если монитор в заданном расположении нарушает **любое** из определённых пороговых значений в 3 из 5 последних выполнений, при отсутствии открытого окна обслуживания для монитора. То есть нарушения должны происходить в одном и том же расположении. В проблему могут быть включены несколько расположений с подобными нарушениями.

Проблема закрывается, если пороговые значения производительности не нарушались в 5 последних выполнениях в каждом из ранее затронутых расположений.

Подробности см. в разделе [Расчёты Synthetic](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-calculations#performance-problems "Узнайте о расчётах метрик Synthetic Monitoring.").
