# HTTP monitors reporting results

# HTTP monitors reporting results

* Explanation
* 13-min read
* Updated on Nov 27, 2023

Перейдите в **Synthetic Classic** и выберите HTTP-монитор, чтобы открыть **страницу Synthetic details**, которая предоставляет обзор результатов выполнения монитора, визуализаций результатов и свойств монитора. Страница Synthetic details, основанная на AI-движке Davis, отображает сводную информацию и графики с готовыми ссылками и фильтрами для перехода к деталям проблемы и редактированию настроек монитора.

![HTTP monitor details](https://dt-cdn.net/images/httpmonitordetails1-1794-d60c8a1236.png)

HTTP monitor details

## Метрические визуализации

Верхняя панель показывает общую инфографику [доступности](#availability) и [производительности](#performance) монитора за выбранный период.

Используйте панель фильтров в верхней части страницы для фильтрации всех HTTP-данных по одному или нескольким расположениям.

Выберите быстрые ссылки в левом верхнем углу для перехода непосредственно к различным карточкам страницы деталей или для доступа к настройкам HTTP-монитора (**Edit**, **Disable**, **Delete**). [**Analyze execution details**](#analyze-last-execution) отображает результаты последнего успешного и неуспешного выполнений для каждого расположения в формате JSON.

Теги, применённые к HTTP-монитору, отображаются под именем монитора. Нажмите **Add tag** для добавления дополнительных тегов. Обратите внимание: теги можно добавлять и удалять только со страницы деталей.

Фиолетовые полосы над временными линиями доступности или производительности обозначают окна обслуживания.

![HTTP details with maintenance window](https://dt-cdn.net/images/httpdetailsmw1-1889-5247b78e7d.png)

HTTP details with maintenance window

* Отображение проблем и получение оповещений в течение окон обслуживания зависит от того, как вы [настроили окна обслуживания](/managed/analyze-explore-automate/notifications-and-alerting/maintenance-windows/define-maintenance-window "Создавайте окна обслуживания и определяйте их область.").
* [Окна обслуживания можно исключать из расчётов доступности](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-calculations#m-windows-availability "Поймите расчёты метрик Synthetic Monitoring.") с помощью глобальной настройки.

## Доступность

Инфографика доступности в верхней части страницы отображает доступность монитора за выбранный период с деталями о простое и затронутых расположениях (если есть). Инфографика длительности сбоя (простоя) и затронутых расположений отображается, если монитор недоступен в одном или нескольких расположениях. Количество затронутых расположений также отображается, если по каким-либо причинам данные от каких-либо расположений отсутствуют.

Обратите внимание: монитор может быть недоступен в одном или всех расположениях даже без настроенных [пороговых значений сбоев (глобальных или локальных)](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic#outage-handling "Узнайте о настройке HTTP-мониторов."). Продолжительность сбоя — это сумма всех простоев за выбранный период без учёта перекрывающихся простоев. Подробнее о расчёте доступности и простоя см. в разделе [Расчёты Synthetic](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-calculations "Поймите расчёты метрик Synthetic Monitoring.").

Карточка **Availability** показывает общую доступность по всем расположениям мониторинга с аннотациями глобальных/локальных сбоев и глобальных/локальных отсутствующих данных (например, при отключённом мониторе).

Наведите курсор на график **All locations** для просмотра информации о количестве расположений со сбоями или отсутствующими данными в любой момент времени. Разверните **All locations** для просмотра графиков доступности каждого расположения. Выберите блок простоя (красный) или доступности (фиолетовый) на любой временной линии доступности и нажмите **Analyze** для фильтрации всех HTTP-данных по длительности этого блока.

Нажмите **Pin to dashboard**, чтобы закрепить плитку **HTTP monitor** на классическом дашборде.

## Производительность

Инфографика производительности в верхней части страницы деталей отображает среднюю производительность HTTP-монитора, то есть время отклика для суммы всех запросов по всем расположениям за выбранный период. Дополнительные метрики фиксируются для каждого запроса и отображаются в [карточке запросов](#http-requests).

Карточка **Performance** показывает линии трендов минимального и максимального времени отклика для суммы всех запросов, где заштрихованная область представляет разницу между двумя значениями в любой момент времени. Обратите внимание: если монитор работает из одного расположения, линии тренда минимума и максимума совпадают.

![HTTP monitor performance](https://dt-cdn.net/images/httpdetailsperformanceminmax2-2216-152212e727.png)

HTTP monitor performance

При желании можно просматривать линии трендов среднего суммарного времени отклика всех запросов (**Response time by location**) или среднего **Response time by request** по всем расположениям.

Если монитор нарушает [пороговое значение производительности](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic#performance-thresholds "Узнайте о настройке HTTP-мониторов.") — для суммы всех запросов или отдельного запроса, над графиком производительности на время проблемы появляется сплошная красная линия. Кроме того, любое пороговое значение для суммы всех запросов отображается пунктирной красной линией. Нажмите сплошную красную полосу для отображения ссылки на страницу обзора проблемы.

![HTTP monitor performance problem](https://dt-cdn.net/images/httpdetailsperformanceviolation2-2217-2a18a384e0.png)

## Анализ деталей выполнения

Нажмите **Analyze execution details** в верхней части страницы деталей или **Analyze last failed execution** в карточке **Failed requests** для просмотра последних успешных, неуспешных и [выполнений по требованию](/managed/observe/digital-experience/synthetic-monitoring/general-information/on-demand-executions "Выполняйте синтетические мониторы по требованию из публичных или частных расположений.") HTTP-монитора для каждого расположения в формате JSON. Нажмите [**Executions difference**](#filters) для просмотра цветовой разницы между последними успешными, неуспешными или выполнениями по требованию.

Тело ответа и захваченные [HTTP-метрики](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/http-monitor-metrics-classic "Узнайте о метриках производительности, собираемых для HTTP-мониторов.") также отображаются вместе с кодом HTTP-статуса для каждого запроса. Если монитор содержит [учётные данные](/managed/manage/credential-vault "Храните и управляйте учётными данными в хранилище."), переданные в URL запроса, заголовках HTTP или теле запроса, в JSON отображается только идентификатор учётных данных в соответствующем элементе. (Эта функция требует ActiveGate версии 1.229.)

Можно ограничить отображение конфиденциальной информации (например, учётных данных, возвращённых в теле ответа) в деталях выполнения, выбрав **Do not store and display request and response bodies, header values, and peer certificate details in execution details** в [настройках монитора](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic "Узнайте о настройке HTTP-мониторов.").

Это нужно делать для каждого запроса, для которого нужно ограничить отображение. Тела запроса и ответа, значения заголовков запроса и ответа, а также детали одноранговых сертификатов заменяются замещающим текстом.

![Execution details hidden fields](https://dt-cdn.net/images/analyze-execution-details-hidden-948-0e70ad67cb.png)

Execution details hidden fields

Для успешных или неуспешных выполнений вкладки содержат JSON-данные для всех расположений. [Используйте фильтры](#filters) вверху или слева от каждой вкладки для фильтрации деталей выполнения. Можно просматривать и **Download** соответствующий отфильтрованный JSON-файл для успешных или неуспешных выполнений. Для выполнений по требованию можно загрузить JSON-файл для единственного выбранного выполнения. Также можно **Copy** URL окружения для получения JSON-файлов через API. Обратите внимание: GET-запросы API для получения выполнений синтетических мониторов требуют API-токенов.

![Download JSON and API link](https://dt-cdn.net/images/http-analyze-execution-download-1889-126037a9ff.png)

Download JSON and API link

Атрибут `customLogs` отображает метку времени, уровень журнала и сообщение для каждой строки пользовательского журнала, заданной с помощью [методов скриптинга](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic "Узнайте, как применять предварительные и последующие скрипты к запросам.") `api.info()`, `api.warn()`, `api.fail()` или `api.error()`. (Эта функция требует Dynatrace версии 1.255+ для публичных расположений и ActiveGate версии 1.255+ для частных расположений.)

![customLogs attribute](https://dt-cdn.net/images/http-analyze-execution-customlogs-2182-b4164106c0.png)

customLogs attribute

Вкладка **On-demand execution** перезаписывается при каждом выполнении по требованию. Используйте раскрывающийся список для выбора любого выполнения по требованию для монитора за предыдущие шесть часов. Если выполнения по требованию находятся в [режимах **Standard** или **Disable problem detection**](/managed/observe/digital-experience/synthetic-monitoring/general-information/on-demand-executions#ui "Выполняйте синтетические мониторы по требованию из публичных или частных расположений."), детали также записываются на вкладки последнего неуспешного/успешного выполнения. Обратите внимание: в этих режимах, если монитор завершается неуспешно из-за нарушения порогового значения производительности, выполнение появляется на вкладках успешных и выполнений по требованию.

## Размер ответа

Карточка **Response size** показывает линии трендов минимального и максимального размера ответа для суммы всех запросов, где любая заштрихованная область представляет разницу между двумя значениями в любой момент времени. Линии трендов могут совпадать (как показано ниже), например, когда монитор работает из одного расположения. Однако размер ответа может варьироваться, например, когда разные ответы отправляются в зависимости от расположения.

![Min and max response size for all requests taken together](https://dt-cdn.net/images/httpdetailsresponsesize-1119-5f3f338a3e.png)

Для отслеживания таких различий можно включить **Show response size breakdown per location**. Обратите внимание: если отфильтровать всю страницу деталей по каким-либо расположениям, переключатель становится недоступен.

## HTTP-запросы

HTTP-монитор может состоять из одного или нескольких HTTP-запросов. Карточка **HTTP requests** даёт обзор всех выполненных запросов, их порядок, имя, URL запроса и используемый HTTP-метод. Для каждого запроса карточка **HTTP requests** разбивает производительность (**Response time**) по следующим метрикам (подробнее см. в разделе [Метрики HTTP-мониторов](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/http-monitor-metrics-classic "Узнайте о метриках производительности, собираемых для HTTP-мониторов.")):

* **DNS lookup time**
* **TCP connect time**
* **TLS handshake time**
* **Waiting**
* **Download**

**Response time** — это сумма этих различных метрик.

Разверните запрос из списка для просмотра всех метрик производительности на одном графике. Выберите метрику в легенде для её удаления/добавления в график производительности. Нажмите **Edit request** для перехода к настройкам монитора из этой карточки.

![HTTP requests card](https://dt-cdn.net/images/httpdetailsrequests2-1096-3c247f0d9a.png)

HTTP requests card

Вкладка **Status codes** показывает временную линию возвращённых HTTP-кодов статуса для конкретного запроса. (Карточка [HTTP status codes](#status-codes) показывает возвращённые HTTP-коды статуса для HTTP-монитора в целом.)

Если запрос нарушает своё [пороговое значение производительности для конкретного события](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic#performance-thresholds "Узнайте о настройке HTTP-мониторов."), он выделяется красным. Разверните запрос, чтобы увидеть тайминги производительности и нарушенное пороговое значение. На время проблемы над составным графиком появляется сплошная красная линия; пороговое значение запроса отображается пунктирной красной линией. Нажмите сплошную красную полосу для отображения ссылки на страницу обзора проблемы.

![HTTP request performance violation](https://dt-cdn.net/images/httpdetailsrequestviolation2-1096-dbe95d7ef3.png)

Нажмите **Add tag** для добавления дополнительных тегов. Обратите внимание: теги можно добавлять и удалять только со страницы деталей.

## Сервисы

Карточка **Services** отображает все отслеживаемые сервисы, автоматически связанные с HTTP-монитором. Эта карточка появляется, когда отслеживаемый HTTP-эндпоинт размещён на хосте под управлением OneAgent и URL эндпоинта обрабатывается одним из обнаруженных сервисов. Связь между HTTP-монитором и сервисом устанавливается на основе данных, предоставленных OneAgent. Поэтому перед тем как сервис может быть связан с монитором, необходимо завершить хотя бы одно выполнение монитора. Выберите ссылку для отображаемого сервиса для перехода на страницу обзора сервиса, отфильтрованную по HTTP-монитору.

![Services card](https://dt-cdn.net/images/httpdetailsservice-931-dc29452459.png)

Services card

## Отслеживаемые приложения

HTTP-мониторы позволяют отслеживать внутренние ресурсы и API-эндпоинты, например ключевые бэкенд-API для входа в систему или операций поиска, используемых мобильными приложениями. Такие HTTP-мониторы можно связывать с отслеживаемыми мобильными, веб- или пользовательскими приложениями. Нажмите **Assign monitor to application**. (Приложение можно связать напрямую в [настройках монитора](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic#setup "Узнайте о настройке HTTP-мониторов.").)

![Link an application](https://dt-cdn.net/images/httpdetailslinkapp2-1102-1f74517f99.png)

Link an application

Если для приложений, с которыми работает синтетический монитор, включён [Real User Monitoring (RUM)](/managed/observe/digital-experience/rum-concepts/rum-overview "Узнайте о Real User Monitoring, ключевых метриках производительности, мониторинге мобильных приложений и многом другом."), Dynatrace автоматически связывает RUM-приложения с монитором, и отображается карточка **Monitored applications**. Здесь можно просматривать ключевые метрики приложения и переходить непосредственно к данным RUM.

После связывания HTTP-монитора с приложением доступность синтетического монитора отображается непосредственно в деталях приложения, а Davis автоматически связывает обнаруженные проблемы синтетического мониторинга с привязанным приложением.

Обратите внимание: нельзя блокировать трафик Synthetic Monitoring для RUM-приложений путём [исключения ботов, поисковых роботов или IP-адресов расположений Synthetic](/managed/observe/digital-experience/web-applications/additional-configuration/exclude-browsers-robots-and-spiders-from-monitoring "Отключите Real User Monitoring для определённых IP-адресов, браузеров, ботов и поисковых роботов.").

## Проблемы

Карточка **Problems** показывает проблемы производительности (нарушение порогового значения) и доступности (локальный или глобальный сбой) при включении соответствующих пороговых значений в [настройках монитора](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic "Узнайте о настройке HTTP-мониторов."). Разверните карточку для просмотра активных и разрешённых проблем за выбранный период.

Подробнее об определении пороговых значений производительности и доступности см. в разделе [Настройка HTTP-мониторов](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic "Узнайте о настройке HTTP-мониторов."). Подробнее о расчёте доступности и производительности, генерации и снятии проблем см. в разделе [Расчёты Synthetic](/managed/observe/digital-experience/synthetic-monitoring/general-information/synthetic-calculations "Поймите расчёты метрик Synthetic Monitoring."). Подробнее о рабочем процессе и концепциях оповещений, включая настройку профилей и шаблонов уведомлений, см. в разделе [Обзор оповещений Synthetic](/managed/observe/digital-experience/synthetic-monitoring/analysis-and-alerting/synthetic-alerting-overview "Узнайте о концепциях и рабочем процессе оповещений Synthetic.").

Существует три основных типа проблем для HTTP-мониторов:

* Глобальный сбой (доступность)
* Локальный сбой (доступность)
* Нарушение порогового значения производительности (проблема производительности для суммы всех запросов или отдельных запросов)

* Проблемы производительности могут объединять нарушения пороговых значений как на уровне монитора, так и на уровне запросов.
* Одновременно возникающие проблемы могут объединяться в одну проблему (**Multiple application problems**).

## События

Карточка **Events** показывает все события, составляющие проблемы. В списке и на временной линии отображаются события как активных, так и разрешённых проблем.

Наведите курсор на временной слот на временной линии событий, чтобы увидеть тип и количество событий, созданных за этот интервал. Выберите временной слот для просмотра всплывающей подсказки с событиями, произошедшими в нём.

Выберите тип события, например **HTTP monitor location slowdown**, чтобы увидеть список событий. Для каждого расположения, в котором монитор нарушает пороговые значения производительности на уровне запроса или монитора, создаётся одно событие замедления. Выберите отдельное событие для просмотра деталей.

[Метод `api.fail()`](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/pre-and-post-scripting-for-http-monitors-classic#end "Узнайте, как применять предварительные и последующие скрипты к запросам.") можно использовать для определения пользовательского **Failure message**, которое появляется в карточке Events при сбое. Пользовательские сообщения журнала также появляются в атрибуте `customLogs` в [деталях выполнения HTTP-монитора](#analyze-last-execution).

## Неуспешные запросы

Карточка **Failed requests** показывает количество каждого неуспешного запроса в выполнениях HTTP-монитора за выбранный период, разбитое по имени запроса и коду ошибки. На временном линейном графике отображается количество кодов ошибок — наведите курсор на любой временной слот для просмотра количества различных кодов статуса за этот интервал. Выберите код ошибки в легенде для его скрытия/отображения на графике.

Выберите имя запроса для его редактирования в [настройках монитора](/managed/observe/digital-experience/synthetic-monitoring/http-monitors-classic/configure-http-monitors-classic "Узнайте о настройке HTTP-мониторов."). Нажмите [**Analyze last failed execution**](#analyze-last-execution) для просмотра последних неуспешных и успешных выполнений в формате JSON.

![Failed requests card](https://dt-cdn.net/images/httpdetailsfailedrequests-945-1510d2a6e1.png)

Failed requests card

Обратите внимание: монитор, испытывающий простой из-за неуспешных запросов, вызывает проблемы сбоя только при настроенных пороговых значениях сбоев.

## HTTP-коды статуса

Карточка **HTTP status codes** отображает временную линию возвращённых HTTP-кодов статуса для выполнений HTTP-монитора в целом, то есть код статуса последнего выполненного запроса в мониторе, успешного или неуспешного. Если монитор содержит несколько запросов, например три, и монитор завершается неуспешно на втором запросе, третий запрос не выполняется. Сообщаемый код статуса относится ко второму запросу.

Наведите курсор на любой временной слот на временной линии для просмотра количества различных кодов статуса за этот интервал. Выберите код статуса в легенде для его скрытия/отображения на графике.
