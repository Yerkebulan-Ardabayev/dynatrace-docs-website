# Настройка автоматической инжекции

# Настройка автоматической инжекции

* How-to guide
* Updated on Jul 01, 2025

После развёртывания OneAgent в режиме полного стека мониторинга на хосте он по умолчанию автоматически инжектирует RUM JavaScript в HTML веб-приложений для технологий, перечисленных в разделе [Technology support - Real User Monitoring - Web servers and applications](/managed/ingest-from/technology-support#rum-auto-injection "Find technical details related to Dynatrace support for specific platforms and development frameworks."). Для запуска инжекции достаточно перезапустить процессы веб-сервера после установки OneAgent.

По умолчанию инжекция следует набору [встроенных правил автоматической инжекции](#where-is-rum-js-injected), однако в некоторых сценариях может потребоваться определить [пользовательское правило инжекции](#custom-injection-rule). Пользовательские правила инжекции также поддерживают [исключение отдельных страниц из инжекции](#disable-injection), что позволяет настроить [ручную вставку для страниц автоматически инжектируемого приложения](#manual-insertion-using-oneagent).

RUM JavaScript отвечает за сбор RUM-данных из браузеров ваших клиентов. Он отправляет эти данные в виде XHR POST-запросов на конечную точку маяка, которая выполняет первичную проверку перед пересылкой данных в ваш Dynatrace Cluster. Конечная точка маяка по умолчанию в сценариях автоматической инжекции предоставляется OneAgent, инструментирующим ваш веб-сервер или сервер приложений, однако возможны альтернативные конфигурации, см. раздел [Configure beacon endpoint for web applications](/managed/observe/digital-experience/web-applications/additional-configuration/beacon-endpoint "Change the default beacon endpoint URL and send RUM beacons to Dynatrace infrastructure or another instrumented web server.").

Фрагмент RUM JavaScript, инжектируемый в HTML вашего приложения, может иметь разные форматы, каждый из которых разработан для удовлетворения конкретных требований, описанных в разделе [Select a snippet format](/managed/observe/digital-experience/web-applications/initial-setup/snippet-formats "Select a format for the RUM JavaScript snippet that best fits your specific use case").
Для настройки формата, используемого при автоматической инжекции, обратитесь к разделу [Configure the snippet format](#configure-snippet-format).

OneAgent инжектирует RUM JavaScript в каждую страницу на основе текущей конфигурации каждый раз, когда инструментированный веб-сервер или сервер приложений доставляет страницу. Однако политика кэширования страницы определяет, насколько часто OneAgent имеет возможность выполнять эту инжекцию. Для решения этой проблемы OneAgent включает функцию оптимизации заголовков управления кэшем. Подробнее о том, как работает эта функция и каковы её ограничения, см. в разделе [Cache control header optimizations](#cache-header-optimization).

Если у вас нет доступа к веб-серверу, на котором размещено ваше приложение, и поэтому вы не можете установить OneAgent, вы можете осуществлять мониторинг приложения с помощью [агентлесс RUM](/managed/observe/digital-experience/web-applications/initial-setup/set-up-agentless-real-user-monitoring "Set up agentless monitoring for your web applications.").

## Встроенные правила автоматической инжекции

OneAgent определяет ответы, содержащие HTML-контент, и инжектирует RUM JavaScript в раздел head каждой страницы. Он не изменяет другие ресурсы, такие как изображения, CSS-файлы, JSON- или XML-документы или обычный текст.

OneAgent старается инжектировать RUM JavaScript как первый скрипт на странице, что помогает обеспечить надёжные результаты мониторинга. Инжекция работает только с валидным HTML, соответствующим базовым принципам — например, все элементы должны быть правильно открыты и закрыты. OneAgent следует встроенным правилам, указанным ниже, чтобы определить наиболее подходящее место в HTML-документе для инжекции RUM JavaScript.

### Спецификация правил автоматической инжекции

| Название | Условие | Действие | Переопределяет |
| --- | --- | --- | --- |
| <!DOCTYPE> | Тег `<!DOCTYPE>` не является `<!DOCTYPE html>`. | Прервать и не выполнять инжекцию. Допускается только одно объявление Doctype. |  |
| initial tag | Перед `<html>` встречается тег, не являющийся одним из следующих: `<!DOCTYPE ...>`, `<html>`, `<link>`, `<meta>`, `<script>`, `<style>`. | Прервать без инжекции. |  |
| <?xml?> | Встречается спецификация `<?xml ...?>`. | Игнорировать и продолжить сканирование документа. | non-<meta> tag, non-<head> tag, initial tag |
| <html> tag | Встречается тег `<html>`. | Если ранее найдена потенциальная точка инжекции — выполнить инжекцию там и прекратить сканирование. Используется самая ранняя из найденных точек. В противном случае продолжить сканирование документа. |  |
| <script src> tag | В `<head>` найден тег `<script src="...">`. | Если ранее найдена потенциальная точка инжекции — выполнить инжекцию там и прекратить сканирование. Используется самая ранняя из найденных точек. В противном случае выполнить инжекцию перед этим тегом `<script>` и прекратить сканирование. | non-<meta> tag, <base>/<meta> tag |
| <title>/<noscript> tag | Встречается тег `<title>` или `<noscript>`. | Игнорировать всё до тега `</title>` или `</noscript>`, затем продолжить сканирование документа. | <body> tag |
| <base>/<meta> tag | Встречается тег `<base>` или `<meta>`. | Отменить любую условную точку инжекции, найденную ранее. Продолжить сканирование документа. | любая условная точка инжекции, найденная ранее по: non-<meta> tag, unclosed <meta>, non-<head> tag |
| non-<meta> tag | В `<head>` найден тег, не являющийся `<meta>` или `<title>`. | Выполнить инжекцию перед ним (условная инжекция). Продолжить сканирование документа — этот выбор может быть переопределён. |  |
| unclosed <meta> | Тег `</head>` встречается после тега `<meta>`, который не закрыт ни тегом `</meta>`, ни XML-стилем `<meta ... />`. | Добавить `</meta>` с последующей инжекцией, оба перед `</head>` (условная инжекция). Продолжить сканирование документа — этот выбор может быть переопределён. |  |
| comment | Встречается тег `<!- comment ->`. | Игнорировать и продолжить сканирование документа. | non-<meta> tag, non-<head> tag |
| non-<head> tag | После `<html>`, но до `<head>` найден тег, не являющийся `<head>` или `<body>`. | Выполнить инжекцию перед ним (условная инжекция). Продолжить сканирование документа — этот выбор может быть переопределён. |  |
| </head> tag | Встречается тег `</head>`. | Если ранее найдена потенциальная точка инжекции — выполнить инжекцию там и прекратить сканирование. Используется самая ранняя. В противном случае продолжить сканирование документа. |  |
| flush | Только Java. Flush вызывается на инжектирующем потоке/writer'е вне секции `<head></head>`, и ожидается подтверждение условной инжекции. | Отменить условную точку инжекции, передать flush и продолжить сканирование для поиска новой точки. | любая условная точка инжекции, найденная ранее по: non-<head> tag |
| flush in <head> | Только Java. Flush вызывается на инжектирующем потоке/writer'е внутри `<head></head>`, и ожидается подтверждение условной инжекции. | Сохранить точку инжекции и продолжить сканирование. Игнорировать flush. |  |
| <body> tag | Встречается тег `<body>`. | Сканирование документа останавливается. Используется любая потенциальная точка инжекции, найденная ранее. Используется самая ранняя. Если ни одно правило ранее не совпало — выполнить инжекцию после `<body>`. |  |
| end of file | Достигнут конец файла, и сканирование не завершилось ранее. | Не выполнять инжекцию. | любая условная точка инжекции, найденная ранее по: non-<meta> tag, unclosed <meta>, non-<head tag |
| parse error | Содержимое документа не имеет базовой структуры, ожидаемой от HTML-тегов и атрибутов. | Сканирование документа останавливается. Если ранее найдена потенциальная точка инжекции — выполнить инжекцию там и прекратить сканирование. Используется самая ранняя. Если ни одно правило не предоставляет более ранней точки инжекции — не выполнять инжекцию. |  |

Если вы не уверены, был ли RUM JavaScript успешно инжектирован, или испытываете проблемы с настройкой RUM, обратитесь к разделу [Web applications: Issues with RUM JavaScript](https://dt-url.net/up03l19).

## Явное указание места инжекции RUM JavaScript

В некоторых случаях OneAgent не может выполнить инжекцию с помощью [встроенных правил](#where-is-rum-js-injected). Как правило, это связано с некорректным HTML, хотя другие редкие условия также могут помешать набору правил определить подходящую точку инжекции. В таких сценариях можно обеспечить автоматическую инжекцию, определив пользовательское правило инжекции, явно указывающее место инжекции RUM JavaScript в HTML-документе.

Чтобы создать пользовательское правило инжекции, указывающее точку инжекции в HTML-документе:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Injection** > **Custom injection rules**.
5. В разделе **Define custom injection rules** выберите **Add custom rule**.
6. Определите **Operator** и, при необходимости, соответствующий **URL pattern**, чтобы указать, к каким страницам применяется правило.
7. В разделе **Rule** выберите **Before specific HTML** или **After specific HTML** и введите соответствующий HTML-шаблон.
8. Выберите **Save changes** для создания пользовательского правила инжекции.

## Отключение инжекции для отдельных страниц

Чтобы отключить инжекцию для одной или нескольких страниц вашего приложения:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Injection** > **Custom injection rules**.
5. В разделе **Define custom injection rules** выберите **Add custom rule**.
6. Определите **Operator** и, при необходимости, соответствующий **URL pattern**, чтобы указать, к каким страницам применяется правило.
7. В разделе **Rule** выберите **Do not inject**.
8. Выберите **Save changes** для создания пользовательского правила инжекции.

Это правило отключает только инжекцию и [оптимизацию заголовков управления кэшем](#cache-header-optimization); оно не отключает полностью функциональность RUM. Например, cookie могут по-прежнему устанавливаться. Если вам нужен способ выборочного включения RUM, обратитесь к инструкциям в разделе [Roll out RUM selectively for your applications](/managed/observe/digital-experience/web-applications/initial-setup/selective-rum-rollout "Roll out RUM selectively after installing OneAgent in full-stack monitoring mode on your hosts").

## Настройка формата фрагмента

Чтобы настроить формат фрагмента для автоматической инжекции:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Injection** > **Automatic injection**.
5. В разделе **Snippet format** выберите желаемый формат.

Подробнее о различных форматах фрагментов см. в разделе [Select a snippet format](/managed/observe/digital-experience/web-applications/initial-setup/snippet-formats "Select a format for the RUM JavaScript snippet that best fits your specific use case").

## Оптимизация заголовков управления кэшем

Включено по умолчанию

Политика кэширования вашего приложения напрямую влияет на то, насколько часто OneAgent может инжектировать RUM JavaScript в страницу с использованием текущей конфигурации. Для обеспечения своевременного применения изменений конфигурации функция оптимизации заголовков управления кэшем изменяет HTTP-заголовки таким образом, чтобы условные запросы приводили к промаху кэша при изменении конфигурации RUM, давая OneAgent возможность выполнить инжекцию. Подробнее об этих оптимизациях и конкретных изменениях заголовков описано ниже.

Спецификация оптимизации заголовков управления кэшем

Когда оптимизация заголовков управления кэшем включена и конфигурация RUM изменилась, запросы, которые в противном случае привели бы к ответу `304 Not Modified`, возвращают `200 OK`, давая OneAgent возможность инжектировать RUM JavaScript в HTML вашего приложения. Если конфигурация RUM не изменилась, запросы возвращают ответ `304 Not Modified`.

### Изменяемые заголовки

OneAgent изменяет следующие заголовки ответа и запроса:

#### Заголовки ответа

* `ETag`
* `Last-Modified`

#### Заголовки запроса

* `If-None-Match`
* `If-Modified-Since`
* `If-Unmodified-Since`[1](#fn-1-1-def)
* `If-Match`[1](#fn-1-1-def)

1

Не изменяется для range-запросов, которые в целом исключены из мониторинга RUM.

Ниже описано, как OneAgent изменяет заголовки управления кэшем.

#### При изменении конфигурации RUM

* Входящие заголовки запроса `If-None-Match` и `If-Modified-Since` удаляются.
* Изменения в заголовках `If-Unmodified-Since` и `If-Match` отменяются.
* Если используются оба заголовка `ETag` и `Last-Modified`:

  + К заголовку `ETag` добавляется суффикс.
  + Из заголовка `Last-Modified` вычитается одна секунда.
* Если используется только заголовок `ETag`:

  + К заголовку `ETag` добавляется суффикс.
* Если используется только заголовок `Last-Modified`:

  + Создаётся заголовок `ETag`.
  + Из заголовка `Last-Modified` вычитается одна секунда.

#### При отсутствии изменений конфигурации RUM

* До обработки запроса веб-сервером или сервером приложений изменения в заголовках отменяются, чтобы сервер не видел этих изменений.

Когда приложение состоит из нескольких инструментированных уровней, изменения заголовков `ETag` и `Last-Modified` применяются на каждом уровне.

OneAgent не изменяет заголовки `Expires` и `Cache-Control`, которые явно определяют время истечения срока действия ресурсов. Для обеспечения своевременных обновлений рекомендуется устанавливать более короткие сроки истечения в этих заголовках.

### Отключение оптимизации заголовков управления кэшем

В редких случаях оптимизация заголовков управления кэшем может влиять на функциональность вашего приложения. В таком случае можно отключить эту функцию.
При отключении развёртывание новых конфигураций RUM будет полностью зависеть от существующего поведения кэширования вашего приложения.

Чтобы отключить оптимизацию заголовков управления кэшем:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Injection** > **Automatic injection**.
5. Отключите параметр **Optimize the value of cache control headers for use with Dynatrace Real User Monitoring**.
6. Выберите **Save changes**.

### Отключение оптимизации заголовков управления кэшем для не-HTML ресурсов

В некоторых случаях OneAgent может изменять заголовки управления кэшем для не-HTML ресурсов. Это происходит потому, что изменения заголовков применяются до того, как можно определить точный тип содержимого. Хотя OneAgent исключает определённые ресурсы из оптимизации заголовков управления кэшем на основе суффикса URI, учесть все типы не-HTML контента невозможно.

Если вы хотите предотвратить изменения заголовков управления кэшем для определённых не-HTML ресурсов, создайте [пользовательское правило инжекции, отключающее инжекцию для этих ресурсов](#disable-injection).

## Использование ручной вставки для страниц автоматически инжектируемого приложения

Если OneAgent развёрнут в группах процессов и автоматическая инжекция поддерживается для вашей технологии, OneAgent по умолчанию инжектирует RUM JavaScript автоматически. Однако при желании его можно вставить вручную во все или определённые страницы.

Сначала необходимо подавить автоматическую инжекцию, как описано в разделе [Disable injection for particular pages](#disable-injection). Затем для ручной вставки RUM JavaScript:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Injection** > **Manual insertion**.
5. Выберите необходимый [формат фрагмента](/managed/observe/digital-experience/web-applications/initial-setup/snippet-formats "Select a format for the RUM JavaScript snippet that best fits your specific use case") и скопируйте предоставленный фрагмент.
6. Вставьте фрагмент в HTML перед всеми остальными скриптами.

Кроме того, фрагмент можно получить через [API](/managed/dynatrace-api/environment-api/rum/real-user-monitoring-javascript-code "Learn how you can use the Dynatrace API to set up and maintain your manually injected applications using the Real User Monitoring JavaScript API."), что позволяет интегрировать его вставку в процесс сборки. Это обеспечивает постоянную работу приложения с актуальной конфигурацией.
