# Создание пользовательских имён для пользовательских действий веб-приложений

# Создание пользовательских имён для пользовательских действий веб-приложений

* How-to guide
* 12-min read
* Updated on Jan 27, 2023

Имена [пользовательских действий](/managed/observe/digital-experience/rum-concepts/user-actions "Learn what user actions are and how they help you understand what users do with your application.") генерируются автоматически на основе URL загрузок или XHR и зачастую труднозапоминаемы. При большом количестве пользовательских запросов и взаимодействий в день отслеживание пользовательских действий становится затруднительным без возможности логической группировки.

Правила именования пользовательских действий позволяют генерировать последовательные и читаемые имена, которые легко группировать и отслеживать в приложении.

Правила именования создаются и настраиваются для каждого приложения индивидуально, после чего используются в Dynatrace в:

* Графиках пользовательского опыта
* Графиках
* Целях конверсии (например, `add-to-shopping-cart` и `newsletter-sign-up`)
* Ключевых пользовательских действиях
* Списке топ-потребителей

Для удобного создания и поддержки правил именования пользовательских действий можно использовать placeholder'ы, входные параметры и разные правила для действий Load и XHR.

## Определение имени действия

Dynatrace пытается назначать осмысленные имена для действий. Для этого проверяются несколько свойств действия — например, inner HTML, заголовок и подсказка HTML-элемента, инициирующего действие. Это может быть кнопка или якорь. Также предпринимается попытка получить заголовок при наличии более сложной HTML-структуры с несколькими вложенными тегами.

RUM JavaScript использует несколько методов для выбора имени, наилучшим образом подходящего для действия. Начиная с самого внутреннего HTML-узла, по которому щёлкнули (кнопка, тег изображения или ссылка), проверяется следующее в порядке приоритета:

1. [Атрибут `data-dtname`](#change-action-name-via-data-dtname)
2. `nodeName` — например, изображение, якорь или ввод
   Поиск прекращается при обнаружении тега `html`, `body`, `head` или элемента `document`.
3. `innerText` или `textContent`

Если ни один из них не возвращает разумный результат, RUM JavaScript применяет рекурсивный алгоритм, проверяющий различные вещи в зависимости от `nodeName` текущего HTML-узла. Если ничего не найдено, проверяется родительский узел.

## Добавление и использование placeholder'ов

Placeholder'ы позволяют добавлять динамические элементы в правила пользовательского именования.

Placeholder'ы полезны, когда нужно включать динамические входные значения (например, URL записей блога, посещённых пользователями) в имена пользовательских действий, или когда нужно удалить или заменить определённые части входного значения (например, динамические идентификаторы пользовательских сессий) и сгруппировать их под общим именем пользовательского действия.

Placeholder'ы состоят из трёх элементов:

* Входное значение — например, URL страницы
* Шаг обработки, извлекающий или заменяющий части входных данных (необязательно)
* Имя placeholder'а, используемое в правиле именования пользовательских действий

Для удобства Dynatrace предоставляет набор предопределённых placeholder'ов: `pageUrl`, `sourceUrl`, `xhrUrl`, `Top XHR URL` и `pageTitle`. Для лучшего понимания placeholder'ов рассмотрим пример: XHR-действие инициирует нажатие кнопки подтверждения на странице оформления заказа. Нажатие кнопки подтверждения (исходное действие) сопровождается одним из следующих целевых действий:

* Инициируются несколько связанных XHR-запросов по первому веб-запросу — например, `xhrUrl` или `Top XHR URL`.
* Инициируется новое действие загрузки, перенаправляющее на страницу подтверждения. В этом случае есть `sourceUrl` — URL, где завершился процесс оформления заказа, и `pageUrl` — URL, где отображается подтверждение.

Такие цепочки и корреляции действий лучше всего видны на графике водопада. Можно использовать `sourceURL` и `pageURL` или их XHR-эквиваленты `XHRUrl` и `Top XHR URL` для правильного именования действий.

Предоставленный набор placeholder'ов нельзя настроить, но их можно сразу использовать для создания правил именования. Также можно создавать собственные placeholder'ы.

### Создание placeholder'а

Чтобы создать placeholder:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **User actions**.
5. В разделе **Placeholders** выберите **Add placeholder** и укажите тип входных данных.

   Для типов данных **CSS selector**, **JavaScript variable**, **Meta tag** и **Cookie value** также необходимо указать значение.

   ![Add a placeholder](https://dt-cdn.net/images/screenshot-2020-08-27-at-15-22-45-1308-aba05796c1.png)

   Add a placeholder
6. Необязательно: добавьте шаги обработки и условия и просмотрите placeholder, чтобы убедиться, что он работает корректно.
7. Необязательно: при добавлении шага обработки **extract** или **extract with regular expression** выберите, использовать ли пустой результат по умолчанию или выбранные входные данные в качестве запасного варианта.
8. Назовите placeholder и выберите **Save**.

На одно приложение можно добавить до 50 placeholder'ов.

### Шаги обработки

Доступны следующие шаги обработки:

* **extract**: требует разделитель начала и разделитель конца извлечения. Разделители не включаются в извлечение. Если начальный разделитель отсутствует, извлечение начинается с начала входных данных. Если конечный разделитель отсутствует, извлечение заканчивается в конце входных данных. Необязательно: можно использовать пустой результат по умолчанию или выбранные входные данные в качестве запасного варианта.
* **replace**: требует разделитель начала и разделитель конца замены. Необязательно: можно указать текст замены. Если текст замены не указан, совпавшие символы отбрасываются.
* **replace text**: требует текст для идентификации и сопоставления. Необязательно: можно указать текст замены. Если текст замены не указан, совпавший текст отбрасывается.
* **replace IDs**: требует идентификатор для идентификации и сопоставления. Необязательно: можно указать замещающие идентификаторы. Если замещающий идентификатор не указан, совпавший идентификатор отбрасывается.
* **extract with regular expression**: требует регулярное выражение для идентификации и извлечения. Необязательно: можно использовать пустой результат по умолчанию или выбранные входные данные в качестве запасного варианта.
* **replace with regular expression**: требует регулярное выражение для идентификации заменяемой строки. Необязательно: можно указать замещающее регулярное выражение. Если оно не указано, совпавшее регулярное выражение отбрасывается. В настоящее время поддерживается только одна группа захвата `(.*)`.

При добавлении нескольких шагов обработки к placeholder'у они применяются последовательно: первый шаг применяется к выбранным входным данным, каждый последующий — к результату предыдущего шага.

## Создание имён действий

Dynatrace предоставляет возможность создавать пользовательские действия загрузки и XHR. Для создания правил именования можно использовать новые или существующие placeholder'ы из списка **Placeholders**.

В этом примере используется `CampaignID` для создания нового правила именования, идентифицирующего загрузку страницы как просмотр кампании с добавлением `CampaignID` к имени действия.

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **User actions**.
5. Прокрутите страницу вниз до раздела **User action naming rules** и выберите вкладку **Naming rules for load actions** или **Naming rules for XHR actions**.
6. Выберите **Add naming rule**.
7. Введите шаблон именования с использованием placeholder'а. Убедитесь, что placeholder заключён в фигурные скобки (например: `{CampaignID}`).
8. Необязательно: определите условия, при которых также можно использовать placeholder'ы для уточнения применимости правил.

   В нашем примере для создания правила именования, применяемого только к специальной рекламной кампании с `CampaignID` равным **3**, можно использовать условия и placeholder'ы для сопоставления только нужных действий.

   ![Creating custom user actions](https://dt-cdn.net/images/campaignid-974-5548a30b62.png)

   Creating custom user actions
9. Необязательно: используйте раздел **Preview your rule** для проверки нового правила именования.
10. Выберите **Save**.
11. Используйте элементы управления **Move up/down** для изменения приоритета правил.

    Первое совпадающее правило в таблице применяется первым.

    ![User action naming rules](https://dt-cdn.net/images/user-action-naming-2077-281508f770.png)

    User action naming rules

На одно приложение можно добавить до 250 правил именования пользовательских действий.

## Расширенная настройка имён действий

Доступны следующие дополнительные параметры настройки имён пользовательских действий:

* **Split user actions by domain**: отключите этот параметр, если разные домены не должны приводить к разным пользовательским действиям. Например, если приложение доступно по двум доменам, `www.domain-one.com` и `www.domain-two.com`, и разбивка по доменам включена, `load of page /index` для `www.domain-one.com` будет другим пользовательским действием, чем `load of page /index` для `www.domain-two.com`.
* **Case-insensitive naming**: отключите этот параметр, если имена пользовательских действий должны сохраняться с учётом регистра. Именование без учёта регистра позволяет выполнять поиск по всем вариантам регистра. Например, пользовательские действия `Loading of Page /` и `Loading of PAGE /` будут соответствовать `loading of page /`.
* **Prioritize load actions and use value of last detected**: включите этот параметр для использования улучшенного распространения действий загрузки. Когда в ходе действия присутствует загрузка (документный запрос), действие помечается как действие загрузки.
  Кроме того, включите этот параметр для использования значений placeholder'ов (например, `pageUrl`) последнего обнаруженного действия загрузки для именования пользовательских действий. Рассмотрим пример с начальным действием `click /easytravel/search`, которое инициирует перезагрузку страницы с `/easytravel` на `/easytravel/search` и, следовательно, действие загрузки `loading of page /easytravel/search`, которое, в свою очередь, инициирует другой XHR-вызов.

  + При включённом параметре используется последнее обнаруженное действие загрузки, и поэтому эта цепочка действий помечается как действие загрузки `loading of page /easytravel/search`.
  + При отключённом параметре действие помечается как `click /easytravel/search`, поскольку в предыдущей версии распространения загрузки она не распространяется в сценариях с XHR-запросами.

## Примеры использования

Ниже рассматриваются примеры правил именования пользовательских действий из предыдущей версии и показано, как создавать эквивалентные правила в новой версии.

#### Извлечение ценной информации из входных данных

Примеры из предыдущей версии:

* **Extract** `/some-path-before/(.*).some-defined-text-after from path` из path
* **Extract** `Loading of page /sales/(.*)` из имени действия
* **Extract** `(.*)` из заголовка страницы
* **Extract** `(.*)` из XHR URL

Тех же результатов теперь можно достичь с помощью placeholder'ов и шага обработки.
С помощью шага обработки **extract** можно определить начальные и конечные разделители, точно указывающие, откуда извлекать значения. Таким образом можно заменить многие правила именования, ранее требовавшие регулярных выражений.

#### Именование действий на основе URL или пути

Примеры из предыдущей версии:

* **Design step**: если имя действия содержит `design`
* **Config data call**: имя действия соответствует регулярному выражению `*/config/*`
* **Product**: путь начинается с `/product/`
* **Search**: XHR URL начинается с `/api/search/dynamic`

В предыдущей версии имя действия обычно использовалось как входное значение для переименования автоматически обнаруженных Dynatrace имён. Теперь можно использовать placeholder по умолчанию в правиле именования, например `pageUrlPath` или `pageUrl`, для получения той же информации в сочетании с условием **begins with** или **contains**.

Пример из предыдущей версии:

* `Customer Contact Request for UFO`: XHR url соответствует regex `.*/api/request/.*/contact?category=UFO`

#### Удаление или маскирование частей входных данных

Примеры из предыдущей версии:

* **Remove ID**: `(\\?|&)_id=:*?(?=&|$)`
* **Remove conversation**: `(\\?|&)conversation=:*?(?=&|$)`
* **Remove params**: `\?.*`
* **Remove searched car**: `(?<=search-results\/cars\/)[a-zA-Z0-9]*`

Удаление и маскирование частей данных теперь проще реализуется с помощью подхода «удаление между». Для удаления значения определённого параметра строки запроса, например идентификатора пользователя, определите ключ как конечный разделитель и `&` как следующий символ, вводящий другие параметры.

Для большей гибкости или, например, для удаления всех параметров URL используйте **replace by regular expression**.

#### Консолидация правил именования с помощью placeholder'ов

Рассмотрим оформление заказа в интернет-магазине. Для именования действий загрузки для каждого шага в предыдущей версии требовалось определять по одному правилу для каждого шага.

В этом случае на каждый шаг оформления назначалось одно правило:

* **Loading of Enter personal details**: имя действия начинается с `/checkout/index.php?step=details`
* **Loading of Select payment method**: имя действия начинается с `/checkout/index.php?step=payment`
* **Loading of Order summary**: имя действия начинается с `/checkout/index.php?step=summary`
* **Loading of Order confirmation**: имя действия начинается с `/checkout/index.php?step=confirmation`

В новой версии можно создать placeholder, использующий переменную JavaScript `adobePageName` как входные данные, которые могут быть уже доступны в существующем слое данных аналитического инструмента (например, Adobe Analytics).

## Установка имени действия через `data-dtname`

Если стандартное обнаружение имени действия не подходит, можно задать пользовательский атрибут `data-dtname` в HTML-тегах и использовать его как заголовок. Например, следующему действию:

```
<label for="txtFirstname">Firstname</label> <input data-dtname="Firstname Text Input" type="text" value="firstname" name="firstname" title="Firstname" id="txtFirstname" />
```

можно назначить заголовок `click on "Firstname Text Input"`.

Чтобы пользовательский атрибут отображался как имя пользовательского действия, необходимо также [настроить правило именования](#create-action-names), включающее placeholder `{elementIdentifier (default)}`.

При использовании разных имён атрибутов для разных инструментов можно выбрать альтернативу `data-dtname`, которую Dynatrace будет использовать для именования пользовательских действий.
