# Определение приложений для Real User Monitoring

# Определение приложений для Real User Monitoring

* How-to guide
* 14-min read
* Updated on Apr 23, 2024

После установки [OneAgent](/managed/ingest-from/dynatrace-oneagent "Understand the important concepts related to OneAgent and find out how to install and operate OneAgent on different platforms.") в режиме полного стека мониторинга на хосте он отслеживает все приложения, работающие на этом хосте. В качестве отправной точки все данные мониторинга инкапсулируются в приложение-placeholder под названием **My web application**. Это приложение-placeholder предоставляется для большей гибкости — именно пользователь решает, как организовать свои приложения.

Dynatrace предлагает следующие подходы к определению приложений для RUM.

| Подход к определению приложения | Тип инжекции RUM JavaScript | Когда использовать |
| --- | --- | --- |
| [Рекомендуемый подход](#automated-approach) | Автоматическая инжекция | Когда веб-приложения легко идентифицируются по доменам и нужно создать приложение из уже захваченного трафика RUM-мониторинга. |
| [Подход с правилами определения приложений](#application-detection-rules) | Автоматическая инжекция | Когда нужно создавать новые приложения или разделения трафика по доменам недостаточно. Правила определения приложений используются для определения более сложных шаблонов группировки трафика RUM-мониторинга. |
| [Ручной подход](#alternative-approach) (агентлесс-мониторинг) | Ручная вставка | Когда нет доступа к хосту веб-приложения, но есть доступ к коду приложения. |

Формат сниппета можно выбрать как для автоматически инжектируемых, так и для приложений с ручной вставкой. Подробнее см. в разделе [Выбор формата сниппета](/managed/observe/digital-experience/web-applications/initial-setup/snippet-formats "Select a format for the RUM JavaScript snippet that best fits your specific use case").

Java Servlet-based applications

При использовании RUM для приложения на основе Java Servlet обязательно перезапустите процесс после завершения начальной настройки. RUM-сенсор не включается до перезапуска.

## Рекомендуемый подход

Приложение-placeholder **My web application** агрегирует трафик со всех обнаруженных доменов. Оно служит отправной точкой для сопоставления идентифицированных доменов с отдельными приложениями в среде.

1. Перейдите в **Web**.
2. Выберите приложение-placeholder **My web application**.
3. Прокрутите страницу вниз до панели **Top 3 included domains**.
   На этой панели отображаются домены с наибольшим количеством пользовательских действий, обнаруженных OneAgent в среде.
4. Выберите **View full details**.
5. В списке **Top domains** выберите кнопку со стрелкой в столбце **Transfer domain** для развёртывания записи домена.
6. Следуйте одному из вариантов ниже:

   * Чтобы добавить новое приложение, выберите **Create new application**. Приложение будет создано и отображено на странице **Applications**.
   * Чтобы добавить домен к существующему приложению, выберите **Transfer**.

Можно заменить автоматически сгенерированное имя на пользовательское имя приложения.

Чтобы переименовать приложение:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **General settings** > **Application name**.
5. Введите предпочтительное имя в поле вверху страницы. Имена приложений должны быть уникальными.

### Возможные проблемы рекомендуемого подхода

**Ненаблюдаемый компонент перезаписывает URL**. При использовании рекомендуемого подхода для нового или обновлённого приложения автоматически создаётся правило определения приложения. Это правило сопоставляет пользовательские действия, захваченные на домене, с приложением. OneAgent использует часть хоста URL для определения домена. Если ненаблюдаемый компонент между браузером и первым инструментированным уровнем перезаписывает хост, необходимо [настроить определение имени хоста](#rum-appdetection-uninstrumentedcomponent). Иначе OneAgent может не применять правило определения приложения.

**Действия по-прежнему сопоставляются с приложением-placeholder**. Если пользовательские действия по-прежнему сопоставляются с **My web application** вместо приложения, см. раздел [Как быстро вступают в силу изменения правил определения приложений?](#rum-appdetection-rule-changes).

## Подход с правилами определения приложений

Для создания новых приложений, изменения существующих сопоставлений или определения более сложных правил на основе не только доменов, но и URL используйте страницу настроек **Application detection**.

URL для определения приложений имеют структуру `scheme://host:port/path?query`, где строка запроса необязательна, а стандартные порты `80` для HTTP и `443` для HTTPS опускаются. URL не включает возможный идентификатор фрагмента, как в `scheme://host:port/path?query#fragment`, поскольку правила определения приложений оцениваются на серверной стороне приложения, а идентификатор фрагмента используется только браузером и не добавляется к веб-запросам.

Чтобы добавить правило определения приложения:

1. Перейдите в **Settings** > **Web and mobile monitoring** > **Application detection**. Отображается список правил определения приложений.
   Для каждого приложения, определённого с помощью [рекомендуемого подхода](#automated-approach), правило определения автоматически генерируется и добавляется в конец списка. Правила применяются последовательно, приоритет имеют правила вверху списка.
2. Выберите **Add detection rule**.
3. Используйте предложенные параметры для создания подходящего правила определения для приложения.
   Правила можно применять к новым или существующим приложениям, а также создавать правила на основе URL или домена (хоста) приложения.
4. Выберите **Save** для создания правила определения приложения.
   Правило помещается в конец списка правил определения приложений.
5. Необязательно: выберите и удерживайте **Drag row** ![Drag handle](https://dt-cdn.net/images/drag-handle-turquoise-600-1aa0e5ea00.svg "Drag handle") рядом с именем правила и переместите его вверх или вниз в списке для изменения приоритета.

Можно создать до 1000 правил определения приложений на среду.

### Возможные проблемы подхода с правилами определения приложений

**Ненаблюдаемый компонент перезаписывает URL**. Если ненаблюдаемый компонент между браузером и первым инструментированным уровнем перезаписывает URL, необходимо принять особые меры (подробнее см. [Что делать, если ненаблюдаемый компонент перезаписывает части URL?](#rum-appdetection-uninstrumentedcomponent)). Иначе OneAgent может не применять правила определения приложений.

**Действия по-прежнему сопоставляются с предыдущим приложением**. Если пользовательские действия по-прежнему сопоставляются с предыдущим приложением вместо требуемого, см. раздел [Как быстро вступают в силу изменения правил определения приложений?](#rum-appdetection-rule-changes).

### FAQ по рекомендуемому подходу и подходу с правилами

Как Dynatrace применяет правила определения приложений?

Правила определения приложений оцениваются на серверной стороне приложения OneAgent'ом на первом инструментированном уровне — ближайшем к браузеру. Результат передаётся последующим OneAgent'ам с использованием заголовка `x-dynatrace-application`. При таком подходе любая перезапись URL между инструментированными серверными уровнями приложения не влияет на результат определения приложения.

OneAgent добавляет параметр `app` в инжектированный RUM JavaScript в HTML-код приложения. Этот параметр содержит идентификатор обнаруженного приложения.

![App parameter in the RUM JavaScript](https://dt-cdn.net/images/injected-app-parameter-2505-7ee0111d11.png)

App parameter in the RUM JavaScript

При отправке маяков с захваченными пользовательскими действиями RUM JavaScript включает параметр `app` в строку запроса.

![App parameter added to the query string](https://dt-cdn.net/images/app-parameter-in-beacon-query-1922-30c91adb9b.png)

App parameter added to the query string

На кластере Dynatrace параметр `app` в строке запроса маяка используется для назначения пользовательских действий приложениям.

Чтобы проверить идентификатор приложения для сравнения с параметром `app`:

1. В Dynatrace перейдите в **Web**.
2. Выберите приложение.
3. Проверьте URL текущей страницы. Он содержит параметр `uemapplicationId`, например `uemapplicationId=APPLICATION-4CC1C5694BF9B3AB`. Его значение без префикса `APPLICATION-` является искомым идентификатором приложения.

   ![Checking the application ID in the Dynatrace web UI](https://dt-cdn.net/images/application-id-2559-b6b9f31629.png)

   Checking the application ID in the Dynatrace web UI

Что делать, если ненаблюдаемый компонент перезаписывает части URL?

Как описано в разделе «Как Dynatrace применяет правила определения приложений?», правила определения приложений оцениваются OneAgent'ом на первом инструментированном уровне. Однако между браузером и первым инструментированным уровнем может находиться ненаблюдаемый компонент (обратный прокси или балансировщик нагрузки), перезаписывающий URL. В результате URL для определения приложения отличается от URL, изначально запрошенного браузером.

### Хост перезаписан

Компоненты, такие как прокси и балансировщики нагрузки, можно настроить для передачи исходной информации о хосте бэкенду в заголовке запроса — например, в [`X-Forwarded-Host`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/X-Forwarded-Host) или [`Forwarded`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Forwarded). OneAgent может получить исходный хост из одного из этих заголовков или из любого проприетарного заголовка того же формата, что и `X-Forwarded-Host`.

![Passing the original host information in the X-Forwarded-Host header](https://dt-cdn.net/images/host-name-determination-1093-9025d73d9b.png)

Passing the original host information in the X-Forwarded-Host header

Чтобы использовать исходное имя хоста при определении приложения:

1. Настройте ненаблюдаемый компонент для добавления заголовка с исходным именем хоста (если это не делается по умолчанию).
   Например, если исходное имя хоста, запрошенное браузером, — `www.example.com`, а компонент перенаправляет запрос на внутренний хост `backend.com`, компонент должен добавить заголовок запроса с `www.example.com`.
2. В Dynatrace перейдите в **Settings** > **Web and mobile monitoring** > **Host name determination**.
3. Если добавленный компонентом заголовок отсутствует в списке, выберите **Add HTTP header**.
4. Заголовки запроса в списке обрабатываются последовательно, приоритет имеют те, что вверху. Переместите новую запись выше заголовка `Host`, который есть в каждом HTTP/1.1-запросе.

Кроме того, можно использовать внутреннее имя хоста в правиле определения приложения. Однако OneAgent использует результат определения имени хоста не только для определения приложения, но и для [автоматического определения домена cookie](/managed/observe/digital-experience/web-applications/additional-configuration/configure-cookie-attributes#default-cookie-domain "Learn which RUM cookie attributes you can configure and how."). Поэтому при отсутствии настройки определения имени хоста необходимо вручную настроить домен cookie согласно разделу [Ручная настройка домена cookie](/managed/observe/digital-experience/web-applications/additional-configuration/configure-cookie-attributes#manual-cookie-domain-config "Learn which RUM cookie attributes you can configure and how."). Рекомендуемый выбор — [eTLD+1](https://web.dev/same-site-same-origin/) домена, который OneAgent определяет автоматически.

### Путь URL перезаписан

В отличие от части хоста URL, нет функции определения исходного пути URL, поскольку компоненты редко устанавливают заголовки, передающие исходный путь последующим уровням. Поэтому часть пути URL, удалённую при перезаписи ненаблюдаемым компонентом, нельзя использовать в правиле определения приложения.

Как быстро вступают в силу изменения правил определения приложений?

При изменении правил определения приложений обновлённые правила обычно передаются OneAgent в течение минуты. OneAgent категоризирует входящие запросы согласно правилам и устанавливает параметр `app` в инжектированном RUM JavaScript. RUM JavaScript затем отправляет маяки с обновлённым параметром `app`, определяющим категоризацию захваченных пользовательских действий.

Следующие факторы могут задержать вступление в силу обновлённых правил:

* HTML-код предоставляется через CDN или другой кэш. В этом случае OneAgent может инжектировать RUM JavaScript только когда HTML-код вытеснен из кэша и снова запрошен с инструментированного сервера-источника.
* Приложение указывает длительный срок истечения HTML-документа в заголовке ответа `Expires` или директиве `max-age` заголовка `Cache-Control`. Кэш может предоставлять его указанное время, включая инжектированный параметр `app`.
* Если приложение не указывает явный срок истечения, это не означает, что страница не кэшируется. Кэш может использовать страницу без повторной валидации, пока не истекут условия кэширования.
* Одностраничные приложения перезаписывают текущую страницу динамически вместо загрузки новой. Если HTML-код был загружен в браузерную вкладку до изменения правил, категоризация XHR-действий не отражает обновлённую конфигурацию. Это изменится при обновлении страницы.

## Агентлесс-подход

Если нет доступа к веб-серверу (и OneAgent нельзя установить на хост), но есть доступ к коду приложения, используйте [агентлесс RUM](/managed/observe/digital-experience/web-applications/initial-setup/set-up-agentless-real-user-monitoring "Set up agentless monitoring for your web applications.").

Для некоторых технологий [автоматическая инжекция RUM JavaScript](/managed/observe/digital-experience/web-applications/initial-setup/rum-injection "Configure automatic injection of the RUM JavaScript into the pages of your applications") не поддерживается даже при возможности установки OneAgent. Например, хотя OneAgent может отслеживать серверную часть приложения Heroku, он не может инжектировать RUM JavaScript в страницы. В таких случаях используйте агентлесс RUM и вручную добавляйте RUM JavaScript.

Список технологий и серверов, поддерживающих автоматическую инжекцию RUM JavaScript, см. в разделе [Technology support - Real User Monitoring - Web servers and applications](/managed/ingest-from/technology-support#rum-auto-injection "Find technical details related to Dynatrace support for specific platforms and development frameworks.").

## Рекомендации

* Определяйте приложения на основе принадлежности команд, чтобы удобно использовать [зоны управления](/managed/manage/identity-access-management/permission-management/management-zones "Learn about management zones concepts, how to define management zones, and how to make the most of them.") для ограничений доступа.
* Разумно определять приложения на основе стека технологий, чтобы применялись правильные настройки. Например, активация поддержки конкретных XHR-фреймворков обычно требуется только для отдельных частей крупного приложения и для конкретных правил именования.
* **Приложение-placeholder My web application**

  + Не переименовывайте приложение-placeholder **My web application** — оно включает все пользовательские действия на всех доменах, не включённых в правило определения. При переименовании может быть сложно отличить его от других приложений.
  + Это приложение-placeholder нельзя удалить.
  + Если **My web application** не найдено, его могли переименовать. Для поиска откройте любое другое приложение и замените идентификатор приложения в URL на `EA7C4B59F27D43EB`.
  + В новых средах этот placeholder отображается только после поступления трафика от автоматически инжектируемого приложения.
* Разделение приложений по доменам работает лучше всего, поскольку Dynatrace не может коррелировать пользовательские действия между доменами с конкретными пользовательскими сессиями. Эта корреляция выполняется через cookie и работает только если cookie может быть установлен на одном домене. Например, пользовательские действия для `www.dynatrace.com` и `blog.dynatrace.com` могут захватываться в одном приложении, поскольку cookie может быть установлен для `dynatrace.com`. Однако трафик `www.dynatrace.com` и `www.internal-dynatrace.com` не может быть захвачен в одной пользовательской сессии.
* Группируйте приложения с низким трафиком. Если создаётся приложение на основе домена с менее чем 10 действиями в минуту, Dynatrace не будет автоматически обнаруживать аномалии для него.
* Правила определения приложений обрабатываются последовательно для каждого запроса. Чем больше правил, тем больше время обработки — размещайте правила для приложений с наибольшим трафиком вверху списка.
* Более конкретные правила определения приложений должны быть определены первыми, более общие — в конце списка.
