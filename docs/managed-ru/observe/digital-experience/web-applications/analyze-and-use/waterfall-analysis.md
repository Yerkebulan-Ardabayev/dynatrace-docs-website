# Анализ водопада

# Анализ водопада

* How-to guide
* 7-min read
* Updated on Jan 24, 2023

Dynatrace захватывает данные о пользовательском опыте и производительности, отслеживая отдельные пользовательские действия. Как правило, пользовательское действие начинается с нажатия на HTML-элемент управления (например, кнопку или ссылку). Браузер затем загружает запрошенные данные — либо переходя на новую страницу, либо через XHR/fetch-запрос. Затем выполняются JavaScript-коллбэки, строится или изменяется DOM-дерево, и веб-приложение снова готово к новому пользовательскому действию. Представление **Waterfall analysis** в Dynatrace — идеальный инструмент для детального анализа всех данных мониторинга пользовательских действий.

Чтобы открыть анализ водопада для пользовательского действия:

1. Перейдите в **Web**.
2. Выберите приложение, содержащее нужное пользовательское действие.
3. Прокрутите страницу приложения вниз до раздела **Top 3 user actions**.
4. Выберите пользовательское действие (или нажмите **View full details** для доступа к дополнительным пользовательским действиям).
5. На странице **User action** выберите **View analysis in waterfall chart** для открытия **Waterfall analysis**.

## Данные в представлении водопада

Представление анализа водопада в основном строится на W3C [таймингах](/managed/observe/digital-experience/rum-concepts/user-action-metrics "Learn what metrics Dynatrace calculates for user actions and find out what each metric indicates."). Представление организует записи водопада в три раздела:

**Document requests** — Отображает весь контент, идентифицированный как загрузка страницы для пользовательского действия. Включает исходный HTML, а также другой контент, отображаемый во фреймах. Каждая запись содержит детали, захваченные W3C navigation timings.

**XHR requests** — Отображает все ресурсы, отправленные через XHR или fetch(). Записи содержат все детали, доступные через W3C resource timings, а также время выполнения JavaScript-коллбэков.

**Resources** — Отображает все детали W3C timings для всех ресурсов.

Пользовательское действие может содержать более одной загрузки страницы или XHR. Поэтому исходный запрос выделен фиолетовым цветом. [Тайминги](/managed/observe/digital-experience/rum-concepts/user-action-metrics "Learn what metrics Dynatrace calculates for user actions and find out what each metric indicates.") исходного запроса отображаются в виде вертикальных событийных линий над графиком.

## Топ результатов

[Топ результатов](/managed/observe/digital-experience/web-applications/analyze-and-use/visually-complete-top-findings "Lean how to leverage Visually complete top findings provided in the waterfall analysis.") для водопадов помогает понять пользовательский опыт видимой без прокрутки части для реальных пользователей. Начальные загрузки страниц формируют первые впечатления новых посетителей веб-приложения. Кроме того, топ результатов предоставляет удобный способ понять, какие веб-запросы влияют на [ключевые метрики производительности](/managed/observe/digital-experience/rum-concepts/user-action-metrics#kpm "Learn what metrics Dynatrace calculates for user actions and find out what each metric indicates.") и являются потенциальными кандидатами для оптимизации.

Для каждого пользовательского действия, просматриваемого в **Waterfall analysis**, автоматически идентифицируются и представляются следующие топ результаты:

* Несжатые текстовые ресурсы
* Ресурсы размером более 100 кБ
* Ресурсы с частотой кэширования браузером менее 50%
* Медленные собственные, сторонние или CDN ресурсы (> 200 мс)
* Контактируемые сторонние и CDN домены
* Собственные, сторонние и CDN ресурсы

## Агрегированное представление

В то время как в инструментах разработчика браузера всегда приходится анализировать отдельные выполнения, агрегированный водопад Dynatrace позволяет получить сводное представление нескольких выполнений одних и тех же пользовательских действий от различных реальных пользователей. Агрегации рассчитываются из выполнений пользовательских действий в временном диапазоне анализа и группируются в 3 агрегированных водопада; можно переключаться между агрегированными представлениями водопада, отображающими самые медленные 10%, самые быстрые 10% и ±5% от медианы. Это помогает выявлять запросы и паттерны, связанные с высокой или низкой производительностью.

### Редко встречающиеся ресурсы

Для поддержания агрегированного представления **Waterfall analysis** в приемлемом и читаемом размере ресурсы, встречающиеся менее чем в 30% экземпляров пользовательских действий, перемещаются в группу **Rarely occurring**. Эту группу можно развернуть для просмотра редко встречающихся ресурсов.

## Представление отдельного экземпляра

Хотя автоматические результаты — отличный способ начать анализ водопада пользовательских действий и найти способы улучшения опыта клиентов, иногда необходимо более глубоко изучить данные производительности.

Хотя агрегированная информация полезна для обзора этапов обработки пользовательских действий и использования ресурсов, иногда необходимо анализировать отдельные экземпляры конкретных пользовательских действий для понимания влияния определённых браузеров, геолокаций или других переменных. Поэтому доступен анализ водопада отдельных экземпляров пользовательских действий. Отдельные экземпляры пользовательских действий перечислены на вкладке **Instances of this action**.

### Сквозная видимость и детализация в распределённые трассировки

В представлении **Waterfall analysis** при анализе производительности отдельного экземпляра пользовательского действия, для которого доступны [распределённые трассировки](/managed/observe/application-observability/distributed-traces "Gain observability into highly distributed, cloud-native architectures to effectively trace and analyze transactions in real time."), доступен топ результат **Traces**. В примере ниже показано, что для выбранного экземпляра пользовательского действия доступны `52 Traces` для анализа. Выберите плитку **Traces** для получения ссылки на список связанных распределённых трассировок. Топ результаты также выделяют все ресурсные записи в графике анализа водопада, для которых доступны распределённые трассировки.

![Waterfall analysis](https://dt-cdn.net/images/waterfall-1-2000-30253b2468.png)

Waterfall analysis

Следует учитывать следующие ограничения:

* Сквозная видимость требует наличия экземпляра OneAgent на серверной стороне. Топ результаты отображаются только при наличии распределённых трассировок.
* Для процессов под высокой нагрузкой Dynatrace OneAgent автоматически регулирует объём данных, отправляемых на Dynatrace Server. Поэтому некоторые распределённые трассировки могут не захватываться из-за [адаптивного управления трафиком](/managed/ingest-from/dynatrace-oneagent/adaptive-traffic-management/adaptive-traffic-management-managed "Improve your Dynatrace Managed environment health and performance with the adaptive features of traffic management, load reduction, and capture control.").
* Распределённые трассировки для XHR-вызовов между разными источниками недоступны в анализе водопада по умолчанию. Для их обеспечения необходимо реализовать один из подходов, описанных в разделе [Связывание XHR-действий между разными источниками и их распределённых трассировок](/managed/observe/digital-experience/web-applications/initial-setup/link-cross-origin-xhrs "Enable the correlation between cross-origin XHR actions and distributed traces.").
* Dynatrace оценивает различные факторы для корреляции ресурсов водопада (на основе W3C resource timings) и распределённых трассировок (захваченных OneAgent). Одним из факторов является URL. Если части URL не идентичны на клиентской и серверной стороне — например, из-за правил перезаписи — кнопка **View trace** может не отображаться. В таких случаях выберите плитку **Traces** в топ результатах для доступа ко всем распределённым трассировкам.

## Настройки приложения, влияющие на анализ водопада

В настройках приложения можно настраивать пороговые значения для топ результатов, а в глобальных настройках — определять типы ресурсов и правила очистки URL.

### Настройка пороговых значений результатов водопада

Чтобы настроить пороговые значения результатов водопада:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **Advanced setup**.
5. Скорректируйте нужные пороговые значения в разделе **Waterfall finding thresholds**.

### Определение типа ресурса

В представлении анализа водопада можно группировать ресурсы по типу. Поскольку MIME-тип недоступен из JavaScript, тип ресурса определяется по расширению файла. Если некоторые ресурсы не имеют правильного расширения, можно настроить правила соответствия URL для определения их типов.

Чтобы настроить правило соответствия URL для определения типа ресурса:

1. Перейдите в **Settings**.
2. Выберите **Web and mobile monitoring** > **Resource types**.

### Определение правил очистки URL

В агрегированном представлении **Waterfall analysis** ресурсы агрегируются по URL. Часто идентификаторы и другие случайные числа препятствуют правильной агрегации ресурсов. В этом случае некоторые важные ресурсы могут непреднамеренно оказаться в списке **Rarely occurring**.

Чтобы определить правило очистки URL:

1. Перейдите в **Settings** > **Web and mobile monitoring** > **Resource URL cleanup rules**.
2. Определите [регулярные выражения](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Learn how to use regular expressions in the context of Dynatrace.") для удаления идентификаторов.
