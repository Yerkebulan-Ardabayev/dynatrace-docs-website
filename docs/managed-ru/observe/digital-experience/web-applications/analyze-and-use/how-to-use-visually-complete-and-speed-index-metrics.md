# Использование метрик «Visually complete» и «Speed index»

* How-to guide
* 9-min read
* Published Jul 03, 2018

Для более глубокой оценки пользовательского опыта была [представлена новая, улучшенная версия алгоритма Visually complete](https://www.dynatrace.com/news/blog/more-visibility-into-user-experience-with-new-web-performance-metrics-and-enhanced-visually-complete/), обеспечивающая более точный контроль над расчётами. В сочетании с дополнительными метриками производительности улучшенная версия Visually complete даёт точные, стабильные измерения и более широкие возможности управления тем, что включается в расчёты. Для глобального перехода на новую версию во всей среде перейдите в **Web** и выберите любое приложение из списка. Затем прокрутите страницу вниз до раздела **Enable Visually complete 2.0 and new web performance metrics** и выберите **Enable for all applications**.

Поддержка более ранней версии Visually complete прекратится начиная с версии RUM JavaScript 1.211 для RUM и Synthetic версии 1.212.

Visually complete — это точечная метрика, измеряющая момент завершения загрузки видимой области страницы. Значения метрики Visually complete, как правило, меньше аналогичных метрик (например, времени загрузки страницы и показателя DOM interactive), поскольку пользователи воспринимают завершение загрузки страницы до того, как загружены 100% фоновых элементов. Оптимизация времени визуально завершённой загрузки важнее оптимизации других показателей загрузки страницы с точки зрения пользовательского опыта, так как отражает время ожидания пользователями полной загрузки контента, видимого без прокрутки. Результаты сортируются по расположению, устройству, операционной системе и типу браузера.

Visually complete означает, что все элементы в видимой области веб-страницы загружены на 100%:

* Элементы
* Изображения
* IFrames (только изображения)

Видимая область простирается от верхней части страницы до нижней границы окна браузера (прокрутка пользователем не влияет на область страницы, используемую для измерения).

Преимущества Visually complete:

* Позволяет видеть точное время, которое уходит у конечных пользователей на получение нужной информации.
* Обеспечивает истинно бизнес-ориентированную метрику с точки зрения пользователя.
* Даёт 100% видимость реального пользовательского опыта независимо от устройства.
* Объединяет ИТ и бизнес в части связи пользовательского опыта с бизнес-результатами.
* Ускоряет повышение производительности.
* Оптимизирует решения в разработке, эксплуатации и пользовательском опыте.
* Может сочетаться с мощными графиками водопада для понимания и улучшения времени рендеринга страниц.

Вместе с Speed index Visually complete показывает общее время загрузки страницы целиком. Поэтому эти метрики собираются только для основных фреймов, но не для субфреймов. В случаях, когда фреймы имеют разное происхождение и невозможно определить родительский фрейм, сбор Speed index и Visually complete также предпринимается для субфреймов.

## Расчёт Visually complete

#### Расчёт Visually complete для действий загрузки

Visually complete начинает наблюдение за содержимым DOM после инжекции RUM JavaScript.

Наблюдение прекращается по срабатыванию события Load event end, при условии что последний добавленный элемент страницы появился более одной секунды назад. При наличии повторяющихся изменений на странице Visually complete автоматически завершается по таймауту через 3 секунды после срабатывания Load event end. Для расчёта используются все элементы, присутствующие на странице в момент завершения наблюдения.

Visually complete проверяет, являются ли добавленные элементы видимыми или находятся ли они в видимой области. Если да, то время появления последнего видимого элемента на странице используется в качестве Visually complete. Для внешних ресурсов, таких как изображения, используется Resource Timings API браузера для получения времени загрузки, а не момент размещения тега `<img>` на странице.

#### Расчёт Visually complete для XHR-действий

Для XHR-действий наблюдение за содержимым DOM начинается с `callback begin` XHR-действия и завершается через 50 миллисекунд после `callback end`. Любые мутации с внешними ресурсами, такими как изображения, таблицы стилей или IFrames, заставляют RUM JavaScript добавлять слушатель загрузки к этим ресурсам. Это продлевает время XHR-действия до последнего ресурса, добавленного в наблюдаемом временном диапазоне.

Visually complete затем рассчитывается для этих ресурсов. Visually complete может отличаться от времени окончания действия. Но когда Visually complete имеет значение `-1` (значение по умолчанию) — например, когда к DOM не добавляются видимые ресурсы — для Visually complete используется время окончания действия.

Причина, по которой для XHR-действий поддерживаются только ресурсы, а не все элементы, заключается в следующем: при параллельных XHR-действиях и нескольких наблюдениях одновременно невозможно соотнести появляющиеся на странице элементы с нужным действием.

## Расчёт Speed index

Speed index полезен для сравнения пользовательского опыта на различных страницах. Чем меньше значение, тем лучше пользовательский опыт страницы. Подробное определение Speed index и описание его расчёта см. в [документации WebPagetest](https://dt-url.net/gl03s4g).

Алгоритм в RUM JavaScript совпадает с алгоритмом WebPageTest, но использует видимые элементы, захваченные из Visually complete, — «реальные» элементы страницы. Преимущество перед сравнением скриншотов (webpagetest.org) в том, что Visually complete осведомлён об элементах страницы и их тайминге. Сравнение скриншотов отражает только визуальные изменения страницы. Например, если бы на странице была вращающаяся GIF-анимация, показатели Visually complete и Speed index на webpagetest.org оказались бы неточными.

Speed index вычисляет индекс, измеряя прогресс загрузки веб-страницы между определёнными маркерами вместе с процентом незагруженных элементов в определённый момент времени.

* (200 мс × 100%) (ничего не загружено до 200 мс)
* (200 мс × 60%) (к первому маркеру уже загружено 40%; то есть 60% должно быть загружено для достижения 100%)
* (200 мс × 40%) (40% должно быть загружено для достижения 100%)
* Speed index = (200 мс × 100%) + (200 мс × 60%) + (200 мс × 40%) = 400

Speed index применим только к действиям загрузки, так как использует всю страницу как эталон для расчёта индекса. Например, если на странице есть маленькая иконка, загружающаяся 5 секунд, тогда как остальная часть страницы загружается за 2 секунды, Visually complete составит 5 секунд (иконка — последний элемент). Но Speed index будет близок к 2 секундам, поскольку основная часть страницы загрузилась за 2 секунды.

## Параметры настройки улучшенного Visually complete

В версиях Dynatrace 1.194 и более поздних новый алгоритм Visually complete предоставляет больший контроль над расчётом метрики, позволяя задавать следующие параметры:

* **Threshold**: минимальная видимая площадь элемента (в пикселях), при которой он учитывается в Visually complete и Speed index. То есть в расчёты включаются только элементы, площадь которых превышает заданное минимальное значение. Для определения порогового значения используйте свойство `vct`.
* **Inactivity timeout for load actions**: время, которое модуль Visually complete ожидает бездействия и отсутствия дальнейших мутаций на странице после действия загрузки. Для определения таймаута бездействия используйте свойство `vcit`.
* **Mutation timeout for XHR and custom actions**: время, которое модуль Visually complete ожидает после закрытия XHR или пользовательского действия перед началом расчёта. Для определения таймаута мутации используйте свойство `vcx`.
* **Excluded URL list**: регулярные выражения для определения URL изображений и IFrames, исключаемых из обнаружения модулем Visually complete. Для создания списка используйте свойство `iub`.
* **Ignored mutation list**: CSS-селекторы запросов для указания узлов мутации (элементов, которые меняются), игнорируемых при расчёте Visually complete и Speed index. Для создания списка используйте свойство `mb`.

Чтобы настроить эти параметры:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **Content capture**.
5. Добавьте нужный параметр.

   * `Threshold`:
     Используйте значение от 0 до 10000; значение по умолчанию — 50.
   * `Inactivity timeout for load actions`:
     Время в миллисекундах — используйте значение от 0 до 30000; значение по умолчанию — 1000.
   * `Mutation timeout for XHR and custom actions`:
     Время в миллисекундах — используйте значение от 0 до 5000; значение по умолчанию — 50.
   * `Excluded URL list`:
     Используйте символ | в качестве разделителя между записями, например `\.dynatrace\.com\/login|\.dynatrace\.com\/logout`. Разделение между разными протоколами URL не поддерживается; исключается каждый протокол URL.
   * `Ignored mutation list`:
     Используйте символ запятой в качестве разделителя между записями, например `#someDomElement@someattribute,.someOtherDomElement`.

## Использование Visually complete в качестве ключевой метрики производительности

В версиях Dynatrace 1.193 и более ранних Visually complete является стандартной [ключевой метрикой производительности](/managed/observe/digital-experience/web-applications/analyze-and-use/work-with-key-performance-metrics "Learn how to use the right key performance metrics to optimize user experience data for each of your applications."), применяемой для оценки производительности всех действий загрузки и XHR-действий. Тайминги Visually complete и Speed index по умолчанию захватываются для всех приложений с включённым Real User Monitoring.

Можно изменить ключевую метрику производительности для действий загрузки и XHR-действий. Также можно отключить захват таймингов Visually complete и Speed index, однако повторное включение захвата сбросит периоды эталонного обнаружения аномалий, определённые для приложения.

Чтобы установить ключевую метрику производительности для приложения:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. Выберите **General settings** > **Load actions** / **XHR actions** / **Custom actions**.
5. В разделе **Key performance metric** выберите ключевую метрику производительности из списка для каждого типа пользовательских действий.

Чтобы отключить захват таймингов Visually complete и Speed index:

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **Content capture**.
5. Отключите параметр **Visually complete & Speed index**.

### Использование Visually complete с одностраничными приложениями

Для фреймворков одностраничных приложений, таких как Angular, Visually complete обеспечивает ещё более точные тайминги для пользовательских действий. С Visually complete тайминги не останавливаются при получении ответов XHR. Благодаря этому можно получить дополнительные сведения о динамической загрузке ресурсов, инициируемой изменениями DOM-элементов на основе JavaScript из ответа XHR-запроса.

### Корреляция производительности пользователей с бизнес-метриками

Производительность — один из главных факторов пользовательского опыта и успешных бизнес-результатов. Метрика Visually complete обеспечивает этот анализ в готовом виде.

### Visually complete для анализа пользовательских действий

Visually complete и Speed index захватываются для каждого шага в пользовательском действии.

1. Перейдите в **Web**.
2. Выберите приложение для анализа.
3. В разделе **Performance analysis** выберите раздел действий для отображения данных о пользовательских действиях.
4. На плитке **Top 3 user actions** выберите **View full details**.
5. В разделе **Key user actions** или **Top 100 user actions** выберите действие, чтобы увидеть его тайминги вклада и временную шкалу действия на плитке **Contributors breakdown**. Временная шкала показывает тайминги для всех соответствующих стадий действия, включая **Speed index** и **Visually complete**. Метрика производительности, используемая для шага, отображается над временной шкалой, позволяя быстро определить, используется ли «User action duration» или «Visually complete» в качестве метрики производительности.

### Устранение неполадок с Visually complete и Speed index

При использовании в качестве ключевой метрики производительности:

* Visually complete рассчитывается при изменении визуального элемента на странице, и это изменение передаётся в каждое последующее действие загрузки и XHR-действие (но не асинхронные веб-запросы, не влияющие на DOM).
* Speed index доступен только для действий загрузки.

Если данные Visually complete или Speed index неточны, проверьте следующие страницы сообщества Dynatrace:

* [Web applications: Visually complete and Speed index metrics not shown](https://dt-url.net/pw438xh)
* [Web applications: Inaccurate Visually complete and Speed index metrics](https://dt-url.net/jd638ur)
