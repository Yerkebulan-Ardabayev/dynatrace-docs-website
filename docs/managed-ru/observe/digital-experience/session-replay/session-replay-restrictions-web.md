# Технические ограничения Session Replay для веб-приложений

* Explanation
* 4-min read
* Updated on Feb 27, 2024

Session Replay совместим со страничными приложениями, одностраничными приложениями (SPA) и приложениями, использующими IFrame. Однако применяются определённые ограничения.

* [Поддерживаемые версии браузеров для Session Replay](/managed/discover-dynatrace/get-started/dynatrace-ui/dynatrace-web-ui-requirements "Узнайте, в каких браузерах работает Dynatrace Managed.")

## Технологические ограничения

Следующие технологии не поддерживаются:

* Frames
* Canvas
* WebGL
* Web Animations API
* Плагины, такие как Adobe Flash Player, Java applets и нативные технологии (не HTML)
* Документы, такие как PDF-файлы и документы Word
* Псевдоклассы CSS :visited, :focus и :active
* Movie replay

## IFrame

Запись IFrame и вложенных IFrame поддерживается. Однако применяются следующие ограничения:

* В каждый IFrame должен быть внедрён RUM JavaScript. Если RUM JavaScript внедрён только в родительскую страницу, а не в IFrame, содержимое IFrame и взаимодействия не будут отображаться в воспроизведении. Это связано с тем, что просмотреть содержимое IFrame с родительской страницы невозможно.
* Запись IFrame работает только при наличии RUM JavaScript на странице. Если RUM JavaScript есть только в IFrame (который не является родительской страницей), воспроизведение покажет его как верхний фрейм записи. В таком случае взаимодействия в родительском фрейме (например, прокрутка) не будут записаны.
* Одно и то же приложение должно мониторировать родительскую страницу и IFrame. При использовании разных приложений для родительской страницы и IFrame последний не будет корректно отображаться при воспроизведении.

## Ресурсы

По умолчанию Session Replay [захватывает и хранит CSS-ресурсы](/managed/observe/digital-experience/session-replay/configure-session-replay-web#sr-resource-capturing "Настройте потребление мониторинга и параметры конфиденциальности данных для Session Replay.") во время записи сессии. Однако Session Replay не может воспроизвести [URL-адреса blob/object](https://w3c.github.io/FileAPI/#DefinitionOfScheme).

Если [захват ресурсов для Session Replay](/managed/observe/digital-experience/session-replay/configure-session-replay-web#sr-resource-capturing "Настройте потребление мониторинга и параметры конфиденциальности данных для Session Replay.") отключён, применяются следующие ограничения. Эти ограничения также распространяются на изображения и шрифты даже при включённом захвате ресурсов.

* Ресурсы должны быть доступны в браузере, используемом для воспроизведения. Должно быть прямое соединение с сервером, обслуживающим ресурсы.
* Ресурсы не могут быть защищены. Если ресурсы защищены, убедитесь, что проигрыватель может получить к ним доступ без аутентификации.
* Ресурсы не могут быть персонализированы. Это относится к ресурсам, которые различаются для каждого пользователя, но обслуживаются по одному URL. Однако это ограничение не распространяется на URL, которые различаются для каждого персонализированного ресурса. Пример: аватары, использующие один URL для передачи разных изображений разным пользователям.

Для преодоления этих ограничений рекомендуется [включить захват ресурсов для Session Replay](/managed/observe/digital-experience/session-replay/configure-session-replay-web#sr-resource-capturing "Настройте потребление мониторинга и параметры конфиденциальности данных для Session Replay.").

## Динамические формы

Если вы динамически обновляете значения в форме и они не отображаются в записи Session Replay, проверьте код приложения и убедитесь в способе обновления. Возможно, вы обновляете свойство `.value` элемента формы. Хотя это допустимый способ обновления, он не вызывает никаких событий на странице, и Dynatrace не узнаёт об изменении. В качестве альтернативного решения рекомендуется обновлять значение атрибута (для элементов `<input>`) или свойство `.textContent` (для элементов `<textarea>`). Таким образом Dynatrace получит уведомление об изменениях и запишет их корректно.

## Объём данных

В среднем минута записи и воспроизведения сессии потребляет 100 КБ дискового пространства. Это также необходимый объём исходящей полосы пропускания от клиентской системы до серверов Dynatrace. Однако это значение не гарантировано, поскольку зависит от веб-приложения. Приложения с очень частыми изменениями DOM, большим начальным размером или активным использованием IFrame могут генерировать большие объёмы данных.

## Передача данных и OneAgent

Убедитесь, что используется OneAgent версии 1.193+, поскольку он поддерживает высокий объём передачи beacon.

Поддержка высокого объёма beacon доступна для всех агентов с возможностью Real User Monitoring:

* Apache
* NGINX
* Java
* IIS
* Node.JS

## Поддержка приложений

Веб-приложения разнообразны. Хотя Session Replay создан для работы с любым веб-приложением и включает поддержку IFrame и одностраничных приложений, могут встречаться случаи, когда воспроизведение пользовательских взаимодействий не будет идеальным. По этой причине Session Replay может не воссоздать пользовательский опыт надлежащим образом для всех веб-приложений.

Сайты, изменяющие нативные API браузера, могут вызывать конфликты с кодом JavaScript-агента и препятствовать корректной записи страницы.

## Среда

Session Replay должен передавать данные из браузеров клиентов в ваш кластер Dynatrace. Передача данных может завершаться неудачей в зависимости от среды, доступности сети, скорости, сетевых задержек, проблем с подключением, пропускной способности и других факторов.
