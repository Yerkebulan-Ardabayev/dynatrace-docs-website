# Настройка Session Replay для веб-приложений

# Настройка Session Replay для веб-приложений

* How-to guide
* 22-min read
* Updated on Mar 05, 2026

Вы можете настроить потребление мониторинга и параметры конфиденциальности данных для Session Replay. В следующих разделах описаны все возможные параметры конфигурации:

* [**Cost and traffic control**](#cost-traffic-control) для сокращения количества записываемых сессий пользователей
* [**Opt-in mode**](#sr-opt-in-mode) для определения, какие части сессии пользователя должны записываться, и предоставления пользователям возможности дать согласие на запись своих сессий
* [**URL exclusion**](#sr-url-exclusion) для исключения страниц и представлений из записи
* [**Masking**](#sr-masking) для предотвращения записи и отображения личных данных пользователей
* [**Resource capture**](#sr-resource-capturing) для захвата и хранения стилей (CSS) во время записи сессии
* [**User permissions and management zones**](#sr-permissions-and-management-zones) для управления доступом к записям сессий

Dynatrace ввёл несколько параметров конфигурации Session Replay для защиты личных данных клиентов. Перед включением Session Replay и настройкой параметров конфиденциальности, описанных на этой странице, убедитесь, что ваша организация предприняла все остальные [необходимые шаги для защиты данных клиентов](/managed/manage/data-privacy-and-security "Узнайте о мерах безопасности Dynatrace для защиты личных данных.").

Возможность воспроизводить записанные сессии с настройками [маскирования](#sr-masking) или без них [контролируется разрешениями](#sr-permissions). Разрешения **Replay sessions with masking** и **Replay sessions without masking** доступны на уровне среды и зоны управления.

## Управление стоимостью и трафиком

После включения Session Replay все сессии пользователей, анализируемые с помощью RUM, также записываются с Session Replay. Однако вы можете ограничить количество сессий, записываемых Session Replay, чтобы снизить объём данных. Для этого настройте параметры управления стоимостью и трафиком.

### Изменение количества записываемых сессий

Для ограничения количества сессий, записываемых с Session Replay:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **General settings** > **Enablement and cost control**.
5. В разделе **Session Replay** введите новое значение для **Cost and traffic control**.

### Расчёт количества записываемых сессий

Для определения фактического процента сессий, записываемых Session Replay, следует учитывать настройку [управления стоимостью и трафиком для RUM](/managed/observe/digital-experience/web-applications/additional-configuration/configure-cost-and-traffic-control-web "Используйте настройки управления стоимостью и трафиком в Dynatrace для снижения потребления сессий в веб-приложениях."), заданную для вашего приложения (общий процент сессий, анализируемых с RUM).

#### Формула

Используйте следующую формулу для расчёта фактического процента сессий, записанных Session Replay:

**Фактический % сессий, записанных SR = управление стоимостью и трафиком для RUM × управление стоимостью и трафиком для Session Replay**

#### Пример расчёта

| Параметр | Значение |
| --- | --- |
| Управление стоимостью и трафиком для RUM | 50% |
| Управление стоимостью и трафиком для Session Replay | 20% |
| Фактический % сессий, записанных Session Replay | 50% × 20% = 10% |
| Общее количество сессий | 2500 |
| Количество сессий, анализируемых с RUM | 2500 × 50% = 1250 |
| Количество сессий, записанных Session Replay | 2500 × 10% = 250 |

#### Сессии, охватывающие несколько приложений

При наличии нескольких приложений и переходе пользователя из одного в другое Dynatrace создаёт только одну сессию, в контексте которой записывается вся активность. Такая сессия привязывается ко всем приложениям, посещённым пользователем.

Для определения того, должна ли многоприложенческая сессия записываться Session Replay, Dynatrace использует конфигурацию управления стоимостью и трафиком того приложения, в которое пользователь вошёл первым.

Например, если управление стоимостью и трафиком для Session Replay установлено на `1%` для приложения A и `100%` для приложения B, при переходе пользователя из A в B Dynatrace использует значение `1%` для определения необходимости записи. Это может приводить к меньшему или большему проценту сессий, записанных Session Replay, по сравнению с настроенными значениями.

## Режим opt-in

Режим opt-in Session Replay позволяет определить, какие части сессии должны записываться и когда начинается запись. Например, можно записывать сессии в следующих случаях:

* Как только любой пользователь входит в систему
* Только для определённых клиентов для предоставления им расширенной поддержки
* Только для определённых страниц приложения

Этот механизм позволяет реализовать согласие конечных пользователей на запись сессий.

При включении режима opt-in Session Replay для веб-приложения запись активной сессии начинается только после вызова метода [`enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay) на глобальном объекте `dtrum`. Глобальный объект `dtrum` доступен, если [RUM JavaScript](/managed/observe/digital-experience/web-applications/initial-setup/snippet-formats "Выберите формат RUM JavaScript-сниппета для вашего случая") [внедрён автоматически](/managed/observe/digital-experience/web-applications/initial-setup/rum-injection "Настройте автоматическое внедрение RUM JavaScript в страницы ваших приложений") или [вставлен вручную](/managed/observe/digital-experience/web-applications/initial-setup/set-up-agentless-real-user-monitoring "Настройте мониторинг без агента для ваших веб-приложений.").

Эта команда запускает запись сессии. Session Replay остаётся активным, и запись начинается автоматически на всех последующих посещаемых страницах в рамках той же сессии или до вызова [`dtrum.disableSessionReplay()`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#disablesessionreplay).

Метод [`dtrum.enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay) включает параметр `ignoreCostControl`, позволяющий записывать определённые сессии в обход значения, заданного в разделе [Cost and traffic control](#cost-traffic-control).

Если включён [режим opt-in Real User Monitoring](/managed/observe/digital-experience/web-applications/additional-configuration/configure-real-user-monitoring-according-to-gdpr#user-opt-in-mode-gdpr "Узнайте о настройках конфиденциальности Dynatrace для соответствия веб-приложений требованиям защиты данных."), перед включением Session Replay необходимо включить RUM, например:

`dtrum.enable();`
`dtrum.enableSessionReplay(true);`

### Пример

Рассмотрим следующий сценарий. Как владелец приложения вы хотите записывать все сессии, включающие страницы со 2-й по 5-ю. Активность на страницах 1 и 6 должна быть исключена из записи. На следующей схеме показано, где требуются методы [`dtrum.enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay) и [`dtrum.disableSessionReplay()`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#disablesessionreplay).

![Иллюстрация записи сессии](https://dt-cdn.net/images/session-recording-illustration-568-a36c3c47ad.png)

Иллюстрация записи сессии

В таких случаях можно отображать баннер согласия для включения записи при переходе пользователя на страницу 2. При выборе пользователем **Accept** приложение вызывает метод [`dtrum.enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay) и начинает запись.

![Пример баннера opt-in Session Replay](https://dt-cdn.net/images/banner-example-2569-b86bd49e59.png)

Пример баннера opt-in Session Replay

Для хранения истории согласия пользователя в браузере можно использовать cookie. Содержимое этого cookie проверяется при каждой сессии для определения необходимости отображения баннера. Например, если cookie хранит согласие под именем `sessionReplayConsent`, поток приложения будет следующим:

1. Приложение проверяет значение cookie `sessionReplayConsent`.
2. Если значение равно `true`, вызывается [`dtrum.enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay).
3. Если значение равно `false`, отображается баннер согласия.
4. При предоставлении согласия вызывается [`dtrum.enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay).
5. Согласие пользователя записывается в cookie `sessionReplayConsent`.

С этим cookie Session Replay остаётся активным до 5-й страницы приложения.

После выхода пользователя со страницы 5 можно использовать метод [`dtrum.disableSessionReplay()`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#disablesessionreplay) для остановки записи и удалить cookie согласия.

Методы JavaScript для включения и отключения Session Replay можно использовать без отображения баннера. Например, для записи сессии при каждом входе пользователя используйте [`dtrum.enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay) для начала и [`dtrum.disableSessionReplay()`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#disablesessionreplay) после успешного выхода для остановки. Это обеспечивает полный контроль над началом и завершением Session Replay.

### Включение режима opt-in Session Replay

Режим opt-in Session Replay по умолчанию отключён.

Для включения режима opt-in Session Replay:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **General settings** > **Data privacy** > **Session Replay**.
5. Включите **Enable opt-in mode for Session Replay**.

При этих настройках Session Replay неактивен в браузерах конечных пользователей, и сессии не записываются до вызова метода [`dtrum.enableSessionReplay(ignoreCostControl: boolean)`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enablesessionreplay) из приложения.

Если вы не включаете **Enable opt-in mode for Session Replay**, все сессии пользователей записываются с самого начала до вызова [`dtrum.disableSessionReplay()`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#disablesessionreplay).

## Исключение URL

Исключение URL применяется к страницам и представлениям. Для исключения страницы из записи Session Replay определите регулярное выражение для сопоставления с URL конкретной страницы. Можно настроить правила для отдельных веб-страниц, целых сайтов и одностраничных приложений.

* Если URL соответствует правилу исключения, это правило применяется, а все последующие игнорируются.
* Если URL не соответствует ни одному правилу, все подходящие страницы записываются Session Replay.
* Если Session Replay включён и правила исключения URL не определены, все страницы записываются по умолчанию.

![Session Replay - исключение URL](https://dt-cdn.net/images/sr-url-exclusion-700-395161c0b1.png)

Session Replay - исключение URL

Для исключения страниц из записи:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **General settings** > **Data privacy** > **Session Replay**.
5. В разделе **URL exclusion** выберите **Add exclusion rule**.

   ![Исключение URL](https://dt-cdn.net/images/url-exclusion-1343-e10f29c49d.png)

   Исключение URL
6. Введите регулярное выражение для сопоставления с URL страницы и выберите **Add rule**.

## Маскирование

Session Replay записывает каждое взаимодействие пользователя. Поэтому маскирование конфиденциальных данных крайне важно. Настройки маскирования позволяют защитить конфиденциальные данные при записи и воспроизведении сессий. Вы можете задавать отдельные правила маскирования для записи и воспроизведения, позволяя применять уровни маскирования, контролируемые [разрешениями пользователей](#sr-permissions).

Session Replay реализует функциональность маскирования, обеспечивая либо отсутствие захвата личных данных в момент записи, либо их маскирование в момент воспроизведения.

Опция маскирования маскирует только буквенно-цифровые символы; символы форматирования (точки, запятые, двоеточия) не маскируются. Таким образом, при воспроизведении сессий можно проверять формат данных, не видя фактической информации.

Например, рассмотрим поле email-адреса в обычной веб-форме. Пользователь вводит свой адрес:

![Поле email с содержимым](https://dt-cdn.net/images/2018-10-25-12-34-34-312-26653a5668.png)

Поле email с содержимым

Session Replay маскирует эти данные и отображает звёздочки вместо нечисловых символов:

![Поле email с маскированным содержимым](https://dt-cdn.net/images/2018-10-25-12-40-06-317-b9aa2a3f8c.png)

Поле email с маскированным содержимым

Маскированные данные — отображаемые при воспроизведении как звёздочки (`*****`) для нечислового ввода или нули (`0000`) для числового — либо никогда не покидают клиентский браузер (маскирование при захвате), либо захватываются, но маскируются при воспроизведении. Воспроизведение сессий [контролируется разрешениями](#sr-permissions).

Session Replay предоставляет два варианта настройки маскирования контента.

* [Атрибут `data-dtrum-mask`](#mask-data-via-code)
* [Страница конфигурации Session Replay](#mask-data-via-ui)

### Маскирование с атрибутом `data-dtrum-mask`

Атрибут `data-dtrum-mask` требует изменения кода приложения и безопасен по дизайну. Он позволяет учитывать элементы с конфиденциальной информацией на этапах проектирования и реализации. Рекордер автоматически определяет и маскирует контент (текст, значения ввода и атрибутов) и взаимодействия (движения курсора и прокрутки) в узле с этим атрибутом и его потомках.

Код приложения должен быть изменён для включения атрибута `data-dtrum-mask`.

![Атрибут data-dtrum-mask в коде](https://dt-cdn.net/images/data-dtrum-mask-example-655-045d2f8d23.png)

Атрибут data-dtrum-mask в коде

### Маскирование через интерфейс

Страница настройки Session Replay в Dynatrace предоставляет более гибкий подход. Вы можете изменить конфигурацию в соответствии с требованиями к записи сессий. При этом нет необходимости изменять код приложения.

Эта страница настроек также предоставляет опции маскирования для скрытия взаимодействий с определёнными элементами, которые могут непреднамеренно раскрывать конфиденциальные данные. Например, список с несколькими вариантами ответа на вопрос о вероисповедании или поле пола пользователя. Даже с маскированным текстом можно определить ответ пользователя, увидев выбранный вариант.

Для настройки маскирования Session Replay:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **General settings** > **Data privacy** > **Session Replay**.
5. В разделе **Content masking preferences** выберите [предопределённую опцию маскирования](#masking-preset-options) для записи и воспроизведения.
6. Если вы выбрали **Allow list** или **Block list**, добавьте нужные правила маскирования.

#### Уровни маскирования контента

Предопределённые [опции маскирования](#masking-preset-options) Session Replay доступны как для записи, так и для воспроизведения:

* **Настройки маскирования для записи** управляют маскированием данных в момент записи («маскирование при захвате»). Маскированные данные никогда не покидают клиентский браузер и не захватываются. При установке более строгого уровня **Recording masking settings** те же настройки применяются и к **Playback masking settings**, что также затрагивает все ранее записанные сессии.
* **Настройки маскирования для воспроизведения** влияют на маскирование данных в момент воспроизведения («маскирование при отображении»). Данные, захваченные при записи, могут быть маскированы при воспроизведении.

Вы можете определять правила маскирования как для записи, так и для воспроизведения.

Правила маскирования воспроизведения предназначены для дополнительного уровня маскирования поверх правил записи. Настройки маскирования воспроизведения не могут быть менее строгими, чем настройки записи.

Вы можете использовать [разрешения пользователей](#sr-permissions) для управления возможностью воспроизведения сессий с применением правил маскирования или без них.

#### Опции маскирования контента

Следующие предопределённые опции маскирования используются для ограничения захвата и воспроизведения личных и конфиденциальных данных:

| Опция маскирования | Что маскируется | Когда использовать |
| --- | --- | --- |
| **Mask all** | Весь текст, пользовательский ввод, значения атрибутов и изображения | Используйте для тестирования Session Replay с гарантией отсутствия сбора конфиденциальных данных. Вы по-прежнему сможете видеть взаимодействия пользователей. Используйте для устранения неполадок, когда важен порядок взаимодействий пользователя с элементами управления. |
| **Mask user input** | Весь пользовательский ввод, включая варианты в выпадающих списках | Выберите эту опцию, если конфиденциальные данные поступают только от ввода пользователя. |
| **Allow list** | Все элементы из опции **Mask all**, кроме указанных вами | Рекомендуется для большинства приложений: позволяет собирать только необходимые данные. Гарантирует, что даже при последующих изменениях кода новые элементы с конфиденциальной информацией не будут записаны. Элементы определяются CSS-селектором. |
| **Block list** | Только элементы, указанные в этом списке блокировки | При выборе этой опции отображается список всех правил из **Mask all**. Используйте его для разблокировки элементов и атрибутов, которые хотите захватывать. Вы также можете создавать собственные дополнительные правила. |

**Mask user input** является опцией маскирования по умолчанию начиная с Dynatrace версии 1.262. Ранее по умолчанию использовалась опция **Mask all**.

Опции **Mask all**, **Mask user input** и **Allow list** не скрывают взаимодействия пользователей с элементами. С опцией **Block list** можно настроить скрытие взаимодействий с маскированными элементами.

## Захват ресурсов

С Dynatrace Session Replay стили (CSS) захватываются и сохраняются во время записи сессии. Нет необходимости хранить CSS-файлы в публично доступном месте, помечать ресурсы как изменённые или предоставлять разрешения процессу, который может получить доступ к защищённым стилям. Это полностью автоматизированный процесс, обеспечивающий захват ресурсов и их доступность для будущего воспроизведения.

Захват ресурсов для Session Replay включён по умолчанию.

### Включение захвата ресурсов

Для включения и настройки захвата ресурсов:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **Resource capture for Session Replay**.
5. Включите **Enable resource capture**.
6. Необязательно. Для исключения ресурсов на определённых страницах выберите **Add exclusion rule**, введите регулярное выражение и нажмите **Add rule**.

   Правила исключения обрабатываются следующим образом:

   * Если URL соответствует правилу исключения, правило применяется, а последующие игнорируются.
   * Если URL не соответствует ни одному правилу, все ресурсы захватываются.
   * Если Session Replay включён и правила исключения URL не определены, все CSS-ресурсы доступны для записи.

### Примечания и ограничения для захвата ресурсов

* Если ресурс не был захвачен или недоступен в хранилище ресурсов Dynatrace, он извлекается из исходного источника.
* Изображения и шрифты не захватываются. При воспроизведении они извлекаются из исходного местоположения, поэтому применяются определённые [ограничения](/managed/observe/digital-experience/session-replay/session-replay-restrictions-web#resources "Узнайте об ограничениях, применяемых к Session Replay."). Для улучшения воспроизведения Session Replay используйте **Session Replay browser extension**.
* Dynatrace не захватывает ресурсы размером более 3 МБ.
* Стандартный [срок хранения](/managed/manage/data-privacy-and-security/data-privacy/data-retention-periods "Проверьте сроки хранения данных различных типов.") захваченных ресурсов составляет 35 дней.
* Dynatrace захватывает ресурсы для не более чем 0,1% сессий, записанных Session Replay. Обычно этого достаточно для корректного воспроизведения, поскольку ресурс, захваченный для одной сессии, используется повторно для других. Однако захват всех ресурсов для приложения с низким трафиком может занять некоторое время.

  Расчёт сессий с захваченными ресурсами

  **Сессий с захваченными ресурсами = Всего сессий × Управление стоимостью RUM × Управление стоимостью SR × 0,1%**

  | Параметр | Пояснение | Пример |
  | --- | --- | --- |
  | Всего сессий | Общее число сессий пользователей | 1 000 000 |
  | [Управление стоимостью и трафиком RUM](/managed/observe/digital-experience/web-applications/additional-configuration/configure-cost-and-traffic-control-web "Используйте управление стоимостью и трафиком в Dynatrace для снижения потребления сессий.") | Процент сессий, захватываемых и анализируемых с RUM | 50% |
  | [Управление стоимостью и трафиком Session Replay](#cost-traffic-control) | Процент сессий, записываемых Session Replay | 10% |

  В нашем примере ресурсы захватываются для 50 сессий (1 000 000 × 50% × 10% × 0,1% = 50).

## Изменение Content Security Policy для Session Replay

Session Replay использует web workers для обработки данных во время записи, разгружая логику из UI-потока для повышения производительности. RUM JavaScript достигает этого путём загрузки кода как blob. Для этого необходимо добавить `blob:` в директиву `worker-src` правил CSP, в дополнение к адаптациям, описанным в [Изменение Content Security Policy для RUM](/managed/observe/digital-experience/web-applications/additional-configuration/modify-csp-for-rum "Узнайте, как включить и изменить CSP для приложений, мониторируемых с RUM."). Правила CSP можно определить через HTTP-заголовок `Content-Security-Policy` или с помощью тега `<meta>` в разделе `<head>` HTML-документа.

Если `blob:` не разрешён в `worker-src` (или в качестве запасного варианта в `script-src`), при загрузке страницы произойдёт нарушение CSP, и весь код будет выполняться в UI-потоке.

Если вы не можете запускать web workers на сайте или хотите отключить их использование Session Replay, добавьте пользовательское свойство конфигурации.

Для отключения использования web workers Session Replay:

1. Перейдите в **Web**.
2. Выберите приложение, которое хотите настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**...**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **Custom configuration properties**.
5. Выберите **Add a custom configuration property**.
6. Введите свойство `srbw=0` и сохраните изменения.

Не рекомендуется запускать код Session Replay в UI-потоке вместо web workers, поскольку это снизит производительность страницы.

## Часто задаваемые вопросы

Session Replay потребляет много места?

Примерно 100 КБ данных генерируется за каждую минуту сессии, и около 500 КБ хранилища потребляет полностью записанная сессия.

Все ли сессии занимают одинаковый объём?

Нет. Объём сессии зависит от множества факторов: размера приложения, длительности сессии и активности пользователя.

Размеры сессий варьируются от 100 КБ до более чем 1 МБ.

Сессии записываются как видео?

Нет. Для записи сессий Session Replay отслеживает изменения в DOM-дереве веб-приложения. Каждое визуальное изменение соответствует изменению в DOM. Session Replay захватывает и воспроизводит эти изменения. Поскольку все данные текстовые, небольшие размеры достигаются благодаря компактному представлению и сжатию.

Какое потребление полосы пропускания?

Поскольку сжатие выполняется на стороне клиента, потребление полосы пропускания совпадает с потреблением хранилища: около 100 КБ в минуту или 500 КБ за сессию.

Это выгодно для записи сессий пользователей на мобильных устройствах с ограниченным мобильным трафиком.

Влияет ли сжатие на браузер клиента?

Session Replay разработан с минимальным воздействием на UI-поток. Большинство задач, включая эффективный алгоритм сжатия, выполняются рабочим потоком в фоне без вмешательства в пользовательский интерфейс.

Применяются ли правила маскирования на сервере Dynatrace?

[Правила маскирования для записи](/managed/observe/digital-experience/session-replay/configure-session-replay-web#masking-preset-options "Настройте потребление мониторинга и параметры конфиденциальности данных для Session Replay.") передаются на клиент, обеспечивая по умолчанию отсутствие отправки конфиденциальных данных за пределы браузера. [Настройки маскирования](/managed/observe/digital-experience/session-replay/configure-session-replay-web#sr-masking "Настройте потребление мониторинга и параметры конфиденциальности данных для Session Replay.") для записи и воспроизведения настраиваются.

Что делает расширение браузера Session Replay и как его установить?

Расширение браузера Session Replay позволяет проксировать загрузку ресурсов при воспроизведении сессии. Его можно использовать для улучшения воспроизведения, например в следующих случаях:

* Ресурсы шрифтов и стилей загружаются с HTTPS-источников.
* Ресурсы загружаются с HTTP-источников.

Расширение можно установить из Chrome Web Store: [`Session Replay browser extension`](https://chromewebstore.google.com/detail/session-replay-browser-ex/hjbdnbhpfiionafiooklnafmaojjfljh)

Каковы рекомендации по защите конфиденциальности пользователей?

Конфиденциальность должна быть встроена в приложение с самого начала его создания. Любая функция, связанная с обработкой личных данных, должна оцениваться на предмет рисков конфиденциальности на ранних этапах. Dynatrace рекомендует помечать конфиденциальные элементы и поля ввода. Например, при обнаружении HTML-атрибута [`data-dtrum-mask`](#mask-data-via-code) данные автоматически маскируются.

Рекомендуется начинать с опции **Mask all** и постепенно корректировать настройки, если вы считаете, что часть заблокированных элементов можно безопасно разрешить.

При наличии конфиденциальных данных в записях используйте более строгие правила маскирования воспроизведения для ограничения отображаемой информации. Используйте [разрешения пользователей](#sr-permissions) для управления возможностью воспроизведения сессий с применением правил маскирования или без них.

Что происходит при некорректной настройке правил маскирования и доступности конфиденциальных данных при Session Replay?

После случайного отображения конфиденциальных данных, как только правила маскирования будут настроены корректно, Session Replay применяет последнюю конфигурацию ко всем записанным сессиям, включая записанные до применения правил. Обновлённые правила применяются в момент воспроизведения, и [пользователи с правом воспроизведения](#sr-permissions) не смогут просматривать маскированные данные. Правила маскирования воспроизведения игнорируются для пользователей с правом воспроизведения без маскирования.

Как настроить приложение без маскирования элементов?

Используйте подход **Block list** и удалите все предопределённые правила.

Когда лучше всего использовать Mask all?

**Mask all** — лучшая опция для тестирования Session Replay без риска раскрытия конфиденциальных данных. Также удобна при использовании Session Replay для устранения неполадок, когда важен порядок взаимодействий.

Почему данные Session Replay отображаются частично при воспроизведении?

На странице деталей сессии временная шкала **Session Replay** показывает события как [Real User Monitoring (RUM)](/managed/observe/digital-experience/rum-concepts/rum-overview "Узнайте о мониторинге реальных пользователей, ключевых метриках производительности и многом другом."), так и Session Replay.

При воспроизведении сессии вы можете заметить, что, несмотря на отображение всех действий и событий на временной шкале, проигрыватель пропускает часть событий и переходит к концу. Это обычно означает отсутствие данных Session Replay для этой части сессии.

Наиболее распространённые причины частичного отсутствия данных:

* Сессия охватывала второе приложение, для которого Session Replay был отключён.
* Запись Session Replay была приостановлена, поскольку страница [исключена из записи](#sr-url-exclusion).
* Данные Session Replay доступны, но у вас [нет разрешения на доступ к ним](#no-permission).
