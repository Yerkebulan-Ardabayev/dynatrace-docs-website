# Игнорирование ошибок веб-запросов для пользовательских приложений в RUM Classic

* Практическое руководство
* Чтение за 1 минуту
* Опубликовано 30 января 2023 г.

По умолчанию Dynatrace помечает все коды состояния ответа `4xx` и `5xx` как [ошибки](/managed/observe/digital-experience/rum-classic/rum-concepts/user-and-error-events#error "Узнайте о событиях пользователей и ошибок, а также о типах событий пользователей и ошибок, фиксируемых Dynatrace.") веб-запросов. Однако возможно, что некоторые неуспешные HTTP-коды на самом деле не указывают на проблему с приложением. Например, иногда код состояния `401 Unauthorized` просто показывает, что пользователь решил не входить в систему. Другой пример, код `429 Too Many Requests`, который может указывать на проблемы с ограничением скорости запросов, не вызванные приложением.

Чтобы игнорировать определённые HTTP-коды состояния, нужно настроить, какие коды состояния не должны помечаться как ошибки. При создании правила исключения можно добавлять как отдельные HTTP-коды состояния, например `401`, так и диапазоны кодов состояния, например `400-499`.

Создание правила исключения

1. Перейти в **Frontend**.
2. Выбрать приложение, которое нужно настроить.
3. Выбрать **More** (**…**) > **Edit** в верхнем правом углу плитки с именем приложения.
4. В настройках приложения выбрать **Request errors**.
5. Выбрать **Add exclusion rule** и указать HTTP-код состояния или диапазон кодов, которые должны считаться успешными.

   ![Настройка правила исключения](https://dt-cdn.net/images/mobile-exclusion-rules-2124-75133d34c8.png)

   Настройка правила исключения

После настройки правила исключения веб-запросы с ранее ошибочными кодами состояния будут учитываться как успешные веб-запросы при расчёте частоты ошибок. Это повлияет на рейтинг Apdex приложения, частоту ошибок запросов и список ошибок веб-запросов, а также на некоторые пользовательские действия и пользовательские метрики.

Например, на скриншотах страницы обзора приложения ниже видно снижение частоты ошибок веб-запросов. Это произошло после того, как для приложения было настроено правило игнорирования кода состояния ответа `401 Unauthorized`. Также заметно снижение частоты ошибок для основных провайдеров приложения, а также более высокий рейтинг Apdex и более низкая частота ошибок для одного из пользовательских действий.
