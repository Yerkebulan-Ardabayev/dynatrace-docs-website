# Использование метрик Visually complete и Speed index в RUM Classic

* Практическое руководство
* 9 минут на чтение
* Опубликовано 03 июля 2018 г.

Чтобы дать больше видимости пользовательского опыта в приложении, [представлена новая, улучшенная версия алгоритма Visually complete﻿](https://www.dynatrace.com/news/blog/more-visibility-into-user-experience-with-new-web-performance-metrics-and-enhanced-visually-complete/), которая даёт больше контроля над расчётами Visually complete. В сочетании с дополнительными метриками производительности улучшенная версия Visually complete даёт точные, стабильные измерения и больше контроля над тем, что включается в расчёты Visually complete. Чтобы перейти на новую версию глобально во всей среде, нужно открыть **Web** и выбрать любое приложение из списка. Затем прокрутить вниз до **Enable Visually complete 2.0 and new web performance metrics** и выбрать **Enable for all applications**.

Поддержка более ранней версии Visually complete закончится с версией RUM JavaScript 1.211 для RUM и версией Synthetic 1.212.

Visually complete, это точечная во времени метрика, которая измеряет момент, когда видимая область страницы завершила загрузку. Метрики Visually complete обычно короче по продолжительности, чем сопоставимые метрики (например, время загрузки страницы и измерения DOM interactive), потому что пользователи воспринимают страницу как полностью загруженную ещё до того, как загрузятся 100% фоновых элементов страницы. Оптимизация времени visually complete ценнее оптимизации других показателей времени загрузки страницы с точки зрения пользовательского опыта, так как она отражает время, которое реальные пользователи тратят на ожидание полной загрузки контента над сгибом страницы (above-the-fold). Результаты можно сортировать по расположению, устройству, операционной системе и типу браузера.

Visually complete означает, что все элементы в видимой области веб-страницы загружены на 100%:

* Элементы
* Изображения
* IFrames (только изображения)

Видимая область простирается от верха страницы до низа окна браузера (прокрутка страницы пользователем не влияет на область страницы, используемую для измерения).

Visually complete даёт следующие преимущества:

* Позволяет увидеть, сколько времени точно требуется конечным пользователям, чтобы увидеть нужную им информацию.
* Даёт настоящую бизнес-значимую метрику с точки зрения пользователя.
* Даёт 100% видимость реального пользовательского опыта независимо от устройства.
* Согласует ИТ и бизнес, связывая пользовательский опыт с бизнес-результатами.
* Ускоряет улучшение производительности.
* Оптимизирует решения в области разработки, эксплуатации и пользовательского опыта.
* Может комбинироваться с мощными водопадными диаграммами (waterfall charts) для понимания и улучшения времени рендеринга страницы.

Наряду со Speed index, Visually complete показывает общее время, затраченное на загрузку всей страницы. Поэтому эти метрики собираются только для основных фреймов (main frames), а не для подфреймов. В случаях, когда фреймы кросс-доменные и нет возможности определить родительский фрейм, сбор Speed index и Visually complete предпринимается и для подфреймов.

## Расчёт Visually complete

#### Расчёт Visually complete для load-действий

Visually complete начинает наблюдать за содержимым DOM сразу после внедрения RUM JavaScript.

Наблюдение прекращается, когда срабатывает событие Load event end и с момента добавления последнего элемента страницы прошло больше секунды. Если на странице происходят повторяющиеся изменения, Visually complete автоматически завершается по таймауту через 3 секунды после срабатывания Load event end. Для расчёта используются все элементы, являющиеся частью страницы на момент, когда Visually complete прекращает наблюдение за страницей.

Visually complete проверяет, видимы ли добавленные на страницу элементы или находятся ли они в видимой области. Если да, то для Visually complete используется время появления на странице последнего видимого элемента. Для внешних ресурсов, таких как изображения, Visually complete использует resource timings API браузера для получения времени загрузки, а не время размещения тега `<img>` на странице.

#### Расчёт Visually complete для XHR-действий

Для XHR-действий наблюдение за содержимым DOM начинается с `callback begin` XHR-действия и заканчивается через 50 миллисекунд после `callback end`. Любые мутации с внешними ресурсами, такими как изображения, таблицы стилей или IFrames, приводят к тому, что RUM JavaScript добавляет слушатель загрузки (load listener) к этим ресурсам. Это продлевает время XHR-действия до момента добавления последнего ресурса в пределах наблюдаемого временного интервала.

Затем для этих ресурсов рассчитывается Visually complete. Значение Visually complete может отличаться от времени окончания действия. Но когда Visually complete установлен в `-1` (значение по умолчанию), например, когда в DOM не добавлено видимых ресурсов, для Visually complete используется время окончания действия.

Причина, по которой для XHR-действий поддерживаются только ресурсы, а не все элементы, в том, что при наличии параллельных XHR-действий и нескольких одновременных наблюдений элементы, появляющиеся на странице, нельзя сопоставить с нужным действием.

## Расчёт Speed index

Speed index полезен для сравнения пользовательского опыта различных страниц. Чем меньше число, тем лучше пользовательский опыт страницы. Подробное определение Speed index и базовое описание расчёта Speed index приведены на [странице документации WebPagetest﻿](https://dt-url.net/gl03s4g).

Решение в составе RUM JavaScript использует тот же алгоритм, что и WebPageTest, но использует видимые элементы, зафиксированные в рамках visually complete. Это «реальные» элементы страницы. Преимущество перед сравнением скриншотов (webpagetest.org) в том, что Visually complete учитывает элементы страницы и их тайминги. Сравнение скриншотов отражает только визуальные изменения страницы. Так, например, если бы на странице был вращающийся GIF, который непрерывно меняется, время Visually complete и Speed index на webpagetest.org было бы неточным.

Speed index рассчитывает индекс, измеряя прогресс загрузки веб-страницы между определёнными маркерами, вместе с процентом элементов, которые остаются незагруженными в данный момент времени.

* (200 мс × 100%) (до 200 мс ничего не загружено)
* (200 мс × 60%) (к первому маркеру уже загружено 40%; то есть до 100% нужно загрузить ещё 60%)
* (200 мс × 40%) (до 100% нужно загрузить ещё 40%)
* Speed index = (200 мс × 100%) + (200 мс × 60%) + (200 мс × 40%) = 400

Speed index применим только к load-действиям, потому что при расчёте индекса он использует в качестве ориентира всю страницу. Например, если на странице есть маленький значок, загрузка которого занимает пять секунд, а остальная часть страницы загружается за две секунды, то Visually complete составит пять секунд, потому что значок, это последний элемент на странице. Но Speed index будет около двух секунд, потому что основная часть страницы загрузилась за две секунды.

## Настройки конфигурации для улучшенного Visually complete

В версиях Dynatrace 1.194 и более поздних новый алгоритм Visually complete даёт больше контроля над тем, как рассчитывается метрика, позволяя задавать следующие настройки:

* **Threshold**: эта настройка определяет минимальную видимую область на элемент (в пикселях), при которой элемент учитывается в Visually complete и Speed index. Это означает, что в расчёты будут включены только элементы, у которых площадь в пикселях не меньше заданной минимальной. Для определения порога используется свойство `vct`.
* **Inactivity timeout for load actions**: время, в течение которого модуль Visually complete ожидает бездействия и отсутствия дальнейших мутаций на странице после load-действия. Для определения таймаута бездействия используется свойство `vcit`.
* **Mutation timeout for XHR and custom actions**: время, в течение которого модуль Visually complete ожидает после завершения XHR- или пользовательского действия, прежде чем начать расчёт. Для определения таймаута мутации используется свойство `vcx`.
* **Excluded URL list**: с помощью регулярных выражений определяются URL изображений и iFrame, которые нужно исключить из обнаружения модулем Visually complete. Для создания списка используется свойство `iub`.
* **Ignored mutation list**: CSS-селекторы запросов, задающие узлы мутаций (изменяющиеся элементы), которые нужно игнорировать при расчёте Visually complete и Speed index. Для создания списка используется свойство `mb`.

Для настройки этих параметров

1. Открыть **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Capturing** > **Content capture**.
5. Добавить нужную настройку.

   * `Threshold`:  
     использовать значение от 0 до 10000; значение по умолчанию, 50.
   * `Inactivity timeout for load actions`:  
     время в миллисекундах, использовать значение от 0 до 30000; значение по умолчанию, 1000.
   * `Mutation timeout for XHR and custom actions`:  
     время в миллисекундах, использовать значение от 0 до 5000, значение по умолчанию, 50.
   * `Excluded URL list`:  
     в качестве разделителя между записями использовать символ вертикальной черты, например, `\.dynatrace\.com\/login|\.dynatrace\.com\/logout`. Разделение между разными протоколами URL не поддерживается; исключается каждый протокол URL.
   * `Ignored mutation list`:  
     в качестве разделителя между записями использовать запятую, например, `#someDomElement@someattribute,.someOtherDomElement`.

### Добавление Visually complete и других метрик на дашборд

Метрика Visually complete и другие расчётные метрики для RUM можно использовать для создания дашбордов по метрикам, которые представляют наибольший интерес. Прежде чем добавлять метрику на дашборд, нужно [создать метрику](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/rum-calculated-metrics-web "Create calculated metrics as well as custom charts based on calculated metrics for your web applications."). Помимо хорошо известной метрики Visually complete, здесь также добавлены две из трёх метрик "web vitals": **First input delay (FID)** и **Largest contentful paint (LCP)**.

## Использование Visually complete в качестве ключевой метрики производительности

В Dynatrace версий 1.193 и более ранних Visually complete, это ключевая [метрика производительности](/managed/observe/digital-experience/rum-classic/web-applications/analyze-and-use/work-with-key-performance-metrics "Learn how to use the right key performance metrics to optimize user experience data for each of your applications."), применяемая по умолчанию для оценки производительности всех load actions и XHR actions. Метрики Visually complete и Speed index захватываются по умолчанию для всех приложений с включённым Real User Monitoring.

Можно изменить ключевую метрику производительности, используемую для load actions и XHR actions. Также можно отключить захват метрик Visually complete и Speed index, однако повторное включение захвата сбрасывает референсные периоды обнаружения аномалий, заданные для приложения.

Чтобы задать ключевую метрику производительности для приложения

1. Перейти в **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. Выбрать **General settings** > **Load actions** / **XHR actions** / **Custom actions**.
5. В разделе **Key performance metric** выбрать ключевую метрику производительности из списка для каждого типа пользовательских действий.

Чтобы отключить захват метрик Visually complete и Speed index

1. Перейти в **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Capturing** > **Content capture**.
5. Отключить настройку **Visually complete & Speed index**.

### Использование Visually complete с single-page приложениями

Для фреймворков single-page приложений, таких как Angular, Visually complete даёт ещё более точные тайминги для пользовательских действий. При использовании Visually complete тайминги не останавливаются при получении ответов XHR. Благодаря этому можно получить дополнительную информацию о динамической загрузке ресурсов, вызванной изменениями элементов DOM на основе JavaScript в ответ на XHR-вызов.

### Сопоставление производительности пользователей с бизнес-метриками

Производительность, это один из основных факторов, влияющих на пользовательский опыт и успешные бизнес-результаты. Метрика Visually complete предоставляет такой анализ из коробки.

### Visually complete для анализа пользовательских действий

Visually complete и Speed index захватываются для каждого шага пользовательского действия.

1. Перейти в **Web**.
2. Выбрать приложение для анализа.
3. В разделе **Performance analysis** выбрать раздел actions, чтобы отобразить данные пользовательских действий.
4. На плитке **Top 3 user actions** выбрать **View full details**.
5. В разделе **Key user actions** или **Top 100 user actions** выбрать действие, чтобы увидеть тайминги его вклада и таймлайн действия для шага на плитке **Contributors breakdown**. Таймлайн действия показывает тайминги для всех соответствующих стадий действия, включая **Speed index** и **Visually complete**. Метрика производительности, используемая для шага, отображается над таймлайном, поэтому можно быстро определить, что используется в качестве метрики производительности: "User action duration" или "Visually complete".

### Устранение неполадок Visually complete и Speed index

При использовании в качестве ключевой метрики производительности:

* Visually complete вычисляется при изменении визуального элемента на странице, и это изменение распространяется с каждым последующим пользовательским действием load и XHR (но не с асинхронными веб-запросами, которые не влияют на DOM).
* Speed index доступен только для load actions.

Если точные данные Visually complete или Speed index не отображаются, проверьте одну из следующих страниц в сообществе Dynatrace.

* [Web applications: Visually complete and Speed index metrics not shown﻿](https://dt-url.net/pw438xh)
* [Web applications: Inaccurate Visually complete and Speed index metrics﻿](https://dt-url.net/jd638ur)
