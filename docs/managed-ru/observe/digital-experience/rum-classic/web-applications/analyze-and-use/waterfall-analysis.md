# Waterfall-анализ в RUM Classic

# Waterfall-анализ в RUM Classic

* Руководство
* Чтение: 7 минут
* Обновлено 24 января 2023 г.

Dynatrace фиксирует данные о пользовательском опыте и производительности, отслеживая отдельные действия пользователя. Обычно действие пользователя начинается с клика по элементу управления HTML (например, по кнопке или ссылке). Затем браузер загружает запрошенные данные, либо переходя на новую страницу, либо через XHR/fetch-вызов. После этого выполняются JavaScript-колбэки, строится или изменяется дерево DOM, и веб-приложение снова готово к новому действию пользователя. Представление Dynatrace **Waterfall analysis** идеально подходит для детального анализа всех подобных данных мониторинга действий пользователя.

Чтобы открыть waterfall-анализ для действия пользователя

1. Перейти в **Web**.
2. Выбрать приложение, в котором есть нужное для анализа действие пользователя.
3. Прокрутить страницу приложения вниз до раздела **Top 3 user actions**.
4. Выбрать действие пользователя (или выбрать **View full details**, чтобы получить доступ к дополнительным действиям пользователя).
5. На странице **User action** выбрать **View analysis in waterfall chart**, чтобы открыть **Waterfall analysis**.

## Данные, отображаемые в waterfall-представлении

Представление waterfall-анализа построено главным образом на основе W3C [timings](/managed/observe/digital-experience/rum-classic/rum-concepts/user-action-metrics "Узнать, какие метрики вычисляет Dynatrace для действий пользователя, и что означает каждая метрика."). Записи waterfall организованы в три раздела:

**Document requests**, отображает весь контент, идентифицированный как загрузка страницы для данного действия пользователя. Сюда входит исходный HTML, а также любой другой контент, отображаемый во фреймах. Каждая запись содержит детали, зафиксированные W3C navigation timings.

**XHR requests**, отображает все ресурсы, отправленные через XHR или fetch(). Эти записи содержат все детали, доступные через W3C resource timings, а также время выполнения JavaScript-колбэка.

**Resources**, отображает все детали W3C timing для всех ресурсов.

Действие пользователя может содержать более одной загрузки страницы или XHR-запроса. Поэтому исходный запрос выделен фиолетовым цветом. [Timings](/managed/observe/digital-experience/rum-classic/rum-concepts/user-action-metrics "Узнать, какие метрики вычисляет Dynatrace для действий пользователя, и что означает каждая метрика.") исходного запроса отображаются в виде вертикальных линий событий над диаграммой.

## Top findings

[Top findings](/managed/observe/digital-experience/rum-classic/web-applications/analyze-and-use/visually-complete-top-findings "Узнать, как использовать Visually complete top findings, предоставляемые в waterfall-анализе.") для waterfall помогают понять пользовательский опыт реальных пользователей в верхней части экрана (above-the-fold). Первоначальные загрузки страниц формируют первое впечатление новых посетителей веб-приложения. Кроме того, top findings дают простой способ понять, какие веб-запросы влияют на [ключевые метрики производительности](/managed/observe/digital-experience/rum-classic/rum-concepts/user-action-metrics#kpm "Узнать, какие метрики вычисляет Dynatrace для действий пользователя, и что означает каждая метрика."), и, следовательно, являются потенциальными кандидатами для оптимизации.

Для каждого действия пользователя, просматриваемого в **Waterfall analysis**, автоматически определяются и представляются следующие top findings (пример ниже):

* Несжатые текстовые ресурсы
* Ресурсы размером более 100 кБ
* Ресурсы, у которых доля кэширования браузером менее 50%[1](#fn-1-1-def)
* Медленные (> 200 мс) ресурсы первой стороны, третьих сторон или CDN
* Домены третьих сторон и CDN, к которым были обращения
* Ресурсы первой стороны, третьих сторон и CDN

1

Чтобы определить, был ли ресурс, вероятно, взят из кэша, используется следующий индикатор, описанный в [спецификации W3C﻿](https://www.w3.org/TR/resource-timing-2/): "It is possible for transferSize value to be lower than encodedBodySize: when a cached response is successfully revalidated the transferSize reports the size of the response HTTP headers incurred during the revalidation, and encodedBodySize reports the size of the previously retrieved payload body."

В примере ниже обнаружен несжатый текстовый ресурс. При выборе этой плитки finding соответствующий ресурс автоматически выделяется ниже в списке **Resources**.

## Агрегированное представление

Инструменты разработчика браузера позволяют смотреть только отдельные выполнения, а агрегированный waterfall Dynatrace даёт возможность получить сводное представление о нескольких выполнениях одного и того же действия пользователя различными реальными пользователями. Эти агрегации вычисляются на основе выполнений действий пользователя в рамках анализируемого периода времени и распределяются по трём агрегированным waterfall-диаграммам. Можно выбирать между представлениями агрегированного waterfall, отображающими самые медленные 10%, самые быстрые 10% и +-5% от медианы. Это помогает выявлять запросы и паттерны, связанные с высокой или низкой производительностью.

### Редко встречающиеся ресурсы

Чтобы представление агрегированного **Waterfall analysis** оставалось разумного и удобочитаемого размера, ресурсы, которые встречаются менее чем в 30% экземпляров действий пользователя, перемещаются в группу ресурсов **Rarely occurring**. Эту группу можно развернуть, чтобы увидеть редко встречающиеся ресурсы. Обратить внимание на все детали, возвращаемые в панели подсказки при наведении.

## Представление отдельного экземпляра

Автоматические findings отлично подходят для начала работы с waterfall-анализом действий пользователя и для поиска способов улучшения пользовательского опыта приложения, но иногда нужно глубже погрузиться в данные о производительности действий пользователя.

Агрегированная информация полезна для получения общего представления об этапах обработки действий пользователя и использовании ресурсов, но иногда необходимо проанализировать отдельные экземпляры конкретных действий пользователя, чтобы понять влияние определённых браузеров, геолокаций или других переменных. Поэтому есть возможность просматривать waterfall-анализ отдельных экземпляров действий пользователя. Отдельные экземпляры действий пользователя перечислены на вкладке **Instances of this action**.

### Сквозная видимость и переход к distributed traces

Top finding под названием **Traces** доступен, когда представление **Waterfall analysis** используется для анализа производительности отдельного экземпляра действия пользователя, для которого доступны [distributed traces](/managed/observe/application-observability/distributed-traces "Получить видимость в высокораспределённых, cloud-native архитектурах для эффективной трассировки и анализа транзакций в реальном времени."). В примере ниже показано, что для анализа доступны `52 Traces`, связанные с выбранным экземпляром действия пользователя. Выбрать плитку **Traces**, чтобы получить ссылку на список связанных distributed traces. Top findings также выделяют в диаграмме waterfall-анализа все записи ресурсов, для которых доступны distributed traces.

![Waterfall analysis](https://dt-cdn.net/images/waterfall-1-2000-30253b2468.png)

Waterfall analysis

Обратить внимание на следующие ограничения:

* Для сквозной видимости требуется работающий на стороне сервера экземпляр OneAgent. Top findings видны только тогда, когда distributed traces действительно доступны.
* Для процессов, работающих под высокой нагрузкой, Dynatrace OneAgent автоматически регулирует объём данных, отправляемых на Dynatrace Server. Поэтому некоторые distributed traces могут не захватываться из-за [Adaptive Traffic Management](/managed/ingest-from/dynatrace-oneagent/adaptive-traffic-management/adaptive-traffic-management-managed "Улучшить работоспособность и производительность среды Dynatrace Managed с помощью адаптивных функций управления трафиком, снижения нагрузки и контроля захвата.").
* Distributed traces для кросс-доменных (cross-origin) XHR-вызовов недоступны в waterfall-анализе «из коробки». Чтобы сделать их доступными, нужно реализовать один из подходов, описанных в статье [Link cross-origin XHR user actions and their distributed traces in RUM Classic](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/link-cross-origin-xhrs "Включить корреляцию между кросс-доменными XHR-действиями и distributed traces.").
* Dynatrace оценивает разные факторы для сопоставления ресурсов waterfall (на основе W3C resource timings) и distributed traces (захваченных OneAgent). Один из факторов, URL. Если части URL не совпадают на клиентской и серверной стороне, например, из-за правил перезаписи, кнопка **View trace** может не отображаться. В таких случаях нужно выбрать плитку **Traces**, показанную в top findings, чтобы перейти ко всем distributed traces.

## Настройки приложения, влияющие на анализ waterfall

В настройках приложения можно настроить пороговые значения для основных находок, а в глобальных настройках, задать типы ресурсов и правила очистки URL.

### Настройка пороговых значений находок waterfall

Настройка пороговых значений находок waterfall

1. Перейти в раздел **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Capturing** > **Advanced setup**.
5. Настроить нужные пороговые значения в разделе **Waterfall finding thresholds**.

### Определение типа ресурса

В представлении Waterfall analysis ресурсы можно группировать по их типу. Поскольку MIME-тип недоступен из JavaScript, тип ресурса определяется на основе расширения файла. Если у некоторых ресурсов нет корректного расширения, можно настроить правила сопоставления URL для определения этих типов ресурсов.

Настройка правила сопоставления URL для определения типа ресурса

1. Перейти в раздел **Settings**.
2. Выбрать **Web and mobile monitoring** > **Resource types**.

![Waterfall analysis](https://dt-cdn.net/images/img3-2000-ba4f2dd61f.png)

### Определение правил очистки URL

В агрегированном представлении **Waterfall analysis** ресурсы агрегируются на основе их URL. Часто идентификаторы и другие случайные числа мешают корректной агрегации ресурсов. В этом случае некоторые важные ресурсы могут непреднамеренно попасть в список **Rarely occurring** ресурсов.

Определение правила очистки URL

1. Перейти в **Settings** > **Web and mobile monitoring** > **Resource URL cleanup rules**.
2. Задать [регулярные выражения](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Learn how to use regular expressions in the context of Dynatrace.") для удаления идентификаторов.
