# Тегирование конкретных пользователей для веб-приложений в RUM Classic

* How-to guide
* 5 минут на чтение
* Обновлено 01 апр. 2026 г.

Одна из ключевых возможностей Real User Monitoring, это способность уникально идентифицировать отдельных пользователей в разных браузерах, устройствах и пользовательских сессиях.

С помощью [тегов пользователей](/managed/observe/digital-experience/rum-classic/rum-concepts/user-and-error-events#user-tagging "Узнайте о событиях пользователей и ошибок и о типах событий пользователей и ошибок, которые фиксирует Dynatrace.") можно анализировать поведение и опыт конкретного пользователя с помощью [анализа пользовательских сессий](/managed/observe/digital-experience/rum-classic/session-segmentation/user-sessions "Узнайте о сегментации пользовательских сессий и атрибутах фильтрации.").
По умолчанию Dynatrace присваивает каждому новому пользователю уникальный случайный ID. Однако можно назначать более осмысленные пользовательские теги, составленные, например, из имён пользователей или email-адресов.

Для веб-приложений можно настроить пользовательское тегирование с помощью RUM JavaScript API или метаданных страницы приложения.

## Тегирование пользователей через RUM JavaScript API

Используйте метод [`identifyUser`﻿](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#identifyuser) [RUM JavaScript API](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/customize-rum "Узнайте, как настроить Real User Monitoring Classic с помощью JavaScript API."), чтобы задать тег пользователя.

## Тегирование пользователей на основе исходного кода страницы

Этот подход к тегированию пользователей работает за счёт захвата доступных данных в исходном коде страницы приложения: имена пользователей или email-адреса довольно часто встречаются в тексте DOM-элемента или в переменной JavaScript. Полный список источников данных страницы см. в разделе [Доступные типы источников](#available-page-data-sources).

Например, демонстрационное приложение **easyTravel** отображает имя пользователя в приветственном сообщении в правом верхнем углу главной страницы (см. изображение ниже). С помощью инструментов разработчика, встроенных в большинство браузеров, можно сгенерировать уникальный CSS-селектор для этого конкретного элемента.

### Добавление правила тега пользователя

После того как определено, где в исходном коде страницы приложения находятся имена пользователей, можно создать правило тега пользователя.

Чтобы добавить правило тега пользователя

1. Перейдите в раздел **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Capturing** > **User tags**.
5. Выберите **Add user tag rule**.
6. Выберите **Source type** и имя источника для этого правила тега пользователя.
   Доступные типы источников

   * **CSS selector**. Используется, когда идентификатор пользователя виден на странице. Этот механизм захватывает значение `innerText/textContent` первого совпадения (доступно для браузеров, поддерживающих `querySelector`). Чтобы получить значение конкретного атрибута элемента, добавьте символ «@», за которым следует имя атрибута, например, `#someDomElement@someAttribute`.
   * **JavaScript variable**. Используется, когда идентификатор пользователя доступен как переменная JavaScript в объекте `window`. Например, если указать переменную JavaScript `userName`, RUM JavaScript получит значение из `window.userName`.
   * **Meta tag**:. Используется, когда идентификатор пользователя присутствует в одном из мета-тегов исходного кода страницы. Здесь указывается имя мета-тега, которое Dynatrace использует для захвата его значения `content`.
   * **Cookie value**. Используется, когда идентификатор пользователя присутствует в одном из существующих cookie.

     Если в качестве источника данных нужно использовать `Cookie value`, убедитесь, что у cookie отсутствует атрибут `HttpOnly`. В противном случае RUM JavaScript не сможет прочитать значения cookie, поскольку cookie с атрибутом `HttpOnly` недоступны для JavaScript.
   * **Query string**. Используется, когда идентификатор пользователя является частью определённого параметра строки запроса, который можно указать здесь.
   * **Server-side request attribute**. Используется, когда нужно использовать серверный [атрибут запроса](/managed/observe/application-observability/services/request-attributes "Узнайте, что такое атрибуты запросов, и как использовать их на всех уровнях всех представлений анализа сервисов."), чтобы тегировать каждую пользовательскую сессию.  
     После того как [атрибут запроса определён](/managed/observe/application-observability/services/request-attributes#define-request-attributes "Узнайте, что такое атрибуты запросов, и как использовать их на всех уровнях всех представлений анализа сервисов."), задайте **Server-side request attribute** в качестве **Source type**. Обратите внимание, что атрибут запроса может захватываться не для каждого действия пользователя и не для каждой пользовательской сессии, поскольку серверная распределённая трассировка может не захватываться из-за [Adaptive Traffic Management](/managed/ingest-from/dynatrace-oneagent/adaptive-traffic-management/adaptive-traffic-management-managed "Улучшите работоспособность и производительность среды Dynatrace Managed с помощью адаптивных функций управления трафиком, снижения нагрузки и контроля захвата.").
7. Опционально Включите **Ignore case**, если нужно преобразовать тег пользователя в нижний регистр.
8. Опционально Чтобы обеспечить чистое извлечение значения имени пользователя, включите **Apply cleanup rule** и введите регулярное выражение.
9. Выберите **Add user tag rule**.

### Проверка конфигурации тега пользователя

Чтобы убедиться, что Dynatrace корректно применил конфигурацию тега пользователя, сохраните конфигурацию и перезагрузите страницу приложения. После этого посмотрите на внедрённый JavaScript в обновлённом исходном коде страницы приложения.

Как видно в примере ниже, теперь в выражениях метаданных этой страницы указано свойство с именем `md=`.

Чтобы найти сессии конкретного пользователя

1. Перейдите в раздел **Session Segmentation**.
2. Выберите поле фильтра в верхней части страницы и выберите атрибут **User tag**.
3. Выберите тег пользователя из списка и выберите **Apply**.

На странице **User sessions** теперь отображается список сессий, связанных с этим конкретным пользователем. Можно выбрать конкретную сессию, чтобы просмотреть её детали. Либо выберите имя пользователя, чтобы перейти на [страницу обзора этого пользователя](/managed/observe/digital-experience/rum-classic/session-segmentation/analyze-all-sessions-of-a-single-user "Узнайте о поведении пользователя, анализируя профиль пользователя (оценку пользовательского опыта) и активность сессий.").

## Ограничения и примечания

* На одно приложение можно добавить до 20 правил тегов пользователей.
* Все захваченные значения обрезаются до максимальной длины в 100 символов.
* Если для идентификации пользователей используется RUM JavaScript API, API может переопределить любые существующие правила метаданных, настроенные ранее. Теги пользователей могут переопределять любые другие теги на основе метаданных.
* Все настроенные теги пользователей захватываются на каждой странице. RUM JavaScript используется для проверки каждого отслеживаемого действия пользователя на соответствие каждому созданному правилу тега пользователя, поэтому список правил тегов пользователей стоит держать минимальным, чтобы снизить накладные расходы.
* В качестве тега для всей сессии используется последнее действие пользователя в сессии, содержащее тег.
* Настройка конфиденциальности данных [Do Not Track](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/configure-real-user-monitoring-according-to-gdpr#do-not-track-gdpr "Узнайте о настройках конфиденциальности, которые предоставляет Dynatrace, чтобы обеспечить соответствие ваших веб-приложений региональным нормам защиты данных.") может влиять на тегирование пользователей.

  Если для веб-приложения включена опция **Comply with "Do Not Track" browser settings** и у конкретного пользователя в браузере включена опция «Do Not Track», информация о теге пользователя не отправляется. Однако если у пользователя опция «Do Not Track» в браузере отключена, его тег пользователя захватывается.
* Если организация по соображениям конфиденциальности не разрешает отслеживание отдельных пользователей, можно вместо этого определить теги пользователей, соответствующие названиям команд или отделов. Таким образом можно отслеживать опыт отдельных пользователей, не раскрывая никакой идентифицирующей информации.
