# Настройка атрибутов cookie RUM Classic

* Практическое руководство
* Чтение 4 мин
* Обновлено 22 апр. 2026 г.

Dynatrace Real User Monitoring Classic использует набор HTTP-cookie, см. [Cookies и клиентское хранилище для RUM и Session Replay](/managed/manage/data-privacy-and-security/data-privacy/rum-cookies-and-web-storage "Узнайте, как Dynatrace RUM и Session Replay используют cookie, веб-хранилище и IndexedDB."). Некоторые атрибуты cookie можно настраивать.

Чтобы получить доступ к этим параметрам

1. Перейдите в **Web**.
2. Выберите приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выберите **More** (**…**) > **Edit**.
4. В настройках приложения выберите **Injection** > **Cookie**.

В следующих разделах описаны доступные параметры.

## Настройка атрибутов cookie, важных для конфиденциальности данных

Если политика безопасности компании требует атрибутов cookie `Secure` и `SameSite`, RUM нужно настроить на их установку, поскольку по умолчанию они не задаются.

Dynatrace RUM не поддерживает атрибут `HttpOnly`. Поскольку cookie с `HttpOnly` недоступны для JavaScript, JavaScript RUM не может их читать или изменять. Нужно убедиться, что инфраструктура не добавляет атрибут `HttpOnly`, поскольку это нарушит мониторинг.

### Настройка атрибута Secure

Атрибут cookie `Secure` гарантирует, что браузеры отправляют cookie только через защищённые соединения.

Перед включением атрибута cookie `Secure` нужно убедиться, что приложение полностью обслуживается через защищённые соединения. Иначе будет потеряна видимость любого незашифрованного HTTP-трафика.

Чтобы установить атрибут cookie `Secure`, включите **Cookie** > **Use the Secure cookie attribute for cookies set by Dynatrace**.

### Настройка атрибута SameSite

Атрибут cookie `SameSite` определяет, отправляет ли браузер cookie с межсайтовыми запросами. Подробное объяснение этого атрибута и его значений см. в [SameSite cookies explained﻿](https://web.dev/samesite-cookies-explained/).

Чтобы установить атрибут cookie `SameSite`, перейдите в **Cookie** > **SameSite cookie attribute** и выберите одно из значений **None**, **Lax** и **Strict**.

## Настройка атрибута Domain

Атрибуты cookie `Domain` и `Path` определяют область действия cookie. Эта область определяет, отправляет ли браузер cookie вместе с запросом и может ли клиентский JavaScript получить к нему доступ для заданного URL.

* Dynatrace всегда устанавливает атрибут `Path` для cookie RUM в значение `/`, поэтому область действия cookie охватывает все пути URL в пределах домена. Атрибут `Path` настроить нельзя.
* По умолчанию атрибут `Domain` определяется автоматически, но его также можно настроить вручную для каждого приложения.

В следующих разделах объясняется, как Dynatrace определяет домен cookie по умолчанию и когда может понадобиться настроить его вручную.

### Домен cookie по умолчанию

По умолчанию Dynatrace определяет домен cookie автоматически, выбирая [эффективный домен верхнего уровня плюс один (eTLD+1)﻿](https://web.dev/same-site-same-origin/) от URL запроса. Например, Dynatrace выбирает домен cookie `example.com` для `www.example.com` и домен cookie `example.co.uk` для `www.example.co.uk`. Это позволяет Dynatrace фиксировать непрерывную сессию, даже когда пользователи перемещаются между поддоменами при взаимодействии с приложениями, например, с `www.example.com` на `shop.example.com`.

Домен cookie определяется либо на стороне сервера, силами OneAgent, либо на стороне клиента, силами JavaScript RUM.

* Если JavaScript RUM [внедряется автоматически](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection "Настройте автоматическое внедрение JavaScript RUM на страницы приложений."), домен cookie обычно определяет OneAgent, используя результат определения имени хоста как отправную точку. Если неинструментированный компонент переписывает хостовую часть URL, нужно убедиться, что [определение имени хоста настроено корректно](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/define-your-applications-via-the-my-web-application-placeholder#rum-appdetection-uninstrumentedcomponent "Узнайте, как определить приложения, следуя предложенному, ручному подходу или подходу с правилами обнаружения приложений.").
* При использовании [agentless RUM](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/set-up-agentless-real-user-monitoring "Настройте agentless-мониторинг для веб-приложений.") домен cookie определяет JavaScript RUM.

### Настройка домена cookie вручную

В большинстве сценариев настраивать домен cookie вручную не требуется. Однако ручная настройка полезна в следующих случаях:

* Нужно разделить пользовательские действия разных поддоменов на отдельные пользовательские сессии.
* Автоматическое определение домена cookie не работает, потому что неинструментированный компонент переписывает хостовую часть URL и не передаёт исходную информацию о хосте в заголовке запроса. Предпочтительное решение, настроить компонент на добавление такого заголовка, см. [Что делать, если неинструментированный компонент переписывает части URL?](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/define-your-applications-via-the-my-web-application-placeholder#rum-appdetection-uninstrumentedcomponent "Узнайте, как определить приложения, следуя предложенному, ручному подходу или подходу с правилами обнаружения приложений."). Ручная настройка домена cookie также может служить обходным решением.

Чтобы настроить домен cookie, перейдите в **Cookie** > **Domain to be used for cookie placement** и введите нужный домен.

Типичные проблемы и ограничения ручной настройки

##### Настроенный домен cookie является публичным суффиксом

Браузеры не позволяют устанавливать cookie с атрибутом `Domain`, выходящим за границы одной организации. Поэтому установка домена cookie в значение [публичного суффикса﻿](https://publicsuffix.org/list/) не сработает.

##### В настроенное приложение сопоставлены несколько доменов без общего eTLD+1

Поскольку домен cookie настраивается для каждого приложения отдельно, все домены, сопоставленные с приложением, должны иметь как минимум общий eTLD+1.

Например, нельзя вручную настроить домен cookie для приложения, в которое сопоставлены и `www.example.com`, и `www.example.co.uk`. Если выбрать домен cookie `example.com`, браузер отклонит cookie RUM для запросов к `www.example.co.uk`.

##### Вложенные домены cookie

Если домен cookie настраивается вручную для всех или части приложений, нужно убедиться, что домены cookie приложений не вложены друг в друга.

Рассмотрим следующий пример. Домен `www.example.com` сопоставлен с приложением **Example app**, а домен `shop.example.com`, с приложением **Shopping app**. По умолчанию домен cookie `example.com` определяется автоматически для обоих приложений. Если вручную установить домен cookie для **Shopping app** в значение `shop.example.com`, а пользователи перемещаются туда-сюда между двумя приложениями, могут возникать неоднозначные ситуации. Cookie RUM для **Example app** используют домен cookie `example.com` и поэтому также применяются к **Shopping app**, у которого есть собственный набор cookie RUM с доменом `shop.example.com`. При такой конфигурации Dynatrace может случайным образом разбивать захваченные данные RUM на короткие пользовательские сессии, а пользовательские действия и распределённые трассировки могут связываться не так, как ожидается.
