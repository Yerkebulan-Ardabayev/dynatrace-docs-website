Список разрешённых источников для маяков позволяет указать источники, от которых конечные точки маяков [приложения](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/beacon-endpoint "Change the default beacon endpoint URL and send RUM beacons to Dynatrace infrastructure or another instrumented web server.") должны принимать кросс-доменные RUM-маяки.

## Маяки с одинаковым и разным источником

RUM JavaScript отправляет RUM-маяки, чтобы передать собранные данные в Dynatrace. В зависимости от метода внедрения существует две настройки по умолчанию:

* **Автоматически внедряемые приложения > маяки с тем же источником**

  Когда [RUM JavaScript внедряется автоматически](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection "Configure automatic injection of the RUM JavaScript into the pages of your applications"), RUM-маяки отправляются обратно на веб- или прикладной сервер, на котором размещено приложение с автоматическим внедрением; OneAgent предоставляет конечную точку маяка.

  По умолчанию маяки автоматически внедряемых приложений являются **маяками с тем же источником**, поскольку протокол, хост и порт запросов маяка и страницы, с которой они отправляются, совпадают.

  Если выбрана одна из [альтернативных настроек конечной точки маяка](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/beacon-endpoint "Change the default beacon endpoint URL and send RUM beacons to Dynatrace infrastructure or another instrumented web server.") (когда маяки автоматически внедряемого приложения отправляются на кластер ActiveGate или на инструментированный сервер в другом домене), RUM-маяки становятся **кросс-доменными маяками**.
* **Приложения без агента > кросс-доменные маяки**

  При использовании [мониторинга без агента](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/set-up-agentless-real-user-monitoring "Set up agentless monitoring for your web applications.") RUM-маяки отправляются на конечную точку маяка, которая является частью кластера ActiveGate.

  Для приложений без агента RUM-маяки являются **кросс-доменными маяками**, поскольку они отправляются в другой домен.

Браузеры соблюдают политику одинакового источника, которая по умолчанию разрешает скриптам отправлять запросы только на тот же источник. Для отправки кросс-доменных запросов нужно использовать механизм совместного использования ресурсов между источниками (CORS), который позволяет серверам указывать источники, которым разрешён доступ к серверу. Поэтому кросс-доменные RUM-маяки должны использовать CORS. В этом случае:

* Браузер добавляет заголовок `Origin` к кросс-доменному маяку.
* По умолчанию конечная точка маяка добавляет заголовок `Access-Control-Allow-Origin` к каждому ответу, разрешая источник, указанный в заголовке `Origin`.

С помощью списка разрешённых источников для маяков можно указать, от каких источников конечные точки маяков должны принимать RUM-маяки.

## Указание источников маяков для CORS

Создайте правило источника маяков, чтобы указать, от каких источников OneAgent и кластер ActiveGate должны принимать RUM-маяки.

Сразу после добавления первого правила источника маяков приложения, не соответствующие этому правилу, перестанут собирать RUM-данные, если только их маяки не отправляются на тот же источник и не обрабатываются OneAgent.

Чтобы добавить правило источника маяков

1. Перейдите в **Settings** > **Web and mobile monitoring** > **Beacon origins for CORS**.
2. Выберите **Add item**.
3. Укажите корректный шаблон для источника, который нужно задать.

   ![Add a beacon origin rule](https://dt-cdn.net/images/add-beacon-origin-rule-1912-783d3ea903.png)

   Add a beacon origin rule

Можно добавить до 20 правил источников маяков на среду.

## Применение списка разрешённых источников для маяков в различных сценариях

![Beacon origin allowlist flowchart](https://dt-cdn.net/images/beacon-origin-allowlist-after-settings-conversion-716-6a55b387ba.png)

Beacon origin allowlist flowchart

Эта блок-схема показывает, как Dynatrace применяет список разрешённых источников для маяков в различных сценариях. Используйте её, чтобы понять, разрешён ли конкретный источник маяка.

* Если список разрешённых источников для маяков пуст, RUM-маяки от любого источника принимаются всеми конечными точками маяков.
* Если источник есть в списке разрешённых, RUM-маяк от этого источника принимается. В кросс-доменном случае источник копируется в заголовок `Access-Control-Allow-Origin` ответа, а ответ маяка возвращает код состояния HTTP `200 OK`.
* Если источника нет в списке разрешённых, кросс-доменный RUM-маяк от этого источника отклоняется. Маяк завершается с кодом состояния `403 Forbidden` и сообщением вида `Value in Origin Header is not allowed`.
* OneAgent не применяет список разрешённых источников для маяков к маякам с тем же источником.
