# Выбор формата сниппета в RUM Classic

* Практическое руководство
* Опубликовано 01 июля 2025

RUM JavaScript, независимо от того, [внедряется он автоматически](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection "Configure automatic injection of the RUM JavaScript into the pages of your applications") или [вставляется вручную](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/set-up-agentless-real-user-monitoring "Set up agentless monitoring for your web applications.") в веб-приложение, состоит из двух ключевых компонентов:

* **Monitoring code**, JavaScript-код, который обеспечивает возможности RUM, такие как захват действий пользователя.
* **Configuration**, конфигурация приложения, которая изначально используется кодом мониторинга. Позже она обновляется через ответ RUM beacon при изменении конфигурации.

Dynatrace RUM предоставляет несколько форматов сниппетов, которые встраивают эти компоненты в страницы разными способами, каждый из которых рассчитан на определённые требования.

Эта страница [предлагает рекомендации](#snippet-format-recommendations), которые помогут выбрать формат сниппета, лучше всего соответствующий требованиям, а затем приводит [подробное описание каждого формата](#snippet-format-details).

## Рекомендации по выбору формата сниппета

Для большинства случаев использования рекомендуется следующее:

* Использовать [JavaScript Tag](#js-tag) для ручной вставки.
* Использовать [OneAgent JavaScript Tag](#oneagent-js-tag) для автоматического внедрения.

Если нужно задействовать функцию браузера subresource integrity (SRI), чтобы убедиться, что код мониторинга не был изменён, использовать [OneAgent JavaScript tag with SRI](#oneagent-js-tag-sri) для ручной вставки и автоматического внедрения.

Чтобы избежать блокирующего парсинг выполнения кода мониторинга, нужно настроить параметр **script execution** выбранного формата сниппета на значение **async** или **defer**. Инструкции по настройке этого параметра приведены ниже в документации по каждому отдельному формату сниппета. Стоит учитывать, что пока код мониторинга ещё не полностью загружен и инициализирован, некоторые тайминги и действия пользователя будут потеряны.

## Доступные форматы сниппетов

Тег JavaScript

OneAgent тег JavaScript

OneAgent тег JavaScript с SRI

Встроенный код

Фрагмент кода

**Тег JavaScript** ссылается на внешний файл, который включает и код мониторинга, и конфигурацию. Тег JavaScript доступен только для ручной вставки.

[Получить тег JavaScript через API](/managed/dynatrace-api/environment-api/rum/rum-manual-insertion-tags/get-javascript-tag)

##### Обновления

Короткий срок кэширования внешнего файла обеспечивает регулярные автоматические обновления, поэтому тег JavaScript, это идеальный выбор, если не нужно самостоятельно следить за обновлением вставленного фрагмента после изменений конфигурации.

##### Источник кода мониторинга

Внешний файл загружается с CDN.

##### Срок кэширования

Внешний файл, содержащий код мониторинга и конфигурацию, обновляется согласно сроку кэширования, который по умолчанию составляет один час и может настраиваться. Доступны небольшие значения, от нескольких часов до нескольких дней, чтобы конфигурация обновлялась регулярно. Срок кэширования задаёт, как долго браузеры могут кэшировать файл. Если CDN настроен согласно описанию в разделе [Подготовка кластера к продуктивной эксплуатации](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/set-up-agentless-real-user-monitoring#managed-cluster-production-ready "Настройка мониторинга реальных пользователей без агента для веб-приложений."), он кэширует файл на один час.

Настройка срока кэширования

1. Перейти в раздел **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В верхнем правом углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Injection** > **Manual insertion**.
5. В разделе **JavaScript tag** в выпадающем списке под **Cache monitoring code and configuration for** выбрать нужный срок кэширования.

##### Выполнение скрипта

По умолчанию код мониторинга загружается и выполняется браузером синхронно. Чтобы избежать блокирующего парсинг поведения, можно настроить добавление атрибутов `async` или `defer`.

* При использовании атрибута `async` код мониторинга загружается параллельно с парсингом страницы и выполняется сразу же, как только становится доступен.
* При использовании атрибута `defer` код мониторинга также загружается параллельно, но выполнение откладывается до завершения парсинга страницы.

При использовании обоих атрибутов часть таймингов и пользовательских действий будет потеряна, пока код мониторинга ещё не загружен и не инициализирован полностью.

Настройка выполнения скрипта

1. Перейти в раздел **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В верхнем правом углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Injection** > **Manual insertion**.
5. В разделе **JavaScript tag** установить **Script execution attribute** в значение **async**, **defer** или **No attribute**.
6. Скопировать фрагмент и вставить его на страницу.

При получении тега JavaScript через API можно управлять выполнением скрипта, передав параметр. Подробности см. в разделе [GET JavaScript tag](/managed/dynatrace-api/environment-api/rum/rum-manual-insertion-tags/get-javascript-tag "Получение самого актуального тега JavaScript для ручной вставки.").

##### Добавление атрибута `crossorigin="anonymous"`

🟢 Включено по умолчанию

Внешний скрипт, на который ссылается тег JavaScript, обслуживается с CDN, что приводит к кросс-доменному запросу. Чтобы включить сбор подробных сообщений об ошибках JavaScript и W3C-таймингов ресурсов, нужно включить атрибут `crossorigin="anonymous"` в тег скрипта.

Включение добавления атрибута `crossorigin="anonymous"`

1. Перейти в раздел **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В верхнем правом углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Injection** > **Manual insertion**.
5. В разделе **JavaScript tag** включить **Add the crossorigin=anonymous attribute**.
6. Скопировать фрагмент и вставить его на страницу.

При получении тега JavaScript через API можно управлять добавлением атрибута `crossorigin="anonymous"`, передав параметр. Подробности см. в разделе [GET JavaScript tag](/managed/dynatrace-api/environment-api/rum/rum-manual-insertion-tags/get-javascript-tag "Получение самого актуального тега JavaScript для ручной вставки.").

**OneAgent тег JavaScript** включает конфигурацию и ссылается на внешний файл, содержащий код мониторинга. Он доступен как для ручной вставки, так и для автоматического внедрения.

[Получить OneAgent тег JavaScript через API](/managed/dynatrace-api/environment-api/rum/rum-manual-insertion-tags/get-oneagent-javascript-tag)

##### Обновления

При ручной вставке OneAgent тега JavaScript его также нужно вручную обновлять при каждом изменении конфигурации. Чтобы он оставался актуальным, рекомендуется использовать его вместе с полностью автоматизированными обновлениями через API. Использование OneAgent тега JavaScript из веб-интерфейса не рекомендуется, если только своевременность обновлений конфигурации не является критичной.

В случае автоматического внедрения OneAgent всегда внедряет OneAgent тег JavaScript с использованием актуальной конфигурации. При этом стоит учитывать, что политика кэширования приложения может влиять на то, как часто у OneAgent есть возможность выполнить внедрение. Подробнее см. в разделе [Оптимизации заголовка Cache-Control](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection#cache-header-optimization "Настройка автоматического внедрения RUM JavaScript на страницы приложений").

##### Источник кода мониторинга

В случае мониторинга без агента код мониторинга доставляется CDN. В случае автоматического внедрения или [ручной вставки для страниц приложения с автоматическим внедрением](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection#manual-insertion-using-oneagent "Настройка автоматического внедрения RUM JavaScript на страницы приложений"), файл по умолчанию доставляется OneAgent, который инструментирует приложение. Также можно загружать его с CDN, см. раздел [Запрос кода мониторинга с CDN](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/configure-monitoring-code-source#request-rum-monitoring-code-from-cdn "Настройка источника кода Real User Monitoring Classic под конкретные требования.").

##### Срок кэширования

И браузеры, и CDN могут кэшировать код мониторинга на срок до одного года. Такой продолжительный срок кэширования становится возможным благодаря встраиванию в URL файла версии RUM JavaScript и хеша активных модулей.

**OneAgent тег JavaScript с SRI** позволяет воспользоваться функцией браузера subresource integrity (SRI), чтобы убедиться, что код мониторинга не был изменён, см. раздел [Использование Subresource Integrity (SRI) для кода Real User Monitoring Classic](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/subresource-integrity "Использование функции браузера Subresource Integrity (SRI) для обеспечения целостности кода Real User Monitoring Classic."). Поддерживается как для автоматического внедрения, так и для ручной вставки.

**Inline code** содержит и конфигурацию, и код мониторинга RUM, что позволяет минимизировать число веб-запросов. При этом код мониторинга Session Replay не встраивается инлайн, поэтому при использовании Session Replay по-прежнему будет дополнительный запрос. Если сайт состоит из множества отдельных страниц, использование inline-кода может быть невыгодным, так как оно увеличивает размер каждого документа. Однако это подходящий вариант для одностраничных приложений (SPA).

[Получить inline-код через API](/managed/dynatrace-api/environment-api/rum/rum-manual-insertion-tags/get-inline-code)

##### Обновления

При ручной вставке inline-кода его также нужно вручную обновлять при каждом изменении конфигурации. Чтобы код оставался актуальным, рекомендуется использовать его вместе с полностью автоматизированными обновлениями через API. Использование inline-кода из веб-интерфейса не рекомендуется, если своевременность обновлений конфигурации не критична.

При автоматической инъекции OneAgent всегда внедряет inline-код, используя текущую конфигурацию. Однако политика кеширования приложения может влиять на то, как часто OneAgent получает возможность выполнить внедрение. Подробнее см. [Оптимизация заголовков управления кешем](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection#cache-header-optimization "Configure automatic injection of the RUM JavaScript into the pages of your applications").

##### Источник кода мониторинга

При agentless-мониторинге код мониторинга Session Replay доставляется через CDN. При автоматической инъекции или [ручной вставке для страниц приложения с автоматической инъекцией](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection#manual-insertion-using-oneagent "Configure automatic injection of the RUM JavaScript into the pages of your applications") код мониторинга Session Replay по умолчанию доставляется OneAgent, который инструментирует приложение. Также можно загружать его через CDN, см. [Запрос кода мониторинга из CDN](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/configure-monitoring-code-source#request-rum-monitoring-code-from-cdn "Configure the Real User Monitoring Classic code source for your specific requirements.").

##### Продолжительность кеширования

Так как код мониторинга RUM встроен инлайн, его продолжительность кеширования совпадает с продолжительностью кеширования страницы, которая определяется настройками кеша веб-сервера. Код мониторинга Session Replay может кешироваться браузерами и CDN на срок до года. Такая продолжительность кеширования достигается за счёт встраивания версии RUM JavaScript в URL файла.

##### Выполнение скрипта

Код мониторинга загружается и вычисляется браузером синхронно.

**Code snippet** содержит конфигурацию и базовую функциональность кода мониторинга. Доступен как для ручной вставки, так и для автоматической инъекции.

* При ручной вставке он внедряет на страницу ещё один элемент script, который ссылается на внешний файл, содержащий полную функциональность кода мониторинга.
* При автоматической инъекции этот второй элемент script внедряется OneAgent при внедрении code snippet.

Внешний файл, содержащий полную функциональность кода мониторинга, можно настроить на синхронную или асинхронную загрузку с помощью атрибута `defer`, см. [Выполнение скрипта](#code-snippet-script-execution).

##### Обновления

При ручной вставке обновления кода мониторинга и конфигурации становятся доступны после первой перезагрузки страницы. Чтобы актуальная конфигурация была доступна с самого начала, code snippet нужно регулярно обновлять. Это также необходимо для гарантии совместимости между code snippet и внешним файлом, которую тестируют на протяжении года.

При автоматической инъекции OneAgent всегда внедряет code snippet, используя текущую конфигурацию. Однако политика кеширования приложения может влиять на то, как часто OneAgent получает возможность выполнить внедрение. Подробнее см. [Оптимизация заголовков управления кешем](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection#cache-header-optimization "Configure automatic injection of the RUM JavaScript into the pages of your applications").

##### Источник кода мониторинга

При agentless-мониторинге внешний файл, содержащий полную функциональность кода мониторинга, доставляется через CDN. При автоматической инъекции или [ручной вставке для страниц приложения с автоматической инъекцией](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection#manual-insertion-using-oneagent "Configure automatic injection of the RUM JavaScript into the pages of your applications") он по умолчанию доставляется OneAgent, который инструментирует приложение. Также можно загружать его через CDN, см. [Запрос кода мониторинга из CDN](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/configure-monitoring-code-source#request-rum-monitoring-code-from-cdn "Configure the Real User Monitoring Classic code source for your specific requirements.").

##### Продолжительность кеширования

Кеширование code snippet совпадает с кешированием страницы, которое определяется настройками веб-сервера. Внешний файл может кешироваться браузерами и CDN на срок до года. Такая продолжительность кеширования достигается за счёт встраивания версии RUM JavaScript и хеша активных модулей в URL файла.

##### Выполнение скрипта

Внешний файл, обеспечивающий полную функциональность кода мониторинга, можно настроить на синхронную или асинхронную загрузку с помощью атрибута `defer`. При асинхронной загрузке скрипт скачивается параллельно с парсингом HTML и выполняется только после того, как весь документ полностью распарсен. Пока код мониторинга ещё не загружен полностью, часть таймингов и пользовательских действий будет потеряна.

Настройка выполнения скрипта для ручной вставки

1. Перейти в **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Injection** > **Manual insertion**.
5. В разделе **Code snippet** в выпадающем списке под **Load the monitoring code** выбрать **synchronously** или **deferred**.
6. Скопировать сниппет и вставить его на страницу.

Настройка выполнения скрипта для автоматической инъекции

1. Перейти в раздел **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Injection** > **Automatic injection**.
5. В разделе **Snippet format** выбрать **Code snippet**.
6. В разделе **Load the monitoring code** выбрать вариант **synchronously** или **deferred**.
7. Выбрать **Save changes**.
