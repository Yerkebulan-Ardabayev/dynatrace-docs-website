# Создание пользовательских названий действий для веб-приложений в RUM Classic

* Практическое руководство
* Чтение: 12 мин
* Обновлено 27 января 2023 г.

Названия [действий пользователя](/managed/observe/digital-experience/rum-classic/rum-concepts/user-actions "Узнайте, что такое действия пользователя и как они помогают понять, что пользователи делают в приложении.") создаются автоматически на основе URL загрузки или XHR-запросов, вызвавших действие, и их часто трудно читать и запоминать. Когда приложение обрабатывает много пользовательских запросов и взаимодействий в день, отслеживать действия пользователя становится сложно, если нет возможности логически их группировать.

Правила именования действий пользователя дают возможность создавать последовательные и удобочитаемые названия действий, которые легко группировать и отслеживать во всём приложении.

Такие правила именования действий можно создавать и настраивать индивидуально для каждого приложения, а затем использовать в Dynatrace в:

* Диаграммах пользовательского опыта
* Графиках
* Целях конверсии (например, `add-to-shopping-cart` и `newsletter-sign-up`)
* Ключевых действиях пользователя
* Списке основных потребителей

Чтобы упростить создание и поддержку правил именования действий пользователя, можно использовать заполнители, входные параметры и даже разные правила для Load- и XHR-действий.

## Определение названия действия

Dynatrace пытается назначать действиям осмысленные названия. Для этого проверяется несколько свойств действия, таких как внутренний HTML, подпись и подсказка HTML-элемента, вызвавшего действие. Этим элементом может быть кнопка или ссылка. Также предпринимается попытка получить подпись, если структура HTML более сложная и содержит несколько вложенных тегов.

RUM JavaScript использует несколько техник для выбора названия, наиболее подходящего действию. Проверка начинается с самого внутреннего HTML-узла, по которому был клик, например кнопки, тега изображения или ссылки, и проверяется следующее в порядке приоритета:

1. [атрибут `data-dtname`](#change-action-name-via-data-dtname)
2. `nodeName`, например изображение, ссылка или поле ввода  
   Проверка останавливается при обнаружении тега `html`, тега `body`, тега `head` или элемента `document`.
3. `innerText` или `textContent`

Если ни один из этих вариантов не дал приемлемого результата, RUM JavaScript начинает применять рекурсивный алгоритм, который проверяет разные параметры в зависимости от `nodeName` текущего проверяемого HTML-узла. Если ничего не найдено, проверяется родительский узел.

## Добавление и использование заполнителей

Заполнители позволяют добавлять динамические элементы в пользовательские правила именования.

Заполнители полезны в случаях, когда нужно включить в название действия пользователя динамические входные значения (например, URL записей блога, которые посещают пользователи), либо когда нужно удалить или заменить определённые части входного значения (например, динамические идентификаторы пользовательской сессии) и сгруппировать их под общим названием действия пользователя.

Заполнитель состоит из следующих трёх элементов:

* Входное значение, например URL страницы
* Шаг обработки, извлекающий или заменяющий части входного значения (необязательно)
* Название заполнителя, которое используется в правиле именования действия пользователя

Чтобы упростить работу с заполнителями, Dynatrace предоставляет набор предопределённых заполнителей, таких как `pageUrl`, `sourceUrl`, `xhrUrl`, `Top XHR URL` и `pageTitle`. Чтобы лучше понять заполнители, рассмотрим следующий пример: XHR-действие вызывает выбор кнопки подтверждения на странице оформления заказа. За выбором кнопки подтверждения (в данном случае, это исходное действие) следует одно из следующих целевых действий:

* первый веб-запрос вызывает несколько последовательных XHR-запросов, например `xhrUrl` или `Top XHR URL`.
* запускается новое действие загрузки, перенаправляющее на страницу подтверждения. В этом случае есть `sourceUrl`, это URL, на котором завершился процесс оформления заказа, и `pageUrl`, это URL, на котором отображается подтверждение оформления заказа.

Такие цепочки и корреляции действий лучше всего видны на водопадной диаграмме. Можно использовать `sourceURL` и `pageURL` или эквиваленты для XHR, `XHRUrl` и `Top XHR URL`, чтобы правильно называть действия в случаях, когда возникают такие корреляции.

Предоставленный набор заполнителей не подлежит настройке, но его можно сразу использовать для создания собственных правил именования. Также можно создавать собственные заполнители.

### Создание заполнителя

Чтобы создать заполнитель

1. Перейти в **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Capturing** > **User actions**.
5. В разделе **Placeholders** выбрать **Add placeholder** и выбрать тип входных данных.

   Для типов данных **CSS selector**, **JavaScript variable**, **Meta tag** и **Cookie value** также нужно указать значение.

   ![Добавление заполнителя](https://dt-cdn.net/images/screenshot-2020-08-27-at-15-22-45-1308-aba05796c1.png)

   Добавление заполнителя
6. Необязательно Добавить шаги обработки и условия, а также предварительно просмотреть заполнитель, чтобы проверить, работает ли он как задумано.
7. Необязательно Если добавлен шаг обработки **extract** или **extract with regular expression**, выбрать, использовать ли в качестве резервного варианта пустой результат (по умолчанию) или выбранное входное значение.
8. Назвать заполнитель и выбрать **Save**.

Можно добавить до 50 заполнителей на приложение.

### Шаги обработки

Чтобы помочь максимально эффективно использовать заполнители, представлены следующие шаги обработки:

* **extract**: требует разделитель, с которого должно начинаться извлечение, и разделитель, на котором извлечение должно закончиться. Разделители не входят в извлекаемую часть. Если начальный разделитель отсутствует, извлечение начинается с начала входного значения. Если конечный разделитель отсутствует, извлечение останавливается в конце входного значения. Опционально можно использовать в качестве резервного варианта пустой результат (по умолчанию) или выбранное входное значение.
* **replace**: требует разделитель, с которого должна начинаться замена, и разделитель, на котором замена должна закончиться. Опционально можно также указать текст замены. Если текст замены не указан, совпавшие символы отбрасываются.
* **replace text**: требует текст, который нужно найти и сопоставить. Опционально можно указать текст замены. Если текст замены не указан, совпавший текст отбрасывается.
* **replace IDs**: требует идентификатор, который нужно найти и сопоставить. Опционально можно указать идентификаторы замены. Если идентификатор замены не указан, совпавший идентификатор отбрасывается.
* **extract with regular expression**: требует регулярное выражение для поиска и извлечения. Опционально можно использовать в качестве резервного варианта пустой результат (по умолчанию) или выбранное входное значение.
* **replace with regular expression**: требует регулярное выражение для поиска строки, подлежащей замене. Опционально можно указать регулярное выражение замены. Если оно не указано, совпавшее регулярное выражение отбрасывается. В настоящее время поддерживается только одна группа захвата `(.*)`.

Если к заполнителю добавлено несколько шагов обработки, они применяются последовательно. Это означает, что первый шаг обработки применяется к выбранному входному значению, а каждый последующий шаг применяется к результату предыдущего шага обработки.

## Создание названий действий

Dynatrace предоставляет возможность создавать пользовательские действия загрузки, а также XHR-действия. Можно использовать новые заполнители или существующие заполнители, доступные в списке **Placeholders**, для создания новых правил именования.

В этом примере используется `CampaignID` для создания нового пользовательского правила именования, которое определяет загрузку страницы как просмотр кампании с добавлением `CampaignID` к названию действия.

1. Перейти в **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **Capturing** > **User actions**.
5. Прокрутить вниз до **User action naming rules** и выбрать вкладку **Naming rules for load actions** или **Naming rules for XHR actions**.
6. Выбрать **Add naming rule**.
7. Ввести шаблон именования с использованием заполнителя. Убедиться, что заполнитель заключён в фигурные скобки (пример: `{CampaignID}`).
8. Необязательно Определить условия, при которых также можно использовать заполнители для дополнительного уточнения применимости правил.

   В данном примере, чтобы создать пользовательское правило именования действий, которое применяется только к специальной промо-кампании, где `CampaignID` равен **3**, можно использовать условия и заполнители, чтобы правило соответствовало только нужным действиям, которые должны получить соответствующее название.

   ![Создание пользовательских действий](https://dt-cdn.net/images/campaignid-974-5548a30b62.png)

   Создание пользовательских действий
9. Необязательно Использовать раздел **Preview your rule**, чтобы проверить, работает ли новое правило именования как ожидается.
10. Выбрать **Save**.
11. Использовать элементы управления **Move up/down**, чтобы изменить приоритет правил.

    Первое подходящее правило в таблице применяется первым.

    ![Правила именования действий пользователя](https://dt-cdn.net/images/user-action-naming-2077-281508f770.png)

    Правила именования действий пользователя

Можно добавить до 250 правил именования действий пользователя на приложение.

## Расширенная настройка названий действий

Для имён пользовательских действий доступны следующие дополнительные параметры конфигурации:

* **Split user actions by domain**: отключите эту настройку, если разные домены не должны приводить к разделению пользовательских действий. Например, если приложение доступно на двух доменах, `www.domain-one.com` и `www.domain-two.com`, и разделение по доменам включено, то `load of page /index` для `www.domain-one.com` будет считаться отдельным пользовательским действием от `load of page /index` для `www.domain-two.com`.
* **Case-insensitive naming**: отключите эту настройку, если имена пользовательских действий нужно хранить с учётом регистра. Регистронезависимое именование позволяет искать по всем вариантам регистра. Например, действия `Loading of Page /` и `Loading of PAGE /` в обоих случаях будут отображаться как `loading of page /`.
* **Prioritize load actions and use value of last detected**: включите эту настройку, если нужно использовать улучшенное распространение load-действий. Когда во время действия присутствует load (по сути, запрос документа), действие помечается как load-действие.  
  Дополнительно включите эту настройку, чтобы использовать значения-заполнители, такие как `pageUrl`, из последнего load-действия для именования пользовательских действий. Например, рассмотрим начальное пользовательское действие `click /easytravel/search`, которое запускает перезагрузку страницы с `/easytravel` на `/easytravel/search` и, соответственно, load-действие `loading of page /easytravel/search`, которое, в свою очередь, запускает ещё один XHR-вызов.

  + При включённой настройке используется последнее обнаруженное load-действие, поэтому вся эта цепочка действий помечается как load-действие `loading of page /easytravel/search`.
  + При отключённой настройке действие помечается как `click /easytravel/search`, поскольку в предыдущей версии распространения load-действий load-действие не распространяется в сценариях, где также присутствуют XHR-запросы.

## Варианты использования и примеры

В следующих разделах рассмотрены примеры правил именования пользовательских действий, созданных в предыдущей версии, и показано, как эквивалентные правила можно создать с помощью новой, расширенной версии правил именования пользовательских действий.

#### Извлечение ценной информации из входных данных

Примеры из предыдущей версии правил именования пользовательских действий:

* **Extract** `/some-path-before/(.*).some-defined-text-after from path` from path
* **Extract** `Loading of page /sales/(.*)` from action name
* **Extract** `(.*)` from page title
* **Extract** `(.*)` from XHR URL

Теперь того же результата можно достичь с помощью заполнителей и этапа обработки.  
С помощью этапа обработки **extract** можно задать начальный и конечный разделители, которые точно определяют, откуда извлекаются значения. Таким образом, можно заменить многие правила именования, для которых раньше требовались регулярные выражения.

![Example for extract](https://dt-cdn.net/images/action-naming-examples-1-2880-6cb1cca2e3.png)

Пример для extract

#### Именование действий на основе их URL или пути

Примеры из предыдущей версии именования пользовательских действий:

* **Design step**: if action name contains `design`
* **Config data call**: action name matches regular expression `*/config/*`
* **Product**: path begins with `/product/`
* **Search**: XHR URL begins with `/api/search/dynamic`

В предыдущей версии правил именования пользовательских действий имя действия обычно использовалось как входное значение для переименования имён действий, автоматически обнаруженных Dynatrace. Это часто вызывало путаницу, поскольку пользователи не понимали, что определённые ими имена действий используются как входные значения.

Здесь нужно было различать автоматически обнаруженное Dynatrace имя действия (возможное входное значение) и итоговое имя действия, заданное пользователем через правило именования. Во всех этих случаях пользователь мог просто использовать путь или URL страницы либо URL XHR вместо этого.

В новой версии можно использовать заполнитель по умолчанию в правиле именования, например `pageUrlPath` или `pageUrl`, чтобы получить ту же информацию, а затем условие **begins with** или **contains**, чтобы сделать то же самое.

Пример для переименования

И наоборот, для большей гибкости можно использовать условие **match regular expression**.

Например, прежние правила из предыдущей версии именования пользовательских действий:

* `Customer Contact Request for UFO`: XHR url matches regex match `.*/api/request/.*/contact?category=UFO`

![Example for regex](https://dt-cdn.net/images/action-naming-examples-3-2880-882b31b031.png)

Пример для regex

#### Удаление или маскирование частей входных данных

Примеры из предыдущей версии правил именования пользовательских действий:

* **Remove ID**: `(\\?|&)_id=:*?(?=&|$)`
* **Remove conversation**: `(\\?|&)conversation=:*?(?=&|$)`
* **Remove params**: `\?.*`
* **Remove searched car**: `(?<=search-results\/cars\/)[a-zA-Z0-9]*`

Удаление и маскирование частей данных теперь проще выполняется с помощью подхода "remove in-between". Если нужно удалить значение определённого параметра строки запроса, например ID пользователя, задайте ключ как конечный разделитель, а `&` как следующий символ, вводящий другие параметры. Если во входном значении, к которому применяется сопоставление, нет конечного разделителя, вы получите уведомление.

![Example for renaming](https://dt-cdn.net/images/action-naming-examples-4-2880-c6afd6f33c.png)

Пример для переименования

Для большей гибкости, например, чтобы удалить все параметры URL, используйте **replace by regular expression**.

#### Объединение правил именования с помощью заполнителей

Рассмотрим в качестве примера оформление заказа в интернет-магазине. Чтобы назвать load-действия для каждого шага в предыдущей версии, нужно было сначала определить одно правило именования действия для каждого отдельного шага.

В этом случае на каждый шаг оформления заказа назначается отдельное правило:

* **Loading of Enter personal details**: action name begins with `/checkout/index.php?step=details`
* **Loading of Select payment method**: action name begins with `/checkout/index.php?step=payment`
* **Loading of Order summary**: action name begins with `/checkout/index.php?step=summary`
* **Loading of Order confirmation**: action name begins with `/checkout/index.php?step=confirmation`

В новой версии можно создать заполнитель, использующий в качестве входных данных JavaScript-переменную `adobePageName`, которая может быть уже доступна в существующем data layer инструмента аналитики, например Adobe Analytics.

## Задание имени действия через `data-dtname`

Если стандартное определение имени действия не подходит для ваших целей, можно также задать пользовательский атрибут `data-dtname` внутри HTML-тегов и использовать его в качестве подписи. Например, следующее действие:

```
<label for="txtFirstname">Firstname</label> <input data-dtname="Firstname Text Input" type="text" value="firstname" name="firstname" title="Firstname" id="txtFirstname" />
```

может получить подпись `click on "Firstname Text Input"`.

Чтобы пользовательский атрибут отображался как имя пользовательского действия, также нужно [настроить правило именования](#create-action-names), включающее заполнитель `{elementIdentifier (default)}`.

Если для разных инструментов используются разные имена атрибутов, можно задать альтернативу `data-dtname`, которую Dynatrace сможет использовать для именования пользовательских действий.
