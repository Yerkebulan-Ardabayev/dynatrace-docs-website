# Определение приложений для Real User Monitoring Classic

# Определение приложений для Real User Monitoring Classic

* Практическое руководство
* Чтение занимает 14 минут
* Обновлено 23 апр. 2024 г.

После установки [OneAgent](/managed/ingest-from/dynatrace-oneagent "Understand the important concepts related to OneAgent and find out how to install and operate OneAgent on different platforms.") на хост в режиме полного мониторинга стека он отслеживает все приложения, работающие на этом хосте. В качестве отправной точки все данные мониторинга инкапсулируются в приложении-заполнителе с именем **My web application**. Такое приложение-заполнитель предлагается для большей гибкости: именно пользователь решает, как организовать свои приложения.

Dynatrace предлагает следующие подходы к определению приложения для RUM.

| Подход к обнаружению приложений | Тип внедрения RUM JavaScript | Когда использовать |
| --- | --- | --- |
| [Рекомендуемый подход](#automated-approach) | Автоматическое внедрение | Когда веб-приложения легко идентифицировать по доменам и нужно создать приложение из уже захваченного трафика мониторинга RUM. |
| [Подход на основе правил обнаружения приложений](#application-detection-rules) | Автоматическое внедрение | Когда нужно создать новые приложения или когда разделения трафика по доменам недостаточно. Правила обнаружения приложений позволяют определять более сложные шаблоны для группировки трафика мониторинга RUM в приложения. |
| [Ручной подход](#alternative-approach) (также называемый «мониторинг без агента») | Внедрение вручную | Когда нет доступа к хосту веб-приложения, но есть доступ к коду приложения. |

Формат внедрения можно выбрать как для автоматически внедряемых, так и для вручную вставляемых приложений. Подробнее см. в разделе [Выбор формата фрагмента кода в RUM Classic](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/snippet-formats "Select a format for the RUM JavaScript snippet that best fits your specific use case").

Приложения на основе Java Servlet

При использовании RUM для приложения на основе Java Servlet после завершения первоначальной настройки процесс нужно перезапустить. Сенсор RUM не включается до перезапуска.

## Рекомендуемый подход

Приложение-заполнитель **My web application** агрегирует трафик со всех обнаруженных доменов. Это приложение может служить отправной точкой для сопоставления обнаруженных доменов с отдельными приложениями в среде.

1. Перейти в раздел **Web**.
2. Выбрать приложение-заполнитель **My web application**.
3. Прокрутить вниз до панели **Top 3 included domains**.  
   На этой панели показаны домены с наибольшим числом пользовательских действий, обнаруженных OneAgent в среде.
4. Выбрать **View full details**.
5. В списке **Top domains** выбрать кнопку со стрелкой в столбце **Transfer domain**, чтобы развернуть запись домена.
6. Выполнить одно из следующих действий:

   * Чтобы добавить новое приложение, выбрать **Create new application**. Приложение создаётся и отображается на странице **Applications**.
   * Чтобы добавить домен в существующее приложение, выбрать **Transfer**.

Поскольку может понадобиться более понятное имя для приложения, автоматически сгенерированное имя можно заменить на собственное имя приложения.

Переименование приложения

1. Перейти в раздел **Web**.
2. Выбрать приложение, которое нужно настроить.
3. В правом верхнем углу страницы обзора приложения выбрать **More** (**…**) > **Edit**.
4. В настройках приложения выбрать **General settings** > **Application name**.
5. Ввести предпочитаемое имя в поле в верхней части страницы. Обратите внимание, что имена приложений должны быть уникальными.

### Возможные проблемы для рекомендуемого подхода

**Неинструментированный компонент переписывает URL**. При использовании рекомендуемого подхода для нового или обновлённого приложения автоматически создаётся правило обнаружения приложения. Это правило сопоставляет пользовательские действия, зафиксированные на домене, с приложением. OneAgent использует часть URL с именем хоста для определения домена. Если неинструментированный компонент между браузером и первым инструментированным уровнем переписывает хост, нужно [настроить определение имени хоста](#rum-appdetection-uninstrumentedcomponent). Иначе OneAgent может не суметь применить правило обнаружения приложения.

**Действия по-прежнему сопоставляются с приложением-заполнителем**. Если пользовательские действия по-прежнему сопоставляются с **My web application** вместо нужного приложения, см. раздел [Как быстро вступают в силу изменения правил обнаружения приложений?](#rum-appdetection-rule-changes).

## Подход на основе правил обнаружения приложений

Если нужно создать больше приложений, изменить существующие сопоставления приложений или определить более сложные правила, основанные не только на доменах, но и на URL, можно использовать страницу настроек **Application detection**.

URL-адреса, используемые для обнаружения приложений, имеют структуру `scheme://host:port/path?query`, где строка запроса необязательна, а порты по умолчанию `80` для HTTP и `443` для HTTPS опускаются. URL не включает возможный идентификатор фрагмента, как в `scheme://host:port/path?query#fragment`, поскольку правила обнаружения приложений вычисляются на серверной стороне приложения, а идентификатор фрагмента используется только браузером и не добавляется к веб-запросам.

Добавление правила обнаружения приложения

1. Перейти в **Settings** > **Web and mobile monitoring** > **Application detection**. Отображается список правил обнаружения приложений.  
   Для каждого приложения, определённого с помощью [рекомендуемого подхода](#automated-approach), автоматически создаётся правило обнаружения, которое добавляется в конец списка. Правила применяются последовательно, при этом правила в верхней части списка имеют приоритет над правилами, расположенными ниже.
2. Выбрать **Add detection rule**.
3. Использовать параметры, предложенные на странице, чтобы создать подходящее правило обнаружения для приложения.  
   Правила можно применять к новым или существующим приложениям, а также создавать их на основе URL приложения или домена (хоста).
4. Выбрать **Save**, чтобы создать правило обнаружения приложения.  
   Правило размещается в самом конце списка правил обнаружения приложений.
5. Необязательно: выбрать и удерживать **Drag row** ![Drag handle](https://dt-cdn.net/images/drag-handle-turquoise-600-1aa0e5ea00.svg "Drag handle") рядом с именем правила и переместить правило вверх или вниз по списку, чтобы изменить его приоритет.

В одной среде можно создать до 1000 правил обнаружения приложений.

### Возможные проблемы для подхода на основе правил обнаружения приложений

**Неинструментированный компонент переписывает URL**. Если неинструментированный компонент между браузером и первым инструментированным уровнем переписывает URL, нужно учесть особые обстоятельства (подробнее см. в разделе [Что можно сделать, если неинструментированный компонент переписывает части URL?](#rum-appdetection-uninstrumentedcomponent)). Иначе OneAgent может не суметь применить правила обнаружения приложений.

**Действия по-прежнему сопоставляются с предыдущим приложением**. Если пользовательские действия по-прежнему сопоставляются с предыдущим приложением вместо нужного, см. раздел [Как быстро вступают в силу изменения правил обнаружения приложений?](#rum-appdetection-rule-changes).

### Часто задаваемые вопросы для рекомендуемого подхода и подхода на основе правил обнаружения приложений

Как Dynatrace применяет правила обнаружения приложений?

Правила обнаружения приложений вычисляются на серверной стороне приложения OneAgent на первом инструментированном уровне, ближайшем к браузеру. Результат передаётся последующим OneAgent с помощью заголовка `x-dynatrace-application`. При таком подходе любое переписывание URL, которое может происходить между инструментированными серверными уровнями приложения, не влияет на результат обнаружения приложения.

OneAgent добавляет параметр `app` в RUM JavaScript, внедряемый в HTML-код приложения. Этот параметр содержит идентификатор обнаруженного приложения.

![App parameter in the RUM JavaScript](https://dt-cdn.net/images/injected-app-parameter-2505-7ee0111d11.png)

Параметр app в RUM JavaScript

При отправке beacon-запросов с зафиксированными пользовательскими действиями RUM JavaScript включает этот параметр `app` в строку запроса.

![App parameter added to the query string](https://dt-cdn.net/images/app-parameter-in-beacon-query-1922-30c91adb9b.png)

Параметр app, добавленный в строку запроса

В кластере Dynatrace параметр `app` в строке запроса beacon используется для назначения пользовательских действий приложениям.

Проверка идентификатора приложения для сравнения с параметром app

1. В Dynatrace перейти в раздел **Web**.
2. Выбрать приложение.
3. Проверить URL текущей страницы. Этот URL содержит параметр `uemapplicationId`, например `uemapplicationId=APPLICATION-4CC1C5694BF9B3AB`. Его значение без префикса `APPLICATION-` и есть искомый идентификатор приложения.

   ![Checking the application ID in the Dynatrace web UI](https://dt-cdn.net/images/application-id-2559-b6b9f31629.png)

   Проверка идентификатора приложения в веб-интерфейсе Dynatrace

Что можно сделать, если неинструментированный компонент переписывает части URL?

Как объяснено в разделе [Как Dynatrace применяет правила обнаружения приложений?](#rum-appdetection-appparam), правила обнаружения приложений вычисляются OneAgent на первом инструментированном уровне приложения. Однако между браузером и первым инструментированным уровнем приложения может находиться неинструментированный компонент, например обратный прокси или балансировщик нагрузки, который переписывает URL. В результате URL, используемый для обнаружения приложения, отличается от URL, изначально запрошенного браузером. В этом случае нужно учесть особые обстоятельства.

### Хост переписывается

Компоненты вроде прокси и балансировщиков нагрузки можно настроить так, чтобы они передавали информацию об исходном хосте в бэкенд через заголовок запроса, например, в заголовке [`X-Forwarded-Host`﻿](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/X-Forwarded-Host) или [`Forwarded`﻿](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Forwarded). OneAgent может получить исходный хост из одного из этих заголовков или из любого собственного заголовка, имеющего тот же формат, что и `X-Forwarded-Host`.

![Passing the original host information in the X-Forwarded-Host header](https://dt-cdn.net/images/host-name-determination-1093-9025d73d9b.png)

Передача информации об исходном хосте в заголовке X-Forwarded-Host

Как использовать исходное имя хоста при определении приложения

1. Настроить неинструментированный компонент так, чтобы он добавлял заголовок, передающий исходное имя хоста (если он не делает этого по умолчанию).  
   Например, если исходное имя хоста, запрошенное браузером, было `www.example.com`, а неинструментированный компонент перенаправил запрос на внутренний хост `backend.com`, то компонент должен добавить заголовок запроса, передающий `www.example.com`, как показано на диаграмме последовательности выше.
2. В Dynatrace перейти в **Settings** > **Web and mobile monitoring** > **Host name determination**.
3. Если заголовок, добавленный компонентом, отсутствует в списке, выбрать **Add HTTP header**.
4. Заголовки запроса в списке обрабатываются последовательно, приоритет у тех, что находятся выше по списку. Нужно переместить новую запись выше заголовка `Host`, который входит в каждый запрос HTTP/1.1.

Также можно использовать внутреннее имя хоста в правиле определения приложения. Однако OneAgent использует результат определения имени хоста не только для определения приложения, но и для [автоматического определения домена cookie](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/configure-cookie-attributes#default-cookie-domain "Learn which RUM cookie attributes you can configure and how."). Поэтому, если определение имени хоста не настроено, как описано выше, домен cookie нужно настроить вручную, как описано в разделе [Ручная настройка домена cookie](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/configure-cookie-attributes#manual-cookie-domain-config "Learn which RUM cookie attributes you can configure and how."). Рекомендуемый вариант, это [eTLD+1﻿](https://web.dev/same-site-same-origin/) домена, который OneAgent определяет автоматически.

### Путь URL перезаписывается

В отличие от части хоста в URL, функции определения исходного пути URL не существует, поскольку для компонентов нечасто встречается практика устанавливать заголовки, которые сообщают исходный путь последующим уровням. Поэтому часть пути URL, удалённую в результате перезаписи URL на неинструментированном компоненте, нельзя использовать в правиле определения приложения.

Как быстро вступают в силу изменения правил определения приложения?

При изменении правил определения приложения обновлённые правила обычно доводятся до OneAgent в течение минуты. OneAgent классифицирует входящие запросы согласно настроенным правилам определения приложения и соответствующим образом задаёт параметр `app` во внедрённом RUM JavaScript. Затем RUM JavaScript передаёт свои маяки (beacons) с использованием обновлённого параметра `app`, который определяет, как классифицируются захваченные действия пользователя. Подробнее см. в разделе [Как Dynatrace применяет правила определения приложения?](#rum-appdetection-appparam).

Следующие факторы могут задержать вступление в силу обновлённых правил определения приложения:

* HTML-код обслуживается через CDN или другой кеш. В этом случае OneAgent может внедрить RUM JavaScript только тогда, когда HTML-код будет вытеснен из кеша и запрошен заново у инструментированного исходного сервера.
* Приложение указывает длительное время истечения срока действия для HTML-документа с помощью заголовка ответа `Expires` или директивы `max-age` заголовка ответа `Cache-Control`. Кеш может обслуживать документ столько времени, сколько указано в этих заголовках, включая внедрённый параметр `app`. Изменение в правилах определения приложения не может отразиться немедленно, поскольку кеш не отправляет запрос к исходному серверу, пока срок действия документа не истёк.
* Если приложение не указывает явное время истечения срока действия, это не значит, что страница не может кешироваться (см. [RFC7234: Calculating heuristic freshness﻿](https://datatracker.ietf.org/doc/html/rfc7234#section-4.2.2)). Кеш может использовать страницу без повторной проверки столько, сколько сочтёт нужным, если только ответ не содержит директив, которые это запрещают, например, `Cache-Control: no-cache`.
* Одностраничные приложения перезаписывают текущую страницу динамически вместо загрузки новой страницы с сервера. Если HTML-код был загружен во вкладку браузера до изменения правил определения приложения, то классификация захваченных действий пользователя XHR ещё не отражает обновлённую конфигурацию. Это меняется, как только страница обновляется.

## Подход agentless RUM

Если нет доступа к веб-серверу, и поэтому нельзя установить OneAgent на хост, но есть доступ к коду приложения, следует выбрать [agentless RUM](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/set-up-agentless-real-user-monitoring "Set up agentless monitoring for your web applications.").

Для некоторых технологий [автоматическое внедрение RUM JavaScript](/managed/observe/digital-experience/rum-classic/web-applications/initial-setup/rum-injection "Configure automatic injection of the RUM JavaScript into the pages of your applications") не поддерживается, даже если OneAgent можно установить. Например, OneAgent может отслеживать серверную часть приложения Heroku, но не может внедрить RUM JavaScript в страницы приложения. В этом случае нужно использовать agentless RUM и вручную добавить RUM JavaScript на страницы приложения.

Список технологий и серверов, поддерживающих автоматическое внедрение RUM JavaScript, см. в разделе [Technology support - Real User Monitoring - Web servers and applications](/managed/ingest-from/technology-support#rum-auto-injection "Find technical details related to Dynatrace support for specific platforms and development frameworks.").

## Рекомендуемые практики и советы

* Определяйте приложения на основе владения командой, чтобы удобно использовать [management zones](/managed/manage/identity-access-management/permission-management/management-zones "Learn about management zones concepts, how to define management zones, and how to make the most of them.") для ограничения доступа.
* Имеет смысл определять приложения на основе их технологического стека, чтобы применялись правильные настройки и упрощалось управление конкретными параметрами. Например, включение поддержки конкретных XHR-фреймворков обычно требуется только для отдельных частей большого приложения и для конкретных правил именования пользовательских действий. Разделение таких больших приложений на более мелкие, исходя из используемых технологий или владения командой, может помочь.
* Placeholder-приложение **My web application**

  + Не рекомендуется переименовывать placeholder-приложение **My web application**, поскольку оно включает все пользовательские действия на всех доменах, не попавших ни в одно правило распознавания приложений. Если переименовать **My web application**, его будет сложно отличить от остальных приложений.
  + Удалить это placeholder-приложение невозможно.
  + Если не удаётся найти placeholder-приложение **My web application**, возможно, кто-то его переименовал. Чтобы найти его, откройте любое другое приложение и замените ID приложения в URL на `EA7C4B59F27D43EB`. Произойдёт переход на **My web application**.
  + В новых окружениях это placeholder-приложение появляется только после того, как начинает поступать трафик от автоматически внедрённого (auto-injected) приложения.
* Разделение приложений по доменам работает лучше всего, поскольку Dynatrace не может сопоставить пользовательские действия на разных доменах с конкретными пользовательскими сессиями. Это сопоставление выполняется через cookie и поэтому работает только в том случае, если cookie можно установить на одном и том же домене. Например, пользовательские действия для `www.dynatrace.com` и `blog.dynatrace.com` можно фиксировать в рамках одного приложения, поскольку cookie можно установить для `dynatrace.com`. Однако трафик для `www.dynatrace.com` и `www.internal-dynatrace.com` нельзя зафиксировать в рамках одной пользовательской сессии. Разделять пользовательские действия по домену всё равно можно, но пользовательские сессии не могут включать действия сразу с нескольких доменов. Подробнее см. [User sessions in RUM Classic](/managed/observe/digital-experience/rum-classic/rum-concepts/user-session "Learn how a user session is defined, when a user session starts or ends, how user session duration is calculated, and more.").
* Группируйте приложения с низким трафиком. Если создать приложение на основе домена, у которого меньше 10 действий в минуту, Dynatrace не будет автоматически обнаруживать аномалии для этого нового приложения. Dynatrace полагается на стабильный трафик приложения, чтобы правильно [строить многомерные базовые линии и автоматически сообщать о проблемах приложения](/managed/dynatrace-intelligence/anomaly-detection/static-thresholds-infrastructure "Learn about the fixed thresholds used by Dynatrace to determine when a detected slowdown or error-rate increase justifies the generation of a new problem event."). Настройки обнаружения аномалий приложения можно изменить на странице **Anomaly detection** в настройках приложения. Хотя эта рекомендация противоречит указанной выше, объединение приложений с низким трафиком иногда оправдано.
* Правила распознавания приложений обрабатываются последовательно для каждого запроса. Чем больше правил, тем больше времени на обработку, а поскольку правила обрабатываются внутри OneAgent, правила для приложений с наибольшим трафиком стоит размещать в начале списка. Как только одно правило совпадает, дальнейшая обработка запроса не выполняется.
* Более специфичные правила распознавания приложений следует определять первыми, а более общие правила должны находиться в конце списка правил распознавания приложений.  
  Предположим, нужно создать приложение с именем `A`, на которое будут сопоставлены следующие два домена:

  `http://www.mydomain.com/cms/directory/shop/index.html`  
  `http://shop.differentdomain.de/index.html`

  и ещё одно приложение `B` для следующего домена:

  `http://newdomain.co.uk/hello/shop.html`

  Если создать общее правило группировки на основе значения `shop`, все три домена будут сгруппированы в одно и то же веб-приложение для мониторинга. Поэтому сначала нужно определить более специфичное правило, например «если URL заканчивается на `shop.html`», чтобы только третий URL сопоставлялся с приложением `B`. После этого можно спокойно определить общее правило на основе значения `shop`, поскольку третий URL уже будет сопоставлен с предыдущим приложением и, соответственно, не попадёт в приложение `A`.
* В зависимости от требований можно скорректировать потребление мониторинга и настроить Real User Monitoring соответствующим образом.
* Для мониторинга трафика в рамках одного приложения можно использовать [user actions and session properties](/managed/observe/digital-experience/rum-classic/web-applications/additional-configuration/define-user-action-and-session-properties "Define custom string, numeric, and date properties for your monitored web applications."). Для мониторинга отдельных приложений и получения полной картины по потреблению для каждого приложения можно настроить отдельные приложения, использовать теги для разделения метрик и определить management zones.
