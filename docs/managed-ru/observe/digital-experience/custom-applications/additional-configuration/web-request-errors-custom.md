# Игнорирование ошибок веб-запросов в пользовательских приложениях

* How-to guide
* 1-min read
* Published Jan 30, 2023

По умолчанию Dynatrace помечает все коды ответа `4xx` и `5xx` как [ошибки](/managed/observe/digital-experience/rum-concepts/user-and-error-events#error "Узнайте о событиях пользователей и ошибках, а также о типах событий, фиксируемых Dynatrace.") веб-запросов. Однако некоторые неуспешные HTTP-коды могут не указывать на проблему с вашим приложением. Например, код `401 Unauthorized` может просто означать, что пользователь решил не входить в систему. Другой пример — код `429 Too Many Requests`, который может указывать на проблемы с ограничением частоты запросов, не вызванные вашим приложением.

Чтобы игнорировать определённые HTTP-коды состояния, настройте, какие из них не должны помечаться как ошибки. При создании правила исключения можно указать как отдельный HTTP-код, например `401`, так и диапазон кодов, например `400-499`.

Чтобы создать правило исключения:

1. Перейдите в **Frontend**.
2. Выберите приложение, которое нужно настроить.
3. Нажмите **More** (**…**) > **Edit** в правом верхнем углу плитки с именем вашего приложения.
4. В настройках приложения выберите **Request errors**.
5. Нажмите **Add exclusion rule** и укажите HTTP-код состояния или диапазон кодов, которые следует считать успешными.

   ![Настройка правила исключения](https://dt-cdn.net/images/mobile-exclusion-rules-2124-75133d34c8.png)

   Настройка правила исключения

После настройки правила исключения веб-запросы с ранее ошибочными кодами состояния будут учитываться как успешные при расчёте частоты ошибок. Это повлияет на рейтинг Apdex приложения, частоту ошибок запросов, список ошибок веб-запросов, а также на некоторые метрики действий пользователей и пользовательские метрики.

Например, на скриншотах страницы обзора приложения ниже виден спад частоты ошибок веб-запросов. Это произошло после настройки правила игнорирования кода ответа `401 Unauthorized` для данного приложения. Также обратите внимание на снижение частоты ошибок у топ-провайдеров приложения, а также на более высокий рейтинг Apdex и меньшую частоту ошибок для одного из пользовательских действий.
