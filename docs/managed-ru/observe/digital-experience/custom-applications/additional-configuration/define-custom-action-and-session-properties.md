# Определение свойств действий и сессий пользователей в пользовательских приложениях

# Определение свойств действий и сессий пользователей в пользовательских приложениях

* How-to guide
* 1-min read
* Published Jan 27, 2022

Dynatrace собирает большой объём информации о производительности ваших приложений. Вы можете дополнить эту информацию ценными метаданными и преобразовать их в свойства действий и сессий пользователей.

Свойства действий и сессий — это метаданные в формате «ключ-значение», по которым можно выполнять фильтрацию в различных аналитических представлениях Dynatrace. Эти свойства удобны для создания сложных запросов, сегментаций или агрегаций на основе захваченных метаданных. Их можно использовать на страницах **User sessions** и **User sessions query**. Подробнее о том, как использовать эти свойства, см. [Использование свойств действий и сессий пользователей в пользовательских приложениях](/managed/observe/digital-experience/custom-applications/analyze-and-use/action-and-session-properties-custom "Свойства действий и сессий — метаданные в формате «ключ-значение», обеспечивающие дополнительную видимость и более глубокий анализ взаимодействия конечных пользователей.").

Ниже приведена информация о настройке таких свойств и связанные примеры конфигурации. Чтобы использовать свойства действий и сессий, необходимо сначала отправить нужные метаданные в Dynatrace, а затем [добавить свойства в веб-интерфейсе Dynatrace](#add-properties).

## Отправка метаданных в Dynatrace

Перед определением свойств действий и сессий необходимо начать передачу нужных метаданных в Dynatrace. Это можно сделать двумя способами:

* [Передача пользовательских значений через SDK](#report-value-sdk)
* [Определение атрибутов запроса](#add-request-attribute)

### Передача значения через SDK

Вы можете передавать метаданные в формате «ключ-значение» в исходном коде вашего пользовательского приложения через вызов API. Подробнее см. [Методы API Dynatrace OpenKit | Передача значения](/managed/ingest-from/extend-dynatrace/openkit/dynatrace-openkit-api-methods#report-values "Узнайте, как использовать Dynatrace OpenKit с точки зрения разработчика.").

Чтобы проверить, достигают ли данные, переданные в коде вашего приложения, Dynatrace, перейдите в детали пользовательского действия, которое должно содержать эти данные, и прокрутите вниз до раздела **Reported values**.

![Страница деталей пользовательского действия со значениями, переданными через SDK](https://dt-cdn.net/images/user-action-details-with-reported-values-2048-b44e8bca3e.png)

Страница деталей пользовательского действия со значениями, переданными через SDK

### Определение атрибута запроса

Кроме того, вы можете добавить нужные метаданные к серверным запросам, определить атрибуты запросов и использовать их для создания свойств действий и сессий.

Атрибуты запросов извлекаются из URL веб-запросов, HTTP-заголовков запросов и других метаданных запросов. Эти атрибуты представляют собой пары «ключ-значение», по которым можно выполнять фильтрацию во многих аналитических и distributed tracing-представлениях Dynatrace.

Подробнее см. [Атрибуты запросов](/managed/observe/application-observability/services/request-attributes "Узнайте, что такое атрибуты запросов и как их использовать во всех аналитических представлениях сервисов.").

## Добавление свойств действий и сессий

После передачи значений через код приложения или определения атрибутов запросов информация, захваченная Dynatrace, обогащается ценными метаданными. Вы можете «повысить» эти метаданные до свойств действий и сессий в настройках приложения.

Данные можно сохранять на разных «уровнях хранения»:

* **User action property**: данные сохраняются в определённом свойстве на уровне каждого действия пользователя, в котором Dynatrace может получить метаданные.
* **User session property**. Последнее захваченное значение сохраняется в определённом свойстве на уровне сессии.
* **Both options**: данные сохраняются на обоих уровнях.

Чтобы определить свойство действия или сессии:

1. Перейдите в **Frontend** и выберите приложение для настройки.
2. Нажмите **More** (**…**) > **Edit**.
3. В настройках приложения выберите **Session and user action properties** и нажмите **Add property**.
4. Выберите **Expression type** и при необходимости **Data type**.

   * Если вы добавили метаданные в запросы и [определили атрибуты запросов](#add-request-attribute), выберите **Server-side request attribute**.
   * Если вы [передавали пользовательские значения через SDK](#report-value-sdk), выберите **SDK reported value**.
5. Укажите **Name** или **Request attribute name**, **Display name** и **Key**.

   | Параметр | Пояснение |
   | --- | --- |
   | Name / Request attribute name | Для типа SDK reported value — это имя переданного значения из кода вашего приложения. **Name** чувствителен к регистру и должен точно совпадать с записью в коде. Для типа server-side request attribute — это имя атрибута запроса, заданного в **Settings** > **Server-side service monitoring** > **Request attributes**. |
   | Display name | Имя свойства, отображаемое в веб-интерфейсе Dynatrace, например на странице деталей сессии или действия. |
   | Key | Имя свойства, используемое для его идентификации и последующего поиска в [USQL](/managed/observe/digital-experience/session-segmentation/custom-queries-segmentation-and-aggregation-of-session-data "Узнайте, как получать доступ к данным сессий и выполнять запросы."). |
6. Выберите хотя бы один [тип хранения](#storage-levels): свойство действия пользователя, свойство сессии или оба.
7. Для типа хранения свойства сессии выберите один из типов агрегации.
8. Необязательно: чтобы ограничить захватываемые значения, включите **Apply cleanup rule** и укажите [регулярное выражение](/managed/manage/tags-and-metadata/reference/regular-expressions-in-dynatrace "Узнайте, как использовать регулярные выражения в контексте Dynatrace.").

## Примеры свойств действий и сессий

Ниже приведены примеры нескольких свойств действий и сессий, настроенных для примерного веб-приложения easyTravel.

Journey ID

Member status

Total purchase amount

Credit card

![Journey ID](https://dt-cdn.net/images/property-journey-id-2136-1e20f89c85.png)

Journey ID

Свойство **Journey ID** с типом данных `long` добавляется к каждому действию и сессии пользователя в приложении easyTravel. Для сессии сохраняется значение последнего действия.

![Свойство Member status](https://dt-cdn.net/images/property-member-status-2136-1665ea176c.png)

Свойство Member status

Свойство **Member status** фиксирует статус участия в программе лояльности для всей сессии. В данном случае отдельные действия пользователя не важны, так как статус лояльности обычно остаётся неизменным на протяжении всей сессии.

![Свойство Purchase amount](https://dt-cdn.net/images/property-purchase-amount-2136-8a92eff1b1.png)

Свойство Purchase amount

Свойство **Total purchase amount** представляет суммарную стоимость всех туров, забронированных через приложение easyTravel. Обратите внимание: это свойство сессии является суммой значений `purchase_amount` из нескольких пользовательских действий.

![Свойство Credit card type](https://dt-cdn.net/images/property-credit-card-type-2136-8a271f7d47.png)

Свойство Credit card type

Свойство **Credit card type** — пример «повышения» серверного атрибута запроса до свойств действия и сессии.

## Ограничения

* Максимальное количество свойств на одно приложение — 200.
* Максимальное количество свойств действия на одно приложение — 20.
* Свойства действий и сессий типа `String` ограничены 100 символами после применения правила очистки.
* До 20 свойств на приложение предоставляются бесплатно. Дополнительные свойства потребляют DEM-единицы.
  См. [DEM-единицы](/managed/license/monitoring-consumption-classic/digital-experience-monitoring-units#dem-units "Узнайте, как рассчитывается потребление Dynatrace Digital Experience Monitoring на основе DEM-единиц.") (обратите внимание на записи **Session property** и **Action property** в таблице) и [Бесплатный уровень свойств действий и сессий](/managed/license/monitoring-consumption-classic/digital-experience-monitoring-units#free-action-and-session-properties "Узнайте, как рассчитывается потребление Dynatrace Digital Experience Monitoring.").

## Примечания

* Dynatrace начинает захватывать свойства действий и сессий только после того, как вы [определите их в настройках приложения](#add-properties).
* Вы можете проверить, сколько свойств уже используется и сколько ещё можно добавить. Перейдите в настройки приложения > **Session and user action properties** и прокрутите вниз до раздела **Property usage quotas**.
