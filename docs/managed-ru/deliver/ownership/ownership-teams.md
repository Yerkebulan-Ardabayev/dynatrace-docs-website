# Create and manage teams for entity ownership

# Create and manage teams for entity ownership

* How-to guide
* 8-min read
* Updated on Nov 07, 2023

Владение отслеживаемыми сущностями определяется в рамках **команд**. Вы можете настраивать команды-владельцы в веб-интерфейсе, через API и с помощью [Configuration as Code](/managed/deliver/configuration-as-code "Use Dynatrace configuration as code via Monaco or Terraform."). Каждая команда имеет **уникальный идентификатор**, который является основой для [применения владения к сущностям](/managed/deliver/ownership/assign-ownership "Assign owners to entities using entity metadata like labels, environment variables, and tags.") различными методами (метки и аннотации Kubernetes, метаданные хоста, переменные окружения или теги). Помимо этого, определения команд включают имена, описания, важную информацию маршрутизации для уведомлений (через Microsoft Teams, Slack, Jira и email), обязанности (например, эксплуатация или безопасность) и дополнительные полезные ссылки.

Настройки команд основаны на [фреймворке Settings 2.0](/managed/manage/settings/settings-20 "Introduction to the Settings 2.0 framework"), который определяет каждую конфигурацию команды как объект настроек на основе схемы владения — независимо от того, используете ли вы API, Configuration as Code, импортируемые данные из сторонних баз данных или веб-интерфейс по пути **Settings** > **Ownership**.

Информация о владении, прикреплённая к отслеживаемым сущностям, помогает сопоставить ваше окружение Dynatrace с нужными командами для совместной работы, устранения проблем и эскалации уязвимостей.

## Разрешения

Для создания, редактирования или удаления команды через API необходимо **любое** из следующих разрешений.

* Разрешение токена `settings.write` [token permission](/managed/manage/identity-access-management/access-tokens-and-oauth-clients/access-tokens "Learn the concept of an access token and its scopes.")
* [Политика IAM](/managed/manage/identity-access-management/permission-management/manage-user-permissions-policies "Working with policies") `ALLOW settings:objects:write, settings:schemas:read WHERE settings:schemaId = "builtin:ownership.teams";`

Для создания команд через веб-интерфейс необходимо разрешение **Manage monitoring settings** [permission](/managed/manage/identity-access-management/permission-management/role-based-permissions "Role-based permissions") на уровне окружения или зоны управления.

## Создание команды через веб-интерфейс

Для создания команды через веб-интерфейс Dynatrace

1. Перейдите в **Settings** > **Ownership** > **Teams**.

   ![Ownership teams settings page](https://dt-cdn.net/images/ownership-teams-page-2212-be3abe3c7d.png)

   Страница настроек команд владельцев
2. Выберите **Add team**.
3. Обязательно Начните с указания **Team name**.
4. Обязательно **Team identifier** заполняется автоматически на основе имени команды. Используйте идентификатор по умолчанию или задайте свой.

   **Требования**

   Идентификаторы команд являются основой для [применения владения к сущностям](/managed/deliver/ownership/assign-ownership "Assign owners to entities using entity metadata like labels, environment variables, and tags.") независимо от метода. Основные идентификаторы команд должны быть уникальными в вашем окружении. Однако можно повторно использовать идентификатор ранее удалённой команды.

   * Идентификаторы должны быть уникальными; два основных идентификатора команды не могут совпадать.
   * Идентификаторы должны быть от 1 до 100 символов в длину.
   * Они должны начинаться и заканчиваться буквой (`[a-zA-Z]`).
   * Между первым и последним символами могут содержать дефисы (`-`), символы подчёркивания (`_`) и буквенно-цифровые символы. Пробелы и другие специальные символы не поддерживаются.
   * После создания **основной идентификатор команды не может быть изменён**.
5. Добавьте **Description**, помогающее отличить команду, например, по управляемым сущностям или по обязанностям. Это описание добавляется к имени команды при отображении на странице **Ownership teams**.
6. При необходимости укажите **Supplementary identifiers**. Выберите **Add supplementary identifier** для добавления до трёх дополнительных идентификаторов, например, для идентификации подкоманд. При назначении владения сущностям можно использовать основной идентификатор команды или любой из дополнительных идентификаторов для назначения сущности именованной команде.

   * Дополнительные идентификаторы должны соответствовать тем же требованиям к символам, что и основной ID команды.
   * Дополнительный идентификатор не может совпадать с основным идентификатором той же команды.
   * В рамках команды дополнительные идентификаторы должны быть уникальными.
   * Дополнительный идентификатор может совпадать с основным или дополнительным идентификатором другой команды.
   * В отличие от основного идентификатора команды, дополнительные идентификаторы **можно изменять и удалять**.
7. Выберите все **Responsibilities**, применимые к команде. Хотя это необязательно, эти переключатели предоставляют критически важные метаданные с первого взгляда и помогают указать соответствующие контактные данные.

   * **Development** — команды, ответственные за разработку и поддержку базового программного кода
   * **Security** — команды, оценивающие влияние и приоритет уязвимостей и реагирующие на них
   * **Operations** — команды, развёртывающие программное обеспечение и управляющие им с акцентом на производительность и доступность
   * **Infrastructure** — команды, ответственные за установку и обслуживание IT-оборудования
   * **Line of business** — команды, ответственные за достижение целей клиентов и бизнеса
8. Добавьте все **Contact details**, применимые к команде. Вы можете предоставить конкретную информацию маршрутизации для целевых уведомлений через email, Jira, Microsoft Teams и Slack.

   Можно добавить столько записей, сколько необходимо. Создавайте отдельную запись для каждого email-адреса, Slack-канала, Microsoft Teams или проекта Jira.

   1. Выберите **Add contact information**.
   2. Выберите **Integration type**.
   3. Укажите сведения:

      * **Email** — укажите один **Email**-адрес.
      * **Jira** — введите **Project** (название проекта Jira), **Default assignee** Обязательно и **URL** проекта.
      * **MS Teams** — укажите **Team** (название команды) и **URL** (ссылку на канал команды из MS Teams).
      * **Slack** — укажите **Channel** (название канала) и **URL**.
9. Укажите любые **Links**, описывающие команду или её обязанности.
10. Выберите **Add link**.
11. Выберите тип ссылки **Type**, например **Documentation** или **Wiki**.
12. Укажите **URL**.

    Можно добавить столько записей, сколько необходимо. Создавайте отдельную запись для каждой ссылки.
13. Выберите **Add additional information** для добавления дополнительных пар ключ-значение в поля **Name** и **Value**, например, названия центров затрат или информацию об организационной иерархии. Также можно указать необязательный **URL**.

    Ключи для дополнительной информации:

    * Должны быть от 1 до 100 символов в длину.
    * Могут содержать буквы, цифры, специальные символы и пробелы.
14. Укажите **External ID** для использования исключительно в целях автоматизации, например при импорте или обновлении информации о команде. Например, используйте это поле для хранения уникального ID, присвоенного команде во внешней системе, такой как Active Directory.

    * Внешний ID можно указать только при создании команды.
    * После создания внешний ID **не может быть изменён**.
    * Строка может начинаться, заканчиваться и содержать специальные символы, пробелы, цифры и буквы.

    * Внешние ID нельзя использовать для применения информации о владении к сущностям в парах ключ-значение; для назначения сущностей командам-владельцам используйте основной ID команды и дополнительные ID.
15. **Save changes**.

Подробнее об оптимальной настройке команд см. в разделе [Рекомендации по владению сущностями](/managed/deliver/ownership/best-practices "Tips and best practices to ensure that entities have adequate ownership coverage").

## Создание команды через API и Configuration as Code

Для создания команды через [**Settings API**](/managed/dynatrace-api/environment-api/settings "Find out what the Dynatrace Settings API offers.") требуется [JSON-схема](/managed/dynatrace-api/environment-api/settings/schemas/get-schema "View a settings schema via the Dynatrace API.") конфигурации владения (`builtin:ownership.config`), предоставляемая Dynatrace. Схему можно получить с помощью Settings API. Конфигурацию команды следует передавать в виде JSON-payload.

[Configuration as Code](/managed/deliver/configuration-as-code "Use Dynatrace configuration as code via Monaco or Terraform.") позволяет использовать JSON-шаблоны и YAML-файлы для создания одной или нескольких команд одновременно.

## Управление командами

Управлять (создавать, удалять, редактировать) командами-владельцами можно через **Settings** > **Ownership** > **Teams**, а также через [Settings API](/managed/dynatrace-api/environment-api/settings "Find out what the Dynatrace Settings API offers.").

На странице настроек **Ownership teams** перечислены все команды (и их описания) в вашем окружении, созданные как через веб-интерфейс, так и через API. Используйте поле **Filter** для сужения списка по любой строке в имени или описании команды.

Из этого представления можно **Delete** команду, а также раскрыть **Details** для просмотра и редактирования информации о команде. Однако основной идентификатор команды изменить или удалить нельзя.

### История изменений

Выберите ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More") в правом верхнем углу страницы **Ownership teams**, затем выберите **Revision history** для просмотра деталей изменений, включая создание, изменение и удаление команд. Изменения перечислены с **Timestamp** (временной меткой), **Source** (источником изменения — **Web UI** или **API**) и **User** (пользователем, вносившим изменение).

![History of ownership teams](https://dt-cdn.net/images/ownership-teams-revisions-1462-e55719add4.png)

История команд владельцев

Разверните для просмотра **Details** любого изменения. Изменения выделены цветом для отображения **Original value** и **New value**. На изображении ниже показан удалённый дополнительный идентификатор.

![Revision simple view](https://dt-cdn.net/images/ownership-teams-revision1-1849-055f7f5d2e.png)

Простой вид изменения

**Переключитесь в расширенный режим** и выберите **Show diff** для просмотра тех же изменений в виде JSON-объектов. **Key** в расширенном режиме — это уникальный ID настройки, присвоенный каждой команде. Выберите ключ для просмотра сведений о соответствующей команде.

![Revision YAML view](https://dt-cdn.net/images/ownership-teams-revision2-1848-06b3770890.png)

Вид изменения в формате YAML

Также можно фильтровать историю изменений для конкретной команды — вставьте любой **Key** ID настройки в поле **Filter**.

### Дополнительное управление командами

Для каждой команды предусмотрены дополнительные удобства управления.

1. Разверните сведения о команде на странице **Ownership teams**.
2. Выберите ![More](https://dt-cdn.net/images/more-icon-01c8b008ca.svg "More") в правом верхнем углу сведений о команде для доступа к [**Revision history**](#revision-history) для конкретной команды, полезным фрагментам **API** для управления командой и возможности **Duplicate** (дублировать) сведений о команде как основы для создания новой.

   ![Team-management links](https://dt-cdn.net/images/ownership-team-specific-links-1462-4d8ea5d2df.png)

   Ссылки управления командой

Фрагменты API для чтения, создания, обновления и удаления команд доступны как для Windows, так и для Linux. Хотя фрагменты API привязаны к конкретной команде, их можно изменять, например, удалив идентификатор команды для создания новой. Обязательно укажите свой токен доступа.

![Team-management API snippets](https://dt-cdn.net/images/ownership-teams-api-snippets-763-e9a0f5eb88.png)

Фрагменты API для управления командой
