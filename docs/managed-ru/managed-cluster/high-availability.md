# Premium High Availability

# Premium High Availability

* Explanation
* 3-min read
* Published Mar 23, 2022

Dynatrace Managed поддерживает развёртывания с высокой доступностью в одном или нескольких центрах обработки данных. Такие развёртывания состоят из нескольких равнозначных узлов, выполняющих одни и те же сервисы.

Для оптимальной конфигурации переключения при отказе следуйте приведённым ниже рекомендациям.

## Избыточность

Планируйте развёртывание минимум трёх узлов Managed Cluster. Все узлы автоматически реплицируют данные между собой, поэтому, как правило, существует два дополнительных реплика помимо первичного шарда. Задержка между узлами должна составлять около 10 мс или менее.

Вся конфигурация Managed Cluster и его окружений (включая все события, пользовательские сессии и метрики) распределена по всем узлам кластера, при этом поддерживаются две копии данных. Это позволяет Dynatrace продолжать полноценную работу даже после потери узла. Производительность и доступность Managed Cluster могут снизиться при потере двух и более узлов — в зависимости от распределения данных и требуемого уровня согласованности.

Данные событий [Log Monitoring](/managed/analyze-explore-automate/log-monitoring "Learn how to enable Log Monitoring, the insights that Log Monitoring can provide, and more.") реплицируются в хранилище Elasticsearch для обеспечения высокой доступности и оптимизации затрат на хранилище. Таким образом, если один узел выходит из строя, Dynatrace располагает копией данных на другом узле. Однако сбой двух узлов делает часть событий лога недоступными. После восстановления узлов данные снова становятся доступными; в противном случае данные теряются.

Необработанные данные транзакций (стеки вызовов, запросы к базе данных, трассировка уровня кода и т.д.) не реплицируются между узлами — они равномерно распределяются по всем узлам. Поэтому при отказе узла Dynatrace может точно оценить недостающие данные. Это возможно потому, что такие данные, как правило, недолговечны, а большой объём необработанных данных, собираемых Dynatrace, гарантирует достаточный набор данных на каждом узле даже при временной недоступности одного из них.

Если требуется региональная отказоустойчивость (когда все узлы Managed Cluster в одном территориальном домене могут выйти из строя), распределяйте узлы по отдельным физическим площадкам, используя один из следующих вариантов:

* Два территориальных расположения можно реализовать только с [высокой доступностью для нескольких центров обработки данных](/managed/managed-cluster/high-availability/premium-high-availability "Learn how Dynatrace Premium High Availability provides near-zero downtime and data resilience through multi-data-center deployments for Managed Clusters.").
* Три расположения с низкой задержкой можно реализовать с помощью [rack-aware развёртывания Managed](/managed/managed-cluster/high-availability/rack-awareness "Understand the steps required to create a rack-aware Dynatrace Managed deployment.").
* Шесть расположений можно реализовать, сочетая высокую доступность для нескольких центров обработки данных и rack-aware развёртывание в каждом центре.

Replication factor равный трём гарантирует, что в каждом расположении хранятся все данные метрик и событий.

Для установок Dynatrace Managed, развёрнутых в глобально распределённых центрах обработки данных (с задержкой выше 10 мс), требуется Premium High Availability — решение с полностью автоматическим переключением при отказе в случае полного отказа центра обработки данных. Оно расширяет существующие возможности высокой доступности Dynatrace Managed, обеспечивая географическую избыточность для глобально распределённых предприятий, которым необходимо работать с критически важными сервисами без зависимости от внешних решений для репликации и балансировки нагрузки.  
См. [Высокая доступность для нескольких центров обработки данных](/managed/managed-cluster/high-availability/premium-high-availability "Learn how Dynatrace Premium High Availability provides near-zero downtime and data resilience through multi-data-center deployments for Managed Clusters.").

## Аппаратное обеспечение

Во избежание потери данных конфигурации, метрик и логов развёртывайте каждый узел на отдельном хосте. Размещайте узлы на оборудовании с одинаковыми характеристиками — в особенности это касается дисков, CPU и RAM — для минимизации деградации производительности при недоступности части узлов. При аппаратном сбое затрагиваются только данные на отказавшей машине. Данные метрик и конфигурация не пострадают, так как все узлы их реплицируют. Данные, хранящиеся исключительно на этом узле — распределённые трассировки и Session Replay — будут потеряны. Однако репрезентативное значение счётчиков доступно на других узлах. Деградация производительности минимальна, поскольку все узлы работают на одинаковом оборудовании с равномерно распределённой нагрузкой.

## Вычислительная ёмкость

При планировании Managed Cluster закладывайте дополнительный запас ёмкости с расчётом на возможный отказ узла. Кластеры, работающие на 100% своей вычислительной мощности, не имеют ресурса для компенсации потери узла и рискуют потерять данные при его отказе. Развёртывания, рассчитанные на отказ узла, должны иметь запас вычислительной ёмкости на треть выше типичного потребления.

При отказе узла NGINX, выполняющий балансировку нагрузки системы, автоматически перенаправляет весь трафик OneAgent на оставшиеся рабочие узлы — никаких действий пользователя не требуется, кроме замены отказавшего узла.
