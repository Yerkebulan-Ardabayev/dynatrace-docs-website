# Настройка балансировщика нагрузки для Dynatrace Managed

* Published May 19, 2022

Кластер Dynatrace Managed включает NGINX — многофункциональный высокопроизводительный HTTP-сервер и обратный прокси с открытым исходным кодом. NGINX, установленный на каждом узле кластера, выполняет балансировку нагрузки и маршрутизацию трафика между узлами. NGINX специально настроен для выполнения следующих задач:

* **Постоянство сеансов и привязка**

  После открытия сеанса веб-интерфейса на заданном узле кластера все последующие запросы перенаправляются на этот узел до тех пор, пока пользователь не завершит сеанс (не выйдет из системы).
* **Проверки работоспособности узлов**

  Если узел недоступен, он помечается как неработоспособный, и трафик перенаправляется только на работоспособные узлы.
* **Разрешение DNS** (необязательно)

  DNS-записи управляются для кластера: DNS разрешает адреса всех узлов кластера. Это обеспечивает дополнительную балансировку нагрузки на основе DNS (порядок IP-адресов при разрешении DNS случаен). Автоматическое управление DNS для кластера является необязательным — его можно отключить и использовать собственный DNS.

Дополнительная балансировка нагрузки кластера

NGINX — это неотъемлемая часть развёртывания Dynatrace Managed. Если требуется дополнительный уровень балансировки нагрузки для организации единой точки доступа, можно настроить балансировщик нагрузки перед кластером Dynatrace Managed.

## Соображения при настройке

* **Соответствие требованиям**

  В вашей организации могут действовать строгие правила управления TLS-сертификатами и контроля доступа. Например, политика может требовать, чтобы весь входящий трафик проходил через одобренный организацией центральный балансировщик нагрузки или прокси-сервер.
* **Расширенное переключение при отказе**

  В случае сбоя аппаратного обеспечения или сети узла его IP-адрес может по-прежнему оставаться в записи домена. Это может привести к таймаутам подключения, если неработоспособный IP-адрес будет выбран при разрешении DNS. Внешний балансировщик нагрузки может автоматически исключать неработоспособный узел, что улучшает работу пользователей.
* **Геораспределённая балансировка нагрузки**

  Для кластеров, развёрнутых в двух центрах обработки данных (с использованием функции Premium HA), рекомендуется использовать дополнительный балансировщик нагрузки, который может оптимизировать доступ пользователей и обеспечить переключение при отказе между центрами обработки данных. См. [Premium HA для нескольких центров обработки данных](/managed/managed-cluster/high-availability/premium-high-availability "Learn how Dynatrace Premium High Availability provides near-zero downtime and data resilience through multi-data-center deployments for Managed Clusters.").

## Рекомендации по настройке балансировщика нагрузки

Конфигурация балансировщика нагрузки зависит от конкретного варианта его использования. Ниже приведены общие рекомендации.

### Настройка целевых узлов

Список целевых узлов должен содержать IP-адреса узлов Dynatrace Managed, предназначенных для приёма веб-трафика, включая пользовательский трафик и вызовы REST API. Как правило, список должен содержать все узлы кластера. Этот список необходимо обновлять после добавления или удаления узла.

В Cluster Management Console можно исключить узел из трафика веб-интерфейса. Эта функция влияет на конфигурацию NGINX и автоматически предоставляемый домен (`xyz.dynatrace-managed.com`). При использовании внешнего балансировщика нагрузки узлы кластера необходимо исключать из списка целевых узлов — иначе они по-прежнему будут получать сетевой трафик. См. [Возможности узла кластера](/managed/managed-cluster/configuration/cluster-node-capabilities "Find out how to enable/disable a cluster node via the Web UI or API call").

Трафик должен перенаправляться на HTTPS-порт 443 NGINX. Путь/параметры запроса не должны изменяться никаким образом; пользовательские суффиксы пути не поддерживаются.

### Таймауты

Рекомендуется устанавливать таймаут ответа целевого узла не менее значения, настроенного в NGINX по умолчанию (120 секунд).

### Проверки работоспособности

Рекомендуется настроить проверки работоспособности для всех целевых узлов. Если проверки работоспособности настроены не для всех целевых узлов, балансировщик нагрузки может попытаться перенаправить трафик на узел, не охваченный проверкой, который может быть недоступен из-за сбоя.

Используйте URL `/rest/health` для проверки работоспособности с ожидаемым кодом ответа `HTTP 200`.

### Постоянство сеансов и привязка

Постоянство сеансов и привязка настраиваются и реализуются кластером Dynatrace Managed. Дополнительная конфигурация не требуется.

### Домен и SSL-сертификаты

(необязательно)

Настройка доменного имени, указывающего на балансировщик нагрузки, и TLS-сертификата является необязательной. Можно отключить автоматическое управление доменом и сертификатами в Dynatrace. См. [Конфигурация DNS для Dynatrace Managed](/managed/managed-cluster/configuration/dns-configuration-in-managed "Learn how to configure DNS entries for Dynatrace Managed.").

## Трафик веб-интерфейса

Если планируется использовать балансировщик нагрузки для трафика веб-интерфейса, помимо [общих рекомендаций](#general-guidelines) по настройке балансировщика нагрузки, необходимо настроить следующий параметр.

### Настройка публичной конечной точки

1. В Cluster Management Console (CMC) перейдите в **Settings** и выберите **Public endpoints**.
2. Измените **Dynatrace Web UI URL** так, чтобы он указывал на ваш балансировщик нагрузки.

Чат недоступен

Если вы изменили URL веб-интерфейса Dynatrace, который обычно используется для доступа к интерфейсу кластера Dynatrace, или используете несколько доменов для доступа к кластеру Dynatrace, это может повлиять на встроенную помощь в чате. Тем не менее вы можете получить к ней доступ через REST API:

1. Повторите шаг 1, описанный выше.
2. Следуйте инструкциям по перенаправлению из **Public endpoints** в интерфейсе.
3. Используйте следующий токен API для создания **Additional address**.

   POST `/endpoint/webUiAddress`
4. Используйте следующий токен API для доступа к встроенной помощи в чате.

   POST `/endpoint/additionalWebUiAddresses`

## Трафик OneAgent

Балансировка нагрузки менее актуальна для трафика OneAgent, поскольку OneAgent способен переключаться между конечными точками и фактически выполнять балансировку нагрузки на стороне клиента. Если планируется использовать балансировщик нагрузки для трафика OneAgent, помимо [общих рекомендаций](#general-guidelines) по настройке балансировщика нагрузки, необходимо настроить следующий параметр.

### Настройка конечной точки OneAgent

1. В Cluster Management Console (CMC) перейдите в **Deployment Status** и выберите **Cluster nodes**.
2. На странице сведений каждого узла выберите **Configure** и в разделе **Customize node endpoints** измените **IP address and port for OneAgent traffic** так, чтобы адрес указывал на балансировщик нагрузки, а не непосредственно на узел.

## Трафик RUM/Mobile/Synthetic

Если планируется использовать балансировщик нагрузки там, где применяется RUM без агента или мониторинг мобильных приложений, настоятельно рекомендуется настроить балансировщик нагрузки либо перед кластерными ActiveGate, либо непосредственно перед кластером. Из-за возможных всплесков TLS-соединений и запросов большого объёма перенос завершения TLS на балансировщик нагрузки является целесообразным, особенно если балансировщик имеет аппаратное ускорение TLS и/или возможности автомасштабирования. Это позволяет защитить кластер от перегрузок и разделить пути доступа для трафика beacon и трафика веб-интерфейса/OneAgent.

Если планируется использовать балансировщик нагрузки для трафика RUM/Mobile/Synthetic, помимо [общих рекомендаций](#general-guidelines) по настройке балансировщика нагрузки, необходимо настроить следующий параметр.

### Настройка публичной конечной точки

1. В Cluster Management Console (CMC) перейдите в **Settings** и выберите **Public endpoints**.
2. Измените **Cluster ActiveGate URL** так, чтобы он указывал на ваш балансировщик нагрузки.  
   Это можно сделать, даже если кластерные ActiveGate отсутствуют и/или балансировщик нагрузки указывает непосредственно на кластер Managed.
