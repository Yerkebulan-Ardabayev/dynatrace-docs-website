# Аппаратные требования

# Аппаратные требования

* Reference
* 7-min read
* Updated on May 09, 2026

Перед установкой Dynatrace Managed ознакомьтесь с аппаратными требованиями. Точное планирование ресурсов для Managed Cluster не всегда возможно, особенно в средах с растущим трафиком. Хотя предварительный анализ полезен, важнее иметь возможность наращивать мощность по мере роста потребностей в мониторинге. Планируйте масштабирование по следующим направлениям:

* **Горизонтально** — добавляйте больше узлов. Поддерживается до 30 узлов кластера.
* **Вертикально** — добавляйте больше RAM или CPU на узел.
* **Хранилище** — используйте изменяемые дисковые тома (см. раздел [Рекомендации по хранилищу](#storage) ниже).

В производственных средах для обеспечения надёжности и избыточности данных требуется минимальный кластер из 3 узлов. Подробнее см. в разделе [Многоузловые установки](#multi-node-install).

## Аппаратные требования

Аппаратные требования в следующих таблицах являются оценочными, основанными на типичных средах и нагрузках. Значения в конкретных средах могут отличаться. Определения столбцов:

* **Минимальные характеристики узла** — CPU и RAM должны быть выделены исключительно для Dynatrace. Отключите режим энергосбережения CPU. Частота CPU должна быть не менее 2 ГГц; хосты должны иметь не менее 32 ГБ RAM.
* **Хранилище транзакций** — распределяется по всем узлам, избыточно не хранится. В многоузловых кластерах общее хранилище транзакций делится на количество узлов.
* **Долгосрочное хранилище метрик** — в многоузловых установках сохраняются три копии хранилища метрик. При четырёх и более узлах требование к хранилищу на узел уменьшается.

  Держите долгосрочное хранилище метрик ниже 2 ТБ на узел. Стабильность и операционная устойчивость обеспечиваются до 4 ТБ на узел. Рост постоянного хранилища ухудшает производительность, устойчивость и доступность кластера, а также создаёт проблемы при добавлении узлов. Если требуется больше хранилища, добавьте ещё один узел для снижения требования на узел.

## Требования к хранилищу

Dynatrace Managed хранит несколько типов данных мониторинга. Рекомендации:

* Храните бинарные файлы Dynatrace и хранилище данных на отдельных точках монтирования, чтобы хранилище данных можно было изменять независимо.
* Не храните данные Dynatrace на корневом томе во избежание сложностей при последующем изменении размера.
* Исключите пути хранения данных из антивирусного сканирования для предотвращения проблем с согласованностью данных.
* Монтируйте различные типы хранилищ данных на отдельные дисковые тома для максимальной гибкости и производительности.
* Используйте изменяемые дисковые разделы, например с Logical Volume Manager (LVM).
* Используйте одинаковый размер раздела на всех узлах кластера.

Размер диска может варьироваться в зависимости от использования. Например, в кластере из 2 узлов, где у каждого узла 10 ТБ и хранилище транзакций на обоих узлах составляет лишь 1,5 ТБ, третьему узлу требуется минимум 1 ТБ: (1,5 ТБ + 1,5 ТБ) / 3 = 1 ТБ. В аналогичном кластере, где диски заполнены на 9 ТБ, третьему узлу требуется минимум 6 ТБ: (9 ТБ + 9 ТБ) / 3 = 6 ТБ.

Значение ниже рассчитанного минимума является неправильной конфигурацией. Обратите внимание, что несколько источников данных — например, Session Replay — влияют на использование диска и также могут вызвать неправильную конфигурацию.

### Требования к файловой системе

Dynatrace Managed работает со всеми распространёнными файловыми системами. Используйте быстрое локальное хранилище, подходящее для нагрузок баз данных. Dynatrace Managed также поддерживает зашифрованные файловые системы при условии соответствия дисковых IOPS требованиям и прозрачности шифрования на уровне операционной системы (например, шифрование Amazon EBS или Azure Storage).

* Высоколатентные удалённые тома, такие как NFS или CIFS, не поддерживаются в качестве основного хранилища; NFS достаточно только для резервных копий.
* Amazon Elastic File System (EFS) не поддерживается в качестве основного хранилища для Elasticsearch, так как это может привести к повреждению индексов.

## Требования для Log Monitoring

Требования для [Log Monitoring Classic](/managed/analyze-explore-automate/log-monitoring "Learn how to enable Log Monitoring, the insights that Log Monitoring can provide, and more."):

* Все узлы кластера должны иметь не менее 64 ГБ RAM.

Дополнительные рекомендации:

* Для более устойчивой конфигурации добавляйте узлы кластера, а не увеличивайте аппаратные ресурсы на каждом узле.
* Равномерно распределяйте дополнительное хранилище Elasticsearch по всем узлам кластера.
* При добавлении CPU или RAM сохраняйте одинаковую конфигурацию всех узлов.
* Обновляйте лимит приёма данных кластера через Cluster Management Console или [REST API](/managed/dynatrace-api/cluster-api/cluster-api-v2/log-monitoring/post-update-log-events-per-cluster-for-log-monitoring "Learn how to update the total log events per cluster limit based on the cluster resources size using API.") при каждом изменении аппаратного обеспечения кластера.

  1. Для этого войдите в **Cluster Management Console**.
  2. Перейдите в **Environments**, выберите среду и настройте лимит в **Cluster overload prevention settings**.

## Многоузловые установки

В производственных средах для обеспечения надёжности и избыточности данных требуется минимальный кластер из 3 узлов. Все узлы многоузлового кластера должны:

* Иметь одинаковую аппаратную конфигурацию
* Синхронизироваться по NTP
* Находиться в одном часовом поясе
* Взаимодействовать через частную сеть на [необходимых портах](/managed/managed-cluster/installation/cluster-node-ports "Review the network ports required by Dynatrace Managed and configure your firewall for inbound and outbound communication.")
* Иметь сетевую задержку между узлами не более 10 мс
* Иметь системных пользователей Dynatrace Managed с одинаковыми идентификаторами UID:GID на всех узлах
