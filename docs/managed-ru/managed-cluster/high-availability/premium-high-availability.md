# Premium HA для нескольких центров обработки данных

* Explanation
* Updated on Apr 30, 2026

Dynatrace Premium High Availability (Premium HA) — это готовое самодостаточное решение, обеспечивающее практически нулевое время простоя и непрерывный мониторинг без потери данных в сценариях переключения при отказе. Оно позволяет снизить вычислительные расходы и затраты на хранилище за счёт устранения необходимости в отдельных резервных хостах аварийного восстановления и связанной инфраструктуре для хранения и передачи резервных копий.

Хотя дополнительные узлы в связанном центре обработки данных (ЦОД) увеличивают общую вычислительную ёмкость кластера, этот прирост нелинеен. При планировании ёмкости рассматривайте узлы дополнительного ЦОД как резервные, а не как расширение мощности — поскольку дополнительный ЦОД содержит копию всех данных Cassandra и Elasticsearch из основного ЦОД.

Важные замечания

* Premium HA доступен для клиентов с лицензионной моделью [Dynatrace classic licensing](/managed/license/monitoring-consumption-classic "Understand how Dynatrace monitoring consumption is calculated for classic licensing.") или [Dynatrace Platform Subscription](/managed/license "About Dynatrace Platform Subscription (DPS), the licensing model for all Dynatrace capabilities.").
* Кластер Premium HA требует подключения к [Dynatrace Mission Control](/managed/managed-cluster/basics/mission-control-proactive-support "Mission Control proactively monitors your Managed Cluster, provides software updates, and keeps your installation secure and reliable.") и поэтому доступен только для Managed-кластеров в режиме онлайн.
* Минимальное количество узлов для кластера Premium HA — 6 (по 3 узла на ЦОД).
* Максимальное количество узлов для кластера Premium HA — 30 (по 15 узлов на ЦОД).
* Миграция на Premium HA необратима.
* Оба ЦОД в кластере Premium HA должны быть симметрично сконфигурированы.

## Восстановление при сегментированном кластере

Если одна часть кластера теряет связь с другой, это не обязательно означает недоступность этой части кластера — возможно, это просто проблема сетевого соединения. Необходимо определить, какая часть кластера будет считаться выжившей. Разрывы сетевого соединения между центрами обработки данных продолжительностью до трёх часов устраняются автоматически. Для предотвращения несогласованности данных при более длительных сбоях рекомендуется остановить серверный сервис на всех узлах в затронутом ЦОД. Сервисы можно запустить после стабилизации сетевого соединения.

Для обработки ситуации, когда одна часть кластера недоступна, [Dynatrace Mission Control](/managed/managed-cluster/basics/mission-control-proactive-support "Mission Control proactively monitors your Managed Cluster, provides software updates, and keeps your installation secure and reliable.") отслеживает состояние всех узлов и автоматически назначает одну из частей кластера основной (выжившей). Во время восстановления это назначение определяет порядок повторной синхронизации всех частей кластера.

## Шардирование и репликация данных

Используя виртуальные rack'и, Dynatrace High Availability хранит три копии всех данных конфигурации, метрик и пользовательских сессий в каждом ЦОД. Это обеспечивает оптимальную производительность и надёжность в сценариях переключения при отказе.

Необработанные данные транзакций (такие как распределённые трассировки, стеки вызовов и запросы к базе данных) случайным образом распределяются по всем ЦОД, чтобы на каждом ЦОД всегда был доступен статистически репрезентативный набор данных.

Данные синхронизируются между ЦОД асинхронно. Это устраняет требование к задержке 10 мс, которое применяется ко всем межЦОД-кластерам. Синхронизация данных минимизирует потребление полосы пропускания между ЦОД и предотвращает потерю данных при отказе ЦОД.

При сбоях продолжительностью менее трёх часов Premium HA автоматически повторно синхронизирует данные между ЦОД.

## Маршрутизация данных телеметрии

Для управления потоком данных телеметрии к узлам кластера в различных ЦОД можно использовать сетевые зоны. Несмотря на то что Premium HA реализует различные оптимизации для снижения межЦОД-трафика, для обеспечения избыточности данных рекомендуется разрешить ActiveGate отправлять данные в оба ЦОД. OneAgent и ActiveGate можно настроить с предпочтением определённых сетевых зон, сохраняя при этом возможность переключения на другую часть кластера при отказе ЦОД. Для этих целей также можно использовать балансировщики нагрузки.

При активно-пассивных развёртываниях приложений не отключайте ActiveGate в пассивных частях развёртывания. Это гарантирует, что все компоненты инфраструктуры Dynatrace остаются активными в сценариях аварийного восстановления и позволяет выполнить переключение при отказе без перенастройки или повторного обнаружения.

## Технические сведения

Premium HA требует ОС с поддержкой cgroups версии 1.0 и systemd версии 219 или выше (например, RHEL/CentOS 7+).

Узлы продолжают взаимодействовать друг с другом через обычные порты, описанные ранее. Порты, которые должны быть открыты между узлами одного ЦОД, совпадают с портами, необходимыми внутри кластера, когда кластер охватывает два ЦОД.

Соединения между узлами в разных ЦОД должны быть зашифрованы. Dynatrace не создаёт и не устанавливает необходимые сертификаты — это необходимо сделать вручную. Поддерживается сетевая задержка RTT до 100 мс. Потребление полосы пропускания зависит от ряда факторов. За подробной информацией обратитесь к специалисту по продуктам Dynatrace.

Можно выполнить миграцию однокластерного ЦОД (или межЦОД-кластера без привязки к ЦОД) в двойной ЦОД-кластер Premium HA. За подробной информацией обратитесь к специалисту по продуктам Dynatrace.
