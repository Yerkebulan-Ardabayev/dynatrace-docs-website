# Конвертация в rack-aware с использованием восстановления

* Published Feb 04, 2021

Данный метод является универсальным и подходит для крупных кластеров и более сложных изменений топологии. Однако процесс восстановления занимает значительное время, а из-за ежедневной частоты резервного копирования возможна некоторая потеря данных. Хранилище транзакций ([распределённые трассировки](/managed/observe/application-observability/distributed-traces "Gain observability into highly distributed, cloud-native architectures to effectively trace and analyze transactions in real time.")) не включается в резервную копию.

Процесс конвертации Managed-кластера в rack-aware через резервное копирование и восстановление

## Подготовка

* Убедитесь, что у кластера есть актуальная резервная копия.  
  См. [Резервное копирование и восстановление кластера](/managed/managed-cluster/operation/back-up-and-restore-a-cluster "Understand the steps and commands required to restore a Dynatrace Managed cluster.").
* Подготовьте новые узлы.  
  Убедитесь, что размер дискового раздела, выделенного под хранилище Cassandra на новом узле, достаточен для размещения всей базы данных Cassandra (с запасом для компактирования и новых данных). Размер диска должен как минимум вдвое превышать суммарный объём хранилища Cassandra всех существующих узлов кластера. Рекомендуется хранить данные Cassandra на отдельном томе, чтобы избежать проблем с дисковым пространством, связанных с различными типами данных.

## Восстановление кластера в новые rack'и

Порядок восстановления кластера в новые rack'и

1. Остановите существующий кластер, чтобы исключить подключение двух кластеров с одним и тем же ID к Dynatrace Mission Control.  
   См. [Запуск/остановка/перезапуск кластера](/managed/managed-cluster/operation/start-stop-restart-node "Properly shut down and restart Dynatrace Managed nodes using command line parameters.").
2. Выполните официальную процедуру восстановления кластера ([Резервное копирование и восстановление кластера](/managed/managed-cluster/operation/back-up-and-restore-a-cluster#cluster-restore "Understand the steps and commands required to restore a Dynatrace Managed cluster.")), используя следующий изменённый шаг:

   Запуск восстановления Dynatrace на каждом узле

   Параллельно на каждом узле запустите установщик Dynatrace Managed со следующими параметрами:

   * `--rack-name` — определяет rack, к которому будет принадлежать этот узел.
   * `--rack-dc` — определяет центр обработки данных, к которому будет принадлежать этот узел.
   * `--restore` — переключает установщик в режим восстановления.
   * `--cluster-ip` — IPv4-адрес узла, на котором запускается установщик.
   * `--cluster-nodes` — разделённый запятыми список идентификаторов и IP-адресов всех узлов кластера, включая тот, на котором запускается установщик, в формате `<node_id>:<node_ip>,<node_id>:<node_ip>`.
   * `--seed-ip` — IPv4-адрес seed-узла.
   * `backup-file` — путь к файлу резервной копии `*.tar`, включающий путь к точке монтирования общего хранилища, UUID кластера, идентификатор узла, номер версии резервной копии и файл `*.tar` в следующем формате:  
     `<path-to-backup>/<UUID>/node_<node_id>/files/<backup_version_number>/<backup_file>`

   Пример пути к резервной копии

   В данном примере пути:
   `/mnt/backup/bckp/c9dd47f0-87d7-445e-bbeb-26429fac06c6/node_1/files/19/backup-001.tar`
   части пути имеют следующий смысл:

   * `<path-to-backup>` = `/mnt/backup/bckp/`
   * `<UUID>` = `c9dd47f0-87d7-445e-bbeb-26429fac06c6`
   * `<node_id>` = `1`
   * `<backup_version_number>` = `19`

   Пока выполняется резервное копирование, могут присутствовать две директории резервных копий с разными номерами версий:

   * Директория с меньшим номером версии содержит старую резервную копию. Она будет удалена после завершения резервного копирования.
   * Директория с большим номером версии содержит резервную копию в процессе создания.

   Номер версии резервной копии автоматически увеличивается с каждым запуском резервного копирования.

   Получите идентификаторы и IP-адреса из инвентаризации, составленной до начала восстановления.

   Например:  
   `10.176.41.168` — IP-адрес узла для восстановления  
   `1: 10.176.41.168, 3: 10.176.41.169, 5: 10.176.41.170` — идентификаторы узлов и новые IP-адреса всех узлов кластера

   ```
   sudo ./tmp/backup-001-dynatrace-managed-installer.sh



   --rack-name rack2



   --rack-dc datacenter1



   --restore



   --cluster-ip "10.176.41.168"



   --cluster-nodes "1:10.176.41.168,3:10.176.41.169,5:10.176.41.170"



   --seed-ip "10.176.41.169"



   --backup-file /mnt/backup/bckp/c9dd47f0-87d7-445e-bbeb-26429fac06c6/node_1/files/19/backup-001.tar
   ```
3. При загрузке данных в процессе восстановления они сразу размещаются в целевых rack'ах.

По завершении конвертации rack'и отобразятся на странице **Deployment status** в Cluster Management Console:
