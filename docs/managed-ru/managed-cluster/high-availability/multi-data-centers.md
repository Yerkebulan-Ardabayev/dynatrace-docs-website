# Высокая доступность в нескольких дата-центрах

* Пояснение
* Чтение за 5 минут
* Обновлено 07 июля 2026 г.

Premium High Availability (PHA) расширяет высокую доступность Dynatrace Managed на два дата-центра (ДЦ). PHA, это самодостаточное решение, которое поддерживает мониторинг без потери данных и с минимальным простоем при отказе ДЦ.

PHA хранит полную активную копию данных во втором ДЦ. Активная копия убирает необходимость в отдельных резервных хостах аварийного восстановления и инфраструктуре переноса резервных копий, снижая затраты на вычисления и хранение.

## Требования и ограничения

Перед планированием развёртывания PHA нужно ознакомиться со следующими требованиями и ограничениями:

* Использовать лицензионную модель, поддерживающую PHA: либо [Dynatrace Classic](/managed/license/monitoring-consumption-classic "Understand how Dynatrace classic monitoring consumption is calculated, including host units, DDUs, DEM units, and Application Security units."), либо [Dynatrace Platform Subscription](/managed/license "Dynatrace Platform Subscription, capability rate cards, hybrid licensing, and previous license models.").
* Подключить Managed Cluster к [Dynatrace Mission Control (MC)](/managed/managed-cluster/basics/mission-control-proactive-support "Mission Control proactively monitors your Managed Cluster, provides software updates, and keeps your installation secure and reliable."). PHA доступен только для онлайн-кластеров Managed.
* Развернуть не менее 6 узлов, по 3 узла на каждый ДЦ.
* Развернуть не более 30 узлов, не более 15 узлов на ДЦ.
* Обеспечить симметричный размер обоих ДЦ.
* Спланировать миграцию. Миграцию на PHA нельзя отменить.
* Использовать операционную систему, поддерживающую `cgroups` версии 1.0 и `systemd` версии 219 или новее, например Red Hat Enterprise Linux 7 или CentOS 7.
* Открыть одинаковые порты между узлами внутри одного ДЦ и в Managed Cluster, охватывающем два ДЦ.
* Шифровать соединения между узлами в разных ДЦ. Dynatrace Managed не создаёт и не устанавливает необходимые сертификаты.
* Поддерживать сетевую задержку кругового обхода на уровне 100 мс или меньше.

## Региональная отказоустойчивость

Региональная отказоустойчивость означает, что все узлы Managed Cluster в одной локации могут выйти из строя без простоя. Для этого нужно распределить узлы Managed Cluster по отдельным физическим локациям одним из следующих способов:

* Использовать PHA для двух локаций, которым нужны географическая избыточность и автоматическое переключение при отказе.
* Использовать [развёртывание с учётом стоек](/managed/managed-cluster/high-availability/rack-awareness "Learn how rack-aware deployment groups Dynatrace Managed Cluster nodes into three fault domains to ensure resilience against a full rack outage and data loss.") для трёх локаций с низкой задержкой.
* Совместить PHA с [развёртыванием с учётом стоек](/managed/managed-cluster/high-availability/rack-awareness "Learn how rack-aware deployment groups Dynatrace Managed Cluster nodes into three fault domains to ensure resilience against a full rack outage and data loss.") в каждом ДЦ для шести локаций.

Коэффициент репликации, равный трём, обеспечивает наличие всех данных метрик и событий в каждой локации.

## Планирование мощности

Нужно строить Managed Cluster с учётом дополнительной мощности и возможного отказа узла. Managed Cluster, работающий на 100 процентах своей вычислительной мощности, не имеет резерва для компенсации потери узла. Развёртывания, работающие на полной мощности, могут терять данные при отказе узла.

Развёртывания, спланированные с учётом отказа узла, должны иметь вычислительную мощность на треть выше типичной загрузки.

При планировании мощности в развёртывании PHA узлы дополнительного ДЦ нужно рассматривать как резервные, а не как расширение мощности. Дополнительный ДЦ хранит копию всех данных Cassandra и Elasticsearch из исходного ДЦ.

При отказе узла NGINX автоматически перенаправляет весь трафик OneAgent на оставшиеся рабочие узлы. Нужно только заменить отказавший узел.

## Репликация данных между дата-центрами

Dynatrace Managed распределяет всю конфигурацию Managed Cluster и его сред по всем узлам. Реплицируемые данные включают все события, пользовательские сессии и метрики. Managed Cluster хранит две копии этих данных, поэтому Dynatrace Managed может продолжать работу после потери одного узла.

Потеря двух или более узлов может повлиять на производительность и доступность Managed Cluster. Влияние зависит от распределения данных и требуемого уровня согласованности данных.

Данные событий [Log Monitoring](/managed/analyze-explore-automate/log-monitoring "Learn how to enable Log Monitoring, the insights that Log Monitoring can provide, and more.") реплицируются в хранилище Elasticsearch для обеспечения высокой доступности и оптимизации затрат на хранение. В результате, если узел выходит из строя, копия есть на другом узле. Однако отказ двух узлов делает некоторые события логов недоступными. Если узлы снова поднимаются, данные снова становятся доступны. В противном случае Dynatrace Managed теряет данные.

Dynatrace Managed не реплицирует сырые данные транзакций, такие как стеки вызовов, операторы базы данных и видимость на уровне кода, между узлами. Dynatrace Managed равномерно распределяет эти данные по всем узлам.

В результате при отказе узла Dynatrace Managed может оценить недостающие данные. Такая оценка возможна, потому что сырые данные транзакций обычно недолговечны. Dynatrace Managed также собирает большой объём сырых данных, поэтому у каждого узла всё равно остаётся достаточно большой набор данных, даже если другой узел недоступен в течение некоторого времени.

### Шардирование и репликация данных

Используя виртуальные стойки, PHA хранит три копии всех данных конфигурации, метрик и пользовательских сессий в каждом ДЦ. Хранение в трёх копиях обеспечивает оптимальную производительность и надёжность в сценариях переключения при отказе.

PHA распределяет сырые данные транзакций, такие как распределённые трассировки, стеки вызовов и операторы базы данных, по всем ДЦ. Распределение между ДЦ делает статистически репрезентативный набор данных доступным в каждом ДЦ.

PHA синхронизирует данные между ДЦ асинхронно. Асинхронная синхронизация устраняет требование задержки в 10 мс, которое применяется к Managed Cluster, охватывающим несколько ДЦ. Синхронизация данных минимизирует потребление пропускной способности между ДЦ и предотвращает потерю данных при отказе ДЦ.

Во время простоев продолжительностью до 72 часов PHA автоматически ресинхронизирует данные между ДЦ.

## Маршрутизация телеметрии

PHA использует сетевые зоны для маршрутизации телеметрии на узлы Managed Cluster в разных ДЦ. Сетевые зоны позволяют OneAgent и ActiveGate отдавать предпочтение локальным маршрутам, сохраняя при этом возможность переключения на другой ДЦ во время простоя. Инструкции по настройке приведены в разделе [Репликация узлов между дата-центрами для PHA](/managed/managed-cluster/high-availability/add-data-center#review-network-zone-setup "Learn how to replicate Managed Cluster nodes across two data centers to set up a Premium High Availability deployment with cross-data center replication.").

## Переключение при отказе и восстановление

PHA автоматически обрабатывает простои ДЦ продолжительностью до 72 часов и использует Mission Control для определения работоспособной части Managed Cluster во время восстановления. Поведение при переключении описано в разделе [Переключение при отказе между дата-центрами](/managed/managed-cluster/high-availability/failover "Learn how the Premium High Availability multi-data center failover mechanism detects node outages and transfers responsibility to a healthy data center."). Восстановление после более длительных простоев описано в разделе [Аварийное восстановление дата-центра из другого дата-центра](/managed/managed-cluster/high-availability/recover-from-data-center "Recover a data center when Premium High Availability can't repair it within 72 hours by restoring or recreating it from another data center.").

Managed Cluster можно перенести из одного ДЦ или из Managed Cluster, уже охватывающего несколько ДЦ, в развёртывание PHA с двумя ДЦ. Шаги миграции описаны в разделе [Репликация узлов между дата-центрами для PHA](/managed/managed-cluster/high-availability/add-data-center "Learn how to replicate Managed Cluster nodes across two data centers to set up a Premium High Availability deployment with cross-data center replication.").

## Размещение оборудования

Чтобы предотвратить потерю данных конфигурации, метрик и логов, каждый узел нужно развернуть на отдельном хосте. Узлы нужно развернуть на оборудовании с одинаковыми характеристиками, особенно диска, процессора и памяти. Идентичное оборудование минимизирует снижение производительности при недоступности некоторых узлов.

Аппаратный отказ влияет только на данные, находящиеся на вышедшей из строя машине. Отказ не влияет на данные метрик или конфигурацию, так как их реплицируют все узлы. Dynatrace Managed теряет только данные, хранившиеся на этом узле, такие как распределённые трассировки и записи сессий. Однако на других узлах всё равно остаётся достаточно похожих данных для оценок.

Соответствующее оборудование и сбалансированные рабочие нагрузки снижают снижение производительности при недоступности узлов.
