# Конвертация в rack-aware с использованием репликации

* Updated on Nov 24, 2025

Данный метод подходит для небольших кластеров Dynatrace Managed, где один узел может содержать полный реплика. Если текущий размер хранилища метрик (база данных Cassandra) на узел превышает 1 ТБ, используйте метод восстановления кластера. Существующие узлы можно использовать для последовательной переустановки (один за другим) с rack-aware параметрами. Также можно использовать дополнительное оборудование в качестве новых узлов: удалять один узел и одновременно добавлять новый с rack-aware параметрами.

Преимущество этого подхода — отсутствие влияния на доступность кластера. Однако для обеспечения доступности необходимо предоставить кластеру Managed время на использование нативных методов репликации для сохранения данных. Следует учитывать, что удаление и добавление узла в кластер занимает время — оно зависит от размера хранилища метрик и скорости диска или сети. Некоторые операции с кластером могут занимать до одного-двух дней. В течение этого времени Managed-кластер блокирует все остальные операции (добавление и удаление узлов, обновление и резервное копирование). При переустановке на существующем хосте подождите 72 часа перед повторным подключением хоста к кластеру.

## Подготовка

1. Подготовьте базу данных Cassandra к распределению реплик по кластеру (требуется Python 2.7).

   Этот шаг применим только к Dynatrace Managed версии 1.304 и более ранних.

   На одном из существующих узлов измените keyspace для изменения настроек snitch с помощью Cassandra CQL:

   ```
   sudo <managed-installation-dir>/cassandra/bin/cqlsh <node_IP>
   ```

   Где `<managed-installation-dir>` — директория с бинарными файлами Dynatrace Managed, а `<node-IP>` — IP-адрес текущего узла (узла, с которого выполняется команда).

   В результате вы перейдёте в CQL-оболочку:

   ```
   Connected to 00aa0a0a-1ab1-11a1-aaa1-0a0a0aa1a1aa at xx.xxx.xx.xxx:9042.



   [cqlsh 5.0.1 | Cassandra 3.0.23 | CQL spec 3.4.0 | Native protocol v4]



   Use HELP for help.



   cqlsh>
   ```

   Затем выполните:

   ```
   ALTER KEYSPACE ruxitdb WITH REPLICATION = {'class': 'NetworkTopologyStrategy', 'datacenter1':3};
   ```

   По завершении введите: `exit`.
2. Подготовьте кластер для добавления узлов в разные rack'и.

   На каждом существующем узле отредактируйте `/etc/dynatrace.conf` и скорректируйте следующие настройки:

   ```
   CASSANDRA_NODE_RACK = rack1



   CASSANDRA_NODE_RACK_DC = datacenter1



   ELASTICSEARCH_NODE_RACK = rack1
   ```

   Выполните `sudo /opt/dynatrace-managed/installer/reconfigure.sh` для применения изменений конфигурации узла кластера.

   Выполнять последовательно

   Выполняйте скрипт `reconfigure.sh` последовательно, так как он инициирует перезапуск процессов. Параллельное выполнение на нескольких узлах создаёт риск простоя базы данных Cassandra.
3. Подготовьте новые узлы.

   Убедитесь, что размер дискового раздела, выделенного под хранилище Cassandra на новом узле, достаточен для размещения всей базы данных Cassandra (с запасом для компактирования и новых данных). Размер диска должен как минимум вдвое превышать суммарный объём хранилища Cassandra всех существующих узлов кластера. Рекомендуется хранить данные Cassandra на отдельном томе, чтобы избежать проблем с дисковым пространством, связанных с различными типами данных.
4. Убедитесь, что настройки rack'ов корректно соответствуют физическим rack'ам и центрам обработки данных. В идеале по завершении конвертации в каждом rack'е должно быть одинаковое количество узлов.

## Расширение кластера в новые rack'и

При первоначальном развёртывании Dynatrace Managed все узлы кластера по умолчанию группируются в единый default rack в default datacenter. Даже если развёртывание не является rack-aware, все узлы уже находятся в `datacenter1` `rack1`. Во избежание добавления нового узла в default rack не используйте значение `rack1` для параметра `--rack-name` нового rack'а.

При конвертации кластера в том же центре обработки данных используйте значение параметра `datacenter1` по умолчанию.

Порядок расширения кластера в новые rack'и

1. Добавьте новый узел с rack-параметрами в кластер.

   Используйте процедуру [Добавление нового узла кластера](/managed/managed-cluster/installation/add-cluster-node "Add a node to your Managed Cluster by downloading the installer, running it on the target host, and monitoring synchronization progress.") и добавьте параметры `--rack-name` и `--rack-dc` к команде установки. Например:

   ```
   /bin/sh dynatrace-managed-installer.sh --seed-auth abcdefjhij1234567890 --rack-name rack2 --rack-dc datacenter1
   ```

   Дождитесь полного вступления узла в кластер. В зависимости от размера базы данных процесс Cassandra bootstrapping может занять несколько дней.

   Подсказка

   Если узел испытывает затруднения с обработкой нагрузки данных, можно временно остановить серверный процесс Dynatrace на этом узле и освободить дополнительные циклы CPU для Cassandra.
2. Добавьте ещё один узел в тот же rack с теми же rack-параметрами.  
   В этот раз Cassandra bootstrapping должен выполниться быстрее.
3. Продолжайте добавлять новые узлы в `rack2`. Как только в `rack2` окажется достаточное количество узлов (1/3 от целевого размера кластера), начинайте добавлять новые узлы с rack-параметрами в `rack3`, соблюдая требования к дисковому пространству.
4. Как только достаточное количество узлов окажется также в `rack3`, начинайте удалять исходные узлы, настроенные без rack awareness (находящиеся в default rack'е `rack1`).

По завершении конвертации rack'и отобразятся на странице **Deployment status** в Cluster Management Console:
