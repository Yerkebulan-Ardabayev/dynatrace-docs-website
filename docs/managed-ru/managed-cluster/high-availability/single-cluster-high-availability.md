# Высокая доступность единого кластера

* Объяснение
* 3 мин чтения
* Обновлено 03 июля 2026

Dynatrace Managed обеспечивает высокую доступность в пределах одного Managed Cluster. Managed Cluster работает на трёх или более равноправных узлах, которые используют общие сервисы и реплицируют данные друг другу. Благодаря этому Managed Cluster продолжает работать при отказе узла, и нужно только заменить неисправный узел.

Единый Managed Cluster переносит потерю отдельных узлов, но не потерю целой локации. Чтобы достичь региональной отказоустойчивости, при которой все узлы Managed Cluster в одной локации могут выйти из строя без простоя, используется Premium High Availability (PHA) между дата-центрами. Подробнее в разделе [Высокая доступность нескольких дата-центров](/managed/managed-cluster/high-availability/multi-data-centers "Understand how Dynatrace Managed Premium High Availability provides failover, data resilience, and data routing across data centers.").

## Резервирование узлов

Разворачивается минимум три узла Managed Cluster. Все узлы реплицируют данные по Managed Cluster, поэтому обычно, помимо основного шарда, существует две реплики. Задержку между узлами нужно поддерживать на уровне около 10 мс или меньше.

Dynatrace Managed распределяет всю конфигурацию Managed Cluster и его окружений по всем узлам, включая все события, пользовательские сессии и метрики. Managed Cluster поддерживает две копии этих данных, поэтому Dynatrace Managed продолжает работать после потери одного узла. Потеря двух или более узлов может повлиять на производительность и доступность Managed Cluster, в зависимости от распределения данных и требуемого уровня согласованности данных.

Данные событий [Log Monitoring](/managed/analyze-explore-automate/log-monitoring "Learn how to enable Log Monitoring, the insights that Log Monitoring can provide, and more.") реплицируются в хранилище Elasticsearch для обеспечения высокой доступности и оптимизации затрат на хранение. Благодаря этому, если узел выходит из строя, другой узел хранит копию данных. Однако отказ двух узлов делает часть событий журнала недоступными. Если узлы возвращаются в строй, данные снова становятся доступны. В противном случае Dynatrace Managed теряет эти данные.

Dynatrace Managed не реплицирует между узлами необработанные данные транзакций, такие как стеки вызовов, операторы базы данных и данные видимости на уровне кода. Вместо этого он равномерно распределяет эти данные по всем узлам. Благодаря этому, когда узел выходит из строя, Dynatrace Managed может оценить недостающие данные. Такая оценка надёжна, потому что необработанные данные транзакций обычно недолговечны. Dynatrace Managed также собирает большой объём этих данных, поэтому каждый узел хранит достаточно большой набор данных, даже когда другой узел недоступен.

## Отказоустойчивость узлов

Если узел выходит из строя, балансировщик нагрузки NGINX автоматически перенаправляет весь трафик OneAgent на оставшиеся рабочие узлы. Нужно только заменить неисправный узел.

## Размещение оборудования

Чтобы предотвратить потерю конфигурации, метрик и данных журналов, каждый узел разворачивается на отдельном хосте. Узлы разворачиваются на оборудовании с одинаковыми характеристиками, особенно диска, процессора и памяти, чтобы минимизировать снижение производительности при недоступности некоторых узлов.

Аппаратный отказ влияет только на данные, находящиеся на неисправной машине. Отказ не влияет на данные метрик или конфигурацию, так как их реплицируют все узлы. Dynatrace Managed теряет только данные, хранившиеся на этом узле, такие как распределённые трассировки и записи сессий. Однако другие узлы сохраняют репрезентативный набор аналогичных данных.

## Планирование ёмкости

Managed Cluster нужно строить с учётом дополнительной ёмкости и возможного отказа узла. Managed Cluster, работающий на 100% своей вычислительной мощности, не имеет резерва для компенсации потери узла и может терять данные при отказе узла. Нужно планировать вычислительную мощность на треть выше типичной загрузки.
