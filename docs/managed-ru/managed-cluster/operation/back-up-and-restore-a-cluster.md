# Резервное копирование и восстановление кластера

* Опубликовано 22 марта 2018 г.

Для настройки автоматического резервного копирования кластера Dynatrace Managed

1. Войти в **Cluster Management Console**.
2. Перейти в **Settings** > **Backup**.
3. Включить резервное копирование кластера и выбрать область:

   * Пользовательские сессии могут содержать конфиденциальную информацию.  
     **Исключить пользовательские сессии из резервной копии, чтобы соответствовать требованиям GDPR**.
   * Исключить данные метрик временных рядов из резервной копии, если исторические данные не важны и нужно сохранить только данные конфигурации.
   * Включить резервное копирование событий Log Monitoring.
4. (Необязательно) Выбрать дата-центр. Этот шаг нужен только при развёртывании с несколькими дата-центрами (развёртывание Premium High Availability). Подробнее о развёртываниях Premium High Availability см. [Отказоустойчивость между несколькими дата-центрами](/managed/managed-cluster/high-availability/multi-data-centers "Узнать, как Dynatrace Managed Premium High Availability обеспечивает отказоустойчивость, устойчивость данных и маршрутизацию данных между дата-центрами.") и [Аварийное восстановление из резервной копии](/managed/managed-cluster/high-availability/recover-from-backup "Узнать, как восстановить дата-центр Premium High Availability из резервной копии в развёртывании Dynatrace Managed с несколькими дата-центрами после утраты дата-центра.")
5. Указать полный путь к смонтированному сетевому файловому хранилищу, где хранятся архивы резервных копий.
6. Настроить время старта.

## Автоматическое резервное копирование кластера

Можно автоматически создавать резервные копии данных конфигурации Dynatrace Managed (правила именования, теги, management zones, профили оповещений и другое), данных метрик временных рядов и пользовательских сессий. Для максимальной отказоустойчивости нужно сохранять файлы резервных копий во внешнем расположении.

* Каждый узел должен быть подключён к общей файловой системе, например Network File Sharing (NFS), и эта общая файловая система должна быть смонтирована **в один и тот же общий каталог на каждом узле**.  
  Чтобы проверить права пользователя, нужно выполнить следующую команду на каждом узле:

  ```
  su - dynatrace -s /bin/bash -c "touch /nfs/dynatrace/backup/$(uname -n)"



  ls -ltr /path/to/backup/
  ```
* Пользователь, от имени которого работают службы Dynatrace, должен иметь права на чтение и запись в общей файловой системе.
* Точка монтирования общей файловой системы должна быть доступна при перезапуске системы.
* Можно добавить точку монтирования в `fstab` или использовать инструмент управления дисками, чтобы сделать монтирование общей файловой системы постоянным.
* Протокол, используемый для передачи данных, зависит от конфигурации. Рекомендуется NFSv4. Из-за низкой производительности и отказоустойчивости CIFS не рекомендуется.

Точка монтирования общей файловой системы при загрузке системы

Если точка монтирования общей файловой системы недоступна при загрузке системы, Dynatrace не запустится на этом узле. Это может привести к недоступности кластера. Нужно вручную отключить резервное копирование, чтобы Dynatrace мог запуститься.

### Хранилище метрик и конфигурации

Dynatrace хранит предыдущую резервную копию до завершения новой.

#### Характеристики резервного копирования

* Снимок создаётся ежедневно.
* Любые данные, реплицированные между узлами, также сохраняются в резервной копии (дедупликация не выполняется).
* Dynatrace исключает наиболее часто изменяющиеся семейства столбцов (исключённые семейства столбцов составляют около 80% общего объёма хранилища), а также данные с разрешением 1 минута и 5 минут.

### Elasticsearch

Файлы Elasticsearch хранятся в несжатом двоичном формате. Хотя данные реплицируются между узлами и, помимо основного шарда, существуют две реплики, резервная копия исключает реплицированные данные.

Не удалять резервные копии Elasticsearch

Резервное копирование Elasticsearch выполняется инкрементально, поэтому системе нужно иметь возможность добавлять недавние изменения к предыдущей резервной копии. Не удалять файлы резервных копий Elasticsearch.

#### Характеристики резервного копирования

* Снимок создаётся, по умолчанию, каждые 2 часа и является инкрементальным.  
  Изначально Dynatrace копирует весь набор данных, а затем создаёт снимки различий. Более старые снимки постепенно удаляются, когда им исполняется пять (5) дней.
* Поскольку Dynatrace хранит некоторые из более старых снимков, размер резервной копии растёт независимо от текущего размера на диске. Снимки инкрементальны, но Elasticsearch со временем объединяет сегменты данных, из-за чего в резервной копии появляются определённые дубликаты.

Резервное копирование хранилища транзакций отсутствует

Данные хранилища транзакций не резервируются, поэтому при восстановлении из резервных копий возможны пробелы в данных глубокого мониторинга (например, в распределённых трассировках и трассировках уровня кода). По умолчанию данные хранилища транзакций хранятся только 10 дней. С точки зрения долгосрочной перспективы включать данные хранилища транзакций в резервные копии не нужно.

## Восстановление кластера

Чтобы восстановить кластер, нужно выполнить шаги ниже.

### Перед началом

* Чтобы восстановить кластер на том же хосте, что и исходный кластер, нужно сначала удалить его.
* Нужно убедиться, что машины, подготовленные для восстановления кластера, имеют аппаратную конфигурацию и разметку дисков, схожие с исходным кластером, и достаточную мощность для обработки нагрузки после восстановления.

  Рекомендуется восстанавливать кластер на том же количестве узлов, что и в кластере, с которого сделана резервная копия. В исключительных случаях возможно восстановление в кластер с числом узлов, меньшим не более чем на два по сравнению с кластером резервной копии. При попытке восстановления в кластер, число узлов которого меньше исходного более чем на два, есть риск потери конфигурации кластера.
* Нужно убедиться, что существующий кластер остановлен, чтобы предотвратить подключение двух кластеров с одинаковым ID к Dynatrace Mission Control. См. [Запуск/остановка/перезапуск кластера](/managed/managed-cluster/operation/start-stop-restart-node "Правильное завершение работы и перезапуск узлов Dynatrace Managed с помощью параметров командной строки.").
* Нужно убедиться, что системные пользователи, созданные для Dynatrace Managed, имеют одинаковые идентификаторы UID:GID на всех узлах.
* На каждом целевом узле смонтировать хранилище резервных копий, например, в `/mnt/backup`. Этот путь далее в шагах обозначается как `<path-to-backup>`.
* Убедиться, что установщик имеет права на чтение NFS. Например:
  `sudo adduser dynatrace && sudo chown -R dynatrace:dynatrace <path-to-backup>`
* Составить инвентарь своего кластера. Эта информация понадобится в ходе восстановления.

  + Идентификаторы узлов в кластере. Резервная копия каждого узла хранится в отдельном каталоге, имя которого основано на идентификаторе узла, в формате `node_<node-id>` (например, `node_1`, `node_5` и т. д.).
  + IPv4-адреса новых машин.
  + IP-адрес целевой машины для каждого узла.
  + ID узла, который станет seed-узлом в кластере.

### Восстановление из резервной копии

Чтобы восстановить кластер, выполни шаги ниже:

1. **Копирование установщика на целевые узлы**
   Для восстановления кластера нужно использовать точно ту же версию установщика, что и в исходном кластере. Скопируй установщик из `<path-to-backup>/<UUID>/node_<node-id>/` на локальный диск каждого целевого узла.

   Например `cp <path-to-backup>/<UUID>/node_<node-id>/files/<backup-version-number>/backup-001-dynatrace-managed-installer.sh /tmp/`
2. **Запуск восстановления Dynatrace на каждом узле**
   Параллельно на каждом узле запусти установщик Dynatrace Managed со следующими параметрами:

   * `--restore`, переключает установщик в режим восстановления.
   * `--cluster-ip`, IPv4-адрес узла, на котором запускается установщик.
   * `--cluster-nodes`, список идентификаторов и IP-адресов всех узлов кластера через запятую, включая узел, на котором запускается установщик, в следующем формате `<node-id>:<node-ip>,<node-id>:<node-ip>`.
   * `--seed-ip`, IPv4-адрес seed-узла.
   * `backup-file`, путь к файлу резервной копии `*.tar`, включающий путь к точке монтирования общего файлового хранилища, ID кластера, ID узла, версию резервной копии и файл резервной копии `*.tar` в следующем формате:
     `<path-to-backup>/<UUID>/node_<node-id>/files/<backup-version-number>/<backup-file>`

   Пример пути к резервной копии

   В этом примере пути:

   `/mnt/backup/bckp/c9dd47f0-87d7-445e-bbeb-26429fac06c6/node_1/files/19/backup-001.tar`

   части пути следующие:

   * `<path-to-backup>` = `/mnt/backup/bckp/`
   * `<UUID>` = `c9dd47f0-87d7-445e-bbeb-26429fac06c6`
   * `<node-id>` = `1`
   * `<backup-version-number>` = `19`

   Пока резервное копирование выполняется, могут присутствовать две директории резервных копий с разными номерами версий:

   * Директория с меньшим номером версии содержит старую резервную копию. Она будет удалена после завершения резервного копирования.
   * Директория с большим номером версии содержит резервную копию, которая выполняется в данный момент.

   Номер версии резервной копии увеличивается автоматически при каждом запуске резервного копирования.

   Получи ID и IP-адреса из инвентарного списка, который был создан перед началом работы.

   Например:
   `10.176.41.168`, IP-адрес узла для восстановления
   `1: 10.176.41.168, 3: 10.176.41.169, 5: 10.176.41.170`, ID узлов и новые IP-адреса всех узлов кластера

   ```
   sudo /tmp/backup-001-dynatrace-managed-installer.sh



   --restore



   --cluster-ip "10.176.41.168"



   --cluster-nodes "1:10.176.41.168,3:10.176.41.169,5:10.176.41.170"



   --seed-ip "10.176.41.169"



   --backup-file /mnt/backup/bckp/c9dd47f0-87d7-445e-bbeb-26429fac06c6/node_1/files/19/backup-001.tar
   ```
3. **Запуск firewall, Cassandra и Elasticsearch**
   На каждом узле по очереди запусти firewall, Cassandra и Elasticsearch с помощью скрипта запуска:

   ```
   /opt/dynatrace-managed/launcher/firewall.sh start



   /opt/dynatrace-managed/launcher/cassandra.sh start



   /opt/dynatrace-managed/launcher/elasticsearch.sh start
   ```
4. **Проверка состояния Cassandra**
   На каждом узле проверь, что Cassandra запущена. Выполни команду:
   `<dynatrace-install-dir>/utils/cassandra-nodetool.sh status`

   Все узлы восстановленного кластера должны быть перечислены в ответе со следующими значениями:
   `Status = Up`
   `State = Normal`
5. **Проверка состояния Elasticsearch**
   На каждом узле проверь, что Elasticsearch запущена. Выполни команду:
   `curl -s -N -XGET 'http://localhost:9200/_cluster/health?pretty' | grep status`

   Должен быть получен следующий ответ:
   `"status" : "green"`
   или для конфигурации из одного узла:
   `"status" : "yellow"`
6. **Восстановление базы данных Elasticsearch**
   На seed-узле выполни следующую команду:
   `<dynatrace-install-dir>/utils/restore-elasticsearch-data.sh <path-to-backup>/<UUID>`
7. **Восстановление файлов метрик и данных конфигурации**
   На каждом узле по очереди, начиная с seed-узла, выполни следующую команду:
   `<dynatrace-install-dir>/utils/restore-cassandra-data.sh <path-to-backup>/<UUID>/node_<node-id>/files/<backup-version-number>/backup-001.tar`

   Дождись, пока Cassandra полностью настроит кластер. Используй команду:
   `<dynatrace-install-dir>/utils/cassandra-nodetool.sh status`

   * Должен быть получен следующий ответ:
     `Status = Up`
     `State = Normal`
8. Опционально **Восстановление Cassandra**

   Восстановление можно запускать только на кластерах с более чем одним узлом.

   Последовательно на всех узлах запусти восстановление Cassandra:

   ```
   <dynatrace-install-dir>/utils/repair-cassandra-data.sh
   ```

   Это нужно для обеспечения согласованности данных между узлами. **Выполнение этого шага может занять несколько часов**.

   Чтобы восстановить только данные конфигурации без временных рядов метрик, выполни следующую команду:

   ```
   <dynatrace-install-dir>/utils/repair-cassandra-data.sh 1
   ```
9. **Запуск Dynatrace**

   На каждом узле по очереди, начиная с seed-узла, выполни следующую команду:

   ```
   <dynatrace-install-dir>/launcher/dynatrace.sh start
   ```

   Дождись возможности войти в Cluster Management Console.
10. Опционально **Удаление оставшихся ссылок на старые узлы**
    Если было принято решение восстановить меньше узлов, чем было в исходном кластере, удали узлы, помеченные как `Offline` в Cluster Management Console. Дополнительную информацию см. в разделе [Удаление узла кластера](/managed/managed-cluster/operation/remove-a-cluster-node "Learn how to remove a new cluster node using either the command prompt or the Cluster Management Console.")
11. **Переключение OneAgent на новый адрес кластера**
    Если кластер изначально был настроен с использованием DNS для OneAgent, нужно только обновить DNS-записи, как описано в следующем шаге.

    В противном случае нужно настроить Cluster ActiveGate (или OneAgent, если ActiveGate не используются) с новым целевым адресом и перезапустить их. Если Cluster ActiveGate отсутствуют, но есть Environment ActiveGate, это нужно сделать на Environment ActiveGate.

    В противном случае нужно настроить и перезапустить Cluster ActiveGate (или OneAgent, если ActiveGate не используются) с новым целевым адресом.

    Настройка нового целевого адреса на ActiveGate

    1. Останови службу ActiveGgate.
    2. Обнови параметр `seedServerUrl` в `config.properties`.
    3. Обнови `cluster.properties` новыми URL.
    4. Удали файл `connectivity_history.properties`.

    Выполни следующий вызов API кластера для каждого узла, заменив `<node-id>` на идентификатор узла, `<node-ip>` на IPv4-адрес узла и `<api-token>` на действительный токен Cluster API.

    ```
    curl -ikS -X PUT -d <node-ip> https://<node_ip>:8021/api/v1.0/onpremise/endpoint/publicIp/agents/<node-id>?Api-Token=<api-token> -H  "accept: application/json" -H  "Content-Type: application/json"
    ```

    Должен быть получен ответ `200`, как в примере ниже:

    ```
    HTTP/1.1 200 OK



    Date: Tue, 19 Feb 2019 17:49:06 GMT



    X-Robots-Tag: noindex



    Server: ruxit server



    Content-Length: 0
    ```
12. Опционально **Обновление DNS-записей кластера**
    Если восстановление кластера привело к изменению IP-адресов, обнови DNS-записи.

    * Если используется автоматическое управление доменом и сертификатами, выполни следующий вызов API кластера для каждого узла, заменив `<node-id>` на идентификатор узла, `<node-ip>` на IPv4-адрес узла и `<api-token>` на действительный [токен API](/managed/dynatrace-api/basics/dynatrace-api-authentication "Find out how to get authenticated to use the Dynatrace API.").

    ```
    curl -ikS -X PUT -d <node-ip> https://<node-ip>:8021/api/v1.0/onpremise/endpoint/publicIp/domain/<node-id>?Api-Token=<api-token> -H  "accept: application/json" -H  "Content-Type: application/json"
    ```

    Должен быть получен ответ `200`, как в примере ниже:

    ```
    HTTP/1.1 200 OK



    Date: Tue, 19 Feb 2019 17:49:06 GMT



    X-Robots-Tag: noindex



    Server: ruxit server



    Content-Length: 0
    ```

    * Если используется собственный DNS, обнови домен кластера на новый IP-адрес.
13. **Включение резервного копирования**
    Чтобы предотвратить перезапись предыдущего снимка, резервное копирование автоматически отключается после восстановления. По завершении восстановления резервное копирование нужно включить снова.

    1. Войди в **Cluster Management Console**.
    2. Перейди в **Settings** > **Backup**.
    3. Включи **Enable cluster backup** и подтверди полный путь к архиву резервной копии, а также настрой время ежедневного резервного копирования.

### Отключение резервного копирования вручную

В определённых ситуациях требуется вручную отключить резервное копирование кластера. Например, если точка монтирования общей файловой системы недоступна при загрузке системы, Dynatrace не запустится на этом узле. В этом случае резервное копирование нужно отключить вручную, чтобы дать Dynatrace возможность запуститься.

1. Отредактируй файл `<install-dir>/elasticsearch/config/elasticsearch.yml`.
2. Удали строку с параметром `path.repo:`.
   Например:

   ```
   network.host: [ _local_, "10.10.10.10" ]



   network.publish_host: 10.10.10.10



   path.data: /var/opt/dynatrace-managed/elasticsearch



   path.repo: <REMOVE THIS LINE>
   ```
3. Сохрани файл и перезапусти систему. См. [Запуск/остановка/перезапуск кластера](/managed/managed-cluster/operation/start-stop-restart-node "Properly shut down and restart Dynatrace Managed nodes using command line parameters.")
