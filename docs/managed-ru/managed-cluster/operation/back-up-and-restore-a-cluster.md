# Backup and restore a cluster

# Backup and restore a cluster

* Published Mar 22, 2018

Для настройки автоматического резервного копирования кластера Dynatrace Managed

1. Войдите в **Cluster Management Console**.
2. Перейдите в **Settings** > **Backup**.
3. Включите резервное копирование кластера и выберите область охвата:

   * Пользовательские сессии могут содержать конфиденциальные данные.  
     **Исключите пользовательские сессии из резервной копии для соблюдения требований GDPR**.
   * Исключите данные метрик временных рядов из резервной копии, если исторические данные не актуальны и вы хотите сохранить только данные конфигурации.
   * Включите резервное копирование событий Log Monitoring.
4. (Опционально) Выберите центр обработки данных. Этот шаг необходим только при развёртывании с несколькими центрами обработки данных (Premium High Availability). Подробнее о таких развёртываниях см. в разделах [Высокая доступность для нескольких центров обработки данных](/managed/managed-cluster/high-availability/premium-high-availability "Learn how Dynatrace Premium High Availability provides near-zero downtime and data resilience through multi-data-center deployments for Managed Clusters.") и [Аварийное восстановление из резервной копии](/managed/managed-cluster/high-availability/data-center-disaster-recovery-from-backup "Learn how to recover lost data center from backup in a multiple data center deployment.").
5. Укажите полный путь к смонтированному сетевому хранилищу, где будут храниться архивы резервных копий.
6. Настройте время начала резервного копирования.

## Автоматическое резервное копирование кластера

Вы можете автоматически создавать резервные копии данных конфигурации Dynatrace Managed (правила именования, теги, зоны управления, профили оповещений и другое), данных метрик временных рядов и пользовательских сессий. Для максимальной отказоустойчивости сохраняйте файлы резервных копий в удалённом хранилище.

* Каждый узел должен быть подключён к общей файловой системе — например, Network File Sharing (NFS) — а общая файловая система должна быть смонтирована **в одной и той же общей директории на каждом узле**.  
  Для проверки прав пользователя выполните следующую команду на каждом узле:

  ```
  su - dynatrace -s /bin/bash -c "touch /nfs/dynatrace/backup/$(uname -n)"



  ls -ltr /path/to/backup/
  ```
* Пользователь, запускающий сервисы Dynatrace, должен иметь права на чтение и запись в общей файловой системе.
* Точка монтирования общей файловой системы должна быть доступна при перезагрузке системы.
* Точку монтирования можно добавить в `fstab` или использовать инструмент управления дисками для обеспечения постоянного монтирования общей файловой системы.
* Протокол передачи данных зависит от вашей конфигурации. Рекомендуем NFSv4. CIFS не рекомендуется из-за низкой производительности и надёжности.

Точка монтирования общей файловой системы при загрузке системы

Если точка монтирования общей файловой системы недоступна при загрузке системы, Dynatrace не запустится на этом узле. Это может привести к недоступности кластера. Для запуска Dynatrace необходимо вручную отключить резервное копирование.

### Хранилище метрик и конфигурации

Dynatrace сохраняет предыдущую резервную копию до завершения создания новой.

#### Характеристики резервной копии

* Снимок выполняется ежедневно.
* Все данные, реплицированные между узлами, также включаются в резервную копию (дедупликация не выполняется).
* Dynatrace исключает наиболее часто изменяемые семейства столбцов (исключаемые семейства составляют около 80% общего объёма хранилища), а также данные с разрешением 1 минута и 5 минут.

### Elasticsearch

Файлы Elasticsearch хранятся в несжатом бинарном формате. Хотя данные реплицируются между узлами и существуют два дополнительных реплика помимо первичного шарда, резервная копия не включает реплицированные данные.

Не удаляйте резервные копии Elasticsearch

Elasticsearch выполняет резервное копирование инкрементально, поэтому необходима возможность добавлять последние изменения к предыдущей резервной копии. Не удаляйте файлы резервных копий Elasticsearch.

#### Характеристики резервной копии

* По умолчанию снимок создаётся каждые 2 часа и является инкрементальным.  
  Изначально Dynatrace копирует весь набор данных, а затем создаёт снимки изменений. Более старые снимки постепенно удаляются по истечении пяти (5) дней.
* Поскольку Dynatrace сохраняет часть более старых снимков, размер резервной копии растёт независимо от текущего размера данных на диске. Снимки инкрементальные, однако Elasticsearch объединяет сегменты данных со временем, что приводит к появлению определённых дубликатов в резервной копии.

Резервное копирование хранилища транзакций не выполняется

Данные хранилища транзакций не включаются в резервную копию, поэтому после восстановления резервной копии могут наблюдаться пробелы в подробных данных мониторинга (например, в распределённых трассировках и трассировках уровня кода). По умолчанию данные хранилища транзакций хранятся только 10 дней. С долгосрочной точки зрения включать данные хранилища транзакций в резервные копии не требуется.

## Восстановление кластера

Для восстановления кластера выполните следующие шаги.

### Прежде чем начать

* Для восстановления кластера на том же хосте, что и исходный кластер, сначала деинсталлируйте его.
* Убедитесь, что машины, подготовленные для восстановления кластера, имеют схожую аппаратную конфигурацию и разметку дисков, что и исходный кластер, а также достаточную ёмкость для обработки нагрузки после восстановления.

  Рекомендуем восстанавливать кластер с тем же количеством узлов, что и в резервной копии. В исключительных случаях допускается восстановление на кластер, содержащий до двух узлов меньше, чем в резервной копии. Попытка восстановления на кластер с более чем двумя отсутствующими узлами по сравнению с оригинальным создаёт риск потери конфигурации кластера.
* Убедитесь, что существующий кластер остановлен, чтобы исключить подключение двух кластеров с одним и тем же ID к Dynatrace Mission Control. См. раздел [Запуск/остановка/перезапуск кластера](/managed/managed-cluster/operation/start-stop-restart-node "Properly shut down and restart Dynatrace Managed nodes using command line parameters.").
* Убедитесь, что системные пользователи, созданные для Dynatrace Managed, имеют одинаковые идентификаторы UID:GID на всех узлах.
* На каждом целевом узле смонтируйте хранилище резервных копий, например, в `/mnt/backup`. В дальнейших шагах этот путь обозначается как `<path-to-backup>`.
* Убедитесь, что установщик имеет права на чтение NFS. Например:
  `sudo adduser dynatrace && sudo chown -R dynatrace:dynatrace <path-to-backup>`
* Составьте инвентаризацию кластера. Эти данные потребуются в процессе восстановления.

  + Идентификаторы узлов в кластере. Резервная копия каждого узла хранится в отдельной директории, имя которой основано на идентификаторе узла в формате `node_<node-id>` (например, `node_1`, `node_5` и т.д.).
  + IPv4-адреса новых машин.
  + IP-адрес целевой машины для каждого узла.
  + Идентификатор узла, который станет seed-узлом в кластере.

### Восстановление из резервной копии

Для восстановления кластера выполните следующие шаги:

1. **Скопируйте установщик на целевые узлы**  
   Для восстановления кластера необходимо использовать точно ту же версию установщика, что и в оригинальной инсталляции. Скопируйте установщик из `<path-to-backup>/<UUID>/node_<node-id>/` на локальный диск каждого целевого узла.

   Например: `cp <path-to-backup>/<UUID>/node_<node-id>/files/<backup-version-number>/backup-001-dynatrace-managed-installer.sh /tmp/`
2. **Запустите восстановление Dynatrace на каждом узле**  
   Параллельно на каждом узле запустите установщик Dynatrace Managed со следующими параметрами:

   * `--restore` — переключает установщик в режим восстановления.
   * `--cluster-ip` — IPv4-адрес узла, на котором запускается установщик.
   * `--cluster-nodes` — разделённый запятыми список идентификаторов и IP-адресов всех узлов кластера, включая тот, на котором запускается установщик, в формате `<node-id>:<node-ip>,<node-id>:<node-ip>`.
   * `--seed-ip` — IPv4-адрес seed-узла.
   * `backup-file` — путь к файлу резервной копии `*.tar`, включающий путь к точке монтирования общего хранилища, UUID кластера, идентификатор узла, номер версии резервной копии и файл `*.tar` в следующем формате:  
     `<path-to-backup>/<UUID>/node_<node-id>/files/<backup-version-number>/<backup-file>`

   Пример пути к резервной копии

   В данном примере пути:

   `/mnt/backup/bckp/c9dd47f0-87d7-445e-bbeb-26429fac06c6/node_1/files/19/backup-001.tar`

   части пути имеют следующий смысл:

   * `<path-to-backup>` = `/mnt/backup/bckp/`
   * `<UUID>` = `c9dd47f0-87d7-445e-bbeb-26429fac06c6`
   * `<node-id>` = `1`
   * `<backup-version-number>` = `19`

   Пока выполняется резервное копирование, могут присутствовать две директории резервных копий с разными номерами версий:

   * Директория с меньшим номером версии содержит старую резервную копию. Она будет удалена после завершения резервного копирования.
   * Директория с большим номером версии содержит резервную копию в процессе создания.

   Номер версии резервной копии автоматически увеличивается с каждым запуском резервного копирования.

   Получите идентификаторы и IP-адреса из инвентаризации, составленной до начала восстановления.

   Например:  
   `10.176.41.168` — IP-адрес узла для восстановления  
   `1: 10.176.41.168, 3: 10.176.41.169, 5: 10.176.41.170` — идентификаторы узлов и новые IP-адреса всех узлов кластера

   ```
   sudo /tmp/backup-001-dynatrace-managed-installer.sh



   --restore



   --cluster-ip "10.176.41.168"



   --cluster-nodes "1:10.176.41.168,3:10.176.41.169,5:10.176.41.170"



   --seed-ip "10.176.41.169"



   --backup-file /mnt/backup/bckp/c9dd47f0-87d7-445e-bbeb-26429fac06c6/node_1/files/19/backup-001.tar
   ```
3. **Запустите фаервол, Cassandra и Elasticsearch**  
   Поочерёдно на каждом узле запустите фаервол, Cassandra и Elasticsearch с помощью скрипта launcher:

   ```
   /opt/dynatrace-managed/launcher/firewall.sh start



   /opt/dynatrace-managed/launcher/cassandra.sh start



   /opt/dynatrace-managed/launcher/elasticsearch.sh start
   ```
4. **Проверьте состояние Cassandra**  
   На каждом узле проверьте, запущена ли Cassandra. Выполните команду:
   `<dynatrace-install-dir>/utils/cassandra-nodetool.sh status`

   Все узлы восстановленного кластера должны отображаться в ответе со следующими значениями:  
   `Status = Up`
   `State = Normal`
5. **Проверьте состояние Elasticsearch**  
   На каждом узле проверьте, запущен ли Elasticsearch. Выполните команду:  
   `curl -s -N -XGET 'http://localhost:9200/_cluster/health?pretty' | grep status`

   Ожидаемый ответ:  
   `"status" : "green"`  
   или для одноузлового кластера:  
   `"status" : "yellow"`
6. **Восстановите базу данных Elasticsearch**  
   На seed-узле выполните следующую команду:
   `<dynatrace-install-dir>/utils/restore-elasticsearch-data.sh <path-to-backup>/<UUID>`
7. **Восстановите файлы метрик и данных конфигурации**  
   Поочерёдно на каждом узле, начиная с seed-узла, выполните следующую команду:
   `<dynatrace-install-dir>/utils/restore-cassandra-data.sh <path-to-backup>/<UUID>/node_<node-id>/files/<backup-version-number>/backup-001.tar`

   Дождитесь полной готовности кластера Cassandra. Используйте команду:
   `<dynatrace-install-dir>/utils/cassandra-nodetool.sh status`

   * Ожидаемый ответ:
     `Status = Up`
     `State = Normal`
8. Опционально **Выполните восстановление Cassandra**

   Восстановление можно выполнять только на кластерах с более чем одним узлом.

   Последовательно на всех узлах запустите восстановление Cassandra:

   ```
   <dynatrace-install-dir>/utils/repair-cassandra-data.sh
   ```

   Это необходимо для обеспечения согласованности данных между узлами. **Выполнение этого шага может занять несколько часов**.

   Для восстановления только данных конфигурации без метрик временных рядов выполните следующую команду:

   ```
   <dynatrace-install-dir>/utils/repair-cassandra-data.sh 1
   ```
9. **Запустите Dynatrace**

   Поочерёдно на каждом узле, начиная с seed-узла, выполните следующую команду:

   ```
   <dynatrace-install-dir>/launcher/dynatrace.sh start
   ```

   Дождитесь возможности входа в Cluster Management Console.
10. Опционально **Удалите оставшиеся ссылки на старые узлы**  
    Если вы решили восстановить кластер с меньшим количеством узлов, чем в исходном, удалите узлы, помеченные как `Offline` в Cluster Management Console. Подробнее см. раздел [Удаление узла кластера](/managed/managed-cluster/operation/remove-a-cluster-node "Learn how to remove a new cluster node using either the command prompt or the Cluster Management Console.").
11. **Переключите OneAgents на новый адрес кластера**  
    Если кластер был изначально настроен с использованием DNS для OneAgents, достаточно обновить записи DNS, как описано в следующем шаге.

    В противном случае необходимо настроить Cluster ActiveGates (или OneAgents, если ActiveGates не используются) с новым целевым адресом и перезапустить их. Если Cluster ActiveGates отсутствуют, но есть Environment ActiveGates, это следует сделать на Environment ActiveGates.

    В противном случае необходимо настроить и перезапустить Cluster ActiveGates (или OneAgents, если ActiveGates не используются) с новым целевым адресом.

    Настройка нового целевого адреса на ActiveGate

    1. Остановите сервис ActiveGate.
    2. Обновите параметр `seedServerUrl` в `config.properties`.
    3. Обновите `cluster.properties` с новыми URL.
    4. Удалите файл `connectivity_history.properties`.

    Выполните следующий вызов API кластера для каждого узла, заменив `<node-id>` на идентификатор узла, `<node-ip>` на IPv4-адрес узла и `<api-token>` на действующий токен Cluster API.

    ```
    curl -ikS -X PUT -d <node-ip> https://<node_ip>:8021/api/v1.0/onpremise/endpoint/publicIp/agents/<node-id>?Api-Token=<api-token> -H  "accept: application/json" -H  "Content-Type: application/json"
    ```

    Ожидаемый ответ `200`, как в примере ниже:

    ```
    HTTP/1.1 200 OK



    Date: Tue, 19 Feb 2019 17:49:06 GMT



    X-Robots-Tag: noindex



    Server: ruxit server



    Content-Length: 0
    ```
12. Опционально **Обновите DNS-записи кластера**  
    Если в результате восстановления кластера изменились IP-адреса, обновите DNS-записи.

    * Если используется автоматическое управление доменом и сертификатами, выполните следующий вызов API кластера для каждого узла, заменив `<node-id>` на идентификатор узла, `<node-ip>` на IPv4-адрес узла и `<api-token>` на действующий [токен API](/managed/dynatrace-api/basics/dynatrace-api-authentication "Find out how to get authenticated to use the Dynatrace API.").

    ```
    curl -ikS -X PUT -d <node-ip> https://<node-ip>:8021/api/v1.0/onpremise/endpoint/publicIp/domain/<node-id>?Api-Token=<api-token> -H  "accept: application/json" -H  "Content-Type: application/json"
    ```

    Ожидаемый ответ `200`, как в примере ниже:

    ```
    HTTP/1.1 200 OK



    Date: Tue, 19 Feb 2019 17:49:06 GMT



    X-Robots-Tag: noindex



    Server: ruxit server



    Content-Length: 0
    ```

    * Если используется собственная DNS, обновите домен кластера на новый IP-адрес.
13. **Включите резервное копирование**  
    Во избежание перезаписи предыдущего снимка резервное копирование автоматически отключается после восстановления. После завершения восстановления необходимо снова включить резервное копирование.

    1. Войдите в **Cluster Management Console**.
    2. Перейдите в **Settings** > **Backup**.
    3. Включите **Enable cluster backup** и подтвердите полный путь к архиву резервной копии и расписание ежедневного резервного копирования.

### Ручное отключение резервного копирования

В ряде ситуаций требуется вручную отключить резервное копирование кластера. Например, если точка монтирования общей файловой системы недоступна при загрузке системы, Dynatrace не запустится на этом узле. В таком случае необходимо вручную отключить резервное копирование, чтобы Dynatrace смог запуститься.

1. Отредактируйте файл `<install-dir>/elasticsearch/config/elasticsearch.yml`.
2. Удалите строку с параметром `path.repo:`.
   Например:

   ```
   network.host: [ _local_, "10.10.10.10" ]



   network.publish_host: 10.10.10.10



   path.data: /var/opt/dynatrace-managed/elasticsearch



   path.repo: <УДАЛИТЬ ЭТУ СТРОКУ>
   ```
3. Сохраните файл и перезагрузите систему. См. раздел [Запуск/остановка/перезапуск кластера](/managed/managed-cluster/operation/start-stop-restart-node "Properly shut down and restart Dynatrace Managed nodes using command line parameters.")
