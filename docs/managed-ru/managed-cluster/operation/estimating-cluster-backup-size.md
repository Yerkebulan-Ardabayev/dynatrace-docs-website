# Estimate cluster backup size

# Estimate cluster backup size

* Published Aug 05, 2019

Следуйте приведённым инструкциям для оценки размера резервной копии кластера по хранилищу метрик и хранилищу Elasticsearch. Общий размер резервной копии кластера можно приблизительно оценить как сумму оценок для хранилища метрик и хранилища Elasticsearch.

### Общая формула

1. Оцените размер резервной копии хранилища метрик.

   Как правило, он составляет 20% от суммы хранилища метрик на всех узлах. Количество узлов не влияет на формулу.

   Доступное пространство для хранения резервных копий

   Текущая резервная копия хранилища метрик сохраняется до завершения следующего резервного копирования и создания новой резервной копии. До удаления первой резервной копии необходимое пространство для хранения резервной копии хранилища метрик, таким образом, удваивается по сравнению с фактическим размером резервной копии. По этой причине рекомендуем закладывать удвоенный расчётный объём для хранения резервной копии хранилища метрик.
2. Оцените размер резервной копии хранилища Elasticsearch.

   Расчётный необходимый размер резервной копии Elasticsearch определяется исходя из размера хранилища Elasticsearch. Как правило, он меньше суммы хранилища Elasticsearch на всех узлах. Хотя резервная копия Elasticsearch не содержит реплики данных, для оценки используется общий объём хранилища — из-за инкрементальных снимков.
3. Оцените размер резервной копии кластера.

   На основе предыдущих оценок рассчитайте расчётный размер резервной копии как 20% хранилища метрик + хранилище Elasticsearch.

### Пример расчёта

Предположим, требуется оценить размер резервной копии для кластера из трёх узлов:

1. Рассчитайте объём хранилища метрик для всего кластера.  
   Проверьте размер хранилища метрик на диске **на каждом узле**. Значения должны незначительно отличаться между узлами.

   ```
   du -sh /var/opt/dynatrace-managed/cassandra/



   885GB
   ```

   В данном примере кластер состоит из трёх узлов, и на одном узле 885 ГБ хранилища метрик, поэтому можно оценить общий объём хранилища метрик как: `3 * 885 ГБ = 2,6 ТБ`.
2. Рассчитайте объём хранилища Elasticsearch для всего кластера.  
   Проверьте размер хранилища Elasticsearch на диске **на каждом узле**. Значения должны незначительно отличаться между узлами.

   ```
   du -sh /var/opt/dynatrace-managed/elasticsearch/



   1.5TB
   ```

   В данном примере кластер состоит из трёх узлов, и на одном узле 1,5 ТБ хранилища Elasticsearch, поэтому можно оценить общий объём хранилища Elasticsearch как: `3 * 1,5 ТБ = 4,5 ТБ`.
3. Имея оценки для хранилища метрик и Elasticsearch, рассчитаем размер резервной копии как 20% от хранилища метрик + оценку хранилища Elasticsearch: `(2,6 ТБ * 0,2) + 4,5 ТБ = 5,02 ТБ`.

   Расчётный **размер одной резервной копии** составил 5,02 ТБ. Для оценки **минимального необходимого объёма**, который следует выделить под резервные копии, необходимо удвоить резервную копию хранилища метрик. Итого: `4,5 ТБ + ((2,6 ТБ * 0,2) * 2) = 5,54 ТБ`.

Точность оценки

Данная оценка будет неточной, если расчёт выполнялся на новом кластере, который только начал накапливать данные. По мере роста хранилища на диске будет увеличиваться и размер резервной копии.

### Резервная копия только конфигурации

Занимает до 10% хранилища метрик вместо 20% (то есть на 50% меньше хранилища метрик).

### Исключение пользовательских сессий

В средах с высокой интенсивностью RUM (50 тыс. пользовательских сессий в минуту) пользовательские сессии могут занимать до 99% общего объёма хранилища Elasticsearch. Поэтому их исключение может полностью убрать оценку хранилища Elasticsearch из общей формулы. В других средах сложно оценить снижение размера резервной копии.
