# Migrate a cluster

# Migrate a cluster

* Updated on Jul 17, 2025

Варианты миграции кластера зависят от особенностей вашей среды развёртывания (задержка, расположение и объём трафика), допустимого времени простоя и политик работы с данными. Рекомендуем выполнять миграцию кластеров Dynatrace Managed одним из следующих способов:

* [Использовать процедуры резервного копирования и восстановления кластера](#backup-method)
* [Использовать встроенный механизм репликации узлов](#replication-method)
* [Использовать перенастройку IP-адреса](/managed/managed-cluster/operation/ip-reconfiguration "Learn how to reconfigure your node's IP address.")

Каждый метод имеет преимущества и недостатки, описанные ниже.

## Метод резервного копирования

Этот вариант предполагает создание резервной копии исходного кластера, перенос данных резервной копии и её восстановление на целевом кластере.  
Перед выбором этого варианта миграции учтите следующее:

#### Преимущества

* Поскольку вы вручную переносите данные кластера на целевой кластер, задержка в вашей среде развёртывания не влияет на этот процесс. Рекомендуем этот тип миграции для развёртываний с **высокой задержкой**.

#### Недостатки

* Мониторинг будет недоступен с момента начала резервного копирования кластера до завершения его восстановления на целевом кластере.
* По умолчанию резервное копирование Cassandra выполняется ежедневно, поэтому точка восстановления кластера соответствует последней выполненной резервной копии, что может привести к потере данных за период до 24 часов. Можно запланировать миграцию ближе ко времени резервного копирования, чтобы минимизировать потери данных, однако с учётом времени ручного переноса данных и восстановления нового кластера определённые потери данных неизбежны.
* Поскольку хранилище транзакций (данные распределённых трассировок с типичным сроком хранения 10 дней) не включается в резервную копию, данные хранилища транзакций теряются при данном методе миграции.

### Миграция кластера с использованием резервного копирования и восстановления

Для миграции узлов Dynatrace Managed с использованием метода резервного копирования и восстановления:

1. Настройте целевые узлы и все необходимые сетевые компоненты в целевой среде развёртывания.

   * Убедитесь, что целевые узлы соответствуют аппаратным требованиям.
   * В типичном сценарии предполагается, что количество целевых узлов равно количеству исходных узлов или превышает его.
2. Создайте резервную копию текущего кластера Dynatrace Managed.

   * См. раздел [Резервное копирование кластера](/managed/managed-cluster/operation/back-up-and-restore-a-cluster#back-up "Understand the steps and commands required to restore a Dynatrace Managed cluster.")
3. Остановите текущий кластер Dynatrace Managed.
   Войдите на Linux-машину, где установлены узлы Dynatrace Managed, с правами root и выполните команду `dynatrace.sh` с параметром `stop`.
   По умолчанию скрипт расположен в `/opt/dynatrace-managed/launcher`.

   ```
   [root@localhost]# dynatrace.sh stop
   ```
4. Выполните восстановление из резервной копии на целевой кластер Dynatrace Managed.

   * См. раздел [Восстановление кластера](/managed/managed-cluster/operation/back-up-and-restore-a-cluster#restore "Understand the steps and commands required to restore a Dynatrace Managed cluster.")

## Метод репликации

Метод репликации гарантирует непрерывность мониторинга в процессе миграции. Вы временно расширяете кластер, добавляя новые узлы, ждёте автоматической репликации данных на новые узлы и после завершения репликации поочерёдно удаляете старые узлы.

Миграция в другой центр обработки данных

Метод репликации применим даже в случае, если целевые узлы находятся в другом центре обработки данных. Его можно использовать для центров обработки данных в непосредственной близости друг от друга, с RTT между узлами не более 10 мс и достаточной пропускной способностью между центрами.

В сценариях, где прямое соединение между центрами обработки данных отсутствует, **используйте VPN для связи между центрами**.

Поскольку кластер требует постоянного соединения между узлами, сетевое соединение между центрами обработки данных должно быть стабильным и защищённым. Учтите, что трафик внутри кластера (то есть между узлами) не шифруется. Поэтому для межцентрового трафика требуется зашифрованное соединение, например с использованием VPN.

#### Преимущества

* **Нулевое время простоя** и **отсутствие потери данных**.

#### Недостатки

* Метод репликации хорошо работает только для развёртываний с **низкой задержкой** или **низким трафиком**.
* Ориентировочное время репликации, включая решардинг данных и ожидание истечения срока хранения нереплицированных данных хранилища транзакций, как правило, составляет 10 дней.

### Миграция кластера с использованием репликации

Для миграции узлов Dynatrace Managed с использованием метода репликации:

1. Настройте целевые узлы и все необходимые сетевые компоненты.

   В типичном сценарии предполагается, что количество целевых узлов не меньше количества исходных узлов. Убедитесь, что целевые узлы соответствуют аппаратным требованиям и что между текущими узлами и целевыми узлами есть неограниченное сетевое соединение. Все порты, открытые на текущих узлах, должны быть также открыты на новых целевых узлах.

   * См. раздел [Сетевые порты Dynatrace Server](/managed/managed-cluster/installation/cluster-node-ports "Review the network ports required by Dynatrace Managed and configure your firewall for inbound and outbound communication.")
2. Поочерёдно добавляйте новые узлы Dynatrace Managed в текущий кластер.

   Перед добавлением очередного узла убедитесь, что все существующие узлы кластера работоспособны и доступны. При миграции нескольких узлов убедитесь, что новый узел полностью запущен, прежде чем переходить к следующему.
3. [Отключите трафик OneAgent](/managed/managed-cluster/configuration/cluster-node-capabilities "Find out how to enable/disable a cluster node via the Web UI or API call") на исходных узлах, но не останавливайте их до завершения репликации данных и истечения срока хранения нереплицированных данных хранилища транзакций.

   Для проверки прогресса репликации данных выполните на новом узле команду `cassandra-nodetool.sh` с параметром `status`:

   ```
   sudo $DT_DIR/utils/cassandra-nodetool.sh status
   ```

   Результат должен выглядеть примерно так:

   ```
   Datacenter: datacenter1



   ===============Status=Up/Down



   / State=Normal/Leaving/Joining/Moving



   â Address Load Tokens Owns (effective) Host ID Rack



   UN 10.176.41.167 18.82 GB 256 100.0% 3af25127-4f99-4f43-afc3-216d7a2c10f8 rack1



   UN 10.176.41.154 19.44 GB 256 100.0% 5a618559-3a73-42ec-83f0-32d28e08beec rack1
   ```

   Значение **Load** не должно существенно отличаться между узлами, а в поле **Status** у всех узлов должно отображаться `UN`.

   ![Изображение: узел присоединяется к кластеру](https://dt-cdn.net/images/managednodejoining-400-23d1badb9f.png)

   Изображение: узел присоединяется к кластеру

   Хранилище транзакций

   Отключение узла фактически переводит хранилище транзакций в режим **только для чтения**, однако узел по-прежнему способен обрабатывать запросы.
4. Поочерёдно удаляйте исходные узлы.

   Рекомендуется отложить удаление исходного (старого) узла до истечения срока хранения нереплицированных данных хранилища транзакций. Срок хранения этих данных настраивается, однако как правило составляет 10 дней.

   Dynatrace не позволяет удалять несколько узлов одновременно, поэтому убедитесь, что решардинг данных завершён перед удалением каждого последующего узла. Решардинг данных обычно занимает несколько часов в зависимости от объёма данных и пропускной способности сети между узлами. В крайних случаях он может занять несколько дней.
