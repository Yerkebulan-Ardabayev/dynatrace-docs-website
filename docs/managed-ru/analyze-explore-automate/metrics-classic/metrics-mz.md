# Фильтрация метрик по зоне управления

* Чтение: 4 мин
* Обновлено 12 июля 2023 г.

Данные метрик можно фильтровать по [зоне управления](/managed/manage/identity-access-management/permission-management/management-zones "Узнайте о концепции зон управления, о том, как их определять и как максимально использовать.") двумя способами.

* Первый способ — настроить [правила на основе сущностей](#entity-based-rules) для зоны управления, чтобы сопоставлять метрики с соответствующим типом сущности.
* Второй способ — настроить [размерные правила](#dimensional-rules) для зоны управления, чтобы сопоставлять ряды метрик по ключам метрик или измерений.

В следующих разделах описан каждый метод и [соображения безопасности](#security) при фильтрации данных метрик по зоне управления.

## Правила на основе сущностей для фильтрации данных метрик

Можно определить [правила через пользовательский интерфейс](/managed/manage/identity-access-management/permission-management/management-zones/management-zone-rules#ui "Определите правила для ограничения сущностей, доступных в зоне управления.") или [текстовые правила](/managed/manage/identity-access-management/permission-management/management-zones/management-zone-rules#text "Определите правила для ограничения сущностей, доступных в зоне управления.") для предоставления доступа к сущностям в зоне управления. Пользователи зоны управления автоматически получают доступ к метрикам, **тип сущности** которых соответствует сущностям в данной зоне. Обратите внимание, что тип сущности метрики — это основная сущность, связанная с этой метрикой. Хотя метрика может иметь несколько **измерений на основе сущностей**, это не то же самое, что тип сущности.

* [Конечной точки GET дескриптора метрики в Metrics API](/managed/dynatrace-api/environment-api/metric-v2/get-descriptor "Просмотр дескриптора метрики через Metrics v2 API."); проверьте [элемент `entityType`](/managed/dynatrace-api/environment-api/metric-v2/get-descriptor#response "Просмотр дескриптора метрики через Metrics v2 API.").

### Ограничения

Существуют два ограничения при фильтрации данных метрик в зонах управления с помощью правил на основе сущностей.

1. Фильтрация метрик по правилам на основе сущностей работает только для **типа сущности** метрики. Метрика может иметь несколько **измерений на основе сущностей**, таких как `dt.entity.host` и `dt.entity.disk`, но у большинства метрик только один тип сущности. Если правило зоны управления на основе сущностей ссылается на сущность, которая не является типом сущности метрики, ряды метрики не будут видны в зоне управления, даже если измерение метрики на основе сущностей совпадает с правилом.

   Например, если правило зоны управления сопоставляет сущности `QUEUE`, начинающиеся с определённой строки, метрика `builtin:queue.incoming_requests` не будет видна в зоне управления. Это связано с тем, что, несмотря на наличие измерений на основе сущностей `dt.entity.service` и `dt.entity.queue`, тип сущности метрики — `SERVICE`, который не соответствует правилу для сущностей `QUEUE`.
2. Если метрика имеет **несколько типов сущностей**, фильтрация по зоне управления не применяется. Это ограничение затрагивает метрики с измерениями `dt.entity.monitored_entity` или `dt.entity.device_application`.

## Размерные правила для данных метрик

Если тип сущности метрики не соответствует правилу на основе сущностей в зоне управления, можно определить размерное правило для сопоставления **измерения метрики на основе сущностей**. Например, для метрики `builtin:queue.incoming_requests` необходимо определить размерное правило для измерения `dt.entity.queue`. Обратите внимание, что значение условия измерения должно быть идентификатором сущности (например, `'dt.entity.queue' equals 'QUEUE-43717C10AF1BD1E0'`), а не именем сущности (см. изображение ниже).

## Соображения безопасности

Ниже приведены соображения безопасности при отображении данных метрик и меры по снижению рисков.

### Ряды метрик, видимые во всех зонах управления

Фильтрация по зоне управления не работает для метрик с несколькими типами сущностей. Следующие метрики подвержены этому ограничению:

* `builtin:billing.ddu.metrics.byEntity`
* `builtin:billing.ddu.metrics.byEntityRaw`
* `builtin:billing.ddu.log.byEntity`
* `builtin:billing.ddu.events.byEntity`
* `builtin:billing.ddu.serverless.byEntity`
* `builtin:billing.ddu.traces.byEntity`
* Несколько метрик `builtin:apps.other.` с измерением `dt.entity.device_application`

Из-за этого ограничения пользователи, которым не должен быть предоставлен доступ к этим рядам метрик, могут всё равно их видеть независимо от зоны управления.

### Преобразования селектора метрик

[Преобразования селектора метрик](/managed/dynatrace-api/environment-api/metric-v2/metric-selector "Настройте селектор метрик для Metric v2 API.") `parents` и `names` добавляют информацию к запросам данных метрик, что может раскрыть идентификатор родительской отслеживаемой сущности (например, идентификатор хоста, являющегося родительским для процесса) или отображаемое имя сущности в случае преобразования `names`.

### Последствия

Несмотря на то что эти оговорки заслуживают внимания, практическое воздействие на безопасность ограничено. Данные [метрик с несколькими типами сущностей](#multiple-entity-types), как правило, не являются конфиденциальными. Для сред с повышенными требованиями к безопасности рекомендуется учитывать особенности [преобразований селектора метрик](#metric-selector-transformations) (например, идентификаторы сущностей, такие как `HOST-27A71FA663E7F352`, и отображение имён хостов) при предоставлении пользователю разрешения на чтение данных метрик.

Рекомендуется особо тщательно выбирать, какие измерения передаются в составе данных для **конфиденциальных пользовательских метрик**, чтобы обеспечить эффективность желаемых ограничений доступа на чтение.
