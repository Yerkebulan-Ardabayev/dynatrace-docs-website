# Методы маскирования чувствительных данных (Logs Classic)

* Чтение: 4 мин
* Обновлено 26 февраля 2026 г.

Log Monitoring Classic

Журналы могут содержать чувствительные данные, требующие маскирования перед использованием для аналитики, наблюдаемости, безопасности и других целей. Dynatrace предоставляет инструменты, позволяющие выполнять требования по защите данных и соответствию нормативным требованиям, продолжая извлекать ценность из журналов.

Выберите один или оба метода в зависимости от информационной архитектуры и конфигурации журналов:

* **Маскирование при захвате**  
  Этот подход позволяет маскировать чувствительные данные в вашем окружении, на хостах или процессах до их передачи в Dynatrace SaaS с помощью OneAgent.
* **Маскирование при приёме**  
  Этот подход позволяет маскировать чувствительные данные после их поступления в Dynatrace SaaS и до записи на диск (хранения).

Рекомендуется сочетать маскирование данных при захвате с маскированием при приёме для создания двух уровней защиты.

## Маскирование журналов при захвате

Маскирование при захвате предполагает идентификацию и маскирование чувствительных частей записей журнала до их передачи в Dynatrace. Для этого можно использовать OneAgent для сбора журналов. OneAgent имеет встроенный механизм маскирования чувствительных данных, который можно детально настроить на уровне хоста, группы хостов или окружения. Используйте маскирование при захвате с OneAgent для:

* Гарантии того, что чувствительные данные никогда не покинут ваше окружение.
* Избегания использования нескольких инструментов.
* Упрощения конфигурации.

Для маскирования данных таким способом следуйте инструкциям в разделе [Маскирование чувствительных данных (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/log-monitoring-configuration/sensitive-data-masking "Маскируйте чувствительную информацию в данных журнала с помощью Log Monitoring Classic.").

Маскирование перед отправкой журналов в общий API приёма

Если журналы отправляются в общий API приёма Dynatrace и требуется маскировать чувствительные данные при захвате, необходимо либо:

* Маскировать данные путём настройки источника журналов
* Либо настроить отправщик или перенаправитель журналов, маскирующий чувствительные данные перед отправкой в Dynatrace (в вашем окружении)

## Маскирование журналов при приёме

С помощью этого метода можно маскировать данные после их поступления в Dynatrace, устанавливая правила обработки журналов. После обработки данные отправляются на хранение и доступны для дальнейшего анализа. Ключевое преимущество этого метода состоит в том, что он поддерживает потоки данных из всех каналов приёма журналов, включая:

* OneAgent
* Общий API приёма
* Расширения Dynatrace

Для маскирования чувствительных данных в Dynatrace этим методом необходимо создать правило обработки, специфичное для атрибутов, которые нужно замаскировать. См. [Log Monitoring Classic](/managed/analyze-explore-automate/log-monitoring/log-processing#lprules "Создайте правила обработки журналов, преобразующие входящие данные журнала для лучшего анализа или дальнейшей обработки."). Например, для маскирования IP-адреса можно задать правила обработки, как показано в примере ниже.

### Пример: маскирование любого атрибута

Если требуется изменить содержимое или любой другой атрибут, его необходимо объявить как `INOUT` (доступный для записи) с помощью команды `USING`. Функция `REPLACE_PATTERN` — очень мощный инструмент, полезный для маскирования части атрибута.

* В следующем примере маскируется IP-адрес: последний октет заменяется значением 0.

  Определение правила обработки:

  ```
  USING(INOUT ip)



  | FIELDS_ADD(ip: IPADDR(ip) & 0xFFFFFF00l)
  ```

  Пример данных журнала:

  ```
  {



  "content":"Lorem ipsum",



  "timestamp": "1656009021053",



  "ip": "192.168.0.12"



  }
  ```

  Результат после преобразования:

  ```
  {



  "content": "Lorem ipsum",



  "timestamp": "1656009021053",



  "ip": "192.168.0.0"



  }
  ```
* В следующем примере маскируется IP-адрес: последний октет заменяется значением `xxx`.

  Определение правила обработки:

  ```
  USING(INOUT ip)



  | FIELDS_ADD(ip: REPLACE_PATTERN(ip, "(INT'.'INT'.'INT'.'):not_masked INT", "${not_masked}xxx"))
  ```

  Пример данных журнала:

  ```
  {



  "content":"Lorem ipsum",



  "timestamp": "1656009021053",



  "ip": "192.168.0.12"



  }
  ```

  Результат после преобразования:

  ```
  {



  "content": "Lorem ipsum",



  "timestamp": "1656009021053",



  "ip": "192.168.0.xxx"



  }
  ```
* В следующем примере весь адрес электронной почты маскируется с использованием `sha1` (Secured Hash Algorithm).

  Определение правила обработки:

  ```
  USING(INOUT email)



  | FIELDS_ADD(email: REPLACE_PATTERN(email, "LD:email_to_be_masked", "${email_to_be_masked|sha1}"))
  ```

  Пример данных журнала:

  ```
  {



  "content":"Lorem ipsum",



  "timestamp": "1656009924312",



  "email": "john.doe@dynatrace.com"



  }
  ```

  Результат после преобразования:

  ```
  {



  "content": "Lorem ipsum",



  "timestamp": "1656009924312",



  "email": "9940e79e41cbf7cc452b137d49fab61e386c602d"



  }
  ```
* В следующем примере из поля содержимого маскируются IP-адрес, адрес электронной почты и номер кредитной карты.

  Определение правила обработки:

  ```
  USING(INOUT content)



  | FIELDS_ADD(content: REPLACE_PATTERN(content, "



  (LD 'ip: '):p1                                   // Lorem ipsum ip:



  (INT'.'INT'.'INT'.'):ip_not_masked               // 192.168.0.



  INT                                              // 12



  ' email: ':p2                                    //  email:



  LD:email_name '@' LD:email_domain                // john.doe@dynatrace.com



  ' card number: ': p3                             //  card number:



  CREDITCARD:card                                  // 4012888888881881



  ", "${p1}${ip_not_masked}xxx${p2}${email_name|md5}@${email_domain}${p3}${card|sha1}"))
  ```

  Пример данных журнала:

  ```
  {



  "timestamp": "1656010291511",



  "content": "Lorem ipsum ip: 192.168.0.12 email: john.doe@dynatrace.com card number: 4012888888881881 dolor sit amet"



  }
  ```

  Результат после преобразования:

  ```
  {



  "content": "Lorem ipsum ip: 192.168.0.xxx email: abba0b6ff456806bab66baed93e6d9c4@dynatrace.com card number: 62163a017b168ad4a229c64ae1bed6ffd5e8fb2d dolor sit amet",



  "timestamp": "1656010291511"



  }
  ```
