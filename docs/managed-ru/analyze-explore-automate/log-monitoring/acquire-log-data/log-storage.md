# Правила приёма журналов (Logs Classic)

* Учебное руководство
* Чтение: 16 мин
* Обновлено 18 января 2023 г.

Log Monitoring Classic

Dynatrace версии 1.252+ OneAgent версии 1.243+

Если вы используете OneAgent версии ранее 1.243 и Dynatrace Cluster версии ранее 1.252, перейдите в раздел [Log Sources and Storage](/managed/upgrade/unavailable-in-managed "Выбранная функция недоступна в Dynatrace Managed.").

Dynatrace позволяет включать и исключать конкретные источники журналов для анализа. С помощью [фреймворка управления идентификацией и доступом (IAM) Dynatrace](/managed/manage/identity-access-management/permission-management/manage-user-permissions-policies "Работа с политиками") можно контролировать, кто из пользователей может изменять конфигурации в той или иной области.

Конфигурация основана на правилах, использующих матчеры по иерархии, пути журнала и группам процессов. Эти правила определяют, какие файлы журналов из числа обнаруженных OneAgent — автоматически или заданных как пользовательские источники — принимаются.

## Правило приёма журналов

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Нажмите **Add rule** и укажите имя конфигурации.
   По умолчанию кнопка **Include in storage** включена, что означает сохранение элементов, соответствующих этому правилу, в Dynatrace. Альтернативно можно выбрать тип правила **Exclude from storage**.
3. Разверните **Details** нового правила и нажмите **Add matcher** для создания конкретного совпадения.
   Одно правило может включать несколько матчеров.

   За исключением атрибута **Log source** в Windows (пути к файлам нечувствительны к регистру), матчеры чувствительны к регистру.
4. Выберите атрибут сопоставления:

| Атрибут | Описание | Логика выпадающего списка поиска |
| --- | --- | --- |
| **Process group** | Сопоставление по ID группы процессов. | Отображаются сущности, видимые за последние 3 дня. |
| **Log source** | Сопоставление по пути журнала или полному имени журнала событий Windows; поддерживаются подстановочные символы в виде звёздочки. | Можно вводить вручную. Без ограничения по времени. |
| **Log source origin** | Сопоставление по детектору, использованному агентом журналов для обнаружения файла журнала. | Можно вводить вручную. Без ограничения по времени. |
| **Log content** | Сопоставление по содержимому журнала; поддерживаются подстановочные символы в виде звёздочки. | Можно вводить вручную. Без ограничения по времени. |
| **Log record level** | Сопоставление по уровню записи журнала. Поддерживаются значения: `alert`, `critical`, `debug`, `emergency`, `error`, `info`, `none`, `notice`, `severe`, `warn`. | Можно вводить вручную. Без ограничения по времени. |
| **journald unit** | Сопоставление по любым из выбранных единиц journald. | Можно вводить вручную. Без ограничения по времени. |
| **Host tag** | Сопоставление по тегу хоста. | Можно вводить вручную. Без ограничения по времени. |
| **Kubernetes container name** | Сопоставление по имени контейнера Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **Kubernetes namespace name** | Сопоставление по имени пространства имён Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **Kubernetes deployment name** | Сопоставление по любому из выбранных деплойментов. | Можно вводить вручную. |
| **Kubernetes pod annotation** | Сопоставление по любым из выбранных аннотаций пода. Правильный формат: `key=value`. | Можно вводить вручную. |
| **Kubernetes pod label** | Сопоставление по любым из выбранных меток пода. Правильный формат: `key=value`. | Можно вводить вручную. |
| **Kubernetes workload name** | Сопоставление по любым из выбранных имён рабочих нагрузок. | Отображаются атрибуты, видимые за последние 90 дней. |
| **Kubernetes workload kind** | Сопоставление по любым из выбранных типов рабочих нагрузок. | Можно вводить вручную. |
| **Docker container name** | Сопоставление по имени контейнера. | Отображаются атрибуты, видимые за последние 90 дней. |
| **DT entity container group ID** | Сопоставление по любой из выбранных групп контейнеров. | Можно вводить вручную. Без ограничения по времени. |
| **Process technology** | Сопоставление по имени технологии. | Можно вводить вручную. Без ограничения по времени. |
| **Windows log record event ID** | Сопоставление по любым из выбранных ID событий. | Можно вводить вручную. Без ограничения по времени. |
| **Windows log record source** | Сопоставление по любым из выбранных атрибутов источника. | Можно вводить вручную. Без ограничения по времени. |
| **Windows log record task category** | Сопоставление по любым из выбранных атрибутов категории задачи. | Можно вводить вручную. Без ограничения по времени. |
| **Windows log record operational code** | Сопоставление по любым из выбранных атрибутов кода операции. | Можно вводить вручную. Без ограничения по времени. |
| **Windows log record user name** | Сопоставление по любым из выбранных атрибутов имени пользователя. | Можно вводить вручную. Без ограничения по времени. |
| **Windows log record keywords** | Сопоставление по любым из выбранных атрибутов ключевых слов. | Можно вводить вручную. Без ограничения по времени. |

Подстановочный символ поддерживается для любого значения атрибута и может использоваться несколько раз в одном значении. Однако некоторые атрибуты, например Process Group, имеют ограниченный предопределённый список возможных значений.

5. Нажмите **Add value** и из **Values** выберите обнаруженные элементы данных журналов.
6. Нажмите **Save changes**.

Определённые правила можно переупорядочивать; они выполняются в том порядке, в котором отображаются на странице **Log storage**.

7. Для активации правила включите переключатель **Active**.

## Сопоставление списка правил с данными журналов

Сопоставление происходит в предопределённой иерархии, правила выполняются сверху вниз. Иерархия сопоставления:

1. Правила конфигурации хоста
2. Правила конфигурации группы хостов
3. Правила конфигурации тенанта

## Области конфигурации

Поддерживаются три иерархических области: хост, группа хостов и тенант. Область с наименьшим возможным набором правил имеет приоритет над более широкими наборами.

![Приоритет правил приёма журналов](https://dt-cdn.net/images/log-storage-rule-priority-white-1491-d1d8126129.png)

Приоритет правил приёма журналов

1. Правила хранения журналов, настроенные для хоста, имеют приоритет над правилами для группы хостов.
2. Правила хранения журналов, настроенные для группы хостов, имеют приоритет над правилами для тенанта.

### Область хоста

Доступна через **Host settings** для конкретного хоста.

1. Перейдите в **Hosts**.
2. Найдите и выберите нужный хост.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**...**) > **Settings**.
4. В настройках хоста перейдите в **Log Monitoring** > **Log ingest rules**.
5. Настройте загрузку в хранилище, добавляя правила с атрибутами, соответствующими данным журналов.

### Область группы хостов

Доступна через страницу **Host**.

1. Перейдите в **Hosts** и выберите интересующий хост.
2. На странице обзора хоста выберите **Properties and tags**.
3. На панели **Properties and tags** найдите свойство **Host group**.
4. Выберите имя группы хостов для отображения всех хостов в ней.
5. Выберите имя группы хостов в любой строке.
6. В настройках группы хостов выберите **Log Monitoring** > **Log ingest rules**.
7. Настройте загрузку в хранилище, добавляя правила.

### Область тенанта

Доступна в меню настроек.

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Настройте загрузку в хранилище, добавляя правила.

## REST API

Для управления правилами приёма журналов через Settings API:

* Просмотр схемы
* Список сохранённых объектов конфигурации
* Просмотр отдельного объекта конфигурации
* Создание, редактирование или удаление объекта конфигурации

Идентификатор схемы (`schemaId`): `builtin:logmonitoring.log-storage-settings`.

## Часто задаваемые вопросы

Будут ли работать более старые версии OneAgent?

Версии OneAgent ранее `1.243` не будут отправлять данные; в ответ они получат пустой белый список.

Важен ли порядок элементов конфигурации?

Да, элементы конфигурации сопоставляются сверху вниз, то есть верхнее значение является наиболее приоритетным.

Как долго ждать применения конфигурации на хосте?

Применяется в течение 90 секунд.

Уменьшает ли добавление матчера по содержимому количество событий журналов, отправляемых в Dynatrace?

Да. Матчер по содержимому сужает область событий журналов согласно заданным критериям.

Снижает ли фильтрация содержимого стоимость DDU и/или использование сети?

Да. Фильтрация содержимого на стороне OneAgent снижает как затраты DDU, так и использование сети.
