# Добавление файлов логов вручную (Logs Classic)

* Чтение: 7 мин
* Обновлено 18 января 2023 г.

Log Monitoring Classic

Начиная с OneAgent версии 1.251 и Dynatrace Cluster версии 1.254, настоятельно рекомендуется переключиться на [Конфигурацию пользовательских источников логов](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/log-custom-source "Настройте пользовательские источники логов, не обнаруженные автоматически."), которая имеет множество [преимуществ](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/log-custom-source#clscadvantages "Настройте пользовательские источники логов, не обнаруженные автоматически.") перед ручным добавлением файлов логов.

Dynatrace позволяет добавлять отдельные файлы логов вручную в случаях, когда они не обнаруживаются автоматически. Достаточно добавить один путь к логу на уровне группы процессов, чтобы охватить все процессы в группе на всех отслеживаемых хостах.

### Добавление файла логов в группу процессов

Чтобы вручную добавить файл логов в группу процессов

1. В Dynatrace перейдите в **Technologies & Processes** и выберите группу процессов.
2. На странице группы процессов выберите экземпляр процесса, для которого нужно добавить новый файл логов.
3. В разделе **Log files** на странице выберите ссылку **Configure more log files**.  
   Если с процессом не связаны файлы логов, ищите раздел **No log sources detected**.
4. На странице **Process group settings** выберите вкладку **Log monitoring**.
5. Нажмите **Add new log for monitoring**.
6. Введите путь к файлу логов, который нужно связать с этой группой процессов. Допускаются только абсолютные пути.

   * Проверьте [требования к путям логов](#considerations-for-adding-text-log-files-manually), чтобы убедиться в их поддержке.
   * Убедитесь, что [включён мониторинг](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/add-log-files-sources-v2 "Узнайте, как включать и исключать источники логов для анализа.") добавленных файлов логов.
7. Нажмите **Save**.

### Журналы событий Windows

Стандартные журналы событий Windows обнаруживаются автоматически, поэтому следующие файлы добавлять вручную не нужно:

* `Windows Security Log`
* `Windows Application Log`
* `Windows System Log`
* `IIS Event Logs`

Чтобы вручную добавить пользовательский журнал событий Windows для процесса

1. В Dynatrace перейдите в **Hosts**.
2. Выберите хост, содержащий файл журнала событий Windows.
3. В разделе **Processes** на обзорной странице хоста выберите процесс, для которого нужно добавить новый файл логов.  
   Или перейдите в **Technologies & Processes**, выберите группу процессов, затем на странице группы выберите экземпляр процесса.
4. В разделе **Log files** нажмите **Add more**.
5. На странице **Manually added process logs** нажмите **Add new log for monitoring**.
6. Укажите имя лога.  
   Для поиска пользовательского имени лога можно использовать встроенный **Event Viewer** Windows. Убедитесь, что вы указываете полное имя, как показано в примере ниже.

   Получение полного имени журнала событий Windows.

   Как показано в следующем примере, можно отобразить имя лога, щёлкнув правой кнопкой мыши выбранный журнал событий и выбрав **Properties**:

   ![Экран свойств Windows Event Viewer.](https://dt-cdn.net/images/windows-event-viewer-properties1-547-41612bfc3c.png)

   Экран свойств Windows Event Viewer.

   Скопируйте **Full Name**:

   ![Свойства лога Windows Viewer.](https://dt-cdn.net/images/windows-event-viewer-properties2-1261-03ad4741e4.png)

   Свойства лога Windows Viewer.
7. Нажмите **Save**.

### Настройка разрешений в сетевой файловой системе (NFS)

При работе с логами в NFS разрешения доступа строго соблюдаются. Это справедливо несмотря на расширенные возможности OneAgent, позволяющие ему обращаться к локальным файлам без прав на чтение для пользователя `dtuser`.

Чтобы разрешить OneAgent обрабатывать и принимать ресурсы, смонтированные через NFS:

* Каждый нелокальный каталог по пути должен иметь как минимум разрешения `read` и `execute`.
* Каждый файл логов должен иметь разрешение `read`.

Пример добавляемого файла логов:

`/mnt/nfs/logs/app1/test.log`  
где `/mnt/nfs` смонтирован на внешний ресурс NFS.

В этом сценарии оба каталога `/mnt/nfs/logs` и `/mnt/nfs/logs/app1` должны иметь разрешения `r-x` для «остальных», а каждый файл `test.log.*` — разрешение `r--` для «остальных», как показано ниже:

```
$ ls -l /mnt/nfs



drwxr-xr-x 3 1001 1002 4096 Sep 8 17:11 logs
```

```
$ ls -l /mnt/nfs/log



drwxr-xr-x 3 1001 1002 4096 Sep 8 17:11 app1
```

```
$ ls -l /mnt/nfs/logs/app1



-rw-rw-r-- 3 1001 1002 100 Jul 19 14:22 test.log



-rw-rw-r-- 3 1001 1002 100 Jul 19 14:23 test.log.1



-rw-rw-r-- 3 1001 1002 100 Jul 19 14:24 test.log.2
```

### Требования к добавлению текстовых файлов логов вручную

Dynatrace поддерживает файлы логов, которые обновлялись за последние 7 дней **И** содержат полный путь **И** соответствуют **хотя бы одному из дополнительных критериев**:

* Файлы расположены в каталоге с именем `log` или `logs` (до 2 уровней) или в `/var/log/`.  
  **ОК (уровень 1):**  
  `c:\...\logs\custom.txt`  
  `c:\...\log\custom.txt`

  **ОК (уровень 2):**  
  `c:\...\logs\app1\custom.txt`  
  `c:\...\log\app2\custom.txt`

  **НЕ ОК (уровень 3 и выше):**  
  `c:\...\logs\app1\app11\custom.txt`  
  `c:\...\log\app2\app22\custom.txt`
* Пользовательский путь к логам должен указывать на файлы, а не на каталог:  
  **НЕ ОК:**  
  `/var/log/customdir/`
* Файлы имеют расширение `.log`, `-log`, `_log` (например, `apache.log`, `apache.myapp-log`, `apache.myapp_log`)
* Каждый файл с другим расширением, например `.txt`, поддерживается, если его имя содержит префикс `.log`, `-log` или `_log`, которому предшествует хотя бы один символ (например, `apache.log.txt`, `apache-log.txt` или `apache_log.txt`)

  Файл `log.txt` сам по себе не принимается (если только не находится в каталоге `log` или `logs`).
* Имя файла — `catalina.out`

Правильными считаются только имена файлов, содержащие полный путь. Поэтому `AlertServiceLog.log` отклоняется, несмотря на соответствие другим критериям, но `c:\customdir\AlertServiceLog.log` принимается.

Дополнительно:

* Пользовательские пути к логам могут начинаться или заканчиваться подстановочным символом (\*).  
  **ОК:**  
  `/var/log/customdir/*`  
  `/var/log/customdir/file*`  
  `/var/log/customdir/*.log`
* Можно добавить группу логов (набор файлов) с помощью символа звёздочки (**\***) в имени файла.  
  Например, `/var/log/error-*.log` будет соответствовать `error-1.log`, `error-some-ID.log` и т. д., и набор файлов будет отображаться как единый файл `error-*.log`.

Пользовательские пути к логам

Убедитесь, что файлы логов добавлены в хранилище логов после настройки пользовательских путей. См. [Источники логов и хранилище (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/add-log-files-sources-v2 "Узнайте, как включать и исключать источники логов для анализа.")

Пример:  
Если файл называется `AlertServiceLog.txt`, описанные выше сценарии применяются следующим образом:

| Файл | Принят? | Причина |
| --- | --- | --- |
| AlertServiceLog.txt | НЕ ОК | Не содержит полного пути. |
| c:\customdir\AlertServiceLog.txt | НЕ ОК | Содержит полный путь, но ни одно дополнительное требование не выполнено. |
| c:…\log\AlertServiceLog.txt | ОК | Требование 1-го уровня выполнено. |
| c:…\log\app2\AlertServiceLog.txt | ОК | Требование 2-го уровня выполнено. |
| c:…\log\app2\app22\AlertServiceLog.txt | НЕ ОК | Уровень > 2 |
| c:\customdir\file.log | ОК | Расширение `.log` |
| c:\customdir\file-log | ОК | Расширение `-log` |
| /var/log/customdir/AlertServiceLog\_log\* | ОК | Подстановочный символ принят (выполнен хотя бы один дополнительный критерий: расширение `_log`). |
| c:\customdir\AlertServiceLog.log.txt | ОК | Наличие `.log` в файле с расширением `.txt` |
| c:\customdir\AlertServiceLog\_log.txt | ОК | Наличие `_log` в файле с расширением `.txt` |
| c:\customdir\AlertServiceLog-log.txt | ОК | Наличие `-log` в файле с расширением `.txt` |

### Дополнительно:

* Поддерживаются только абсолютные пути к файлам:

  + **Unix:** путь должен начинаться с `/`.  
    Пути внутри следующих папок не поддерживаются:

  ```
  etc



  boot



  proc



  dev



  bin



  sbin



  lib



  usr



  .ssh
  ```

  + **Windows:** путь должен начинаться с буквы диска (например, `C:\`).  
    Пути внутри следующих папок не поддерживаются:

  ```
  windows



  winnt
  ```
* Файлы логов, добавленные вручную, затрагивают все процессы на всех хостах, принадлежащих настроенной группе процессов.
* Чтобы добавить файлы логов из контейнера:

1. Настройте контейнер так, чтобы файл логов записывался на диск.
2. Укажите OneAgent это расположение, задав абсолютный путь к месту записи файлов логов контейнером.

* Можно добавлять нумерованные наборы ротируемых файлов логов с помощью символа решётки (**#**).  
  Например, `/var/log/error-#.log` будет соответствовать `error-1.log`, `error-2.log` и т. д., и набор файлов будет отображаться как единый файл `error-#.log`.
* Можно глобально отключить возможность настройки пользовательских файлов логов, установив:  
  `AppLogRemoteConfiguration = false`

  **На Linux**:  
  `/var/lib/dynatrace/oneagent/agent/config/ruxitagentloganalytics.conf`

  **На Windows**:  
  `%PROGRAMDATA%\dynatrace\oneagent\agent\config\ruxitagentloganalytics.conf`

Использование шаблона

Если установка OneAgent (новая или обновлённая) не содержит файл `ruxitagentloganalytics.conf`, используйте `ruxitagentloganalytics.conf.template` как шаблон и создайте собственный файл `ruxitagentloganalytics.conf`. Затем скопируйте файл `ruxitagentloganalytics.conf` в следующий каталог:

* Linux: `/var/lib/dynatrace/oneagent/agent/config/`
* Windows: `%PROGRAMDATA%\dynatrace\oneagent\agent\config\`
