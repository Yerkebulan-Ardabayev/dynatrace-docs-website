# Пользовательский источник журналов (Logs Classic)

* Учебное руководство
* Чтение: 14 мин
* Обновлено 19 ноября 2024 г.

Log Monitoring Classic

OneAgent версии 1.251+ Dynatrace Cluster версии 1.254+

В OneAgent версии 1.249 и более ранних необходимо [добавлять файлы журналов вручную](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/add-log-files-manually-v2 "Узнайте, как вручную добавлять файлы журналов для анализа.").

Настройка пользовательских источников журналов позволяет вручную добавлять источники журналов, которые не были обнаружены автоматически.

* Автообнаружение может не идентифицировать источник журнала, если файл журнала не открыт для записи в процессе выполнения.
* Также может не найти источники журналов, не относящиеся ни к одному процессу или являющиеся частью кратковременных процессов.

В таких случаях можно определять, настраивать и кастомизировать источники журналов по своим нуждам.

* Начиная с OneAgent версии 1.251 можно переключиться на улучшенную версию добавления вручную — настройку пользовательских источников журналов. Подключите её, выбрав баннер на [странице настройки пользовательских источников журналов](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/log-custom-source#mainclscuipage "Настройте пользовательские источники журналов, не обнаруженные автоматически.").

Весь процесс состоит из двух частей:

1. Определение источника (настройка пользовательских источников журналов), описанное на этой странице.
2. Получение журналов (добавление журналов в хранилище), описанное в разделе [Правила приёма журналов](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/log-storage "Настройте хранение файлов журналов, уже известных OneAgent.").

Для хранения пользовательских журналов необходимо выполнить оба шага.

## Преимущества настройки пользовательских источников журналов

По сравнению с добавлением журналов в более ранних версиях OneAgent процесс теперь имеет несколько улучшений.

Настройка пользовательских источников журналов позволяет:

* Определять источники журналов без контекста группы процессов (добавление группы процессов опционально). К одному источнику журналов можно добавить до трёх групп процессов.
* Определять правила в трёх областях применения (хост, группа хостов, окружение).
* Изменять правила безопасности через файлы, размещённые на хосте.
* Добавлять правило с несколькими путями в рамках одной группы процессов.
* Использовать подстановочные символы в директориях.
* Использовать специализированный API.

Автоматическая миграция устаревшей конфигурации пользовательских источников журналов: каждое из существующих правил мигрирует в область окружения с соответствующим контекстом группы процессов. Имена мигрированных правил имеют префикс `auto-migrated`.

## Поддерживаемые области применения

Поддерживаются три иерархических области: хост, группа хостов и окружение. Чем уже область, тем выше её приоритет.

* Правила источников журналов, настроенные для хоста, имеют приоритет над правилами для группы хостов.
* Правила источников журналов, настроенные для группы хостов, имеют приоритет над правилами для окружения Dynatrace.

### Хосты

Настройка пользовательских источников журналов на уровне хоста

1. Перейдите в **Hosts**.
2. Найдите и выберите нужный хост для отображения страницы обзора хоста.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**...**) > **Settings**.

4. В настройках хоста перейдите в **Log Monitoring** > **Custom log sources**.

### Группы хостов

Настройка пользовательских источников журналов на уровне группы хостов

1. Перейдите в **Hosts** и выберите интересующий хост.
2. На странице обзора хоста выберите **Properties and tags**.
3. На панели **Properties and tags** найдите свойство **Host group**, чтобы увидеть имя группы хостов, к которой принадлежит выбранный хост.

   Свойство **Host group** не отображается, если выбранный хост не принадлежит ни одной группе хостов.
4. Выберите имя группы хостов для отображения всех хостов в этой группе. Откроется страница **OneAgent deployment**, отфильтрованная по выбранной группе хостов. Каждый перечисленный хост содержит ссылку **Host group:** `<имя группы>`, где `<имя группы>` — имя настраиваемой группы хостов.
5. Выберите имя группы хостов в любой строке.

6. В настройках группы хостов выберите **Log Monitoring** > **Custom log sources**

### Окружение

Настройка пользовательских источников журналов на уровне окружения

1. Перейдите в **Settings** > **Log Monitoring** > **Custom log sources**.

## Настройка источника журналов

1. Перейдите на страницу **Custom log sources** на уровне хоста, группы хостов или окружения, как описано выше.
2. Выберите **Add custom log source** и добавьте **Rule name**.
3. Опционально: привяжите правило к группе процессов, выбрав имя группы процессов из выпадающего меню.

Правила, назначенные группам процессов, отсутствующим на данном хосте, будут проигнорированы для этого хоста.

4. В разделе **Custom log source paths** выберите **Log source type**. Доступны два типа источников:

   * **Log path**
   * **Windows Event Log**
     Каждое правило применяется к одному типу источника журналов. Можно выбрать либо **Log path**, либо **Windows Event Log**.
5. Для определения полного пути к файлу журнала выберите **Add custom log source path**, введите полный путь (например, `/var/lib/*.log` или `/var/log/sys.bin`) и нажмите **Add Path**. На один источник журналов можно добавить до 100 значений. При выборе **Windows Event Log** обратитесь к [примеру Windows event](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/add-log-files-manually-v2#Windoweventlog "Узнайте, как вручную добавлять файлы журналов для анализа.") для указания правильного пути.

Иногда OneAgent не может определить метод кодирования отслеживаемого файла. Это может быть связано с отсутствием маркера BOM, определяющего метод кодирования, или использованием специфической кодировки для диакритических символов. Поддерживаемые кодировки: UTF-8, UTF-16 Little и Big Endian, Shift\_JIS, EUC\_JP (японский язык) и EUC\_KR (корейский язык).

6. Опционально: выберите **Show advanced** для раскрытия панели, позволяющей определить список атрибутов для обогащения каждой записи журнала из заданных источников. Введите ключ атрибута или выберите его из списка, затем введите значение атрибута. Доступно только при выборе **Log source type** — Log Path.

   Обогащение журналов пользовательскими атрибутами.

   При использовании подстановочных символов в пути журнала может понадобиться различать пути, совпавшие с подстановочными символами. В таких случаях можно определить атрибуты, использующие полный путь к файлу или часть пути, совпавшую с подстановочными символами.

   Для определения такого атрибута выполните следующие шаги:

   1. Введите ключ или выберите его из списка.
   2. В поле значения атрибута используйте токен `${N}`, где `N` — индекс подстановочного символа, начиная с 1. `${0}` имеет особое значение и раскрывается в полный путь к файлу журнала.

   В одном атрибуте можно использовать несколько токенов `${N}` и комбинировать их с другими символами. Например, `worker:${1}-${2}`.

   Если токен `${N}` ссылается на индекс подстановочного символа, превышающий их количество в пути журнала, он не будет заменён, и `${N}` останется в значении атрибута. Ключ атрибута должен содержать только латинские буквенно-цифровые символы (верхнего или нижнего регистра), точки (`.`), знаки подчёркивания (`_`), дефисы (`-`) или двоеточия (`:`). Он не должен начинаться с префикса `dt.` и не должен совпадать ни с одним из следующих:

   ```
   process.technology, log.source, log.content, timestamp, container.name, winlog.level, winlog.eventid, winlog.provider, winlog.opcode, winlog.task, winlog.keywords, winlog.username, k8s.namespace.name, k8s.container.name, k8s.deployment.name
   ```

   На один путь журнала можно определить до 100 атрибутов. Минимальная версия: OneAgent 1.285.
7. Нажмите **Save changes**.
8. Для активации правила включите переключатель **Active**.

## Требования к правам доступа к файлам журналов

Чтобы OneAgent мог принимать файл журнала, ему необходимы права на чтение этого файла.

Способ предоставления прав зависит от ОС, на которой развёрнут OneAgent.

OneAgent поддерживает три операционные системы:

* Linux
* Windows
* AIX

Когда требуется явное предоставление прав:

| Диски | Linux | Windows | AIX |
| --- | --- | --- | --- |
| Локальные диски | Действий не требуется | Действия могут потребоваться | Действий не требуется |
| Удалённые диски | Действия могут потребоваться | Действия могут потребоваться | Действий не требуется |

### Linux

* Файлы журналов на локальных дисках: действий не требуется.
  По умолчанию OneAgent имеет необходимые [Linux capabilities](https://man7.org/linux/man-pages/man7/capabilities.7.html) для чтения этих файлов.
* Файлы журналов на удалённом диске: необходимо предоставить следующие права для учётной записи, под которой работает OneAgent (по умолчанию — `dtuser`):

  + Права на чтение и выполнение `(r+x)` для каждой директории в пути к файлу журнала.
  + Права на чтение `(r)` самого файла журнала — для учётной записи, под которой работает OneAgent (по умолчанию `dtuser`).

    Файлы журналов на удалённом диске

    Отсутствие прав доступа является одной из наиболее распространённых причин отсутствия приёма журналов с удалённых дисков на Linux.

    При наличии доступа к отслеживаемому хосту можно быстро проверить, является ли отсутствие прав причиной проблемы, выполнив команду `sudo -u dtuser cat /path/to/file.log` с указанием правильного полного пути, при условии что `dtuser` — учётная запись, под которой работает OneAgent.

### Windows

OneAgent работает под учётной записью `LocalSystem`.

Следующие права обычно предоставляются автоматически для файлов журналов на локальных дисках:

* Право на чтение для каждого файла журнала.
* Права на чтение и выполнение для директорий в пути для этой учётной записи.

Для удалённых дисков они могут быть не установлены. Необходимо проверить их наличие и предоставить необходимые права.
Подробнее: [https://www.ntfs.com/ntfs-permissions-setting.htm](https://www.ntfs.com/ntfs-permissions-setting.htm).

OneAgent не может получить доступ к файлам, требующим авторизации через дополнительные диалоги UI, например всплывающее окно Windows Explorer.

### AIX

По умолчанию OneAgent имеет необходимые права root для чтения этих файлов.

## Сопоставление файлов журналов

При настройке пользовательского источника журналов соблюдайте следующие правила:

* Пользовательские пути журналов должны быть абсолютными; относительные пути отклоняются. Абсолютный путь имеет следующий вид:

  + Для Windows: `любая буква:\`
  + Для Linux: начинается с `/`
  + Для NFS: начинается с `//hostname/`
* Путь Windows Event Log в Windows Event System должен быть относительным.
* Пользовательские источники журналов могут содержать подстановочные символы:

  + `#` заменяет строку цифр или символ решётки `#`. Может использоваться только в именах файлов.
  + `*` заменяет строку любых символов, включая сам `*`. Не заменяет слэш (`/`) или обратный слэш (`\`). `*` может использоваться как в именах файлов, так и в директориях.

Кроме того, каждый добавляемый путь пользовательского источника журналов должен быть проверен OneAgent и соответствовать его правилам безопасности (правилам сопоставления файлов). На стороне OneAgent применяются следующие правила безопасности:

## Правила безопасности

Dynatrace определяет правила безопасности для пользовательских источников журналов. Эти правила управляют доступом OneAgent к источникам журналов, защищая от несанкционированного доступа и злоупотребления данными. Предопределённые правила безопасности можно добавлять или переопределять в конфигурационном файле на хосте, где установлен OneAgent, обеспечивая гибкость адаптации под конкретные требования безопасности.

Правила запрещают пути журналов в критических системных директориях (таких как `/etc`, `/boot`, `/proc` и других), пути, содержащие `.ssh`, пути с расширением `.pem` и пути в директориях, начинающихся с точки (скрытые директории). Допустимые пути журналов должны иметь расширение log с определёнными разделителями, находиться в первом или втором уровне директории `log` или `logs`, располагаться на любом уровне директории `/var/log`, либо иметь имя файла `catalina.out`.

При проверке безопасности правила учитывают разрешённые пути символических ссылок, подчёркивая важность фактического расположения файла, а не пути символической ссылки.

* Путь журнала не находится ни в одной из следующих директорий: `/etc`, `/boot`, `/proc`, `/dev`, `/bin`, `/sbin`, `/usr`, `WindowsRoot:\windows`, `WindowsRoot:\winnt`. Однако `Windows|winnt\system32\winevt\Logs` принимается, И
* Путь журнала не содержит `.ssh`, И
* Путь журнала не имеет расширения `.pem`, И
* Путь журнала не находится в директории, имя которой начинается с `.` (например, `/.hidden`), И
* Путь журнала должен иметь расширение `log`, разделённое `.`, `-` или `_` (может следовать ещё одно расширение с тем же набором разделителей), ИЛИ

  + Путь журнала должен находиться на первом или втором уровне директории `log` или `logs`, ИЛИ
  + Путь журнала должен находиться на любом уровне директории `/var/log`, ИЛИ
  + Путь журнала должен иметь имя файла `catalina.out`.

Файлы с путями, не соответствующими одному или нескольким критериям, не принимаются.
После выполнения указанных условий происходит сопоставление файлов журналов. Ознакомьтесь с [правилами сопоставления файлов журналов](#log-file-matching).

Если настроенный для приёма файл журнала является символической ссылкой, Dynatrace проверит его путь (путь, полученный после перехода по символической ссылке) по правилам безопасности.

### Переопределение правил безопасности

Предопределённые правила безопасности можно добавлять или переопределять только в конфигурационном файле на хосте, где установлен OneAgent.

* Сохраняйте изменения в отдельном файле в директории постоянной конфигурации OneAgent.

  + `/var/lib/dynatrace/oneagent/agent/config/logmodule` на Linux и UNIX
  + `%PROGRAMDATA%\dynatrace\oneagent\agent\config\logmodule` на Windows

    В указанных директориях допускается любой файл журнала с суффиксом `.json`.
* Не редактируйте файл, содержащий предопределённые правила:

  + `/opt/dynatrace/oneagent/agent/conf` на Linux и UNIX
  + `%PROGRAMFILES%\dynatrace\oneagent\agent\conf` на Windows
* Правила, определённые пользователем в пользовательской конфигурации, имеют приоритет над правилами по умолчанию. При этом первое совпадающее правило определяет, проходит ли путь проверку безопасности. Формат файла переопределения конфигурации (файла, сохранённого в директории постоянной конфигурации) должен соответствовать формату файла с предопределёнными правилами.

### Файл конфигурации переопределения

* Существует предопределённый шаблон директорий, выполняемый справа налево. Например, `/log/` совпадёт с `/log/file` и `/var/log/file`, но не с `/log/dir/file`
* Совпадает только одна директория. Например, `/log/*/` совпадёт с `/log/dir/file`, но не с `/log/dir/dir2/file`
* Выражение `[-.\\_]` означает, что для совпадения должен присутствовать один из символов в квадратных скобках

### Пример файла конфигурации переопределения

Файл имеет следующую структуру:

* `allowed-log-paths-configuration`: Отмечает массив правил.

Каждое правило состоит из трёх пар ключ-значение со следующими обязательными ключами:

* `directory-pattern`
* `file-pattern`
* `action`

Описание ключей:

* `directory-pattern`: Объект задаёт шаблон для сопоставления директорий. Шаблон директорий выполняется справа налево: например, `/log/` совпадёт с `/log/file.txt` и `/var/log/file.txt`, но не с `/log/dir/file.txt`. Применяются следующие правила:

  + Директория сопоставляется подстановочным символом `*`. Например, `/log/*/` совпадёт с `/log/dir/file.txt`, но не с `/log/dir/dir2/file.txt`.
  + `**` совпадает с любым количеством поддиректорий. Например, шаблон `/log/dir/**/file.txt` совпадёт с `/log/dir/dir1/dir2/dir3/file.txt`.
  + `^` совпадает с началом пути. Он привязывает шаблон к началу проверяемого пути. Например, `^/usr/*/` совпадёт с путями, начинающимися с `/usr/`, такими как `/usr/log/file.txt` и `/usr/local/file.txt`, но не совпадёт с `/some/usr/log/file.txt`.
    Для Windows-путей якорь может также пропускать букву диска. Например, шаблон `^/Users/Public/` совпадёт с фактическим путём `C:\Users\Public\file.txt`. JSON рассматривает `\` как escape-символ, поэтому при указании Windows-путей можно использовать `C:\\Users\\Public` или `C:/Users/Public`, но не `C:\Users\Public`.

  Специальные символы `*`, `**` и `^` можно комбинировать в одном шаблоне директорий для создания более сложных правил сопоставления. Например, шаблон `^/log/**/dir/*/*/` совпадёт с путём `/log/some/deep/dir/and/deeper/file.txt`.
* `file-pattern`: Объект задаёт шаблон для сопоставления файлов в директориях, совпавших с `directory-pattern`. Шаблон применяется с полным совпадением. Это означает, что шаблон `*.txt` совпадёт с `error.txt`, но не с `error.txt.1`.
  Для корректного обнаружения файлов с ротацией шаблон файла должен включать подстановочный символ в конце. Например, для совпадения с файлами, ротирующимися от `error.txt` до `error.txt.1`, шаблон файла должен быть `*.txt*`.
* `action`: Объект задаёт действие для совпавшего файла: `EXCLUDE` или `INCLUDE`.

Выражение `[-.\\_]` означает, что для совпадения должен присутствовать один из символов в квадратных скобках.

Пример файла конфигурации переопределения:

```
{



"allowed-log-paths-configuration":[



{



"directory-pattern":"/",



"file-pattern":"*.pem",



"action":"EXCLUDE" // или INCLUDE



},



... ваши правила ...



]



}
```

## Примеры правил безопасности OneAgent

Каждый добавляемый путь пользовательского источника журналов должен быть проверен OneAgent и соответствовать его правилам безопасности (правилам сопоставления файлов). На стороне OneAgent применяются некоторые предопределённые правила безопасности. Примеры правил исключения и включения для UNIX, Linux и Windows приведены в таблице ниже.

| Операционная система | Шаблон директорий | Шаблон файлов | Действие |
| --- | --- | --- | --- |
| UNIX | `/` | `*.pem` | EXCLUDE |
| UNIX | `/` | `*[-.\\_]log[-.\\_]*` | INCLUDE |
| Linux | `/.ssh/` | \* | EXCLUDE |
| Linux | `/` | `*[-.\\_]log` | INCLUDE |
| Windows | `/.*/` | \* | EXCLUDE |
| Windows | `/windows/system32/winevt/Logs/` | \* | INCLUDE |
