# Приём JSON и TXT журналов (Logs Classic)

* Пояснение
* Чтение: 5 мин
* Обновлено 04 мая 2026 г.

Log Monitoring Classic

API приёма журналов принимает журналы в форматах JSON, TXT и OTLP. На этой странице описаны форматы JSON и text. Документацию по OTLP см. в разделе [OTLP](/managed/ingest-from/opentelemetry/otlp-api/ingest-logs#otlp-structured-logs "Узнайте, как Dynatrace принимает записи журналов OpenTelemetry и какие ограничения применяются.").

Эндпоинт приёма собирает и пытается автоматически преобразовать данные журналов, содержащие следующие JSON-элементы:

* Содержимое журнала
* Временная метка
* Атрибуты в формате ключ-значение

Подробнее об ограничениях: [Ограничения по умолчанию Log Monitoring (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/log-monitoring-limits "Ограничения по умолчанию для последней версии Dynatrace Log Monitoring.").

## Преобразование данных и автоматический парсинг JSON

API приёма журналов собирает и пытается автоматически преобразовывать данные журналов. Каждая запись журнала из принятого пакета сопоставляется с одной записью журнала Dynatrace, содержащей три специальных атрибута: `timestamp`, `loglevel`, `content` и атрибуты ключ-значение. Эти четыре свойства устанавливаются на основе ключей, присутствующих во входном JSON-объекте.

### Временная метка

* `timestamp` устанавливается на основе значения первого найденного ключа из следующего списка (в порядке приоритета): `timestamp`, `@timestamp`, `_timestamp`, `eventtime`, `date`, `published_date`, `syslog.timestamp`, `time`, `epochSecond`, `startTime`, `datetime`, `ts`, `timeMillis`, `@t`.
* Поддерживаемые форматы: Unix epoch time в UTC (секунды, миллисекунды или секунды с дробной частью — доступно с Dynatrace версии 1.339), `RFC3339` и `RFC3164`.

  Для неподдерживаемых форматов временных меток используется текущая временная метка, а значение неподдерживаемого формата сохраняется в атрибуте `unparsed_timestamp`.
* Записи журналов старше [лимита возраста журналов](/managed/analyze-explore-automate/log-monitoring/log-monitoring-limits "Ограничения по умолчанию для последней версии Dynatrace Log Monitoring.") отбрасываются. Временные метки более чем на 10 минут опережающие текущее время заменяются текущим временем.
* Если в записи журнала нет поддерживаемого ключа временной метки, по умолчанию используется текущая временная метка.
* Если в временной метке отсутствует часовой пояс, по умолчанию используется UTC.

### Уровень журнала

* `loglevel` устанавливается на основе значения первого найденного ключа из следующего списка (в порядке приоритета): `loglevel`, `status`, `severity`, `level`, `syslog.severity`.
* Значение по умолчанию: `NONE`.

### Содержимое

* `content` устанавливается на основе значения первого найденного ключа из следующего списка (в порядке приоритета): `content`, `message`, `payload`, `body`, `log`.
* Значение по умолчанию: пустая строка.

### Атрибуты

* Атрибуты журнала содержат все остальные ключи входного JSON-объекта, кроме использованных для `timestamp`, `loglevel` и `content`.
* Атрибуты первого уровня желательно сопоставлять с семантическими атрибутами, чтобы Dynatrace мог связать их с контекстом. Подробнее: [Семантические атрибуты (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/logs-classic-ingestion-api/log-classic-semantic-attributes "Поддерживаемые семантические атрибуты, индексируемые в Log Monitoring Classic.").
* Все ключи атрибутов преобразуются в нижний регистр, значения — в строки. Пользовательские и семантические атрибуты в общем случае можно использовать в запросах.
* Автоматический атрибут: к каждой записи журнала, принятой через API, автоматически добавляется атрибут `dt.auth.origin`. Этот атрибут содержит публичную часть ключа API, с которым источник журнала авторизуется для подключения к универсальному API приёма журналов.

#### Атрибуты нескалярных типов

Вложенные объекты в атрибутах журнала преобразуются в плоские пары ключ-значение.

Когда атрибут журнала содержит объект, каждое вложенное свойство становится отдельным атрибутом. Этот процесс работает для атрибутов до пятого уровня; атрибуты глубже игнорируются.

Типы массивов сохраняются как массивы, но содержащиеся типы унифицируются до единого типа.

* Сложные значения (такие как массивы или объекты) сопоставляются со строковыми значениями JSON.
* Если хотя бы одно значение в массиве является строкой или должно быть преобразовано в строку (например, объект или массив), целевой тип всего массива — строка.
* Если все значения в исходном массиве числовые, целевой тип массива — числовой.
* Null-значения совместимы с любым типом.

| Входные данные | Выходные данные эндпоинта API приёма журналов |
| --- | --- |
| `{"content": "Transaction successfully processed.", "transaction": {"id": "TXN12345", "amount": 250.75}, "auditTrail": ["Created", "Approved", 3]}` | `{"content": "Transaction successfully processed.", "transaction.id": "TXN12345", "transaction.amount": 250.75, "auditTrail": ["Created", "Approved", "3"]}` |

##### Конфликты имён

При сохранении атрибутов в плоском виде на стороне Dynatrace могут возникать конфликты имён, если атрибуты на разных уровнях имеют одинаковое имя. Dynatrace разрешает это, добавляя к дублирующимся атрибутам префикс `overwritten[COUNTER].`. Значение счётчика указывает, сколько раз имя атрибута уже встречалось как дубликат. Например:

| Входные данные | Выходные данные |
| --- | --- |
| `{"host.name": "abc", "host": {"name": "xyz"}}` | `{"host.name": "abc", "overwritten1.host.name": "xyz"}` |

## Обработка атрибутов API приёма журналов

API приёма журналов дополнительно принимает атрибуты через:

* Параметры запроса
* Специальный заголовок: `X-Dynatrace-Attr`

Эти атрибуты объединяются с атрибутами из тела записи журнала.

### Атрибуты из параметров запроса

* Все параметры запроса, переданные на эндпоинт API приёма журналов, добавляются к атрибутам тела записи журнала.
* Если ключ параметра встречается несколько раз, все значения сохраняются как многозначный атрибут.
* Ключи и значения следуют тем же правилам парсинга атрибутов, что и атрибуты тела.

### Атрибуты через заголовок (X-Dynatrace-Attr)

API поддерживает специальный заголовок для передачи дополнительных атрибутов:

`X-Dynatrace-Attr: region=eu-central-1&team=core`

### Правила приоритета атрибутов

При наличии атрибутов в нескольких местах API приёма журналов применяет следующий порядок приоритета:

* Параметры запроса (наивысший приоритет)
* Заголовок X-Dynatrace-Attr
* Тело записи журнала (наименьший приоритет)

При переопределении атрибутов тела параметрами запроса или заголовком:

* Итоговое значение атрибута устанавливается согласно правилам приоритета.
* Значения из тела журнала сохраняются под ключами `overwrittenN.<attribute_key>`, где N — порядковый номер.
