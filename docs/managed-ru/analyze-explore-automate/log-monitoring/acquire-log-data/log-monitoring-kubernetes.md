# Мониторинг журналов в Kubernetes (Logs Classic)

* Учебное руководство
* Чтение: 10 мин
* Обновлено 08 октября 2025 г.

Log Monitoring Classic

Dynatrace Log Monitoring поддерживает сбор журналов из систем оркестрации контейнеров Kubernetes через OneAgent.

В качестве альтернативы сбору журналов на основе OneAgent можно передавать журналы в Dynatrace через [API приёма журналов](/managed/upgrade/unavailable-in-managed "Выбранная функция недоступна в Dynatrace Managed.") с помощью таких интеграций, как [Fluent Bit](/managed/upgrade/unavailable-in-managed "Выбранная функция недоступна в Dynatrace Managed."), [Fluentd](/managed/upgrade/unavailable-in-managed "Выбранная функция недоступна в Dynatrace Managed."), [Logstash](/managed/upgrade/unavailable-in-managed "Выбранная функция недоступна в Dynatrace Managed.") или [Dynatrace Collector](/managed/ingest-from/opentelemetry/collector#dt-collector-dist "Узнайте, как использовать OpenTelemetry Collector, включая Dynatrace OTel Collector, для приёма телеметрии из OpenTelemetry.").

## Поддерживаемые функции

Dynatrace Log Monitoring поддерживает различные платформы на базе Kubernetes, такие как Upstream Kubernetes или Red Hat OpenShift, использующие **containerd** или **CRI-O** в качестве среды выполнения контейнеров.

**Docker** не соответствует CRI (Container Runtime Interface). По этой причине настройки Kubernetes с использованием **Docker** поддерживаются лишь частично. Kubernetes объявил **Docker** устаревшим в качестве среды выполнения контейнеров после версии v1.20.

Подробнее о поддерживаемых версиях Kubernetes: [Поддержка Dynatrace Operator и известные проблемы](/managed/ingest-from/technology-support/support-model-and-issues "Как Dynatrace поддерживает версии Kubernetes и Red Hat OpenShift и известные проблемы").

## Автообнаружение журналов Kubernetes

OneAgent автоматически обнаруживает журналы, записываемые контейнеризованным приложением в потоки **stdout**/**stderr**. Kubernetes Engine сохраняет эти потоки в файл на узле Kubernetes. OneAgent обнаруживает эти файлы журналов и представляет журналы контейнеров в источнике журналов `Container Output`.

Журналы, записываемые напрямую в файловую систему пода, OneAgent не обнаруживает. В таком случае используйте интеграцию с отправщиком журналов, например Fluent Bit.

## Обогащение журналов метаданными Kubernetes

Модуль журналов OneAgent дополняет принятые журналы следующими метаданными Kubernetes: `k8s.cluster.name`, `k8s.cluster.uid`, `k8s.namespace.name`, `k8s.workload.name`, `k8s.workload.kind`, `dt.entity.kubernetes_cluster`, `k8s.pod.name`, `k8s.pod.uid`, `k8s.container.name`, `dt.entity.kubernetes_node`. Эти метаданные используются для сопоставления журналов с моделью сущностей кластеров Kubernetes, пространств имён, рабочих нагрузок и подов.

Кроме того, в записи журналов добавляются аннотации пода, начинающиеся с префикса `metadata.dynatrace.com/`.

### Управление приёмом журналов с помощью метаданных Kubernetes

Приём журналов из Kubernetes можно контролировать с помощью правил приёма журналов в Dynatrace. Эти правила можно настраивать на уровне кластера Kubernetes для специфичного для кластера приёма журналов. Правила используют матчеры для метаданных Kubernetes и других общих атрибутов записей журналов для определения, какие журналы следует принимать.
Стандартные функции обработки журналов OneAgent, включая [маскирование чувствительных данных](/managed/analyze-explore-automate/log-monitoring/methods-of-masking-sensitive-data "Выберите оптимальный метод маскирования чувствительных данных."), [настройку временных меток](/managed/analyze-explore-automate/log-monitoring/log-monitoring-configuration/timestamp-configuration "Определяйте правила мониторинга журналов для управления временными метками данных журналов.") и [автоматическое обогащение](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/lm-log-data-transformation "Универсальный приём журналов автоматически преобразует данные журналов в выходные значения атрибута loglevel.") записей журналов, также доступны и включены.

Используйте следующие рекомендуемые атрибуты сопоставления при настройке приёма журналов из Kubernetes.

| Атрибут | Описание | Логика выпадающего списка поиска |
| --- | --- | --- |
| **Kubernetes namespace name** | Сопоставление выполняется по имени пространства имён Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **Kubernetes container name** | Сопоставление выполняется по имени контейнера Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **Kubernetes deployment name** | Сопоставление выполняется по имени рабочей нагрузки Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **Kubernetes pod annotation** | Сопоставление выполняется по любой из выбранных аннотаций пода. Правильный формат: `key=value`. | Можно вводить вручную. |
| **Kubernetes pod label** | Сопоставление выполняется по любой из выбранных меток пода. Правильный формат: `key=value`. | Можно вводить вручную. |
| **Kubernetes workload name** | Сопоставление выполняется по любому из выбранных имён рабочих нагрузок. | Можно вводить вручную. |
| **Kubernetes workload kind** | Сопоставление выполняется по любому из выбранных типов рабочих нагрузок. | Можно вводить вручную. |
| **Log content** | Сопоставление выполняется по содержимому журнала; подстановочные символы поддерживаются в виде звёздочки. | Можно вводить вручную. Без ограничения по времени. |
| **Log record level** | Сопоставление выполняется по уровню записи журнала. Поддерживаются следующие значения: `alert`, `critical`, `debug`, `emergency`, `error`, `info`, `none`, `notice`, `severe`, `warn`. | Можно вводить вручную. Без ограничения по времени. |
| **Log source origin** | Сопоставление выполняется по детектору, использованному OneAgent для обнаружения файла журнала. | Можно вводить вручную. Без ограничения по времени. |
| **Process group** | Сопоставление выполняется по ID группы процессов. Также требуется запуск OneAgent на узле. | Отображаются сущности, видимые за последние 3 дня. |
| **Process technology** | Сопоставление выполняется по имени технологии. Также требуется запуск OneAgent на узле. | Можно вводить вручную. Без ограничения по времени. |
| **DT entity container group ID** | Сопоставление выполняется по любой из выбранных групп контейнеров. Также требуется запуск OneAgent на узле. | Можно вводить вручную. Без ограничения по времени. |

## Настройка приёма журналов из Kubernetes

Для приёма журналов из Kubernetes необходимо определить правила приёма журналов. Конфигурация основана на иерархии правил, использующих матчеры для метаданных Kubernetes и других общих атрибутов записей журналов. Эти правила определяют, какие файлы журналов из числа обнаруженных OneAgent принимаются.

| Атрибут | Описание | Логика выпадающего списка поиска |
| --- | --- | --- |
| **K8s namespace name** | Сопоставление выполняется по имени пространства имён Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **K8s container name** | Сопоставление выполняется по имени контейнера Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **K8s deployment name** | Сопоставление выполняется по имени пода Kubernetes. | Отображаются атрибуты, видимые за последние 90 дней. |
| **Log content** | Сопоставление выполняется по содержимому журнала; подстановочные символы поддерживаются в виде звёздочки. | Можно вводить вручную. Без ограничения по времени. |
| **Log record level** | Сопоставление выполняется по уровню записи журнала. Поддерживаются значения: `alert`, `critical`, `debug`, `emergency`, `error`, `info`, `none`, `notice`, `severe`, `warn`. | Можно вводить вручную. Без ограничения по времени. |

### Иерархия правил приёма журналов

Правила приёма журналов могут быть определены в области окружения, а также на уровне хоста или группы хостов. Иерархия сопоставления:

1. Правила конфигурации хоста;
2. Правила конфигурации группы хостов;
3. Правила конфигурации тенанта.

Сопоставление происходит в предопределённой иерархии, правила выполняются сверху вниз. Это означает, что если правило выше в списке совпадает с определёнными данными журнала, нижние правила игнорируются. Элементы, совпавшие в конфигурациях более высокого уровня, перезаписываются конфигурациями более низкого уровня при совпадении с теми же данными журнала. Если ни одно правило не совпало, файл не отправляется.

## Варианты использования

Ознакомьтесь с вариантами использования приёма журналов из окружений Kubernetes с помощью Dynatrace.

### Приём всех журналов из определённого пространства имён

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Нажмите **Add rule** и укажите имя конфигурации в поле **Rule name**.
   Убедитесь, что кнопка **Include in storage** включена, чтобы журналы, соответствующие этой конфигурации, сохранялись в Dynatrace.
3. Нажмите **Add condition**.
4. В выпадающем списке **Matcher attribute** выберите **K8s namespace name**.
5. Выберите пространство имён из выпадающего списка в поле **Value** и нажмите **Add matcher**.
6. Нажмите **Save changes**.

### Приём журналов из определённого пространства имён и контейнера

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Нажмите **Add rule** и укажите имя конфигурации в поле **Rule name**.
   Убедитесь, что кнопка **Include in storage** включена.
3. Нажмите **Add condition**.
4. В выпадающем списке **Matcher attribute** выберите **K8s namespace name**.
5. Выберите пространство имён из выпадающего списка в поле **Value** и нажмите **Add matcher**.
6. Добавьте новый матчер, на этот раз выберите **K8s container name** и введите имя контейнера в поле **Value**. В этом шаге можно добавить несколько имён контейнеров.
7. Нажмите **Save changes**.

### Приём всех журналов Kubernetes, исключая определённые пространства имён

1. Перейдите в **Settings** и выберите **Log Monitoring** > **Log ingest rules**.
2. Нажмите **Add rule** и укажите имя конфигурации в поле **Rule name**.
   Убедитесь, что кнопка **Include in storage** включена.
3. Нажмите **Add condition**.
4. В выпадающем списке **Matcher attribute** выберите **K8s namespace name**.
5. Вставьте звёздочку (\*) в поле **Value** как заместитель для всех доступных пространств имён кластера.
6. Нажмите **Add matcher**.
7. Нажмите **Save changes**.
8. На экране правил **Log ingest** добавьте ещё одно правило и выберите параметр **Exclude from storage**.
9. В поле **Value** добавьте пространства имён, которые нужно исключить при приёме журналов Kubernetes.
10. Нажмите **Add matcher**.
11. Нажмите **Save changes**.

На экране **Log ingest rules** расставьте настроенные правила так, чтобы правило исключения пространств имён находилось сверху, а правило включения всех пространств имён — снизу.

## REST API

Для управления правилами приёма журналов можно использовать Settings API:

* Просмотр схемы;
* Список сохранённых объектов конфигурации;
* Просмотр отдельного объекта конфигурации;
* Создание, редактирование или удаление объекта конфигурации.

Идентификатор схемы: `builtin:logmonitoring.log-storage-settings`.
