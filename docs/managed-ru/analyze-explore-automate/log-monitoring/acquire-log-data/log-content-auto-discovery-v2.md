# Автоматическое обнаружение содержимого логов (Logs Classic)

* Пояснение
* Чтение: 3 мин
* Обновлено 25 марта 2026 г.

Log Monitoring Classic

По умолчанию Dynatrace автоматически обнаруживает все новые файлы логов, соответствующие описанным ниже требованиям.

## Автоматическое обнаружение по умолчанию

Dynatrace автоматически обнаруживает, анализирует и сохраняет (если [выбрано для хранения](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/log-storage "Настройте хранение файлов логов, уже известных OneAgent.")) логи каждые 60 секунд.

Хранение автоматически обнаруженных файлов в Dynatrace зависит от правил приёма логов.

По умолчанию модуль логов OneAgent автоматически обнаруживает следующие категории файлов логов:

* **Системные логи**  
  На Windows:

  + `Windows Security Log`
  + `Windows Application Log`
  + `Windows System Log`

  На Linux:

  + `/var/log/messages`
  + `/var/log/syslog`

  На AIX:

  + `/var/log/messages` (поддерживается OneAgent версии 1.335+)
  + `/var/log/syslog` (поддерживается всеми версиями OneAgent)
* **Файлы логов**, открытые работающими процессами. Подробности см. в разделе [Автоматическое обнаружение содержимого логов (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/log-content-auto-discovery-v2#autodiscoveryrequirements "Узнайте об автоматическом обнаружении содержимого логов и требованиях для его работы.")
* **IIS Logs** (только Windows) — как журналы событий, так и обычные файлы логов
* **Логи контейнеров** (только Linux) в Kubernetes, OpenShift и неинструментированном Docker. Подробности см. в разделе [Мониторинг логов в Kubernetes (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/acquire-log-data/log-monitoring-kubernetes#autodiscovery_requirements "Узнайте, как осуществлять мониторинг логов в Kubernetes.")

### Атрибуты, выбираемые в журналах событий Windows

Для журналов событий Windows Log Monitoring обнаруживает следующие поля и отправляет их как пользовательские атрибуты:

| Имя семантического атрибута | Свойство события |
| --- | --- |
| `winlog.keywords` | `Event.RenderingInfo.Keywords` |
| `winlog.username` | `Event.System.Security.<xmlattr>.UserID` |
| `winlog.level` | `Event.RenderingInfo.Level` |
| `winlog.eventid` | `Event.System.EventID` |
| `winlog.provider` | `Event.System.Provider.<xmlattr>.Name` |
| `winlog.task` | `Event.System.Task` |
| `winlog.opcode` | `Event.RenderingInfo.Opcode` |

## Требования к автоматическому обнаружению

Файл логов должен соответствовать всем следующим требованиям для автоматического обнаружения:

* Файл логов должен быть открыт [важным процессом](/managed/observe/infrastructure-observability/process-groups/basic-concepts/which-are-the-most-important-processes "Отображение наиболее важных процессов для мониторинга и группировки.").
* Файл логов должен существовать не менее одной минуты.

  Неподдерживаемые метки времени

  Файлам с неподдерживаемой меткой времени автоматически присваивается время чтения файла.
* Логи должны иметь поддерживаемую кодировку символов.
  По умолчанию поддерживается UTF-8. Другие поддерживаемые типы включают UTF-8 BOM, а также, если файлы содержат маркер порядка байтов (BOM), UTF-16LE и UTF-16BE.

  Двоичные логи

  Двоичные файлы логов не обнаруживаются автоматически. Можно использовать пользовательские источники логов с параметром **Allow binary format** для приёма двоичных файлов.
* Размер файла логов должен быть не менее 0,5 КБ.
* Файл логов должен был обновляться (записываться) за последние 7 дней.  
  Файлы логов, не обновлявшиеся в течение последних 7 дней при активном Log Monitoring, не будут отображаться на дашбордах.
* Файл логов должен находиться в папке `log` или `logs` (или в её подпапках):

  + Примеры допустимых путей:  
    `c:\log\log_file.txt`  
    `c:\logs\NewFolder\log_file.txt`
  + Пример недопустимого пути:  
    `c:\log\NewFolder\NewFolder\log_file.txt`

  или имя файла логов должно содержать строку `log`, которой предшествует или за которой следует точка (`.`) или символ подчёркивания (`_`):

  + Примеры допустимых имён файлов:  
    `c:\NewFolder\abc.log`  
    `c:\NewFolder\0865842.log.txt`
  + Пример недопустимого имени файла:  
    `c:\NewFolder\logfile.txt`

## Отключение автоматического обнаружения логов

Если вы не хотите, чтобы Dynatrace автоматически обнаруживал новые файлы логов на конкретном отслеживаемом хосте, можно отключить автоматическое обнаружение.

1. На хосте откройте файл конфигурации аналитики логов для редактирования.

   * **На Linux**:  
     `/var/lib/dynatrace/oneagent/agent/config/ruxitagentloganalytics.conf`
   * **На Windows**:  
     `%PROGRAMDATA%\dynatrace\oneagent\agent\config\ruxitagentloganalytics.conf`
2. Установите следующее:  
   `AppLogAutoDetection = false`

Перезапуск OneAgent не требуется.

## Ограничения при автоматическом обнаружении логов с использованием OneAgent

Файлы логов в OneAgent:

* не могут быть удалены ранее чем через минуту после создания;
* должны дополняться (старое содержимое не обновляется);
* должны иметь текстовое содержимое;
* должны быть открыты постоянно (не только на короткие периоды добавления записей);
* должны быть открыты в режиме записи;
* не должны быть меньше настроенного порога размера (по умолчанию: 500 байт) для проверки на двоичное содержимое.

Максимальное количество источников логов на экземпляр группы процессов по умолчанию составляет 200. Это значение настраивается через параметр **Maximum number of log sources per process group instance** в ![Settings](https://dt-cdn.net/images/settings-icon-256-38e1321b51.webp "Settings") **Settings** > **Log Monitoring** > **Advanced log settings**.

В стандартных средах модуль логов OneAgent поддерживает до 10 000 файлов в одном каталоге с логами и 200 МБ нового содержимого логов в минуту. При больших объёмах данных — особенно на порядок выше — модуль логов OneAgent, скорее всего, справится. Для предварительной проверки вашей конфигурации обратитесь в службу поддержки Dynatrace.

В особых случаях, например при очень низкой производительности оборудования, ограничения модуля логов OneAgent могут быть более строгими.

## Неподдерживаемые сценарии автоматического обнаружения

Сценарии, не поддерживаемые в процессе автоматического обнаружения ротируемых логов:

* Генерация ротируемых логов со сменой каталога. Этот процесс может привести к созданию большого количества неагрегированных и/или неполных логов, а также к перегрузке ресурсов.
* Генерация ротируемых логов с немедленным сжатием, когда приложение обращается к файлу с тем же именем. При достижении критерия ротации (например, требуемого размера файла) файл перемещается в другое место и немедленно сжимается.  
  Пример: `/var/log/application.log -> /var/log/application.log.1.gz -> /var/log/application.log.2.gz -> /var/log/application.log.3.gz`. Этот процесс также может привести к неполному созданию логов.
