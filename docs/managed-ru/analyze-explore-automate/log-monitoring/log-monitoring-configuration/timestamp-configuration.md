# Конфигурация временных меток/разбиения (Logs Classic)

# Конфигурация временных меток/разбиения (Logs Classic)

* Чтение: 7 мин
* Обновлено 10 апреля 2026 г.

Log Monitoring Classic

Dynatrace позволяет определять правила, управляющие временными метками данных журналов.

## Обнаружение временных меток

По умолчанию Log Monitoring автоматически обнаруживает только наиболее распространённое и однозначное подмножество поддерживаемых форматов дат. Подробнее см. в разделе [Поддерживаемые форматы временных меток (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/log-monitoring-configuration/timestamp-data-format "Поддерживаемые временные метки для последней версии Dynatrace Log Monitoring.").

### Временная метка не обнаружена

Если Log Monitoring не может определить формат времени, каждая строка журнала обрабатывается как отдельная запись с автоматически присвоенной временной меткой (временная метка наблюдения) с точностью до одной минуты, за исключением строк, начинающихся с пробельных символов (пробел, табуляция), которые считаются продолжением предыдущей записи.

### Ограничение поиска временных меток

Независимо от формата, временная метка обычно находится в первых 64 символах записи журнала. Однако она может встречаться и в других местах — в этом случае можно увеличить этот лимит на странице конфигурации OneAgent: **Log Monitoring** > **Timestamp/Splitting patterns**.

### Правила временных меток

Независимо от расположения в записи журнала, временная метка может быть записана в нескольких форматах. Dynatrace поддерживает некоторые форматы временных меток по умолчанию, но иногда несколько форматов могут соответствовать входящим данным журнала и временная метка может быть определена по неверному шаблону.

По этой причине Log Monitoring также позволяет определять конкретный формат даты с помощью правил временных меток, задающих, что следует считать временной меткой в записи журнала. Правила содержат шаблон временной метки, часовой пояс и matchers.

* **Pattern** — определяет, что следует считать временной меткой в строке журнала.
* **Timestamp search limit** — задаёт количество символов в каждой строке журнала, начиная от начала строки, в пределах которых ищется временная метка.
* **Timestamps in indented lines** — включите эту опцию, если не нужно анализировать временные метки в строках, начинающихся с пробельных символов.
* **Time zone** — определяет часовой пояс временной метки. Необязательно, если шаблон временной метки включает индикатор часового пояса (`%z`).
* **Matcher** — сужает область применения правила и применяет шаблон временной метки только к совпадающим записям журнала. Поскольку атрибут `log.content` нельзя использовать в matchers шаблона временной метки, максимальная детализация — источник журнала. Такая детализация обусловлена тем, что шаблон временной метки используется для разбиения содержимого источника журнала на отдельные записи, то есть применяется до определения значения атрибута `log.content` (или любых других атрибутов на уровне отдельной записи).

  + При создании нескольких правил, соответствующих одним и тем же данным журнала, все определённые форматы времени учитываются при поиске.
  + Если хотя бы одно правило соответствует данному журналу, предопределённые форматы к нему не применяются.

### Несколько шаблонов временных меток в одном источнике журнала

При приёме записей журнала OneAgent анализирует их на наличие временной метки, используя список шаблонов matcher.

* Если запись содержит одну временную метку, OneAgent сопоставляет её со списком шаблонов, начиная с первого. Если временная метка соответствует одному из шаблонов, OneAgent использует её как начало записи журнала.
* Если запись содержит две и более временных метки, OneAgent начинает сопоставление с шаблона, совпавшего в предыдущей записи.

Таким образом, даже если первая временная метка записи соответствует первому шаблону в списке, OneAgent может выбрать вторую — потому что предыдущая запись соответствовала второму шаблону.
Пример приведён в блоке кода ниже.

```
Log entry 1: Pattern 1, Pattern 2



Log entry 2: Pattern 2



Log entry 3: Pattern 1, Pattern 2
```

Для записи 3, даже если «Pattern 1» стоит первым в списке шаблонов, OneAgent выберет «Pattern 2».

Чтобы OneAgent учитывал только нужные шаблоны временных меток, тщательно выбирайте шаблоны в списке. В примере выше, чтобы гарантировать использование Pattern 1, удалите Pattern 2 из списка. Это может привести к тому, что сообщения, содержащие только метки Pattern 2, не будут обработаны.

Например, если файл журнала содержит шаблоны `%FT%T` (2024-01-01T12:30:01) и `%F %T %Z` (2024-01-01 12:30:01 UTC), и OneAgent успешно совпадает с первым шаблоном в строке, этот шаблон будет приоритетным для последующих строк.

## Поддерживаемые области

Поддерживаются три иерархических области: хост, группа хостов и окружение.

Иерархические области объединяются в единый список в следующем порядке:

1. Правила хоста
2. Правила группы хостов
3. Правила окружения

Приоритет правил конфигурации временных меток

OneAgent получает объединённый список (объединённые списки из соответствующих хостов, групп хостов и окружений) без указания того, в каких областях они определены.

### Область хоста

Область хоста доступна через **Host settings** для конкретного хоста.

1. Перейдите в **Hosts**.
2. Найдите и выберите нужный хост для перехода на страницу обзора хоста.
3. В правом верхнем углу страницы обзора хоста выберите **More** (**...**) > **Settings**.
4. В настройках хоста выберите **Log Monitoring** > **Timestamp/Splitting patterns**.
5. Настройте разбиение данных, добавив правила с набором matchers, задающих, что следует считать временной меткой в записи журнала.

### Область группы хостов

Область группы хостов доступна через страницу **Host**.

1. Перейдите в **Hosts** и выберите нужный хост.
2. На странице обзора хоста выберите **Properties and tags**.
3. На панели **Properties and tags** найдите свойство **Host group**, чтобы увидеть название группы хостов, к которой принадлежит выбранный хост.

   Свойство **Host group** не отображается, если выбранный хост не принадлежит ни одной группе хостов.
4. Выберите название группы хостов, чтобы отобразить все хосты в этой группе. Откроется страница **OneAgent deployment**, отфильтрованная по выбранной группе хостов. Для каждого хоста в списке есть ссылка **Host group:** `<имя группы>`.
5. Выберите название группы хостов в любой строке.
6. В настройках группы хостов выберите **Log Monitoring** > **Timestamp/Splitting patterns**.
7. Настройте разбиение данных, добавив правила с набором matchers, задающих, что следует считать временной меткой в записи журнала.

### Область окружения

Область окружения доступна в меню настроек.

1. Перейдите в **Settings** > **Log Monitoring** > **Timestamp/Splitting patterns**.
2. Настройте разбиение данных, добавив правила с набором matchers для идентификации чувствительных данных.

## Создание правила

Чтобы добавить правило (на уровне хоста, группы хостов или окружения) для интерпретации временных меток входящих данных журнала:

1. Нажмите **Add rule** для начала настройки правила.
2. **Rule name**

   Название, отображаемое для данной конфигурации.
3. **Pattern**

   Введите шаблон, который следует считать датой из журналов. Подробнее о форматах временных меток см. в разделе [Поддерживаемые форматы временных меток (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/log-monitoring-configuration/timestamp-data-format "Поддерживаемые временные метки для последней версии Dynatrace Log Monitoring.") и [библиотеке Date](https://dt-url.net/o8034wt).

   Необходимо указать как минимум год, месяц, день, часы, минуты и секунды, хотя допустимы альтернативные форматы для них. Можно включить индикатор часового пояса (`%z`) или задать часовой пояс отдельно в определении правила.

   Правила без шаблона могут переопределять часовой пояс только для стандартных поддерживаемых временных меток.
4. **Timestamp search limit**

   Используйте это поле для определения количества символов в каждой строке журнала, в пределах которых ищется временная метка. Если нужно игнорировать временные метки и разбивать журналы по [правилам по умолчанию](/managed/upgrade/unavailable-in-managed "Данная функция недоступна в Dynatrace Managed."), установите это значение в `0`. Используйте это поле для перезаписи глобального [ограничения поиска временных меток](/managed/upgrade/unavailable-in-managed "Данная функция недоступна в Dynatrace Managed.") (по умолчанию 64 байта).
5. **Time zone**

   Выберите часовой пояс для применения к данному шаблону.
   Этот параметр недоступен, если часовой пояс уже указан в шаблоне временной метки (`%z`).

   Можно выбрать `Local time zone` для использования часового пояса хоста, на котором запущен OneAgent.
6. Нажмите **Add matcher**, чтобы создать конкретное условие сопоставления для данного правила и сузить его область применения.

   В одном правиле можно указать несколько matchers. Например, правило конфигурации временной метки можно применить к журналам из конкретного контейнера, пространства имён или источника журналов. Несколько matchers с одинаковым атрибутом используют логику AND, тогда как matchers с несколькими назначенными значениями используют логику OR.

   | Атрибут matcher | Сопоставление основано на |
   | --- | --- |
   | **Process group** | Идентификатор группы процессов. |
   | **Log source** | Путь к журналу; поддерживаются символы подстановки в виде звёздочки. |
   | **K8s container name** | Имя контейнера Kubernetes. |
   | **K8s namespace name** | Имя пространства имён Kubernetes. |
   | **K8s deployment name** | Имя развёртывания Kubernetes. |
   | **Container name** | Имя контейнера. |
   | **DT entity container group ID** | Идентификатор группы контейнеров сущности DT. |
   | **Process technology** | Имя технологии. |
7. Выберите атрибут сопоставления.
8. Нажмите **Value** и из списка **Value** выберите обнаруженные элементы данных журнала.

   К выбранному атрибуту можно добавить несколько значений. Например, один matcher может указывать `Log source` и соответствовать значениям `/var/log/syslog` и `Windows Application Log`. Используйте звёздочки (`*`) в качестве символов подстановки для частичного совпадения.
9. Нажмите **Save changes**.

Правила выполняются в том порядке, в котором отображаются на странице **Timestamp/Splitting patterns**.

Задержка распространения при изменении порядка правил

При изменении порядка правил (для изменения порядка их выполнения) предусмотрите задержку распространения в две-три минуты между сохранением изменения и его вступлением в силу.

Переключатель **Active**

Начиная с OneAgent версии 1.249, можно активировать/деактивировать правила с помощью переключателя **Active**. Для эффективного управления правилами рекомендуется обновить OneAgent до версии 1.249. Если на хосте с версией OneAgent ниже 249 настроены правила, их нельзя будет деактивировать — в этом случае нужно удалить такие правила с помощью кнопки **Delete** на уровне правила или через REST API.

Правила выполняются в том порядке, в котором отображаются на странице **Timestamp/Splitting patterns**.

## REST API

Для управления конфигурацией временных меток можно использовать Settings API:

* Просмотр схемы
* Список сохранённых объектов конфигурации
* Просмотр отдельного объекта конфигурации
* Создание, редактирование или удаление существующего объекта конфигурации

Чтобы проверить текущую версию схемы конфигурации временных меток, выведите список всех доступных схем и найдите идентификатор `builtin:logmonitoring.timestamp-configuration`.

Объекты конфигурации временных меток доступны для настройки в следующих областях:

* `environment` — объект конфигурации применяется ко всем хостам в данном окружении.
* `host_group` — объект конфигурации применяется ко всем хостам, назначенным данной группе хостов.
* `host` — объект конфигурации применяется только к данному хосту.

Чтобы создать конфигурацию временных меток через API:

1. [Создайте токен доступа](/managed/dynatrace-api/basics/dynatrace-api-authentication#create-token "Узнайте, как пройти аутентификацию для использования Dynatrace API.") с разрешениями **Write settings** (`settings.write`) и **Read settings** (`settings.read`).
2. Используйте endpoint [GET a schema](/managed/dynatrace-api/environment-api/settings/schemas/get-schema "Просмотрите схему настроек через Dynatrace API."), чтобы узнать формат JSON для отправки конфигурации. Идентификатор схемы (`schemaId`) для конфигурации временных меток — `builtin:logmonitoring.timestamp-configuration`. Пример JSON-payload с конфигурацией временных меток:

   ```
   [



   {



   "insertAfter":"uAAZ0ZW5hbnQABnRlbmFudAAkMGUzYmY2ZmYtMDc2ZC0zNzFmLhXaq0",



   "schemaId": "builtin:logmonitoring.timestamp-configuration",



   "schemaVersion": "0.1.0",



   "scope": "tenant",



   "value": {



   "config-item-title": "Added from REST API",



   "date-time-pattern": "%Y-%m-%d %H:%M:%S",



   "timezone": "CET",



   "matchers": [



   {



   "attribute": "dt.entity.process_group",



   "operator": "MATCHES",



   "values": [



   "PROCESS_GROUP-05F00CBACF39EBD1"



   ]



   },



   {



   "attribute": "log.source",



   "operator": "MATCHES",



   "values": [



   "Windows System Log",



   "Windows Security Log"



   ]



   }



   ]



   }



   }



   ]
   ```
3. Используйте endpoint [POST an object](/managed/dynatrace-api/environment-api/settings/objects/post-object "Создайте или проверьте объект настроек через Dynatrace API.") для отправки конфигурации.
