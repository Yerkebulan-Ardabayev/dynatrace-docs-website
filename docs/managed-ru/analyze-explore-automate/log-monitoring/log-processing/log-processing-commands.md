# Команды обработки журналов (Logs Classic)

* Обзор
* Чтение: 2 мин
* Обновлено 18 января 2023 г.

Log Monitoring Classic

Команды обработки журналов и событий представляют собой операторы и предложения DQL. **Определение правила** позволяет задавать последовательность команд для достижения желаемой цели правила обработки.

**Определение правила** обработки состоит из последовательности команд (слева направо), объединённых с помощью символа вертикальной черты (`|`). Концептуально это аналогично конвейеризации команд Unix, где вывод команды слева от черты становится вводом команды справа. Пайплайн всегда должен начинаться с команды, создающей структурированный поток данных для последующих команд.

## PARSE

Команда `PARSE` анализирует выбранное поле с помощью предоставленного шаблонного выражения, обогащая запись журнала экспортированными значениями.

### Примеры

* [Извлечение атрибутов из разных форматов в рамках одного шаблонного выражения.](/managed/analyze-explore-automate/log-monitoring/log-processing/log-processing-examples#lpexample5 "Примеры сценариев обработки журналов.")
* [Извлечение конкретных полей из JSON-содержимого.](/managed/analyze-explore-automate/log-monitoring/log-processing/log-processing-examples#lpexample4 "Примеры сценариев обработки журналов.")

## USING

Команда USING задаёт входные данные для определённого преобразования, перечисляя поля с опциями, которые могут передаваться из входящего события в преобразование. Эта команда должна быть первой в определении процессора.

Формат объявления полей: `USING(<field_expression>, <field_expression>, ...)`  
Формат выражения поля: `[IN|INOUT] name: [type[]?]`

Команда `USING` в определении процессора эквивалентна оператору `USING(IN content:STRING)`.

Опции:

* `IN` — поле только для чтения, опция по умолчанию.
* `INOUT` — поле с возможностью записи.
* `type` — один из следующих типов: `STRING`, `BOOLEAN`, `INTEGER`, `LONG`, `DOUBLE`, `DURATION`, `TIMESTAMP`, `IPADDR`.
* `[]` — обозначает поле как массив, то есть несколько значений атрибута будут переданы.
* `?` — обозначает поле как необязательное, то есть преобразование будет выполнено, даже если такого атрибута нет в событии.

Пример:

```
USING(content, event.type:STRING?, INOUT attribute:INT[]) | ...
```

В этом случае из входящего события в преобразование передаются три поля:  
`content` — строковое поле только для чтения  
`event.type` — необязательная строка только для чтения  
`attribute` — записываемый массив целых чисел

## FIELDS\_ADD

Команда `FIELDS_ADD` задаёт список литералов, функций, имён полей и вычисляемых выражений через запятую.

Формат:

```
FIELDS_ADD( [ alias':' ]
{ *
| alias.*
| '@'position
| function
| expression
| literal
}
[ , ... ]
)
```

Каждому выражению можно присвоить псевдоним. Псевдонимы, определённые в команде select, могут использоваться в последующих командах пайплайна.

### Пример

В следующем примере вычисляется сумма `field1` и `field2`, результат помещается в поле `mySum`.

| field1 | field2 |
| --- | --- |
| 300 | 650 |

```
FIELDS_ADD(mySum:(field1 + field2))
```

| field1 | field2 | mySum |
| --- | --- | --- |
| 300 | 650 | 950 |

## FIELDS\_RENAME

Команда `FIELDS_RENAME` переименовывает указанные поля записи. Порядок полей и остальные поля сохраняются.

Формат: `FIELDS_RENAME(newName:field, name2:field2, ...)`

### Пример

В следующем примере поле `ip` переименовывается в `src_addr`, а поле `f` — в `value`.

| t | ip | s | f |
| --- | --- | --- | --- |
| 2020-09-23 15:14:42.947 +0300 | 0.0.0.0 | 0ho0 | 0.0 |

```
FIELDS_RENAME(src_addr:ip, value:f)
```

| t | src\_addr | s | value |
| --- | --- | --- | --- |
| 2020-09-23 15:14:42.947 +0300 | 0.0.0.0 | 0ho0 | 0.0 |

## FIELDS\_REMOVE

Команда `FIELDS_REMOVE` удаляет указанные поля из выходного потока. Команда полезна в скриптах для удаления чувствительных полей.

Формат: `FIELDS_REMOVE(field, field, ...)`

### Пример

В следующем примере добавляются поля `t`, `ip`, `s` и `f`. Затем удаляются поля `s` и `ip`, оставляя поля `t` и `f`.

```
FIELDS_REMOVE(s, ip)
```

| t | f |
| --- | --- |
| 2020-09-23 15:17:39.497 +0300 | 0.0 |
| 2020-09-23 15:17:39.498 +0300 | 1.0 |
| 2020-09-23 15:17:39.499 +0300 | 2.0 |

## FILTER\_OUT

Команда `FILTER_OUT` отбрасывает всю запись с помощью выражения, возвращающего булево значение (boolean\_expression). Это команда негативной фильтрации. Она может состоять из нескольких булевых выражений, объединённых операторами AND, OR и NOT. Если условие выполнено, вся запись отбрасывается и не передаётся на следующий этап. Например, можно отбрасывать все записи с атрибутом уровня журнала `DEBUG` или `debug`.

Формат: `FILTER_OUT(loglevel IN ['DEBUG', 'debug'])`

### Примеры

* В следующем примере отбрасываются записи, где `_raw_text` (исходный текст записи) содержит строку `vpns` или `_raw_text` равен `NULL`:

  ```
  FILTER_OUT(contains(_raw_text, '/vpns/'))
  ```
* В следующем примере отбрасываются записи, где строковое поле `type` равно `logout` или `type` равен `NULL`:

  ```
  FILTER_OUT(type = 'logout')
  ```
* В следующем примере отбрасываются записи, где поле временной метки `last_modified` больше 7 дней назад или `last_modified` равен `NULL`:

  ```
  FILTER_OUT(last_modified >= now()[-7 day])
  ```
