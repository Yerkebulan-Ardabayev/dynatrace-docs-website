# Метрики журналов (Logs Classic)

* Учебное руководство
* Чтение: 7 мин
* Обновлено 18 января 2023 г.

Log Monitoring Classic

Dynatrace Log Monitoring позволяет не только просматривать и анализировать журналы, но и создавать метрики на основе данных журналов и использовать их в Dynatrace наравне с любой другой метрикой. Их можно добавлять на панель мониторинга, включать в анализ и создавать пользовательские оповещения.

Ценообразование на метрики журналов основано на модели Davis data units (DDU). Ознакомьтесь с разделом [DDU для метрик](/managed/license/monitoring-consumption-classic/davis-data-units/metric-cost-calculation "Узнайте, как рассчитать потребление и затраты DDU для отслеживаемых метрик.").

В зависимости от параметров, выбранных при создании метрики журнала, значение новой метрики может представлять:

* **Количество записей журнала** (доступно с Dynatrace версии 1.206+)
  Значение метрики представляет собой количество записей журнала, соответствующих запросу.
* **Значение атрибута** (доступно с Dynatrace версии 1.229+)
  Значение метрики может представлять одну из агрегаций, задаваемых в [Data Explorer](/managed/analyze-explore-automate/explorer "Запрашивайте метрики и преобразовывайте результаты для получения нужных инсайтов.").

При приёме данных журналов Dynatrace применяет заданный запрос и в зависимости от выбора **Measure** значение метрики представляет: количество записей, соответствующих запросу, или одно из следующих значений для указанного атрибута: `Average`, `Count`, `Maximum`, `Minimum`, `Sum`, `Median`, `Percentile 10th`, `Percentile 75th` или `Percentile 90th`. Указанный атрибут должен быть числового типа.

## Создание метрики журнала

Все метрики на основе матчера Log Monitoring имеют ключ метрики с префиксом `log.`.

Доступность метрики журнала в Dynatrace

Созданная метрика журнала доступна только при приёме новых данных журналов, соответствующих условию в поле **Matcher**. Убедитесь, что новые данные журнала были приняты до использования метрики в других областях Dynatrace.

Диапазон временных меток для принятия данных при создании метрики — от **1 часа** в прошлом до **10 минут** в будущем.

Есть два способа создания метрики на основе данных журналов:

### Создание метрики через настройки

1. Перейдите в **Settings** > **Log Monitoring** > **Metrics extraction** и нажмите **Add log metric**.
2. В **Metric key** добавьте имя метрики к ключу `log.`.
3. В **Matcher** введите запрос Log Monitoring для фильтрации данных журнала. Подробнее: [Просмотрщик журналов](/managed/analyze-explore-automate/log-monitoring/analyze-log-data/log-viewer#query-syntax "Узнайте, как использовать просмотрщик журналов Dynatrace.").
4. Выберите **Measure**:

   * **Occurrence of log records** — количество записей журнала, совпавших с запросом.
   * **Attribute value** — набор агрегатов для значения атрибута совпавших записей. При выборе этого варианта также укажите **Attribute**.
5. Опционально выберите **Add dimension** для добавления измерений.
   Измерения позволяют разбивать количество метрик журнала по конкретному атрибуту.
6. Нажмите **Save changes** для создания метрики журнала.

### Создание метрики через просмотрщик журналов

1. Перейдите в **Logs**.
2. Создайте запрос в просмотрщике журналов или используйте **Available attributes**.
3. Нажмите **Create metric**. Запрос появится на странице **Log metrics**.
4. Опционально добавьте измерения.
5. Добавьте имя метрики к ключу `log.` и сохраните изменения.

### Ограничения уникальных измерений

Если создаются метрики с высокой кардинальностью (много уникальных измерений), можно достигнуть лимита 1 000 000 измерений на метрику или 50 000 000 для всех метрик. Для смягчения этих ограничений:

* Сужайте запрос для уменьшения количества журналов в метрике.
* Снижайте кардинальность измерений.
* Отключайте метрику, превышающую лимит.

## Редактирование метрики журнала

Для просмотра, включения, отключения, удаления или изменения метрик, созданных из данных журналов, перейдите в **Settings** > **Log Monitoring** > **Metrics extraction**.

Изменение ключа метрики создаёт новую метрику. Исторические данные будут доступны только со старым ключом метрики.

## Пример

В этом примере создаётся и строится график метрики журнала, сохраняется на панель мониторинга и создаётся оповещение.

1. Перейдите в **Logs** для открытия просмотрщика журналов.
2. Создайте запрос для фильтрации интересующих данных. В этом примере: `status="error"`
3. Нажмите **Create metric**.
   Откроется страница **Log metrics** с заполненным полем **Matcher**.
4. Введите ключ метрики (уникальное имя). По умолчанию каждый ключ начинается с префикса `log.`.
   В этом примере: `log.error_PGI`
5. Нажмите **Add dimension** и выберите измерение `dt.entityprocess_group_instance` из списка.

   Без измерения Dynatrace будет считать ошибки глобально. В этом примере нужно видеть распределение статуса ошибки по экземплярам групп процессов. Измерение `dt.entityprocess_group_instance` позволит точно видеть, где возник статус ошибки.
6. Нажмите **Save changes**.

После определения метрики можно:

* **График:** Перейдите в **Data Explorer**, выберите метрику `log.error_PGI` и нажмите **Run query**.
* **Панель мониторинга:** После создания графика нажмите **Pin to dashboard** для добавления на классическую панель.
* **Оповещение:** Перейдите в **Settings** > **Anomaly detection** > **Metric events**, нажмите **Add metric event** и создайте пользовательское событие на основе `log.error_PGI`.
