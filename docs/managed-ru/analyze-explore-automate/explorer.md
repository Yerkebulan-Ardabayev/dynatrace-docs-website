# Data Explorer

# Data Explorer

* Чтение: 7 мин
* Обновлено 3 октября 2025 г.

Data Explorer значительно расширяет возможности запросов и визуализации метрик.

* Анализируйте многомерные метрики — встроенные в Dynatrace или поступающие из StatsD, Prometheus, Telegraf и других источников.
* Выбирайте тип визуализации.
* Добавляйте визуализацию непосредственно на классические дашборды для удобного доступа и совместного использования.
* Обеспечивает основу для расчёта метрик в визуализациях на классических дашбордах.

Быстрый старт

Если вы предпочитаете сразу перейти к практике, попробуйте [быстрый старт Data Explorer](/managed/analyze-explore-automate/explorer/explorer-quick-start "С помощью Data Explorer создайте первый запрос, выберите визуализацию и закрепите результаты на дашборде."). Вы узнаете, как:

* Использовать шаблон
* Создать простую визуализацию с нуля
* Использовать визуализацию непосредственно в Data Explorer
* Закрепить визуализацию на дашборде для дальнейшего использования и совместного доступа

Если же вас интересуют подробности — читайте ниже.

### Пространственная агрегация

Пространственная агрегация позволяет указать, как точки данных запроса метрики должны агрегироваться по измерениям.

Запрос всегда возвращает статистически наиболее точные результаты, даже если отдельные метрики предоставляют разные статистические показатели в зависимости от природы каждой метрики.

Чтобы изменить агрегацию, выберите её из списка сразу после имени метрики в редакторе запросов:

![Data Explorer: редактор запросов: выбор пространственной агрегации](https://dt-cdn.net/images/aggregation-select-579-2c3184b61a.png)

Data Explorer: редактор запросов: выбор пространственной агрегации

Для каждой метрики доступны одинаковые варианты пространственной агрегации: `Auto`, `Average`, `Count`, `Maximum`, `Minimum`, `Sum`, `Median`, `Percentile 10th`, `Percentile 75th` и `Percentile 90th`.

### Split by

По умолчанию запрос не разделяет по измерениям, используя агрегацию метрики. При разделении по измерению, например по хосту, агрегация применяется для каждого хоста.

Чтобы разделить по измерению

1. Если **Split by** ещё не отображается в редакторе запросов, нажмите значок «+» и выберите `Split by` из списка.
2. Установите **Split by** — измерение, по которому нужно разделить запрос.

#### Пример

Если метрика строки — `CPU usage %`, единственное доступное измерение — `Host`. В поле **Split by** выберите `Host`.

### Sort by

По умолчанию результаты сортируются по убыванию в соответствии с выбранной агрегацией.

Чтобы задать порядок сортировки

1. Если **Sort by** ещё не отображается в редакторе запросов, нажмите значок «+» и выберите `Sort by` из списка.
2. Установите **Sort by** — измерение, по которому нужно сортировать.
3. Выберите порядок сортировки: `ASC` (по возрастанию) или `DESC` (по убыванию).

### Rate

Чтобы задать частоту

1. Если **Rate** ещё не отображается в редакторе запросов, нажмите значок «+» и выберите `Rate` из списка.
2. Установите **Rate** в значение `None`, `Per second`, `Per minute` или `Per hour`.

### Filter by

Область действия определяется любым заданным фильтром. По умолчанию область — `(include all)`.

Чтобы отфильтровать запрос (изменить область действия)

1. Если **Filter by** ещё не отображается в редакторе запросов, нажмите значок «+» и выберите `Filter by` из списка.
2. В поле **Filter by**:

   * Выберите доступное измерение
   * Укажите атрибут
   * Укажите значение атрибута

Можно добавлять несколько фильтров.

#### Пример

Если метрика — **Action count (by Apdex category) [web]** (`builtin:apps.web.actionCount.category`) и нужно отфильтровать по конкретному веб-приложению `My web application`:

1. Если **Filter by** ещё не отображается, добавьте его.
2. В поле **Filter by** выберите `Web application`, затем `Name`, затем `My web application`.

См. также [Автоматически расширенная фильтрация](#auto-extended-filtering)

### Limit

По умолчанию, если метрики разделены по измерению, отображается не более 20 метрик.

Чтобы задать явное ограничение

1. Если **Limit** ещё не отображается в редакторе запросов, нажмите значок «+» и выберите `Limit` из списка.
2. Установите **Limit** в значение `1`, `10`, `20` или `100`.

Для визуализаций Honeycomb и Table это ограничение можно обойти, используя **Limit** > **Maximum** или включив **Advanced mode** и удалив `:limit(100)` из запроса.

### Global limit

**Global limit** выравнивает записи по нескольким метрикам с одинаковой размерностью (трансформация **Split by**). Если для метрики задан **Global limit**, в результат включаются только записи, совпадающие по измерениям этой метрики.

**Global limit** отображается автоматически при выполнении всех следующих условий:

* Выбрана визуализация **Table**.
* В таблицу добавлены минимум две метрики.
* Минимум две метрики имеют одинаковую трансформацию **Split by**.

Только одна метрика может быть отмечена как **Global limit**. Метрика с **Global limit** переопределяет трансформацию **Limit** всех остальных метрик с той же трансформацией **Split by**. В этом случае **Limit** для этих метрик выглядит отключённым.

В **Advanced mode** параметр **Global limit** всегда виден независимо от условий.

### Default

Чтобы задать значение по умолчанию

1. Если **Default** ещё не отображается в редакторе запросов, нажмите значок «+» и выберите `Default` из списка.
2. Установите **Default** — нужное значение по умолчанию.

### Timeshift

Чтобы задать сдвиг по времени

1. Если **Timeshift** ещё не отображается в редакторе запросов, нажмите значок «+» и выберите `Timeshift` из списка.
2. Установите **Timeshift** — число (положительное или отрицательное).
3. Выберите единицу (second, minute, hour, day, week) из соседнего списка.

#### Пример

Чтобы сдвинуть назад на две минуты:

1. Установите **Timeshift** в `-2`
2. Установите единицу `minute`

## Базовые линии

Для выявления аномалий можно использовать базовые линии — добавлять доверительный коридор к линии метрики на графике. Это позволяет видеть, когда значение выходит за пределы коридора.
Расчёт базовой линии основан на модели [сезонной базовой линии](/managed/dynatrace-intelligence/ai-models/seasonal-baseline "Как Davis предлагает пороговые значения сезонной базовой линии для набора объектов."), используемой для создания [событий метрик](/managed/dynatrace-intelligence/anomaly-detection/metric-events "Узнайте о событиях метрик в Dynatrace") для обнаружения аномалий.

* Базовые линии применяются только к визуализации `Graph`.
* Базовые линии не добавляются к плитке дашборда при закреплении графика.
* Временной диапазон для вычисления базовой линии определяется выбранным разрешением:

  | Диапазон разрешения | Примеры разрешения | Временной диапазон базовой линии |
  | --- | --- | --- |
  | resolution < 5 минут | + 1 минута | **предыдущие 14 дней** |
  | 5 минут ≥ resolution < 1 часа | + 5 минут, + 10 минут, + 30 минут | **предыдущие 28 дней** |
  | 1 час ≥ resolution < 1 дня | + 1 час, + 6 часов, + 12 часов | **400 дней** |
  | resolution ≥ 1 дня | + 1 день, + 1 неделя, + 1 месяц | **5 лет** |

### Добавление базовой линии

1. На графике выберите линию метрики, для которой нужна базовая линия.
2. В появившемся окне нажмите **Add baseline**.
3. Подождите, пока вычисляется базовая линия (`Loading`). После этого график перерисовывается с отображением базовой линии для выбранной метрики.

### Скрытие и отображение базовой линии

Базовые линии отображаются отдельно в легенде графика. Например, при добавлении базовой линии для метрики `CPU usage %` в визуализации `Graph` легенда показывает `CPU usage %` и `CPU usage % - baseline`. Нажимайте записи легенды для включения/выключения отображения.

### Удаление базовой линии

1. На графике выберите линию метрики, для которой нужно удалить базовую линию.
2. В появившемся окне нажмите **Remove baseline**. График перерисовывается без базовой линии.

### Сравнение с базовыми линиями событий метрик

Между базовыми линиями в Data Explorer и событиями метрик могут быть различия. Эти функции предлагают разные подходы для разных контекстов. В целом, конфигурация Data Explorer фиксирована, а конфигурация событий метрик настраивается.

|  | **Data Explorer** | **Metric events** |
| --- | --- | --- |
| **Samples** | `5` | Настраивается |
| **Violating samples** | `3` | Настраивается |
| **Dealerting samples** | `5` | Настраивается |
| **Alert on no data** | `false` | Настраивается |
| **Tolerance** (влияет на ширину доверительного коридора) | `4` | Настраивается (диапазон: `0.1`–`10`) |
| **Resolution** (влияет на детализацию) | Настраивается | `1` минута |
| **Training time** | Мгновенно | Ежедневно |

Подробности о сезонной базовой линии см. в разделе [Seasonal baseline](/managed/dynatrace-intelligence/ai-models/seasonal-baseline "Как Davis предлагает пороговые значения сезонной базовой линии для набора объектов.").

### Часто задаваемые вопросы о базовых линиях

Как рассчитывается базовая линия?

Расчёт базовой линии основан на модели сезонной базовой линии, используемой для создания [событий метрик](/managed/dynatrace-intelligence/anomaly-detection/metric-events "Узнайте о событиях метрик в Dynatrace") для обнаружения аномалий. Подробности о внутренней работе модели см. в разделе [Seasonal baseline](/managed/dynatrace-intelligence/ai-models/seasonal-baseline "Как Davis предлагает пороговые значения сезонной базовой линии для набора объектов.").

Почему базовая линия отличается от предварительного просмотра модели сезонной базовой линии?

Хотя модель базовой линии основана на модели [сезонной базовой линии](/managed/dynatrace-intelligence/ai-models/seasonal-baseline "Как Davis предлагает пороговые значения сезонной базовой линии для набора объектов."), результирующие базовые линии могут различаться по нескольким причинам:

* **Разрешение**: базовая линия в Data Explorer выводится из данных в зависимости от выбранного разрешения, как описано выше. Модель сезонной базовой линии для события метрики всегда обучается на данных с разрешением 1 минута. При разных разрешениях базовые линии различаются, поскольку и значения метрик различаются.
* **Временной диапазон базовой линии**: временной диапазон определяется выбранным разрешением, как описано выше. Поскольку конфигурации событий метрик всегда используют данные с разрешением 1 минута, временной диапазон обучения может отличаться.
* **Параметры модели**: базовая линия в Data Explorer использует фиксированные параметры по умолчанию:

  + Tolerance = 4
  + Alert condition = 'Alert if metric is outside'
  + No alert on missing data
    Если параметры в конфигурации событий метрик отличаются, результирующая базовая линия может быть иной.

## Коррелированные метрики

Dynatrace Davis® учитывает специфические для предметной области знания и топологию при вычислении связанных сигналов наблюдаемости. Davis ранжирует наиболее релевантные сигналы вверху, а оценка Davis для каждого обнаруженного сигнала показывает, насколько точно сигнал соответствует поведению референсного сигнала за выбранный временной диапазон. [Подробнее об AI Davis®](/managed/dynatrace-intelligence "Ознакомьтесь с возможностями Davis AI.").

### Добавление коррелированных метрик

Этот параметр доступен только при использовании **Split by** в запросе.

1. Перейдите в **Data Explorer** (стандартный или расширенный режим), создайте запрос метрики, разделённой по связанному измерению, и отобразите его в визуализации `Graph`.

   Коррелированные метрики доступны *только* при следующих условиях:

   * Выбрана визуализация `Graph`
   * Запрос содержит **Split by** по измерению, связанному с выбранными данными

   Попробуйте этот пример:

   ![Коррелированные метрики: запрос: стандартный режим](https://dt-cdn.net/images/query-cpu-usage-1217-d0a8083c84.png)

   Коррелированные метрики: запрос: стандартный режим

   В **Advanced mode** это выглядит так:

   ![Коррелированные метрики: запрос: Advanced mode](https://dt-cdn.net/images/query-cpu-usage-advanced-mode-1220-9fb8efd9b1.png)

   Коррелированные метрики: запрос: Advanced mode

   ```
   builtin:host.cpu.usage:splitBy("dt.entity.host"):sort(value(auto,descending)):limit(20)
   ```
2. Нажмите **Run query**, чтобы построить график.
3. Выберите (нажмите) линию на графике, чтобы отобразить всплывающее окно со связанными параметрами.
4. В всплывающем окне нажмите **See correlated metrics**.

   На боковой панели **Davis for Correlation analysis** перечислены метрики, коррелирующие с выбранными данными согласно корреляционному анализу Davis AI. Корреляция определяется формой ряда, а не значениями.

   Что показывает анализатор?

   * **Reference signal** — выбранный на графике ряд данных. Формы других рядов метрик сравниваются с формой этого ряда.
   * **Connected signals** — другие ряды метрик с похожей формой, отсортированные от наиболее до наименее похожих. Чем похожее форма, тем теснее корреляция.

     Для каждой коррелированной метрики анализатор показывает:

     + Имя метрики
     + Измерение
     + Идентификатор объекта

     Корреляции иногда группируются.
5. На боковой панели нажмите любую перечисленную метрику, чтобы автоматически добавить её в текущий запрос.

   * Можно добавлять несколько коррелированных метрик в запрос
   * Одну и ту же метрику можно добавлять несколько раз, а затем редактировать запрос
6. После добавления коррелированных метрик нажмите **Run query** для обновления графика.

### Часто задаваемые вопросы о коррелированных метриках

Что означает «корреляция» в данном контексте?

Для определения корреляции анализатор проверяет форму рядов данных, а не значения. Два ряда с очень похожими формами считаются коррелированными.

Почему параметр «See correlated metrics» недоступен?

Возможные причины отсутствия параметра «See correlated metrics»:

* Не выбрана визуализация `Graph`
* В запросе не задан **Split by** по измерению
* Запрос не был выполнен для построения графика
* Не выбрана (не нажата) ни одна линия на графике

Что означает «No connected signals found»?

Если отображается `No connected signals found`, возможные причины:

* Слишком малая дисперсия в выборке (например, метрика, отображаемая в виде прямой линии)
* Слишком мало точек данных в выборке (например, за очень короткий временной диапазон)

## Автоматически расширенная фильтрация

Автоматически расширенные фильтры используют топологию Dynatrace (модель объектов) для предложения дополнительных измерений фильтрации, недоступных в исходной метрике. Они работают как на уровне плитки, так и на уровне дашборда.

* На уровне плитки выбирайте их при задании **Filter by** в Data Explorer, просто выбрав исходное измерение метрики с назначенной связью. Например, метрика, отслеживающая производительность для событий Synthetic, имеет связь с монитором Synthetic. Используя топологию, можно сначала выбрать связанное событие Synthetic в фильтре, а затем, помимо имени, тега, идентификатора или состояния работоспособности, получить дополнительный параметр для выбора связанного монитора.
* На уровне дашборда Dynatrace автоматически расширяет метрики там, где это возможно, чтобы при передаче динамического фильтра он мог применяться к плитке с этой метрикой.

## Примечания и ограничения

* Максимум 10 метрик на визуализацию
* До 100 рядов на метрику

  Для визуализации Honeycomb это ограничение можно обойти: включите **Advanced mode** и удалите `:limit(100)` из запроса.
* В отличие от многомерного анализа, Data Explorer использует долгосрочные данные метрик, а не [данные трассировок и запросов](/managed/observe/application-observability/multidimensional-analysis#data-source "Настройте представление многомерного анализа и сохраните его как вычисляемую метрику."), поэтому значения на визуализациях могут отличаться от значений в многомерном анализе.
* Для предотвращения проблем производительности на плитках дашборда, созданных с помощью Data Explorer, максимальное количество точек данных для запроса на плитке — 4000. На основе выбранного временного диапазона и применённого пользовательского разрешения Dynatrace прогнозирует количество точек данных. Если прогнозируемое значение превышает 4000, Dynatrace автоматически переключается на достаточно высокое разрешение, чтобы удержать число точек данных ниже 4000.

  Это ограничение не распространяется на визуализации в самом Data Explorer, где можно иметь более 4000 точек данных. Оно применяется только к плиткам дашборда, созданным с помощью Data Explorer, где комбинация разрешения/временного диапазона на дашборде приводит к более 4000 точек данных.
* Примеры нотации порядка величины в Dynatrace:

  | Обозначение | Множитель | Значение |
  | --- | --- | --- |
  | k | 10^3 | кило, тысяча |
  | M | 10^6 | мега, миллион |
  | G | 10^9 | гига, миллиард |
  | T | 10^12 | тера, триллион |

  Подробности см. в разделе [Нотация порядка величины](/managed/discover-dynatrace/get-started/dynatrace-ui/order-of-magnitude-notation "Нотация порядка величины Dynatrace для отображения значений метрик").

### Трансформация свёртки и разрешение

Учтите, что параметр **Fold transformation** влияет на разрешение.

* Если **Fold transformation** установлен в `Auto` для визуализации `Table`, `Single value`, `Top list` или `Honeycomb`, используется разрешение `Inf` (бесконечность) для обратной совместимости. Если выбранный селектор метрик не поддерживает разрешение `Inf`, трансформация `fold` автоматически добавляется в конец запроса.
* Если **Fold transformation** установлен в значение, отличное от `Auto`, используется `fold`.

Поскольку все селекторы метрик запрашиваются с использованием одного и того же механизма (либо `fold`, либо `Inf`), добавление нового селектора, требующего `fold`, может изменить результат других селекторов.

Чтобы просмотреть фактический запрос, используемый Data Explorer, перейдите в раздел **Result** в Data Explorer и нажмите значок «…» > **Copy request**.
