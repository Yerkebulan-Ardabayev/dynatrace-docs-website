# Профили оповещений о проблемах

* Чтение: 4 мин
* Обновлено 26 августа 2025 г.

Профили оповещений о проблемах управляют доставкой уведомлений о проблемах по каналам оповещений организации на основе предварительно заданных фильтров — по серьёзности проблемы, её продолжительности, пользовательским событиям и тегам. Профили оповещений позволяют точно контролировать, при каких условиях отправляются уведомления о проблемах, а при каких — нет. Это относится ко всем push-уведомлениям о проблемах, отправляемым через [мобильное приложение Dynatrace](/managed/analyze-explore-automate/notifications-and-alerting/push-notifications-via-the-dynatrace-mobile-app "Узнайте, как подключить среды Dynatrace к мобильному приложению для получения оповещений о проблемах.") и отображаемым в веб-интерфейсе Dynatrace. Профили оповещений также используются для настройки фильтрованных интеграций уведомлений о проблемах с такими сторонними системами, как [Slack](/managed/analyze-explore-automate/notifications-and-alerting/problem-notifications/slack-integration "Настройте интеграцию уведомлений о проблемах со Slack."), [Opsgenie](/managed/analyze-explore-automate/notifications-and-alerting/problem-notifications/opsgenie-integration "Настройте интеграцию Opsgenie с Dynatrace.") и [PagerDuty](/managed/analyze-explore-automate/notifications-and-alerting/problem-notifications/pagerduty-integration "Узнайте, как отправлять уведомления о проблемах из Dynatrace в PagerDuty.").

Каждая среда мониторинга имеет профиль оповещений по умолчанию, который определяет уровень серьёзности, при котором отправляется оповещение об обнаруженной проблеме.

Администратор среды может создавать новые профили оповещений, обеспечивающие детальный контроль над маршрутизацией оповещений о проблемах для отдельных команд или сторонних интеграций, таких как Slack, HipChat и других.

Чтобы просмотреть профили оповещений среды, перейдите в **Settings** > **Alerting** > **Problem alerting profiles**. Профиль `Default` всегда присутствует в списке. Его можно изменить, но нельзя удалить.

## Область применения профиля оповещений

Профили оповещений предоставляют мощный механизм фильтрации уведомлений о проблемах. Комбинируя критерии фильтрации, можно создавать пользовательские профили — например, задерживать уведомления о проблемах в средах разработки, немедленно оповещая о проблемах в производственной среде. Область применения профиля оповещений определяется:

* Зоной управления.
* Правилами серьёзности.
* Фильтрами событий.

Эти условия объединяются по логике **И**. Событие должно выполнять **все** условия, чтобы вызвать уведомление на основе профиля.

Сопоставление тегов

Сопоставление тегов в профилях оповещений является точным. Если указать только ключ, тег будет сопоставляться с сущностями, имеющими данный ключ без значения. Например, если указать ключ `Apps` без значения, он совпадёт с сущностью, имеющей ключ `Apps` с пустым значением. Однако он не совпадёт с сущностью `AppsRedis`, у которой тот же ключ, но непустое значение.

Для сопоставления тегов «ключ-значение» необходимо указывать и ключ, и значение.

### Зона управления

Фильтр зоны управления сокращает объём данных, анализируемых профилем оповещений. Вместо проверки всех данных среды можно сосредоточиться только на данных из указанной зоны управления.

По умолчанию новый профиль оповещений использует `All` (все зоны управления), то есть фильтр не применяется. В большинстве случаев рекомендуется применять фильтр зоны управления для сужения области применения профиля до зоны ответственности команды. Имейте в виду, что зоны управления могут перекрываться. Если проблема обнаружена в сервисе, определённом в нескольких зонах управления, применяется несколько фильтров.

### Правила серьёзности

Правила серьёзности фильтруют события по уровню серьёзности. Для каждого профиля оповещений можно задать до 100 правил серьёзности. Правила объединяются по логике **ИЛИ**. Событие, выполняющее **любое** из правил, вызывает уведомление на основе профиля.

Доступные критерии:

* [Уровень серьёзности](/managed/dynatrace-intelligence/root-cause-analysis/event-analysis-and-correlation/event-categories "Узнайте о различных категориях событий и поддерживаемых типах событий, уровнях серьёзности и логике их формирования.").
* Как долго проблема открыта перед отправкой оповещения — это позволяет избегать оповещений о малосерьёзных проблемах, не влияющих на клиентский опыт и не требующих немедленного внимания.
* Необязательно: отслеживаемые сущности, имеющие любой или все из указанных тегов.

Критерии правил объединяются по логике **И**. Для применения правила должны быть выполнены **все** критерии.

### Фильтры событий

Правила событий фильтруют события по их свойствам. Для каждого профиля оповещений можно задать до 20 правил событий. Особенно для сценариев автоматического устранения неисправностей полезно запускать конкретные действия на основе детальной информации, фиксируемой при нештатных ситуациях — например, вызывать оповещения при авариях процессов.

Доступные критерии:

* Предопределённые события — тип события
* Пользовательские события — заголовок, описание и свойства события

Каждый критерий можно инвертировать, используя параметр **negate** (отрицание). Например, критерий **begins with** превращается в **does not begin with**.

Правила объединяются по следующей логике:

1. Все правила, содержащие отрицательные критерии, группируются по логике **И**.
2. Все остальные правила группируются по логике **ИЛИ**.
3. Эти две группы (с отрицанием и без) объединяются по логике **И**.

## Создание профиля оповещений

Чтобы создать профиль оповещений:

1. Перейдите в **Settings** > **Alerting** > **Problem alerting profiles** и выберите **Add alerting profile**.
2. Введите имя нового профиля в поле **Create new alerting profile** и нажмите **Create**.
3. Определите фильтр [зоны управления](/managed/manage/identity-access-management/permission-management/management-zones "Узнайте о концепции зон управления, как их определять и как максимально использовать.").
4. Определите правила [уровня серьёзности](/managed/dynatrace-intelligence/root-cause-analysis/event-analysis-and-correlation/event-categories "Узнайте о различных категориях событий и поддерживаемых типах событий, уровнях серьёзности и логике их формирования.") для профиля.
5. Определите фильтр событий на основе одного из вариантов:

   * `Predefined` — конкретный встроенный тип события.
   * `Custom` — заголовок, описание и свойства события.
6. Проверьте сводку критериев нового профиля оповещений и при необходимости измените их.
7. Нажмите **Done** в верхнем правом углу, чтобы сохранить новый профиль оповещений.

## API профилей оповещений

Помимо веб-интерфейса Dynatrace, профилями оповещений о проблемах можно управлять через [Settings API](/managed/dynatrace-api/configuration-api/alerting-profiles-api "Узнайте, что предлагает API профилей оповещений Dynatrace."). Используйте схему **Problem alerting profiles** (`builtin:alerting.profile`).
