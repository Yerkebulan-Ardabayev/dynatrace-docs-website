# Автоадаптивные пороги для обнаружения аномалий

* Explanation
* 3-min read
* Updated on Jan 28, 2026

Автоадаптивные пороги — это динамический подход к базовому мониторингу, при котором эталонное значение для обнаружения аномалий изменяется со временем. Главное преимущество перед [статическим порогом](/managed/dynatrace-intelligence/anomaly-detection/static-thresholds "Когда использовать статический порог для обнаружения аномалий.") состоит в том, что эталонное значение динамически адаптируется со временем — вам не нужно заранее знать порог и вручную обновлять множество статических порогов для метрик, поведение которых меняется со временем.

Если конфигурация обнаружения аномалий включает несколько объектов, каждый объект получает собственный автоадаптивный порог, и каждый порог оценивается независимо. Например, если область действия конфигурации включает пять хостов, Dynatrace вычисляет и оценивает пять независимых порогов.

Существует **ограничение в 100 конфигураций метрических событий на среду**, вне зависимости от того, сколько отдельных порогов содержит каждая конфигурация.

## Автоадаптивный порог vs. статический порог

Рассмотрим пример, в котором адаптивный порог имеет преимущество перед статически заданным. График ниже показывает измеренное время записи диска в миллисекундах. Это нестабильная метрика, значение которой резко возрастает в зависимости от интенсивности операций записи на диск. Если определить порог для каждого диска в данной ИТ-системе на основе начальных данных (начало графика), мы установим статический порог в 20 миллисекунд. Однако впоследствии нагрузка на диск возрастёт, и статически заданный порог начнёт генерировать множество ложноположительных оповещений. Чтобы этого избежать, потребовалось бы вручную задать новый порог и обновить конфигурацию.

Автоадаптивные пороги, напротив, ежедневно автоматически адаптируют эталонные пороги на основе измерений за предыдущие семь дней. Если поведение метрики изменяется, порог адаптируется автоматически.

## Расчёт порога

Эталонными значениями для расчёта порога служат значения метрики за последние семь дней.

* Измерения за каждую минуту используются для расчёта 99-го перцентиля всех измерений. Это определяет соответствующий **базис**.
* Межквартильный размах между 25-м и 75-м перцентилями используется как **флуктуация сигнала**, которая может добавляться к базису. С помощью параметра `number of signal fluctuation` (n × флуктуация сигнала) можно управлять тем, сколько раз флуктуация сигнала добавляется к базису для формирования фактического порога оповещения.

Ещё один важный параметр для динамических порогов — скользящее окно, используемое для сравнения текущих измерений с рассчитанным порогом. Оно определяет, как часто рассчитанный порог должен нарушаться в рамках скользящего окна времени для формирования события (нарушения не обязаны быть последовательными). Такой подход помогает избежать чрезмерно агрессивных оповещений при единичных нарушениях. Скользящее окно можно задать максимум в 60 минут.

По умолчанию любые 3 минуты из скользящего окна в 5 минут должны нарушать порог для формирования события. То есть событие должно иметь 3 нарушающих минуты в любом скользящем окне продолжительностью 5 минут.
