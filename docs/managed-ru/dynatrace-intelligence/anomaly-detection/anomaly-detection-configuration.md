# Конфигурация обнаружения аномалий

# Конфигурация обнаружения аномалий

* Explanation
* 7-min read
* Updated on Apr 17, 2026

Конфигурация обнаружения аномалий включает несколько компонентов:

* Источник данных — временной ряд, который оценивается. Например, конкретная метрика.
* Тип и параметры анализатора: способ оценки данных.
* Скользящее окно: период, в течение которого оцениваются данные.
* Шаблон события: тип шаблона, формируемого конфигурацией.

После настройки и активации конфигурация отслеживает данные и формирует событие при выполнении условий. Чтобы убедиться в корректной работе конфигурации и получении оповещений о нужных событиях, можно просмотреть результаты её работы в предварительном просмотре:

* Предварительный просмотр метрического события визуально отображает поведение события. Можно изменять настройки и наблюдать, как они влияют на конфигурацию.

## Источник данных

Источник данных предоставляет временной ряд, который оценивается Davis:

* Метрика определяет временной ряд. Это может быть как отдельная метрика, определяемая ключом метрики, так и [выражение метрики](/managed/dynatrace-api/environment-api/metric-v2/metric-expressions "Используйте выражения метрик для применения арифметических операций в запросах к точкам данных через Metrics API v2.").

Если ваши данные имеют задержку, её необходимо компенсировать в конфигурации с помощью параметра **Query offset**. Укажите значение в минутах.

## Тип и параметры анализатора

Параметры анализатора определяют, как Davis оценивает данные, предоставляемые источником данных. Конкретный набор параметров зависит от типа анализа:

* [Автоадаптивный порог](/managed/dynatrace-intelligence/anomaly-detection/auto-adaptive-threshold "Как Dynatrace адаптирует пороги для нескольких объектов в рамках конфигурации обнаружения аномалий.") — Dynatrace автоматически рассчитывает порог и динамически адаптирует его к поведению данных.
* [Сезонный базис](/managed/dynatrace-intelligence/ai-models/seasonal-baseline "Как Davis предлагает пороги сезонного базиса для набора объектов.") — Dynatrace создаёт доверительный диапазон для данных с сезонными закономерностями.
* [Статический порог](/managed/dynatrace-intelligence/anomaly-detection/static-thresholds "Когда использовать статический порог для обнаружения аномалий.") — порог, не изменяющийся со временем.

| Параметр | Описание |
| --- | --- |
| Number of signal fluctuations | Автоадаптивный порог состоит из двух компонентов: базиса и флуктуации сигнала. Этот параметр определяет, сколько раз флуктуация сигнала добавляется к базису. Подробнее см. в разделе [Расчёт порога](/managed/dynatrace-intelligence/anomaly-detection/auto-adaptive-threshold#calculation "Как Dynatrace адаптирует пороги для нескольких объектов в рамках конфигурации обнаружения аномалий."). |
| Threshold | Этот параметр определяет значение статического порога и, при наличии, его единицу измерения. Выберите **Suggest values**, чтобы использовать значение, рассчитанное Davis на основе предыдущих данных. |
| Tolerance | Этот параметр определяет [допуск](/managed/dynatrace-intelligence/ai-models/seasonal-baseline#parameters "Как Davis предлагает пороги сезонного базиса для набора объектов.") сезонной модели. Более высокий допуск означает более широкий доверительный диапазон, что ведёт к меньшему числу срабатываний событий. |
| Alert condition | Этот параметр определяет условие формирования события: если метрика превышает, опускается ниже или выходит за пределы порога. |
| Missing data alert | Этот параметр определяет, активно ли оповещение об отсутствующих данных для конфигурации. При активности оно объединяется с пороговым условием по логике **ИЛИ**. Параметр находится в разделе **Advanced properties** конфигурации. |

## Оповещение об отсутствующих данных

Dynatrace предоставляет возможность настроить оповещение об отсутствии данных в метрике. Если оповещение активно, Dynatrace регулярно проверяет, содержит ли скользящее окно конфигурации обнаружения аномалий какие-либо измерения. Например, если скользящее окно установлено на **3 нарушающих выборки в течение любых 5 минут**, Dynatrace формирует оповещение при отсутствии данных в течение 3 минут.

Условие отсутствия данных и пороговое условие объединяются по логике **ИЛИ**.

Рекомендуется отключать оповещения об отсутствующих данных для разреженных потоков данных, где измерения не ожидаются через регулярные интервалы, так как это приводит к шторму оповещений.

Для ожидаемых данных с опозданием (например, метрик облачной интеграции с задержкой 5 минут) используйте длинные скользящие окна, покрывающие задержки. Для задержки в 5 минут используйте скользящее окно не менее 10 минут.

Заполнитель `{missing_data_samples}` в описании события заменяется на количество минут без полученных данных.

## Скользящее окно

Скользящее окно конфигурации обнаружения аномалий определяет, сколько одноминутных выборок должны нарушать порог в течение определённого периода. При достижении заданного числа нарушений Dynatrace формирует событие. Цель — избежать чрезмерно агрессивных оповещений при единичных нарушениях, когда каждое измерение, нарушающее порог, формирует событие.

Событие остаётся открытым до тех пор, пока метрика не вернётся в пределы порога на определённое число одноминутных выборок в рамках того же скользящего окна, после чего Dynatrace закрывает событие. Удержание события в открытом состоянии помогает избежать избыточных оповещений, добавляя новые нарушения порога к существующей проблеме вместо формирования новой.

Настройки скользящего окна находятся в разделе **Advanced properties** конфигурации. По умолчанию:

* Для формирования события необходимо, чтобы любые 3 одноминутные выборки из 5 нарушали порог.
* Для закрытия события необходимо 5 одноминутных выборок с нормальными значениями.

Для каждого из трёх типов анализатора можно задать скользящее окно до 60 минут.

Рассмотрим пример статического порога в 90% использования CPU.

Explanation of the sliding window and de-alerting

Анализ события начинается с первой нарушающей выборки в скользящем окне. Когда число нарушающих выборок достигает заданного порога, анализ события прекращается и формируется проблема. Несмотря на прекращение анализа события, само событие остаётся открытым до выполнения критериев снятия оповещения:

* Число нарушающих выборок должно быть меньше порогового числа для формирования проблемы.
* Число «нормальных» выборок должно быть больше или равно числу выборок для снятия оповещения.

Для закрытия события **оба** критерия должны быть выполнены.

Значения по умолчанию (3 нарушающих выборки в скользящем окне из 5 для формирования проблемы, 5 выборок для снятия оповещения) хорошо подходят для большинства конфигураций. Однако может потребоваться их изменение (например, из-за шума в измерениях).

## Шаблон события

Шаблон события определяет характеристики события, формируемого при нарушении порога. Необходимо указать как минимум имя и тип события.

* Для быстрого понимания события имя должно быть кратким и понятным описанием ситуации, например `High network activity` или `CPU saturation`.
* Имя может включать заполнители, такие как `{threshold}` или `{alert_condition}`. В фактическом событии заполнители заменяются реальными значениями. Для просмотра доступных заполнителей введите `{` в поле ввода. Доступны следующие заполнители:

  + `{alert_condition}` — условие оповещения (выше/ниже порога).

    Если для параметра **Alert condition** задано значение `Alert if the metric is outside`, заполнители `{threshold}`, `{severity}` и `{baseline}` будут недоступны.
  + `{baseline}` — нарушенное значение базиса.
  + `{dims}` — список всех измерений (и их значений) метрики, нарушившей порог. Можно также указать конкретное измерение: `{dims:dt.entity.<entity>}`. Для получения списка доступных измерений вашей метрики выполните запрос GET metric descriptor.
  + `{entityname}` — имя затронутого объекта.
  + `{metricname}` — имя метрики, нарушившей порог.
  + `{missing_data_samples}` — число выборок с отсутствующими данными. Доступно только если включено оповещение об отсутствующих данных.

    `{missing_data_samples}` в описании события

    Рекомендуется включать заполнитель `{missing_data_samples}` в описание события, чтобы понять, вызвана ли проблема отсутствием данных или нарушениями порога.
  + `{severity}` — серьёзность события.
  + `{threshold}` — нарушенное значение порога.

Дополнительные параметры можно указать в виде пар «ключ — значение». Список возможных свойств событий см. в [Semantic Dictionary](/managed/upgrade/unavailable-in-managed "Выбранный раздел недоступен в Dynatrace Managed.").
