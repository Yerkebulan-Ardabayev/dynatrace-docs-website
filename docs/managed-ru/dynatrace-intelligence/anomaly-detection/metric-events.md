# Метрические события

# Метрические события

* Overview
* 5-min read
* Updated on Mar 04, 2024

Davis® AI Dynatrace автоматически анализирует аномальные ситуации в вашей ИТ-инфраструктуре и пытается определить соответствующее влияние и первопричину. Davis опирается на широкий спектр источников информации: транзакционный вид ваших сервисов и приложений, а также все события, формируемые на отдельных узлах топологии Smartscape®. Одним из источников событий в Dynatrace являются метрические события, то есть события на основе метрических данных. Они настраиваются в глобальных настройках вашей среды и видны всем пользователям Dynatrace в вашей среде. Существует два типа метрических событий в зависимости от способа запроса метрики для оценки события:

* [Ключ метрики](/managed/dynatrace-intelligence/anomaly-detection/metric-events/metric-key-events "Узнайте о метрических событиях на основе ключа метрики."). События на основе ключа метрики оценивают входящие измерения одной метрики. С этим типом запроса можно использовать только статические пороги.
* [Селектор метрики](/managed/dynatrace-intelligence/anomaly-detection/metric-events/metric-selector-events "Узнайте о метрических событиях на основе селектора метрики."). События на основе селектора метрики оценивают сложный запрос, определяемый [селектором](/managed/dynatrace-api/environment-api/metric-v2/metric-selector "Настройте селектор метрики для Metric v2 API."). Этот тип запроса может включать исторические данные и даже [арифметические операции](/managed/dynatrace-api/environment-api/metric-v2/metric-expressions "Используйте выражения метрик для применения арифметических операций в запросах к точкам данных через Metrics API v2.") с несколькими метриками.

Dynatrace использует три стратегии мониторинга для таких событий:

* [Статический порог](/managed/dynatrace-intelligence/anomaly-detection/static-thresholds "Когда использовать статический порог для обнаружения аномалий.") — порог, не изменяющийся со временем.
* [Автоадаптивный порог](/managed/dynatrace-intelligence/anomaly-detection/auto-adaptive-threshold "Как Dynatrace адаптирует пороги для нескольких объектов в рамках конфигурации обнаружения аномалий.") — Dynatrace автоматически рассчитывает порог и динамически адаптирует его к поведению метрики.
* [Сезонный базис](/managed/dynatrace-intelligence/ai-models/seasonal-baseline "Как Davis предлагает пороги сезонного базиса для набора объектов.") — Dynatrace рассчитывает доверительный диапазон для метрики с сезонными закономерностями.

* Автоадаптивные пороги и сезонный базис доступны только для событий на основе селектора метрики.
* Количество конфигураций метрических событий (как на основе ключа метрики, так и на основе селектора метрики) ограничено **10 000**.

## Обзор метрических событий

Dynatrace предоставляет обзор всех ваших метрических событий с информацией об ограничениях и сбоях метрических событий за последние 24 часа. Для перехода к обзору выберите **Settings** > **Anomaly detection** > **Metric events**.

![Metric events overview](https://dt-cdn.net/images/metric-events-overview-1194-c3db36a9d3.png)

Metric events overview

В обзоре ограничений отображается, какая часть каждого типа ограничений использована. Ссылка для просмотра ведёт в Data Explorer с предзаполненным запросом для отображения наиболее нагружающих конфигураций.

Обзор метрических событий включает конфигурации, испытывавшие проблемы в течение последних 24 часов. Отслеживаемые проблемы:

* `THROTTLED`: конфигурация достигла лимитов запросов селектора метрики.
* `QUERY_FAILED`: запрос метрики конфигурации завершился ошибкой. Возможные причины сбоя запроса:

  + Management zones или теги более не доступны.
  + Данные метрики более не доступны.
  + Запрашиваемые данные недоступны в разрешении 1 минута.
* `CONFIG_BLOCKED`: конфигурация создаёт значительную нагрузку и была заблокирована во избежание негативного влияния на другие пользовательские оповещения.

## Management zones в метрических событиях

Если у вас есть разрешения на уровне management zone, вы можете создавать конфигурации метрических событий, привязанные к этим management zones. Такие конфигурации используют только данные из указанной management zone. Также можно просматривать все конфигурации management zone, использующие management zone, к которой у вас есть доступ.

Для создания метрических событий без охвата management zone необходим доступ администратора.

Management zones, настроенные через [правила для многомерных данных](/managed/manage/identity-access-management/permission-management/management-zones/management-zone-rules#logs-metrics "Определите правила для ограничения объектов, доступных в рамках management zone."), не поддерживаются для метрических событий.

## Осведомлённость о топологии

Осведомлённость о топологии и контексте — ключевые темы платформы наблюдаемости Dynatrace. Данные — такие как метрики, трассировки, события и логи — не просто сообщаются и хранятся на платформе. Эти данные богаты ссылками на топологию, откуда они поступили. Например, для метрик процессов каждое измерение ссылается на связанные хосты и процессы. Davis AI использует эту топологическую информацию для автоматического обнаружения первопричин и анализа влияния обнаруженных аномалий. То же самое применимо ко всем метрическим событиям в вашей среде.

Когда конфигурация обнаружения аномалий формирует событие, Dynatrace автоматически определяет наиболее релевантный объект для связи с событием. При обнаружении нескольких ссылок на объекты автоматически выбирается наиболее релевантная. Например, если метрика со ссылкой как на хост, так и на процесс приводит к событию, событие формируется на процессе.

[Ingestion метрик](/managed/ingest-from/extend-dynatrace/extend-metrics "Узнайте, как расширить наблюдаемость метрик в Dynatrace.") позволяет отправлять все типы измерений метрик, независимо от числа связанных объектов. Существуют следующие сценарии:

### Измерения не связаны ни с каким объектом

Если определить метрическое событие для нетопологической метрики, результирующее событие будет сформировано для самой среды мониторинга, а не для конкретного объекта Smartscape.

Пример: показатели выручки для всех розничных магазинов по географическому региону

```
business.revenue,shop=shop111,city=NewYork 234



business.revenue,shop=shop999,city=Atlanta 499
```

### Измерения связаны с одним объектом

Если определить метрическое событие для измерения, связанного с одним объектом, результирующее событие будет сформировано для этого объекта.

Пример: выполнения пакетных заданий, измеряемые на отслеживаемом хосте, где измерение связано с хостом

```
batchjob.executions,dt.entity.host=HOST-1111111,hostname=hostA,ip=53.43.23.12 23



batchjob.executions,dt.entity.host=HOST-2222222,hostname=hostB,ip=53.43.23.12 23
```

### Измерения связаны с несколькими объектами

Когда для каждого измерения указано несколько объектов, Dynatrace выбирает наиболее подходящий объект для формирования события. В случае хоста и процесса измерение предположительно относится к процессу, а не к хосту, поэтому событие формируется на процессе.

Пример: число запусков пакетных заданий, измеряемое для процесса на отслеживаемом хосте, где измерение связано как с процессом, так и с хостом

```
batchjob.executions,dt.entity.host=HOST-1,dt.entity.process_group_instance=PROCESS-GROUP-INSTANCE-1,hostname=hostA,ip=53.43.23.12 23



batchjob.executions,dt.entity.host=HOST-2222222,dt.entity.process_group_instance=PROCESS-GROUP-INSTANCE-2,hostname=hostB,ip=53.43.23.12 23
```
