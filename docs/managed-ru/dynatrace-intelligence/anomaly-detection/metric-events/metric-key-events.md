# События на основе ключа метрики

# События на основе ключа метрики

* How-to guide
* 4-min read
* Updated on May 22, 2025

События на основе ключа метрики опираются на входящие необработанные измерения одной метрики. Для этого типа события доступна только стратегия мониторинга на основе статического порога. Можно отслеживать все измерения метрики в рамках одной конфигурации (например, можно создать оповещение для 20 000 CPU в одной конфигурации метрического события). В качестве защитной меры Dynatrace ограничивает такие конфигурации лимитом в 200 одновременных оповещений. Можно сузить область действия события до конкретных измерений.

Кроме того, применяется ограничение в **10 000** конфигураций (как на основе ключа метрики, так и на основе селектора метрики) на среду.

## Область действия событий на основе ключа метрики

Многие метрики Dynatrace поставляются несколькими объектами; их число может легко достигать тысяч. Вероятно, вам не нужно, чтобы событие охватывало все эти объекты одновременно. Для сужения области действия события можно задать фильтры на основе правил.

Доступны два типа фильтров:

* **Фильтры по объектам** сужают область на основе указанных критериев (например, имени объекта или тега). Конкретный набор доступных критериев зависит от метрики. При указании нескольких фильтров применяется логика **И**.
* **Фильтры по измерениям** отбирают объекты на основе указанных кортежей (уникальных комбинаций метрика–ключ измерения–значение измерения). При указании нескольких фильтров применяется логика **И**. Например, можно задать фильтр по измерению, который выбирает только пользовательские действия с iOS-устройств для метрического события на основе метрики **Action count**.

## Создание события на основе ключа метрики

1. Перейдите в раздел **Settings** > **Anomaly Detection** > **Metric events** и выберите **Add metric event**.
2. В поле **Summary** укажите краткое, понятное описание события.
3. В разделе **Query definition** настройте запрос метрики:

   1. Выберите тип запроса **Metric key**.
   2. Выберите метрику для вашего метрического события. Можно указать ключ или отображаемое имя метрики. Начните вводить текст, чтобы увидеть список предложений.
   3. Выберите тип агрегации для метрики (при наличии).
   4. Выберите management zone. Для метрического события оцениваются только данные из этой зоны. Оставьте это поле пустым, чтобы использовать все данные, предоставляемые метрикой.
4. Необязательно: в разделе **Advanced query definition** укажите смещение запроса (в минутах).  
   Смещение необходимо для метрик с задержкой; в противном случае метрическое событие может генерировать ложные оповещения.
5. Необязательно: добавьте фильтры по объектам на основе правил.
6. Необязательно: выберите измерения, которые должно учитывать событие.
7. Определите стратегию мониторинга. Для запросов на основе ключа метрики доступны только статические пороги.

   1. Укажите значение порога. Выберите **Use suggested threshold**, чтобы использовать значение, основанное на предыдущих данных.
   2. При наличии выберите единицу измерения порога.
   3. Выберите поведение [оповещения об отсутствующих данных](/managed/dynatrace-intelligence/anomaly-detection/anomaly-detection-configuration#missing-data "Как настроить оповещение об отсутствующих измерениях.").  
      Если оповещение об отсутствующих данных включено, оно объединяется с пороговым условием по логике **ИЛИ**.
   4. Выберите условие оповещения: оповещать, если метрика выше или ниже порога.
   5. Необязательно: в разделе **Advanced model properties** укажите скользящее окно для сравнения.  
      Скользящее окно определяет, как часто порог (автоматически рассчитанный или заданный вручную) должен нарушаться в рамках скользящего окна времени для формирования события (нарушения не обязаны быть последовательными). Это помогает избежать чрезмерно агрессивных оповещений при единичных нарушениях. Можно задать скользящее окно до 60 минут.
8. Проверьте предварительный просмотр оповещения и оцените эффективность вашей конфигурации.

   1. Выберите значения измерений для отображения в предварительном просмотре.
   2. Выберите временной диапазон предварительного просмотра. Можно получить оповещения за один, три или семь дней.
9. Укажите **заголовок** события. Заголовок должен быть кратким и понятным описанием ситуации, например `High network activity` или `CPU saturation`.
10. В разделе **Description** создайте информативное сообщение события. Сообщения о событиях помогают понять природу события. Можно использовать следующие заполнители:

    * `{alert_condition}` — условие оповещения (выше/ниже порога).
    * `{baseline}` — нарушенное значение базиса.
    * `{dims}` — список всех измерений (и их значений) метрики, нарушившей порог. Можно также указать конкретное измерение: `{dims:dt.entity.<entity>}`. Для получения списка доступных измерений метрики выполните запрос [GET metric descriptor](/managed/dynatrace-api/environment-api/metric-v2/get-descriptor "Просмотрите дескриптор метрики через Metrics v2 API.").
    * `{entityname}` — имя затронутого объекта.
    * `{metricname}` — имя метрики, нарушившей порог.
    * `{missing_data_samples}` — число выборок с отсутствующими данными. Доступно только если включено оповещение об отсутствующих данных.

      `{missing_data_samples}` в описании события

      Рекомендуется включать заполнитель `{missing_data_samples}` в описание события, чтобы понять, вызвана ли проблема отсутствием данных или нарушениями порога.
    * `{severity}` — серьёзность события.
    * `{threshold}` — нарушенное значение порога.
11. Выберите тип для формируемых событий.
12. Определите стратегию **объединения** для формируемых событий.  
    Если объединение разрешено, Davis® AI попытается объединить это событие с существующими проблемами; в противном случае каждый раз формируется новая проблема.
13. Необязательно: задайте дополнительные свойства в виде пар «ключ — значение», прикрепляемых к событию.
14. Нажмите **Save changes**.
