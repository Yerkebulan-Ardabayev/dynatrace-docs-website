# События на основе селектора метрики

* How-to guide
* 4-min read
* Updated on May 22, 2025

Селектор метрики — мощный инструмент для указания данных, которые необходимо считывать при оценке метрического события. Он предоставляет две основные возможности:

* [Преобразования метрик](/managed/dynatrace-api/environment-api/metric-v2/metric-selector "Настройте селектор метрики для Metric v2 API.") для преобразования считываемых данных.
* [Выражения метрик](/managed/dynatrace-api/environment-api/metric-v2/metric-expressions "Используйте выражения метрик для применения арифметических операций в запросах к точкам данных через Metrics API v2.") для объединения одной или нескольких метрик в другой результат с помощью простой математики.

С помощью селектора метрики Davis может получать доступ к историческим данным метрики и изучать нормальное поведение вашей среды, что позволяет использовать автоадаптивные пороги в метрическом событии. Однако применяются некоторые ограничения:

* **100 000** отслеживаемых измерений метрик на среду
* **10 000** конфигураций метрических событий (как на основе ключа метрики, так и на основе селектора метрики) на среду
* **1 000** отслеживаемых измерений на конфигурацию метрического события (статический или автоадаптивный порог)
* **500** отслеживаемых измерений на конфигурацию метрического события (сезонный базис)
* **100** селекторов метрик на стратегию мониторинга. Можно иметь 100 конфигураций с автоадаптивным порогом и 100 со статическим порогом.

## Область действия событий на основе селектора метрики

Сам селектор определяет область действия события на основе селектора метрики. Важно понимать последствия настройки селектора, охватывающего измерения от тысяч отдельных источников. Dynatrace применяет защитные ограничения к обнаружению аномалий в части числа измерений метрик, которые можно отслеживать в одной среде мониторинга, во избежание операционных проблем. Чтобы узнать, как сузить область действия конфигурации, см. раздел [**Преобразование Filter**](/managed/dynatrace-api/environment-api/metric-v2/metric-selector#filter "Настройте селектор метрики для Metric v2 API.").

## Объединение метрик

С помощью возможностей выражений метрик можно реализовать оповещения с перспективой сверху вниз, а не оповещать о каждом компоненте по отдельности.

Например, можно отслеживать паттерны логов на нескольких хостах. Вычисляя суммарное количество обнаруженных паттернов логов по всем релевантным лог-файлам, Dynatrace может обнаруживать аномалии паттернов в агрегированном потоке логов, а не в отдельных счётчиках на лог-файл.
Если по многим объектам встречаются редкие значения (например, счётчик ошибок по нескольким процессам одного типа), агрегированное обнаружение аномалий сверху вниз значительно более устойчиво к ложноположительным оповещениям, чем обнаружение по отдельному счётчику ошибок на процесс.

## Создание события на основе селектора метрики

1. Перейдите в раздел **Settings** > **Anomaly Detection** > **Metric events** и выберите **Add metric event**.
2. В поле **Summary** укажите краткое, понятное описание события.
3. В разделе **Query definition** настройте запрос метрики:

   1. Выберите тип запроса **Metric selector**.
   2. Укажите необходимый селектор метрики.
4. Выберите management zone. Для метрического события оцениваются только данные из этой зоны. Оставьте это поле пустым, чтобы использовать все данные, запрашиваемые селектором метрики.
5. Необязательно: в разделе **Advanced query definition** укажите смещение запроса (в минутах).  
   Смещение необходимо для метрик с задержкой; в противном случае метрическое событие может генерировать ложные оповещения.
6. Определите стратегию мониторинга

   1. Выберите тип модели:

      * Автоадаптивный порог — Dynatrace рассчитывает порог автоматически и динамически адаптирует его к поведению метрики.
      * Статический порог — порог, не изменяющийся со временем.
      * Сезонный базис — Dynatrace создаёт доверительный диапазон для метрики с сезонными закономерностями.
   2. Для статического порога укажите его значение. Выберите **Use suggested threshold**, чтобы использовать значение, основанное на предыдущих данных.
   3. Выберите поведение [оповещения об отсутствующих данных](/managed/dynatrace-intelligence/anomaly-detection/anomaly-detection-configuration#missing-data "Как настроить оповещение об отсутствующих измерениях.").  
      Если оповещение об отсутствующих данных включено, оно объединяется с пороговым условием по логике **ИЛИ**.
   4. Выберите условие оповещения: оповещать, если метрика выше, ниже или за пределами порога.
   5. Необязательно: в разделе **Advanced model properties** укажите [скользящее окно](/managed/dynatrace-intelligence/anomaly-detection/anomaly-detection-configuration#sliding-window "Как настроить оповещение об отсутствующих измерениях.") для сравнения.  
      Скользящее окно определяет, как часто порог (автоматически рассчитанный или заданный вручную) должен нарушаться в рамках скользящего окна времени для формирования события (нарушения не обязаны быть последовательными). Это помогает избежать чрезмерно агрессивных оповещений при единичных нарушениях. Можно задать скользящее окно до 60 минут.
7. Проверьте предварительный просмотр оповещения и оцените эффективность вашей конфигурации.

   1. Выберите значения измерений для отображения в предварительном просмотре.
   2. Выберите временной диапазон предварительного просмотра. Можно получить оповещения за один, три или семь дней.
8. Укажите **заголовок** события. Заголовок должен быть кратким и понятным описанием ситуации, например `High network activity` или `CPU saturation`.
9. В разделе **Description** создайте информативное сообщение события. Сообщения о событиях помогают понять природу события. Можно использовать следующие заполнители:

   * `{alert_condition}` — условие оповещения (выше/ниже порога).
   * `{baseline}` — нарушенное значение базиса.
   * `{dims}` — список всех измерений (и их значений) метрики, нарушившей порог. Можно также указать конкретное измерение: `{dims:dt.entity.<entity>}`. Для получения списка доступных измерений метрики выполните запрос [GET metric descriptor](/managed/dynatrace-api/environment-api/metric-v2/get-descriptor "Просмотрите дескриптор метрики через Metrics v2 API.").
   * `{entityname}` — имя затронутого объекта.
   * `{metricname}` — имя метрики, нарушившей порог.
   * `{missing_data_samples}` — число выборок с отсутствующими данными. Доступно только если включено оповещение об отсутствующих данных.

     `{missing_data_samples}` в описании события

     Рекомендуется включать заполнитель `{missing_data_samples}` в описание события, чтобы понять, вызвана ли проблема отсутствием данных или нарушениями порога.
   * `{severity}` — серьёзность события.
   * `{threshold}` — нарушенное значение порога.
10. Выберите **тип события** для формируемых событий.
11. Включите или отключите **Allow merge** для определения стратегии объединения формируемых событий.  
    Если **Allow merge** включён, Davis AI попытается объединить это событие с существующими проблемами; если отключён, каждый раз формируется новая проблема.
12. Необязательно: задайте дополнительные свойства в виде пар «ключ — значение», прикрепляемых к событию.
13. Нажмите **Save changes**.
