# Настройка чувствительности обнаружения аномалий для сервисов

# Настройка чувствительности обнаружения аномалий для сервисов

* How-to guide
* 6-min read
* Updated on Jan 28, 2026

Dynatrace обнаруживает четыре типа аномалий для сервисов: **деградация времени отклика**, **рост частоты ошибок**, **падение нагрузки на сервис** и **всплески нагрузки на сервис**. Каждый тип аномалии обнаруживается независимо и формирует собственные проблемы и оповещения.

Чтобы настроить глобальную конфигурацию обнаружения аномалий для сервисов

1. Перейдите в раздел **Settings**.
2. Выберите **Anomaly detection** > **Services**. Здесь можно настроить обнаружение для каждого типа аномалий.

## Деградация времени отклика

Этот тип обнаружения аномалий отслеживает время отклика ваших сервисов и формирует оповещение, если метрика нарушает заданные пороги. Dynatrace может обнаруживать деградацию на основе автоматического построения базиса (относительный порог) или фиксированных порогов (абсолютный порог).

Dynatrace оценивает деградацию по двум категориям:

* Все ответы: оповещение формируется, если медианное время отклика всех запросов деградирует за пределы обоих порогов — абсолютного и относительного.
* 10% наиболее медленных ответов: оповещение формируется, если время отклика 10% наиболее медленных запросов деградирует за пределы обоих порогов.

Чтобы настроить обнаружение деградации времени отклика

Автоматическое построение базиса

Фиксированные пороги

1. Включите **Detect response time degradation** и выберите **automatically** из списка.
2. Задайте значения деградации в остальных полях. Нарушение **любого** критерия формирует оповещение.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого сервис считается малонагруженным. Сервисы с меньшей нагрузкой исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго сервис должен находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

1. Включите **Detect response time degradation** и выберите **using fixed thresholds** из списка.
2. Задайте значения деградации в остальных полях. Нарушение **любого** критерия формирует оповещение.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого сервис считается малонагруженным. Сервисы с меньшей нагрузкой исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго сервис должен находиться в аномальном состоянии, прежде чем будет сформировано оповещение.
5. В списке **Sensitivity** выберите чувствительность порога:

   * Низкая: используется высокая статистическая достоверность — кратковременные нарушения (например, из-за всплеска нагрузки) не формируют оповещений.
   * Средняя: используется умеренная статистическая достоверность — оповещение не формируется при каждом единичном нарушении.
   * Высокая: статистическая достоверность не применяется. Каждое нарушение формирует оповещение.

При фиксированных порогах в описание проблемы включается значение фиксированного порога и величина его превышения.

## Рост частоты ошибок

Этот тип обнаружения аномалий отслеживает частоту ошибок ваших сервисов и формирует оповещение, если частота превышает заданные пороги. Dynatrace может обнаруживать рост частоты ошибок на основе автоматического построения базиса или фиксированных порогов.

Чтобы настроить обнаружение роста частоты ошибок

Автоматическое построение базиса

Фиксированные пороги

1. Включите **Detect increase in failure rate** и выберите **automatically** из списка.
2. Укажите значения **relative %** и **absolute %**, при превышении которых должны отправляться оповещения. Для формирования оповещения должны быть нарушены **оба** порога.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого сервис считается малонагруженным. Сервисы с меньшей нагрузкой исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго сервис должен находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

1. Включите **Detect increase in failure rate** и выберите **using fixed thresholds** из списка.
2. Укажите значение **absolute %**, при превышении которого должны отправляться оповещения.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого сервис считается малонагруженным. Сервисы с меньшей нагрузкой исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго сервис должен находиться в аномальном состоянии, прежде чем будет сформировано оповещение.
5. В списке **Sensitivity** выберите чувствительность порога:

   * Низкая: используется высокая статистическая достоверность — кратковременные нарушения (например, из-за всплеска нагрузки) не формируют оповещений.
   * Средняя: используется умеренная статистическая достоверность — оповещение не формируется при каждом единичном нарушении.
   * Высокая: статистическая достоверность не применяется. Каждое нарушение формирует оповещение.

При фиксированных порогах в описание проблемы включается значение фиксированного порога и величина его превышения.

## Падение нагрузки на сервис

Этот тип обнаружения аномалий изучает нормальное поведение нагрузки вашего сервиса в течение 7 дней и формирует оповещение при значительном падении нагрузки.

Чтобы настроить обнаружение падения нагрузки на сервис

1. Включите **Detect service load drops**.
2. Укажите наблюдаемый порог нагрузки для получения оповещений при падении нагрузки.
3. Необязательно: чтобы избежать случайных оповещений, задайте, как долго сервис должен находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

## Всплески нагрузки на сервис

Этот тип обнаружения аномалий изучает нормальное поведение нагрузки вашего сервиса в течение 7 дней и формирует оповещение при значительном росте нагрузки.

Чтобы настроить обнаружение всплесков нагрузки на сервис

1. Включите **Detect service load spikes**.
2. Укажите наблюдаемый порог нагрузки для получения оповещений при всплесках нагрузки.
3. Необязательно: чтобы избежать случайных оповещений, задайте, как долго сервис должен находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

## Период эталона

Davis автоматически формирует базисы в течение недавнего периода эталона. Период эталона по умолчанию — последние **7 дней**.

Если данные мониторинга, собранные в течение периода эталона, более не актуальны — например, если вы развернули новую версию приложения с существенными изменениями и получаете большое количество оповещений — нажмите **Reset**, чтобы установить новый базис. Davis удалит предыдущий период эталона и немедленно начнёт сбор данных для нового.

## Пороги для конкретного сервиса

В качестве альтернативы глобальному определению порогов для всей среды можно задать детально настроенные пороги для отдельных сервисов. Пороги на уровне сервиса переопределяют глобальные пороги для данного сервиса, тогда как для остальных сервисов продолжают действовать глобальные настройки. Вернуться к глобально определённым порогам можно в любой момент.

Чтобы изменить настройки порогов для конкретного сервиса

1. Перейдите в раздел **Services**.
2. Выберите сервис, который необходимо настроить.
3. В контекстном меню (**…**) выберите **Settings**.
4. Выберите **Anomaly detection**.
5. Отключите **Use global anomaly detection settings**.
6. Задайте пороги на уровне сервиса так же, как описано выше для глобальных настроек.

## Пороги для конкретного веб-запроса

Понимая базисную производительность отдельных запросов к сервису, Davis может интеллектуально анализировать сильно различающееся время отклика нескольких запросов в рамках одного сервиса. Например, запрос `orange.jsf` может иметь медианное время отклика `200 мс`, тогда как запрос `orange-booking-payment.jsf` в том же сервисе может быть быстрее — с медианным временем отклика `25 мс`. Davis понимает такие различия и формирует оповещение, если время отклика любого из запросов начинает превышать установленные базисы.

Тем не менее существуют сценарии, в которых параметризация алгоритмов автоматического построения базиса на уровне запроса может быть полезной:

* установить более низкие пороги времени отклика для конкретных, критически важных для бизнеса запросов;
* установить более высокие пороги времени отклика для нестабильных, не критических для бизнеса запросов;
* задать фиксированный, а не относительный порог для запросов с определённым SLA;
* задать фиксированный, а не относительный порог для запросов с определённым таймаутом;
* отключить оповещения для конкретных, не критических для бизнеса запросов.

Все эти сценарии также можно реализовать, задав пороги на уровне запроса для [ключевого веб-запроса](/managed/observe/application-observability/services-classic/monitor-key-requests "Узнайте, как вести тщательный мониторинг запросов, критически важных для вашего бизнеса."). Чтобы настроить порог для ключевого запроса

1. Перейдите в раздел **Services**.
2. Выберите сервис, который необходимо настроить.
3. В контекстном меню (**…**) выберите **Settings**.
4. Выберите **Anomaly detection**.
5. Прокрутите страницу до раздела **Set thresholds on key requests** и разверните меню нужного ключевого запроса.
6. Отключите **Use service or global settings**.
7. Задайте пороги на уровне запроса так же, как описано выше для глобальных настроек.
