# Настройка чувствительности обнаружения аномалий для приложений

# Настройка чувствительности обнаружения аномалий для приложений

* How-to guide
* 5-min read
* Published Sep 06, 2021

Dynatrace обнаруживает четыре типа аномалий для приложений: **деградация ключевых показателей производительности**, **падение трафика**, **всплески трафика** и **рост частоты ошибок**. Каждый тип аномалии обнаруживается независимо и формирует собственные проблемы и оповещения.

Чтобы настроить глобальную конфигурацию обнаружения аномалий для приложений

1. Перейдите в раздел **Settings** > **Anomaly detection** и выберите **Web applications**, **Mobile apps** или **Custom apps**.

## Деградация ключевых показателей производительности

Этот тип обнаружения аномалий отслеживает [ключевые показатели производительности](/managed/observe/digital-experience/rum-concepts/user-action-metrics#kpm "Узнайте, какие метрики Dynatrace вычисляет для пользовательских действий и что означает каждая из них.") и формирует оповещение, если метрика нарушает заданный порог. Dynatrace может обнаруживать деградацию на основе автоматического построения базиса или фиксированных порогов.

Dynatrace оценивает деградацию по двум категориям — все действия и 10% наиболее медленных — и формирует оповещение, если **любая** метрика нарушает порог.

Чтобы настроить обнаружение деградации ключевых показателей производительности

Автоматическое построение базиса

Фиксированные пороги

1. Включите **Detect key performance metric time degradations** и выберите **automatically** из списка.
2. Задайте значения деградации в остальных полях. Нарушение **любого** критерия формирует оповещение.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого приложение считается малонагруженным. Приложения с меньшим трафиком исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго приложение должно находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

1. Включите **Detect key performance metric time degradations** и выберите **using fixed thresholds** из списка.
2. Задайте значения деградации в остальных полях. Нарушение **любого** критерия формирует оповещение.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого приложение считается малонагруженным. Приложения с меньшим трафиком исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго приложение должно находиться в аномальном состоянии, прежде чем будет сформировано оповещение.
5. В списке **Sensitivity** выберите чувствительность порога:

   * Низкая: используется высокая статистическая достоверность — кратковременные нарушения (например, из-за всплеска нагрузки) не формируют оповещений.
   * Средняя: используется умеренная статистическая достоверность — оповещение не формируется при каждом единичном нарушении.
   * Высокая: статистическая достоверность не применяется. Каждое нарушение формирует оповещение.

При фиксированных порогах в описание проблемы включается значение фиксированного порога и величина его превышения.

## Падение трафика

Этот тип обнаружения аномалий изучает нормальное поведение трафика вашего приложения в течение 7 дней и формирует оповещение при значительном падении трафика.

Чтобы настроить обнаружение падения трафика

1. Включите **Detect traffic drops**.
2. Укажите наблюдаемый порог трафика для получения оповещений при падении трафика.

## Всплески трафика

Этот тип обнаружения аномалий изучает нормальное поведение трафика вашего приложения в течение 7 дней и формирует оповещение при значительном росте трафика.

Чтобы настроить обнаружение всплесков трафика

1. Включите **Detect traffic spikes**.
2. Укажите наблюдаемый порог трафика для получения оповещений при всплесках трафика.

## Рост частоты ошибок

Этот тип обнаружения аномалий отслеживает частоту ошибок ваших приложений и формирует оповещение, если частота превышает заданные пороги. Dynatrace может обнаруживать рост частоты ошибок на основе автоматического построения базиса или фиксированных порогов.

Чтобы настроить обнаружение роста частоты ошибок

Автоматическое построение базиса

Фиксированные пороги

1. Включите **Detect increase in failure rate** и выберите **automatically** из списка.
2. Укажите значения **relative %** и **absolute %**, при превышении которых должны отправляться оповещения. Для формирования оповещения должны быть нарушены **оба** порога.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого приложение считается малонагруженным. Приложения с меньшим трафиком исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго приложение должно находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

1. Включите **Detect increase in failure rate** и выберите **using fixed thresholds** из списка.
2. Укажите значение **absolute %**, при превышении которого должны отправляться оповещения.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого приложение считается малонагруженным. Приложения с меньшим трафиком исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго приложение должно находиться в аномальном состоянии, прежде чем будет сформировано оповещение.
5. В списке **Sensitivity** выберите чувствительность порога:

   * Низкая: используется высокая статистическая достоверность — кратковременные нарушения (например, из-за всплеска нагрузки) не формируют оповещений.
   * Средняя: используется умеренная статистическая достоверность — оповещение не формируется при каждом единичном нарушении.
   * Высокая: статистическая достоверность не применяется. Каждое нарушение формирует оповещение.

При фиксированных порогах в описание проблемы включается значение фиксированного порога и величина его превышения.

## Период эталона

Davis автоматически формирует базисы в течение недавнего периода эталона. Период эталона по умолчанию — последние **7 дней**.

Если данные мониторинга, собранные в течение периода эталона, более не актуальны — например, если вы развернули новую версию приложения с существенными изменениями и получаете большое количество оповещений — нажмите **Reset**, чтобы установить новый базис. Davis удалит предыдущий период эталона и немедленно начнёт сбор данных для нового.

## Пороги для конкретного приложения

В качестве альтернативы глобальному определению порогов для всей среды можно задать детально настроенные пороги для отдельных приложений. Пороги на уровне приложения переопределяют глобальные пороги для данного приложения, тогда как для остальных приложений продолжают действовать глобальные настройки. Вернуться к глобально определённым порогам можно в любой момент.

Чтобы изменить настройки порогов для конкретного приложения

1. Перейдите в раздел **Web**, **Mobile**, **Frontend** или **Custom Applications** в зависимости от типа приложения.
2. Выберите приложение, которое необходимо настроить.
3. В контекстном меню (**…**) выберите **Edit**.
4. Выберите **Anomaly detection**.
5. Отключите **Use global anomaly detection settings**.
6. Задайте пороги на уровне приложения так же, как описано выше для глобальных настроек.
