# Настройка чувствительности обнаружения аномалий для сервисов баз данных

# Настройка чувствительности обнаружения аномалий для сервисов баз данных

* How-to guide
* 5-min read
* Updated on Jan 28, 2026

Dynatrace обнаруживает пять типов аномалий для сервисов баз данных: **деградация времени отклика**, **рост частоты ошибок**, **падение нагрузки на сервис**, **всплески нагрузки на сервис** и **неудачные подключения к базе данных**. Каждый тип аномалии обнаруживается независимо и формирует собственные проблемы и оповещения.

Чтобы настроить глобальную конфигурацию обнаружения аномалий для сервисов баз данных

1. Перейдите в раздел **Settings**.
2. Выберите **Anomaly detection** > **Database services**. Здесь можно настроить обнаружение для каждого типа аномалий.

## Деградация времени отклика

Этот тип обнаружения аномалий отслеживает время отклика ваших баз данных и формирует оповещение, если метрика нарушает заданные пороги. Dynatrace может обнаруживать деградацию на основе автоматического построения базиса или фиксированных порогов.

Dynatrace оценивает деградацию по двум категориям — все ответы и 10% наиболее медленных — и формирует оповещение, если **любое** время отклика нарушает порог.

Чтобы настроить обнаружение деградации времени отклика

Автоматическое построение базиса

Фиксированные пороги

1. Включите **Detect response time degradation** и выберите **automatically** из списка.
2. Задайте значения деградации в остальных полях. Нарушение **любого** критерия формирует оповещение.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого база данных считается малонагруженной. Базы данных с меньшей нагрузкой исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго база данных должна находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

1. Включите **Detect response time degradation** и выберите **using fixed thresholds** из списка.
2. Задайте значения деградации в остальных полях. Нарушение **любого** критерия формирует оповещение.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого база данных считается малонагруженной. Базы данных с меньшей нагрузкой исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго база данных должна находиться в аномальном состоянии, прежде чем будет сформировано оповещение.
5. В списке **Sensitivity** выберите чувствительность порога:

   * Низкая: используется высокая статистическая достоверность — кратковременные нарушения (например, из-за всплеска нагрузки) не формируют оповещений.
   * Средняя: используется умеренная статистическая достоверность — оповещение не формируется при каждом единичном нарушении.
   * Высокая: статистическая достоверность не применяется. Каждое нарушение формирует оповещение.

При фиксированных порогах в описание проблемы включается значение фиксированного порога и величина его превышения.

## Рост частоты ошибок

Этот тип обнаружения аномалий отслеживает частоту ошибок ваших сервисов баз данных и формирует оповещение, если частота превышает заданные пороги. Dynatrace может обнаруживать рост частоты ошибок на основе автоматического построения базиса или фиксированных порогов.

Чтобы настроить обнаружение роста частоты ошибок

Автоматическое построение базиса

Фиксированные пороги

1. Включите **Detect increase in failure rate** и выберите **automatically** из списка.
2. Укажите значения **relative %** и **absolute %**, при превышении которых должны отправляться оповещения. Для формирования оповещения должны быть нарушены **оба** порога.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого база данных считается малонагруженной. Базы данных с меньшей нагрузкой исключаются из оценки.
4. Необязательно: чтобы избежать случайных оповещений, задайте, как долго база данных должна находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

1. Включите **Detect increase in failure rate** и выберите **using fixed thresholds** из списка.
2. Укажите значение **absolute %**, при превышении которого должны отправляться оповещения.
3. Необязательно: чтобы избежать избыточных оповещений, задайте порог **actions/min**, ниже которого база данных считается малонагруженной. Базы данных с меньшей нагрузкой исключаются из оценки.

5. В списке **Sensitivity** выберите чувствительность порога:

   * Средняя: используется умеренная статистическая достоверность — оповещение не формируется при каждом единичном нарушении.
   * Высокая: статистическая достоверность не применяется. Каждое нарушение формирует оповещение.

## Падение нагрузки на сервис

Этот тип обнаружения аномалий изучает нормальное поведение нагрузки вашего сервиса базы данных в течение 7 дней и формирует оповещение при значительном падении нагрузки.

Чтобы настроить обнаружение падения нагрузки на сервис базы данных

1. Включите **Detect service load drops**.
2. Укажите наблюдаемый порог нагрузки для получения оповещений при падении нагрузки.
3. Необязательно: чтобы избежать случайных оповещений, задайте, как долго база данных должна находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

## Всплески нагрузки на сервис

Этот тип обнаружения аномалий изучает нормальное поведение нагрузки вашего сервиса базы данных в течение 7 дней и формирует оповещение при значительном росте нагрузки.

Чтобы настроить обнаружение всплесков нагрузки на сервис базы данных

1. Включите **Detect service load spikes**.
2. Укажите наблюдаемый порог нагрузки для получения оповещений при всплесках нагрузки.
3. Необязательно: чтобы избежать случайных оповещений, задайте, как долго база данных должна находиться в аномальном состоянии, прежде чем будет сформировано оповещение.

## Период эталона

Davis автоматически формирует базисы в течение недавнего периода эталона. Период эталона по умолчанию — последние **7 дней**.

Если данные мониторинга, собранные в течение периода эталона, более не актуальны — например, если вы развернули новую версию приложения с существенными изменениями и получаете большое количество оповещений — нажмите **Reset**, чтобы установить новый базис. Davis удалит предыдущий период эталона и немедленно начнёт сбор данных для нового.

## Пороги для конкретного сервиса базы данных

В качестве альтернативы глобальному определению порогов для всей среды можно задать детально настроенные пороги для отдельных сервисов баз данных. Пороги на уровне базы данных переопределяют глобальные пороги для данного сервиса базы данных, тогда как для остальных баз данных продолжают действовать глобальные настройки. Вернуться к глобально определённым порогам можно в любой момент.

Чтобы изменить настройки порогов для конкретного сервиса базы данных

1. Перейдите в раздел **Database Services**.
2. Выберите базу данных, которую необходимо настроить.
3. В контекстном меню (**…**) выберите **Edit**.
4. Выберите **Anomaly detection**.
5. Отключите **Use global anomaly detection settings**.
6. Задайте пороги на уровне базы данных так же, как описано выше для глобальных настроек.

## Пороги для конкретного запроса к базе данных

Davis может интеллектуально анализировать запросы с сильно различающимися характеристиками в рамках одной базы данных. Некоторые запросы, однако, могут требовать особого внимания в силу их важности для критически важных бизнес-функций. Это достигается с помощью ключевых запросов.

Чтобы настроить пороги для ключевого запроса

1. Перейдите в раздел **Database Services**.
2. Выберите базу данных, которую необходимо настроить.
3. В контекстном меню (**…**) выберите **Edit**.
4. Выберите **Anomaly detection**.
5. Прокрутите страницу до раздела **Set thresholds on key requests** и разверните меню нужного ключевого запроса.
6. Отключите **Use service or global settings**.
7. Задайте пороги на уровне запроса так же, как описано выше для глобальных настроек.
