# Статические пороги для обнаружения аномалий

* Explanation
* 2-min read
* Updated on Apr 05, 2024

Статический порог представляет собой жёсткое ограничение, которое метрика не должна нарушать. Поскольку статические пороги не изменяются со временем, они являются важным инструментом мониторинга для определения критических границ нормальной работы.

Важно выбрать между статическим и [адаптивным порогом](/managed/dynatrace-intelligence/anomaly-detection/auto-adaptive-threshold "Как Dynatrace адаптирует пороги для нескольких объектов в рамках конфигурации обнаружения аномалий.") в зависимости от вашего сценария использования.

Например, статический порог можно использовать для ограничения общего использования памяти известным процессом. В этом случае статический порог превосходит адаптивный, поскольку если потребление памяти медленно растёт со временем, адаптивный порог просто растёт вместе с ним, не формируя проблем и в конечном счёте приводя к скрытой утечке памяти.

На иллюстрациях ниже потребление памяти стабильно растёт на протяжении 30 дней. Статически заданный порог в 40 МБ обнаружит аномальное поведение процесса, тогда как адаптивный порог будет расти вместе со значением метрики.

Помимо значения порога, можно указать, как часто порог должен нарушаться в рамках скользящего временного окна для формирования события (нарушения не обязаны быть последовательными). Это помогает избежать чрезмерно агрессивных оповещений при единичных нарушениях порога. Можно задать скользящее окно до 60 минут.

По умолчанию любые 3 минуты из скользящего окна в 5 минут должны нарушать порог для формирования события. То есть событие потребует 3 нарушающих минуты в любом скользящем окне продолжительностью 5 минут.
