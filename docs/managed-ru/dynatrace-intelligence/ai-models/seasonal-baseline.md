# Сезонный базис

# Сезонный базис

* Explanation
* 5-min read
* Published Sep 28, 2022

Сезонный базис — это динамический подход к базовому мониторингу для систем с выраженными сезонными закономерностями. Пример сезонной закономерности — метрика, возрастающая в рабочие часы и снижающаяся в остальное время. Для такого случая затруднительно настроить конфигурацию оповещений на основе статического или автоадаптивного порога. Если задать фиксированный порог, ориентированный на поведение в рабочие часы, аномалия в нерабочее время останется незамеченной. Davis® AI **изучает сезонное поведение** ваших метрик и автоматически создаёт для него доверительный диапазон.

Если конфигурация метрического события включает несколько объектов, каждый объект получает собственный сезонный базис, и каждый базис оценивается независимо. Например, если область действия события охватывает пять сервисов, Dynatrace вычисляет и оценивает пять независимых базисов.

## Пример базиса

В этом примере рассматривается число пользовательских действий в приложении. Реальные взаимодействия пользователей часто следуют сезонности рабочих часов и выходных дней, и наш временной ряд также демонстрирует ежедневные пики.

При оценке по статическому порогу эти ежедневные пики приводят к ложноположительным оповещениям (красная аннотация на графике).

## Параметры модели базиса

Модель сезонного базиса использует значения метрики за каждую минуту **последних 14 дней** в качестве эталона для расчёта базиса. Модель изучает обычное поведение метрики и стремится обнаруживать регулярно повторяющиеся закономерности. Для вычисления доверительного диапазона применяется **вероятностное прогнозирование**, охватывающее ожидаемый диапазон будущего значения.

Ширину доверительного диапазона можно настроить с помощью параметра **tolerance** (допуск). Более высокий допуск означает более широкий доверительный диапазон, что ведёт к меньшему числу срабатываний событий. Допустимый диапазон допуска — от `0.1` до `10`, значение по умолчанию — `4`.

Как следует из названия, сезонная модель эффективна для данных с сезонными закономерностями (например, ежедневными пиками). Поскольку она потребляет значительные ресурсы и требует больше времени для изучения поведения, предусмотрены проверки корректности для обеспечения оптимальной производительности. [Проверка](#training) оценивает каждый кортеж (уникальные комбинации метрика–измерение–значение измерения). Если проверка не пройдена, вместо сезонной используется простая модель, и в предварительном просмотре отображается соответствующее сообщение.

Простая модель вычисляет доверительный диапазон на основе робастной оценки распределения входных данных, и рассчитанный интервал остаётся постоянным во времени. Для простой модели параметр **tolerance** управляет шириной доверительного диапазона так же, как и для сезонной модели.

Если в метаданных метрики заданы **границы метрики** (минимальное или максимальное значение), они автоматически учитываются в модели. Если границы не заданы, а входные данные содержат только положительные значения, модель автоматически устанавливает минимальную граничную отметку `0`. При наличии минимального или максимального значения доверительный диапазон не выходит за эти границы.

Модель (сезонная или простая) обновляется ежедневно.

Ещё один важный параметр для сезонных базисов — [скользящее окно](/managed/dynatrace-intelligence/anomaly-detection/anomaly-detection-configuration#sliding-window "Как настроить оповещение об отсутствующих измерениях."), используемое для сравнения текущих измерений с доверительным диапазоном. Оно определяет, как часто доверительный диапазон должен нарушаться в рамках скользящего окна времени, чтобы сформировать событие (нарушения не обязаны быть последовательными). Такой подход помогает снизить количество ложноположительных оповещений. Скользящее окно можно задать максимум в 60 минут.

## Обучение модели базиса

Модель сезонного базиса использует значения метрики за каждую минуту последних 14 дней для обучения и обучается на кортеж (уникальные комбинации метрика–измерение–значение измерения). Обучение происходит ежедневно.

На изображении ниже представлен обзор процесса обучения.

![seasonal baseline model overview](https://dt-cdn.net/images/seasonal-baseline-model-training-1187-8bd05de831.png)

seasonal baseline model overview

На первом шаге — **проверке данных** — оценивается целесообразность обучения сезонной модели на данных метрики. Вот некоторые причины перехода на простую модель:

* в данных много пропущенных значений;
* в данных много выбросов;
* в данных недостаточно изменчивости;
* в данных не обнаружена сезонность.

Если эта проверка не пройдена, вместо сезонной используется **простая модель**. В случае успеха выполняется следующий шаг — **предобработка входных данных**. На этом шаге данные подготавливаются для обучения модели квантильной регрессии, которая изучает сезонное поведение на основе трёх ключевых характеристик (признаков) данных:

* **Авторегрессионные признаки** — десять последних исторических точек данных (небольшие пропуски интерполируются).
* **Признаки времени и дня** — определяют день или конкретное время суток, что помогает выявлять закономерности, характерные только для определённого дня.
* **Устойчивые сезонные закономерности** — регулярно повторяющиеся закономерности, извлечённые с помощью преобразования Фурье.

Эти признаки используются для обучения **модели квантильной регрессии** — вероятностной регрессионной модели, устойчивой к выбросам благодаря оценке квантилей.

Следующий шаг — **валидация модели**, которая проверяет надёжность прогнозируемых квантилей на подмножестве данных. Если эта проверка не пройдена, вместо сезонной используется **простая модель**.

По завершении обучения Dynatrace использует сезонную модель для обнаружения аномалий. Каждую минуту модель формирует **одношаговый прогноз вперёд** доверительного диапазона на следующую минуту. После получения фактической точки данных спрогнозированный доверительный интервал сравнивается с фактическим значением для проверки того, находится ли значение в пределах прогнозируемых границ.
