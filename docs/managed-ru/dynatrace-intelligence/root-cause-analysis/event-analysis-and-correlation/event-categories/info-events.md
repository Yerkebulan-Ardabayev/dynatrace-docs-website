На этой странице представлена информация о поддерживаемых информационных событиях и логике их возникновения.

## Аннотация

`CUSTOM_ANNOTATION`

События аннотаций могут отправляться в Dynatrace сторонними инструментами для выделения интересных периодов времени. Существует несколько каналов, которые можно использовать для автоматической передачи аннотаций в Dynatrace, включая [Events API v2](/managed/dynatrace-api/environment-api/events-v2 "Узнайте, что можно сделать с помощью Dynatrace Events API v2.") и плагины мониторинга.

## Развёртывание

`CUSTOM_DEPLOYMENT`

События развёртывания могут отправляться в Dynatrace сторонними инструментами для сообщения о развёртываниях ПО вместе с важными метаданными, связанными с бизнес-процессами, такими как авторизатор, владелец продукта и данные о версии. Существует несколько каналов для автоматической передачи событий развёртывания в Dynatrace, включая Dynatrace REST API и плагины мониторинга.

## Соответствие паттерну лога на хосте или процессе

`LOG_MATCHED`

Можно определить пользовательские правила сопоставления паттернов логов, применяющиеся к лог-файлам как на уровне хоста, так и на уровне процесса. Если пользовательский порог числа совпадений паттернов в минуту нарушается, будет сформировано информационное событие лога, отображаемое на уровне хоста или процесса.

### Применимые объекты Dynatrace

К этому событию применимы все объекты Dynatrace.

## Частота ошибок Elastic load balancer

`ELASTIC_LOAD_BALANCER_HIGH_FAILURE_RATE`

Dynatrace автоматически сообщает о числе неудачных попыток подключения к балансировщику нагрузки AWS. По умолчанию Dynatrace формирует оповещение, если число неудачных попыток подключения превышает 10 в минуту в течение не менее 3 из 5 одноминутных интервалов наблюдения.

### Применимые объекты Dynatrace

К этому событию применимы следующие объекты Dynatrace:

* Elastic load balancer

## Помечен для завершения

`MARKED_FOR_TERMINATION`

По умолчанию Dynatrace формирует проблему, когда отслеживаемый объект становится недоступным. Однако в некоторых случаях может потребоваться намеренно остановить объект (например, отключить мониторинг хоста во время окна обслуживания). Чтобы избежать лишних уведомлений и оповещений о проблемах, назначьте это событие любому объекту, который вы собираетесь остановить. В течение следующих 60 минут это предотвратит формирование Dynatrace проблемы при недоступности объекта.

Параметр **Alert on graceful host shutdown** перекрывает событие `MARKED_FOR_TERMINATION` и формирует оповещение тем не менее.

## Завершение из-за нехватки памяти

Событие «Завершение из-за нехватки памяти» указывает на то, что OOM Killer был запущен для завершения контейнера, поскольку он превысил ограничение памяти контейнера.

* Это может указывать на необходимость изменить ограничение памяти для контейнера или на то, что контейнер использовал неожиданно большой объём памяти.
* Это событие не формирует оповещение, однако можно настроить [пользовательское событие для оповещения](/managed/dynatrace-intelligence/anomaly-detection/metric-events "Узнайте о метрических событиях в Dynatrace.") на основе метрики `Out of memory kills`.
