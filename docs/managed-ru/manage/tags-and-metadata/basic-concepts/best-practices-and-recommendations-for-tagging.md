# Лучшие практики и рекомендации по тегированию

# Лучшие практики и рекомендации по тегированию

* Explanation
* 5-min read
* Published May 03, 2018

Тегирование — мощный механизм. Однако для получения максимальной пользы оно должно применяться осознанно и осмысленно. В помощь предоставлены конкретные рекомендации и лучшие практики, описанные ниже. С [автотегированием на основе метаданных](/managed/manage/tags-and-metadata/setup/how-to-define-tags#automatic "Узнайте, как определять и применять теги вручную и автоматически.") теги могут генерироваться автоматически и назначаться отслеживаемым объектам на основе конкретных значений метаданных, которые Dynatrace определяет автоматически.

В дополнение к правилам автоматического тегирования Dynatrace поддерживает определение тегов через системные переменные окружения и ручное тегирование. Для ручного тегирования удобен запрос [POST tags](/managed/dynatrace-api/environment-api/custom-tags/post-tags "Назначение пользовательских тегов отслеживаемым объектам через Dynatrace API.") Custom tags API, позволяющий назначать тег большому набору объектов.

## Метаданные хоста и тегирование

Для каждого из хостов можно [определять теги](/managed/observe/infrastructure-observability/hosts/configuration/define-tags-and-metadata-for-hosts#edit-the-host-tag-configuration-file "Узнайте, как помечать тегами и задавать дополнительные свойства для отслеживаемого хоста."), а также [дополнительные метаданные](/managed/observe/infrastructure-observability/hosts/configuration/define-tags-and-metadata-for-hosts#edit-the-host-metadata-configuration-file "Узнайте, как помечать тегами и задавать дополнительные свойства для отслеживаемого хоста.").

Рекомендуется определять дополнительные метаданные на уровне развёрнутой системы хоста. Следует передавать как можно больше дополнительных метаданных. Эти данные могут использоваться Dynatrace для определения тегов, зон управления или динамически в диаграммах, дашбордах и других элементах. Также рекомендуется стандартизировать ряд метаданных об организации. Их можно автоматически сохранять в соответствующий конфигурационный файл в процессе развёртывания.

Важные аспекты для стандартизации:

* Владелец / команда / бизнес-единица / направление бизнеса
* Окружение: staging / production
* Версия (если применимо)
* Важность / уровень (актуально для профилей оповещений о проблемах)

Для тегов хостов рекомендуется не определять теги на уровне развёрнутой системы. Причина в том, что это трудоёмко и требует тщательного планирования. Кроме того, такой подход сложно изменять. Прибегайте к нему только в случаях, когда то, что нужно тегировать, меняется редко и необходимо создать глобальный фильтр (возможный пример: важность/уровень). Другой случай — если ваш рабочий процесс требует определять теги во время развёртывания в автоматическом режиме.

## Метаданные приложений и тегирование

Рекомендации для хостов справедливы и для приложений, и для процессов. Как правило, стоит думать о дополнительных и стандартных метаданных, а не о тегах.

Для приложений вроде WebSphere или Tomcat используйте [переменную окружения](/managed/observe/infrastructure-observability/process-groups/configuration/define-your-own-process-group-metadata "Настройте собственные метаданные, связанные с процессами.") `DT_CUSTOM_PROP` для определения метаданных. Эта переменная должна присутствовать при запуске приложения. Для WebSphere это можно сделать в консоли WebSphere в разделе JVM. Для Tomcat и других просто определите её в скрипте запуска.

При работе с Kubernetes используйте [аннотации Kubernetes](/managed/observe/infrastructure-observability/container-platform-monitoring/kubernetes-monitoring/leverage-tags-defined-in-kubernetes-deployments "Организация и фильтрация приложений с помощью меток и аннотаций Kubernetes/OpenShift.") для определения метаданных. Dynatrace автоматически подхватывает все аннотации. То же самое относится к [AWS](/managed/ingest-from/amazon-web-services/integrate-with-aws/cloudwatch-metrics/limit-api-calls-to-aws-using-tags "Добавьте и настройте теги AWS."). Dynatrace также подхватывает теги AWS, но в данном случае — как теги, а не метаданные. Разумеется, их всё равно можно использовать для определения дополнительных тегов или в любых других целях.

Для определения [тегов Dynatrace в Cloud Foundry](/managed/observe/infrastructure-observability/container-platform-monitoring/cloud-foundry-monitoring/organize-cf-deployments-by-tags "Автоматически организуйте и фильтруйте все отслеживаемые приложения с помощью тегов Cloud Foundry.") можно использовать один или несколько экземпляров сервиса Cloud Foundry, в имени которых содержится подстрока `dynatrace`.

## Автоматическое тегирование

Рекомендуется настраивать [правила автоматического тегирования](/managed/manage/tags-and-metadata/setup/how-to-define-tags#automatic "Узнайте, как определять и применять теги вручную и автоматически.") на основе существующих или пользовательских метаданных для определения наборов фильтров для диаграмм, оповещений и других целей. Эти теги и правила можно изменять и адаптировать в любое время; они применяются практически мгновенно без каких-либо изменений в отслеживаемом окружении или приложениях.

## CMDB, API и автоматизация

Если у вас есть CMDB, в которой вы хотите определять и поддерживать теги, это можно делать, используя API для синхронизации этих тегов с Dynatrace. Это означает возможность определять теги и назначать их объектам через API. Данный подход следует использовать для общих тегов, которые уже поддерживаются за пределами Dynatrace.

## Зоны управления

[Зоны управления](/managed/manage/identity-access-management/permission-management/management-zones "Узнайте о концепции зон управления, как их определять и эффективно использовать.") — важная концепция Dynatrace, особенно значимая в крупных окружениях, так как они позволяют организовать работу больших организаций. Зоны управления основаны на той же идее, что и автоматические теги, но разработаны специально для определения групп объектов, которые либо принадлежат вместе, либо имеют общие уровни безопасности.

Рекомендуется определять зоны для всех команд, указывая, что они должны видеть, и для различных направлений бизнеса. Также имеет смысл добавлять разные зоны для разных команд поддержки, чтобы большой объём данных стал управляемым и были введены уровни безопасности. Каждый пользователь может иметь доступ к нескольким зонам. Зоны могут пересекаться, поэтому не нужно беспокоиться о назначении объекта в конкретную зону. Общие объекты могут находиться во многих или всех зонах.

Лучший способ создания зон управления — определить правила на основе метаданных объектов (включая пользовательские метаданные). Однако если вы хотите поддерживать зоны управления за пределами Dynatrace, их также можно строить на основе тегов.

## Проблемы, зоны управления и теги

[Проблемы](/managed/dynatrace-intelligence "Ознакомьтесь с возможностями Davis AI.") также можно фильтровать по тегам. По сути, это означает возможность сосредоточиться на проблемах, связанных с объектами, имеющими определённый тег. Таким же образом интеграции уведомлений могут использовать теги. Можно настроить отправку только тех проблем, которые касаются объектов с определённым тегом, в конкретную [интеграцию уведомлений](/managed/analyze-explore-automate/notifications-and-alerting/problem-notifications "Узнайте, как интегрировать сторонние системы уведомлений о проблемах с Dynatrace.").

## Правила именования и соглашения об именах

Dynatrace автоматически присваивает имена, однако они не всегда позволяют быстро определить, к какому приложению или сервису относится объект. Для решения этой задачи рекомендуется использовать [правила именования сервисов](/managed/observe/application-observability/services/service-detection/service-detection-v1/customize-service-naming "Используйте правила именования для настройки и улучшения автоматического именования сервисов.") и [правила именования групп процессов](/managed/observe/infrastructure-observability/process-groups/configuration/pg-naming "Способы настройки именования групп процессов."). Это можно сделать в Dynatrace на основе метаданных, импортированных из отслеживаемых приложений.

Настоятельно рекомендуется добавлять к именам групп процессов и приложений префикс, связанный с бизнес-единицей (`BU` или `LOB`) или другим обозначением владельца. Такой стандарт именования будет автоматически применяться к новым и существующим объектам и может быть расширен в будущем в соответствии с возникающими потребностями. Это также одна из причин, по которым имеет смысл определять минимальный набор метаданных, которые каждое приложение может предоставить.
