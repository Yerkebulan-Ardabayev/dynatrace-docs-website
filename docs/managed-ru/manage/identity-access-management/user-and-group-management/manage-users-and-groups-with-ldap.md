# Управление пользователями и группами через LDAP

* Published Jul 19, 2017

Кластер Dynatrace Managed можно подключить к каталогу LDAP для аутентификации, управления пользователями и управления группами.

При интеграции с LDAP пользователи из внешнего ресурса LDAP получают доступ к Dynatrace. Пользователей можно назначать в группы Dynatrace вручную или автоматически на основе данных LDAP.

## Начало работы

1. Перейдите в **User authentication** > **User repository**.
2. Выберите **External LDAP server** из списка.  
   Откроется страница **LDAP configuration**.

   Пример страницы 'LDAP configuration'

   ![Настройка LDAP](https://dt-cdn.net/images/live-example-obfuscated-1218-f4bcd0eb01.png "Настройка LDAP")

   После переключения на аутентификацию LDAP

   1. Локальные учётные записи (кроме учётной записи администратора) перестанут работать: вход с локальной учётной записью станет невозможен.
   2. Учётная запись администратора, созданная при установке, продолжит работать независимо от выбранного провайдера аутентификации.
3. На странице **LDAP configuration** отображается трёхшаговый процесс настройки.

   1. [Настройка подключения](#connection-configuration)
   2. [Настройка запроса групп](#groups-query-configuration)
   3. [Настройка запроса пользователей](#users-query-configuration)

   Подробности настройки см. ниже.

## Настройка подключения

Пример раздела 'Connection configuration'

![Параметры подключения](https://dt-cdn.net/images/1-connection-configuration-278-e88f7c30f4.png "Параметры подключения")

1. Введите **Host address** вашего LDAP-сервера.

   * Для включения шифрованного соединения с LDAP-сервером выберите **Use SSL**. Значение **Port number** изменится автоматически, но при необходимости его можно изменить вручную.
2. При необходимости измените **Port number**:

   * 389 — порт по умолчанию для стандартного LDAP-соединения; убедитесь, что ваш LDAP-сервер действительно использует его.
   * 636 — порт по умолчанию для защищённого соединения LDAPS (LDAP over SSL).
3. Укажите **Bind DN** (Distinguished Name) для учётной записи пользователя LDAP, например в формате:  
   `CN=UserName,OU=OU-name,DC=DomainName,DC=DomainExtension`  
   или любой другой допустимой строки LDAP.

   **Bind DN**, как правило, является системным пользователем (не реальным человеком), используемым для подключения к серверу LDAP. С точки зрения LDAP-сервера это просто пользователь, который читает данные и поэтому не нуждается в доступе на запись, но должен иметь доступ на чтение всех данных, которые будет получать сервер Dynatrace.
4. Введите **Password** пользователя LDAP, указанного в **Bind DN**.
5. Необязательно: если на вашем LDAP-сервере настроены переходы по ссылкам (referral), задайте значение **Maximum referral hops**.
6. Нажмите **Test connection**, чтобы проверить доступность LDAP-сервера из Dynatrace Managed. В ходе теста подключения:

   1. Dynatrace пытается определить тип используемого LDAP-сервера.
   2. На основе предыдущего шага Dynatrace предоставляет настройки по умолчанию для запросов групп и пользователей.

После успешного подключения можно переходить к настройке групп и пользователей.

* Группы необходимо создавать вручную администратором.
* Пользователи создаются автоматически, но только после успешной попытки аутентификации.

## Настройка запроса групп

После успешного теста подключения становится активным шаг **Groups query**. Если LDAP-интеграция нужна только для аутентификации (управление группами и назначение разрешений выполняются в Dynatrace Managed), установите флажок **Assign users to groups automatically based on LDAP query** и переходите к настройке [запроса пользователей](#users-query-configuration). В противном случае следуйте инструкции ниже.

Пример раздела 'Groups query'

![Параметры запроса групп](https://dt-cdn.net/images/2-groups-query-278-94ee3c12f7.png "Параметры запроса групп")

1. Введите строки запроса в соответствующие поля для возврата групп, которые нужно интегрировать с Dynatrace.

   * Каталог LDAP организован в виде дерева. **Base DN** для запроса групп — это запись, содержащая поддерево с вашими группами. На примере ниже есть два поддерева с группами пользователей: `OU=Groups,DC=dynatrace,DC=org` и `OU=Lab,DC=dynatrace,DC=org`:

   ![Пример настройки LDAP](https://dt-cdn.net/images/ldapexample1-275-b06a413d1e.png "Пример настройки LDAP")

   Если нужно назначать пользователей в группы из обоих поддеревьев, укажите Base DN для запроса групп как `DC=dynatrace,DC=org` (родительскую запись). Для назначения только в группы поддерева `OU=Lab,DC=dynatrace,DC=org` укажите это поддерево в качестве Base DN.

   * Можно ввести строку **Filter** для LDAP, чтобы сузить количество возвращаемых групп. Фильтр должен содержать сведения о классе объектов, к которому относятся записи групп. Например, для Active Directory фильтр по умолчанию:

     `(objectClass=group)`

     для OpenLDAP:

     `(objectClass=groupOfNames)`

     Для ограничения числа используемых групп фильтр можно дополнить условиями по имени группы. Например, фильтр `(&(objectClass=group)(CN=PL_*))` ограничивает группы теми, у которых `objectClass` равен `group`, а атрибут `CN` (общее имя) начинается с `PL_`. Можно использовать любой допустимый запрос LDAP. LDAP не чувствителен к регистру.
   * Настройте **Group Id attribute**. Этот атрибут используется только в определённых случаях. Подробнее см. в разделе [Сопоставление пользователей и групп](#matching-users-and-groups) ниже. Если не применимо — установите то же значение, что и **Group name attribute**.
   * Настройте **Group name attribute** — атрибут, содержащий имя группы, обычно называемый `name` (например, для Active Directory) или `cn` (например, для OpenLDAP). Значения **Group name attribute** в вашем каталоге LDAP должны совпадать с именами групп LDAP на странице **User groups**. LDAP не чувствителен к регистру.

     Пример

     Если соответствующая запись группы в каталоге LDAP:

     | Атрибут | Значение |
     | --- | --- |
     | `name` | `My_TestGroup1` |

     Настройте следующее:

     + В Dynatrace, **User authentication** > **User repository** (страница **LDAP configuration**), в шаге **Groups query**, установите **Group name attribute** в значение `name` (имя атрибута).
     + В Dynatrace, **User authentication** > **User groups**, отредактируйте или добавьте группу и добавьте `My_TestGroup1` (значение атрибута) в поле **LDAP groups**.

   Имя группы LDAP на странице **User groups** по умолчанию совпадает с именем, указанным при создании группы.

   * Настройте **Group members attribute**. Этот атрибут подробно рассмотрен в разделе [Сопоставление пользователей и групп](#matching-users-and-groups) ниже.
2. Нажмите **Test query** для проверки настроек и корректности запроса.

## Настройка запроса пользователей

После успешного теста подключения становится активным шаг **Users query**.

Пример раздела 'Users query'

![Параметры запроса пользователей](https://dt-cdn.net/images/3-users-query-278-51fae6437d.png "Параметры запроса пользователей")

1. Введите строки запроса в соответствующие поля для возврата пользователей, которых нужно интегрировать с Dynatrace.

   * Каталог LDAP организован в виде дерева. **Base DN** для запроса пользователей — это запись, содержащая поддерево с вашими пользователями. Например, на изображении ниже есть два поддерева с пользователями:

     `OU=Functional,OU=Accounts,DC=dynatrace,DC=org` и `OU=Primary,OU=Accounts,DC=dynatrace,DC=org`

   ![Пример настройки LDAP](https://dt-cdn.net/images/ldapexample2-286-fa119f53e5.png "Пример настройки LDAP")

   Применительно к дереву из примера:

   * Для аутентификации пользователей из обоих поддеревьев (`OU=Functional` и `OU=Primary`) установите Base DN равным родительской записи этих поддеревьев:  
     `OU=Accounts,DC=dynatrace,DC=org`
   * Для аутентификации только из поддерева `OU=Primary` в `OU=Accounts` установите Base DN:  
     `OU=Primary,OU=Accounts,DC=dynatrace,DC=org`
   * Для дополнительного ограничения системных пользователей до поддерева `OU=EU` в `OU=Primary` установите Base DN:  
     `OU=EU,OU=Primary,OU=Accounts,DC=dynatrace,DC=org`
   * Можно ввести строку **Filter** для сужения числа возвращаемых пользователей. Фильтр должен содержать сведения о классе объектов. Например, для Active Directory и OpenLDAP фильтр по умолчанию:

     `(objectClass=person)`

     Для ограничения аутентифицируемых пользователей фильтр можно дополнить любым допустимым запросом LDAP. Например, фильтр  
     `(&(objectClass=user)(|(department=101)(department=102)(department=103)))`  
     ограничивает пользователей теми, у которых `objectClass` равен `user`, а атрибут `department` имеет одно из указанных значений.  
     LDAP не чувствителен к регистру.
   * Настройте **Login attribute** — атрибут для входа в систему.
   * При необходимости настройте **First name attribute**, **Last name attribute** и **Email attribute**.
   * Настройте **Group membership attribute**. Этот атрибут подробно рассмотрен в разделе [Сопоставление пользователей и групп](#matching-users-and-groups) ниже.
2. Нажмите **Test query** для проверки настроек и корректности запроса.

Тестовые запросы (для групп и пользователей) проверяют только корректность Base DN, фильтров и обязательных атрибутов — атрибута имени группы для групп и атрибута входа для пользователей.

* Тестовые запросы не выдают ошибок при неправильной настройке необязательных атрибутов.
* Тестовые запросы не проверяют правильность назначения пользователей в группы.

## Сопоставление пользователей и групп

Существует несколько способов сопоставления пользователей с группами в LDAP-серверах. Ознакомьтесь с примерами и выберите подходящий.

### Пример сопоставления 1

Если **Group members attribute** (например, `member` или `uniqueMember`) в записи группы LDAP содержит DN пользователя:

* Настройка **Group Id attribute** не требуется. Можно установить **Group Id attribute** равным **Group name attribute**.

### Пример сопоставления 2

Если **Group membership attribute** (например, `memberOf` или `isMemberOf`) в записи пользователя содержит DN группы:

* Настройка **Group Id attribute** также не требуется, поскольку для сопоставления пользователя с группой используется DN группы. Можно установить **Group Id attribute** равным **Group name attribute**.

### Пример сопоставления 3

Если **Group membership attribute** (например, `gid` или `group`) в записи пользователя содержит идентификатор группы:

* **Group Id attribute** должен быть настроен на атрибут, хранящий ссылочное значение.

**Пример записи группы в каталоге LDAP:**

| Атрибут | Значение |
| --- | --- |
| `gidNumber` | `6380` |

**Пример записи пользователя в каталоге LDAP:**

| Атрибут | Значение |
| --- | --- |
| `gid` | `6380` |

В этом случае:

* **Group Id attribute** (в **Groups query**) должен быть настроен на `gidNumber`
* **Group membership attribute** (в **Users query**) должен быть настроен на `gid`

## Сопоставление групп Dynatrace Managed с группами LDAP

Сведения о разрешениях групп пользователей, доступных в Dynatrace Managed, см. в разделе [Группы пользователей и разрешения](/managed/manage/identity-access-management/user-and-group-management/user-groups-and-permissions "Узнайте о поддерживаемых разрешениях и политиках, о том, как назначать их группам и управлять пользователями и группами.").

После успешной настройки групп и пользователей из LDAP необходимо назначить роли окружений мониторинга группам из вашего каталога пользователей. По умолчанию импортируемые группы не имеют разрешений на доступ к окружениям.

Пользователи не смогут получить доступ к окружению мониторинга до выполнения этого шага.

1. Выберите **User authentication** > **User groups**.
2. Из списка групп, импортированных из LDAP, выберите имена групп для настройки.
3. Можно назначить права администратора кластера определённой группе, включив параметр **Grant global administrator permissions to this group**. Все учётные записи пользователей этой группы получат права администратора.
4. В разделе **Permissions** вручную установите разрешения для каждого окружения.

## Ограничения

Dynatrace не поддерживает несколько доменов для аутентификации пользователей LDAP.
