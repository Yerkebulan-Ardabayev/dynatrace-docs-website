# Группы пользователей, разрешения и политики

* Published Jul 17, 2018

Для предоставления доступа к окружению мониторинга или серверу Dynatrace необходимо настроить группы пользователей в Dynatrace Managed.

## Управление группами и пользователями

При установке Dynatrace Managed создаётся учётная запись администратора по умолчанию. Она существует независимо от выбранного типа аутентификации (внутренняя база данных или LDAP). Учётная запись администратора по умолчанию имеет разрешения уровня кластера.

Управлять пользователями и группами можно через Cluster Management Console.

### Создание группы

1. В Cluster Management Console выберите **User authentication** > **User groups**.
2. Нажмите **Add new group**.
3. Назначьте разрешения новой группе.

   * Чтобы назначить административные разрешения группе, установите параметр **Grant global administrator permissions to this group** в положение **On**. Группа получит права доступа ко всем окружениям.
   * Для назначения индивидуальных прав доступа для каждого окружения введите разделённый запятыми список имён групп LDAP, которые должны быть сопоставлены с этой группой пользователей.

### Создание пользователя

1. В Cluster Management Console выберите **User authentication > User accounts**.
2. Нажмите **Add new user**.

   Этот вариант доступен только при использовании внутренней базы данных, а не LDAP.

### Добавление пользователя в группы

1. В Cluster Management Console выберите **User authentication > User accounts**.
2. Выберите пользователя.
3. Нажмите **Add** в разделе **Add group assignments**.

   Группу нельзя назначить, если для неё не указаны разрешения.

## Разрешения

Группам можно назначать предопределённый набор разрешений. После создания группы в неё можно добавлять пользователей. Пользователи могут входить в несколько групп и наследовать разрешения всех групп, в которые они входят. Группы можно изменять или создавать новые по мере необходимости.

### Разрешения кластера

Пользователи, назначенные в группы с этим разрешением, автоматически получают права администратора для всех окружений. Они имеют доступ к Cluster Management Console и могут управлять окружениями мониторинга и сервером Dynatrace. Такие пользователи также могут:

* Добавлять новые узлы сервера Dynatrace
* Обновлять сервер Dynatrace
* Управлять пользователями и группами пользователей Dynatrace Managed
* Устанавливать Dynatrace OneAgent в любое окружение мониторинга
* Настраивать параметры мониторинга для любого окружения

### Разрешения окружения

Для доступа к разрешениям окружения

1. В Cluster Management Console перейдите в **Environments** и выберите нужное окружение.
2. В разделе **Assign environment permissions to user groups** можно добавлять или удалять разрешения окружения для групп пользователей.

Dynatrace предоставляет следующие разрешения уровня окружения. Выберите все применимые:

* **View environment:** разрешает доступ к окружению только для чтения. Одного этого разрешения недостаточно для изменения настроек или установки OneAgent.

  Разрешение **View environment** необходимо для любого другого разрешения уровня окружения, поэтому при выборе любого другого разрешения уровня окружения оно выбирается автоматически.

* **Change monitoring settings:** разрешает изменение всех настроек окружения. Для установки OneAgent необходимо предоставить разрешение **Download & install OneAgent**.
* **Download & install OneAgent:** разрешает загрузку OneAgent и его установку на хостах. Для изменения/редактирования настроек необходимо предоставить разрешение **Change monitoring settings**.

* **View logs:** разрешает доступ к конфиденциальным данным файлов журналов на вкладке **Logs**.

  Роль **View logs** обходит существующий условный доступ, определённый в границах политик через [разрешение storage logs](/managed/manage/identity-access-management/permission-management/manage-user-permissions-policies/advanced/iam-policystatements#storage-logs-read "Полный справочник политик IAM и соответствующих условий для всех сервисов Dynatrace."). Чтобы соблюдать эти условия, рекомендуется выразить те же разрешения **View logs** в операторе политики, используя эквивалентное [разрешение просмотра журналов окружения](/managed/manage/identity-access-management/permission-management/manage-user-permissions-policies/advanced/iam-policystatements#environment-roles-logviewer "Полный справочник политик IAM и соответствующих условий для всех сервисов Dynatrace.").

* **View sensitive request data:** разрешает просмотр потенциальных [персональных данных, собираемых Dynatrace](/managed/manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace "Узнайте, какие типы данных конечных пользователей могут собираться при мониторинге Dynatrace."), включая загрузку дампов памяти. Пользователи без этого разрешения видят, что точка данных существует, но персональные данные маскируются звёздочками (`*****`). Также позволяет вручную запускать [дампы памяти](/managed/observe/application-observability/profiling-and-optimization/memory-dump-analysis "Узнайте, как Dynatrace позволяет запускать, загружать и анализировать дампы памяти для Java и Node.js.").

* **Configure request capture data:** разрешает настройку правил захвата атрибутов запросов. Их можно использовать для захвата таких элементов, как HTTP-заголовки или параметры POST, для хранения, фильтрации и поиска. При установке этого разрешения **Change monitoring settings** включается автоматически.

* **Replay session data with masking:** разрешает воспроизведение [записанных пользовательских сессий](/managed/observe/digital-experience/session-replay "Узнайте, как использовать Session Replay для понимания и устранения ошибок.") с применением [правил маскирования воспроизведения](/managed/observe/digital-experience/session-replay/configure-session-replay-web#masking-preset-options "Настройка параметров Session Replay.") во время воспроизведения. Данные, маскированные при записи, никогда не захватываются и всегда маскируются при воспроизведении.
* **Replay session data without masking:** разрешает воспроизведение записанных пользовательских сессий без применения правил маскирования. Данные, маскированные при записи, всегда маскируются при воспроизведении.

* **Manage security problems:** разрешает просмотр уязвимостей, выявленных Dynatrace Application Security, и управление ими.
* **View security problems:** разрешает просмотр (но не управление) уязвимостей, выявленных Dynatrace Application Security.

Подробные сведения о разрешениях Application Security см. в разделе [Точная настройка разрешений](/managed/secure/application-security#restrict-permissions-managed "Доступ к функциям Dynatrace Application Security.").

* **Manage support tickets:** разрешает доступ ко всем обращениям в поддержку, созданным для данного окружения.

### Разрешения зоны управления

Для доступа к разрешениям зоны управления

1. В Cluster Management Console перейдите в **User authentication** > **User groups** и щёлкните значок карандаша для редактирования нужной группы.
2. В разделе **Management zone permissions** можно добавлять или удалять разрешения для нужной зоны управления.

Dynatrace предоставляет следующие разрешения уровня зоны управления. (Разрешения **Download & install OneAgent** и **Configure request capture data** недоступны, так как не применяются к зонам управления.) Выберите все применимые:

* **View environment:** разрешает доступ только для чтения к объектам в зоне управления. Для изменения/редактирования настроек необходимо предоставить разрешение **Change monitoring settings**.

  Разрешение **View environment** необходимо для любого другого разрешения уровня зоны управления, поэтому при выборе любого другого разрешения оно выбирается автоматически.

* **Manage monitoring settings:** разрешает изменение настроек объектов в зоне управления, например создание или редактирование синтетических мониторов. Также предоставляет доступ к некоторым пунктам глобального меню настроек, но разрешает вносить изменения только для назначенных зон управления.

* **View logs:** разрешает доступ к конфиденциальным данным файлов журналов на вкладке **Logs** для хостов, явно включённых в зону управления. Недостаточно предоставить доступ уровня зоны управления к группам хостов, которым принадлежат эти хосты — подробнее см. в разделе [Правила зон управления](/managed/manage/identity-access-management/permission-management/management-zones/management-zone-rules "Определение правил для ограничения объектов, доступных в зоне управления.").

* **View sensitive request data:** разрешает просмотр потенциальных [персональных данных, собираемых Dynatrace](/managed/manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace "Узнайте, какие типы данных конечных пользователей могут собираться при мониторинге Dynatrace."), для объектов в зоне управления. Пользователи без этого разрешения видят, что точка данных существует, но персональные данные маскируются звёздочками (`*****`) — см. также [**Разрешения окружения**](#environment) выше.

* **Replay session data with masking:** разрешает воспроизведение [записанных пользовательских сессий](/managed/observe/digital-experience/session-replay "Узнайте, как использовать Session Replay.") с применением [правил маскирования воспроизведения](/managed/observe/digital-experience/session-replay/configure-session-replay-web#masking-preset-options "Настройка параметров Session Replay.") во время воспроизведения. Данные, маскированные при записи, всегда маскируются при воспроизведении.
* **Replay session data without masking:** разрешает воспроизведение записанных пользовательских сессий без применения правил маскирования. Данные, маскированные при записи, всегда маскируются при воспроизведении.

  Для работы разрешений Session Replay в зоне управления пользователю также необходим доступ к соответствующим приложениям.

  + Если пользовательская сессия охватывает несколько приложений, не все из которых назначены зоне управления, пользователи могут видеть и воспроизводить сессию. Однако пользовательские действия, связанные с приложением, к которому доступ отсутствует, маскируются.
  + Кнопки воспроизведения недоступны для пользователей с доступом к приложениям, но без разрешения на воспроизведение сессий.

* **Manage security problems:** разрешает просмотр уязвимостей, выявленных Dynatrace Application Security, и управление ими.
* **View security problems:** разрешает просмотр (но не управление) уязвимостей, выявленных Dynatrace Application Security.

Подробные сведения о разрешениях Application Security см. в разделе [Точная настройка разрешений](/managed/secure/application-security#restrict-permissions-managed "Доступ к функциям Dynatrace Application Security.").

### Соотношение разрешений окружения и зоны управления

При предоставлении любого разрешения, кроме **View environment**, на уровне окружения разрешение **View environment** также автоматически включается. Аналогично, при предоставлении любого разрешения, кроме **View environment**, на уровне зоны управления разрешение **View environment** включается для зоны.

[Зоны управления](/managed/manage/identity-access-management/permission-management/management-zones "Концепции зон управления, настройка и оптимальное использование.") предназначены для обеспечения целевого и ограниченного доступа к определённым объектам в окружении. Если необходимо предоставить разрешение пользователям, работающим в зоне управления, рекомендуется использовать [разрешения уровня зоны управления](#mz). Любое разрешение, предоставленное на уровне окружения, заменяет и дополняет разрешения уровня зоны управления. Иными словами, разрешения зоны управления не могут ограничивать разрешения, уже предоставленные на уровне окружения.

Рассмотрим пример зоны управления, содержащей три хоста из пяти в окружении. Если предоставить разрешение **View logs** для зоны управления, наблюдатели увидят вкладку **Logs** с информацией по трём хостам в зоне. Однако если убрать это разрешение на уровне зоны и предоставить его на уровне окружения, пользователи смогут:

* Получить доступ ко **всем** зонам управления через фильтр в строке меню.
* Видеть вкладку **Logs** для всех пяти хостов в окружении при просмотре **всех** зон управления.
* Видеть вкладку **Logs** для трёх хостов в назначенной зоне управления при переключении на неё.

## Политики IAM

Разрешениями пользователей также можно управлять с помощью политик IAM, обеспечивающих более детальный контроль доступа. С помощью политик можно создавать собственные разрешения доступа и назначать их группам пользователей как на уровне кластера, так и на уровне окружения — аналогично обычному механизму разрешений.

Подробнее см. в разделе [Работа с политиками](/managed/manage/identity-access-management/permission-management/manage-user-permissions-policies "Работа с политиками").
