# Правила зон управления

* How-to guide
* 16-min read
* Updated on Oct 03, 2025

Зоны управления состоят из одного или нескольких правил, определяющих и ограничивающих объекты или размерные данные (такие как журналы и метрики), доступные в зоне управления.

При выборе зоны управления в **Settings** > **Preferences** > **Management zones** отображаются все настроенные правила. Для отдельных правил доступны кнопки **Disable/Enable**.

![Правила зоны управления](https://dt-cdn.net/images/mz-rules-1a-1227-bcb209a202.png "Правила зоны управления")

Подробнее:

* О том, как данные журналов могут приниматься и автоматически назначаться зонам управления, см. в разделе [Зоны управления и принятые данные журналов (Logs Classic)](/managed/analyze-explore-automate/log-monitoring/analyze-log-data/management-zones-and-log-monitoring "Узнайте, как принятые данные журналов назначаются зонам управления.").
* О том, как [добавить сервисный уровень в зону управления](/managed/deliver/service-level-objectives-classic/configure-and-monitor-slo#mz "Создание, настройка и мониторинг сервисных уровней в Dynatrace."), чтобы пользователи с доступом к зоне могли просматривать статус SLO и бюджет ошибок на странице **Service-level objectives**.

## Типы правил

Зоны управления предлагают **определение правил на основе интерфейса** для:

* [Отслеживаемых объектов](#ui)
* [Размерных данных (журналы и метрики)](#logs-metrics)

Можно выбрать тип правила, а затем создать условия правила на основе свойств объекта/данных, операторов и значений.

При создании правил для некоторых объектов можно распространить доступ на связанные топологические объекты без создания дополнительного правила. Подробнее см. в разделе [Применение правил зон управления](#rules-apply) ниже.

Для правил на основе интерфейса для размерных данных журналов и метрик можно определить условия на основе имени файла журнала, ключей метрик и ключей/значений измерений. Поддерживаются встроенные, вычисляемые и принятые метрики.

Дополнительная информация о Metrics API v2

Используйте мощный [селектор метрик](/managed/dynatrace-api/environment-api/metric-v2/metric-selector "Настройка селектора метрик для Metric v2 API.") [Metrics API v2](/managed/dynatrace-api/environment-api/metric-v2 "Получение информации о метриках через Metrics v2 API.") для ключей метрик, ключей и значений измерений:

* [GET список всех доступных метрик](/managed/dynatrace-api/environment-api/metric-v2/get-all-metrics "Получение списка всех метрик в вашем окружении мониторинга через Metrics v2 API.").
* [GET сведения о конкретной метрике](/managed/dynatrace-api/environment-api/metric-v2/get-descriptor "Просмотр дескриптора метрики через Metrics v2 API.") для проверки ключей измерений.
* [GET список точек данных](/managed/dynatrace-api/environment-api/metric-v2/get-data-points "Чтение точек данных одной или нескольких метрик через Metrics v2 API.") для проверки значений измерений.

Пользователи автоматически получают доступ к журналам и метрикам, связанным с объектами, включёнными в их назначенные зоны управления.

**[Текстовые правила](#text)** используют мощный [селектор объектов](/managed/dynatrace-api/environment-api/entity-v2/entity-selector "Настройка селектора объектов для конечных точек Environment API.") Environment API v2 для определения объектов. Текстовые правила позволяют определять доступ к объектам на основе всех типов объектов, свойств, значений и связей, предоставляемых [Monitored entities API v2](/managed/dynatrace-api/environment-api/entity-v2 "Изучите Dynatrace Monitored entities API.").

Преимущества текстовых правил:

* Детальные и сфокусированные определения объектов без необходимости просматривать ограниченный набор вариантов в интерфейсе.
* Тогда как правила на основе интерфейса допускают некоторое [распространение доступа к объектам](#rules-apply) на основе связей, в текстовых правилах можно явно использовать связи для фильтрации выбора объектов. Вы сами решаете, какие связи использовать для фильтрации объектов.
* Можно определять текстовые правила для включения пользовательских типов объектов, атрибутов и связей в зоны управления.

Для одного окружения мониторинга можно добавить:

* 5 000 зон управления по умолчанию. По вопросам обращайтесь к специалисту Dynatrace через онлайн-чат.
* 300 правил зон управления на основе интерфейса для объектов (150 для Dynatrace версии 1.323 и более ранних).
* 300 правил зон управления на основе интерфейса для размерных данных (150 для Dynatrace версии 1.323 и более ранних).
* 300 правил на основе селектора объектов для зон управления (150 для Dynatrace версии 1.323 и более ранних).
* 100 000 условий для всех правил зон управления в совокупности (не распространяется на [правила на основе селектора объектов](#text)).
* Любой отдельный объект максимум в 200 зонах управления (выполните вызов API [GET an entity](/managed/dynatrace-api/environment-api/entity-v2/get-entity "Просмотр параметров отслеживаемого объекта через Dynatrace API.") для просмотра зон управления объекта).

Ознакомьтесь с документацией о [том, как оптимизировать производительность правил зон управления при масштабировании](/managed/manage/tags-and-metadata/basic-concepts/best-practice-tagging-at-scale "Оптимизация правил автоматического тегирования и зон управления для ускорения процесса автоматического назначения.").

### Добавление правила на основе интерфейса для объектов

Примеры различных типов правил см. в разделе [Примеры](#examples).

1. [Выберите/создайте зону управления](/managed/manage/identity-access-management/permission-management/management-zones/set-up-management-zones "Создание зон управления и назначение прав доступа к ним.") и нажмите **Add a new rule**.
2. Выберите **Monitored entity** в качестве **Rule type**. (Дополнительную информацию о [правилах для размерных данных](#logs-metrics) и [текстовых правилах](#text) через **Entity selector** см. ниже.)
3. Выберите тип объекта, к которому должно применяться правило (**Rule applies to**), например **Web applications**.

   ![Выбор типа объекта](https://dt-cdn.net/images/mz-rules-entity-type-571-8a758524db.png "Выбор типа объекта")

4. Каждый объект можно определить и ограничить несколькими условиями. Нажмите **Add condition**.
5. Выберите **Property**, **Operator** и **Value** условия (там, где применимо). Например, можно указать, что **Web application name begins with** заданной строкой. Допускается ввод до 80 символов; подстановочные знаки не допускаются; регулярные выражения разрешены в операторах условий **contains regex** и **does not contain regex**.
6. При вводе текстовой строки укажите, следует ли применять **Case sensitive** (учёт регистра).

   ![Условия правила](https://dt-cdn.net/images/mz-rules-3a-958-3a6e99d6a0.png "Условия правила")

7. Для некоторых объектов можно распространить доступ на связанные топологические объекты без создания дополнительного правила. Например, включите соответствующие переключатели для добавления базовых хостов и групп процессов при определении правила для **Services**.

   ![Включение связанных объектов](https://dt-cdn.net/images/mz-rules-4a-879-f4b10ae36b.png "Включение связанных объектов")

8. Нажмите **Add condition** для добавления условия (или **Remove condition** для его удаления) по необходимости.

   * Для каждого правила необходимо определить не менее одного условия.
   * Условия применяются с логикой `AND` — для применения правила к объекту должны выполняться все условия.
   * Количество условий для одного правила не ограничено. Однако суммарный лимит — 100 000 условий для всех правил в окружении.
9. Нажмите **Preview**, чтобы увидеть объекты, соответствующие правилу и бывшие активными за последние 72 часа. При применении зоны управления будут отображаться все объекты, соответствующие правилам за заданный период. Обратите внимание, что **Preview** доступен только для правил на основе объектов.

   ![Предварительный просмотр правила](https://dt-cdn.net/images/mz-rules-5a-957-2b8d1b7d14.png "Предварительный просмотр правила")

10. Нажмите **Save changes**.

### Добавление правила на основе интерфейса для размерных данных

Примеры различных типов правил см. в разделе [Примеры](#examples).

1. [Выберите/создайте зону управления](/managed/manage/identity-access-management/permission-management/management-zones/set-up-management-zones "Создание зон управления и назначение прав доступа к ним.") и нажмите **Add a new rule**.
2. Выберите **Dimensional data** для **Rule type**. (Дополнительную информацию о [правилах для объектов](#ui) и [текстовых правилах](#text) через **Entity selector** см. ниже.)
3. Выберите **Type** данных, к которым должно применяться правило.

   * Для правила на основе интерфейса для встроенной, вычисляемой или принятой метрики выберите **METRIC**.
   * Для правила на основе интерфейса для журналов выберите **LOG**.
4. Каждое правило можно определить и ограничить несколькими условиями. Нажмите **Add condition**.
5. Выберите **Type**, **Key** (при необходимости), **Operator** и **Value** условия. В любом текстовом поле допускается ввод до 80 символов; подстановочные знаки не допускаются.

   Типы условий:

   * Для атрибута журнала или измерения метрики — **DIMENSION**.

     ![Условие атрибута журнала или измерения метрики](https://dt-cdn.net/images/mz-rules-data-dimension-580-3843630e79.png "Условие атрибута журнала или измерения метрики")

   * Для имени файла журнала — **LOG\_FILE\_NAME**. Имя файла журнала должно соответствовать атрибуту `log.source`.

     ![Условие имени файла журнала](https://dt-cdn.net/images/mz-rules-log-filename-580-7f89e2cbd4.png "Условие имени файла журнала")

   * Для ключа метрики — **METRIC\_KEY**.

     ![Условие ключа метрики](https://dt-cdn.net/images/mz-rules-metric-key-580-fac4735560.png "Условие ключа метрики")

   * Для комбинированного условия журнала или метрики — **ANY**.

   Допустимые операторы: **begins with** и **equals**.
6. Нажмите **Add condition** для добавления условия (или **Remove condition** для его удаления) по необходимости.

   * Для каждого правила необходимо определить не менее одного условия.
   * Условия применяются с логикой `AND` — все условия должны выполняться для применения правила.
   * Количество условий для одного правила не ограничено. Однако суммарный лимит — 100 000 условий для всех правил в окружении.
   * **Preview** недоступен для правил размерных данных.
7. Нажмите **Save changes**.

### Добавление текстового правила на основе селектора объектов

1. [Выберите/создайте зону управления](/managed/manage/identity-access-management/permission-management/management-zones/set-up-management-zones "Создание зон управления и назначение прав доступа к ним.") и нажмите **Add a new rule**.
2. В поле **Rule type** выберите **Entity selector**.
3. Для заполнения строки **Entity Selector** может потребоваться выполнение [вызовов Monitored entities API v2](/managed/dynatrace-api/environment-api/entity-v2 "Изучите Dynatrace Monitored entities API.") для получения типов объектов, свойств, значений и связей. Подробнее о составлении определения объекта см. в [документации по селектору объектов](/managed/dynatrace-api/environment-api/entity-v2/entity-selector "Настройка селектора объектов для конечных точек Environment API."). Примеры различных типов правил см. в разделе [Примеры](#examples).

   **Важные части строки селектора объектов**

   * `type(<entityType>)` определяет тип объекта, к которому применяется правило. Например, тип объекта для хостов — `host`, для групп процессов — `process_group`. Тип объекта не чувствителен к регистру. В `<entityType>` можно указать только одну запись.

     Выполните вызов API [GET all entity types](/managed/dynatrace-api/environment-api/entity-v2/get-all-entity-types "Просмотр всех типов отслеживаемых объектов в вашем окружении через Dynatrace API.") для получения списка всех типов объектов в вашем окружении (включая пользовательские) и их свойств.

     Кроме того, можно указать несколько идентификаторов отдельных объектов с помощью критерия `entityId`. Выполните вызов API [GET entities list](/managed/dynatrace-api/environment-api/entity-v2/get-entities-list "Просмотр списка отслеживаемых объектов через Dynatrace API.") для получения списка фактических объектов в вашем окружении и их свойств.
   * Свойства и значения объектов фильтруют список объектов, к которым применяется правило. Например:

     + `entityName.startsWith("Book")` фильтрует объекты, имя которых начинается с `Book`.
     + `serviceType(WEB_REQUEST_SERVICE)` фильтрует сервисы веб-запросов.

     Выполните вызов API [GET entity type](/managed/dynatrace-api/environment-api/entity-v2/get-entity-type "Просмотр сведений о типе отслеживаемого объекта через Dynatrace API.") для любого типа объекта (например, `service`), чтобы получить список всех его свойств (например, `serviceType`). Также можно выполнить вызов API [GET entities list](/managed/dynatrace-api/environment-api/entity-v2/get-entities-list "Просмотр списка отслеживаемых объектов через Dynatrace API.") для получения списка фактических объектов с их значениями свойств (например, `WEB_REQUEST_SERVICE`).
   * Связи дополнительно уточняют выбор объектов, определяя объект через его связь с другим. Связи бывают двух видов.

     + `fromRelationship` означает, что направление связи — **от данного объекта** (то есть запрашиваемого) к связанному. При запросе всех сервисов, которые вызывает сервис A, это связь «от (сервиса A)» к другим сервисам.
     + `toRelationship` означает, что направление связи — от связанного объекта **к данному объекту** (то есть запрашиваемому). При запросе всех сервисов, которые вызывают сервис A, это связь «к (сервису A)».

     Выполните вызов API [GET entity type](/managed/dynatrace-api/environment-api/entity-v2/get-entity-type "Просмотр сведений о типе отслеживаемого объекта через Dynatrace API.") для любого типа объекта, чтобы получить список его связей from/to и связанных типов объектов. Также можно выполнить вызов API [GET entities list](/managed/dynatrace-api/environment-api/entity-v2/get-entities-list "Просмотр списка отслеживаемых объектов через Dynatrace API.") для получения фактических объектов с их значениями связей (например, объект типа `service` может иметь связь `calls` «от себя» к другому `service`).
4. Нажмите **Preview**, чтобы увидеть объекты, соответствующие правилу и бывшие активными за последние 72 часа. (При применении зоны управления будут отображаться все объекты, соответствующие правилам за заданный период.)

   ![Предварительный просмотр правила на основе селектора объектов](https://dt-cdn.net/images/mz-rules-6a-998-ebeb2f11bd.png "Предварительный просмотр правила на основе селектора объектов")

5. Нажмите **Save changes**.

## Применение правил зон управления

* Условия применяются с логикой `AND` — все условия в правиле должны выполняться для его применения к объекту.
* Правила применяются с логикой `OR` — любое правило должно применяться для включения объекта в зону управления.
* При создании правил для некоторых объектов можно распространить доступ на связанные топологические объекты без создания дополнительного правила. Например, при создании правила для сервисов можно включить базовые хосты и группы процессов. См. раздел [Добавление правила на основе интерфейса](#ui) выше.

  В других случаях распространение доступа на связанные топологические объекты происходит неявно. Например, при предоставлении доступа к хосту в зоне управления пользовательские метрики, принятые через этот хост, также становятся доступны в зоне управления. Обратите внимание, что это не включает все измерения данных метрик, а только те, которые были отправлены для вашего хоста.

  В случаях, когда такое распространение недоступно, необходимо явно создавать правила для объектов, которые вы хотите включить в зону управления. Например, правило зоны управления, применяемое к **Host groups**, не даёт автоматического доступа к хостам в этих группах — необходимо явно добавить правила для **Hosts**, как показано в разделе [Примеры](#examples) ниже.

  Зоны управления всегда неявно распространяются на следующие связанные объекты. Однако это не распространяется на правила на основе селектора объектов.

  | От | К |
  | --- | --- |
  | Process Group | Process Group Instance, Container Group, Container Group Instance |
  | Hypervisor | VCenter |
  | Host | EC2 Instance, Container Group Instance |
  | AWS Credentials | AWS availability zone, AWS lambda function, AWS application load balancer, AWS network load balancer, EC2 instance, Custom device, Custom device group, Auto scaling group, Relational database service |
  | Synthetic Test | Application |

* При добавлении объекта с тегами в зону управления через API в момент создания объекта возможна задержка назначения зоны управления в зависимости от количества и сложности правил тегирования. Рекомендации по ускорению назначения тегов и зон управления см. в разделе [Лучшие практики масштабирования правил тегирования и зон управления](/managed/manage/tags-and-metadata/basic-concepts/best-practice-tagging-at-scale "Оптимизация правил автоматического тегирования и зон управления.").
* Нельзя определять правила зон управления, где селектор объектов одной зоны фильтрует по другой зоне управления. Предикаты зон управления, такие как `mzID` или `mzName`, не допускаются в строках селектора объектов. Это означает, что, например, нельзя определить зону управления A как содержащую хосты, принадлежащие зоне управления B. Правила зон управления на основе других зон увеличивают количество запусков механизма условной обработки и могут значительно замедлить назначение зон управления. Дополнительную информацию см. в разделе [Лучшие практики масштабирования правил тегирования и зон управления](/managed/manage/tags-and-metadata/basic-concepts/best-practice-tagging-at-scale "Оптимизация правил автоматического тегирования и зон управления.").

  В качестве обходного решения используйте одинаковую строку селектора объектов в обеих зонах. Например:

  + Зона управления A содержит правило `type(SERVICE),entityName.startsWith("myService"),tag("my:tag")`.
  + Зона управления B должна включать сервисы, вызывающие сервисы из зоны управления A. Для этого определите правило `type(SERVICE),fromRelationships.calls(type(SERVICE),entityName.startsWith("myService"),tag("my:tag"))`.

## Примеры

Пример 1: правила зоны управления, предоставляющие доступ к хостам определённых групп; пользователи, назначенные в зону, могут фильтровать по доступным группам хостов.

1. Настройте правило для хостов.

   1. Выберите **Monitored entity** как **Rule type**.
   2. Выберите **Hosts** в поле **Rule applies to**.
   3. Добавьте условие.
   4. Выберите **Host group** как **Property**.
   5. Выберите или найдите группу хостов в поле **Value**.
   6. Нажмите **Preview** для просмотра списка соответствующих объектов.

      ![Предварительный просмотр правила для хостов](https://dt-cdn.net/images/mz-sample-1-1a-1026-c1e2a8cd09.png "Предварительный просмотр правила для хостов")

   7. Нажмите **Save changes**.

   Добавьте правило таким образом для каждого набора хостов на группу хостов.

2. Настройте правило для групп хостов.

   Чтобы пользователи видели группы хостов, содержащие хосты в зоне управления, необходимо создать правила для групп хостов — по одному для каждой группы. Это позволяет пользователям фильтровать по группам хостов на странице **Hosts**.

   1. Выберите **Monitored entity** как **Rule type**.
   2. Выберите **Host groups** в поле **Rule applies to**.
   3. Добавьте условие.
   4. Выберите **Host group name** как **Property**.
   5. Определите условие для имени группы хостов, например строку, содержащуюся в имени группы.
   6. Нажмите **Preview** для просмотра списка соответствующих групп.

      ![Правило для группы хостов](https://dt-cdn.net/images/mz-sample-1-2a-1025-fd96aea449.png "Правило для группы хостов")

   7. Нажмите **Save changes**.

   Добавьте правило таким образом для каждой группы хостов, которую нужно включить в зону управления. Имена групп хостов отображаются в фильтре на странице **Hosts** только при наличии соответствующего правила зоны управления. Как правило, следует предоставлять доступ к тем же группам хостов, которым принадлежат хосты в вашей зоне управления.

При применении зоны управления пользователь видит только назначенные хосты и может фильтровать страницу **Hosts** по **Host group**. Группы хостов соответствуют тем, что определены в правилах зоны управления.

![Фильтр групп хостов](https://dt-cdn.net/images/mz-sample-1-3-1612-2b773ed1d7.png "Фильтр групп хостов")

Пример 2: правила зоны управления, предоставляющие доступ ко всем синтетическим мониторам

Правила зон управления можно определять для трёх типов синтетических объектов:

* Браузерные мониторы
* HTTP-мониторы
* Мониторы третьих сторон

Для предоставления доступа ко всем синтетическим мониторам необходимо определить правила для всех мониторов каждого типа.

![Зоны управления: синтетические объекты](https://dt-cdn.net/images/mz-sample-2-1a-823-7985b67d21.png "Зоны управления: синтетические объекты")

Например, чтобы создать правило для всех **Browser monitors**:

Укажите условие **Browser monitor name exists**. Нажмите **Preview** для просмотра соответствующих объектов.

![Правило для браузерных мониторов](https://dt-cdn.net/images/mz-sample-2-2a-1119-c1287aafd6.png "Правило для браузерных мониторов")

Аналогичным образом настройте правила для HTTP-мониторов и мониторов третьих сторон, чтобы охватить все синтетические мониторы в вашем окружении.

Если пользователю должна быть разрешена возможность создавать или редактировать синтетические мониторы в зоне управления, необходимо предоставить разрешение **Manage monitoring settings** через [разрешения](/managed/manage/identity-access-management/permission-management/role-based-permissions "Разрешения на основе ролей") на уровне зоны управления или окружения.

Пример 3: правила зоны управления, предоставляющие доступ к конкретным измерениям определённых принятых метрик

Можно предоставить доступ к принятой метрике, отфильтрованной по значению измерения, чтобы в зоне управления были доступны только конкретные измерения данной метрики (например, для построения графиков и анализа).

1. Выберите объект **Dimensional data** типа **METRIC**.
2. Заполните условие для имени метрики (**METRIC\_KEY**). Можно предоставить доступ к конкретной метрике, указав полное имя, или к набору метрик, указав начальную строку (например, `business.revenue`).

   ![Правило зоны управления для имени пользовательской метрики](https://dt-cdn.net/images/mz-sample-3-1-955-64a30c6239.png "Правило зоны управления для имени пользовательской метрики")

3. Для фильтрации по конкретным измерениям метрик добавьте условие для измерения. Выберите **DIMENSION** и укажите имя (**Key**) и значение (**Value**) измерения. Например, для фильтрации бизнес-метрик по восточному региону США укажите имя измерения `region` и значение `useast`.

   ![Размерные данные метрик в зоне управления](https://dt-cdn.net/images/mz-sample-3-2-937-3087a6c4e0.png "Размерные данные метрик в зоне управления")

4. Нажмите **Save changes**.

Если зона управления уже предоставляет доступ к хосту, через который принимается пользовательская метрика и её измерения, доступ к этой метрике предоставляется автоматически — явное правило не требуется. Обратите внимание, что это не включает все измерения пользовательской метрики, а только те, которые были отправлены для вашего хоста.

Пример 4: правила зоны управления, предоставляющие доступ к конкретным измерениям определённых принятых журналов

Можно предоставить доступ к журналам, отфильтрованным по значению атрибута (в качестве альтернативы фильтрации журналов по объектам).

1. В поле **Rule type** выберите **Dimensional data**.
2. В поле **Type** выберите **LOG**.
3. Добавьте условие для журналов.

   1. Для типа условия выберите **DIMENSION**.
   2. Укажите имя атрибута журнала в поле **Key**. Можно указать только одно значение атрибута.
   3. Добавьте фразу для поиска в поле **Value**.
   4. Для **Operator** можно выбрать `begins with` или `equals` для введённого **Value**. Например, для фильтрации журналов по конкретному имени развёртывания Kubernetes укажите ключ `k8s.deployment.name` и значение `begins with` `automation-server`.
4. Нажмите **Save changes**.

![Пример правила зоны управления для журналов](https://dt-cdn.net/images/mz-log-example-726-67634de796.png "Пример правила зоны управления для журналов")

Если зона управления уже предоставляет доступ к хосту, через который принимаются журналы, доступ к этим журналам предоставляется автоматически. Это означает, что явное размерное правило для таких журналов не требуется.

Пример 5: правило на основе селектора объектов с использованием связей

Для фильтрации сервисов, напрямую вызывающих сервис с именем `JourneyService`, можно выполнить вызов API [GET all entity types](/managed/dynatrace-api/environment-api/entity-v2/get-all-entity-types "Просмотр всех типов отслеживаемых объектов через Dynatrace API.") для проверки типа объекта и связей для сервисов.

На основе полученных данных можно создать правило на основе селектора объектов для типа `service` со связью `calls` «от себя» к типу `service` с именем `JourneyService`:

`type(SERVICE),fromRelationship.calls(type(SERVICE),entityName(JourneyService))`

Это также можно записать как `type(SERVICE),fromRelationship.calls(type(SERVICE) AND entityName.equals(JourneyService))`.

![Правило на основе селектора объектов с использованием связей](https://dt-cdn.net/images/mz-sample-4-1-954-606bb32f43.png "Правило на основе селектора объектов с использованием связей")
