# Настройка параметров конфиденциальности данных

* Практическое руководство
* Чтение 5 мин
* Updated on Feb 26, 2026

Dynatrace позволяет детально контролировать, какие данные вы хотите собирать. В зависимости от ваших сценариев использования вы можете настроить Dynatrace так, чтобы получать нужный объём [данных из мониторируемых окружений](/managed/manage/data-privacy-and-security/data-privacy/personal-data-captured-by-dynatrace "Узнайте, какие типы пользовательских данных могут собираться при мониторинге Dynatrace и какие методы доступны для маскирования персональных данных конечных пользователей."). Однако помните, что ответственность за защиту персональных данных ваших клиентов при использовании Dynatrace лежит на вас.

Предусмотрено несколько уровней маскирования: настраиваемые на уровне всего окружения (применяются ко всему тенанту) и более детальные — для следующих мониторируемых сущностей:

* Приложения
* Сервисы (группы процессов)
* Источники логов

Чтобы открыть глобальные настройки конфиденциальности данных окружения:

1. Перейдите в **Settings** > **Preferences** > **Data privacy**. Откроется страница **Data privacy**.
2. Переключайтесь между вкладками, чтобы ознакомиться со всеми параметрами конфиденциальности.

Если не указано иное, все параметры на странице **Data privacy** применяются как к данным, собираемым из браузеров конечных пользователей, так и к данным, собираемым OneAgent на стороне сервера.

Помимо настройки параметров на странице **Data privacy**, вы также можете [маскировать логи, собираемые OneAgent](#log-masking), [ограничивать доступ к просмотру персональных данных](#restrict-view-access) и [помечать некоторые атрибуты запросов как конфиденциальные](#conf-attribute).

Обратите внимание, что Dynatrace маскирует данные в соответствии с [тремя уровнями защиты данных](/managed/manage/data-privacy-and-security/data-privacy/levels-of-data-protection "Узнайте, как Dynatrace защищает данные конечных пользователей, применяя ситуативные уровни защиты."): **при захвате**, **при хранении** и **при отображении**. В следующих разделах параметры конфиденциальности сгруппированы по этим уровням защиты.

## Маскирование при захвате

При маскировании на этапе захвата данные маскируются при первом контакте с Dynatrace. Исходные данные не покидают мониторируемое окружение.

### Маскирование на стороне OneAgent

OneAgent версии 1.277+ для URL-адресов  
OneAgent версии 1.285+ для сообщений об исключениях

Чтобы открыть этот параметр, перейдите в **Settings** > **Preferences** > **Data privacy** > **OneAgent-side masking**.

🔴 По умолчанию отключено

OneAgent может маскировать чувствительные точки данных при первом контакте. Конфигурация применяется непосредственно на OneAgent; настроенные точки данных не отправляются на серверы Dynatrace и становятся недоступны.

* В зависимости от сценариев использования и потребностей в данных вы можете создавать пользовательские правила маскирования для точек данных, описанных в таблице ниже.
* Dynatrace также предоставляет встроенные правила маскирования, описанные в таблице ниже. По умолчанию эти правила отключены во всех платных (не пробных) окружениях Dynatrace.

  | Тип чувствительных данных | Строка маскирования для встроенного правила OneAgent версий 1.330 и ниже | Строка маскирования для встроенного правила OneAgent версии 1.331+ |
  | --- | --- | --- |
  | Адреса электронной почты в URL-адресах [1](#fn-1-1-def) | `<masked>` | `<masked-email-oa>` |
  | Значения параметров запросов | `<masked>` | `<masked-value-oa>` |
  | IBAN и номера платёжных карт | `<masked>` | `<masked-financial-oa>` |
  | Идентификаторы и последовательные числа | `<masked>` | `<masked-digits-oa>` |

  1

  OneAgent версии 1.309+ может маскировать адреса электронной почты, содержащие символы в процентном кодировании (например, `%40`). Поддерживается как URL-кодирование, так и base64URL-кодирование.

Эти параметры можно применить к конкретным мониторируемым группам процессов или глобально ко всему окружению.

* Для конкретной группы процессов: перейдите в **Technologies & Processes** > категория группы процессов > группа процессов > **Settings** > **OneAgent-side masking**.
* Глобально: перейдите в **Settings** > **Preferences** > **Data privacy** > **OneAgent-side masking**.

Параметры маскирования на стороне OneAgent не влияют на [Dynatrace RUM JavaScript](/managed/observe/digital-experience/rum-concepts/applications#web "Узнайте о мониторируемых приложениях в Real User Monitoring и поддерживаемых типах приложений."). Для веб-приложений используйте [параметр **Mask personal data in URIs**](/managed/observe/digital-experience/web-applications/additional-configuration/configure-real-user-monitoring-according-to-gdpr#mask-uris "Узнайте о параметрах конфиденциальности Dynatrace для обеспечения соответствия вашего веб-приложения требованиям по защите данных."), чтобы управлять маскированием чувствительных точек данных в URL-адресах.

Чтобы воспользоваться преимуществами нескольких уровней защиты, настраивайте параметры маскирования при захвате независимо от параметров маскирования при хранении и при отображении. Обратите внимание, что данные, замаскированные при захвате, становятся недоступны. Дополнительное применение параметров маскирования при хранении создаёт второй уровень защиты, что может быть полезно для различных требований соответствия.

### Маскирование логов

В Log Management and Analytics вы можете воспользоваться [полностью настраиваемыми правилами маскирования](/managed/upgrade/unavailable-in-managed "Данная функция недоступна в Dynatrace Managed.") для логов, собираемых OneAgent.

### Маскирование IP-адресов конечных пользователей

Чтобы открыть этот параметр, перейдите в **Settings** > **Preferences** > **Data privacy** > **IP masking**.

🟢 По умолчанию включено

Dynatrace собирает IP-адреса и GPS-координаты конечных пользователей для определения региона, из которого они обращаются к вашему приложению.

При включённом параметре **Mask end-user IP addresses and GPS coordinates** Dynatrace маскирует IP-адреса конечных пользователей и GPS-координаты, собранные средствами Real User Monitoring и серверного мониторинга. Последний октет мониторируемых IPv4-адресов и последние 80 бит IPv6-адресов заменяются нулями. GPS-координаты округляются до 1 десятичного знака (~10 км). Маскирование выполняется внутри приложения, мониторируемого процесса или конечной точки обработки маяков, поэтому данные уже замаскированы до отправки (в транзите) в кластер Dynatrace. Поиск местоположения выполняется с использованием анонимизированных IP-адресов и GPS-координат.

Параметр **Mask end-user IP addresses and GPS coordinates** > **Mask all IP addresses** включён по умолчанию для новых окружений.

Для мобильных приложений Dynatrace использует координаты устройства, полученные через GPS или Wi-Fi. Если приложение имеет разрешение на использование геолокационной информации, Dynatrace использует её для определения ближайшего к сообщённому GPS-местоположению города. В противном случае Dynatrace использует [базу данных MaxMind Geo2](https://www.maxmind.com/).

### Маскирование действий пользователей

Чтобы открыть этот параметр, перейдите в **Settings** > **Preferences** > **Data privacy** > **General**.

🔴 По умолчанию отключено

Параметр **Mask user actions (web applications only)** влияет только на Real User Monitoring для веб-приложений. При включении этого параметра Dynatrace использует общие значения для имён действий пользователей.

Когда Dynatrace обнаруживает действие пользователя, вызывающее загрузку страницы или AJAX/XHR-запрос, оно формирует имя действия пользователя на основе:

* Типа пользовательского события, например `click on...`, `loading of page...` или `keypress on...`
* Заголовка, подписи, метки, значения, ID, className или другого доступного свойства связанного HTML-элемента, например изображения, кнопки, флажка или текстового поля

В большинстве случаев подход к именованию действий пользователей по умолчанию работает хорошо, давая такие имена, как:

* `click on "Search" on page /search.html`
* `keypress on "Feedback" on page /contact.html`
* `touch on "Homescreen" of page /list.jsf`

В редких случаях адреса электронной почты, имена пользователей или другие конфиденциальные данные могут непреднамеренно попасть в имена действий пользователей. Это происходит, когда конфиденциальные данные включены в метку, атрибут или другое значение HTML-элемента, что приводит к именам действий вида `click on "My Account Number: 1231231"`. Если такие конфиденциальные данные присутствуют в именах действий пользователей вашего приложения, включите **Mask user actions (web applications only)**. Этот параметр заменяет конкретные имена и значения HTML-элементов общими именами.

При включённом маскировании имён действий пользователей приведённые выше имена отображаются следующим образом:

* `click on INPUT on page /search.html`
* `keypress on TEXTAREA on page /contact.html`
* `touch on DIV of page /list.jsf`

Чтобы избежать сбора персональных данных для действий пользователей в мобильных приложениях, ознакомьтесь с информацией о [маскировании действий пользователей в мобильных приложениях](/managed/observe/digital-experience/mobile-applications/additional-configuration/configure-rum-privacy-mobile#user-action-masking "Воспользуйтесь параметрами конфиденциальности Dynatrace для обеспечения соответствия ваших мобильных приложений требованиям по защите данных.").

## Маскирование при хранении

При маскировании на этапе хранения данные отправляются на сервер Dynatrace для оптимального анализа и маскируются перед сохранением.

### Маскирование персональных данных в URI

Чтобы открыть этот параметр, перейдите в **Settings** > **Preferences** > **Data privacy** > **General**.

🔴 По умолчанию отключено

Dynatrace собирает полные URI запросов, отправляемых из браузеров настольных компьютеров и мобильных устройств, а также URI запросов, отправляемых и принимаемых внутри мониторируемых серверных процессов. URI могут содержать персональные данные, такие как имена пользователей, пароли или идентификаторы.

При включённом параметре **Mask personal data in URIs** Dynatrace обнаруживает персональные данные — адреса электронной почты, IBAN, номера платёжных карт, IP-адреса, UUID и другие идентификаторы — в URI, заголовках, сообщениях об исключениях и данных, собранных для атрибутов запросов. Он маскирует эти данные при хранении, заменяя их строкой `<masked>`. Значения параметров запросов также заменяются строкой `<masked>`. Идентификаторы и числа должны содержать не менее 5 десятичных или шестнадцатеричных цифр для маскирования.

Примеры маскирования URI

| Тип персональных данных | Пример до маскирования | Пример после маскирования |
| --- | --- | --- |
| Адрес электронной почты | `https://example.com/user/john.doe@example.com/profile` | `https://example.com/user/<masked>/profile` |
| Значение параметра запроса | `https://example.com?country=Austria&city=Linz` | `https://example.com?country=<masked>&city=<masked>` |
| Номер платёжной карты | `https://example.com/checkout?card=4111111111111111` | `https://example.com/checkout?card=<masked>` |
| IP-адрес | `https://192.168.10.25/dashboard` | `https://<masked>/dashboard` |

В результате персональные данные маскируются при анализе распределённых трасс, анализе ошибок, именах действий пользователей для RUM и в других местах Dynatrace.

Обратите внимание на разницу между данными, замаскированными звёздочками `*****`, и данными, замаскированными строкой `<masked>`. Подробнее см. в разделе [Маскирование при отображении](/managed/manage/data-privacy-and-security/configuration/configure-global-privacy-settings#masking-at-display "Узнайте, как настроить маскирование конфиденциальности данных для IP-адресов, геолокации и имён действий пользователей.").

## Маскирование при отображении

При маскировании на этапе отображения данные хранятся в исходном виде, но доступны только выбранным вами пользователям.

### Ограниченный доступ к просмотру персональных данных

Dynatrace автоматически считает определённые собираемые точки данных конфиденциальными и отображает их только пользователям, имеющим разрешение **View sensitive request data** ([разрешения](/managed/manage/identity-access-management/permission-management/role-based-permissions "Разрешения на основе ролей")). Остальные пользователи видят, что точка данных существует, но персональные данные маскируются звёздочками `*****`.

Если ваша организация собирает персональные данные пользователей, такие как адреса электронной почты, IP-адреса или пароли, в ходе мониторинга, следует ограничить доступ к просмотру этих персональных данных, чтобы только авторизованные пользователи могли их видеть. Также обратите внимание, что только пользователи с разрешением **View sensitive request data** могут переопределять параметры маскирования данных.

Следующие типы данных считаются конфиденциальными и маскируются при отображении:

* Атрибуты запросов, [помеченные как конфиденциальные](#conf-attribute)
* IP-адреса клиентов
* Сообщения об исключениях
* Параметры запросов URL
* HTTP-заголовки
* HTTP POST-параметры
* Исходные захваченные значения аргументов метода (результирующий атрибут запроса обрабатывается отдельно)

### Конфиденциальные атрибуты запросов

[Атрибуты запросов](/managed/observe/application-observability/services/request-attributes "Узнайте, что такое атрибуты запросов и как использовать их на всех уровнях представлений анализа сервисов.") — это пары ключ-значение метаданных, по которым можно выполнять фильтрацию во всех представлениях анализа сервисов и распределённых трасс Dynatrace.

Dynatrace позволяет выбирать, следует ли [помечать атрибут запроса как конфиденциальный](/managed/observe/application-observability/services/request-attributes#confidential "Узнайте, что такое атрибуты запросов и как использовать их на всех уровнях представлений анализа сервисов."). Для управления атрибутами запросов необходимо иметь разрешение **Manage capturing of sensitive request data** ([разрешения](/managed/manage/identity-access-management/permission-management/role-based-permissions "Разрешения на основе ролей")).

## Режим opt-in

Чтобы открыть этот параметр, перейдите в **Settings** > **Preferences** > **Data privacy** > **General**.

🔴 По умолчанию отключено

Чтобы предоставить конечным пользователям возможность решать, следует ли отслеживать их активность, включите режим opt-in.

По умолчанию RUM автоматически создаёт [куки](/managed/manage/data-privacy-and-security/data-privacy/rum-cookies-and-web-storage#dynatrace-rum-cookies "Узнайте, как RUM и Session Replay используют куки, веб-хранилище и IndexedDB."). При включённом параметре **Data-collection and opt-in mode** ни OneAgent, ни RUM JavaScript не устанавливают куки, а RUM JavaScript не собирает никаких данных. После того как конечный пользователь принимает вашу политику куки, вы можете активировать RUM для этого пользователя через вызов JavaScript API [`dtrum.enable()`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#enable). Используя вызов [`dtrum.disable()`](https://docs.dynatrace.com/javascriptapi/doc/types/dtrum.html#disable), вы можете реализовать диалог, позволяющий конечным пользователям прекратить отправку данных мониторинга в Dynatrace даже после того, как они ранее согласились на это и был вызван `dtrum.enable()`.

## Do Not Track

Чтобы открыть этот параметр, перейдите в **Settings** > **Preferences** > **Data privacy** > **General**.

🟢 По умолчанию включено

Ещё один способ защиты конфиденциальности конечных пользователей — функция «Do Not Track». Когда пользователь включает эту функцию, его браузер добавляет HTTP-заголовок запроса `DNT` ко всем исходящим веб-запросам. Этот заголовок указывает, что пользователь предпочитает не быть отслеживаемым.

После включения параметра **Comply with "Do Not Track" browser settings** доступны два варианта:

* **Capture anonymous user sessions for "Do Not Track"-enabled browsers**: при обнаружении заголовка `DNT` Dynatrace собирает данные RUM, но исключает всю персональную информацию, которая может привести к идентификации пользователя. IP-адрес маскируется, а информация о [пользовательском теге](/managed/observe/digital-experience/rum-concepts/user-and-error-events#user-tagging "Узнайте о событиях пользователей и ошибках и типах событий, собираемых Dynatrace.") не отправляется.

  При включённом [параметре **User tracking**](#user-tracking) Dynatrace по-прежнему устанавливает постоянный куки для обнаружения возвращающихся пользователей.
* **Turn Real User Monitoring off for "Do Not Track"-enabled browsers**: при обнаружении заголовка `DNT` Dynatrace не собирает никаких данных из браузеров с включённым параметром «Do Not Track».

Если отключить **Comply with "Do Not Track" browser settings**, Dynatrace будет игнорировать настройку браузера «Do Not Track» и заголовок `DNT`.

Параметр **Comply with "Do Not Track" browser settings** > **Capture anonymous user sessions for "Do Not Track"-enabled browsers** включён по умолчанию для всех окружений и приложений.

## Отслеживание пользователей

Чтобы открыть этот параметр, перейдите в **Settings** > **Preferences** > **Data privacy** > **General**.

🔴 По умолчанию отключено

Параметр **Use persistent cookies for user tracking** позволяет включить или отключить использование постоянных куки для идентификации возвращающихся пользователей.

При включении RUM JavaScript устанавливает в браузерах конечных пользователей постоянный куки `rxVisitor`, указывающий, что данный браузер ранее использовался для доступа к вашему приложению. При отключении RUM Classic теряет возможность связывать сессии с одним пользователем при перезапуске браузера. Узнайте подробнее о том, [как хранится этот куки](/managed/manage/data-privacy-and-security/data-privacy/rum-cookies-and-web-storage#web-storage "Узнайте, как RUM и Session Replay используют куки, веб-хранилище и IndexedDB.").
