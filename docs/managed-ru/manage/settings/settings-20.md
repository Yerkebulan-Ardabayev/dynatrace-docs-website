# Фреймворк настроек Dynatrace

# Фреймворк настроек Dynatrace

* Explanation
* 4-min read
* Published Jan 18, 2023

Фреймворк Settings 2.0 предоставляет единый инструмент для управления различными конфигурациями в Dynatrace. Независимо от того, используете ли вы веб-интерфейс Dynatrace или [Settings API](/managed/dynatrace-api/environment-api/settings "Узнайте, что предлагает Dynatrace Settings API."), вы управляете одними и теми же конфигурациями с одинаковым эффектом.

Каждая конфигурация Settings 2.0 представляет собой *объект настроек* (определяющий конкретный набор параметров) и основана на *схеме настроек* (определяющей, какие параметры включает конфигурация). Схемы создаются и управляются Dynatrace; создавать собственные схемы нельзя. Объекты, напротив, полностью находятся под вашим контролем.

## Область применения и иерархия настроек

Многие настройки могут быть заданы для разных областей (для всего окружения мониторинга или конкретного объекта). Область по умолчанию — глобальная (всё окружение мониторинга).

Конфигурации, доступные через **Settings** (в Dynatrace перейдите в **Settings**), влияют на всё окружение. Вы можете проверить наличие переопределений на уровне объекта для настройки уровня окружения. Выберите **More** (**…**) > **Hierarchy and overrides**. Отсюда можно легко перейти к конфигурациям на уровне объекта, которые, в свою очередь, содержат информацию о родительской настройке.

![Global settings with overrides](https://dt-cdn.net/images/settings-global-1717-6ffb544dd6.png)

Глобальные настройки с переопределениями

Наиболее конкретная настройка всегда имеет приоритет. Например, конфигурация на уровне хоста переопределяет конфигурацию на уровне группы хостов, а конфигурация на уровне группы хостов, в свою очередь, переопределяет конфигурацию на уровне окружения.

![Entity-level settings](https://dt-cdn.net/images/settings-host-level-1238-af32c1f6d5.png)

Настройки на уровне объекта

Если настройка доступна на нескольких уровнях, управляйте ею на максимально высоком уровне. Настройки высокого уровня автоматически применяются ко всем дочерним объектам; применять их вручную не требуется.

## Упорядоченные настройки

Для некоторых настроек порядок элементов существенен. Например, правила именования оцениваются в определённом порядке, и применяется первое совпадающее правило.

Упорядоченные настройки можно определять на глобальном уровне и уровне объекта. Однако иерархия здесь работает немного иначе — настройки уровня объекта не переопределяют глобальные настройки, а расширяют их. Список настроек уровня объекта добавляется перед глобальным списком; таким образом, правила уровня объекта оцениваются первыми. Если ни одно из правил уровня объекта не применяется, оценка продолжается с глобальными правилами.

Упорядоченные элементы легко переставляются перетаскиванием в нужную позицию.

![Settings - reorder list](https://dt-cdn.net/images/settings-reorder-1139-338c062399.png)

Настройки — переупорядочение списка

## История конфигурации

Каждая конфигурация Settings 2.0 ведёт историю изменений, отслеживая изменения конфигурации и лиц, их внёсших. Для доступа к истории выберите **More** (**…**) > **Revision history**.

![Settings history](https://dt-cdn.net/images/settings-history-1242-a32e589da1.png)

## Разрешения и доступ

Доступ к настройкам управляется с помощью [политик IAM](/managed/manage/identity-access-management/permission-management/manage-user-permissions-policies "Работа с политиками"). Политики позволяют создавать гибкий и детализированный доступ к конфигурациям: пользователи имеют доступ только к тем настройкам, для которых им назначены разрешения. Для вступления политик в силу дополнительные разрешения не требуются. Политики предоставляют доступ к конфигурациям как через веб-интерфейс Dynatrace, так и через Settings API, если на уровне схемы не настроено иное.

Если требуется настроить детализированный доступ к определённым объектам, это можно сделать через зоны управления или группы хостов. Обратите внимание, что объект может входить в несколько зон управления, но только в одну группу хостов, что делает подход на основе групп хостов более прямолинейным, тогда как подход на основе зон управления обеспечивает большую гибкость.

Сведения о настройке политик доступа для настроек см. в разделе [Справочник по политикам IAM](/managed/manage/identity-access-management/permission-management/manage-user-permissions-policies/advanced/iam-policystatements#settings "Полный справочник по политикам IAM и соответствующим условиям для всех сервисов Dynatrace."). Ниже приведены примеры политик.

Предоставление доступа на чтение и запись к настройкам определений целевых показателей качества сервиса

```
ALLOW settings:objects:read, settings:objects:write, settings:schemas:read



WHERE settings:schemaId = "builtin:monitoring.slo";
```

Предоставление доступа на чтение и запись ко всем настройкам cloud automation

```
ALLOW settings:objects:read, settings:objects:write, settings:schemas:read



WHERE settings:schemaGroup = " group:cloud-automation";
```

Предоставление доступа только на чтение к определениям зон управления

```
ALLOW settings:objects:read, settings:schemas:read



WHERE settings:schemaId = "builtin:management-zones";
```

Предоставление доступа на чтение и запись ко всем настройкам группы хостов easyTravel и её хостов

```
ALLOW settings:schemas:read;



ALLOW settings:objects:read, settings:objects:write



WHERE settings:entity.hostGroup = "easyTravel";
```

Предоставление доступа на чтение и запись ко всем настройкам всех объектов с контекстом безопасности «easyTravel»

```
ALLOW settings:schemas:read;



ALLOW settings:objects:read, settings:objects:write



WHERE settings:dt.security_context = "easyTravel";
```

Предоставление доступа на чтение и запись ко всем настройкам всех объектов в зоне управления easyTravel

Эта политика также влияет на глобальные настройки, применяемые к зоне управления easyTravel.

```
ALLOW settings:schemas:read;



ALLOW settings:objects:read, settings:objects:write



WHERE environment:management-zone = "easyTravel";
```

Предоставление доступа на чтение и запись к настройкам аномалий инфраструктуры для хостов в зоне управления easyTravel

```
ALLOW settings:schemas:read;



ALLOW settings:objects:read, settings:objects:write



WHERE settings:schemaId = "builtin:anomaly-detection.infrastructure-hosts" AND environment:management-zone = "easyTravel";
```
